07.7 In Depth — chapter focus

The WIA Smart Wheelchair standard treats a power wheelchair as a layered data flow: the chair's motor and sensor stack → on-device safety supervisor → wire envelope (telemetry, route, safety event) → server-side care-team dashboards and federation → response envelope → on-chair haptic and audio feedback. Each step has a published shape; the goal of this chapter is to make that shape and its conformance gate explicit so an integrator can build a Smart-Wheelchair-conformant component without reading the entire specification first.

The relevant Phase artifacts are: a JSON envelope (Phase 1), a signed HTTP endpoint (Phase 2), a federation handshake when crossing organisational boundaries between care providers (Phase 3), and a bridge profile for ROS 2 navigation stacks and CANopen motor controllers (Phase 4). Implementations are tested by the open conformance suite at https://github.com/WIA-Official/wia-smart-wheelchair-conformance against a mock chair host.

Two recurring design choices show up across every chapter and are worth flagging once here. First, every wheelchair envelope carries an explicit comfort_profile_id pointer; a power chair affects the user's body directly, so the standard treats user comfort and clinical context as first-class concerns. Second, every wire envelope is signed with Ed25519 (IETF RFC 8032) and the federation handshake reuses the same shape that WIA-SOCIAL Phase 3 §5 fixes; this keeps the cross-standard mental model uniform.

07.8 Reference Implementation Walk-Through

The reference container wia/smart-wheelchair-host:1.0.0 implements every Phase 2 endpoint and ships with the open conformance suite preconfigured. A typical local exploration looks like:

docker run -p 8080:8080 wia/smart-wheelchair-host:1.0.0
curl -N "http://localhost:8080/sw/telemetry/wheelchair-001" \
     -H "Accept: text/event-stream"

The container ships with a small library of mock telemetry traces so an integrator can exercise every endpoint without driving an actual chair. Once the integration is working against the mock, the operator swaps in their real ROS 2 stack and CAN bus interface.

07.9 Operational Considerations

Wheelchair infrastructure has three operational considerations that integrators consistently underestimate. First, telemetry retention is a clinical-record decision; the standard makes retention a per-consent decision so an operator can size storage against the actual scope (insurance audit, fall-incident reconstruction, etc.). Second, safety events are time-critical; the host MUST page the on-call clinician within 60 seconds of a critical-severity event. Third, safety supervisor firmware updates require coordination with the navigation stack; the standard's envelope versioning lets the supervisor and navigator advance independently within a major version line.
