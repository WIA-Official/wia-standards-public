HSM 아키텍처 및 구성요소

내부 구조와 하드웨어 설계 원리

HSM 내부 아키텍처 개요

HSM의 보안성은 하드웨어 아키텍처 설계에서 시작됩니다. 일반 서버와 달리 HSM은 보안을 최우선으로 설계된 전용 하드웨어로, 다층 방어 메커니즘과 격리된 처리 환경을 통해 암호화 키와 연산을 보호합니다. 이 장에서는 HSM의 내부 구조, 주요 구성요소, 그리고 각 요소가 전체 보안 체계에 기여하는 방식을 상세히 살펴봅니다.

HSM 전체 아키텍처: ┌──────────────────────────────────────────────────────┐ │ 변조 방지 외부 케이스 │ │ ┌────────────────────────────────────────────────┐ │ │ │ 변조 감지 메시 레이어 │ │ │ │ ┌──────────────────────────────────────────┐ │ │ │ │ │ 보안 처리 환경 (Trust Anchor) │ │ │ │ │ │ │ │ │ │ │ │ ┌────────────┐ ┌─────────────────┐ │ │ │ │ │ │ │ 암호화 │ │ 키 저장소 │ │ │ │ │ │ │ │ 프로세서 │◄──►│ (비휘발성) │ │ │ │ │ │ │ └─────┬──────┘ └─────────────────┘ │ │ │ │ │ │ │ │ │ │ │ │ │ ┌─────▼──────┐ ┌─────────────────┐ │ │ │ │ │ │ │ TRNG │ │ 펌웨어 ROM │ │ │ │ │ │ │ │ (난수발생기)│ │ (불변) │ │ │ │ │ │ │ └────────────┘ └─────────────────┘ │ │ │ │ │ │ │ │ │ │ │ │ ┌────────────────────────────────────┐ │ │ │ │ │ │ │ 보안 부팅 및 무결성 검증 │ │ │ │ │ │ │ └────────────────────────────────────┘ │ │ │ │ │ └──────────────────────────────────────────┘ │ │ │ │ │ │ │ │ ┌──────────────────────────────────────────┐ │ │ │ │ │ I/O 컨트롤러 (네트워크/PCIe/USB) │ │ │ │ │ └──────────────────────────────────────────┘ │ │ │ └────────────────────────────────────────────────┘ │ │ │ │ 변조 감지 센서: 전압, 온도, 주파수, 방사선, 케이스 │ └──────────────────────────────────────────────────────┘

보안 프로세서 (Secure Cryptographic Processor)

HSM의 핵심은 보안 프로세서입니다. 이 전용 칩은 모든 암호화 연산을 수행하며, 키 자료에 대한 직접 접근 권한을 가진 유일한 구성요소입니다. 일반 범용 CPU와 달리 보안 프로세서는 다음과 같은 특화된 기능을 제공합니다.

구성요소 기능 보안 특징
암호화 가속기 RSA, ECC, AES 등 하드웨어 가속 사이드채널 공격 방지 설계, 상수 시간 연산
보안 메모리 키 및 중간 연산 결과 저장 물리적으로 읽기 불가능, 전원 차단 시 자동 삭제
암호화 코프로세서 전용 수학 연산 (모듈러 지수, 타원곡선) 타이밍 공격 방지, 마스킹 적용
보안 부팅 ROM 신뢰 루트, 펌웨어 검증 변경 불가능한 코드, 서명 검증
격리 실행 환경 신뢰 실행 환경 (TEE) 메모리 암호화, 격리된 주소 공간

암호화 가속기 상세

현대 HSM은 다양한 암호화 알고리즘에 대한 하드웨어 가속기를 내장하고 있어, 소프트웨어 구현 대비 10~100배 빠른 성능을 제공합니다.

알고리즘 일반 CPU 성능 HSM 가속기 성능 배수
RSA-2048 서명 ~500 ops/sec 10,000+ ops/sec 20배
RSA-2048 검증 ~15,000 ops/sec 30,000+ ops/sec 2배
ECDSA P-256 서명 ~2,000 ops/sec 20,000+ ops/sec 10배
AES-256 GCM ~500 MB/sec 5,000+ MB/sec 10배
SHA-256 해싱 ~200 MB/sec 2,000+ MB/sec 10배

진정한 난수 생성기 (TRNG)

암호화의 강도는 난수의 품질에 직접적으로 의존합니다. HSM은 소프트웨어 난수 생성기(PRNG) 대신 물리적 현상을 기반으로 한 진정한 난수 생성기(True Random Number Generator, TRNG)를 사용합니다.

TRNG 구현 방식

방식 물리적 원리 장점 단점
열 잡음 저항의 전자 열운동 저렴, 소형화 가능 온도에 영향받음
링 오실레이터 지터(jitter) 누적 FPGA 구현 용이 예측 가능한 패턴 가능
방사성 붕괴 알파/베타 입자 방출 완전히 예측 불가능 비싸고 규제 대상
광자 도착 시간 양자역학적 불확정성 최고 품질의 무작위성 복잡하고 고가

TRNG 품질 검증

HSM의 TRNG는 NIST SP 800-90B 표준에 따라 엔트로피를 검증받습니다. 지속적인 건강 테스트를 통해 난수 품질을 모니터링합니다.

TRNG 연속 건강 테스트: 1. 반복 카운트 테스트 (Repetition Count Test) - 연속된 같은 값이 과도하게 나타나는지 검사 - 임계값: 일반적으로 5~10회 2. 적응형 비율 테스트 (Adaptive Proportion Test) - 특정 윈도우 내에서 동일 값 출현 빈도 검사 - 윈도우 크기: 512~1024 샘플 3. 자가상관 테스트 - 이전 값과의 상관관계 검사 - 주기성 패턴 감지 실패 시 조치: - TRNG 출력 차단 - 경고 이벤트 생성 - 시스템 관리자 알림 - 선택적으로 HSM 작동 중단

키 저장소 (Key Storage)

HSM은 암호화 키를 안전하게 저장하기 위해 특수 설계된 비휘발성 메모리를 사용합니다. 이 메모리는 물리적 추출이 불가능하도록 보호됩니다.

키 저장 기술

저장 방식 기술 특징
배터리 백업 RAM SRAM + 리튬 배터리 빠른 접근, 전원 차단 시 휘발 가능
플래시 메모리 암호화된 NAND/NOR 플래시 영구 저장, 하드웨어 암호화 필수
EEPROM 전기적 소거 가능 ROM 소용량, 높은 내구성
eFuse 일회성 프로그램 가능 퓨즈 루트 키 저장, 변경 불가
PUF (Physical Unclonable Function) 제조 공정 변이 활용 복제 불가능한 고유 키 파생

키 암호화 키 (KEK) 계층

HSM 내부에서도 키는 암호화되어 저장됩니다. 마스터 키 암호화 키(KEK)를 사용한 계층적 키 관리 구조를 적용합니다.

HSM 키 계층 구조: ┌───────────────────────┐ │ Hardware Root Key │ │ (PUF 또는 eFuse) │ ← 하드웨어에 내장 └───────────┬───────────┘ 변경 불가 │ ┌───────────▼───────────┐ │ Master KEK │ │ (초기화 시 생성) │ ← 관리자만 접근 └───────────┬───────────┘ │ ┌───────────▼───────────┐ │ Domain KEK │ │ (파티션별) │ ← 멀티테넌트 격리 └───────────┬───────────┘ │ ┌───────────▼───────────┐ │ User Keys │ │ (애플리케이션 키) │ ← 실제 사용 키 └───────────────────────┘ 키 래핑 프로세스: 1. 사용자 키를 Domain KEK로 암호화 2. Domain KEK를 Master KEK로 암호화 3. Master KEK를 Hardware Root Key로 암호화 4. 암호화된 키만 비휘발성 메모리에 저장

변조 방지 및 감지 메커니즘

HSM의 물리적 보안은 다층 방어 체계로 구성됩니다. 각 계층은 서로 다른 유형의 공격을 방어합니다.

외부 케이스 보안

  • 에폭시 밀봉: 주요 칩을 불투명 에폭시로 완전 덮어 물리적 프로빙 방지
  • 메시 레이어: PCB 전체를 감싸는 미세 전도성 메시, 드릴링 시도 시 단락
  • 케이스 스위치: 케이스 개봉 시 즉시 감지하는 마이크로 스위치
  • 변조 방지 나사: 특수 형상의 나사로 일반 공구로 열 수 없음

환경 센서

센서 유형 감지 대상 정상 범위 공격 예시
전압 센서 입력 전압 변동 ±5% 이내 전압 글리치 공격
주파수 센서 클럭 주파수 조작 ±1% 이내 클럭 글리치 공격
온도 센서 작동 온도 0°C ~ 50°C 냉각/가열 공격
방사선 센서 X-ray, 레이저 백그라운드 수준 칩 분석 시도
전자기 센서 EM 펄스 정상 EMI 범위 EMP 공격

네트워크 및 통신 인터페이스

HSM은 다양한 물리적 인터페이스를 통해 호스트 시스템과 통신합니다. 각 인터페이스는 보안과 성능 요구사항에 따라 선택됩니다.

인터페이스 대역폭 지연시간 사용 사례
PCIe 3.0 x8 ~6.8 GB/s < 1 μs 초고성능 암호화, HFT
10 GbE ~1.2 GB/s < 100 μs 데이터센터 배포
1 GbE ~120 MB/s < 1 ms 일반 엔터프라이즈
USB 3.0 ~400 MB/s < 10 ms 휴대용 HSM, 개발자 토큰
시리얼 (RS-232) ~100 KB/s < 10 ms 레거시 시스템, 관리 콘솔

안전한 통신 프로토콜

HSM과 호스트 간 통신은 암호화되고 인증됩니다. 주요 프로토콜은 다음과 같습니다.

HSM 통신 보안 계층: 애플리케이션 계층: - PKCS#11 API (암호화 작업 요청) - REST API (클라우드 HSM) 전송 보안 계층: - TLS 1.3 (상호 인증) * 클라이언트 인증서 검증 * Perfect Forward Secrecy * 세션 암호화 네트워크 계층: - IPsec (네트워크 HSM) - VPN 터널 - 격리된 관리 VLAN 물리 계층: - 전용 네트워크 세그먼트 - 스위치 포트 보안

펌웨어 및 소프트웨어 구조

HSM의 펌웨어는 보안과 신뢰성을 위해 특별히 설계됩니다. 일반 OS와 달리 최소화된 신뢰 기반(Trusted Computing Base, TCB)을 유지합니다.

펌웨어 계층 구조

HSM 펌웨어 스택: ┌─────────────────────────────────────┐ │ 애플리케이션 API 레이어 │ │ - PKCS#11 구현 │ │ - JCE Provider │ │ - REST API 핸들러 │ └──────────────┬──────────────────────┘ │ ┌──────────────▼──────────────────────┐ │ 암호화 서비스 레이어 │ │ - 키 관리 (생성, 저장, 폐기) │ │ - 암호화 연산 (서명, 암호화 등) │ │ - 난수 생성 │ └──────────────┬──────────────────────┘ │ ┌──────────────▼──────────────────────┐ │ 하드웨어 추상화 레이어 (HAL) │ │ - 암호화 가속기 드라이버 │ │ - TRNG 드라이버 │ │ - 메모리 관리 │ └──────────────┬──────────────────────┘ │ ┌──────────────▼──────────────────────┐ │ 보안 부팅 및 무결성 검증 │ │ - 서명된 펌웨어 검증 │ │ - 측정된 부팅 (Measured Boot) │ │ - 롤백 방지 │ └──────────────┬──────────────────────┘ │ ┌──────────────▼──────────────────────┘ │ 하드웨어 (보안 프로세서, 메모리 등) │ └──────────────────────────────────────┘

보안 부팅 프로세스

HSM은 전원 투입 시마다 무결성을 검증하는 보안 부팅 프로세스를 수행합니다.

HSM 부팅 시퀀스: 1. 하드웨어 초기화 ├─ 보안 프로세서 리셋 ├─ TRNG 시작 및 테스트 └─ 센서 초기화 (온도, 전압 등) 2. 부팅 ROM 실행 (변경 불가) ├─ 하드웨어 자가진단 (POST) ├─ 펌웨어 서명 검증 │ └─ 실패 시 → 부팅 중단 └─ 롤백 방지 확인 3. 펌웨어 로드 및 검증 ├─ 펌웨어를 보안 메모리로 로드 ├─ 해시 계산 및 비교 └─ 측정값을 PCR에 저장 4. 암호화 서비스 초기화 ├─ KEK 계층 복원 ├─ 키 저장소 마운트 └─ 암호화 알고리즘 자가진단 5. 보안 상태 검증 ├─ 변조 센서 상태 확인 ├─ 이전 제로화 이벤트 검사 └─ 감사 로그 무결성 검증 6. API 서비스 시작 ├─ 네트워크 인터페이스 활성화 └─ 클라이언트 연결 수락 시작 부팅 시간: 일반적으로 30초~2분

파티셔닝 및 멀티테넌시

현대 HSM은 단일 물리적 장치를 여러 논리적 파티션으로 나누어 여러 애플리케이션이나 조직이 안전하게 공유할 수 있습니다.

HSM 파티션 격리

격리 메커니즘 구현 방식 보안 수준
암호화 격리 각 파티션마다 별도의 KEK 사용 높음 - 키 유출 시에도 다른 파티션 보호
메모리 격리 하드웨어 메모리 보호 단위(MPU) 높음 - 물리적 주소 공간 분리

감사 로그 격리 파티션별 독립 로그 파일 중간 - 로그 접근 제어 필요

요약

HSM의 아키텍처는 보안을 최우선으로 설계되었습니다. 핵심 구성요소로는 암호화 전용 보안 프로세서, 진정한 난수 생성기(TRNG), 변조 방지 키 저장소, 다층 물리적 보안 메커니즘이 있습니다. 보안 프로세서는 RSA, ECC, AES 등의 암호화 알고리즘에 대한 하드웨어 가속기를 내장하여 일반 CPU 대비 10~100배 빠른 성능을 제공합니다.

TRNG는 열 잡음, 링 오실레이터, 광자 도착 시간 등 물리적 무작위성을 활용하여 예측 불가능한 난수를 생성하며, NIST SP 800-90B 표준에 따라 지속적으로 품질을 검증합니다. 키 저장소는 플래시 메모리, EEPROM, PUF 등의 기술을 사용하며, 마스터 KEK를 정점으로 한 계층적 키 래핑 구조로 보호됩니다.

물리적 보안은 에폭시 밀봉, 메시 레이어, 케이스 스위치 등 외부 보호와 전압·온도·주파수·방사선 센서를 통한 환경 모니터링으로 구성됩니다. 변조 시도 감지 시 밀리초 내에 모든 키를 제로화합니다. 통신 인터페이스는 PCIe, 이더넷, USB 등 다양하며, TLS 1.3, IPsec 등으로 보안이 유지됩니다.

HSM 펌웨어는 최소화된 신뢰 기반(TCB)을 유지하며, 보안 부팅을 통해 매 전원 투입 시 무결성을 검증합니다. 현대 HSM은 파티셔닝을 통해 멀티테넌시를 지원하며, 암호화·메모리·인증·감사 로그를 격리하여 안전한 공유를 가능하게 합니다.

  2. 암호화 가속기가 일반 CPU 대비 높은 성능을 제공하는 이유를 설명하세요.
  3. 진정한 난수 생성기(TRNG)와 소프트웨어 난수 생성기(PRNG)의 차이를 설명하고, HSM이 TRNG를 사용하는 이유를 논하세요.
  4. TRNG의 연속 건강 테스트 2가지를 설명하고, 테스트 실패 시 조치를 기술하세요.
  5. HSM 키 저장소에 사용되는 5가지 기술을 비교하고 각각의 장단점을 설명하세요.
  6. HSM의 키 암호화 키(KEK) 계층 구조를 그림으로 그리고 각 계층의 역할을 설명하세요.
  7. HSM의 물리적 변조 방지 메커니즘 5가지를 나열하고 각각이 방어하는 공격 유형을 설명하세요.
  8. HSM의 환경 센서 5가지와 각 센서가 감지하는 공격 유형을 설명하세요.
  9. HSM의 보안 부팅 프로세스를 단계별로 설명하고, 각 단계의 중요성을 논하세요.
  10. HSM 파티셔닝의 격리 메커니즘 4가지를 설명하고 멀티테넌시 환경에서의 중요성을 논하세요.

실무 적용 체크리스트

  • 매니페스트 무결성: 모든 발급은 SHA-256 또는 Sigstore 서명을 부착해 변조를 차단합니다. 발급 후 변경은 새 매니페스트로 처리 되며 이전 매니페스트는 supersede 마킹과 함께 보존됩니다.
  • 보관 기간: ISO/IEC 17065:2012 감사 기대치를 충족하기 위해 7년 이상의 보관 정책을 명시합니다. 관할 규정이 더 길면 그것을 따릅니다.
  • 버전 관리: 정정·확장은 시맨틱 버저닝(SemVer 2.0.0)을 따릅니다. 필수 요건의 추가는 마이너, 호환 단절은 메이저 버전 갱신을 트리거합니다. 메이저 변경은 최소 90일 병행 호환 기간을 거치며 그 기간 동안 신·구 두 버전 모두 WIA 등록부에 Stable로 표기 됩니다.
  • 부정결과 보고: 누락된 지표는 사유와 함께 매니페스트에 기록 되어야 합니다. 사유가 없는 누락은 감사 시 실패로 간주됩니다.
  • 키 회전: 매니페스트 서명 키는 배포 정책에 따라 회전됩니다. 이전 키는 공개된 키 이력에 보존되어 과거 매니페스트의 사후 검증을 보장합니다.

참조 토폴로지

구현체의 운용 토폴로지는 다음 4가지 중 하나로 매니페스트에 선언됩니다: 단일 테넌트 엣지(조직당 1개 런타임), 멀티 테넌트 게이트웨이(헤더 기반 격리로 다중 테넌트 서비스), 연합 메시(다수 런타임이 디렉토리 서비스 통해 상호 인식), 에어갭 배치(라이브 연결 없이 서명된 패키지를 out-of-band 채널로 전송). 각 토폴로지는 참조 테스트 벡터를 포함하며 구현체가 자신의 배포가 표준 패턴과 일치하는지 검증할 수 있도록 합니다.
