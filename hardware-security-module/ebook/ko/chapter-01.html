제1장

HSM 소개

하드웨어 보안 모듈의 기초와 중요성

HSM이란 무엇인가?

하드웨어 보안 모듈(Hardware Security Module, HSM)은 암호화 키를 안전하게 생성, 저장, 관리하고 암호화 연산을 수행하는 물리적 컴퓨팅 장치입니다. HSM은 변조 방지 하드웨어 내부에서 민감한 암호화 작업을 수행함으로써 소프트웨어 기반 보안 솔루션이 제공할 수 없는 수준의 보호를 제공합니다.

일반적인 서버나 애플리케이션은 메모리, 디스크, 네트워크를 통해 암호화 키가 노출될 위험이 있습니다. 반면 HSM은 전용 하드웨어 칩셋 내부에서 모든 암호화 작업을 수행하며, 키 자료가 HSM 경계 밖으로 평문 형태로 나가지 않도록 보장합니다. 이러한 물리적 격리가 HSM의 핵심 가치입니다.

HSM의 핵심 기능

  • 안전한 키 생성: 하드웨어 기반 난수 생성기로 예측 불가능한 암호화 키 생성
  • 키 저장 및 보호: 변조 방지 하드웨어 내부에 암호화 키를 안전하게 저장
  • 암호화 연산: 서명, 검증, 암호화, 복호화 작업을 HSM 내부에서 수행
  • 물리적 보안: 침입 시도 감지 시 자동으로 키를 삭제하는 변조 방지 메커니즘
  • 규정 준수: FIPS 140-2, Common Criteria 등 국제 보안 표준 인증
  • 감사 로깅: 모든 암호화 작업에 대한 상세한 감사 추적 기록

HSM이 중요한 이유

현대 디지털 인프라는 암호화에 전적으로 의존합니다. 온라인 뱅킹, 전자상거래, 클라우드 서비스, 의료 기록, 정부 통신—이 모든 것이 암호화 키로 보호됩니다. 만약 이러한 키가 손상되면 그 결과는 치명적일 수 있습니다.

실제 보안 침해 사례

역사적으로 암호화 키 관리 실패는 심각한 보안 사고로 이어졌습니다:

  • 코드 서명 키 탈취: 공격자가 소프트웨어 제조사의 서명 키를 훔쳐 악성코드에 정상 서명을 첨부
  • SSL/TLS 인증서 위조: 인증 기관(CA)의 키 유출로 허위 HTTPS 인증서 발급
  • 결제 시스템 침해: ATM이나 POS 단말기의 암호화 키 노출로 카드 정보 유출
  • 데이터베이스 암호화 실패: 데이터 암호화 키가 같은 서버에 평문으로 저장되어 무용지물

HSM은 이러한 위협에 대한 근본적인 해결책입니다. 키를 안전한 하드웨어 경계 내부에 격리함으로써, 서버가 완전히 침해되더라도 암호화 키는 보호됩니다.

HSM의 종류

HSM은 다양한 형태로 제공되며, 각각 특정 용도에 최적화되어 있습니다.

1. 네트워크 HSM (Network-Attached HSM)

네트워크 HSM은 독립 실행형 네트워크 어플라이언스로, 여러 서버와 애플리케이션이 공유할 수 있습니다. 일반적으로 1U~2U 랙 마운트 형태이며, 이더넷을 통해 연결됩니다.

네트워크 HSM 토폴로지: ┌─────────────────┐ │ 데이터센터 │ │ │ │ ┌───────────┐ │ │ │ 웹 서버들 │ │ │ └─────┬─────┘ │ │ │ │ │ ┌─────▼─────┐ │ │ │ 네트워크 │ │ │ │ 스위치 │ │ │ └─────┬─────┘ │ │ │ │ │ ┌─────▼─────┐ │ │ │ 네트워크 │ │ │ │ HSM │ │ │ │ (HA 클러스터)│ │ └───────────┘ │ └─────────────────┘ 특징: - 고성능 (초당 수천~수만 건 처리) - 이중화 구성으로 고가용성 제공 - 중앙 집중식 키 관리 - 다수의 애플리케이션 동시 지원

2. PCIe HSM (내장형)

PCIe HSM은 서버 내부에 직접 설치되는 카드 형태로, 낮은 지연시간이 필요한 애플리케이션에 적합합니다.

  • 장점: 극도로 낮은 지연시간, 네트워크 오버헤드 없음
  • 단점: 물리적 서버 접근 필요, 확장성 제한
  • 용도: 고빈도 거래, 실시간 암호화가 필요한 애플리케이션

3. USB HSM (휴대용)

USB 형태의 소형 HSM으로, 개인 키 보호와 코드 서명에 주로 사용됩니다.

  • 장점: 휴대 가능, 저렴한 비용
  • 단점: 제한된 성능, 물리적 분실 위험
  • 용도: 개발자 코드 서명, 관리자 인증, 이동 근무

4. 클라우드 HSM

AWS CloudHSM, Azure Dedicated HSM, Google Cloud HSM 등 클라우드 제공업체가 관리하는 HSM 서비스입니다.

  • 장점: 하드웨어 관리 불필요, 탄력적 확장, 글로벌 배포
  • 단점: 클라우드 종속성, 네트워크 지연
  • 용도: 클라우드 네이티브 애플리케이션, SaaS 제품

HSM의 주요 사용 사례

금융 서비스

애플리케이션 HSM 역할 규정
ATM/POS PIN 처리 PIN 암호화, 검증, 키 관리 PCI PIN Security

공개 키 기반 구조 (PKI)

인증 기관(Certificate Authority, CA)은 디지털 인증서 발급 시 HSM을 사용하여 루트 CA 키를 보호합니다. 이 키가 손상되면 전체 PKI 신뢰 체계가 붕괴되므로, HSM 보호가 필수적입니다.

PKI 계층 구조와 HSM: ┌──────────────────────┐ │ 루트 CA │ │ (오프라인 HSM) │ ← 최고 보안 레벨 └──────────┬───────────┘ 에어갭 환경 │ ┌──────────▼───────────┐ │ 중간 CA │ │ (온라인 HSM) │ ← 운영 환경 └──────────┬───────────┘ 네트워크 연결 │ ┌──────────▼───────────┐ │ 발급 CA │ │ (HSM 클러스터) │ ← 고성능 필요 └──────────┬───────────┘ 초당 수천 건 │ ┌──────────▼───────────┐ │ SSL/TLS 인증서 │ │ (최종 사용자) │ └──────────────────────┘

클라우드 서비스 제공자

AWS, Azure, Google Cloud 등 대형 클라우드 플랫폼은 고객 데이터 암호화를 위해 HSM을 광범위하게 사용합니다.

  • 데이터 암호화: S3, Blob Storage 등의 데이터 암호화 키 보호
  • 키 관리 서비스: AWS KMS, Azure Key Vault의 백엔드 HSM
  • 고객 관리 키: BYOK(Bring Your Own Key) 시나리오 지원
  • 규정 준수: HIPAA, GDPR, PCI-DSS 요구사항 충족

코드 서명

소프트웨어 제조사는 소프트웨어 무결성을 보장하기 위해 코드 서명 키를 HSM에 저장합니다. 이는 공급망 공격을 방지하는 중요한 방어책입니다.

# HSM을 사용한 코드 서명 예제 (개념적) # 1. HSM에 연결 hsm_session = connect_to_hsm(slot=0, pin="USER_PIN") # 2. 서명 키 로드 (키는 HSM 내부에만 존재) signing_key = hsm_session.get_key(label="code-signing-key") # 3. 소프트웨어 해시 계산 file_hash = sha256("my-software.exe") # 4. HSM 내부에서 서명 수행 (키 노출 없음) signature = hsm_session.sign( key=signing_key, data=file_hash, algorithm="RSA-PSS" ) # 5. 서명 첨부 attach_signature("my-software.exe", signature) # 결과: 사용자는 서명을 검증하여 소프트웨어 출처 확인 # 공격자가 서버를 침해해도 서명 키를 얻을 수 없음

HSM의 물리적 보안 메커니즘

HSM의 가장 중요한 특징 중 하나는 변조 방지(tamper-resistant) 설계입니다. 다양한 물리적 보안 메커니즘이 통합되어 있습니다.

변조 감지 기술

메커니즘 작동 원리 대응 조치
봉인 센서 케이스 개봉 시 전기 회로 단락 즉시 전원 차단 및 키 삭제
메시 레이어 PCB 전체를 둘러싼 미세 전도성 메시 메시 손상 감지 시 제로화
전압/온도 센서 비정상적인 전압·온도 모니터링 공격 패턴 감지 시 작동 중단
방사선 센서 X-ray 등 분석 시도 감지 센서 트리거 시 키 제로화
에폭시 밀봉 중요 칩을 에폭시로 완전 밀봉 물리적 추출 불가능

제로화 (Zeroization) 프로세스

변조 시도가 감지되면 HSM은 자동으로 모든 암호화 키를 삭제합니다. 이 프로세스는 밀리초 내에 완료되며, 복구가 불가능합니다.

제로화 트리거 조건: 1. 물리적 침입 - 케이스 개봉 감지 - 메시 레이어 손상 - 드릴링/밀링 시도 2. 환경 공격 - 전압 글리치 공격 - 클럭 조작 - 온도 범위 초과 (< -10°C 또는 > 60°C) 3. 논리적 공격 - 과도한 인증 실패 (예: 10회) - 불법 명령 패턴 - 펌웨어 무결성 검증 실패 제로화 절차: 1. 모든 휘발성 메모리 덮어쓰기 (난수로 3회) 2. 비휘발성 키 저장소 삭제 3. 보안 상태를 "제로화됨"으로 설정 4. HSM 작동 중단 (재초기화 필요)

HSM 구매 시 고려사항

조직에 적합한 HSM을 선택하려면 여러 요소를 평가해야 합니다.

성능 요구사항

  • 처리량: 초당 필요한 서명/암호화 작업 수 (TPS)
  • 지연시간: 각 작업의 최대 허용 지연시간 (밀리초)
  • 동시 연결: HSM에 연결할 애플리케이션/서버 수
  • 알고리즘 지원: RSA, ECC, AES 등 필요한 알고리즘

인증 및 규정 준수

  • FIPS 140-2 레벨: Level 3 이상 권장 (금융, 정부)
  • Common Criteria: EAL4+ 인증 여부
  • 산업별 인증: PCI-HSM (결제), ANSSI (프랑스), BSI (독일)

운영 요구사항

  • 고가용성: 클러스터링, 이중화, 자동 장애조치 지원
  • 백업 및 복구: 키 백업 메커니즘, 재해 복구 절차
  • 관리 인터페이스: CLI, GUI, REST API 제공 여부
  • 감사 로깅: 상세한 로그 기록 및 SIEM 통합

요약

하드웨어 보안 모듈(HSM)은 암호화 키를 물리적으로 보호하고 암호화 연산을 안전하게 수행하는 전용 하드웨어 장치입니다. 소프트웨어 암호화와 달리 HSM은 변조 방지 하드웨어 내부에서 모든 작업을 수행하며, 키가 평문으로 노출되지 않도록 보장합니다.

HSM은 네트워크 어플라이언스, PCIe 카드, USB 장치, 클라우드 서비스 등 다양한 형태로 제공되며, 각각 특정 용도에 최적화되어 있습니다. 금융 서비스의 PIN 처리, PKI의 루트 CA 보호, 클라우드 데이터 암호화, 소프트웨어 코드 서명 등 중요한 보안 작업에 필수적으로 사용됩니다.

HSM의 핵심 가치는 물리적 보안 메커니즘에 있습니다. 봉인 센서, 메시 레이어, 전압 모니터링 등 다층 방어를 통해 침입 시도를 감지하고, 감지 시 밀리초 내에 모든 키를 자동으로 삭제(제로화)합니다. 이러한 특징으로 FIPS 140-2 Level 3~4 인증을 받아 규제 산업에서 신뢰받고 있습니다.

HSM 선택 시에는 성능 요구사항(TPS, 지연시간), 필요한 인증(FIPS, Common Criteria, PCI), 운영 요구사항(고가용성, 백업, 관리) 등을 종합적으로 고려해야 합니다.

복습 문제

  1. HSM의 정의와 주요 기능 3가지를 설명하세요.
  2. 소프트웨어 암호화와 HSM 기반 암호화의 차이점을 보안, 성능, 비용 관점에서 비교하세요.
  3. 네트워크 HSM, PCIe HSM, USB HSM, 클라우드 HSM의 장단점을 각각 설명하세요.
  4. 금융 서비스에서 HSM이 사용되는 4가지 사례를 들고, 각각의 규제 요구사항을 설명하세요.
  5. PKI에서 루트 CA 키를 HSM으로 보호하는 것이 왜 중요한지 설명하세요.
  6. HSM의 변조 감지 메커니즘 5가지를 나열하고 각각의 작동 원리를 설명하세요.
  7. 제로화(Zeroization)란 무엇이며, 어떤 상황에서 트리거되는지 설명하세요.
  8. 코드 서명에 HSM을 사용하는 이유와 공급망 공격 방지 효과를 설명하세요.
  9. HSM 선택 시 고려해야 할 성능 요구사항 4가지를 나열하세요.
  10. FIPS 140-2 인증이 HSM에 중요한 이유를 설명하고, Level 3와 Level 4의 차이를 기술하세요.

본 표준은 다음 ALLOW 출처를 단일 진실원으로 삼아 데이터 교환·서명· 감사 증적 인터페이스를 제공합니다. 구현체는 이들 원전을 대체하지 않고, 데이터 교환·감사 증적·자동 검증을 위한 상호운용 계층을 정의합니다.

  • OpenAPI Specification 3.1 — REST API 정의
  • JSON Schema 2020-12 — 스키마 검증
  • ISO/IEC 27001:2022 — 정보보안 경영시스템
  • ISO/IEC 27018:2019 — 공용 클라우드의 PII 보호
  • ISO/IEC 17065:2012 — 제품 인증 기관
  • CycloneDX 1.5 / SPDX 2.3 — 소프트웨어 BOM
  • Sigstore (DSSE 봉투, Rekor 투명성 로그) — 서명 앵커링
  • in-toto Attestation Framework 1.0 — 빌드 출처 검증
  • IETF RFC 9700 — OAuth 2.1
  • IETF RFC 9457 — Problem Details for HTTP APIs
  • IETF RFC 8615 — well-known URIs

실무 적용 체크리스트

  • 매니페스트 무결성: 모든 발급은 SHA-256 또는 Sigstore 서명을 부착해 변조를 차단합니다. 발급 후 변경은 새 매니페스트로 처리 되며 이전 매니페스트는 supersede 마킹과 함께 보존됩니다.
  • 보관 기간: ISO/IEC 17065:2012 감사 기대치를 충족하기 위해 7년 이상의 보관 정책을 명시합니다. 관할 규정이 더 길면 그것을 따릅니다.
  • 버전 관리: 정정·확장은 시맨틱 버저닝(SemVer 2.0.0)을 따릅니다. 필수 요건의 추가는 마이너, 호환 단절은 메이저 버전 갱신을 트리거합니다. 메이저 변경은 최소 90일 병행 호환 기간을 거치며 그 기간 동안 신·구 두 버전 모두 WIA 등록부에 Stable로 표기 됩니다.
  • 부정결과 보고: 누락된 지표는 사유와 함께 매니페스트에 기록 되어야 합니다. 사유가 없는 누락은 감사 시 실패로 간주됩니다.
  • 키 회전: 매니페스트 서명 키는 배포 정책에 따라 회전됩니다. 이전 키는 공개된 키 이력에 보존되어 과거 매니페스트의 사후 검증을 보장합니다.

참조 토폴로지

구현체의 운용 토폴로지는 다음 4가지 중 하나로 매니페스트에 선언됩니다: 단일 테넌트 엣지(조직당 1개 런타임), 멀티 테넌트 게이트웨이(헤더 기반 격리로 다중 테넌트 서비스), 연합 메시(다수 런타임이 디렉토리 서비스 통해 상호 인식), 에어갭 배치(라이브 연결 없이 서명된 패키지를 out-of-band 채널로 전송). 각 토폴로지는 참조 테스트 벡터를 포함하며 구현체가 자신의 배포가 표준 패턴과 일치하는지 검증할 수 있도록 합니다.

거버넌스 및 변경 관리

모든 변경 제안과 정정 요청은 GitHub Issues 트래커에 표준 이름 레이블로 접수됩니다. WIA 표준 워킹그룹은 매 마이너 릴리스 사이클의 시작 시점에 열린 이슈를 검토하고, 결정 로그를 릴리스 노트와 함께 공개합니다. 정정(에라타)은 패치 버전으로 발급되며, 새로운 의무 요건의 추가는 마이너, 호환 단절은 메이저 버전 갱신을 트리거합니다. 안전·인체· 프라이버시 관련 이슈는 fast-track 절차로 처리됩니다.
