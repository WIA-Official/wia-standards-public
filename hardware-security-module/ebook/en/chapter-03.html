CHAPTER 3

Cryptographic Operations

Mastering Digital Signatures, Encryption, and Advanced Cryptographic Functions

Overview of Cryptographic Operations

Hardware Security Modules are designed to perform cryptographic operations with maximum security and performance. Unlike software-based cryptography where keys and intermediate values may be exposed in system memory, HSMs ensure that all cryptographic operations occur within the protected hardware boundary. This chapter explores the core cryptographic operations supported by HSMs and best practices for their use.

Understanding these operations is essential for application developers integrating with HSMs, security architects designing cryptographic systems, and operations teams managing HSM deployments. Each operation type has specific use cases, performance characteristics, and security considerations that must be understood for effective implementation.

Digital Signatures

Digital signatures are one of the most important cryptographic operations performed by HSMs. They provide authentication (proving who created the signature), integrity (detecting any changes to the signed data), and non-repudiation (the signer cannot deny having signed the data). Digital signatures are fundamental to PKI, code signing, document signing, blockchain transactions, and secure communications.

RSA Signatures

RSA remains one of the most widely used signature algorithms despite being developed in the 1970s. Its mathematical foundation on the difficulty of factoring large numbers has proven resilient, though key sizes have increased over time to maintain security against advancing computational capabilities.

RSA Signature Schemes

Scheme Padding Method Security Level Recommended Use
PKCS#1 v1.5 Deterministic padding Medium Legacy systems, backward compatibility
RSA-PSS Probabilistic padding with salt High New applications, maximum security
ISO/IEC 9796 Message recovery padding Medium Specialized applications, bandwidth-constrained

RSA Key Size Recommendations

Current Best Practices (2025):

  • 2048-bit: Minimum for new keys, acceptable until 2030
  • 3072-bit: Recommended for long-term security (beyond 2030)
  • 4096-bit: Maximum security, slower performance
  • 1024-bit and below: Deprecated, should not be used

ECDSA Signatures

Elliptic Curve Digital Signature Algorithm (ECDSA) provides security equivalent to RSA but with much smaller key sizes. A 256-bit ECDSA key provides roughly the same security as a 3072-bit RSA key, resulting in faster operations, smaller signatures, and reduced bandwidth requirements.

Standard Elliptic Curves

Curve Key Size Security Bits Primary Applications
P-256 (secp256r1) 256 bits 128 bits TLS, general purpose, US government
P-384 (secp384r1) 384 bits 192 bits High-security applications, classified data
P-521 (secp521r1) 521 bits 260 bits Maximum security, future-proofing
secp256k1 256 bits 128 bits Bitcoin, Ethereum, blockchain

EdDSA Signatures

Edwards-curve Digital Signature Algorithm (EdDSA) is a modern signature scheme designed to avoid many pitfalls of earlier algorithms. It provides high performance, strong security, and resistance to side-channel attacks. The most common variant is Ed25519, which is widely used in modern protocols.

EdDSA Advantages

  • Deterministic signatures (no random number generation during signing)
  • High performance (faster than ECDSA for both signing and verification)
  • Complete Edwards curves eliminating exceptional cases
  • Built-in protection against fault attacks
  • Small signatures (64 bytes for Ed25519)
  • Widely supported in modern software

Encryption and Decryption

HSMs support both symmetric and asymmetric encryption, each with distinct use cases and performance characteristics. Symmetric encryption is used for bulk data encryption, while asymmetric encryption typically encrypts symmetric keys or small amounts of data.

Symmetric Encryption

Symmetric encryption uses the same key for both encryption and decryption. It's fast and efficient, making it suitable for encrypting large amounts of data.

AES Modes of Operation

Mode Provides Use Cases Notes
GCM Confidentiality + Integrity Network protocols, disk encryption, general purpose Recommended for most applications
CBC Confidentiality only Legacy systems, database encryption Requires separate MAC for integrity
CTR Confidentiality only Parallelizable encryption, streaming Good performance, requires unique nonce
ECB Confidentiality (weak) None (deprecated) Not recommended - reveals patterns
XTS Confidentiality Disk/storage encryption Specialized for storage devices

Asymmetric Encryption

Asymmetric encryption uses a public key for encryption and a private key for decryption. While slower than symmetric encryption, it's essential for key exchange and scenarios where the encryptor and decryptor don't share a pre-established key.

RSA Encryption Padding

Padding Scheme Security Max Plaintext Size Recommendation
OAEP (SHA-256) High Key size - 66 bytes Recommended for all new applications
OAEP (SHA-1) Medium Key size - 42 bytes Legacy compatibility only
PKCS#1 v1.5 Low Key size - 11 bytes Deprecated - vulnerable to attacks

Hybrid Encryption Pattern

Most applications use hybrid encryption: symmetric encryption for data, asymmetric encryption for the symmetric key:

Hybrid Encryption Process: 1. Generate random symmetric key (AES-256) 2. Encrypt data with symmetric key using AES-GCM 3. Encrypt symmetric key with recipient's RSA public key (OAEP) 4. Send encrypted data + encrypted key + IV + auth tag Decryption Process: 1. Decrypt symmetric key using RSA private key (in HSM) 2. Decrypt data using recovered symmetric key (in HSM) 3. Verify authentication tag 4. Return plaintext to application

Message Authentication Codes

Message Authentication Codes (MACs) provide integrity and authenticity without confidentiality. They're faster than digital signatures and don't require asymmetric cryptography, making them ideal for high-throughput scenarios where both parties share a secret key.

HMAC (Hash-based MAC)

HMAC combines a cryptographic hash function with a secret key to produce a MAC:

CMAC (Cipher-based MAC)

CMAC uses a block cipher (typically AES) instead of a hash function. It's preferred in environments where AES hardware acceleration is available:

  • Based on AES encryption in CBC mode
  • Fixed output size (128 bits for AES-128)
  • Provable security properties
  • NIST approved (SP 800-38B)

Key Wrapping and Unwrapping

Key wrapping is the process of encrypting one cryptographic key with another. This is essential for secure key backup, key distribution, and key migration between systems.

RSA Key Transport

RSA can wrap symmetric keys for transport to another party:

  • Symmetric key encrypted with recipient's RSA public key
  • Only recipient with private key can unwrap
  • Use RSA-OAEP for padding
  • Maximum wrapped key size limited by RSA key size

Hash Operations

While hashing doesn't require key material and can be done efficiently in software, HSMs often provide hash operations for convenience and to ensure data never leaves the secure boundary.

Supported Hash Functions

Hash Function Output Size Security Status Common Uses
SHA-256 256 bits Secure General purpose, signatures, Bitcoin
SHA-384 384 bits Secure High security applications
SHA-512 512 bits Secure Maximum security, 64-bit platforms
SHA-3 (256/384/512) Variable Secure Alternative to SHA-2, future-proofing
SHA-1 160 bits Broken Legacy only - not recommended

Key Derivation Functions

Key Derivation Functions (KDFs) generate cryptographic keys from passwords, shared secrets, or other key material. HSMs implement KDFs to ensure derived keys maintain high entropy and are protected from brute-force attacks.

HKDF (HMAC-based KDF)

HKDF expands a short shared secret into multiple cryptographic keys:

PBKDF2 (Password-Based KDF)

PBKDF2 derives keys from passwords using many iterations to slow down brute-force attacks:

  • Configurable iteration count (minimum 100,000 for 2025)
  • Salt prevents rainbow table attacks
  • HMAC-based construction
  • Widely supported and standardized

Random Number Generation

HSMs provide high-quality random numbers essential for key generation, nonces, initialization vectors, and other cryptographic operations.

Random Number Quality Levels

Type Source Use Cases
True Random (TRNG) Physical entropy source Seed generation, master keys, long-term keys
Cryptographic (DRBG) Deterministic generator seeded by TRNG Session keys, IVs, nonces, high-volume operations
Pseudo-Random Software algorithm Non-cryptographic uses only

Chapter Summary

Cryptographic operations performed within HSMs benefit from hardware protection ensuring keys never leave the secure boundary. Digital signatures using RSA, ECDSA, or EdDSA provide authentication, integrity, and non-repudiation for various applications from PKI to blockchain. Each algorithm offers different trade-offs between security, performance, and compatibility.

Encryption operations support both symmetric (AES) and asymmetric (RSA, ECC) algorithms. AES-GCM is recommended for most applications as it provides both confidentiality and integrity. Hybrid encryption combines the efficiency of symmetric encryption with the key exchange capabilities of asymmetric encryption.

Message authentication codes (HMAC, CMAC) provide integrity and authenticity for high-throughput scenarios. Key wrapping enables secure key backup and distribution. Hash functions and key derivation functions round out the comprehensive cryptographic toolkit provided by modern HSMs.

Understanding the characteristics, trade-offs, and appropriate use cases for each operation type is essential for building secure, efficient cryptographic systems. Always use current best practices: RSA-PSS for signatures, AES-GCM for encryption, and modern curves like P-256 or Ed25519 for elliptic curve operations.
