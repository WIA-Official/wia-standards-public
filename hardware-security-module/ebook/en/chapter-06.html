CHAPTER 6

Compliance and Certification

Meeting Regulatory Requirements and Security Standards

Overview of HSM Certification

Hardware Security Modules operate in highly regulated environments where security failures can have severe financial, legal, and reputational consequences. To ensure HSMs meet stringent security requirements, multiple certification programs have been established by government agencies, industry consortia, and standards organizations. These certifications provide independent validation that an HSM correctly implements security controls and resists various attack vectors.

Certification is not merely a checkbox exercise—it represents months or years of rigorous testing, documentation, and validation. Certified HSMs undergo extensive security analysis including physical tamper testing, cryptographic algorithm validation, side-channel attack resistance, and security policy review. Organizations deploying HSMs in regulated industries often require specific certifications as part of their compliance obligations.

Why Certification Matters

  • Regulatory Compliance: Many regulations explicitly require certified cryptographic modules
  • Risk Mitigation: Independent validation reduces security implementation risks
  • Customer Trust: Certifications provide evidence of security commitment
  • Insurance Requirements: Cyber insurance may require certified security controls
  • Procurement Standards: Government and enterprise buyers often mandate certifications

FIPS 140-2 Certification

FIPS 140-2 (Federal Information Processing Standard Publication 140-2) is the gold standard for cryptographic module validation. Published by NIST (National Institute of Standards and Technology), FIPS 140-2 defines four security levels, with Level 3 and Level 4 representing the highest grades of physical and logical security appropriate for HSMs.

FIPS 140-2 Security Levels

Level Security Requirements Typical Use Cases
Level 1 Software encryption, basic security requirements Low-security applications, testing environments
Level 2 Physical tamper-evidence, role-based authentication Commercial applications, general enterprise use
Level 3 Tamper-detection and response, identity-based authentication Financial services, healthcare, government
Level 4 Complete physical protection, environmental failure protection Military, critical infrastructure, high-value payments

FIPS 140-2 Level 3 Requirements

Most enterprise HSMs target FIPS 140-2 Level 3, which mandates:

  • Physical Tamper Detection: Sensors detect physical intrusion attempts and trigger zeroization
  • Identity-Based Authentication: Strong authentication mechanisms beyond simple passwords
  • Environmental Failure Protection: Detection of voltage, temperature, and frequency attacks
  • Cryptographic Key Entry/Output: Secure methods for key import/export using encryption or split knowledge
  • Secure Firmware Updates: Signed and authenticated firmware with rollback protection
  • Audit Logging: Comprehensive logging of security-relevant events

FIPS 140-2 Testing Areas

Testing Area Requirements Validation Method
Cryptographic Module Specification Complete documentation of module boundaries and interfaces Document review, architectural analysis
Cryptographic Module Ports and Interfaces Defined data input, output, control, and status interfaces Interface testing, boundary analysis
Roles, Services, and Authentication Distinct roles with appropriate authentication strength Role-based access testing, authentication strength analysis
Finite State Model Defined states and transitions with error handling State machine validation, error condition testing
Physical Security Tamper evidence, detection, and response mechanisms Physical penetration testing, environmental testing
Operational Environment Operating system security if applicable OS configuration review, access control testing
Cryptographic Key Management Secure key generation, storage, distribution, destruction Key lifecycle testing, entropy analysis
Electromagnetic Interference/Compatibility (EMI/EMC) Emissions within acceptable limits FCC Part 15 Class A testing
Self-Tests Power-up and conditional self-tests Known-answer tests, pairwise consistency tests
Design Assurance Configuration management, secure distribution Development process review, delivery assurance
Mitigation of Other Attacks Resistance to documented attack methods Side-channel analysis, fault injection testing

FIPS 140-2 Validation Process

FIPS 140-2 Validation Timeline (Typical): Phase 1: Pre-Testing (2-4 months) - Security policy documentation - Algorithm implementations (CAVP testing) - Vendor self-testing - Documentation preparation Phase 2: Lab Testing (3-6 months) - Accredited lab (CMVP) testing - Physical security evaluation - Cryptographic boundary verification - Environmental testing Phase 3: CMVP Review (2-6 months) - NIST/CMVP review of test reports - Resolution of findings - Additional testing if needed - Final validation Phase 4: Certificate Issuance - Certificate number assigned - Public listing on NIST website - Security policy publication Total Duration: 7-16 months (typical) Cost: $50,000 - $200,000+ (depending on complexity)

FIPS 140-3 Transition

FIPS 140-3, published in 2019, is the successor to FIPS 140-2 and incorporates ISO/IEC 19790:2012 (Security requirements for cryptographic modules). The transition period allows both standards to coexist, with FIPS 140-2 testing ending in 2026.

Key Changes in FIPS 140-3

Area FIPS 140-2 FIPS 140-3
Foundation Standalone standard Based on ISO/IEC 19790
Software Modules Limited software support Enhanced software module requirements
Algorithm Transitions Fixed algorithm requirements Support for algorithm transitions (e.g., SHA-3, post-quantum)
Authentication Password-based authentication Enhanced authentication including biometrics
Side-Channel Attacks Optional mitigation Mandatory testing for specific attacks
Firmware Updates Basic requirements Enhanced secure update mechanisms

FIPS 140-3 Side-Channel Attack Testing

One significant enhancement in FIPS 140-3 is mandatory testing for certain side-channel attacks, particularly for Level 3 and Level 4 modules. This includes:

  • Timing Attacks: Verification that cryptographic operations execute in constant time
  • Power Analysis: Testing resistance to simple and differential power analysis (SPA/DPA)
  • Electromagnetic Analysis: Evaluation of EM emissions during cryptographic operations
  • Fault Injection: Resistance to voltage glitching and clock manipulation

Common Criteria (ISO/IEC 15408)

Common Criteria (CC) is an international standard (ISO/IEC 15408) for computer security certification. Unlike FIPS 140, which focuses specifically on cryptographic modules, Common Criteria provides a framework for evaluating all aspects of IT security products. HSMs are often evaluated under Common Criteria using Protection Profiles specific to cryptographic devices.

Common Criteria Evaluation Assurance Levels (EAL)

EAL Level Name Assurance Description
EAL1 Functionally Tested Basic assurance through testing
EAL2 Structurally Tested Developer testing with some design examination
EAL3 Methodically Tested and Checked Methodical testing and development controls
EAL4 Methodically Designed, Tested, and Reviewed Rigorous commercial development (common for HSMs)
EAL5 Semiformally Designed and Tested Semiformal design and testing
EAL6 Semiformally Verified Design and Tested Semiformal verification of design
EAL7 Formally Verified Design and Tested Formal verification (extremely rare, highest assurance)

Common Criteria vs. FIPS 140

Aspect FIPS 140-2/3 Common Criteria
Scope Cryptographic modules only Any IT security product
Geography US/Canada (CMVP) International (CCRA member countries)
Requirements Fixed security requirements Flexible, based on Protection Profile or Security Target
Levels 4 security levels 7 evaluation assurance levels
Duration 7-16 months typical 12-24 months typical (EAL4+)
Cost $50K-$200K $100K-$500K+ (EAL4+)

PCI-DSS Requirements

The Payment Card Industry Data Security Standard (PCI-DSS) is mandatory for any organization that processes, stores, or transmits credit card information. PCI-DSS doesn't certify HSMs directly but requires cryptographic operations to be performed using validated cryptographic modules—typically FIPS 140-2 Level 3 or higher HSMs.

PCI-DSS Cryptographic Requirements

Requirement Description HSM Role
Requirement 3 Protect stored cardholder data HSM encrypts stored PANs (Primary Account Numbers)
Requirement 4 Encrypt transmission of cardholder data HSM generates and protects TLS/SSL keys
Requirement 8 Identify and authenticate access HSM provides secure authentication tokens
Requirement 10 Track and monitor access HSM maintains tamper-evident audit logs

PCI PIN Security Requirements

For organizations processing PIN transactions, PCI PIN Security Requirements mandate even stricter controls:

PCI PIN Security - HSM Requirements: 1. PIN Processing - All PIN processing in secure cryptographic devices (SCD) - Devices must be PCI-approved (typically FIPS 140-2 Level 3+) - PIN never available in clear text outside SCD 2. Key Management - Dual control for key administration - Split knowledge for key components - Cryptographic separation of PIN and data keys - Key change procedures documented and followed 3. Device Management - Physical security controls for HSM installation - Tamper-evident seals and monitoring - Secure courier delivery for HSMs - Decommissioning procedures (zeroization) 4. Monitoring - Continuous monitoring for tampering - Immediate response to tamper events - Audit trail of all key management operations

Industry-Specific Certifications

Beyond general security certifications, specific industries have additional HSM certification requirements tailored to their unique security and regulatory needs.

Financial Services

  • SWIFT CSP: Customer Security Programme requirements for SWIFT messaging
  • EMV Certification: For payment card processing and EMV key management
  • PCI-HSM: Payment Card Industry HSM security requirements
  • RBI Guidelines: Reserve Bank of India cryptographic controls (India)
  • MAS TRM: Monetary Authority of Singapore Technology Risk Management

Government and Defense

  • NSA Suite B: NSA-approved cryptographic algorithms (transitioning to CNSA)
  • ANSSI: French Network and Information Security Agency certification
  • BSI: German Federal Office for Information Security certification
  • CESG: UK National Technical Authority for Information Assurance

Healthcare

  • HIPAA: Health Insurance Portability and Accountability Act (requires encryption)
  • HITECH: Health Information Technology for Economic and Clinical Health Act
  • FDA Guidance: For medical device cryptographic security

Compliance Implementation Strategies

Achieving and maintaining compliance requires ongoing effort beyond initial certification. Organizations must implement comprehensive compliance programs that address people, processes, and technology.

Compliance Program Components

Component Activities Frequency
Policy Development Document cryptographic policies, procedures, standards Initial + Annual review
Risk Assessment Identify and evaluate cryptographic risks Annual + after major changes
Security Training Train personnel on HSM operations and security Onboarding + Annual refresher
Access Control Implement and review privileged access controls Continuous + Quarterly review
Audit Logging Monitor and review HSM audit logs Continuous monitoring + Daily review
Vulnerability Management Assess and remediate HSM vulnerabilities Monthly scans + Quarterly assessments
Incident Response Respond to security incidents involving HSMs As needed + Annual drill
Compliance Audits Internal and external compliance assessments Annual external + Quarterly internal

Documentation Requirements

Compliance Documentation (Typical): 1. Security Policies - Cryptographic security policy - Key management policy - Access control policy - Incident response policy 2. Procedures - HSM initialization procedures - Key generation procedures - Backup and recovery procedures - Emergency zeroization procedures 3. Standards - Cryptographic algorithm standards - Key length and lifecycle standards - Authentication strength requirements - Physical security standards 4. Records - HSM inventory and configuration - Key ceremony records - Access control matrices - Audit log archives - Training records - Incident reports 5. Evidence - FIPS 140 certificates - Common Criteria certificates - PCI attestation of compliance (AoC) - Penetration test reports - Vulnerability scan results

Audit and Assessment

Regular audits verify ongoing compliance with certification requirements and organizational policies. Both internal and external audits play critical roles in maintaining security posture.

Audit Types and Scope

Audit Type Scope Auditor Frequency
Internal Audit Policy compliance, operational procedures Internal audit team Quarterly
PCI-DSS Audit Payment card data protection requirements Qualified Security Assessor (QSA) Annual
SOC 2 Type II Security, availability, confidentiality controls CPA firm Annual
ISO 27001 Information security management system Accredited certification body Annual surveillance + Triennial recertification
Penetration Test Technical security testing of HSM infrastructure Ethical hacking firm Annual or biannual

Common Audit Findings

  • Insufficient Documentation: Missing or outdated policies, procedures, or records
  • Weak Access Controls: Excessive privileges or shared accounts
  • Inadequate Monitoring: Audit logs not reviewed or retained properly
  • Configuration Drift: HSM configurations not matching documented standards
  • Training Gaps: Personnel lacking required security training
  • Patch Management: Delayed application of security updates
  • Backup Failures: Key backup procedures not tested or documented
  • Physical Security: Inadequate controls for HSM physical access

Maintaining Certification Status

Certifications are not permanent—they require ongoing maintenance, recertification, and sometimes revalidation when products are updated or regulations change.

Change Management for Certified HSMs

Critical Principle: Any change to a FIPS 140-2/3 validated module may invalidate the certification. Changes must be carefully evaluated and may require revalidation.

Changes Requiring Revalidation:

  • Cryptographic algorithm changes or additions
  • Physical enclosure modifications
  • Firmware updates affecting security functionality
  • Security policy changes
  • Changes to cryptographic boundary

Changes Potentially Allowed Under Existing Certificate:

  • Bug fixes not affecting security functions
  • Documentation clarifications
  • Non-security feature additions
  • Performance optimizations (if timing-attack analysis shows no impact)
Change Management Workflow: 1. Change Request └─> Document proposed change └─> Identify affected components └─> Assess certification impact 2. Impact Analysis └─> Review FIPS 140 security policy └─> Consult with validation lab └─> Determine if revalidation needed 3. Approval Process └─> Security review board approval └─> Compliance team sign-off └─> Executive approval for revalidation costs 4. Implementation └─> If no revalidation: implement change └─> If revalidation required: initiate validation process 5. Documentation └─> Update security policy └─> Update operational procedures └─> Notify customers of certification status

Chapter Summary

HSM certification provides independent validation of security controls and is essential for regulatory compliance across multiple industries. FIPS 140-2 and its successor FIPS 140-3 represent the gold standard for cryptographic module validation, with Level 3 and Level 4 providing the highest assurance appropriate for enterprise HSMs. The FIPS validation process is rigorous, expensive, and time-consuming, typically requiring 7-16 months and costing $50,000 to $200,000 or more.

Common Criteria offers an international alternative with flexible security requirements based on Protection Profiles and Security Targets. While Common Criteria and FIPS serve different purposes, they are complementary—many HSMs pursue both certifications to meet diverse customer requirements.

Industry-specific regulations like PCI-DSS mandate the use of certified cryptographic modules for payment processing, particularly for PIN transactions. Organizations operating in regulated industries must implement comprehensive compliance programs addressing policies, procedures, training, monitoring, and auditing.

Maintaining certification requires ongoing vigilance. Changes to certified modules may invalidate certifications and require expensive revalidation. Regular internal and external audits verify continued compliance and identify areas for improvement.
