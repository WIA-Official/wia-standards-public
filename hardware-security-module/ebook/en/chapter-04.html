CHAPTER 4

Key Management

Comprehensive Key Lifecycle Management and Best Practices

The Importance of Key Management

Key management is arguably the most critical aspect of cryptographic security. Even the strongest algorithms become worthless if keys are improperly generated, stored, distributed, or destroyed. Hardware Security Modules provide a comprehensive framework for managing cryptographic keys throughout their entire lifecycle—from generation through secure destruction—ensuring keys remain protected at every stage.

Poor key management has been the root cause of numerous security breaches. Keys stored in configuration files, transmitted via email, or left on decommissioned systems have led to data breaches affecting millions. HSMs address these vulnerabilities by enforcing rigorous key management policies and providing secure mechanisms for every key lifecycle operation.

Key Lifecycle States

The WIA-SEC-014 standard defines a comprehensive key lifecycle with distinct states and allowed transitions. Understanding these states is essential for implementing proper key management policies.

State Description Allowed Operations Typical Duration
Pre-Active Key generated but not yet authorized for use Export, destroy, activate Hours to days
Active Key approved and available for cryptographic operations All cryptographic operations per key policy Months to years
Suspended Temporarily disabled, can be reactivated Reactivate, revoke Days to weeks
Deactivated No longer used for new operations, verification only Verify signatures, decrypt legacy data Years (archival)
Compromised Known or suspected compromise None (immediate revocation required) None
Destroyed Securely erased, permanently unrecoverable None Permanent

Key Generation

Secure key generation is the foundation of cryptographic security. Keys must be generated with sufficient entropy, proper parameters, and within a secure environment. HSMs provide the ideal environment for key generation, ensuring keys are born secure and never exist outside the protected boundary.

On-Device Key Generation (Recommended)

The gold standard for key generation is creating keys directly within the HSM:

  • Private keys never exist outside the HSM's secure boundary
  • Hardware TRNG provides high-quality entropy
  • Proper algorithm parameters enforced by HSM firmware
  • Immediate protection under HSM's master encryption key
  • Audit trail created automatically

Key Import (When Necessary)

Sometimes keys must be generated elsewhere and imported into the HSM. This should be minimized and carefully controlled:

Import Security Requirements:

  • Keys must be wrapped (encrypted) during import using secure key wrapping
  • Use temporary secure channels (e.g., TLS 1.3 with mutual authentication)
  • Verify key integrity using checksums or MACs
  • Log all import operations with full audit trail
  • Destroy wrapped key material after successful import
  • Consider marking imported keys as "imported" for tracking

Key Storage and Protection

Once generated, keys must be protected at rest. HSMs implement multiple layers of protection to ensure keys remain confidential and integrity-protected throughout their storage lifetime.

Key Attributes and Policies

Each key has attributes that control its usage and protection:

Attribute Purpose Example Values
CKA_TOKEN Persistent vs. session key true (persistent), false (session)
CKA_PRIVATE Requires authentication to use true (protected), false (public)
CKA_SENSITIVE Cannot be revealed in plaintext true (for private/secret keys)
CKA_EXTRACTABLE Can be exported from HSM false (recommended for critical keys)
CKA_SIGN Can be used for signing true/false
CKA_DECRYPT Can be used for decryption true/false
CKA_UNWRAP Can unwrap other keys true/false

Key Hierarchy

HSMs use a hierarchical key structure to protect all keys:

Level 0 - Device Master Key (DMK): Embedded in hardware during manufacturing, never changes, protects all other keys. If DMK is compromised, the entire device is compromised.

Level 1 - Master KEKs (Key Encryption Keys): Generated during HSM initialization, encrypted under DMK, used to protect different classes of keys (user keys, admin keys, etc.).

Level 2 - Domain KEKs: Per-partition or per-tenant keys that isolate different users/applications, encrypted under Master KEKs.

Level 3 - Operational Keys: User keys for signing, encryption, etc., encrypted under Domain KEKs. These are the keys applications actually use.

Key Backup and Recovery

While keys should ideally never leave the HSM, backup is essential for business continuity and disaster recovery. The WIA-SEC-014 standard provides secure mechanisms for key backup that maintain security while enabling recovery.

Backup Methods

1. Encrypted Key Export (Key Wrapping)

Keys can be exported in wrapped (encrypted) form for backup:

2. M-of-N Secret Sharing

Critical keys (especially master keys) can be split into N shares, requiring M shares to reconstruct:

  • Common configurations: 3-of-5, 2-of-3, 5-of-7
  • Each share is cryptographically useless alone
  • Shares distributed to different key custodians
  • Reconstruction requires physical presence of M custodians
  • Based on Shamir's Secret Sharing algorithm

Key Ceremony Example: When initializing a new root CA, the master key might be generated within the HSM, then backed up using 5-of-7 secret sharing. The seven shares are distributed to seven senior security officers in sealed envelopes. To restore the key (e.g., after HSM failure), any five of the seven officers must gather in a secure facility to provide their shares.

3. HSM Clustering and Replication

Modern HSMs support clustering where keys are automatically replicated across multiple devices:

  • Synchronous or asynchronous replication
  • Geographic distribution for disaster recovery
  • Automatic failover when primary HSM fails
  • Load balancing across cluster members
  • Cryptographically protected replication channels

Key Rotation

Regular key rotation is a fundamental security practice that limits the impact of key compromise and meets compliance requirements. The WIA-SEC-014 standard supports both manual and automated key rotation.

Why Rotate Keys?

  • Limit Exposure: If a key is compromised, rotation limits the amount of data encrypted under that key
  • Compliance: Many regulations mandate periodic key rotation (e.g., PCI-DSS)
  • Cryptoperiod Limits: Cryptographic best practices recommend maximum usage periods
  • Algorithm Migration: Facilitates transition to stronger algorithms or larger key sizes
  • Personnel Changes: When key custodians leave the organization

Rotation Strategies

Strategy How It Works Use Cases
Scheduled Rotation Keys rotated on fixed schedule (monthly, yearly) Data encryption keys, session keys
Usage-Based Rotation Rotate after N operations or bytes encrypted High-throughput encryption systems
Event-Driven Rotation Rotate on specific events (employee departure, breach) Access control, authentication keys
Versioned Keys Multiple key versions active simultaneously Allows gradual migration, backward compatibility

Key Distribution

Distributing keys securely to their intended recipients is a complex challenge. HSMs provide several mechanisms for secure key distribution.

Public Key Distribution

Public keys can be distributed freely, but their authenticity must be verified:

  • Certificates: Public keys wrapped in X.509 certificates signed by trusted CA
  • Key Servers: Centralized repositories with integrity protection
  • Direct Exchange: Out-of-band verification (fingerprints, QR codes)
  • Blockchain: Distributed ledger for public key publication

Symmetric Key Distribution

Symmetric keys require secure distribution mechanisms:

Method Mechanism Security Level
Key Wrapping Encrypt under recipient's KEK High
RSA Key Transport Encrypt under recipient's RSA public key High
Diffie-Hellman Derive shared secret from DH exchange High
Physical Transport USB HSM or smart card Very High (for initialization)

Key Destruction

Proper key destruction is as important as secure key generation. Keys must be destroyed when they're no longer needed, when compromised, or when required by policy or regulation.

Secure Deletion Process

HSM Key Destruction Procedure: 1. Verify authorization (may require multiple approvals) 2. Check no dependencies (no keys wrapped under this key) 3. Create audit log entry BEFORE destruction 4. Overwrite key storage with random data (3+ passes) 5. Overwrite with zeros 6. Overwrite with ones 7. Final random overwrite 8. Mark storage location as available 9. Create destruction confirmation audit entry 10. Notify key stakeholders Total time: < 100ms Result: Key is cryptographically unrecoverable

Destruction Considerations

  • Ensure all copies destroyed (backups, replicas, wrapped versions)
  • Consider data recovery requirements before destroying decryption keys
  • Regulatory retention periods may require keeping keys for years
  • Certificate revocation before destroying signing keys
  • Document destruction in compliance records

Key Escrow and Recovery

Some scenarios require designated parties to recover keys in emergencies. This must be balanced carefully against security requirements.

Escrow Use Cases

  • Law enforcement access to encrypted data (controversial)
  • Corporate data recovery when employee unavailable
  • Regulatory compliance in certain industries
  • Succession planning for critical cryptographic operations

Security Considerations: Key escrow introduces significant risks. Escrowed keys become high-value targets for attackers. If implemented, use M-of-N recovery with split knowledge and dual control. Strictly limit who can access escrowed keys and maintain detailed audit logs of all access.

Chapter Summary

Effective key management is the foundation of cryptographic security. The key lifecycle encompasses generation, storage, distribution, rotation, backup, and destruction—each phase requiring careful controls and procedures. HSMs provide the secure environment and tools necessary to implement proper key management throughout this lifecycle.

Key generation should occur within the HSM using hardware random number generation to ensure keys are born secure. Storage protection uses hierarchical encryption, with hardware-protected master keys safeguarding all operational keys. Key attributes control usage and extractability, implementing the principle of least privilege.

Backup and recovery mechanisms including key wrapping, secret sharing, and HSM clustering enable business continuity while maintaining security. Regular key rotation limits exposure and meets compliance requirements. Secure destruction ensures keys are cryptographically unrecoverable when no longer needed.

Organizations must develop comprehensive key management policies addressing all lifecycle phases, assign clear responsibilities, implement technical controls, and maintain detailed audit trails. The WIA-SEC-014 standard provides a framework for these practices, ensuring keys remain protected from generation through destruction.
