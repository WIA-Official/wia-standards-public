Chapter 7: CI/CD Operations and Optimization

Running a CI/CD system is not a set-and-forget endeavor. It requires continuous monitoring, optimization, and troubleshooting to maintain performance and reliability. This chapter covers the operational aspects of managing CI/CD infrastructure at scale.

Pipeline Monitoring

Effective monitoring provides visibility into pipeline health, performance trends, and potential issues before they impact development teams.

Key Metrics to Track

Metric Description Target Alert Threshold
Build Duration Time from trigger to completion < 10 minutes > 20 minutes
Success Rate Percentage of successful builds > 95% < 80%
Queue Time Time waiting for runner < 30 seconds > 5 minutes
Flaky Test Rate Tests that pass/fail inconsistently < 1% > 5%
MTTR Mean time to recovery after failure < 30 minutes > 2 hours

        expr: |

Performance Optimization

Pipeline performance directly impacts developer productivity. Slow pipelines lead to context switching, batched commits, and delayed feedback.

Build Time Analysis

# GitHub Actions - Build Time Analysis
name: Build Time Analysis
on:
  workflow_run:
    workflows: ["CI Pipeline"]
    types: [completed]

jobs:
  analyze:
    runs-on: ubuntu-latest
    steps:
      - name: Fetch workflow run data
        uses: actions/github-script@v7
        with:
          script: |
            const run = await github.rest.actions.getWorkflowRun({
              owner: context.repo.owner,
              repo: context.repo.repo,
              run_id: context.payload.workflow_run.id
            });

            const jobs = await github.rest.actions.listJobsForWorkflowRun({
              owner: context.repo.owner,
              repo: context.repo.repo,
              run_id: context.payload.workflow_run.id
            });

            // Analyze step durations
            const stepAnalysis = jobs.data.jobs.flatMap(job =>
              job.steps.map(step => ({
                job: job.name,
                step: step.name,
                duration: new Date(step.completed_at) - new Date(step.started_at)
              }))
            );

            // Find slowest steps
            const slowest = stepAnalysis
              .sort((a, b) => b.duration - a.duration)
              .slice(0, 5);

            console.log('Slowest steps:', slowest);

            // Send to monitoring
            // await sendMetrics(stepAnalysis);

Optimization Strategies

# Optimization Checklist

## Caching Optimization
- [ ] Dependencies cached with lock file hash
- [ ] Build outputs cached between runs
- [ ] Docker layer caching enabled
- [ ] Remote cache for monorepos (Turborepo, Nx)

## Parallelization
- [ ] Independent jobs run in parallel
- [ ] Test suites split across shards
- [ ] Matrix builds for multi-platform testing
- [ ] Parallel step execution where possible

## Resource Optimization
- [ ] Right-sized runner instances
- [ ] Spot/preemptible instances for non-critical jobs
- [ ] Scheduled cleanup of old artifacts
- [ ] Efficient base images (Alpine, distroless)

## Pipeline Design
- [ ] Fail-fast for quick feedback
- [ ] Conditional job execution
- [ ] Path-based filtering for monorepos
- [ ] Staged pipelines (fast feedback first)

Caching Optimization Example

# Advanced Caching Strategy
name: Optimized Build
on: [push]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # Restore all caches in parallel
      - name: Restore dependency cache
        uses: actions/cache/restore@v4
        id: deps-cache
        with:
          path: node_modules
          key: deps-${{ hashFiles('pnpm-lock.yaml') }}

      - name: Restore build cache
        uses: actions/cache/restore@v4
        id: build-cache
        with:
          path: |
            .next/cache
            dist
          key: build-${{ runner.os }}-${{ hashFiles('src/**') }}
          restore-keys: |
            build-${{ runner.os }}-

      # Only install if cache miss
      - name: Install dependencies
        if: steps.deps-cache.outputs.cache-hit != 'true'
        run: pnpm install --frozen-lockfile

      # Incremental build
      - name: Build
        run: pnpm build

      # Save caches for future runs
      - name: Save dependency cache
        uses: actions/cache/save@v4
        if: steps.deps-cache.outputs.cache-hit != 'true'
        with:
          path: node_modules
          key: deps-${{ hashFiles('pnpm-lock.yaml') }}

      - name: Save build cache
        uses: actions/cache/save@v4
        with:
          path: |
            .next/cache
            dist
          key: build-${{ runner.os }}-${{ hashFiles('src/**') }}

Flaky Tests

# Flaky Test Detection and Quarantine
name: Test with Flaky Detection
on: [push]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Run tests with retries
        run: |
          npm test -- --retries=2 --reporter=json > test-results.json

      - name: Analyze flaky tests
        run: |
          node << 'EOF'
          const results = require('./test-results.json');
          const flaky = results.testResults
            .flatMap(f => f.assertionResults)
            .filter(t => t.retryReasons && t.retryReasons.length > 0);

          if (flaky.length > 0) {
            console.log('Flaky tests detected:');
            flaky.forEach(t => console.log(`  - ${t.fullName}`));

            // Report to tracking system
            // await reportFlakyTests(flaky);
          }
          EOF

      - name: Upload flaky test report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: flaky-test-report
          path: test-results.json

Resource Exhaustion

# GitHub Actions - Resource Monitoring
name: Build with Resource Monitoring
on: [push]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Start resource monitoring
        run: |
          # Monitor resources in background
          while true; do
            echo "$(date) - Memory: $(free -m | awk 'NR==2{print $3}')MB used"
            echo "$(date) - Disk: $(df -h / | awk 'NR==2{print $5}') used"
            sleep 30
          done &
          echo $! > /tmp/monitor.pid

      - name: Build
        run: npm run build

      - name: Stop monitoring
        if: always()
        run: kill $(cat /tmp/monitor.pid) 2>/dev/null || true

Capacity Planning

# Capacity Planning Model
"""
WIA-CI Capacity Planning Calculator

Inputs:
- daily_commits: Average commits per day
- avg_build_time: Average build duration (minutes)
- peak_multiplier: Peak vs average ratio
- target_queue_time: Maximum acceptable queue time (minutes)

Output:
- Required runner capacity
"""

def calculate_runners(daily_commits, avg_build_time, peak_multiplier, target_queue_time):
    # Convert to hourly rate
    hourly_builds = daily_commits / 8  # Assume 8 working hours

    # Peak hour builds
    peak_builds = hourly_builds * peak_multiplier

    # Required capacity (builds per hour)
    builds_per_runner_hour = 60 / avg_build_time

    # Minimum runners for peak load
    min_runners = peak_builds / builds_per_runner_hour

    # Add buffer for queue time target
    buffer_factor = 1 + (avg_build_time / (target_queue_time + avg_build_time))

    return ceil(min_runners * buffer_factor)

# Example: 200 commits/day, 10 min builds, 2x peak, 2 min queue target
# calculate_runners(200, 10, 2, 2) = 10 runners
