Chapter 5: Security in CI/CD Pipelines

Security must be integrated throughout the CI/CD pipeline, not bolted on at the end. This chapter covers secrets management, Static and Dynamic Application Security Testing (SAST/DAST), supply chain security, and practices that embed security into every stage of software delivery.

The DevSecOps Approach

DevSecOps extends DevOps principles to include security as a shared responsibility. Rather than treating security as a gate at the end of development, it becomes an integral part of every stage.

# DevSecOps Pipeline Stages

├──────────┬──────────┬──────────┬──────────┬─────────────────┤
│   Plan   │  Code    │  Build   │  Test    │     Deploy      │
├──────────┼──────────┼──────────┼──────────┼─────────────────┤

│ Modeling │ commit   │ SCA      │ Pen Test │ Scanning        │
│          │ Hooks    │ Secrets  │ API      │ Runtime         │

│ Stories  │ Linters  │          │          │                 │
└──────────┴──────────┴──────────┴──────────┴─────────────────┘
Shift-Left Security: The cost of fixing a security vulnerability increases exponentially as it progresses through the development lifecycle. Finding issues during coding costs 6x less than finding them in testing, and 15x less than finding them in production.

Secrets Management

Secrets - API keys, credentials, certificates - are essential for CI/CD operations but represent significant security risks if mishandled. Never store secrets in code repositories.

Secrets Management Best Practices

  • External Secrets Store: Use dedicated secret management services (HashiCorp Vault, AWS Secrets Manager, Azure Key Vault)
  • Environment-Specific Secrets: Maintain separate secrets for development, staging, and production
  • Rotation Policies: Implement automatic secret rotation on a regular schedule
  • Audit Logging: Log all secret access for compliance and incident response
  • Least Privilege: Grant minimal necessary access to secrets

GitHub Actions Secrets Configuration

# GitHub Actions - Secure Secret Usage
name: Secure Deployment
on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production  # Requires approval for production secrets

    permissions:
      contents: read
      id-token: write  # Required for OIDC

    steps:
      - uses: actions/checkout@v4

      # Use OIDC for cloud authentication (no long-lived credentials)
      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789:role/github-actions
          aws-region: us-east-1

      # Secrets accessed via environment
      - name: Deploy application
        env:
          DATABASE_URL: ${{ secrets.DATABASE_URL }}
          API_KEY: ${{ secrets.API_KEY }}
        run: |
          # Secrets are masked in logs
          npm run deploy

Secret Scanning

# GitHub Actions - Secret Detection
name: Secret Scanning
on: [push, pull_request]

jobs:
  detect-secrets:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      # Gitleaks - detect secrets in code
      - name: Gitleaks scan
        uses: gitleaks/gitleaks-action@v2
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}

      # TruffleHog - high-signal secret detection
      - name: TruffleHog scan
        uses: trufflesecurity/trufflehog@main
        with:
          path: ./
          base: main
          extra_args: --only-verified

Static Application Security Testing (SAST)

SAST tools analyze source code to identify security vulnerabilities without executing the application. They catch issues like SQL injection, XSS, and insecure cryptography early in development.

CodeQL Analysis

# GitHub Actions - CodeQL SAST
name: CodeQL Analysis
on:
  push:
    branches: [main]
  pull_request:
  schedule:
    - cron: '0 0 * * 1'  # Weekly deep scan

jobs:
  analyze:
    runs-on: ubuntu-latest
    permissions:
      security-events: write
      packages: read
      actions: read
      contents: read

    strategy:
      fail-fast: false
      matrix:
        language: [javascript, python]

    steps:
      - uses: actions/checkout@v4

      - name: Initialize CodeQL
        uses: github/codeql-action/init@v3
        with:
          languages: ${{ matrix.language }}
          queries: security-extended,security-and-quality

      - name: Autobuild
        uses: github/codeql-action/autobuild@v3

      - name: Perform CodeQL Analysis
        uses: github/codeql-action/analyze@v3
        with:
          category: "/language:${{ matrix.language }}"

Semgrep Custom Rules

# .semgrep.yml - Custom security rules
rules:
  - id: wia-no-eval
    patterns:
      - pattern: eval(...)
    message: "eval() is dangerous and should not be used"
    severity: ERROR
    languages: [javascript, typescript]

  - id: wia-sql-injection
    patterns:
      - pattern: |
          $QUERY = "..." + $USER_INPUT + "..."
          $DB.query($QUERY)
    message: "Potential SQL injection vulnerability"
    severity: ERROR
    languages: [javascript, typescript]

  - id: wia-hardcoded-secret
    patterns:
      - pattern-regex: '(password|secret|api_key)\s*=\s*["\'][^"\']{8,}["\']'
    message: "Hardcoded secret detected"
    severity: ERROR
    languages: [javascript, typescript, python]
# GitHub Actions - Semgrep
name: Semgrep
on: [push, pull_request]

jobs:
  semgrep:
    runs-on: ubuntu-latest
    container:
      image: semgrep/semgrep
    steps:
      - uses: actions/checkout@v4

      - name: Run Semgrep
        run: semgrep scan --config auto --config .semgrep.yml --sarif --output semgrep.sarif

      - name: Upload SARIF
        uses: github/codeql-action/upload-sarif@v3
        with:
          sarif_file: semgrep.sarif
SAST Tool Selection: Use multiple SAST tools as they have different strengths. CodeQL excels at data flow analysis, Semgrep is fast and customizable, and language-specific tools often catch nuanced issues.

Dynamic Application Security Testing (DAST)

DAST tools test running applications by simulating attacks. They find runtime vulnerabilities that SAST cannot detect, such as authentication flaws and server misconfigurations.

OWASP ZAP Integration

# GitHub Actions - OWASP ZAP DAST
name: DAST Security Scan
on:
  push:
    branches: [main]
  schedule:
    - cron: '0 2 * * *'  # Nightly security scan

jobs:
  zap-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build and start application
        run: |
          docker-compose up -d
          sleep 30  # Wait for app to be ready

      - name: OWASP ZAP Baseline Scan
        uses: zaproxy/action-baseline@v0.10.0
        with:
          target: 'http://localhost:3000'
          rules_file_name: '.zap/rules.tsv'
          cmd_options: '-a'

      - name: Upload ZAP Report
        uses: actions/upload-artifact@v4
        if: always()
        with:
          name: zap-report
          path: report_html.html

Software Composition Analysis (SCA)

SCA tools analyze third-party dependencies for known vulnerabilities. With modern applications using hundreds of dependencies, SCA is essential for supply chain security.

Dependency Vulnerability Scanning

# GitHub Actions - Comprehensive SCA
name: Dependency Security
on:
  push:
  pull_request:
  schedule:
    - cron: '0 0 * * *'

jobs:
  sca:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # GitHub's built-in dependency scanning
      - name: Dependency Review
        uses: actions/dependency-review-action@v4
        if: github.event_name == 'pull_request'
        with:
          fail-on-severity: high
          deny-licenses: GPL-3.0, AGPL-3.0

      # Snyk for deep vulnerability analysis
      - name: Snyk vulnerability scan
        uses: snyk/actions/node@master
        env:
          SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}
        with:
          args: --severity-threshold=high --sarif-file-output=snyk.sarif

      - name: Upload Snyk results
        uses: github/codeql-action/upload-sarif@v3
        with:
          sarif_file: snyk.sarif

        with:

            sbom.json
            sbom-spdx.json

Container Security

Container images require special security attention as they package both application code and system dependencies.

# GitHub Actions - Container Security
name: Container Security
on:
  push:
    paths:
      - 'Dockerfile'
      - 'docker-compose.yml'

jobs:
  scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build image
        run: docker build -t myapp:${{ github.sha }} .

      # Trivy vulnerability scanner
      - name: Trivy scan
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: 'myapp:${{ github.sha }}'
          format: 'sarif'
          output: 'trivy-results.sarif'
          severity: 'CRITICAL,HIGH'

      - name: Upload Trivy results
        uses: github/codeql-action/upload-sarif@v3
        with:
          sarif_file: trivy-results.sarif

      # Dockerfile best practices
      - name: Hadolint
        uses: hadolint/hadolint-action@v3.1.0
        with:
          dockerfile: Dockerfile
          failure-threshold: error

Security Compliance Automation

# Compliance as Code
name: Compliance Check
on:
  push:
  schedule:
    - cron: '0 0 * * 0'

jobs:
  compliance:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Check OWASP Top 10 compliance
        run: |
          echo "Verifying OWASP Top 10 controls..."
          # A01 - Broken Access Control
          grep -r "authorize" src/ || echo "::warning::No authorization checks found"
          # A02 - Cryptographic Failures
          ! grep -r "md5\|sha1" src/ || echo "::warning::Weak crypto detected"
          # A03 - Injection
          npm audit --audit-level=moderate

      - name: Generate compliance report
        run: |
          node scripts/generate-compliance-report.js > compliance-report.json

      - name: Upload compliance artifacts
        uses: actions/upload-artifact@v4
        with:
          name: compliance-report
          path: compliance-report.json

Security in CI/CD is not a one-time setup but an ongoing practice. Regular updates to security rules, continuous monitoring for new vulnerabilities, and security training for developers are essential for maintaining a secure software delivery pipeline.
