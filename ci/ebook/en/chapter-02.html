Chapter 2: CI/CD Pipeline Architecture

A well-designed CI/CD architecture is the foundation of reliable and efficient software delivery. This chapter explores pipeline design patterns, runner infrastructure, orchestration strategies, and artifact management practices that enable teams to build scalable and maintainable CI/CD systems.

Pipeline Design Patterns

Pipeline design patterns provide reusable solutions to common CI/CD challenges. Understanding these patterns enables architects to design systems that are both powerful and maintainable.

The Linear Pipeline

The simplest pipeline pattern where stages execute sequentially. Each stage must complete successfully before the next begins. This pattern is ideal for straightforward applications with linear dependencies.

# Linear Pipeline Pattern
# GitHub Actions Implementation
name: Linear Pipeline
on: [push]

jobs:
  build-test-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install
        run: npm ci

      - name: Lint
        run: npm run lint

      - name: Test
        run: npm test

      - name: Build
        run: npm run build

      - name: Deploy
        run: npm run deploy
        if: github.ref == 'refs/heads/main'

The Fan-Out/Fan-In Pattern

This pattern parallelizes independent tasks to reduce overall pipeline duration, then converges at synchronization points. Essential for complex applications with multiple test suites or build targets.

# Fan-Out/Fan-In Pattern
# GitLab CI Implementation
stages:
  - build
  - test
  - deploy

build:
  stage: build
  script:
    - npm ci
    - npm run build
  artifacts:
    paths:
      - dist/

# Fan-Out: Parallel test jobs
unit-tests:
  stage: test
  needs: [build]
  script:
    - npm run test:unit

integration-tests:
  stage: test
  needs: [build]
  script:
    - npm run test:integration

e2e-tests:
  stage: test
  needs: [build]
  script:
    - npm run test:e2e

security-scan:
  stage: test
  needs: [build]
  script:
    - npm audit

# Fan-In: Deploy requires all tests
deploy:
  stage: deploy
  needs: [unit-tests, integration-tests, e2e-tests, security-scan]
  script:
    - npm run deploy
  only:
    - main
WIA-CI Architecture Principle: Design pipelines with the "fail fast" mentality. Place quick-running checks (linting, formatting) early in the pipeline to provide rapid feedback, reserving longer-running tests for later stages.

The Matrix Strategy Pattern

Matrix builds test code across multiple combinations of configurations, operating systems, or runtime versions. This pattern ensures compatibility across diverse environments.

# Matrix Strategy Pattern
# GitHub Actions Implementation
name: Matrix Build
on: [push, pull_request]

jobs:
  test:
    strategy:
      fail-fast: false
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        node: [18, 20, 22]
        exclude:
          - os: windows-latest
            node: 18

    runs-on: ${{ matrix.os }}

    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js ${{ matrix.node }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}

      - name: Install and Test
        run: |
          npm ci
          npm test

Runner Infrastructure

Runners are the execution environments where pipeline jobs run. Choosing the right runner infrastructure is critical for performance, cost, and security.

Types of Runners

Runner Type Pros Cons Use Cases
Hosted (Cloud) Zero maintenance, scalable Limited customization, cost at scale Most projects, open source
Self-Hosted Full control, cost-effective Maintenance overhead Enterprise, compliance needs
Container-Based Consistent environments Container image management Microservices, reproducibility
Serverless Pay-per-use, auto-scaling Cold start latency Sporadic builds, cost optimization

Self-Hosted Runner Configuration

apiVersion: actions.summerwind.dev/v1alpha1

  name: wia-ci-runners
  namespace: actions-runner-system

  replicas: 5
  template:
    spec:

          cpu: "2"
          memory: "4Gi"
      volumeMounts:
        - name: docker-socket
          mountPath: /var/run/docker.sock
      volumes:
        - name: docker-socket
          hostPath:
            path: /var/run/docker.sock

Orchestration Strategies

Orchestration defines how pipelines coordinate across multiple repositories, services, and environments. Effective orchestration enables complex deployment scenarios while maintaining reliability.

Monorepo Orchestration

In monorepo setups, changes to shared code may require rebuilding multiple dependent projects. Smart change detection minimizes unnecessary builds.

# Monorepo Change Detection
# GitHub Actions with Path Filters
name: Monorepo CI
on:
  push:
    branches: [main]
  pull_request:

jobs:
  detect-changes:
    runs-on: ubuntu-latest
    outputs:
      api: ${{ steps.changes.outputs.api }}
      web: ${{ steps.changes.outputs.web }}
      shared: ${{ steps.changes.outputs.shared }}
    steps:
      - uses: actions/checkout@v4
      - uses: dorny/paths-filter@v2
        id: changes
        with:
          filters: |
            api:
              - 'packages/api/**'
              - 'packages/shared/**'
            web:
              - 'packages/web/**'
              - 'packages/shared/**'
            shared:
              - 'packages/shared/**'

  build-api:
    needs: detect-changes
    if: needs.detect-changes.outputs.api == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm run build --workspace=packages/api

  build-web:
    needs: detect-changes
    if: needs.detect-changes.outputs.web == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm run build --workspace=packages/web

Multi-Repository Orchestration

When services span multiple repositories, coordination becomes more complex. Event-driven architectures and centralized orchestration help manage dependencies.

# Cross-Repository Trigger
# GitHub Actions - Trigger Downstream Pipeline
name: Trigger Downstream
on:
  workflow_dispatch:
  push:
    tags: ['v*']

jobs:
  trigger-downstream:
    runs-on: ubuntu-latest
    steps:
      - name: Trigger Consumer Repo
        uses: peter-evans/repository-dispatch@v2
        with:
          token: ${{ secrets.CROSS_REPO_TOKEN }}
          repository: your-org/consumer-service
          event-type: dependency-updated
          client-payload: |
            {
              "version": "${{ github.ref_name }}",
              "source": "${{ github.repository }}"
            }
Orchestration Best Practice: Implement idempotent pipelines that can be safely re-run without side effects. This enables automatic retries and manual re-triggering without fear of duplicate deployments or corrupted state.

Artifact Management

Artifacts are the outputs of build processes - compiled binaries, container images, test reports, and documentation. Effective artifact management ensures traceability and enables reliable deployments.

Artifact Types and Storage

Artifact Type Storage Solution Retention Policy
Build Outputs CI Platform Storage 30-90 days
Container Images Container Registry Based on tags (latest, semver)
Package Libraries Package Registry (npm, PyPI) Permanent for releases
Test Reports Object Storage (S3, GCS) 90-365 days
Documentation Static Hosting Permanent, versioned

Container Image Versioning

# Container Image Build and Push
# GitHub Actions with GHCR
name: Build Container
on:
  push:
    branches: [main]
    tags: ['v*']

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write

    steps:
      - uses: actions/checkout@v4

      - name: Login to Container Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Extract Metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=ref,event=branch
            type=semver,pattern={{version}}
            type=sha,prefix=

      - name: Build and Push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

Architecture Decision Framework

The WIA-CI standard provides a decision framework for architectural choices:

# WIA-CI Architecture Decision Framework

1. SCALE ASSESSMENT
   - Team size: < 10, 10-50, 50-200, 200+
   - Build frequency: hourly, hundreds/day, thousands/day
   - Repository strategy: monorepo, polyrepo, hybrid

2. RUNNER SELECTION
   if (compliance_requirements || specialized_hardware):
       use self_hosted_runners
   elif (cost_sensitive && predictable_load):
       use reserved_capacity
   else:
       use hosted_runners

3. PARALLELIZATION STRATEGY
   if (independent_test_suites):
       use fan_out_pattern
   if (multiple_target_environments):
       use matrix_strategy
   if (long_running_tests):
       split_and_parallelize

4. ARTIFACT STRATEGY
   - Immutable artifacts with semantic versioning
   - Retention based on environment criticality
   - Signed artifacts for production deployments
WIA-CI Maturity Model - Level 2 (Architecture):
  • Multi-stage pipeline with clear separation
  • Parallel job execution where applicable
  • Artifact versioning and storage
  • Branch-based pipeline variations

A solid architectural foundation enables teams to iterate quickly while maintaining reliability. The patterns and practices in this chapter provide the building blocks for CI/CD systems that can grow with your organization.
