챕터 3: 빌드 시스템과 의존성 관리

빌드 시스템은 모든 CI 파이프라인의 엔진을 형성하며 소스 코드를 배포 가능한 아티팩트로 변환합니다. 이 챕터에서는 기술 스택 전반의 빌드 자동화, 의존성 관리 전략, 그리고 빌드 성능을 극적으로 향상시키는 캐싱 기법을 다룹니다.

빌드 시스템 기초

빌드 시스템은 소스 코드의 컴파일, 변환, 패키징을 조율합니다. 현대 빌드 시스템은 복잡한 의존성 그래프를 처리하고, 증분 빌드를 지원하며, CI 플랫폼과 원활하게 통합되어야 합니다.

빌드 시스템 특성

특성 설명 영향
결정론적 동일한 입력이 동일한 출력을 생성 재현 가능한 빌드
증분적 변경된 컴포넌트만 재빌드 더 빠른 반복
병렬 독립적인 작업을 동시에 실행 빌드 시간 단축
캐시 가능 이전 빌드 출력을 저장하고 재사용 극적인 속도 향상
격리됨 외부 환경으로부터 격리 일관된 결과

WIA-CI 빌드 표준: 모든 빌드 구성은 선언적이고 버전 관리되어야 합니다. 특정 실행 환경이나 수동 설정 단계에 의존하는 명령형 빌드 스크립트는 피하세요.

Go 빌드 시스템

# Makefile - Go 빌드 구성
.PHONY: all build test lint clean

VERSION := $(shell git describe --tags --always)
BUILD_TIME := $(shell date -u +"%Y-%m-%dT%H:%M:%SZ")
LDFLAGS := -ldflags "-X main.Version=${VERSION} -X main.BuildTime=${BUILD_TIME}"

all: lint test build

build:
	CGO_ENABLED=0 go build ${LDFLAGS} -o bin/app ./cmd/app

test:
	go test -v -race -coverprofile=coverage.out ./...

lint:
	golangci-lint run ./...

clean:
	rm -rf bin/ coverage.out

의존성 관리

의존성은 코드가 의존하는 외부 라이브러리와 패키지입니다. 효과적인 의존성 관리는 보안 위험을 최소화하면서 재현 가능한 빌드를 보장합니다.

의존성 관리 원칙

  • 락 파일: 재현 가능한 빌드를 보장하기 위해 항상 락 파일(package-lock.json, pnpm-lock.yaml, Cargo.lock)을 커밋하세요
  • 시맨틱 버전 관리: 의존성에 대한 semver 범위를 이해하고 적절히 사용하세요
  • 최소 의존성: 정기적으로 감사하고 사용하지 않는 의존성을 제거하세요
  • 보안 스캐닝: 의존성 취약점 스캐닝을 CI에 통합하세요
  • 프라이빗 레지스트리: 내부 패키지와 퍼블릭 레지스트리 프록시를 위해 프라이빗 레지스트리를 사용하세요
# 의존성 감사 통합
# GitHub Actions
name: 의존성 보안
on:
  push:
  schedule:
    - cron: '0 0 * * *'  # 일일 스캔

jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Node.js 설정
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: 의존성 설치
        run: npm ci

      - name: npm audit 실행
        run: npm audit --audit-level=moderate

      - name: 오래된 패키지 확인
        run: npm outdated || true

      - name: Snyk 보안 스캔 실행
        uses: snyk/actions/node@master
        env:
          SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}
        with:
          args: --severity-threshold=high

의존성 캐싱 전략

# 다층 캐싱 전략
# GitHub Actions
name: 캐싱을 통한 빌드
on: [push]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # 락 파일 기반 node_modules 캐시
      - name: node_modules 캐시
        uses: actions/cache@v4
        with:
          path: node_modules
          key: ${{ runner.os }}-node-${{ hashFiles('pnpm-lock.yaml') }}
          restore-keys: |
            ${{ runner.os }}-node-

      # 빌드 출력물 캐시
      - name: 빌드 캐시
        uses: actions/cache@v4
        with:
          path: |
            dist
            .next/cache
          key: ${{ runner.os }}-build-${{ github.sha }}
          restore-keys: |
            ${{ runner.os }}-build-

      - name: 의존성 설치
        run: pnpm install --frozen-lockfile

      - name: 빌드
        run: pnpm build

고급 캐싱 전략

효과적인 캐싱은 빌드 시간을 50-90% 줄일 수 있습니다. WIA-CI 표준은 최적의 성능을 위해 여러 캐싱 레이어를 구현할 것을 권장합니다.

캐싱 레이어

레이어 내용 키 전략 일반적인 절감
의존성 node_modules, .m2, vendor 락 파일 해시 30-60초
빌드 캐시 컴파일 출력, 중간물 소스 해시 + 구성 1-5분
Docker 레이어 이미지 레이어 레이어 콘텐츠 해시 2-10분
테스트 캐시 테스트 결과, 스냅샷 테스트 파일 해시 30-120초
캐시 무효화 전략: 적절한 세분성으로 캐시 키를 설계하세요. 너무 구체적인 키는 캐시 미스를 초래하고, 너무 광범위한 키는 오래된 아티팩트를 복원할 수 있습니다. 캐시 키에 OS, 도구 버전, 관련 구성 해시를 포함하세요.

빌드 재현성

재현 가능한 빌드는 동일한 소스 코드가 항상 동일한 바이너리를 생성하도록 보장합니다. 이는 보안 감사와 프로덕션 문제 디버깅에 매우 중요합니다.

# 재현 가능한 빌드를 위한 Dockerfile
FROM node:20-alpine AS builder

# 결정론적 환경 설정
ENV NODE_ENV=production
ENV SOURCE_DATE_EPOCH=0

WORKDIR /app

# 의존성 파일 먼저 복사 (더 나은 캐싱)
COPY package.json pnpm-lock.yaml ./

# 정확한 버전으로 의존성 설치
RUN corepack enable && pnpm install --frozen-lockfile

# 소스 코드 복사
COPY . .

# 결정론적 출력으로 빌드
RUN pnpm build

# 프로덕션 이미지
FROM node:20-alpine AS runner
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./

CMD ["node", "dist/index.js"]
WIA-CI 성숙도 모델 - 레벨 3 (빌드 우수성):
  • 환경 전반에 걸친 재현 가능한 빌드
  • 70%+ 캐시 히트율의 다층 캐싱
  • 대부분의 변경에 대해 5분 이내의 빌드 시간
  • 보안 스캐닝을 통한 자동화된 의존성 업데이트

빌드 시스템과 의존성 관리를 마스터하는 것은 효율적인 CI 파이프라인에 필수적입니다. 이 챕터의 기법들은 팀과 코드베이스에 맞게 확장되는 더 빠르고 더 신뢰할 수 있는 빌드를 달성하는 데 도움이 될 것입니다.
