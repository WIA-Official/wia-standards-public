챕터 2: CI/CD 파이프라인 아키텍처

잘 설계된 CI/CD 아키텍처는 신뢰할 수 있고 효율적인 소프트웨어 배포의 기반입니다. 이 챕터에서는 파이프라인 설계 패턴, 러너 인프라, 오케스트레이션 전략, 그리고 확장 가능하고 유지보수 가능한 CI/CD 시스템을 구축하기 위한 아티팩트 관리 실천을 탐구합니다.

파이프라인 설계 패턴

파이프라인 설계 패턴은 일반적인 CI/CD 문제에 대한 재사용 가능한 솔루션을 제공합니다. 이러한 패턴을 이해하면 아키텍트는 강력하면서도 유지보수 가능한 시스템을 설계할 수 있습니다.

선형 파이프라인

스테이지가 순차적으로 실행되는 가장 간단한 파이프라인 패턴입니다. 각 스테이지는 다음이 시작되기 전에 성공적으로 완료되어야 합니다. 이 패턴은 선형적 의존성을 가진 단순한 애플리케이션에 이상적입니다.

# 선형 파이프라인 패턴
# GitHub Actions 구현
name: 선형 파이프라인
on: [push]

jobs:
  build-test-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: 설치
        run: npm ci

      - name: 린트
        run: npm run lint

      - name: 테스트
        run: npm test

      - name: 빌드
        run: npm run build

      - name: 배포
        run: npm run deploy
        if: github.ref == 'refs/heads/main'

팬아웃/팬인 패턴

이 패턴은 독립적인 작업을 병렬화하여 전체 파이프라인 기간을 줄인 다음 동기화 지점에서 수렴합니다. 여러 테스트 스위트나 빌드 대상을 가진 복잡한 애플리케이션에 필수적입니다.

# 팬아웃/팬인 패턴
# GitLab CI 구현
stages:
  - build
  - test
  - deploy

build:
  stage: build
  script:
    - npm ci
    - npm run build
  artifacts:
    paths:
      - dist/

# 팬아웃: 병렬 테스트 작업
unit-tests:
  stage: test
  needs: [build]
  script:
    - npm run test:unit

integration-tests:
  stage: test
  needs: [build]
  script:
    - npm run test:integration

e2e-tests:
  stage: test
  needs: [build]
  script:
    - npm run test:e2e

security-scan:
  stage: test
  needs: [build]
  script:
    - npm audit

# 팬인: 배포는 모든 테스트 필요
deploy:
  stage: deploy
  needs: [unit-tests, integration-tests, e2e-tests, security-scan]
  script:
    - npm run deploy
  only:
    - main
WIA-CI 아키텍처 원칙: "빨리 실패하기" 마인드셋으로 파이프라인을 설계하세요. 빠르게 실행되는 검사(린팅, 포맷팅)를 파이프라인 초기에 배치하여 신속한 피드백을 제공하고, 더 오래 실행되는 테스트는 이후 스테이지에 예약하세요.

매트릭스 전략 패턴

매트릭스 빌드는 여러 구성, 운영 체제 또는 런타임 버전 조합에서 코드를 테스트합니다. 이 패턴은 다양한 환경에서의 호환성을 보장합니다.

# 매트릭스 전략 패턴
# GitHub Actions 구현
name: 매트릭스 빌드
on: [push, pull_request]

jobs:
  test:
    strategy:
      fail-fast: false
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        node: [18, 20, 22]
        exclude:
          - os: windows-latest
            node: 18

    runs-on: ${{ matrix.os }}

    steps:
      - uses: actions/checkout@v4

      - name: Node.js ${{ matrix.node }} 설정
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}

      - name: 설치 및 테스트
        run: |
          npm ci
          npm test

러너 인프라

러너는 파이프라인 작업이 실행되는 실행 환경입니다. 올바른 러너 인프라를 선택하는 것은 성능, 비용, 보안에 매우 중요합니다.

러너 유형

러너 유형 장점 단점 사용 사례
호스팅 (클라우드) 유지보수 없음, 확장 가능 제한된 커스터마이징, 규모에 따른 비용 대부분의 프로젝트, 오픈 소스
셀프 호스팅 완전한 제어, 비용 효율적 유지보수 오버헤드 엔터프라이즈, 규정 준수 요구
컨테이너 기반 일관된 환경 컨테이너 이미지 관리 마이크로서비스, 재현성
서버리스 사용량 기반 과금, 자동 확장 콜드 스타트 지연 산발적 빌드, 비용 최적화

셀프 호스팅 러너 구성

apiVersion: actions.summerwind.dev/v1alpha1

오케스트레이션 전략

오케스트레이션은 파이프라인이 여러 저장소, 서비스, 환경에 걸쳐 어떻게 조정되는지를 정의합니다. 효과적인 오케스트레이션은 신뢰성을 유지하면서 복잡한 배포 시나리오를 가능하게 합니다.

모노레포 오케스트레이션

모노레포 설정에서 공유 코드의 변경은 여러 종속 프로젝트의 재빌드가 필요할 수 있습니다. 스마트 변경 감지는 불필요한 빌드를 최소화합니다.

# 모노레포 변경 감지
# GitHub Actions with Path Filters
name: 모노레포 CI
on:
  push:
    branches: [main]
  pull_request:

jobs:
  detect-changes:
    runs-on: ubuntu-latest
    outputs:
      api: ${{ steps.changes.outputs.api }}
      web: ${{ steps.changes.outputs.web }}
      shared: ${{ steps.changes.outputs.shared }}
    steps:
      - uses: actions/checkout@v4
      - uses: dorny/paths-filter@v2
        id: changes
        with:
          filters: |
            api:
              - 'packages/api/**'
              - 'packages/shared/**'
            web:
              - 'packages/web/**'
              - 'packages/shared/**'
            shared:
              - 'packages/shared/**'

  build-api:
    needs: detect-changes
    if: needs.detect-changes.outputs.api == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm run build --workspace=packages/api

  build-web:
    needs: detect-changes
    if: needs.detect-changes.outputs.web == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm run build --workspace=packages/web
오케스트레이션 모범 사례: 부작용 없이 안전하게 재실행할 수 있는 멱등성 파이프라인을 구현하세요. 이를 통해 중복 배포나 손상된 상태에 대한 우려 없이 자동 재시도와 수동 재트리거가 가능합니다.

아티팩트 관리

아티팩트는 빌드 프로세스의 출력물입니다 - 컴파일된 바이너리, 컨테이너 이미지, 테스트 리포트, 문서 등. 효과적인 아티팩트 관리는 추적성을 보장하고 신뢰할 수 있는 배포를 가능하게 합니다.

아티팩트 유형 및 저장소

아티팩트 유형 저장 솔루션 보존 정책
빌드 출력물 CI 플랫폼 저장소 30-90일
컨테이너 이미지 컨테이너 레지스트리 태그 기반 (latest, semver)
패키지 라이브러리 패키지 레지스트리 (npm, PyPI) 릴리스용 영구
테스트 리포트 객체 저장소 (S3, GCS) 90-365일
문서 정적 호스팅 영구, 버전 관리

컨테이너 이미지 버전 관리

# 컨테이너 이미지 빌드 및 푸시
# GitHub Actions with GHCR
name: 컨테이너 빌드
on:
  push:
    branches: [main]
    tags: ['v*']

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write

    steps:
      - uses: actions/checkout@v4

      - name: 컨테이너 레지스트리 로그인
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: 메타데이터 추출
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=ref,event=branch
            type=semver,pattern={{version}}
            type=sha,prefix=

      - name: 빌드 및 푸시
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

아키텍처 결정 프레임워크

WIA-CI 표준은 아키텍처 선택을 위한 결정 프레임워크를 제공합니다:

# WIA-CI 아키텍처 결정 프레임워크

1. 규모 평가
   - 팀 규모: < 10, 10-50, 50-200, 200+
   - 빌드 빈도: 시간당, 수백/일, 수천/일
   - 저장소 전략: 모노레포, 폴리레포, 하이브리드

2. 러너 선택
   if (규정_준수_요구사항 || 전문_하드웨어):
       셀프_호스팅_러너_사용
   elif (비용_민감 && 예측_가능한_부하):
       예약_용량_사용
   else:
       호스팅_러너_사용

3. 병렬화 전략
   if (독립적인_테스트_스위트):
       팬아웃_패턴_사용
   if (다중_타겟_환경):
       매트릭스_전략_사용
   if (장시간_실행_테스트):
       분할_및_병렬화

4. 아티팩트 전략
   - 시맨틱 버전 관리를 통한 불변 아티팩트
   - 환경 중요도에 따른 보존
   - 프로덕션 배포를 위한 서명된 아티팩트
WIA-CI 성숙도 모델 - 레벨 2 (아키텍처):
  • 명확한 분리가 있는 다단계 파이프라인
  • 적용 가능한 곳에서 병렬 작업 실행
  • 아티팩트 버전 관리 및 저장
  • 브랜치 기반 파이프라인 변형

견고한 아키텍처 기반은 팀이 신뢰성을 유지하면서 빠르게 반복할 수 있게 합니다. 이 챕터의 패턴과 실천은 조직과 함께 성장할 수 있는 CI/CD 시스템을 위한 구성 요소를 제공합니다.
