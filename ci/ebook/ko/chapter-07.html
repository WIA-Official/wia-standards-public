챕터 7: CI/CD 운영 및 최적화

CI/CD 시스템 운영은 설정하고 잊어버리는 일이 아닙니다. 성능과 신뢰성을 유지하기 위해 지속적인 모니터링, 최적화, 문제 해결이 필요합니다. 이 챕터에서는 대규모 CI/CD 인프라 관리의 운영적 측면을 다룹니다.

파이프라인 모니터링

효과적인 모니터링은 파이프라인 상태, 성능 추세, 그리고 개발 팀에 영향을 미치기 전 잠재적 문제에 대한 가시성을 제공합니다.

추적해야 할 핵심 메트릭

메트릭 설명 목표 알림 임계값
빌드 시간 트리거에서 완료까지 시간 10분 이내 20분 초과
성공률 성공한 빌드 비율 95% 초과 80% 미만
큐 대기 시간 러너 대기 시간 30초 이내 5분 초과
불안정 테스트율 일관성 없이 통과/실패하는 테스트 1% 미만 5% 초과
MTTR 실패 후 평균 복구 시간 30분 이내 2시간 초과

성능 최적화

파이프라인 성능은 개발자 생산성에 직접적인 영향을 미칩니다. 느린 파이프라인은 컨텍스트 전환, 배치 커밋, 지연된 피드백으로 이어집니다.

빌드 시간 분석

# GitHub Actions - 빌드 시간 분석
name: 빌드 시간 분석
on:
  workflow_run:
    workflows: ["CI 파이프라인"]
    types: [completed]

jobs:
  analyze:
    runs-on: ubuntu-latest
    steps:
      - name: 워크플로우 실행 데이터 가져오기
        uses: actions/github-script@v7
        with:
          script: |
            const run = await github.rest.actions.getWorkflowRun({
              owner: context.repo.owner,
              repo: context.repo.repo,
              run_id: context.payload.workflow_run.id
            });

            const jobs = await github.rest.actions.listJobsForWorkflowRun({
              owner: context.repo.owner,
              repo: context.repo.repo,
              run_id: context.payload.workflow_run.id
            });

            // 단계별 시간 분석
            const stepAnalysis = jobs.data.jobs.flatMap(job =>
              job.steps.map(step => ({
                job: job.name,
                step: step.name,
                duration: new Date(step.completed_at) - new Date(step.started_at)
              }))
            );

            // 가장 느린 단계 찾기
            const slowest = stepAnalysis
              .sort((a, b) => b.duration - a.duration)
              .slice(0, 5);

            console.log('가장 느린 단계:', slowest);

            // 모니터링으로 전송
            // await sendMetrics(stepAnalysis);

# 최적화 체크리스트

## 캐싱 최적화
- [ ] 락 파일 해시로 의존성 캐시
- [ ] 실행 간 빌드 출력 캐시
- [ ] Docker 레이어 캐싱 활성화
- [ ] 모노레포용 원격 캐시 (Turborepo, Nx)

## 병렬화
- [ ] 독립 작업 병렬 실행
- [ ] 샤드 간 테스트 스위트 분할
- [ ] 다중 플랫폼 테스트용 매트릭스 빌드
- [ ] 가능한 곳에서 병렬 단계 실행

## 리소스 최적화
- [ ] 적절한 크기의 러너 인스턴스
- [ ] 비중요 작업에 스팟/선점형 인스턴스
- [ ] 오래된 아티팩트 정기 정리
- [ ] 효율적인 베이스 이미지 (Alpine, distroless)

## 파이프라인 설계
- [ ] 빠른 피드백을 위한 fail-fast

캐싱 최적화 예시

# 고급 캐싱 전략
name: 최적화된 빌드
on: [push]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # 모든 캐시 병렬 복원
      - name: 의존성 캐시 복원
        uses: actions/cache/restore@v4
        id: deps-cache
        with:
          path: node_modules
          key: deps-${{ hashFiles('pnpm-lock.yaml') }}

      - name: 빌드 캐시 복원
        uses: actions/cache/restore@v4
        id: build-cache
        with:
          path: |
            .next/cache
            dist
          key: build-${{ runner.os }}-${{ hashFiles('src/**') }}
          restore-keys: |
            build-${{ runner.os }}-

      # 캐시 미스 시에만 설치
      - name: 의존성 설치
        if: steps.deps-cache.outputs.cache-hit != 'true'
        run: pnpm install --frozen-lockfile

      # 증분 빌드
      - name: 빌드
        run: pnpm build

      # 향후 실행을 위해 캐시 저장
      - name: 의존성 캐시 저장
        uses: actions/cache/save@v4
        if: steps.deps-cache.outputs.cache-hit != 'true'
        with:
          path: node_modules
          key: deps-${{ hashFiles('pnpm-lock.yaml') }}

      - name: 빌드 캐시 저장
        uses: actions/cache/save@v4
        with:
          path: |
            .next/cache
            dist
          key: build-${{ runner.os }}-${{ hashFiles('src/**') }}
10배 빌드 시간 개선: 일반적인 최적화 여정은 공격적인 캐싱, 병렬화, 증분 빌드를 구현하여 빌드 시간을 20분 이상에서 2분 이내로 줄일 수 있습니다. 빌드 시간을 핵심 엔지니어링 메트릭으로 추적하세요.

# 불안정 테스트 감지 및 격리
name: 불안정 감지 테스트
on: [push]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: 재시도로 테스트 실행
        run: |
          npm test -- --retries=2 --reporter=json > test-results.json

      - name: 불안정 테스트 분석
        run: |
          node << 'EOF'
          const results = require('./test-results.json');
          const flaky = results.testResults
            .flatMap(f => f.assertionResults)
            .filter(t => t.retryReasons && t.retryReasons.length > 0);

          if (flaky.length > 0) {
            console.log('불안정 테스트 감지됨:');
            flaky.forEach(t => console.log(`  - ${t.fullName}`));

            // 추적 시스템에 보고
            // await reportFlakyTests(flaky);
          }
          EOF

      - name: 불안정 테스트 보고서 업로드
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: flaky-test-report
          path: test-results.json

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: 리소스 모니터링 시작
        run: |
          # 백그라운드에서 리소스 모니터링
          while true; do
            echo "$(date) - 메모리: $(free -m | awk 'NR==2{print $3}')MB 사용"
            echo "$(date) - 디스크: $(df -h / | awk 'NR==2{print $5}') 사용"
            sleep 30
          done &
          echo $! > /tmp/monitor.pid

      - name: 빌드
        run: npm run build

      - name: 모니터링 중지
        if: always()
        run: kill $(cat /tmp/monitor.pid) 2>/dev/null || true

용량 계획

# 용량 계획 모델
"""
WIA-CI 용량 계획 계산기

입력:
- daily_commits: 일일 평균 커밋
- avg_build_time: 평균 빌드 시간 (분)
- peak_multiplier: 피크 대 평균 비율
- target_queue_time: 최대 허용 큐 대기 시간 (분)

출력:
- 필요한 러너 용량
"""

def calculate_runners(daily_commits, avg_build_time, peak_multiplier, target_queue_time):
    # 시간당 비율로 변환
    hourly_builds = daily_commits / 8  # 8 근무 시간 가정

    # 피크 시간 빌드
    peak_builds = hourly_builds * peak_multiplier

    # 필요한 용량 (시간당 빌드)
    builds_per_runner_hour = 60 / avg_build_time

    # 피크 부하를 위한 최소 러너
    min_runners = peak_builds / builds_per_runner_hour

    # 큐 시간 목표를 위한 버퍼 추가
    buffer_factor = 1 + (avg_build_time / (target_queue_time + avg_build_time))

    return ceil(min_runners * buffer_factor)

# 예시: 200 커밋/일, 10분 빌드, 2배 피크, 2분 큐 목표
# calculate_runners(200, 10, 2, 2) = 10 러너
WIA-CI 운영 성숙도:
  • 레벨 1: 기본 모니터링, 수동 문제 해결
  • 레벨 2: 대시보드, 알림, 문서화된 런북
  • 레벨 3: 자동화된 최적화, 셀프서비스 디버깅
  • 레벨 4: 예측적 용량, 자동화된 복구

CI/CD의 운영 우수성은 지속적인 여정입니다. 메트릭의 정기적인 검토, 선제적 최적화, 도구에 대한 투자는 개발자 생산성과 시스템 신뢰성에서 배당금을 지불합니다.
