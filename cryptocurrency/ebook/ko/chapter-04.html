🪙 Chapter 4: Layer 1 - 데이터 형식

WIA-FIN-003 Cryptocurrency Standard | 트랜잭션 스키마와 데이터 구조

4.1 서론: 왜 데이터 형식이 중요한가?

인터넷의 성공은 표준화된 데이터 형식 덕분입니다. IP 패킷, HTTP 요청/응답, HTML 문서 - 모두 엄격하게 정의된 형식을 따릅니다. 마찬가지로, 암호화폐 생태계가 진정으로 상호운용 가능하려면 통일된 데이터 형식이 필수적입니다.

현재 상황은 혼란스럽습니다. 비트코인은 독자적인 트랜잭션 형식을, 이더리움은 RLP 인코딩을, 솔라나는 Borsh를 사용합니다. 각 체인의 주소 형식도 다릅니다. 이는 마치 각 웹사이트가 고유한 HTML 문법을 사용하는 것과 같습니다 - 불가능하지는 않지만, 극도로 비효율적입니다.

WIA-FIN-003 Layer 1은 이 문제를 해결합니다. 모든 블록체인이 이해할 수 있는 공통 언어를 제공하면서도, 각 체인의 고유한 특성을 존중합니다.

💡 핵심 개념: Layer 1은 "lingua franca"(공통어)입니다. 각 블록체인은 자체 내부 형식을 유지하지만, WIA-FIN-003 형식으로 변환하여 다른 체인과 통신합니다. 이는 마치 각국이 자국어를 사용하지만 국제 회의에서는 영어를 사용하는 것과 같습니다.

4.2 Universal Transaction Schema (UTS)

Universal Transaction Schema는 WIA-FIN-003의 핵심입니다. 이는 모든 암호화폐 거래를 표현할 수 있는 통일된 JSON 스키마입니다.

4.2.1 핵심 설계 목표

목표 설명 구현 방법
완전성 모든 유형의 거래 표현 확장 가능한 메타데이터 필드
간결성 불필요한 데이터 최소화 선택적 필드, 압축 인코딩
검증 가능성 모든 필드 독립적 검증 명시적 타입, 체크섬, 서명
확장성 미래 기능 추가 가능 버전 필드, 플러그인 아키텍처
인간 가독성 개발자가 이해 가능 JSON 기본 형식, 명확한 필드명
효율성 네트워크 대역폭 절약 선택적 바이너리 인코딩 (Protobuf, CBOR)

4.3 주소 형식 표준 (WIA Address Format)

현재 암호화폐 주소는 체인마다 다릅니다: 비트코인은 Base58, 이더리움은 0x 16진수, 솔라나는 Base58 등. 이는 사용자 혼란과 실수를 야기합니다. WIA-FIN-003는 통일된 주소 형식을 제안합니다.

4.3.1 WIA 주소 형식

형식: wia:{chain}:{address}:{checksum}

구성 요소:
• wia - 프로토콜 식별자 (고정)
• chain - 블록체인 식별자 (예: btc, eth, sol)
• address - 원본 주소 (체인 고유 형식 유지)
• checksum - CRC32 체크섬 (선택적, 오타 감지)

예시:
비트코인: wia:btc:1A1zP1eP5QGefi2DMPTfTL5SLmv7DivfNa:c8f2
이더리움: wia:eth:0x742d35Cc6634C0532925a3b844Bc9e7595f0bEb:a3d9
솔라나: wia:sol:DYw8jCTfwHNRJhhmFcbXvVDTqWMEVFBX6ZKUmG5CNSKK:7e1b

4.3.2 체인 식별자 레지스트리

Chain ID 블록체인 네트워크 주소 형식
btc Bitcoin Mainnet Base58Check (P2PKH/P2SH), Bech32 (SegWit)
btc-testnet Bitcoin Testnet Base58Check (P2PKH/P2SH), Bech32 (SegWit)
eth Ethereum Mainnet 0x + 40자 16진수
eth-sepolia Ethereum Sepolia Testnet 0x + 40자 16진수
sol Solana Mainnet Base58 (32-44자)
polygon Polygon Mainnet 0x + 40자 16진수
bnb BNB Chain Mainnet 0x + 40자 16진수
avax Avalanche C-Chain 0x + 40자 16진수

4.3.3 주소 검증 알고리즘

// TypeScript 예제: WIA 주소 검증
import CRC32 from 'crc-32';

interface WIAAddress {
  protocol: string;   // "wia"
  chain: string;      // "btc", "eth", etc.
  address: string;    // 원본 주소
  checksum?: string;  // CRC32 체크섬
}

function parseWIAAddress(wiaAddr: string): WIAAddress {
  const parts = wiaAddr.split(':');

  if (parts.length < 3) {
    throw new Error('Invalid WIA address format');
  }

  if (parts[0] !== 'wia') {
    throw new Error('Not a WIA address');
  }

  return {
    protocol: parts[0],
    chain: parts[1],
    address: parts[2],
    checksum: parts[3] || undefined
  };
}

function validateWIAAddress(wiaAddr: string): boolean {
  try {
    const parsed = parseWIAAddress(wiaAddr);

    // 1. 프로토콜 검증
    if (parsed.protocol !== 'wia') return false;

    // 2. 체인 검증
    const validChains = ['btc', 'eth', 'sol', 'polygon', ...];
    if (!validChains.includes(parsed.chain)) return false;

    // 3. 주소 형식 검증 (체인별)
    if (!validateNativeAddress(parsed.chain, parsed.address)) {
      return false;
    }

    // 4. 체크섬 검증 (있는 경우)
    if (parsed.checksum) {
      const calculatedChecksum = calculateChecksum(
        parsed.chain + ':' + parsed.address
      );
      if (calculatedChecksum !== parsed.checksum) {
        return false;
      }
    }

    return true;
  } catch (error) {
    return false;
  }
}

function calculateChecksum(data: string): string {
  const crc = CRC32.str(data);
  // CRC32를 16진수 4자리로 변환
  return (crc >>> 0).toString(16).slice(-4);
}

function validateNativeAddress(chain: string, address: string): boolean {
  switch (chain) {
    case 'btc':
      // 비트코인 주소 검증 (Base58Check, Bech32)
      return /^[13][a-km-zA-HJ-NP-Z1-9]{25,34}$/.test(address) ||
             /^bc1[a-z0-9]{39,59}$/.test(address);

    case 'eth':
    case 'polygon':
    case 'bnb':
      // EVM 호환 체인 주소 검증
      return /^0x[a-fA-F0-9]{40}$/.test(address);

    case 'sol':
      // 솔라나 주소 검증 (Base58, 32-44자)
      return /^[1-9A-HJ-NP-Za-km-z]{32,44}$/.test(address);

    default:
      return false;
  }
}

// 사용 예제
const addresses = [
  "wia:btc:1A1zP1eP5QGefi2DMPTfTL5SLmv7DivfNa:c8f2",
  "wia:eth:0x742d35Cc6634C0532925a3b844Bc9e7595f0bEb:a3d9",
  "wia:sol:DYw8jCTfwHNRJhhmFcbXvVDTqWMEVFBX6ZKUmG5CNSKK:7e1b"
];

addresses.forEach(addr => {
  console.log(`${addr}: ${validateWIAAddress(addr) ? '✓ Valid' : '✗ Invalid'}`);
});

4.5 인코딩 및 직렬화

JSON은 인간이 읽기 쉽지만, 네트워크 전송에는 비효율적입니다. WIA-FIN-003는 상황에 따라 다양한 인코딩을 지원합니다.

4.5.1 지원 인코딩 형식

형식 용도 크기 속도 가독성
JSON 개발, 디버깅, API 응답 기준 (100%) 보통 높음
Protobuf 노드 간 통신, 저장 ~30% 빠름 낮음
CBOR IoT, 모바일 앱 ~40% 빠름 낮음
MessagePack 범용 바이너리 ~50% 빠름 낮음
Avro 빅데이터, 스트리밍 ~35% 매우 빠름 낮음

4.5.3 크기 비교 예제

동일한 트랜잭션의 인코딩 크기 비교:

JSON (pretty-printed):
- 크기: 2,456 bytes
- 장점: 인간이 읽을 수 있음, 디버깅 쉬움
- 단점: 크기가 큼, 파싱 느림

JSON (minified):
- 크기: 1,823 bytes
- 장점: JSON 대비 25% 감소
- 단점: 여전히 큼

Protobuf:
- 크기: 547 bytes (JSON 대비 70% 감소!)
- 장점: 매우 작음, 빠른 직렬화/역직렬화
- 단점: 인간이 읽을 수 없음, 스키마 필요

CBOR:
- 크기: 729 bytes (JSON 대비 60% 감소)
- 장점: 작음, 자체 설명적
- 단점: Protobuf보다 큼

MessagePack:
- 크기: 912 bytes (JSON 대비 50% 감소)
- 장점: JSON과 호환, 구현 간단
- 단점: Protobuf/CBOR보다 큼

결론:
- 개발/디버깅 → JSON
- 프로덕션 P2P 통신 → Protobuf
- 모바일/IoT → CBOR
- 범용 → MessagePack

4.6 실전 구현: TypeScript SDK

이론을 실제 코드로 구현해봅시다. WIA-FIN-003 트랜잭션을 생성하고 검증하는 TypeScript SDK 예제입니다.

4.6.1 트랜잭션 빌더

// wia-transaction-builder.ts
import { createHash } from 'crypto';
import * as secp256k1 from 'secp256k1';

export class WIATransactionBuilder {
  private tx: Partial;

  constructor() {
    this.tx = {
      wia_version: "1.0.0",
      schema_version: "1.0",
      timestamp: new Date().toISOString(),
    };
  }

  setType(type: TransactionType): this {
    this.tx.tx_type = type;
    return this;
  }

  setSender(address: string, privateKey: Buffer): this {
    this.tx.sender = {
      address,
      address_type: this.detectAddressType(address),
      public_key: this.derivePublicKey(privateKey),
      signature: null  // 나중에 서명
    };
    return this;
  }

  setReceiver(address: string, amount: string, asset: Asset): this {
    this.tx.receiver = {
      address,
      address_type: this.detectAddressType(address),
      amount,
      asset
    };
    return this;
  }

  setFee(fee: Fee): this {
    this.tx.fee = fee;
    return this;
  }

  setMetadata(metadata: Metadata): this {
    this.tx.metadata = metadata;
    return this;
  }

  async build(privateKey: Buffer): Promise {
    // 1. Nonce 생성
    this.tx.nonce = Date.now();

    // 2. Transaction ID 생성
    this.tx.tx_id = this.generateTxId();

    // 3. 서명 생성
    const signature = await this.sign(privateKey);
    this.tx.sender!.signature = signature;

    // 4. 검증
    this.validate();

    return this.tx as WIATransaction;
  }

  private generateTxId(): string {
    const data = JSON.stringify({
      sender: this.tx.sender?.address,
      receiver: this.tx.receiver?.address,
      amount: this.tx.receiver?.amount,
      nonce: this.tx.nonce,
      timestamp: this.tx.timestamp
    });

    const hash = createHash('sha256').update(data).digest('hex');
    return `wia:tx:0x${hash}`;
  }

  private async sign(privateKey: Buffer): Promise {
    // 서명할 데이터 준비
    const message = this.getSigningMessage();
    const messageHash = createHash('sha256').update(message).digest();

    // ECDSA 서명
    const { signature, recid } = secp256k1.ecdsaSign(
      messageHash,
      privateKey
    );

    // DER 인코딩
    const derSignature = secp256k1.signatureExport(signature);

    return {
      algorithm: "ECDSA_secp256k1",
      value: "0x" + Buffer.from(derSignature).toString('hex'),
      signed_at: new Date().toISOString()
    };
  }

  private getSigningMessage(): string {
    return JSON.stringify({
      wia_version: this.tx.wia_version,
      tx_type: this.tx.tx_type,
      timestamp: this.tx.timestamp,
      nonce: this.tx.nonce,
      sender: this.tx.sender?.address,
      receiver: this.tx.receiver?.address,
      amount: this.tx.receiver?.amount,
      asset: this.tx.receiver?.asset,
      fee: this.tx.fee
    });
  }

  private validate(): void {
    if (!this.tx.sender) throw new Error("Sender required");
    if (!this.tx.receiver) throw new Error("Receiver required");
    if (!this.tx.fee) throw new Error("Fee required");
    if (!this.tx.tx_type) throw new Error("Transaction type required");

    // 주소 검증
    if (!validateWIAAddress(this.tx.sender.address)) {
      throw new Error("Invalid sender address");
    }
    if (!validateWIAAddress(this.tx.receiver.address)) {
      throw new Error("Invalid receiver address");
    }

    // 금액 검증
    const amount = parseFloat(this.tx.receiver.amount);
    if (isNaN(amount) || amount <= 0) {
      throw new Error("Invalid amount");
    }
  }

  private detectAddressType(address: string): string {
    const parsed = parseWIAAddress(address);

    switch (parsed.chain) {
      case 'btc':
        if (address.includes('1')) return 'p2pkh';
        if (address.includes('3')) return 'p2sh';
        if (address.includes('bc1')) return 'p2wpkh';
        return 'unknown';

      case 'eth':
      case 'polygon':
      case 'bnb':
        return 'externally_owned_account';  // 또는 스마트 컨트랙트 확인

      default:
        return 'unknown';
    }
  }

  private derivePublicKey(privateKey: Buffer): string {
    const publicKey = secp256k1.publicKeyCreate(privateKey);
    return "0x" + Buffer.from(publicKey).toString('hex');
  }
}

// 사용 예제
async function example() {
  const privateKey = Buffer.from('your-private-key-hex', 'hex');

  const tx = await new WIATransactionBuilder()
    .setType('transfer')
    .setSender('wia:eth:0x742d35Cc...', privateKey)
    .setReceiver(
      'wia:eth:0x8B3192f3...',
      '100.0',
      {
        type: 'ERC20',
        symbol: 'USDC',
        contract: 'wia:eth:0xA0b86991c...',
        decimals: 6,
        chain: 'ethereum-mainnet'
      }
    )
    .setFee({
      amount: '0.002',
      currency: 'ETH',
      gas_limit: 65000,
      gas_price: '30',
      payer: 'sender'
    })
    .setMetadata({
      memo: 'Payment for services'
    })
    .build(privateKey);

  console.log('Created transaction:', tx);

  // JSON으로 출력
  console.log(JSON.stringify(tx, null, 2));

  // Protobuf로 직렬화 (선택적)

📌 챕터 요약

Layer 1 - 데이터 형식의 핵심:

  1. Universal Transaction Schema (UTS): 모든 블록체인 거래를 표현할 수 있는 통일된 JSON 스키마로, 상호운용성의 기반을 제공합니다.
  2. WIA 주소 형식: wia:{chain}:{address}:{checksum} 형식으로 모든 블록체인 주소를 통일하여 사용자 혼란을 줄이고 오류를 방지합니다.
  3. 자산 메타데이터 표준: 토큰, NFT, 스테이블코인의 메타데이터를 표준화하여 일관된 정보 제공과 검증을 가능하게 합니다.
  4. 다중 인코딩 지원: JSON (개발), Protobuf (프로덕션), CBOR (IoT) 등 상황에 맞는 최적 인코딩을 선택할 수 있습니다.
  5. 실용적 SDK: TypeScript로 구현된 트랜잭션 빌더로 개발자가 쉽게 WIA-FIN-003 트랜잭션을 생성하고 검증할 수 있습니다.
🎯 실전 적용: Layer 1 데이터 형식은 WIA-FIN-003의 가장 기본적인 계층입니다. 이 표준을 채택하면 여러분의 블록체인 프로젝트는 즉시 다른 모든 WIA 호환 시스템과 통신할 수 있습니다. 마치 웹사이트가 HTTP를 따르면 모든 브라우저에서 접근 가능한 것과 같습니다.
