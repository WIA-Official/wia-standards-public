Chapter 2: Risk Assessment & Hazard Analysis

2.1 Introduction to Risk Assessment

Risk assessment is the systematic process of identifying hazards, analyzing risks, and evaluating whether safety measures are adequate. For physical AI systems, risk assessment must account for the unique challenges posed by autonomous decision-making, learning systems, and complex human-robot interactions.

The WIA-AI-SAFETY-PHYSICAL standard requires comprehensive risk assessment as the foundation of all safety activities. This assessment must be performed iteratively throughout the system lifecycle and updated whenever changes occur.

Risk Equation:
Risk = Severity x Probability x Exposure

Where:
- Severity: Potential harm magnitude (minor injury to fatality)
- Probability: Likelihood of hazardous event occurring
- Exposure: Frequency and duration of human presence in hazard zone

2.2 Hazard Identification Methods

Systematic hazard identification ensures no significant risks are overlooked:

Structured What-If Technique (SWIFT)

A systematic brainstorming approach using structured questions:

Question Category Example Questions Typical Hazards Found
System Failure What if the sensor fails? What if communication is lost? Uncontrolled motion, collision
Human Error What if operator gives wrong command? Unintended operation, safety bypass
Environmental What if lighting conditions change? Perception failure, navigation error
AI Behavior What if AI misidentifies an object? Wrong action, unexpected motion
Integration What if timing is misaligned? Race conditions, coordination failure

Hazard and Operability Study (HAZOP)

HAZOP uses guide words to systematically explore deviations from design intent:

// HAZOP Analysis Structure
interface HAZOPAnalysis {
    node: string;           // System component being analyzed
    parameter: string;      // Design parameter (speed, force, position)
    guideWord: GuideWord;   // NO, MORE, LESS, REVERSE, OTHER, etc.
    deviation: string;      // Resulting deviation from intent
    causes: string[];       // Potential causes of deviation
    consequences: string[]; // Potential consequences
    safeguards: string[];   // Existing protective measures
    recommendations: string[]; // Additional measures needed
}

enum GuideWord {
    NO = "Complete negation of design intent",
    MORE = "Quantitative increase",
    LESS = "Quantitative decrease",
    REVERSE = "Opposite direction/logic",
    PART_OF = "Incomplete fulfillment",
    AS_WELL_AS = "Additional unintended activity",
    OTHER = "Complete substitution"
}

// Example HAZOP Entry
const exampleEntry: HAZOPAnalysis = {
    node: "Robot Arm Joint 3",
    parameter: "Angular Velocity",
    guideWord: GuideWord.MORE,
    deviation: "Joint moves faster than commanded",
    causes: [
        "Motor controller fault",
        "Encoder failure reporting lower speed",
        "Software calculation error"
    ],
    consequences: [
        "Increased impact force on collision",
        "Loss of positional accuracy",
        "Mechanical stress on joint"
    ],
    safeguards: [
        "Hardware speed governor",
        "Secondary encoder",
        "Watchdog timer"
    ],
    recommendations: [
        "Add independent velocity monitoring",
        "Implement velocity envelope checking"
    ]
};

2.3 Failure Mode and Effects Analysis (FMEA)

FMEA systematically analyzes potential failure modes and their effects on system safety:

Risk Priority Number (RPN) = Severity x Occurrence x Detection
  • Each factor rated 1-10
  • RPN range: 1-1000
  • High RPN items require immediate action
  • Typical threshold: RPN > 100 requires mitigation

2.4 Fault Tree Analysis (FTA)

FTA works backwards from an undesired event (top event) to identify all possible causes:

// Fault Tree Example: Human Injury from Robot
/*
                    [Human Injury]
                          |
                        (OR)
              ____________|____________
             |            |            |
      [Collision]   [Crushing]   [Entanglement]
           |
         (AND)
      ____|____
     |         |
[Robot in  [Human in
 motion]   workspace]
     |         |
   (OR)      (OR)
    |       __|__
    |      |     |
[Normal][Unauth][Sensor
 op]    entry]  fail]

*/

interface FaultTreeNode {
    id: string;
    type: 'EVENT' | 'GATE' | 'BASIC';
    gateType?: 'AND' | 'OR' | 'NOT' | 'XOR';
    description: string;
    probability?: number;
    children?: FaultTreeNode[];
}

function calculateTopEventProbability(node: FaultTreeNode): number {
    if (node.type === 'BASIC') {
        return node.probability || 0;
    }

    const childProbs = node.children?.map(calculateTopEventProbability) || [];

    switch (node.gateType) {
        case 'OR':
            // P(A OR B) = P(A) + P(B) - P(A)P(B)
            return childProbs.reduce((acc, p) => acc + p - acc * p, 0);
        case 'AND':
            // P(A AND B) = P(A) * P(B)
            return childProbs.reduce((acc, p) => acc * p, 1);
        default:
            return 0;
    }
}

// Example Calculation
const collisionTree: FaultTreeNode = {
    id: 'TOP',
    type: 'GATE',
    gateType: 'AND',
    description: 'Collision with human',
    children: [
        {
            id: 'MOTION',
            type: 'BASIC',
            description: 'Robot in motion',
            probability: 0.8 // Robot operates 80% of time
        },
        {
            id: 'HUMAN',
            type: 'GATE',
            gateType: 'OR',
            description: 'Human in collision zone',
            children: [
                { id: 'AUTH', type: 'BASIC', description: 'Authorized entry', probability: 0.1 },
                { id: 'UNAUTH', type: 'BASIC', description: 'Unauthorized entry', probability: 0.01 },
                { id: 'SENSOR', type: 'BASIC', description: 'Sensor failure', probability: 0.001 }
            ]
        }
    ]
};

// Collision probability = 0.8 * (0.1 + 0.01 + 0.001 - ...) ≈ 0.088

2.6 AI-Specific Risk Considerations

Physical AI systems introduce unique risk factors that must be specifically addressed:

Machine Learning Risk Factors

Risk Factor Description Mitigation Strategy
Distribution Shift Real-world differs from training data Out-of-distribution detection, conservative fallback
Adversarial Inputs Deliberately crafted deceptive inputs Input validation, anomaly detection
Concept Drift Environment changes over time Continuous monitoring, periodic retraining
Uncertainty Blindness Model doesn't know what it doesn't know Uncertainty quantification, confidence thresholds
Reward Hacking Finding loopholes in objective function Multiple safety constraints, human oversight 
// AI Risk Assessment Framework
class AIRiskAssessor {
    assessModelRisk(model: AIModel, context: DeploymentContext): RiskProfile {
        const risks: Risk[] = [];

        // Distribution shift risk
        const distributionRisk = this.assessDistributionShift(
            model.trainingData,
            context.expectedInputs
        );
        risks.push(distributionRisk);

        // Adversarial robustness
        const adversarialRisk = this.assessAdversarialRobustness(model);
        risks.push(adversarialRisk);

        // Uncertainty calibration
        const uncertaintyRisk = this.assessUncertaintyCalibration(model);
        risks.push(uncertaintyRisk);

        // Failure mode analysis
        const failureModes = this.identifyAIFailureModes(model);
        risks.push(...failureModes);

        return this.aggregateRisks(risks);
    }

    assessDistributionShift(training: DataProfile, deployment: DataProfile): Risk {
        // Statistical comparison of training vs deployment data
        const divergence = this.calculateKLDivergence(training, deployment);

        if (divergence > 0.5) {
            return {
                type: 'DISTRIBUTION_SHIFT',
                severity: 'HIGH',
                description: 'Significant mismatch between training and deployment',
                mitigation: 'Retrain model on deployment-representative data'
            };
        }
        // ... additional analysis
    }

    assessConfidenceThreshold(model: AIModel): number {
        // Determine safe confidence threshold for decisions
        // Below this threshold, system should request human assistance
        const calibrationError = this.measureCalibrationError(model);

        // Higher calibration error = higher confidence threshold needed
        return 0.5 + (calibrationError * 0.4); // Range: 0.5 - 0.9
    }
}

2.7 Quantitative Risk Assessment

Quantitative methods provide numerical risk estimates essential for safety-critical decisions:

Probabilistic Risk Assessment (PRA)

// Probabilistic Risk Assessment
interface PRAResult {
    annualizedRisk: number;           // Expected incidents per year
    fatalityRisk: number;             // Probability of fatality per year
    expectedCost: number;             // Annual expected loss in currency
    confidenceInterval: [number, number]; // 90% confidence bounds
}

function calculateAnnualizedRisk(
    hazardRate: number,      // Hazards per operating hour
    operatingHours: number,  // Hours per year
    exposureFactor: number,  // Fraction of time humans exposed
    injuryProbability: number // P(injury | hazard occurs)
): number {
    return hazardRate * operatingHours * exposureFactor * injuryProbability;
}

// Example: Collaborative Robot Application
const cobot = {
    hazardRate: 1e-6,        // One hazard per million hours
    operatingHours: 4000,     // 4000 hours/year operation
    exposureFactor: 0.5,      // Human present 50% of time
    injuryProbability: 0.01   // 1% chance of injury if hazard occurs
};

const annualRisk = calculateAnnualizedRisk(
    cobot.hazardRate,
    cobot.operatingHours,
    cobot.exposureFactor,
    cobot.injuryProbability
);

// Result: 2e-8 or about 1 injury per 50 million years
// This would be considered acceptable for most applications

2.8 Risk Mitigation Strategies

Once risks are identified and assessed, appropriate mitigation strategies must be selected:

Strategy Description Effectiveness Example
Elimination Remove the hazard entirely 100% Remove sharp edges from design
Substitution Replace with less hazardous option 90%+ Use pneumatic instead of hydraulic
Engineering Controls Physical safeguards 70-90% Light curtains, safety interlocks
Administrative Controls Procedures and training 50-70% Operating procedures, training
PPE Personal protection 30-50% Safety glasses, gloves

2.10 Continuous Risk Monitoring

Risk assessment is not a one-time activity but requires continuous monitoring throughout operation:

// Real-Time Risk Monitoring System
class RuntimeRiskMonitor {
    private riskIndicators: Map<string, RiskIndicator> = new Map();
    private alertThresholds: AlertThresholds;

    registerIndicator(indicator: RiskIndicator): void {
        this.riskIndicators.set(indicator.id, indicator);
    }

    async monitorContinuously(): Promise<void> {
        while (true) {
            const currentRisk = await this.calculateCurrentRisk();

            if (currentRisk.level > this.alertThresholds.warning) {
                this.raiseWarning(currentRisk);
            }

            if (currentRisk.level > this.alertThresholds.critical) {
                this.initiateRiskMitigation(currentRisk);
            }

            this.logRiskMetrics(currentRisk);
            await this.sleep(100); // 10Hz monitoring
        }
    }

    calculateCurrentRisk(): CurrentRiskProfile {
        let aggregateRisk = 0;
        const contributingFactors: RiskFactor[] = [];

        for (const [id, indicator] of this.riskIndicators) {
            const value = indicator.getCurrentValue();
            const contribution = indicator.weight * value;
            aggregateRisk += contribution;

            if (contribution > 0.1) {
                contributingFactors.push({
                    indicator: id,
                    value,
                    contribution
                });
            }
        }

        return {
            level: aggregateRisk,
            timestamp: Date.now(),
            factors: contributingFactors,
            trend: this.calculateTrend()
        };
    }
}

Key Takeaways

  • Risk assessment must be systematic, comprehensive, and continuously updated
  • Multiple hazard identification methods (SWIFT, HAZOP, FMEA, FTA) provide complementary perspectives
  • Risk = Severity x Probability x Exposure provides the fundamental framework
  • AI systems introduce unique risks including distribution shift, adversarial inputs, and uncertainty blindness
  • Quantitative risk assessment enables objective safety decisions
  • Risk mitigation follows a hierarchy: elimination, substitution, engineering controls, administrative controls, PPE
  • Comprehensive documentation is essential for compliance and continuous improvement
  • Runtime risk monitoring enables dynamic safety management
