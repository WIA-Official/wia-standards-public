Cross-Chapter Anchors

This chapter sits inside the WIA Screen Reader 8-chapter sequence. Each chapter is intended to be readable standalone but gains depth when read in order. The standard's 4-Phase specification is cross-referenced throughout:

  • Phase 1 (Data Format) — pronunciation hint, accessibility tree node, reading order plan, braille mapping, user profile, telemetry frame envelopes.
  • Phase 2 (API Interface) — local IPC for NVDA / VoiceOver / Orca, browser-extension message-passing, cloud-relay HTTPS endpoints.
  • Phase 3 (Federation Protocol) — pronunciation hint sharing across vendors, profile portability across devices, telemetry consent enforcement, curator trust scoring.
  • Phase 4 (Integration) — NVDA addon, VoiceOver daemon, TalkBack module, Orca plugin, Chrome / Firefox / Edge browser extensions, EPUB 3 Accessibility integration, WIA family composition (WIHP for Korean phonetics, WIA Braille for the 8-dot extension, WIA-OMNI-API for credentials, WIA-ACCESSIBILITY for accommodations).

Conformance and Reference Implementation

A black-box conformance test suite is open-sourced at https://github.com/WIA-Official/wia-screen-reader-conformance. The reference implementation ships as separate addons / daemons / extensions per platform, all published from the same monorepo at WIA-Official/wia-screen-reader-impl. Implementers can either consume the reference implementations directly or build vendor-specific implementations that pass the conformance suite.

Why This Standard Matters

Screen-reader accessibility has historically been a per-vendor concern. NVDA users build pronunciation lexicons that VoiceOver users cannot consume. Korean braille readers use a Korean-specific 8-dot extension that English Grade 2 readers cannot understand. WIA Screen Reader exists to make these resources portable: a hint curated by a Korean medical-terminology educator works for an Android TalkBack reader on a Spanish keyboard, because the underlying envelope shapes are uniform.

The 211-language support (via WIHP) is the standard's headline feature, but the more important architectural choice is the user profile portability described in Phase 3 §5: a user can move between work NVDA, home VoiceOver, and mobile TalkBack without losing their voice rate, braille grade, or hint subscriptions.

Implementer Notes

Three operational concerns recur across screen-reader integrations and are baked into the standard:

  1. Latency budget for live announcements. Live regions (chat messages, form errors, search results) need to reach the user under the screen-reader's perceived latency threshold. The standard recommends an end-to-end budget of 250 ms from page event to spoken output for polite live regions, 100 ms for assertive, and 500 ms for the WIA-Screen-Reader-extension gentle tier.
  2. Memory footprint of pronunciation hint cache. A Korean medical reader subscribed to two curators may cache 50,000+ hints. The standard recommends an LRU cache with a 100 MB ceiling and per-language eviction priorities tied to the user's preferred language.
  3. Braille display I/O. Refreshable braille displays often connect via USB or Bluetooth with vendor-proprietary protocols. The standard does not define the wire protocol to the display itself; it defines only the JSON envelope that the host emits. Driver code remains per-vendor.

Accessibility-First Design

Every choice in this standard is filtered through the question: "does this make the experience better for the visually impaired user, or just easier for the implementer?" When the two conflict, the user wins. That is why the standard mandates explicit consent for telemetry, refuses to log raw page URLs in any host-side data, and requires implementations to surface name-source provenance so the user can debug why a button reads as a particular phrase.
