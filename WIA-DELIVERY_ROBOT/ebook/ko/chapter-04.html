📖 Chapter 04

통신 프로토콜 (Phase 3)

1. 통신 프로토콜 개요

WIA-DELIVERY_ROBOT Phase 3는 배송 로봇 시스템의 안정적이고 효율적인 통신을 보장하기 위한 포괄적인 프로토콜을 정의합니다. 이 프로토콜은 데이터 포맷(Phase 1)과 API 인터페이스(Phase 2)를 기반으로, 실제 운영 환경에서 발생할 수 있는 다양한 통신 시나리오를 다룹니다.

1.1 프로토콜의 목적

  • 신뢰성: 네트워크 장애 시에도 메시지 전달 보장
  • 일관성: 모든 시스템이 동일한 상태를 유지
  • 효율성: 대역폭과 처리 능력의 최적 활용
  • 보안성: 메시지의 무결성과 기밀성 보호
  • 확장성: 증가하는 로봇 수와 트래픽에 대응

1.2 프로토콜 계층 구조

┌─────────────────────────────────────────┐ │ 응용 계층 (Application Layer) │ │ - 로봇 제어 명령 │ │ - 배송 작업 관리 │ │ - 상태 보고 │ ├─────────────────────────────────────────┤ │ 프레젠테이션 계층 │ │ - JSON/Protocol Buffers │ │ - 데이터 압축/암호화 │ ├─────────────────────────────────────────┤ │ 세션 계층 (Session Layer) │ │ - 연결 관리 │ │ - 인증 및 권한 │ ├─────────────────────────────────────────┤ │ 전송 계층 (Transport Layer) │ │ - TCP/UDP │ │ - WebSocket │ │ - MQTT/AMQP │ ├─────────────────────────────────────────┤ │ 네트워크 계층 (Network Layer) │ │ - IPv4/IPv6 │ │ - 라우팅 │ └─────────────────────────────────────────┘

2. 메시지 구조 및 포맷

2.1 메시지 프레임 구조

모든 WIA-DELIVERY_ROBOT 메시지는 다음과 같은 3계층 구조를 따릅니다:

+------------------+------------------+------------------+ | Header (64B) | Payload (var) | Signature (64B) | +------------------+------------------+------------------+ | | | ← 최소 128B → | ← 최대 16MB → | ← 64B 고정 → | +------------------+------------------+------------------+

2.2 헤더 필드 상세

{ "version": "1.0", // 프로토콜 버전 "messageType": "request", // 메시지 타입 "messageId": "msg-550e8400...", // 고유 메시지 ID (UUID) "correlationId": "corr-123abc...", // 요청-응답 연결 ID "source": "ROBOT-SEOUL-001", // 발신자 ID "destination": "CONTROL-CENTER-01", // 수신자 ID "timestamp": "2025-01-15T14:30:00Z", // 생성 시각 "priority": "normal", // 우선순위 (low/normal/high/critical) "ttl": 300, // 유효 시간 (초) "compression": "gzip", // 압축 방식 "encryption": "aes256" // 암호화 방식 }

2.3 메시지 타입 분류

타입 코드 설명 응답 필요
REQUEST 0x01 작업 요청 메시지 ✓
RESPONSE 0x02 요청에 대한 응답 -
BROADCAST 0x03 전체 시스템 공지 -
ALERT 0x04 긴급 알림 ✓
SYNC 0x05 상태 동기화 ✓
HEARTBEAT 0x06 연결 유지 신호 -
ACK 0x07 수신 확인 -
NACK 0x08 수신 거부 -

3. 운영 프로토콜

3.1 초기화 프로토콜 (Initialization Protocol)

로봇이 시스템에 연결될 때 수행되는 5단계 프로세스:

1. CONNECT: TCP/TLS 연결 수립 Client → Server: SYN Server → Client: SYN-ACK Client → Server: ACK 2. AUTHENTICATE: 인증 정보 교환 Client → Server: { "messageType": "request", "payload": { "action": "authenticate", "credentials": { "robotId": "ROBOT-SEOUL-001", "apiKey": "wia_live_sk_...", "timestamp": "2025-01-15T14:30:00Z" } } } 3. CONFIGURE: 시스템 설정 교환 4. SYNC: 상태 동기화 5. READY: 운영 준비 완료

3.2 데이터 교환 프로토콜

일반적인 데이터 교환은 5단계 요청-응답 사이클을 따릅니다:

  1. REQUEST: 클라이언트가 요청 전송
  2. VALIDATE: 서버가 요청 검증
  3. PROCESS: 서버가 작업 처리
  4. RESPOND: 서버가 응답 전송
  5. CONFIRM: 클라이언트가 수신 확인

3.3 종료 프로토콜 (Shutdown Protocol)

1. NOTIFY: 종료 의도 알림 2. FLUSH: 대기 중인 작업 완료 3. SYNC: 최종 상태 동기화 4. DISCONNECT: 연결 종료 5. CLEANUP: 리소스 해제

4. 안전 프로토콜

4.1 오류 처리 레벨

레벨 심각도 조치 예시
Level 1 Warning 로그 기록, 계속 운영 네트워크 지연, 임시 센서 오류
Level 2 Caution 알림 전송, 제한 운영 배터리 부족, GPS 불안정
Level 3 Alert 새 작업 중단, 복구 시작 통신 두절, 중요 센서 고장
Level 4 Critical 긴급 정지, 즉시 조사 충돌 위험, 시스템 침해

4.2 복구 프로토콜 (Recovery Protocol)

복구 절차: 1. ISOLATE: 영향받은 시스템 격리 2. ASSESS: 피해 범위 평가 3. RESTORE: 백업에서 복원 4. VERIFY: 복원 검증 5. RESUME: 운영 재개 6. DOCUMENT: 사고 기록

⚠️ 복구 프로토콜 주의사항

  • 자동 복구 한계: Level 3 이상은 인간 개입 필수
  • 백업 주기: 중요 데이터는 최소 15분마다 백업
  • 복구 우선순위: 안전 관련 시스템 > 배송 작업 > 통계 데이터
  • 문서화: 모든 복구 과정은 실시간으로 기록

5. 동기화 프로토콜

5.1 상태 동기화

여러 시스템 간의 상태 일관성을 유지하기 위한 프로토콜:

{ "protocol": "WIA_SYNC_V1", "syncType": "full", "participants": ["ROBOT-SEOUL-001", "CONTROL-CENTER-01"], "method": "consensus", "timestamp": "2025-01-15T14:30:00Z", "checkpoints": [...] }

5.2 시간 동기화 프로토콜

배송 로봇 시스템의 모든 구성 요소가 정확한 시간을 유지하는 것은 매우 중요합니다:

  1. 타임스탬프 교환
  2. 왕복 시간(RTT) 계산
  3. 네트워크 지연 보정
  4. 기준 시간 검증
  5. 시간 조정

5.3 데이터 동기화 전략

전략 설명 장점 단점
Full Sync 전체 데이터 동기화 완전한 일관성 대역폭 소모 큼
Incremental Sync 변경 사항만 동기화 효율적 복잡한 구현
Event-based Sync 이벤트 발생 시 동기화 실시간성 순서 보장 필요
Periodic Sync 주기적 동기화 예측 가능 지연 발생 가능

6. 보안 프로토콜

6.1 암호화 요구사항

🔒 필수 암호화 사양

  • 전송 계층: TLS 1.3 (최소 TLS 1.2)
  • 저장 데이터: AES-256-GCM
  • 키 교환: ECDHE (Elliptic Curve Diffie-Hellman Ephemeral)
  • 디지털 서명: Ed25519 (또는 ECDSA P-256)
  • 해시 함수: SHA-256 (최소 SHA-256)

6.2 인증서 계층 구조

Root CA (WIA Certificate Authority) │ ├─ Intermediate CA (Robot Manufacturers) │ └─ Service Certificate (Delivery Robots) │ └─ Intermediate CA (Service Providers) └─ Server Certificate (Control Centers)

  • 운영 로그: 일반적인 시스템 작업 기록
  • 오류 로그: 오류 및 예외 상황 기록

8. 프로토콜 확장성

8.1 플러그인 아키텍처

WIA-DELIVERY_ROBOT 프로토콜은 새로운 기능을 플러그인 형태로 추가할 수 있도록 설계되었습니다. 플러그인 시스템을 통해 압축, 암호화, 메트릭 수집 등의 기능을 독립적으로 개발하고 통합할 수 있습니다.

8.2 버전 협상 메커니즘

클라이언트와 서버는 연결 초기화 시 지원하는 프로토콜 버전을 교환하고, 가장 높은 공통 버전을 자동으로 선택합니다. 이를 통해 신규 버전과 구버전 간의 원활한 상호 운용성을 보장합니다.

🎯 핵심 요점

  • 메시지 구조: Header-Payload-Signature 3계층 구조로 확장성과 보안성 확보
  • 8가지 메시지 타입: REQUEST, RESPONSE, BROADCAST, ALERT, SYNC, HEARTBEAT, ACK, NACK
  • 초기화 5단계: CONNECT → AUTHENTICATE → CONFIGURE → SYNC → READY
  • 4단계 오류 처리: Warning, Caution, Alert, Critical로 체계적 대응
  • 종료 5단계: NOTIFY → FLUSH → SYNC → DISCONNECT → CLEANUP
  • 보안 계층: TLS 1.3, AES-256-GCM, Ed25519 서명으로 다층 보안
  • 5가지 로그 타입: 운영, 오류, 보안, 성능, 감사 로그로 완벽한 추적

📝 복습 문제

  1. WIA-DELIVERY_ROBOT 메시지 프레임의 3계층 구조를 설명하고, 각 계층의 크기와 역할을 기술하시오.
  2. 8가지 메시지 타입(REQUEST, RESPONSE, BROADCAST, ALERT, SYNC, HEARTBEAT, ACK, NACK)의 용도와 응답 필요 여부를 비교하시오.
  3. 초기화 프로토콜의 5단계(CONNECT, AUTHENTICATE, CONFIGURE, SYNC, READY)를 순서대로 설명하시오.
  4. 4단계 오류 처리 레벨(Warning, Caution, Alert, Critical)의 차이점과 각 레벨에서 수행되는 조치를 설명하시오.
  5. 복구 프로토콜의 6단계(ISOLATE, ASSESS, RESTORE, VERIFY, RESUME, DOCUMENT)를 설명하고, 각 단계의 중요성을 기술하시오.
  6. NTP 기반 시간 동기화의 5단계 프로세스를 설명하고, 왜 정확한 시간 동기화가 중요한지 서술하시오.
  7. 필수 암호화 사양 5가지(전송 계층, 저장 데이터, 키 교환, 디지털 서명, 해시 함수)를 나열하고 설명하시오.
  8. 5가지 로그 카테고리(운영, 오류, 보안, 성능, 감사)의 보존 기간과 그 이유를 설명하시오.
