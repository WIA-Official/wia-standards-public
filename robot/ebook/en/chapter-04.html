Chapter 4: DDS Communication Protocols

4.1 DDS/RTPS Overview

ROS 2 uses the DDS (Data Distribution Service) standard (OMG DDS 1.4) and its wire protocol RTPS (Real-Time Publish-Subscribe, OMG RTPS 2.5) as the default middleware. DDS provides a data-centric publish/subscribe (DCPS) model with declarative QoS policies, enabling fine-grained control over delivery reliability, latency, and resource consumption — critical requirements for safety-critical robot systems.

WIA-ROBOT supports three RMW (ROS Middleware) implementations:

RMWDDS ImplementationUse Case
rmw_fastrtps_cppeProsima FastDDS 3.xDefault; best performance on Linux
rmw_cyclonedds_cppEclipse Cyclone DDS 0.10Low-latency, good for embedded
rmw_connextddsRTI Connext DDS 7.xEnterprise; DO-178C avionics cert

4.2 QoS Profiles

WIA-ROBOT defines three QoS profile tiers:

Tier 1 — Safety-Critical Commands

Used for: /emergency_stop, /wia_robot/execute_trajectory action goals, /robot_description.

qos_commands:
  reliability: RELIABLE           # guaranteed delivery, retransmit on loss
  durability: TRANSIENT_LOCAL     # late joiners receive last N messages
  history: KEEP_LAST
  depth: 10
  deadline: {nsec: 100000000}     # 100 ms -- ISO 10218-1 budget
  liveliness: AUTOMATIC
  liveliness_lease_duration: {nsec: 500000000}  # 500 ms watchdog

Tier 2 — High-Rate Telemetry

Used for: /joint_states (50 Hz), /imu/data_raw, /wrench (1 kHz).

qos_telemetry_fast:
  reliability: BEST_EFFORT        # drop stale frames; prioritise latency
  durability: VOLATILE            # no late-joiner buffering
  history: KEEP_LAST
  depth: 1
  lifespan: {nsec: 25000000}      # discard after 25 ms (2 frames at 50 Hz)

Tier 3 — Map/State

Used for: /map (SLAM occupancy grid), /tf_static, /robot_state_publisher outputs.

qos_map_static:
  reliability: RELIABLE
  durability: TRANSIENT_LOCAL
  history: KEEP_LAST
  depth: 1

4.3 ROS 2 Domain ID Isolation

ROS_DOMAIN_ID (0–232) partitions DDS communication. WIA-ROBOT deployment guidelines:

Domain IDEnvironment
0Development / simulation default
10–19Clinical / ward robots (one ID per ward)
20–29Industrial / factory floor robots
100–109Test bench / QA environments
200–209Isaac Sim virtual robots
export ROS_DOMAIN_ID=12   # Ward 2 exoskeleton fleet
ros2 run wia_robot_gateway gateway_node

4.4 FastDDS 3.x Configuration

FastDDS 3.x (eProsima) is the default RMW for WIA-ROBOT. Key XML configuration for WIA safety-critical profiles: 

<!-- FASTRTPS_DEFAULT_PROFILES_FILE=/opt/wia_robot/dds/wia_profiles.xml -->
<profiles>
  <participant profile_name="wia_safety" is_default_participant="true">
    <rtps>
      <builtin>
        <discovery_config>
          <leaseDuration><sec>DURATION_INFINITY</sec></leaseDuration>
          <initial_announcements>
            <count>5</count>
            <period><nanosec>100000000</nanosec></period>
          </initial_announcements>
        </discovery_config>
      </builtin>
      <sendBufferSize>1048576</sendBufferSize>
      <receiveBufferSize>4194304</receiveBufferSize>
    </rtps>
  </participant>
</profiles>

4.5 MQTT 5 Bridge for Clinical Systems

Hospital EMR and nurse call systems often cannot run a full ROS 2 stack. The WIA-ROBOT MQTT 5 bridge (wia_robot_mqtt_bridge) re-publishes critical events to an MQTT 5 broker, enabling integration with clinical dashboards over standard hospital network infrastructure. 

bridge_config:
  mqtt_broker: mqtts://hospital-iot.internal:8883
  client_id: wia_robot_bridge_ward2
  tls_cert: /etc/wia_robot/mqtt_client.crt
  tls_key: /etc/wia_robot/mqtt_client.key

  subscriptions:
    - ros2_topic: /emergency_stop
      mqtt_topic: wia/robot/{robot_id}/estop
      qos: 2        # MQTT QoS 2 = exactly-once (matches RELIABLE DDS)

    - ros2_topic: /joint_states
      mqtt_topic: wia/robot/{robot_id}/joints
      qos: 0        # MQTT QoS 0 = at-most-once (matches BEST_EFFORT DDS)
      rate_limit_hz: 5   # downsample from 50 Hz to 5 Hz for clinical dashboard

4.6 Troubleshooting DDS Discovery

# Verify topic discovery
ros2 topic list
ros2 topic info /joint_states --verbose
ros2 topic hz /joint_states        # expect ~50 Hz

# Check DDS participant count
ros2 multicast receive              # test multicast reachability
ros2 daemon status                  # is the ROS 2 daemon running?

# Force unicast if multicast blocked (hospital networks often block multicast)
export RMW_FASTRTPS_USE_QOS_FROM_XML=1
export FASTRTPS_DEFAULT_PROFILES_FILE=/opt/wia_robot/dds/unicast_profiles.xml
