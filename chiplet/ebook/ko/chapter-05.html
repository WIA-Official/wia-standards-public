챕터 5: 칩렛 보안

칩렛 아키텍처는 전통적인 모놀리식 칩 보안을 넘어서는 고유한 보안 과제를 도입합니다. 이 챕터에서는 하드웨어 보안, 공급망 무결성, 안티 위조 기술, 하드웨어 트로이 목마 탐지, 인증 메커니즘 및 다중 공급업체 칩렛 생태계를 위한 포괄적인 보안 프레임워크를 다룹니다.

칩렛 시스템의 보안 과제

칩렛 시스템의 모듈식 특성은 확장된 공격 표면을 만듭니다. 칩렛이 여러 공급업체에서 소싱되고, 다른 파운드리에서 제조되며, 제3자 패키징 하우스에서 조립될 수 있으므로 악의적인 간섭의 기회가 크게 증가합니다.

다중 공급업체 보안 문제

단일 칩렛 시스템에는 5개의 다른 공급업체의 구성 요소, 3개의 다른 파운드리에서 제조, 4번째 위치에서 조립, 또 다른 곳에서 테스트가 포함될 수 있습니다. 각 핸드오프는 포괄적인 보안 조치를 통해 해결해야 하는 잠재적 취약점을 나타냅니다.

공급망 보안

칩렛 공급망을 보호하려면 물리적 보안, 암호화 인증 및 포괄적인 추적 시스템을 결합하는 다층 접근 방식이 필요합니다.

칩렛 인증 프레임워크

// 칩렛 인증 시스템
interface ChipletAuthentication {
    // 물리적 복제 불가 기능 (PUF)
    puf: {
        type: "SRAM" | "RingOscillator" | "Arbiter" | "Butterfly";
        bitLength: number;        // 256-4096 비트
        reliability: number;      // % 안정 비트
        uniqueness: number;       // % 칩 간 변동

        // PUF 기반 키 생성
        generateKey(): Promise;
        authenticate(challenge: Challenge): Promise;
    };

    // 고유 식별자 (UID)
    uid: {
        format: "128-bit UUID";
        storage: "퓨즈" | "EEPROM" | "PUF 파생";
        cryptoBinding: boolean;
    };

    // 인증서 체인
    certificate: {
        type: "X.509v3";
        issuer: "WIA-CA" | "Vendor-CA";
        attributes: ChipletAttributes;
        validityPeriod: Date[];
        revocationCheck: "OCSP" | "CRL";
    };
}

// 인증 프로토콜
class ChipletAuthProtocol {
    async authenticateChiplet(chiplet: ChipletId): Promise {
        // 1단계: UID 읽기
        const uid = await this.readUID(chiplet);

        // 2단계: 인증서 체인 확인
        const cert = await this.getCertificate(uid);
        const chainValid = await this.verifyCertChain(cert);

        // 3단계: PUF 챌린지-응답
        const challenge = this.generateChallenge();
        const response = await this.sendChallenge(chiplet, challenge);
        const pufValid = await this.verifyPUFResponse(response, cert);

        // 4단계: 폐기 상태 확인
        const notRevoked = await this.checkRevocation(cert);

        return {
            authenticated: chainValid && pufValid && notRevoked,
            chipletId: uid,
            certificate: cert,
            trustLevel: this.calculateTrustLevel(cert)
        };
    }
}

하드웨어 트로이 목마 탐지

하드웨어 트로이 목마는 칩렛 보안에 대한 가장 심각한 위협 중 하나입니다. 탐지에는 설계 시간 기술, 제조 시간 테스트 및 런타임 모니터링의 조합이 필요합니다.

하드웨어 트로이 목마 특성

하드웨어 트로이 목마는 일반적으로 세 가지 구성 요소를 가집니다: 트로이 목마를 활성화하는 트리거 메커니즘, 악의적 기능을 실행하는 페이로드, 탐지를 회피하는 스텔스 기능입니다. 칩렛 시스템은 공급망의 어느 지점에서든 삽입될 수 있는 트로이 목마를 다루어야 합니다.

챕터 요약

칩렛 보안은 공급망 무결성, 하드웨어 트로이 목마 위협, 인증 및 보안 통신을 다루는 포괄적인 접근 방식이 필요합니다. WIA-CHIPLET 보안 프레임워크는 보안 요구사항과 실제 구현 고려사항의 균형을 맞추면서 다중 공급업체 칩렛 생태계에서 신뢰를 가능하게 하는 표준 및 인증 프로세스를 제공합니다.
