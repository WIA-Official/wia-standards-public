Chapter 5: Chiplet Security

Chiplet architecture introduces unique security challenges that extend beyond traditional monolithic chip security. This chapter addresses supply chain security, hardware trojan detection, authentication mechanisms, and comprehensive security frameworks for multi-vendor chiplet ecosystems.

Security Challenges in Chiplet Systems

The modular nature of chiplet systems creates an expanded attack surface. With chiplets potentially sourced from multiple vendors, manufactured at different foundries, and assembled by third-party packaging houses, the opportunities for malicious interference multiply significantly.

The Multi-Vendor Security Problem

A single chiplet system might include components from five different vendors, manufactured at three different foundries, assembled at a fourth location, and tested at yet another. Each handoff represents a potential vulnerability point that must be addressed through comprehensive security measures.

Supply Chain Security

Securing the chiplet supply chain requires a multi-layered approach combining physical security, cryptographic authentication, and comprehensive tracking systems.

Chiplet Authentication Framework

// Chiplet Authentication System
interface ChipletAuthentication {
    // Physical Unclonable Function (PUF)
    puf: {
        type: "SRAM" | "RingOscillator" | "Arbiter" | "Butterfly";
        bitLength: number;        // 256-4096 bits
        reliability: number;      // % stable bits
        uniqueness: number;       // % inter-chip variation

        // PUF-based key generation
        generateKey(): Promise<CryptoKey>;
        authenticate(challenge: Challenge): Promise<Response>;
    };

    // Unique Identifier (UID)
    uid: {
        format: "128-bit UUID";
        storage: "Fuse" | "EEPROM" | "PUF-derived";
        cryptoBinding: boolean;
    };

    // Certificate Chain
    certificate: {
        type: "X.509v3";
        issuer: "WIA-CA" | "Vendor-CA";
        attributes: ChipletAttributes;
        validityPeriod: Date[];
        revocationCheck: "OCSP" | "CRL";
    };
}

// Authentication Protocol
class ChipletAuthProtocol {
    async authenticateChiplet(chiplet: ChipletId): Promise<AuthResult> {
        // Step 1: Read UID
        const uid = await this.readUID(chiplet);

        // Step 2: Verify certificate chain
        const cert = await this.getCertificate(uid);
        const chainValid = await this.verifyCertChain(cert);

        // Step 3: PUF challenge-response
        const challenge = this.generateChallenge();
        const response = await this.sendChallenge(chiplet, challenge);
        const pufValid = await this.verifyPUFResponse(response, cert);

        // Step 4: Check revocation status
        const notRevoked = await this.checkRevocation(cert);

        return {
            authenticated: chainValid && pufValid && notRevoked,
            chipletId: uid,
            certificate: cert,
            trustLevel: this.calculateTrustLevel(cert)
        };
    }
}

Hardware Trojan Detection

Hardware trojans represent one of the most serious threats to chiplet security. Detection requires a combination of design-time techniques, manufacturing-time testing, and runtime monitoring.

Hardware Trojan Characteristics

Hardware trojans typically have three components: a trigger mechanism that activates the trojan, a payload that executes the malicious function, and stealth features that evade detection. Chiplet systems must address trojans that could be inserted at any point in the supply chain.

Secure Die-to-Die Communication

The die-to-die interface in chiplet systems must be secured against eavesdropping, tampering, and man-in-the-middle attacks. This is especially critical in multi-vendor systems where chiplets from different sources communicate.

// Secure D2D Implementation
class SecureD2DLink {
    private sessionKey: CryptoKey;
    private sequenceNumber: bigint;

    async establishSecureSession(peer: ChipletId): Promise<void> {
        // Mutual authentication
        await this.authenticatePeer(peer);

        // Key exchange
        const ephemeralKey = await this.generateEphemeralKey();
        const peerPublic = await this.exchangePublicKey(peer, ephemeralKey.publicKey);

        // Derive session key
        this.sessionKey = await this.deriveSessionKey(ephemeralKey.privateKey, peerPublic);

        // Initialize sequence number
        this.sequenceNumber = BigInt(0);
    }

    async sendSecure(data: Uint8Array): Promise<void> {
        // Increment sequence number
        this.sequenceNumber++;

        // Construct authenticated encryption
        const nonce = this.constructNonce(this.sequenceNumber);
        const aad = this.constructAAD(this.sequenceNumber);

        // Encrypt and authenticate
        const ciphertext = await crypto.subtle.encrypt(
            { name: "AES-GCM", iv: nonce, additionalData: aad },
            this.sessionKey,
            data
        );

        // Transmit
        await this.transmit(ciphertext);
    }
}
