🔒 Chapter 6: Security and Compliance

Protecting Student Data and Privacy

Security and compliance are not optional features but fundamental requirements for responsible learning analytics. This chapter covers essential security practices, privacy regulations, and compliance frameworks.

6.1.2 FERPA Compliance for Learning Analytics

What is FERPA?

The Family Educational Rights and Privacy Act (FERPA) protects the privacy of student education records. For learning analytics, key considerations include:

  • Educational Records Definition: Analytics data tied to individual students counts as educational records
  • Directory Information: Some data (name, major) can be disclosed without consent if designated as directory information
  • Legitimate Educational Interest: School officials with legitimate educational interest can access records
  • Student Rights: Students (or parents if under 18) can inspect and request corrections to records
  • Third-Party Disclosure: Requires consent except for specific exceptions (school officials, compliance with judicial order, etc.)

Common FERPA Violation: Displaying student analytics data on publicly visible dashboards or leaderboards that reveal individual performance. Even anonymized data can violate FERPA if students can be re-identified.

6.1.3 GDPR Requirements

Principles for Processing Personal Data:

  • Lawfulness, Fairness, Transparency: Clear communication about data processing
  • Purpose Limitation: Collect for specified, explicit purposes only
  • Data Minimization: Adequate, relevant, and limited to what's necessary
  • Accuracy: Keep data accurate and up to date
  • Storage Limitation: Retain only as long as necessary
  • Integrity and Confidentiality: Appropriate security measures
  • Accountability: Demonstrate compliance

Individual Rights Under GDPR:

Right Description Implementation
Right to Access Obtain copy of personal data Provide data export feature, respond within 30 days
Right to Rectification Correct inaccurate data Allow profile updates, verify accuracy
Right to Erasure "Right to be forgotten" Delete data when no longer necessary, with exceptions
Right to Restrict Limit processing Flag records, suppress from analytics
Right to Portability Receive data in machine-readable format Export to JSON, CSV
Right to Object Object to processing Opt-out mechanisms, justified overrides

6.2 Data Security

6.2.1 Encryption Standards

Encryption at Rest:

  • Database Encryption: AES-256 for all databases containing student data
  • File System Encryption: Encrypt volumes storing analytics data
  • Backup Encryption: Ensure backups are encrypted with same standards
  • Key Management: Use hardware security modules (HSM) or key management services
  • Key Rotation: Regularly rotate encryption keys (annually minimum)

Encryption in Transit:

  • TLS 1.3: Use latest TLS for all network communications
  • Certificate Management: Valid certificates from trusted CAs, automated renewal
  • Perfect Forward Secrecy: Use cipher suites supporting PFS
  • HSTS: HTTP Strict Transport Security to prevent downgrade attacks

6.2.2 Access Control

Principle of Least Privilege:

Role-Based Access Control Matrix:

                     | View Own | View Class | View All | Export | Admin |
-------------------- |----------|------------|----------|--------|-------|
Student              |    ✓     |            |          |        |       |
Teaching Assistant   |    ✓     |     ✓      |          |        |       |
Instructor           |    ✓     |     ✓      |          |   ✓    |       |
Department Chair     |    ✓     |     ✓      |    ✓     |   ✓    |       |
Analytics Admin      |    ✓     |     ✓      |    ✓     |   ✓    |   ✓   |
System Admin         |          |            |          |        |   ✓   |

✓ = Permission granted

Access Control Implementation:

  • Multi-Factor Authentication (MFA): Require MFA for all analytics system access
  • Single Sign-On (SSO): Integrate with institutional identity management
  • Session Management: Timeout after 30 minutes inactivity, secure session tokens
  • Audit Logging: Log all access to student data with user, time, action
  • Regular Reviews: Quarterly access reviews, remove unnecessary permissions

6.2.3 Security Monitoring

Security Events to Monitor:

  • Failed login attempts (potential brute force attacks)
  • Unusual data access patterns (e.g., bulk exports, access to many records)
  • Privilege escalation attempts
  • After-hours access from unusual locations
  • Changes to security configurations
  • API rate limit violations

6.3 Privacy-Preserving Techniques

6.3.1 Anonymization and Pseudonymization

Anonymization: Irreversibly removing personally identifiable information

  • Techniques: Aggregation, generalization, data suppression
  • Use Cases: Research datasets, public reporting, cross-institution benchmarking
  • Limitations: Risk of re-identification through combination with other data

Pseudonymization: Replacing identifiers with pseudonyms

  • Techniques: Hashing, tokenization, encryption
  • Use Cases: Analytics processing, data sharing with controlled access
  • Advantages: Reversible with proper authorization, maintains data utility

K-Anonymity: Ensure each record is indistinguishable from at least k-1 other records. For example, if k=5, each student's data should be identical to at least 4 others on identifying attributes (age, major, etc.).

6.3.2 Differential Privacy

What is Differential Privacy?

A mathematical framework that provides privacy guarantees by adding carefully calibrated noise to query results. The presence or absence of any individual's data has minimal impact on query output.

Implementation Approaches:

  • Global Differential Privacy: Add noise at data collection before storing
  • Local Differential Privacy: Add noise to individual data before sending to system
  • Privacy Budget (ε): Total privacy loss allowed, smaller ε = stronger privacy
  • Query Budget: Limit number of queries to prevent privacy leakage

6.4 Data Breach Response

6.4.1 Incident Response Plan

Preparation Phase:

  • Establish incident response team (IT security, legal, compliance, communications)
  • Define breach classification criteria (severity levels)
  • Create notification templates (students, regulators, media)
  • Set up secure communication channels
  • Conduct regular drills and simulations

Detection and Analysis:

  • Identify scope: What data was accessed? How many individuals affected?
  • Determine cause: How did the breach occur? What vulnerabilities were exploited?
  • Assess risk: What is the potential harm to individuals?
  • Document everything: Maintain detailed timeline and evidence

Containment and Eradication:

  • Immediate containment: Isolate affected systems, revoke compromised credentials
  • Root cause elimination: Patch vulnerabilities, strengthen controls
  • Verification: Confirm threat is eliminated before restoration

Notification Requirements:

Regulation Timeline Who to Notify
GDPR 72 hours to regulator, without undue delay to individuals Supervisory authority, affected individuals
FERPA No specific timeline Affected students/parents, Department of Education
State Laws (US) Varies by state (often "without unreasonable delay") Affected individuals, state attorney general

6.5 Ethical Data Use

6.5.1 Ethics Review Process

When to Conduct Ethics Review:

  • New analytics use cases or algorithms
  • Changes to data collection or processing
  • Research involving student data
  • High-stakes decisions based on analytics (admissions, financial aid)

Ethics Review Criteria:

Criterion Questions to Consider
Beneficence Who benefits? How do benefits outweigh risks?
Non-maleficence What are potential harms? How are they minimized?
Justice Are benefits and burdens fairly distributed? Any bias?
Autonomy Do students have meaningful choice? Is consent informed?
Transparency Can stakeholders understand how analytics work?

6.5.2 Algorithmic Accountability

Explainability Requirements:

  • Model Documentation: Describe what algorithm does, training data, performance metrics
  • Individual Explanations: Explain why a particular prediction was made
  • Feature Importance: Identify which factors most influenced predictions
  • Counterfactual Explanations: "If X changed to Y, outcome would be Z"
  • Contestability: Mechanisms for students to challenge decisions

6.6 Vendor Management

6.6.1 Third-Party Risk Assessment

Vendor Security Evaluation:

  • SOC 2 Type II certification
  • ISO 27001 compliance
  • Privacy certifications (Privacy Shield, EU-US Data Privacy Framework)
  • Regular penetration testing results
  • Data breach history and response
  • Subprocessor disclosure and management

6.6.2 Data Processing Agreements

Essential DPA Clauses:

  • Purpose Limitation: Vendor processes data only for specified purposes
  • Confidentiality: Vendor maintains confidentiality, trains staff
  • Security Measures: Vendor implements appropriate technical and organizational measures
  • Sub-processing: Require approval for subcontractors
  • Data Return/Deletion: Return or delete data upon contract termination
  • Audit Rights: Institution can audit vendor compliance
  • Breach Notification: Vendor must notify of any data breaches
  • Indemnification: Vendor liability for privacy violations

6.7 Compliance Auditing

6.7.1 Regular Compliance Checks

Quarterly Compliance Checklist:

  • Review and update privacy notices
  • Audit user access permissions
  • Verify encryption is functioning
  • Test backup and recovery procedures
  • Review security logs for anomalies
  • Update data inventory and flow diagrams
  • Assess vendor compliance
  • Review and process data subject requests

6.7.2 Documentation Requirements

Essential Documentation:

  • Data Inventory: What data is collected, where stored, how used
  • Privacy Impact Assessment: Risks and mitigation for each use case
  • Consent Records: Evidence of informed consent obtained
  • Security Policies: Access control, encryption, incident response procedures
  • Training Records: Staff training on privacy and security
  • Vendor Agreements: Contracts and DPAs with third parties
  • Audit Logs: System access and data processing activities
  • Breach Records: Incidents, investigations, remediation
