## 이론

### "학습한 순간부터 모델은 낡기 시작한다"

ML 엔지니어링을 10년 한 선배가 내게 해준 말이다. 학습 직후 F1 0.92였던 모델이 6개월 뒤 0.78로 떨어져 있었고, 나는 그 이유를 몰랐다. 로그를 뒤지고 코드를 뒤졌지만 파이프라인은 멀쩡했다. 문제는 **입력 데이터의 분포가 바뀐 것**이었다. 사용자들이 올리는 고문서가 학습 당시엔 300dpi 스캔본 중심이었는데, 반 년 사이 휴대폰 카메라 촬영본이 60%를 넘었다. 모델은 그 분포를 본 적이 없었다.

이 경험 이후 나는 **"배포는 시작이지 끝이 아니다"** 를 원칙으로 삼는다. 프로덕션 ML 시스템은 실시간 감시·점진 개선·자동 롤백의 세 축으로 유지된다.

### 드리프트 3종류

| 종류 | 정의 | 감지 지표 |
|---|---|---|
| **Covariate Shift (데이터 드리프트)** | 입력 분포 P(X) 변화 | KL Divergence, PSI, Wasserstein |
| **Concept Drift (개념 드리프트)** | P(Y\|X) 관계 변화 | 예측 정확도 하락 |
| **Label Drift** | P(Y) 분포 변화 | 클래스 비율 변화 |

고문서 복원처럼 **입력은 이미지, 출력은 복원 품질 지표**인 태스크에서는 주로 Covariate Shift와 Concept Drift가 문제. Label Drift는 분류 태스크 관심사.

### SLO와 에러 예산

Google SRE 책의 핵심 개념을 ML 시스템에 적용:

```
SLO 예: "P95 응답시간 < 5초, 가용성 99.5%, 복원 SSIM > 0.85"
에러 예산 = 100% - SLO = 0.5% = 월 3.6시간 다운타임 허용
```

에러 예산이 남아 있으면 **공격적 실험·빠른 배포** 허용. 소진되면 **안정화 모드** — 신기능 중단, 기존 장애 수정. 이게 품질과 속도의 자동 균형 장치다.

내 고문서 복원 서비스 SLO:
- **가용성**: 99.5% (월 3.6h 다운타임)
- **지연**: P95 < 10초, P99 < 30초
- **품질**: SSIM_inside > 0.70 (월 평균)
- **실패율**: 모델 에러 < 2%

### 점진 배포의 4단계

```
1. Shadow Deployment   — 신 모델 병행 실행, 결과 비교만 (사용자 영향 0)
2. Canary (1~5%)        — 일부 사용자에게만 신 모델
3. A/B (50/50)          — 통계적 유의성 확보
4. Full Rollout         — 전면 전환
   ↓ 문제 감지 시
5. Automatic Rollback   — SLO 위반 시 즉시 이전 버전으로
```

**Shadow**가 결정적이다. 프로덕션 트래픽과 동일한 입력으로 신 모델을 돌려 **출력만 로그**에 저장. 사용자는 기존 모델 결과만 보지만, 우리는 두 모델 비교로 **실전 성능 차이**를 안전하게 측정한다.

### 재학습 트리거 설계

**시간 기반**: "매월 1일" — 단순하지만 불필요한 학습 발생  
**성능 기반**: "SSIM 평균 하락 5%p" — 이상적이지만 검출 지연 있음  
**드리프트 기반**: "PSI > 0.2 지속 3일" — 선제적 대응 가능 ⭐

실무 조합: **드리프트 감지 → 수동 검토 → 재학습 승인**. 완전 자동은 위험 (잘못된 트리거로 매주 학습 돌면 비용·품질 모두 악화).

# 1. 드리프트 감지 — PSI + Wasserstein
# ──────────────────────────────────────────────
class DriftDetector:
    """
    PSI(Population Stability Index) + Wasserstein distance로 분포 이동 감지.
    PSI 해석:
      < 0.1  안정
      0.1~0.2 관찰 필요
      > 0.2  유의미한 드리프트
    """

    def __init__(self, baseline_features: pd.DataFrame):
        """
        baseline_features: 학습 당시 입력 특성 샘플 (N × features)
        예: [brightness, resolution, mask_ratio, aspect_ratio, ...]
        """
        self.baseline = baseline_features
        self.baseline_bins: Dict[str, np.ndarray] = {}
        self._precompute_bins()

    def _precompute_bins(self, n_bins: int = 10):
        """각 feature를 10 bin으로 분할 (decile 기준)"""
        for col in self.baseline.columns:
            self.baseline_bins[col] = np.quantile(
                self.baseline[col].dropna(),
                np.linspace(0, 1, n_bins + 1),
            )

    def compute_psi(self, current: pd.DataFrame, col: str) -> float:
        """PSI = Σ (current% - baseline%) × ln(current% / baseline%)"""
        bins = self.baseline_bins[col]
        base_hist, _ = np.histogram(self.baseline[col], bins=bins)
        curr_hist, _ = np.histogram(current[col],       bins=bins)

        base_pct = base_hist / max(base_hist.sum(), 1)
        curr_pct = curr_hist / max(curr_hist.sum(), 1)

        # 0 division 방지
        base_pct = np.where(base_pct == 0, 1e-6, base_pct)
        curr_pct = np.where(curr_pct == 0, 1e-6, curr_pct)

        psi = np.sum((curr_pct - base_pct) * np.log(curr_pct / base_pct))
        return float(psi)

    def detect(self, current: pd.DataFrame) -> Dict:
        """모든 feature의 PSI + 종합 판정"""
        from scipy.stats import wasserstein_distance

        results = {}
        severity = "stable"

        for col in self.baseline.columns:
            psi = self.compute_psi(current, col)
            wd  = wasserstein_distance(self.baseline[col], current[col])
            verdict = (
                "stable" if psi < 0.1 else
                "warning" if psi < 0.2 else
                "drifted"
            )
            results[col] = {"psi": round(psi, 3), "wasserstein": round(wd, 3),
                            "verdict": verdict}

            # 전체 심각도는 최악 기준
            if verdict == "drifted":
                severity = "drifted"
            elif verdict == "warning" and severity != "drifted":
                severity = "warning"

        return {
            "timestamp":    datetime.now(timezone.utc).isoformat(),
            "severity":     severity,
            "per_feature":  results,
            "recommended_action": self._recommend(severity),
        }

    @staticmethod
    def _recommend(severity: str) -> str:
        return {
            "stable":   "현재 유지, 주간 모니터링",
            "warning":  "원인 분석 + 일간 모니터링",
            "drifted":  "재학습 후보, 3일 지속되면 학습 트리거",
        }[severity]

# 2. SLO 관리 + 에러 예산
# ──────────────────────────────────────────────
@dataclass
class SLOConfig:
    name:                str
    metric:              str                # "availability" | "latency_p95" | "quality_ssim"
    target:              float
    target_comparator:   Literal["gte", "lte"]   # SSIM은 gte, latency는 lte
    window_days:         int = 28             # 슬라이딩 윈도우


@dataclass
class SLOStatus:
    config:         SLOConfig
    current_value:  float
    compliant:      bool
    error_budget_remaining_pct: float       # 0~100
    burn_rate:      float                   # 소진 속도 (1.0 = 정상 속도)


class SLOManager:
    """
    SLO 상태 조회 + 에러 예산 계산 + 경고 발생
    """
    def __init__(self, configs: List[SLOConfig]):
        self.configs = {c.name: c for c in configs}

    def evaluate(
        self,
        metric_history: pd.DataFrame,    # [timestamp, metric_name, value]
    ) -> List[SLOStatus]:
        results = []
        now = datetime.now(timezone.utc)

        for name, cfg in self.configs.items():
            window_start = now - timedelta(days=cfg.window_days)
            filtered = metric_history[
                (metric_history["metric_name"] == cfg.metric)
                & (pd.to_datetime(metric_history["timestamp"]) >= window_start)
            ]
            if filtered.empty:
                continue

            values = filtered["value"].values

            # 준수 판정
            if cfg.target_comparator == "gte":
                compliant_points = (values >= cfg.target).sum()
            else:
                compliant_points = (values <= cfg.target).sum()

            compliance_rate = compliant_points / len(values)
            error_budget_total = 1.0 - (cfg.target if cfg.metric == "availability" else 0.99)
            error_rate = 1.0 - compliance_rate
            budget_consumed = error_rate / max(error_budget_total, 1e-6)
            budget_remaining = max(0, 1 - budget_consumed) * 100

            results.append(SLOStatus(
                config=cfg,
                current_value=float(np.mean(values)),
                compliant=compliance_rate >= (cfg.target if cfg.metric == "availability" else 0.95),
                error_budget_remaining_pct=round(budget_remaining, 1),
                burn_rate=round(budget_consumed * (28 / cfg.window_days), 2),
            ))
        return results

# 3. 섀도우 배포 — 두 모델 병행 비교
# ──────────────────────────────────────────────
class ShadowDeployment:
    """
    Production 모델과 Candidate 모델을 동일 입력에 돌려 결과 비교 로그.
    사용자는 Production 결과만 받음.
    """
    def __init__(
        self,
        production_model,
        candidate_model,
        log_path: str = "./logs/shadow.jsonl",
    ):
        self.production = production_model
        self.candidate  = candidate_model
        self.log_path   = Path(log_path)
        self.log_path.parent.mkdir(parents=True, exist_ok=True)

    async def serve(self, request: dict) -> dict:
        """실제 서빙은 Production, 섀도우로 Candidate 동시 실행 + 비교 로그"""
        import asyncio
        prod_task      = asyncio.create_task(self._run(self.production, request))
        candidate_task = asyncio.create_task(self._run(self.candidate,  request))

        # Production 결과 대기 (사용자 응답)
        prod_result = await prod_task

        # Candidate는 별도 처리, 사용자 응답 블로킹 X
        async def log_candidate():
            try:
                cand_result = await candidate_task
                self._log_comparison(request, prod_result, cand_result)
            except Exception as e:
                self._log_comparison(request, prod_result, {"error": str(e)})

        asyncio.create_task(log_candidate())
        return prod_result

    async def _run(self, model, request):
        return await model.infer(request)

    def _log_comparison(self, request, prod, cand):
        record = {
            "timestamp":  datetime.now(timezone.utc).isoformat(),
            "request_id": request.get("id"),
            "prod": {"ssim": prod.get("ssim"), "latency": prod.get("latency_ms")},
            "cand": {"ssim": cand.get("ssim"), "latency": cand.get("latency_ms")},
        }
        with open(self.log_path, "a", encoding="utf-8") as f:
            f.write(json.dumps(record) + "\n")

    def analyze_shadow_log(self, min_samples: int = 1000) -> dict:
        """섀도우 로그 분석 — Candidate가 Production 대비 얼마나 좋은가"""
        records = [json.loads(l) for l in open(self.log_path, encoding="utf-8")]
        if len(records) < min_samples:
            return {"status": "insufficient_samples", "count": len(records)}

        df = pd.DataFrame([{
            "prod_ssim":    r["prod"].get("ssim"),
            "cand_ssim":    r["cand"].get("ssim") if "error" not in r["cand"] else None,
            "prod_latency": r["prod"].get("latency"),
            "cand_latency": r["cand"].get("latency") if "error" not in r["cand"] else None,
        } for r in records]).dropna()

        return {
            "samples":           len(df),
            "ssim_improvement":  float(df["cand_ssim"].mean() - df["prod_ssim"].mean()),
            "latency_change_ms": float(df["cand_latency"].mean() - df["prod_latency"].mean()),
            "candidate_wins":    int((df["cand_ssim"] > df["prod_ssim"]).sum()),
            "recommendation":    self._judge_candidate(df),
        }

    @staticmethod
    def _judge_candidate(df: pd.DataFrame) -> str:
        ssim_gain = df["cand_ssim"].mean() - df["prod_ssim"].mean()
        lat_cost  = df["cand_latency"].mean() - df["prod_latency"].mean()

        if ssim_gain > 0.02 and lat_cost < 500:
            return "promote_to_canary"
        if ssim_gain > 0.01:
            return "promote_with_latency_tuning"
        return "keep_production"

# 4. A/B 테스트 결정 — 통계적 유의성
# ──────────────────────────────────────────────
def ab_decision(
    group_a_scores: np.ndarray,
    group_b_scores: np.ndarray,
    alpha: float = 0.05,
    min_effect_size: float = 0.01,
) -> dict:
    """
    두 그룹 비교 — Welch's t-test + effect size.
    통계적 유의 + 실질적 효과 모두 만족해야 승자 선언.
    """
    from scipy.stats import ttest_ind

    t_stat, p_value = ttest_ind(group_a_scores, group_b_scores, equal_var=False)
    mean_diff = float(group_b_scores.mean() - group_a_scores.mean())

    if p_value >= alpha:
        verdict = "inconclusive"
    elif abs(mean_diff) < min_effect_size:
        verdict = "no_practical_difference"
    elif mean_diff > 0:
        verdict = "b_wins"
    else:
        verdict = "a_wins"

    return {
        "verdict":        verdict,
        "p_value":        round(p_value, 4),
        "mean_diff":      round(mean_diff, 4),
        "sample_size":    {"a": len(group_a_scores), "b": len(group_b_scores)},
    }

# 5. 자동 롤백 핸들러
# ──────────────────────────────────────────────
class AutoRollback:
    """
    SLO 위반 지속 감지 시 자동 롤백.
    5분 내 연속 3회 임계 초과면 발동.
    """
    def __init__(
        self,
        slo_manager: SLOManager,
        deployment_controller,     # k8s 또는 자체 디플로이어
        consecutive_threshold: int = 3,
        check_interval_sec: int = 60,
    ):
        self.slo = slo_manager
        self.deploy = deployment_controller
        self.threshold = consecutive_threshold
        self.interval = check_interval_sec
        self.consecutive_violations = 0

    def on_slo_check(self, statuses: List[SLOStatus]) -> None:
        critical = [s for s in statuses if not s.compliant and s.burn_rate > 2.0]
        if critical:
            self.consecutive_violations += 1
            print(f"⚠️  SLO 위반 연속 {self.consecutive_violations}회")

            if self.consecutive_violations >= self.threshold:
                self.trigger_rollback(critical)
                self.consecutive_violations = 0
        else:
            self.consecutive_violations = 0

    def trigger_rollback(self, critical_statuses):
        """이전 안정 버전으로 롤백"""
        previous_version = self.deploy.get_previous_stable_version()
        print(f">> 자동 롤백: {self.deploy.current_version()} → {previous_version}")
        self.deploy.rollback_to(previous_version)
        # Slack/PagerDuty 알림
        self._alert(critical_statuses, previous_version)

    def _alert(self, statuses, rolled_back_to):
        # 실제 구현은 webhook 전송
        msg = (
            f"🔥 AutoRollback: 버전 {rolled_back_to} 로 롤백\n"
            f"위반 SLO: {[s.config.name for s in statuses]}"
        )
        print(msg)

## 함정 카드

1. **"학습 끝나면 프로젝트 끝"** → ML 프로젝트 총 수명 중 **개발 30%, 배포 이후 운영 70%**. 모니터링·드리프트·재학습을 설계하지 않은 모델은 6개월 뒤 사용 불가 상태가 흔하다. 처음부터 MLOps 고려가 필수.

2. **"PSI 임계값 0.2를 절대 기준"** → 0.2는 일반 권장이지만 태스크마다 다르다. 고문서 도메인은 입력 변동 커서 0.25가 합리적일 수 있고, 의료 영상은 0.15로 엄격하게. 도메인별 **자체 임계** 정립이 필수.

3. **"섀도우 배포 결과를 바로 프로모션"** → 1,000 샘플 비교해서 신 모델이 0.01 더 좋다고 즉시 교체하면 위험. **통계적 유의성 + 실질적 효과 + 에지 케이스 검토** 3박자. `ab_decision()` 같은 규칙 + 1주일 관찰.

4. **"자동 롤백을 광범위하게 설정"** → 일시 네트워크 스파이크·리전 장애도 SLO 위반. **consecutive_threshold**를 3~5회로 두고 **burn_rate > 2.0** 조건 엄격하게. 너무 민감하면 롤백 반복으로 서비스 불안정. 너무 둔감하면 실제 장애 방치.

5. **"재학습을 전자동화"** → 매주 드리프트 → 학습 → 배포를 자동화하면 **에러가 증폭**된다. 데이터 오염 한 번에 전체 파이프라인 부패. 반드시 **드리프트 감지 → 사람 승인 → 학습** 단계 유지. 자동화는 트리거까지만.

## 연습문제

**문제 1.** 서비스 SLO가 "SSIM > 0.70, 가용성 99.5%"인데 지난 28일 중 11일이 SSIM 미달이다. 에러 예산 소진율과 다음 행동은?

> **해설:** SSIM SLO 준수율 = (28-11)/28 = **60.7%**, 목표 95% 대비 34.3%p 미달. 에러 예산 총 5% 중 이미 39.3% 초과 소진 — **에러 예산 음수**. 행동 순서: ① 즉시 안정화 모드 — 신기능 중단, SSIM 회복 전용 ② 원인 분석 — 드리프트 감지기로 입력 분포 확인, 최근 배포된 모델 버전 체크 ③ 롤백 후보 — 이전 안정 버전 존재 시 즉시 복구. 28일 윈도우 SLO는 **한 번 깨지면 회복까지 수 주 소요**. 장기 관점 플래닝.

**문제 2.** 섀도우 배포에서 Candidate 모델이 SSIM +0.03 개선, 대신 지연 +800ms 증가. 프로모션 판단은?

> **해설:** **프로모션 유보, 최적화 우선**. SSIM 0.03 개선은 유의미(최소 임계 0.01 × 3). 하지만 지연 800ms 추가는 **사용자 체감 치명적** (P95가 10초 → 10.8초면 SLO 가능하나, P99는 더 악화). 실제 사용 맥락 질문: ① 이 기능이 배치 처리면 지연 허용 — 프로모션 ② 실시간 대화형이면 거부 — LCM·양자화·증류 등 경량화 후 재시도. 품질 개선과 지연 악화의 **트레이드오프는 사용자 시나리오가 결정**. 숫자 하나로 결정하지 말 것.

**문제 3.** 자동 롤백이 2주에 3번 발생했다. 원인 후보와 개선은?

> **해설:** **자동 롤백은 "마지막 방어선"이지 "정상 운영"이 아니다**. 2주 3회면 체계 문제. 원인 후보: ① **SLO가 너무 엄격** — 가용성 99.9%를 99.5%로 현실화 ② **신 모델 품질이 불안정** — 배포 전 검증 단계 부실 ③ **인프라 이슈** — GPU 스로틀링, 네트워크 지터 등 모델 외 요인 ④ **consecutive_threshold 너무 낮음** — 3회를 5회로. 개선 프로세스: ① 롤백 각각 포스트모템 작성 ② 근본 원인 분류 ③ SLO 현실성 재검토 ④ 자동 롤백 조건 재설계. 자동화는 도구지 면피 수단 아님.
