3.1 The Complexity of Multimodal Routing

Journey planning in MaaS environments presents unique computational and user experience challenges that go far beyond traditional single-mode navigation. While routing for cars or public transit individually are well-solved problems, combining multiple transport modes introduces combinatorial complexity that requires sophisticated algorithms and careful optimization.

Consider a typical urban journey from home to office. A naive routing system might suggest:

  • Drive your personal car (fastest, but expensive parking)
  • Take the subway (economical, but crowded during rush hour)
  • Walk the entire distance (impractical for 8km)

A multimodal MaaS platform instead computes combinations like:

  • Bike share (2km) → Subway (6km) → Walk (0.5km)
  • Walk (0.8km) → Bus (7km) → E-scooter (0.5km)
  • Taxi (entire distance) on rainy days

Each option involves different transfer points, waiting times, costs, physical effort, and environmental impact. The optimal choice depends on user preferences, real-time conditions, and contextual factors like weather, time of day, and baggage.

3.2 Graph-Based Transportation Networks

Multimodal routing begins with representing the transportation network as a weighted directed graph where:

Node Types

  • Location Nodes: Physical points (addresses, intersections, landmarks)
  • Transit Stops: Bus stops, subway stations, bike docks
  • Transfer Nodes: Points where mode changes occur
  • Virtual Nodes: Logical connection points for scheduling

Edge Types

  • Walking Edges: Pedestrian connections between nearby nodes
  • Transit Edges: Scheduled public transport connections
  • On-Demand Edges: Ride-hailing, car-sharing (dynamically computed)
  • Micromobility Edges: Bike/scooter routes based on current availability
  • Transfer Edges: Mode changes (with associated time penalties)

Edge Weights

Each edge carries multiple weight dimensions:

Dimension Unit Example Value
Duration Minutes 12 minutes
Cost Currency $2.50
Distance Kilometers 3.2 km
Carbon CO₂ kg 0.15 kg
Comfort Score (0-100) 75
Accessibility Boolean/Score Wheelchair-accessible

3.3 Core Routing Algorithms

Dijkstra's Algorithm Adaptations

The classic Dijkstra's shortest path algorithm forms the foundation but requires modifications for multimodal contexts:

function multimodalDijkstra(origin, destination, departureTime, preferences):
    queue = PriorityQueue()
    distances = Map()  // node → minimum cost
    previousNodes = Map()  // for path reconstruction

    queue.add((origin, 0, departureTime, null))  // (node, cost, time, mode)
    distances[origin] = 0

    while not queue.isEmpty():
        (currentNode, currentCost, currentTime, currentMode) = queue.poll()

        if currentNode == destination:
            return reconstructPath(previousNodes, destination)

        for edge in getOutgoingEdges(currentNode, currentTime):
            // Skip if mode change violates user preferences
            if not isAllowedMode(edge.mode, preferences):
                continue

            // Add transfer penalty if mode changes
            transferPenalty = (edge.mode != currentMode) ? TRANSFER_TIME : 0

            newCost = currentCost + edge.cost + transferPenalty
            newTime = currentTime + edge.duration + transferPenalty

            // Multi-objective optimization: cost, time, comfort
            score = computeScore(newCost, newTime, edge, preferences)

            if score < distances.getOrDefault(edge.toNode, INFINITY):
                distances[edge.toNode] = score
                previousNodes[edge.toNode] = (currentNode, edge)
                queue.add((edge.toNode, newCost, newTime, edge.mode))

    return null  // No path found

Time-Dependent Routing

Public transit operates on schedules, requiring time-aware algorithms. The Connection Scan Algorithm (CSA) efficiently handles this by scanning connections in chronological order:

  • Preprocess all transit connections into a sorted array
  • Scan through connections, updating earliest arrival times
  • Time complexity: O(C) where C is number of connections (typically linear)
  • Space complexity: O(S) where S is number of stops

CSA is particularly effective for public transit networks with thousands of stops and millions of daily connections, outperforming Dijkstra's algorithm in these scenarios.

Heuristic-Guided Search (A* Algorithm)

For large networks, A* algorithm with appropriate heuristics dramatically reduces search space:

function aStarMultimodal(origin, destination, departureTime):
    openSet = PriorityQueue()
    gScore = Map()  // actual cost from origin
    fScore = Map()  // estimated total cost (gScore + heuristic)

    gScore[origin] = 0
    fScore[origin] = heuristic(origin, destination)
    openSet.add((origin, fScore[origin]))

    while not openSet.isEmpty():
        current = openSet.poll()

        if current == destination:
            return reconstructPath(cameFrom, destination)

        for neighbor in getNeighbors(current):
            tentativeGScore = gScore[current] + dist(current, neighbor)

            if tentativeGScore < gScore.getOrDefault(neighbor, INFINITY):
                cameFrom[neighbor] = current
                gScore[neighbor] = tentativeGScore
                fScore[neighbor] = gScore[neighbor] + heuristic(neighbor, destination)
                openSet.add((neighbor, fScore[neighbor]))

    return null

// Admissible heuristic: straight-line distance / max speed
function heuristic(node, destination):
    straightLineDistance = haversineDistance(node, destination)
    return straightLineDistance / MAX_TRANSIT_SPEED

3.4 Multi-Objective Optimization

Users rarely optimize for a single dimension. Real preferences involve trade-offs between multiple objectives:

Pareto Optimal Solutions

A journey is Pareto optimal if no other journey is better in all dimensions. MaaS systems should present multiple Pareto-optimal options:

  • Fastest Route: Minimizes total travel time
  • Cheapest Route: Minimizes monetary cost
  • Eco-Friendly Route: Minimizes carbon emissions
  • Balanced Route: Optimizes weighted combination

Weighted Scoring Function

Convert multi-dimensional preferences into a single score for ranking:

function computeRouteScore(route, preferences):
    score = 0

    // Weight factors from user preferences (sum to 1.0)
    score += preferences.timeWeight * normalize(route.duration)
    score += preferences.costWeight * normalize(route.cost)
    score += preferences.comfortWeight * normalize(route.comfortLevel)
    score += preferences.ecoWeight * normalize(route.carbonFootprint)

    // Penalties for undesirable features
    if route.numberOfTransfers > 2:
        score += TRANSFER_PENALTY * route.numberOfTransfers

    if route.requiresWalkingInRain and weather.isRaining:
        score += RAIN_WALK_PENALTY

    return score

3.5 Real-Time Data Integration

Static routing on fixed schedules is insufficient. Modern MaaS must incorporate real-time conditions:

Data Sources

  • Vehicle Positions: GPS tracking of buses, trains, shared bikes/scooters
  • Service Alerts: Delays, cancellations, route changes
  • Traffic Conditions: Congestion, accidents, road closures
  • Weather: Rain, snow, extreme temperatures affecting mode choice
  • Availability: Real-time bike dock occupancy, scooter battery levels
  • Demand Patterns: Predicted wait times based on current demand

Dynamic Recomputation

When real-time conditions change during a journey, the system must recompute routes on-the-fly. This requires:

  • Efficient incremental algorithms that update existing routes rather than recomputing from scratch
  • Push notifications to users when better alternatives emerge
  • Graceful handling of missed connections with automatic rebooking

3.6 Transfer Optimization

Transfers between modes are often the most frustrating part of multimodal journeys. Effective transfer optimization considers:

Transfer Time Estimation

Accurate modeling of transfer duration including:

  • Physical distance between platforms/stops
  • Vertical movement (stairs, escalators, elevators)
  • Queuing time for ticketing or bike checkout
  • Buffer time for reliability (varies by mode and time of day)
Transfer Type Minimum Time Recommended Time
Same Platform 2 minutes 3 minutes
Adjacent Platform 4 minutes 6 minutes
Different Station (same building) 8 minutes 12 minutes
Walk to nearby stop 5-15 minutes 8-20 minutes
Bike dock checkout 3 minutes 5 minutes

Transfer Minimization

When multiple routes have similar total journey time, prefer those with fewer transfers. Research shows users strongly prefer direct connections even if slightly slower.

3.7 Accessibility-Aware Routing

Inclusive MaaS platforms must provide accessible routing for users with disabilities, elderly travelers, and parents with strollers.

Accessibility Attributes

  • Step-Free Access: Elevator/ramp availability at stations
  • Audio/Visual Announcements: For hearing/vision impaired
  • Low-Floor Vehicles: Buses and trams with wheelchair access
  • Assistance Services: Staff support at major hubs
  • Smooth Surfaces: Avoiding cobblestone or rough terrain

Custom Routing Constraints

When accessibility mode is enabled:

  • Filter out routes with stairs or steep slopes
  • Prefer wider sidewalks and accessible crossings
  • Increase transfer buffer times to account for reduced mobility
  • Prioritize stations with confirmed elevator functionality
  • Suggest companion assistance when needed

3.8 Context-Aware Journey Planning

Advanced MaaS systems personalize routing based on user context and historical behavior.

Contextual Factors

Factor Impact on Routing
Time of Day Rush hour: prefer less crowded alternatives
Weather Rain: avoid long walks, prefer covered transit
Day of Week Weekend: reduced transit frequency, more leisure pace
User Location Near home: familiar routes; unknown area: clearer instructions
Journey Purpose Commute: optimize time; leisure: prefer scenic routes
Baggage Heavy bags: avoid stairs, prefer direct routes

Learning from Behavior

Machine learning models can predict user preferences based on historical choices:

  • Preferred departure times for regular commutes
  • Mode preferences (e.g., always chooses bus over subway)
  • Willingness to walk vs. wait for next connection
  • Price sensitivity patterns

3.9 Presentation and UX Considerations

Even perfect routing algorithms fail if users can't understand the results. Effective journey presentation includes:

Route Visualization

  • Map View: Geographic representation with mode-specific colors
  • Step-by-Step Instructions: Clear, actionable directions
  • Timeline View: Horizontal timeline showing journey phases
  • Alternative Comparisons: Side-by-side comparison table

Key Information Hierarchy

Present most critical information prominently:

  1. Total duration and arrival time
  2. Total cost
  3. Number of transfers
  4. Walking distance
  5. Environmental impact (carbon saved)
  6. Detailed step-by-step breakdown

Real-Time Updates During Journey

  • Countdown to next departure
  • Vehicle arrival predictions
  • Disruption alerts with alternative suggestions
  • Proactive rebooking for missed connections

3.10 Chapter Summary

Multimodal journey planning is the technological heart of MaaS, requiring sophisticated graph algorithms, real-time data integration, and multi-objective optimization. Representing transportation networks as weighted directed graphs enables adaptation of classical algorithms like Dijkstra's and A* to handle time-dependent schedules, mode transfers, and diverse user preferences.

Effective routing must balance multiple objectives—time, cost, comfort, sustainability—presenting Pareto-optimal alternatives rather than a single "best" route. Real-time data from vehicle tracking, traffic monitors, and weather services enables dynamic recomputation as conditions change during journeys.

Transfers between modes require careful modeling of physical distances, vertical movement, and buffer times for reliability. Accessibility-aware routing ensures inclusive transportation access for users with disabilities. Context-aware personalization using machine learning adapts routes based on time, weather, purpose, and learned user preferences.

Clear presentation through maps, timelines, and step-by-step instructions transforms complex routing algorithms into intuitive user experiences. By optimizing both algorithmic efficiency and human comprehension, MaaS platforms embody 弘益人間—benefiting all humanity through seamless urban mobility.
