2.1 Architectural Principles for MaaS

Building a successful MaaS platform requires adherence to fundamental architectural principles that ensure scalability, reliability, and maintainability. The WIA-AUTO-025 standard emphasizes these core tenets:

Microservices-Based Design

Modern MaaS platforms employ microservices architecture, decomposing the system into small, independently deployable services. Each microservice owns a specific business capability:

  • User Service: Authentication, authorization, profile management
  • Journey Planning Service: Multimodal route calculation and optimization
  • Booking Service: Reservation management across providers
  • Payment Service: Transaction processing, billing, invoicing
  • Provider Integration Service: Adapters for each transport provider API
  • Real-Time Data Service: Live vehicle positions, disruption alerts
  • Analytics Service: Usage patterns, reporting, insights

API-First Approach

Every component exposes well-defined RESTful APIs following the WIA-AUTO-025 specification. This enables:

  • Third-party integrations (maps applications, digital assistants)
  • Multi-platform support (mobile apps, web, kiosks)
  • Future extensibility without core system changes
  • Independent testing and deployment

Event-Driven Communication

Services communicate through asynchronous events published to message queues, providing loose coupling and resilience. Example events include:

  • JourneyBooked - User completes a booking
  • VehicleLocationUpdated - Provider sends real-time position
  • PaymentProcessed - Transaction completed
  • ServiceDisruption - Transport disruption detected

2.2 Reference Architecture

The WIA-AUTO-025 standard defines a layered reference architecture that provides a blueprint for MaaS implementations:

Layer Purpose Key Components
Presentation Layer User interfaces and API gateways Mobile apps, web portals, API gateway
Application Layer Business logic and orchestration Microservices, workflow engines
Integration Layer External service connectivity Provider adapters, data transformers
Data Layer Persistent storage and caching Databases, cache, object storage
Infrastructure Layer Cloud resources and networking Compute, storage, load balancers

2.3 Core Microservices Deep Dive

Journey Planning Service

The most complex component, responsible for calculating optimal multimodal routes. Key algorithms include:

Connection Scan Algorithm (CSA): Efficiently computes earliest arrival times for public transit by scanning all connections in chronological order. Time complexity: O(n) where n is the number of connections.

Modified Dijkstra's Algorithm: Finds shortest paths across different transport modes, with edge weights representing time, cost, or composite scores based on user preferences.

Transfer Patterns: Precomputed hub-and-spoke patterns accelerate queries by reducing the search space for common origin-destination pairs.

// Pseudocode for multimodal journey planning
function planJourney(origin, destination, departureTime, preferences):
    // Initialize priority queue with origin
    queue = PriorityQueue()
    queue.add(Journey(origin, departureTime, 0))

    visited = Set()

    while not queue.isEmpty():
        current = queue.poll()

        if current.location == destination:
            return reconstructPath(current)

        if current.location in visited:
            continue

        visited.add(current.location)

        // Expand to all possible next segments
        for mode in [WALK, BUS, SUBWAY, BIKE_SHARE, ...]:
            segments = getAvailableSegments(current, mode, preferences)
            for segment in segments:
                newJourney = extend(current, segment)
                queue.add(newJourney)

    return null  // No route found

Booking Service

Manages the complete booking lifecycle with state machine pattern:

  • PENDING: Temporary reservation created
  • CONFIRMED: Payment successful, booking guaranteed
  • IN_PROGRESS: User actively on journey
  • COMPLETED: Journey finished successfully
  • CANCELLED: User or system cancelled booking
  • FAILED: Technical or payment failure

The service must handle distributed transactions across multiple providers, implementing the Saga pattern to maintain consistency even when individual operations fail.

Payment Service

Processes transactions through multiple payment gateways while maintaining PCI DSS compliance. Key features include:

  • Token storage instead of raw card data
  • Support for credit cards, digital wallets, bank accounts
  • Subscription billing and recurring charges
  • Multi-currency support
  • Refund and dispute handling
  • Revenue splitting among transport providers

2.5 Integration Patterns

Connecting with diverse transport providers requires flexible integration approaches.

Adapter Pattern

Each provider gets a dedicated adapter translating between their proprietary API and WIA-AUTO-025 standard interface. This isolates provider-specific logic, making it easy to add or swap providers.

Circuit Breaker Pattern

Prevents cascading failures when a provider API becomes unavailable. After detecting failures, the circuit breaker temporarily stops calling the failing service, allowing it to recover while protecting the overall system.

Caching Strategy

Static data (station locations, route maps) cached with long TTL. Semi-static data (schedules) cached with moderate TTL. Real-time data (vehicle positions) cached briefly (10-30 seconds) to reduce API load while maintaining freshness.

2.6 Scalability Considerations

Horizontal Scaling

All microservices designed as stateless pods that can be replicated across multiple servers. Auto-scaling policies add or remove instances based on CPU, memory, or request rate metrics.

Database Sharding

Large datasets (journey history, transactions) partitioned across multiple database instances using consistent hashing on user_id. This distributes load and avoids single points of failure.

Geographic Distribution

For MaaS platforms serving multiple cities or countries, deploy regional clusters with edge caching to minimize latency. User requests routed to nearest region.

2.7 Security Architecture

Authentication and Authorization

OAuth 2.0 and OpenID Connect for user authentication. JWT tokens for API authorization. Role-based access control (RBAC) for different user types (regular users, corporate admins, system operators).

Data Protection

  • TLS 1.3 for all network communication
  • Encryption at rest for PII and payment data
  • Field-level encryption for highly sensitive data
  • Regular security audits and penetration testing

Privacy by Design

GDPR and privacy regulation compliance built into architecture:

  • Data minimization - collect only necessary information
  • Purpose limitation - use data only for stated purposes
  • Right to erasure - user data deletion mechanisms
  • Data portability - export user data in standard formats

2.8 Monitoring and Observability

Three Pillars of Observability

Metrics: Quantitative measurements like request rate, error rate, latency. Collected via Prometheus/Grafana or similar tools.

Logs: Detailed event records for debugging. Centralized using ELK stack (Elasticsearch, Logstash, Kibana) or similar. Structured JSON format for machine parsing.

Traces: Distributed tracing across microservices using OpenTelemetry. Tracks complete request flow through the system, identifying bottlenecks.

Key Performance Indicators

Journey Planning Time < 2s > 5s

2.9 Disaster Recovery and Business Continuity

Backup Strategy

Automated daily backups of all databases with point-in-time recovery capability. Critical configuration stored in version control. Regular restore testing to verify backup integrity.

Failure Modes and Mitigation

  • Provider API Outage: Gracefully degrade by excluding failed provider from route options
  • Payment Gateway Failure: Failover to backup payment processor
  • Database Failure: Promote read replica to primary, reroute traffic
  • Regional Datacenter Outage: DNS failover to backup region

2.10 Chapter Summary

Effective MaaS platform architecture requires careful attention to microservices design, API-first development, and event-driven communication. The WIA-AUTO-025 reference architecture provides a proven blueprint with five layers: presentation, application, integration, data, and infrastructure.

Core services like journey planning, booking, and payment must handle complex algorithms, distributed transactions, and high-volume requests while maintaining sub-second response times. Well-defined data schemas ensure interoperability, while integration patterns like adapters and circuit breakers enable reliable provider connectivity.

Scalability through horizontal scaling, sharding, and geographic distribution allows MaaS platforms to serve millions of users. Security architecture incorporating OAuth 2.0, encryption, and privacy by design protects sensitive user data. Comprehensive monitoring provides visibility into system health and performance.

By adhering to these architectural principles aligned with 弘益人間—benefiting all humanity—development teams can build MaaS platforms that are robust, scalable, and maintainable as they evolve to meet growing urban mobility needs.
