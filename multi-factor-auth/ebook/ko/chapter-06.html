기업용 MFA

기업 환경에서의 MFA 구현은 SSO 통합, 페더레이션, 조건부 액세스 등 추가적인 고려사항이 필요합니다. 엔터프라이즈 ID 관리와 MFA의 통합을 살펴봅니다.

6.1 SSO (Single Sign-On) 통합

SSO는 사용자가 한 번의 로그인으로 여러 애플리케이션에 접근할 수 있게 해줍니다. MFA와 SSO를 결합하면 보안과 편의성을 모두 확보할 수 있습니다.

6.1.1 SSO 프로토콜

프로토콜 표준 사용 사례 MFA 통합
SAML 2.0 OASIS 엔터프라이즈 웹 앱 AuthnContext로 MFA 요구
OAuth 2.0 + OIDC IETF/OpenID 모던 웹/모바일 앱 acr_values로 MFA 요구
WS-Federation OASIS Microsoft 환경 ClaimType으로 MFA 요구

6.1.2 SAML MFA 통합

<!-- SAML 인증 요청에서 MFA 요구 -->
<samlp:AuthnRequest
    xmlns:samlp="urn:oasis:names:tc:SAML:2.0:protocol"
    ID="_abc123"
    Version="2.0"
    IssueInstant="2025-01-15T10:00:00Z"
    Destination="https://idp.example.com/sso">

    <saml:Issuer>https://sp.example.com</saml:Issuer>

    <!-- MFA 요구 컨텍스트 -->
    <samlp:RequestedAuthnContext Comparison="minimum">
        <saml:AuthnContextClassRef>
            urn:oasis:names:tc:SAML:2.0:ac:classes:PasswordProtectedTransport
        </saml:AuthnContextClassRef>
        <saml:AuthnContextClassRef>
            urn:oasis:names:tc:SAML:2.0:ac:classes:MobileTwoFactorContract
        </saml:AuthnContextClassRef>
    </samlp:RequestedAuthnContext>

</samlp:AuthnRequest>

6.1.3 OIDC MFA 통합

// OpenID Connect 인증 요청
const authUrl = new URL('https://idp.example.com/authorize');

authUrl.searchParams.set('client_id', 'your-client-id');
authUrl.searchParams.set('response_type', 'code');
authUrl.searchParams.set('scope', 'openid profile email');
authUrl.searchParams.set('redirect_uri', 'https://app.example.com/callback');
authUrl.searchParams.set('state', generateState());
authUrl.searchParams.set('nonce', generateNonce());

// MFA 요구 (Authentication Context Class Reference)
authUrl.searchParams.set('acr_values', 'urn:mfa');

// 또는 특정 인증 수준 요구
// authUrl.searchParams.set('acr_values', 'urn:mfa:hardware_key');

window.location.href = authUrl.toString();

// 토큰 검증 시 ACR 확인
const idToken = await verifyIdToken(tokenResponse.id_token);

if (idToken.acr !== 'urn:mfa') {
    // MFA가 수행되지 않음 - 추가 인증 요구
    throw new Error('MFA 인증이 필요합니다');
}

6.2 ID 제공자 (Identity Provider)

기업 환경에서는 중앙화된 ID 제공자(IdP)를 통해 인증을 관리합니다. 주요 IdP 플랫폼과 MFA 기능을 비교합니다.

Microsoft Entra ID (Azure AD)

  • Microsoft Authenticator 푸시
  • FIDO2 보안 키
  • Windows Hello for Business
  • 조건부 액세스 정책

Okta

  • Okta Verify 푸시
  • WebAuthn/FIDO2
  • SMS/음성 OTP
  • 적응형 MFA

Google Workspace

  • Google Prompt
  • Titan 보안 키
  • Google Authenticator
  • 고급 보호 프로그램

Ping Identity

  • PingID
  • FIDO2/WebAuthn
  • 위험 기반 인증
  • 행동 분석

6.3 조건부 액세스

조건부 액세스는 사용자, 디바이스, 위치, 애플리케이션 등의 조건에 따라 동적으로 접근 정책을 적용하는 방식입니다.

6.3.1 조건부 액세스 정책 예시

# Azure AD 조건부 액세스 정책 (JSON)
{
  "displayName": "고위험 로그인 시 MFA 요구",
  "state": "enabled",
  "conditions": {
    "users": {
      "includeGroups": ["all-employees"]
    },
    "applications": {
      "includeApplications": ["all"]
    },
    "signInRiskLevels": ["medium", "high"],
    "locations": {
      "excludeLocations": ["trusted-offices"]
    },
    "platforms": {
      "includePlatforms": ["all"]
    }
  },
  "grantControls": {
    "operator": "OR",
    "builtInControls": [
      "mfa",
      "compliantDevice"
    ]
  }
}

6.3.2 조건부 액세스 시나리오

시나리오 조건 액세스 제어
회사 네트워크 내 신뢰할 수 있는 IP 범위 MFA 우회 가능
관리형 디바이스 Intune/MDM 규정 준수 접근 허용
BYOD 개인 디바이스 MFA + 제한된 앱 접근
고위험 로그인 이상 행동 감지 MFA + 비밀번호 변경
민감 데이터 접근 DLP 분류 데이터 하드웨어 키 MFA

6.4 페더레이션

페더레이션은 서로 다른 조직 간에 ID 및 인증 정보를 공유하는 방식입니다. 파트너사, 고객사, 클라우드 서비스 간의 SSO를 가능하게 합니다.

6.4.1 페더레이션 신뢰 관계

🤝 페더레이션 유형

  • 허브 앤 스포크: 중앙 IdP가 여러 SP와 연결
  • 메시: 모든 참여자가 서로 직접 연결
  • 브리지: 중개 서비스가 다른 페더레이션 연결

6.4.2 B2B 페더레이션 MFA

// B2B 게스트 사용자 MFA 정책
const b2bMfaPolicy = {
  name: "B2B Guest MFA Policy",

  conditions: {
    userType: "guest",  // 외부 사용자
    applications: ["sensitive-apps"],
  },

  requirements: {
    // 게스트는 항상 MFA 필요
    mfaRequired: true,

    // 홈 테넌트의 MFA 신뢰 여부
    trustHomeTenantMfa: {
      enabled: true,
      // 신뢰할 수 있는 파트너 목록
      trustedPartners: [
        "partner-a.com",
        "partner-b.com"
      ],
      // 미신뢰 파트너는 우리 MFA 요구
      fallbackToResourceMfa: true
    },

    // 최소 인증 강도
    minAuthStrength: "phishing_resistant"
  }
};

6.5 특권 계정 관리 (PAM)

관리자 계정과 같은 특권 계정은 추가적인 보호가 필요합니다. PAM 솔루션과 MFA를 통합하여 특권 접근을 보호합니다.

6.5.1 특권 계정 MFA 요구사항

계정 유형 MFA 요구 세션 제한
도메인 관리자 하드웨어 키 필수 1시간, JIT 접근
클라우드 관리자 FIDO2 필수 4시간, 위치 제한
데이터베이스 관리자 MFA + 승인 워크플로우 작업별 세션
서비스 계정 관리형 ID/인증서 자동 순환

6.5.2 Just-In-Time (JIT) 접근

// JIT 특권 접근 요청 흐름
interface PrivilegedAccessRequest {
  requesterId: string;
  targetRole: string;
  justification: string;
  duration: number;  // 분
  resources: string[];
}

async function requestPrivilegedAccess(request: PrivilegedAccessRequest) {
  // 1. MFA 검증 (하드웨어 키 필수)
  const mfaResult = await verifyMfa({
    userId: request.requesterId,
    requiredMethod: 'hardware_key',
    context: {
      action: 'privileged_access_request',
      role: request.targetRole
    }
  });

  if (!mfaResult.verified) {
    throw new Error('하드웨어 보안 키 인증이 필요합니다');
  }

  // 2. 승인 워크플로우 시작
  const approvalRequest = await createApprovalWorkflow({
    requester: request.requesterId,
    approvers: getApproversForRole(request.targetRole),
    details: request,
    timeout: 3600000  // 1시간 내 승인 필요
  });

  // 3. 승인 완료 시 임시 권한 부여
  approvalRequest.on('approved', async () => {
    const tempCredentials = await grantTemporaryAccess({
      userId: request.requesterId,
      role: request.targetRole,
      duration: request.duration,
      resources: request.resources
    });

    // 4. 감사 로그
    await auditLog({
      event: 'PRIVILEGED_ACCESS_GRANTED',
      actor: request.requesterId,
      role: request.targetRole,
      duration: request.duration,
      approver: approvalRequest.approvedBy
    });

    return tempCredentials;
  });
}

6.6 기업 MFA 배포 전략

📋 단계별 배포 계획

  1. 파일럿 (1-2주): IT 팀 및 보안 팀 대상
  2. 조기 수용자 (2-4주): 기술 친화적 부서
  3. 단계적 확대 (4-8주): 부서별 순차 배포
  4. 전사 적용 (8-12주): 모든 직원 의무화
  5. 강화 (지속): 피싱 저항 방식으로 업그레이드
