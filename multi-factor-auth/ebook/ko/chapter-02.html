인증 요소 유형

다중 인증의 핵심은 서로 다른 유형의 인증 요소를 조합하는 것입니다. 이 장에서는 지식 요소, 소유 요소, 생체 요소의 특성과 각각의 장단점을 상세히 살펴봅니다.

2.1 인증 요소 개요

인증 요소는 사용자의 신원을 증명하기 위해 사용되는 정보 또는 특성입니다. 각 요소 유형은 고유한 특성을 가지며, 다양한 공격 벡터에 대해 서로 다른 저항력을 보입니다.

🧠

지식 요소

Something You Know

사용자만 알고 있는 정보. 비밀번호, PIN, 보안 질문 등이 해당됩니다.

📱

소유 요소

Something You Have

사용자가 물리적으로 소유한 것. 스마트폰, 보안 토큰, 스마트카드 등입니다.

👆

생체 요소

Something You Are

사용자의 고유한 생물학적 특성. 지문, 얼굴, 홍채, 음성 등입니다.

2.1.1 추가 인증 요소

전통적인 세 가지 요소 외에도 현대 시스템에서는 추가적인 요소들을 고려합니다:

요소 유형 설명 예시
위치 요소 사용자의 물리적 위치 GPS, IP 지오로케이션, 사무실 네트워크
시간 요소 접근 시도 시간 업무 시간, 시간대, 접근 패턴
행동 요소 사용자의 행동 패턴 타이핑 패턴, 마우스 움직임, 앱 사용 패턴
컨텍스트 요소 환경 및 상황 정보 디바이스 상태, 네트워크 유형, 요청 리소스

2.2 지식 요소 (Knowledge Factors)

지식 요소는 가장 오래되고 널리 사용되는 인증 방법입니다. 구현이 간단하지만 여러 보안 취약점을 가지고 있습니다.

2.2.1 비밀번호

비밀번호는 가장 일반적인 지식 요소입니다. 효과적인 비밀번호 정책이 중요합니다:

🔑 NIST 비밀번호 가이드라인 (SP 800-63B)

  • 최소 길이: 8자 이상 (16자 이상 권장)
  • 복잡성 규칙 폐지: 특수문자 강제 요구 제거
  • 유출 검사: 알려진 유출 비밀번호 목록과 대조
  • 주기적 변경 폐지: 침해 증거 없이 변경 강제 금지
  • 암호 관리자 허용: 붙여넣기 기능 차단 금지
# 비밀번호 강도 검증 예시 (Python)
import hashlib
import requests

def check_password_strength(password):
    # 길이 검사
    if len(password) < 12:
        return False, "비밀번호는 12자 이상이어야 합니다"

    # Have I Been Pwned API로 유출 검사
    sha1 = hashlib.sha1(password.encode()).hexdigest().upper()
    prefix, suffix = sha1[:5], sha1[5:]

    response = requests.get(f"https://api.pwnedpasswords.com/range/{prefix}")

    if suffix in response.text:
        return False, "이 비밀번호는 유출된 적이 있습니다"

    return True, "안전한 비밀번호입니다"

2.2.2 PIN (개인 식별 번호)

PIN 유형 자릿수 사용 사례 보안 수준
숫자 PIN 4-6자리 ATM, 스마트폰 잠금 낮음 (시도 제한 필수)
영숫자 PIN 6-8자리 보안 시스템 중간
패턴 PIN 4-9포인트 모바일 디바이스 낮음-중간

2.2.3 보안 질문

보안 질문은 계정 복구에 주로 사용되지만, 보안 수준이 낮아 MFA 요소로는 권장되지 않습니다. 답변이 소셜 미디어나 공개 정보에서 추측 가능한 경우가 많습니다.

⚠️ 보안 질문의 문제점

  • 추측 가능: "어머니의 결혼 전 성"은 공개 기록에서 찾을 수 있음
  • 불변성: 답변을 변경할 수 없음 (첫 번째 애완동물 이름)
  • 문화적 편향: 일부 질문은 특정 문화에만 적용
  • 망각: 사용자가 답변을 잊어버리는 경우 빈번

2.3 소유 요소 (Possession Factors)

소유 요소는 사용자가 물리적으로 소유하고 있는 것을 기반으로 합니다. 지식 요소보다 탈취가 어렵지만, 분실 위험이 있습니다.

2.3.1 하드웨어 토큰

토큰 유형 작동 방식 장점 단점
OTP 토큰 일회용 비밀번호 생성 오프라인 작동, 피싱 부분 저항 배터리, 동기화 필요
스마트카드 인증서 기반 인증 높은 보안, 다목적 카드 리더기 필요
USB 보안 키 FIDO2/U2F 프로토콜 피싱 저항, 사용 편의 USB 포트 필요
NFC 키 근거리 무선 통신 모바일 친화적 NFC 지원 필요

2.3.2 소프트웨어 토큰

스마트폰 앱을 통해 구현되는 소프트웨어 기반 토큰입니다. 하드웨어 토큰에 비해 배포와 관리가 용이합니다. 

# TOTP 알고리즘 구현 (Python)
import hmac
import hashlib
import struct
import time

def generate_totp(secret, time_step=30):
    """
    시간 기반 일회용 비밀번호 생성
    RFC 6238 구현
    """
    # 현재 시간을 time_step으로 나눈 카운터
    counter = int(time.time() // time_step)

    # 카운터를 8바이트 빅엔디언으로 변환
    counter_bytes = struct.pack('>Q', counter)

    # HMAC-SHA1 계산
    hmac_hash = hmac.new(
        secret.encode(),
        counter_bytes,
        hashlib.sha1
    ).digest()

    # Dynamic Truncation
    offset = hmac_hash[-1] & 0x0F
    code = struct.unpack('>I', hmac_hash[offset:offset+4])[0]
    code = (code & 0x7FFFFFFF) % 1000000

    return f"{code:06d}"

# 사용 예시
secret = "JBSWY3DPEHPK3PXP"  # Base32 인코딩된 시크릿
otp = generate_totp(secret)
print(f"현재 OTP: {otp}")

장점 단점
사용 편의성 (원터치) MFA 피로 공격에 취약

2.4 생체 요소 (Inherence Factors)

생체 요소는 사용자의 고유한 생물학적 또는 행동적 특성을 사용합니다. 분실이나 공유가 불가능하여 높은 보안성을 제공합니다.

2.4.1 생리적 생체 인식

유형 FAR (오수락률) FRR (오거부률) 사용 사례
지문 인식 0.001% 0.1% 스마트폰, 출입 통제
얼굴 인식 0.1% 2.5% 스마트폰, 공항 보안
홍채 인식 0.0001% 0.2% 고보안 시설, 국경 통제
정맥 패턴 0.0001% 0.01% ATM, 의료 시스템
음성 인식 2% 5% 콜센터, 음성 비서

2.4.2 행동적 생체 인식

행동적 생체 인식은 사용자의 행동 패턴을 분석하여 지속적으로 신원을 확인합니다. 명시적인 인증 단계 없이 백그라운드에서 작동할 수 있습니다.

  • 키스트로크 다이나믹스: 타이핑 속도, 리듬, 압력 패턴
  • 마우스 제스처: 마우스 이동 속도, 클릭 패턴, 궤적
  • 터치스크린 제스처: 스와이프 속도, 압력, 각도
  • 걸음걸이 분석: 보행 패턴 (웨어러블 디바이스)
  • 앱 사용 패턴: 앱 사용 순서, 시간대, 빈도

2.4.3 생체 인식 보안 고려사항

🔒 생체 인식 보안 모범 사례

  • 라이브니스 검출: 사진이나 영상 재생 공격 방지
  • 템플릿 보호: 생체 데이터의 안전한 저장 (취소 가능한 생체 인식)
  • 프레젠테이션 공격 방어: 가짜 지문, 마스크 등 탐지
  • 다중 모달: 여러 생체 특성 조합으로 정확도 향상
  • 대체 수단: 생체 인식 실패 시 백업 인증 방법

2.5 요소 조합 전략

효과적인 MFA는 서로 다른 유형의 요소를 조합하여 각 요소의 약점을 보완합니다. 동일 유형의 요소를 여러 개 사용하는 것은 진정한 MFA가 아닙니다.

2.5.1 권장 조합

보안 수준 요소 조합 사용 사례
기본 비밀번호 + SMS OTP 일반 소비자 서비스
향상 비밀번호 + TOTP 앱 비즈니스 애플리케이션
높음 비밀번호 + 하드웨어 키 금융, 의료 시스템
매우 높음 비밀번호 + 하드웨어 키 + 생체 정부, 국방, 핵심 인프라
