MFA 기술

다중 인증을 구현하는 다양한 기술과 프로토콜을 살펴봅니다. TOTP, 푸시 알림, 하드웨어 토큰, SMS 기반 인증 등 각 기술의 작동 원리와 보안 특성을 이해합니다.

3.1 TOTP (시간 기반 일회용 비밀번호)

TOTP(Time-based One-Time Password)는 RFC 6238에 정의된 표준으로, 현재 시간과 공유 비밀 키를 기반으로 일회용 비밀번호를 생성합니다.

3.1.1 TOTP 작동 원리

# TOTP 알고리즘 상세

1. 시간 카운터 계산
   T = floor((현재_유닉스_시간 - T0) / X)
   - T0: 기준 시간 (일반적으로 0)
   - X: 시간 스텝 (기본값 30초)

2. HMAC-SHA1 해시 계산
   HS = HMAC-SHA1(K, T)
   - K: Base32 인코딩된 공유 비밀 키
   - T: 8바이트 빅엔디언 카운터

3. Dynamic Truncation
   offset = HS[19] & 0xf
   code = (HS[offset] & 0x7f) << 24
        | (HS[offset+1] & 0xff) << 16
        | (HS[offset+2] & 0xff) << 8
        | (HS[offset+3] & 0xff)

4. 코드 생성
   OTP = code mod 10^digits  # 보통 6자리

3.1.2 TOTP 구현 예시

// TypeScript TOTP 구현
import * as crypto from 'crypto';

interface TOTPOptions {
  secret: string;      // Base32 인코딩된 시크릿
  digits?: number;     // 자릿수 (기본: 6)
  period?: number;     // 시간 스텝 (기본: 30초)
  algorithm?: string;  // 해시 알고리즘 (기본: SHA1)
}

function generateTOTP(options: TOTPOptions): string {
  const {
    secret,
    digits = 6,
    period = 30,
    algorithm = 'sha1'
  } = options;

  // Base32 디코딩
  const key = base32Decode(secret);

  // 시간 카운터 계산
  const counter = Math.floor(Date.now() / 1000 / period);
  const counterBuffer = Buffer.alloc(8);
  counterBuffer.writeBigInt64BE(BigInt(counter));

  // HMAC 계산
  const hmac = crypto.createHmac(algorithm, key);
  hmac.update(counterBuffer);
  const hash = hmac.digest();

  // Dynamic Truncation
  const offset = hash[hash.length - 1] & 0xf;
  const code = (hash[offset] & 0x7f) << 24
             | (hash[offset + 1] & 0xff) << 16
             | (hash[offset + 2] & 0xff) << 8
             | (hash[offset + 3] & 0xff);

  // 자릿수에 맞춰 반환
  const otp = code % Math.pow(10, digits);
  return otp.toString().padStart(digits, '0');
}

// 검증 함수 (시간 드리프트 허용)
function verifyTOTP(token: string, options: TOTPOptions, window = 1): boolean {
  const period = options.period || 30;

  for (let i = -window; i <= window; i++) {
    const testTime = Date.now() + (i * period * 1000);
    const expected = generateTOTP({
      ...options,
      // 시간 오프셋 적용
    });

    if (token === expected) {
      return true;
    }
  }
  return false;
}

3.2 푸시 알림 인증

푸시 기반 인증은 사용자의 스마트폰에 인증 요청을 전송하고, 사용자가 승인하면 인증이 완료되는 방식입니다.

3.2.1 푸시 인증 흐름

# 푸시 인증 시퀀스

사용자        클라이언트        서버           푸시 서비스      모바일 앱
  │              │              │                 │              │
  │──로그인 요청──▶│              │                 │              │
  │              │──인증 요청───▶│                 │              │
  │              │              │──푸시 전송──────▶│              │
  │              │              │                 │──알림 표시──▶│
  │              │              │                 │              │
  │              │◀─────────────────대기─────────────────────────│
  │              │              │                 │              │
  │              │              │                 │◀──승인/거부──│
  │              │              │◀──응답 전달─────│              │
  │              │◀──인증 결과──│                 │              │

3.2.2 넘버 매칭 (Number Matching)

MFA 피로 공격을 방지하기 위해, 로그인 화면에 표시된 숫자를 모바일 앱에서 입력하도록 요구하는 방식입니다.

🔢 넘버 매칭 구현

// 넘버 매칭 인증 흐름
1. 서버가 2자리 랜덤 숫자 생성 (예: 47)
2. 웹 로그인 화면에 숫자 표시: "47을 입력하세요"
3. 푸시 알림에 로그인 컨텍스트 표시:
   - 앱: "Example Corp 로그인"
   - 위치: "서울, 대한민국"
   - 디바이스: "Chrome on Windows"
   - 시간: "2025-01-15 14:30 KST"
4. 사용자가 앱에서 "47" 입력
5. 서버에서 숫자 일치 확인 후 인증 완료

3.3.2 FIDO2/WebAuthn 프로토콜

// WebAuthn 등록 (Registration) 예시
async function registerWebAuthn() {
  // 서버에서 챌린지 및 옵션 가져오기
  const options = await fetch('/webauthn/register/options').then(r => r.json());

  // 자격 증명 생성
  const credential = await navigator.credentials.create({
    publicKey: {
      challenge: Uint8Array.from(options.challenge, c => c.charCodeAt(0)),
      rp: {
        name: "WIA 보안 시스템",
        id: "wia-security.example.com"
      },
      user: {
        id: Uint8Array.from(options.userId, c => c.charCodeAt(0)),
        name: options.userName,
        displayName: options.displayName
      },
      pubKeyCredParams: [
        { alg: -7, type: "public-key" },   // ES256
        { alg: -257, type: "public-key" }  // RS256
      ],
      authenticatorSelection: {
        authenticatorAttachment: "cross-platform",  // 외부 보안 키
        userVerification: "required",               // PIN/생체 필수
        residentKey: "preferred"                    // 패스키 지원
      },
      timeout: 60000,
      attestation: "direct"
    }
  });

  // 서버로 자격 증명 전송
  await fetch('/webauthn/register/verify', {
    method: 'POST',
    body: JSON.stringify({
      id: credential.id,
      rawId: bufferToBase64(credential.rawId),
      response: {
        clientDataJSON: bufferToBase64(credential.response.clientDataJSON),
        attestationObject: bufferToBase64(credential.response.attestationObject)
      },
      type: credential.type
    })
  });
}

3.4 SMS 기반 인증

SMS OTP는 가장 널리 배포된 2FA 방법이지만, 여러 보안 취약점으로 인해 고가치 계정에는 권장되지 않습니다.

3.4.1 SMS 인증의 취약점

📱 SIM 스와핑

공격자가 이동통신사를 속여 피해자의 전화번호를 자신의 SIM 카드로 이전하는 공격입니다.

📡 SS7 공격

통신 프로토콜의 취약점을 이용해 SMS 메시지를 가로채는 공격입니다.

🦠 모바일 악성코드

디바이스에 설치된 악성 앱이 SMS 메시지를 읽고 전달하는 공격입니다.

🎭 실시간 피싱

피싱 사이트가 실시간으로 OTP를 요청하고 실제 서비스에 전달하는 공격입니다.

3.4.2 SMS 대안 권장사항

현재 방식 권장 대안 이점
SMS OTP TOTP 앱 오프라인 작동, SIM 스와핑 면역
SMS OTP 푸시 알림 + 넘버 매칭 컨텍스트 확인, 피싱 저항
SMS OTP 하드웨어 키 완전한 피싱 저항

3.5 이메일 기반 인증

이메일로 인증 링크나 OTP를 전송하는 방식은 MFA보다는 비밀번호 재설정이나 초기 등록에 주로 사용됩니다.

3.5.1 매직 링크 (Magic Link)

# 매직 링크 인증 흐름

1. 사용자가 이메일 주소 입력
2. 서버가 고유 토큰 생성 (JWT 또는 랜덤 문자열)
3. 토큰을 포함한 링크를 이메일로 전송
   https://app.example.com/auth/magic?token=eyJhbG...
4. 사용자가 링크 클릭
5. 서버가 토큰 검증 (서명, 만료, 사용 여부)
6. 검증 성공 시 세션 생성

# 보안 고려사항
- 토큰 만료 시간: 15-30분
- 일회용 사용 (사용 후 무효화)
- IP 바인딩 (선택적)
- 레이트 리미팅
