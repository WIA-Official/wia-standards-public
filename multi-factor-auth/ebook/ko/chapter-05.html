구현 패턴

MFA 시스템의 실제 구현 패턴을 살펴봅니다. 사용자 등록 흐름, 복구 프로세스, 단계별 인증(Step-up Authentication), 그리고 다양한 아키텍처 패턴을 다룹니다.

5.1 사용자 등록 흐름

MFA 등록은 사용자가 인증 수단을 시스템에 등록하는 과정입니다. 보안과 사용자 경험의 균형을 맞추는 것이 중요합니다.

5.1.1 등록 흐름 설계

│                    MFA 등록 흐름                                 │

  │                          │                          │

5.1.2 등록 API 구현

// TOTP 등록 API (Node.js/Express)
import speakeasy from 'speakeasy';
import QRCode from 'qrcode';

// 1단계: TOTP 시크릿 생성
app.post('/api/mfa/totp/setup', async (req, res) => {
  const userId = req.user.id;

  // 시크릿 생성
  const secret = speakeasy.generateSecret({
    name: `WIA보안시스템:${req.user.email}`,
    issuer: 'WIA Security',
    length: 32
  });

  // 임시 저장 (아직 검증되지 않음)
  await db.users.update(userId, {
    totpSecretPending: secret.base32,
    totpSetupAt: new Date()
  });

  // QR 코드 생성
  const qrCodeUrl = await QRCode.toDataURL(secret.otpauth_url);

  res.json({
    secret: secret.base32,  // 수동 입력용
    qrCode: qrCodeUrl,
    expiresIn: 600  // 10분 내 검증 필요
  });
});

// 2단계: OTP 검증 및 등록 완료
app.post('/api/mfa/totp/verify', async (req, res) => {
  const { token } = req.body;
  const userId = req.user.id;

  const user = await db.users.findById(userId);

  // 토큰 검증
  const verified = speakeasy.totp.verify({
    secret: user.totpSecretPending,
    encoding: 'base32',
    token: token,
    window: 1  // ±30초 허용
  });

  if (!verified) {
    return res.status(400).json({ error: '잘못된 인증 코드입니다' });
  }

  // 등록 완료
  await db.users.update(userId, {
    totpSecret: user.totpSecretPending,
    totpSecretPending: null,
    totpEnabled: true,
    mfaEnabled: true
  });

  // 복구 코드 생성
  const recoveryCodes = generateRecoveryCodes(8);
  await db.recoveryCodes.create(userId, recoveryCodes);

  res.json({
    success: true,
    recoveryCodes: recoveryCodes,
    message: '복구 코드를 안전한 곳에 저장하세요'
  });
});

5.2 복구 프로세스

사용자가 MFA 디바이스를 분실하거나 접근할 수 없을 때를 대비한 복구 메커니즘이 필수적입니다.

5.2.1 복구 코드

특성 권장 사항
개수 8-16개
길이 8-12자 (영숫자)
사용 방식 일회용 (사용 후 무효화)
저장 해시하여 저장 (bcrypt)
표시 등록 시 한 번만 표시
재생성 MFA 인증 후 가능

5.2.2 복구 흐름 구현

// 복구 코드 생성
function generateRecoveryCodes(count: number): string[] {
  const codes: string[] = [];
  const charset = 'ABCDEFGHJKLMNPQRSTUVWXYZ23456789';  // 혼동 문자 제외

  for (let i = 0; i < count; i++) {
    let code = '';
    for (let j = 0; j < 8; j++) {
      code += charset[crypto.randomInt(charset.length)];
      if (j === 3) code += '-';  // 가독성을 위한 하이픈
    }
    codes.push(code);
  }
  return codes;
}

// 복구 코드로 로그인
app.post('/api/auth/recover', async (req, res) => {
  const { email, password, recoveryCode } = req.body;

  // 1. 비밀번호 확인
  const user = await authenticateUser(email, password);
  if (!user) {
    return res.status(401).json({ error: '자격 증명이 올바르지 않습니다' });
  }

  // 2. 복구 코드 확인
  const normalizedCode = recoveryCode.replace(/-/g, '').toUpperCase();
  const codes = await db.recoveryCodes.findByUserId(user.id);

  const validCode = codes.find(c =>
    !c.usedAt && bcrypt.compareSync(normalizedCode, c.hash)
  );

  if (!validCode) {
    await logSecurityEvent(user.id, 'RECOVERY_CODE_FAILED');
    return res.status(401).json({ error: '유효하지 않은 복구 코드입니다' });
  }

  // 3. 복구 코드 사용 처리
  await db.recoveryCodes.markUsed(validCode.id);

  // 4. 세션 생성 및 MFA 재등록 요구
  const session = await createSession(user.id, {
    requireMfaReenrollment: true,
    recoveryUsed: true
  });

  await logSecurityEvent(user.id, 'RECOVERY_CODE_USED');

  res.json({
    sessionToken: session.token,
    remainingCodes: codes.filter(c => !c.usedAt).length - 1,
    message: '새 MFA 방법을 등록해야 합니다'
  });
});

5.3 단계별 인증 (Step-up Authentication)

단계별 인증은 민감한 작업을 수행할 때 추가 인증을 요구하는 패턴입니다. 사용자 경험을 해치지 않으면서 보안을 강화합니다.

5.3.1 Step-up 인증 시나리오

작업 기본 인증 Step-up 요구
프로필 조회 세션 토큰 없음
비밀번호 변경 세션 토큰 현재 비밀번호 + MFA
결제 정보 변경 세션 토큰 MFA (최근 5분 이내)
대량 자금 이체 세션 토큰 MFA + 추가 승인
계정 삭제 세션 토큰 비밀번호 + MFA + 대기 기간

5.3.2 Step-up 구현

// Step-up 인증 미들웨어
interface StepUpOptions {
  maxAge?: number;        // MFA 인증 후 유효 시간 (초)
  requiredLevel?: string; // 필요한 인증 수준
}

function requireStepUp(options: StepUpOptions = {}) {
  const { maxAge = 300, requiredLevel = 'mfa' } = options;

  return async (req, res, next) => {
    const session = req.session;

    // 마지막 MFA 인증 시간 확인
    const lastMfaAt = session.lastMfaAuthAt;
    const elapsed = (Date.now() - lastMfaAt) / 1000;

    if (!lastMfaAt || elapsed > maxAge) {
      return res.status(403).json({
        error: 'STEP_UP_REQUIRED',
        message: '이 작업을 수행하려면 추가 인증이 필요합니다',
        requiredLevel: requiredLevel,
        authUrl: '/api/auth/step-up'
      });
    }

    // 인증 수준 확인
    if (session.authLevel < requiredLevel) {
      return res.status(403).json({
        error: 'INSUFFICIENT_AUTH_LEVEL',
        message: '더 강력한 인증이 필요합니다',
        requiredLevel: requiredLevel,
        currentLevel: session.authLevel
      });
    }

    next();
  };
}

// 사용 예시
app.post('/api/settings/payment',
  authenticate,
  requireStepUp({ maxAge: 300 }),  // 5분 이내 MFA 필요
  async (req, res) => {
    // 결제 정보 업데이트 로직
  }
);

app.post('/api/transfers/large',
  authenticate,
  requireStepUp({ maxAge: 60, requiredLevel: 'hardware_key' }),
  async (req, res) => {
    // 대량 이체 로직 (1분 이내 하드웨어 키 인증 필요)
  }
);

5.4 아키텍처 패턴

5.4.1 중앙화 vs 분산화

🏗️ MFA 아키텍처 비교

패턴 장점 단점
중앙화 (IdP) 일관된 정책, 단일 관리 단일 실패점, 지연
분산화 (앱별) 유연성, 독립적 복잡성, 일관성 부족
하이브리드 균형 잡힌 접근 설계 복잡성

5.4.2 마이크로서비스 MFA

// MFA 서비스 인터페이스 (gRPC)
service MFAService {
  // 인증 방법 관리
  rpc ListAuthenticators(ListRequest) returns (AuthenticatorList);
  rpc RegisterAuthenticator(RegisterRequest) returns (RegisterResponse);
  rpc RemoveAuthenticator(RemoveRequest) returns (RemoveResponse);

  // 인증 수행
  rpc InitiateChallenge(ChallengeRequest) returns (Challenge);
  rpc VerifyChallenge(VerifyRequest) returns (VerifyResponse);

  // 복구
  rpc GenerateRecoveryCodes(RecoveryRequest) returns (RecoveryCodes);
  rpc ValidateRecoveryCode(ValidateRequest) returns (ValidateResponse);
}

message ChallengeRequest {
  string user_id = 1;
  string session_id = 2;
  AuthenticatorType preferred_type = 3;
  map context = 4;  // IP, 디바이스 등
}

message Challenge {
  string challenge_id = 1;
  AuthenticatorType type = 2;
  bytes challenge_data = 3;  // TOTP면 없음, WebAuthn이면 챌린지
  int64 expires_at = 4;
}

5.5 신뢰할 수 있는 디바이스

자주 사용하는 디바이스를 신뢰 디바이스로 등록하면 MFA 프롬프트 빈도를 줄여 사용자 경험을 개선할 수 있습니다.

5.5.1 디바이스 핑거프린팅

// 디바이스 식별자 생성
interface DeviceFingerprint {
  userAgent: string;
  screenResolution: string;
  timezone: string;
  language: string;
  platform: string;
  cookiesEnabled: boolean;
  localStorage: boolean;
  hardwareConcurrency: number;
  deviceMemory?: number;
  // 캔버스 핑거프린트는 프라이버시 우려로 제외 권장
}

async function generateDeviceId(fp: DeviceFingerprint): Promise {
  const data = JSON.stringify(fp);
  const hash = await crypto.subtle.digest(
    'SHA-256',
    new TextEncoder().encode(data)
  );
  return btoa(String.fromCharCode(...new Uint8Array(hash)));
}

// 신뢰 디바이스 등록
app.post('/api/devices/trust', async (req, res) => {
  const { deviceId, name } = req.body;
  const userId = req.user.id;

  // MFA 검증 필요
  if (!req.session.mfaVerified) {
    return res.status(403).json({ error: 'MFA 인증이 필요합니다' });
  }

  await db.trustedDevices.create({
    userId,
    deviceId: await hashDeviceId(deviceId),
    name: name || 'Unknown Device',
    createdAt: new Date(),
    expiresAt: addDays(new Date(), 30)  // 30일 후 만료
  });

  res.json({ success: true });
});

5.5.2 적응형 MFA 정책

조건 위험 점수 MFA 요구
신뢰 디바이스 + 알려진 위치 낮음 선택적
신뢰 디바이스 + 새 위치 중간 필수
새 디바이스 + 알려진 위치 중간 필수
새 디바이스 + 새 위치 높음 필수 + 알림
불가능한 이동 감지 매우 높음 차단 + 경고
