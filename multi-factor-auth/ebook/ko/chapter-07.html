제7장

보안 고려사항

MFA는 보안을 크게 강화하지만 완벽하지는 않습니다. MFA를 우회하려는 공격 벡터, 피싱 저항 인증, 위험 기반 인증 등 MFA 시스템의 보안을 강화하는 방법을 알아봅니다.

7.1 MFA 우회 공격

공격자들은 MFA를 우회하기 위한 다양한 기법을 발전시켜 왔습니다. 주요 공격 벡터와 대응 방법을 이해하는 것이 중요합니다.

🎭 실시간 피싱 (AiTM)

역방향 프록시를 통해 자격증명과 MFA 토큰을 실시간으로 가로채는 공격입니다.

위험도: 매우 높음

📱 SIM 스와핑

이동통신사를 속여 피해자의 전화번호를 공격자의 SIM으로 이전하는 공격입니다.

위험도: 높음

😫 MFA 피로 공격

반복적인 푸시 알림으로 사용자가 실수로 승인하도록 유도하는 공격입니다.

위험도: 중간-높음

🔑 세션 하이재킹

MFA 완료 후 세션 토큰을 탈취하여 인증을 우회하는 공격입니다.

위험도: 중간

7.1.1 Adversary-in-the-Middle (AiTM) 공격

AiTM 공격은 가장 정교한 MFA 우회 기법 중 하나입니다. 공격자는 역방향 프록시를 설정하여 실제 서비스처럼 보이는 피싱 사이트를 운영합니다.

공격 단계 설명 방어
유인 피싱 이메일로 가짜 사이트 유도 보안 인식 교육, 이메일 필터링
자격증명 캡처 사용자가 가짜 사이트에 로그인 비밀번호 관리자 (URL 검증)
MFA 중계 MFA 프롬프트를 실시간 전달 FIDO2 (원본 바인딩)
세션 탈취 인증 후 세션 쿠키 캡처 토큰 바인딩, 짧은 세션

7.1.2 MFA 피로 공격 방어

# MFA 피로 공격 방어 전략

1. 넘버 매칭 구현
   - 로그인 화면에 숫자 표시
   - 사용자가 앱에서 동일한 숫자 입력
   - 무작위 승인 방지

2. 컨텍스트 정보 표시
   - 로그인 시도 위치
   - 디바이스 정보
   - 애플리케이션 이름
   - 시간

3. 레이트 리미팅
   - 시간당 푸시 알림 수 제한
   - 연속 거부 시 잠금
   - 보안 팀 알림

4. 피싱 저항 대안
   - 하드웨어 보안 키 사용
   - 푸시 알림 제거 옵션

⚠️ 실제 사례: Uber 침해 (2022)

공격자는 Uber 계약업체의 자격증명을 획득한 후, 1시간 이상 MFA 푸시 알림을 반복 전송했습니다. 결국 계약업체가 피로를 느끼고 승인했고, 공격자는 내부 시스템에 접근하여 민감한 데이터를 획득했습니다.

교훈: 넘버 매칭과 레이트 리미팅이 있었다면 방지 가능했습니다.

7.2 피싱 저항 인증

피싱 저항 MFA는 피싱 공격에 면역인 인증 방식입니다. FIDO2/WebAuthn이 대표적인 피싱 저항 프로토콜입니다.

7.2.1 FIDO2의 피싱 저항 원리

🔒 FIDO2가 피싱을 막는 방법

메커니즘 설명
원본 바인딩 자격증명이 특정 도메인에 암호학적으로 연결됨. 가짜 사이트에서는 유효한 서명 생성 불가
공개키 암호화 개인키가 인증기를 떠나지 않음. 가로챌 수 있는 비밀 정보 없음
채널 바인딩 TLS 채널 정보가 인증에 포함. 프록시 공격 탐지
사용자 확인 물리적 상호작용 필요 (터치, 생체). 원격 공격 불가

7.2.2 WebAuthn 구현 예시

// WebAuthn 인증 흐름
async function authenticateWithWebAuthn() {
  // 1. 서버에서 챌린지 가져오기
  const options = await fetch('/api/webauthn/authenticate/options')
    .then(r => r.json());

  // 2. 브라우저가 자격증명 요청
  const assertion = await navigator.credentials.get({
    publicKey: {
      challenge: base64ToArrayBuffer(options.challenge),
      allowCredentials: options.allowCredentials.map(c => ({
        id: base64ToArrayBuffer(c.id),
        type: 'public-key'
      })),
      userVerification: 'required',
      timeout: 60000
    }
  });

  // 3. 서버에 인증 데이터 전송
  const response = await fetch('/api/webauthn/authenticate/verify', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      id: assertion.id,
      rawId: arrayBufferToBase64(assertion.rawId),
      response: {
        authenticatorData: arrayBufferToBase64(
          assertion.response.authenticatorData
        ),
        clientDataJSON: arrayBufferToBase64(
          assertion.response.clientDataJSON
        ),
        signature: arrayBufferToBase64(assertion.response.signature)
      },
      type: assertion.type
    })
  });

  return response.json();
}

7.3 위험 기반 인증 (RBA)

위험 기반 인증은 컨텍스트 신호를 분석하여 동적으로 인증 요구사항을 조정합니다. 저위험 상황에서는 마찰을 줄이고, 고위험 상황에서는 보안을 강화합니다.

7.3.1 위험 신호

신호 유형 예시 위험 영향
디바이스 신뢰 알려진/관리형 디바이스 신뢰 디바이스 → 위험 감소
위치 IP 지오로케이션, 불가능한 이동 새 국가 → 위험 증가
행동 로그인 시간, 앱 사용 패턴 이상 행동 → 위험 증가
네트워크 VPN, Tor, 기업 네트워크 익명 네트워크 → 위험 증가

7.3.2 적응형 인증 구현

// 위험 기반 인증 엔진
interface AuthContext {
  userId: string;
  deviceId: string;
  ipAddress: string;
  userAgent: string;
  timestamp: Date;
  requestedResource: string;
}

interface RiskScore {
  score: number;  // 0.0 ~ 1.0
  factors: RiskFactor[];
}

async function evaluateRisk(context: AuthContext): Promise {
  const factors: RiskFactor[] = [];
  let score = 0;

  // 디바이스 신뢰 평가
  const device = await getDeviceTrust(context.deviceId);
  if (device.isManaged) {
    score -= 0.2;
    factors.push({ type: 'device_managed', impact: -0.2 });
  }
  if (device.isKnown) {
    score -= 0.15;
    factors.push({ type: 'device_known', impact: -0.15 });
  }

  // 위치 분석
  const location = await geolocate(context.ipAddress);
  const lastLocation = await getLastLoginLocation(context.userId);

  if (isImpossibleTravel(location, lastLocation, context.timestamp)) {
    score += 0.5;
    factors.push({ type: 'impossible_travel', impact: 0.5 });
  }

  if (location.isNewCountry) {
    score += 0.25;
    factors.push({ type: 'new_country', impact: 0.25 });
  }

  // 네트워크 평판
  const network = await analyzeNetwork(context.ipAddress);
  if (network.isTorExit) {
    score += 0.4;
    factors.push({ type: 'tor_exit', impact: 0.4 });
  }

  return {
    score: Math.max(0, Math.min(1, score)),
    factors
  };
}

async function getRequiredAuth(risk: RiskScore): Promise {
  if (risk.score < 0.2) {
    return { mfaRequired: false };
  } else if (risk.score < 0.5) {
    return { mfaRequired: true, methods: ['totp', 'push'] };
  } else if (risk.score < 0.8) {
    return { mfaRequired: true, methods: ['hardware_key'] };
  } else {
    return { blocked: true, reason: '의심스러운 활동 감지' };
  }
}

7.4 세션 보안

MFA 이후의 세션 보안도 중요합니다. 세션 토큰이 탈취되면 MFA를 완전히 우회할 수 있습니다.

7.4.1 세션 보호 전략

🔐 세션 보안 체크리스트

  • 짧은 세션 수명: 민감 작업은 짧은 세션 타임아웃
  • 디바이스 바인딩: 세션을 특정 디바이스에 연결
  • 지속적 검증: 세션 중 컨텍스트 변화 모니터링
  • Step-up 인증: 민감 작업 시 재인증 요구
  • 이상 탐지: 비정상 세션 활동 감지

7.5 복구 프로세스 보안

계정 복구는 종종 보안의 약한 고리가 됩니다. 복구 프로세스가 MFA만큼 강력해야 합니다.

7.5.1 안전한 복구 방법

방법 보안 수준 권장 사항
백업 보안 키 매우 높음 등록 시 2개 이상 키 등록, 오프라인 보관
복구 코드 높음 일회용, 해시 저장, 안전한 곳에 보관
관리자 복구 중간 신원 확인 필수, 감사 로그, 대기 시간
이메일 복구 낮음 가능하면 피함, 이메일 계정도 MFA 보호
보안 질문 매우 낮음 사용 금지 권장
