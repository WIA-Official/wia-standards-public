Chapter 02

Authentication Factors

Knowledge, Possession, and Inherence - The Building Blocks of MFA

2.1 The Three Factor Categories

Authentication factors are credentials used to verify a user's identity. For authentication to be truly "multi-factor," it must combine factors from at least two different categories. The three fundamental categories established by NIST and ISO standards are:

🧠
Knowledge Factors

Something you know

  • Passwords
  • PINs
  • Security questions
  • Passphrases
📱
Possession Factors

Something you have

  • Smartphones
  • Hardware tokens
  • Smart cards
  • Security keys
👤
Inherence Factors

Something you are

  • Fingerprints
  • Facial recognition
  • Voice patterns
  • Behavioral biometrics
⚠️ Common Misconception

Using two factors from the same category (e.g., password + security question) is NOT multi-factor authentication—it's two-step verification. True MFA requires factors from different categories.

2.2 Knowledge Factors: Something You Know

Knowledge factors are credentials stored in the user's memory. While the most common authentication method, they're also the most vulnerable to various attacks.

Password-Based Authentication

Password Vulnerabilities

  • Phishing: Users tricked into revealing passwords to fake login pages
  • Credential Stuffing: Automated testing of leaked username/password pairs
  • Brute Force: Systematic trying of all possible combinations
  • Rainbow Tables: Pre-computed hash tables for rapid cracking
  • Keylogging: Malware recording every keystroke
  • Shoulder Surfing: Visual observation during password entry
  • Social Engineering: Manipulating users to disclose passwords

Security Questions: A Flawed Approach

Security questions have fallen out of favor in modern authentication due to fundamental weaknesses:

Question TypeExampleVulnerabilityAttack Success Rate
Personal History "Mother's maiden name?" Public records, social media 40-60%
Biographical "City of birth?" Limited options, guessable 20-30%
Favorites "Favorite color?" Extremely limited options 30-50%
Pet Names "First pet's name?" Social media exposure 25-40%
⚠️ WIA-SEC Position on Security Questions

Security questions are NOT recommended as an authentication factor in the WIA-SEC standard. They should only be used as account recovery hints, never as a primary or secondary authentication method. Consider backup codes or email-based recovery instead.

2.3 Possession Factors: Something You Have

Possession factors require the user to physically possess a device or object. These are significantly more secure than knowledge factors because they cannot be guessed or socially engineered as easily.

Smartphone-Based Authentication

Smartphones have become the most popular possession factor due to their ubiquity and versatility.

MethodHow It WorksSecurity LevelUser ExperiencePhishing Resistance
SMS OTP 6-digit code sent via text message Low-Medium Good ❌ No (SIM swap attacks)

Push Notification Approve/deny prompt on registered device Medium-High Excellent ⚠️ Vulnerable to fatigue
Number Matching Enter displayed number in app High Very Good ✅ Yes
WebAuthn/FIDO2 Cryptographic challenge-response Very High Excellent ✅ Yes (phishing-proof)

2.4 Inherence Factors: Something You Are

Biometric authentication leverages unique physical or behavioral characteristics. These factors cannot be forgotten, shared easily, or stolen in the traditional sense, making them powerful authentication tools.

FAR (False Acceptance Rate): Probability an unauthorized user is incorrectly accepted

FRR (False Rejection Rate): Probability an authorized user is incorrectly rejected

Privacy Considerations

⚠️ Biometric Data Protection

Under GDPR and similar regulations, biometric data is classified as "special category" personal data requiring enhanced protection. The WIA-SEC standard mandates:

  • On-device storage only: Biometric templates must never leave the user's device
  • Local matching: Authentication matching occurs on the device, not in the cloud
  • Irreversible transformation: Store mathematical representations, not actual biometric images
  • User consent: Explicit opt-in required with clear explanation of data use
  • Revocability: Users must be able to delete biometric data at any time

NIST Authenticator Assurance Levels (AAL)

NIST Special Publication 800-63B defines three authentication assurance levels:

LevelRequirementsTypical Use CasesExample Combinations

AAL2 Multi-factor with at least one possession or inherence factor Moderate-risk: financial, healthcare, PII Password + TOTP, Password + SMS, Biometric + PIN
AAL3 Hardware-based cryptographic authenticator High-risk: national security, critical infrastructure PIV card + PIN, FIDO2 security key + biometric

✅ WIA-SEC Recommendation

For optimal security-usability balance, implement passwordless authentication with FIDO2 + biometric as the primary path, with password + TOTP as a fallback for users without compatible devices.

2.7 Chapter Summary

📋 Key Takeaways

1
Three factor categories form the foundation

Knowledge (something you know), Possession (something you have), and Inherence (something you are) must be combined from different categories for true MFA.

2
Not all factors are created equal

FIDO2 hardware keys and device-bound biometrics provide the strongest protection, while SMS OTP and security questions are weak.

3
Phishing resistance is critical

Only cryptographically-bound authenticators (FIDO2, WebAuthn) are truly phishing-proof.

4
Biometric data must stay on-device

Privacy regulations and best practices require biometric templates to remain in secure device enclaves, never transmitted to servers.

5
NIST AAL framework guides implementation

AAL2 (multi-factor) should be the baseline for any system handling sensitive data.
