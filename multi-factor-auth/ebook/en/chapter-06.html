6.1 Single Sign-On (SSO) with MFA

Single Sign-On allows users to authenticate once and gain access to multiple applications without re-entering credentials. When combined with MFA, SSO provides both security and convenience at enterprise scale.

SSO Protocols

ProtocolPrimary Use CaseMFA SupportComplexityAdoption
SAML 2.0 Enterprise web applications ✅ Full support High (XML-based) Very High (legacy standard)
OAuth 2.0 API authorization, mobile apps ⚠️ Authorization only, not authentication Medium Very High
OpenID Connect (OIDC) Modern web and mobile applications ✅ Full support (via ACR values) Medium (JSON-based) High (growing rapidly)
Kerberos Windows Active Directory environments ⚠️ Limited (requires PKINIT) High (enterprise infrastructure) High (Windows domains)

6.2 Identity Federation

Federation enables users to access resources across organizational boundaries using a single set of credentials. This is essential for B2B partnerships, mergers, and multi-organization collaboration.

Federation Scenarios

ScenarioDescriptionMFA ConsiderationsCommon Protocols
Enterprise to SaaS Employees access third-party applications Enterprise IdP enforces MFA policy SAML 2.0, OIDC
B2B Partner Access Partner employees access shared resources Trust partner IdP MFA or require step-up SAML 2.0 federation
Consumer Social Login Users log in with Google/Facebook/Apple Rely on social provider MFA OAuth 2.0 + OIDC
Government Cross-Agency Federal employees access multiple agencies PIV card required (AAL3) SAML 2.0 with X.509
Academic Federation Students access resources across universities Home institution MFA policy applies SAML 2.0 (Shibboleth, eduGAIN)

Trust and MFA in Federation

When trusting external identity providers, organizations must decide whether to:

  • Accept Partner MFA: Trust that partner's MFA is sufficient
  • Require Step-Up: Demand additional authentication for sensitive resources
  • Mandate Minimum AAL: Specify minimum authenticator assurance level in federation agreement
  • Attribute-Based Access: Use SAML attributes to make access decisions based on authentication method

✅ WIA-SEC Federation Best Practices
  • Explicit Trust Agreements: Document MFA requirements in federation contracts
  • Attribute Verification: Validate MFA claims in SAML assertions/OIDC tokens
  • Step-Up for Sensitive Data: Require additional authentication regardless of partner's baseline
  • Audit Partner Changes: Monitor when partner modifies their MFA policies
  • Revocation Mechanisms: Ability to immediately revoke federation trust
  • Metadata Management: Automate federation metadata updates and signature verification

6.3 Conditional Access Policies

Conditional Access (also called context-aware access) enforces dynamic authentication requirements based on real-time risk assessment, user attributes, and resource sensitivity.

Conditional Access Dimensions

DimensionConditionsExample Policy
User/Group Department, role, privilege level Require hardware key for IT admins
Location IP range, country, corporate network Allow password-only from office network
Device Managed/unmanaged, OS, compliance state Require MFA on personal devices
Application Sensitivity level, data classification Require AAL3 for payroll system
Real-time Risk Impossible travel, anomalous behavior Block login if risk score > 70
Time Business hours, off-hours, time since last auth Require re-auth every 4 hours for contractors

Microsoft Azure AD Conditional Access Example

Azure AD Policy Configuration 
{
  "displayName": "Require MFA for Azure Management from Untrusted Locations",
  "state": "enabled",
  "conditions": {
    "users": {
      "includeGroups": ["IT-Admins", "Global-Admins"],
      "excludeUsers": ["break-glass-admin@company.com"]
    },
    "applications": {
      "includeApplications": ["Microsoft Azure Management"]
    },
    "locations": {
      "includeLocations": ["All"],
      "excludeLocations": ["Corporate-Network", "Home-Office-VPN"]
    },
    "platforms": {
      "includePlatforms": ["all"]
    },
    "deviceStates": {
      "includeStates": ["All"],
      "excludeStates": ["Compliant", "Hybrid-AD-Joined"]
    },
    "signInRiskLevels": ["medium", "high"]
  },
  "grantControls": {
    "operator": "AND",
    "builtInControls": [
      "mfa",
      "compliantDevice"
    ],
    "termsOfUse": ["azure-management-tou-id"]
  },
  "sessionControls": {
    "signInFrequency": {
      "value": 4,
      "type": "hours",
      "isEnabled": true
    },
    "persistentBrowser": {
      "mode": "never",
      "isEnabled": true
    }
  }
}

Common Conditional Access Patterns

  • Baseline Protection: Require MFA for all cloud apps
  • Admin Protection: Require phishing-resistant MFA (FIDO2) for privileged roles
  • Risky Sign-in Protection: Require MFA + password change if risk detected
  • Legacy Authentication Blocking: Block apps that don't support modern auth/MFA
  • Device Compliance: Require managed device OR MFA for data access
  • Geographic Restrictions: Block or challenge logins from specific countries
  • Application-Specific: Different MFA requirements per app sensitivity
⚠️ Conditional Access Pitfalls
  • Lockout Risk: Always exclude break-glass accounts from all policies
  • Policy Conflicts: Multiple overlapping policies can create unexpected behavior
  • User Experience: Too many prompts lead to MFA fatigue and workarounds
  • Test Mode: Use "report-only" mode before enabling enforcement
  • Monitoring: Track policy failures to identify usability issues

6.5 Zero Trust Architecture

Zero Trust assumes no implicit trust based on network location. Every access request is authenticated, authorized, and encrypted, with MFA as a core component.

Zero Trust Principles

  • Verify Explicitly: Always authenticate and authorize based on all available data points
  • Least Privilege Access: Limit user access with Just-In-Time and Just-Enough-Access (JIT/JEA)
  • Assume Breach: Minimize blast radius and segment access. Verify end-to-end encryption

MFA in Zero Trust

Zero Trust ComponentMFA IntegrationImplementation
Identity Strong MFA as primary control Phishing-resistant MFA for all users
Devices Device + user authentication Certificate-based device auth + MFA
Applications Per-app authentication policies Different MFA requirements per sensitivity
Data MFA for data access and export Step-up auth when accessing PII/PHI
Network No network-based trust MFA required regardless of network location
Visibility & Analytics Continuous authentication monitoring Behavioral biometrics, anomaly detection

6.6 Chapter Summary

📋 Key Takeaways

1
SSO + MFA provides the best enterprise UX

Users authenticate once with MFA and gain access to all applications seamlessly via SAML 2.0 or OpenID Connect.

2
Federation requires explicit MFA trust agreements

Define minimum authenticator assurance levels and accepted methods in federation contracts.

3
Conditional Access enables risk-based policies

Adjust authentication requirements dynamically based on user, device, location, and application context.

4
Enterprise IdPs offer comprehensive MFA

Azure AD, Okta, Ping, and Auth0 provide integrated MFA with SSO, federation, and adaptive authentication.

5
Zero Trust requires MFA everywhere

Verify explicitly with phishing-resistant MFA regardless of network location or device trust.
