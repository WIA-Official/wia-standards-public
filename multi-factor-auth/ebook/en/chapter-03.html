3.1 Time-Based One-Time Passwords (TOTP)

TOTP is one of the most widely deployed MFA technologies, used by Google Authenticator, Microsoft Authenticator, Authy, and dozens of other apps. It generates time-sensitive codes that change every 30 seconds based on a shared secret.

⏱️
TOTP (RFC 6238)

How it works: A cryptographic hash function combines a shared secret key with the current timestamp to generate a 6-8 digit code that remains valid for a short time window (typically 30 seconds).

TOTP Algorithm Implementation 
// TOTP Generation Algorithm
function generateTOTP(secret, timeStep = 30, digits = 6) {
  // 1. Get current Unix timestamp
  const timestamp = Math.floor(Date.now() / 1000);

  // 2. Calculate time counter
  const counter = Math.floor(timestamp / timeStep);

  // 3. Convert counter to 8-byte array (big-endian)
  const counterBytes = new ArrayBuffer(8);
  const view = new DataView(counterBytes);
  view.setBigUint64(0, BigInt(counter), false);

  // 4. Generate HMAC-SHA1 hash
  const hmac = crypto.createHmac('sha1', base32Decode(secret));
  hmac.update(Buffer.from(counterBytes));
  const hash = hmac.digest();

  // 5. Dynamic truncation
  const offset = hash[hash.length - 1] & 0x0f;
  const truncated = (
    ((hash[offset] & 0x7f) << 24) |
    ((hash[offset + 1] & 0xff) << 16) |
    ((hash[offset + 2] & 0xff) << 8) |
    (hash[offset + 3] & 0xff)
  );

  // 6. Generate OTP
  const otp = (truncated % Math.pow(10, digits)).toString();
  return otp.padStart(digits, '0');
}

// Example usage
const secret = "JBSWY3DPEHPK3PXP"; // Base32-encoded secret
const code = generateTOTP(secret);
console.log(`Current TOTP: ${code}`); // Output: "123456"

⚠️ TOTP Security Considerations
  • QR Code Exposure: Screenshot of enrollment QR code can compromise account forever
  • Phishing Vulnerability: Users can be tricked into entering TOTP codes on fake sites
  • Backup Codes: Essential but often stored insecurely (screenshots, notes apps)
  • Device Loss: Lost phone = lost access unless backup configured
  • Man-in-the-Middle: TOTP codes can be intercepted and used in real-time

3.2 HMAC-Based One-Time Passwords (HOTP)

HOTP (RFC 4226) is the predecessor to TOTP, using a counter instead of time. Each code generation increments the counter, ensuring each code is used exactly once.

📖 When to Use HOTP

HOTP is appropriate for hardware tokens, smartcards, and scenarios where accurate time synchronization cannot be guaranteed. For smartphone-based authentication, TOTP is superior due to automatic time synchronization via NTP.

3.3 Push Notification Authentication

Push notification-based MFA sends an approval request directly to a user's registered device, offering superior user experience compared to manual code entry.

📲
Push Authentication Flow
  1. User initiates login on primary device (laptop/desktop)
  2. Server sends push notification to user's registered mobile device
  3. User receives notification with login details (location, IP, device type)
  4. User approves or denies the authentication request
  5. Server receives approval and grants access

✅ WIA-SEC Push Authentication Requirements
  • Number Matching: Mandatory for all push notifications
  • Context Display: Show IP, location, device type, and timestamp
  • Rate Limiting: Maximum 5 push requests per hour per account
  • Deny Tracking: Lock account after 10 denied requests within 24 hours
  • Timeout: Push requests expire after 90 seconds

3.4 SMS and Email One-Time Passwords

SMS and email OTPs are among the most accessible MFA methods but also the most vulnerable. Understanding their risks is essential for proper implementation.

Malware (SMS reader) Medium $500-$5,000 70-90% Mobile security, app permissions
⚠️ NIST Deprecation of SMS

NIST Special Publication 800-63B (Digital Identity Guidelines) deprecated SMS-based OTP in 2016, stating that "due to the risk that SMS messages may be intercepted or redirected, implementers of new systems SHOULD carefully consider alternative authenticators."

WIA-SEC Position: SMS OTP is acceptable only as a last-resort fallback method, never as the primary MFA mechanism. Require additional security measures (device binding, risk scoring) when SMS is used.

3.5 Hardware Security Keys and FIDO2

Hardware security keys represent the gold standard for MFA, offering cryptographic phishing resistance that software-based methods cannot match.

FIDO2 Architecture

FIDO2 consists of two complementary standards:

  • WebAuthn: W3C web standard for passwordless authentication in browsers
  • CTAP2: Client-to-Authenticator Protocol for communication between devices and authenticators
FIDO2 Registration Flow 
// WebAuthn Registration (Simplified)
async function registerSecurityKey(username) {
  // 1. Request challenge from server
  const response = await fetch('/auth/register/start', {
    method: 'POST',
    body: JSON.stringify({ username })
  });
  const options = await response.json();

  // 2. Browser API creates credential
  const credential = await navigator.credentials.create({
    publicKey: {
      challenge: base64Decode(options.challenge),
      rp: { name: "WIA-SEC Demo", id: "example.com" },
      user: {
        id: base64Decode(options.userId),
        name: username,
        displayName: username
      },
      pubKeyCredParams: [
        { type: "public-key", alg: -7 },  // ES256
        { type: "public-key", alg: -257 } // RS256
      ],
      authenticatorSelection: {
        authenticatorAttachment: "cross-platform", // Hardware key
        userVerification: "required",
        residentKey: "required" // Passwordless capable
      },
      timeout: 60000,
      attestation: "direct"
    }
  });

  // 3. Send public key to server for storage
  await fetch('/auth/register/complete', {
    method: 'POST',
    body: JSON.stringify({
      id: credential.id,
      rawId: base64Encode(credential.rawId),
      response: {
        attestationObject: base64Encode(credential.response.attestationObject),
        clientDataJSON: base64Encode(credential.response.clientDataJSON)
      }
    })
  });

  return { success: true, credentialId: credential.id };
}

✅ Why FIDO2 is Phishing-Proof

Origin Binding: The authenticator cryptographically binds credentials to the exact domain (origin). A credential created for "bank.com" will NOT work on "bank-login.com" or any other domain, making phishing mathematically impossible.

Challenge-Response: Each authentication uses a unique challenge, preventing replay attacks.

No Shared Secrets: Only public keys are stored on servers, eliminating database breach risk.
