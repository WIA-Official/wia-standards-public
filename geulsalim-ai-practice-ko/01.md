- 고문서 복원 파이프라인의 데이터 흐름을 처음부터 끝까지 설계할 수 있다

## 이론

### 실전 책의 시작 — 기초에서 한 단계 위로

글살림 AI 기초에서 텍스트→이미지 LoRA를 완성했다면, 실전 책의 목표는 더 구체적이다: **실제 손상된 이미지를 복원**하는 Inpainting 파인튜닝.

단순 텍스트→이미지와 Inpainting의 핵심 차이:

| | 텍스트→이미지 | Inpainting |
|---|---|---|
| 입력 | 텍스트 프롬프트 | 텍스트 + 원본 이미지 + 마스크 |
| 출력 | 완전 새 이미지 | 마스킹 영역만 채운 이미지 |
| 학습 데이터 | (텍스트, 이미지) 페어 | (이미지, 마스크, 타깃) 트리플 |
| 핵심 과제 | 텍스트 의미 학습 | 문맥 보존 + 결손 복원 |

### 고문서 복원 Inpainting 파이프라인

```
[입력 파이프라인]
원본 고문서 이미지
    ↓
손상 마스크 생성 (Otsu / SAM / 수동)
    ↓
(이미지, 마스크) 페어 전처리
    ↓
1024×1024 정규화 + WebDataset 저장

[학습 파이프라인]
SDXL Inpainting 베이스모델 로드
    ↓
LoRA rank 16 어댑터 삽입 (UNet 어텐션)
    ↓
10,000장 × 3 에폭 학습 (A100 기준 ~6시간)
    ↓
val_loss + SSIM 기반 최고 체크포인트 선택

[추론 파이프라인]
손상 이미지 업로드
    ↓
자동 마스크 감지 (Otsu or SAM)
    ↓
SDXL Inpainting LoRA 추론 (20 DDIM 스텝)
    ↓
품질 검증 (SSIM ≥ 0.85)
    ↓
감사 로그 + 워터마킹
    ↓
복원 이미지 저장/반환
```

### 기초 파인튜닝 vs Inpainting 파인튜닝 — 실전 차이점

나는 기초 LoRA로 시작해 Inpainting으로 전환할 때 **3가지 놀라운 점**을 발견했다:

1. **데이터 요구량이 10배** — 텍스트→이미지에서 100장이면 충분했지만, Inpainting은 최소 1,000장 이상이어야 복원 다양성이 확보된다. 마스크 패턴이 너무 적으면 모델이 특정 마스크 형태에만 과적합된다.

2. **마스크 품질이 결과 품질을 좌우** — 직사각형 마스크만으로 학습하면 실제 고문서의 불규칙한 손상 패턴에 대응 못 한다. 최소 5가지 이상의 마스크 유형으로 학습.

3. **`strength` 파라미터가 핵심** — 기초에서는 `strength`를 거의 다루지 않지만, Inpainting에서는 0.7~0.9 범위를 상황에 따라 정밀 조정해야 한다.

### 실전 프로젝트 스펙

**목표**: 조선 후기 풍속화 1만 장 복원 모델 학습

**데이터**: AI-Hub 한국 이미지 + 국립중앙박물관 공개 자료
**GPU**: A100 40GB × 1 (Lambda Labs, $1.29/hr)
**예상 학습 시간**: 6~8시간 (~$9)
**최종 모델 크기**: ~150MB (LoRA 어댑터)

## 실전 코드

```python
# Inpainting 파인튜닝 환경 준비 + 스펙 검증
# pip install diffusers transformers torch accelerate peft

import os
import torch


def verify_inpainting_environment():
    """
    Inpainting 파인튜닝에 필요한 환경 체크
    """
    print("=" * 60)
    print("글살림 AI 실전 — 환경 검증")
    print("=" * 60)
    
    # GPU 확인
    if torch.cuda.is_available():
        gpu_name = torch.cuda.get_device_name(0)
        vram_gb  = torch.cuda.get_device_properties(0).total_memory / 1e9
        print(f"GPU: {gpu_name} ({vram_gb:.0f} GB VRAM)")
        
        # 최소 요구사항 체크
        min_vram = 12.0
        if vram_gb < min_vram:
            print(f"⚠️  VRAM {vram_gb:.0f}GB < 최소 {min_vram}GB")
            print("   해결: gradient_checkpointing + batch_size=1 + gradient_accum=4")
        else:
            print(f"✅ VRAM 충분 ({vram_gb:.0f}GB ≥ {min_vram}GB)")
    else:
        print("❌ GPU 없음 — 학습 불가. GPU 클라우드 필요")
    
    # 패키지 버전 확인
    packages = ["diffusers", "transformers", "accelerate", "peft", "xformers"]
    for pkg in packages:
        try:
            mod = __import__(pkg)
            ver = getattr(mod, "__version__", "unknown")
            print(f"  {pkg}: {ver}")
        except ImportError:
            print(f"  {pkg}: ❌ 미설치 (pip install {pkg})")
    
    # 디스크 공간
    import shutil
    free_gb = shutil.disk_usage(".").free / 1e9
    print(f"\n디스크 여유 공간: {free_gb:.0f}GB")
    if free_gb < 50:
        print("⚠️  50GB 이상 권장 (모델 + 데이터셋)")


def estimate_training_cost(
    n_images: int,
    n_epochs: int = 3,
    gpu_price_per_hour: float = 1.29,  # A100 Lambda Labs
    images_per_second: float = 0.1,   # A100 기준 SDXL batch=1 약 10초/이미지
) -> dict:
    """
    학습 시간 및 비용 추정
    """
    total_steps  = (n_images * n_epochs)
    train_hours  = total_steps / (images_per_second * 3600)
    total_cost   = train_hours * gpu_price_per_hour
    
    return {
        "total_steps":    total_steps,
        "train_hours":    round(train_hours, 1),
        "total_cost_usd": round(total_cost, 2),
        "recommendation": (
            "A100 최적"    if total_cost < 20 else
            "H100 고려"    if total_cost < 50 else
            "데이터 줄이기"
        )
    }


if __name__ == "__main__":
    verify_inpainting_environment()
    
    print("\n학습 비용 추정")
    print("-" * 40)
    scenarios = [
        ("실험용",    1_000, 2),
        ("실전 기본", 5_000, 3),
        ("실전 완전", 10_000, 3),
    ]
    for name, n, epochs in scenarios:
        est = estimate_training_cost(n, epochs)
        print(f"[{name}] {n:,}장 × {epochs}에폭 = "
              f"{est['train_hours']}시간 ≈ ${est['total_cost_usd']} "
              f"({est['recommendation']})")
```

## 함정 카드

1. **"기초 LoRA와 동일한 데이터로 시작한다"** → Inpainting 학습 데이터는 반드시 `(이미지, 마스크, 타깃)` 트리플이어야 한다. 텍스트→이미지 데이터셋은 그대로 사용 불가. 마스크 생성 파이프라인 별도 구축 필요.

2. **"SDXL Inpainting 전용 체크포인트를 사용하지 않는다"** → `stabilityai/stable-diffusion-xl-base-1.0`은 Inpainting용이 아니다. 반드시 `diffusers/stable-diffusion-xl-1.0-inpainting-0.1`을 베이스로 사용. 일반 SDXL에 Inpainting 파인튜닝하면 결과가 최악이다.

3. **"배치 크기 1로 A100에서 학습한다"** → A100 40GB에서 batch_size=4 또는 2는 가능하다. batch=1은 gradient 노이즈가 크고 GPU 이용률이 낮다. `gradient_accumulation_steps`로 유효 배치를 8로 맞추면서 실제 배치는 2~4가 최적.

4. **"마스크 1가지 유형만 사용한다"** → 직사각형, 브러시, 불규칙 곡선, Otsu 기반 실제 손상 마스크를 균등하게 섞어야 실제 고문서 손상에 일반화된다. 단일 마스크 유형 학습은 특정 형태에만 좋은 결과.

5. **"결과가 즉시 완벽하지 않다고 포기한다"** → Inpainting 파인튜닝은 텍스트→이미지보다 어렵다. 200~300 스텝에서 결과가 아직 거칠어도 정상. 500 스텝부터 눈에 띄는 개선. 1,000 스텝 체크포인트부터 사용 가능 품질.

## 연습문제

**문제 1.** SDXL 텍스트→이미지와 SDXL Inpainting의 UNet 아키텍처 차이를 설명하라. Inpainting 모델은 어떤 추가 채널을 입력으로 받는가?

> **해설:** SDXL Inpainting의 UNet은 표준 4채널(RGBA latent) 외에 **마스크 채널(1채널) + 마스킹된 이미지 latent(4채널) = 9채널**을 입력으로 받는다. 이를 "9-in" 채널이라 부른다. 기반 SDXL의 첫 convolution 레이어를 4채널에서 9채널로 확장하여 초기화. 이 추가 채널 덕분에 모델이 "어디를 채울지"를 명시적으로 알 수 있다.

**문제 2.** `strength=0.8`과 `strength=0.95`의 차이가 고문서 복원에 미치는 영향을 설명하라.

> **해설:** `strength`는 원본 이미지에 noising하는 timestep 수를 결정. `strength=0.8` → 총 1000 timestep 중 800스텝부터 시작 = 원본 구조의 20% 보존. `strength=0.95` → 950스텝부터 = 원본의 5%만 보존 = 거의 새로 생성. 고문서 복원에서는 주변 맥락(원본 종이 질감, 먹의 방향)을 살려야 하므로 `strength=0.7~0.85`가 적합. 너무 높으면 원본과 괴리된 결과가 나온다.

**문제 3.** 10,000장 데이터로 A100 하나를 사용해 학습할 때, 총 GPU 비용을 최소화하면서 품질을 유지하는 설정을 제안하라.

> **해설:** 최적 설정: `batch_size=2, gradient_accumulation=4` (유효 배치 8), `max_train_steps=5000` (3 에폭 기준), `gradient_checkpointing=True`, `fp16=True`. 예상 시간: 5,000 스텝 × (2이미지 × 배치 처리 ~5초) / 3600 = 2.8시간. 비용: 2.8 × $1.29 = $3.6. `--resume_from_checkpoint`로 체크포인트 재개 시 중단 손실 방지.
