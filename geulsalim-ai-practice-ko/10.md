# 챕터 10 · Mixed Precision + Gradient Checkpointing

## 학습 목표
- bf16 Mixed Precision으로 메모리 절반 + 속도 2배를 얻는 원리와 설정 방법을 설명할 수 있다
- Gradient Checkpointing으로 activation 메모리를 40~50% 줄이는 대신 속도 20% 손해를 받아들이는 트레이드오프를 평가할 수 있다
- Gradient Accumulation으로 유효 배치 크기를 유지하면서 VRAM을 쥐어짜는 설정을 적용할 수 있다

---

## 이론

### fp16 vs bf16 — SDXL은 왜 bf16인가

두 포맷 모두 16비트지만 비트 할당이 다르다:

| | 지수(exponent) | 가수(mantissa) | 표현 범위 |
|---|---|---|---|
| fp32 | 8 | 23 | 매우 넓음 |
| fp16 | 5 | 10 | ±65,504 (좁음) |
| bf16 | **8** | 7 | fp32와 동일 범위 |

SDXL 학습에서 activation이 자주 65,504를 초과한다. fp16은 오버플로우 → `inf` → `NaN` 폭증. **bf16은 지수가 fp32와 같아 오버플로우 없음**. 대신 가수 정밀도는 fp16보다 낮아 세밀한 값 구분이 약하지만 학습에는 문제없다.

실무 기준 (2026-04):
- **A100/H100/RTX 30·40 시리즈**: bf16 네이티브 지원, 첫 선택
- **V100·T4·RTX 20 시리즈**: bf16 미지원 → fp16 + Loss Scaling (GradScaler)
- **AMD MI250/MI300**: bf16 네이티브 지원

나는 V100에서 SDXL을 학습시키려다 fp16 NaN을 일주일 반복한 뒤, RunPod A100으로 옮기고 bf16으로 바꿨더니 한 번에 학습이 완주됐다. **하드웨어가 bf16을 지원하면 묻지도 따지지도 말고 bf16.**

### Mixed Precision의 "Mixed"

모든 연산을 fp16/bf16으로 하면 가중치 업데이트에서 정밀도 손실이 누적된다. Mixed Precision의 실제 구조:

```
가중치:        fp32로 저장 (마스터 복사본)
Forward:       bf16으로 계산 (autocast 안에서)
Loss:          bf16
Backward:      bf16
Optimizer:     fp32 가중치에 bf16 gradient를 캐스팅해 업데이트
```

PyTorch에서 `torch.autocast(device_type="cuda", dtype=torch.bfloat16)` 컨텍스트 매니저가 이를 자동화한다.

### Gradient Checkpointing — 시간을 메모리로 바꾼다

Backward에는 forward의 모든 activation이 필요하다. UNet의 레이어 140개 × 배치 × 해상도 = GB 단위 activation이 GPU 메모리에 쌓인다.

Checkpointing은 activation을 **중간만 저장하고 나머지는 backward에서 재계산**한다:
- 메모리: 40~50% 감소
- 속도: 20~25% 감소 (재계산 비용)

```python
unet.enable_gradient_checkpointing()
```

한 줄이면 된다. SDXL 학습에서는 **거의 무조건 켜는 기본 설정**이다.

### Gradient Accumulation — 배치를 가상으로 크게

배치 크기 16으로 학습하고 싶지만 VRAM은 batch=2만 허용한다면:

```
accumulation_steps = 8
for step in range(N):
    for micro_step in range(accumulation_steps):
        loss = compute_loss(micro_batch)
        (loss / accumulation_steps).backward()    # gradient 누적
    optimizer.step()
    optimizer.zero_grad()
```

8번의 작은 배치 gradient가 합쳐져 유효 배치 16이 된다. 수렴 행동이 거의 동일하다. **LoRA 학습의 사실상 표준 패턴**.

## 실전 코드

```python
# Mixed Precision + Gradient Checkpointing + Accumulation 통합 학습 루프
# pip install accelerate>=0.28 diffusers peft torch

from accelerate import Accelerator
from accelerate.utils import ProjectConfiguration, set_seed
import torch
import torch.nn.functional as F
from diffusers import StableDiffusionXLInpaintPipeline, DDPMScheduler
from diffusers.optimization import get_cosine_schedule_with_warmup
from peft import LoraConfig
from torch.optim import AdamW


# ──────────────────────────────────────────────
# 1. Accelerator 초기화 (bf16 + logging)
# ──────────────────────────────────────────────
def build_accelerator(
    output_dir: str,
    accumulation_steps: int = 4,
    mixed_precision: str = "bf16",   # "fp16", "bf16", "no"
):
    cfg = ProjectConfiguration(project_dir=output_dir, logging_dir=f"{output_dir}/logs")
    accelerator = Accelerator(
        gradient_accumulation_steps = accumulation_steps,
        mixed_precision             = mixed_precision,
        log_with                    = "wandb",
        project_config              = cfg,
    )
    # 시드 재현성
    set_seed(42)
    return accelerator


# ──────────────────────────────────────────────
# 2. 모델·옵티마이저 준비 (메모리 최적화 옵션 전부)
# ──────────────────────────────────────────────
def setup_memory_optimized_training(accelerator: Accelerator):
    MODEL_ID = "diffusers/stable-diffusion-xl-1.0-inpainting-0.1"
    pipe = StableDiffusionXLInpaintPipeline.from_pretrained(
        MODEL_ID, torch_dtype=torch.float32,
    )

    # VAE, Text Encoder: 고정 + fp16 (추론만 하므로 fp16 무관)
    pipe.vae.to(accelerator.device, dtype=torch.float16)
    pipe.vae.requires_grad_(False)
    pipe.text_encoder.to(accelerator.device, dtype=torch.float16)
    pipe.text_encoder.requires_grad_(False)
    pipe.text_encoder_2.to(accelerator.device, dtype=torch.float16)
    pipe.text_encoder_2.requires_grad_(False)

    # UNet: fp32 마스터 + LoRA
    unet = pipe.unet
    lora_cfg = LoraConfig(
        r=16, lora_alpha=16,
        target_modules=["to_q", "to_k", "to_v", "to_out.0"],
    )
    unet.add_adapter(lora_cfg)
    for n, p in unet.named_parameters():
        p.requires_grad = "lora" in n.lower()

    # ★ Gradient Checkpointing 활성화
    unet.enable_gradient_checkpointing()
    # 학습 시 xformers보다 PyTorch SDPA가 안정적
    unet.set_attn_processor(
        {k: None for k in unet.attn_processors.keys()}  # 기본 AttnProcessor로 리셋
    )

    scheduler = DDPMScheduler.from_pretrained(MODEL_ID, subfolder="scheduler")

    return pipe, unet, scheduler


def build_optimizer(unet, lr: float = 1e-4):
    trainable = [p for p in unet.parameters() if p.requires_grad]

    # 8-bit Adam (bitsandbytes) — 옵티마이저 state 4배 압축
    # LoRA 학습 파라미터 15M에서는 fp32 Adam이 60MB 정도라 큰 차이 없음
    # 하지만 풀 파인튜닝이면 4GB → 1GB로 절약
    try:
        import bitsandbytes as bnb
        optimizer = bnb.optim.AdamW8bit(trainable, lr=lr, weight_decay=1e-2)
        print(">> 8-bit AdamW 활성화")
    except ImportError:
        optimizer = AdamW(trainable, lr=lr, weight_decay=1e-2)
        print(">> 표준 fp32 AdamW")
    return optimizer


# ──────────────────────────────────────────────
# 3. 학습 루프 (accelerator.accumulate 사용)
# ──────────────────────────────────────────────
def training_loop(
    accelerator: Accelerator,
    unet,
    vae,
    noise_scheduler,
    loader,
    prompt_embeds,
    pooled_embed,
    total_steps: int = 30000,
):
    optimizer = build_optimizer(unet, lr=1e-4)
    lr_scheduler = get_cosine_schedule_with_warmup(
        optimizer, num_warmup_steps=500,
        num_training_steps=total_steps,
    )

    unet, optimizer, loader, lr_scheduler = accelerator.prepare(
        unet, optimizer, loader, lr_scheduler,
    )
    accelerator.init_trackers("sdxl-inpaint-lora", config={"rank": 16, "lr": 1e-4})

    global_step = 0
    progress_loss = []

    for epoch in range(999):  # 스텝 기반 종료
        for batch in loader:
            with accelerator.accumulate(unet):
                # VAE 인코딩 (fp16)
                with torch.no_grad():
                    latents = vae.encode(
                        batch["pixel_values"].to(accelerator.device, dtype=torch.float16)
                    ).latent_dist.sample() * vae.config.scaling_factor
                    masked_latents = vae.encode(
                        batch["masked_pixel_values"].to(accelerator.device, dtype=torch.float16)
                    ).latent_dist.sample() * vae.config.scaling_factor
                latents        = latents.float()
                masked_latents = masked_latents.float()

                mask = batch["mask"].to(accelerator.device, dtype=torch.float32)
                mask_small = F.interpolate(
                    mask, size=latents.shape[-2:], mode="nearest",
                )

                noise = torch.randn_like(latents)
                timesteps = torch.randint(
                    0, noise_scheduler.config.num_train_timesteps,
                    (latents.shape[0],), device=accelerator.device, dtype=torch.long,
                )
                noisy_latents = noise_scheduler.add_noise(latents, noise, timesteps)
                unet_input = torch.cat([noisy_latents, masked_latents, mask_small], dim=1)

                add_time_ids = torch.tensor(
                    [[1024, 1024, 0, 0, 1024, 1024]] * latents.shape[0],
                    device=accelerator.device, dtype=torch.float32,
                )

                # ★ autocast — bf16 forward
                noise_pred = unet(
                    unet_input, timesteps,
                    encoder_hidden_states = prompt_embeds.expand(latents.shape[0], -1, -1),
                    added_cond_kwargs = {
                        "text_embeds": pooled_embed.expand(latents.shape[0], -1),
                        "time_ids":    add_time_ids,
                    },
                ).sample

                loss = F.mse_loss(noise_pred.float(), noise.float())

                accelerator.backward(loss)

                if accelerator.sync_gradients:
                    accelerator.clip_grad_norm_(
                        [p for p in unet.parameters() if p.requires_grad],
                        max_norm=1.0,
                    )
                optimizer.step()
                lr_scheduler.step()
                optimizer.zero_grad()

            # sync_gradients=True일 때만 "한 optimizer step 완료"
            if accelerator.sync_gradients:
                progress_loss.append(loss.item())
                global_step += 1

                if global_step % 50 == 0:
                    avg = sum(progress_loss[-50:]) / min(50, len(progress_loss))
                    accelerator.log({
                        "train/loss":       avg,
                        "train/lr":         lr_scheduler.get_last_lr()[0],
                    }, step=global_step)
                    accelerator.print(
                        f"step {global_step:6d} | loss {avg:.4f} | "
                        f"lr {lr_scheduler.get_last_lr()[0]:.2e}"
                    )

                if global_step >= total_steps:
                    accelerator.end_training()
                    return


# ──────────────────────────────────────────────
# 4. 실행 — accelerate launch로
# ──────────────────────────────────────────────
if __name__ == "__main__":
    accelerator = build_accelerator(
        output_dir="./training_output",
        accumulation_steps=4,      # micro_batch 2 × accum 4 = 유효 배치 8
        mixed_precision="bf16",
    )
    pipe, unet, scheduler = setup_memory_optimized_training(accelerator)

    # DataLoader, 프롬프트 임베딩은 챕터 05, 08 코드 조합
    # ...
    # training_loop(accelerator, unet, pipe.vae, scheduler, loader, ...)
```

**실행:**
```bash
accelerate config      # 대화형 설정 (bf16, gradient_accumulation 확인)
accelerate launch train.py --num_processes=1
```

**메모리 사용 비교 (SDXL Inpainting LoRA rank 16, A100 80GB 실측):**

| 설정 | VRAM | Batch | 속도 (it/s) |
|---|---|---|---|
| fp32 기본 | 68 GB | 1 | 0.6 |
| bf16 only | 42 GB | 2 | 1.8 |
| bf16 + Grad Ckpt | 24 GB | 4 | 1.4 |
| bf16 + Grad Ckpt + Accum=4 | 24 GB | 8 유효 | 0.35 (step/s) |

`bf16 + Grad Ckpt`가 최적 구간. Accumulation은 유효 배치 필요할 때만.

## 함정 카드

1. **"fp16을 쓰고 NaN을 왜 나는지 모른다"** → SDXL은 fp16 NaN이 악명 높다. 가중치 중 일부가 65,504 초과 activation을 유발. 해결: ① bf16 하드웨어면 즉시 전환 ② fp16만 가능하면 `GradScaler(init_scale=2**10)` 작게 시작 ③ VAE를 fp32로 유지 (encode/decode에서 NaN 다발). 내 V100 경험상 **fp16 SDXL은 공식 지원이라 봐선 안 된다**.

2. **"Gradient Checkpointing과 Xformers Attention을 동시에 쓴다"** → 두 최적화가 activation 처리 방식이 충돌해 드물게 NaN 또는 backward 오류. SDXL 학습 시 `enable_xformers...`를 끄고 PyTorch 2.0+ SDPA만 쓴다. 추론에만 Xformers.

3. **"gradient accumulation 안에서 `optimizer.zero_grad()`를 잘못 호출"** → accumulation_steps=4면 4번의 backward 후 한 번만 zero_grad. `accelerator.accumulate` 컨텍스트를 쓰면 자동화되지만 수동으로 할 때는 `if (step+1) % accum == 0: optimizer.step(); zero_grad()` 로직 필수.

4. **"VAE만 fp16으로 두고 forward 시 캐스팅을 빠뜨린다"** → pipeline에서 `vae.to(dtype=fp16)` 해도 입력 텐서가 fp32면 `RuntimeError: expected scalar type Half but got Float`. 입력도 `.to(dtype=torch.float16)` 명시.

5. **"checkpointing을 켜고도 배치를 늘려도 OOM이 안 뚫린다"** → activation은 줄었어도 **CUDA 커널 워크스페이스**가 고정 사용. `torch.cuda.empty_cache()`, `PYTORCH_CUDA_ALLOC_CONF=max_split_size_mb:128` 환경변수로 메모리 조각화 완화. 이래도 부족하면 rank를 8로.

## 연습문제

**문제 1.** A100 40GB에서 SDXL Inpainting LoRA를 돌리려 한다. 아래 세 설정 중 학습이 실제 작동할 가능성이 가장 높은 것은?

(A) fp32 + batch=2 + checkpoint off  
(B) bf16 + batch=1 + checkpoint off  
(C) bf16 + batch=2 + checkpoint on + accum=4

> **해설:** **(C)**. (A)는 fp32 VRAM 사용이 80GB급이라 40GB엔 불가. (B)는 bf16에 batch=1이면 약 30GB로 겨우 맞지만 학습이 느리고 batch=1은 BatchNorm 안정성 위험. (C)는 activation 절반 + batch 2로 한 스텝 VRAM 약 24GB, accum=4로 유효 배치 8, 40GB 여유 있게 동작. 실무에서 40GB 환경 표준.

**문제 2.** bf16 학습 중 `accelerator.backward(loss)`에서 "The provided loss tensor is not on the same device as the model"이 뜬다. 가장 흔한 원인은?

> **해설:** LPIPS나 커스텀 loss 계산에서 중간 텐서를 `.to("cuda:0")`로 하드코딩했으나 Accelerate가 `cuda:1`에 모델을 올린 경우(멀티 GPU). 또는 배치 일부가 CPU에 남아있는 경우. 해결: 모든 디바이스 지정을 `accelerator.device`로 통일. `loss.device == unet.device` 체크 로그 추가로 디버깅.

**문제 3.** Gradient Accumulation 4로 설정했지만 실제 학습 곡선이 batch=2 기본 학습과 거의 똑같이 보인다. 왜인가?

> **해설:** **정상이다**. Accumulation은 "유효 배치 크기"만 늘릴 뿐 학습 동역학은 큰 배치 학습과 유사해야 한다. 동일해 보이면 구현이 맞는 것. 다만 차이가 나는 구간: ① BatchNorm이 있는 모델(SDXL 무관) ② `optimizer.step` 횟수가 1/4이므로 스케줄러의 warmup·cosine이 실제 스텝 기준으로 작동해야 함. `get_cosine_schedule_with_warmup`에 전달하는 `total_steps`가 **optimizer 스텝 기준**인지 **batch 스텝 기준**인지 혼동하면 스케줄이 어긋난다 — Accelerate는 전자로 처리.
