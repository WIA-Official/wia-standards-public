# 챕터 15 · 추론 최적화 — 배치·CFG·스케줄러·LCM

## 학습 목표
- DPMSolver++·LCM 등 최신 스케줄러로 inference step을 30→6으로 줄이면서 품질을 유지하는 트레이드오프를 설명할 수 있다
- 배치 추론으로 GPU 처리량을 3배 이상 올리는 코드를 작성할 수 있다
- CFG(Classifier-Free Guidance) 스케일과 step 수 최적 조합을 자체 데이터셋으로 결정할 수 있다

---

## 이론

### 추론 비용의 지배 요소

1024 Inpainting 1회 = UNet forward 30회 + VAE encode 1 + VAE decode 1.
A100 기준 ≈ 9초. 같은 구조로 1만 장 처리 = 25시간 = $30.

비용을 줄이는 레버 3가지:
- **스텝 수**: 30 → 6 (5배) — LCM 또는 Lightning
- **배치**: 1 → 4 (3~4배) — VRAM 여유
- **해상도**: 1024 → 768 (1.78배) — 품질 손실

### 스케줄러 선택 (2026-04 기준)

| 스케줄러 | Step | 품질 | LoRA 호환 | 메모 |
|---|---|---|---|---|
| DDIM | 50 | ⭐⭐⭐ | ✅ | 가장 안정, 느림 |
| DPMSolver++ 2M | 20~30 | ⭐⭐⭐⭐ | ✅ | 실무 표준 ⭐ |
| Euler a | 25 | ⭐⭐⭐ | ✅ | 노이지하지만 다양성 |
| LCM-LoRA | 4~8 | ⭐⭐⭐ | ✅ (적용 필요) | 별도 LCM-LoRA 로드 |
| SDXL Lightning | 4 | ⭐⭐⭐⭐ | ⚠️ (경우별) | Distilled 가중치 |

**실무 권장 조합:**
- 배치 처리·품질 우선: **DPMSolver++ 2M Karras + 25 step + CFG 7.5**
- 실시간 서비스: **LCM-LoRA + 6 step + CFG 2.0**
- 최고 속도(5-sec UX): **SDXL Lightning 4-step + CFG 1.0**

### CFG(Classifier-Free Guidance) 수학

```
pred = pred_uncond + scale × (pred_cond - pred_uncond)
```

- `scale = 1.0`: 프롬프트 무시
- `scale = 7.5`: 기본 (SDXL 공식)
- `scale = 10+`: 프롬프트 강조, 과채도 아티팩트

**핵심 비용**: CFG는 한 step에 **두 번의 UNet forward**(조건·무조건). 즉 실제 step = 표시 step × 2. `cfg_scale=1.0`이면 조건 패스만 돌려 **2배 빠름**. LCM·Lightning은 CFG 1.0으로 학습되어 있어 이 최적화 활용.

### 배치 추론의 함정

Inpainting 파이프라인에 배치 입력 시 주의:
- `prompt=[p] * N` 또는 단일 프롬프트 + 이미지 리스트
- `image=[img1, img2, ...]`, `mask_image=[m1, m2, ...]`
- 배치 내 **해상도가 모두 같아야** 함 (1024×1024 정규화)

VRAM: batch=1이 14GB이면 batch=4는 약 28GB. A100 40GB면 **배치 3**이 안전.

## 실전 코드

```python
# 추론 최적화 — 배치 + DPMSolver++ + LCM-LoRA
# pip install diffusers transformers torch accelerate

import torch
import time
from pathlib import Path
from PIL import Image
from diffusers import (
    StableDiffusionXLInpaintPipeline,
    DPMSolverMultistepScheduler,
    LCMScheduler,
)
from peft import LoraConfig
from safetensors.torch import load_file


DEVICE = "cuda"


# ──────────────────────────────────────────────
# 1. 최적화된 파이프라인 빌더
# ──────────────────────────────────────────────
class OptimizedInpaintPipeline:
    """
    DPMSolver++ 기본, LCM 옵션 제공.
    torch.compile 사용.
    """
    BASE_MODEL = "diffusers/stable-diffusion-xl-1.0-inpainting-0.1"
    LCM_LORA   = "latent-consistency/lcm-lora-sdxl"    # Inpainting 호환 LCM-LoRA

    def __init__(
        self,
        lora_path:    str,
        mode:         str = "dpm",       # "dpm" | "lcm"
        compile_unet: bool = True,
    ):
        # (1) 베이스 파이프라인
        pipe = StableDiffusionXLInpaintPipeline.from_pretrained(
            self.BASE_MODEL, torch_dtype=torch.float16, variant="fp16",
        ).to(DEVICE)
        pipe.set_progress_bar_config(disable=True)

        # (2) 스타일 LoRA 로드
        lora_cfg = LoraConfig(
            r=16, lora_alpha=16,
            target_modules=["to_q", "to_k", "to_v", "to_out.0"],
        )
        pipe.unet.add_adapter(lora_cfg, adapter_name="style")
        state = load_file(lora_path, device="cpu")
        pipe.unet.load_state_dict(state, strict=False)

        # (3) 모드별 스케줄러
        if mode == "dpm":
            pipe.scheduler = DPMSolverMultistepScheduler.from_config(
                pipe.scheduler.config,
                use_karras_sigmas=True,
                algorithm_type="dpmsolver++",
            )
            self.default_steps = 25
            self.default_cfg   = 7.5
        elif mode == "lcm":
            # LCM-LoRA 추가 로드
            pipe.load_lora_weights(self.LCM_LORA, adapter_name="lcm")
            pipe.set_adapters(["style", "lcm"], adapter_weights=[1.0, 1.0])
            pipe.scheduler = LCMScheduler.from_config(pipe.scheduler.config)
            self.default_steps = 6
            self.default_cfg   = 2.0
        else:
            raise ValueError(f"Unknown mode: {mode}")

        # (4) 메모리·속도 최적화
        pipe.enable_vae_tiling()             # 추론은 tiling OK (학습과 다름)
        pipe.unet.to(memory_format=torch.channels_last)

        # (5) torch.compile (2~3배 추가 속도, 첫 호출만 컴파일 5분)
        if compile_unet:
            pipe.unet = torch.compile(pipe.unet, mode="reduce-overhead", fullgraph=False)
            print(">> torch.compile 활성화 (첫 호출 시 컴파일 1-5분 소요)")

        self.pipe = pipe
        self.mode = mode

    # ──────── 단일 추론 ────────
    def infer(
        self,
        image:   Image.Image,
        mask:    Image.Image,
        prompt:  str = "Korean traditional document restoration",
        negative_prompt: str = "modern, photograph, digital art",
        steps:   int = None,
        cfg:     float = None,
        seed:    int = 42,
    ) -> Image.Image:
        generator = torch.Generator(DEVICE).manual_seed(seed)
        with torch.autocast("cuda", dtype=torch.float16):
            result = self.pipe(
                prompt=prompt,
                negative_prompt=negative_prompt if self.mode == "dpm" else None,
                image=image,
                mask_image=mask,
                num_inference_steps=steps or self.default_steps,
                guidance_scale=cfg or self.default_cfg,
                generator=generator,
            ).images[0]
        return result

    # ──────── 배치 추론 ────────
    def infer_batch(
        self,
        images:  list[Image.Image],
        masks:   list[Image.Image],
        prompt:  str = "Korean traditional document restoration",
        steps:   int = None,
        cfg:     float = None,
        seed:    int = 42,
    ) -> list[Image.Image]:
        assert len(images) == len(masks), "이미지·마스크 길이 불일치"
        B = len(images)
        generators = [torch.Generator(DEVICE).manual_seed(seed + i) for i in range(B)]

        with torch.autocast("cuda", dtype=torch.float16):
            results = self.pipe(
                prompt=[prompt] * B,
                image=images,
                mask_image=masks,
                num_inference_steps=steps or self.default_steps,
                guidance_scale=cfg or self.default_cfg,
                generator=generators,
            ).images
        return results


# ──────────────────────────────────────────────
# 2. 처리량 벤치마크
# ──────────────────────────────────────────────
def benchmark(
    pipeline: OptimizedInpaintPipeline,
    test_image: Image.Image,
    test_mask:  Image.Image,
    n_warmup: int = 2,
    n_measure: int = 10,
    batch_sizes: list = [1, 2, 4],
) -> dict:
    """배치 크기별 처리량 측정"""
    results = {}
    for B in batch_sizes:
        imgs = [test_image] * B
        masks = [test_mask] * B

        # 워밍업 (torch.compile 컴파일 완료 대기)
        for _ in range(n_warmup):
            _ = pipeline.infer_batch(imgs, masks)

        torch.cuda.synchronize()
        start = time.perf_counter()
        for _ in range(n_measure):
            _ = pipeline.infer_batch(imgs, masks)
        torch.cuda.synchronize()
        elapsed = time.perf_counter() - start

        throughput = B * n_measure / elapsed
        results[B] = {
            "sec_per_batch":  elapsed / n_measure,
            "sec_per_image":  elapsed / (B * n_measure),
            "images_per_sec": throughput,
            "vram_gb":        torch.cuda.max_memory_allocated() / 1024**3,
        }
        torch.cuda.reset_peak_memory_stats()
        print(f"batch={B}: {throughput:.2f} imgs/s, "
              f"{results[B]['sec_per_image']:.2f}s/img, "
              f"VRAM {results[B]['vram_gb']:.1f}GB")
    return results


# ──────────────────────────────────────────────
# 3. CFG/Step 스윕
# ──────────────────────────────────────────────
def sweep_cfg_steps(
    pipeline: OptimizedInpaintPipeline,
    image: Image.Image, mask: Image.Image,
    target: Image.Image,
) -> None:
    """품질-속도 트레이드오프 측정"""
    from torchmetrics.image import StructuralSimilarityIndexMeasure
    ssim_fn = StructuralSimilarityIndexMeasure().to(DEVICE)
    import numpy as np

    def to_t(img):
        arr = np.array(img.convert("RGB")).astype(np.float32) / 255.0
        return torch.from_numpy(arr).permute(2, 0, 1).unsqueeze(0).to(DEVICE)

    combos = [(steps, cfg) for steps in [6, 15, 25, 50] for cfg in [2.0, 5.0, 7.5, 10.0]]
    for steps, cfg in combos:
        t0 = time.perf_counter()
        pred = pipeline.infer(image, mask, steps=steps, cfg=cfg)
        t = time.perf_counter() - t0

        ssim = ssim_fn(to_t(pred), to_t(target)).item()
        print(f"steps={steps:3d} cfg={cfg:.1f}: SSIM {ssim:.3f}, time {t:.2f}s")


# ──────────────────────────────────────────────
# 4. 메인
# ──────────────────────────────────────────────
if __name__ == "__main__":
    # DPM 모드 (품질 우선)
    dpm_pipe = OptimizedInpaintPipeline(
        lora_path="./output/lora_final.safetensors",
        mode="dpm",
        compile_unet=True,
    )
    # LCM 모드 (속도 우선)
    lcm_pipe = OptimizedInpaintPipeline(
        lora_path="./output/lora_final.safetensors",
        mode="lcm",
        compile_unet=True,
    )

    # 벤치마크
    test_img  = Image.open("./test/damaged_01.jpg").convert("RGB").resize((1024, 1024))
    test_mask = Image.open("./test/damaged_01_mask.png").convert("L").resize((1024, 1024))

    print("=== DPM 모드 ===")
    benchmark(dpm_pipe, test_img, test_mask)
    print("\n=== LCM 모드 ===")
    benchmark(lcm_pipe, test_img, test_mask)

    # CFG/Step 최적 조합 탐색
    target = Image.open("./test/clean_01.jpg").convert("RGB").resize((1024, 1024))
    print("\n=== CFG × Step 스윕 ===")
    sweep_cfg_steps(dpm_pipe, test_img, test_mask, target)
```

**내 실측 (A100 40GB, 1024 해상도):**
```

=== LCM 모드 (6 step, CFG 2.0) ===
```

## 함정 카드

1. **"LCM-LoRA를 먼저 로드하고 스타일 LoRA를 나중에 로드"** → PEFT 어댑터 순서가 결과에 영향. 권장: 스타일 먼저, LCM 나중, `set_adapters(weights=[1.0, 1.0])`. LCM 가중치를 너무 크게 주면(1.5+) 스타일이 묻힌다.

2. **"torch.compile 컴파일 시간을 잊고 첫 요청 타임아웃"** → `compile_unet=True`면 **첫 호출에서 2~5분**이 정상. 서비스 시작 시 워밍업 호출로 컴파일을 끝낸 뒤 트래픽 수신. 컴파일 결과는 프로세스 종료 시 사라지므로 매 프로세스 시작마다 재컴파일.

3. **"배치 크기가 최대 처리량 지점을 넘었다"** → A100 40GB에서 배치 6 이상이면 OOM 직전까지 가다 GPU 스로틀링 or OOM. "batch=4가 최적"이 SDXL Inpainting의 현실적 기준. H100 80GB면 8까지.

4. **"DPMSolver++ 설정에서 `use_karras_sigmas=False`로 뒀다"** → Karras 시그마 스케줄은 25 step에서 40 step 품질에 준한다. 끄면 같은 step에서 품질 10% 저하. SDXL에선 거의 무조건 `True`.

5. **"Negative prompt 없이 돌린다"** → 복원 태스크에서 "modern, photograph, low quality, blurry" 같은 negative로 원치 않는 스타일 억제 가능. 하지만 LCM 모드에선 negative prompt가 무시되니 DPM에서만 의미 있음.

## 연습문제

**문제 1.** LCM 6-step 결과가 DPM 25-step보다 SSIM이 0.03 낮지만 추론 속도는 4배 빠르다. 어떤 서비스에서 LCM을 쓸 것인가?

> **해설:** **미리보기·드래프트 UX**에 LCM, **최종 저장**에 DPM. 예: 사용자가 마스크를 그리며 실시간 복원 미리보기(0.5초 응답)는 LCM, "저장" 버튼 클릭 시 DPM 재생성 2.5초. 또는 배치 자동처리에서 "1차 스크리닝"은 LCM, "최종 납품본"은 DPM. 품질과 속도가 요구사항에 따라 분리되는 곳이면 LCM이 유용.

**문제 2.** `enable_vae_tiling`을 켠 추론 결과에서 1024 이미지 중앙에 미세한 가로·세로 라인이 보인다. 원인과 해결은?

> **해설:** VAE tiling은 이미지를 타일로 쪼개 개별 디코딩 후 병합하므로 **타일 경계 아티팩트** 발생. 기본 overlap은 0.25지만 약한 경우에 이 라인이 남는다. 해결: ① `pipe.vae.enable_tiling(overlap=0.5)` — overlap 증가 (메모리 +25%) ② 1024 해상도면 tiling 불필요, `disable_tiling` ③ 2048 등 고해상도에서만 tiling 켜라. 1024 × batch 4가 A100 40GB에 맞으므로 tiling 불필요.

**문제 3.** 같은 이미지에 `cfg=5.0`과 `cfg=10.0` 결과 중 SSIM은 5.0이 높고 LPIPS는 10.0이 높다. 어느 쪽이 "더 좋은 복원"인가?

> **해설:** **상황에 따라 다름**. SSIM 높음 = 원본과 구조적 유사. LPIPS 높음 = 사람 눈에 부자연 아티팩트. CFG 10은 프롬프트 강조로 "더 선명하지만 과장된" 복원. 고문서 복원은 "원본 느낌 보존"이 더 중요하므로 **SSIM 우선 → CFG 5 선택**. 단, 마스크 안쪽 내용 생성 품질이 부족하면 CFG를 7.5로 올리는 게 실용 균형. 의사 결정: **양쪽 임계값** 설정 (SSIM>0.85 & LPIPS<0.20) 후 범위 내에서 CFG 선택.
