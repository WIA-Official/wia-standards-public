Detection Envelopes and the MITRE ATT&CK Mapping

The detection_event envelope is the smallest unit of shareable observation in this standard. It exists for one reason: when an EDR fires, a SIEM correlates, or a canary trips, the resulting record needs to be unambiguous to every downstream consumer without custom translation logic per pair of systems.

2.1 The envelope schema

An envelope carries an identifier, a tenant, an observation timestamp, a confidence score, an evidence hash, an array of MITRE ATT&CK v15 Enterprise triples, and an Ed25519 signature over the canonical JSON form (RFC 8785 JCS).

{
  "event_id": "01HXR8E2ZQ9V8K3M5W7P0T2N4F",
  "tenant_id": "did:wia:tenant:acme-eu",
  "observed_at": "2026-04-27T14:22:18.412Z",
  "mitre_attack": [
    { "tactic": "TA0040", "technique": "T1486" },
    { "tactic": "TA0040", "technique": "T1490" }
  ],
  "confidence": 0.94,
  "evidence_hash": "sha384-...",
  "severity": "CRITICAL",
  "signature": { "alg": "Ed25519", "value": "..." }
}

2.2 Why MITRE alignment is mandatory

Every detector eventually develops a vendor-specific ontology. One EDR calls it "Mass File Encryption," another calls it "Crypto Activity," a third calls it "Ransomware Behaviour Pattern 7." Without a common reference frame, downstream correlation collapses into string matching, and string matching collapses into noise. ATT&CK is mandatory because it is the one ontology every mature SOC already speaks and because ATT&CK identifiers are stable across versions.

2.3 Confidence is calibrated, not heuristic

The confidence field is a calibrated posterior probability, not a marketing number. A detector emitting 0.94 is asserting that, over the population of observations it has emitted at this score in the past, roughly 94 % were confirmed true positives in subsequent triage. Vendors that cannot produce calibrated scores SHOULD emit null for this field rather than a heuristic.

2.4 Detector classes

The standard recognises four detector classes: behavioural canary, cryptographic anomaly, identity exfiltration, and lateral movement. Each class carries different operational implications. A canary trip is high-confidence by construction (legitimate traffic should not touch a honeyfile) but narrow. A cryptographic-anomaly detector has broader coverage but a higher base-rate false-positive risk during backup windows or installer rollouts.

2.5 Triage state and analyst feedback

Analyst dispositions — true positive, false positive, benign positive — flow back through triage_state envelopes that reference the original event by event_id. The standard never mutates the original signed envelope; the triage state is a separate signed envelope, so that the chain of custody on the original detection remains intact even after a closing disposition is recorded.

2.6 Federation gating

Detection envelopes flagged with severity ≥ HIGH and confidence ≥ 0.85 are eligible for cross-jurisdictional ISAC propagation, gated by the tenant's share_with allowlist. The default is safe: nothing leaves the tenant unless the tenant explicitly opted in, even when the detection is severe.

2.7 Replay defence

Detection envelopes carry a 96-bit nonce and are accepted with a clock skew tolerance of ±300 seconds. Replays within a 600-second cache window are rejected. The defence is not negotiable; replayed detections are how an attacker hides re-entry by drowning the SOC in known-noise events.

Worked example

Consider a mid-size hospital network running 18 EHR servers, 240 imaging workstations, 42 IoT medical devices, and a separate biomedical engineering subnet. The SOC has one full-time analyst, two contracted analysts, and an MSSP relationship. The cyber-insurer requires annual posture evidence; the national health regulator requires 24-hour breach notification.

In a pre-standard world, the SOC re-keys every observation into the EDR portal, the SIEM, the MSSP ticketing system, the insurer questionnaire, and the regulator filing form. The total work per real incident exceeds the analyst headcount by a factor of three to five during the first 72 hours, and that is the work done after the technical containment is finished.

In a post-standard world, the same observation is signed once, propagated automatically to every consumer that has subscribed to the tenant under appropriate TLP and federation rules, and rendered into each consumer’s preferred form by a downstream adaptor. The SOC analyst spends the first 72 hours doing analysis and decision-making, not data entry. That is the entire return on signing the envelope.
