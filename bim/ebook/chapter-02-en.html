Chapter 2: BIM Maturity Levels (LOD)

Level of Development Framework for Model Progression

2.1 Understanding Level of Development

Level of Development (LOD) is a fundamental concept in BIM that defines the degree of completeness and reliability of information contained within model elements at various stages of design and construction. LOD provides a common framework for communicating expectations about model content, enabling project teams to understand exactly what level of detail and accuracy is present in the model at any given point.

The LOD specification, originally developed by the American Institute of Architects (AIA) and now maintained by BIMForum, defines five primary levels ranging from LOD 100 (conceptual) to LOD 500 (as-built verification). Each level represents a progressive increase in geometric detail, information richness, and reliability for downstream uses such as quantity takeoffs, coordination, and construction.

LOD vs. LOI

It is important to distinguish between Level of Development (LOD) and Level of Information (LOI). While LOD encompasses both geometry and data, some frameworks separate geometric detail (LOG - Level of Geometry) from non-graphic information (LOI). WIA-BIM uses the integrated LOD approach but recognizes that some projects may require specifying LOG and LOI independently.

2.2 LOD Levels Explained

LOD 100 - Conceptual

Elements are represented as symbols or generic representations. At this level, model elements are not geometric representations but rather placeholders indicating the existence and approximate location of components. LOD 100 is suitable for early-stage programming, massing studies, and space allocation.

  • Geometry: Symbol, mass, or placeholder only
  • Information: Existence and approximate location
  • Reliability: Cannot be used for measurements or quantities
  • Use Cases: Conceptual design, feasibility studies, area calculations

LOD 200 - Schematic Design

Elements are modeled as generic systems or assemblies with approximate quantities, size, shape, location, and orientation. Non-graphic information may be attached. LOD 200 models support schematic design decisions and preliminary cost estimation.

  • Geometry: Generic size, shape, location, orientation
  • Information: Approximate quantities and system types
  • Reliability: Suitable for preliminary analysis and estimation
  • Use Cases: Schematic design, outline specifications, preliminary scheduling

LOD 300 - Design Development

Elements are modeled with specific assemblies accurate in terms of quantity, size, shape, location, and orientation. Full 3D geometric coordination is possible. LOD 300 is the typical baseline for clash detection and coordination activities.

  • Geometry: Accurate size, shape, location, orientation
  • Information: Specific assemblies, detailed properties
  • Reliability: Suitable for coordination and quantity takeoffs
  • Use Cases: Design development, 3D coordination, detailed cost estimation

LOD 350 - Construction Documentation

Elements include detailing necessary for coordination with other building systems, including supports, connections, and interfaces. LOD 350 extends LOD 300 to include sufficient detail for trade coordination and shop drawing preparation.

  • Geometry: Includes connections and interface details
  • Information: Installation details, support requirements
  • Reliability: Suitable for construction coordination
  • Use Cases: Trade coordination, clash resolution, shop drawing support

LOD 400 - Fabrication/Assembly

Elements are modeled with sufficient detail and accuracy for fabrication of the represented component. This includes fabrication, assembly, and installation information. LOD 400 supports direct digital fabrication workflows.

  • Geometry: Fabrication-level detail and tolerances
  • Information: Complete fabrication and installation data
  • Reliability: Suitable for direct fabrication
  • Use Cases: Digital fabrication, prefabrication, CNC machining

LOD 500 - As-Built/Record

Elements represent field-verified representation of the component as actually constructed. LOD 500 is not about additional geometric detail but rather verification that the model accurately reflects built conditions. This level supports facility management and operations.

  • Geometry: Field-verified accurate representation
  • Information: As-built data, installation dates, warranties
  • Reliability: Verified against actual construction
  • Use Cases: Facility management, asset management, operations

2.4 WIA-BIM LOD Specification Schema

// WIA-BIM LOD Specification Interface
interface LODSpecification {
    projectId: string;
    version: string;
    elements: LODElement[];
}

interface LODElement {
    elementType: string;              // IFC entity type
    lodRequirements: LODRequirement[];
}

interface LODRequirement {
    phase: ProjectPhase;              // Design phase
    requiredLOD: LODLevel;            // Minimum LOD required
    author: string;                   // Responsible party
    geometry: GeometryRequirement;    // Geometric detail
    information: InfoRequirement[];   // Required properties
}

type LODLevel = 100 | 200 | 300 | 350 | 400 | 500;

type ProjectPhase =
    | 'CONCEPT'
    | 'SCHEMATIC_DESIGN'
    | 'DESIGN_DEVELOPMENT'
    | 'CONSTRUCTION_DOCUMENTS'
    | 'CONSTRUCTION'
    | 'AS_BUILT';

interface GeometryRequirement {
    representation: 'SYMBOL' | 'GENERIC' | 'SPECIFIC' | 'FABRICATION' | 'VERIFIED';
    tolerance?: number;               // Dimensional tolerance in mm
    includeConnections?: boolean;     // Include interface geometry
}

interface InfoRequirement {
    propertySet: string;              // IFC property set name
    property: string;                 // Property name
    required: boolean;                // Mandatory flag
    dataType: string;                 // Expected data type
}

2.5 LOD Validation API

// WIA-BIM SDK: LOD Validation Service
import { WIABimClient, IfcModel, LODValidator } from '@wia/bim-sdk';

class LODValidationService {
    private validator: LODValidator;

    constructor(lodSpec: LODSpecification) {
        this.validator = new LODValidator(lodSpec);
    }

    async validateModel(model: IfcModel, phase: ProjectPhase): Promise<ValidationReport> {
        const elements = await model.getAllElements();
        const results: ElementValidation[] = [];

        for (const element of elements) {
            const requirement = this.validator.getRequirement(element.type, phase);
            if (!requirement) continue;

            const validation = await this.validateElement(element, requirement);
            results.push(validation);
        }

        return {
            modelId: model.globalId,
            phase: phase,
            timestamp: new Date().toISOString(),
            totalElements: results.length,
            compliant: results.filter(r => r.compliant).length,
            nonCompliant: results.filter(r => !r.compliant).length,
            details: results
        };
    }

    private async validateElement(
        element: IfcElement,
        requirement: LODRequirement
    ): Promise<ElementValidation> {
        const geometryValid = await this.validateGeometry(element, requirement.geometry);
        const infoValid = await this.validateInformation(element, requirement.information);

        return {
            globalId: element.globalId,
            name: element.name,
            type: element.type,
            requiredLOD: requirement.requiredLOD,
            geometryCompliant: geometryValid.compliant,
            geometryIssues: geometryValid.issues,
            informationCompliant: infoValid.compliant,
            informationIssues: infoValid.issues,
            compliant: geometryValid.compliant && infoValid.compliant
        };
    }

    private async validateGeometry(
        element: IfcElement,
        requirement: GeometryRequirement
    ): Promise<{ compliant: boolean; issues: string[] }> {
        const issues: string[] = [];
        const geometry = await element.getGeometry();

        // Check representation type
        const actualRep = this.determineRepType(geometry);
        if (this.repLevel(actualRep) < this.repLevel(requirement.representation)) {
            issues.push(`Geometry representation insufficient: ${actualRep} < ${requirement.representation}`);
        }

        // Check tolerance if specified
        if (requirement.tolerance && geometry.precision > requirement.tolerance) {
            issues.push(`Geometry precision exceeds tolerance: ${geometry.precision}mm > ${requirement.tolerance}mm`);
        }

        // Check connections if required
        if (requirement.includeConnections) {
            const connections = await element.getConnections();
            if (connections.length === 0) {
                issues.push('Connection geometry required but not present');
            }
        }

        return { compliant: issues.length === 0, issues };
    }
}

// Example Usage
const spec = await loadLODSpecification('project-lod-spec.json');
const service = new LODValidationService(spec);
const model = await client.loadModel('architecture.ifc');
const report = await service.validateModel(model, 'DESIGN_DEVELOPMENT');

console.log(`LOD Validation: ${report.compliant}/${report.totalElements} elements compliant`);

BIM Execution Plan Integration

LOD requirements should be documented in the project's BIM Execution Plan (BEP). The WIA-BIM standard provides templates for LOD matrices that align with project milestones, ensuring all stakeholders understand their modeling responsibilities throughout the project lifecycle.

2.6 Best Practices for LOD Management

  • Define LOD requirements early: Establish LOD expectations during project setup and document in the BEP
  • Assign authoring responsibility: Clearly define which party is responsible for developing each element to its required LOD
  • Implement progressive development: Plan for incremental LOD progression through project phases
  • Validate continuously: Use automated LOD validation tools to ensure compliance throughout the project
  • Document deviations: Record any agreed-upon deviations from standard LOD requirements
  • Consider downstream uses: Select LOD based on intended model uses, not arbitrary targets
