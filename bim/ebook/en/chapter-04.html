4.1 Schema Design Philosophy

The WIA-BIM Data Format layer establishes the foundational vocabulary and structure for representing building information. This schema is designed to be comprehensive yet flexible, enabling rich data representation while maintaining compatibility with existing standards.

Design Goal: Create a universal data format that can represent any building element with sufficient detail for design, construction, and operations while remaining efficient enough for real-time collaboration.

4.1.1 Schema Principles

Principle Description Implementation
Self-Describing Data includes its own type information Type fields in every object
Normalized No data duplication Reference-based relationships
Versioned Schema evolution support Version headers, migration paths
Extensible Custom properties allowed Namespace prefixes for extensions
Validated Structural correctness ensured JSON Schema validation

4.4 Property Sets

Property Sets (Psets) define standardized collections of properties for specific object types, ensuring consistent data representation across all implementations.

4.4.1 Standard Property Sets

Property Set Applies To Key Properties
Pset_WallCommon All walls Reference, LoadBearing, ExtendToStructure, IsExternal
Pset_DoorCommon All doors Reference, FireRating, AcousticRating, SecurityRating
Pset_SpaceCommon All spaces Reference, NetPlannedArea, FinishFloorHeight, OccupancyNumber
Pset_MaterialOptical Materials VisibleTransmittance, SolarTransmittance, ReflectanceBack
Pset_ManufacturerTypeInfo Products GlobalTradeItemNumber, ModelReference, ModelLabel

4.6 Relationship Modeling

WIA-BIM uses explicit relationship objects to define connections between elements, enabling rich semantic understanding of the building model.

4.6.1 Relationship Types

Relationship Description Example
RelContainedIn Spatial containment Door contained in Wall
RelConnects Physical connection Wall connects to Wall
RelSpaceBoundary Space delimitation Wall bounds Space
RelVoids Opening in element Opening voids Wall
RelFills Element fills opening Door fills Opening
RelAggregates Part-whole assembly CurtainWall aggregates Panels
RelAssigns Assignment Equipment assigned to System

4.6.2 Relationship Structure

  "relationships": [

4.7 Schema Validation

4.7.1 Validation Rules

// WIA-BIM Validation Schema (JSON Schema format)
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "$id": "https://wia.org/schemas/bim/v1.0/object.json",
  "title": "WIA-BIM Object",
  "type": "object",
  "required": ["objectType", "guid", "name"],
  "properties": {
    "objectType": {
      "type": "string",
      "enum": ["Wall", "Door", "Window", "Space", "..."]
    },
    "guid": {
      "type": "string",
      "pattern": "^[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$"
    },
    "name": {
      "type": "string",
      "minLength": 1,
      "maxLength": 255
    },
    "properties": {
      "type": "object",
      "additionalProperties": true
    }
  },
  "allOf": [
    {
      "if": { "properties": { "objectType": { "const": "Door" } } },
      "then": { "$ref": "#/definitions/doorProperties" }
    }
  ]
}

4.8 Chapter Summary

Key Takeaways

  • WIA-BIM schema is self-describing, normalized, versioned, and extensible
  • Object hierarchy provides consistent structure from base class to specific elements
  • Property sets standardize data representation across all implementations
  • Multiple geometry representations support different use cases and LODs
  • Explicit relationships enable rich semantic understanding of building models
  • JSON Schema validation ensures data quality and compliance
