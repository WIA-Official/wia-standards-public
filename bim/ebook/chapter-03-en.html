Chapter 3: Common Data Environment (CDE)

Collaborative Platforms for BIM Information Management

3.1 What is a Common Data Environment?

A Common Data Environment (CDE) is the single source of truth for all project information throughout the asset lifecycle. As defined by ISO 19650, the CDE is an agreed source of information for any given project or asset, used to collect, manage, and disseminate documents, models, and other data containers through a managed process.

The CDE concept represents a fundamental shift from traditional file-based document management to a collaborative, process-driven approach where information flows through defined states and gates. This ensures that the right information reaches the right people at the right time, with full traceability and audit capability.

CDE vs Traditional File Sharing

Unlike simple cloud storage or file sharing systems, a CDE provides structured workflows, approval processes, version control, and status management. It is not just where files live but how they progress through their lifecycle from work-in-progress to shared, published, and archived states.

3.2 ISO 19650 CDE Framework

ISO 19650 defines the CDE workflow using four distinct states, often referred to as "suitability codes" or status levels. Information containers (models, drawings, documents) progress through these states as they mature and gain approval. 

CDE Workflow States (ISO 19650)
================================

+----------------+     +----------------+     +----------------+     +----------------+
|   WORK IN      |     |    SHARED      |     |   PUBLISHED    |     |   ARCHIVED     |
|   PROGRESS     |---->|                |---->|                |---->|                |
|                |     |                |     |                |     |                |
| - Individual   |     | - Team review  |     | - Client       |     | - Historical   |
|   authoring    |     | - Coordination |     |   approved     |     |   record       |
| - Not for use  |     | - Suitable for |     | - Issued for   |     | - Superseded   |
| - Draft state  |     |   coordination |     |   construction |     |   versions     |
+----------------+     +----------------+     +----------------+     +----------------+

Status Codes:
- S0: Work In Progress (not suitable for coordination)
- S1: Suitable for coordination
- S2: Suitable for information
- S3: Suitable for review and comment
- S4: Suitable for stage approval
- A1-A6: Authorized for specific purposes (construction, etc.)

3.5 CDE API Integration

// WIA-BIM SDK: CDE Operations API
import { WIACDEClient, Document, Model, Status } from '@wia/bim-cde';

class CDEService {
    private client: WIACDEClient;

    constructor(config: CDEConfig) {
        this.client = new WIACDEClient({
            endpoint: config.endpoint,
            projectId: config.projectId,
            apiKey: config.apiKey
        });
    }

    // Upload model to CDE with metadata
    async uploadModel(
        file: File,
        metadata: ModelMetadata
    ): Promise<UploadResult> {
        const container = await this.client.createContainer({
            name: this.generateName(metadata),
            type: 'MODEL',
            discipline: metadata.discipline,
            status: Status.WORK_IN_PROGRESS,
            revision: metadata.revision
        });

        const result = await this.client.uploadFile(container.id, file);

        // Validate IFC content
        const validation = await this.client.validateIFC(result.fileId);
        if (!validation.passed) {
            throw new ValidationError(validation.errors);
        }

        return {
            containerId: container.id,
            fileId: result.fileId,
            name: container.name,
            status: container.status,
            validationReport: validation
        };
    }

    // Transition model through workflow states
    async transitionStatus(
        containerId: string,
        newStatus: Status,
        approver: string,
        comments?: string
    ): Promise<TransitionResult> {
        // Validate transition is allowed
        const container = await this.client.getContainer(containerId);
        const allowedTransitions = this.getAllowedTransitions(container.status);

        if (!allowedTransitions.includes(newStatus)) {
            throw new WorkflowError(
                `Cannot transition from ${container.status} to ${newStatus}`
            );
        }

        // Execute transition with audit trail
        return await this.client.transition({
            containerId,
            fromStatus: container.status,
            toStatus: newStatus,
            approver,
            comments,
            timestamp: new Date().toISOString()
        });
    }

    // Query containers by status and discipline
    async queryContainers(
        filters: QueryFilters
    ): Promise<Container[]> {
        return await this.client.query({
            status: filters.status,
            discipline: filters.discipline,
            dateRange: filters.dateRange,
            originator: filters.originator,
            includeSuperseded: false
        });
    }

    // Create federated model from discipline models
    async createFederatedModel(
        sourceContainers: string[],
        federatedName: string
    ): Promise<FederatedModel> {
        const sources = await Promise.all(
            sourceContainers.map(id => this.client.getLatestFile(id))
        );

        return await this.client.federate({
            name: federatedName,
            sources: sources.map(s => ({
                containerId: s.containerId,
                fileId: s.fileId,
                transform: s.coordinateTransform
            })),
            coordinationSpace: 'PROJECT_ORIGIN'
        });
    }

    private generateName(metadata: ModelMetadata): string {
        return [
            metadata.projectCode,
            metadata.originator,
            metadata.volume || 'ZZ',
            metadata.level || 'ZZ',
            metadata.form,
            metadata.discipline,
            metadata.number.toString().padStart(4, '0')
        ].join('-');
    }

    private getAllowedTransitions(current: Status): Status[] {
        const transitions: Record<Status, Status[]> = {
            [Status.WORK_IN_PROGRESS]: [Status.SHARED],
            [Status.SHARED]: [Status.WORK_IN_PROGRESS, Status.PUBLISHED],
            [Status.PUBLISHED]: [Status.ARCHIVED],
            [Status.ARCHIVED]: []
        };
        return transitions[current] || [];
    }
}

3.6 Access Control and Permissions

Robust access control is essential for CDE security and workflow integrity. WIA-BIM defines role-based access control (RBAC) with discipline-specific permissions.

Role WIP Access Shared Access Published Access Approve
Author Read/Write (own) Read Read No
Discipline Lead Read/Write (discipline) Read/Write Read To Shared
BIM Coordinator Read (all) Read/Write Read To Published
Project Manager Read (all) Read Read/Write All
Client None None Read No

Audit Trail Requirements

WIA-BIM mandates complete audit trails for all CDE operations including uploads, downloads, status transitions, and access attempts. Audit logs must be immutable, timestamped, and retained for the asset lifecycle plus applicable legal retention periods.

3.7 Version Control Best Practices

  • Never overwrite: Always create new versions rather than replacing files
  • Meaningful revision codes: Use P01, P02 for preliminary, C01, C02 for construction
  • Link revisions to reasons: Every revision must have documented rationale
  • Maintain relationships: Track dependencies between related documents
  • Supersede explicitly: Mark replaced versions as superseded with clear links
  • Archive strategically: Move to archive only after replacement is approved
