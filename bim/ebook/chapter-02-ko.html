2장: BIM 성숙도 수준 (LOD)

모델 진행을 위한 개발 수준 프레임워크

2.1 개발 수준 이해하기

개발 수준(LOD, Level of Development)은 설계 및 시공의 다양한 단계에서 모델 요소에 포함된 정보의 완성도와 신뢰성을 정의하는 BIM의 기본 개념입니다. LOD는 모델 내용에 대한 기대를 전달하기 위한 공통 프레임워크를 제공하여, 프로젝트 팀이 특정 시점에 모델에 존재하는 세부 수준과 정확도를 정확히 이해할 수 있게 합니다.

미국 건축가 협회(AIA)에서 원래 개발하고 현재 BIMForum에서 관리하는 LOD 사양은 LOD 100(개념)에서 LOD 500(준공 검증)까지 5개의 주요 레벨을 정의합니다. 각 레벨은 기하학적 세부 사항, 정보 풍부성 및 수량 산출, 조정 및 시공과 같은 하류 용도에 대한 신뢰성의 점진적인 증가를 나타냅니다.

LOD vs. LOI

개발 수준(LOD)과 정보 수준(LOI)을 구별하는 것이 중요합니다. LOD는 기하학과 데이터를 모두 포함하지만, 일부 프레임워크는 기하학적 세부 사항(LOG - 기하학 수준)을 비그래픽 정보(LOI)와 분리합니다. WIA-BIM은 통합 LOD 접근 방식을 사용하지만 일부 프로젝트에서 LOG와 LOI를 독립적으로 지정해야 할 수 있음을 인정합니다.

2.2 LOD 레벨 설명

LOD 100 - 개념

요소는 기호 또는 일반적인 표현으로 나타납니다. 이 수준에서 모델 요소는 기하학적 표현이 아니라 구성 요소의 존재와 대략적인 위치를 나타내는 자리 표시자입니다. LOD 100은 초기 단계 프로그래밍, 매싱 연구 및 공간 배분에 적합합니다.

  • 기하학: 기호, 매스 또는 자리 표시자만
  • 정보: 존재 및 대략적인 위치
  • 신뢰성: 측정이나 수량에 사용할 수 없음
  • 활용 사례: 개념 설계, 타당성 조사, 면적 계산

LOD 200 - 기본 설계

요소는 대략적인 수량, 크기, 형상, 위치 및 방향을 가진 일반적인 시스템 또는 어셈블리로 모델링됩니다. 비그래픽 정보가 첨부될 수 있습니다. LOD 200 모델은 기본 설계 결정 및 예비 비용 추정을 지원합니다.

  • 기하학: 일반적인 크기, 형상, 위치, 방향
  • 정보: 대략적인 수량 및 시스템 유형
  • 신뢰성: 예비 분석 및 추정에 적합
  • 활용 사례: 기본 설계, 개략 사양, 예비 일정

LOD 300 - 실시 설계

요소는 수량, 크기, 형상, 위치 및 방향 측면에서 정확한 특정 어셈블리로 모델링됩니다. 완전한 3D 기하학적 조정이 가능합니다. LOD 300은 간섭 검토 및 조정 활동의 일반적인 기준선입니다.

  • 기하학: 정확한 크기, 형상, 위치, 방향
  • 정보: 특정 어셈블리, 상세 속성
  • 신뢰성: 조정 및 수량 산출에 적합
  • 활용 사례: 실시 설계, 3D 조정, 상세 비용 추정

LOD 350 - 시공 문서

요소에는 지지대, 연결부 및 인터페이스를 포함하여 다른 건물 시스템과의 조정에 필요한 세부 사항이 포함됩니다. LOD 350은 LOD 300을 확장하여 공종 조정 및 제작도 준비에 충분한 세부 사항을 포함합니다.

  • 기하학: 연결부 및 인터페이스 세부 사항 포함
  • 정보: 설치 세부 사항, 지지 요구 사항
  • 신뢰성: 시공 조정에 적합
  • 활용 사례: 공종 조정, 간섭 해결, 제작도 지원

LOD 400 - 제작/조립

요소는 표현된 구성 요소의 제작에 충분한 세부 사항과 정확도로 모델링됩니다. 여기에는 제작, 조립 및 설치 정보가 포함됩니다. LOD 400은 직접 디지털 제작 워크플로우를 지원합니다.

  • 기하학: 제작 수준의 세부 사항 및 공차
  • 정보: 완전한 제작 및 설치 데이터
  • 신뢰성: 직접 제작에 적합
  • 활용 사례: 디지털 제작, 프리패브, CNC 가공

LOD 500 - 준공/기록

요소는 실제로 시공된 구성 요소의 현장 검증된 표현을 나타냅니다. LOD 500은 추가 기하학적 세부 사항이 아니라 모델이 시공된 조건을 정확하게 반영하는지 확인하는 것입니다. 이 수준은 시설 관리 및 운영을 지원합니다.

  • 기하학: 현장 검증된 정확한 표현
  • 정보: 준공 데이터, 설치 날짜, 보증
  • 신뢰성: 실제 시공과 대조하여 검증됨
  • 활용 사례: 시설 관리, 자산 관리, 운영

2.4 WIA-BIM LOD 사양 스키마

// WIA-BIM LOD 사양 인터페이스
interface LODSpecification {
    projectId: string;                // 프로젝트 ID
    version: string;                  // 버전
    elements: LODElement[];           // 요소 목록
}

interface LODElement {
    elementType: string;              // IFC 엔티티 유형
    lodRequirements: LODRequirement[];// LOD 요구사항
}

interface LODRequirement {
    phase: ProjectPhase;              // 설계 단계
    requiredLOD: LODLevel;            // 필수 최소 LOD
    author: string;                   // 책임자
    geometry: GeometryRequirement;    // 기하학적 세부사항
    information: InfoRequirement[];   // 필수 속성
}

type LODLevel = 100 | 200 | 300 | 350 | 400 | 500;

type ProjectPhase =
    | 'CONCEPT'                       // 개념
    | 'SCHEMATIC_DESIGN'              // 기본 설계
    | 'DESIGN_DEVELOPMENT'            // 실시 설계
    | 'CONSTRUCTION_DOCUMENTS'        // 시공 문서
    | 'CONSTRUCTION'                  // 시공
    | 'AS_BUILT';                     // 준공

interface GeometryRequirement {
    representation: 'SYMBOL' | 'GENERIC' | 'SPECIFIC' | 'FABRICATION' | 'VERIFIED';
    tolerance?: number;               // mm 단위 치수 공차
    includeConnections?: boolean;     // 인터페이스 기하학 포함
}

interface InfoRequirement {
    propertySet: string;              // IFC 속성 세트 이름
    property: string;                 // 속성 이름
    required: boolean;                // 필수 플래그
    dataType: string;                 // 예상 데이터 유형
}

2.5 LOD 검증 API

// WIA-BIM SDK: LOD 검증 서비스
import { WIABimClient, IfcModel, LODValidator } from '@wia/bim-sdk';

class LODValidationService {
    private validator: LODValidator;

    constructor(lodSpec: LODSpecification) {
        this.validator = new LODValidator(lodSpec);
    }

    async validateModel(model: IfcModel, phase: ProjectPhase): Promise<ValidationReport> {
        const elements = await model.getAllElements();
        const results: ElementValidation[] = [];

        for (const element of elements) {
            const requirement = this.validator.getRequirement(element.type, phase);
            if (!requirement) continue;

            const validation = await this.validateElement(element, requirement);
            results.push(validation);
        }

        return {
            modelId: model.globalId,
            phase: phase,
            timestamp: new Date().toISOString(),
            totalElements: results.length,
            compliant: results.filter(r => r.compliant).length,
            nonCompliant: results.filter(r => !r.compliant).length,
            details: results
        };
    }

    private async validateElement(
        element: IfcElement,
        requirement: LODRequirement
    ): Promise<ElementValidation> {
        const geometryValid = await this.validateGeometry(element, requirement.geometry);
        const infoValid = await this.validateInformation(element, requirement.information);

        return {
            globalId: element.globalId,
            name: element.name,
            type: element.type,
            requiredLOD: requirement.requiredLOD,
            geometryCompliant: geometryValid.compliant,
            geometryIssues: geometryValid.issues,
            informationCompliant: infoValid.compliant,
            informationIssues: infoValid.issues,
            compliant: geometryValid.compliant && infoValid.compliant
        };
    }
}

// 사용 예시
const spec = await loadLODSpecification('project-lod-spec.json');
const service = new LODValidationService(spec);
const model = await client.loadModel('architecture.ifc');
const report = await service.validateModel(model, 'DESIGN_DEVELOPMENT');

console.log(`LOD 검증: ${report.compliant}/${report.totalElements}개 요소 준수`);

BIM 실행 계획 통합

LOD 요구 사항은 프로젝트의 BIM 실행 계획(BEP)에 문서화되어야 합니다. WIA-BIM 표준은 프로젝트 마일스톤과 일치하는 LOD 매트릭스 템플릿을 제공하여, 모든 이해관계자가 프로젝트 수명주기 전반에 걸친 모델링 책임을 이해하도록 합니다.

2.6 LOD 관리 모범 사례

  • LOD 요구 사항을 조기에 정의: 프로젝트 설정 중 LOD 기대치를 수립하고 BEP에 문서화
  • 작성 책임 할당: 각 요소를 필요한 LOD로 개발하는 책임 당사자를 명확히 정의
  • 점진적 개발 구현: 프로젝트 단계를 통한 점진적 LOD 진행 계획
  • 지속적 검증: 자동화된 LOD 검증 도구를 사용하여 프로젝트 전반에 걸쳐 준수 보장
  • 편차 문서화: 표준 LOD 요구 사항에서 합의된 편차를 기록
  • 하류 용도 고려: 임의 목표가 아닌 의도된 모델 용도에 따라 LOD 선택
