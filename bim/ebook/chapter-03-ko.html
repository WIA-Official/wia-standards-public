3장: 공통 데이터 환경 (CDE)

BIM 정보 관리를 위한 협업 플랫폼

3.1 공통 데이터 환경이란?

공통 데이터 환경(CDE, Common Data Environment)은 자산 수명주기 전반에 걸쳐 모든 프로젝트 정보의 단일 진실 소스입니다. ISO 19650에서 정의한 바와 같이, CDE는 관리되는 프로세스를 통해 문서, 모델 및 기타 데이터 컨테이너를 수집, 관리 및 배포하는 데 사용되는 주어진 프로젝트 또는 자산에 대한 합의된 정보 소스입니다.

CDE 개념은 정보가 정의된 상태와 게이트를 통해 흐르는 협업적, 프로세스 중심 접근 방식으로의 전통적인 파일 기반 문서 관리에서 근본적인 전환을 나타냅니다. 이를 통해 적절한 정보가 적절한 시간에 적절한 사람에게 전달되며, 완전한 추적성과 감사 기능을 제공합니다.

CDE vs 전통적 파일 공유

단순한 클라우드 스토리지나 파일 공유 시스템과 달리, CDE는 구조화된 워크플로우, 승인 프로세스, 버전 관리 및 상태 관리를 제공합니다. 이는 단순히 파일이 존재하는 곳이 아니라 작업 진행 중에서 공유, 게시, 보관 상태까지 수명주기를 통해 파일이 어떻게 진행되는지를 나타냅니다.

3.2 ISO 19650 CDE 프레임워크

ISO 19650은 종종 "적합성 코드" 또는 상태 수준이라고 하는 네 가지 구별되는 상태를 사용하여 CDE 워크플로우를 정의합니다. 정보 컨테이너(모델, 도면, 문서)는 성숙하고 승인을 받으면서 이러한 상태를 통해 진행됩니다. 

CDE 워크플로우 상태 (ISO 19650)
================================

+----------------+     +----------------+     +----------------+     +----------------+
|   작업 중       |     |    공유        |     |    게시        |     |    보관        |
|  (WIP)         |---->|   (SHARED)     |---->|  (PUBLISHED)   |---->|  (ARCHIVED)    |
|                |     |                |     |                |     |                |
| - 개인 작성     |     | - 팀 검토      |     | - 고객 승인     |     | - 이력 기록    |
| - 사용 불가     |     | - 조정         |     | - 시공용 발행   |     | - 대체된 버전  |
| - 초안 상태     |     | - 조정에 적합   |     |                |     |                |
+----------------+     +----------------+     +----------------+     +----------------+

상태 코드:
- S0: 작업 진행 중 (조정에 부적합)
- S1: 조정에 적합
- S2: 정보 제공에 적합
- S3: 검토 및 의견에 적합
- S4: 단계 승인에 적합
- A1-A6: 특정 목적 승인 (시공 등)

3.4 명명 규칙

일관된 명명 규칙은 CDE 효율성에 매우 중요합니다. WIA-BIM은 실용적인 구현을 위해 조정된 ISO 19650-2 권장 명명 규칙 구조를 채택합니다.

필드 설명 예시
프로젝트 프로젝트 식별 코드 PRJ001
작성자 콘텐츠를 생성하는 조직 ABC (건축사무소)
기능 분류 볼륨/존/시스템 ZZ (전체 프로젝트)
공간 분류 레벨/영역 L01 (1층)
형식 파일 유형 M3 (3D 모델)
분야 설계 분야 A (건축)
번호 일련번호 0001 
// ISO 19650-2 규칙을 따르는 파일 명명 예시
// 형식: [프로젝트]-[작성자]-[볼륨]-[레벨]-[유형]-[분야]-[번호]

PRJ001-ABC-ZZ-L01-M3-A-0001.ifc      // 건축 모델 1층
PRJ001-XYZ-ZZ-L01-M3-S-0001.ifc      // 구조 모델 1층
PRJ001-DEF-ZZ-ZZ-M3-M-0001.ifc       // 기계설비 모델 (전체 층)
PRJ001-ABC-ZZ-L02-DR-A-0001.pdf      // 건축 도면 2층
PRJ001-MGR-ZZ-ZZ-CR-C-0001.bcf       // 간섭 보고서 (조정)

3.5 CDE API 통합

// WIA-BIM SDK: CDE 작업 API
import { WIACDEClient, Document, Model, Status } from '@wia/bim-cde';

class CDEService {
    private client: WIACDEClient;

    constructor(config: CDEConfig) {
        this.client = new WIACDEClient({
            endpoint: config.endpoint,
            projectId: config.projectId,
            apiKey: config.apiKey
        });
    }

    // 메타데이터와 함께 모델을 CDE에 업로드
    async uploadModel(
        file: File,
        metadata: ModelMetadata
    ): Promise<UploadResult> {
        const container = await this.client.createContainer({
            name: this.generateName(metadata),
            type: 'MODEL',
            discipline: metadata.discipline,
            status: Status.WORK_IN_PROGRESS,
            revision: metadata.revision
        });

        const result = await this.client.uploadFile(container.id, file);

        // IFC 내용 검증
        const validation = await this.client.validateIFC(result.fileId);
        if (!validation.passed) {
            throw new ValidationError(validation.errors);
        }

        return {
            containerId: container.id,
            fileId: result.fileId,
            name: container.name,
            status: container.status,
            validationReport: validation
        };
    }

    // 워크플로우 상태를 통해 모델 전환
    async transitionStatus(
        containerId: string,
        newStatus: Status,
        approver: string,
        comments?: string
    ): Promise<TransitionResult> {
        // 전환 허용 여부 검증
        const container = await this.client.getContainer(containerId);
        const allowedTransitions = this.getAllowedTransitions(container.status);

        if (!allowedTransitions.includes(newStatus)) {
            throw new WorkflowError(
                `${container.status}에서 ${newStatus}로 전환할 수 없습니다`
            );
        }

        // 감사 추적과 함께 전환 실행
        return await this.client.transition({
            containerId,
            fromStatus: container.status,
            toStatus: newStatus,
            approver,
            comments,
            timestamp: new Date().toISOString()
        });
    }

    // 상태 및 분야별 컨테이너 조회
    async queryContainers(
        filters: QueryFilters
    ): Promise<Container[]> {
        return await this.client.query({
            status: filters.status,
            discipline: filters.discipline,
            dateRange: filters.dateRange,
            originator: filters.originator,
            includeSuperseded: false
        });
    }

    // 분야별 모델에서 통합 모델 생성
    async createFederatedModel(
        sourceContainers: string[],
        federatedName: string
    ): Promise<FederatedModel> {
        const sources = await Promise.all(
            sourceContainers.map(id => this.client.getLatestFile(id))
        );

        return await this.client.federate({
            name: federatedName,
            sources: sources.map(s => ({
                containerId: s.containerId,
                fileId: s.fileId,
                transform: s.coordinateTransform
            })),
            coordinationSpace: 'PROJECT_ORIGIN'
        });
    }

    private generateName(metadata: ModelMetadata): string {
        return [
            metadata.projectCode,
            metadata.originator,
            metadata.volume || 'ZZ',
            metadata.level || 'ZZ',
            metadata.form,
            metadata.discipline,
            metadata.number.toString().padStart(4, '0')
        ].join('-');
    }
}

3.6 접근 제어 및 권한

강력한 접근 제어는 CDE 보안 및 워크플로우 무결성에 필수적입니다. WIA-BIM은 분야별 권한을 가진 역할 기반 접근 제어(RBAC)를 정의합니다.

역할 WIP 접근 공유 접근 게시 접근 승인
작성자 읽기/쓰기 (본인) 읽기 읽기 불가
분야 책임자 읽기/쓰기 (분야) 읽기/쓰기 읽기 공유까지
BIM 코디네이터 읽기 (전체) 읽기/쓰기 읽기 게시까지
프로젝트 관리자 읽기 (전체) 읽기 읽기/쓰기 전체
고객 없음 없음 읽기 불가

감사 추적 요구사항

WIA-BIM은 업로드, 다운로드, 상태 전환 및 접근 시도를 포함한 모든 CDE 작업에 대한 완전한 감사 추적을 의무화합니다. 감사 로그는 변경 불가능하고, 타임스탬프가 찍히며, 자산 수명주기와 해당 법적 보존 기간 동안 유지되어야 합니다.

3.7 버전 관리 모범 사례

  • 절대 덮어쓰지 않기: 파일을 교체하는 대신 항상 새 버전 생성
  • 의미 있는 리비전 코드: 예비용 P01, P02, 시공용 C01, C02 사용
  • 리비전과 이유 연결: 모든 리비전에는 문서화된 근거가 있어야 함
  • 관계 유지: 관련 문서 간의 종속성 추적
  • 명시적으로 대체: 대체된 버전을 명확한 링크와 함께 대체됨으로 표시
  • 전략적으로 보관: 대체본이 승인된 후에만 보관으로 이동
