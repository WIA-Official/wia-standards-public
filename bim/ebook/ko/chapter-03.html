Chapter 03: 협업 프로세스

BIM 프로젝트의 성공은 효과적인 협업에 달려 있습니다. CDE(Common Data Environment), BIM 워크플로우, 협업 도구, 정보 교환 프로토콜 등 효과적인 협업을 위한 프로세스와 환경에 대해 상세히 알아봅니다.

3.1 CDE (Common Data Environment)

CDE(Common Data Environment)는 프로젝트 정보를 수집, 관리, 배포하기 위한 단일화된 정보 소스입니다. ISO 19650에서 정의한 CDE는 BIM 협업의 핵심 인프라로, 모든 프로젝트 참여자가 동일한 정보에 접근하고 활용할 수 있도록 합니다.

CDE의 정의 (ISO 19650)

"프로젝트 또는 자산에 대한 정보를 수집, 관리, 배포하기 위해 합의된 정보 소스. 정보 컨테이너의 집합체와 해당 컨테이너를 관리하는 수단을 제공하는 솔루션."

CDE의 4가지 영역

CDE는 정보의 상태에 따라 4가지 영역으로 구분됩니다. 각 영역은 정보의 성숙도와 공유 범위를 나타냅니다.

CDE 영역 구조 (ISO 19650)
=====================================

┌─────────────────────────────────────────────────────────────┐
│                      CDE (공통 데이터 환경)                 │
├───────────────┬───────────────┬───────────────┬─────────────┤
│   작업 진행   │    공유       │    발행       │   아카이브  │
│   (WIP)       │   (SHARED)    │  (PUBLISHED)  │  (ARCHIVE)  │
├───────────────┼───────────────┼───────────────┼─────────────┤
│ • 개인/팀 작업│ • 팀 간 공유  │ • 공식 배포   │ • 승인 완료 │
│ • 검토 전     │ • 검토 중     │ • 승인 완료   │ • 장기 보관 │
│ • 미완성      │ • 조율 중     │ • 계약 의무   │ • 기록 보존 │
│               │               │               │             │
│ Status:       │ Status:       │ Status:       │ Status:     │
│ S0            │ S1, S2        │ S3, S4        │ S5+         │
└───────────────┴───────────────┴───────────────┴─────────────┘
        │               │               │               │
        ▼               ▼               ▼               ▼
    내부 작업       협업 검토       외부 배포       영구 보존

정보 흐름: WIP → SHARED → PUBLISHED → ARCHIVE

정보 컨테이너 상태 코드

코드 상태 설명
S0 작업 진행 작업 중, 공유 전
S1 조율용 공유 기하학적 조율을 위한 공유
S2 정보용 공유 참조 목적의 공유
S3 검토 및 의견 리드 디자이너 검토용
S4 승인용 공유 발주처 승인 요청용
A 승인 완료 공식 승인된 정보
B 조건부 승인 수정 조건 하 승인

3.2 BIM 워크플로우

BIM 워크플로우는 프로젝트 참여자들이 정보를 생산, 공유, 활용하는 일련의 과정입니다. 효과적인 워크플로우는 역할 분담, 정보 교환 시점, 검토 프로세스 등을 명확히 정의합니다.

BIM 협업 워크플로우
=====================================

[설계 단계]
                                    
건축 ─────┬─────────────────────────────────────────────► 설계 완료
          │                                                   ▲
          │  ①모델 공유                                       │
          ▼                                                   │
구조 ─────┼─────────────────────────────────────────────►     │
          │  ②참조/협의                                       │
          ▼                                                   │
기계 ─────┼─────────────────────────────────────────────►     │
          │                                                   │
          ▼  ③통합 검토                                       │
전기 ─────┼─────────────────────────────────────────────►     │
          │                                                   │
          ▼                                                   │
┌─────────┴─────────┐     ┌─────────────────┐               │
│   통합 모델       │ ──► │  충돌 검토      │ ──► 이슈 해결 ─┘
│   (Federated)     │     │  (Clash Review) │
└───────────────────┘     └─────────────────┘

주요 교환 시점:
• 매주: 분야별 모델 업로드
• 격주: 통합 충돌 검토 회의
• 월간: 설계 리뷰 미팅

협업 회의 유형

BIM 조율 회의 (BIM Coordination Meeting)

충돌 검토 결과를 바탕으로 분야 간 이슈를 해결하는 회의입니다. 매주 또는 격주 단위로 진행하며, 충돌 리포트, 이슈 로그를 기반으로 논의합니다.

설계 검토 회의 (Design Review Meeting)

3D 모델을 활용한 설계안 검토 회의입니다. 발주처, 설계팀, 시공사가 참여하여 설계 의도와 시공성을 검토합니다.

3.3 정보 교환 요구사항 (EIR)

EIR(Exchange Information Requirements)은 발주처가 BIM 정보에 대한 요구사항을 정의한 문서입니다. 프로젝트 시작 전에 작성되며, BIM 실행 계획(BEP)의 기준이 됩니다.

EIR의 구성 요소
  • 기술적 요구사항: 소프트웨어, 데이터 형식, LOD, 좌표계
  • 관리적 요구사항: 납품 일정, 검토 절차, 역할 분담
  • 상업적 요구사항: 계약 조건, 보안, 지적재산권
EIR 템플릿 예시
=====================================

1. 프로젝트 정보
   ├── 프로젝트명: [프로젝트명]
   ├── 발주처: [발주처명]
   ├── 위치: [프로젝트 위치]
   └── 규모: [연면적, 층수]

2. BIM 목표
   ├── 설계 조율 및 충돌 검토
   ├── 물량 산출 및 비용 관리
   ├── 시공 시뮬레이션 (4D)
   └── 시설 관리 정보 인계

3. 기술적 요구사항
   ├── 소프트웨어
   │   ├── 저작도구: Revit 2024, Tekla 2024
   │   ├── 검토도구: Navisworks, BIM Collab
   │   └── 교환형식: IFC 4.0, BCF 2.1
   │
   ├── LOD 요구사항
   │   ├── 기본설계: LOD 200
   │   ├── 실시설계: LOD 300
   │   └── 시공: LOD 400
   │
   └── 좌표계
       ├── 측지계: GRS80
       ├── 투영: UTM-K (EPSG:5186)
       └── 프로젝트 원점: [좌표]

4. 관리적 요구사항
   ├── 모델 납품 일정
   ├── 검토 절차
   └── 품질 기준

5. 상업적 요구사항
   ├── 데이터 소유권
   ├── 보안 등급
   └── 계약 조건

3.4 BIM 실행 계획 (BEP)

BEP(BIM Execution Plan)는 EIR에 대한 응답으로 작성되는 문서로, 프로젝트에서 BIM을 어떻게 수행할 것인지를 상세히 기술합니다. 수급자가 작성하며, 착수 전과 착수 후로 구분하여 작성할 수 있습니다.

BEP 주요 내용
=====================================

1. 프로젝트 정보
   └── 기본 정보, 참여 조직, 연락처

2. BIM 목표 및 활용
   ├── 주요 BIM 활용 분야
   │   ├── 설계 조율
   │   ├── 3D 시각화
   │   ├── 충돌 검토
   │   ├── 물량 산출
   │   └── 4D 시뮬레이션
   │
   └── 단계별 목표
       ├── 설계 단계: [목표]
       ├── 시공 단계: [목표]
       └── 운영 단계: [목표]

3. 조직 및 역할
   ├── BIM 관리자 (BIM Manager)
   ├── BIM 조율자 (BIM Coordinator)
   └── BIM 모델러 (BIM Modeler)

4. BIM 프로세스
   ├── 모델 개발 프로세스
   ├── 협업 프로세스
   ├── 검토 프로세스
   └── 승인 프로세스

5. 기술 환경
   ├── 소프트웨어
   ├── 하드웨어
   ├── 네트워크
   └── CDE 플랫폼

6. 모델 관리
   ├── 명명 규칙
   ├── 좌표계 설정
   ├── 모델 분할 전략
   └── 버전 관리

7. 납품 계획
   ├── 납품 일정
   ├── 납품 형식
   └── 품질 검증 방법

3.5 충돌 검토 프로세스

충돌 검토(Clash Detection)는 BIM의 핵심 가치 중 하나입니다. 다양한 분야의 모델을 통합하여 물리적 간섭, 공간 부족, 시공 순서 문제 등을 사전에 파악합니다.

충돌의 유형
  • Hard Clash: 물리적으로 중첩되는 경우 (배관 vs 구조체)
  • Soft Clash: 여유 공간 부족 (유지보수 공간, 단열재 등)
  • Workflow Clash: 시공 순서 충돌 (4D 관점)

3.6 BCF (BIM Collaboration Format)

BCF는 buildingSMART에서 개발한 개방형 이슈 관리 형식입니다. BIM 모델의 특정 위치와 관련된 이슈를 소프트웨어 중립적으로 공유할 수 있습니다.

BCF 이슈 구조
=====================================

BCF 파일 (.bcfzip)
├── bcf.version              # BCF 버전 정보
├── extensions.xml           # 확장 정의
└── [GUID]/                  # 이슈별 폴더
    ├── markup.bcf           # 이슈 내용
    │   ├── Topic            # 이슈 제목, 설명
    │   ├── Comment          # 코멘트 목록
    │   ├── Viewpoint        # 뷰포인트 참조
    │   └── BimSnippet       # BIM 스니펫
    │
    ├── viewpoint.bcfv       # 카메라 위치, 가시성
    │   ├── Components       # 관련 요소
    │   ├── Camera           # 카메라 설정
    │   └── ClippingPlanes   # 단면 설정
    │
    └── snapshot.png         # 스냅샷 이미지

BCF Topic 예시:
=====================================

  덕트와 보 간섭
  
    B동 3층 기계실 상부 덕트가 
    구조 보와 200mm 간섭 발생
  
  High
  Active
  kim@mep.com
  2025-02-15

3.7 역할과 책임

BIM 프로젝트에서는 전통적 역할 외에 BIM 관련 역할이 추가됩니다. 각 역할의 책임을 명확히 정의하고 협업 체계를 구축하는 것이 중요합니다.

역할 주요 책임 필요 역량
BIM 관리자 BIM 전략, 표준, 프로세스 총괄 BIM 전문성, 관리 능력
BIM 조율자 분야 간 모델 조율, 충돌 관리 다분야 이해, 조율 능력
분야별 BIM 리드 분야별 모델 품질, 납품 관리 해당 분야 전문성
BIM 모델러 모델 작성, 정보 입력 소프트웨어 숙련도
정보 관리자 CDE 운영, 정보 보안 IT, 데이터 관리

3.8 협업 도구

BIM 협업을 위한 다양한 도구와 플랫폼이 존재합니다. 프로젝트 규모, 요구사항, 예산에 따라 적절한 도구를 선택합니다.

CDE 플랫폼

  • Autodesk Construction Cloud (BIM 360 / ACC)
  • Bentley ProjectWise
  • Trimble Connect
  • Aconex (Oracle)
  • PlanGrid

모델 검토 도구

  • Navisworks (Autodesk)
  • Solibri Model Checker
  • BIMcollab
  • SimpleBIM

3.9 정리

효과적인 BIM 협업은 명확한 프로세스, 적절한 도구, 그리고 역할 분담에 의해 실현됩니다. CDE를 중심으로 정보를 관리하고, EIR과 BEP를 통해 요구사항과 실행 계획을 정의하며, BCF를 활용한 이슈 관리로 협업 효율을 높일 수 있습니다.

핵심 요약
  • CDE는 프로젝트 정보의 단일화된 소스
  • 4가지 영역: WIP, Shared, Published, Archive
  • EIR은 발주처의 BIM 요구사항 문서
  • BEP는 EIR에 대한 응답으로 실행 계획을 정의
  • 충돌 검토는 BIM의 핵심 가치
  • BCF는 개방형 이슈 관리 형식
  • 역할별 책임 명확화가 협업 성공의 열쇠
