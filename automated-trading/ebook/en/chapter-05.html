1. API Architecture Overview

Modern trading APIs typically provide both REST endpoints for request-response operations and WebSocket connections for real-time streaming data. Understanding when to use each protocol is essential for efficient system design.

Protocol Use Cases Latency Advantages Disadvantages
REST API Account info, order submission, history 50-200ms Simple, stateless, widely supported Higher latency, polling overhead
WebSocket Market data streams, order updates 5-50ms Real-time, bi-directional, efficient Stateful, connection management
FIX Protocol Institutional trading, OMS integration 1-10ms Industry standard, feature-rich Complex, requires parsing
Binary Protocols High-frequency trading <1ms Ultra-low latency, compact Vendor-specific, complexity

class MarketDataClient {
  private config: MarketDataConfig;
  private ws: WebSocket | null = null;
  private subscriptions: Set = new Set();
  private eventHandlers: Map = new Map();
  private reconnectAttempts: number = 0;
  private maxReconnectAttempts: number = 5;

  constructor(config: MarketDataConfig) {
    this.config = config;
  }

  async connect(): Promise {
    return new Promise((resolve, reject) => {
      this.ws = new WebSocket(this.config.wsEndpoint);

      this.ws.onopen = () => {
        console.log('Market data WebSocket connected');
        this.reconnectAttempts = 0;
        this.resubscribeAll();
        resolve();
      };

      this.ws.onmessage = (event) => {
        this.handleMessage(JSON.parse(event.data));
      };

      this.ws.onerror = (error) => {
        console.error('WebSocket error:', error);
        reject(error);
      };

      this.ws.onclose = () => {
        console.log('WebSocket closed');
        this.handleReconnect();
      };
    });
  }

  subscribeTicker(symbol: string, callback: (ticker: Ticker) => void): void {
    const channel = `ticker.${symbol}`;
    this.subscriptions.add(channel);
    this.addEventHandler(channel, callback);

    if (this.ws?.readyState === WebSocket.OPEN) {
      this.sendSubscription(channel);
    }
  }

  subscribeOrderBook(
    symbol: string,
    callback: (orderBook: OrderBookSnapshot) => void
  ): void {
    const channel = `orderbook.${symbol}`;
    this.subscriptions.add(channel);
    this.addEventHandler(channel, callback);

    if (this.ws?.readyState === WebSocket.OPEN) {
      this.sendSubscription(channel);
    }
  }

  subscribeTrades(symbol: string, callback: (trade: Trade) => void): void {
    const channel = `trades.${symbol}`;
    this.subscriptions.add(channel);
    this.addEventHandler(channel, callback);

    if (this.ws?.readyState === WebSocket.OPEN) {
      this.sendSubscription(channel);
    }
  }

  unsubscribe(channel: string): void {
    this.subscriptions.delete(channel);
    this.eventHandlers.delete(channel);

    if (this.ws?.readyState === WebSocket.OPEN) {
      this.ws.send(JSON.stringify({
        type: 'unsubscribe',
        channel
      }));
    }
  }

  async getTicker(symbol: string): Promise {
    const response = await fetch(
      `${this.config.restEndpoint}/ticker/${symbol}`,
      {
        headers: this.getAuthHeaders()
      }
    );

    if (!response.ok) {
      throw new Error(`Failed to fetch ticker: ${response.statusText}`);
    }

    return await response.json();
  }

  async getOrderBook(
    symbol: string,
    depth: number = 20
  ): Promise {
    const response = await fetch(
      `${this.config.restEndpoint}/orderbook/${symbol}?depth=${depth}`,
      {
        headers: this.getAuthHeaders()
      }
    );

    if (!response.ok) {
      throw new Error(`Failed to fetch order book: ${response.statusText}`);
    }

    return await response.json();
  }

  async getRecentTrades(
    symbol: string,
    limit: number = 100
  ): Promise {
    const response = await fetch(
      `${this.config.restEndpoint}/trades/${symbol}?limit=${limit}`,
      {
        headers: this.getAuthHeaders()
      }
    );

    if (!response.ok) {
      throw new Error(`Failed to fetch trades: ${response.statusText}`);
    }

    return await response.json();
  }

  async getCandles(
    symbol: string,
    interval: '1m' | '5m' | '15m' | '1h' | '4h' | '1d',
    startTime?: number,
    endTime?: number,
    limit: number = 100
  ): Promise {
    const params = new URLSearchParams({
      interval,
      limit: limit.toString()
    });

    if (startTime) params.append('startTime', startTime.toString());
    if (endTime) params.append('endTime', endTime.toString());

    const response = await fetch(
      `${this.config.restEndpoint}/candles/${symbol}?${params}`,
      {
        headers: this.getAuthHeaders()
      }
    );

    if (!response.ok) {
      throw new Error(`Failed to fetch candles: ${response.statusText}`);
    }

    return await response.json();
  }

  private handleMessage(message: any): void {
    const { channel, data } = message;

    if (this.eventHandlers.has(channel)) {
      const handlers = this.eventHandlers.get(channel)!;
      handlers.forEach(handler => handler(data));
    }
  }

  private sendSubscription(channel: string): void {
    if (this.ws?.readyState === WebSocket.OPEN) {
      this.ws.send(JSON.stringify({
        type: 'subscribe',
        channel,
        auth: this.generateAuthToken()
      }));
    }
  }

  private resubscribeAll(): void {
    for (const channel of this.subscriptions) {
      this.sendSubscription(channel);
    }
  }

  private addEventHandler(channel: string, handler: Function): void {
    if (!this.eventHandlers.has(channel)) {
      this.eventHandlers.set(channel, []);
    }
    this.eventHandlers.get(channel)!.push(handler);
  }

  private handleReconnect(): void {
    if (this.reconnectAttempts < this.maxReconnectAttempts) {
      this.reconnectAttempts++;
      const delay = Math.min(1000 * Math.pow(2, this.reconnectAttempts), 30000);

      console.log(`Reconnecting in ${delay}ms (attempt ${this.reconnectAttempts})`);

      setTimeout(() => {
        this.connect().catch(console.error);
      }, delay);
    } else {
      console.error('Max reconnection attempts reached');
    }
  }

  private getAuthHeaders(): HeadersInit {
    const timestamp = Date.now().toString();
    const signature = this.generateSignature(timestamp);

    return {
      'X-API-KEY': this.config.apiKey,
      'X-TIMESTAMP': timestamp,
      'X-SIGNATURE': signature
    };
  }

  private generateSignature(timestamp: string): string {
    // HMAC-SHA256 signature
    const crypto = require('crypto');
    const message = timestamp + this.config.apiKey;
    return crypto
      .createHmac('sha256', this.config.apiSecret)
      .update(message)
      .digest('hex');
  }

  private generateAuthToken(): string {
    return this.generateSignature(Date.now().toString());
  }

  disconnect(): void {
    if (this.ws) {
      this.ws.close();
      this.ws = null;
    }
    this.subscriptions.clear();
    this.eventHandlers.clear();
  }
}

class TradingAPIClient {
  private config: TradingAPIConfig;
  private rateLimiter: RateLimiter;

  constructor(config: TradingAPIConfig) {
    this.config = config;
    this.rateLimiter = new RateLimiter({
      maxRequests: 100,
      windowMs: 60000  // 100 requests per minute
    });
  }

  async placeOrder(params: OrderParams): Promise {
    await this.rateLimiter.checkLimit();

    const endpoint = '/api/v3/order';
    const body = {
      ...params,
      timestamp: Date.now()
    };

    const signature = this.signRequest(body);
    body['signature'] = signature;

    const response = await fetch(`${this.config.baseURL}${endpoint}`, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        'X-API-KEY': this.config.apiKey
      },
      body: JSON.stringify(body)
    });

    if (!response.ok) {
      const error = await response.json();
      throw new TradingAPIError(
        error.code,
        error.msg,
        response.status
      );
    }

    return await response.json();
  }

  async cancelOrder(
    symbol: string,
    orderId: string
  ): Promise {
    await this.rateLimiter.checkLimit();

    const endpoint = '/api/v3/order';
    const params = {
      symbol,
      orderId,
      timestamp: Date.now()
    };

    const signature = this.signRequest(params);
    const queryString = new URLSearchParams({
      ...params,
      signature
    } as any).toString();

    const response = await fetch(
      `${this.config.baseURL}${endpoint}?${queryString}`,
      {
        method: 'DELETE',
        headers: {
          'X-API-KEY': this.config.apiKey
        }
      }
    );

    if (!response.ok) {
      const error = await response.json();
      throw new TradingAPIError(
        error.code,
        error.msg,
        response.status
      );
    }

    return await response.json();
  }

  private signRequest(params: any): string {
    const crypto = require('crypto');
    const queryString = Object.keys(params)
      .sort()
      .map(key => `${key}=${params[key]}`)
      .join('&');

    return crypto
      .createHmac('sha256', this.config.apiSecret)
      .update(queryString)
      .digest('hex');
  }
}

class RateLimiter {
  private requests: number[] = [];
  private maxRequests: number;
  private windowMs: number;

  constructor(config: { maxRequests: number; windowMs: number }) {
    this.maxRequests = config.maxRequests;
    this.windowMs = config.windowMs;
  }

  async checkLimit(): Promise {
    const now = Date.now();
    this.requests = this.requests.filter(
      time => now - time < this.windowMs
    );

    if (this.requests.length >= this.maxRequests) {
      const oldestRequest = this.requests[0];
      const waitTime = this.windowMs - (now - oldestRequest);
      await new Promise(resolve => setTimeout(resolve, waitTime));
      return this.checkLimit();
    }

    this.requests.push(now);
  }
}

4. FIX Protocol

Financial Information eXchange (FIX) Protocol is the industry-standard messaging protocol for electronic trading. FIX provides rich message types for order management, executions, and market data.

Message Type FIX MsgType Purpose Key Fields
New Order Single D Submit new order ClOrdID, Symbol, Side, OrderQty, OrdType
Execution Report 8 Order status update OrderID, ExecType, OrdStatus, LeavesQty
Order Cancel Request F Cancel existing order OrigClOrdID, ClOrdID, Symbol, Side
Market Data Request V Subscribe to market data MDReqID, SubscriptionRequestType
Market Data Snapshot W Market data update Symbol, MDEntryType, MDEntryPx, MDEntrySize

5. Rate Limiting and Best Practices

API Integration Best Practices

  • Respect Rate Limits: Track request counts and implement exponential backoff
  • Use WebSockets for Streaming: Avoid polling for real-time data
  • Implement Retry Logic: Handle transient failures with exponential backoff
  • Validate Responses: Never assume API responses are well-formed
  • Log All Requests: Maintain audit trail for debugging and compliance
  • Use Testnet: Thoroughly test on testnet before deploying to production
  • Monitor API Health: Track latency, error rates, and availability

Summary

Key Takeaways

  • Protocol Selection: Use REST for stateless operations, WebSocket for streaming, FIX for institutional integration
  • Error Handling: Robust error handling and retry logic are essential for production reliability
  • Authentication: HMAC signatures and timestamp validation prevent replay attacks
  • Rate Limiting: Respect exchange limits to avoid bans; implement client-side rate limiting
  • Monitoring: Track API performance metrics to detect issues early

Reference Standards and Implementation Notes

This standard is designed to interoperate with established international frameworks. Implementations SHOULD treat the following as the single source of truth and use this PHASE document as the interoperability layer that maps existing data into a machine-readable, signed, and auditable shape.

  • OpenAPI Specification 3.1 — REST API definition
  • JSON Schema 2020-12 — schema validation
  • ISO/IEC 27001:2022 — information security management
  • ISO/IEC 27018:2019 — PII protection in public clouds
  • ISO/IEC 17065:2012 — product certification bodies
  • CycloneDX 1.5 / SPDX 2.3 — software bill of materials
  • Sigstore (DSSE envelope, Rekor transparency log) — signature anchoring
  • in-toto Attestation Framework 1.0 — build provenance
  • IETF RFC 9700 — OAuth 2.1
  • IETF RFC 9457 — Problem Details for HTTP APIs
  • IETF RFC 8615 — well-known URIs

Practical Implementation Checklist

  • Manifest integrity — every published manifest is signed via Sigstore DSSE. Tampering is detectable without trusting an intermediary. Updates are issued as new manifests; the prior manifest is preserved with a supersede marker.
  • Retention — keep signed evidence for at least seven years to satisfy ISO/IEC 17065:2012 audit expectations. Some regulators require longer retention; in that case the deployment policy MUST extend the window rather than relying on the defaults.
  • Versioning — follow Semantic Versioning 2.0.0. Major version bumps require at least a 90-day overlap with the prior major version on every WIA-published reference implementation.
  • Negative-result reporting — a skipped indicator without recorded justification is treated as a failed indicator at audit time.
  • Key rotation — manifest signing keys are rotated per the deployment policy. The previous key is preserved in a published key history so that auditors can verify historical manifests offline.

Reference Topologies

Implementations declare their operational topology in the manifest: single-tenant edge (one runtime per organization), multi-tenant gateway (shared runtime with header-based isolation), federated mesh (multiple runtimes peer to one another and publish manifests to a directory service), or air-gapped batch (signed evidence package transported out-of-band). Each topology ships with reference test vectors so that implementations can validate their deployment matches the canonical pattern.

Open Governance

Issues, errata, and proposals are tracked at github.com/WIA-Official/wia-standards/issues with the appropriate label. The WIA Standards working group reviews open issues at the start of every minor release cycle and publishes the resulting decision log alongside the release notes. Errata are issued as patch releases; new normative requirements trigger minor bumps; backwards-incompatible changes trigger major bumps.
