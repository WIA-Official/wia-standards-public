Chapter 3: API 인터페이스

3.1 2단계 소개: API 인터페이스

1단계에서 확립된 표준화된 데이터 형식을 기반으로, WIA-LEGAL-001 표준의 2단계는 디지털 법원 시스템에 대한 프로그래밍 방식 접근을 가능하게 하는 포괄적인 응용 프로그래밍 인터페이스(API)를 도입합니다. 이러한 API는 정적 데이터 스키마를 전자 접수 시스템, 사건 관리 플랫폼, 가상 심리 애플리케이션 및 타사 법률 기술 통합을 지원하는 동적이고 상호작용적인 서비스로 변환합니다.

API 계층은 데이터와 기능 사이의 중요한 다리 역할을 하며, 개발자, 법률 실무자 및 법원 관리자가 잘 정의되고 안전하며 표준화된 인터페이스를 통해 디지털 법원 시스템과 상호작용할 수 있도록 합니다. RESTful 및 GraphQL API 옵션을 모두 제공함으로써 WIA-LEGAL-001은 단순한 문서 검색부터 증거 제출, 심리 조정, 관할권 간 사건 관리를 포함하는 복잡한 다중 시스템 워크플로까지 다양한 통합 시나리오를 수용합니다.

3.2 API 아키텍처 및 설계 원칙

3.2.1 핵심 설계 원칙

API 설계는 일관성, 사용성 및 장기적 유지 관리성을 보장하는 기본 원칙을 준수합니다:

  • RESTful 리소스 지향 설계: 각 엔티티(사건, 문서, 심리, 증거)는 고유한 URI를 가진 리소스로 표현되며, 표준 HTTP 메서드(GET, POST, PUT, DELETE)를 통해 조작됩니다.
  • 무상태 통신: 각 API 요청에는 처리에 필요한 모든 정보가 포함되어 서버 측 세션 종속성을 제거하고 수평적 확장을 가능하게 합니다.
  • 버전화된 엔드포인트: 모든 API 엔드포인트는 버전 식별자를 포함하여(예: /v1/cases) 시스템 진화 중 하위 호환성을 지원합니다.
  • 일관된 오류 처리: 의미 있는 HTTP 상태 코드, 오류 코드 및 설명 메시지를 가진 표준화된 오류 응답 형식을 제공합니다.
  • 기본 보안: 모든 엔드포인트는 인증 및 권한 부여가 필요하며, 공개 접근은 명시적으로 구성됩니다.
  • 속도 제한 및 조절: 남용으로부터 보호하고 사용자 간 공정한 리소스 할당을 보장합니다.

API 서비스는 개발, 테스트 및 프로덕션 워크플로를 지원하기 위해 여러 환경에서 작동합니다:

3.3 핵심 API 엔드포인트

3.3.1 인증 엔드포인트

인증은 API 보안의 기반을 형성하며, 법원 시스템 리소스에 대한 접근을 제어하고 적절한 자격 증명을 가진 권한 있는 당사자가 작업을 수행하도록 보장합니다:

POST   /auth/login       # 사용자 인증 및 액세스 토큰 획득
POST   /auth/refresh     # 만료된 액세스 토큰 갱신
POST   /auth/logout      # 인증 세션 무효화
GET    /auth/verify      # 토큰 유효성 검증
POST   /auth/mfa/enable  # 다중 인증 활성화
POST   /auth/mfa/verify  # 로그인 중 MFA 코드 검증

인증 시스템은 무상태 인증을 위해 JWT(JSON Web Tokens)와 함께 OAuth 2.0을 구현합니다. 액세스 토큰은 보안 노출을 최소화하기 위해 제한된 수명(일반적으로 1시간)을 가지며, 리프레시 토큰은 재인증 없이 연장된 기간 동안 새로운 액세스 토큰을 획득할 수 있게 합니다.

3.3.2 사건 관리 엔드포인트

사건 관리 엔드포인트는 초기 접수부터 처분 및 항소까지 전체 사건 수명 주기를 지원하는 법원 사건 데이터에 대한 포괄적인 접근을 제공합니다:

GET    /cases                    # 사건 목록 조회 (필터링/페이지네이션)
POST   /cases                    # 새 사건 접수
GET    /cases/{caseId}           # 사건 세부 정보 검색
PUT    /cases/{caseId}           # 사건 정보 업데이트
DELETE /cases/{caseId}           # 사건 보관 (소프트 삭제)
GET    /cases/{caseId}/timeline  # 시간순 사건 이벤트 조회
GET    /cases/{caseId}/parties   # 모든 당사자 목록 조회
POST   /cases/{caseId}/parties   # 사건에 당사자 추가
PUT    /cases/{caseId}/status    # 사건 상태 업데이트
GET    /cases/{caseId}/statistics # 사건 메트릭 조회

3.3.3 문서 접수 엔드포인트

문서 엔드포인트는 암호화 검증과 함께 제출, 유효성 검사, 서비스 및 법원 문서 검색을 포함한 전자 접수 워크플로를 처리합니다:

GET    /documents                    # 문서 목록 조회
POST   /documents                    # 새 문서 접수
GET    /documents/{docId}            # 문서 검색
PUT    /documents/{docId}            # 문서 메타데이터 업데이트
DELETE /documents/{docId}            # 문서 제거 (허용된 경우)
GET    /documents/{docId}/versions   # 문서 개정 이력 조회
POST   /documents/{docId}/sign       # 디지털 서명 적용
GET    /documents/{docId}/download   # 문서 파일 다운로드
POST   /documents/{docId}/serve      # 문서 전자 송달

3.3.4 심리 관리 엔드포인트

심리 엔드포인트는 일정 조정, 참가자 조정 및 녹음 및 녹취록 접근을 포함하여 가상 및 대면 법원 절차를 모두 관리합니다:

GET    /hearings                     # 심리 목록 조회
POST   /hearings                     # 새 심리 일정 조정
GET    /hearings/{hearingId}         # 심리 세부 정보 조회
PUT    /hearings/{hearingId}         # 심리 업데이트
DELETE /hearings/{hearingId}         # 심리 취소
POST   /hearings/{hearingId}/join    # 가상 심리 참여
GET    /hearings/{hearingId}/recording  # 심리 녹음 접근
GET    /hearings/{hearingId}/transcript # 녹취록 접근
POST   /hearings/{hearingId}/notify     # 심리 알림 전송

3.5 오류 처리 및 HTTP 상태 코드

포괄적인 오류 처리는 요청이 실패할 때 개발자가 명확하고 실행 가능한 피드백을 받을 수 있도록 보장합니다. API는 자세한 오류 객체와 결합된 표준 HTTP 상태 코드를 사용합니다.

표 3.2: HTTP 상태 코드 및 의미
코드 상태 의미 일반적인 원인
200 OK 요청 성공 성공적인 GET, PUT 작업
201 Created 리소스가 성공적으로 생성됨 성공적인 POST 작업
204 No Content 성공적인 삭제 성공적인 DELETE 작업
400 Bad Request 잘못된 요청 형식 잘못된 JSON, 필수 필드 누락
401 Unauthorized 인증 필요 액세스 토큰 누락 또는 무효
403 Forbidden 권한 부족 사용자에게 필요한 역할 또는 범위 부족
404 Not Found 리소스가 존재하지 않음 잘못된 사건 ID, 문서 ID 등
409 Conflict 리소스 상태 충돌 중복 사건 번호, 동시 수정
422 Unprocessable Entity 유효성 검사 실패 잘못된 데이터 값, 비즈니스 규칙 위반
429 Too Many Requests 속도 제한 초과 시간 창에서 너무 많은 API 호출
500 Internal Server Error 서버 측 오류 데이터베이스 실패, 예기치 않은 예외
503 Service Unavailable 일시적 사용 불가 유지 관리, 시스템 과부하

3.6 인증 및 권한 부여

3.6.1 OAuth 2.0 인증 흐름

API는 웹 애플리케이션의 경우 권한 부여 코드 흐름과 서버 간 통합의 경우 클라이언트 자격 증명 흐름을 사용하는 OAuth 2.0을 구현합니다.

로그인 요청:

POST /v1/auth/login
Content-Type: application/json

{
  "username": "attorney@lawfirm.com",
  "password": "SecurePassword123!",
  "mfa_code": "123456"
}

로그인 응답:

{
  "access_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refresh_token": "refresh_abc123def456...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "scope": "case:read case:write document:file hearing:schedule"
}

3.6.2 권한 부여 범위

세분화된 권한 부여는 OAuth 범위를 통해 특정 API 작업에 대한 접근을 제어합니다:

표 3.3: API 권한 부여 범위
범위 설명 일반적인 역할
case:read 사건 정보 조회 모든 인증된 사용자
case:write 사건 생성 및 업데이트 변호사, 법원 서기
document:file 법원 문서 접수 변호사, 본인 소송 당사자
document:read 접수된 문서 접근 사건 참가자, 공개 (봉인되지 않음)
hearing:schedule 심리 일정 조정 판사, 법원 서기
hearing:join 가상 심리 참여 사건 참가자
evidence:submit 증거 제출 변호사, 당사자
order:issue 법원 명령 발행 판사만
admin:manage 관리 기능 시스템 관리자

3.7 弘益人間 철학과 API 설계

弘益人間(홍익인간) 철학의 API 구현

WIA-LEGAL-001 API는 다음을 통해 "널리 인간을 이롭게 하라"는 원칙을 구현합니다:

  • 개방형 표준: 모든 관할권이 구현할 수 있도록 완전한 API 문서를 공개적으로 게시
  • 공정한 접근: 남용을 방지하면서 합법적인 대량 사용을 허용하도록 설계된 속도 제한
  • 개발자 경험: 명확한 문서, 일관된 패턴 및 유용한 오류 메시지로 채택 장벽 감소
  • 접근성: API는 장애인이나 제한된 기술 접근 권한을 가진 사용자를 위한 특수 인터페이스 생성 가능
  • 투명성: 감사 로그 및 공개 엔드포인트로 법원 시스템 책임성 지원

챕터 요약

핵심 요점:

  1. 2단계는 WIA-LEGAL-001 데이터 형식을 전자 접수, 사건 관리 및 가상 심리를 가능하게 하는 상호작용 서비스로 변환하는 RESTful 및 GraphQL API를 도입합니다.
  2. API 아키텍처는 무상태 통신, 버전화된 엔드포인트, 일관된 오류 처리 및 기본 보안 설계 원칙을 포함한 현대적 모범 사례를 따릅니다.
  3. 핵심 엔드포인트 카테고리(인증, 사건, 문서, 심리, 증거)는 문서 송달 및 증거 보관 연쇄 추적과 같은 특수 법률 워크플로와 함께 포괄적인 CRUD 작업을 제공합니다.
  4. JWT 토큰을 사용하는 OAuth 2.0은 안전한 인증을 제공하며, 세분화된 권한 부여 범위는 사용자 역할 및 권한에 따라 특정 작업에 대한 접근을 제어합니다.
  5. 투명한 헤더가 있는 속도 제한은 남용을 방지하면서 합법적인 대량 사용을 허용하며, 진단 세부 정보가 포함된 포괄적인 오류 응답은 신속한 문제 해결 및 개발을 촉진합니다.
  6. API 설계는 전 세계 디지털 법원 시스템 채택의 장벽을 낮추는 개방적이고 접근 가능하며 잘 문서화된 인터페이스를 제공함으로써 弘益人間을 구현합니다.

복습 질문

  1. OAuth 2.0 인증 흐름에서 액세스 토큰과 리프레시 토큰의 차이점을 설명하십시오. API가 왜 둘 다 사용합니까?
  2. 사용자가 접근 권한이 없는 사건에 대한 문서를 접수하려고 할 때 반환되어야 하는 HTTP 상태 코드는 무엇입니까? 401, 403, 404의 차이점을 설명하십시오.
  3. API 속도 제한의 목적을 설명하십시오. 읽기 작업과 쓰기 작업에 왜 다른 속도 제한이 적용됩니까?
  4. API가 접수 프로세스 중 문서 무결성을 어떻게 보장합니까? SHA-256 해시와 디지털 서명이 어떤 역할을 합니까?
  5. RESTful과 GraphQL API 아키텍처를 비교하십시오. 법원 시스템 통합에서 GraphQL을 선호할 수 있는 시나리오는 무엇입니까?
  6. 오류 응답 형식에 포함되는 정보는 무엇입니까? 이것이 개발자와 최종 사용자를 모두 어떻게 지원합니까?
  7. 권한 부여 범위가 최소 권한 원칙을 어떻게 구현합니까? 제한된 범위를 가진 역할의 예를 제시하십시오.
