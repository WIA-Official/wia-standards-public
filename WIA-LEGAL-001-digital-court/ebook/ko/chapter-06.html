Chapter 6: 보안 및 인증

6.1 보안 아키텍처 개요

디지털 법원 시스템의 보안은 다층 방어(Defense in Depth) 전략을 채택합니다. 이는 네트워크 계층부터 애플리케이션, 데이터, 사용자 인증에 이르기까지 여러 보안 계층을 구현하여 단일 보안 메커니즘의 실패가 전체 시스템을 침해하지 않도록 보장합니다. 법률 시스템의 특성상 보안은 단순한 기술적 요구 사항이 아니라 정의의 무결성과 공정성을 유지하는 데 필수적인 요소입니다.

WIA-LEGAL-001 보안 프레임워크는 CIA 트라이어드(기밀성, 무결성, 가용성)를 핵심 원칙으로 하며, 법적 맥락에서 특히 중요한 비부인성(Non-repudiation)과 감사 가능성(Auditability)을 추가합니다. 모든 보안 메커니즘은 NIST, ISO 27001, GDPR, SOC 2와 같은 국제 표준 및 규정을 준수하도록 설계되었습니다.

6.2 암호화 요구 사항

6.2.1 저장 데이터 암호화 (Data at Rest)

모든 저장 데이터는 AES-256-GCM(Galois/Counter Mode)을 사용하여 암호화됩니다. GCM 모드는 암호화와 인증을 동시에 제공하여 데이터 무결성과 기밀성을 모두 보장합니다. 암호화 키는 Hardware Security Module(HSM)에 저장되며, FIPS 140-2 Level 3 이상 인증을 받은 HSM을 사용해야 합니다. 

// 저장 데이터 암호화 구현 예제
import crypto from 'crypto';

class DataEncryption {
  private hsmClient: HSMClient;

  constructor(hsmConfig: HSMConfig) {
    this.hsmClient = new HSMClient(hsmConfig);
  }

  async encryptData(plaintext: Buffer, metadata: any): Promise {
    // HSM에서 데이터 암호화 키(DEK) 생성
    const dek = await this.hsmClient.generateDataKey({
      algorithm: 'AES-256-GCM',
      keyId: 'master-key-id'
    });

    // 초기화 벡터(IV) 생성 (96 bits for GCM)
    const iv = crypto.randomBytes(12);

    // 인증 태그를 위한 추가 인증 데이터(AAD)
    const aad = Buffer.from(JSON.stringify({
      version: '1.0',
      timestamp: new Date().toISOString(),
      metadata: metadata
    }));

    // AES-256-GCM으로 암호화
    const cipher = crypto.createCipheriv('aes-256-gcm', dek.plaintext, iv);
    cipher.setAAD(aad);

    const ciphertext = Buffer.concat([
      cipher.update(plaintext),
      cipher.final()
    ]);

    const authTag = cipher.getAuthTag();

    return {
      ciphertext: ciphertext.toString('base64'),
      encryptedKey: dek.ciphertext, // HSM으로 암호화된 DEK
      iv: iv.toString('base64'),
      authTag: authTag.toString('base64'),
      aad: aad.toString('base64'),
      algorithm: 'AES-256-GCM',
      keyRotationDate: new Date(Date.now() + 90 * 24 * 60 * 60 * 1000) // 90일
    };
  }

  async decryptData(encrypted: EncryptedData): Promise {
    // HSM에서 DEK 복호화
    const dek = await this.hsmClient.decrypt({
      ciphertext: encrypted.encryptedKey,
      keyId: 'master-key-id'
    });

    const decipher = crypto.createDecipheriv(
      'aes-256-gcm',
      dek.plaintext,
      Buffer.from(encrypted.iv, 'base64')
    );

    decipher.setAAD(Buffer.from(encrypted.aad, 'base64'));
    decipher.setAuthTag(Buffer.from(encrypted.authTag, 'base64'));

    const plaintext = Buffer.concat([
      decipher.update(Buffer.from(encrypted.ciphertext, 'base64')),
      decipher.final()
    ]);

    return plaintext;
  }

  async rotateKeys(): Promise {
    // 90일마다 자동 키 순환
    const oldKeyId = 'master-key-id';
    const newKeyId = await this.hsmClient.createKey({
      algorithm: 'AES-256',
      usage: 'encrypt-decrypt'
    });

    // 모든 DEK를 새 마스터 키로 재암호화
    await this.reEncryptAllDEKs(oldKeyId, newKeyId);

    // 이전 키 비활성화 (90일 후 삭제)
    await this.hsmClient.scheduleKeyDeletion(oldKeyId, 90);
  }
}

6.2.2 전송 중 데이터 보안 (Data in Transit)

모든 네트워크 통신은 TLS 1.3 이상을 사용해야 하며, Perfect Forward Secrecy(PFS)를 지원하는 암호 스위트만 허용됩니다. 이는 세션 키가 손상되더라도 과거 통신이 복호화될 수 없도록 보장합니다. 

// TLS 1.3 서버 구성 (Node.js)
import https from 'https';
import fs from 'fs';

const tlsOptions = {
  // TLS 1.3 최소 버전 강제
  minVersion: 'TLSv1.3',
  maxVersion: 'TLSv1.3',

  // 인증서 및 키
  cert: fs.readFileSync('/path/to/cert.pem'),
  key: fs.readFileSync('/path/to/key.pem'),

  // PFS를 지원하는 암호 스위트만 허용
  ciphers: [
    'TLS_AES_256_GCM_SHA384',
    'TLS_CHACHA20_POLY1305_SHA256',
    'TLS_AES_128_GCM_SHA256'
  ].join(':'),

  // 클라이언트 인증서 요구 (상호 TLS)
  requestCert: true,
  rejectUnauthorized: true,
  ca: fs.readFileSync('/path/to/ca.pem'),

  // HSTS (HTTP Strict Transport Security)
  honorCipherOrder: true,

  // 세션 재사용 비활성화 (최대 보안)
  sessionTimeout: 300,
  sessionIdContext: 'digitalcourt'
};

const server = https.createServer(tlsOptions, (req, res) => {
  // HSTS 헤더 설정 (2년, 서브도메인 포함, preload)
  res.setHeader(
    'Strict-Transport-Security',
    'max-age=63072000; includeSubDomains; preload'
  );

  // 추가 보안 헤더
  res.setHeader('X-Frame-Options', 'DENY');
  res.setHeader('X-Content-Type-Options', 'nosniff');
  res.setHeader('X-XSS-Protection', '1; mode=block');
  res.setHeader(
    'Content-Security-Policy',
    "default-src 'self'; script-src 'self'; style-src 'self'"
  );

  // 요청 처리
  handleRequest(req, res);
});

server.listen(443);

6.3.1 인증 레벨

6.3.2 FIDO2/WebAuthn 구현

// WebAuthn 등록 및 인증
import { generateRegistrationOptions, verifyRegistrationResponse } from '@simplewebauthn/server';

class WebAuthnService {
  async registerDevice(userId: string, userName: string) {
    // 등록 옵션 생성
    const options = generateRegistrationOptions({
      rpName: 'WIA Digital Court',
      rpID: 'digitalcourt.wia.org',
      userID: userId,
      userName: userName,

      // 인증자 요구 사항
      authenticatorSelection: {
        authenticatorAttachment: 'cross-platform', // USB 키 등
        requireResidentKey: true,
        userVerification: 'required' // 생체 인증 또는 PIN
      },

      // 지원 알고리즘
      supportedAlgorithmIDs: [-7, -257], // ES256, RS256

      // 타임아웃
      timeout: 60000,

      // 증명 유형
      attestationType: 'direct'
    });

    // 챌린지를 세션에 저장
    await this.saveChallenge(userId, options.challenge);

    return options;
  }

  async verifyRegistration(userId: string, response: any) {
    const expectedChallenge = await this.getChallenge(userId);

    const verification = await verifyRegistrationResponse({
      response: response,
      expectedChallenge: expectedChallenge,
      expectedOrigin: 'https://digitalcourt.wia.org',
      expectedRPID: 'digitalcourt.wia.org'
    });

    if (verification.verified) {
      // 인증자 저장
      await this.saveAuthenticator(userId, {
        credentialID: verification.registrationInfo.credentialID,
        credentialPublicKey: verification.registrationInfo.credentialPublicKey,
        counter: verification.registrationInfo.counter,
        credentialDeviceType: verification.registrationInfo.credentialDeviceType,
        credentialBackedUp: verification.registrationInfo.credentialBackedUp,
        registeredAt: new Date()
      });

      return { success: true };
    }

    return { success: false, error: 'Verification failed' };
  }

  async authenticate(userId: string) {
    const authenticators = await this.getAuthenticators(userId);

    const options = generateAuthenticationOptions({
      rpID: 'digitalcourt.wia.org',
      allowCredentials: authenticators.map(auth => ({
        id: auth.credentialID,
        type: 'public-key',
        transports: ['usb', 'nfc', 'ble']
      })),
      userVerification: 'required',
      timeout: 60000
    });

    await this.saveChallenge(userId, options.challenge);

    return options;
  }
}

// TypeScript RBAC 검증 미들웨어
interface Permission {
  resource: string;
  action: string;
}

class RBACService {
  private rolePermissions: Map>;

  constructor(config: RBACConfig) {
    this.rolePermissions = this.loadPermissions(config);
  }

  hasPermission(userRoles: string[], permission: Permission): boolean {
    const requiredPermission = `${permission.resource}:${permission.action}`;

    for (const role of userRoles) {
      const permissions = this.rolePermissions.get(role);
      if (permissions?.has(requiredPermission)) {
        return true;
      }
    }

    return false;
  }

  async checkAccess(
    userId: string,
    resource: string,
    action: string
  ): Promise {
    // 사용자 역할 조회
    const user = await this.getUserWithRoles(userId);

    // 권한 검증
    const hasPermission = this.hasPermission(user.roles, { resource, action });

    if (!hasPermission) {
      // 감사 로그 기록
      await this.auditLog.record({
        event: 'access_denied',
        userId: userId,
        resource: resource,
        action: action,
        timestamp: new Date()
      });
    }

    return hasPermission;
  }

  // Express 미들웨어
  requirePermission(resource: string, action: string) {
    return async (req, res, next) => {
      const userId = req.user.id;
      const hasAccess = await this.checkAccess(userId, resource, action);

      if (!hasAccess) {
        return res.status(403).json({
          error: 'Forbidden',
          message: '이 작업을 수행할 권한이 없습니다.'
        });
      }

      next();
    };
  }
}

  }

6.5 감사 로깅

6.5.1 포괄적인 감사 추적

모든 시스템 활동은 불변 감사 로그에 기록됩니다. 로그는 변조를 방지하기 위해 블록체인에 주기적으로 앵커링되며, 최소 7년간 보존됩니다. 감사 로그는 규정 준수, 포렌식 조사, 분쟁 해결에 필수적입니다. 

// 감사 로그 구조
interface AuditLog {
  // 고유 식별자
  logId: string;

  // 이벤트 정보
  eventType: string; // 'document.accessed', 'case.modified', etc.
  eventCategory: 'access' | 'modification' | 'deletion' | 'authentication' | 'authorization';

  // 행위자 정보
  actor: {
    did: string;
    userId: string;
    role: string;
    ipAddress: string;
    userAgent: string;
    location?: {
      country: string;
      city: string;
    };
  };

  // 리소스 정보
  resource: {
    type: 'case' | 'document' | 'hearing' | 'evidence' | 'order';
    id: string;
    caseId?: string;
    metadata?: any;
  };

  // 작업 정보
  action: string; // 'read', 'write', 'delete', 'download', etc.
  result: 'success' | 'failure' | 'denied';
  resultDetails?: string;

  // 타임스탬프
  timestamp: string; // ISO 8601

  // 세션 정보
  sessionId: string;

  // 변경 사항 (수정 작업인 경우)
  changes?: {
    before: any;
    after: any;
    fields: string[];
  };

  // 블록체인 앵커
  blockchainAnchor?: {
    txHash: string;
    blockNumber: number;
    chainId: number;
  };

  // 보존 정책
  retentionPolicy: {
    minimumYears: number;
    deleteAfter?: string;
  };
}

// 감사 로깅 서비스
class AuditLogService {
  private logStore: LogStore;
  private blockchain: BlockchainService;

  async record(log: Partial): Promise {
    // 로그 ID 생성
    const logId = generateUUID();

    // 완전한 로그 객체 생성
    const fullLog: AuditLog = {
      logId,
      ...log,
      timestamp: new Date().toISOString(),
      retentionPolicy: {
        minimumYears: 7
      }
    } as AuditLog;

    // 불변 저장소에 기록
    await this.logStore.append(fullLog);

    // 중요 이벤트는 즉시 블록체인에 앵커링
    if (this.isCriticalEvent(log.eventType)) {
      const anchor = await this.blockchain.anchor({
        logId: logId,
        hash: this.hashLog(fullLog),
        timestamp: fullLog.timestamp
      });

      fullLog.blockchainAnchor = anchor;
      await this.logStore.update(logId, { blockchainAnchor: anchor });
    }

    // 의심스러운 활동 감지
    await this.detectAnomalies(fullLog);
  }

  async search(query: AuditLogQuery): Promise {
    // 검색 권한 확인
    if (!await this.hasAuditPermission(query.requesterId)) {
      throw new Error('감사 로그 조회 권한이 없습니다.');
    }

    // 검색 수행
    const results = await this.logStore.search(query);

    // 검색 자체도 감사 로그에 기록
    await this.record({
      eventType: 'audit_log.searched',
      eventCategory: 'access',
      actor: { did: query.requesterId },
      resource: { type: 'audit_log', id: 'search' },
      action: 'search',
      result: 'success',
      metadata: { query }
    });

    return results;
  }

  private isCriticalEvent(eventType: string): boolean {
    const criticalEvents = [
      'order.issued',
      'verdict.signed',
      'evidence.submitted',
      'case.sealed',
      'access.denied.security_violation'
    ];

    return criticalEvents.includes(eventType);
  }

  private async detectAnomalies(log: AuditLog): Promise {
    // 이상 패턴 감지 (ML 기반)
    const anomalyScore = await this.anomalyDetector.score(log);

    if (anomalyScore > 0.8) {
      // 보안팀에 알림
      await this.alertSecurityTeam({
        severity: 'high',
        log: log,
        anomalyScore: anomalyScore,
        reason: '비정상적인 접근 패턴 감지'
      });
    }
  }
}

6.7 弘益人間과 보안

弘益人間(홍익인간) - 모두를 위한 보안

WIA-LEGAL-001의 보안 프레임워크는 다음을 통해 "널리 인간을 이롭게 하라"는 원칙을 실현합니다:

  • 평등한 보호: 모든 사용자가 동일한 수준의 보안 보호를 받으며, 사회적 지위나 재력과 무관합니다.
  • 프라이버시 권리: 강력한 암호화와 접근 제어로 모든 사람의 법적 프라이버시를 보호합니다.
  • 투명한 감사: 불변 감사 로그는 권력 남용을 방지하고 정의의 투명성을 보장합니다.
  • 접근성: 다단계 인증은 보안을 제공하면서도 합법적인 사용자의 접근을 방해하지 않도록 설계되었습니다.
  • 국제 표준 준수: GDPR, CCPA 등 글로벌 프라이버시 규정을 준수하여 모든 지역의 권리를 보호합니다.

⚠️ 보안 모범 사례

  • 모든 프로덕션 시스템에서 TLS 1.3을 강제하고 이전 버전을 비활성화하십시오.
  • HSM을 사용하여 암호화 키를 보호하고 정기적으로 키를 순환하십시오.
  • 모든 중요 작업에 레벨 3 이상의 인증을 요구하십시오.
  • 감사 로그를 정기적으로 검토하고 이상 패턴을 모니터링하십시오.
  • 정기적인 보안 감사와 침투 테스트를 수행하십시오.
  • 보안 인시던트 대응 계획을 수립하고 정기적으로 훈련하십시오.

챕터 요약

핵심 요점:

  1. 다층 방어 전략은 네트워크부터 애플리케이션, 데이터, 인증에 이르기까지 여러 보안 계층을 구현하여 단일 실패점을 방지합니다.
  2. 모든 저장 데이터는 AES-256-GCM으로 암호화되며, 키는 FIPS 140-2 Level 3 이상의 HSM에 보관되고 90일마다 순환됩니다.
  3. 4단계 인증 시스템은 작업의 민감도에 따라 비밀번호, TOTP, FIDO2 하드웨어 키, 생체 인증을 요구합니다.
  4. 역할 기반 접근 제어(RBAC)는 최소 권한 원칙에 따라 각 역할에 필요한 최소한의 권한만 부여합니다.
  5. 불변 감사 로그는 모든 시스템 활동을 기록하며, 중요 이벤트는 블록체인에 앵커링되어 변조를 방지합니다.
  6. 시스템은 GDPR, CCPA, HIPAA, SOC 2, ISO 27001, NIST 800-53 등 주요 국제 규정 및 표준을 준수합니다.

복습 질문

  1. 다층 방어(Defense in Depth) 전략이 무엇이며, 디지털 법원 시스템에서 구현되는 7가지 보안 계층을 설명하십시오.
  2. AES-256-GCM의 GCM 모드가 왜 중요하며, 일반 CBC 모드와 어떻게 다릅니까?
  3. Perfect Forward Secrecy(PFS)가 무엇이며, 법원 시스템에서 왜 필수적입니까?
  4. 4단계 인증 레벨을 설명하고, 각 레벨이 적용되는 작업의 예를 제시하십시오.
  5. FIDO2/WebAuthn이 전통적인 비밀번호 기반 인증보다 안전한 이유는 무엇입니까?
  6. RBAC 시스템에서 '최소 권한 원칙'이 어떻게 구현되며, 이것이 보안에 미치는 영향은 무엇입니까?
  7. 감사 로그를 블록체인에 앵커링하는 이점은 무엇이며, 어떤 이벤트가 즉시 앵커링되어야 합니까?
  8. GDPR과 CCPA의 주요 차이점은 무엇이며, 디지털 법원 시스템은 어떻게 두 규정을 모두 준수합니까?

다음 챕터 미리보기
