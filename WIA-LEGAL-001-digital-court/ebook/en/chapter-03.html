Chapter 3: API Interface Overview (Phase 2)

3.1 Introduction to Phase 2: API Interface

Building upon the standardized data formats established in Phase 1, Phase 2 of the WIA-LEGAL-001 standard introduces comprehensive Application Programming Interfaces (APIs) that enable programmatic access to digital court systems. These APIs transform the static data schemas into dynamic, interactive services that power electronic filing systems, case management platforms, virtual hearing applications, and third-party legal technology integrations.

The API layer serves as the critical bridge between data and functionality, allowing developers, legal practitioners, and court administrators to interact with digital court systems through well-defined, secure, and standardized interfaces. By providing both RESTful and GraphQL API options, WIA-LEGAL-001 accommodates diverse integration scenarios, from simple document retrieval to complex multi-system workflows involving evidence submission, hearing coordination, and cross-jurisdictional case management.

3.2 API Architecture and Design Principles

The WIA-LEGAL-001 API architecture follows industry best practices for modern web services, emphasizing security, scalability, reliability, and developer experience. The architecture supports multiple deployment models, from single-court installations to nationwide systems handling millions of cases.

3.2.1 Core Design Principles

The API design adheres to fundamental principles that ensure consistency, usability, and long-term maintainability:

  • RESTful Resource-Oriented Design: Each entity (case, document, hearing, evidence) is represented as a resource with a unique URI, manipulated through standard HTTP methods (GET, POST, PUT, DELETE)
  • Stateless Communication: Each API request contains all necessary information for processing, eliminating server-side session dependencies and enabling horizontal scaling
  • Versioned Endpoints: All API endpoints include version identifiers (e.g., /v1/cases) to support backward compatibility during system evolution
  • Consistent Error Handling: Standardized error response formats with meaningful HTTP status codes, error codes, and descriptive messages
  • Security by Default: All endpoints require authentication and authorization; public access is explicitly configured rather than assumed
  • Rate Limiting and Throttling: Protection against abuse and ensuring fair resource allocation across users

3.2.2 Base URL Structure

The API service operates across multiple environments to support development, testing, and production workflows:

Environment Base URL Purpose Data Persistence

3.3 Core API Endpoints

The WIA-LEGAL-001 API organizes endpoints into logical categories corresponding to major court system functions. Each category provides a complete CRUD (Create, Read, Update, Delete) interface where applicable, with additional specialized operations for legal-specific workflows.

3.3.1 Authentication Endpoints

Authentication forms the foundation of API security, controlling access to court system resources and ensuring operations are performed by authorized parties with proper credentials.

POST   /auth/login       # Authenticate user and obtain access token
POST   /auth/refresh     # Refresh expired access token
POST   /auth/logout      # Invalidate authentication session
GET    /auth/verify      # Verify token validity
POST   /auth/mfa/enable  # Enable multi-factor authentication
POST   /auth/mfa/verify  # Verify MFA code during login

The authentication system implements OAuth 2.0 with JWT (JSON Web Tokens) for stateless authentication. Access tokens have limited lifespans (typically 1 hour) to minimize security exposure, while refresh tokens enable obtaining new access tokens without re-authentication for extended periods.

3.3.2 Case Management Endpoints

Case management endpoints provide comprehensive access to court case data, supporting the entire case lifecycle from initial filing through disposition and appeals.

GET    /cases                    # List cases (with filtering/pagination)
POST   /cases                    # File new case
GET    /cases/{caseId}           # Retrieve case details
PUT    /cases/{caseId}           # Update case information
DELETE /cases/{caseId}           # Archive case (soft delete)
GET    /cases/{caseId}/timeline  # Get chronological case events
GET    /cases/{caseId}/parties   # List all parties
POST   /cases/{caseId}/parties   # Add party to case
PUT    /cases/{caseId}/status    # Update case status
GET    /cases/{caseId}/statistics # Get case metrics

3.3.3 Document Filing Endpoints

Document endpoints handle the electronic filing workflow, including submission, validation, service, and retrieval of court documents with cryptographic verification.

GET    /documents                    # List documents
POST   /documents                    # File new document
GET    /documents/{docId}            # Retrieve document
PUT    /documents/{docId}            # Update document metadata
DELETE /documents/{docId}            # Remove document (if permitted)
GET    /documents/{docId}/versions   # Get document revision history
POST   /documents/{docId}/sign       # Apply digital signature
GET    /documents/{docId}/download   # Download document file
POST   /documents/{docId}/serve      # Electronically serve document

3.3.4 Hearing Management Endpoints

Hearing endpoints manage both virtual and in-person court proceedings, including scheduling, participant coordination, and access to recordings and transcripts.

GET    /hearings                     # List hearings
POST   /hearings                     # Schedule new hearing
GET    /hearings/{hearingId}         # Get hearing details
PUT    /hearings/{hearingId}         # Update hearing
DELETE /hearings/{hearingId}         # Cancel hearing
POST   /hearings/{hearingId}/join    # Join virtual hearing
GET    /hearings/{hearingId}/recording  # Access hearing recording
GET    /hearings/{hearingId}/transcript # Access transcript
POST   /hearings/{hearingId}/notify     # Send hearing notifications

3.3.5 Evidence Management Endpoints

Evidence endpoints manage digital evidence submission, chain of custody tracking, authenticity verification, and admissibility determinations.

GET    /evidence                      # List evidence items
POST   /evidence                      # Submit new evidence
GET    /evidence/{evidenceId}         # Get evidence details
PUT    /evidence/{evidenceId}         # Update evidence metadata
POST   /evidence/{evidenceId}/verify  # Verify evidence authenticity
GET    /evidence/{evidenceId}/chain   # Get chain of custody
POST   /evidence/{evidenceId}/custody # Add custody record
GET    /evidence/{evidenceId}/download # Download evidence file

3.5 Error Handling and HTTP Status Codes

Comprehensive error handling ensures developers receive clear, actionable feedback when requests fail. The API uses standard HTTP status codes combined with detailed error objects.

Table 3.2: HTTP Status Codes and Their Meanings
Code Status Meaning Common Causes
200 OK Request succeeded Successful GET, PUT operations
201 Created Resource created successfully Successful POST operations
204 No Content Successful deletion Successful DELETE operations
400 Bad Request Invalid request format Malformed JSON, missing required fields
401 Unauthorized Authentication required Missing or invalid access token
403 Forbidden Insufficient permissions User lacks required role or scope
404 Not Found Resource doesn't exist Invalid case ID, document ID, etc.
409 Conflict Resource state conflict Duplicate case number, concurrent modification
422 Unprocessable Entity Validation failed Invalid data values, business rule violation
429 Too Many Requests Rate limit exceeded Too many API calls in time window
500 Internal Server Error Server-side error Database failure, unexpected exception
503 Service Unavailable Temporary unavailability Maintenance, system overload

3.7 Rate Limiting and Quotas

Rate limiting prevents API abuse and ensures fair resource allocation. Limits vary by endpoint category and user tier.

Table 3.4: API Rate Limits by Category
Endpoint Category Rate Limit Time Window Burst Allowance
Authentication 10 requests 1 minute No burst
Case Read Operations 100 requests 1 minute +20 burst
Case Write Operations 20 requests 1 minute +5 burst
Document Filing 50 requests 1 hour +10 burst
Hearing Operations 30 requests 1 minute +10 burst
Evidence Operations 25 requests 1 hour +5 burst

Rate limit information is included in response headers:

HTTP/1.1 200 OK
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 95
X-RateLimit-Reset: 1640606400

弘益人間 (Hongik Ingan) Philosophy in API Design

The WIA-LEGAL-001 API embodies the principle of "Benefit All Humanity" through:

  • Open Standards: Complete API documentation published openly for any jurisdiction to implement
  • Fair Access: Rate limits designed to prevent abuse while allowing legitimate high-volume use
  • Developer Experience: Clear documentation, consistent patterns, and helpful error messages reduce barriers to adoption
  • Accessibility: API enables creation of specialized interfaces for users with disabilities or limited technical access
  • Transparency: Audit logs and public endpoints support court system accountability
