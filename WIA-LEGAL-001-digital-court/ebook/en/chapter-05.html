Chapter 5: Protocol Specifications (PHASE-3)

5.1 Introduction to Communication Protocols

Phase 3 of the WIA-LEGAL-001 standard represents the critical infrastructure layer that enables real-time, secure communication within digital court systems. While Phase 1 established standardized data formats and Phase 2 defined the API interfaces, Phase 3 specifies the underlying protocols that make virtual hearings, secure messaging, encrypted document transfers, and blockchain-based record keeping possible.

The protocol specifications in this phase address fundamental questions about how digital court systems communicate: How are video streams transmitted with minimal latency? How can we ensure end-to-end encryption for sensitive legal communications? What protocols guarantee the integrity of evidence as it moves through the system? These questions and many others are answered through the comprehensive protocol framework detailed in this chapter.

Key Principle: Protocol design in legal systems must balance three sometimes-competing requirements: security (protecting confidential information), accessibility (enabling participation from any device or location), and auditability (maintaining comprehensive records of all communications).

5.2 Virtual Hearing Protocol Architecture

5.2.1 WebRTC-Based Video Conferencing

The virtual hearing protocol is built on WebRTC (Web Real-Time Communication), an open-source framework that enables peer-to-peer audio and video communication directly in web browsers without requiring plugins or additional software. This choice ensures maximum accessibility while maintaining high quality and low latency.

5.2.2 Connection Flow Sequence

The establishment of a virtual courtroom session follows a carefully orchestrated sequence of steps to ensure security, proper identity verification, and optimal media quality: 

1. Authentication Phase
   ├─ User authenticates with court server (OAuth 2.0 + MFA)
   ├─ Session token issued (JWT with 1-hour expiration)
   └─ Role and permissions verified

2. Room Creation Phase
   ├─ Secure virtual courtroom instantiated
   ├─ Unique room ID generated
   ├─ Recording infrastructure initialized
   └─ Access controls configured per case confidentiality

3. Participant Join Phase
   ├─ Identity verification against case registry
   ├─ Biometric verification (optional, for high-security cases)
   ├─ Device capability detection
   └─ Network quality assessment

4. Media Negotiation Phase
   ├─ Session Description Protocol (SDP) exchange
   ├─ ICE candidate gathering
   ├─ STUN/TURN server selection
   └─ Optimal codec selection based on bandwidth

5. Encrypted Streaming Phase
   ├─ DTLS-SRTP encrypted media streams established
   ├─ Continuous quality monitoring
   ├─ Adaptive bitrate adjustment
   └─ Packet loss mitigation

6. Recording Phase
   ├─ Server-side encrypted recording initiated
   ├─ Multi-track recording (separate audio/video per participant)
   ├─ Real-time transcription generation
   └─ Blockchain anchoring of recording hash

7. Transcription Phase
   ├─ Real-time speech-to-text conversion
   ├─ Speaker identification and labeling
   ├─ Legal terminology recognition
   └─ Timestamped transcript generation

5.3 Secure Messaging Protocol

5.3.1 End-to-End Encryption Architecture

Communication between parties in a legal case requires the highest level of security. The WIA-LEGAL-001 secure messaging protocol implements the Signal Protocol, widely recognized as the gold standard for end-to-end encrypted messaging. This ensures that only the intended recipients can read messages, even if the server infrastructure is compromised.

5.4.2 Chunked Upload Implementation

Large legal documents (such as multi-volume depositions or extensive evidence compilations) may be hundreds of megabytes or even gigabytes in size. The chunked upload mechanism ensures reliable transfer even over unstable network connections. 

// JavaScript example: Chunked document upload
class SecureDocumentUploader {
  constructor(apiClient, chunkSize = 5 * 1024 * 1024) { // 5MB chunks
    this.client = apiClient;
    this.chunkSize = chunkSize;
  }

  async uploadDocument(file, caseId, metadata) {
    // Step 1: Calculate file hash
    const fileHash = await this.calculateSHA256(file);

    // Step 2: Initialize upload session
    const session = await this.client.post('/documents/upload/init', {
      caseId: caseId,
      filename: file.name,
      fileSize: file.size,
      fileHash: fileHash,
      metadata: metadata
    });

    // Step 3: Upload file in chunks
    const totalChunks = Math.ceil(file.size / this.chunkSize);

    for (let i = 0; i < totalChunks; i++) {
      const start = i * this.chunkSize;
      const end = Math.min(start + this.chunkSize, file.size);
      const chunk = file.slice(start, end);

      const chunkHash = await this.calculateSHA256(chunk);

      await this.client.post(`/documents/upload/${session.uploadId}/chunk`, {
        chunkIndex: i,
        totalChunks: totalChunks,
        chunkHash: chunkHash,
        data: chunk
      });

      // Report progress
      this.onProgress && this.onProgress({
        uploadedChunks: i + 1,
        totalChunks: totalChunks,
        percentage: ((i + 1) / totalChunks) * 100
      });
    }

    // Step 4: Finalize upload
    const result = await this.client.post(
      `/documents/upload/${session.uploadId}/complete`,
      {
        finalHash: fileHash
      }
    );

    return result.documentId;
  }

  async calculateSHA256(data) {
    const buffer = await data.arrayBuffer();
    const hashBuffer = await crypto.subtle.digest('SHA-256', buffer);
    const hashArray = Array.from(new Uint8Array(hashBuffer));
    return hashArray.map(b => b.toString(16).padStart(2, '0')).join('');
  }
}

pragma solidity ^0.8.19;

contract WIADigitalCourtRecords {
    struct CourtRecord {
        bytes32 recordId;
        bytes32 caseId;
        string recordType;
        bytes32 contentHash;
        string ipfsHash;
        address submittedBy;
        uint256 timestamp;
        bytes signature;
        bool sealed;
    }

    mapping(bytes32 => CourtRecord) public records;
    mapping(bytes32 => bytes32[]) public caseRecords;

    event RecordAnchored(
        bytes32 indexed recordId,
        bytes32 indexed caseId,
        string recordType,
        uint256 timestamp
    );

    modifier onlyAuthorizedCourt() {
        require(
            authorizedCourts[msg.sender],
            "Only authorized courts can anchor records"
        );
        _;
    }

    function anchorRecord(
        bytes32 _recordId,
        bytes32 _caseId,
        string memory _recordType,
        bytes32 _contentHash,
        string memory _ipfsHash,
        bytes memory _signature,
        bool _sealed
    ) public onlyAuthorizedCourt returns (bool) {
        require(
            records[_recordId].timestamp == 0,
            "Record already exists"
        );

        records[_recordId] = CourtRecord({
            recordId: _recordId,
            caseId: _caseId,
            recordType: _recordType,
            contentHash: _contentHash,
            ipfsHash: _ipfsHash,
            submittedBy: msg.sender,
            timestamp: block.timestamp,
            signature: _signature,
            sealed: _sealed
        });

        caseRecords[_caseId].push(_recordId);

        emit RecordAnchored(
            _recordId,
            _caseId,
            _recordType,
            block.timestamp
        );

        return true;
    }

    function verifyRecord(
        bytes32 _recordId,
        bytes32 _contentHash
    ) public view returns (bool) {
        CourtRecord memory record = records[_recordId];
        return record.contentHash == _contentHash;
    }

    function getCaseRecords(
        bytes32 _caseId
    ) public view returns (bytes32[] memory) {
        return caseRecords[_caseId];
    }
}

5.6 Event Streaming Protocol

5.6.1 Real-Time Updates Architecture

Digital court systems require real-time notifications for case updates, document filings, hearing schedules, and other time-sensitive events. The event streaming protocol uses Server-Sent Events (SSE) for unidirectional server-to-client updates and WebSocket for bidirectional communication. 

// Event Stream Consumer Example
class CourtEventStream {
  constructor(apiEndpoint, accessToken) {
    this.endpoint = apiEndpoint;
    this.token = accessToken;
    this.eventSource = null;
    this.handlers = {};
  }

  connect(caseId) {
    const url = `${this.endpoint}/events/stream?caseId=${caseId}`;

    this.eventSource = new EventSource(url, {
      headers: {
        'Authorization': `Bearer ${this.token}`
      }
    });

    // Generic message handler
    this.eventSource.onmessage = (event) => {
      const data = JSON.parse(event.data);
      this.handleEvent(data);
    };

    // Specific event type handlers
    this.eventSource.addEventListener('case.updated', (event) => {
      const data = JSON.parse(event.data);
      this.trigger('case.updated', data);
    });

    this.eventSource.addEventListener('document.filed', (event) => {
      const data = JSON.parse(event.data);
      this.trigger('document.filed', data);
    });

    this.eventSource.addEventListener('hearing.scheduled', (event) => {
      const data = JSON.parse(event.data);
      this.trigger('hearing.scheduled', data);
    });

    this.eventSource.addEventListener('order.issued', (event) => {
      const data = JSON.parse(event.data);
      this.trigger('order.issued', data);
    });

    this.eventSource.onerror = (error) => {
      console.error('EventSource error:', error);
      this.reconnect();
    };
  }

  on(eventType, handler) {
    if (!this.handlers[eventType]) {
      this.handlers[eventType] = [];
    }
    this.handlers[eventType].push(handler);
  }

  trigger(eventType, data) {
    if (this.handlers[eventType]) {
      this.handlers[eventType].forEach(handler => handler(data));
    }
  }

  disconnect() {
    if (this.eventSource) {
      this.eventSource.close();
      this.eventSource = null;
    }
  }

  reconnect() {
    setTimeout(() => {
      this.disconnect();
      this.connect();
    }, 5000); // Reconnect after 5 seconds
  }
}
