Chapter 6: Security and Authentication

6.1 The Security Imperative in Digital Courts

Security in digital court systems is not merely a technical concern—it is a fundamental requirement for maintaining the integrity of the justice system itself. Courts handle some of society's most sensitive information: criminal evidence, medical records, trade secrets, child custody disputes, and matters of national security. A breach in court security could compromise ongoing investigations, endanger witnesses, violate attorney-client privilege, or undermine public trust in the judicial system.

The WIA-LEGAL-001 standard approaches security with a "defense in depth" philosophy, implementing multiple layers of protection so that if one layer is compromised, others remain intact. This chapter details the comprehensive security architecture that protects digital court systems from unauthorized access, data breaches, and malicious attacks while maintaining the accessibility that justice requires.

Security Principle: In legal systems, security measures must be transparent and auditable. Users must understand how their information is protected, and courts must be able to prove that security measures were properly applied. This transparency builds trust while maintaining protection.

6.2 Multi-Factor Authentication (MFA)

6.2.1 Authentication Levels

The WIA-LEGAL-001 standard defines four levels of authentication, each building upon the previous to provide increasing levels of security. The appropriate level is determined by the sensitivity of the operation being performed and the role of the user.

Level Authentication Method Use Cases Attack Resistance
Level 1 Username + Strong Password Public record access, basic browsing Low (vulnerable to credential theft)
Level 2 Level 1 + TOTP/SMS Case filing, document access Medium (resistant to password theft)
Level 3 Level 2 + Hardware Security Key Sealed documents, evidence submission High (phishing-resistant)
Level 4 Level 3 + Biometric Verification Judicial orders, sensitive operations Very High (multi-modal verification)

6.2.2 FIDO2/WebAuthn Implementation

For Level 3 and Level 4 authentication, the WIA-LEGAL-001 standard mandates support for FIDO2/WebAuthn, a modern, phishing-resistant authentication standard supported by all major browsers and operating systems. Unlike traditional two-factor methods (SMS codes or TOTP), FIDO2 uses cryptographic keys that cannot be phished, as they are bound to the specific domain of the court system. 

// JavaScript: WebAuthn Registration Example
async function registerSecurityKey(userId, userName) {
  // Request challenge from server
  const challengeResponse = await fetch('/auth/webauthn/register/begin', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ userId, userName })
  });

  const options = await challengeResponse.json();

  // Create credential with user's security key
  const credential = await navigator.credentials.create({
    publicKey: {
      challenge: base64ToArrayBuffer(options.challenge),
      rp: {
        name: "WIA Digital Court",
        id: "digitalcourt.wia.org"
      },
      user: {
        id: base64ToArrayBuffer(options.user.id),
        name: userName,
        displayName: options.user.displayName
      },
      pubKeyCredParams: [
        { type: "public-key", alg: -7 },  // ES256
        { type: "public-key", alg: -257 } // RS256
      ],
      authenticatorSelection: {
        authenticatorAttachment: "cross-platform",
        requireResidentKey: false,
        userVerification: "preferred"
      },
      timeout: 60000,
      attestation: "direct"
    }
  });

  // Send credential to server for verification
  const response = await fetch('/auth/webauthn/register/complete', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      id: credential.id,
      rawId: arrayBufferToBase64(credential.rawId),
      response: {
        clientDataJSON: arrayBufferToBase64(credential.response.clientDataJSON),
        attestationObject: arrayBufferToBase64(credential.response.attestationObject)
      },
      type: credential.type
    })
  });

  return response.json();
}

// Authentication (login) with security key
async function authenticateWithSecurityKey(userId) {
  const challengeResponse = await fetch('/auth/webauthn/login/begin', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ userId })
  });

  const options = await challengeResponse.json();

  const assertion = await navigator.credentials.get({
    publicKey: {
      challenge: base64ToArrayBuffer(options.challenge),
      timeout: 60000,
      rpId: "digitalcourt.wia.org",
      allowCredentials: options.allowCredentials.map(cred => ({
        type: "public-key",
        id: base64ToArrayBuffer(cred.id)
      })),
      userVerification: "preferred"
    }
  });

  const response = await fetch('/auth/webauthn/login/complete', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      id: assertion.id,
      rawId: arrayBufferToBase64(assertion.rawId),
      response: {
        clientDataJSON: arrayBufferToBase64(assertion.response.clientDataJSON),
        authenticatorData: arrayBufferToBase64(assertion.response.authenticatorData),
        signature: arrayBufferToBase64(assertion.response.signature),
        userHandle: arrayBufferToBase64(assertion.response.userHandle)
      },
      type: assertion.type
    })
  });

  return response.json();
}

6.3 Role-Based Access Control (RBAC)

6.3.1 Role Hierarchy and Permissions

Not everyone in a digital court system should have access to everything. The principle of least privilege dictates that users should have only the minimum permissions necessary to perform their duties. The WIA-LEGAL-001 standard defines a comprehensive role hierarchy with granular permissions.

Role Primary Permissions Restrictions Auth Level
Judge Full case access, issue orders, seal/unseal documents, preside hearings Limited to assigned cases, all actions logged Level 4
Attorney File documents, access case files, participate in hearings Only cases where representing a party Level 3
Party (Self-Rep) View own case, file documents, attend hearings Cannot access sealed docs, limited case history Level 2
Court Clerk Case management, scheduling, document processing Cannot issue orders or seal documents Level 3
Evidence Custodian Evidence intake, chain of custody management Cannot access case merits or deliberations Level 3
Public Observer View public records, attend public hearings No access to sealed/confidential information Level 1
System Administrator System maintenance, user management Cannot access case content, all actions audited Level 4

interface Permission {
  resource: string;    // e.g., "case", "document", "hearing"
  action: string;      // e.g., "read", "write", "delete", "seal"
  condition?: string;  // e.g., "assigned_to_user", "is_public"
}

interface Role {
  roleId: string;
  roleName: string;
  permissions: Permission[];
  inheritsFrom?: string[];
}

        { resource: 'document', action: 'read_all', condition: 'case_assigned' },
        { resource: 'document', action: 'seal' },
        { resource: 'document', action: 'unseal' },
        { resource: 'hearing', action: 'preside' },
        { resource: 'order', action: 'issue' },
        { resource: 'evidence', action: 'rule_admissibility' }
      ]
    });

    // Define Attorney role
    this.roles.set('attorney', {
      roleId: 'attorney',
      roleName: 'Attorney',
      permissions: [
        { resource: 'case', action: 'read', condition: 'represents_party' },
        { resource: 'document', action: 'file' },
        { resource: 'document', action: 'read_unsealed', condition: 'case_access' },
        { resource: 'hearing', action: 'participate' },
        { resource: 'motion', action: 'file' },
        { resource: 'evidence', action: 'submit' }
      ]
    });

    // Define Party role
    this.roles.set('party', {
      roleId: 'party',
      roleName: 'Party (Self-Represented)',
      permissions: [
        { resource: 'case', action: 'read', condition: 'is_party' },
        { resource: 'document', action: 'read_own' },
        { resource: 'document', action: 'file' },
        { resource: 'hearing', action: 'attend' }
      ]
    });

    // Additional roles defined similarly...

  async checkPermission(
    userId: string,
    resource: string,
    action: string,
    context: any
  ): Promise {
    const userRoles = await this.getUserRoles(userId);

    for (const roleName of userRoles) {
      const role = this.roles.get(roleName);
      if (!role) continue;

      for (const permission of role.permissions) {
        if (permission.resource === resource && permission.action === action) {
          if (!permission.condition) return true;

          // Evaluate condition
          const conditionMet = await this.evaluateCondition(
            permission.condition,
            userId,
            context
          );

          if (conditionMet) return true;
        }
      }
    }

    return false;
  }

  private async evaluateCondition(
    condition: string,
    userId: string,
    context: any
  ): Promise {
    switch (condition) {
      case 'assigned_to_user':
        return context.case?.assignedJudge === userId;

      case 'represents_party':
        return context.case?.parties.some((p: any) =>
          p.attorney === userId
        );

      case 'is_party':
        return context.case?.parties.some((p: any) =>
          p.partyId === userId
        );

      case 'is_public':
        return context.document?.confidentiality === 'public';

      default:
        return false;

6.4 Encryption Standards

6.4.1 Data at Rest Encryption

All data stored in digital court systems must be encrypted to prevent unauthorized access in the event of a storage breach or physical theft of servers. The WIA-LEGAL-001 standard mandates AES-256-GCM (Galois/Counter Mode) encryption for all data at rest, with keys stored in Hardware Security Modules (HSMs).

Data Type Encryption Method Key Management Rotation Schedule
Case Records AES-256-GCM HSM (AWS KMS / Azure Key Vault) Annual
Documents AES-256-GCM + per-document keys Master key in HSM, doc keys encrypted Per document version
Evidence Files AES-256-GCM + integrity hash HSM with blockchain hash anchoring Never (immutable)
Hearing Recordings AES-256-GCM + streaming encryption HSM with multi-part encryption 90 days
Database (PII) Transparent Data Encryption (TDE) HSM-managed master key Quarterly
Backups AES-256-GCM with separate keys Separate HSM, geographically distributed Every backup

6.4.2 Data in Transit Encryption

All communication between clients and servers, between services, and between jurisdictions must be encrypted to prevent eavesdropping and man-in-the-middle attacks. The standard requires TLS 1.3 as the minimum version, with perfect forward secrecy (PFS) enabled. 

// NGINX TLS Configuration Example
server {
    listen 443 ssl http2;
    server_name digitalcourt.wia.org;

    # TLS 1.3 only (most secure)
    ssl_protocols TLSv1.3;

    # Certificate configuration
    ssl_certificate /etc/ssl/certs/digitalcourt.wia.org.crt;
    ssl_certificate_key /etc/ssl/private/digitalcourt.wia.org.key;

    # Modern cipher suite (TLS 1.3)
    ssl_ciphers 'TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256';
    ssl_prefer_server_ciphers on;

    # OCSP stapling
    ssl_stapling on;
    ssl_stapling_verify on;
    ssl_trusted_certificate /etc/ssl/certs/ca-bundle.crt;

    # HSTS (HTTP Strict Transport Security)
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;

    # Certificate Transparency
    ssl_ct on;
    ssl_ct_static_scts /etc/ssl/scts;

    # Additional security headers
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
    add_header Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline';" always;

    # Session configuration
    ssl_session_timeout 1d;
    ssl_session_cache shared:SSL:50m;
    ssl_session_tickets off;

    location / {
        proxy_pass http://backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

6.5 Compliance and Regulatory Requirements

6.5.1 Data Protection Regulations

Digital court systems must comply with various data protection regulations depending on their jurisdiction and the types of cases they handle. The WIA-LEGAL-001 standard is designed to meet or exceed the requirements of major international regulations.

Regulation Jurisdiction Key Requirements WIA-LEGAL-001 Compliance
GDPR European Union Right to access, right to be forgotten, data minimization, consent management Full compliance with exception for legal proceedings
CCPA/CPRA California, USA Consumer rights, opt-out mechanisms, data sale restrictions Full compliance, transparency in data handling
HIPAA USA (Healthcare) Protected health information (PHI) safeguards, business associate agreements HIPAA-compliant when handling medical records
SOC 2 Type II International Security, availability, confidentiality, privacy controls Annual SOC 2 audit recommended
ISO 27001 International Information security management system (ISMS) Certification available for implementations
CJIS USA (Criminal Justice) FBI security requirements for criminal justice data CJIS-compliant for criminal court systems

6.5.2 Accessibility Standards

Security must not come at the cost of accessibility. Courts serve everyone, including people with disabilities. The WIA-LEGAL-001 standard requires compliance with WCAG 2.1 Level AA at minimum, ensuring that security features like multi-factor authentication work with screen readers and other assistive technologies.

6.7 Security Monitoring and Threat Detection

6.7.1 Security Information and Event Management (SIEM)

Continuous monitoring is essential for detecting and responding to security threats in real-time. The WIA-LEGAL-001 standard recommends implementing a SIEM solution that aggregates logs from all system components and uses machine learning to detect anomalous behavior. 

// Example: Security Event Detection Rules
const securityRules = [
  {
    name: "Multiple Failed Login Attempts",
    condition: (events) => {
      const failedLogins = events.filter(e =>
        e.type === 'auth.failed' &&
        e.userId === events[0].userId
      );
      return failedLogins.length >= 5 &&
             withinTimeWindow(failedLogins, 15 * 60 * 1000); // 15 minutes
    },
    severity: "MEDIUM",
    action: "lock_account_and_notify"
  },

  {
    name: "Access to Sealed Document Without Authorization",
    condition: (events) => {
      return events.some(e =>
        e.type === 'document.accessed' &&
        e.document.confidentiality === 'sealed' &&
        !e.user.permissions.includes('access_sealed')
      );
    },
    severity: "HIGH",
    action: "immediate_alert_and_audit"
  },

  {
    name: "Unusual Download Volume",
    condition: (events) => {
      const downloads = events.filter(e => e.type === 'document.downloaded');
      const totalSize = downloads.reduce((sum, d) => sum + d.size, 0);
      const userAverage = getUserHistoricalAverage(events[0].userId);
      return totalSize > userAverage * 10; // 10x normal
    },
    severity: "HIGH",
    action: "throttle_and_investigate"
  },

  {
    name: "Geographic Anomaly",
    condition: (events) => {
      const lastKnownLocation = getLastKnownLocation(events[0].userId);
      const currentLocation = events[0].geoLocation;
      const distance = calculateDistance(lastKnownLocation, currentLocation);
      const timeDiff = events[0].timestamp - lastKnownLocation.timestamp;

      // Impossible travel: >500km in <1 hour
      return distance > 500 && timeDiff < 3600000;
    },
    severity: "HIGH",
    action: "require_additional_authentication"
  },

  {
    name: "Privilege Escalation Attempt",
    condition: (events) => {
      return events.some(e =>
        e.type === 'permission.denied' &&
        e.attemptedAction === 'elevate_privileges'
      );
    },
    severity: "CRITICAL",
    action: "immediate_lock_and_investigate"
  }
];
