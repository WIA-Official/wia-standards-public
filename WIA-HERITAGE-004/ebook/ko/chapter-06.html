Phase 3: 프로토콜

WIA-HERITAGE-004: Intangible Heritage Standard

06.1 Phase 3의 목표와 범위

Phase 3는 실시간 통신과 복잡한 분산 워크플로우를 위한 프로토콜을 정의합니다. Phase 2의 API가 데이터의 조회와 업로드를 담당한다면, Phase 3의 프로토콜은 라이브 공연 스트리밍, 다중 카메라 동기화, 모션 캡처 데이터의 실시간 전송 등 시간에 민감한 작업을 가능하게 합니다.

2024년 판소리 세계 축제에서는 서울, 전주, 뉴욕, 파리의 4개 공연장이 동시에 연결되어 라이브 협연을 진행했습니다. 이는 WIA-HERITAGE-004 Phase 3 프로토콜 덕분에 가능했으며, 4개 도시 간 평균 지연시간은 280ms에 불과했습니다.

🌐 Phase 3 핵심 목표
  • WebRTC 라이브 스트리밍: 500ms 이하 초저지연 실시간 방송
  • 다중 카메라 동기화: 20대 이상 카메라의 나노초 단위 타임코드 동기화
  • 모션 캡처 실시간 전송: 초당 120프레임 모션 데이터 지연 없는 전송
  • 블록체인 무결성 보장: 변조 방지 타임스탬프와 해시 체인
  • 분산 협업 녹화: 다수의 원격지에서 동시 녹화 및 실시간 편집
  • QoS(Quality of Service): 네트워크 품질에 따른 적응형 품질 조정

구현 난이도 및 필요 자원

Phase 3는 고급 네트워크 및 실시간 시스템 전문 지식이 필요합니다:

  • 전문 인력: 실시간 통신 엔지니어, 네트워크 전문가, DevOps 엔지니어
  • 인프라: 저지연 서버 (Edge Computing), WebRTC 중계 서버, 블록체인 노드
  • 네트워크: 최소 100Mbps 업로드 대역폭, 안정적인 연결
  • 장비: 타임코드 동기화 장치, 네트워크 카메라, 모션 캡처 시스템
  • 예산: 초기 $50,000-150,000, 월 운영비 $2,000-10,000

06.2 WebRTC 초저지연 라이브 스트리밍

무형 문화유산 라이브 공연을 전 세계에 실시간으로 방송하려면 지연시간이 1초 이하여야 관객이 현장감을 느낄 수 있습니다. 전통적인 RTMP/HLS는 5-30초의 지연이 발생하지만, WebRTC(Web Real-Time Communication)는 500ms 이하의 지연을 실현합니다.

WebRTC 아키텍처

구성 요소역할프로토콜
송신자 (Broadcaster)공연장 카메라, 마이크getUserMedia API, MediaStream
시그널링 서버피어 간 연결 중개WebSocket, SDP (Session Description Protocol)
TURN/STUN 서버NAT 통과, 방화벽 우회ICE (Interactive Connectivity Establishment)
SFU (Selective Forwarding Unit)다중 수신자에게 스트림 분배WebRTC, VP9/H.264
수신자 (Viewer)전 세계 관객 브라우저RTCPeerConnection

WebRTC 연결 설정 순서

// 1단계: 송신자가 미디어 스트림 획득
const stream = await navigator.mediaDevices.getUserMedia({
  video: { width: 1920, height: 1080, frameRate: 60 },
  audio: { sampleRate: 48000, channelCount: 2 }
});

// 2단계: RTCPeerConnection 생성
const pc = new RTCPeerConnection({
  iceServers: [
    { urls: 'stun:stun.heritage.gugak.go.kr:3478' },
    { urls: 'turn:turn.heritage.gugak.go.kr:3478', username: 'user', credential: 'pass' }
  ]
});

// 3단계: 로컬 스트림 추가
stream.getTracks().forEach(track => pc.addTrack(track, stream));

// 4단계: Offer 생성 및 시그널링 서버로 전송
const offer = await pc.createOffer();
await pc.setLocalDescription(offer);
signalingServer.send({ type: 'offer', sdp: offer.sdp });

// 5단계: 원격 Answer 수신 및 설정
signalingServer.on('answer', async (answer) => {
  await pc.setRemoteDescription(answer);
});

// 6단계: ICE 후보 교환
pc.onicecandidate = (event) => {
  if (event.candidate) {
    signalingServer.send({ type: 'ice-candidate', candidate: event.candidate });
  }
};

// 연결 성공 시 라이브 스트리밍 시작
pc.onconnectionstatechange = () => {
  console.log('Connection state:', pc.connectionState);
  if (pc.connectionState === 'connected') {
    console.log('Live streaming started!');
  }
};

다중 화질 라이브 스트리밍 (Simulcast)

수신자의 네트워크 대역폭에 맞춰 자동으로 화질 조정:

const pc = new RTCPeerConnection({...});
const sender = pc.addTrack(videoTrack, stream);

// 동시에 3가지 품질 전송
await sender.setParameters({
  encodings: [
    { rid: 'h', maxBitrate: 5000000, scaleResolutionDownBy: 1.0 },  // 1080p
    { rid: 'm', maxBitrate: 2000000, scaleResolutionDownBy: 2.0 },  // 540p
    { rid: 'l', maxBitrate: 800000, scaleResolutionDownBy: 4.0 }    // 270p
  ]
});

06.3 다중 카메라 정밀 동기화

무형 문화유산 기록에서 다중 각도 촬영은 필수적입니다. 판소리 공연 시 창자의 표정(정면), 손동작(측면), 고수의 북 치는 장면(측면), 전체 무대(와이드)를 동시에 촬영하여 나중에 편집합니다. 이때 각 카메라의 타임코드가 정확히 일치해야 립싱크가 맞습니다.

06.4 모션 캡처 실시간 전송

전통 무용의 모션 캡처 데이터는 초당 120프레임, 17개 센서에서 68차원 데이터(위치 xyz × 3, 회전 quaternion × 4)를 생성합니다. 이는 초당 약 140KB의 데이터이며, 여러 무용수를 동시에 캡처하면 대역폭이 급증합니다. 실시간 전송을 위한 최적화가 필수적입니다.

  mocapSystem.on('frame', (frameData) => {

    const json = {
      timestamp: Date.now(),
      frameNumber: frameData.frameNumber,
      performers: [

          id: "performer-1",

            { name: "LeftShoulder", position: [5.0, 97.0, 0.0], rotation: [-0.15, 0.10, -0.03, 0.98] },

데이터 압축 및 최적화

대역폭 절약을 위한 압축 기법:

  • Quaternion 압축: 32비트 float × 4 → 16비트 fixed-point × 3 (w는 계산으로 복원)
  • Delta 인코딩: 절대 위치 대신 이전 프레임과의 차이만 전송
  • 키프레임 + 보간: 매 프레임 전송 대신 키프레임만 전송, 중간은 클라이언트에서 보간
  • 선택적 전송: 움직임이 거의 없는 관절은 업데이트 생략
// 압축 전: 140 KB/초
{
  "joints": [
    { "name": "Hips", "position": [0.0, 92.5, 0.0], "rotation": [0, 0, 0, 1] },
    ...
  ]
}

// Delta 인코딩 후: 35 KB/초
{
  "keyframe": false,
  "delta": [
    { "name": "Hips", "dp": [0.01, -0.02, 0.0], "dq": [0.001, 0, 0] },
    ...
  ]
}

06.5 블록체인 기반 무결성 보장

무형 문화유산 디지털 아카이브의 가장 큰 위협 중 하나는 데이터 변조입니다. 누군가 판소리 녹음의 일부를 삭제하거나 수정하면, 그 기록의 진위를 의심하게 됩니다. 블록체인 기술은 데이터의 생성 시점, 수정 이력, 원본 여부를 증명 가능하게 만듭니다.

블록체인 아카이빙 아키텍처

WIA-HERITAGE-004는 하이브리드 모델을 사용합니다:

  • 대용량 미디어 파일: IPFS(InterPlanetary File System) 분산 스토리지에 저장
  • 메타데이터 및 해시: Ethereum/Polygon 블록체인에 기록
  • 타임스탬프: Bitcoin 블록체인 OP_RETURN으로 최종 검증

Ethereum 스마트 컨트랙트

메타데이터와 해시를 블록체인에 영구 기록:

// Solidity 스마트 컨트랙트
pragma solidity ^0.8.0;

contract HeritageArchive {
    struct Performance {
        string performanceId;
        string title;
        string ipfsCid;
        bytes32 sha256Hash;
        uint256 timestamp;
        address uploader;
        string metadata; // JSON-LD URL
    }

    mapping(string => Performance) public performances;
    string[] public performanceIds;

    event PerformanceArchived(
        string indexed performanceId,
        string ipfsCid,
        bytes32 sha256Hash,
        uint256 timestamp
    );

    function archivePerformance(
        string memory _performanceId,
        string memory _title,
        string memory _ipfsCid,
        bytes32 _sha256Hash,
        string memory _metadata
    ) public {
        require(
            bytes(performances[_performanceId].performanceId).length == 0,
            "Performance already archived"
        );

        performances[_performanceId] = Performance({
            performanceId: _performanceId,
            title: _title,
            ipfsCid: _ipfsCid,
            sha256Hash: _sha256Hash,
            timestamp: block.timestamp,
            uploader: msg.sender,
            metadata: _metadata
        });

        performanceIds.push(_performanceId);

        emit PerformanceArchived(_performanceId, _ipfsCid, _sha256Hash, block.timestamp);
    }

    function verifyPerformance(
        string memory _performanceId,
        bytes32 _sha256Hash
    ) public view returns (bool) {
        Performance memory perf = performances[_performanceId];
        return perf.sha256Hash == _sha256Hash;
    }

    function getPerformance(
        string memory _performanceId
    ) public view returns (Performance memory) {
        return performances[_performanceId];
    }
}

06.7 QoS (Quality of Service) 및 네트워크 최적화

실시간 스트리밍과 모션 캡처 전송은 네트워크 품질에 매우 민감합니다. 패킷 손실, 지연, 지터는 즉시 시청 경험을 저하시킵니다. QoS 메커니즘은 네트워크 조건에 맞춰 자동으로 품질을 조정합니다.

적응형 비트레이트 제어

// 네트워크 상태 모니터링
const networkMonitor = {
  bandwidth: 5000000,  // 5 Mbps
  latency: 120,        // 120 ms
  packetLoss: 0.02,    // 2%
  jitter: 15           // 15 ms
};

// 품질 레벨 자동 선택
function selectQualityLevel(networkMonitor) {
  if (networkMonitor.bandwidth > 10000000 && networkMonitor.latency < 50) {
    return { resolution: '4K', bitrate: 15000000, fps: 60 };
  } else if (networkMonitor.bandwidth > 5000000 && networkMonitor.latency < 100) {
    return { resolution: '1080p', bitrate: 5000000, fps: 30 };
  } else if (networkMonitor.bandwidth > 2000000) {
    return { resolution: '720p', bitrate: 2000000, fps: 30 };
  } else {
    return { resolution: '360p', bitrate: 800000, fps: 24 };
  }
}

// 실시간 품질 조정
setInterval(() => {
  const currentNetwork = measureNetworkQuality();
  const quality = selectQualityLevel(currentNetwork);

  if (quality.resolution !== currentQuality.resolution) {
    console.log(`Switching quality: ${currentQuality.resolution} → ${quality.resolution}`);
    switchStreamQuality(quality);
  }
}, 5000);  // 5초마다 체크

06.8 장 요약

✅ 핵심 내용
  1. Phase 3는 실시간 통신 프로토콜로 라이브 스트리밍, 다중 카메라 동기화, 분산 협업을 가능하게 합니다.
  2. WebRTC는 500ms 이하 초저지연 라이브 스트리밍을 제공하며, Simulcast로 다중 화질 동시 전송이 가능합니다.
  3. PTP (IEEE 1588)는 네트워크 카메라를 마이크로초 단위로 동기화하여 다중 각도 촬영의 완벽한 립싱크를 보장합니다.
  4. 모션 캡처 실시간 전송은 WebSocket과 Delta 인코딩으로 대역폭을 73% 절감하며 28ms 지연으로 VR 실시간 애니메이션을 구현합니다.
  5. 블록체인과 IPFS는 데이터 무결성을 영구 보장하며, 누구나 파일의 진위를 검증할 수 있습니다.
  6. 분산 협업 녹화는 세계 각지의 전승자가 동시에 공연하고 녹화할 수 있게 하며, NTP로 나노초 단위 동기화를 달성합니다.
  7. QoS 메커니즘은 네트워크 상태에 따라 자동으로 품질을 조정하여 안정적인 스트리밍을 보장합니다.

06.9 복습 문제

  1. WebRTC가 전통적인 RTMP/HLS 스트리밍보다 낮은 지연시간을 제공하는 이유를 P2P 연결 구조와 연관지어 설명하세요.
  2. PTP (IEEE 1588) 타임코드 동기화가 다중 카메라 녹화에서 왜 중요한지 설명하고, 립싱크 오류가 발생하는 구체적인 상황을 예로 들어보세요.
  3. 모션 캡처 데이터의 Delta 인코딩 기법을 설명하고, 어떻게 대역폭을 절감하는지 수치적으로 논의하세요.
  4. 블록체인 기반 아카이빙에서 IPFS와 Ethereum의 역할 분담을 설명하고, 왜 대용량 파일을 블록체인에 직접 저장하지 않는지 논의하세요.
  5. 분산 협업 녹화에서 서울, 뉴욕, 파리의 장비를 동기화하는 과정을 NTP와 타임존 차이를 고려하여 단계별로 설명하세요.
  6. QoS 적응형 비트레이트 제어가 무형 문화유산 라이브 스트리밍에서 중요한 이유를 설명하고, 네트워크 불안정 시 자동 품질 조정 알고리즘을 제시하세요.
