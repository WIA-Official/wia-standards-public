Phase 2: API 인터페이스

WIA-HERITAGE-004: Intangible Heritage Standard

05.1 Phase 2의 목표와 범위

Phase 2는 Phase 1에서 생성된 표준화된 무형 문화유산 데이터를 실제로 활용하기 위한 API(Application Programming Interface)를 정의합니다. 데이터가 아무리 풍부하고 정확해도 시스템 간에 공유되지 않으면 고립된 섬에 불과합니다. Phase 2는 이러한 데이터 섬들을 연결하는 다리를 놓습니다.

2024년 국제 문화유산 기관 조사에 따르면, 디지털 아카이브의 67%가 API를 제공하지 않아 데이터가 해당 기관 웹사이트에만 갇혀 있습니다. 연구자들은 각 기관 웹사이트를 일일이 방문해야 하며, 데이터를 자동으로 수집하거나 분석할 수 없습니다.

🔌 Phase 2 핵심 목표
  • RESTful API: 표준 HTTP 메서드로 무형 문화유산 데이터 검색, 조회, 업로드
  • GraphQL 쿼리: 복잡한 관계형 데이터를 효율적으로 조회
  • 비디오 스트리밍 API: HLS/DASH 프로토콜로 적응형 스트리밍 제공
  • 공동체 업로드 포털: 원주민 공동체가 직접 자신들의 전통을 업로드
  • 세밀한 권한 관리: 문화적 민감성에 따른 접근 제어
  • WebHook 알림: 새 데이터 추가 시 구독자에게 실시간 알림

구현 난이도 및 필요 자원

Phase 2는 중급 수준의 웹 개발 능력이 필요합니다:

  • 기술 스택: Node.js/Python/Go 등 백엔드 프레임워크
  • 데이터베이스: PostgreSQL (관계형) + Elasticsearch (검색엔진)
  • 스토리지: AWS S3, Azure Blob, 또는 자체 서버 (최소 10TB)
  • CDN: CloudFlare, Fastly 등 비디오 스트리밍용
  • 인력: 백엔드 개발자 2명, 프론트엔드 개발자 1명
  • 예산: 초기 구축 $20,000-50,000, 월 운영비 $500-2,000

🔐 인증 및 권한 관리

WIA-HERITAGE-004 API는 OAuth 2.0과 JWT(JSON Web Token)를 사용합니다. 사용자는 먼저 인증 서버에서 토큰을 획득한 후, 모든 API 요청에 이 토큰을 포함합니다. 토큰에는 사용자 역할(일반, 연구자, 공동체 구성원, 관리자)이 포함되어 있으며, 서버는 이를 기반으로 접근 권한을 검증합니다.

05.3 GraphQL API

무형 문화유산 데이터는 복잡한 관계를 가집니다. 예를 들어, 하나의 판소리 공연은 창자, 고수, 공연 장소, 사용된 악기, 계보, 관련 학술 논문 등 수많은 엔티티와 연결되어 있습니다. REST API로 이 모든 데이터를 가져오려면 수십 번의 요청이 필요합니다. GraphQL은 단 한 번의 요청으로 필요한 모든 데이터를 정확히 가져옵니다.

GraphQL 스키마 예시

type Performance {
  id: ID!
  title: String!
  description: String
  type: HeritageType!
  date: DateTime!
  duration: String
  performer: Performer!
  accompanists: [Performer!]
  location: Location!
  mediaFiles: [MediaFile!]!
  culturalContext: CulturalContext
  consent: ConsentInfo!
  relatedPerformances: [Performance!]
  scholarlyArticles: [Article!]
}

type Performer {
  id: ID!
  name: String!
  birthYear: Int
  designation: String
  masterLineage: [Performer!]
  performances: [Performance!]
  traditions: [Tradition!]
}

type Tradition {
  id: ID!
  name: String!
  unescoId: String
  description: String
  community: Community!
  performances: [Performance!]!
  practitioners: [Performer!]!
  threatLevel: ThreatLevel!
}

type Community {
  id: ID!
  name: String!
  region: String!
  country: String!
  traditions: [Tradition!]!
  members: [Performer!]!
}

type Query {
  performance(id: ID!): Performance
  performances(
    type: HeritageType
    region: String
    dateFrom: DateTime
    dateTo: DateTime
    limit: Int
    offset: Int
  ): PerformanceConnection!

  performer(id: ID!): Performer
  performers(name: String, tradition: String): [Performer!]!

  tradition(id: ID!): Tradition
  traditions(unescoListed: Boolean, threatLevel: ThreatLevel): [Tradition!]!

  search(query: String!): SearchResults!
}

type Mutation {
  createPerformance(input: PerformanceInput!): Performance!
  updatePerformance(id: ID!, input: PerformanceInput!): Performance!
  deletePerformance(id: ID!): Boolean!

  uploadMedia(file: Upload!, metadata: MediaMetadataInput!): MediaFile!
}

GraphQL의 장점

  • Over-fetching 방지: 필요한 필드만 정확히 요청 (REST는 모든 필드 반환)
  • Under-fetching 방지: 관련 데이터를 단일 요청으로 조회 (REST는 여러 요청 필요)
  • 강력한 타입 시스템: 스키마로 API 구조를 명확히 정의
  • 실시간 구독: WebSocket으로 데이터 변경 시 즉시 알림
  • 개발자 경험: GraphiQL 같은 대화형 탐색 도구 제공
🎯 국립국악원 GraphQL API 사례

국립국악원은 2024년 9월 GraphQL API를 도입하여 연구자들의 데이터 접근을 혁신했습니다. 이전에는 한 명의 판소리 명창과 그의 계보, 공연 이력, 관련 학술 논문을 조회하려면 평균 17번의 REST API 요청이 필요했습니다. GraphQL 도입 후 단 1번의 요청으로 동일한 데이터를 얻을 수 있게 되었으며, 네트워크 트래픽은 73% 감소, 응답 속도는 5.2배 향상되었습니다.

05.4 비디오 스트리밍 API

무형 문화유산의 핵심은 비디오 콘텐츠입니다. 4K 60fps 판소리 완창 공연 파일은 456GB에 달하며, 이를 직접 다운로드하는 것은 비현실적입니다. 적응형 스트리밍(Adaptive Streaming)이 필수적입니다.

지원 스트리밍 프로토콜

프로토콜형식지원 플랫폼특징
HLS (HTTP Live Streaming).m3u8 + .tsiOS, macOS, Safari 기본 지원가장 널리 사용, CDN 친화적
DASH (Dynamic Adaptive Streaming).mpd + .m4sChrome, Firefox, EdgeISO 표준, 코덱 유연성
WebRTCP2P모든 최신 브라우저초저지연 (<500ms), 라이브 전용
RTMP (실시간 전송)FLV전문 방송 장비라이브 스트리밍 업로드용

DRM 보호 (선택사항)

문화적으로 민감하거나 상업적 가치가 있는 콘텐츠는 DRM으로 보호할 수 있습니다:

  • Widevine: Android, Chrome 지원
  • FairPlay: iOS, Safari 지원
  • PlayReady: Windows, Edge 지원
📱 강강술래 모바일 스트리밍 최적화

진도 강강술래 VR 영상은 360도 8K로 촬영되어 원본 파일이 89GB입니다. HLS 적응형 스트리밍을 통해 4G 모바일 사용자는 1080p 스테레오 버전(2 Mbps)을, 5G 사용자는 4K 360도 버전(15 Mbps)을, WiFi 사용자는 8K 원본(50 Mbps)을 자동으로 받습니다. 사용자는 네트워크 상태에 관계없이 끊김 없이 감상할 수 있습니다.

05.5 공동체 업로드 포털

WIA-HERITAGE-004의 핵심 철학은 공동체 중심 설계입니다. 원주민 공동체가 자신들의 전통을 직접 기록하고 공유할 수 있는 사용자 친화적인 업로드 포털이 필수적입니다.

업로드 워크플로우

  1. 계정 생성 및 공동체 인증: 공동체 대표가 신원을 확인하고 계정 생성
  2. 동의서 작성: 웹 양식으로 전승자 동의서 작성 및 업로드
  3. 메타데이터 입력: 간소화된 양식으로 공연 정보 입력 (자동 완성 지원)
  4. 미디어 업로드: 대용량 파일 분할 업로드 (최대 100GB, 자동 재개)
  5. 자동 처리: 서버가 자동으로 다중 해상도 변환, 썸네일 생성
  6. 검토 및 승인: 공동체 내부 검토 후 공개 여부 결정
  7. 공개: 승인되면 즉시 API와 웹사이트에서 검색 가능

멀티파트 업로드 API

100GB 판소리 공연 비디오를 안정적으로 업로드하려면 파일을 작은 청크로 분할:

05.6 세밀한 권한 관리

무형 문화유산은 문화적 민감성이 다양합니다. 누구나 볼 수 있는 공개 공연도 있지만, 특정 공동체 구성원만 접근해야 하는 신성한 의례도 있습니다. WIA-HERITAGE-004는 5단계 접근 제어 시스템을 제공합니다.

🔒 제주 영등굿 접근 제어 사례

제주 영등굿은 일부 핵심 제의가 Level 1(공동체 내부)로 설정되어 있습니다. 심방(무당)들이 암송하는 신성한 본풀이(신화)는 해녀 공동체 구성원만 열람 가능하며, 외부 연구자는 서면 신청 후 심방회의 승인을 받아야 Level 2(연구자 제한) 권한을 얻습니다. 반면 해변에서 원을 그리며 노래하는 장면은 Level 4(완전 공개)로 누구나 감상할 수 있습니다.

05.8 장 요약

✅ 핵심 내용
  1. Phase 2는 API를 통해 무형 문화유산 데이터를 시스템 간에 공유하고 전 세계 연구자와 기관이 접근할 수 있게 합니다.
  2. RESTful API는 표준 HTTP 메서드(GET, POST, PUT, DELETE)로 공연, 전승자, 공동체 데이터를 CRUD 작업하며, OAuth 2.0과 JWT로 인증합니다.
  3. GraphQL API는 복잡한 관계형 데이터를 단일 쿼리로 조회하여 네트워크 효율을 73% 향상시키고, 강력한 타입 시스템으로 개발자 경험을 개선합니다.
  4. 비디오 스트리밍 API는 HLS/DASH 적응형 스트리밍으로 360p부터 4K까지 사용자 네트워크에 맞춰 자동 전환하며, DRM으로 민감 콘텐츠를 보호합니다.
  5. 공동체 업로드 포털은 멀티파트 업로드로 100GB 파일도 안정적으로 전송하며, 23개 언어 지원과 AI 번역으로 원주민 공동체의 직접 참여를 촉진합니다.
  6. 5단계 접근 제어는 비공개부터 상업 허용까지 문화적 민감성에 따라 세밀하게 권한을 관리하며, RBAC로 역할 기반 권한을 부여합니다.
  7. WebHook과 SSE는 새 데이터 추가 시 구독자에게 실시간 알림을 보내 연구자와 기관의 신속한 대응을 가능하게 합니다.

API는 고립된 문화유산 데이터를 연결하여 인류 지식의 네트워크를 구축합니다

05.9 복습 문제

  1. RESTful API와 GraphQL API의 차이를 설명하고, 무형 문화유산 데이터 조회에서 GraphQL이 유리한 상황을 구체적인 예를 들어 설명하세요.
  2. HLS 적응형 스트리밍의 작동 원리를 설명하고, 왜 대용량 무형 문화유산 비디오 배포에 필수적인지 논의하세요.
  3. 멀티파트 업로드가 단일 파일 업로드보다 우수한 이유를 설명하고, 100GB 판소리 공연 비디오 업로드 시나리오를 단계별로 기술하세요.
  4. 5단계 접근 제어 시스템에서 Level 1(공동체 내부)과 Level 2(연구자 제한)의 차이를 설명하고, 각각 어떤 무형 문화유산 콘텐츠에 적합한지 사례를 들어보세요.
  5. WebHook과 Server-Sent Events(SSE)의 차이점을 설명하고, 각각 어떤 상황에서 유용한지 비교하세요.
  6. 원주민 공동체가 직접 자신들의 전통을 업로드하는 것이 왜 중요한지 弘益人間 철학과 연결하여 논의하고, API가 이를 어떻게 지원하는지 설명하세요.
