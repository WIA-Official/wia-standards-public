WIA-HERITAGE-004 표준 아키텍처 개요

WIA-HERITAGE-004: Intangible Heritage Standard

03.1 설계 철학과 핵심 원칙

WIA-HERITAGE-004 표준의 아키텍처는 무형 문화유산의 독특한 특성을 반영하여 설계되었습니다. 단순한 기술 사양이 아닌, 공동체와 전통을 존중하는 철학적 기반 위에 구축된 통합 시스템입니다.

5대 설계 원칙

  • 공동체 주권(Community Sovereignty): 모든 디지털 자산에 대한 최종 통제권은 원주민 공동체에 있으며, 기술은 이를 지원하는 도구일 뿐입니다. 공동체가 원하지 않으면 아무리 기술적으로 우수해도 구현하지 않습니다.
  • 점진적 증강(Progressive Enhancement): Phase 1의 기본 기능만으로도 유용하며, 상위 단계는 선택적 추가입니다. 스마트폰 하나로도 표준을 준수할 수 있고, 최첨단 연구소에서도 활용할 수 있는 확장성을 제공합니다.
  • 상호운용성 우선(Interoperability First): 폐쇄적 생태계가 아닌 개방형 표준을 지향합니다. 기존의 CIDOC-CRM, Dublin Core, IIIF, Schema.org 등 국제 표준과 완벽하게 호환됩니다.
  • 미래 증명(Future-Proof): 100년 후에도 데이터를 읽을 수 있도록 설계되었습니다. 특정 벤더 기술에 종속되지 않으며, 모든 핵심 형식은 문서화된 개방형 표준입니다.
  • 문화적 맥락 중심(Context-Centric): 데이터보다 맥락이 중요합니다. 영상 파일 자체보다 그것이 왜, 어디서, 누구에 의해, 어떤 의미로 만들어졌는지에 더 많은 메타데이터를 할애합니다.
🏗️ 아키텍처 설계 목표
  • 단순성: 복잡한 기술 지식 없이도 Phase 1 구현 가능
  • 확장성: 10개 항목부터 1억 개 항목까지 처리 가능한 아키텍처
  • 탄력성: 네트워크 장애, 하드웨어 고장에도 데이터 손실 없음
  • 보안성: 종단 간 암호화, 블록체인 무결성 검증
  • 접근성: 저대역폭 환경, 구형 기기에서도 작동

03.2 4단계 아키텍처 상세

WIA-HERITAGE-004의 핵심은 4단계(Phase) 아키텍처입니다. 각 단계는 독립적이면서도 통합될 때 시너지를 발휘합니다.

Phase핵심 기능주요 기술의존성

Phase 1: 데이터 형식 (Data Format Layer)

가장 기본이 되는 계층으로, 무형 문화유산 데이터를 표준화된 형식으로 저장합니다:

  • 메타데이터 스키마: JSON-LD 기반, 최소 15개 필수 필드 (제목, 설명, 생성일, 공동체 ID, 접근 권한 등)
  • 비디오: H.265/HEVC (압축) 또는 ProRes (보존), 최소 1080p 30fps
  • 오디오: FLAC 96kHz/24bit (무손실), AAC 256kbps (스트리밍)
  • 3D/모션: glTF 2.0 (3D 모델), BVH/FBX (모션 캡처)
  • 이미지: JPEG 2000 (보존), WebP (웹)
  • 문서: PDF/A-3 (아카이브), Markdown (편집 가능)

Phase 1만 구현해도 데이터 교환이 가능하며, 외장 하드에 저장하여 물리적으로 전달할 수 있습니다.

Phase 2: API 인터페이스 (API Layer)

네트워크를 통한 데이터 접근을 제공합니다:

  • RESTful API: 간단한 CRUD(생성, 읽기, 수정, 삭제) 작업
  • GraphQL: 복잡한 관계형 쿼리, 정확히 필요한 데이터만 요청
  • 인증/인가: OAuth 2.0, JWT 토큰, 역할 기반 접근 제어(RBAC)
  • 검색 API: 전문 검색, 패싯 필터링, 다국어 지원
  • 업로드 API: 청크 업로드, 재개 가능, 자동 메타데이터 추출
  • 스트리밍 API: 적응형 비트레이트, 다중 화질, 자막 동기화

Phase 2부터는 웹 브라우저나 모바일 앱을 통한 전 세계 접근이 가능합니다.

Phase 3: 프로토콜 (Protocol Layer)

고급 네트워크 기능과 실시간 통신을 지원합니다:

  • 라이브 스트리밍: WebRTC 기반 지연시간 1초 이하의 실시간 방송
  • P2P 동기화: IPFS를 활용한 탈중앙화 저장 및 공유
  • 다중 사용자 협업: 실시간 공동 편집, 주석 달기
  • CDN 통합: 전 세계 엣지 서버 캐싱으로 빠른 로딩
  • 오프라인 동기화: 네트워크 없이 작업 후 자동 병합

Phase 4: 통합 (Integration Layer)

최신 기술과의 통합으로 혁신적 경험을 제공합니다:

  • AI 분석 엔진: 자동 태깅, 동작 분석, 음악 전사
  • VR/AR 플랫폼: WebXR 기반 몰입형 체험
  • 블록체인 레지스트리: Ethereum/Polygon 스마트 계약으로 출처 증명
  • 지식 그래프: 전통 간 관계를 시맨틱 웹으로 연결
  • 다국어 번역: 신경망 기계 번역 + 인간 검수

03.3 시스템 컴포넌트 구조

WIA-HERITAGE-004 참조 구현은 마이크로서비스 아키텍처를 따릅니다. 각 서비스는 독립적으로 배포, 확장, 업데이트 가능합니다.

핵심 컴포넌트

컴포넌트역할기술 스택
메타데이터 서비스JSON-LD 메타데이터 저장 및 검색PostgreSQL + Elasticsearch
미디어 서비스비디오/오디오 인코딩 및 스트리밍FFmpeg, NGINX RTMP
권한 관리 서비스접근 제어, 사용자 인증Keycloak, OAuth 2.0

데이터 흐름

사용자가 전통 춤 공연 영상을 업로드하는 예시:

  1. 업로드: 클라이언트 → 미디어 서비스 (청크 업로드)
  2. 인코딩: 미디어 서비스 → FFmpeg (다중 해상도 트랜스코딩)
  3. 메타데이터 추출: 미디어 서비스 → AI 분석 서비스 (자동 태깅)
  4. 저장: 미디어 파일 → S3/Azure Blob, 메타데이터 → PostgreSQL
  5. 인덱싱: PostgreSQL → Elasticsearch (검색 가능하게)
  6. 블록체인 등록: 메타데이터 서비스 → 블록체인 서비스 (해시 저장)
  7. 알림: 시스템 → 공동체 대표 (검수 요청)

03.4 기존 표준과의 상호운용성

WIA-HERITAGE-004는 고립된 표준이 아닙니다. 기존 문화유산 및 메타데이터 표준과 완벽하게 통합됩니다.

CIDOC-CRM 통합

CIDOC Conceptual Reference Model은 박물관계의 사실상 표준입니다. WIA-HERITAGE-004의 모든 메타데이터는 CIDOC-CRM 엔티티로 매핑 가능합니다:

  • WIA Performance → CIDOC E7 Activity
  • WIA Performer → CIDOC E39 Actor
  • WIA Tradition → CIDOC E55 Type
  • WIA Community → CIDOC E74 Group
  • WIA Digital Asset → CIDOC E73 Information Object

Dublin Core 매핑

Dublin Core 15개 핵심 요소는 모두 WIA-HERITAGE-004에 대응됩니다:

Dublin CoreWIA-HERITAGE-004

dc:creatorwia:performer / wia:community
dc:subjectwia:keywords / wia:tradition_type
dc:descriptionwia:description / wia:cultural_context
dc:datewia:performance_date / wia:creation_date
dc:typewia:heritage_domain (음악, 춤, 의례 등)
dc:formatwia:media_format
dc:rightswia:access_level / wia:license

IIIF 통합

International Image Interoperability Framework와 통합하여 고해상도 이미지 및 A/V 자료를 표준 방식으로 제공합니다:

  • IIIF Image API: 고해상도 사진을 타일 방식으로 줌 가능
  • IIIF Presentation API: 멀티미디어 자료를 순서대로 프레젠테이션
  • IIIF A/V API: 비디오/오디오를 IIIF 뷰어에서 재생

Schema.org 통합

웹 검색 엔진 최적화를 위해 Schema.org 마크업 지원:

  • schema:PerformingArtsEvent
  • schema:Dance / schema:MusicEvent
  • schema:CreativeWork
  • schema:VideoObject / schema:AudioObject

이를 통해 Google, Bing 등 검색 엔진에서 풍부한 결과(Rich Snippets)로 표시됩니다.

03.5 보안 및 프라이버시 아키텍처

문화적으로 민감한 자료를 다루므로 보안은 선택이 아닌 필수입니다.

다층 보안 모델

  • 전송 계층: TLS 1.3 이상 강제, 중간자 공격 방지
  • 저장 계층: AES-256 암호화, 접근 수준별 다른 키 사용
  • 애플리케이션 계층: 역할 기반 접근 제어, 속성 기반 접근 제어(ABAC)
  • 블록체인 계층: 무결성 검증, 변조 불가능한 감사 로그

접근 제어 메커니즘

4단계 접근 권한(공개, 제한, 커뮤니티 전용, 금지)을 기술적으로 강제합니다:

  • 공개: 익명 접근 허용, CDN 캐싱
  • 제한: 등록 사용자 + 공동체 승인, 워터마크 자동 삽입
  • 커뮤니티 전용: 공동체 멤버 신원 확인, VPN 터널
  • 금지: 디지털화 자체 금지, 메타데이터만 "존재함" 표시

프라이버시 보호

  • 개인정보 최소화: 전승자가 익명을 원하면 이름 대신 가명 사용
  • 얼굴 블러링: 동의하지 않은 관객의 얼굴 자동 모자이크
  • 위치 난독화: 정확한 GPS 대신 지역 수준으로만 공개 (무단 방문 방지)
  • GDPR/CCPA 준수: 잊혀질 권리, 데이터 이동권 보장

03.6 확장성 및 성능 설계

소규모 박물관부터 국제 기구까지 다양한 규모에 대응합니다.

수평적 확장(Horizontal Scaling)

  • 무상태 서비스: 모든 API 서버는 상태를 저장하지 않아 무한 복제 가능
  • 데이터베이스 샤딩: 지역별, 전통 유형별로 데이터베이스 분산
  • CDN 활용: 정적 미디어는 CloudFlare/Akamai로 엣지 캐싱
  • 메시지 큐: RabbitMQ/Kafka로 비동기 작업 분산 처리

성능 최적화

작업최적화 전최적화 후기법

저대역폭 환경 지원

아프리카, 남미 등 네트워크 인프라가 취약한 지역을 고려:

  • 경량 모드: 이미지 품질 자동 저하, JS 번들 크기 최소화
  • 오프라인 우선: Service Worker로 오프라인 캐싱
  • 점진적 로딩: 중요한 메타데이터 먼저, 미디어는 나중에
  • 압축: Brotli 압축으로 전송량 70% 감소

03.8 장 요약

✅ 핵심 내용
  1. 5대 설계 원칙: 공동체 주권, 점진적 증강, 상호운용성 우선, 미래 증명, 문화적 맥락 중심으로 아키텍처를 설계했습니다.
  2. 4단계 아키텍처: 데이터 형식(Phase 1), API 인터페이스(Phase 2), 프로토콜(Phase 3), 통합(Phase 4)으로 점진적 채택이 가능합니다.
  3. 마이크로서비스 구조: 8개 핵심 컴포넌트(메타데이터, 미디어, 권한, 검색, 3D, AI, 블록체인, VR)가 독립적으로 작동합니다.
  4. 기존 표준 호환: CIDOC-CRM, Dublin Core, IIIF, Schema.org와 완벽하게 통합되어 기존 시스템과 상호운용됩니다.
  5. 다층 보안: 전송, 저장, 애플리케이션, 블록체인 계층에서 4단계 접근 권한을 강제합니다.
  6. 확장성 설계: 수평적 확장, 샤딩, CDN으로 글로벌 규모까지 성능을 보장합니다.

03.9 복습 문제

  1. WIA-HERITAGE-004의 5대 설계 원칙을 설명하고, 각 원칙이 무형 문화유산의 어떤 특성을 반영하는지 논하세요.
  2. 4단계 아키텍처에서 각 Phase의 핵심 기능과 의존성을 설명하세요. 왜 Phase 1만으로도 유용한지, 상위 Phase는 어떤 추가 가치를 제공하는지 구체적인 예를 들어 설명하세요.
  3. 마이크로서비스 아키텍처의 장점을 설명하고, 전통 춤 영상 업로드 시나리오를 통해 각 컴포넌트 간 데이터 흐름을 추적하세요.
  4. WIA-HERITAGE-004가 CIDOC-CRM 및 Dublin Core와 어떻게 매핑되는지 설명하고, 이러한 상호운용성이 왜 중요한지 논하세요.
  5. 4단계 접근 권한(공개, 제한, 커뮤니티 전용, 금지)을 기술적으로 어떻게 강제하는지 보안 메커니즘을 설명하세요.
  6. 저대역폭 환경에서도 WIA-HERITAGE-004가 작동할 수 있는 최적화 기법들을 나열하고, 각각이 어떻게 성능을 개선하는지 설명하세요.

03.10 다음 장 예고

제4장에서는 Phase 1 데이터 형식 계층을 심층적으로 탐구합니다. JSON-LD 메타데이터 스키마의 모든 필드, 멀티미디어 파일 형식 선택 기준, 파일 이름 규칙, 디렉토리 구조, 검증 도구 등을 상세한 코드 예제와 함께 다룹니다. 실제로 스마트폰으로 촬영한 영상을 WIA-HERITAGE-004 표준에 맞게 패키징하는 전 과정을 단계별로 안내합니다.
