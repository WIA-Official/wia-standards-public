제2장: 아키텍처와 WCAG 원칙

접근 가능한 사용자 인터페이스를 구축하기 위해서는 견고한 아키텍처 설계와 WCAG 원칙에 대한 깊은 이해가 필요합니다. 이 장에서는 접근성 중심의 시스템 아키텍처, WCAG의 네 가지 핵심 원칙의 상세 내용, 그리고 컴포넌트 기반 설계 방법을 다룹니다.

접근성 중심 아키텍처

접근성 중심 아키텍처는 접근성을 핵심 설계 원칙으로 삼아 시스템의 모든 계층에서 접근성을 보장합니다. 다음은 WIA-ACCESSIBLE_UI의 권장 아키텍처 구조입니다:

┌─────────────────────────────────────────────────────────────┐
│                    사용자 인터페이스 계층                      │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐        │
│  │ 시맨틱 HTML │  │ WAI-ARIA   │  │ 포커스 관리  │        │
│  └─────────────┘  └─────────────┘  └─────────────┘        │
├─────────────────────────────────────────────────────────────┤
│                    컴포넌트 계층                             │

│  │ 버튼/링크   │  │ 폼 컨트롤   │  │ 모달/대화상자│        │

WCAG 원칙 1: 인식의 용이성 (Perceivable)

정보와 사용자 인터페이스 구성요소는 사용자가 인식할 수 있는 방식으로 제공되어야 합니다. 이 원칙은 사용자가 제시된 정보를 인지할 수 있어야 함을 의미합니다.

1.1 텍스트 대체 수단

모든 비텍스트 콘텐츠에는 동등한 목적을 제공하는 텍스트 대체 수단이 제공되어야 합니다:

<!-- 정보 전달 이미지 -->
<img src="chart.png"
     alt="2024년 분기별 매출 추이: 1분기 120억, 2분기 135억, 3분기 145억, 4분기 160억원">

<!-- 기능적 이미지 (버튼 내 아이콘) -->
<button type="button" aria-label="검색">
    <img src="search-icon.svg" alt="" role="presentation">
</button>

<!-- 장식용 이미지 -->
<img src="decorative-line.png" alt="" role="presentation">

<!-- 복잡한 이미지 (상세 설명 필요) -->
<figure>
    <img src="system-architecture.png"
         alt="시스템 아키텍처 다이어그램"
         aria-describedby="arch-desc">
    <figcaption id="arch-desc">
        시스템은 3개 계층으로 구성됩니다: 프레젠테이션 계층은
        React 기반 SPA로 구현되고, 비즈니스 로직 계층은
        Node.js Express 서버가 담당하며, 데이터 계층은
        PostgreSQL과 Redis를 사용합니다.
    </figcaption>
</figure>

1.2 시간 기반 미디어

사전 녹음된 오디오 및 비디오 콘텐츠에 대해 대체 수단을 제공해야 합니다:

  • 자막: 동기화된 텍스트로 대화 및 중요한 소리 정보 제공
  • 오디오 설명: 시각적 정보를 음성으로 설명
  • 대본: 전체 콘텐츠의 텍스트 버전 제공

1.3 적응 가능

콘텐츠는 정보나 구조를 잃지 않고 다양한 방식으로 제시될 수 있어야 합니다:

<!-- 시맨틱 구조 사용 -->
<article>
    <header>
        <h1>기사 제목</h1>
        <time datetime="2024-12-15">2024년 12월 15일</time>
    </header>

    <section>
        <h2>첫 번째 섹션</h2>
        <p>섹션 내용...</p>
    </section>

    <aside>
        <h3>관련 정보</h3>
        <p>부가 정보...</p>
    </aside>
</article>

<!-- 테이블 헤더 연결 -->
<table>
    <caption>월별 판매 실적</caption>
    <thead>
        <tr>
            <th scope="col">제품명</th>
            <th scope="col">1월</th>
            <th scope="col">2월</th>
        </tr>
    </thead>
    <tbody>
        <tr>
            <th scope="row">제품 A</th>
            <td>100</td>
            <td>120</td>
        </tr>
    </tbody>
</table>

WCAG 원칙 2: 운용의 용이성 (Operable)

사용자 인터페이스 구성요소와 탐색은 운용 가능해야 합니다. 사용자가 인터페이스를 조작할 수 있어야 합니다.

2.1 키보드 접근성

모든 기능은 키보드만으로 사용할 수 있어야 합니다:

// 드롭다운 키보드 이벤트 핵심 처리
menu.addEventListener('keydown', (e) => {
    switch (e.key) {
        case 'Escape':
            closeMenu();
            triggerButton.focus();
            break;
        case 'ArrowDown':
            e.preventDefault();
            focusNextItem();
            break;
        case 'ArrowUp':
            e.preventDefault();
            focusPrevItem();
            break;
    }
});

2.2 충분한 시간

사용자가 콘텐츠를 읽고 사용하는 데 충분한 시간을 제공해야 합니다. 시간 제한이 있는 경우 연장, 조정, 해제 옵션을 제공합니다.

2.3 발작 및 신체 반응

초당 3회 이상 깜빡이는 콘텐츠를 포함하지 않아야 합니다.

WCAG 원칙 3: 이해의 용이성 (Understandable)

정보와 사용자 인터페이스 운용은 이해할 수 있어야 합니다. 콘텐츠는 읽기 쉽고 예측 가능하게 작동해야 합니다.

3.1 가독성

텍스트 콘텐츠는 읽기 쉽고 이해하기 쉬워야 합니다:

  • 페이지 언어 명시: <html lang="ko">
  • 부분 언어 변경 표시: <span lang="en">Accessibility</span>
  • 약어 설명: <abbr title="World Wide Web">WWW</abbr>

3.2 예측 가능성

웹 페이지는 예측 가능한 방식으로 나타나고 운용되어야 합니다:

원칙: 포커스를 받거나 입력이 변경될 때 사용자가 예상하지 못한 컨텍스트 변경이 발생하면 안 됩니다. 폼 제출은 버튼 클릭 등 명시적인 사용자 행동으로만 이루어져야 합니다.

3.3 입력 지원

사용자가 실수를 방지하고 수정할 수 있도록 돕습니다:

<!-- 오류 식별 및 제안 -->
<label for="email">이메일 <span aria-hidden="true">*</span></label>
<input type="email" id="email"
       aria-required="true"
       aria-invalid="true"
       aria-describedby="email-error">
<span id="email-error" role="alert">
    올바른 이메일 형식을 입력해주세요.
</span>

WCAG 원칙 4: 견고성 (Robust)

콘텐츠는 다양한 사용자 에이전트(브라우저, 보조기술 포함)에서 안정적으로 해석될 수 있도록 충분히 견고해야 합니다.

4.1 호환성

현재 및 미래의 사용자 에이전트와의 호환성을 극대화합니다:

  • 유효한 HTML 마크업 사용
  • 고유한 ID 값 보장
  • 사용자 정의 컨트롤에 적절한 이름, 역할, 값 제공

접근성 컴포넌트 설계

다음은 접근 가능한 컴포넌트를 설계할 때 고려해야 할 사항입니다:

버튼 컴포넌트

  • 명확한 레이블 (텍스트 또는 aria-label)
  • 키보드 포커스 가능 (tabindex)
  • 상태 표시 (aria-pressed, aria-expanded)
  • 비활성화 상태 표시 (aria-disabled)

폼 컴포넌트

  • 레이블과 입력 필드 연결
  • 필수 입력 표시 (aria-required)
  • 오류 상태 표시 (aria-invalid)
  • 도움말 및 오류 메시지 연결 (aria-describedby)

모달 컴포넌트

  • 포커스 트랩 구현
  • ESC 키로 닫기
  • 열림 상태 알림 (aria-modal, role="dialog")
  • 닫힐 때 원래 요소로 포커스 복귀
모범 사례: 컴포넌트 라이브러리를 구축할 때 접근성 테스트를 자동화하고, 모든 컴포넌트가 WCAG 2.1 AA 수준을 충족하도록 보장하세요.

요약

이 장에서는 접근성 중심 아키텍처의 구조와 WCAG의 네 가지 핵심 원칙(POUR)을 상세히 살펴보았습니다. 인식의 용이성, 운용의 용이성, 이해의 용이성, 견고성의 원칙을 이해하고 적용함으로써 모든 사용자가 이용할 수 있는 인터페이스를 구축할 수 있습니다. 다음 장에서는 시각적 접근성의 구체적인 구현 방법을 다룹니다.
