제7장: 테스트와 모니터링

접근성은 한 번 구현하면 끝나는 것이 아니라 지속적인 테스트와 모니터링이 필요합니다. 이 장에서는 자동화 테스트 도구, 수동 테스트 방법, CI/CD 통합, 접근성 모니터링, 그리고 성능 지표에 대해 다룹니다. 체계적인 테스트 전략을 통해 접근성 품질을 유지하고 개선할 수 있습니다.

테스트 전략 개요

효과적인 접근성 테스트는 자동화 테스트와 수동 테스트를 결합합니다. 자동화 테스트는 전체 WCAG 위반의 약 30-40%만 감지할 수 있으므로, 수동 테스트와 사용자 테스트가 필수적입니다.

테스트 피라미드:
  • 자동화 테스트 (기반): CI/CD에 통합된 린터, 유닛 테스트
  • 수동 테스트 (중간): 키보드 테스트, 화면 판독기 테스트
  • 사용자 테스트 (정상): 실제 장애인 사용자 참여 테스트

자동화 테스트 도구

axe-core

Deque Systems가 개발한 오픈소스 접근성 테스트 엔진입니다. 브라우저 확장, CLI, 테스트 프레임워크 통합 등 다양한 방식으로 사용할 수 있습니다.

  • WCAG 2.1 A/AA 규칙 포함
  • 오탐(false positive) 비율이 낮음
  • Jest, Cypress, Playwright 등과 통합 가능
// axe-core를 Jest와 함께 사용
// 설치: npm install --save-dev @axe-core/react jest-axe

import { render } from '@testing-library/react';
import { axe, toHaveNoViolations } from 'jest-axe';
import MyComponent from './MyComponent';

expect.extend(toHaveNoViolations);

describe('MyComponent 접근성', () => {
    it('접근성 위반이 없어야 한다', async () => {
        const { container } = render(<MyComponent />);
        const results = await axe(container);
        expect(results).toHaveNoViolations();
    });

    it('특정 규칙만 검사', async () => {
        const { container } = render(<MyComponent />);
        const results = await axe(container, {
            runOnly: {
                type: 'tag',
                values: ['wcag2a', 'wcag2aa']
            }
        });
        expect(results).toHaveNoViolations();
    });
});

// Cypress와 함께 사용
// cypress/support/commands.js
import 'cypress-axe';

// cypress/e2e/accessibility.cy.js
describe('접근성 테스트', () => {
    beforeEach(() => {
        cy.visit('/');
        cy.injectAxe();
    });

    it('홈페이지에 접근성 위반이 없어야 한다', () => {
        cy.checkA11y();
    });

    it('특정 영역만 검사', () => {
        cy.checkA11y('main', {
            rules: {
                'color-contrast': { enabled: true },
                'valid-lang': { enabled: true }
            }
        });
    });

    it('위반 시 상세 정보 출력', () => {
        cy.checkA11y(null, null, (violations) => {
            violations.forEach(violation => {
                cy.log(`${violation.id}: ${violation.description}`);
                violation.nodes.forEach(node => {
                    cy.log(`  - ${node.html}`);
                });
            });
        });
    });
});

Lighthouse

Google Chrome에 내장된 감사 도구로, 접근성 점수와 개선 권고사항을 제공합니다. CI/CD에서 CLI로 실행할 수 있습니다.

// Lighthouse CI 설정
// lighthouserc.js
module.exports = {
    ci: {
        collect: {
            url: ['http://localhost:3000/', 'http://localhost:3000/about'],
            numberOfRuns: 3
        },
        assert: {
            assertions: {
                'categories:accessibility': ['error', { minScore: 0.9 }],
                'aria-allowed-attr': 'error',
                'aria-hidden-body': 'error',
                'aria-hidden-focus': 'error',
                'button-name': 'error',
                'color-contrast': 'warn',
                'document-title': 'error',
                'html-has-lang': 'error',
                'image-alt': 'error',
                'label': 'error',
                'link-name': 'error',
                'list': 'error',
                'meta-viewport': 'error'
            }
        },
        upload: {
            target: 'temporary-public-storage'
        }
    }
};

// package.json scripts
// "lhci": "lhci autorun"
// CI에서 실행: npm run lhci

Pa11y

명령줄 접근성 테스트 도구로, HTML CodeSniffer 또는 axe를 사용합니다. CI/CD 파이프라인에 쉽게 통합됩니다.

// Pa11y 설정 파일 (.pa11yci.json)
{
    "defaults": {
        "standard": "WCAG2AA",
        "runners": ["axe"],
        "timeout": 30000
    },
    "urls": ["http://localhost:3000/"]
}

// 실행: npx pa11y-ci

수동 테스트 체크리스트

자동화 도구로 감지할 수 없는 문제를 찾기 위한 수동 테스트 항목입니다:

테스트 영역 확인 항목
키보드 탐색 Tab 키로 모든 대화형 요소에 도달 가능한가?
포커스 표시 현재 포커스된 요소가 시각적으로 명확한가?
포커스 순서 탭 순서가 논리적인 읽기 순서를 따르는가?
포커스 관리 모달 열림/닫힘 시 포커스가 적절히 이동하는가?
확대/축소 200% 확대 시에도 콘텐츠가 잘리지 않는가?
색상 대비 텍스트와 배경의 대비가 충분한가?
색상 의존 색상 외의 방법으로도 정보가 전달되는가?
시맨틱 제목 계층이 논리적인가? (h1 > h2 > h3)

CI/CD 통합

# GitHub Actions 접근성 테스트 워크플로우
# .github/workflows/accessibility.yml

name: Accessibility Tests

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  accessibility:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Build application
        run: npm run build

      - name: Start server
        run: npm start &
        env:
          CI: true

      - name: Wait for server
        run: npx wait-on http://localhost:3000

      - name: Run axe tests
        run: npm run test:a11y

      - name: Run Lighthouse CI
        run: |
          npm install -g @lhci/cli
          lhci autorun

      - name: Run Pa11y
        run: npx pa11y-ci

      - name: Upload results
        uses: actions/upload-artifact@v4
        if: always()
        with:
          name: accessibility-results
          path: |
            .lighthouseci/
            pa11y-results/

접근성 모니터링

프로덕션 환경에서 지속적으로 접근성을 모니터링하여 회귀를 방지합니다:

주요 모니터링 지표

  • WCAG 위반 수 (심각도별)
  • Lighthouse 접근성 점수
  • 페이지별 접근성 문제 분포
  • 시간에 따른 접근성 점수 추이
// 접근성 모니터링 대시보드 데이터 수집
class AccessibilityMonitor {
    constructor() {
        this.results = [];
    }

    async runAudit(url) {
        const axeResults = await this.runAxe(url);
        const lighthouseResults = await this.runLighthouse(url);

        const report = {
            url,
            timestamp: new Date().toISOString(),
            axe: {
                violations: axeResults.violations.length,
                passes: axeResults.passes.length,
                incomplete: axeResults.incomplete.length,
                bySeverity: this.countBySeverity(axeResults.violations)
            },
            lighthouse: {
                score: lighthouseResults.score,
                audits: lighthouseResults.audits
            }
        };

        this.results.push(report);
        await this.saveReport(report);
        await this.checkThresholds(report);

        return report;
    }

    countBySeverity(violations) {
        return violations.reduce((acc, v) => {
            acc[v.impact] = (acc[v.impact] || 0) + 1;
            return acc;
        }, {});
    }

    async checkThresholds(report) {
        const thresholds = {
            minLighthouseScore: 90,
            maxCriticalViolations: 0,
            maxSeriousViolations: 3
        };

        const alerts = [];

        if (report.lighthouse.score < thresholds.minLighthouseScore) {
            alerts.push(`Lighthouse 점수가 ${thresholds.minLighthouseScore} 미만입니다.`);
        }

        if (report.axe.bySeverity.critical > thresholds.maxCriticalViolations) {
            alerts.push(`심각한 위반이 발견되었습니다.`);
        }

        if (alerts.length > 0) {
            await this.sendAlerts(alerts);
        }
    }

    async sendAlerts(alerts) {
        // Slack, 이메일 등으로 알림 전송
        console.log('접근성 알림:', alerts);
    }
}

성능과 접근성 균형

성능 최적화 시 접근성 고려:
  • 이미지 지연 로딩 시 대체 텍스트 유지
  • 코드 분할 시 중요한 접근성 기능 포함
  • CSS/JS 지연 로딩 시 기본 스타일 유지
  • 서버 사이드 렌더링으로 초기 접근성 보장

접근성 보고서 작성

정기적인 접근성 보고서를 통해 개선 상황을 추적하고 이해관계자와 공유합니다:

보고서 포함 항목:
  1. 테스트 범위 (페이지 수, 기능 수)
  2. 발견된 문제 요약 (심각도별 분류)
  3. WCAG 적합성 수준 평가
  4. 이전 보고서 대비 개선/악화 사항
  5. 우선순위별 개선 권고사항
  6. 예상 수정 일정
