Chapter 02

Architecture and WCAG Principles

2.1 The POUR Framework

The POUR principles form the foundation of WCAG and accessible design. Every success criterion traces back to these four principles.

P - Perceivable

Content must be presentable to users in ways they can perceive. Covers text alternatives, captions, and distinguishable content.

O - Operable

Interface components must be operable. Covers keyboard accessibility, sufficient time, and navigability.

U - Understandable

Information and operation must be understandable. Includes readable text and predictable behavior.

R - Robust

Content must be robust enough for diverse user agents including assistive technologies.

2.2 Conformance Levels

Level A Minimum accessibility - addresses severe barriers.

Level AA Required by most legislation - reasonable balance of impact and effort.

2.3 Semantic HTML Foundation

First Rule of ARIA

Don't use ARIA unless necessary. Native HTML elements provide built-in semantics, keyboard support, and AT compatibility.

// WRONG: ARIA-heavy approach
<div role="button" tabindex="0" onclick="toggle()">
  Toggle Feature
</div>

// CORRECT: Native HTML element
<button type="button" onclick="toggle()">
  Toggle Feature
</button>

// Semantic Element Guide
const semanticElements = {
  navigation: '<nav> with aria-label',
  mainContent: '<main> for primary content',
  sections: '<section> with accessible name',
  headings: '<h1>-<h6> in logical hierarchy',
  forms: '<form> with <label> elements'
};

2.4 Document Structure

Screen reader users navigate by landmarks and headings. Proper document structure enables efficient navigation.

<body>
  <a href="#main" class="skip-link">Skip to main</a>

  <header>
    <nav aria-label="Primary">...</nav>
  </header>

  <main id="main">
    <h1>Page Heading</h1>
    <section aria-labelledby="s1">
      <h2 id="s1">Section Title</h2>
    </section>
  </main>

  <aside aria-label="Related">...</aside>
</body>

2.5 Heading Hierarchy

  • One H1 per page describing the page's primary purpose
  • No skipped levels - progress from H1 to H2 to H3
  • Descriptive content - headings should be meaningful out of context

2.7 Component Patterns

The WAI-ARIA Authoring Practices Guide (APG) provides established patterns for common interactive components. These patterns specify expected keyboard behavior, ARIA roles and properties, and focus management for menus, tabs, dialogs, and more.

Key Component Patterns
  • Modal Dialog: Traps focus, closes on Escape, returns focus to trigger
  • Tabs: Arrow keys navigate between tabs, Tab key moves to panel
  • Menu: Arrow keys navigate items, Enter activates, Escape closes
  • Accordion: Space/Enter toggles panels, focus remains on headers

2.8 Summary

Accessible architecture begins with understanding POUR principles and WCAG conformance levels. By prioritizing semantic HTML, proper document structure, and logical heading hierarchy, you create a solid foundation for accessible interfaces. ARIA should enhance, not replace, native HTML semantics. These architectural decisions form the backbone of all accessible implementations covered in subsequent chapters.
