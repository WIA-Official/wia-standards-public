Chapter 06

Assistive Technology Integration

6.1 Understanding Assistive Technologies

Assistive technologies (AT) are hardware and software that help people with disabilities interact with computers. Building accessible interfaces requires understanding how these technologies work.

TechnologyUsersHow It Works
Screen ReadersBlind, low visionConverts content to speech/braille
MagnifiersLow visionEnlarges portions of screen
Voice ControlMotor impairmentsControl through spoken commands
Switch DevicesSevere motor impairmentsSingle buttons to navigate

6.2 The Accessibility Tree

Screen readers build a virtual representation of the page called the accessibility tree. It contains roles, names, states, and relationships.

// DOM Element
<button aria-pressed="false" disabled>
  Submit Order
</button>

// Resulting Accessibility Tree Node
{
  role: "button",
  name: "Submit Order",
  states: {
    pressed: false,
    disabled: true,
    focusable: false
  }
}

// Screen Reader Announcement:
// "Submit Order button, disabled"

6.3 ARIA Deep Dive

The Five Rules of ARIA
  1. Don't use ARIA if native HTML works
  2. Don't change native semantics
  3. All interactive elements must be keyboard accessible
  4. Don't hide focusable elements with aria-hidden
  5. All interactive elements must have accessible names

Accessible Names

// Priority order for accessible name:

// 1. aria-labelledby
<span id="label">Name</span>
<input aria-labelledby="label" />

// 2. aria-label
<button aria-label="Close dialog">X</button>

// 3. Native labeling
<label for="email">Email</label>
<input id="email" />

// 4. Content (buttons, links)
<button>Save Changes</button>

6.4 Live Regions

// Announce dynamic content changes

// Polite (wait for screen reader to finish)
<div role="status" aria-live="polite">
  Form saved successfully
</div>

// Assertive (interrupt current speech)
<div role="alert" aria-live="assertive">
  Error: Please fix highlighted fields
</div>

// JavaScript announcer utility
class Announcer {
  constructor() {
    this.region = document.createElement('div');
    this.region.setAttribute('aria-live', 'polite');
    this.region.className = 'visually-hidden';
    document.body.appendChild(this.region);
  }

  announce(message) {
    this.region.textContent = '';
    setTimeout(() => {
      this.region.textContent = message;
    }, 100);
  }
}

6.6 Platform Accessibility APIs

Browsers expose accessibility information through platform-specific APIs that assistive technologies consume. Windows uses UI Automation (UIA), macOS and iOS use NSAccessibility, Linux uses ATK/AT-SPI, and Android uses AccessibilityNodeInfo. Understanding these APIs helps debug accessibility issues.

6.7 Browser Developer Tools

Modern browsers provide accessibility inspection tools. Chrome DevTools includes an Accessibility pane showing the accessibility tree, computed properties, and potential issues. Lighthouse provides automated accessibility audits. Firefox and Safari offer similar capabilities.

6.8 Summary

Integrating with assistive technologies requires understanding how they consume accessibility information and what users expect. By properly implementing semantic HTML, ARIA, live regions, and testing with actual screen readers, you build interfaces that work seamlessly for all users regardless of their assistive technology.
