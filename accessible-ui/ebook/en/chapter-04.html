Chapter 04

Interaction Patterns

4.1 Keyboard Accessibility

Keyboard accessibility is critical for users who cannot use a mouse due to motor or visual impairments. WCAG 2.1.1 requires all functionality be operable through keyboard.

Core Requirements
  • All interactive elements must be focusable
  • Focus order must be logical and predictable
  • Focus must be visible at all times
  • No keyboard traps - users can always Tab away

4.2 Focus Management

// Focus Management Utilities
class FocusManager {
  static focusableSelector =
    'a[href], button:not([disabled]), input:not([disabled]), ' +
    'select:not([disabled]), textarea:not([disabled]), ' +
    '[tabindex]:not([tabindex="-1"])';

  static getFocusableElements(container) {
    return Array.from(
      container.querySelectorAll(this.focusableSelector)
    ).filter(el => el.offsetParent !== null);
  }

  static createFocusTrap(container) {
    const focusable = this.getFocusableElements(container);
    const first = focusable[0];
    const last = focusable[focusable.length - 1];

    container.addEventListener('keydown', (e) => {
      if (e.key !== 'Tab') return;

      if (e.shiftKey && document.activeElement === first) {
        e.preventDefault();
        last.focus();
      } else if (!e.shiftKey && document.activeElement === last) {
        e.preventDefault();
        first.focus();
      }
    });
  }
}

  border-radius: 4px;

4.5 Touch Accessibility

WCAG 2.2 requires touch targets to be at least 24x24 CSS pixels at Level AA. Recommend 44x44 pixels for primary actions.

/* Touch Target Sizing */
.button {
  min-height: 44px;
  min-width: 44px;
  padding: 12px 24px;
  margin: 8px; /* Spacing between targets */
}

/* Icon button with expanded touch area */
.icon-button {
  position: relative;
  width: 24px;
  height: 24px;
}
.icon-button::before {
  content: '';
  position: absolute;
  top: 50%;
  left: 50%;
  transform: translate(-50%, -50%);
  width: 44px;
  height: 44px;
}

4.6 Motion and Animation

/* Respect reduced motion preference */
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}

// JavaScript check
const prefersReducedMotion =
  window.matchMedia('(prefers-reduced-motion: reduce)').matches;

4.7 Voice Control Considerations

Voice control software like Dragon NaturallySpeaking, Voice Control on macOS/iOS, and Voice Access on Android enable users with motor impairments to interact through spoken commands. Ensure visible labels match accessible names for reliable voice activation.

Voice Control Best Practices
  • Visible button text should match the accessible name
  • Use unique, descriptive link text instead of "click here"
  • Provide text alternatives for icon-only buttons
  • Test interfaces with voice control software

4.8 Summary

Accessible interaction patterns ensure all users can operate your interface regardless of input method. Implementing proper keyboard support, visible focus indicators, adequate touch targets, and respecting motion preferences creates inclusive interactive experiences for the widest possible audience.
