Chapter 05

Security and Authentication

5.1 The Security-Accessibility Balance

Security mechanisms often create barriers for users with disabilities. CAPTCHAs, time-limited sessions, and complex passwords can exclude users who cannot perceive challenges or respond within time limits. WCAG 2.2 Success Criterion 3.3.8 addresses authentication accessibility.

Key Principle

Security and accessibility are not opposing forces. Both serve the goal of enabling users to safely and effectively use digital products.

5.2 CAPTCHA Accessibility

CAPTCHA Issues
  • Visual CAPTCHAs inaccessible to blind users
  • Audio CAPTCHAs inaccessible to deaf users
  • Both difficult for cognitive disabilities
// Accessible CAPTCHA Strategy
const captchaStrategy = {
  // Level 1: Invisible protection (preferred)
  invisible: {
    method: 'Behavioral analysis + honeypot + rate limiting',
    implementation: `
      <!-- Honeypot field -->
      <div class="visually-hidden" aria-hidden="true">
        <input type="text" name="website" tabindex="-1" />
      </div>
    `
  },

  // Level 2: Low-friction challenge
  lowFriction: `
    <input type="checkbox" id="human" required />
    <label for="human">I am a human</label>
  `,

  // Level 3: Simple logic question (last resort)
  logicQuestion: 'What is 3 + 4?'
};

5.3 Accessible Password Fields

<div class="form-field">
  <label for="password">Password</label>

  <div class="password-wrapper">
    <input
      type="password"
      id="password"
      autocomplete="current-password"
      aria-describedby="pw-requirements"
    />

    <button
      type="button"
      aria-label="Show password"
      aria-pressed="false"
      onclick="togglePassword(this)"
    >
      <svg aria-hidden="true">...</svg>
    </button>
  </div>

  <div id="pw-requirements">
    At least 8 characters
  </div>
</div>

5.4 Two-Factor Authentication

  • SMS codes: Extend expiration time for users who need more time
  • TOTP apps: 30-second codes can be challenging; offer alternatives
  • Phone calls: Provide text alternatives for deaf users
  • Passkeys: Increasingly accessible option using biometrics
<!-- Accessible OTP Input -->
<fieldset>
  <legend>Enter verification code</legend>
  <p id="otp-help">
    Code expires in <span aria-live="polite">10 minutes</span>
  </p>

  <input
    type="text"
    inputmode="numeric"
    pattern="[0-9]{6}"
    autocomplete="one-time-code"
    aria-describedby="otp-help"
  />

  <button type="button">Resend code</button>
</fieldset>

5.5 Session Timeouts

WCAG 2.2.1 requires users can turn off, adjust, or extend time limits.

<dialog id="session-warning" aria-labelledby="sw-title">
  <h2 id="sw-title">Session Expiring</h2>
  <p>Your session expires in 2 minutes.</p>

  <button onclick="extendSession()">
    Extend Session
  </button>
  <button onclick="logout()">
    Log Out
  </button>
</dialog>
Error Message Best Practices
  • Use clear, plain language that describes the problem
  • Provide specific guidance to resolve issues
  • Don't reveal whether username or password was incorrect
  • Announce errors via aria-live regions to screen readers
  • Move focus to error messages or first invalid field

5.6 Passkeys and Modern Authentication

Passkeys represent a significant accessibility improvement over traditional passwords. They eliminate the need to remember complex passwords and leverage biometric authentication built into devices, benefiting users with cognitive disabilities who struggle with password management.

5.7 Summary

Balancing security and accessibility requires thoughtful design that considers diverse user needs while maintaining robust protection. By implementing invisible protection layers, providing accessible authentication alternatives, managing sessions considerately, and handling errors clearly, you create secure experiences that work for everyone.
