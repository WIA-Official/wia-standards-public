Chapter 04: Phase 1 — Data Format

4.1 Phase 1 in one paragraph

Phase 1 specifies JSON envelopes for transportation network data. Every envelope is signed with Ed25519 (IETF RFC 8032).

4.2 Route envelope

{
  "wia_transportation_network_version": "1.0.0",
  "type": "route",
  "route_id": "rt_mta_M15-SBS",
  "agency_id": "did:wia:agency:mta-nyc",
  "short_name": "M15-SBS",
  "long_name": "First/Second Avs Select Bus Service",
  "mode": "bus",
  "color": "#EE352E",
  "stops": ["stop_M15_001", "stop_M15_002"],
  "signature": { "alg": "Ed25519", "value": "..." }
}

4.3 Schedule + trip update + vehicle position envelopes

Standard GTFS / GTFS-Realtime structure: schedule defines planned trips, trip_update predicts arrivals and delays, vehicle_position reports real-time GPS + occupancy. The standard preserves all three fields with WIA signature wrappers.

4.4 Service alert envelope

Disruption announcement. Carries affected_routes, severity (info/warning/critical), cause (technical/weather/security/strike/maintenance), effect (no_service/reduced_service/significant_delays/detour/additional_service), and a rider-facing description in multiple languages.

4.5 Fare product envelope

Fare structure for a route or system. Distance-based, time-based, zone-based, or flat-rate. Includes transfer rules and discount eligibility (senior, student, low-income).

4.6 Shared vehicle envelope

Micromobility availability. Required: vehicle_id, vehicle_type (bike, e-bike, scooter, e-moped), location, battery_pct (for electric), is_reserved, is_disabled, last_reported.

4.7 Rider trip envelope

Completed trip record with explicit consent envelope reference. The standard does not allow rider trip envelopes to flow without an active consent; downstream consumers MUST refuse rider_trip envelopes whose consent is missing or revoked.

Implementer Notes

Three operational concerns recur in transportation network infrastructure and are baked into the standard. First, vehicle_position update frequency: most US transit agencies emit every 15-30 seconds; high-frequency rail systems emit every second. Hosts MUST provision sufficient throughput. Second, rider trip consent revocation: when a rider revokes consent, the data must stop flowing within minutes; the consent envelope MUST carry an explicit revocation timestamp. Third, GTFS feed updates: agencies typically publish weekly GTFS updates; the standard\x27s schedule envelope versioning lets downstream consumers detect new feeds without polling.

Multi-Modal Trip Planning Pattern

The standard's core use case is multi-modal trip planning: a traveller wants to go from point A to point B and the planner needs to combine bus, rail, micromobility, and walking. The standard makes this possible without per-vendor integration: the traveller's app subscribes to one WIA-UNI-008 host that aggregates feeds from all participating providers in the city. Trip search returns multi-modal options ranked by total time, total cost, or carbon footprint. The traveller selects an option and the app initiates rider_trip envelopes with explicit consent for the trip duration only.
