Chapter 05: Phase 2 — API Interface

5.1 Phase 2 in one paragraph

Phase 2 specifies the HTTPS surface for transportation network data. Transport is TLS 1.3 (IETF RFC 8446) only.

5.2 Discovery

Every host publishes a discovery document at /.well-known/wia-transportation-network.

5.3 Route + schedule endpoints

POST /tn/route                   (signed; agency authority)
GET  /tn/route/{route_id}
GET  /tn/route?agency_id={id}&mode={m}
POST /tn/schedule                (signed; agency authority)
GET  /tn/schedule/{schedule_id}

5.4 Real-time endpoints

POST /tn/vehicle-position        (signed; agency authority; high frequency)
GET  /tn/vehicle-position/stream?agency_id={id}    (Server-Sent Events)
POST /tn/trip-update             (signed; agency authority)
GET  /tn/trip-update/stream?route_id={id}    (Server-Sent Events)

5.5 Service alert + fare endpoints

POST /tn/service-alert           (signed; agency authority)
GET  /tn/service-alert?affected_route={id}
POST /tn/fare-product            (signed; agency authority)
GET  /tn/fare-product?agency_id={id}

5.6 Shared vehicle endpoints

POST /tn/shared-vehicle          (signed; operator authority)
GET  /tn/shared-vehicle/stream?bbox={...}    (Server-Sent Events)

5.7 Rider trip endpoints

POST /tn/rider-trip              (signed; rider consent envelope required)
GET  /tn/rider-trip?rider_id={id}    (rider only; full audit log)

5.8 Authentication

Writes require HTTP Message Signatures per IETF RFC 9421. Public reads (route, schedule, real-time, service alerts, fare products, shared vehicle availability) are anonymous; rider trip reads require rider authentication or explicit research consent envelope.

Implementer Notes

Three operational concerns recur in transportation network infrastructure and are baked into the standard. First, vehicle_position update frequency: most US transit agencies emit every 15-30 seconds; high-frequency rail systems emit every second. Hosts MUST provision sufficient throughput. Second, rider trip consent revocation: when a rider revokes consent, the data must stop flowing within minutes; the consent envelope MUST carry an explicit revocation timestamp. Third, GTFS feed updates: agencies typically publish weekly GTFS updates; the standard\x27s schedule envelope versioning lets downstream consumers detect new feeds without polling.

Multi-Modal Trip Planning Pattern

The standard's core use case is multi-modal trip planning: a traveller wants to go from point A to point B and the planner needs to combine bus, rail, micromobility, and walking. The standard makes this possible without per-vendor integration: the traveller's app subscribes to one WIA-UNI-008 host that aggregates feeds from all participating providers in the city. Trip search returns multi-modal options ranked by total time, total cost, or carbon footprint. The traveller selects an option and the app initiates rider_trip envelopes with explicit consent for the trip duration only.
