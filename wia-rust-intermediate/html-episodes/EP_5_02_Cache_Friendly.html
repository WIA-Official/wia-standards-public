struct Particle {
x: f32, y: f32, z: f32,
vx: f32, vy: f32, vz: f32,
}
let particles: Vec<Particle> = vec![...];

EP 5-02

성능 비교 Performance Comparison

AoS (Array of Structs)

for p in &mut particles {
p.x += p.vx;
}
🐌 느림

각 입자마다 전체 구조체 로드
캐시 라인 낭비

Loads entire struct per particle
Wastes cache lines

SoA (Struct of Arrays)

for i in 0..n {
particles.x[i] += particles.vx[i];
}
⚡ 빠름

연속된 데이터 접근
캐시 히트율 극대화

Sequential data access
Maximizes cache hits

데이터 정렬 Data Alignment

// 캐시 라인 크기에 맞춰 정렬 (보통 64 bytes)
#[repr(align(64))]
struct CacheAligned {
data: [u8; 64],
}
// False sharing 방지
#[repr(align(64))]
struct Counter {
value: AtomicU64,
_padding: [u8; 56], // 64 - 8
}
📏 align(64) - 캐시 라인 정렬

🚫 False Sharing 방지
다른 스레드가 같은 캐시 라인 수정 시
성능 저하 발생

✨ 패딩으로 해결 📏 align(64) - Cache line aligned

🚫 Prevent False Sharing
Performance degrades when
threads modify same cache line

✨ Solve with padding

프리페칭 활용 Utilizing Prefetching

use std::intrinsics::prefetch_read_data;
fn process_data(data: &[u64]) {
for i in 0..data.len() {
// 다음 데이터를 미리 캐시에 로드
if i + 8 < data.len() {
unsafe {
prefetch_read_data(
&data[i + 8], 3 // locality
);
}
}
// 현재 데이터 처리
process(data[i]);
}
}

실전 팁 Practical Tips

캐시 친화적 설계 원칙

1️⃣ 순차 접근 - 배열은 Vec, 링크드 리스트 피하기

2️⃣ 데이터 국소성 - 자주 사용하는 데이터를 가까이

3️⃣ 정렬 활용 - 캐시 라인 크기(64B) 고려

4️⃣ False Sharing 방지 - 스레드별 데이터 분리

5️⃣ 벤치마크 필수 - criterion으로 측정

Cache-Friendly Design Principles

1️⃣ Sequential Access - Use Vec, avoid linked lists

2️⃣ Data Locality - Keep frequently used data close

3️⃣ Use Alignment - Consider cache line size (64B)

4️⃣ Prevent False Sharing - Separate thread data

5️⃣ Always Benchmark - Measure with criterion
