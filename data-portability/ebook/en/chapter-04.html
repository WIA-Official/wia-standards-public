Chapter 4: Phase 1 - Universal Data Format

Chapter Overview: Deep dive into Phase 1 specifications including JSON schemas, data package structure, metadata requirements, GDPR Article 20 compliance, and validation rules.

4.1 Package Structure Specification

Every WIA-LEG-008 data package follows a standardized directory structure ensuring consistency across all implementations:

data-package-v2.0-{userId}-{timestamp}/
├── manifest.json                 # Required: Package metadata
├── profile.json                  # Required: User profile
├── activity/
│   ├── posts.json               # User-generated posts
│   ├── comments.json            # Comments on others' content
│   ├── reactions.json           # Likes, votes, favorites
│   └── shares.json              # Retweets, reposts
├── relationships/
│   ├── friends.json             # Bidirectional connections
│   ├── following.json           # Asymmetric following
│   ├── followers.json           # Asymmetric followers
│   ├── blocks.json              # Blocked accounts
│   └── groups.json              # Group memberships
├── preferences/
│   ├── settings.json            # User preferences
│   ├── privacy.json             # Privacy controls
│   └── notifications.json       # Notification settings
├── media/
│   ├── photos/                  # Image files
│   ├── videos/                  # Video files
│   ├── audio/                   # Audio files
│   └── documents/               # PDFs, files
├── messages/
│   ├── conversations.json       # Private messages metadata
│   └── threads/                 # Individual conversation files
├── commerce/
│   ├── purchases.json           # Purchase history
│   ├── reviews.json             # Product reviews
│   └── wishlist.json            # Saved items
└── verification/
    ├── checksums.json           # SHA-256 hashes
    ├── signature.json           # Digital signature
    └── timestamp.json           # RFC 3161 timestamp

4.5 Relationship Schema

Social connections standardized for cross-platform portability:

Field Type Description
relationshipId string Unique identifier for the relationship
type enum friend, following, follower, blocked, muted
targetUserId string ID of the other user
targetUsername string Username of the other user
established datetime When relationship was created
mutual boolean Whether relationship is bidirectional
labels array User-defined tags (family, close_friend, etc.)

4.6 Media Handling

4.6.1 Embedding vs. Referencing

Media files can be included in two ways:

  • Embedded: Files packaged within the export archive (recommended for portability)
  • Referenced: URLs to external media (useful for large files, requires active hosting)

4.7 GDPR Article 20 Compliance

WIA-LEG-008 Phase 1 fully implements GDPR Article 20 requirements:

GDPR Requirement WIA-LEG-008 Implementation
Structured format JSON with defined schemas ($schema references)
Commonly used JSON (RFC 8259), UTF-8 encoding
Machine-readable Parseable JSON with semantic structure
Data provided by user Clear attribution in activity streams
Right to transmit Standardized import/export enables direct transfer
Without hindrance No technical barriers, standard formats

4.8 Validation Rules

4.8.1 Required Fields

Minimum required fields for each schema:

  • Manifest: standard, version, generated, user.userId, package.packageId
  • Profile: userId, username OR email (at least one identifier)
  • Activity: id, type, actor, object, published
  • Relationship: relationshipId, type, targetUserId, established

4.8.2 Data Type Validation

Field Type Format Validation
userId string 1-255 characters, alphanumeric + underscore
email string RFC 5322 email format
datetime string ISO 8601 (YYYY-MM-DDTHH:MM:SSZ)
url string RFC 3986 URI format
language string BCP 47 language tag (en-US, es-MX)

4.9 Internationalization

Supporting global users requires proper internationalization:

  • Character Encoding: UTF-8 for all text fields
  • Language Tags: BCP 47 codes (en, es, zh-CN, ar)
  • Localized Content: Support for multilingual posts and profiles
  • Right-to-Left: Proper handling of Arabic, Hebrew scripts
  • Emoji Support: Full Unicode emoji support
  • Timezone: All timestamps in UTC with timezone metadata

4.10 Versioning and Evolution

The standard evolves while maintaining backward compatibility:

  • Semantic Versioning: MAJOR.MINOR.PATCH (2.0.0, 2.1.0, 2.1.1)
  • Schema URLs: Version-specific schema references
  • Deprecation Policy: 2-year notice before removing fields
  • Extension Fields: Custom fields prefixed with x_ allowed
  • Forward Compatibility: Parsers ignore unknown fields gracefully
