How the simulator works

The simulator runs entirely in the browser. The JSON Schema for the manifest is fetched from the WIA Standards public registry and cached locally. Validation produces a Problem Details (RFC 9457) error report when the manifest is invalid, and a signed conformance summary when the manifest is valid.

The simulator does not transmit the manifest to any server; all processing is local. To submit a manifest for live conformance verification, use the management API documented in spec/PHASE-2-API-INTERFACE.md.

Conformance tiers

  • Surface — accepted data formats; self-attested.
  • Verified — annual third-party audit against the four PHASE documents.
  • Anchored — continuous evidence package per Annex G; SBOM signed (CycloneDX 1.5 + Sigstore).

Reference standards

This simulator and the underlying schema reference only ALLOW citations per the WIA citation policy: OpenAPI Specification 3.1, JSON Schema 2020-12, IETF RFC 9457, IETF RFC 9700, IETF RFC 6585, ISO/IEC 27001:2022, Sigstore (DSSE envelope, Rekor transparency log), CycloneDX 1.5.

Time and integrity

The standard assumes synchronized clocks (NTPv4 stratum-2 or PTP) and SHA-256 hash integrity checks. The manifest signing key is rotated per the deployment policy; the previous key is preserved in a published key history so that auditors can verify historical manifests offline.

Validator details

The simulator implements the structural layer of the JSON Schema 2020-12 validator. Full constraint-level validation (pattern, format, additionalProperties) is performed only by the canonical validator hosted at the WIA Standards public registry. For automated CI integration, use the CLI tool at cli/data-portability.sh, which exits with a non-zero status code when the manifest is invalid and emits a Problem Details (RFC 9457) error report on stderr.
