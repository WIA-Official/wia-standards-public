6장: 3단계 - 통신 프로토콜

6.1 AIRI 프로토콜 개요

AI-로봇 인터페이스(AIRI) 프로토콜은 실시간, 안전 중요 AI-로봇 통신을 위해 설계된 WIA-AI-ROBOT 표준의 핵심 통신 프로토콜입니다.

6.2 메시지 형식 사양

AIRI 메시지는 선택적 JSON 인코딩과 함께 효율적인 바이너리 형식을 사용합니다:

바이너리 메시지 구조

AIRI 메시지 형식 (바이너리):
┌──────────┬──────────┬──────────┬────────────┬─────────┬────────┐
│  Magic   │ Version  │  Type    │  Length    │ Payload │   CRC  │
│ (4바이트)│ (2바이트)│ (2바이트)│  (4바이트) │ (N바이트│(4바이트│
└──────────┴──────────┴──────────┴────────────┴─────────┴────────┘

Magic: 0x41495249 (ASCII로 'AIRI')
Version: 프로토콜 버전 (예: v1.0의 경우 0x0100)
Type: 메시지 유형 식별자
Length: 페이로드 길이 (바이트)
Payload: 메시지 데이터 (유형별로 다름)
CRC: 오류 감지를 위한 CRC32 체크섬

메시지 유형

유형 코드 메시지 유형 목적
0x0001 HANDSHAKE 연결 수립
0x0002 HEARTBEAT 연결 유지 및 상태 확인
0x0010 COMMAND 로봇 명령 실행
0x0011 COMMAND_ACK 명령 확인
0x0012 STATUS 로봇 상태 업데이트
0x0020 TELEMETRY 센서 데이터 스트리밍
0x0030 AI_REQUEST AI 추론 요청
0x0031 AI_RESPONSE AI 추론 결과
0x0040 SAFETY_STOP 비상 정지 신호
0x00FF ERROR 오류 알림

6.3 연결 수립

AIRI 프로토콜은 핸드셰이크 프로세스를 사용하여 보안 연결을 수립합니다:

클라이언트                                    서버
  │                                           │
  │─────── 핸드셰이크 요청 ──────────────────>│
  │  (기능, 인증 토큰, 버전)                   │
  │                                           │
  │<──────── 핸드셰이크 응답 ──────────────────│
  │  (수락됨, session_id, 구성)               │
  │                                           │
  │─────── 하트비트 ──────────────────────>  │
  │                                           │
  │<──────── 하트비트 ACK ─────────────────────│
  │                                           │
  │           [연결 활성]                     │

6.4 실시간 통신

AIRI는 서로 다른 지연 시간 요구사항을 위한 여러 전송 메커니즘을 지원합니다:

전송 지연 시간 신뢰성 사용 사례
WebSocket 10-50ms 높음 (TCP) 명령, 상태, 일반 데이터
UDP 1-10ms 최선 노력 고주파 원격 측정
MQTT 50-200ms 구성 가능 QoS 플릿 조정, 이벤트
gRPC 5-30ms 높음 (HTTP/2) AI 추론, RPC 호출

6.5 안전 인터록

다층 안전 메커니즘은 위험한 작업을 방지합니다:

안전 아키텍처

계층 1: 명령 검증
├── 구문 확인
├── 매개변수 범위 검증
├── 기능 확인
└── 작업 공간 경계 확인

계층 2: 충돌 감지
├── 리소스 충돌 확인
├── 동시 명령 검증
└── 우선 순위 해결

계층 3: 안전 규칙
├── 속도 제한 시행
├── 힘 제한 확인
├── 안전 구역 준수
└── 충돌 예측

계층 4: 실시간 모니터링
├── 센서 기반 장애물 감지
├── 비상 정지 모니터링
└── 와치독 타이머

계층 5: 안전 장치
├── 연결 손실 처리
├── 하트비트 타임아웃
└── 안전 상태 보장

안전 정지 프로토콜

// 비상 정지 메시지 (최고 우선 순위)
{
  "type": "safety_stop",
  "priority": "critical",
  "timestamp": "2025-01-15T10:30:45.123Z",
  "reason": "obstacle_detected|emergency_button|connection_loss",
  "source": "sensor|operator|system",
  "stop_type": "immediate|controlled"
}

// 로봇은 반드시:
1. 100ms 이내에 모든 동작 정지
2. 안전하게 위치 유지
3. 확인 전송
4. 재개 명령 대기
안전 중요: 안전 정지 명령은 모든 대기열을 우회하며 100ms 이내에 처리되어야 합니다. 명시적인 운영자 승인 없이 재정의할 수 없습니다.

6.6 오류 처리 및 복구

견고한 오류 처리는 시스템 신뢰성을 보장합니다:

오류 분류

오류 수준 조치 복구
치명적 즉시 정지 수동 개입 필요
오류 현재 명령 중단 자동 재시도 또는 대체
경고 로그 및 계속 확대 모니터링
정보 로그만 조치 불필요

6.7 상태 동기화

분산 시스템 간 일관된 상태 유지:

// 버전 제어가 있는 상태 업데이트
{
  "type": "state_update",
  "robot_id": "rb_001",
  "timestamp": "2025-01-15T10:30:45.123Z",
  "version": 1234, // 단조 증가
  "state": {
    "position": {"x": 4.2, "y": 3.0, "z": 0.0},
    "mode": "executing",
    "battery": 85
  },
  "checksum": "sha256_hash_of_state"
}

// 충돌 해결: 버전과 함께 마지막 쓰기 우선
if (incoming.version > local.version) {
  local.state = incoming.state;
  local.version = incoming.version;
}

6.8 서비스 품질(QoS)

서로 다른 메시지 유형은 서로 다른 전달 보장이 필요합니다:

메시지 유형 QoS 수준 보장
명령 QoS 2 정확히 한 번 전달
상태 QoS 1 최소 한 번 전달
원격 측정 QoS 0 최선 노력 (최신 값)
안전 정지 QoS 2 정확히 한 번, 높은 우선 순위

6.9 대역폭 최적화

네트워크 리소스의 효율적인 사용:

압축 전략

// 페이로드 크기에 따른 적응형 압축
function selectCompression(payload) {
  if (payload.size < 1KB) {
    return "none"; // 오버헤드가 가치 없음
  } else if (payload.type === "image") {
    return "jpeg"; // 이미지용 손실 압축
  } else if (payload.type === "point_cloud") {
    return "lz4"; // 빠른 압축
  } else {
    return "gzip"; // 범용
  }
}

// 원격 측정용 델타 인코딩
{
  "type": "telemetry_delta",
  "base_version": 1000,
  "changes": {
    "position.x": 4.2, // 변경된 필드만
    "battery": 84
  }
}

6.10 프로토콜 보안

통신을 보호하기 위한 보안 조치:

보안 계층

  • 전송 보안: 모든 연결에 TLS 1.3 암호화
  • 인증: 인증서 또는 토큰을 사용한 상호 인증
  • 메시지 무결성: CRC32 체크섬 및 선택적 HMAC 서명
  • 재생 보호: 타임스탬프 검증 및 nonce 추적
  • 명령 권한 부여: 실행 전 권한 확인

핵심 내용

  • AIRI 프로토콜은 AI-로봇 시스템을 위한 실시간, 안전 중요 통신을 제공합니다
  • Magic, Version, Type, Length, Payload, CRC 필드가 있는 바이너리 메시지 형식
  • 핸드셰이크부터 안전 정지까지 10가지 표준 메시지 유형
  • 다양한 지연 시간 요구사항을 위한 여러 전송 옵션 (WebSocket, UDP, MQTT, gRPC)
  • 명령 검증부터 안전 장치까지 5계층 안전 아키텍처
  • 안전 정지 명령은 100ms 이내에 처리되어야 하며 재정의할 수 없습니다
  • 오류 처리에는 분류, 재시도 전략, 복구 절차가 포함됩니다
  • QoS 수준은 다양한 메시지 유형에 적절한 전달 보장을 보장합니다
  • TLS 암호화, 인증, 메시지 서명, 재생 보호를 통한 보안
