Chapter 5: Phase 2 - API Interface Definition

5.1 API Design Principles

Phase 2 of the WIA-AI-ROBOT standard defines comprehensive API interfaces that bridge AI systems and robots. These APIs follow RESTful principles while supporting real-time requirements:

Principle Description Implementation
RESTful Design Standard HTTP methods and status codes GET, POST, PUT, DELETE operations
Statelessness Each request contains all necessary information JWT tokens, session in request
Resource-Oriented URLs represent resources /robots/{id}/commands, /sensors/{id}
Versioned APIs support multiple versions /v1/, /v2/ in URL path
Well-Documented OpenAPI specifications Swagger/OpenAPI 3.0

5.3 Robot Control APIs

Unified interface for controlling diverse robot platforms:

Command Execution API

POST /v1/robots/{robot_id}/commands
Content-Type: application/json
Authorization: Bearer {token}

Request:
{
  "command_id": "cmd_123",
  "type": "navigate_to",
  "parameters": {
    "target": {"x": 5.0, "y": 3.2, "z": 0.0},
    "speed": 0.5
  },
  "priority": "normal",
  "safety": {
    "require_validation": true

{
  "command_id": "cmd_123",
  "status": "accepted",
  "estimated_duration": 30000,
  "status_url": "/v1/robots/{robot_id}/commands/cmd_123"

5.8 Error Handling

Standard HTTP status codes and error formats:

Status Code Meaning Example
200 OK Successful request GET status successful
201 Created Resource created New command created
202 Accepted Request accepted for processing Command queued
400 Bad Request Invalid request format Missing required field
401 Unauthorized Authentication required Missing token
403 Forbidden Insufficient permissions Cannot control this robot
404 Not Found Resource doesn't exist Robot ID not found
409 Conflict State conflict Robot busy with another task
422 Unprocessable Valid format, invalid content Command violates safety rules
500 Server Error Internal server error Unexpected failure
503 Unavailable Service temporarily unavailable Robot offline

Error Response Format

HTTP 422 Unprocessable Entity
Content-Type: application/json

{
  "error": {
    "code": "SAFETY_VIOLATION",
    "message": "Command violates workspace boundaries",
    "details": {
      "command_id": "cmd_123",
      "violation": "target position outside allowed workspace",
      "target": {"x": 10.0, "y": 5.0},
      "workspace_max": {"x": 5.0, "y": 5.0}
    },
    "timestamp": "2025-01-15T10:30:45Z"
  }
}

5.10 API Documentation Best Practices

Comprehensive API documentation using OpenAPI:

openapi: 3.0.0
info:
  title: WIA-AI-ROBOT API
  version: 1.0.0
  description: Standard API for AI-Robot integration
servers:
  - url: https://api.wia-robot.org/v1
paths:
  /robots/{robot_id}/commands:
    post:
      summary: Execute robot command
      parameters:
        - name: robot_id
          in: path
          required: true
          schema:
            type: string
      requestBody:
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/Command'
      responses:
        '202':
          description: Command accepted
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/CommandResponse'

Key Takeaways

  • WIA-AI-ROBOT APIs follow RESTful principles with real-time support
  • AI inference endpoints provide standardized access to LLMs and vision AI
  • Robot control APIs offer unified interfaces for diverse robot platforms
  • Telemetry streaming uses WebSocket and SSE for real-time data delivery
  • Configuration APIs enable dynamic capability discovery and parameter management
  • Fleet management APIs coordinate multiple robots efficiently
  • Multiple authentication methods support different use cases (API keys, JWT, OAuth, mTLS)
  • Standard HTTP status codes and error formats ensure consistent error handling
  • Rate limiting protects services from overload
  • OpenAPI specifications provide comprehensive API documentation
