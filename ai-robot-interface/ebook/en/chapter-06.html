Chapter 6: Phase 3 - Communication Protocol

6.1 AIRI Protocol Overview

The AI-Robot Interface (AIRI) Protocol is the core communication protocol of the WIA-AI-ROBOT standard, designed for real-time, safety-critical AI-robot communication. It provides low-latency message exchange, reliability guarantees, and built-in safety mechanisms.

Protocol Stack

┌─────────────────────────────────────┐
│   Application Layer (Commands/Data) │
├─────────────────────────────────────┤
│   AIRI Protocol (Message Format)    │
├─────────────────────────────────────┤
│   Transport (WebSocket/MQTT/UDP)    │
├─────────────────────────────────────┤
│   Network (TCP/IP or UDP/IP)        │
├─────────────────────────────────────┤
│   Physical (Ethernet/WiFi/5G)       │
└─────────────────────────────────────┘

6.2 Message Format Specification

AIRI messages use an efficient binary format with optional JSON encoding:

Binary Message Structure

AIRI Message Format (Binary):
┌──────────┬──────────┬──────────┬────────────┬─────────┬────────┐
│  Magic   │ Version  │  Type    │  Length    │ Payload │   CRC  │
│ (4 bytes)│ (2 bytes)│ (2 bytes)│  (4 bytes) │ (N bytes│(4 bytes│
└──────────┴──────────┴──────────┴────────────┴─────────┴────────┘

Magic: 0x41495249 ('AIRI' in ASCII)
Version: Protocol version (e.g., 0x0100 for v1.0)
Type: Message type identifier
Length: Payload length in bytes
Payload: Message data (varies by type)
CRC: CRC32 checksum for error detection

Message Types

Type Code Message Type Purpose
0x0001 HANDSHAKE Connection establishment
0x0002 HEARTBEAT Keep-alive and health check
0x0010 COMMAND Robot command execution
0x0011 COMMAND_ACK Command acknowledgment
0x0012 STATUS Robot status update
0x0020 TELEMETRY Sensor data streaming
0x0030 AI_REQUEST AI inference request
0x0031 AI_RESPONSE AI inference result
0x0040 SAFETY_STOP Emergency stop signal
0x00FF ERROR Error notification

  │  (capabilities, auth token, version)      │
  │                                           │
  │<──────── HANDSHAKE Response ──────────────│
  │  (accepted, session_id, config)           │
  │                                           │
  │─────── HEARTBEAT ──────────────────────>  │
  │                                           │

6.5 Safety Interlocks

Multi-layer safety mechanisms prevent dangerous operations:

Safety Architecture

Layer 1: Command Validation
├── Syntax check
├── Parameter range validation
├── Capability verification
└── Workspace boundary check

Layer 2: Conflict Detection
├── Resource conflict check
├── Concurrent command validation
└── Priority resolution

Layer 3: Safety Rules
├── Speed limit enforcement
├── Force limit verification
├── Safety zone compliance
└── Collision prediction

Layer 4: Real-Time Monitoring
├── Sensor-based obstacle detection
├── Emergency stop monitoring
└── Watchdog timer

Layer 5: Fail-Safe
├── Connection loss handling
├── Heartbeat timeout
└── Safe state guarantee

Safety Stop Protocol

// Emergency stop message (highest priority)
{
  "type": "safety_stop",
  "priority": "critical",
  "timestamp": "2025-01-15T10:30:45.123Z",
  "reason": "obstacle_detected|emergency_button|connection_loss",
  "source": "sensor|operator|system",
  "stop_type": "immediate|controlled",
  "affected_robots": ["rb_001", "rb_002"]
}

// Robot must:
1. Stop all motion within 100ms
2. Hold position safely
3. Send acknowledgment
4. Wait for resume command
Safety Critical: Safety stop commands bypass all queues and must be processed within 100ms. They cannot be overridden without explicit operator authorization.

6.6 Error Handling and Recovery

Robust error handling ensures system reliability:

Error Classification

Error Level Action Recovery
Critical Immediate stop Manual intervention required
Error Abort current command Automatic retry or fallback
Warning Log and continue Monitor for escalation
Info Log only No action needed

Retry Strategy

// Exponential backoff with jitter
function retryCommand(command, attempt = 0) {
  const maxAttempts = 3;
  const baseDelay = 1000; // 1 second

  if (attempt >= maxAttempts) {
    return {status: "failed", reason: "max_retries_exceeded"};
  }

  try {
    return executeCommand(command);
  } catch (error) {
    if (isRetriable(error)) {
      const delay = baseDelay * Math.pow(2, attempt) +
                    random(0, 1000); // jitter
      sleep(delay);
      return retryCommand(command, attempt + 1);
    } else {
      throw error; // Non-retriable error
    }
  }
}

Maintaining consistent state across distributed systems:

State Update Protocol

// State update with version control
{
  "type": "state_update",
  "robot_id": "rb_001",
  "timestamp": "2025-01-15T10:30:45.123Z",
  "version": 1234, // Monotonically increasing
  "state": {
    "position": {"x": 4.2, "y": 3.0, "z": 0.0},
    "mode": "executing",
    "current_command": "cmd_123",
    "battery": 85
  },
  "checksum": "sha256_hash_of_state"
}

// Conflict resolution: last-write-wins with version
if (incoming.version > local.version) {
  local.state = incoming.state;
  local.version = incoming.version;
}

6.9 Bandwidth Optimization

Efficient use of network resources:

Compression Strategies

// Adaptive compression based on payload size
function selectCompression(payload) {
  if (payload.size < 1KB) {
    return "none"; // Overhead not worth it
  } else if (payload.type === "image") {
    return "jpeg"; // Lossy for images
  } else if (payload.type === "point_cloud") {
    return "lz4"; // Fast compression
  } else {
    return "gzip"; // General purpose
  }
}

// Delta encoding for telemetry
{
  "type": "telemetry_delta",
  "base_version": 1000,
  "changes": {
    "position.x": 4.2, // Only changed fields
    "battery": 84
  }
}

6.10 Protocol Security

Security measures to protect communication:

Security Layers

  • Transport Security: TLS 1.3 encryption for all connections
  • Authentication: Mutual authentication using certificates or tokens
  • Message Integrity: CRC32 checksums and optional HMAC signatures
  • Replay Protection: Timestamp validation and nonce tracking
  • Command Authorization: Permission checks before execution

Message Signing

{
  "type": "command",
  "command_id": "cmd_123",
  "payload": {...},
  "security": {
    "timestamp": "2025-01-15T10:30:45.123Z",
    "nonce": "random_value_12345",
    "signature": "HMAC-SHA256(payload + timestamp + nonce)",
    "signer": "ai_service_001"
  }
}

// Verification steps:
1. Check timestamp (must be within 5 minutes)
2. Verify nonce not seen before
3. Validate signature
4. Check signer authorization

Key Takeaways

  • AIRI Protocol provides real-time, safety-critical communication for AI-robot systems
  • Binary message format with Magic, Version, Type, Length, Payload, and CRC fields
  • Ten standard message types from handshake to safety stop
  • Handshake process establishes secure connections with capability negotiation
  • Multiple transport options (WebSocket, UDP, MQTT, gRPC) for different latency needs
  • Five-layer safety architecture from command validation to fail-safe mechanisms
  • Safety stop commands must be processed within 100ms and cannot be overridden
  • Error handling includes classification, retry strategies, and recovery procedures
  • QoS levels ensure appropriate delivery guarantees for different message types
  • Security through TLS encryption, authentication, message signing, and replay protection
