Chapter 4: Posthumous Authentication and Access Control

While Chapter 3 focused on revoking the deceased's credentials, Chapter 4 addresses a equally critical challenge: granting controlled, auditable access to authorized parties who need to manage the deceased's digital estate. Executors, beneficiaries, and family members require legitimate access to accounts, data, and digital assets, but this access must be carefully managed to prevent fraud, protect privacy, and honor the deceased's wishes.

4.1 Executor Authorization Framework

The WIA-LEG-010 standard defines a comprehensive framework for authorizing executors and other authorized parties to access deceased individuals' digital resources. This framework balances security, legal validity, and practical usability.

4.1.1 Authorization Hierarchy

The standard recognizes four levels of authorized access:

  • Primary Executor: Court-appointed or will-designated executor with full authority over the digital estate
  • Co-Executors: Additional executors with shared authority, requiring multi-party approval for major actions
  • Specific Beneficiaries: Individuals granted access to specific accounts or data types
  • Professional Advisors: Lawyers, accountants, or digital estate specialists with limited, time-bound access

4.1.2 Executor Credential Issuance

async function issueExecutorCredentials(request) {
  // Validate legal authority
  const legalValidation = await validateLegalAuthority({
    executorId: request.executorId,
    deceasedId: request.deceasedId,
    legalDocuments: request.documents, // probate, will, court order
    jurisdiction: request.jurisdiction
  });

  if (!legalValidation.valid) {
    throw new Error(`Legal validation failed: ${legalValidation.reason}`);
  }

  // Generate time-locked credential
  const credential = {
    credentialId: generateUUID(),
    credentialType: 'executor_access',
    executorId: request.executorId,
    deceasedId: request.deceasedId,
    issuedAt: new Date().toISOString(),
    expiresAt: calculateExpiration(request.expiryPeriod || '1y'),
    permissions: request.permissions || ['read', 'export'],
    requiresMultiSig: request.coExecutors?.length > 0,
    multiSigConfig: {
      requiredSignatures: Math.ceil((request.coExecutors?.length || 0) / 2) + 1,
      signers: [request.executorId, ...(request.coExecutors || [])]
    },
    legalBasis: {
      documentType: legalValidation.documentType,
      documentId: legalValidation.documentId,
      issuingCourt: legalValidation.court,
      verificationHash: legalValidation.verificationHash
    }
  };

  // Sign credential with system key
  const signedCredential = await signCredential(credential);

  // Store in executor registry
  await storeExecutorCredential(signedCredential);

  // Generate access tokens
  const accessToken = await generateAccessToken(signedCredential);
  const refreshToken = await generateRefreshToken(signedCredential);

  return {
    credential: signedCredential,
    accessToken: accessToken,
    refreshToken: refreshToken,
    expiresIn: 3600, // 1 hour
    tokenType: 'Bearer'
  };
}

4.2 Multi-Party Authorization

For high-value assets or sensitive operations, WIA-LEG-010 supports multi-party authorization requiring signatures from multiple executors or witnesses before granting access.

4.2.1 Multi-Signature Protocol

async function initiateMultiSigAccess(accessRequest) {
  const { resourceId, executorId, coExecutors, operation } = accessRequest;

  // Create multi-sig proposal
  const proposal = {
    proposalId: generateUUID(),
    resourceId: resourceId,
    operation: operation,
    initiatedBy: executorId,
    requiredSignatures: Math.ceil(coExecutors.length / 2) + 1,
    signatures: [
      {
        signerId: executorId,
        timestamp: new Date().toISOString(),
        signature: await signProposal(accessRequest, executorId)
      }
    ],
    status: 'pending',
    createdAt: new Date().toISOString(),
    expiresAt: new Date(Date.now() + 7 * 24 * 60 * 60 * 1000).toISOString() // 7 days
  };

  // Store proposal
  await storeMultiSigProposal(proposal);

  // Notify co-executors
  await notifyCoExecutors(coExecutors, {
    proposalId: proposal.proposalId,
    operation: operation,
    initiatedBy: executorId,
    requiresAction: true
  });

  return proposal;
}

async function addSignature(proposalId, executorId, signature) {
  const proposal = await getMultiSigProposal(proposalId);

  // Validate executor is authorized signer
  if (!proposal.coExecutors.includes(executorId)) {
    throw new Error('Executor not authorized to sign this proposal');
  }

  // Verify signature
  const validSignature = await verifySignature(
    proposal,
    executorId,
    signature
  );

  if (!validSignature) {
    throw new Error('Invalid signature');
  }

  // Add signature
  proposal.signatures.push({
    signerId: executorId,
    timestamp: new Date().toISOString(),
    signature: signature
  });

  // Check if threshold met
  if (proposal.signatures.length >= proposal.requiredSignatures) {
    proposal.status = 'approved';

    // Execute the operation
    await executeAuthorizedOperation(proposal);
  }

  await updateMultiSigProposal(proposal);

  return proposal;
}

4.3 Time-Locked Access Controls

Some access should be delayed or limited by time constraints. For example, certain data might only become accessible 30 days after death, or executor access might automatically expire after one year.

4.3.1 Time-Lock Implementation

class TimeLockController {
  async createTimeLock(config) {
    const timeLock = {
      lockId: generateUUID(),
      resourceId: config.resourceId,
      authorizedParty: config.authorizedParty,
      unlockConditions: {
        type: config.lockType, // 'absolute_time', 'relative_time', 'event_based'
        unlockTime: config.unlockTime,
        baseEvent: config.baseEvent, // e.g., 'death_verification'
        offset: config.offset // e.g., '+30 days'
      },
      status: 'locked',
      createdAt: new Date().toISOString()
    };

    await this.storeTimeLock(timeLock);
    await this.scheduleUnlockCheck(timeLock);

    return timeLock;
  }

  async checkUnlockConditions(lockId) {
    const lock = await this.getTimeLock(lockId);
    const now = new Date();

    switch (lock.unlockConditions.type) {
      case 'absolute_time':
        return now >= new Date(lock.unlockConditions.unlockTime);

      case 'relative_time':
        const baseEventTime = await this.getEventTime(
          lock.unlockConditions.baseEvent
        );
        const unlockTime = this.addTimeOffset(
          baseEventTime,
          lock.unlockConditions.offset
        );
        return now >= unlockTime;

      case 'event_based':
        return await this.checkEventOccurred(
          lock.unlockConditions.requiredEvent
        );
    }
  }

  async attemptUnlock(lockId, requestingParty) {
    const lock = await this.getTimeLock(lockId);

    // Verify requesting party is authorized
    if (lock.authorizedParty !== requestingParty) {
      throw new Error('Unauthorized access attempt');
    }

    // Check unlock conditions
    const conditionsMet = await this.checkUnlockConditions(lockId);

    if (!conditionsMet) {
      const estimatedUnlock = await this.estimateUnlockTime(lock);
      throw new Error(
        `Time lock not yet released. Estimated unlock: ${estimatedUnlock}`
      );
    }

    // Unlock and grant access
    lock.status = 'unlocked';
    lock.unlockedAt = new Date().toISOString();
    lock.unlockedBy = requestingParty;

    await this.updateTimeLock(lock);

    // Generate access token
    return await this.generateTimeLockAccessToken(lock);
  }
}

4.4.2 Immutable Audit Chain

Audit logs are linked in a blockchain-style chain where each log entry references the hash of the previous entry, making tampering detectable: 

class ImmutableAuditLog {
  async logAccess(accessEvent) {
    // Get previous log entry
    const previousLog = await this.getLastLogEntry(accessEvent.deceasedId);

    // Create new log entry
    const logEntry = {
      ...accessEvent,
      previousHash: previousLog ? previousLog.hash : '0x0',
      sequenceNumber: previousLog ? previousLog.sequenceNumber + 1 : 1
    };

    // Calculate hash of this entry
    logEntry.hash = await this.calculateHash(logEntry);

    // Sign entry
    logEntry.signature = await this.signLogEntry(logEntry);

    // Store entry
    await this.storeLogEntry(logEntry);

    // Periodically anchor to blockchain
    if (logEntry.sequenceNumber % 100 === 0) {
      await this.anchorToBlockchain(logEntry.hash);
    }

    return logEntry;
  }

  async verifyAuditChain(deceasedId) {
    const allLogs = await this.getAllLogEntries(deceasedId);

    for (let i = 1; i < allLogs.length; i++) {
      const current = allLogs[i];
      const previous = allLogs[i - 1];

      // Verify chain linkage
      if (current.previousHash !== previous.hash) {
        return {
          valid: false,
          error: `Chain break at entry ${i}`,
          tampered: true
        };
      }

      // Verify hash
      const calculatedHash = await this.calculateHash(current);
      if (calculatedHash !== current.hash) {
        return {
          valid: false,
          error: `Hash mismatch at entry ${i}`,
          tampered: true
        };
      }

      // Verify signature
      const validSignature = await this.verifySignature(current);
      if (!validSignature) {
        return {
          valid: false,
          error: `Invalid signature at entry ${i}`,
          tampered: true
        };
      }
    }

    return {
      valid: true,
      totalEntries: allLogs.length,
      firstEntry: allLogs[0].timestamp,
      lastEntry: allLogs[allLogs.length - 1].timestamp
    };
  }
}

4.5 Privacy-Preserving Access

Executors may need to access certain information without viewing all content. Zero-knowledge proofs and selective disclosure enable this:

  • Selective Disclosure: Executors can prove account balance without seeing transaction history
  • Redacted Access: Sensitive content (medical records, private messages) can be redacted while preserving metadata
  • Purpose Limitation: Access is granted only for specific stated purposes
  • Minimization: Only minimum necessary data is revealed

4.6 Revocation of Posthumous Access

Executor credentials can be revoked if legal authority changes, fraud is detected, or the estate is closed: 

async function revokeExecutorAccess(revocationRequest) {
  const { credentialId, reason, revokedBy, legalBasis } = revocationRequest;

  // Retrieve credential
  const credential = await getExecutorCredential(credentialId);

  // Validate revocation authority
  await validateRevocationAuthority(revokedBy, credential, legalBasis);

  // Revoke all active sessions
  await revokeActiveSessions(credentialId);

  // Invalidate tokens
  await invalidateTokens(credentialId);

  // Update credential status
  credential.status = 'revoked';
  credential.revokedAt = new Date().toISOString();
  credential.revokedBy = revokedBy;
  credential.revocationReason = reason;

  await updateExecutorCredential(credential);

  // Publish revocation to registry
  await publishRevocation({
    credentialId: credentialId,
    revokedAt: credential.revokedAt,
    reason: reason
  });

  // Audit log
  await logAuditEvent({
    eventType: 'executor_access_revoked',
    credentialId: credentialId,
    revokedBy: revokedBy,
    reason: reason
  });

  return credential;
}

4.7 Best Practices for Implementation

  • Always verify legal authority before issuing executor credentials
  • Implement multi-party authorization for high-value or sensitive resources
  • Use time-locks appropriately to prevent premature access
  • Maintain comprehensive, immutable audit logs
  • Design for privacy: minimize data exposure even to authorized parties
  • Implement automatic expiration of executor credentials
  • Provide clear interfaces for family members to understand what was accessed
