Chapter 3: Identity Revocation Mechanisms

Once a death has been verified through the protocols described in Chapter 2, the next critical phase is systematic revocation of the deceased person's digital credentials. This process must be comprehensive (covering all credentials), rapid (minimizing windows for fraud), secure (preventing unauthorized access during transition), and graceful (allowing for legitimate posthumous access as covered in Chapter 4).

This chapter explores the technical architecture for cascading credential revocation, the challenges of reaching across heterogeneous identity systems, and the algorithms that make large-scale revocation feasible while maintaining security and auditability.

3.1 The Identity Credential Landscape

Modern digital identity is not a single monolithic credential but a complex web of interrelated authentication mechanisms, each with different properties, lifetimes, and revocation requirements.

3.1.1 Credential Types and Characteristics

  • Password-based credentials: Traditional username/password combinations. Simple to revoke but often cached or stored in browser password managers, requiring additional cleanup.
  • Session tokens: Temporary credentials issued after authentication. May persist in cookies, local storage, or HTTP headers. Revocation requires invalidating both server-side sessions and client-side storage.
  • OAuth/OpenID tokens: Delegated authorization credentials used for single sign-on. Revoking the primary identity must cascade to all connected services—potentially dozens of third-party applications.
  • API keys and access tokens: Long-lived credentials for programmatic access. Often embedded in scripts, applications, or IoT devices. May require coordination with device manufacturers for revocation.
  • Biometric credentials: Fingerprints, facial recognition templates, iris scans, voice prints. Present unique challenges as they're tied to a physical body and may continue to function for a period after death.
  • Hardware tokens: Physical security keys (YubiKey, smart cards). Revocation must occur even if the physical device is never recovered.
  • Blockchain credentials: Decentralized identifiers (DIDs), verifiable credentials, private keys. No central authority to contact; revocation must occur through cryptographic means.
  • Derived credentials: Credentials created from or dependent on other credentials. Revoking a parent must cascade to all children.

3.1.2 Credential Dependency Graph

Credentials exist in a dependency graph where some credentials enable the creation or use of others. Consider this example: 

Email Account (root)
  ├── Password Reset Tokens
  ├── Two-Factor Authentication Setup
  │   ├── SMS 2FA
  │   ├── Authenticator App
  │   └── Backup Codes
  ├── OAuth Applications (SSO)
  │   ├── Social Media Login (Facebook, Google)
  │   ├── Enterprise SaaS (Slack, GitHub)
  │   └── Financial Services (PayPal, Stripe)
  ├── API Keys
  │   ├── Cloud Provider Keys (AWS, Azure)
  │   ├── Payment Processor Keys
  │   └── IoT Device Keys
  └── Session Tokens
      ├── Browser Sessions
      ├── Mobile App Sessions
      └── Desktop Application Sessions

Revoking the root email account should cascade through this entire tree, but in practice, many of these dependencies are not tracked or enforced by systems, requiring explicit enumeration and revocation at each level.

3.2 Cascading Revocation Algorithm

The WIA-LEG-010 standard specifies a breadth-first search (BFS) algorithm for systematic credential revocation that ensures all credentials are discovered and revoked while respecting dependency relationships.

3.2.1 Core Algorithm

async function cascadeRevocation(identityId, deathProof) {
  // Initialize tracking structures
  const queue = [identityId];
  const discovered = new Set();
  const revoked = new Set();
  const failed = new Set();
  const auditLog = [];

  // BFS traversal of credential graph
  while (queue.length > 0) {
    const currentCred = queue.shift();

    // Skip if already processed
    if (discovered.has(currentCred)) {
      continue;
    }

    discovered.add(currentCred);

    try {
      // Discover dependent credentials
      const dependents = await discoverDependentCredentials(
        currentCred,
        deathProof
      );

      // Add to queue for processing
      queue.push(...dependents.filter(d => !discovered.has(d)));

      // Attempt revocation
      const revocationResult = await revokeCredential(
        currentCred,
        deathProof,
        {
          reason: 'death_verified',
          cascading: true,
          timestamp: new Date().toISOString()
        }
      );

      if (revocationResult.success) {
        revoked.add(currentCred);
        auditLog.push({
          credential: currentCred,
          status: 'revoked',
          timestamp: revocationResult.timestamp,
          method: revocationResult.method
        });
      } else {
        failed.add(currentCred);
        auditLog.push({
          credential: currentCred,
          status: 'failed',
          reason: revocationResult.error,
          timestamp: new Date().toISOString()
        });
      }
    } catch (error) {
      failed.add(currentCred);
      auditLog.push({
        credential: currentCred,
        status: 'error',
        error: error.message,
        timestamp: new Date().toISOString()
      });
    }
  }

  // Generate comprehensive revocation report
  return {
    identityId: identityId,
    totalDiscovered: discovered.size,
    successfulRevocations: revoked.size,
    failedRevocations: failed.size,
    successRate: (revoked.size / discovered.size) * 100,
    revokedCredentials: Array.from(revoked),
    failedCredentials: Array.from(failed),
    auditLog: auditLog,
    timestamp: new Date().toISOString()
  };
}

3.2.2 Credential Discovery

Discovering all credentials associated with an identity is challenging because there's no universal registry. The standard defines multiple discovery mechanisms: 

async function discoverDependentCredentials(credentialId, deathProof) {
  const discovered = [];

  // 1. Query credential registries
  const registryResults = await queryCredentialRegistries(
    credentialId,
    deathProof
  );
  discovered.push(...registryResults);

  // 2. Parse OAuth authorization server metadata
  if (credentialId.type === 'oauth') {
    const oauthClients = await discoverOAuthClients(credentialId);
    discovered.push(...oauthClients);
  }

  // 3. Scan blockchain for DID credentials
  if (credentialId.type === 'did') {
    const blockchainCreds = await scanBlockchainForCredentials(
      credentialId.did
    );
    discovered.push(...blockchainCreds);
  }

  // 4. Query platform-specific APIs
  const platformCreds = await queryPlatformAPIs(credentialId);
  discovered.push(...platformCreds);

  // 5. Check for hardware tokens in device registry
  const hardwareTokens = await queryDeviceRegistry(credentialId);
  discovered.push(...hardwareTokens);

  // Deduplicate and return
  return Array.from(new Set(discovered));
}

3.3 OAuth and SSO Revocation

OAuth and single sign-on (SSO) credentials present particular challenges because they enable access to dozens or hundreds of third-party services. Revoking the primary identity must notify all relying parties.

3.3.1 OAuth Token Revocation Protocol

async function revokeOAuthCredentials(oauthIdentity, deathProof) {
  // Get all authorized applications
  const authorizedApps = await getAuthorizedApplications(oauthIdentity.id);

  const revocationResults = await Promise.allSettled(
    authorizedApps.map(async app => {
      try {
        // Revoke access token
        await revokeOAuthToken({
          token: app.accessToken,
          tokenTypeHint: 'access_token',
          clientId: app.clientId
        });

        // Revoke refresh token if exists
        if (app.refreshToken) {
          await revokeOAuthToken({
            token: app.refreshToken,
            tokenTypeHint: 'refresh_token',
            clientId: app.clientId
          });
        }

        // Notify relying party of death event
        if (app.deathNotificationWebhook) {
          await notifyRelyingParty(app.deathNotificationWebhook, {
            eventType: 'user.deceased',
            userId: oauthIdentity.id,
            proof: deathProof,
            timestamp: new Date().toISOString()
          });
        }

        return { app: app.name, status: 'revoked' };
      } catch (error) {
        return { app: app.name, status: 'failed', error: error.message };
      }
    })
  );

  return revocationResults;
}

3.4 Biometric Credential Purging

Biometric credentials present unique challenges because they're derived from physical characteristics of a body. After death, these biometrics may still technically function for a period (fingerprints remain usable for hours to days, facial recognition may work on a deceased person's face, etc.).

Security Critical

Biometric templates must be purged from all systems immediately upon death verification to prevent postmortem authentication fraud. This includes templates stored in devices (phones, laptops), servers, and biometric databases.

3.4.1 Biometric Template Purging Protocol

async function purgeBiometricTemplates(identityId, deathProof) {
  const biometricSystems = [
    'fingerprint_databases',
    'facial_recognition_systems',
    'iris_scan_databases',
    'voice_print_systems',
    'behavioral_biometrics'
  ];

  const purgeResults = await Promise.allSettled(
    biometricSystems.map(async system => {
      // Query system for templates
      const templates = await queryBiometricSystem(system, identityId);

      // Purge each template
      const purged = await Promise.all(
        templates.map(async template => {
          // Overwrite with random data before deletion (DOD 5220.22-M)
          await secureOverwrite(template.id);

          // Delete template
          await deleteBiometricTemplate(template.id);

          // Log purge action
          await logAuditEvent({
            action: 'biometric_purge',
            system: system,
            templateId: template.id,
            templateType: template.type,
            timestamp: new Date().toISOString(),
            proof: deathProof
          });

          return template.id;
        })
      );

      return {
        system: system,
        templatesPurged: purged.length,
        status: 'completed'
      };
    })
  );

  return purgeResults;
}

3.5 Blockchain and DID Revocation

Decentralized identifiers (DIDs) and blockchain-based credentials cannot be "revoked" in the traditional sense because there's no central authority to contact. Instead, revocation is achieved through cryptographic means and on-chain status updates.

3.5.1 DID Deactivation

async function deactivateDID(did, deathProof, privateKey) {
  // Create DID deactivation document
  const deactivationDoc = {
    '@context': 'https://www.w3.org/ns/did/v1',
    id: did,
    deactivated: true,
    deathProof: {
      certificateHash: deathProof.certificateHash,
      blockchainAnchor: deathProof.blockchainAnchor,
      timestamp: deathProof.timestamp
    }
  };

  // Sign deactivation document
  const signature = await signDocument(deactivationDoc, privateKey);

  // Publish to DID registry
  const txHash = await publishToBlockchain({
    method: 'deactivateDID',
    did: did,
    document: deactivationDoc,
    signature: signature
  });

  // Update verifiable credential status list
  await revokeVerifiableCredentials(did, deathProof);

  return {
    did: did,
    deactivated: true,
    transactionHash: txHash,
    timestamp: new Date().toISOString()
  };
}

3.6 Notification and Communication

Systematic revocation must be accompanied by appropriate notifications to various stakeholders:

  • Executors/Family: Regular status updates on revocation progress
  • Service Providers: Notification of credential revocation and next steps
  • Security Teams: Alerts about any failed revocations or anomalies
  • Regulators: Compliance reports for jurisdictions requiring notification

3.7 Edge Cases and Failure Handling

Not all revocations will succeed on the first attempt. The standard defines handling for common failure scenarios:

  • Service Unavailable: Queue for retry with exponential backoff (max 7 days)
  • Authentication Required: Escalate to executor for manual intervention
  • Credential Not Found: Log as successful (already revoked or never existed)
  • Partial Revocation: Mark as partially successful, continue with other credentials
  • Revocation Rejected: Legal escalation, may require court order

3.8 Performance and Scalability Considerations

Revoking potentially hundreds of credentials across dozens of services requires careful performance optimization:

  • Parallel revocation where dependencies allow (up to 50 concurrent operations)
  • Batching of revocation requests to the same service
  • Caching of service metadata to reduce discovery overhead
  • Priority queuing: high-risk credentials (financial, healthcare) revoked first
  • Asynchronous processing with status callbacks
