Chapter 5: Phase 2 - API Interface

弘益人間 (Hongik Ingan)

APIs enable systems to communicate. Phase 2 provides REST endpoints, GraphQL schemas, and WebSocket connections for real-time food safety data exchange.

5.1 Base API URL

Production: https://api.wia.org/food-safety/v1
Staging:    https://api-staging.wia.org/food-safety/v1

5.2 Authentication

Authorization: Bearer <access_token>
X-API-Key: <api_key>
X-WIA-DID: did:wia:manufacturer:12345

5.3 Product Endpoints

MethodEndpointDescription
POST/productsCreate new product
GET/products/{id}Retrieve product details
PATCH/products/{id}Update product info
DELETE/products/{id}Mark as recalled

5.4 Example: Create Product

POST /products
Content-Type: application/json

{
  "productName": "Fresh Strawberries",
  "category": "fresh-produce",
  "batchNumber": "FARM-20250115-001"
}

Response (201 Created):
{
  "productId": "FS-2025-STR-001",
  "status": "created",
  "qrCode": "https://wia.org/verify/FS-2025-STR-001"
}

5.5 Monitoring Endpoints

POST /monitoring              - Submit monitoring data
GET  /monitoring/{id}/latest   - Get latest readings
GET  /monitoring/alerts        - Get active alerts

5.6 Recall Endpoints

POST  /recalls         - Initiate recall
GET   /recalls/{id}    - Get recall details
PATCH /recalls/{id}/status - Update status

5.7 Error Codes

CodeDescription
400Bad Request - Invalid input
401Unauthorized - Missing auth
404Not Found - Product not found
429Too Many Requests - Rate limit
500Internal Server Error

5.8 Rate Limiting

TierRequests/minuteRequests/day
Free601,000
Standard30010,000
Enterprise1,000100,000

Summary

Phase 2 provides comprehensive REST APIs for product management, safety monitoring, recalls, and traceability. With OAuth 2.0 security, rate limiting, and clear error handling, these APIs enable seamless integration with existing systems.
