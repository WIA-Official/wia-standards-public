Chapter 2: Data Schema

Purpose

This chapter covers the Chapter 2: Data Schema area of the WIA-MED-028 Health Insurance Data standard. The standard defines a cross-vendor data and protocol layer for health-insurance claims, eligibility, prior authorization, and remittance, interoperable with HL7 FHIR R5, X12 EDI 270/271/278/834/835/837, and the NCPDP SCRIPT 2017071 standard.

Reference Standards (CITATION-POLICY ALLOW only)

  • HL7 FHIR R5 — Coverage, ExplanationOfBenefit, Claim, ClaimResponse resources
  • X12 EDI 270/271 — Eligibility inquiry/response
  • X12 EDI 278 — Prior authorization request/response
  • X12 EDI 837/835 — Health care claim / payment
  • NCPDP SCRIPT 2017071 — pharmacy electronic prescribing
  • HIPAA 45 CFR Part 162 — Administrative requirements
  • HIPAA 45 CFR Part 164 — Security and Privacy
  • ISO/IEC 27001:2022, ISO/IEC 27018:2019, ISO/IEC 27701:2019
  • OpenAPI Specification 3.1, JSON Schema 2020-12
  • IETF RFC 9700 (OAuth 2.1), RFC 9457 (Problem Details)
  • Sigstore (DSSE envelope, Rekor transparency log)

Implementation Procedure

Health-insurance data exchange follows a four-stage protocol: eligibility verification, claim submission, adjudication, and remittance. Each stage produces a signed manifest anchored to a Sigstore Rekor entry so that downstream auditors can reconstruct the chain offline.

Eligibility Verification

The provider issues an eligibility inquiry (X12 EDI 270 or HL7 FHIR Coverage operation) to the payer's WIA-MED-028 endpoint. The payer responds within the SLA declared in the manifest (default p95: 2 seconds). Eligibility data is signed by the payer; the provider records both the inquiry and the response in its audit log.

Claim Submission

The provider submits a claim (X12 EDI 837 or HL7 FHIR Claim resource) to the payer. The submission includes the patient's pseudonymous identifier, service codes (CPT/HCPCS), diagnosis codes (ICD-11), and supporting documentation references. The submission is signed by the provider and acknowledged by the payer with a transaction-set acknowledgment.

Adjudication

The payer adjudicates the claim per its medical policy. The adjudication outcome is returned as a ClaimResponse (HL7 FHIR R5) or X12 EDI 277 status response. Denials include the reason code from the published code set and a human-readable description.

Remittance

The payer issues a remittance advice (X12 EDI 835 or HL7 FHIR ExplanationOfBenefit resource) to the provider. The remittance includes the paid amount, adjustment codes, and the patient responsibility amount. The remittance is signed by the payer and anchored to a Sigstore Rekor entry.

Privacy and Security

Personal data exchanged via this protocol is subject to HIPAA 45 CFR Parts 162 and 164 in the United States, GDPR Article 9 in the European Union, and equivalent national health-data privacy regulations. Implementations MUST encrypt data in transit (TLS 1.3) and at rest (AES-256-GCM or stronger), apply role-based access controls, and maintain tamper-evident audit logs.

Reference Standards and Implementation Notes

This standard is designed to interoperate with established international frameworks. Implementations SHOULD treat the following as the single source of truth and use this PHASE document as the interoperability layer that maps existing data into a machine-readable, signed, and auditable shape.

  • OpenAPI Specification 3.1 — REST API definition
  • JSON Schema 2020-12 — schema validation
  • ISO/IEC 27001:2022 — information security management
  • ISO/IEC 27018:2019 — PII protection in public clouds
  • ISO/IEC 17065:2012 — product certification bodies
  • CycloneDX 1.5 / SPDX 2.3 — software bill of materials
  • Sigstore (DSSE envelope, Rekor transparency log) — signature anchoring
  • in-toto Attestation Framework 1.0 — build provenance
  • IETF RFC 9700 — OAuth 2.1
  • IETF RFC 9457 — Problem Details for HTTP APIs
  • IETF RFC 8615 — well-known URIs

Practical Implementation Checklist

  • Manifest integrity — every published manifest is signed via Sigstore DSSE. Tampering is detectable without trusting an intermediary. Updates are issued as new manifests; the prior manifest is preserved with a supersede marker.
  • Retention — keep signed evidence for at least seven years to satisfy ISO/IEC 17065:2012 audit expectations. Some regulators require longer retention; in that case the deployment policy MUST extend the window rather than relying on the defaults.
  • Versioning — follow Semantic Versioning 2.0.0. Major version bumps require at least a 90-day overlap with the prior major version on every WIA-published reference implementation.
  • Negative-result reporting — a skipped indicator without recorded justification is treated as a failed indicator at audit time.
  • Key rotation — manifest signing keys are rotated per the deployment policy. The previous key is preserved in a published key history so that auditors can verify historical manifests offline.

Reference Topologies

Implementations declare their operational topology in the manifest: single-tenant edge (one runtime per organization), multi-tenant gateway (shared runtime with header-based isolation), federated mesh (multiple runtimes peer to one another and publish manifests to a directory service), or air-gapped batch (signed evidence package transported out-of-band). Each topology ships with reference test vectors so that implementations can validate their deployment matches the canonical pattern.

Open Governance

Issues, errata, and proposals are tracked at github.com/WIA-Official/wia-standards/issues with the appropriate label. The WIA Standards working group reviews open issues at the start of every minor release cycle and publishes the resulting decision log alongside the release notes. Errata are issued as patch releases; new normative requirements trigger minor bumps; backwards-incompatible changes trigger major bumps.
