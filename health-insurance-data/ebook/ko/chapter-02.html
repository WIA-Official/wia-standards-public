제2장: 청구 데이터 형식

"정확하고 표준화된 청구 데이터는 의료 제공자가 신속하게 보상을 받고, 보험사가 효율적으로 처리하며, 환자가 명확한 비용 정보를 얻는 기반입니다."

— 청구 데이터 표준화의 중요성

2.1 CMS-1500 양식 (전문의 청구)

CMS-1500은 전문의, 클리닉, 외래 서비스에 사용되는 표준 청구 양식입니다.

📋 주요 섹션 구조

필드 번호 내용 필수 여부
1-13 환자 및 보험 가입자 정보 필수
14-20 진단 및 진료 날짜 필수
21-24 서비스 라인 (CPT 코드) 필수
25-33 제공자 및 결제 정보 필수

🔍 핵심 필드 상세 설명

환자 정보 섹션 (1-13)

  • 필드 1 — 보험 유형 (Medicare, Medicaid, 상업 보험 등)
  • 필드 2 — 환자 성명 (성, 이름, 중간 이름)
  • 필드 3 — 환자 생년월일 및 성별 (MM DD YYYY)
  • 필드 5 — 환자 주소 (거리, 도시, 주, 우편번호)
  • 필드 9 — 타 보험 정보 (COB - Coordination of Benefits)
  • 필드 11 — 가입자 보험 번호 (Member ID)
  • 필드 12-13 — 환자/가입자 서명 및 날짜

진단 정보 (14-21)

  • 필드 14 — 현재 질환 또는 부상 발생일
  • 필드 15 — 기타 날짜 (첫 증상, 유사한 질환)
  • 필드 17 — 의뢰 의사 이름 (필요시)
  • 필드 19 — 추가 청구 정보
  • 필드 21 — ICD-10-CM 진단 코드 (최대 12개)
    • 우선순위 순서로 입력 (주 진단 먼저)
    • 각 서비스 라인과 연결 (필드 24E 참조)

서비스 라인 정보 (24A-24J)

각 서비스마다 다음 정보가 필요합니다:

필드 설명 예시
24A 서비스 날짜 (From-To) 01/15/2025
24B 진료 장소 코드 11 (Office)
24D CPT/HCPCS 코드 99213 (외래 진찰)
24E 진단 포인터 A (필드 21의 첫 진단)
24F 청구 금액 $150.00
24G 서비스 단위 (일수/횟수) 1
24J 제공자 NPI 번호 1234567890

제공자 정보 (25-33)

  • 필드 25 — 제공자 연방 세금 ID 번호 (EIN 또는 SSN)
  • 필드 31 — 제공자 서명 및 날짜
  • 필드 32 — 서비스 시설 정보 (병원, 클리닉 등)
  • 필드 33 — 청구 제공자 정보 (이름, 주소, NPI)

2.2 UB-04 양식 (병원 청구)

UB-04 (CMS-1450)는 병원, 요양시설, 입원 환자 서비스에 사용됩니다.

🏥 UB-04 양식의 주요 차이점

  • 복잡한 수가 구조 — DRG, APR-DRG 등 입원 환자 분류
  • Revenue Code 사용 — 부서별 서비스 분류 (약국, 방사선, 수술실 등)
  • 다중 진단 및 시술 — 최대 18개 진단, 6개 시술 코드
  • Present on Admission (POA) — 입원 시 진단 존재 여부

📊 UB-04 핵심 필드

환자 정보 (FL 1-41)

Form Locator 내용
FL 4 청구 유형 (입원, 외래, 응급실 등)
FL 6 청구 시작/종료일
FL 8 환자 이름
FL 10 환자 생년월일
FL 12 입원일
FL 17 환자 상태 (퇴원 시)

Revenue Code 라인 (FL 42-49)

각 서비스 라인은 다음 정보를 포함합니다:

  • FL 42: Revenue Code — 부서/서비스 유형
    • 0250: 약국
    • 0300: 실험실
    • 0360: 수술실
    • 0450: 응급실
    • 0730: EKG/ECG
  • FL 43: Description — 서비스 설명
  • FL 44: HCPCS/CPT Code — 시술 코드
  • FL 45: Service Date — 서비스 날짜
  • FL 46: Service Units — 단위 (일수, 마일, 분 등)
  • FL 47: Total Charges — 총 청구 금액

진단 및 시술 코드 (FL 66-81)

  • FL 67: Principal Diagnosis — 주 진단 (ICD-10-CM)
  • FL 67A-Q: Other Diagnoses — 추가 진단 (최대 17개)
  • FL 72: POA Indicator — 입원 시 진단 존재 여부
    • Y: Yes (입원 시 존재)
    • N: No (입원 후 발생)
    • U: Unknown (불명)
    • W: 임상적으로 미확정
  • FL 74: Principal Procedure — 주 시술 (ICD-10-PCS)
  • FL 74A-E: Other Procedures — 추가 시술 (최대 5개)

2.3 전자 청구 트랜잭션 (837)

EDI 837 트랜잭션은 종이 양식을 전자 형식으로 변환한 것입니다.

🔄 837 트랜잭션 유형

  • 837P (Professional) — CMS-1500 전자 버전 (의사, 클리닉)
  • 837I (Institutional) — UB-04 전자 버전 (병원)
  • 837D (Dental) — ADA 치과 청구 전자 버전

📦 837 트랜잭션 구조

계층적 구조로 데이터가 조직됩니다:

ISA (Interchange Header) — 전체 파일 시작
  GS (Functional Group Header) — 그룹 시작
    ST (Transaction Set Header) — 개별 청구 시작
      BHT (Beginning of Hierarchical Transaction)

      2000A (Billing Provider) — 청구 제공자
        NM1, N3, N4, REF, PER 세그먼트

      2000B (Subscriber) — 보험 가입자
        NM1, N3, N4, DMG, REF
        SBR (Subscriber Information)

        2010BA (Subscriber Name)
        2010BB (Payer Name) — 보험사 정보

        2300 (Claim Information) — 청구 정보
          CLM (Claim Segment)
          DTP (Date/Time)
          REF (Reference)
          HI (Health Care Diagnosis Code) — 진단 코드

          2400 (Service Line) — 서비스 라인
            LX (Line Number)
            SV1 (Professional Service) — CPT 코드, 금액
            DTP (Service Date)
            REF (Line Item Control Number)

    SE (Transaction Set Trailer) — 개별 청구 종료
  GE (Functional Group Trailer) — 그룹 종료
IEA (Interchange Trailer) — 전체 파일 종료

💡 837P 핵심 세그먼트 예시

CLM (Claim Segment)

CLM*123456789*150***11:B:1*Y*A*Y*Y~

필드 설명:
- 123456789: 청구 ID
- 150: 청구 금액 ($150.00)
- 11:B:1: 진료 장소 코드 (11=Office, B=일반 의사)
- Y: 할당 수락 (Assignment of Benefits)
- A: Medicare 할당 코드
- Y: 서명 확인
- Y: 제공자 서명

HI (Health Care Diagnosis Code)

HI*ABK:E119*ABF:I10*ABF:E7800~

필드 설명:
- ABK: Principal Diagnosis (주 진단)
- E119: ICD-10-CM 코드 (제2형 당뇨병)
- ABF: Admitting Diagnosis (추가 진단)
- I10: 고혈압
- E7800: 순수 고콜레스테롤혈증

SV1 (Professional Service)

SV1*HC:99213*150*UN*1***1~

필드 설명:
- HC: HCPCS/CPT 코드
- 99213: 외래 진찰 코드
- 150: 청구 금액
- UN: Units (단위)
- 1: 서비스 단위 수
- 1: 진단 포인터 (HI 세그먼트의 첫 진단)

2.4 청구 검증 및 오류 방지

✅ 청구 제출 전 체크리스트

  • 환자 정보 확인
    • 성명 철자 정확성
    • 생년월일 정확성
    • 보험 ID 번호 확인
    • 주소 최신화
  • 보험 자격 검증
    • 서비스 날짜에 유효한 보험
    • 서비스가 보장 항목인지 확인
    • 사전 승인 필요 여부
  • 코딩 정확성
    • ICD-10 진단 코드 최신 버전 사용
    • CPT 코드가 진단과 일치
    • Modifier 올바르게 사용
    • Units 정확성
  • 금액 계산
    • Units × Unit Price = Total
    • 수수료 일정(Fee Schedule) 준수
    • 계약 요율 확인

🚫 흔한 청구 오류

오류 유형 원인 해결 방법
인구통계 오류 환자 이름, 생년월일 불일치 보험카드와 대조 확인
자격 문제 서비스 날짜에 보험 비활성 실시간 자격 확인 (270/271)
코딩 오류 진단-시술 불일치 Medical Necessity 검증
중복 청구 동일 서비스 재제출 청구 추적 시스템
지원 문서 누락 의무 기록 미첨부 Attachment 관리 자동화

청구 오류의 80%는 제출 전 자동 검증 시스템으로 예방할 수 있습니다. WIA-MED-028 표준은 이러한 검증 규칙을 명확히 정의하여 청구 거부율을 획기적으로 낮춥니다.
