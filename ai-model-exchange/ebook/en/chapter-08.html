8.3 Case Study 2: Cross-Platform Mobile Deployment

📱 Scenario: Deploying Object Detection to iOS and Android

A company trained YOLOv5 in PyTorch and needed to deploy it on both iOS (CoreML) and Android (TFLite) while maintaining consistent performance.

Solution Architecture

# PyTorch (Training) → ONNX → TFLite (Android)
#                            ↓
#                        CoreML (iOS)

# Step 1: Export PyTorch to ONNX
import torch
from models.yolo import YOLOv5

model = YOLOv5.from_pretrained('yolov5s')
model.eval()

dummy_input = torch.randn(1, 3, 640, 640)
torch.onnx.export(
    model, dummy_input, "yolov5s.onnx",
    opset_version=12, input_names=['images'],
    output_names=['output'], dynamic_axes={'images': {0: 'batch'}}
)

# Step 2: ONNX to CoreML (iOS)
import coremltools as ct
from onnx_coreml import convert

coreml_model = convert(model='yolov5s.onnx')
coreml_model.save('YOLOv5.mlmodel')

# Step 3: ONNX to TensorFlow to TFLite (Android)
from onnx_tf.backend import prepare
import tensorflow as tf

onnx_model = onnx.load("yolov5s.onnx")
tf_model = prepare(onnx_model)
tf_model.export_graph("yolov5_tf")

converter = tf.lite.TFLiteConverter.from_saved_model("yolov5_tf")
converter.optimizations = [tf.lite.Optimize.DEFAULT]
tflite_model = converter.convert()

with open("yolov5s.tflite", "wb") as f:
    f.write(tflite_model)

8.4 Case Study 3: MLOps Pipeline at Scale

🏭 Scenario: Enterprise ML Platform with 100+ Models

A large tech company built a centralized ML platform managing hundreds of models across dozens of teams.

Architecture

  • Training: Kubernetes + Kubeflow for distributed training
  • Registry: MLflow with PostgreSQL backend
  • Storage: S3 for model artifacts
  • Serving: NVIDIA Triton on GPU clusters
  • Monitoring: Prometheus + Grafana
  • CI/CD: GitLab CI with automated testing

Automated Pipeline

# .gitlab-ci.yml
stages:
  - train
  - test
  - optimize
  - register
  - deploy

train:
  stage: train
  script:
    - python train.py --config config.yaml
    - python validate.py --threshold 0.90
  artifacts:
    paths:
      - model.pth
      - metrics.json

test:
  stage: test
  script:
    - pytest tests/test_model.py
    - python benchmark.py --latency-threshold 100
  dependencies:
    - train

optimize:
  stage: optimize
  script:
    - python quantize.py --model model.pth
    - python export_onnx.py
    - python validate_onnx.py
  artifacts:
    paths:
      - model_quantized.pth
      - model.onnx

register:
  stage: register
  script:
    - python register_model.py --name $CI_COMMIT_TAG
  only:
    - tags

deploy:
  stage: deploy
  script:
    - python deploy_triton.py --version $CI_COMMIT_TAG
  when: manual
  only:
    - tags

Impact

  • ⚡ 10x faster model deployment (weeks → days)
  • 🔒 100% model lineage tracking
  • 📊 Centralized monitoring across all models
  • 🤝 Improved collaboration between teams
  • ♻️ Model reuse increased by 40%

8.5 Best Practices Checklist

Model Development

  • ☑️ Use version control (Git) for code
  • ☑️ Track experiments with MLflow/W&B
  • ☑️ Document hyperparameters and configurations
  • ☑️ Save training logs and metrics
  • ☑️ Use reproducible random seeds

Model Export & Conversion

  • ☑️ Export to framework-agnostic format (ONNX)
  • ☑️ Validate numerical equivalence after conversion
  • ☑️ Test on representative data samples
  • ☑️ Document conversion steps and parameters
  • ☑️ Include fallback for unsupported operations

Model Optimization

  • ☑️ Benchmark before optimization (baseline)
  • ☑️ Apply quantization with calibration data
  • ☑️ Measure accuracy impact of each optimization
  • ☑️ Profile inference performance
  • ☑️ Document size/speed/accuracy trade-offs

Model Registry

  • ☑️ Use semantic versioning (MAJOR.MINOR.PATCH)
  • ☑️ Include comprehensive model cards
  • ☑️ Tag models with metadata (task, dataset, metrics)
  • ☑️ Implement access control and permissions
  • ☑️ Maintain staging → production workflow

Model Deployment

  • ☑️ Containerize serving infrastructure (Docker)
  • ☑️ Implement health checks and readiness probes
  • ☑️ Set up monitoring and alerting
  • ☑️ Use load balancing and auto-scaling
  • ☑️ Plan for rollback and disaster recovery

Production Monitoring

  • ☑️ Track latency, throughput, error rates
  • ☑️ Monitor model performance (accuracy drift)
  • ☑️ Log predictions for debugging
  • ☑️ Set up alerts for anomalies
  • ☑️ Implement A/B testing infrastructure

8.6 Common Pitfalls and Solutions

Pitfall Impact Solution
Skipping numerical validation after conversion Silent accuracy degradation Always compare outputs with tolerance checks
Hardcoding model paths Difficult to version/deploy Use configuration files and environment variables
No model versioning Cannot track or rollback Implement semantic versioning from day 1
Missing model cards Poor reproducibility Automate model card generation in pipeline
Deploying without monitoring Production issues go unnoticed Set up comprehensive monitoring before launch
Over-optimizing for speed Accuracy degradation Define acceptable accuracy thresholds
