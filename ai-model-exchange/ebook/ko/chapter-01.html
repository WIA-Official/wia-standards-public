1.1 머신러닝 배포의 진화

머신러닝 모델 배포의 여정은 지난 10년 동안 크게 발전했습니다. 현대 딥러닝 초기(2012년경)에는 연구자들이 주로 단일 프레임워크(Caffe 또는 PyTorch)를 사용했으며, 학습된 것과 동일한 인프라에 모델을 배포했습니다. 이러한 동질적인 환경은 배포를 단순화했지만 유연성과 협업을 제한했습니다.

AI 생태계가 성숙함에 따라 몇 가지 과제가 나타났습니다:

  • 프레임워크 확산: 팀마다 특정 요구사항과 선호도에 따라 다른 프레임워크(PyTorch, TensorFlow, JAX, MXNet)를 채택했습니다.
  • 배포 다양성: 모델이 클라우드 서버, 엣지 디바이스, 모바일 폰, 브라우저, 전문 하드웨어에서 실행되어야 했습니다.
  • 최적화 요구사항: 프로덕션 환경은 연구 프로토타입보다 더 작고 빠른 모델을 요구했습니다.
  • 협업 장벽: 다른 스택을 사용하는 팀 간 모델 공유가 점점 어려워졌습니다.

이러한 과제는 표준화된 모델 교환 형식과 프로토콜의 필요성을 야기했습니다—이 책과 WIA-AI-008 표준의 초점입니다.

1.2 모델 교환이란?

모델 교환은 다음 사이에서 머신러닝 모델을 전송하는 프로세스를 의미합니다:

  • 프레임워크: PyTorch 모델을 TensorFlow로, 또는 JAX를 ONNX로 변환
  • 환경: 학습 서버에서 프로덕션 인프라로 모델 이동
  • 조직: 레지스트리와 저장소를 통한 모델 공유
  • 플랫폼: 클라우드, 엣지, 모바일 또는 웹 환경으로 배포

모델 교환의 핵심 구성요소

구성요소 설명 예시
직렬화 모델 아키텍처와 가중치를 디스크에 저장 torch.save(), tf.saved_model.save()
변환 다른 형식 간 변환 PyTorch → ONNX → TensorFlow Lite
최적화 모델 크기 축소 및 성능 향상 양자화, 프루닝, 증류
패키징 메타데이터 및 종속성과 함께 모델 번들링 모델 카드, 버전 정보, 스키마
배포 레지스트리와 프로토콜을 통한 모델 공유 Hugging Face Hub, TensorFlow Hub, MLflow
서빙 추론을 위한 모델 배포 TorchServe, TF Serving, Triton

1.3 모델 교환이 중요한 이유

적절한 모델 교환 관행을 이해하고 구현하면 ML 수명주기 전반에 걸쳐 상당한 이점을 제공합니다:

협업과 생산성

연구자가 PyTorch에서 모델을 학습하지만 프로덕션 팀이 TensorFlow를 사용하는 경우, 원활한 변환을 통해 더 빠른 반복과 배포가 가능합니다. 팀은 사일로를 만들지 않고 각 작업에 가장 적합한 도구를 선택할 수 있습니다.

성능 최적화

서로 다른 배포 대상은 매우 다른 리소스 제약을 가집니다. 데이터센터 GPU에서 실행되는 모델은 스마트폰의 동일한 모델보다 100배 더 많은 메모리와 컴퓨팅을 가질 수 있습니다. 모델 교환은 양자화, 프루닝, 아키텍처 검색과 같은 기술을 통해 각 대상에 대한 최적화를 가능하게 합니다.

재현성과 버전 관리

적절한 모델 패키징에는 학습 데이터, 하이퍼파라미터, 프레임워크 버전, 성능 지표에 대한 메타데이터가 포함됩니다. 이를 통해 재현 가능한 과학과 모델이 프로덕션에서 예기치 않게 동작할 때 더 쉬운 디버깅이 가능합니다.

오픈 사이언스와 민주화

弘益人間(널리 인간을 이롭게 하라)의 철학은 개방형 모델 공유에 구현됩니다. Hugging Face와 같은 플랫폼은 최첨단 모델에 대한 액세스를 민주화하여 전 세계 연구자들이 서로의 작업을 기반으로 구축할 수 있게 했습니다.

1.5 모델 교환의 과제

프레임워크 비호환성

서로 다른 프레임워크는 다른 계산 그래프, 연산자 세트, 데이터 형식을 사용합니다. PyTorch의 모든 연산이 TensorFlow에 직접적으로 대응하는 것은 아니며, 그 반대도 마찬가지입니다. 사용자 정의 연산과 동적 제어 흐름은 특히 어려운 과제를 제시합니다.

정확도 저하

형식 간 변환 또는 최적화 적용은 수치 오류를 도입할 수 있습니다. FP32에서 INT8로의 양자화는 정확도를 1-3% 감소시킬 수 있습니다. 변환 후 허용 가능한 정확도를 보장하려면 신중한 검증이 필요합니다.

메타데이터 손실

단순한 직렬화 형식은 가중치와 아키텍처만 저장하고 학습 데이터, 전처리 파이프라인, 예상 입력 형식에 대한 중요한 정보를 잃습니다. 포괄적인 모델 카드가 이 과제를 해결합니다.

1.6 WIA-AI-008 표준

WIA-AI-008 표준은 4단계를 통해 이러한 과제를 해결하는 모델 교환을 위한 포괄적인 프레임워크를 제공합니다:

1단계: 데이터 형식 및 패키징
모델 직렬화, 메타데이터 스키마, 모델 카드 형식을 표준화합니다. 모델이 자체 문서화되고 재현 가능하도록 보장합니다.
2단계: API 및 변환 알고리즘
크로스 프레임워크 변환 및 최적화를 위한 API를 정의합니다. 일반적인 변환 경로에 대한 참조 구현을 제공합니다.
3단계: 교환 프로토콜 및 레지스트리
모델 배포, 인증, 액세스 제어를 위한 프로토콜을 지정합니다. 중앙집중식 또는 연합형 모델 레지스트리를 가능하게 합니다.
4단계: 통합 및 모델 서빙
배포 인프라, 모니터링, 프로덕션 모범 사례를 다룹니다. 모델이 대규모로 안정적으로 실행되도록 보장합니다.

1.7 범용 형식: ONNX

Open Neural Network Exchange (ONNX) 형식은 모델 교환에서 중심 역할을 합니다. 2017년 Microsoft와 Facebook(현 Meta)이 만든 ONNX는 신경망을 위한 프레임워크 독립적인 중간 표현을 제공합니다.

왜 ONNX인가?

  • 프레임워크 독립적: 모든 프레임워크에서 내보내고, 모든 프레임워크로 가져오기
  • 광범위한 연산자 지원: 일반적인 딥러닝 연산의 95% 이상 커버
  • 최적화된 추론: ONNX Runtime은 고도로 최적화된 실행 제공
  • 업계 채택: 모든 주요 프레임워크와 클라우드 제공업체의 지원
  • 확장성: 특수 사용 사례를 위한 사용자 정의 연산자 추가 가능

1.8 모델 카드: 문서화 표준

2019년 Google 연구원들이 도입한 모델 카드는 머신러닝 모델에 대한 구조화된 문서를 제공합니다. 투명성, 책임성, 재현성을 다룹니다.

필수 모델 카드 구성요소

  • 모델 세부사항: 아키텍처, 버전, 저자, 라이선스
  • 의도된 사용: 주요 사용 사례 및 범위 외 응용
  • 요인: 관련 인구통계학적, 환경적 또는 기술적 요인
  • 메트릭: 다양한 데이터셋 및 하위 그룹의 성능 지표
  • 학습 데이터: 데이터셋 소스, 전처리, 제한사항
  • 평가 데이터: 테스트 세트 및 평가 방법론
  • 윤리적 고려사항: 알려진 편향, 공정성 문제, 개인정보 보호 영향
  • 주의사항 및 권장사항: 제한사항 및 배포 지침
