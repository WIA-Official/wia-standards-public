Chapter 5: Phase 2 - API Interface & Game Mechanics

REST API Architecture

Phase 2 specifies RESTful APIs that enable educational games to interact with learning platforms, content repositories, and analytics systems. These APIs follow REST principles for simplicity, scalability, and broad compatibility.

API Design Principles

  • Resource-Oriented: URLs represent resources, not actions
  • Stateless: Each request contains all necessary context
  • Standard HTTP Methods: GET, POST, PUT, PATCH, DELETE
  • JSON Responses: All responses use Phase 1 JSON schemas
  • HATEOAS: Responses include links to related resources
  • Versioned: API version in URL path (e.g., /v1/games)

Authentication and Authorization

Secure authentication protects student data and ensures only authorized clients can access learning information. The standard supports multiple authentication methods for different deployment scenarios.

Game Catalog APIs

Catalog APIs enable discovery and selection of educational games. Filtering, searching, and recommendation capabilities help learners find appropriate games for their level and interests.

Key Endpoints

GET /v1/games
  - List all available games
  - Query params: subject, grade, difficulty, language

GET /v1/games/{gameId}
  - Get detailed game information
  - Returns Phase 1 game metadata

GET /v1/games/search?q=algebra
  - Full-text search across game catalog
  - Returns ranked results

GET /v1/games/recommendations/{playerId}
  - Personalized game recommendations
  - Based on learning profile and history

Player Management APIs

Player APIs handle user registration, profile management, and preference settings. Privacy controls ensure students and parents have visibility and control over data collection.

Progress Synchronization APIs

Progress APIs enable saving and retrieving game state, ensuring students can continue their learning journey across devices and sessions. Conflict resolution handles simultaneous updates from multiple devices.

Progress API Examples

POST /v1/progress
  - Save current game progress
  - Request body: Phase 1 progress schema
  - Returns: Sync confirmation with timestamp

GET /v1/progress/{playerId}/{gameId}
  - Retrieve latest progress for player/game
  - Returns: Current progress state

GET /v1/progress/{playerId}
  - Get progress across all games
  - Returns: Array of progress objects

Achievement and Badge APIs

Achievement APIs award badges, verify credentials, and enable learners to showcase their accomplishments. Integration with verifiable credentials allows achievements to be recognized beyond the original platform.

Achievement Operations

GET /v1/achievements/{playerId}
  - List all achievements earned by player

GET /v1/achievements/{achievementId}/verify
  - Verify authenticity of achievement credential

Learning Analytics APIs

Analytics APIs collect detailed learning event data while preserving student privacy. Aggregated insights help teachers and administrators understand learning patterns without exposing individual student data inappropriately.

Analytics Event Types

  • Session Events: Game start, pause, resume, complete
  • Performance Events: Challenge attempts, successes, failures
  • Learning Events: Skill demonstrations, mastery achievements
  • Engagement Events: Time-on-task, hint usage, replay behavior
  • Social Events: Multiplayer interactions, peer help given/received

Analytics Submission Example

POST /v1/analytics/events
Content-Type: application/json

{

      "outcome": "success",
      "score": 10,
      "duration": 85,
      "hintsUsed": 1
    }
  ]
}

Error Handling and Status Codes

Standardized error responses enable clients to handle failures gracefully and provide meaningful feedback to users. All errors follow a consistent JSON structure.

Standard Error Response

{
  "error": {
    "code": "INVALID_PROGRESS_DATA",
    "message": "Progress data failed validation",
    "details": "Field 'currentLevel' exceeds totalLevels",
    "timestamp": "2025-12-25T14:30:00Z",
    "requestId": "req_12345"
  }
}

Rate Limiting and Performance

To ensure system stability and fair resource allocation, API requests are rate-limited. Clients receive headers indicating their current usage and limits.

Rate Limit Headers

X-RateLimit-Limit: 1000 (requests per hour)
X-RateLimit-Remaining: 847
X-RateLimit-Reset: 2025-12-25T15:00:00Z

API Versioning and Deprecation

APIs evolve over time while maintaining backward compatibility. Version numbers in URLs enable smooth transitions when breaking changes are necessary.

  • Current version: v1 (stable, supported)
  • Beta features: v2-beta (preview, may change)
  • Deprecated: v0 (legacy, removal scheduled)
  • Deprecation notice period: minimum 12 months
  • Migration guides provided for all breaking changes
