Chapter 03

Template Security

Protecting biometric templates through cryptographic techniques and secure storage

The Template Security Challenge

Unlike passwords, biometric characteristics cannot be changed if compromised. Once a biometric template is stolen, the associated biometric trait is permanently compromised across all systems using that modality. This fundamental difference demands robust security measures specifically designed for biometric data protection.

Template security encompasses the entire lifecycle of biometric data, from initial capture through enrollment, storage, transmission, matching, and eventual deletion. Each stage presents unique vulnerabilities that must be addressed through appropriate technical and organizational controls.

Critical Security Consideration

Biometric templates must be treated as highly sensitive personal data. Unlike passwords, stolen biometric data cannot be reset. A compromised fingerprint template means that fingerprint is permanently vulnerable to impersonation across all systems, potentially for the lifetime of the individual.

Template Protection Approaches

ISO/IEC 24745 defines a framework for biometric template protection with three primary approaches:

🔐
Cancelable Biometrics

Intentionally distorted templates that can be revoked and reissued

🔑
Biometric Cryptosystems

Binding biometrics with cryptographic keys

🛡️
Secure Computation

Matching without exposing template data

Cancelable Biometrics

Cancelable biometrics applies a non-invertible transformation to the original biometric template before storage. If compromised, a new template can be generated by applying a different transformation, effectively "canceling" the old template.

Transformation Techniques

  • Cartesian Transformation: Divides fingerprint image into blocks and applies random translations
  • Polar Transformation: Converts coordinates to polar system and applies radial/angular distortions
  • Surface Folding: Maps biometric features onto a randomly generated surface
  • Bloom Filter: Converts template to a set membership structure that cannot be reversed
Cancelable Biometric Transformation
===================================

Original Template: T
Transformation Key: K (user-specific, stored separately)
Transformation Function: F (non-invertible)

Protected Template: T' = F(T, K)

Requirements:
├── Non-invertibility: Cannot recover T from T'
├── Diversity: Different K produces unlinkable T'
├── Accuracy: Matching performance minimally degraded
└── Revocability: New K generates new valid T'

Example (Biohashing):
1. Generate random projection matrix R from K
2. Project template: v = R × T
3. Binarize: T'[i] = 1 if v[i] > threshold else 0
Fuzzy Vault Scheme

The fuzzy vault is a biometric cryptosystem that binds a cryptographic key to biometric features. The key can only be retrieved by presenting a biometric sample sufficiently similar to the enrolled template.

Fuzzy Vault Construction
========================

ENROLLMENT:
1. Extract genuine minutiae points: G = {g1, g2, ..., gn}
2. Generate secret polynomial: p(x) = s + a1*x + a2*x^2 + ...
   (coefficients encode the secret key s)
3. For each gi, compute point on polynomial: (gi, p(gi))
4. Generate chaff points: C = {(c1, r1), (c2, r2), ...}
   where ri ≠ p(ci)
5. Store vault: V = G_points ∪ C

AUTHENTICATION:
1. Extract query minutiae: Q = {q1, q2, ..., qm}
2. Find overlapping points in vault
3. Use polynomial interpolation on matching points
4. If sufficient matches, recover polynomial and key

Security Properties:
├── Genuine points hidden among chaff
├── Must match enough minutiae to interpolate
└── Computationally infeasible to guess without biometric
Fuzzy Commitment Scheme

Fuzzy commitment combines error-correcting codes with biometric data to create a secure binding between biometrics and cryptographic keys. The scheme tolerates small variations in biometric samples through the error-correction capability.

Fuzzy Commitment Scheme
=======================

ENROLLMENT:
1. Extract biometric feature vector: b (binary string)
2. Generate random codeword: c from error-correcting code
3. Compute commitment: δ = b ⊕ c (XOR)
4. Store: (δ, h(c)) where h is cryptographic hash

AUTHENTICATION:
1. Extract query feature vector: b'
2. Compute: c' = b' ⊕ δ
3. Decode c' using error-correcting code → c_decoded
4. If h(c_decoded) == h(c), authentication succeeds

Error Tolerance:
├── If Hamming(b, b') < t (correction threshold)
├── Then c_decoded = c (successful recovery)
└── Key derived from c provides authentication

Secure Multi-Party Computation

Secure Multi-Party Computation (SMPC) enables biometric matching without either party revealing their private data. This approach is particularly valuable for centralized biometric databases where the server should not have access to plaintext templates.

Secure Biometric Matching Protocols
===================================

1. HOMOMORPHIC ENCRYPTION APPROACH
   ┌─────────────┐                    ┌─────────────┐
   │   Client    │                    │   Server    │
   │ (Prover)    │                    │ (Verifier)  │
   └──────┬──────┘                    └──────┬──────┘
          │                                  │
          │ 1. Encrypt probe template        │
          │    E(probe) ─────────────────────▶
          │                                  │
          │            2. Compute on encrypted data
          │               E(similarity) = f(E(probe), E(enrolled))
          │                                  │
          │ 3. Return encrypted result       │
          │◀───────────────────── E(result)  │
          │                                  │
          │ 4. Decrypt to get match score    │
          │    result = D(E(result))         │

2. GARBLED CIRCUITS APPROACH
   - Convert matching algorithm to boolean circuit
   - One party garbles circuit, other evaluates
   - Neither learns intermediate values

3. SECRET SHARING APPROACH
   - Split template into n shares
   - Any t shares insufficient to reconstruct
   - Matching computed on shares without reconstruction

Storage Architecture Options

Storage Model Description Advantages Disadvantages
On-Device Template stored on user's personal device User control, privacy, no central target Device loss, limited cross-device use
Smart Card Template stored on tamper-resistant card Portable, secure element protection Card loss, reader requirements
Central Database Templates in organizational server Easier management, 1:N identification High-value target, privacy concerns
Distributed Template split across multiple servers No single point of compromise Complexity, availability requirements

Encryption Requirements

WIA-BIOMETRIC_AUTH mandates the following encryption requirements for template protection:

Template Encryption Specifications
==================================

AT REST:
├── Algorithm: AES-256-GCM (authenticated encryption)
├── Key Derivation: PBKDF2 or Argon2id
├── Key Storage: Hardware Security Module (HSM) or TEE
└── Access Control: Role-based with audit logging

IN TRANSIT:
├── Protocol: TLS 1.3 minimum
├── Cipher Suites: AEAD only (GCM, ChaCha20-Poly1305)
├── Certificate: X.509 with strong validation
└── Perfect Forward Secrecy: Required (ECDHE)

KEY MANAGEMENT:
├── Key Rotation: Annual or upon compromise indication
├── Key Escrow: Prohibited for template encryption keys
├── Key Hierarchy: Separate keys per user or application
└── Key Destruction: Cryptographic erasure supported

TEMPLATE LIFECYCLE:
├── Creation: Encrypted immediately after extraction
├── Storage: Always encrypted, never in plaintext
├── Comparison: Decrypt in secure enclave only
└── Deletion: Cryptographic shredding with key destruction
Best Practice: Defense in Depth

Template security should employ multiple layers of protection. Combine cancelable transformations with encryption, use secure enclaves for matching operations, implement strict access controls, and maintain comprehensive audit logs. No single security measure is sufficient; defense in depth provides resilience against partial compromises.

Hardware Security Integration

Modern devices provide hardware-backed security features that significantly enhance template protection:

  • Trusted Execution Environment (TEE): Isolated processing environment (ARM TrustZone, Intel SGX) for sensitive biometric operations
  • Secure Element (SE): Dedicated tamper-resistant chip for key storage and cryptographic operations
  • Hardware Security Module (HSM): Enterprise-grade key management with FIPS 140-2/3 certification
  • Trusted Platform Module (TPM): Hardware root of trust for platform integrity verification
