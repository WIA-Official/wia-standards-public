장 요약

핵심 요점:

  1. HTTP Accept, Content-Type 및 Accept-Encoding 헤더를 통한 콘텐츠 협상은 클라이언트와 서버가 메시지 형식(JSON, XML, protobuf), 압축(gzip, Brotli) 및 언어(en, ja, ko)에 합의할 수 있게 하여 단일 엔드포인트로 다국어 클라이언트를 지원합니다.
  2. TLS 핸드셰이크 중 ALPN(Application-Layer Protocol Negotiation)은 프로토콜 버전 협상(HTTP/1.1, HTTP/2, HTTP/3)을 가능하게 하여 서버가 동일한 포트에서 여러 프로토콜을 지원할 수 있도록 하면서 클라이언트가 최상의 상호 지원 버전으로 투명하게 업그레이드할 수 있도록 합니다.
  3. API 버전 관리 전략(URI 버전 관리, 헤더 버전 관리, 미디어 유형 버전 관리)은 API 진화 중 하위 호환성을 가능하게 하며 의미 체계 버전 관리(major.minor.patch)는 중단 변경, 기능 추가 및 버그 수정을 나타내어 클라이언트 업그레이드 계획을 안내합니다.
  4. 서비스 디스커버리 프로토콜(DNS-SD, Consul, etcd)은 헬스 체크, 메타데이터 기반 필터링 및 버전 인식 라우팅을 통한 동적 엔드포인트 디스커버리를 가능하게 하여 클라이언트가 하드코딩된 엔드포인트 없이 적절한 서비스 버전을 발견하고 연결할 수 있도록 합니다.
  5. 기능 플래그 및 기능 협상은 OPTIONS 요청, 응답 헤더 또는 서비스 메타데이터를 통해 서버 기능을 광고하여 점진적 기능 롤아웃을 허용하며 클라이언트가 버전 결합 없이 사용 가능한 기능을 감지하고 적응할 수 있도록 합니다.
  6. 프로토콜 폴백 체인(HTTP/3 → HTTP/2 → HTTP/1.1)은 선호하는 프로토콜이 실패할 때 복원력을 제공하며 클라이언트 재시도 논리, 타임아웃 구성 및 연결 캐싱을 통해 서비스 가용성을 유지하면서 최적의 프로토콜 선택을 보장합니다.
  7. GraphQL 스키마 인트로스펙션은 런타임 기능 디스커버리를 가능하게 하여 클라이언트가 사용 가능한 타입, 필드 및 뮤테이션을 동적으로 쿼리할 수 있도록 하며 스키마 진화, 더 이상 사용되지 않는 경고 및 라이브 스키마에서 타입 안전 클라이언트 코드 생성을 지원합니다.

복습 질문

  1. 여러 메시지 형식(JSON, XML, protobuf, MessagePack) 및 압축 알고리즘(gzip, Brotli, LZ4)을 지원하는 콘텐츠 협상 시스템을 설계하십시오. HTTP 헤더 파싱(Accept, Content-Type, Accept-Encoding), 품질 값(q-value) 평가 및 형식/압축 선택을 구현하십시오. 지원되지 않는 형식 요청을 어떻게 처리합니까?
  2. TLS 핸드셰이크 중 ALPN(Application-Layer Protocol Negotiation)을 설명하십시오. ALPN을 사용하여 동일한 포트에서 HTTP/1.1, HTTP/2 및 HTTP/3을 지원하는 서버를 설계하십시오. 클라이언트가 프로토콜 기본 설정을 어떻게 나타냅니까? 공통 프로토콜이 지원되지 않을 때 어떻게 됩니까?
  3. API 버전 관리 전략을 비교하십시오: URI 버전 관리(/v1/users), 헤더 버전 관리(X-API-Version: 2) 및 미디어 유형 버전 관리(application/vnd.api+json;version=3). 하위 호환성, 더 이상 사용되지 않는 경고 및 정상적인 버전 종료를 지원하는 REST API에 대한 버전 관리 체계를 설계하십시오. 중단 변경을 언제 도입하시겠습니까?
  4. 헬스 체크 및 메타데이터 기반 필터링을 통해 Consul 또는 etcd를 사용하여 서비스 디스커버리를 구현하십시오. 서비스를 발견하고 버전 태그로 필터링하며 여러 인스턴스 전반에 걸쳐 재시도 논리를 구현하고 서비스 등록/등록 해제 이벤트를 처리하는 클라이언트를 설계하십시오. 서비스 버전 업그레이드 중 연쇄 장애를 어떻게 방지합니까?
  5. 기능 협상을 통해 기능 플래그 시스템을 설계하십시오. OPTIONS 요청, 사용자 지정 응답 헤더(X-Capabilities) 또는 서비스 메시 메타데이터를 통해 서버 기능 광고를 구현하십시오. 감지된 기능을 기반으로 동작을 조정하는 클라이언트를 구축하십시오. 기능 종속성 및 호환되지 않는 기능 조합을 어떻게 처리합니까?
  6. HTTP/3 → HTTP/2 → HTTP/1.1 재시도 체인을 통해 프로토콜 폴백을 구현하십시오. 타임아웃 전략, 오류 감지(연결 타임아웃, TLS 협상 실패, ALPN 불일치) 및 후속 요청에서 불필요한 재시도 오버헤드를 피하기 위한 프로토콜 캐싱을 설계하십시오. 폴백 속도와 연결 시도 비용의 균형을 어떻게 맞춥니까?
  7. 런타임 기능 디스커버리를 위한 GraphQL 인트로스펙션을 설명하십시오. __schema 및 __type 인트로스펙션 쿼리를 사용하여 스키마를 쿼리하고 쿼리 호환성을 검증하며 더 이상 사용되지 않는 필드를 감지하고 타입 안전 클라이언트 코드를 생성하는 클라이언트를 설계하십시오. 인트로스펙션이 스키마 진화 및 하위 호환성을 어떻게 지원합니까?
