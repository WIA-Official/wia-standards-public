Chapter Summary

Key Takeaways:

  1. Content negotiation through HTTP Accept, Content-Type, and Accept-Encoding headers enables clients and servers to agree on message formats (JSON, XML, protobuf), compression (gzip, Brotli), and language (en, ja, ko), supporting polyglot clients with a single endpoint.
  2. ALPN (Application-Layer Protocol Negotiation) during TLS handshake enables protocol version negotiation (HTTP/1.1, HTTP/2, HTTP/3), allowing servers to support multiple protocols on the same port while clients transparently upgrade to the best mutually supported version.
  3. API versioning strategies (URI versioning, header versioning, media type versioning) enable backward compatibility during API evolution, with semantic versioning (major.minor.patch) indicating breaking changes, feature additions, and bug fixes to guide client upgrade planning.
  4. Service discovery protocols (DNS-SD, Consul, etcd) enable dynamic endpoint discovery with health checking, metadata-based filtering, and version-aware routing, allowing clients to discover and connect to appropriate service versions without hardcoded endpoints.
  5. Feature flags and capability negotiation allow gradual feature rollout by advertising server capabilities through OPTIONS requests, response headers, or service metadata, enabling clients to detect and adapt to available features without version coupling.
  6. Protocol fallback chains (HTTP/3 → HTTP/2 → HTTP/1.1) provide resilience when preferred protocols fail, with client retry logic, timeout configuration, and connection caching ensuring optimal protocol selection while maintaining service availability.
  7. GraphQL schema introspection enables runtime capability discovery, allowing clients to query available types, fields, and mutations dynamically, supporting schema evolution, deprecation warnings, and type-safe client code generation from live schemas.

Review Questions

  1. Design a content negotiation system that supports multiple message formats (JSON, XML, protobuf, MessagePack) and compression algorithms (gzip, Brotli, LZ4). Implement HTTP header parsing (Accept, Content-Type, Accept-Encoding), quality value (q-value) evaluation, and format/compression selection. How do you handle unsupported format requests?
  2. Explain ALPN (Application-Layer Protocol Negotiation) during TLS handshake. Design a server that supports HTTP/1.1, HTTP/2, and HTTP/3 on the same port using ALPN. How does the client indicate protocol preferences? What happens when no common protocol is supported?
  3. Compare API versioning strategies: URI versioning (/v1/users), header versioning (X-API-Version: 2), and media type versioning (application/vnd.api+json;version=3). Design a versioning scheme for a REST API that supports backward compatibility, deprecation warnings, and graceful version sunset. When would you introduce breaking changes?
  4. Implement service discovery using Consul or etcd with health checking and metadata-based filtering. Design a client that discovers services, filters by version tag, implements retry logic across multiple instances, and handles service registration/deregistration events. How do you prevent cascading failures during service version upgrades?
  5. Design a feature flag system with capability negotiation. Implement server capability advertisement through OPTIONS requests, custom response headers (X-Capabilities), or service mesh metadata. Build a client that adapts behavior based on detected capabilities. How do you handle feature dependencies and incompatible capability combinations?
  6. Implement protocol fallback with HTTP/3 → HTTP/2 → HTTP/1.1 retry chain. Design timeout strategies, error detection (connection timeout, TLS negotiation failure, ALPN mismatch), and protocol caching to avoid unnecessary retry overhead on subsequent requests. How do you balance fallback speed with connection attempt cost?
  7. Explain GraphQL introspection for runtime capability discovery. Design a client that queries schema using __schema and __type introspection queries, validates query compatibility, detects deprecated fields, and generates type-safe client code. How does introspection support schema evolution and backward compatibility?
