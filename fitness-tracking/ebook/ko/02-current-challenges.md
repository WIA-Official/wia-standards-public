# 2장: 피트니스 트래킹의 현재 과제

## 개요

피트니스 트래킹 기술의 폭발적 성장에도 불구하고, 생태계는 건강 결과 개선 잠재력을 제한하는 중대한 과제에 직면해 있습니다.

---

## 2.1 기기 및 플랫폼 파편화

### 폐쇄형 정원 문제

**독점 생태계**
- 각 제조업체는 맞춤형 데이터 형식 사용
- 특정 기기 브랜드에 묶인 앱
- 제한된 크로스 플랫폼 호환성
- 벤더 종속으로 사용자 선택 감소

**실제 영향:**
```
사용자 스토리: 사라의 딜레마
사라는 3년 동안 A 브랜드의 피트니스 워치를 사용하며
귀중한 건강 데이터를 축적했습니다. 더 나은 기능을 가진
B 브랜드의 최신 기기로 전환하고 싶지만:
- 기록 데이터를 전송할 수 없음
- 3년간의 추세와 인사이트 손실
- 새 기기로 처음부터 시작해야 함
- 열등한 기능에도 불구하고 A 브랜드에 머물러야 함
```

### 호환되지 않는 데이터 형식

**형식 확산**
- 독점 바이너리 형식
- 맞춤형 JSON 스키마
- 일관되지 않은 필드 이름
- 다양한 단위 시스템

**예시 불일치:**
```json
// 브랜드 A 형식
{
  "distance_km": 5.2,
  "duration_min": 32,
  "avg_pace": "6:09"
}

// 브랜드 B 형식
{
  "distanceInMeters": 5200,
  "durationSeconds": 1920,
  "paceMinPerKm": 6.15
}
```

모두 동일한 활동을 설명하지만 변환 없이는 호환되지 않습니다.

---

## 2.2 데이터 사일로 및 이동성 부족

### 갇힌 데이터

**제한된 내보내기 옵션**
- 많은 플랫폼이 내보내기 기능을 제공하지 않음
- 내보내기 형식이 불완전하거나 독점적
- 기록 데이터 액세스 제한

**데이터 소유권 혼란**
```
일반적인 질문:
❓ "내 심박수 데이터는 누가 소유하나요 - 나 아니면 기기 제조업체?"
❓ "구독을 취소하면 운동 기록을 얻을 수 있나요?"
❓ "회사가 폐업하면 내 데이터는 어떻게 되나요?"
```

---

## 2.3 정확도 및 검증 문제

### 일관되지 않은 계산

**칼로리 불일치**

예시: 동일한 5km 달리기, 다른 앱:
- **앱 A:** 387 칼로리 소모
- **앱 B:** 512 칼로리 소모
- **앱 C:** 441 칼로리 소모

동일한 활동에 대해 **32% 차이**!

**발생 이유:**
- 다른 BMR 공식
- 다른 MET 값
- 다른 심박수 가중치 요소
- 다른 EPOC 계산

### 심박수 구역 혼란

**여러 구역 시스템:**

```
시스템 A (5-구역):
구역 1: 최대 심박수의 50-60%
구역 2: 최대 심박수의 60-70%
구역 3: 최대 심박수의 70-80%
구역 4: 최대 심박수의 80-90%
구역 5: 최대 심박수의 90-100%

시스템 B (7-구역):
구역 1-7: 다른 범위

시스템 C (3-구역):
지방 연소: 최대 심박수의 50-70%
유산소: 최대 심박수의 70-85%
최대: 최대 심박수의 85-100%
```

### 걸음 수 정확도

**걸음 감지의 큰 차이:**
- 걷기: 일반적으로 ±5-15% 오차
- 달리기: 일반적으로 ±3-10% 오차
- 걷지 않는 움직임: 20-50% 거짓 양성

---

## 2.4 개인정보 보호 및 보안 우려

### 데이터 수집 불투명성

**알 수 없는 데이터 관행:**
- 어떤 데이터가 수집되나요?
- 어디에 저장되나요?
- 누가 액세스하나요?
- 얼마나 오래 보관되나요?

**일반적인 사용자 우려:**
```
설문조사 결과 (n=5,000 피트니스 트래커 사용자):
78% - 위치 데이터 개인정보 보호 우려
65% - 건강 데이터 침해 우려
54% - 회사를 신뢰하지 않음
43% - 더 나은 개인정보 제어를 위해 전환할 의향
```

### 보안 취약점

**최근 사건:**

**2023: FitHealth 데이터 침해**
- 6,100만 사용자 계정 손상
- GPS 경로, 집 주소 노출
- 집단 소송 제기

**2022: TrackMe API 노출**
- 보안되지 않은 API 엔드포인트 발견
- 실시간 위치 추적 가능
- 200만 명의 사용자 영향

## 2.5 표준화된 지표 부족

### 독점 점수

**공개되지 않은 알고리즘:**

각 플랫폼에는 고유한 점수 시스템이 있습니다:
- **준비도 점수** (플랫폼 A)
- **회복 지수** (플랫폼 B)
- **피트니스 수준** (플랫폼 C)

**문제:**
- 기기 간 비교 불가
- 과학적 검증 없음
- 불투명한 계산 방법

### 트레이닝 부하 불일치

**여러 호환되지 않는 시스템:**

```typescript
// 플랫폼 A: 단순 지속 시간 기반
trainingLoad = duration_minutes * intensity_factor

// 플랫폼 B: 심박수 기반 TRIMP
trainingLoad = duration * HR_ratio * exp(k * HR_ratio)

// 플랫폼 C: 파워 기반 TSS
trainingLoad = (duration * NP * IF) / (FTP * 36)
```

## 2.6 통합 복잡성

### API 파편화

**개발자 과제:**

상위 10개 피트니스 플랫폼을 지원하려면:
- 10개의 다른 OAuth 흐름 구현
- 10개의 다른 데이터 형식 파싱
- 10개의 다른 속도 제한 처리
- 10개의 다른 API 클라이언트 유지
- 10개의 다른 API 요금 지불

**개발 비용:**
- 플랫폼당 약 **200-500시간** 추정
- 플랫폼당 **$20,000-$50,000** 개발 비용

---

## 2.7 사용자 경험 파편화

### 일관되지 않은 인터페이스

**동일한 데이터, 다른 표현:**

심박수 구역 표시:
- 백분율 (50-60%, 60-70% 등)
- 절대 BPM (100-120, 120-140 등)
- 이름 (회복, 유산소, 역치 등)
- 색상 (회색, 파란색, 녹색, 주황색, 빨간색)

### 지표 과부하

**압도적인 정보:**

한 번의 달리기에 표시될 수 있는 항목:
- 지속 시간, 거리, 페이스
- 평균 심박수, 최대 심박수, 심박수 구역
- 칼로리, 활동 칼로리
- 걸음, 케이던스, 보폭
- 고도 상승, 고도 하강
- VO2 Max, 트레이닝 효과, 회복 시간

한 활동에 대해 **50개 이상의 지표**가 사용자를 압도합니다.

---

## 2.8 비즈니스 모델 충돌

### 구독 페이월

**기능 제한:**
- 기본 지표 무료
- 고급 분석 유료
- 기록 데이터 제한
- 페이월 뒤에 잠긴 내보내기

**사용자 불만:**
"기기에 $300을 지불했는데 내 데이터를 보기 위해 왜 월 $10 구독이 필요한가요?"

---

## 2.9 의료 통합 장벽

### 임상 회의론

**의사 우려:**
- 검증되지 않은 정확도
- 소비자 기기에 대한 FDA 승인 없음
- 책임 위험
- 데이터를 기반으로 처방할 수 없음

---

## 2.10 연구 제한

### 데이터 품질 문제

**연구 과제:**
- 일관되지 않은 데이터 수집 방법
- 기기별 편향
- 자가 선택 편향

---

## 2.11 WIA-IND-012의 경우

### 표준화가 도움이 되는 방법

| 과제 | WIA-IND-012 솔루션 |
|------|-------------------|
| 기기 파편화 | 통합 데이터 형식 |
| 데이터 사일로 | 표준 내보내기/가져오기 |
| 정확도 차이 | 검증된 공식 |
| 개인정보 우려 | 명확한 개인정보 프레임워크 |
| 통합 복잡성 | 단일 API 사양 |

## 핵심 요점

✓ 기기 및 플랫폼 파편화가 사용자 데이터를 독점 생태계에 가둠

✓ 15-30%의 정확도 차이가 신뢰할 수 없는 지표와 사용자 혼란 생성

✓ 개인정보 보호 및 보안 사건으로 피트니스 플랫폼에 대한 사용자 신뢰 저하

✓ WIA-IND-012는 표준화를 통해 이러한 체계적 과제 해결
