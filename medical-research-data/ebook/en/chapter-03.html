Chapter 3: Data Collection and Documentation

Standards for Capturing, Annotating, and Describing Research Data

3.1 Data Management Planning

Effective data collection begins before a single data point is captured. A Data Management Plan (DMP) documents how data will be collected, organized, documented, stored, and shared throughout the research lifecycle. Most major funders now require DMPs as part of grant applications.

The DMP is a living document that evolves with the research project. It should be revisited at project milestones and updated as data management needs change or new requirements emerge.

DMP Tools

Several tools facilitate DMP creation: DMPTool (US) and DMPonline (UK) provide templates aligned with funder requirements. DMP Assistant (Canada) includes bilingual support. These tools generate structured DMPs, track versions, and can export to funder-required formats. Many institutions provide local DMP templates and support services.

3.2 Data Collection Standards

Standardized data collection ensures consistency, quality, and interoperability:

Domain-Specific Standards

Domain Standard Application
Clinical Trials CDISC (CDASH, SDTM, ADaM) Case report forms, submission datasets
Observational OMOP CDM, PCORnet CDM Real-world data harmonization
Genomics GA4GH schemas, VCF, BAM Sequence data, variant calling
Imaging DICOM, BIDS, NIfTI Medical imaging, neuroimaging
Biobanking MIABIS, SPREC Sample collection and processing
Survey/PRO PROMIS, SF-36, validated instruments Patient-reported outcomes

CDISC Standards Suite

CDISC (Clinical Data Interchange Standards Consortium) provides the most comprehensive standards for clinical research data. CDASH standardizes case report form content. SDTM defines submission dataset structure required by FDA and EMA. ADaM specifies analysis-ready datasets. ODM-XML enables electronic data capture exchange. Adoption of CDISC standards is required for regulatory submissions and increasingly expected by journals and repositories.

3.3 Metadata Standards

Metadata—data about data—is essential for discovery, understanding, and reuse:

Metadata Levels

  • Descriptive: Title, creators, subject, keywords, abstract
  • Technical: File format, size, software, creation date
  • Administrative: Rights, license, access conditions
  • Provenance: Collection methods, processing steps, transformations
  • Structural: Relationships between files, components

Metadata Schemas

Schema Scope Use Case
Dublin Core General descriptive Basic dataset description
DataCite Research data citation DOI metadata, repository deposit
DDI Social/health surveys Survey data documentation
ISA-Tab Life sciences experiments Study/assay/sample relationships
DCAT Data catalogs Data portal interoperability

3.4 Documentation Best Practices

Comprehensive documentation enables future users to understand and reuse data:

Essential Documentation

  • README Files: Overview of dataset contents, file organization, key information
  • Codebooks: Variable definitions, value labels, units, missing value codes
  • Data Dictionaries: Detailed descriptions of all data elements
  • Protocol Documentation: Methods, instruments, procedures used
  • Analysis Code: Scripts with comments explaining analytical steps
  • Change Logs: Version history, modifications made over time

README Template Structure

# Dataset Title

## Description
Brief description of the dataset

## Citation
How to cite this dataset

## Data Files
- filename1.csv: Description
- filename2.xlsx: Description

## Variables
Key variables and their definitions

## Collection Methods
How data was collected

## Processing
Data cleaning and transformation steps

## Access Conditions
License, restrictions, contact information

## Version History
Changes across dataset versions

⚠️ Documentation Debt

Delaying documentation creates "documentation debt"—the work required grows exponentially as details fade from memory. Document as you go: create README templates at project start, maintain running lab notebooks, annotate code as it's written, and update documentation with each data version. What seems obvious today will be mysterious in a year.

3.5 File Organization and Naming

Systematic organization improves findability and reduces errors:

File Naming Conventions

  • Descriptive: Include project, type, date, version information
  • Consistent: Apply same pattern across all files
  • Machine-Readable: Avoid spaces, special characters; use underscores or hyphens
  • Sortable: Use YYYYMMDD date format for chronological sorting
  • Documented: Record naming convention in project documentation
Good: 2024-01-15_project-alpha_samples_v02.csv
Bad:  data final FINAL (2).csv

Good: genomics/processed/patient001_rnaseq_normalized.tsv
Bad:  John's analysis results!!.xlsx

✓ Data Collection Best Practices

Create DMP before data collection begins; use established domain standards for data formats; implement systematic file naming and organization; maintain rich metadata using community schemas; document continuously rather than at project end; version control all code and track data versions; and budget adequate resources for data management (2-5% of project cost).

📌 Key Takeaways

  • Data Management Plans (DMPs) document data collection, storage, sharing, and preservation plans; required by most funders
  • Domain-specific standards (CDISC, OMOP, DICOM, GA4GH) ensure consistency and interoperability
  • Metadata at multiple levels (descriptive, technical, administrative, provenance) enables discovery and reuse
  • Essential documentation includes README files, codebooks, data dictionaries, and protocol documentation
  • Systematic file naming and organization improves findability and reduces errors
  • Version control for code (Git) and data tracking ensures reproducibility and change history
