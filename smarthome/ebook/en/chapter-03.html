3.7 The four-phase split, in one paragraph each

Phase 1 (Data Format) defines the JSON envelopes that every conformant hub, device, mobile app, and assistive-technology adapter must round-trip byte-identically. Envelopes carry signed provenance so that a third-party auditor can verify which device produced which observation without trusting the hub's narrative log.

Phase 2 (API) fixes the HTTPS surface for device sync, control, and event subscription. The shape is REST plus a WebSocket event stream. The API also defines emergency-channel semantics: a separate WebSocket guarantees that fire / intrusion / fall / medical alerts do not get drowned by noisy normal events.

Phase 3 (Protocol) handles cross-fabric and offline coordination: how hubs keep working when the cloud is unreachable, how a device commissioned to two fabrics resolves command conflicts, and how mesh / star / hub-and-spoke topologies meet a single conformance bar for resident-facing reliability.

Phase 4 (Integration) bridges to Matter, Z-Wave, Apple HomeKit, Google Home, Samsung SmartThings, and to the WIA family standards (WIA-OMNI-API for credentials, WIA-AIR-SHIELD for transport, WIA-INTENT for clinician/caregiver intent lowering, the WIA accessibility profile registry).

3.9 Where this sits in the WIA family

WIA-OMNI-API holds household credentials and federated identities so residents can install a third-party adapter without disclosing their primary credentials to the adapter. WIA-AIR-SHIELD scores transports for trustworthiness so the hub can refuse to forward sensitive control commands across an unscored or downgraded transport. WIA-INTENT lets a clinician or caregiver express a high-level intent ("evening medicine reminder routine") that lowers to the underlying device commands while respecting the resident's accessibility profile.

3.10 Three failure modes the architecture explicitly defends against

First, a single vendor's exit must not strand the resident. Phase 1 envelopes are signed by the device, not by the vendor cloud, so a cloud shutdown leaves the local fabric intact. Second, a misbehaving bridge adapter must not silently rewrite event content. Bridges are constrained to the documented translation table; outputs are signed and any deviation is auditable. Third, a hub failure must not trap the resident in a state where their accessibility profile cannot be applied. Profiles are replicated to the resident's mobile device and loaded by the bridge adapter on hub recovery.
