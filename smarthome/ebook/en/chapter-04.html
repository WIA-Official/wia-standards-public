4.7 The five envelope families in detail

Phase 1 defines five envelope families. Device envelopes describe a commissioned device: vendor, model, firmware, supported clusters, accessibility-extension capabilities, and the signing key bound at PASE commissioning. Command envelopes carry a single instruction targeting one device or one group; they are signed by the issuing principal (resident, caregiver, scenario engine) and carry an idempotency key. Event envelopes describe a state change observed by the hub or a device; they carry the device's signature so an auditor can distinguish "device reported state X" from "hub claimed device reported state X." Emergency envelopes are events flagged for the emergency channel; they carry an additional severity field and an escalation chain. Profile envelopes hold the resident's accessibility profile (preferred input modalities, alert preferences, scanning timing) and are the gating record for any adapter that customises device behaviour for the resident.

4.9 Round-trip canonicalisation

The signed envelope must survive encode/decode through any conformant intermediate. Phase 1 specifies RFC 8785 JSON Canonicalisation Scheme as the canonical form: keys sorted by code point, no insignificant whitespace, numbers in shortest round-trip form. A receiver re-canonicalises before signature verification; deviation indicates a broken intermediate and the receiver MUST reject the envelope rather than guess.

4.10 Versioning of envelopes

Each envelope carries wia_smarthome_version. Within the 1.x line, additive fields are permitted and consumers must ignore unknown fields. Removed fields require a 2.x major bump and a documented deprecation window. Devices commissioned to a 1.x hub continue to function under a 2.x hub via a documented compatibility envelope translator. The translator itself is signed and is published by the WIA standards repository.
