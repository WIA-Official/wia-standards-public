8.7 Conformance test suite

A black-box conformance test suite is published at github.com/WIA-Official/wia-smarthome-conformance. The suite walks discovery, commissioning (success and PIN-rejection paths), bulk device list, control with idempotency-key collision, normal event subscribe / reconnect with Last-Event-Id, emergency event ack and escalation, group create / fire with FallbackStrategy variants, scenario run, voice command round-trip against a fixture audio bundle, offline-mode device control, multi-fabric conflict resolution, and rate-limit headers and exhaustion behaviour. Hubs publishing Gold conformance must additionally pass the bridge tests for at least two ecosystems and the accessibility-profile registry tests.

8.8 Reference implementation

A reference hub container is published as wia/smarthome-hub:1.0.0. The image ships with the HTTP and WebSocket surface implementing every Phase 2 endpoint, a SQLite store sufficient for typical household scale (up to a few hundred devices), bridges for at least Apple HomeKit and Home Assistant, and a built-in conformance test runner so deployers can self-verify before going to production. Production deployments typically swap SQLite for a small Postgres or LiteFS replicated SQLite, and add a dedicated object store for voice-command audio retention.

8.9 Deployment for a real household

A typical accessibility-aware household deployment proceeds in five steps. (1) The household installs the hub container and commissions it to the home network. (2) An accessibility consultant or self- advocate captures the resident's profile via the registry endpoint; this becomes the gating record for every adapter. (3) Devices are commissioned via PASE; the consultant verifies that critical alert modalities (visual + haptic + auditory as needed) are mirrored to the resident's preferred channels. (4) The bridge to the household's existing vendor ecosystem is enabled. (5) Emergency contacts are configured and the escalation chain is rehearsed with the resident.

8.10 Roadmap

The 1.x line is feature-stable; future minor versions will add bridges as new ecosystems emerge and add new accessibility profile fields as residents and consultants identify gaps. The 2.0 line, no earlier than 2028, anticipates a post-quantum signature suite migration aligned with the broader WIA family roadmap.

8.11 Backwards-compatibility promise

Within the 1.x line every endpoint listed in Phase 2 must remain reachable and must continue to honour the documented status codes and content shapes. Hosts may add optional query parameters, response fields, new endpoints, or media types. Hosts must not remove or repurpose existing ones. Breaking changes ride a major version bump and must be preceded by a 12-month deprecation window per IETF RFC 8594 and RFC 9745; deprecated routes must emit Deprecation and Sunset headers throughout the window.
