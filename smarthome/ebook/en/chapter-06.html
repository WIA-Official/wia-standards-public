6.7 The protocol stack from the hub's view

The hub speaks several southbound protocols (Matter over Wi-Fi, Matter over Thread, Z-Wave 800, Zigbee 3.0, Bluetooth LE for wearables) and a single northbound WIA protocol surface. The translation layer is the heart of Phase 3: it converts each southbound device's native cluster vocabulary into the canonical WIA envelope shape, and converts incoming WIA control envelopes into the southbound protocol the device actually speaks.

6.8 Multi-fabric coordination, in detail

A household frequently has at least one vendor fabric (Apple Home, Google Home, SmartThings) plus the WIA fabric. Phase 3 specifies how the WIA fabric coexists rather than competing. A device commissioned to both fabrics emits events on both; control commands from both fabrics queue at the device with a 50 ms conflict window. Within the window the most recently commissioned fabric's command takes precedence; outside the window each fabric's command is FIFO. This avoids the failure mode where two voice assistants give contradictory instructions in the same second.

6.9 Offline operation

Phase 3 requires that core household functions work without internet: local LAN device list and control, group commands, local-NLU voice commands when the hub has the model installed, and emergency alerts to LAN-connected mobile clients. Cloud-dependent functions (remote access, voice assistants without local NLU, third-party scenario sharing) gracefully degrade and resume on reconnect. A conformant hub must cache the resident's accessibility profile locally so that a cloud outage does not strand the resident in a state where their switch input or their alert preferences are misapplied.

6.10 Replay defence

Every signed envelope carries a 96-bit nonce and an RFC 3339 timestamp. Receivers reject envelopes with skew greater than ±300 seconds, reject envelopes whose (signer, nonce) tuple has been seen within the last 600 seconds, and persist the seen-nonce cache across hub restarts so a power cycle does not re-open the window for a previously-blocked replay.

6.11 Trust list maintenance

Each hub maintains a signed trust list of federated peers (vendor clouds, cross-household guest hubs, caregiver agents). Trust lists are republished at least monthly; peers refuse stale lists older than 60 days. A peer may self-publish a revocation envelope to immediately drop trust between list refresh windows. The trust list is itself an envelope and follows Phase 1 canonicalisation rules.
