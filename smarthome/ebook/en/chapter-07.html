7.7 Bridge inventory

Phase 4 specifies bridges to the four major consumer ecosystems (Apple HomeKit, Google Home, Samsung SmartThings, Amazon Alexa) plus Home Assistant for the open-source community. Each bridge is a bidirectional translator: events flow from the southbound vendor ecosystem into the WIA event stream; commands flow from the WIA control surface into the vendor's native control verbs.

7.8 What a bridge guarantees and does not guarantee

A bridge guarantees that for the device classes covered by both ecosystems (lighting, locks, thermostats, door / window sensors, motion sensors), state and control round-trip with the WIA envelope. A bridge does not guarantee that vendor-specific extensions survive (e.g. Apple's adaptive lighting curve, Google's per-device routine triggers) — those remain accessible only via the vendor's own client.

7.9 The accessibility profile registry

Phase 4 also defines the accessibility profile registry: a per-resident signed record describing preferred input modalities (single switch, AAC, voice, eye-gaze), preferred alert modalities (visual, auditory, haptic, multi-modal), scanning timing, contrast needs, and notification latency tolerance. The registry is exposed read-only to bridge adapters under explicit consent so a third-party adapter can render the resident's preferences correctly without re-asking on every device.

7.10 WIA-INTENT lowering

WIA-INTENT lets a caregiver or clinician express a high-level intent — "evening medicine reminder routine," "guided morning wake-up," "departure check-out (lock + lights off + thermostat eco)" — and have it lowered into the underlying device commands while respecting the resident's accessibility profile. The lowering is auditable: every intent execution emits a chain of envelopes that an auditor can walk from intent down to per-device command.

7.11 Bridge configuration file

A reference Apple HomeKit bridge configuration uses TOML with sections for bridge metadata, credentials (sourced from WIA-OMNI-API), the cluster-mapping table, and delivery tuning (batch size, sync interval, exponential backoff). Operators may embed vendor-specific sections; conformant bridges must ignore unknown sections rather than refuse to start. The reference configuration ships in the WIA-Official conformance repository.

Note on bridge-side caching of accessibility profiles

Bridges may cache the resident's accessibility profile under read consent for at most 24 hours. Beyond that window the bridge MUST re-fetch from the registry so that profile updates (e.g. a resident widening their alert tolerance after recovering some hearing) propagate to all bridged ecosystems within a day. The cache window is shorter than for ordinary device state because accessibility preference drift, when it happens, must reach the resident's bridges quickly to avoid surfacing stale assumptions.
