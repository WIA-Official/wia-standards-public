Chapter 4

Secure Credential Issuance

Cryptographic foundations, decentralized identifiers, and secure workflows for refugee documentation

Introduction to Secure Credential Issuance

Credential issuance is the process by which an authorized entity creates and delivers a credential to an individual. For refugee credentials, this process must be both secure and practical—secure enough to prevent fraud and ensure credential integrity, yet practical enough to operate in the challenging environments where refugees are registered. The WIA standard specifies a comprehensive issuance framework that addresses cryptographic requirements, operational security, and workflow design.

Modern credential issuance builds on decades of cryptographic research and standards development. Digital signatures enable credentials that cannot be forged or modified without detection. Decentralized identifiers enable identity anchors that do not depend on any single authority. Verifiable credentials provide standardized formats for expressing and verifying claims. The WIA standard integrates these technologies into a coherent framework optimized for humanitarian contexts.

This chapter covers the cryptographic foundations of secure credentials, the architecture of decentralized identifiers, the structure of verifiable credentials, and the operational workflows for secure issuance. Understanding these elements is essential for implementing WIA-compliant credential systems that protect both credential integrity and holder privacy.

Cryptographic Foundations

Cryptography provides the mathematical foundation for credential security. Digital signatures prove that credentials were issued by authorized entities and have not been modified. Hash functions create unique fingerprints of credential contents. Public key infrastructure manages the keys necessary for signing and verification. The WIA standard specifies cryptographic algorithms and parameters that provide appropriate security levels for refugee credentials.

Digital Signatures

Digital signatures are the cornerstone of credential security. When an issuer signs a credential, they create a mathematical proof that binds the credential content to the issuer's private key. Anyone with the issuer's public key can verify this proof, confirming both the issuer's identity and the credential's integrity. Any modification to the credential after signing invalidates the signature, making tampering detectable.

The WIA standard supports multiple signature algorithms to accommodate different system requirements and ensure future flexibility. Ed25519 signatures provide excellent performance with strong security and are recommended for most applications. ECDSA with P-256 curves ensures compatibility with existing PKI infrastructure. RSA signatures support legacy system integration where required.

Key Management

Cryptographic keys are the foundation of credential security. Private keys must be protected with extreme care—compromise of an issuer's private key would allow attackers to create fraudulent credentials indistinguishable from legitimate ones. The WIA standard specifies key management requirements covering key generation, storage, usage, rotation, and destruction.

Hardware Security Modules (HSMs) provide the strongest protection for high-value issuer keys. HSMs are tamper-resistant devices that generate keys internally and never expose private keys to software. All signing operations occur within the HSM, ensuring that even a compromised server cannot extract the keys. The WIA standard requires HSM-backed key storage for all production credential issuance systems at IAL2 and above.

Key Management Requirements

Requirement IAL1-2 IAL3-4
Key Generation Cryptographically secure RNG HSM-generated
Key Storage Encrypted, access-controlled HSM-backed
Key Usage Application-level controls HSM policy-enforced
Key Rotation Annual minimum Semi-annual or on compromise
Key Backup Encrypted offline backup HSM key ceremony

Hash Functions

Cryptographic hash functions produce fixed-size digests from arbitrary input data. These digests serve multiple purposes in credential systems: creating compact representations for signing, enabling efficient comparisons, and supporting privacy-preserving techniques. The WIA standard specifies SHA-256 as the minimum hash algorithm, with SHA-3 recommended for new implementations.

Decentralized Identifiers (DIDs)

Decentralized Identifiers provide a new type of identifier designed specifically for verifiable, decentralized digital identity. Unlike traditional identifiers assigned by central authorities (such as government ID numbers or email addresses), DIDs can be created by anyone without requiring permission from any organization. This property is essential for refugee populations who may not have access to traditional identity issuers.

DID Structure

A DID is a URI that identifies a subject and can be resolved to a DID Document containing public keys and other metadata. The DID format follows a standard structure: a scheme identifier ("did"), a method identifier (specifying how the DID operates), and a method-specific identifier (the actual identifier within that method). 

// DID Structure Example
did:wia:refugee:b94d27b9934d3e08a52e52d7da7dabfac484efe37a5380ee9088f7ace2efcde9

┌─────────┬────────────┬─────────────────────────────────────────────────────────┐
│  did:   │    wia:    │    refugee:b94d27b9934d3e08...                          │
├─────────┼────────────┼─────────────────────────────────────────────────────────┤
│ Scheme  │  Method    │  Method-Specific Identifier                             │
│         │            │  (cryptographic hash of initial registration data)       │
└─────────┴────────────┴─────────────────────────────────────────────────────────┘

DID Documents

Every DID resolves to a DID Document—a JSON-LD document containing the public keys, authentication methods, and service endpoints associated with that identity. The DID Document enables verifiers to look up the information needed to verify credentials and communicate with the identity holder.

DID Methods for Refugee Credentials

The WIA standard defines a specific DID method optimized for refugee credentials. The did:wia method provides the characteristics needed for humanitarian contexts: operation without persistent internet connectivity, protection against tracking across contexts, support for key rotation and recovery, and compatibility with existing humanitarian data systems.

DIDs can be anchored to various verifiable data registries including blockchains, distributed ledgers, and UNHCR's existing identity management infrastructure. The choice of anchor affects properties like decentralization, availability, and governance. The WIA standard supports multiple anchoring options to accommodate different organizational requirements and infrastructure constraints.

Verifiable Credentials Architecture

Verifiable Credentials (VCs) are the W3C standard for expressing credentials in a way that is cryptographically secure, privacy-respecting, and machine-verifiable. A VC contains claims about a subject, metadata about the credential itself, and a cryptographic proof. The WIA standard builds on the VC data model with extensions specific to refugee credentials.

Credential Structure

Every verifiable credential contains several required elements: context (defining the terms used), type (classifying the credential), issuer (who created it), issuance date, credential subject (who it's about), and proof (cryptographic signature). Optional elements include expiration date, credential status (for revocation checking), and additional metadata.

Credential Schemas

Each credential type is defined by a schema that specifies the required and optional claims, data types, and validation rules. Schemas enable verifiers to understand what claims a credential should contain and how to validate them. Issuance Workflow

Credential issuance follows a structured workflow that ensures proper verification before credential creation and secure delivery to the holder. The WIA standard defines issuance workflows for different credential types and assurance levels, with common elements across all workflows.

1

Registration Request

The credential subject initiates the process by presenting themselves for registration. Biometric capture establishes identity anchor. Basic biographical information is collected. Existing documentation is reviewed.

2

Identity Verification

Verification procedures appropriate to the requested assurance level are completed. Biometric deduplication checks for existing registrations. Document verification where applicable. Witness attestation or interview as required.

3

DID Generation

A DID is generated for the subject if not already registered. Key pairs are generated in a secure environment. The DID Document is created and published. The subject receives their DID and initial keys.

4

Credential Creation

The credential is constructed according to the appropriate schema. Claims are populated from verified information. Metadata including expiration and status endpoints are added. The issuer signs the credential.

5

Credential Delivery

The signed credential is delivered to the holder. Delivery may be to a mobile wallet, physical card with embedded chip, or secure cloud storage. The holder confirms receipt and capability to present the credential.

Issuer Authorization

Not every organization can issue every type of credential. The WIA standard defines an issuer authorization framework that specifies which organizations can issue which credential types. UNHCR and national refugee agencies can issue refugee status credentials. Educational institutions can issue education credentials. Healthcare providers can issue health credentials. This division of authority ensures that credentials are issued by organizations with appropriate expertise and accountability.

Credential Storage and Wallets

Once issued, credentials must be stored securely and made accessible for presentation. Digital wallets provide the interface for holders to manage their credentials, protecting them from unauthorized access while enabling convenient presentation to verifiers. The WIA standard specifies wallet requirements that ensure security and interoperability.

Wallet Architecture

Credential wallets can be implemented as mobile applications, web applications, or hardware devices. Each approach offers different trade-offs between security, convenience, and accessibility. Mobile wallets provide the best balance for most refugee populations, offering offline capability, biometric protection, and ubiquitous device availability. Hardware wallets offer stronger security for high-value credentials. Cloud wallets provide backup and multi-device access.

Wallet Security Requirements

  • Key Protection: Private keys stored in secure enclave or TEE where available
  • Access Control: Biometric or PIN authentication required for credential access
  • Encryption: All stored credentials encrypted at rest
  • Backup: Encrypted backup capability with key recovery options
  • Revocation: Ability to remotely revoke wallet access if device is lost
  • Audit Log: Record of all credential presentations for holder review

Offline Capability

Refugees often operate in environments with limited or no internet connectivity. Credential wallets must support offline storage and presentation. The holder should be able to present credentials without network access; the verifier should be able to verify signature validity offline. Only credential status (revocation) checking requires network access, and even this can be optimized through pre-fetching status lists.

Revocation and Lifecycle Management

Credentials may need to be revoked before their natural expiration for various reasons: the holder's status may change, the credential may have been issued in error, or the holder may report the credential lost or stolen. The WIA standard provides mechanisms for credential revocation that balance the need for timely status updates against privacy concerns.

Revocation Mechanisms

The WIA standard supports multiple revocation mechanisms. Revocation lists publish the status of all credentials from an issuer, enabling efficient batch checking. Status endpoints provide per-credential status lookups for fresher information. On-chain revocation uses blockchain state for tamper-evident status tracking. The choice of mechanism depends on privacy requirements, infrastructure availability, and verification performance needs.

Credential Updates

Some credentials may need updates without full revocation and reissuance. A refugee's address may change while their identity credential remains valid. Educational credentials may be supplemented with additional qualifications. The WIA standard supports credential amendment mechanisms that add information to existing credentials while preserving the original issuance chain.

Security Considerations

Credential issuance systems face numerous security threats that must be addressed through technical controls, operational procedures, and governance frameworks. This section summarizes key security considerations; detailed security requirements are provided in the WIA technical specifications.

Critical Security Threats

  • Key Compromise: Attacker obtains issuer private key, enabling unlimited forgery
  • Insider Threat: Authorized personnel issue fraudulent credentials
  • System Breach: Attacker gains access to issuance systems
  • Credential Theft: Attacker steals holder's credentials or wallet
  • Identity Fraud: Impersonation during registration process
  • Replay Attack: Attacker re-uses captured credential presentation

Defense Strategies

Defense against these threats requires layered security controls. HSM-backed keys prevent key extraction even if systems are breached. Multi-person issuance workflows prevent individual insiders from issuing credentials alone. Audit logging and anomaly detection identify suspicious activity. Biometric binding prevents credential transfer to imposters. Challenge-response protocols prevent replay attacks.

Chapter Summary

Secure credential issuance requires integrating cryptographic technologies with operational security practices. Digital signatures provide the foundation for credential integrity. Decentralized identifiers enable self-sovereign identity without central authority dependency. Verifiable credentials provide standardized formats for expressing and verifying claims. Structured issuance workflows ensure proper verification before credential creation.

Key Takeaways

  • Digital signatures (Ed25519 recommended) ensure credential integrity and issuer authenticity
  • HSM-backed key management required for production issuance systems
  • Decentralized Identifiers enable identity without central authority dependency
  • Verifiable Credentials follow W3C standards with WIA extensions for refugee contexts
  • Issuance workflows match verification intensity to assurance level requirements
  • Credential wallets must support offline storage and secure presentation
  • Revocation mechanisms enable timely status updates while protecting privacy
