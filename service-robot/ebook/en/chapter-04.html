Chapter 4

Phase 1: Data Format Specification

4.1 Introduction to Data Format Design

Phase 1 of WIA-ROB-004 establishes the foundational data formats that enable unambiguous communication between service robots, management systems, and applications. This phase defines how robots represent their identity, current state, capabilities, sensor data, task information, and operational context using standardized JSON schemas.

The choice of JSON as the primary data interchange format reflects careful consideration of practical requirements. JSON offers human readability, which simplifies debugging and development. It has widespread support across programming languages and platforms. JSON schemas provide robust validation capabilities. The format strikes an excellent balance between expressiveness and simplicity, avoiding the complexity of XML while providing more structure than simple key-value pairs.

The data format specification addresses several critical requirements. It must be unambiguous - any compliant system must interpret a data structure identically. It must be extensible - new fields and capabilities can be added without breaking existing implementations. It must be efficient - data structures should be compact enough for resource-constrained devices. It must be secure - sensitive data must be clearly marked and handled appropriately. Finally, it must be practical - the format should be easy for developers to work with, not an academic exercise in theoretical elegance.

4.2 Core Data Structures

Robot Identity

Every service robot must have a unique identity that persists across reboots, firmware updates, and configuration changes. The Robot Identity structure provides comprehensive identification information that supports both human operators and automated systems.

{
  "identity": {
    "robotId": "ROB-2025-ACME-001",
    "serialNumber": "ACME-SR-2025-123456",
    "manufacturer": "ACME Robotics Inc.",
    "model": "ServiceBot-Pro",
    "firmwareVersion": "2.3.1",
    "hardwareRevision": "C",
    "manufacturerDate": "2025-01-15",
    "certificationId": "WIA-ROB-004-CERT-2025-001",
    "certificationLevel": 2,
    "certificationExpiry": "2027-01-15"
  }
}

The robotId field provides a globally unique identifier following a structured format: standard identifier (ROB), year, manufacturer code, and sequential number. This format enables easy parsing while ensuring uniqueness. The serialNumber is the manufacturer's own serial number, allowing cross-reference with manufacturing and warranty systems.

Certification information is embedded directly in the identity structure. This allows any system interacting with the robot to immediately verify that it meets WIA-ROB-004 requirements at a specific compliance level. The certification expiry date enables automated alerts when recertification is needed.

The operationalState field uses a standardized enumeration: idle, active, charging, maintenance, error, emergency_stop. This high-level state enables quick assessment of robot availability. The healthStatus provides an overall health indicator: healthy, degraded, critical, failed.

Location information supports both indoor and outdoor robots. Indoor robots typically use building-relative coordinates while outdoor robots use GPS. The standard accommodates both through a flexible location structure that includes multiple coordinate systems. This flexibility ensures the format works for delivery robots navigating city streets and cleaning robots working within buildings.

The currentTask structure provides visibility into what the robot is doing and how far along it is. The progress percentage enables user interfaces to show meaningful status updates. The estimated completion time helps with scheduling and coordination.

Errors and warnings are reported as arrays of structured objects, each with a standardized code, severity level, human-readable message, and timestamp. This structured approach enables automated error handling and alerting while providing useful information for human operators.

The supportedTasks array explicitly declares what task types the robot can execute and what parameters each task requires. This enables validation of task assignments before dispatch, preventing errors from incompatible task requests.

4.5 Event and Alert Structures

Events and alerts notify management systems and operators of significant occurrences requiring attention or logging. The event format supports both routine operational events and critical alerts.

{
  "event": {
    "eventId": "EVT-2025-001234",
    "robotId": "ROB-2025-ACME-001",
    "timestamp": "2025-01-20T14:32:15.243Z",
    "eventType": "obstacle_detected",
    "severity": "warning",
    "category": "navigation",
    "description": "Unexpected obstacle detected in path",
    "location": {
      "buildingId": "BLDG-A",
      "floor": 3,
      "coordinates": { "x": 45.2, "y": 23.8 }
    },
    "relatedTask": "TASK-2025-001234",
    "actionTaken": "path_replanned",
    "requiresAcknowledgment": false,
    "metadata": {
      "obstacleType": "temporary",
      "obstacleSize": "small",
      "detectionSensor": "LIDAR-001"
    }
  }
}

Event severity levels (info, warning, error, critical, emergency) enable appropriate filtering and routing. Critical and emergency events might trigger immediate notifications to operators, while info events are simply logged for later analysis.

The category field helps organize events by subsystem, making it easier to identify patterns. For example, frequent navigation events might indicate issues with the robot's mapping or localization systems.

4.7 Data Validation Rules

Phase 1 includes comprehensive validation rules that ensure data integrity and catch errors early. These rules are expressed as JSON Schema definitions that can be used for automated validation in any programming language with JSON Schema support.

📋 Key Validation Requirements

  • Required Fields: Core fields like robotId, timestamp, and operationalState must always be present
  • Type Checking: Fields must match their defined types (string, number, boolean, object, array)
  • Value Constraints: Numeric values must be within defined ranges (e.g., battery 0-100)
  • Format Validation: Timestamps must use ISO 8601 format, IDs must match naming patterns
  • Enumeration Checking: String fields with limited valid values must match enumerated options
  • Cross-Field Validation: Some fields' validity depends on values of other fields
  • Extension Rules: Custom fields must use vendor prefixes to avoid naming conflicts

Implementing these validation rules consistently across all compliant systems ensures that data format violations are caught immediately rather than causing subtle bugs or interoperability failures later.

4.8 Internationalization and Localization

Service robots operate globally, requiring support for multiple languages and regional conventions. Phase 1 addresses internationalization through several mechanisms.

All human-readable strings can include language variants using a language code suffix. For example, a robot description might include both description_en and description_ko fields. Systems should use the language variant matching the user's preference, falling back to English if the preferred language isn't available.

Timestamps always use ISO 8601 format with UTC timezone, eliminating timezone confusion. Display systems convert to local time zones as needed for user interfaces, but all data exchange uses UTC to ensure consistency.

Measurements include explicit unit fields. Rather than assuming metric or imperial units, every measurement specifies its unit (meters, feet, celsius, fahrenheit, etc.). This explicitness prevents conversion errors and supports global deployment.

4.9 Privacy and Data Protection

Phase 1 includes privacy considerations directly in the data format specification. Certain fields are marked as potentially containing personal information, triggering special handling requirements.

The specification defines a privacyLevel attribute that can be attached to any data field:

  • public: No privacy concerns, can be shared freely
  • internal: Should not be exposed outside the organization
  • confidential: Contains sensitive information, restricted access
  • personal: Contains personal data subject to privacy regulations

Management systems must respect these privacy levels when logging, transmitting, or displaying data. Camera images and microphone recordings are automatically classified as personal data, requiring appropriate consent and handling according to applicable privacy regulations like GDPR and CCPA.

  • Comprehensive validation rules ensure data integrity and catch errors early
  • Privacy considerations are built into the data format specification from the ground up
  • Internationalization support enables global deployment with appropriate language and unit handling
