Chapter 6

Phase 3: Communication Protocols

6.1 Protocol Stack Overview

Phase 3 of WIA-ROB-004 specifies the communication protocols that enable reliable, secure, real-time interaction between service robots and management systems. While Phase 2 defined what information is exchanged through APIs, Phase 3 defines how that information is transmitted across networks, ensuring consistent behavior even in challenging wireless environments.

The protocol stack supports multiple transport mechanisms to accommodate different deployment scenarios and performance requirements. HTTP/HTTPS provides the baseline for maximum compatibility. WebSocket enables bidirectional real-time communication for streaming telemetry and responsive control. MQTT offers efficient publish-subscribe messaging for multi-robot coordination and event distribution. The specification ensures that the same logical operations can be performed over any supported transport, allowing system designers to choose based on their specific requirements.

6.2 Transport Layer Protocols

HTTPS for Request-Response

Standard API operations use HTTPS with TLS 1.3 or later for transport security. This provides compatibility with existing web infrastructure, firewall traversal through standard ports, and strong encryption. HTTPS suits operations that follow request-response patterns: querying robot status, creating tasks, updating configuration.

WebSocket for Real-Time Streaming

WebSocket provides persistent bidirectional connections suitable for streaming sensor data, live telemetry, and responsive command-and-control. Once a WebSocket connection is established, both client and server can send messages at any time without request-response overhead. This enables low-latency updates essential for applications like remote operation or real-time monitoring dashboards.

// WebSocket message structure
{
  "protocol": "WIA-ROB-004-WS",
  "version": "1.0",
  "messageType": "telemetry_stream",
  "timestamp": "2025-01-20T14:32:15.243Z",
  "robotId": "ROB-2025-ACME-001",
  "sequenceNumber": 12345,
  "payload": {
    "location": {...},
    "batteryLevel": 85,
    "sensors": {...}
  }
}

MQTT for Publish-Subscribe

MQTT (Message Queuing Telemetry Transport) supports efficient many-to-many communication patterns. Robots publish state updates and events to topics, while management systems and other robots subscribe to topics of interest. This decoupled communication enables scalable fleet coordination where adding new subscribers doesn't require changes to publishers.

WIA-ROB-004 defines a standard topic hierarchy that organizes messages logically:

wia/robots/{robotId}/status
wia/robots/{robotId}/telemetry
wia/robots/{robotId}/events
wia/robots/{robotId}/commands
wia/fleet/{fleetId}/coordination
wia/zone/{zoneId}/presence

6.3 Message Format and Structure

All protocol messages share a common envelope structure that includes protocol version, message type, timestamps, sequence numbers, and payload. This consistent structure simplifies message parsing and routing across different transport mechanisms.

Field Purpose Required
protocol Protocol identifier and version (e.g., "WIA-ROB-004-v1.0") Yes
messageType Type of message (status_update, command, event, etc.) Yes
messageId Unique message identifier for deduplication and tracking Yes
timestamp Message creation time (ISO 8601 UTC) Yes
robotId Identifier of robot sending or receiving message Yes
sequenceNumber Monotonic counter for ordering and gap detection Yes
payload Message-specific data conforming to Phase 1 schemas Yes
signature Cryptographic signature for message authentication Optional

6.4 Security Protocols

Transport Layer Security

All network communication must use TLS 1.3 or later for encryption. Certificate validation is mandatory - systems must verify that they're communicating with authentic robots and servers. Mutual TLS (mTLS) is recommended for high-security deployments where both client and server present certificates.

Message-Level Security

Beyond transport encryption, critical messages can include cryptographic signatures that prevent tampering even if transport security is somehow compromised. Emergency stop commands, configuration changes, and firmware updates always include message-level signatures.

{
  "protocol": "WIA-ROB-004",
  "messageType": "emergency_stop",
  "messageId": "msg-2025-001234",
  "timestamp": "2025-01-20T14:32:15.243Z",
  "robotId": "ROB-2025-ACME-001",
  "payload": {
    "reason": "obstacle_detected",
    "initiator": "safety_system"
  },
  "signature": {
    "algorithm": "ES256",
    "keyId": "safety-key-001",
    "value": "MEUCIQDx..."
  }
}

Anti-Replay Protection

Sequence numbers and message timestamps prevent replay attacks where attackers capture valid messages and retransmit them later. Recipients reject messages with timestamps outside an acceptable time window (typically ±30 seconds) and track recent sequence numbers to detect duplicates.

6.5 Network Resilience

Service robots often operate in challenging wireless environments with intermittent connectivity. Phase 3 includes mechanisms that maintain system coherence despite network disruptions.

Automatic Reconnection

When network connections are lost, robots automatically attempt reconnection with exponential backoff. Initial reconnection attempts occur quickly (1-2 seconds) to handle brief outages. If reconnection fails, backoff intervals increase gradually to avoid overwhelming networks during extended outages. Maximum backoff is capped to ensure robots don't give up trying to reconnect.

Message Buffering

During network outages, robots buffer important messages in persistent storage. When connectivity is restored, buffered messages are transmitted to ensure management systems have complete operational history. Message priority determines buffer depth - critical safety events are always buffered, while routine telemetry may be dropped if buffers fill.

State Synchronization

After reconnection, robots and management systems synchronize state to ensure consistent views. Robots transmit current status, and management systems acknowledge task assignments and configuration. Sequence numbers help identify any messages lost during the outage.

6.6 Quality of Service (QoS)

Different message types have different delivery requirements. Safety-critical messages require guaranteed delivery, while routine telemetry can tolerate occasional loss. Phase 3 defines QoS levels that can be specified per message or topic.

QoS Levels

  • QoS 0 (Best Effort): Message sent once, no confirmation. Suitable for high-frequency telemetry where occasional loss is acceptable.
  • QoS 1 (At Least Once): Message delivered at least once, duplicates possible. Suitable for events and status updates where duplication is tolerable.
  • QoS 2 (Exactly Once): Message delivered exactly once using handshake protocol. Required for commands, configuration changes, and critical events.

6.7 Multi-Robot Coordination

When multiple robots operate in shared spaces, they must coordinate to avoid conflicts and optimize efficiency. Phase 3 defines coordination protocols for common scenarios.

Zone Management

Robots can reserve zones for exclusive use during tasks. For example, a cleaning robot might reserve a room to avoid conflicts with delivery robots. Zone reservation requests use a distributed consensus protocol to handle contention when multiple robots want the same zone simultaneously.

Resource Sharing

Shared resources like charging stations and elevators require coordination. Robots publish resource usage intentions, allowing others to plan accordingly. Priority-based arbitration resolves conflicts when demand exceeds capacity.

Fleet-Wide Optimization

For fleet coordination, robots share information about their locations, current tasks, and future plans. Fleet management systems use this information to optimize task assignment and routing, minimizing total distance traveled and avoiding congestion.

6.8 Monitoring and Diagnostics

Phase 3 includes protocols for monitoring communication health and diagnosing connectivity issues. Heartbeat messages sent periodically verify that connections remain active. Communication statistics (message counts, latencies, errors) help identify network problems. Diagnostic tools can request detailed protocol traces for troubleshooting.

Chapter Summary

Key Takeaways

  • Phase 3 specifies transport protocols (HTTPS, WebSocket, MQTT) for different communication patterns
  • Consistent message envelope structure works across all transport mechanisms
  • Transport-level and message-level security provide defense in depth against attacks
  • Network resilience mechanisms handle intermittent connectivity common in wireless environments
  • Quality of Service levels match delivery guarantees to message criticality
  • Coordination protocols enable efficient multi-robot operation in shared spaces
