Chapter 5

Phase 2: API Interface Design

5.1 RESTful API Architecture

Phase 2 of WIA-ROB-004 defines comprehensive RESTful HTTP APIs that enable applications, management systems, and other robots to interact with service robots in a standardized way. The API design follows REST principles while incorporating modern best practices for security, scalability, and developer experience.

The API is organized into logical resource collections that map to robot capabilities and operational needs. Each resource supports standard HTTP methods (GET, POST, PUT, PATCH, DELETE) with consistent behavior across all endpoints. The API uses JSON for both request and response bodies, aligning with the data formats defined in Phase 1.

5.2 Core API Endpoints

Robot Discovery and Information

GET /api/v1/robots
GET /api/v1/robots/{robotId}
GET /api/v1/robots/{robotId}/capabilities
GET /api/v1/robots/{robotId}/status

The discovery endpoints enable management systems to find available robots and query their capabilities. The status endpoint provides current operational state including location, battery level, current task, and health indicators. These endpoints support both polling and change notification mechanisms.

Task Management

POST /api/v1/robots/{robotId}/tasks
GET /api/v1/robots/{robotId}/tasks
GET /api/v1/robots/{robotId}/tasks/{taskId}
PATCH /api/v1/robots/{robotId}/tasks/{taskId}
DELETE /api/v1/robots/{robotId}/tasks/{taskId}

Task management endpoints handle the complete task lifecycle from creation through execution to completion. The POST endpoint creates new tasks, returning a task ID for tracking. The PATCH endpoint allows task modification (changing priority, updating parameters) while respecting task state constraints. The DELETE endpoint cancels tasks that haven't yet started.

Configuration Management

GET /api/v1/robots/{robotId}/configuration
PUT /api/v1/robots/{robotId}/configuration
GET /api/v1/robots/{robotId}/configuration/history

Configuration endpoints allow authorized users to view and update robot operational parameters. All configuration changes are logged with timestamp and user identity, creating an audit trail. The history endpoint enables reviewing past configuration changes and reverting to previous configurations if needed.

Telemetry and Logging

GET /api/v1/robots/{robotId}/telemetry
GET /api/v1/robots/{robotId}/events
GET /api/v1/robots/{robotId}/logs

Telemetry endpoints provide access to sensor data, performance metrics, and operational statistics. The events endpoint returns structured events and alerts, supporting filtering by severity, category, and time range. The logs endpoint provides access to detailed operational logs for debugging and analysis.

5.3 Authentication and Authorization

WIA-ROB-004 mandates OAuth 2.0 for API authentication, providing industry-standard security with well-understood implementations across programming languages. The standard defines specific scopes that map to robot capabilities and access levels.

OAuth 2.0 Scopes

  • robot:read - View robot status, capabilities, and telemetry
  • robot:write - Modify robot configuration
  • robot:control - Send commands and create tasks
  • robot:admin - Full administrative access including firmware updates
  • fleet:read - View information for all robots in a fleet
  • fleet:manage - Coordinate multiple robots

5.4 Error Handling

The API defines standardized error responses that provide meaningful information for debugging while avoiding security information leaks. All error responses use consistent JSON structure with error codes, messages, and additional context.

{
  "error": {
    "code": "TASK_INVALID_PARAMETERS",
    "message": "The specified destination is not reachable",
    "details": {
      "field": "destination",
      "reason": "Location is outside robot's operational area"
    },
    "requestId": "req-2025-001234",
    "timestamp": "2025-01-20T14:32:15.243Z"
  }
}
HTTP Status Error Code Pattern Usage
400 REQUEST_* Client sent invalid request (malformed JSON, missing required fields)
401 AUTH_* Authentication failed or missing credentials
403 FORBIDDEN_* Authenticated but lacking required permissions
404 NOT_FOUND Requested resource doesn't exist
409 CONFLICT_* Request conflicts with current state
500 INTERNAL_ERROR Server-side error
503 SERVICE_UNAVAILABLE Robot temporarily unavailable

5.5 Rate Limiting and Quality of Service

To ensure fair access and prevent abuse, the API implements rate limiting with different limits for different endpoint categories. Read-only endpoints have higher rate limits than write operations. Emergency commands bypass rate limiting to ensure safety-critical operations always succeed.

Rate limit information is communicated through standard HTTP headers:

X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 987
X-RateLimit-Reset: 1611158400

5.6 Versioning Strategy

The API uses URL-based versioning (v1, v2, etc.) providing clear indication of the API version being used. This approach simplifies API client development and enables smooth transitions when new API versions are released. Multiple API versions can coexist, allowing gradual migration.

5.7 Webhooks and Notifications

For real-time updates, the API supports webhook notifications where management systems register callback URLs to receive push notifications when events occur. This is more efficient than polling for applications that need immediate notification of robot state changes.

POST /api/v1/webhooks
{
  "url": "https://myapp.example.com/robot-events",
  "events": ["task_completed", "error_occurred", "battery_low"],
  "secret": "webhook-secret-key"
}

The webhook system uses HMAC-SHA256 signatures to ensure authenticity of notifications. When a registered event occurs, the system sends an HTTP POST request to the configured URL with the event payload and a signature header. The receiving application validates the signature using the shared secret before processing the event.

Event Categories

Event Category Event Types Typical Use Case
Task Events task_created, task_started, task_completed, task_failed, task_cancelled Workflow orchestration and monitoring
Status Events status_changed, location_updated, battery_low, battery_critical Fleet monitoring dashboards
Error Events error_occurred, warning_raised, emergency_stop, collision_detected Alert systems and incident response
Maintenance Events maintenance_required, firmware_updated, configuration_changed Preventive maintenance scheduling
Sensor Events sensor_anomaly, threshold_exceeded, calibration_needed Predictive analytics and diagnostics

5.8 GraphQL Alternative

In addition to the RESTful API, WIA-ROB-004 Phase 2 also defines an optional GraphQL endpoint for applications that benefit from flexible query structures and reduced over-fetching. GraphQL enables clients to request exactly the data they need in a single query, reducing network round-trips and improving performance for complex data requirements.

POST /api/v1/graphql
{
  query {
    robot(id: "robot-001") {
      id
      name
      status {
        location { x, y, z }
        battery
        currentTask {
          id
          type
          progress
        }
      }
      capabilities {
        navigation
        manipulation
        sensing
      }
    }
  }
}

The GraphQL schema is automatically generated from the robot's capability model, ensuring consistency between REST and GraphQL interfaces. Subscriptions enable real-time updates through WebSocket connections, providing an alternative to webhook notifications for web-based applications.

5.9 API SDKs and Client Libraries

To accelerate development and ensure correct API usage, WIA maintains official SDKs in multiple programming languages. These SDKs handle authentication, request signing, error handling, and retry logic, allowing developers to focus on application logic rather than protocol details.

📦 Supported Languages

  • Python: wia-robot-sdk (PyPI) - Full featured with async support
  • JavaScript/TypeScript: @wia/robot-sdk (npm) - Browser and Node.js compatible
  • Go: github.com/wia/robot-sdk-go - Native Go types and contexts
  • Java: com.wia:robot-sdk (Maven Central) - Enterprise-ready with Spring integration
  • C#: WIA.RobotSDK (NuGet) - .NET Standard 2.0+ compatible
  • Rust: wia-robot (crates.io) - Safe, zero-cost abstractions
