왜 라이프타임? Why Lifetimes?

// 어떤 참조를 반환해야 할까?
fn longest(x: &str, y: &str) -> &str {
if x.len() > y.len() {
x
} else {
y
}
}
// 에러! 반환값의 라이프타임을 알 수 없음
🤔 x를 반환? y를 반환? 컴파일러가 모름 → 라이프타임 명시 필요! 🤔 Return x? Return y? Compiler doesn't know → Need lifetime annotation!

EP 2-04

라이프타임 문법 Lifetime Syntax

fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
if x.len() > y.len() { x } else { y }
}
// 의미: x와 y 중 더 짧은 라이프타임만큼 반환값 유효

'a = "라이프타임 a" (작은따옴표 + 이름)

'a = "lifetime a" (apostrophe + name)

x: &'a str
y: &'a str
return
← 더 짧은 쪽 따라감

fn longest<'a>(x: &'a str, y: &'a str) -> &'a str { ... }

구조체에서 라이프타임 Lifetimes in Structs

// 참조를 필드로 가지면 라이프타임 필요!
struct ImportantExcerpt<'a> {
part: &'a str,
}
let novel = String::from("Call me Ishmael...");
let first = novel.split('.').next().unwrap();
let excerpt = ImportantExcerpt { part: first };
💡 구조체는 참조 필드의 원본보다 오래 살 수 없음! 💡 Struct cannot outlive the data its reference points to!

EP 2-04

생략 규칙 Elision Rules

// 컴파일러가 자동 추론 (생략 가능)
fn first_word(s: &str) -> &str { ... }
// 실제로는 이렇게 해석됨
fn first_word<'a>(s: &'a str) -> &'a str { ... }

생략 규칙 3가지:
1️⃣ 각 참조 매개변수에 별도 라이프타임
2️⃣ 입력이 하나면 출력도 같은 라이프타임
3️⃣ &self가 있으면 출력도 self 라이프타임

3 Elision Rules:
1️⃣ Each ref param gets its own lifetime
2️⃣ One input → output gets same lifetime
3️⃣ With &self → output gets self's lifetime
