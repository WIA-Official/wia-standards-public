Chapter 05: Phase 2 — API Interface

5.1 Phase 2 in one paragraph

Phase 2 specifies the HTTPS surface for sleep monitoring data: session ingest, stage / event / arousal bulk submit, scoring metadata, and re-computation. Transport is TLS 1.3 (IETF RFC 8446) only.

5.2 Discovery

Every host publishes a discovery document at /.well-known/wia-sleep-monitoring listing endpoints, supported session kinds, supported scoring algorithms, and rate limits.

5.3 Session endpoints

POST   /sm/session
GET    /sm/session/{session_id}
PATCH  /sm/session/{session_id}     (state transition)
DELETE /sm/session/{session_id}     (tombstone, audit retention 25y)

5.4 Stage endpoints (bulk)

POST /sm/stage    (bulk submit per session; 960 epochs typical)
GET  /sm/stage?session_id={id}

The bulk pattern matters: a single session has 960 epochs (8 hours x 30s); submitting them one at a time would be inefficient and error-prone.

5.5 Event endpoints

POST / GET / filter on respiratory events. Hosts MUST preserve event ordering by started_at for downstream AHI re-computation.

5.6 Arousal endpoints

POST / GET / filter on arousal events. Hosts MUST refuse arousal submissions whose duration_seconds is below the AASM 3-second minimum.

5.7 Scoring re-computation

POST /sm/scoring/{scoring_metadata_id}/recompute triggers re-running the session through a different algorithm (typically a new AASM Manual edition). The original scoring_metadata is preserved; the new one references the prior via supersedes.

5.8 Authentication

Writes require HTTP Message Signatures per IETF RFC 9421. PHI-bearing reads MUST be authenticated. Cross-vendor reads use Phase 3 federation receipts.

Patient and Clinician Considerations

Sleep data is biometric and high-stakes. Phase 1 carries a clinical_grade flag and per-channel signal quality metrics so consumer wearable estimates do not get conflated with clinical PSG measurements. Phase 3 enforces patient consent before any cross-host data flow, with explicit per-audience scopes (clinical_dashboard, ehr_export, research_de_identified). Phase 4 maps to the AASM compliance level so downstream consumers can filter by appropriateness for their clinical context.

The standard's default audience controls are intentionally restrictive: even the patient's own family / caregiver receives only stage histograms by default, not the raw event stream. Operators MAY relax these defaults per local policy, but the standard's baseline assumes minimum disclosure unless explicitly elevated by consent.
