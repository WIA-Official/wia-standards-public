3장: 보안 및 인증

WIA-FIN-022 오픈뱅킹 표준

3.1 강력한 고객 인증 (SCA)

PSD2는 전자 결제 및 원격 계정 액세스에 대해 강력한 고객 인증을 의무화합니다. SCA는 세 가지 인증 요소 중 최소 두 가지를 요구합니다:

  • 지식: 사용자만 아는 것 (비밀번호, PIN)
  • 소유: 사용자만 가지고 있는 것 (전화, 하드웨어 토큰)
  • 고유성: 사용자 자체 (지문, 얼굴 인식)

동적 연결

결제 거래의 경우 SCA는 동적 연결을 포함해야 하며, 인증 코드를 특정 결제 금액 및 수취인에 바인딩합니다.

3.2 eIDAS 인증서

TPP는 웹사이트 인증(QWAC) 및 전자 봉인(QSeal)을 위한 eIDAS 적격 인증서를 획득해야 합니다.

  • TPP의 신원 및 규제 상태 확인
  • 상호 TLS 인증 활성화
  • 부인 방지를 위한 API 요청 서명

3.3 TLS 및 전송 보안

모든 오픈뱅킹 통신은 강력한 암호화 제품군과 함께 TLS 1.2 이상을 사용해야 합니다:

  • TLS_AES_256_GCM_SHA384 (TLS 1.3)
  • TLS_CHACHA20_POLY1305_SHA256 (TLS 1.3)
  • TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384 (TLS 1.2)

3.4 JWS를 사용한 메시지 서명

중요한 작업(결제, 동의 생성)에는 TPP의 QSeal 인증서를 사용하여 분리된 JWS 서명이 필요합니다.

3.5 액세스 토큰 보안

OAuth 2.0 액세스 토큰은 다음과 같아야 합니다:

  • 수명이 짧음 (결제 흐름의 경우 일반적으로 5-10분)
  • 특정 동의 및 범위에 바인딩됨
  • TLS를 통해서만 전송됨
  • 안전하게 저장됨
  • 고객이 언제든지 취소할 수 있음

3.6 사기 탐지 및 방지

여러 계층의 사기 탐지를 구현합니다:

거래 모니터링

  • 비정상적인 거래 금액 또는 패턴
  • 빠른 거래 시퀀스
  • 새로운 또는 고위험 수취인에 대한 거래
  • 지리적 이상 현상

3.7 보안 모범 사례

은행용

  • 속도 제한 및 DDoS 보호 구현
  • 감사 추적과 함께 모든 API 액세스 로그
  • 비정상적인 TPP 동작 모니터링
  • 정기적인 침투 테스트 및 보안 감사

TPP용

  • 고객 자격 증명을 절대 저장하지 마십시오
  • 인증서 고정 구현
  • 안전한 키 관리 사용 (프로덕션의 경우 HSM)
  • 저장 중인 민감한 데이터 암호화
  • 적절한 세션 관리 구현
