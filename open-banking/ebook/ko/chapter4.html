4장: OAuth 2.0 및 동의 관리

WIA-FIN-022 오픈뱅킹 표준

4.1 OAuth 2.0 인증 프레임워크

오픈뱅킹은 OAuth 2.0을 인증 프레임워크로 사용하며 금융 서비스를 위한 특정 프로필을 사용합니다:

  • 금융급 API (FAPI): 고위험 시나리오를 위한 향상된 보안 프로필
  • PKCE를 사용한 인증 코드 흐름: 웹 및 모바일 앱의 기본 흐름
  • 클라이언트 자격 증명: 사용자 컨텍스트 없이 TPP-to-bank 인증

4.2 인증 코드 흐름

1단계: 동의 생성

계정 정보에 대한 액세스를 요청하는 동의 개체를 만듭니다.

2단계: PKCE를 사용한 인증 요청

PKCE 검증자 및 챌린지를 생성하고 사용자를 인증 엔드포인트로 리디렉션합니다.

3단계: 사용자 인증 및 동의

은행은 SCA로 사용자를 인증하고 다음을 표시하는 동의 화면을 표시합니다:

  • TPP 이름 및 규제 세부 정보
  • 요청된 권한 및 데이터 액세스
  • 액세스 기간
  • 공유할 계정

4단계: 토큰에 대한 코드 교환

인증 코드를 액세스 토큰 및 새로 고침 토큰으로 교환합니다.

4.3 동의 수명 주기 관리

동의 상태

  • AwaitingAuthorisation: 생성되었지만 아직 사용자가 승인하지 않음
  • Authorised: 사용자가 승인함, API 액세스에 사용할 수 있음
  • Rejected: 사용자가 동의를 거부함
  • Revoked: 사용자 또는 TPP가 동의를 취소함
  • Expired: 동의가 만료 시간을 초과함

동의 갱신

계정 액세스 동의는 최대 90일 후에 만료됩니다. TPP는 동의 갱신 흐름을 구현해야 합니다.

4.4 새로 고침 토큰

새로 고침 토큰을 사용하여 사용자 상호 작용 없이 새 액세스 토큰을 얻습니다.

4.5 동의 대시보드

은행은 고객에게 다음을 할 수 있는 대시보드를 제공해야 합니다:

  • 모든 활성 동의 및 액세스 권한이 있는 TPP 보기
  • 상세한 권한 부여 및 액세스한 데이터 확인
  • 액세스 기록 및 API 호출 로그 검토
  • 개별 또는 일괄 동의 취소
  • 동의 만료에 대한 알림 수신

4.6 보안 고려 사항

  • 항상 PKCE를 사용하여 인증 코드 가로채기 방지
  • CSRF 공격을 방지하기 위해 상태 매개변수 검증
  • 새로 고침 토큰에 대한 토큰 회전 구현
  • 수명이 짧은 액세스 토큰 사용 (5-10분)
  • 새로 고침 토큰을 안전하게 저장
  • 토큰 취소 엔드포인트 구현
