Chapter 3: Security and Authentication

WIA-FIN-022 Open Banking Standard

3.1 Strong Customer Authentication (SCA)

PSD2 mandates Strong Customer Authentication for electronic payments and remote account access. SCA requires at least two of three authentication factors:

  • Knowledge: Something only the user knows (password, PIN)
  • Possession: Something only the user has (phone, hardware token)
  • Inherence: Something the user is (fingerprint, face recognition)

3.1.1 Dynamic Linking

For payment transactions, SCA must include dynamic linking - binding the authentication code to the specific payment amount and payee. This prevents man-in-the-middle attacks where attackers modify payment details.

// Example: Dynamic Linking in Payment Authorization { "authentication_challenge": { "method": "mobile_app", "challenge_code": "Please confirm payment", "linked_data": { "amount": "€100.00", "payee": "Coffee Shop Ltd", "account": "****5678" }, "expires_in": 300 } }

3.2 eIDAS Certificates

TPPs must obtain eIDAS qualified certificates for website authentication (QWAC) and electronic seals (QSeal). These certificates:

  • Verify the TPP's identity and regulatory status
  • Enable mutual TLS authentication
  • Sign API requests for non-repudiation
  • Must be issued by qualified trust service providers

Certificate Structure

Subject: CN=FinTech App Ltd, OU=Payment Services, O=FinTech App Ltd, C=GB Issuer: CN=Qualified TSP, O=Trust Services Provider, C=GB Subject Alternative Names: - PSD.GB-FCA.123456789 (Authorization Number) - PSP_AS (Account Information) - PSP_PI (Payment Initiation) Valid From: 2025-01-01 Valid Until: 2027-01-01

3.3 TLS and Transport Security

All Open Banking communications must use TLS 1.2 or higher with strong cipher suites:

  • TLS_AES_256_GCM_SHA384 (TLS 1.3)
  • TLS_CHACHA20_POLY1305_SHA256 (TLS 1.3)
  • TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384 (TLS 1.2)

Mutual TLS (mTLS)

Banks may require mTLS where both client and server present certificates. This provides strong authentication and prevents impersonation attacks.

3.4 Message Signing with JWS

Critical operations (payments, consent creation) require detached JWS signatures using the TPP's QSeal certificate:

3.5 Access Token Security

OAuth 2.0 access tokens must be:

  • Short-lived (typically 5-10 minutes for payment flows)
  • Bound to specific consents and scopes
  • Transmitted only over TLS
  • Stored securely (never in localStorage, use httpOnly cookies or secure storage)
  • Revocable by customers at any time

Token Introspection

3.6 Fraud Detection and Prevention

Implement multiple layers of fraud detection:

Transaction Monitoring

  • Unusual transaction amounts or patterns
  • Rapid sequence of transactions
  • Transactions to new or high-risk payees
  • Geographic anomalies (location vs. typical behavior)

Device Fingerprinting

  • Browser/device characteristics
  • IP address and geolocation
  • Behavioral biometrics (typing patterns, mouse movements)

Risk Scoring

{ "risk_assessment": { "score": 0.15, // 0 = low risk, 1 = high risk "factors": [ {"name": "new_payee", "weight": 0.05}, {"name": "high_amount", "weight": 0.08}, {"name": "unusual_time", "weight": 0.02} ], "action": "allow", // allow | challenge | block "sca_required": true } }

3.7 Security Best Practices

For Banks (ASPSPs):
  • Implement rate limiting and DDoS protection
  • Log all API access with audit trails
  • Monitor for unusual TPP behavior
  • Provide security incident notification to TPPs
  • Regular penetration testing and security audits
For TPPs:
  • Never store customer credentials
  • Implement certificate pinning
  • Use secure key management (HSM for production)
  • Encrypt sensitive data at rest
  • Implement proper session management
  • Regular security training for developers

3.8 Incident Response

Establish clear incident response procedures:

  1. Detection: Automated monitoring and alerting
  2. Assessment: Determine scope and severity
  3. Containment: Limit damage and prevent spread
  4. Notification: Inform affected parties and regulators
  5. Recovery: Restore normal operations
  6. Post-Incident: Review and improve security controls
