Chapter 4: OAuth 2.0 and Consent Management

WIA-FIN-022 Open Banking Standard

4.1 OAuth 2.0 Authorization Framework

Open Banking uses OAuth 2.0 as the authorization framework, with specific profiles for financial services:

  • Financial-grade API (FAPI): Enhanced security profile for high-risk scenarios
  • Authorization Code Flow with PKCE: Primary flow for web and mobile apps
  • Client Credentials: For TPP-to-bank authentication without user context

4.2 Authorization Code Flow

Step 1: Create Consent

POST /account-access-consents Authorization: Bearer {client_credentials_token} { "Data": { "Permissions": [ "ReadAccountsBasic", "ReadAccountsDetail", "ReadBalances", "ReadTransactionsDetail" ], "ExpirationDateTime": "2026-12-31T23:59:59Z" } } Response: { "Data": { "ConsentId": "AAC-123", "Status": "AwaitingAuthorisation" } }

Step 2: Authorization Request with PKCE

// Generate PKCE verifier and challenge const verifier = generateRandomString(128); const challenge = base64UrlEncode(sha256(verifier)); // Redirect user to authorization endpoint GET /authorize? response_type=code& client_id=tpp-client-123& redirect_uri=https://tpp.example.com/callback& scope=accounts& state=random-state-123& code_challenge={challenge}& code_challenge_method=S256& request={signed_jwt}

Step 3: User Authentication and Consent

Bank authenticates user with SCA and displays consent screen showing:

  • TPP name and regulatory details
  • Requested permissions and data access
  • Duration of access
  • Accounts to be shared

Step 4: Exchange Code for Token

POST /token Content-Type: application/x-www-form-urlencoded Authorization: Basic {client_credentials} grant_type=authorization_code& code={authorization_code}& redirect_uri=https://tpp.example.com/callback& code_verifier={verifier} Response: { "access_token": "eyJhbGci...", "token_type": "Bearer", "expires_in": 600, "refresh_token": "refresh_abc...", "scope": "accounts" }

4.3 Consent Lifecycle Management

Consent States

  • AwaitingAuthorisation: Created but not yet authorized by user
  • Authorised: User approved, can be used for API access
  • Rejected: User declined consent
  • Revoked: User or TPP revoked consent
  • Expired: Consent exceeded expiration time
  • Consumed: One-time consent used (payments)

Consent Renewal

Account access consents expire after maximum 90 days. TPPs must implement consent renewal flows:

// Check consent status GET /account-access-consents/{ConsentId} // If expired, create new consent and redirect user POST /account-access-consents // ... redirect user for re-authorization

4.4 Refresh Tokens

Use refresh tokens to obtain new access tokens without user interaction:

POST /token grant_type=refresh_token& refresh_token={refresh_token} Response: { "access_token": "new_token...", "expires_in": 600, "refresh_token": "new_refresh..." // Optional: token rotation }

4.5 Consent Dashboard

Banks must provide customers with a dashboard to:

  • View all active consents and TPPs with access
  • See detailed permission grants and data accessed
  • Review access history and API call logs
  • Revoke consents individually or in bulk
  • Receive notifications about consent expirations

4.6 Security Considerations

Best Practices:
  • Always use PKCE to prevent authorization code interception
  • Validate state parameter to prevent CSRF attacks
  • Implement token rotation for refresh tokens
  • Use short-lived access tokens (5-10 minutes)
  • Store refresh tokens securely (encrypted, httpOnly cookies)
  • Implement token revocation endpoint
  • Log all consent and token operations

4.7 Advanced Patterns

Request Object (JAR)

For enhanced security, authorization parameters can be sent as signed JWT:

// Create signed request object const requestObject = signJWT({ iss: "tpp-client-123", aud: "https://bank.example.com", response_type: "code", client_id: "tpp-client-123", redirect_uri: "https://tpp.example.com/callback", scope: "accounts payments", state: "xyz", nonce: "abc", claims: { id_token: { acr: { essential: true, value: "urn:openbanking:psd2:sca" } }, userinfo: { openbanking_intent_id: { value: "AAC-123", essential: true } } } }, privateKey); // Authorization request GET /authorize? request={requestObject}& client_id=tpp-client-123
