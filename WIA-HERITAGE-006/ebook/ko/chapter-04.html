WIA-HERITAGE-006: Heritage 3D Scan Standard

04.1 E57 포인트 클라우드 포맷

Phase 1의 핵심은 데이터 형식 표준화입니다. WIA-HERITAGE-006은 E57을 포인트 클라우드의 주요 저장 포맷으로 지정합니다. E57(ASTM E2807-11)은 3D 이미징 시스템의 데이터를 저장하기 위한 개방형, 벤더 중립적 표준입니다.

E57 파일 구조

E57 파일은 계층적 XML 구조를 사용하며, 이진 데이터 섹션과 결합됩니다:

구성 요소설명필수 여부
File Header파일 시그니처, 버전 정보필수
Data3D3D 포인트 클라우드 데이터 (XYZ, RGB, 강도)필수
Images2D파노라마 이미지 (선택적)선택
Cartesian Bounds바운딩 박스 (최소/최대 XYZ)권장
Pose스캔 위치 및 방향권장
Index Bounds데이터 범위 및 정밀도필수
Color LimitsRGB 값 범위RGB 데이터 시 필수

WIA 확장 필드

WIA-HERITAGE-006은 E57에 문화유산 특화 확장 필드를 추가합니다:

  • heritage:objectID: 유물/유적의 고유 식별자 (UUID)
  • heritage:catalogNumber: 박물관 카탈로그 번호
  • heritage:culturalPeriod: 문화적 시대 (예: "조선시대", "로마 제국")
  • heritage:materialType: 재질 분류 (석재/목재/금속/도자 등)
  • heritage:conservationState: 보존 상태 (우수/양호/불량/복원됨)
  • heritage:scanConditions: 스캔 환경 (조명, 온도, 습도)
  • heritage:geoLocation: 지리적 위치 (WGS84 좌표)
  • heritage:provenance: 출처 정보 (발굴지, 소유 이력)
📋 E57 필수 메타데이터 (WIA 확장)

WIA-HERITAGE-006 인증을 받으려면 다음 22개 필수 필드를 완성해야 합니다:

  • 기본 정보 (7개): objectID, title, creator, date, description, subject, rights
  • 기술 정보 (8개): scannerModel, scanResolution, pointCount, coordinateSystem, boundingBox, fileSize, compressionType, qualityScore
  • 문화유산 정보 (7개): catalogNumber, culturalPeriod, geoLocation, materialType, dimensions, conservationState, provenance

압축 및 최적화

E57은 내장 압축을 지원하지만, WIA-HERITAGE-006은 추가 최적화 전략을 제공합니다:

기술압축률정밀도 손실적용 대상

LAZ 변환60-80%무손실지리 데이터 통합 시
Octree 서브샘플링90-95%시각적 동등웹 뷰어 (LOD)
통계적 아웃라이어 제거5-15%품질 향상노이즈 많은 스캔

04.2 glTF 2.0 메시 포맷

glTF (GL Transmission Format) 2.0은 3D 씬과 모델의 효율적인 전송 및 로딩을 위한 Khronos Group 표준입니다. "3D의 JPEG"로 불리며, WebGL/WebGPU 기반 웹 렌더링에 최적화되어 있습니다.

glTF 파일 구조

glTF는 JSON 기반 메타데이터와 이진 데이터를 분리합니다:

  • .gltf: JSON 메타데이터 (씬 구조, 노드 계층, 재질 정의)
  • .bin: 이진 버퍼 (버텍스 좌표, 노멀, UV, 인덱스)
  • .glb: 단일 이진 컨테이너 (JSON + 버퍼 통합, 네트워크 전송 최적)
  • 이미지 파일: 텍스처 맵 (PNG, JPEG, KTX2)

PBR 재질 시스템

glTF 2.0은 Physically Based Rendering (PBR)을 표준으로 사용합니다. 문화유산 3D 모델은 다음 텍스처 맵을 활용합니다:

맵 유형용도권장 해상도포맷
Base Color기본 색상 및 알베도2048×2048 (최소 1024)JPEG/PNG
Normal Map표면 디테일 (법선 벡터)2048×2048PNG
Metallic-Roughness금속성/거칠기 (RG 채널)1024×1024PNG
Ambient Occlusion주변 차폐 (그림자)1024×1024JPEG
Emissive자체 발광 (옵션)512×512JPEG

glTF 확장 활용

WIA-HERITAGE-006은 다음 glTF 확장을 활용/제안합니다:

🔧 glTF 표준 확장
  • KHR_mesh_quantization: 16비트 양자화로 버텍스 데이터 40% 압축
  • KHR_texture_basisu: Basis Universal GPU 텍스처 압축 (70-90% 감소)
  • KHR_materials_unlit: 단순 색상 렌더링 (고대 벽화, 도자기)
  • EXT_meshopt_compression: 메시 최적화 압축 (추가 20-30% 감소)
  • KHR_draco_mesh_compression: Draco 기하 압축 (90% 이상)
🏛️ WIA 제안 확장 (WIP)
  • WIA_heritage_metadata: 문화유산 메타데이터 (Dublin Core + CIDOC-CRM)
  • WIA_heritage_annotations: 3D 주석 (손상 영역, 복원 부분 표시)
  • WIA_heritage_provenance: 블록체인 해시, 출처 추적
  • WIA_heritage_multispectral: UV/IR 멀티스펙트럴 텍스처

LOD (Level of Detail) 생성

WIA-HERITAGE-006은 5단계 LOD 피라미드를 권장합니다:

LOD폴리곤 수텍스처파일 크기용도
LOD 0100% (원본)4K500MB연구, 고정밀 분석
LOD 150%2K120MBVR 헤드셋, 데스크톱 뷰어
LOD 225%1K30MB태블릿, 노트북
LOD 310%512px8MB스마트폰, 저사양 PC
LOD 42%256px1MB썸네일, 3G 네트워크

LOD는 자동 생성 도구(Simplygon, meshoptimizer)로 생성하거나, 수동으로 최적화할 수 있습니다.

04.4 텍스처 압축 및 최적화

고해상도 텍스처는 3D 모델 파일 크기의 70-90%를 차지합니다. WIA-HERITAGE-006은 효율적인 압축 전략을 제공합니다.

KTX2/Basis Universal

Basis Universal는 GPU 네이티브 블록 압축 텍스처로 변환하는 초압축 시스템입니다:

원본 포맷원본 크기KTX2 크기압축률품질 손실
PNG (4K 알베도)42.3 MB4.8 MB88.7%시각적 동등
PNG (2K 노멀)18.7 MB3.2 MB82.9%거의 없음
JPEG (4K 사진)6.2 MB2.1 MB66.1%없음

KTX2는 ETC1S (낮은 품질, 최대 압축) 및 UASTC (높은 품질, 중간 압축) 두 모드를 지원합니다. 문화유산용으로는 UASTC를 권장합니다.

텍스처 아틀라스

여러 작은 텍스처를 하나의 큰 아틀라스로 결합하면 Draw Call을 줄이고 성능을 향상시킵니다. WIA-HERITAGE-006은 다음을 권장합니다:

  • 유사 재질 유물들을 하나의 아틀라스로 결합 (예: 청자 10점)
  • 아틀라스 최대 크기: 4096×4096 (모바일 호환성)
  • 채널 패킹: Metallic (R), Roughness (G), AO (B) 동일 텍스처에 저장
  • MIP 맵 자동 생성으로 원거리 렌더링 최적화

04.5 좌표계 및 단위

3D 스캔 데이터는 일관된 좌표계와 단위를 사용해야 상호운용이 가능합니다.

좌표계 표준

좌표계설명적용 대상
WGS84 (EPSG:4326)GPS 좌표계 (위도/경도)글로벌 위치 참조
UTMUniversal Transverse Mercator지역 좌표 (미터 단위)
Local Tangent Plane스캔 위치 기준 로컬 좌표단일 사이트 스캔
Y-up Right-handedglTF 기본 좌표계3D 뷰어/게임 엔진

단위 표준

WIA-HERITAGE-006은 일관된 단위 사용을 요구합니다:

  • 길이: 미터 (m) - glTF는 미터 기준
  • 각도: 라디안 (회전 행렬) 또는 도 (메타데이터)
  • 시간: ISO 8601 (예: "2024-01-15T14:30:00+09:00")
  • 색상: sRGB 색 공간 (0-255 또는 0.0-1.0)

좌표 변환

다른 좌표계 간 변환은 다음 도구를 사용합니다:

  • PROJ: 오픈소스 좌표 변환 라이브러리
  • GDAL: 지리공간 데이터 추상화 라이브러리
  • CloudCompare: GUI 기반 포인트 클라우드 좌표 변환

04.8 장 요약

✅ 핵심 내용
  1. E57 포인트 클라우드: ASTM 표준으로 무손실 압축 지원. WIA 확장 필드 8개 추가 (objectID, culturalPeriod, materialType 등).
  2. glTF 2.0 메시: Khronos 표준으로 PBR 재질 지원. Draco 압축으로 90% 크기 감소. 5단계 LOD 피라미드 권장.
  3. 메타데이터 스키마: Dublin Core 15개 요소 + WIA 확장 7개. JSON-LD 인코딩으로 E57/glTF에 임베드.
  4. 텍스처 최적화: KTX2/Basis Universal로 70-90% 압축. 4K 텍스처 → 4.8MB. 채널 패킹 및 아틀라스 권장.
  5. 좌표계 표준: WGS84 (글로벌), UTM (지역), Y-up Right-handed (glTF). 단위는 미터, 시간은 ISO 8601.
  6. 파일 명명: {countryCode}_{siteID}_{objectID}_{scanDate}_{LOD}_{format}.{ext} 패턴. 표준 디렉토리 구조 7개 폴더.
  7. 품질 검증: 8개 항목 체크리스트. heritage-validator 및 glTF-Validator 도구 사용.

04.9 복습 문제

  1. E57 파일 구조의 7가지 주요 구성 요소를 나열하고, WIA-HERITAGE-006이 추가한 8개 확장 필드의 용도를 설명하세요.
  2. glTF 2.0의 PBR 재질 시스템에서 사용하는 5가지 텍스처 맵의 역할을 설명하고, 각각의 권장 해상도와 포맷을 제시하세요.
  3. WIA-HERITAGE-006 메타데이터 스키마의 22개 필수 필드를 Dublin Core 15개와 WIA 확장 7개로 분류하여 나열하세요.
  4. KTX2/Basis Universal 텍스처 압축의 원리와 효과를 설명하고, 4K PNG 알베도 맵의 압축 전후 크기 비교 사례를 제시하세요.
  5. 5단계 LOD 피라미드의 각 단계별 폴리곤 수, 텍스처 해상도, 파일 크기, 적용 대상을 표로 정리하세요.
  6. Phase 1 구현 완료를 위한 8개 품질 검증 항목을 나열하고, 각각의 검증 방법과 통과 기준을 설명하세요.
