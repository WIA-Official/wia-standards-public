Chapter Summary

Key Takeaways:

  1. DID Methods define the complete lifecycle operations (create, read, update, deactivate) for identifiers on specific platforms - each method specification describes how to encode DIDs, publish DID Documents, and resolve identifiers using platform-specific mechanisms like smart contracts (did:ethr, did:polygon), distributed ledgers (did:indy), or web infrastructure (did:web).
  2. Ethereum-based DID methods (did:ethr) leverage ERC-1056 identity registries for cost-efficient DID management, storing only essential control keys on-chain while enabling off-chain DID Document storage, balancing decentralization with practical gas costs and resolution performance.
  3. Polygon DIDs (did:polygon) provide Layer 2 scalability for DID operations, offering significantly lower transaction costs than Ethereum mainnet while maintaining security through periodic checkpointing to Ethereum, making blockchain-based identity economically viable for high-volume consumer applications.
  4. Hyperledger Indy (did:indy) introduces permissioned ledger architecture optimized for enterprise identity, featuring specialized nodes for DID resolution, built-in schema and credential definition management, and privacy-preserving features including ZKP-based selective disclosure and correlation-resistant identifiers.
  5. DID Resolution transforms DIDs into DID Documents through method-specific resolvers that query appropriate data sources, returning not only the DID Document but also resolution metadata (content-type, resolution time) and document metadata (created, updated, deactivated status) essential for security-conscious verifiers.
  6. Universal Resolvers provide unified interfaces abstracting method-specific resolution logic, enabling applications to resolve any DID method through standard APIs while resolver drivers handle blockchain queries, HTTP requests, or peer-to-peer protocols as required by each method specification.
  7. Method selection impacts operational costs, privacy characteristics, regulatory compliance, and ecosystem interoperability - Universal Identity supports multi-method deployments allowing organizations to choose appropriate methods for different use cases while maintaining consistent application-level APIs.

Review Questions

  1. Explain the DID URL syntax including path, query, and fragment components. How do DID URLs enable fine-grained references to specific keys, service endpoints, or resources within DID Documents, and why is this important for credential verification workflows?
  2. Compare did:ethr and did:polygon in terms of transaction costs, finality time, and security model. For a consumer identity application with 1M users creating DIDs, calculate estimated costs for both methods and explain when each method is economically optimal.
  3. Describe Hyperledger Indy's approach to privacy-preserving DIDs including pairwise DIDs and DID rotation. How does Indy's architecture prevent correlation across verifiers while maintaining cryptographic proof of identity? Compare with public DID approaches on Ethereum.
  4. Analyze the did:web method specification. What are the security implications of using DNS and HTTPS for DID resolution? How can did:web DIDs integrate with Universal Consent to provide consent-aware authentication while avoiding centralized identity provider dependencies?
  5. Design a DID resolution caching strategy for an enterprise application verifying 10,000 credentials per hour. Address cache invalidation, DID Document updates, handling deactivated DIDs, and preventing resolution metadata timing attacks that could leak information about verification patterns.
  6. Evaluate the tradeoffs between method-specific optimizations and universal resolver abstraction. When should applications use method-specific resolution logic versus universal resolvers? How does Universal Identity balance these concerns in its reference implementation?
  7. Discuss cross-chain DID resolution scenarios where credentials issued on one blockchain reference DIDs on another (e.g., Polygon credential referencing Ethereum issuer DID). What are the security considerations, and how should verifiers handle multi-chain verification workflows?
