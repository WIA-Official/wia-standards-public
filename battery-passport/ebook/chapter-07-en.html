Chapter 7

Blockchain for Supply Chain Transparency

Chapter Overview: This chapter explores the implementation of blockchain technology for enhancing supply chain transparency and data integrity in battery passport systems. Distributed ledger technology provides immutable audit trails and enables trust among supply chain participants.

Why Blockchain for Battery Passports

Battery supply chains involve numerous participants across multiple jurisdictions, each maintaining their own records. This fragmentation creates challenges for data verification, traceability, and trust. Blockchain technology addresses these challenges by providing a shared, immutable ledger that all participants can trust without relying on a central authority.

The WIA-BATTERY_PASSPORT standard supports blockchain integration as an optional but recommended layer for applications requiring enhanced trust guarantees. The blockchain layer complements rather than replaces traditional database systems, providing cryptographic proof of data integrity while allowing efficient data storage and retrieval through conventional systems.

Blockchain Architecture Options

Different blockchain architectures offer different trade-offs between decentralization, performance, and governance. The WIA standard supports multiple approaches to accommodate diverse deployment scenarios.

Architecture Characteristics Use Cases
Public Blockchain Fully decentralized, permissionless, high trust Anchoring critical events, public verification
Consortium Blockchain Semi-decentralized, permissioned, industry governance Supply chain tracking, B2B data sharing
Private Blockchain Centralized control, high performance Internal traceability, enterprise deployment
Hybrid Private chain with public anchoring Best of both: performance + transparency

Smart Contract Implementation

Smart contracts automate verification, access control, and business logic for battery passport operations. The WIA standard provides reference implementations for common smart contract patterns.

// Solidity: Battery Passport Smart Contract
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.19;

import "@openzeppelin/contracts/access/AccessControl.sol";
import "@openzeppelin/contracts/utils/Counters.sol";

contract WIABatteryPassport is AccessControl {
    using Counters for Counters.Counter;

    bytes32 public constant MANUFACTURER_ROLE = keccak256("MANUFACTURER");
    bytes32 public constant OPERATOR_ROLE = keccak256("OPERATOR");
    bytes32 public constant RECYCLER_ROLE = keccak256("RECYCLER");

    struct BatteryRecord {
        string batteryId;
        address manufacturer;
        uint256 productionTimestamp;
        bytes32 passportHash;
        LifecycleStatus status;
        uint256 lastUpdateBlock;
    }

    struct LifecycleEvent {
        bytes32 eventHash;
        EventType eventType;
        address reporter;
        uint256 timestamp;
        string ipfsReference;
    }

    enum LifecycleStatus {
        Manufactured,
        InUse,
        SecondLife,
        InRecycling,
        Recycled
    }

    enum EventType {
        Production,
        QualityCheck,
        Shipment,
        Installation,
        Maintenance,
        SoHUpdate,
        OwnershipTransfer,
        SecondLifeTransition,
        RecyclingStart,
        RecyclingComplete
    }

    mapping(string => BatteryRecord) public batteries;
    mapping(string => LifecycleEvent[]) public batteryEvents;

    Counters.Counter private _eventCounter;

    event BatteryRegistered(
        string indexed batteryId,
        address manufacturer,
        bytes32 passportHash
    );

    event LifecycleEventRecorded(
        string indexed batteryId,
        EventType eventType,
        bytes32 eventHash,
        uint256 eventIndex
    );

    event StatusChanged(
        string indexed batteryId,
        LifecycleStatus oldStatus,
        LifecycleStatus newStatus
    );

    constructor() {
        _grantRole(DEFAULT_ADMIN_ROLE, msg.sender);
    }

    function registerBattery(
        string calldata batteryId,
        bytes32 passportHash
    ) external onlyRole(MANUFACTURER_ROLE) {
        require(
            batteries[batteryId].productionTimestamp == 0,
            "Battery already registered"
        );

        batteries[batteryId] = BatteryRecord({
            batteryId: batteryId,
            manufacturer: msg.sender,
            productionTimestamp: block.timestamp,
            passportHash: passportHash,
            status: LifecycleStatus.Manufactured,
            lastUpdateBlock: block.number
        });

        emit BatteryRegistered(batteryId, msg.sender, passportHash);
    }

    function recordEvent(
        string calldata batteryId,
        EventType eventType,
        bytes32 eventHash,
        string calldata ipfsReference
    ) external {
        require(
            batteries[batteryId].productionTimestamp != 0,
            "Battery not registered"
        );

        LifecycleEvent memory newEvent = LifecycleEvent({
            eventHash: eventHash,
            eventType: eventType,
            reporter: msg.sender,
            timestamp: block.timestamp,
            ipfsReference: ipfsReference
        });

        batteryEvents[batteryId].push(newEvent);
        batteries[batteryId].lastUpdateBlock = block.number;

        uint256 eventIndex = batteryEvents[batteryId].length - 1;
        emit LifecycleEventRecorded(
            batteryId,
            eventType,
            eventHash,
            eventIndex
        );
    }

    function verifyPassportHash(
        string calldata batteryId,
        bytes32 expectedHash
    ) external view returns (bool) {
        return batteries[batteryId].passportHash == expectedHash;
    }

    function getEventCount(
        string calldata batteryId
    ) external view returns (uint256) {
        return batteryEvents[batteryId].length;
    }
}

Data Anchoring Strategies

Rather than storing all passport data on-chain (which would be prohibitively expensive and slow), the WIA standard uses anchoring strategies that store cryptographic hashes on the blockchain while keeping full data in off-chain storage.

Hash Anchoring Process

1. Passport data is stored in traditional database

2. SHA-256 hash of data is computed

3. Hash is recorded on blockchain with timestamp

4. Any data modification creates new hash, detectable on verification

Merkle Tree Aggregation

For efficiency, multiple events can be aggregated into a Merkle tree, with only the root hash anchored on-chain. This dramatically reduces on-chain costs while maintaining the ability to verify individual events.

// TypeScript: Merkle Tree Event Aggregation
interface MerkleTreeService {
  addEvent(event: LifecycleEvent): void;
  computeRoot(): string;
  getProof(eventId: string): MerkleProof;
  verifyProof(eventId: string, proof: MerkleProof): boolean;
}

class WIAMerkleService implements MerkleTreeService {
  private tree: MerkleTree;
  private events: Map<string, LifecycleEvent>;

  constructor() {
    this.events = new Map();
    this.tree = new MerkleTree([], sha256);
  }

  addEvent(event: LifecycleEvent): void {
    const eventHash = this.computeEventHash(event);
    this.events.set(event.eventId, event);
    this.tree.addLeaf(eventHash);
  }

  computeRoot(): string {
    return this.tree.getRoot().toString('hex');
  }

  getProof(eventId: string): MerkleProof {
    const event = this.events.get(eventId);
    if (!event) throw new Error('Event not found');

    const eventHash = this.computeEventHash(event);
    const proof = this.tree.getProof(eventHash);

    return {
      eventId: eventId,
      eventHash: eventHash,
      proof: proof.map(p => ({
        position: p.position,
        data: p.data.toString('hex')
      })),
      root: this.computeRoot()
    };
  }

  verifyProof(eventId: string, proof: MerkleProof): boolean {
    const event = this.events.get(eventId);
    if (!event) return false;

    const eventHash = this.computeEventHash(event);
    return this.tree.verify(
      proof.proof,
      eventHash,
      proof.root
    );
  }

  private computeEventHash(event: LifecycleEvent): string {
    const data = JSON.stringify({
      eventId: event.eventId,
      batteryId: event.batteryId,
      eventType: event.eventType,
      timestamp: event.timestamp,
      data: event.eventData
    });
    return sha256(data);
  }
}

// Periodic anchoring service
class AnchoringService {
  private merkleService: MerkleTreeService;
  private blockchain: BlockchainClient;
  private anchorInterval: number = 3600000; // 1 hour

  async anchorPendingEvents(): Promise<AnchorReceipt> {
    const root = this.merkleService.computeRoot();
    const eventCount = this.merkleService.getEventCount();

    const tx = await this.blockchain.anchor({
      root: root,
      eventCount: eventCount,
      timestamp: Date.now()
    });

    return {
      transactionHash: tx.hash,
      blockNumber: tx.blockNumber,
      merkleRoot: root,
      anchoredEvents: eventCount
    };
  }
}

Privacy Considerations

While blockchain provides transparency, some battery passport data requires confidentiality. The WIA standard supports selective disclosure through zero-knowledge proofs and encrypted data compartments.

  • Public Data: Basic identifiers, compliance status, carbon class - visible to all
  • Verified Claims: Third-party attestations without revealing underlying data
  • Encrypted Compartments: Sensitive business data accessible only to authorized parties
  • Zero-Knowledge Proofs: Prove compliance without revealing specific values

Example: Zero-Knowledge Compliance Proof

A battery can prove its carbon footprint is below the regulatory threshold without revealing the exact value. This protects competitive information while enabling verification.
