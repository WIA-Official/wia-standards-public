Chapter 3: Data Standards and Protocols

Interoperability is the cornerstone of effective battery passports. This chapter defines the data formats, communication protocols, and API specifications that enable seamless data exchange between all participants in the battery value chain. Adherence to these standards ensures that battery passport data remains accessible and meaningful regardless of the systems or organizations involved.

Data Format Specifications

The WIA-BATTERY_PASSPORT standard uses JSON as the primary data interchange format. All data exchanges must conform to the defined JSON schemas, which specify required fields, data types, validation rules, and semantic meaning. The schemas are versioned to allow for evolution while maintaining backward compatibility.

Core Schema Structure

{
    "$schema": "https://json-schema.org/draft/2020-12/schema",
    "$id": "https://wia.org/schemas/battery-passport/v1.0",
    "title": "WIA Battery Passport Schema",
    "type": "object",
    "required": ["passportId", "version", "identity", "technical"],
    "properties": {
        "passportId": {
            "type": "string",
            "pattern": "^WIA-BP-[0-9]{4}-[0-9]{6}$",
            "description": "Unique identifier for the battery passport"
        },
        "version": {
            "type": "string",
            "enum": ["1.0"],
            "description": "Schema version"
        },
        "identity": {
            "$ref": "#/$defs/IdentityBlock"
        },
        "technical": {
            "$ref": "#/$defs/TechnicalBlock"
        },
        "materials": {
            "$ref": "#/$defs/MaterialsBlock"
        },
        "carbonFootprint": {
            "$ref": "#/$defs/CarbonFootprintBlock"
        },
        "lifecycle": {
            "$ref": "#/$defs/LifecycleBlock"
        }
    }
}

Communication Protocols

The standard defines multiple communication protocols to accommodate different use cases and integration scenarios. All protocols support both synchronous and asynchronous operations.

REST API Protocol

The primary protocol for most integrations is a RESTful HTTP API. All endpoints use HTTPS with TLS 1.3 minimum. Authentication is performed using OAuth 2.0 with JWT tokens. Rate limiting and throttling protect against abuse while ensuring fair access for all participants.

// REST API Endpoints
Base URL: https://api.wia-passport.org/v1

GET    /passports/{id}           - Retrieve passport
POST   /passports                - Create new passport
PUT    /passports/{id}           - Update passport
DELETE /passports/{id}           - Deactivate passport

GET    /passports/{id}/events    - List lifecycle events
POST   /passports/{id}/events    - Add lifecycle event

GET    /passports/{id}/soh       - Get current SoH
GET    /passports/{id}/history   - Get SoH history

POST   /verify                   - Verify passport authenticity

GraphQL Protocol

For complex queries requiring flexible data selection, a GraphQL endpoint is provided. This is particularly useful for dashboard applications that need to aggregate data from multiple passports or retrieve specific subsets of information.

# GraphQL Query Example
query GetBatteryDetails($id: ID!) {
    passport(id: $id) {
        passportId
        identity {
            manufacturer {
                name
                country
            }
            productionDate
        }
        technical {
            chemistry
            nominalCapacity
            energyDensity
        }
        currentSoH
        lifecycle {
            events(last: 10) {
                eventType
                timestamp
                location {
                    facility
                    country
                }
            }
        }
    }
}

Event Streaming Protocol

For real-time monitoring applications, the standard supports event streaming via WebSocket connections and Apache Kafka topics. This enables immediate notification of lifecycle events and continuous SoH updates.

// WebSocket Connection Example
const ws = new WebSocket('wss://stream.wia-passport.org/v1');

ws.onopen = () => {
    ws.send(JSON.stringify({
        action: 'subscribe',
        passportIds: ['WIA-BP-2025-001234'],
        eventTypes: ['soh_update', 'incident_reported']
    }));
};

ws.onmessage = (event) => {
    const data = JSON.parse(event.data);
    console.log('Received event:', data);
};

Identifier Systems

Unique identification is fundamental to the passport system. The standard defines a hierarchical identifier scheme that ensures global uniqueness while maintaining human readability.

Passport ID Format

Format: WIA-BP-YYYY-NNNNNN

Where:
- WIA    = World Interoperability Alliance prefix
- BP     = Battery Passport indicator
- YYYY   = Year of passport creation
- NNNNNN = Sequential number (000001-999999)

Examples:
- WIA-BP-2025-000001 (First passport of 2025)
- WIA-BP-2025-123456 (123,456th passport of 2025)

Related Identifier Formats

Material Batch:    WIA-MB-YYYY-TTTT-NNNNNN
Cell Batch:        WIA-CB-YYYY-NNNNNN
Event ID:          WIA-EV-YYYYMMDD-NNNNNNNN
Organization ID:   WIA-ORG-CCCC-NNNNNN

Where:
- TTTT = Material type code (LITH, COBL, NICK, etc.)
- CCCC = ISO 3166-1 alpha-2 country code

Data Validation Rules

All data submitted to the passport system must pass validation checks before acceptance. The validation framework includes schema validation, business rule validation, and cross-reference validation.

  • Schema Validation: JSON structure must conform to defined schemas
  • Type Validation: Values must match specified data types
  • Range Validation: Numeric values must fall within valid ranges
  • Format Validation: Strings must match required patterns (dates, IDs, etc.)
  • Reference Validation: Foreign keys must reference existing entities
  • Temporal Validation: Events must have logical chronological order

API Authentication and Authorization

Security is paramount in the battery passport system. The authentication and authorization framework ensures that only verified participants can access data, and that access is appropriately scoped based on roles and permissions.

// OAuth 2.0 Token Request
POST /oauth/token HTTP/1.1
Host: auth.wia-passport.org
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&client_id=org_123456
&client_secret=secret_value
&scope=passport:read passport:write events:write

// Response
{
    "access_token": "eyJhbGciOiJSUzI1NiIs...",
    "token_type": "Bearer",
    "expires_in": 3600,
    "scope": "passport:read passport:write events:write"
}

// Using the Token
GET /v1/passports/WIA-BP-2025-001234 HTTP/1.1
Host: api.wia-passport.org
Authorization: Bearer eyJhbGciOiJSUzI1NiIs...

Data Privacy and Access Control

The battery passport contains both public and confidential information. The access control system ensures appropriate visibility for different stakeholder categories while protecting sensitive business data.

  • Public Access: Basic identity, chemistry type, capacity, certifications
  • Consumer Access: Environmental data, safety records, maintenance history
  • Operator Access: Detailed technical data, operational history
  • Manufacturer Access: Full production data, supply chain details
  • Regulator Access: Complete data set including confidential information
