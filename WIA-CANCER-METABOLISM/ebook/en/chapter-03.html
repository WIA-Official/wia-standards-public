The WIA-CANCER-METABOLISM standard was designed from the ground up with the philosophy of 弘益人間—creating a universal framework that benefits all humanity by enabling seamless collaboration in cancer metabolism research across institutions, countries, and technologies.

3.1 WIA Standard Architecture

The WIA-CANCER-METABOLISM standard represents a comprehensive, layered architecture designed to address every aspect of cancer metabolism data management—from basic data representation to complex clinical integration. This architecture embodies the principle of 弘益人間 by ensuring that researchers, clinicians, and patients worldwide can benefit from shared knowledge and collaborative discovery.

At its foundation, the WIA architecture consists of four interconnected layers, each building upon the previous to create a robust, extensible framework:

🏛️ Four-Layer Architecture

  • Layer 1 - Data Format Layer: Defines the fundamental data structures, schemas, and serialization formats for representing cancer metabolism data
  • Layer 2 - API Layer: Provides standardized interfaces for accessing, querying, and manipulating metabolic data programmatically
  • Layer 3 - Protocol Layer: Establishes communication protocols for secure data exchange between systems and institutions
  • Layer 4 - Integration Layer: Enables integration with existing healthcare systems, research platforms, and clinical workflows

This layered approach provides several critical advantages. First, it allows implementers to adopt the standard incrementally, starting with basic data formats and progressively adding more sophisticated capabilities. Second, it creates clear separation of concerns, making the standard easier to understand, implement, and maintain. Third, it enables different layers to evolve independently while maintaining backward compatibility—a crucial feature for long-term sustainability.

The architecture also incorporates horizontal cross-cutting concerns that span all layers:

  • Security & Privacy: Built-in support for data encryption, access control, and compliance with regulations like HIPAA and GDPR
  • Versioning & Evolution: Systematic approach to standard updates that ensures backward compatibility while enabling innovation
  • Validation & Quality: Comprehensive validation mechanisms to ensure data integrity and scientific accuracy
  • Extensibility: Well-defined extension points that allow domain-specific customizations without breaking interoperability

3.2 Four-Phase Approach

The WIA-CANCER-METABOLISM standard is implemented through a carefully designed four-phase approach. Each phase builds upon the previous one, creating a progressive path from basic data representation to full clinical integration. This phased approach reflects our commitment to 弘益人間 by ensuring that benefits reach the global research community at every stage of implementation.

3.2.1 Phase 1: Data Format Specification

Phase 1 establishes the foundational data formats and schemas that serve as the common language for cancer metabolism research. This phase defines how to represent metabolites, pathways, fluxes, and experimental conditions in a standardized, machine-readable format.

Key components of Phase 1 include:

  • Core Data Schemas: JSON and XML schemas for metabolite profiles, pathway annotations, and flux measurements
  • Controlled Vocabularies: Standardized terminology based on existing ontologies (ChEBI, KEGG, Reactome) with cancer-specific extensions
  • Metadata Specifications: Required and optional metadata fields for experimental provenance, sample information, and analytical methods
  • Quality Metrics: Standardized quality control indicators for mass spectrometry, NMR, and other analytical platforms
  • File Formats: Support for common exchange formats including JSON, XML, CSV, and HDF5

3.2.2 Phase 2: API Specification

Phase 2 defines standardized APIs that enable programmatic access to cancer metabolism data. These RESTful and GraphQL APIs provide consistent interfaces regardless of the underlying data storage or processing systems.

The API specification includes:

  • RESTful Endpoints: Standard HTTP endpoints for CRUD operations on metabolic profiles, pathways, and experiments
  • GraphQL Schema: Flexible query language for complex data retrieval and relationship traversal
  • Authentication & Authorization: OAuth 2.0 and API key-based security mechanisms
  • Rate Limiting: Fair usage policies to ensure system stability
  • Pagination & Filtering: Efficient handling of large datasets
  • Batch Operations: Bulk upload and download capabilities for high-throughput workflows

3.2.3 Phase 3: Protocol Specification

Phase 3 establishes secure communication protocols for data exchange between institutions, research networks, and clinical systems. This phase addresses the critical challenges of secure, compliant data sharing in the healthcare domain.

Protocol features include:

  • TLS/SSL Encryption: End-to-end encryption for all data transmissions
  • Federated Query: Ability to query across multiple institutions without centralizing sensitive data
  • Consent Management: Built-in support for patient consent tracking and enforcement
  • Audit Logging: Comprehensive logging of all data access and modifications
  • Data Anonymization: Standardized approaches to de-identification and pseudonymization

3.2.4 Phase 4: Clinical Integration

Phase 4 completes the standard by defining integration points with existing healthcare IT systems, including electronic health records (EHRs), laboratory information systems (LIS), and clinical decision support systems.

Integration capabilities include:

  • FHIR Compatibility: Mapping to HL7 FHIR resources for seamless EHR integration
  • DICOM Integration: Connection with medical imaging systems for radiomics-metabolomics correlation
  • LIS Connectivity: Direct integration with laboratory workflows and instruments
  • Clinical Reporting: Standardized report formats for clinical interpretation
  • Decision Support: Interfaces for metabolic biomarker-based treatment recommendations

3.3 Design Principles

The WIA-CANCER-METABOLISM standard is built on a foundation of carefully considered design principles that guide every technical decision. These principles ensure the standard remains true to the 弘益人間 philosophy while meeting the practical needs of researchers and clinicians worldwide.

3.3.1 Interoperability First

Interoperability is not just a feature—it's the core mission of the WIA standard. Every design decision is evaluated against the question: "Will this enable or hinder data exchange between different systems?" The standard achieves interoperability through:

  • Format Flexibility: Support for multiple serialization formats (JSON, XML, CSV, HDF5) to accommodate different use cases
  • Standard Vocabularies: Reliance on established ontologies and identifiers rather than creating new terminologies
  • Clear Mappings: Explicit mapping to existing standards like FHIR, DICOM, and LOINC
  • Implementation Guides: Detailed guidance for common integration scenarios

3.3.2 Extensibility Without Fragmentation

Cancer metabolism research is rapidly evolving, with new discoveries and technologies emerging constantly. The standard must accommodate innovation without fragmenting into incompatible variants. We achieve this through:

  • Extension Mechanisms: Well-defined extension points for custom fields and domain-specific data
  • Namespace Management: Controlled extension namespaces prevent naming conflicts
  • Core vs. Optional: Clear distinction between required core elements and optional extensions
  • Community Governance: Transparent process for incorporating extensions into future standard versions

3.3.3 Security and Privacy by Design

Cancer metabolism data often includes sensitive patient information. Security and privacy are not afterthoughts but fundamental architectural requirements:

  • Encryption Standards: Mandatory encryption for data at rest and in transit
  • Access Control: Fine-grained role-based access control (RBAC) and attribute-based access control (ABAC)
  • Anonymization Support: Built-in mechanisms for data de-identification and k-anonymity
  • Regulatory Compliance: Designed to facilitate HIPAA, GDPR, and other regulatory compliance
  • Audit Trails: Comprehensive logging of all data access and modifications

3.3.4 Simplicity Without Sacrificing Power

The standard must be simple enough for small research labs to adopt while powerful enough to support large-scale clinical and pharmaceutical applications. We balance simplicity and power through:

  • Progressive Complexity: Simple use cases require minimal fields; complex scenarios use optional extensions
  • Sensible Defaults: Default values and behaviors that work for most common situations
  • Clear Examples: Extensive example datasets covering common and advanced use cases
  • Tooling Support: Open-source libraries and tools that handle complexity behind simple interfaces

3.3.5 Open and Collaborative

Reflecting 弘益人間, the standard development process is open, transparent, and community-driven:

  • Open Source: All specifications, schemas, and reference implementations are freely available under MIT license
  • Public Governance: Standard evolution is guided by a diverse steering committee representing academia, industry, and clinical practice
  • Community Input: Regular public comment periods and working groups for specialized topics
  • Free Implementation: No licensing fees or restrictions on commercial or research use

3.4 Core Components

The WIA-CANCER-METABOLISM standard comprises several core components that work together to provide comprehensive coverage of cancer metabolism data needs.

3.4.1 Schema Definitions

The schema layer defines the structure and validation rules for all data types in the standard. Schemas are provided in multiple formats to support different technology stacks:

  • JSON Schema: For web applications, APIs, and modern data pipelines
  • XML Schema (XSD): For integration with legacy healthcare systems
  • Protocol Buffers: For high-performance data serialization
  • TypeScript Definitions: For type-safe web and Node.js development

Key schema categories include:

  • Metabolite Schemas: Representing individual metabolites with identifiers, concentrations, and detection metadata
  • Pathway Schemas: Encoding metabolic pathways, reactions, and regulatory relationships
  • Flux Schemas: Representing metabolic flux measurements from isotope tracing experiments
  • Profile Schemas: Complete metabolic profiles combining multiple metabolites and pathways
  • Experiment Schemas: Experimental metadata including methods, instruments, and protocols
  • Sample Schemas: Biological sample information including cancer type, stage, and patient demographics

3.4.2 Validation Rules

Comprehensive validation ensures data quality and scientific accuracy. The standard defines multiple validation levels:

  • Structural Validation: Ensures data conforms to schema requirements (required fields, data types, value ranges)
  • Semantic Validation: Verifies biological and chemical consistency (e.g., metabolite identifiers match names)
  • Cross-Reference Validation: Checks relationships between linked data elements
  • Quality Metrics Validation: Assesses analytical quality indicators (CV%, LOD, signal-to-noise)

3.5 Compatibility with Existing Standards

The WIA-CANCER-METABOLISM standard was designed to complement, not replace, existing healthcare and research standards. Compatibility is achieved through carefully designed mappings and transformation specifications.

3.5.1 HL7 FHIR Integration

Fast Healthcare Interoperability Resources (FHIR) is the dominant standard for healthcare data exchange. WIA provides bidirectional mapping to FHIR resources:

  • Observation Resource: WIA metabolite measurements map to FHIR Observation with LOINC codes
  • DiagnosticReport: WIA metabolic profiles map to FHIR DiagnosticReport
  • Specimen: WIA sample information maps to FHIR Specimen resource
  • Patient: Patient demographics use FHIR Patient resource directly
  • Procedure: Analytical methods map to FHIR Procedure resource
// Example: WIA to FHIR Observation mapping
// WIA metabolite measurement
{
  "id": "CHEBI:16015",
  "name": "lactate",
  "concentration": 12.3,
  "unit": "mmol/kg"
}

// Maps to FHIR Observation
{
  "resourceType": "Observation",
  "status": "final",
  "code": {
    "coding": [{
      "system": "http://loinc.org",
      "code": "2524-7",
      "display": "Lactate [Moles/volume] in Serum or Plasma"
    }]
  },
  "valueQuantity": {
    "value": 12.3,
    "unit": "mmol/kg",
    "system": "http://unitsofmeasure.org",
    "code": "mmol/kg"
  }
}

3.5.2 DICOM Integration

Digital Imaging and Communications in Medicine (DICOM) is essential for integrating metabolic data with medical imaging. WIA supports:

  • Spatial Registration: Linking metabolic measurements to imaging coordinates
  • DICOM SR: Structured reports containing metabolic data
  • Parametric Maps: Metabolic heatmaps as DICOM Parametric Map objects

3.5.3 Laboratory Standards

The standard integrates with existing laboratory information system standards:

  • LOINC: Laboratory observation identifiers for metabolites and assays
  • UCUM: Unified Code for Units of Measure for all quantitative values
  • HL7 v2.x: Legacy messaging for LIS integration

3.5.4 Research Data Standards

Compatibility with research-focused standards ensures seamless integration with existing scientific workflows:

  • ISA-Tab: Investigation-Study-Assay framework for experimental metadata
  • mzML: Mass spectrometry data format (WIA can reference raw mzML files)
  • SBML: Systems Biology Markup Language for pathway models
  • BioPAX: Biological Pathway Exchange for pathway data

3.6 Versioning Strategy

Standards must evolve to accommodate new scientific discoveries and technological advances while maintaining stability for existing implementations. The WIA versioning strategy balances innovation with backward compatibility.

3.6.1 Semantic Versioning

The standard uses semantic versioning (MAJOR.MINOR.PATCH):

  • MAJOR: Incremented for backward-incompatible changes (rare, requires extensive community consultation)
  • MINOR: Incremented for backward-compatible feature additions (annual cadence)
  • PATCH: Incremented for backward-compatible bug fixes and clarifications (as needed)

3.6.2 Deprecation Policy

When features must be removed or changed:

  • Deprecation Notice: At least two MINOR versions before removal
  • Migration Guides: Clear documentation of migration paths
  • Transition Support: Tools for automated data transformation
  • Extended Support: Critical bug fixes for deprecated versions during transition

3.6.3 Version Declaration

All WIA-compliant data must declare the standard version used:

{
  "wia_version": "1.2.0",
  "schema": "cancer-metabolism-profile",
  "backwards_compatible_with": ["1.0.0", "1.1.0"],
  "data": { ... }
}
