🔌 Chapter 5: Phase 2 - API Interface Design

弘益人間 (Hongik Ingan)

"APIs should broadly benefit humanity by making cancer metabolism data universally accessible, interoperable, and actionable across all research institutions and clinical systems worldwide."

In Phase 2 of the WIA-CANCER-METABOLISM standard, we transition from data structure definition to practical implementation through a comprehensive API interface. This chapter explores the design philosophy, architectural patterns, and technical specifications that enable seamless integration of cancer metabolism data into research workflows, clinical decision support systems, and therapeutic development platforms.

The API layer serves as the critical bridge between raw metabolic data and actionable clinical insights. By establishing clear, consistent, and well-documented interfaces, we enable researchers, clinicians, and software developers to build innovative applications that advance our understanding of cancer metabolism and improve patient outcomes.

5.1 API Design Philosophy - RESTful Principles for Medical Data

The WIA-CANCER-METABOLISM API adheres to RESTful (Representational State Transfer) architectural principles, providing a stateless, cacheable, and scalable interface for cancer metabolism data exchange. This design philosophy prioritizes simplicity, discoverability, and compatibility with existing healthcare IT infrastructure.

Core Design Principles

Our API design is guided by six fundamental principles that ensure reliability, security, and usability in clinical and research environments:

  • Resource-Oriented Architecture: All entities (biomarkers, metabolites, pathways, patients) are treated as resources with unique URIs, supporting intuitive navigation and data discovery
  • Stateless Communication: Each request contains all necessary information for processing, eliminating server-side session management and improving scalability
  • Standard HTTP Methods: GET for retrieval, POST for creation, PUT for updates, PATCH for partial modifications, DELETE for removal
  • HATEOAS (Hypermedia as the Engine of Application State): Responses include links to related resources, enabling dynamic API exploration
  • Versioning Strategy: API versions embedded in URLs (e.g., /v1/, /v2/) ensure backward compatibility as the standard evolves
  • Content Negotiation: Support for multiple data formats (JSON, XML, FHIR) through HTTP Accept headers

Medical Data Considerations

Healthcare data presents unique challenges that require special attention in API design:

  • HIPAA Compliance: All patient-identifiable information is encrypted in transit (TLS 1.3) and at rest (AES-256)
  • Audit Logging: Every API request is logged with timestamps, user identifiers, and data access patterns for regulatory compliance
  • Data Provenance: Each resource includes metadata about data source, collection methods, and processing history
  • Temporal Versioning: Resources maintain complete history, allowing researchers to track changes in patient metabolic profiles over time
  • Consent Management: API enforces patient consent preferences, automatically filtering data based on authorization levels

Design Wisdom: "A well-designed medical API should be so intuitive that a researcher can explore cancer metabolism data without reading extensive documentation, yet so robust that it never compromises patient privacy or data integrity."

5.2 RESTful Endpoints - Resource Organization

The WIA-CANCER-METABOLISM API organizes endpoints around four primary resource types, each representing a fundamental entity in cancer metabolism research. This structure mirrors the natural mental model of researchers and clinicians working with metabolic data.

Endpoint Hierarchy

Resource Base Endpoint Description Example
Biomarkers /api/v1/biomarkers Metabolic markers used for cancer detection and monitoring /api/v1/biomarkers/lactate
Metabolites /api/v1/metabolites Chemical compounds involved in metabolic pathways /api/v1/metabolites/glucose
Pathways /api/v1/pathways Interconnected biochemical reaction networks /api/v1/pathways/glycolysis
Patients /api/v1/patients Individual patient metabolic profiles (HIPAA-protected) /api/v1/patients/{id}/metabolites

HTTP Methods and Operations

Each resource endpoint supports standard HTTP methods for CRUD (Create, Read, Update, Delete) operations:

Method Endpoint Pattern Purpose Idempotent
GET /api/v1/biomarkers List all biomarkers with pagination Yes
GET /api/v1/biomarkers/{id} Retrieve specific biomarker details Yes
POST /api/v1/biomarkers Create new biomarker entry No
PUT /api/v1/biomarkers/{id} Replace entire biomarker resource Yes
PATCH /api/v1/biomarkers/{id} Update specific biomarker fields No
DELETE /api/v1/biomarkers/{id} Remove biomarker (soft delete) Yes

Query Parameters and Filtering

The API supports rich querying capabilities through URL parameters, enabling researchers to filter, sort, and paginate results efficiently:

// Filter biomarkers by cancer type
GET /api/v1/biomarkers?cancerType=breast&significance=high

// Search metabolites by molecular weight range
GET /api/v1/metabolites?minWeight=100&maxWeight=500

// Paginate pathway results
GET /api/v1/pathways?page=2&limit=50&sort=name

// Include related resources (expand)
GET /api/v1/patients/12345/metabolites?expand=pathways,biomarkers

// Field selection for bandwidth optimization
GET /api/v1/biomarkers?fields=id,name,level,unit

5.3 Authentication & Authorization - Security Framework

Given the sensitive nature of medical data, the WIA-CANCER-METABOLISM API implements a multi-layered security framework combining OAuth 2.0, API keys, and fine-grained permission scopes. This approach balances robust security with developer-friendly authentication flows.

Authentication Methods

The API supports three authentication methods depending on the use case and security requirements:

1. OAuth 2.0 (Preferred for Production)

OAuth 2.0 provides secure, delegated access without sharing credentials. The API supports the authorization code flow for web applications and the client credentials flow for server-to-server integrations.

// Step 1: Obtain authorization code
GET https://auth.wia-cancer-metabolism.org/oauth/authorize
  ?client_id=YOUR_CLIENT_ID
  &redirect_uri=https://yourapp.com/callback
  &response_type=code
  &scope=biomarkers:read pathways:read patients:read

// Step 2: Exchange code for access token
POST https://auth.wia-cancer-metabolism.org/oauth/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=AUTH_CODE_FROM_STEP_1
&client_id=YOUR_CLIENT_ID
&client_secret=YOUR_CLIENT_SECRET
&redirect_uri=https://yourapp.com/callback

// Response
{
  "access_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "fdb8fdbecf1d03ce5e6125c067733c0d51de209c",
  "scope": "biomarkers:read pathways:read patients:read"
}

2. API Keys (For Development and Testing)

API keys provide a simple authentication mechanism for non-production environments. Keys are passed in the Authorization header or as a query parameter.

// Header-based authentication (recommended)
curl -X GET "https://api.wia-cancer-metabolism.org/v1/biomarkers" \
  -H "Authorization: ApiKey YOUR_API_KEY_HERE"

// Query parameter authentication (for quick testing only)
curl -X GET "https://api.wia-cancer-metabolism.org/v1/biomarkers?apikey=YOUR_API_KEY_HERE"

3. JWT (JSON Web Tokens)

For internal services and microservices architectures, the API accepts signed JWT tokens that contain user identity and permissions.

// JWT token structure
{
  "header": {
    "alg": "RS256",
    "typ": "JWT"
  },
  "payload": {
    "sub": "user@institution.edu",
    "iss": "https://auth.wia-cancer-metabolism.org",
    "aud": "https://api.wia-cancer-metabolism.org",
    "exp": 1735689600,
    "iat": 1735686000,
    "scopes": ["biomarkers:read", "biomarkers:write", "patients:read"]
  }
}

Authorization Scopes

Fine-grained scopes control access to specific resource types and operations:

Scope Description Resource Access
biomarkers:read Read access to biomarker data GET /biomarkers, GET /biomarkers/{id}
biomarkers:write Create and modify biomarkers POST, PUT, PATCH /biomarkers
metabolites:read Read access to metabolite data GET /metabolites, GET /metabolites/{id}
pathways:read Read access to pathway information GET /pathways, GET /pathways/{id}
patients:read Read patient metabolic profiles (HIPAA-restricted) GET /patients/{id}/metabolites
patients:write Update patient data (requires IRB approval) POST, PUT, PATCH /patients/{id}
admin:all Full administrative access (internal use only) All endpoints including DELETE operations

5.5 Error Handling - Standard Error Codes and Messages

Robust error handling is critical for API reliability and developer experience. The WIA-CANCER-METABOLISM API uses standard HTTP status codes complemented by detailed error responses that guide developers toward resolution.

HTTP Status Code Categories

Status Code Category Meaning Action Required
200 OK Success Request completed successfully None
201 Created Success Resource created successfully None
204 No Content Success Deletion or update successful, no content returned None
400 Bad Request Client Error Invalid request syntax or parameters Fix request format
401 Unauthorized Client Error Missing or invalid authentication Provide valid credentials
403 Forbidden Client Error Authenticated but insufficient permissions Request appropriate scopes
404 Not Found Client Error Resource does not exist Verify resource ID
429 Too Many Requests Client Error Rate limit exceeded Implement backoff strategy
500 Internal Server Error Server Error Unexpected server-side failure Retry with exponential backoff
503 Service Unavailable Server Error API temporarily unavailable (maintenance) Check status page, retry later

5.6 Rate Limiting & Quotas - Fair Usage Policies

To ensure API stability and fair resource distribution across all users, the WIA-CANCER-METABOLISM API implements tiered rate limiting based on authentication method and subscription level.

Rate Limit Tiers

Tier Requests/Minute Requests/Day Burst Allowance Use Case
Anonymous 10 1,000 20 Public documentation examples
Developer (API Key) 60 10,000 100 Testing and development
Research Institution 300 100,000 500 Academic research projects
Clinical Integration 1,000 500,000 2,000 Hospital EHR systems
Enterprise 5,000 Unlimited 10,000 Pharmaceutical companies, large-scale research

Best Practices for Rate Limit Management

Implement these strategies to optimize API usage and avoid rate limit errors:

  • Exponential Backoff: When receiving 429 responses, wait progressively longer periods (1s, 2s, 4s, 8s) before retrying
  • Request Batching: Use batch endpoints to retrieve multiple resources in a single request where available
  • Caching: Cache frequently accessed resources locally to reduce redundant API calls
  • Pagination: Request smaller result sets to reduce response time and server load
  • Webhooks: Use webhook subscriptions for real-time updates instead of polling
  • Off-Peak Scheduling: Schedule large data exports during off-peak hours (typically 22:00-06:00 UTC)

5.7 SDK Overview - TypeScript, Python, Java SDKs

To accelerate development and reduce integration complexity, the WIA-CANCER-METABOLISM standard provides official SDKs in three popular programming languages. These SDKs handle authentication, request formatting, error handling, and response parsing automatically.

TypeScript/JavaScript SDK

// Installation
npm install @wia/cancer-metabolism-sdk

// Basic usage
import { WiaCancerMetabolism } from '@wia/cancer-metabolism-sdk';

// Initialize client with OAuth token
const client = new WiaCancerMetabolism({
  accessToken: 'your_oauth_token_here',
  environment: 'production' // or 'sandbox'
});

// Retrieve biomarker data
const lactate = await client.biomarkers.get('lactate');
console.log(`${lactate.name}: ${lactate.normalRange.min}-${lactate.normalRange.max} ${lactate.normalRange.unit}`);

// List all metabolites with filtering
const metabolites = await client.metabolites.list({
  filter: {
    minWeight: 100,
    maxWeight: 500
  },
  sort: 'name',
  page: 1,
  limit: 50
});

// Create new pathway entry
const newPathway = await client.pathways.create({
  name: 'Custom Metabolic Pathway',
  description: 'Experimental pathway discovered in glioblastoma',
  metabolites: ['glucose', 'pyruvate', 'lactate'],
  cancerTypes: ['glioblastoma']
});

5.10 Review Questions

Test your understanding of Phase 2 API design principles with these questions:

  1. Question: Why does the WIA-CANCER-METABOLISM API use stateless communication rather than maintaining server-side sessions?
    Answer: Stateless communication improves scalability by eliminating session storage requirements, enables horizontal scaling across multiple servers, simplifies fault recovery (no session state to restore), and allows requests to be processed by any server in a load-balanced cluster. Each request contains complete authentication and context, making the system more resilient and performant.
  2. Question: What is the difference between a 401 Unauthorized and a 403 Forbidden error, and how should a client respond to each?
    Answer: A 401 error indicates missing or invalid authentication credentials—the client should provide valid credentials (OAuth token, API key, or JWT). A 403 error means the client is authenticated but lacks sufficient permissions (scopes)—the client should request appropriate access rights from an administrator or upgrade their subscription tier. Retrying with the same credentials will not resolve a 403 error.
  3. Question: Explain the purpose of the "expand" query parameter and how it reduces API calls.
    Answer: The expand parameter allows clients to request related resources in a single API call rather than making multiple sequential requests. For example, GET /biomarkers/lactate?expand=pathways,studies returns the biomarker data along with related pathway and study information, reducing 3+ requests to a single request and significantly improving application performance.
  4. Question: Why are rate limits organized into tiers, and what happens when a developer exceeds their tier's limit?
    Answer: Tiered rate limits ensure fair resource distribution while accommodating different use cases (individual research, institutional projects, clinical systems, enterprise applications). When limits are exceeded, the API returns a 429 status code with a Retry-After header indicating when requests can resume. Clients should implement exponential backoff and consider upgrading to a higher tier if they consistently hit limits.
  5. Question: How do the official SDKs improve developer productivity compared to direct API calls?
    Answer: SDKs handle low-level concerns automatically: OAuth token refresh, request retries on transient failures, rate limit backoff, pagination navigation, response parsing into typed objects, error handling with custom exception classes, and request validation. This allows developers to focus on application logic rather than HTTP mechanics, reducing integration time from weeks to hours.
  6. Question: What security measures does the API implement to protect patient metabolic data in compliance with HIPAA?
    Answer: The API implements multiple HIPAA-compliant security layers: TLS 1.3 encryption for data in transit, AES-256 encryption at rest, OAuth 2.0 with fine-grained scopes for access control, comprehensive audit logging of all data access, automatic consent verification before returning patient data, and role-based access control requiring IRB approval for patient data scopes.

5.11 Looking Ahead - Preview of Phase 3 Protocol

Having established a robust API interface in Phase 2, we now turn our attention to Phase 3: the Cancer Metabolism Communication Protocol. While the API defines how applications request and receive metabolic data, the protocol defines how cancer metabolism systems discover each other, negotiate capabilities, and exchange data in real-time clinical and research environments.

What to Expect in Chapter 6

Phase 3 introduces the communication layer that enables:

  • Service Discovery: How metabolomics instruments, laboratory information systems, EHRs, and research databases automatically discover each other on institutional networks and across the internet
  • Capability Negotiation: Protocols for systems to exchange supported features, data formats, and analytical capabilities before initiating data transfer
  • Real-Time Data Streaming: WebSocket and Server-Sent Events for live metabolite measurements from mass spectrometry and NMR systems
  • Message Queue Integration: Asynchronous communication patterns using AMQP, Kafka, and other message brokers for high-throughput laboratory automation
  • Federated Query Processing: Distributed query execution across multiple institutions while preserving patient privacy through differential privacy and secure multi-party computation
  • Event-Driven Architecture: Webhook-based notifications for critical metabolic events (biomarker threshold crossings, pathway activation, treatment response indicators)

From Request-Response to Event-Driven

The API's request-response model works well for on-demand queries, but real-time clinical care and high-throughput research require event-driven architectures that push critical information as soon as it becomes available. Phase 3 protocols bridge this gap, transforming the WIA-CANCER-METABOLISM ecosystem from a passive data repository into an active participant in patient care and research discovery.
