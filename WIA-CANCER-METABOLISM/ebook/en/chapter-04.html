📊 Chapter 4: Phase 1 - Data Format Specification

4.1 Data Model Overview

The WIA-CANCER-METABOLISM data format specification provides a comprehensive, standardized structure for representing cancer metabolism data across all stages of research, clinical application, and therapeutic development. At its core, the data model is designed around the principle of hierarchical organization, enabling both granular detail and high-level analysis.

The data model consists of five primary structural layers that work together to capture the complete metabolic landscape of cancer:

  • Patient Layer: Demographics, clinical context, and treatment history that contextualize metabolic measurements
  • Sample Layer: Biological specimen information including tissue type, collection methods, and processing protocols
  • Measurement Layer: Raw and processed metabolic data from various analytical platforms
  • Pathway Layer: Organized metabolic pathway information linking individual metabolites to biological processes
  • Analysis Layer: Derived insights, biomarker classifications, and computational predictions

This hierarchical structure enables flexible data querying while maintaining strict validation requirements. Each layer contains specific data types with defined schemas, ensuring consistency across institutions while allowing extensibility for emerging measurement technologies and analytical approaches.

The design philosophy emphasizes machine-readability without sacrificing human comprehension. JSON was selected as the primary serialization format due to its widespread adoption, robust tooling ecosystem, and balance between flexibility and structure. The format supports both real-time streaming of metabolic measurements and batch processing of large-scale datasets.

4.2 JSON Schema Definition

The WIA-CANCER-METABOLISM standard defines a comprehensive JSON schema that serves as the authoritative specification for data validation. This schema is versioned using semantic versioning (SemVer) to ensure backward compatibility and controlled evolution of the standard.

The schema architecture follows JSON Schema Draft 2020-12 specifications, providing rich validation capabilities including type checking, range constraints, pattern matching, and cross-field dependencies. Schema composition techniques such as $ref, allOf, and oneOf enable modular schema design and reusability of common definitions.

4.2.1 Root Schema Structure

Every WIA-CANCER-METABOLISM document begins with a root object that establishes the data context and version:

{
  "$schema": "https://wiastandards.com/schemas/cancer-metabolism/v1.0/schema.json",
  "wiaVersion": "1.0.0",
  "documentType": "metabolic-profile",
  "metadata": {
    "createdAt": "2025-12-28T14:30:00Z",
    "institution": "Memorial Sloan Kettering Cancer Center",
    "laboratory": "Metabolomics Core Facility",
    "analyst": "Dr. Sarah Chen",
    "platform": "LC-MS/MS",
    "platformVersion": "Agilent 6546 Q-TOF"
  },
  "patient": { /* Patient data */ },
  "samples": [ /* Sample array */ ],
  "measurements": [ /* Measurement array */ ],
  "pathways": [ /* Pathway array */ ],
  "analysis": { /* Analysis results */ }
}

4.2.2 Schema Validation Rules

The schema enforces strict validation rules to ensure data quality and consistency:

  • Required Fields: All documents must include wiaVersion, documentType, metadata.createdAt, and at least one measurement
  • Type Enforcement: Numeric values must include units; dates must be ISO 8601 formatted; identifiers must match specified patterns
  • Range Validation: Concentration values must be non-negative; percentage values must be 0-100; pH must be 0-14
  • Cross-Field Dependencies: If pathwayId is specified in a measurement, corresponding pathway must exist in pathways array
  • Controlled Vocabularies: Specific fields (cancer type, tissue type, biomarker names) must use values from defined enumerations

4.6 Validation Rules

The WIA-CANCER-METABOLISM standard implements comprehensive validation rules to ensure data quality, biological validity, and computational reliability. These rules operate at multiple levels: syntactic validation (correct JSON structure), semantic validation (meaningful biological values), and cross-field validation (logical consistency between related fields).

4.6.1 Required Field Validation

Every WIA document must include specific required fields depending on the document type:

  • All Documents: wiaVersion, documentType, metadata.createdAt, metadata.institution
  • Metabolic Profiles: At least one measurement in measurements array
  • Patient Data: patientId (pseudonymized), cancerType from controlled vocabulary
  • Measurements: metaboliteId or metaboliteName, value, unit
  • Biomarkers: biomarkerId, name, value, unit, interpretation
  • Pathways: pathwayId, pathwayName, category

4.6.2 Data Type and Range Validation

Field Category Validation Rule Error Example Valid Example
Concentrations Must be ≥0, numeric -2.5 mmol/L ❌ 2.5 mmol/L ✓
Percentages Must be 0-100 125% ❌ 85.5% ✓
pH Values Must be 0-14 pH 15.2 ❌ pH 6.8 ✓
Timestamps ISO 8601 with timezone "2025-12-28" ❌ "2025-12-28T14:30:00Z" ✓
Cancer Types Must use TCGA codes "breast cancer" ❌ "BRCA" ✓
Units Must use UCUM standard "mM" ❌ "mmol/L" ✓

4.6.3 Cross-Field Validation Rules

Complex validation rules ensure logical consistency across related fields:

  1. Pathway References: If a measurement includes pathwayId, corresponding pathway must exist in pathways array
  2. Sample Linking: If patient data is included, all samples must reference valid patientId
  3. Unit Compatibility: Unit must be appropriate for metabolite type (e.g., glucose in mmol/L or mg/dL, not kg/m²)
  4. Temporal Ordering: In time-series data, timestamps must be chronologically ordered
  5. Statistical Consistency: If both mean and stdDev provided, CV should equal (stdDev/mean) × 100
  6. Fold Change Calculation: When tumor and normal values provided, foldChange must equal tumor/normal
  7. Confidence Scores: Quality scores and confidence values must be 0-1 or appropriate percentage ranges

4.6.4 Controlled Vocabularies

Specific fields must use values from defined controlled vocabularies to ensure consistency:

  • Cancer Types: TCGA ontology (BRCA, LUAD, COAD, GBM, etc.)
  • Tissue Types: UBERON anatomy ontology
  • Metabolite IDs: HMDB, ChEBI, or KEGG identifiers
  • Pathway IDs: KEGG, Reactome, or WIA pathway identifiers
  • Units: UCUM (Unified Code for Units of Measure)
  • Detection Methods: Controlled list (LC-MS/MS, GC-MS, NMR, enzymatic assay, etc.)

4.8 Migration from Legacy Formats

Many research institutions and clinical laboratories have existing metabolic data stored in various formats. The WIA standard provides clear migration pathways to convert legacy data into the standardized format while preserving all critical information.

4.8.1 Common Legacy Formats

Format Type Typical Source Key Challenges Migration Strategy
Excel/CSV Manual data entry, exports Inconsistent headers, mixed units, no metadata Header mapping, unit normalization, metadata extraction
Vendor XML Mass spec instruments Proprietary schemas, instrument-specific XPath extraction, field mapping, unit conversion
mzML/mzXML Proteomics/metabolomics Raw spectra focus, limited metadata Peak integration, metabolite identification, annotation
KEGG/HMDB exports Database downloads Reference data without patient context Add patient/sample layers, contextualize measurements
Custom JSON/XML In-house LIMS systems Non-standard schemas Schema mapping, field transformation, validation

4.8.2 Migration Tools and Utilities

The WIA standard includes reference implementation tools for data migration:

  • wia-converter-cli: Command-line tool for batch conversion of CSV/Excel files
  • wia-schema-mapper: Interactive tool for mapping custom schemas to WIA format
  • wia-validator: Validates converted data against WIA schema and reports errors
  • wia-unit-normalizer: Converts various unit systems to UCUM standard
  • Legacy Format Plugins: Pre-built converters for common instrument formats (Agilent, Waters, Bruker, Thermo)

4.8.3 Best Practices for Migration

  1. Inventory Assessment: Catalog all existing data sources, formats, and volumes before migration
  2. Pilot Migration: Start with small representative datasets to validate conversion accuracy
  3. Metadata Preservation: Extract and preserve all available metadata, even if not directly mapped to WIA fields (use extensions)
  4. Quality Control: Validate 100% of converted records using wia-validator before production use
  5. Dual Storage Period: Maintain both legacy and WIA formats during transition period (6-12 months)
  6. Documentation: Document all mapping decisions, unit conversions, and data transformations
  7. Version Control: Use WIA version 1.0 initially, plan upgrade path for future versions
