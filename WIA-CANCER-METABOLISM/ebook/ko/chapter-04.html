4.1 JSON 스키마 기초

WIA-CANCER-METABOLISM 표준은 JSON(JavaScript Object Notation) 형식을 데이터 교환의 기본 형식으로 채택합니다. JSON은 가볍고, 인간이 읽을 수 있으며, 거의 모든 프로그래밍 언어에서 지원되어 의료 정보학 시스템에 이상적입니다.

JSON 스키마는 JSON 데이터의 구조를 정의하고 검증하기 위한 강력한 도구입니다. 스키마를 사용하면 다음과 같은 이점이 있습니다:

  • 버전 관리: 스키마 버전을 통해 하위 호환성 유지

4.2 대사체 데이터 구조

대사체 데이터는 WIA-CANCER-METABOLISM 표준의 핵심입니다. 각 대사체는 고유한 식별자, 농도, 관련 메타데이터를 포함해야 합니다. 다음 표는 주요 대사체 데이터 필드를 정리한 것입니다:

필드명 데이터 타입 필수 여부 설명
metaboliteId string ✓ HMDB, KEGG, ChEBI 등 표준 데이터베이스 ID
name string ✓ 대사체의 화학적 명칭 또는 일반 명칭
concentration number ✓ 측정된 농도 값 (양수)
unit string ✓ 농도 단위 (μM, mM, ng/mL 등)
pathway string 관련 대사 경로 (예: 해당과정, TCA 회로)
detectionMethod string 측정 방법 (LC-MS, GC-MS, NMR 등)
confidence number 측정 신뢰도 (0-1 범위)
referenceRange object 정상 참조 범위 (min, max 값)

4.2.1 대사체 식별자 표준

대사체를 고유하게 식별하기 위해 다음과 같은 표준 데이터베이스 ID를 사용합니다:

  • HMDB (Human Metabolome Database): 인간 대사체에 대한 포괄적인 데이터베이스 (예: HMDB0000122)
  • KEGG (Kyoto Encyclopedia of Genes and Genomes): 대사 경로와 연결된 ID (예: C00031)
  • ChEBI (Chemical Entities of Biological Interest): 생물학적 관심 화학 물질 (예: CHEBI:17234)
  • PubChem CID: 화합물 식별자 (예: CID:5793)

WIA 표준은 우선적으로 HMDB ID를 사용하되, 다른 식별자도 교차 참조로 포함할 것을 권장합니다.

4.2.2 대사체 농도 측정 단위

농도 측정값은 반드시 표준 단위를 사용해야 합니다. WIA-CANCER-METABOLISM은 다음 단위를 공식 지원합니다:

단위 전체 명칭 적용 대상 환산 계수
μM 마이크로몰 혈청, 혈장 1 μM = 10⁻⁶ M
mM 밀리몰 혈청, 혈장 1 mM = 10⁻³ M
ng/mL 나노그램/밀리리터 혈청, 혈장, 소변 1 ng/mL = 10⁻⁹ g/mL
μg/mL 마이크로그램/밀리리터 혈청, 혈장, 소변 1 μg/mL = 10⁻⁶ g/mL
mg/dL 밀리그램/데시리터 혈청, 혈장 1 mg/dL = 10 mg/L
nmol/mg 나노몰/밀리그램 조직 조직 단백질 기준

D 매우 낮음 전임상 연구 또는 초기 탐색적 연구 실험적 용도만

4.4 검증 규칙

데이터 품질을 보장하기 위해 WIA-CANCER-METABOLISM은 여러 수준의 검증 규칙을 정의합니다. 이러한 규칙은 자동화된 도구로 적용할 수 있으며, 데이터 교환 시 오류를 조기에 발견합니다.

4.4.1 필수 필드 검증

다음 필드는 모든 대사체 프로파일에 반드시 포함되어야 합니다:

  • profileId: 고유 프로파일 식별자
  • patientId: 환자 식별자 (익명화)
  • sampleDate: 샘플 채취 일시
  • metabolites: 최소 1개 이상의 대사체 데이터

4.4.2 데이터 타입 검증

각 필드는 정의된 데이터 타입을 준수해야 합니다:

// TypeScript 인터페이스 예제
interface MetaboliteProfile {
  profileId: string;        // 문자열, 패턴: WCM-YYYYMMDD-XXXXXXXX
  patientId: string;        // 문자열
  sampleDate: string;       // ISO 8601 날짜/시간
  sampleType: SampleType;   // 열거형
  cancerType?: string;      // 선택적 문자열
  metabolites: Metabolite[]; // 배열, 최소 1개
  qcMetrics?: QCMetrics;    // 선택적 객체
}

enum SampleType {
  Serum = "serum",
  Plasma = "plasma",
  Urine = "urine",
  Tissue = "tissue",
  Saliva = "saliva"
}

interface Metabolite {
  metaboliteId: string;     // HMDB, KEGG 등
  name: string;
  concentration: number;    // 양수
  unit: ConcentrationUnit;
  pathway?: string;
  detectionMethod?: string;
  confidence?: number;      // 0-1 범위
}

enum ConcentrationUnit {
  Micromolar = "μM",
  Millimolar = "mM",
  NgPerML = "ng/mL",
  UgPerML = "μg/mL",
  MgPerDL = "mg/dL"
}

4.4.3 범위 검증

농도 값은 생리학적으로 타당한 범위 내에 있어야 합니다:

대사체 최소값 최대값 단위 샘플 타입
포도당 2.0 30.0 mM 혈청/혈장
젖산 0.3 20.0 mM 혈청/혈장
글루타민 100 1000 μM 혈청/혈장
피루브산 10 200 μM 혈청/혈장

4.4.4 교차 필드 검증

일부 검증 규칙은 여러 필드 간의 관계를 확인합니다:

  • 단위 일관성: concentration과 unit 필드가 일치해야 함
  • 날짜 순서: sampleDate가 미래 날짜일 수 없음
  • 참조 범위: referenceRange의 min이 max보다 작아야 함
  • 비율 계산: ratio 바이오마커의 calculatedRatio가 numerator/denominator와 일치해야 함

4.7 버전 관리 및 하위 호환성

WIA-CANCER-METABOLISM 표준은 시간이 지남에 따라 발전하므로, 적절한 버전 관리가 필수적입니다. 버전 관리는 시맨틱 버저닝(Semantic Versioning) 2.0을 따릅니다:

  • 메이저 버전 (1.x.x): 하위 호환되지 않는 변경
  • 마이너 버전 (x.1.x): 하위 호환되는 기능 추가
  • 패치 버전 (x.x.1): 하위 호환되는 버그 수정

각 데이터 페이로드는 반드시 버전 정보를 포함해야 합니다:

{
  "version": "1.0.0",
  "schemaUrl": "https://wiastandards.com/schemas/cancer-metabolism/v1.0/metabolite-profile.json",
  // ... 나머지 데이터
}

4.7.1 하위 호환성 가이드라인

새 버전을 개발할 때 다음 원칙을 따릅니다:

  • 기존 필수 필드는 제거하지 않음
  • 새 필드는 선택적(optional)으로 추가
  • 필드 타입은 변경하지 않음 (필요시 새 필드 추가)
  • 열거형(enum) 값은 추가만 가능, 제거 불가
  • 기본값(default value)은 신중하게 변경

}

확장 필드 사용 시 주의사항:

  • 확장 필드는 표준 검증을 통과하지만, 모든 시스템에서 인식되지 않을 수 있음
  • 중요한 정보는 확장 필드에만 의존하지 말고, 표준 필드와 함께 사용
  • 확장 필드 네임스페이스는 기관 또는 프로젝트 고유 식별자 포함 권장
  • 널리 사용되는 확장 필드는 향후 표준에 포함될 수 있음

📌 핵심 요약

  • WIA-CANCER-METABOLISM은 JSON Schema Draft 7 기반의 데이터 형식을 사용하여 대사체 정보를 표준화합니다
  • 대사체 데이터는 HMDB, KEGG 등 표준 식별자와 함께 농도, 단위, 관련 경로 정보를 포함해야 합니다
  • 바이오마커는 증거 수준에 따라 등급이 매겨지며, 임상적 의의와 함께 문서화됩니다
  • 다층 검증 규칙(필수 필드, 데이터 타입, 범위, 교차 검증)이 데이터 품질을 보장합니다
  • 시맨틱 버저닝과 확장 필드를 통해 표준의 유연성과 진화를 지원합니다

🎯 복습 문제

  1. JSON 스키마를 사용하는 주요 이점 4가지를 설명하시오.
  2. 대사체 식별에 사용되는 표준 데이터베이스 ID 시스템 3가지와 각각의 예제를 제시하시오.
  3. 바이오마커 증거 수준 A등급과 C등급의 차이점과 각각의 임상 적용 범위를 설명하시오.
  4. WIA-CANCER-METABOLISM 데이터 검증에서 "교차 필드 검증"이란 무엇이며, 구체적인 예를 2가지 드시오.
  5. 시맨틱 버저닝에서 메이저, 마이너, 패치 버전의 차이를 설명하고, 각각 어떤 변경 사항이 해당하는지 예를 드시오.
  6. 확장 필드(Extension Fields)의 목적과 사용 시 주의사항 3가지를 서술하시오.
