8.1 Standard Overview

The WIA-MEDICAL-DRONE-DELIVERY standard provides a comprehensive framework for implementing autonomous medical logistics systems. Built on the WIA 4-phase architecture (Data, API, Protocol, Integration), this standard enables interoperability between healthcare systems, drone operators, and regulatory infrastructure.

Guided by the principle of 弘益人間 (Hongik Ingan) - "Benefit All Humanity" - this standard aims to bring life-saving medical supplies to every corner of the world through innovative aerial logistics.

8.2 WIA 4-Phase Architecture

The WIA-MEDICAL-DRONE-DELIVERY standard implements the WIA 4-phase architecture pattern, providing structured layers for data management, API access, protocol definition, and system integration.

Architecture Philosophy

The 4-phase architecture separates concerns into distinct layers: Phase 1 (Data) defines the information model, Phase 2 (API) provides programmatic access, Phase 3 (Protocol) enables communication, and Phase 4 (Integration) connects with existing systems. This separation enables flexibility while maintaining consistency.

Phase Name Purpose Components
Phase 1 Data Layer Information modeling and storage Aircraft, flights, cargo, deliveries
Phase 2 API Layer Programmatic access and operations REST APIs, WebSocket, authentication
Phase 3 Protocol Layer Communication and data exchange Message formats, telemetry, C2
Phase 4 Integration Layer External system connectivity EHR, UTM, inventory systems

8.4 Phase 2: API Layer Specification

The API Layer provides programmatic access to medical drone delivery operations.

API Endpoints Overview

Category Endpoint Method Description
Aircraft /api/v1/aircraft GET List all aircraft
/api/v1/aircraft/{id} GET Get aircraft details
/api/v1/aircraft/{id}/status GET Get real-time status
/api/v1/aircraft/{id}/telemetry WSS Stream telemetry
Flights /api/v1/flights GET List flights
/api/v1/flights POST Create flight request
/api/v1/flights/{id} GET Get flight details
/api/v1/flights/{id}/track WSS Track flight
Deliveries /api/v1/deliveries POST Request delivery
/api/v1/deliveries/{id} GET Get delivery status
/api/v1/deliveries/{id}/confirm POST Confirm receipt
Cargo /api/v1/cargo POST Register cargo
/api/v1/cargo/{id} GET Get cargo details
/api/v1/cargo/{id}/temperature GET Get temperature log

Authentication

The API uses OAuth 2.0 with JWT tokens. Scopes include: aircraft:read, flights:manage, deliveries:request, cargo:track, admin:all. Healthcare facility integrations use service accounts with appropriate scope limitations.

8.6 Phase 4: Integration Layer Specification

The Integration Layer connects the drone delivery platform with external systems.

Integration Patterns

System Integration Method Data Exchanged
Hospital EHR FHIR R4, HL7v2 Orders, delivery confirmation
Blood Bank ISBT 128, API Blood orders, tracking
UTM/U-Space USS APIs Flight authorization, deconfliction
Remote ID ASTM F3411 Broadcast identification
Weather Services API METAR, TAF, forecasts
Inventory Systems EDI, API Stock levels, orders

UTM Integration

Integration with UAS Traffic Management systems enables airspace authorization:

  • Flight Plan Submission: Submit intended operations
  • Authorization: Receive airspace clearances
  • Conformance Monitoring: Report actual vs. planned
  • Conflict Resolution: Respond to deconfliction

✓ Implementation Best Practices

  • Start with Phase 1 data model implementation
  • Build APIs incrementally based on use cases
  • Implement healthcare integrations using standard protocols (FHIR, HL7)
  • Ensure UTM integration from initial deployment
  • Design for scalability and multi-operator scenarios

8.8 Reference Architecture

Architecture Components

  • Aircraft Systems: Autopilot, sensors, payload, communications
  • Ground Control: Dispatch, monitoring, C2 infrastructure
  • Platform Services: Flight management, fleet ops, integration hub
  • Healthcare Integration: EHR connectors, ordering interfaces
  • Regulatory Integration: UTM, Remote ID, reporting

📌 Key Takeaways

  • WIA-MEDICAL-DRONE-DELIVERY implements the 4-phase architecture: Data Layer (aircraft, flights, cargo models), API Layer (programmatic access), Protocol Layer (messaging, telemetry), Integration Layer (healthcare, UTM systems)
  • Core data models standardize representation of aircraft, flights, cargo, and deliveries with full chain of custody tracking
  • API layer provides REST and WebSocket interfaces with OAuth 2.0 authentication for secure healthcare integration
  • Protocol layer defines message formats for delivery requests, flight status, telemetry, and cargo monitoring
  • Integration capabilities connect with hospital EHR (FHIR, HL7), blood banks (ISBT 128), UTM systems, and Remote ID
  • The standard maps to FAA, EASA, ASTM, and medical product transport regulations for comprehensive compliance
