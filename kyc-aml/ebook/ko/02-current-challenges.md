# 제2장: KYC/AML 현황의 과제

## 학습 목표

이 장을 마치면 다음을 할 수 있습니다:
- KYC/AML 규정 준수로 인한 막대한 비용 구조를 이해합니다
- 금융기관이 직면한 운영 및 기술적 과제를 파악합니다
- 신원 사기 및 합성 신원의 증가하는 위협을 인식합니다
- 규제 분열과 데이터 품질 문제의 영향을 설명합니다
- 오탐(false positive) 문제가 조사 효율성에 미치는 영향을 분석합니다
- 표준화가 필요한 이유를 이해합니다

---

## 개요

KYC/AML 규정 준수는 금융 보안에 필수적이지만, 업계는 상당한 운영, 기술 및 규제 과제에 직면해 있습니다. 이 장에서는 금융기관, 고객 및 규제 기관에 영향을 미치는 문제점을 검토하고, 이러한 과제가 표준화의 필요성을 어떻게 촉진하는지 살펴봅니다.

---

## 규정 준수 부담

### 비용 폭증

금융기관은 급증하는 규정 준수 비용에 직면해 있습니다:

#### 전 세계 지출
- **2,740억 달러** - 2023년 금융 범죄 규정 준수에 전 세계적으로 지출된 금액
- **18% 증가** - 전년 대비 증가율
- **3,500억 달러** - 2027년까지 예상 도달 금액

#### 기관 규모별 비용 분석

| 기관 유형 | 연간 AML/KYC 비용 | 고객당 비용 |
|------------------|---------------------|-------------------|
| **글로벌 은행** | 5억~10억 달러 이상 | $50 - $100 |
| **지역 은행** | 5천만~2억 달러 | $30 - $75 |
| **핀테크** | 2백만~2천만 달러 | $5 - $25 |
| **신용협동조합** | 50만~5백만 달러 | $20 - $50 |

#### 비용 지출 항목

```
자원 배분:
├── 인력 (45%) - 규정 준수 담당자, 조사관, 분석가
├── 기술 (30%) - 시스템, 라이선스, 통합
├── 외부 서비스 (15%) - 데이터 제공업체, 컨설턴트, 감사자
├── 교육 (5%) - 직원 교육, 인증
└── 과태료 및 개선 (5%) - 벌금, 개선 프로그램
```

### 인력 문제

#### 인력 증가
- 대형 은행: **1,000 - 5,000명의 규정 준수 직원**
- 중견 은행: **100 - 500명의 규정 준수 직원**
- 2008년 이후 규정 준수 인력 **60% 이상 증가**

#### 인재 부족
- AML 조사관에 대한 높은 수요
- 복잡한 기술 요구사항: 법률, 기술, 분석
- 이직률: **연간 25-35%**
- 채용 소요 시간: 고위직의 경우 **3-6개월**

## 수동 프로세스의 비효율성

### 문서 중심 워크플로

전통적인 KYC는 광범위한 수동 검토를 포함합니다:

#### 온보딩 프로세스

```
1일차: 고객이 신청서 + 문서 제출
2-3일차: 분석가가 문서의 진위 여부 검토
4-5일차: 배경 조사 및 스크리닝
6-7일차: 위험 평가 및 승인 결정
8-10일차: 품질 보증 검토
11-14일차: 계좌 활성화

총: 단순한 경우 2-4주, 복잡한 경우 6-12주
```

#### 수동 검토의 문제점

| 문제 | 영향 |
|-------|--------|
| **인적 오류** | 문서의 15-20%가 재검토 필요 |
| **일관성 부족** | 분석가마다 다른 기준 적용 |
| **속도** | 고객당 평균 검토 시간: 30-60분 |
| **확장성** | 물량 급증 처리 불가 (예: 제품 출시) |
| **비용** | 노동 집약적, 거래당 비용 높음 |

### 데이터 입력 병목

**과제:**
- PDF 문서에서 수동 데이터 추출
- 여러 시스템에 정보 재입력
- **30-40%**의 필드에 수정이 필요한 오류 포함
- 기관이나 관할권 간 표준화된 형식 없음

**예시:**
```
고객이 여권 제공:
→ 분석가가 수동으로 입력: 이름, 생년월일, 여권 번호, 발급/만료 날짜
→ CRM 시스템에 데이터 입력
→ 스크리닝 시스템에 동일한 데이터 입력
→ 핵심 뱅킹 시스템에 동일한 데이터 입력
→ 각 입력 지점: 오류 발생 가능성
```

## 신원 사기 및 합성 신원

### 증가하는 사기 위협

#### 수치로 보기
- 신원 사기 손실: 미국에서만 **520억 달러 (2023년)**
- **15명 중 1명** - 매년 영향을 받는 소비자
- **1,440만 명** - 미국의 신원 사기 피해자 (2023년)
- 피해자당 평균 사기 금액: **$1,600**

### 공격 벡터

#### 1. 문서 위조
- 가짜 ID, 여권, 공과금 청구서
- 육안으로 감지하기 어려운 정교한 위조
- 다크웹 마켓플레이스에서 가짜 문서 판매 ($100-$2,000)

**탐지 과제:**
- 법의학 전문 지식 필요
- 수동 검토로는 미묘한 위조품 놓치기 쉬움
- 새로운 위조 기술이 계속 등장

#### 2. 합성 신원

**합성 신원이란?**
실제 정보와 가짜 정보를 결합하여 가상의 신원 생성:
- 실제 사회보장번호 (종종 어린이나 사망자에게서 도용)
- 가짜 이름, 생년월일, 주소
- 신용 기록을 쌓는 데 수년간의 "육성" 기간

**프로세스:**
```
1년차: 합성 신원 생성
        ↓
2년차: 담보 신용카드 신청
        ↓
3년차: 소액 구매로 신용 기록 구축
        ↓
4년차: 더 큰 신용 한도 신청
        ↓
5년차: 모든 신용을 최대한 사용하고 사라짐 ("bust out")
```

**문제의 규모:**
- **5-10%** - 합성 신원으로 인한 부실 채권
- **60억 달러 이상** - 연간 손실
- 가장 빠르게 증가하는 신원 사기 유형
- **80-85%** - 탐지 실패율

#### 3. 계정 탈취 (ATO)

범죄자가 합법적인 계정에 액세스:
- 피싱 공격
- 자격 증명 스터핑 (유출된 비밀번호 테스트)
- SIM 스와핑
- 사회 공학

**영향:**
- 평균 ATO 손실: **사건당 $12,000**
- **24%** - ATO 공격 증가 (2022-2023)
- 합법적인 고객 활동과 구별하기 어려움

#### 4. 자금 운반책 (Money Mules)

불법 자금을 이체하도록 모집된 개인:
- 종종 무의식적인 참가자 (로맨스 사기, 가짜 구인)
- 개인 계정을 사용하여 돈세탁
- 복잡한 조사 과제 생성

## 규제 분열

### 충돌하는 요구사항

전 세계적으로 운영되는 금융기관이 직면한 문제:

#### 1. 관할권 변동

국가마다 다른 규칙:

| 관할권 | 주요 차이점 |
|--------------|-----------------|
| **미국** | 실소유자 규칙, SAR 기준, FinCEN 보고 |
| **유럽연합** | 5MLD/6MLD 요구사항, 국경 간 데이터 공유 |
| **영국** | 브렉시트 후 분기, FCA 감독 |
| **싱가포르** | MAS 통지, 위험 기반 접근 방식 강조 |
| **홍콩** | HKMA 지침, 중국 본토 고려사항 |
| **UAE** | 연방 및 에미리트 수준 요구사항 |

#### 2. 충돌하는 의무

**개인정보 보호 vs. 투명성:**
- **GDPR (EU)**: 데이터 수집 최소화, 삭제 권리
- **AML 법**: 5-10년간 기록 보존, 당국과 공유

**데이터 현지화 vs. 글로벌 스크리닝:**
- 일부 관할권은 데이터를 현지에 저장하도록 요구
- 효과적인 AML은 글로벌 워치리스트 액세스 필요
- 기술적, 법적 복잡성 발생

**충돌 예시:**
```
시나리오: EU 은행이 미국인의 의심스러운 활동 탐지

GDPR 요구사항: EU 외부로 데이터 전송 최소화
미국 요구사항: FinCEN에 의심스러운 활동 보고서 공유
결과: 각 사례마다 복잡한 법적 분석 필요
```

### 규제 불확실성

#### 진화하는 요구사항
- **지속적인 변경**: 새로운 규칙, 업데이트된 지침
- **소급 적용**: 과거 관행이 갑자기 비준수가 됨
- **해석 차이**: 원칙 기반 규칙 적용 방법 불명확

#### 암호화폐 및 디지털 자산
- 규제 프레임워크가 아직 개발 중
- 암호화폐 거래를 위한 "트래블 룰"
- 탈중앙화 금융 (DeFi) 규정 준수 과제
- 대체 불가능한 토큰 (NFT) AML 요구사항 불명확

#### 신흥 기술
- 규정 준수에서 AI/ML: 규제 수용 다양
- 생체 인식 데이터: 개인정보 보호 우려, 동의 요구사항
- 클라우드 컴퓨팅: 데이터 주권 문제
- 오픈 뱅킹: 데이터 공유 프레임워크 미성숙

## 데이터 품질 및 상호 운용성

### 데이터 문제

#### 일관성 없는 형식

모든 시스템이 다른 언어를 사용:

**고객 이름 필드:**
```
시스템 A: "firstName", "lastName"
시스템 B: "given_name", "family_name"
시스템 C: "name" (단일 필드)
시스템 D: "fullName", 분할 필요
시스템 E: "surname", "forename", "middleNames"
```

**주소 형식:**
```
미국 형식:
거리 주소, 도시, 주, 우편번호

영국 형식:
집 번호, 거리, 타운, 카운티, 우편번호

아시아 형식:
국가, 성/주, 도시, 구, 거리, 건물
```

#### 불완전한 데이터

일반적인 문제:
- **35-40%** - 주요 필드가 누락된 고객 기록
- 오래된 정보 (평균적으로 고객은 5-7년마다 이사)
- 시스템 간 일관성 없는 업데이트
- "데이터 완전성"에 대한 표준 없음

#### 데이터 사일로

```
분열 문제:

고객 데이터 (CRM)  ←→  스크리닝 시스템  ←→  거래 모니터링
       ↕                                            ↕
핵심 뱅킹 시스템  ←→  문서 저장소  ←→  사례 관리
```

**결과:**
- 단일 고객 뷰 없음
- 중복 레코드 (15-20% 중복률)
- 일관성 없는 위험 등급
- 조사 지연

### 통합의 악몽

#### API 바벨탑

금융기관은 다음과 통합:
- **10-20개 이상** - 다른 벤더
- **50-100개 이상** - API 엔드포인트
- **제로 표준화** - 벤더 간

**통합 과제:**

| 벤더 유형 | 필요한 맞춤 작업 |
|-------------|---------------------|
| 신원 확인 | 맞춤 데이터 매핑, 웹훅 통합 |
| 스크리닝 제공업체 | 독점 쿼리 형식, 결과 파싱 |
| 거래 모니터링 | 맞춤 규칙 형식, 알림 통합 |
| 문서 관리 | 파일 형식 변환, 메타데이터 표준 |
| 사례 관리 | 워크플로 사용자 정의, 보고 통합 |

#### 맞춤 통합 비용

벤더당 통합:
- **개발 시간**: 2-6개월
- **비용**: $100K - $500K
- **지속적인 유지보수**: 연간 $20K - $100K
- **전환 비용**: 처음부터 다시 구축

**예시:**
```
중견 은행의 스크리닝 벤더 교체:

1-2개월: 요구사항 및 벤더 선택
3-4개월: 데이터 매핑 및 API 개발
5-6개월: 테스트 및 검증
7-8개월: 병렬 실행 및 전환
9-12개월: 문제 해결 및 최적화

총 비용: $400K - $800K
직원 배치: 3-5명의 FTE
```

### 제3자 데이터 과제

#### 데이터 제공업체 환경

기관이 의존하는 업체:
- **신원 확인**: Experian, Equifax, LexisNexis 등
- **제재 스크리닝**: Dow Jones, Refinitiv, ComplyAdvantage 등
- **PEP 데이터베이스**: 여러 지역 제공업체
- **부정적 미디어**: 뉴스 집계기, 전문 서비스
- **기업 등록부**: Dun & Bradstreet, Companies House 등

#### 문제점

**데이터 품질 변동:**
- 정확도: 제공업체 및 지역에 따라 **70-95%**
- 신흥 시장의 커버리지 격차
- 업데이트 빈도: 일일에서 월별
- 소스 간 상충되는 정보

**높은 비용:**
- 구독: 볼륨에 따라 연간 $50K - $2M 이상
- 쿼리당 요금: 검사당 $0.10 - $50
- 글로벌 커버리지를 위해 여러 제공업체 필요

**상호 운용성 없음:**
- 각 제공업체는 독점 데이터 형식 보유
- 제공업체 간 결과 비교 불가
- 데이터 집계 또는 상관 관계 파악 어려움

## 오탐 문제

### 거래 모니터링 위기

**수치:**
- **95-97%** - 알림이 오탐
- **15억 건 이상** - 업계 전체에서 연간 생성된 알림
- **30-60분** - 알림당 평균 조사 시간
- **100만 달러 이상** - 조사관당 연간 비용

### 근본 원인

#### 1. 경직된 규칙 기반 시스템

전통적인 시스템은 정적 규칙 사용:

```
IF (현금_예금 > $10,000) THEN 알림
IF (고위험_국가로_송금) THEN 알림
IF (거래_건수 > 하루 50건) THEN 알림
```

**문제점:**
- 고객 행동 패턴에 적응 불가
- 합법적인 고액 고객이 지속적으로 알림 트리거
- 범죄자들은 임계값 바로 아래로 유지하는 방법 학습

#### 2. 맥락 부족

시스템이 이해하지 못하는 것:
- 고객 직업 (부동산 중개인 = 높은 거래량)
- 생활 이벤트 (주택 구매 = 대규모 송금)
- 계절 패턴 (소매 업체 = 12월 급증)
- 지역 규범 (일부 국가에서는 현금 사용이 일반적)

#### 3. 지나치게 신중한 조정

규제 벌금에 대한 두려움으로 인해:
- "모든 것을 잡기" 위한 낮은 임계값
- 중복 규칙
- 위험 기반 접근 방식에 대한 관용 없음

**예시:**
```
시나리오: 개인 진료소의 치과의사

프로필:
- 연간 수익 $500K
- 하루 50-80건의 결제 (환자 본인부담금)
- 현금, 수표, 카드 혼합
- 정기적인 해외 여행 (컨퍼런스)

전통적인 시스템 알림:
- 높은 거래량 (매일)
- 현금 예금 (매주)
- 해외 ATM 인출 (매월)
- 결과: 연간 200건 이상의 오탐 알림
```

### 조사 병목

**프로세스:**
```
알림 생성
    ↓
대기열 (24-72시간 대기)
    ↓
초기 선별 (10-15분)
    ↓
전체 조사 (30-60분)
    ↓
감독자 검토 (15-30분)
    ↓
SAR 결정
    ↓
문서화 (SAR 제출 시 45-60분)
```

**영향:**
- **90% 이상** - 조사관 시간이 오탐에 소비됨
- 진정한 의심스러운 활동 지연 또는 누락
- 조사관 번아웃
- 지속적인 재교육이 필요한 높은 이직률

## 고객 경험 문제

### 온보딩의 마찰

#### 이탈률

디지털 채널 통계:
- **40-60%** - KYC 확인 중 이탈
- **70-80%** - 프로세스가 10분 이상 걸리면 이탈
- 잃어버린 고객당: **$100-$500** - 낭비된 고객 획득 비용

#### 이탈 이유

| 이유 | 비율 |
|--------|------------|
| 너무 많은 단계/양식 | 35% |
| 불명확한 지침 | 22% |
| 문서 업로드 문제 | 18% |
| 개인정보 보호 우려 | 12% |
| 기술적 문제 | 8% |
| 기타 | 5% |

### 반복적인 확인

**문제:**
고객은 모든 기관에서 재확인해야 함:

```
고객 여정:
A 은행 → 전체 KYC (1시간)
    ↓
B 신용카드 회사 → 전체 KYC (45분)
    ↓
C 투자 플랫폼 → 전체 KYC (1.5시간)
    ↓
D 모기지 대출 기관 → 전체 KYC (2시간 이상)
```

**고객 불만:**
- 동일한 문서를 5-10번 이상 업로드
- 동일한 질문에 반복적으로 답변
- 기존 확인에 대한 인식 없음
- 각 계정마다 몇 주씩 대기

### 개인정보 보호 우려

#### 데이터 수집 범위

고객이 불편해하는 사항:
- 광범위한 개인 정보 요청
- 불명확한 데이터 사용 및 보존
- 제3자와의 공유
- 국제 데이터 전송

#### 보안 우려

- 데이터 유출이 점점 더 흔해짐
- 신원 도용 우려
- 보안 조치에 대한 투명성 부족
- 복잡한 개인정보 보호 정책 (평균: 4,000단어 이상)

**통계:**
- **73%** - 데이터 개인정보 보호에 대해 우려하는 소비자
- **52%** - 개인정보 보호 우려로 인해 거래 포기
- **41%** - 기관이 데이터를 보호할 것이라고 신뢰하지 않음

## 기술 부채

### 레거시 시스템 제약

많은 금융기관이 다음을 사용:

#### 수십 년 된 핵심 시스템
- 1970-1990년대에 구축
- COBOL, 메인프레임 아키텍처
- 배치 처리 (실시간 아님)
- 제한된 통합 기능

#### KYC/AML에 미치는 영향
- 최신 확인 방법 지원 불가
- 실시간 스크리닝 어렵거나 불가능
- 고객 데이터 분산
- API 통합에 복잡한 미들웨어 필요

**예시:**
```
레거시 아키텍처:

핵심 뱅킹 (메인프레임, 1985)
    ↑ (야간 배치)
미들웨어 레이어 (2005)
    ↑ (맞춤 API)
최신 규정 준수 도구 (2020+)

결과: 거래 모니터링에 24시간 지연
```

### 벤더 종속

#### 문제

기관이 갇혀 있는 이유:
- 독점 데이터 형식
- 맞춤 통합
- 높은 전환 비용
- 장기 계약

#### 재무 영향

**전환 비용 예시:**
```
거래 모니터링 시스템 교체:

소프트웨어 마이그레이션: $2M - $5M
데이터 변환: $500K - $1M
통합 재작성: $1M - $3M
직원 교육: $200K - $500K
병렬 실행: $300K - $800K
위험/기회 비용: $1M - $2M

총: $5M - $12M
일정: 18-36개월
```

결과: 기관은 마이그레이션의 고통보다 차선의 시스템을 고수합니다.

## 규제 기술 격차

### 제한된 자동화

많은 프로세스가 여전히 수동:
- **고객 온보딩**: 60-70% 수동 작업
- **거래 검토**: 80-90% 수동 조사
- **SAR 준비**: 70-80% 수동 노력
- **규제 보고**: 50-60% 수동 편집

### AI/ML 채택 장벽

#### 기술적 과제
- 규제 심사를 위한 모델 설명 가능성
- 훈련 데이터 품질 및 양
- 편향 및 공정성 우려
- 오탐음 위험 (실제 범죄 놓침)

#### 조직적 장벽
- 규정 준수 팀의 데이터 과학 기술 부족
- IT 팀의 규정 준수 도메인 지식 부족
- "불 켜놓기"를 위해 우선순위가 지정된 예산
- AI/ML에 저항하는 위험 회피 문화

#### 규제 불확실성
- AI/ML 수용성에 대한 불명확한 지침
- 모델 검증 요구사항 진화 중
- AI가 오류를 범할 경우 책임 문제

### 혁신 역설

```
사이클:

규제 기관이 더 나은 결과 요구
    ↓
기관은 새로운 접근 방식 시도를 두려워함 (규제 위험)
    ↓
비효율적인 레거시 방법 계속 사용
    ↓
낮은 결과 및 높은 비용
    ↓
규제 기관이 압력 증가
    ↓
[사이클 반복]
```

## 국경 간 과제

### 정보 공유 장벽

#### 법적 장애물
- 데이터 보호법이 전송 제한
- 은행 비밀법이 공개 금지
- 상충되는 법원 명령
- 선의의 공유에 대한 안전항 없음

#### 기술적 장애물
- 표준 데이터 형식 없음
- 언어 장벽
- 시간대 지연
- 시스템 비호환성

### 환거래 은행 감소

**문제:**
- 국제 송금에 대한 높은 규정 준수 비용
- **15-20%** - 2011년 이후 환거래 관계 감소
- "위험 제거": 은행이 고위험 지역과의 관계 단절

**영향:**
- 개발도상국의 금융 접근성 감소
- 송금 비용 증가
- 무역 금융 제약
- 비공식 채널로 활동 밀어냄 (AML 목표의 반대)

## 앞으로 나아갈 길

이러한 과제는 극복할 수 없는 것이 아닙니다. 해결책은 다음이 필요합니다:

### 1. 표준화
- 공통 데이터 형식 및 API
- 상호 운용 가능한 시스템
- 공유 프로토콜 및 모범 사례

### 2. 기술 채택
- 패턴 탐지를 위한 AI/ML
- 반복 작업 자동화
- 실시간 처리
- 클라우드 기반 확장성

### 3. 협업
- 정보 공유 프레임워크
- 공유 확인을 위한 유틸리티 모델
- 업계 전체 혁신
- 규제 대화

### 4. 고객 중심성
- 간소화된 확인
- 투명성 및 제어
- 휴대 가능한 신원 자격 증명
- 개인정보 보호 기술
