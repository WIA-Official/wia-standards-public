제3장

Phase 2 — API와 실시간 알림 스트림

이벤트 등록·스크리닝·리스크 점수·여행규칙 교환·실시간 알림 API를 다룹니다.

이벤트와 스크리닝 API

POST /events는 컴플라이언스 이벤트를 등록하고 안정적인 식별자를 반환합니다. POST /screening/sanctions와 POST /screening/pep는 엔터티 정보를 받아 매칭 결과 봉투를 반환합니다. 매칭 결과는 출처별 점수·권장 트리아지 액션·증빙 추적 식별자를 함께 제공하므로, 컴플라이언스 분석가가 한눈에 사유를 검토할 수 있습니다.

리스크 점수 API

POST /risk/score는 사업자의 배포 모델을 호출하여 점수와 설명을 반환합니다. 응답에는 X-Model-Version 헤더가 포함되어 있어, 재현 검증 시 같은 모델 버전을 호출할 수 있습니다. 모델 폐기 예정 시에는 응답에 deprecation 헤더가 추가되어 소비자가 후속 모델로 점진적으로 이행할 수 있게 합니다.

여행규칙 교환 API

POST /travel-rule/exchange는 IVMS 101 봉투를 받아 수신 VASP에게 전달하고 수신 확인을 반환합니다. 엔드포인트는 IVMS 101 스키마 검증, 관할별 임계값 정책 검증, 수신자 공개키 봉투 검증, 메시지 단위 Ed25519 서명을 모두 수행합니다. 전달 실패는 사업자 정책에 따라 재시도되며, 재시도 예산이 소진되면 격리되고 사람의 검토를 위해 에스컬레이션됩니다.

실시간 알림 WebSocket

알림 스트림은 새로운 SAR/STR 초안·제재 리스트 갱신과 사업자 영향 요약·고위험 거래 검토 라우팅·모델 드리프트 임계 초과 등 사업자 의사결정에 즉시 영향을 주는 이벤트를 다중화합니다. 구독자는 알림 클래스별로 필터링할 수 있고, 브로커는 송신 후 24시간 이내 지수적 백오프로 재전송하여 일시적 네트워크 단절에도 누락이 없게 합니다.

감사 추적 API

GET /audit/{eventId}는 모든 상태 전이 시 서명된 불변 감사 추적을 반환합니다. 감사 무결성은 사업자별 일별 머클 트리에 앵커링되며, 일부 사업자는 일별 머클 루트를 공개 블록체인에 추가로 앵커링합니다. 이는 사업자가 사후에 기록을 위변조할 수 없도록 합니다.

요약

이 장에서 다룬 핵심을 한 번 더 정리합니다. 표준은 단순한 사양 문서가 아니라, 산업 전반에 걸쳐 도구와 사람이 같은 언어로 소통하기 위한 약속입니다. 본 장의 주제는 다음 장으로 자연스럽게 이어지므로, 이해가 다소 어려운 부분이 있어도 다음 장을 함께 읽고 나면 큰 그림이 또렷해집니다.

이 표준이 추구하는 가치

WIA Standards는 한 명의 학습자, 한 명의 환자, 한 명의 시민이 자기 데이터의 진정한 주인이 되어 살아갈 수 있도록 도구와 생태계의 마찰을 줄이는 것을 목표로 합니다. 표준이 잘 작동하면 사람은 표준의 존재를 의식하지 않고도 자기 권리를 누립니다. 본 표준의 모든 결정은 이 목표에 비추어 설명될 수 있어야 합니다.

적합성 시험을 통과한 구현은 표준이 보장한 인터페이스와 약속을 그대로 지킵니다. 그러나 표준은 단순히 시험을 통과하는 도구가 아니라, 그 약속이 깨지지 않도록 운영을 설계하는 마음가짐입니다. 모든 변경은 약속을 지키는 방향으로 누적되어야 하며, 약속을 깨는 변경은 충분한 폐기 기간과 공개적 합의를 거쳐야 합니다.

자주 묻는 질문

  1. 이 표준을 채택하면 비용이 얼마나 드나요? 적합성 등급에 따라 다릅니다. Tier 1 자기 선언은 도구 채택 비용만 들고, Tier 2 제3자 평가와 Tier 3 인증은 평가 기관과의 계약 비용이 추가됩니다.
  2. 레거시 시스템과의 호환은 어떻게 보장되나요? 1.x 라인 안에서는 모든 봉투·엔드포인트·프로토콜이 도달 가능해야 합니다. 어댑터 패턴으로 레거시 시스템을 표준 인터페이스 뒤에 두면 점진적 이행이 가능합니다.
  3. 표준이 폐기되면 어떻게 되나요? 12개월 폐기 창과 위원회 2/3 표결을 거쳐야 하므로, 이용자는 충분한 시간을 두고 마이그레이션을 계획할 수 있습니다.
