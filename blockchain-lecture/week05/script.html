45분 세션 1: 스마트 계약 개념과 Solidity 기초

1.1 스마트 계약이란? (10분)

▼
여러분, 지난 주까지 블록체인의 기본 구조를 배웠습니다. 오늘부터는 블록체인 위에서 실행되는 프로그램, 바로 '스마트 계약'을 배워보겠습니다.

스마트 계약(Smart Contract)은 1994년 닉 사보(Nick Szabo)가 처음 제안한 개념으로, 계약 조건이 코드로 작성되어 조건 충족 시 자동 실행되는 프로그램입니다.

  • 자동 실행: 조건이 충족되면 중개자 없이 자동으로 실행
  • 불변성: 한번 배포되면 코드를 변경할 수 없음
  • 투명성: 모든 참여자가 코드와 실행 결과를 확인 가능
  • 분산 실행: 네트워크의 모든 노드에서 동일하게 실행
자판기를 생각해 보세요. 돈을 넣고 버튼을 누르면 음료가 나옵니다. 사람이 개입하지 않죠? 스마트 계약도 마찬가지입니다. 조건이 맞으면 자동으로 실행됩니다.

핵심 포인트

스마트 계약은 단순한 프로그램이 아니라, 블록체인의 합의 메커니즘에 의해 실행이 보장되는 '신뢰 가능한 코드'입니다.

1.2 이더리움과 EVM (10분)

▼
스마트 계약을 가장 먼저 실용화한 플랫폼이 바로 이더리움입니다. 비탈릭 부테린이 2015년에 만들었죠.

EVM(Ethereum Virtual Machine)은 스마트 계약을 실행하는 가상 머신입니다.

  • 튜링 완전: 이론적으로 어떤 계산이든 수행 가능
  • 결정론적: 동일한 입력에 동일한 출력 보장
  • 격리된 환경: 외부 시스템과 직접 통신 불가
  • 가스 메커니즘: 무한 루프 방지, 리소스 사용량에 비례한 비용
EVM을 '전 세계 컴퓨터'라고 부르기도 합니다. 이더리움 네트워크의 모든 노드가 동일한 코드를 실행하니까요. 분산된 하나의 거대한 컴퓨터인 셈입니다.

주의 사항

EVM은 외부 데이터에 직접 접근할 수 없습니다. 외부 데이터가 필요한 경우 오라클(Oracle)을 사용해야 합니다. 이 부분은 이후 주차에서 다룹니다.

1.3 Solidity 언어 소개 (10분)

▼
스마트 계약을 작성하는 대표적인 언어가 Solidity입니다. JavaScript와 C++의 영향을 받았기 때문에, 프로그래밍 경험이 있으면 비교적 쉽게 배울 수 있습니다.

Solidity의 주요 특징:

  • 정적 타입 언어 (변수 타입을 명시적으로 선언)
  • 상속, 인터페이스 지원
  • 이벤트, 모디파이어 등 계약 특화 기능
  • 컴파일 후 EVM 바이트코드로 변환
// SPDX-License-Identifier: MIT pragma solidity ^0.8.0; // 가장 기본적인 스마트 계약 구조 contract MyFirstContract { // 상태 변수 string public name = "대전대학교"; // 함수 function getName() public view returns (string memory) { return name; } }
코드를 한 줄씩 살펴보겠습니다. 첫 줄은 라이선스 표시, 두 번째는 컴파일러 버전 지정입니다. contract 키워드가 클래스처럼 계약을 정의합니다.

1.4 변수와 데이터 타입 (15분)

▼
Solidity의 데이터 타입을 하나씩 살펴봅시다. 크게 값 타입과 참조 타입으로 나뉩니다.

값 타입 (Value Types):

  • bool: true/false
  • uint: 부호 없는 정수 (uint8 ~ uint256)
  • int: 부호 있는 정수
  • address: 20바이트 이더리움 주소
  • bytes1 ~ bytes32: 고정 크기 바이트 배열

참조 타입 (Reference Types):

  • string: 문자열
  • bytes: 가변 크기 바이트 배열
  • array: 배열 (고정/가변)
  • mapping: 키-값 저장소
  • struct: 사용자 정의 구조체
contract DataTypes { bool public isActive = true; uint256 public count = 0; int256 public temperature = -10; address public owner; string public greeting = "안녕하세요"; // 상태 변수의 저장 위치: storage (영구 저장) // 함수 내 변수: memory (임시) 또는 calldata (읽기전용) constructor() { owner = msg.sender; // 배포자의 주소 } }
특히 address 타입이 중요합니다. 이더리움에서 모든 계정은 address로 식별됩니다. msg.sender는 현재 함수를 호출한 주소를 나타내는 전역 변수입니다.

2.1 함수 정의와 호출 (15분)

▼
이제 스마트 계약의 핵심인 함수를 배워보겠습니다. 함수는 계약의 동작을 정의합니다.
contract FunctionExample { uint256 private storedData; // 상태를 변경하는 함수 (트랜잭션 필요, 가스비 발생) function set(uint256 x) public { storedData = x; } // 상태를 읽기만 하는 함수 (가스비 없음) function get() public view returns (uint256) { return storedData; } // 순수 함수 - 상태를 읽지도 변경하지도 않음 function add(uint256 a, uint256 b) public pure returns (uint256) { return a + b; } // payable - 이더를 받을 수 있는 함수 function deposit() public payable { // msg.value로 전송된 이더 확인 } }

함수 키워드 정리:

  • view: 상태를 읽기만 함 (가스비 없음)
  • pure: 상태를 읽지도 변경하지도 않음
  • payable: 이더를 수신할 수 있음
  • 키워드 없음: 상태를 변경함 (트랜잭션, 가스비 필요)
view와 pure 함수는 가스비가 들지 않습니다. 왜냐하면 블록체인의 상태를 변경하지 않으므로 트랜잭션이 필요 없기 때문입니다. 하지만 상태를 변경하는 함수는 모든 노드에서 실행되어야 하므로 가스비가 듭니다.

2.2 가시성 지정자 (15분)

▼
Solidity에는 네 가지 가시성 지정자가 있습니다. 객체지향 프로그래밍의 접근 제어자와 비슷하지만 블록체인 특성이 반영되어 있습니다.
  • public: 어디서든 접근 가능 (내부 + 외부 + 상속 계약)
  • private: 해당 계약 내부에서만 접근 가능
  • internal: 해당 계약 + 상속 계약에서 접근 가능 (기본값)
  • external: 외부에서만 호출 가능 (내부에서 this.func()으로 호출)
contract Visibility { uint256 public publicVar = 1; // 자동 getter 생성 uint256 private privateVar = 2; // 외부 접근 불가 uint256 internal internalVar = 3; // 상속만 가능 function publicFunc() public pure returns (string memory) { return "누구나 호출 가능"; } function privateFunc() private pure returns (string memory) { return "내부에서만 호출"; } function externalFunc() external pure returns (string memory) { return "외부에서만 호출"; } }

보안 주의

private으로 선언해도 블록체인에 저장된 데이터는 누구나 읽을 수 있습니다! private은 다른 계약에서 접근하는 것만 막을 뿐, 데이터 자체는 공개됩니다. 민감한 데이터는 암호화하여 저장해야 합니다.

2.3 WIA-FIN-007 표준 소개 (15분)

▼
이제 WIA-FIN-007 표준에 대해 알아보겠습니다. 이 표준은 금융 분야 스마트 계약의 설계와 구현에 대한 국제 가이드라인입니다.

WIA-FIN-007의 주요 원칙:

  • 감사 가능성(Auditability): 모든 상태 변경이 추적 가능해야 함
  • 접근 제어(Access Control): 역할 기반 접근 제어(RBAC) 적용
  • 에러 처리(Error Handling): 명확한 에러 메시지와 안전한 실패
  • 업그레이드 가능성(Upgradability): 프록시 패턴을 통한 안전한 업그레이드
  • 규제 준수(Compliance): KYC/AML 등 규제 요건 반영
// WIA-FIN-007 준수 계약 기본 구조 contract WIACompliant { address public admin; // 이벤트를 통한 감사 로그 event ActionPerformed(address indexed actor, string action, uint256 timestamp); // 접근 제어 modifier modifier onlyAdmin() { require(msg.sender == admin, "Admin only"); _; } constructor() { admin = msg.sender; } }
WIA-FIN-007은 특히 금융 스마트 계약에서 요구되는 보안, 투명성, 규제 준수를 체계적으로 정의합니다. 대학원 과정에서는 이 표준의 철학과 설계 원칙을 이해하는 것이 중요합니다.

3.2 Hello World 계약 작성 (15분)

▼
화면을 보면서 함께 코드를 작성하겠습니다. 한 줄씩 설명하면서 진행하겠습니다.
// SPDX-License-Identifier: MIT pragma solidity ^0.8.19; contract HelloWorld { // 상태 변수: 블록체인에 영구 저장 string private greeting; address public owner; uint256 public greetingCount; // 이벤트: 로그 기록용 event GreetingChanged(address indexed changer, string newGreeting); // 생성자: 배포 시 한 번만 실행 constructor(string memory _greeting) { greeting = _greeting; owner = msg.sender; greetingCount = 0; } // 인사말 조회 function getGreeting() public view returns (string memory) { return greeting; } // 인사말 변경 (소유자만 가능) function setGreeting(string memory _newGreeting) public { require(msg.sender == owner, "Only owner can change"); greeting = _newGreeting; greetingCount++; emit GreetingChanged(msg.sender, _newGreeting); } }
require 문이 중요합니다. 조건이 거짓이면 트랜잭션이 취소되고 가스비도 환불됩니다. 접근 제어의 기본입니다. 이것이 바로 WIA-FIN-007에서 말하는 접근 제어 패턴의 가장 간단한 형태입니다.

3.4 함수 호출 테스트 (10분)

▼
배포된 계약의 함수를 호출해 봅시다.
  1. getGreeting 클릭 - 초기 인사말 확인
  2. owner 클릭 - 배포자 주소 확인
  3. setGreeting에 새 문자열 입력 후 호출
  4. 다시 getGreeting으로 변경 확인
  5. greetingCount로 변경 횟수 확인
  6. 다른 계정으로 전환 후 setGreeting 시도 - 에러 확인
다른 계정으로 setGreeting을 호출하면 require에 의해 트랜잭션이 실패합니다. 콘솔에서 에러 메시지 "Only owner can change"를 확인할 수 있습니다. 이것이 접근 제어의 기본입니다.
