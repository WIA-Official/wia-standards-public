### 1-2. 스마트 계약이란? (10분)

**교수 멘트:**

> 자, 스마트 계약의 정의부터 봅시다. 1994년에 닉 사보라는 컴퓨터 과학자가 처음 제안한 개념인데요, 당시에는 기술적으로 구현이 어려웠어요. 그러다가 2015년에 이더리움이 등장하면서 비로소 실현된 겁니다.

#### 핵심 개념 정리

- **스마트 계약 (Smart Contract)**: 블록체인 위에서 실행되는 자동화된 프로그램
- **특징**: 한번 배포하면 변경 불가(불변성), 누구나 검증 가능(투명성), 자동 실행(자율성)
- **동작 원리**: 조건(if) → 실행(then) → 결과가 블록체인에 기록

**교수 멘트:**

> 예를 들어볼게요. "만약 A가 B에게 10만원을 보내면, B의 디지털 자산 소유권을 A에게 이전한다." 이런 계약을 코드로 작성해서 블록체인에 올리는 거예요. 조건이 충족되면 자동으로 실행되니까, 중개인이 필요 없어지는 거죠.
>
> 은행 없이도, 변호사 없이도, 부동산 중개사 없이도 거래가 가능해지는 겁니다. 물론 아직 법적인 부분은 정리가 필요하지만요.

#### WIA-FIN-007 표준 소개

**교수 멘트:**

### 1-3. Solidity 기초 문법 (15분)

**교수 멘트:**

> 이제 실제로 스마트 계약을 어떻게 작성하는지 봅시다. 이더리움에서 스마트 계약을 만들 때 가장 많이 사용하는 언어가 **Solidity**입니다. JavaScript와 비슷한 문법을 가지고 있어서, 프로그래밍 경험이 있는 분들은 비교적 빨리 익힐 수 있어요.

#### 기본 구조

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract HelloWorld {
    // 상태 변수
    string public greeting = "Hello, Blockchain!";

    // 함수
    function setGreeting(string memory _greeting) public {
        greeting = _greeting;
    }

    function getGreeting() public view returns (string memory) {
        return greeting;
    }
}
```

**교수 멘트:**

> 하나씩 뜯어봅시다. 맨 위에 `pragma solidity ^0.8.0`은 컴파일러 버전을 지정하는 겁니다. "이 코드는 0.8.0 이상의 Solidity 컴파일러로 컴파일해 주세요"라는 뜻이에요.
>
> `contract`는 다른 언어의 `class`와 비슷합니다. 스마트 계약의 단위죠. 그 안에 변수와 함수를 정의합니다.

#### 변수 타입

| 타입 | 설명 | 예시 |
|------|------|------|
| `uint` | 부호 없는 정수 | `uint256 balance = 100;` |
| `int` | 부호 있는 정수 | `int256 temperature = -10;` |
| `bool` | 불리언 | `bool isActive = true;` |
| `address` | 이더리움 주소 (20바이트) | `address owner = msg.sender;` |
| `string` | 문자열 | `string name = "홍길동";` |
| `bytes` | 바이트 배열 | `bytes32 hash;` |
| `mapping` | 키-값 저장소 | `mapping(address => uint) balances;` |

**교수 멘트:**

> 여기서 주목할 타입이 `address`입니다. 다른 프로그래밍 언어에는 없는, 블록체인 특유의 타입이죠. 이더리움 네트워크의 지갑 주소나 계약 주소를 저장할 때 사용합니다.
>
> 그리고 `mapping`! 이건 정말 많이 씁니다. 파이썬의 딕셔너리, 자바의 HashMap과 비슷한데요, "이 주소의 잔액이 얼마다"를 기록할 때 딱 좋은 구조입니다.

#### 함수의 가시성과 상태 변경자

```solidity
// public: 누구나 호출 가능
function publicFunc() public { }

// private: 계약 내부에서만 호출 가능
function privateFunc() private { }

// view: 상태를 읽기만 함 (가스비 없음)
function viewFunc() public view returns (uint) { }

// pure: 상태를 읽지도 쓰지도 않음
function pureFunc(uint a, uint b) public pure returns (uint) {
    return a + b;
}
```

**교수 멘트:**

> `view`와 `pure`는 꼭 기억해 두세요. `view` 함수는 블록체인의 데이터를 읽기만 하니까 가스비가 안 듭니다. `pure`는 아예 블록체인 데이터를 읽지도 않고 순수하게 계산만 하는 함수입니다.
>
> 반면에 상태를 변경하는 함수는 트랜잭션이 필요하고, 트랜잭션에는 가스비가 들어갑니다. 이 차이를 이해하는 게 중요해요. 나중에 가스 최적화를 할 때 핵심이 됩니다.

#### 실습 단계

- 별도 설치 없이 웹 브라우저에서 바로 사용 가능

**교수 멘트:**

**Step 2: 새 파일 생성**

1. 왼쪽 파일 탐색기에서 `contracts` 폴더 클릭
2. 새 파일 아이콘 클릭
3. 파일명: `HelloWorld.sol`

**Step 3: 코드 작성**

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

// WIA-FIN-007 표준 준수 기본 스마트 계약
contract HelloWorld {
    // 상태 변수: 인사말 저장
    string public greeting;

    // 계약 소유자 주소
    address public owner;

    // 이벤트: 인사말 변경 시 발생
    event GreetingChanged(string oldGreeting, string newGreeting, address changedBy);

    // 생성자: 계약 배포 시 한 번만 실행
    constructor(string memory _initialGreeting) {
        greeting = _initialGreeting;
        owner = msg.sender;
    }

    // 인사말 변경 함수
    function setGreeting(string memory _newGreeting) public {
        string memory oldGreeting = greeting;
        greeting = _newGreeting;
        emit GreetingChanged(oldGreeting, _newGreeting, msg.sender);
    }

    // 인사말 조회 함수 (view - 가스비 없음)
    function getGreeting() public view returns (string memory) {
        return greeting;
    }

    // 소유자 확인 함수
    function isOwner() public view returns (bool) {
        return msg.sender == owner;
    }
}
```

**교수 멘트:**

> 코드를 한 줄씩 같이 작성해 봅시다. 따라 치면서 각 줄이 무슨 역할을 하는지 이해하는 게 중요합니다. 그냥 복붙하면 아무것도 남지 않아요.
>
> 여기서 `event`는 로그 역할을 합니다. 인사말이 바뀔 때마다 기록이 남는 거죠. 프론트엔드에서 이 이벤트를 감지해서 UI를 업데이트할 수도 있습니다.
>
> 그리고 `constructor`는 생성자입니다. 계약을 처음 배포할 때 딱 한 번만 실행되요. 초기값을 설정할 때 사용합니다.

### 3-2. Q&A 및 토론 (10분)

**교수 멘트:**

**예상 질문과 답변:**

**Q1: 스마트 계약을 한번 배포하면 정말 수정이 안 되나요?**

> 네, 기본적으로는 수정이 안 됩니다. 그래서 배포 전에 테스트를 철저히 해야 해요. 하지만 "프록시 패턴"이라는 기법을 사용하면 업그레이드가 가능합니다. 이건 나중에 심화 과정에서 다루겠습니다.

**Q2: Solidity 말고 다른 언어로도 스마트 계약을 만들 수 있나요?**

> 물론이죠. Vyper라는 파이썬 스타일의 언어도 있고, Rust로 작성하는 Solana 체인의 스마트 계약도 있어요. 하지만 이더리움 생태계에서는 Solidity가 압도적이고, 취업 시장에서도 Solidity 수요가 가장 많습니다.

**Q3: 스마트 계약이 해킹될 수도 있나요?**

> 아주 좋은 질문입니다. 네, 실제로 2016년에 The DAO 해킹 사건이 있었어요. 약 360만 ETH가 탈취당했습니다. 그래서 WIA-FIN-007 같은 보안 표준이 중요한 거예요. 다음 주에 보안 관련 내용을 더 자세히 다룰 겁니다.
