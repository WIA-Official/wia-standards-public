45분 세션 1: 조건문, 반복문, mapping, struct, modifier

1.1 조건문과 반복문 (10분)

▼
지난 주에 Solidity 기초를 배웠습니다. 오늘은 더 복잡한 로직을 구현할 수 있는 심화 문법을 배우겠습니다.
contract ControlFlow { // if-else 조건문 function checkAge(uint256 age) public pure returns (string memory) { if (age >= 65) { return "시니어"; } else if (age >= 19) { return "성인"; } else { return "미성년"; } } // for 반복문 - 가스 주의! function sum(uint256 n) public pure returns (uint256) { uint256 total = 0; for (uint256 i = 0; i < n; i++) { total += i; } return total; } }

가스 경고

반복문은 매우 주의해서 사용해야 합니다. 반복 횟수가 많으면 가스 한도를 초과하여 트랜잭션이 실패합니다. 무한 루프는 가스가 소진될 때까지 실행됩니다. 블록 가스 한도(약 3천만)를 초과하면 실행 자체가 불가능합니다.

1.2 mapping과 struct (20분)

▼
mapping은 Solidity에서 가장 많이 사용되는 데이터 구조입니다. 해시 테이블과 유사하게, 키를 값에 매핑합니다.
contract DataStructures { // struct - 사용자 정의 타입 struct Student { string name; uint256 studentId; uint256 score; bool isRegistered; } // mapping - 키-값 저장소 mapping(address => Student) public students; mapping(address => mapping(uint256 => bool)) public courseEnrolled; address[] public studentList; // 열거용 배열 function register(string memory _name, uint256 _id) public { require(!students[msg.sender].isRegistered, "Already registered"); students[msg.sender] = Student(_name, _id, 0, true); studentList.push(msg.sender); } function getStudentCount() public view returns (uint256) { return studentList.length; } }
mapping은 모든 키에 대해 기본값을 반환합니다. 존재하지 않는 키를 조회하면 에러가 아닌 기본값(0, false, "")이 반환됩니다. 그래서 isRegistered 같은 플래그가 필요합니다. 또한 mapping은 열거(iteration)가 불가능하므로, 별도의 배열로 키를 관리해야 합니다.

1.3 modifier (15분)

▼
modifier는 함수 실행 전후에 조건을 검사하는 코드를 재사용할 수 있게 해줍니다. WIA-FIN-007에서 권장하는 접근 제어 패턴의 핵심입니다.
contract AccessControl { address public owner; bool public paused; modifier onlyOwner() { require(msg.sender == owner, "Not owner"); _; // 원래 함수 코드가 여기에 삽입됨 } modifier whenNotPaused() { require(!paused, "Contract is paused"); _; } modifier validAddress(address _addr) { require(_addr != address(0), "Invalid address"); _; } // modifier 여러 개 중첩 가능 function criticalAction() public onlyOwner whenNotPaused { // owner만, 일시정지 아닐 때만 실행 } function pause() public onlyOwner { paused = true; } }
modifier의 밑줄과 세미콜론(__;)은 "원래 함수의 본문을 여기에 넣어라"라는 의미입니다. modifier를 여러 개 사용하면 왼쪽에서 오른쪽으로 순서대로 실행됩니다.

WIA-FIN-007 적용

WIA-FIN-007에서는 역할 기반 접근 제어(RBAC)를 modifier로 구현할 것을 권장합니다. OpenZeppelin의 AccessControl 라이브러리가 표준 구현을 제공합니다.

45분 세션 2: 이벤트, 로깅, 가스 최적화

2.1 이벤트와 로깅 (20분)

▼
이벤트는 블록체인의 로그 시스템입니다. 스마트 계약에서 발생한 중요한 상태 변경을 기록하고, 프론트엔드에서 실시간으로 감지할 수 있게 해줍니다.
contract EventExample { // 이벤트 정의 - indexed 키워드로 필터링 가능 event Transfer( address indexed from, address indexed to, uint256 amount ); event Approval( address indexed owner, address indexed spender, uint256 value ); mapping(address => uint256) public balances; function transfer(address to, uint256 amount) public { require(balances[msg.sender] >= amount, "Insufficient"); balances[msg.sender] -= amount; balances[to] += amount; emit Transfer(msg.sender, to, amount); // 이벤트 발생 } }

이벤트의 특징:

  • indexed: 최대 3개, 이벤트 필터링에 사용 (토픽으로 저장)
  • 가스 효율적: storage에 저장하는 것보다 훨씬 저렴
  • 계약 내에서 읽기 불가: 오직 외부(프론트엔드, 인덱서)에서만 조회
  • WIA-FIN-007: 모든 상태 변경에 이벤트 기록 필수
이벤트는 스마트 계약의 "감사 로그"입니다. WIA-FIN-007에서는 모든 중요한 상태 변경에 이벤트를 필수적으로 발생시키도록 요구합니다. 이를 통해 규제 기관이나 감사인이 모든 활동을 추적할 수 있습니다.

2.2 가스 최적화 기법 (25분)

▼
가스 최적화는 실제 서비스에서 매우 중요합니다. 메인넷에서는 가스비가 실제 비용이니까요.

가스 비용 순서 (높음 -> 낮음):

  1. SSTORE (새 값 저장): ~20,000 가스
  2. SSTORE (값 변경): ~5,000 가스
  3. SLOAD (값 읽기): ~2,100 가스
  4. 메모리 연산: 3 가스~
  5. 스택 연산: 3 가스~
// 나쁜 예: storage를 반복적으로 읽기 function badSum() public view returns (uint256) { uint256 total = 0; for (uint256 i = 0; i < myArray.length; i++) { total += myArray[i]; // 매번 SLOAD } return total; } // 좋은 예: 로컬 변수에 캐싱 function goodSum() public view returns (uint256) { uint256[] memory arr = myArray; // 한 번만 SLOAD uint256 len = arr.length; uint256 total = 0; for (uint256 i = 0; i < len; i++) { total += arr[i]; // memory 읽기 (저렴) } return total; }

주요 최적화 기법:

  • storage 변수를 memory에 캐싱하여 사용
  • 변수 패킹: struct 내 작은 타입을 인접하게 배치 (uint128 + uint128 = 1슬롯)
  • 상수는 constant 또는 immutable 키워드 사용
  • 문자열 대신 bytes32 사용 (고정 길이)
  • 에러 메시지 대신 커스텀 에러 사용 (Solidity 0.8.4+)
  • calldata 사용 (external 함수의 매개변수)
  • 불필요한 storage 읽기/쓰기 최소화
실무에서 가스 최적화는 사용자 경험과 직접 연결됩니다. DeFi 프로토콜에서 트랜잭션 비용이 높으면 사용자가 떠납니다. WIA-FIN-007에서도 "합리적인 가스 효율성"을 요구합니다.

3.2 투표 계약 구현 (25분)

▼
화면을 보면서 함께 코드를 작성하겠습니다.
// SPDX-License-Identifier: MIT pragma solidity ^0.8.19; contract Voting { struct Candidate { uint256 id; string name; uint256 voteCount; } enum State { Created, Voting, Ended } address public admin; State public state; mapping(uint256 => Candidate) public candidates; mapping(address => bool) public voters; mapping(address => bool) public hasVoted; uint256 public candidateCount; uint256 public totalVotes; event CandidateAdded(uint256 id, string name); event VoterRegistered(address voter); event VoteCast(address indexed voter, uint256 candidateId); event StateChanged(State newState); modifier onlyAdmin() { require(msg.sender == admin, "Admin only"); _; } modifier inState(State _state) { require(state == _state, "Invalid state"); _; } constructor() { admin = msg.sender; state = State.Created; } function addCandidate(string memory _name) public onlyAdmin inState(State.Created) { candidateCount++; candidates[candidateCount] = Candidate(candidateCount, _name, 0); emit CandidateAdded(candidateCount, _name); } function registerVoter(address _voter) public onlyAdmin { require(!voters[_voter], "Already registered"); voters[_voter] = true; emit VoterRegistered(_voter); } function startVoting() public onlyAdmin inState(State.Created) { state = State.Voting; emit StateChanged(State.Voting); } function vote(uint256 _candidateId) public inState(State.Voting) { require(voters[msg.sender], "Not registered"); require(!hasVoted[msg.sender], "Already voted"); require(_candidateId > 0 && _candidateId <= candidateCount, "Invalid"); hasVoted[msg.sender] = true; candidates[_candidateId].voteCount++; totalVotes++; emit VoteCast(msg.sender, _candidateId); } function endVoting() public onlyAdmin inState(State.Voting) { state = State.Ended; emit StateChanged(State.Ended); } function getWinner() public view inState(State.Ended) returns (string memory, uint256) { uint256 maxVotes = 0; uint256 winnerId = 0; for (uint256 i = 1; i <= candidateCount; i++) { if (candidates[i].voteCount > maxVotes) { maxVotes = candidates[i].voteCount; winnerId = i; } } return (candidates[winnerId].name, maxVotes); } }
이 코드에서 enum은 상태 머신을 구현합니다. Created, Voting, Ended 세 가지 상태가 있고, modifier로 특정 상태에서만 함수가 실행되도록 제한합니다. 이것이 WIA-FIN-007에서 말하는 상태 관리 패턴입니다.

3.3 배포 및 테스트 (10분)

▼
Remix에서 배포하고 테스트해 봅시다.
  1. 컴파일 후 Remix VM에 배포

  3. registerVoter로 유권자 등록 (다른 계정 주소 사용)

  5. 계정 전환 후 vote 함수로 투표
  6. endVoting 후 getWinner로 결과 확인
