블록체인 개론

Blockchain Introduction

블록체인이란 무엇인가?

블록체인 (Blockchain)은 데이터를 블록(Block) 단위로 묶어 체인(Chain)처럼 연결한 분산 데이터 구조입니다. 각 블록은 이전 블록의 해시값을 포함하고 있어, 한 블록이라도 조작하면 이후 모든 블록의 해시가 깨집니다.

은행 장부를 한 곳이 아니라 수천 명이 동시에 보관하는 것과 같습니다. 한 사람이 장부를 고쳐도 나머지 수천 개와 다르니 바로 들킵니다.

블록체인의 3대 핵심 속성

속성영어의미
탈중앙화Decentralization은행·정부 같은 중앙 관리자 없이 참여자 전체가 네트워크를 운영
투명성Transparency모든 거래 기록을 누구나 열람·검증 가능
불변성Immutability한번 기록된 데이터는 사실상 수정 불가능

분산원장 기술 (DLT)

DLT (Distributed Ledger Technology, 분산원장 기술)는 여러 참여자가 동일한 장부 사본을 각자 보관하는 기술입니다. 블록체인은 DLT의 한 종류이지, 모든 DLT가 블록체인은 아닙니다.

중앙화 vs 분산화

구분중앙화 시스템분산 시스템(블록체인)
관리 주체단일 서버/기관네트워크 참여자 전체
장애 위험SPOF(단일 장애점) 존재한 노드 장애 → 전체 영향 없음
신뢰 방식중앙 기관 신뢰수학·암호학 기반 신뢰
데이터 위변조관리자가 변경 가능합의 없이는 변경 불가

SPOF (Single Point of Failure, 단일 장애점) — 중앙 서버 하나가 고장나면 전체 시스템이 멈추는 구조적 약점. 분산 시스템은 이것을 제거합니다.

DLT 유형

  • 퍼블릭(Public) — 누구나 참여 가능 (예: 비트코인, 이더리움)
  • 프라이빗(Private) — 허가된 참여자만 (예: 기업 내부 블록체인)
  • 컨소시엄(Consortium) — 복수의 기관이 공동 운영 (예: R3 Corda)

블록의 구조

블록은 크게 블록 헤더(Header)와 블록 바디(Body)로 나뉩니다.

머클 트리 (Merkle Tree)

머클 트리는 거래들을 2개씩 짝지어 해시하고, 그 결과를 다시 짝지어 해시하며 올라가 최종 하나의 머클 루트(Merkle Root)를 만드는 이진 트리 구조입니다.

토너먼트 대진표와 비슷합니다. 맨 아래에 각 거래가 있고, 2개씩 합쳐 올라가면서 최종 우승자(=머클 루트)가 결정됩니다. 하나의 거래만 바뀌어도 머클 루트가 완전히 달라집니다.

블록 바디(Body)에는 실제 거래(Transaction) 데이터가 들어갑니다.

한 블록의 데이터를 수정하면 그 블록의 해시가 변하고, 다음 블록의 "이전 블록 해시"와 불일치하므로 연쇄적으로 모든 후속 블록이 깨집니다. 전체 노드의 51% 이상을 동시에 장악하지 않으면 위변조는 사실상 불가능합니다.

Cryptography Fundamentals · WIA-SEC-002

해시 함수 (Hash Function)

해시 함수는 임의 길이의 입력을 고정 길이의 출력으로 변환하는 수학 함수입니다. 블록체인 보안의 근간입니다.

해시 함수의 5가지 핵심 성질

성질영어설명예시
결정성Determinism같은 입력 → 항상 같은 출력"Hello" → 항상 동일한 해시
단방향성Pre-image Resistance출력에서 입력을 역추적 불가해시값으로 원문 복원 불가
충돌 저항성Collision Resistance같은 해시를 만드는 두 입력을 찾기 거의 불가능2^128번 시도해야 발견
눈사태 효과Avalanche Effect입력의 미세한 변화 → 출력의 ~50%가 바뀜"Hello" → "Hello!" 완전 다른 해시
빠른 계산Fast Computation해시 계산은 빠르게 수행수 밀리초 내 완료
"고기 분쇄기"와 같습니다. 고기(입력)를 넣으면 다진 고기(해시)가 나오지만, 다진 고기에서 원래 고기 모양을 복원할 수는 없습니다.

SHA-256

비트코인에서 사용하는 대표적 해시 함수입니다.

  • 설계: NSA에서 설계, NIST가 2001년 표준화
  • 출력 크기: 256비트 = 32바이트 = 64자리 16진수(hex)
  • 가능한 출력 수: 2256 ≈ 1077 (우주의 원자 수보다 많음)

SHA 패밀리 비교

알고리즘출력상태사용처

공개키 암호화 (비대칭 암호화)

비대칭 암호화 (Asymmetric Encryption)는 암호화 키와 복호화 키가 다른 방식입니다.

키 쌍 (Key Pair)

키역할공개 여부
공개키 (Public Key)암호화 / 주소 생성 / 서명 검증모두에게 공개
개인키 (Private Key)복호화 / 서명 생성절대 비공개

개인키 → 공개키: 계산 가능 (수학적 도출)
공개키 → 개인키: 계산 불가능 (이것이 보안의 핵심)

ECC (타원곡선 암호)

ECC (Elliptic Curve Cryptography) — 비트코인과 이더리움은 secp256k1 타원곡선을 사용합니다. RSA보다 짧은 키 길이로 동등한 보안 강도를 제공합니다.

디지털 서명 (Digital Signature)

"이 거래는 내가 진짜 승인한 것이다"를 수학적으로 증명하는 메커니즘입니다.

서명 과정

[서명 생성]
메시지 → SHA-256 해시 → 해시 + 개인키 → ECDSA 알고리즘 → 서명값

[서명 검증]
메시지 → SHA-256 해시 ─┐
서명값 + 공개키 ────────┤ 비교 → 일치하면 ✅ 유효
                          └──────────────────────────
블록체인 보안 3요소: 해시(무결성) + 공개키(신원) + 디지털 서명(인가)

양자내성 암호 (PQC)

PQC (Post-Quantum Cryptography, 양자내성 암호) — 양자 컴퓨터가 기존 암호를 무력화할 수 있는 미래에 대비한 차세대 암호 기술입니다.

양자 위협

  • Shor 알고리즘 — RSA, ECC를 이론적으로 깨뜨릴 수 있음
  • Grover 알고리즘 — 해시 보안 강도를 절반으로 줄임. 하지만 SHA-256은 128비트 보안을 여전히 유지 → 해시 함수는 양자 안전

NIST PQC 표준

  • CRYSTALS-Kyber — 키 교환(암호화) 표준
  • CRYSTALS-Dilithium — 디지털 서명 표준

WIA-SEC-002 권고: 기존 암호 + PQC를 함께 사용하는 하이브리드 방식으로 점진적 전환.

합의 알고리즘

Consensus Algorithms

이중지불 문제 (Double Spending)

이중지불은 같은 디지털 화폐를 두 번 사용하는 것입니다. 디지털 데이터는 복사가 쉬우므로 이를 방지하는 것이 핵심 과제입니다.

  • 전통적 해결: 은행(중앙 기관)이 잔액을 관리
  • 블록체인 해결: 모든 참여자가 거래 내역을 공유하고 합의를 통해 검증

비잔틴 장군 문제

비잔틴 장군 문제 (Byzantine Generals Problem) — 1982년 Lamport 등이 제시. 일부 참여자가 배신자(악의적)일 때 어떻게 합의에 도달할 것인가?

여러 장군이 적의 성을 공격할지 결정해야 하는데, 배신자 장군이 서로 다른 메시지를 보낼 수 있습니다. 전체의 1/3 미만이 배신자라면 합의가 가능합니다.

BFT 조건: 배신자가 전체 참여자의 1/3 미만일 때 합의 달성 가능.

주요 합의 메커니즘 비교

PoW (작업 증명, Proof of Work)

  • 원리: 난이도 조건을 만족하는 해시를 가장 먼저 찾은 채굴자가 블록 생성 권한 획득
  • 보안: 공격 비용 = 전체 해시파워의 51% 이상 필요 (경제적으로 비합리적)
  • 단점: 엄청난 에너지 소비, 느린 처리 속도, 채굴풀 중앙화 경향
  • 대표 체인: 비트코인, 라이트코인

PoS (지분 증명, Proof of Stake)

  • 원리: 코인을 스테이킹(예치)한 검증자 중 블록 생성자 선출
  • 보안: 부정 행위 시 슬래싱(Slashing) — 스테이킹한 자산 몰수
  • 장점: 에너지 효율적, 참여 장벽 낮음
  • 단점: 부의 편중(부익부), Nothing at Stake 문제
  • 대표 체인: 이더리움 2.0, 카르다노, 솔라나
  • The Merge: 이더리움이 2022년 PoW → PoS로 전환한 역사적 사건

DPoS (위임 지분 증명)

  • 원리: 토큰 보유자가 투표로 대표(delegate/witness) 선출 → 대표가 블록 생성
  • 장점: 매우 빠른 속도
  • 단점: 중앙화 경향
  • 대표 체인: EOS, TRON

PBFT (실용적 비잔틴 장애 허용)

  • 원리: 3단계 투표 — Pre-prepare → Prepare → Commit
  • 적합 환경: 허가형(private) 블록체인
  • 대표 체인: Hyperledger Fabric
합의속도에너지탈중앙화보안환경
PoW느림매우 높음높음매우 높음퍼블릭
PoS빠름낮음중간높음퍼블릭
DPoS매우 빠름낮음낮음중간퍼블릭
PBFT빠름낮음낮음높음프라이빗

블록체인 트릴레마

블록체인 트릴레마 (Blockchain Trilemma) — 보안성, 탈중앙화, 확장성 세 가지를 동시에 완벽하게 만족시킬 수 없다는 것을 의미합니다.

  • PoW = 보안·탈중앙화 ↑ but 확장성 ↓
  • DPoS = 확장성 ↑ but 탈중앙화 ↓

해결 시도: 샤딩(Sharding) — 네트워크를 분할해 병렬 처리, 레이어 2(Layer 2) — 메인 체인 위에 별도 처리 계층

비트코인 UTXO 모델

UTXO (Unspent Transaction Output, 미사용 거래 출력) — 비트코인에는 "잔액"이라는 개념이 없습니다. 지갑의 잔액은 사용하지 않은 UTXO들의 합계입니다.

지갑 안에 든 지폐와 같습니다. 10,000원짜리 2장 + 5,000원짜리 1장 = 잔액 25,000원. 커피 3,000원을 사려면 5,000원짜리를 내고 2,000원을 거스름돈으로 받는 것처럼, UTXO를 입력으로 넣고 잔돈 UTXO를 돌려받습니다.

UTXO 거래 흐름

입력(Input): 기존 UTXO를 소비
    ↓
출력(Output): 새로운 UTXO 생성
    - 수신자에게 보내는 금액
    - 발신자에게 돌아오는 거스름돈

수수료 = 입력 합계 - 출력 합계 (채굴자에게)

사토시(Satoshi): 비트코인의 최소 단위. 1 BTC = 100,000,000 사토시

비트코인 총 발행량: 2,100만 BTC (시험 빈출!)

스마트 계약 기초

Smart Contract Basics · WIA-FIN-007

스마트 계약이란?

스마트 계약 (Smart Contract)은 1994년 Nick Szabo가 제안한 개념으로, 계약 조건을 코드로 작성하고 조건 충족 시 자동 실행되는 프로그램입니다.

자판기(Vending Machine)와 같습니다. 돈을 넣고(조건 충족) → 버튼을 누르면(함수 호출) → 음료가 나옵니다(자동 실행). 사람의 개입 없이 코드가 약속을 이행합니다.

스마트 계약의 4대 특성

  • 자동 실행 (Auto-execution) — 조건 충족 시 코드가 자동으로 동작
  • 불변성 (Immutability) — 배포 후 코드 변경 불가
  • 투명성 (Transparency) — 코드를 누구나 확인 가능
  • 분산 실행 (Distributed Execution) — 모든 노드가 동일한 코드를 실행

EVM (이더리움 가상 머신)

EVM (Ethereum Virtual Machine) — 스마트 계약을 실행하는 가상 머신. "세계 컴퓨터(World Computer)"라고 불립니다.

EVM의 핵심 특성

특성설명
튜링 완전이론상 모든 연산 수행 가능
결정적같은 입력 → 항상 같은 결과 (모든 노드 동일)
샌드박스외부 시스템과 직접 통신 불가
가스 메커니즘무한 루프 방지, 자원 사용에 비용 부과
EVM은 외부 데이터에 직접 접근할 수 없습니다. 날씨, 주가 등 외부 정보가 필요하면 오라클(Oracle)이라는 별도의 중개 서비스가 필요합니다.

Solidity 언어 핵심

Solidity는 이더리움 스마트 계약의 주력 프로그래밍 언어입니다. JavaScript와 C++의 영향을 받았으며, 파일 확장자는 .sol입니다.

기본 구조

// SPDX-License-Identifier: MIT       ← 라이선스 선언
pragma solidity ^0.8.19;               ← 컴파일러 버전 지정

contract MyContract {                  ← 계약 정의
    address public owner;              ← 상태 변수

    constructor() {                    ← 배포 시 1회 실행
        owner = msg.sender;            ← 배포자 = 소유자
    }

    function doSomething() public {    ← 함수 정의
        require(msg.sender == owner);  ← 조건 검사
    }
}

  • event / emit — 이벤트 정의 및 발생 (블록체인 로그에 기록)

스마트 계약 심화

Smart Contract Advanced · WIA-FIN-007

제어 흐름과 가스 주의사항

Solidity에서도 if-else 조건문과 for 반복문을 사용합니다.

for 루프에서 반복 횟수가 너무 많으면 가스 한도(블록 가스 리밋: ~3천만)를 초과하여 트랜잭션이 실패합니다. 무한 루프는 가스를 전부 소진합니다.

mapping과 struct

mapping (매핑)

해시 테이블 기반의 키-값 저장소입니다.

mapping(address => uint256) public balances;

// 특징:
// - 존재하지 않는 키 → 기본값 반환 (uint=0, bool=false, string="")
// - 반복(iterate)이 불가능 → 별도 배열로 키를 추적해야 함
// - 중첩 가능: mapping(address => mapping(uint256 => bool))

struct (구조체)

struct Candidate {
    string name;
    uint256 voteCount;
    bool isRegistered;
}
매핑은 모든 키에 대해 기본값을 반환하므로, 데이터 존재 여부를 확인하려면 isRegistered 같은 bool 플래그를 활용하세요.

modifier (수정자)

modifier는 함수 실행 전/후에 재사용 가능한 조건 검사를 삽입하는 패턴입니다.

modifier onlyOwner() {
    require(msg.sender == owner, "Not owner");
    _;    // ← 여기에 원래 함수 본문이 삽입됨
}

modifier whenNotPaused() {
    require(!paused, "Contract paused");
    _;
}

// 사용: 여러 modifier 조합 가능 (왼쪽 → 오른쪽 실행)
function withdraw() public onlyOwner whenNotPaused {
    // ...
}

WIA-FIN-007 권고: 역할 기반 접근 제어(RBAC)에 modifier 활용. OpenZeppelin의 AccessControl 라이브러리가 표준.

이벤트와 로깅

Event는 블록체인의 로그 시스템입니다. 상태 변경을 기록하고 프론트엔드에서 감지할 수 있습니다.

event VoteCast(address indexed voter, uint256 indexed candidateId, uint256 timestamp);

// indexed: 최대 3개, 필터링/검색에 사용
// emit으로 발생시킴
emit VoteCast(msg.sender, _candidateId, block.timestamp);

이벤트의 특징

  • 저장(SSTORE)보다 훨씬 저렴한 가스 비용
  • 스마트 계약 내부에서는 읽을 수 없음 — 외부(프론트엔드, 인덱서)에서만 조회
  • WIA-FIN-007은 모든 상태 변경에 이벤트 기록을 의무화 (감사 추적)

가스 최적화 기법

저장소 비용 계층

연산가스 비용
SSTORE (새 값 저장)~20,000
SSTORE (값 수정)~5,000
SLOAD (값 읽기)~2,100
memory 연산~3
stack 연산~3

최적화 기법 모음

  • 저장소 캐싱: 루프 전에 storage 변수를 memory 변수(로컬)에 복사해서 사용
  • 변수 패킹: struct 내에서 작은 타입을 인접 배치 (uint128 + uint128 = 1 슬롯)
  • constant / immutable: 변하지 않는 값에 사용 → 저장소를 쓰지 않음
  • bytes32 > string: 고정 길이 데이터는 bytes32가 효율적
  • 커스텀 에러: 문자열 에러 대신 error Unauthorized(); 사용 (Solidity 0.8.4+)
  • calldata 활용: external 함수 매개변수는 calldata로

투표 DApp (종합 실습)

6주차의 핵심 실습인 투표 DApp은 지금까지 배운 모든 요소를 결합합니다.

사용된 개념

  • struct — 후보자(Candidate) 정보
  • enum — 상태 머신 (Created → Voting → Ended)
  • mapping — 후보자, 등록 유권자, 투표 여부
  • modifier — onlyAdmin, inState(특정 상태일 때만 실행)
  • event — CandidateAdded, VoterRegistered, VoteCast, StateChanged

ABI (Application Binary Interface) — 스마트 계약과 프론트엔드를 연결하는 인터페이스 규격. Web3.js 또는 ethers.js를 통해 계약의 함수를 호출할 수 있습니다.

디지털 지갑

디지털 지갑의 본질

디지털 지갑은 암호화폐를 "저장"하는 것이 아닙니다. 지갑은 개인키(Private Key)를 관리하는 도구이고, 자산은 블록체인 위에 존재합니다.

"열쇠 보관함"이지 "금고"가 아닙니다. 열쇠를 잃어버리면 금고(블록체인 위 자산)는 영원히 열 수 없습니다.

지갑의 종류

분류유형편의성보안예시
연결 방식핫 월렛 (Hot Wallet)높음보통MetaMask, Trust Wallet
콜드 월렛 (Cold Wallet)낮음매우 높음Ledger, Trezor, 종이 지갑
관리 방식커스터디얼 (Custodial)매우 높음낮음Binance, Coinbase (거래소)
비커스터디얼 (Non-custodial)보통높음MetaMask (자기 관리)
"Not your keys, not your coins" — 커스터디얼 지갑(거래소)에 자산을 맡기면 거래소 파산 시 자산을 잃을 수 있습니다. FTX 파산(2022)이 대표적 사례입니다.
시험 빈출: 커스터디얼 지갑 = 제3자(거래소)가 개인키를 관리하는 지갑

HD 지갑과 BIP-39 니모닉

HD 지갑 (Hierarchical Deterministic) — BIP-32

하나의 마스터 시드에서 무한한 키 쌍을 트리 구조로 생성합니다. 마스터 시드만 백업하면 모든 키를 복원할 수 있습니다.

BIP-39 니모닉 (Mnemonic)

256비트 엔트로피(난수)를 12~24개의 영어 단어로 변환한 것입니다.

  • 단어 목록: 2,048개 단어 (211 = 2048, 각 단어 = 11비트)
  • 체크섬 포함 → 오타 감지 가능
[생성 과정]
엔트로피 (256비트) → SHA-256 체크섬 → 11비트씩 분할 → 단어 매핑

[시드 도출]
니모닉 → PBKDF2 (2048 라운드) → 512비트 시드 → HMAC-SHA512 → 마스터 키
니모닉은 절대로 디지털 기기에 저장하지 마세요. 종이 또는 금속판에 2부 이상 작성하여 다른 물리적 장소에 보관하세요.

BIP-44 파생 경로 (Derivation Path)

m / purpose' / coin_type' / account' / change / address_index

이더리움 첫 번째 주소: m/44'/60'/0'/0/0
비트코인 첫 번째 주소: m/44'/0'/0'/0/0
두 번째 주소:           m/44'/60'/0'/0/1  (index 증가)

키 분실 시나리오

상황결과대응
니모닉 있음✅ 복원 가능아무 지갑 앱에서 복원
니모닉 분실, 지갑 접근 가능⚠️ 긴급즉시 새 지갑 만들고 자산 이전
니모닉 분실 + 지갑 접근 불가❌ 영구 손실복구 방법 없음

소셜 리커버리 (Social Recovery)

Vitalik Buterin이 제안한 방식으로, 신뢰할 수 있는 가디언(Guardian) 그룹을 지정하여 과반수 승인으로 키를 교체할 수 있습니다. ERC-4337 (Account Abstraction)과 결합하여 연구 중입니다.

  • 채널 내에서 오프체인 거래 (빠르고 저렴)
  • 최종 상태만 메인 체인에 기록

DeFi란?

DeFi (Decentralized Finance, 탈중앙 금융) — 은행·증권사 같은 중개자 없이 스마트 계약만으로 금융 서비스를 제공하는 것입니다. 2020년 "DeFi Summer"에 폭발적으로 성장했습니다.

TVL (Total Value Locked) — DeFi 프로토콜에 예치된 총 자산 규모. 피크 시 $2,000억 돌파.

AMM (자동화 시장 조성자)

AMM (Automated Market Maker) — 사람이 아닌 수학 공식이 가격을 결정하고 거래를 처리합니다.

상수곱 공식: x × y = k

x = 토큰 A 수량, y = 토큰 B 수량, k = 상수(변하지 않음)

실제 계산 예시

풀: 100 ETH + 200,000 USDC → k = 20,000,000

1 ETH를 사려면?
  새 ETH 수량 = 99
  새 USDC 수량 = 20,000,000 ÷ 99 = 202,020 USDC
  비용 = 202,020 - 200,000 = 2,020 USDC

10 ETH를 사려면?
  새 ETH = 90
  새 USDC = 20,000,000 ÷ 90 = 222,222 USDC
  비용 = 22,222 USDC → 개당 2,222 USDC (더 비쌈!)

핵심 개념

  • 슬리피지 (Slippage) — 거래 규모가 클수록 기대 가격과 실제 가격의 차이가 커짐. 풀이 클수록 슬리피지 감소
  • 가격 영향 (Price Impact) — 내 거래가 풀 가격에 미치는 영향
수영장에서 물(토큰A)을 퍼내면 수위가 내려가고, 반대쪽(토큰B)이 더 귀해집니다. 수영장이 클수록 한 바가지의 영향이 작습니다.

유동성 풀 (Liquidity Pool)

유동성 공급자 (LP) (Liquidity Provider) — 토큰 쌍을 풀에 예치하여 거래가 가능하게 하는 참여자입니다.

  • LP 토큰: 유동성 제공의 증거로 받는 토큰
  • 수수료 수익: 거래마다 발생하는 수수료(보통 0.3%)를 LP가 분배받음

비영구적 손실 (Impermanent Loss)

비영구적 손실 — 유동성 제공 시, 단순 보유(hodl)보다 손해를 볼 수 있는 현상입니다.

가격 변동비영구적 손실
±25%~0.6%
±50%~2.0%
±100%~5.7%
±200%~13.4%

수수료 수익이 비영구적 손실보다 크면 이익, 작으면 손해입니다.

집중 유동성 (Concentrated Liquidity) — Uniswap V3

LP가 특정 가격 범위에 집중 배치 → 자본 효율성 최대 4,000배 향상.

이자농사 (Yield Farming)

유동성 제공 → LP 토큰 → 스테이킹 → 거버넌스 토큰 보상의 구조입니다.

APY (Annual Percentage Yield) — 복리 효과를 포함한 연간 수익률

유동성 마이닝 (Liquidity Mining) — 보상 토큰을 통한 유동성 유치. 지속 가능성에 대한 우려 존재.

플래시론 (Flash Loan)

플래시론 — 무담보로 거액을 빌려 한 트랜잭션 안에서 빌리고 갚는 대출입니다.

  • 상환 실패 시 → 전체 트랜잭션이 되돌려짐(revert) → 프로토콜 손실 없음
  • 용도: 차익거래(Arbitrage), 담보 교체, 셀프 청산
  • 위험: 플래시론을 악용한 공격 (오라클 조작, 가격 왜곡)
시험 빈출: 플래시론 = 무담보, 같은 트랜잭션 내에서 대출+상환이 이뤄지는 대출

MEV와 프론트러닝

MEV (Maximal Extractable Value) — 트랜잭션 순서 조작으로 추출 가능한 이익.

멤풀 (Mempool) — 확인되지 않은 거래들이 대기하는 풀. 모든 참여자에게 공개.

샌드위치 공격 (Sandwich Attack)

1. 공격자가 멤풀에서 대량 매수 주문 감지
2. 그 주문 앞에 자기 매수를 넣음 (Front-run)
3. 원래 주문이 실행되어 가격 상승
4. 공격자가 비싸진 가격에 매도 (Back-run) → 차익 획득

방어: Commit-Reveal 방식, Flashbots(프라이빗 트랜잭션 제출)

재진입 공격 (Reentrancy Attack)

스마트 계약 보안의 가장 유명한 취약점입니다.

원리

외부 호출(ETH 전송)이 상태 업데이트보다 먼저 실행될 때, 호출받은 계약이 다시 원래 함수로 진입하여 반복적으로 자금을 인출합니다.

The DAO 해킹 (2016)

  • 약 360만 ETH (~$7,000만) 탈취
  • 이더리움 커뮤니티가 투표로 해킹을 되돌리는 하드포크 결정
  • ETH / ETC 분리: 포크에 동의한 체인 = ETH, 반대한 체인 = ETC (Ethereum Classic)
  • 철학적 질문: "코드가 곧 법인가?" (Is Code Law?)

취약한 코드 vs 안전한 코드

// ❌ 취약한 코드 (외부 호출 → 상태 업데이트)
function withdraw() public {
    uint amount = balances[msg.sender];
    (bool success,) = msg.sender.call{value: amount}(""); // 외부 호출 먼저!
    balances[msg.sender] = 0; // 상태 업데이트 나중 → 재진입 가능
}

// ✅ 안전한 코드 (Checks-Effects-Interactions 패턴)
function withdraw() public {
    uint amount = balances[msg.sender];
    balances[msg.sender] = 0;   // Effects: 상태 먼저 업데이트
    (bool success,) = msg.sender.call{value: amount}(""); // Interactions: 그 다음 외부 호출
    require(success);
}
CEI 패턴 (Checks-Effects-Interactions): ①조건 검사(Checks) → ②상태 변경(Effects) → ③외부 호출(Interactions). 재진입 방지의 핵심!

ReentrancyGuard: 뮤텍스(잠금장치) 방식으로 함수 재진입을 원천 차단하는 보호 장치.

정수 오버플로우/언더플로우

오버플로우: uint8 최대값(255)에서 +1 → 0으로 돌아감
언더플로우: uint8 값 0에서 -1 → 255가 됨

  • Solidity < 0.8.0: 기본적으로 취약 → SafeMath 라이브러리 필요
  • Solidity ≥ 0.8.0: 자동 보호 (오버플로우 시 revert)
  • unchecked 블록: 명시적으로 보호 해제 (주의 필요)

BEC Token 사건(2018): batchTransfer에서 _receivers.length * _value가 오버플로우하여 대량 토큰이 무(無)에서 생성됨

51% 공격 — 전체 해시파워의 과반을 장악하여 이중지불을 실행하는 공격입니다.

의료 블록체인

현재 의료 데이터는 병원마다 분산되어 있고, 호환성이 없으며, 환자가 자기 데이터를 통제하지 못합니다.

환자 중심 데이터 관리 모델

  • 환자가 키를 보유하고 접근을 제어
  • 온체인: 접근 권한, 데이터 해시 (가볍게)
  • 오프체인(IPFS): 암호화된 의료 데이터 (무겁게)

환자 동의 스마트 계약

grantAccess(provider, dataType, duration)  // 시간 제한 접근 허용
revokeAccess(provider, dataType)           // 접근 취소

HL7 FHIR: 의료 데이터 형식 표준. 블록체인과 통합하여 병원 간 데이터 호환성 확보.

프라이버시 규제: HIPAA (미국), GDPR (EU) 준수 필수.

시험 에세이 빈출: 의료 블록체인의 장점(데이터 무결성, 안전한 공유, 환자 접근 제어)과 한계(대용량 의료 이미지의 온체인 저장 비용/속도 → 오프체인+해시 접근법 필요)

공급망 관리 (Supply Chain)

위조 상품과 불투명한 출처 문제를 블록체인으로 해결합니다.

  • IoT + 블록체인: 센서 데이터를 변조 불가능하게 기록
  • 사례: IBM Food Trust / Walmart — 망고 원산지 추적: 7일 → 2.2초
  • 사례: De Beers Tracr — 다이아몬드 출처 추적으로 분쟁 다이아몬드(Blood Diamond) 방지

ESG와 탄소 크레딧

탄소 크레딧 토큰화 — 탄소 배출권을 블록체인 토큰으로 발행하여 투명하게 거래합니다.

  • 그린워싱(Greenwashing) 방지 — 실제 감축량을 온체인에서 검증
  • 이중 계상(Double Counting) 방지
  • 프로젝트: Toucan Protocol, KlimaDAO

전자투표 (e-Voting)

전자투표의 요건

  • 비밀성 — 투표 내용 비공개
  • 검증 가능성 — 결과를 독립적으로 확인 가능
  • 강제 투표 방지 — 투표소 밖에서의 강압 차단

영지식 증명 (ZKP, Zero-Knowledge Proof) — 비밀 투표 구현을 위한 핵심 기술. "이 사실이 참이다"를 증명하되, 내용 자체는 공개하지 않는 암호 기법.

현재 활용: DAO 투표 (Snapshot.org), 기업 의결권 행사

탈중앙 신원 (DID / SSI)

SSI (Self-Sovereign Identity, 자기 주권 신원) — 중앙 기관 없이 스스로 신원을 통제합니다.

DID (Decentralized Identifier) — W3C 표준. 블록체인에 등록된 분산 식별자.

VC (Verifiable Credential, 검증 가능한 자격증명)

3자 모델

발행자(Issuer) ──── 자격증명 ────→ 보유자(Holder) ──── 제시 ────→ 검증자(Verifier)
   (예: 대학교)        (졸업장)       (예: 졸업생)       (증명)       (예: 기업)
                                          ↕
                               블록체인 (DID Registry)

응용: 학위 검증, 신원 확인, 난민 신원 제공

Essential Glossary — 가나다순
ABI Application Binary Interface
스마트 계약과 외부 프로그램(프론트엔드)을 연결하는 인터페이스 규격
AMM Automated Market Maker
수학 공식(x×y=k)으로 자동 가격 결정 및 거래를 처리하는 DEX 메커니즘
APY Annual Percentage Yield
복리 효과 포함 연간 수익률
BIP-39 Bitcoin Improvement Proposal 39
개인키를 12~24개 영단어(니모닉)로 변환하는 표준
BIP-44 Bitcoin Improvement Proposal 44
HD 지갑의 키 파생 경로 표준 (m/44'/coin'/account'/change/index)
CBDC Central Bank Digital Currency
중앙은행이 발행하는 디지털 화폐. 도매용(은행간)과 소매용(일반국민)으로 구분
CEI 패턴 Checks-Effects-Interactions
조건검사→상태변경→외부호출 순서로 재진입 공격을 방지하는 코딩 패턴
DeFi Decentralized Finance
중개자 없이 스마트 계약으로 제공되는 탈중앙 금융 서비스
DID Decentralized Identifier
블록체인 기반 분산 식별자. W3C 표준. 자기 주권 신원(SSI)의 핵심
DLT Distributed Ledger Technology
분산원장 기술. 블록체인은 DLT의 한 종류
DPoS Delegated Proof of Stake
투표로 대표를 선출하여 블록 생성을 위임하는 합의 알고리즘
ECC Elliptic Curve Cryptography
타원곡선 암호. 비트코인/이더리움이 사용하는 공개키 암호화 방식 (secp256k1)
EIP-1559 Ethereum Improvement Proposal 1559
가스비 개선. Base Fee(소각) + Priority Fee(검증자) 구조
EIP-2981 NFT Royalty Standard
NFT 로열티 정보를 온체인에 기록하는 표준
EOA Externally Owned Account
개인키로 제어되는 일반 이더리움 사용자 계정
ERC-20 Fungible Token Standard
대체가능 토큰(FT) 표준. 모든 토큰이 동일 가치
ERC-721 Non-Fungible Token Standard
대체불가능 토큰(NFT) 표준. 각 토큰이 고유
ERC-1155 Multi Token Standard
FT와 NFT를 동시에 지원하는 멀티 토큰 표준
ERC-1400 Security Token Standard
증권형 토큰 표준. 양도 제한, 투자자 자격 검증 기능
ERC-4337 Account Abstraction
계정 추상화. 소셜 리커버리 등 사용성 개선을 위한 표준
EVM Ethereum Virtual Machine
스마트 계약 실행 환경. 튜링 완전, 결정적, 샌드박스
Gas 가스
이더리움 연산 수수료 단위. Fee = Gas Used × Gas Price
HD 지갑 Hierarchical Deterministic Wallet
하나의 시드에서 무한한 키를 트리 구조로 생성하는 지갑 (BIP-32)
IPFS InterPlanetary File System
콘텐츠 주소 기반 분산 파일 시스템. CID로 데이터 식별
LTV Loan-to-Value
담보 대비 대출 비율. 예: LTV 66% = 100만원 담보로 66만원 대출
MEV Maximal Extractable Value
트랜잭션 순서 조작으로 추출 가능한 이익. 프론트러닝/샌드위치 공격
Modifier 수정자
Solidity에서 함수 실행 전/후 조건을 재사용 가능하게 삽입하는 패턴
Nonce 논스
채굴 시 해시 조건을 맞추기 위해 조정하는 임의의 값
Oracle 오라클
블록체인 외부 데이터를 스마트 계약에 전달하는 중개 서비스
PBFT Practical Byzantine Fault Tolerance
3단계 투표 합의. 허가형 블록체인용. 전체의 1/3 미만이 악의적일 때 작동
PoS Proof of Stake
코인 스테이킹(예치) 기반 합의. 부정행위 시 슬래싱(자산 몰수)
PoW Proof of Work
연산 경쟁 기반 합의. 에너지 소비 높지만 보안성 최고. 비트코인 사용
PQC Post-Quantum Cryptography
양자내성 암호. CRYSTALS-Kyber(키교환), Dilithium(서명)
RWA Real World Assets
실물 자산을 블록체인 토큰으로 표현하는 자산 토큰화
SBT Soulbound Token
양도 불가능한 토큰. 신원증명, 학위, 자격증 등에 사용
SHA-256 Secure Hash Algorithm 256
256비트 해시 함수. 비트코인의 핵심 해시 알고리즘
Slashing 슬래싱
PoS에서 검증자의 부정행위 시 스테이킹한 자산을 몰수하는 처벌
Slippage 슬리피지
AMM에서 거래 규모로 인한 기대 가격과 실제 가격의 차이
SPOF Single Point of Failure
단일 장애점. 중앙 서버 하나가 전체 시스템을 멈추는 약점
SPV Special Purpose Vehicle
특수목적법인. 자산 토큰화 시 법적 소유권을 연결하는 구조
SSI Self-Sovereign Identity
자기 주권 신원. 중앙기관 없이 스스로 신원을 통제하는 개념
STO Security Token Offering
증권법 적용을 받는 토큰 공개. ICO와 달리 투자자 보호 의무
The Merge 더 머지
2022년 이더리움의 PoW → PoS 전환 이벤트
TVL Total Value Locked
DeFi 프로토콜에 예치된 총 자산 규모
UTXO Unspent Transaction Output
미사용 거래 출력. 비트코인의 잔액 관리 방식
VC Verifiable Credential
검증 가능한 자격증명. DID 생태계에서 발행자-보유자-검증자 모델
ZKP Zero-Knowledge Proof
영지식 증명. 내용을 공개하지 않고 사실의 참을 증명하는 기법
눈사태 효과 Avalanche Effect
해시 함수에서 입력의 미세한 변화가 출력의 ~50%를 바꾸는 성질
머클 트리 Merkle Tree
거래를 2개씩 해시하며 올라가 하나의 루트를 만드는 이진 트리 구조
비영구적 손실 Impermanent Loss
LP 제공 시 단순 보유 대비 손실. 가격 변동이 클수록 커짐
비잔틴 장군 Byzantine Generals Problem
악의적 참여자 존재 시 합의 문제. 배신자 < 1/3이면 합의 가능
샌드위치 공격 Sandwich Attack
멤풀에서 대량 거래를 감지하여 앞뒤로 거래를 끼워 넣는 MEV 공격
소셜 리커버리 Social Recovery
가디언 그룹의 과반 승인으로 지갑 키를 복원하는 방식
스마트 계약 Smart Contract
조건 충족 시 자동 실행되는 블록체인 프로그램. 1994년 Nick Szabo 제안
재진입 공격 Reentrancy Attack
외부 호출로 함수에 재진입하여 자금을 반복 인출하는 공격. The DAO 해킹의 원인
청산 Liquidation
담보 가치 하락 시 자동 매각하여 대출금을 회수하는 메커니즘
트릴레마 Blockchain Trilemma
보안·탈중앙화·확장성을 동시에 만족시킬 수 없는 블록체인의 근본 한계
플래시론 Flash Loan
무담보 대출. 한 트랜잭션 내에서 대출+상환 필수. 실패 시 전체 revert
