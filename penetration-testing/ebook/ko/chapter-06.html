웹 애플리케이션 테스트

웹 애플리케이션의 보안 취약점을 체계적으로 테스트하는 방법을 다룹니다. OWASP 방법론을 기반으로 인증, 인가, 입력 유효성 검증 등 핵심 영역을 테스트합니다.

6.1 웹 애플리케이션 테스트 개요

웹 애플리케이션은 현대 비즈니스의 핵심 구성 요소로, 고객 데이터, 금융 정보, 비즈니스 로직을 처리합니다. 따라서 웹 애플리케이션 보안 테스트는 조직의 전체 보안 전략에서 중요한 부분을 차지합니다.

6.1.1 테스트 범위

  • 프런트엔드: HTML, JavaScript, CSS, 클라이언트 측 로직
  • 백엔드: 서버 측 코드, 비즈니스 로직
  • 데이터베이스: 쿼리, 저장 프로시저, 데이터 보호
  • API: REST, GraphQL, SOAP 엔드포인트
  • 인프라: 웹 서버, 애플리케이션 서버 구성

6.2 Burp Suite 활용

6.2.1 프록시 설정 및 기본 사용

# Burp Suite 프록시 설정
# 1. 브라우저 프록시: 127.0.0.1:8080
# 2. Burp CA 인증서 설치 (HTTPS 트래픽 검사)

Burp Suite 주요 기능:
├── Proxy        - 트래픽 가로채기 및 수정
├── Scanner      - 자동화된 취약점 스캐닝 (Pro)
├── Repeater     - 요청 수동 수정 및 재전송
├── Intruder     - 페이로드 기반 공격
├── Sequencer    - 세션 토큰 분석
├── Decoder      - 인코딩/디코딩
├── Comparer     - 응답 비교
└── Extender     - 확장 기능

6.2.2 수동 테스트 워크플로우

  1. 프록시 활성화 후 애플리케이션 탐색
  2. Site Map에서 엔드포인트 확인
  3. 관심 요청을 Repeater로 전송
  4. 매개변수 수정 및 응답 분석
  5. 취약점 발견 시 Intruder로 자동화

6.3 인증 테스트

6.3.1 인증 취약점 유형

취약점 설명 테스트 방법
브루트포스 반복 로그인 시도 허용 Burp Intruder로 다수 시도
기본 자격 증명 admin/admin 등 기본 계정 일반적인 자격 증명 목록 테스트
사용자 열거 유효 사용자 식별 가능 로그인 오류 메시지 분석
약한 비밀번호 복잡도 요구사항 미비 간단한 비밀번호로 가입 시도
비밀번호 재설정 결함 토큰 예측 가능, 만료 없음 재설정 프로세스 분석

6.3.2 사용자 열거 테스트

# 로그인 응답 비교
# 존재하는 사용자:
POST /login
{"username": "admin", "password": "wrong"}
Response: "Invalid password"

# 존재하지 않는 사용자:
POST /login
{"username": "nonexistent", "password": "wrong"}
Response: "User not found"

# 응답 시간 차이 측정
# 존재하는 계정은 비밀번호 해시 비교로 시간 소요

# Burp Intruder 설정
Attack type: Sniper
Payload: 사용자명 목록
Grep - Match: "Invalid password"

6.4 인가 테스트

6.4.1 접근 제어 취약점

취약점 설명 예시
IDOR 직접 객체 참조 /api/user/123 → /api/user/124
수평적 권한 상승 동등 권한 사용자 접근 다른 사용자의 주문 조회
수직적 권한 상승 높은 권한 기능 접근 일반 사용자가 관리 기능 접근
기능 수준 접근 제어 숨겨진 기능 노출 /admin 페이지 직접 접근

6.4.2 IDOR 테스트

# 원본 요청 (인증된 사용자)
GET /api/orders/1001
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...

Response: {
  "id": 1001,
  "user_id": 100,
  "items": [...],
  "total": 50000
}

# IDOR 테스트 - ID 변경
GET /api/orders/1002
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...  # 동일 토큰

# 취약한 경우: 다른 사용자의 주문 반환
Response: {
  "id": 1002,
  "user_id": 101,  # 다른 사용자
  "items": [...],
  "total": 75000
}

# Burp Intruder로 ID 범위 테스트
Payload: Numbers 1000-2000

6.6 세션 관리 테스트

6.6.1 세션 취약점

취약점 테스트 방법
세션 고정 로그인 전후 세션 ID 변경 여부
세션 하이재킹 HTTPS, Secure/HttpOnly 플래그
세션 만료 비활성 후 세션 유효성
동시 세션 다중 로그인 허용 여부
예측 가능한 토큰 Burp Sequencer 분석

6.7 API 보안 테스트

🔌 REST API 테스트 체크리스트

  • 인증 우회 (토큰 없이 접근)
  • IDOR (객체 ID 변경)
  • 메서드 변경 (GET → POST/PUT/DELETE)
  • 과도한 데이터 노출 (응답 필드 분석)
  • 속도 제한 (Rate Limiting)
  • 입력 유효성 검증 (특수문자, 크기)
  • CORS 설정 (허용 출처)
  • API 버전 관리 (구 버전 접근)
# API 테스트 예시
# 인증 우회
GET /api/v1/users
# Authorization 헤더 없이 요청

# 메서드 변경
DELETE /api/v1/users/1  # 원래 GET만 허용된 경우

# 과도한 데이터 노출
GET /api/v1/users/me
Response: {
  "id": 1,
  "username": "user",
  "password_hash": "...",  # 민감 정보 노출
  "internal_notes": "..."
}

# GraphQL 인트로스펙션
{"query": "{__schema{types{name,fields{name}}}}"}
