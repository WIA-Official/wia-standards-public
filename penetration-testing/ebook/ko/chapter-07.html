제7장

보고서 작성

침투 테스트의 최종 산출물인 보고서 작성 방법을 다룹니다. 효과적인 문서화, 취약점 기술, 권고사항 제시를 통해 고객에게 가치를 전달합니다.

7.1 보고서의 중요성

침투 테스트 보고서는 수행된 모든 작업의 결과물이자, 고객이 받는 주요 산출물입니다. 기술적 발견 사항을 명확하게 전달하고, 경영진과 기술팀 모두가 이해할 수 있어야 합니다. 잘 작성된 보고서는 조직의 보안 개선에 직접적인 영향을 미칩니다.

7.1.1 보고서 목적

  • 발견 사항 문서화: 식별된 모든 취약점과 위험 기록
  • 영향 전달: 취약점이 비즈니스에 미치는 영향 설명
  • 개선 방향 제시: 구체적이고 실행 가능한 권고사항
  • 증거 제공: 규정 준수 및 감사 목적의 문서
  • 우선순위 결정: 리소스 할당을 위한 가이드

7.2 보고서 구조

7.2.1 표준 보고서 템플릿

섹션 대상 독자 내용
표지 전체 프로젝트명, 고객사, 날짜, 분류
요약 경영진 핵심 발견 사항, 위험 수준, 권고사항
범위 및 방법론 전체 테스트 범위, 제약사항, 사용 방법론
발견 사항 요약 전체 취약점 통계, 심각도별 분류
상세 발견 사항 기술팀 각 취약점의 상세 정보
권고사항 전체 개선 조치, 우선순위
부록 기술팀 도구 목록, 상세 로그, 추가 증거

7.3 요약 (Executive Summary)

요약은 경영진을 위한 섹션으로, 기술적 세부 사항 없이 핵심 내용을 전달합니다. 1-2페이지 내에서 다음 내용을 포함해야 합니다:

📋 요약 구성 요소

  • 테스트 개요: 수행 목적, 기간, 범위
  • 전반적 위험 등급: 조직의 보안 상태 요약
  • 핵심 발견 사항: 가장 중요한 3-5개 취약점
  • 긍정적 관찰: 잘 구현된 보안 통제
  • 최우선 권고사항: 즉시 조치가 필요한 항목
  • 결론: 전반적 평가와 다음 단계
예시: 요약 작성

전반적 위험 등급: 높음

테스트 결과, 총 15개의 취약점이 발견되었으며,
이 중 2개는 심각(Critical), 5개는 높음(High) 등급입니다.

주요 발견 사항:
1. SQL 인젝션 취약점으로 전체 데이터베이스 접근 가능
2. 관리자 패널에 기본 자격 증명 사용
3. 민감 데이터 평문 전송

즉각적인 조치가 필요합니다. 특히 SQL 인젝션과
기본 자격 증명 문제는 24시간 내 해결을 권장합니다.

7.4 취약점 기술 방법

7.4.1 취약점 보고 템플릿

항목 설명
제목 간결하고 명확한 취약점 이름
심각도 Critical / High / Medium / Low / Informational
CVSS 점수 표준화된 점수 (0.0-10.0)
영향받는 자산 URL, IP, 구성 요소
설명 취약점의 기술적 설명
영향 악용 시 발생할 수 있는 피해
재현 단계 취약점을 재현하는 상세 단계
증거 스크린샷, 요청/응답, PoC
권고사항 구체적인 개선 방안
참조 CWE, OWASP, 관련 문서

7.4.2 취약점 보고 예시

취약점 ID: VULN-001
제목: 로그인 페이지 SQL 인젝션
심각도: Critical  CVSS: 9.8

영향받는 자산:
- https://example.com/login.php
- 매개변수: username

설명:
로그인 페이지의 username 매개변수가 SQL 인젝션에
취약합니다. 사용자 입력이 적절히 검증되거나 이스케이프
되지 않고 SQL 쿼리에 직접 삽입됩니다.

영향:
- 전체 데이터베이스 접근 및 덤프
- 인증 우회
- 데이터 수정 또는 삭제
- 잠재적 원격 코드 실행

재현 단계:
1. https://example.com/login.php 접속
2. username 필드에 다음 입력: ' OR '1'='1' --
3. password 필드에 아무 값 입력
4. 로그인 버튼 클릭
5. 인증 없이 관리자로 로그인됨

증거:
[스크린샷: 로그인 성공 화면]
[요청/응답 데이터]

권고사항:
1. 준비된 문장(Prepared Statements) 사용
2. 입력 유효성 검증 구현
3. 최소 권한 데이터베이스 계정 사용
4. WAF 규칙 추가

참조:
- CWE-89: SQL Injection
- OWASP: SQL Injection Prevention Cheat Sheet

7.5 권고사항 작성

7.5.1 효과적인 권고사항 원칙

  • 구체적: "보안 강화 필요" 대신 정확한 조치 명시
  • 실행 가능: 기술팀이 바로 적용할 수 있는 수준
  • 우선순위: 위험도와 영향도에 따른 순위
  • 참조 포함: 추가 정보를 위한 링크 제공
  • 현실적: 조직의 환경과 제약 고려

7.5.2 권고사항 예시

취약점 권고사항 우선순위
SQL 인젝션 모든 쿼리에 PDO 또는 준비된 문장 적용 즉시
기본 자격 증명 관리자 계정 비밀번호 변경, 강력한 정책 적용 즉시
XSS 출력 인코딩 적용, CSP 헤더 구현 1주
HTTPS 미사용 전체 사이트 HTTPS 전환, HSTS 적용 1주
정보 노출 서버 배너 제거, 상세 오류 메시지 비활성화 1개월

7.6 보고서 품질 관리

✅ 보고서 검토 체크리스트

  • 모든 취약점에 증거가 첨부되었는가?
  • 재현 단계가 명확하고 따라 하기 쉬운가?
  • 심각도 등급이 일관되게 적용되었는가?
  • 권고사항이 구체적이고 실행 가능한가?
  • 민감 정보가 적절히 마스킹되었는가?
  • 오탈자와 문법 오류가 없는가?
  • 기밀 정보 취급 표시가 되어 있는가?

7.7 보고서 전달

7.7.1 전달 시 고려사항

  • 보안 전송: 암호화된 채널로 전달 (암호화 압축, 보안 파일 공유)
  • 접근 제한: 필요한 인원에게만 공유
  • 브리핑: 주요 이해관계자에게 결과 발표
  • Q&A 세션: 기술팀 질의응답 시간
  • 후속 지원: 개선 조치 지원 가능성
