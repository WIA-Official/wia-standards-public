CHAPTER 07

Reporting and Remediation

Effective documentation, risk communication, and actionable remediation guidance

📝 The Importance of Effective Reporting

The penetration test report is often the only deliverable the client receives. It must effectively communicate findings to both technical and non-technical stakeholders, justify the engagement cost, and provide clear guidance for remediation.

Report Goals

  • Clear Communication: Present findings in understandable language
  • Risk Context: Help stakeholders understand business impact
  • Actionable Guidance: Provide specific remediation steps
  • Evidence-Based: Support claims with proof and screenshots
  • Compliance Support: Address regulatory requirements
  • Knowledge Transfer: Educate the client's security team

Report Audience

Audience Primary Interests Report Section
Executive Leadership (C-Suite) Business risk, compliance, budget Executive Summary
IT Management Resource allocation, priorities, timelines Summary + Recommendations
Security Team Technical details, exploitation steps Detailed Findings
Development Team Code fixes, secure coding practices Remediation Guidance
Compliance/Audit Regulatory requirements, controls Compliance Section

📄 Report Structure

A comprehensive penetration test report typically includes the following sections:

1. Executive Summary

Purpose

High-level overview for non-technical stakeholders, typically 1-2 pages.

Key Components

  • Engagement Overview: Brief description of what was tested
  • Overall Risk Assessment: High-level security posture evaluation
  • Key Findings Summary: Most critical vulnerabilities discovered
  • Business Impact: Potential consequences of exploitation
  • High-Level Recommendations: Strategic security improvements
  • Visual Metrics: Charts showing vulnerability distribution
EXECUTIVE SUMMARY EXAMPLE:

Overview:
SmileStory Inc. engaged WIA Security to conduct a comprehensive
penetration test of their customer portal web application and
associated API infrastructure from October 1-15, 2025.

Risk Assessment:
The overall security posture is MODERATE. While core authentication
mechanisms are properly implemented, several critical vulnerabilities
were identified that could allow unauthorized access to customer data.

Critical Findings:
• SQL Injection in search functionality (CRITICAL)
• Broken Access Control on admin endpoints (HIGH)
• Missing rate limiting on password reset (MEDIUM)

Business Impact:
Successful exploitation could result in:
• Unauthorized access to 50,000+ customer records
• Potential GDPR violations and regulatory fines
• Reputational damage and loss of customer trust

Immediate Actions Required:
1. Patch SQL injection vulnerability within 48 hours
2. Implement proper authorization checks on admin functions
3. Add rate limiting to prevent brute force attacks

2. Scope and Methodology

SCOPE SECTION EXAMPLE:

Testing Period: October 1-15, 2025
Testing Type: Gray-box web application penetration test
Tester: WIA Certified Penetration Tester

In-Scope:
✓ Customer Portal (https://portal.smilestory.com)
✓ Public API (https://api.smilestory.com/v2)
✓ Admin Panel (https://admin.smilestory.com)

Out-of-Scope:
✗ Production database servers
✗ Internal corporate network
✗ Third-party payment processor
✗ Social engineering attacks

Methodology:
Testing followed OWASP Testing Guide v4.2 and PTES standards:
1. Information Gathering
2. Vulnerability Analysis
3. Exploitation
4. Post-Exploitation (limited scope)
5. Reporting

Tools Used:
• Burp Suite Professional
• OWASP ZAP
• SQLmap
• Nmap
• Custom scripts

3. Findings Summary

Overview of all vulnerabilities discovered, categorized by severity:

Severity Count Remediation Priority
CRITICAL 2 Immediate (24-48 hours)
HIGH 5 Urgent (7 days)
MEDIUM 8 Important (30 days)
LOW 12 Standard (90 days)
INFO 6 Best practice improvements

🔍 Detailed Findings

Each vulnerability should be documented with comprehensive details:

Finding Template

Required Elements for Each Finding

  1. Title: Clear, descriptive name
  2. Severity Rating: Critical/High/Medium/Low/Info
  3. CVSS Score: Industry-standard scoring (if applicable)
  4. CWE Reference: Common Weakness Enumeration ID
  5. Description: What the vulnerability is
  6. Impact: Potential consequences of exploitation
  7. Affected Components: Specific systems/endpoints
  8. Steps to Reproduce: Detailed exploitation steps
  9. Proof of Concept: Evidence (screenshots, logs)
  10. Remediation: Specific fix recommendations
  11. References: Additional resources

Sample Detailed Finding

═══════════════════════════════════════════════════════════════
FINDING #1: SQL Injection in Search Functionality
═══════════════════════════════════════════════════════════════

Severity: CRITICAL
CVSS v3.1 Score: 9.8 (Critical)
CWE: CWE-89 (SQL Injection)
OWASP: A03:2021 - Injection

═══════════════════════════════════════════════════════════════
DESCRIPTION
═══════════════════════════════════════════════════════════════

A SQL injection vulnerability exists in the customer search
functionality, allowing unauthenticated attackers to execute
arbitrary SQL queries against the backend database.

═══════════════════════════════════════════════════════════════
IMPACT
═══════════════════════════════════════════════════════════════

Successful exploitation allows an attacker to:
• Extract entire customer database (50,000+ records)
• Modify or delete customer data
• Bypass authentication mechanisms
• Potentially execute operating system commands
• Violate GDPR, PCI-DSS, and other compliance requirements

Estimated business impact:
• Data breach affecting all customers
• Regulatory fines: $2M - $20M (GDPR Article 83)
• Reputational damage
• Legal liability

═══════════════════════════════════════════════════════════════
AFFECTED COMPONENTS
═══════════════════════════════════════════════════════════════

URL: https://portal.smilestory.com/search
Parameter: q (query parameter)
Method: GET
Authentication Required: No

═══════════════════════════════════════════════════════════════
STEPS TO REPRODUCE
═══════════════════════════════════════════════════════════════

1. Navigate to https://portal.smilestory.com/search

2. Enter the following payload in the search box:
   ' OR '1'='1' --

3. Submit the search request

4. Observe that all customer records are returned

5. Verify SQL injection with time-based technique:
   ' AND SLEEP(5) --

6. Response is delayed by 5 seconds, confirming vulnerability

═══════════════════════════════════════════════════════════════
PROOF OF CONCEPT
═══════════════════════════════════════════════════════════════

Request:
GET /search?q=' OR '1'='1' -- HTTP/1.1
Host: portal.smilestory.com

Response: (200 OK - All customer records returned)
[Screenshot: search_sqli_poc.png]

Database extraction:
GET /search?q=' UNION SELECT username,password,email,1,2
FROM users -- HTTP/1.1

Response: Successfully extracted user credentials
[Screenshot: search_sqli_data_extraction.png]

═══════════════════════════════════════════════════════════════
REMEDIATION
═══════════════════════════════════════════════════════════════

IMMEDIATE (Priority 1 - Complete within 48 hours):

1. Implement parameterized queries (prepared statements):

   # Vulnerable code:
   query = "SELECT * FROM customers WHERE name LIKE '%" +
           user_input + "%'"

   # Secure code (Python with SQLAlchemy):
   from sqlalchemy import text
   query = text("SELECT * FROM customers WHERE name LIKE :name")
   result = db.execute(query, {"name": f"%{user_input}%"})

2. Input validation and sanitization:
   • Whitelist allowed characters
   • Reject special characters in search input
   • Limit input length

3. Implement Web Application Firewall (WAF) rules:
   • ModSecurity Core Rule Set
   • Custom rules for SQL injection patterns

SHORT-TERM (Within 7 days):

4. Database security hardening:
   • Remove excessive database permissions
   • Use principle of least privilege
   • Disable command execution features (xp_cmdshell)

5. Implement additional security controls:
   • Add rate limiting to prevent automated exploitation
   • Enable SQL query logging and monitoring
   • Set up alerts for suspicious SQL patterns

LONG-TERM (Within 30 days):

6. Security training:
   • Train developers on secure coding practices
   • Conduct code review focused on input validation
   • Implement SAST in CI/CD pipeline

7. Testing and validation:
   • Perform code audit of all database queries
   • Conduct regression testing after fixes
   • Schedule follow-up penetration test

═══════════════════════════════════════════════════════════════
REFERENCES
═══════════════════════════════════════════════════════════════

  https://cheatsheetseries.owasp.org/cheatsheets/
  SQL_Injection_Prevention_Cheat_Sheet.html

  https://cwe.mitre.org/data/definitions/89.html

Impact Factors:
• Data sensitivity
• System criticality

🎯 Remediation Guidance

Effective Remediation Recommendations

Best Practices for Remediation Advice

  • Specific: Provide exact steps, not vague suggestions
  • Actionable: Development teams should know exactly what to do
  • Prioritized: Include timeline based on severity
  • Verified: Explain how to test that fix is effective
  • Referenced: Link to authoritative guidance and standards
  • Layered: Suggest defense-in-depth approach

Remediation Priorities

Severity Timeline Approach
CRITICAL 24-48 hours Emergency patch, possible service interruption acceptable
HIGH 7 days Urgent fix in next release cycle
MEDIUM 30 days Planned remediation in regular sprint
LOW 90 days Address in regular maintenance window
INFO As feasible Security best practices, future improvements

Remediation Verification

VERIFICATION CHECKLIST:

For each remediated vulnerability:

□ Code changes reviewed and approved
□ Security testing performed (unit tests, integration tests)
□ Vulnerability scanner confirms fix
□ Manual penetration test validates remediation
□ No new vulnerabilities introduced by the fix
□ Documentation updated
□ Security team trained on the fix
□ Monitoring/alerting configured for similar issues

Regression Testing:
• Re-test the specific vulnerability
• Test related functionality
• Verify fix doesn't break existing features
• Confirm defense-in-depth controls remain effective

📈 Metrics and Trends

Include metrics to track security posture over time:

Key Metrics to Include

  • Vulnerability Density: Vulnerabilities per application/system
  • Time to Remediate: Average days from discovery to fix
  • Remediation Rate: Percentage of vulnerabilities fixed
  • Repeat Findings: Previously identified issues reappearing
  • Risk Trend: Overall risk score trending over time

🔄 Follow-Up and Retesting

Retesting Process

  1. Client Notification: Client reports fixes implemented
  2. Retest Scheduling: Schedule validation testing
  3. Focused Testing: Verify specific fixes, check for regressions
  4. Retest Report: Document verification results
  5. Certificate of Remediation: Issue if all critical/high fixed

📝 Report Writing Best Practices

⚠️ Common Report Mistakes to Avoid

  • Overly Technical: Using jargon without explanation
  • Vague Recommendations: "Improve security" without specifics
  • Missing Evidence: Claims without proof
  • Poor Organization: Difficult to navigate report
  • Inconsistent Scoring: Arbitrary severity ratings
  • Copy-Paste Errors: Incorrect client/system names
  • Spelling/Grammar: Unprofessional presentation
  • Missing Context: Not explaining business impact

Quality Checklist

REPORT QUALITY ASSURANCE:

Content:
□ Executive summary is clear and concise
□ All findings include required elements
□ Evidence (screenshots) is clear and annotated
□ Remediation guidance is specific and actionable
□ Risk ratings are consistent and justified
□ References to standards and best practices included

Technical Accuracy:
□ All technical details verified
□ Exploitation steps reproducible
□ No false positives included
□ CVSS scores calculated correctly

Presentation:
□ Professional formatting and layout
□ Table of contents with page numbers
□ Consistent styling throughout
□ No spelling or grammar errors
□ Client name and details correct
□ Appropriate confidentiality markings

Completeness:
□ All in-scope systems addressed
□ Both positive and negative findings included
□ Methodology clearly explained
□ Timeline and scope documented
□ Appendices include supplementary information
