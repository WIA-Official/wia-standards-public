CHAPTER 03

Vulnerability Assessment

Systematic identification, classification, and prioritization of security weaknesses

🎯 Understanding Vulnerability Assessment

Vulnerability assessment is the systematic process of identifying, quantifying, and prioritizing vulnerabilities in a system. Unlike penetration testing which exploits vulnerabilities, vulnerability assessment focuses on discovery and documentation.

Key Objectives

  • Identify all potential security weaknesses
  • Classify vulnerabilities by type and severity
  • Prioritize remediation based on risk
  • Establish a baseline for security posture
  • Track vulnerability trends over time

Assessment vs Testing

Aspect Vulnerability Assessment Penetration Testing
Goal Find and document vulnerabilities Exploit vulnerabilities to assess impact
Scope Broad, comprehensive Targeted, specific objectives
Automation Heavily automated Mix of automated and manual
Frequency Regular (monthly/quarterly) Periodic (quarterly/annually)
Depth Surface to moderate Deep, thorough exploitation

⚙️ Conducting a Vulnerability Scan

Pre-Scan Preparation

  • Define Scope: Identify all systems, networks, and applications to scan
  • Obtain Authorization: Ensure written permission for all targets
  • Schedule Appropriately: Consider business hours and system load
  • Configure Scanner: Set appropriate scan policies and credentials
  • Baseline Documentation: Document current system state

📊 Understanding CVE and CVSS

Common Vulnerabilities and Exposures (CVE)

CVE is a standardized identifier for publicly known security vulnerabilities:

CVE Format

CVE-YYYY-NNNNN

Example: CVE-2021-44228 (Log4Shell)
- CVE: Common Vulnerabilities and Exposures
- 2021: Year of discovery
- 44228: Unique identifier

Common Vulnerability Scoring System (CVSS)

CVSS provides a standardized method for rating vulnerability severity (0.0 - 10.0):

Score Range Severity Description
9.0 - 10.0 CRITICAL Requires immediate action, easily exploitable
7.0 - 8.9 HIGH Should be addressed urgently
4.0 - 6.9 MEDIUM Address in normal course of operations
0.1 - 3.9 LOW Fix when convenient or not critical

CVSS v3.1 Metrics

Base Metrics:
├── Attack Vector (AV): Network, Adjacent, Local, Physical
├── Attack Complexity (AC): Low, High
├── Privileges Required (PR): None, Low, High
├── User Interaction (UI): None, Required
├── Scope (S): Unchanged, Changed
├── Confidentiality Impact (C): None, Low, High
├── Integrity Impact (I): None, Low, High
└── Availability Impact (A): None, Low, High

Temporal Metrics:
├── Exploit Code Maturity
├── Remediation Level
└── Report Confidence

Environmental Metrics:
├── Modified Base Metrics
└── Impact Subscore Modifiers

Real-World Example: Log4Shell (CVE-2021-44228)

CVSS Score: 10.0 (CRITICAL)

Attack Vector: Network (exploitable remotely)
Attack Complexity: Low (easy to exploit)
Privileges Required: None (no authentication needed)
User Interaction: None (automatic exploitation)
Scope: Changed (affects beyond vulnerable component)
CIA Impact: High / High / High (full system compromise)

🔬 Manual Vulnerability Analysis

Automated scanners have limitations. Manual testing is essential for comprehensive assessment:

Why Manual Testing Matters

  • Logic Flaws: Business logic vulnerabilities scanners can't detect
  • Complex Attacks: Multi-step attack chains requiring context
  • False Positive Verification: Confirming scanner findings
  • Custom Applications: Proprietary code and unique implementations
  • Context Understanding: Impact assessment in real-world scenarios

Manual Testing Techniques

Technique Focus Area Tools
Code Review Source code analysis SonarQube, Semgrep, manual review
Configuration Review System and application configs Manual inspection, CIS benchmarks
Architecture Analysis Design and implementation flaws Threat modeling, diagram review
Fuzzing Input validation weaknesses AFL, Burp Intruder, custom scripts
Logic Testing Business logic flaws Manual testing, custom scenarios

🎯 Common Vulnerability Categories

Web Application Vulnerabilities

Vulnerability Description Typical Impact
SQL Injection Malicious SQL code execution Data breach, authentication bypass
Cross-Site Scripting (XSS) Injection of malicious scripts Session hijacking, defacement
CSRF Unauthorized command transmission Unauthorized actions, data modification
Broken Authentication Weak authentication mechanisms Account takeover, identity theft
Security Misconfiguration Insecure default configs Information disclosure, compromise
Insecure Deserialization Unsafe object deserialization Remote code execution

Network Vulnerabilities

  • Unpatched Systems: Missing security updates and patches
  • Weak Encryption: Outdated protocols (SSLv3, TLS 1.0)
  • Open Services: Unnecessary services exposed to network
  • Default Credentials: Unchanged default passwords
  • Network Segmentation: Lack of proper network isolation

Infrastructure Vulnerabilities

  • Outdated Software: End-of-life operating systems and applications
  • Weak Access Controls: Excessive permissions and privileges
  • Unencrypted Communications: Sensitive data transmitted in clear text
  • Missing Security Headers: Lack of protective HTTP headers
  • Exposed Administrative Interfaces: Management panels accessible externally

📋 Vulnerability Prioritization

Not all vulnerabilities are equal. Effective prioritization considers multiple factors:

Risk-Based Prioritization Matrix

Factor High Priority Medium Priority Low Priority
Exploitability Public exploit available Proof of concept exists Theoretical only
Asset Criticality Critical business system Important but not critical Development/test system
Data Sensitivity PII, financial data, secrets Internal business data Public information
Exposure Internet-facing Internal network Isolated system
Threat Intelligence Active exploitation in wild Targeted by threat actors No known threats

Remediation Timeline

Recommended SLAs

  • Critical: 24-48 hours
  • High: 7 days
  • Medium: 30 days
  • Low: 90 days

🛡️ False Positives and Validation

Scanner results require careful validation:

Common False Positive Scenarios

  • Version-Based Detection: Scanner flags version number without confirming vulnerability
  • WAF Interference: Web Application Firewall blocking scanner probes
  • Configuration Differences: Different environment than what scanner expects
  • Custom Implementations: Application behaves differently than standard

Validation Techniques

# Example: Validating SQL Injection finding

# 1. Review scanner evidence
Finding: SQL Injection in /search?q=
Scanner payload: ' OR 1=1--

# 2. Manual verification
# Test with time-based technique
/search?q=' AND SLEEP(5)--

# 3. Observe response time
# If response delayed by ~5 seconds, confirmed vulnerable

# 4. Document validation
Status: CONFIRMED
Evidence: Response delayed 5.2 seconds with SLEEP payload
Recommendation: Implement parameterized queries

📊 Vulnerability Reporting

Effective reporting is crucial for remediation:

Report Components

  • Executive Summary: High-level overview for management
  • Methodology: Scanning approach and tools used
  • Findings Summary: Vulnerability counts by severity
  • Detailed Findings: Each vulnerability with:
    • Description and impact
    • CVSS score and CVE reference
    • Affected systems/components
    • Proof of concept
    • Remediation recommendations
    • References and resources
  • Trend Analysis: Comparison with previous assessments
  • Remediation Roadmap: Prioritized action plan

Best Practices

  • Use clear, non-technical language in executive summary
  • Provide actionable, specific remediation steps
  • Include visual aids (charts, graphs, screenshots)
  • Reference industry standards and compliance requirements
  • Offer to discuss findings and answer questions

🔄 Continuous Vulnerability Management

Vulnerability management is an ongoing process, not a one-time activity:

Vulnerability Management Lifecycle

1. DISCOVER
   ├── Asset inventory
   ├── Regular scanning
   └── Continuous monitoring

2. PRIORITIZE
   ├── Risk assessment
   ├── Business impact analysis
   └── Threat intelligence integration

3. REMEDIATE
   ├── Patch management
   ├── Configuration changes
   └── Compensating controls

4. VERIFY
   ├── Rescanning
   ├── Validation testing
   └── Effectiveness measurement

5. REPORT
   ├── Stakeholder communication
   ├── Metrics and KPIs
   └── Trend analysis

[Loop back to DISCOVER]

Key Performance Indicators (KPIs)

Metric Purpose Target
Mean Time to Detect (MTTD) Speed of vulnerability discovery < 24 hours
Mean Time to Remediate (MTTR) Speed of vulnerability resolution Based on severity
Vulnerability Density Vulnerabilities per asset Trending downward
Remediation Rate % of vulnerabilities fixed > 95% within SLA
Recurrence Rate % of vulnerabilities reappearing < 5%

⚠️ Common Pitfalls

  • Scanner Dependency: Over-reliance on automated tools without manual validation
  • Alert Fatigue: Too many low-priority findings drowning out critical issues
  • Scope Creep: Scanning systems without proper authorization
  • Incomplete Coverage: Missing critical assets from assessment scope
  • Poor Remediation Tracking: Not following up on identified vulnerabilities
