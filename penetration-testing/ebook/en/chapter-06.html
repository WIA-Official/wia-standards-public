CHAPTER 06

Web Application Testing

OWASP Top 10, API security testing, and modern web application vulnerabilities

🌐 Modern Web Application Landscape

Web applications have evolved from simple static websites to complex, dynamic platforms handling sensitive data and critical business functions. This evolution has introduced numerous security challenges that penetration testers must understand and assess.

Why Web Application Security Matters

  • Web apps are the primary attack surface for most organizations
  • They handle sensitive user data, financial transactions, and business logic
  • Internet-facing applications are accessible to attackers worldwide
  • Modern apps integrate with multiple APIs and third-party services
  • Business logic flaws can be as critical as technical vulnerabilities

Testing Methodology

WEB APPLICATION TESTING WORKFLOW:

1. RECONNAISSANCE
   ├── Identify web technologies (Wappalyzer, BuiltWith)
   ├── Discover hidden endpoints and files
   ├── Map application structure
   └── Identify entry points

2. AUTHENTICATION TESTING
   ├── Test login mechanisms
   ├── Password policy assessment
   ├── Session management review
   └── Multi-factor authentication bypass attempts

3. AUTHORIZATION TESTING
   ├── Privilege escalation attempts
   ├── Insecure direct object references
   ├── Path traversal
   └── Forced browsing

4. INPUT VALIDATION
   ├── SQL injection
   ├── Cross-site scripting (XSS)
   ├── Command injection
   ├── LDAP injection
   └── XML/XXE injection

5. BUSINESS LOGIC
   ├── Workflow bypass
   ├── Race conditions
   ├── Price manipulation
   └── Account enumeration

6. CLIENT-SIDE TESTING
   ├── JavaScript analysis
   ├── DOM-based vulnerabilities
   ├── HTML5 security features
   └── WebSocket testing

7. API TESTING
   ├── REST API security
   ├── GraphQL vulnerabilities
   ├── API authentication
   └── Rate limiting

🔝 OWASP Top 10 (2021)

The OWASP Top 10 represents the most critical security risks to web applications:

A01:2021 – Broken Access Control

Risk: CRITICAL - Moved up from #5, now the most common vulnerability

Description

Access control enforces policy such that users cannot act outside of their intended permissions. Failures typically lead to unauthorized information disclosure, modification, or destruction of data.

Common Vulnerabilities

  • Insecure Direct Object References (IDOR)
  • Missing function-level access control
  • Forced browsing to authenticated pages
  • Privilege escalation
# IDOR Example
# User A's profile: /api/user/profile?id=123
# Can access User B's profile: /api/user/profile?id=124

GET /api/user/profile?id=124 HTTP/1.1
Host: example.com
Cookie: session=user_a_session

# Should return 403 Forbidden, but returns User B's data

# Testing approach:
1. Identify parameters referencing objects (id, uid, profile_id)
2. Try accessing other users' resources
3. Test both horizontal (same role) and vertical (different role)
4. Check for predictable identifiers
5. Test with different HTTP methods (GET, POST, PUT, DELETE)

A02:2021 – Cryptographic Failures

Risk: HIGH - Previously known as "Sensitive Data Exposure"

Issue Example Impact
Weak Encryption Using DES, RC4, or weak RSA keys Data can be decrypted
Missing Encryption Transmitting passwords over HTTP Credentials intercepted
Weak Password Hashing MD5, SHA1 without salt Passwords easily cracked
Improper Key Management Hardcoded keys in source code Complete encryption bypass

Risk: CRITICAL - Dropped from #1 but remains critical

A04:2021 – Insecure Design

Risk: HIGH - New category focusing on design flaws

Key Concepts

  • Missing or ineffective security controls by design
  • Threat modeling during design phase
  • Secure design patterns and principles
  • Business logic vulnerabilities

Example: Account Recovery Design Flaw

# Insecure Design:
1. User requests password reset
2. System sends reset link to email
3. Link format: /reset?token=USER_ID_TIMESTAMP
4. Token is predictable

# Secure Design:
1. User requests password reset
2. Generate cryptographically random token
3. Store token with expiration
4. Verify token before allowing reset
5. Invalidate token after use

A05:2021 – Security Misconfiguration

Risk: HIGH - One of the most common issues

Misconfiguration Testing Method Risk
Default Credentials Try admin/admin, admin/password Full system access
Directory Listing Browse to directories without index Information disclosure
Verbose Errors Trigger errors, check responses Information leakage
Missing Security Headers Check HTTP response headers Various attacks enabled
Unnecessary Services Port scan, service enumeration Expanded attack surface

A06:2021 – Vulnerable and Outdated Components

Risk: HIGH - Using components with known vulnerabilities

# Identifying Vulnerable Components

# JavaScript libraries
- Check loaded scripts in browser DevTools
- Look for version numbers in filenames
- Use retire.js or similar tools

# Server-side frameworks
- Analyze HTTP headers: X-Powered-By, Server
- Check error messages for version info
- Use Wappalyzer browser extension

# Example vulnerable libraries (historical):
- jQuery < 3.5.0 (XSS vulnerabilities)
- Log4j < 2.17.0 (Log4Shell RCE)
- Apache Struts < 2.5.26 (RCE vulnerabilities)
- Spring Framework < 5.3.18 (Spring4Shell)

# Testing approach:
1. Identify all components and versions
2. Check CVE databases for known vulnerabilities
3. Verify exploitability in the specific context
4. Test with proof-of-concept exploits (authorized only)

A07:2021 – Identification and Authentication Failures

Risk: CRITICAL - Critical for account security

Testing Checklist

  • Weak Passwords: Test for password complexity requirements
  • Brute Force: Check for rate limiting and account lockout
  • Credential Stuffing: Test with known breached credentials
  • Session Fixation: Check if session ID changes after login
  • Session Timeout: Verify inactive session expiration
  • Password Reset: Test for account enumeration, token security
  • MFA Bypass: Look for ways to circumvent 2FA
# Authentication Testing Examples

# 1. Account Enumeration
POST /login
{"username":"existing_user","password":"wrong"}
→ Response: "Invalid password" (user exists!)

POST /login
{"username":"nonexistent","password":"wrong"}
→ Response: "Invalid credentials" (user doesn't exist!)

# 2. Session Fixation Test
# Before login:
Cookie: SESSIONID=attacker_set_value
# After login:
Cookie: SESSIONID=attacker_set_value (VULNERABLE!)
# Should change to new random session ID

# 3. Weak Password Reset
GET /reset-password?email=victim@example.com
# Check if token is predictable
# Try reusing old tokens
# Test token expiration
# Check for account enumeration

A08:2021 – Software and Data Integrity Failures

Risk: HIGH - New category including insecure deserialization

A09:2021 – Security Logging and Monitoring Failures

Risk: MEDIUM - Enables attackers to persist undetected

A10:2021 – Server-Side Request Forgery (SSRF)

Risk: HIGH - New addition to Top 10

# SSRF Testing Examples

# Basic SSRF
# Application fetches URL provided by user
POST /fetch-url
{"url":"http://internal-server/admin"}

# Cloud metadata exploitation (AWS)
{"url":"http://169.254.169.254/latest/meta-data/iam/security-credentials/"}

# Bypassing filters
http://localhost → http://127.0.0.1
http://127.0.0.1 → http://0.0.0.0
http://0.0.0.0 → http://[::1]
http://example.com → http://example.com@127.0.0.1

# Testing checklist:
1. Identify URL input parameters
2. Try accessing internal resources
3. Test cloud metadata endpoints
4. Attempt to bypass IP filters
5. Check for blind SSRF (out-of-band detection)

🔌 API Security Testing

Modern applications heavily rely on APIs, making API security testing essential:

OWASP API Security Top 10

Risk Description Testing Focus
API1: Broken Object Level Authorization IDOR in APIs Test accessing other users' objects
API2: Broken Authentication Weak API authentication Test auth mechanisms, tokens
API3: Broken Object Property Level Authorization Mass assignment vulnerabilities Add unexpected fields to requests
API4: Unrestricted Resource Consumption No rate limiting Test for DoS via excessive requests
API5: Broken Function Level Authorization Missing authorization on functions Access admin endpoints as user

REST API Testing

# Common API Testing Scenarios

# 1. Mass Assignment
POST /api/users
{
  "username": "newuser",
  "email": "user@example.com",
  "role": "admin"  ← Should not be settable!
}

# 2. Excessive Data Exposure
GET /api/users/123
Response includes:
{
  "id": 123,
  "username": "john",
  "email": "john@example.com",
  "password_hash": "...",  ← Should not be exposed!
  "ssn": "123-45-6789"     ← Should not be exposed!
}

# 3. Lack of Rate Limiting
# Send 1000 requests in 1 second
for i in {1..1000}; do
  curl https://api.example.com/endpoint &
done
# Should be blocked/throttled

# 4. API Versioning Issues
/api/v1/users (deprecated, vulnerable)
/api/v2/users (current, secure)
# Test if v1 is still accessible

# 5. HTTP Method Override
# Try different methods on same endpoint
GET /api/users/123     (Read)
POST /api/users/123    (Should fail or be unauthorized)
PUT /api/users/123     (Update - test authorization)
DELETE /api/users/123  (Delete - test authorization)
PATCH /api/users/123   (Partial update)

GraphQL Security Testing

# GraphQL-Specific Vulnerabilities

# 1. Introspection Query (Info disclosure)
query {
  __schema {
    types {
      name
      fields {
        name
      }
    }
  }
}

# 2. Depth-based DoS
query {
  user {
    posts {
      comments {
        user {
          posts {
            comments {
              # Nested deeply to exhaust resources
            }
          }
        }
      }
    }
  }
}

# 3. Batch Query Attack
query {
  user1: user(id: 1) { name }
  user2: user(id: 2) { name }
  user3: user(id: 3) { name }
  # ... repeat 1000 times
}

# 4. Field Duplication
query {
  user(id: 1) {
    name
    name
    name
    # Repeat to cause processing overhead
  }
}

JWT (JSON Web Token) Testing

# JWT Structure
eyJhbGci... (Header) . eyJzdWIi... (Payload) . SflKxwRJ... (Signature)

# Common JWT Vulnerabilities

# 1. Algorithm Confusion (alg: none)
{
  "alg": "none",
  "typ": "JWT"
}
# Remove signature, try processing

# 2. Algorithm Switching (RS256 → HS256)
# RS256 uses private/public key pair
# HS256 uses symmetric key
# If public key is known, sign with it using HS256

# 3. Weak Secret
# Brute force JWT secret
jwt-cracker   

# 4. Missing Expiration
# Check if 'exp' claim is validated
# Try using old/expired tokens

# 5. Information Disclosure
# Decode payload (base64)
echo "eyJzdWIi..." | base64 -d
# Check for sensitive data in payload

Burp Suite Workflow

# Typical Burp Suite Testing Workflow

1. CONFIGURE PROXY
   Browser → Burp Proxy (127.0.0.1:8080)
   Install Burp CA certificate

2. MAP APPLICATION
   Spider/Crawler → Discover all endpoints
   Analyze sitemap

3. INTERCEPT & MODIFY
   Intercept requests in Proxy
   Modify parameters, headers, cookies
   Forward to application

4. REPEAT & TEST
   Send interesting requests to Repeater
   Modify and resend multiple times
   Analyze responses

5. AUTOMATE ATTACKS
   Send to Intruder
   Configure payload positions
   Run automated attacks (brute force, fuzzing)

6. SCAN FOR VULNERABILITIES
   Active/Passive scanning
   Review scan results
   Validate findings

7. REPORT FINDINGS
   Export issues
   Document exploitable vulnerabilities
   Provide remediation guidance

📋 Web Application Testing Checklist

Comprehensive Testing Checklist

  • ✓ Information Gathering (technologies, endpoints, parameters)
  • ✓ Authentication Testing (login, password reset, MFA)
  • ✓ Session Management (cookies, tokens, timeout)
  • ✓ Authorization (IDOR, privilege escalation, forced browsing)
  • ✓ Input Validation (SQL, XSS, command injection)
  • ✓ Business Logic (workflow bypass, race conditions)
  • ✓ Cryptography (TLS, hashing, encryption)
  • ✓ Error Handling (verbose errors, stack traces)
  • ✓ File Upload (type validation, path traversal)
  • ✓ API Security (authentication, rate limiting, data exposure)
  • ✓ Client-Side (JavaScript analysis, DOM XSS)
  • ✓ Security Headers (CSP, HSTS, X-Frame-Options)
