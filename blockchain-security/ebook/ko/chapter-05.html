제5장: 키 저장소와 지갑 보안

장 개요: 개인키 보안은 블록체인 자산 보호의 기본입니다. 이 장에서는 멀티시그 지갑, 콜드 스토리지 아키텍처, 하드웨어 보안 모듈, 그리고 키 수명주기 관리 모범 사례를 다룹니다.

키 저장소 전략

키 저장소 전략의 선택은 보안 요구사항, 운영 필요성, 위험에 처한 가치에 따라 달라집니다. 서로 다른 접근 방식은 보안과 편의성 사이의 다양한 트레이드오프를 제공합니다.

멀티시그 지갑 구현

멀티시그(다중 서명) 지갑은 거래 승인에 여러 당사자가 필요하여 단일 장애 지점을 제거하고 키 손상에 대한 방어를 제공합니다.

보안 멀티시그 구현

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.19;

contract SecureMultiSig {
    event TransactionProposed(uint256 indexed txId, address indexed proposer);
    event TransactionExecuted(uint256 indexed txId);

    uint256 public constant EXECUTION_DELAY = 24 hours;
    address[] public signers;
    mapping(address => bool) public isSigner;
    uint256 public threshold;
    uint256 public nonce;

    struct Transaction {
        address to;
        uint256 value;
        bytes data;
        uint256 proposedAt;
        uint256 signatureCount;
        bool executed;
    }

    mapping(uint256 => Transaction) public transactions;

    modifier onlySigner() { require(isSigner[msg.sender], "서명자가 아님"); _; }

    constructor(address[] memory _signers, uint256 _threshold) {
        require(_signers.length >= _threshold && _threshold > 0);
        for (uint256 i = 0; i < _signers.length; i++) {
            isSigner[_signers[i]] = true;
            signers.push(_signers[i]);
        }
        threshold = _threshold;
    }

    function proposeTransaction(address to, uint256 value, bytes calldata data)
        external onlySigner returns (uint256 txId) {
        txId = nonce++;
        transactions[txId] = Transaction(to, value, data, block.timestamp, 1, false);
        emit TransactionProposed(txId, msg.sender);
    }

    function executeTransaction(uint256 txId) external onlySigner {
        Transaction storage txn = transactions[txId];
        require(!txn.executed && txn.signatureCount >= threshold);
        require(block.timestamp >= txn.proposedAt + EXECUTION_DELAY);
        txn.executed = true;
        (bool success, ) = txn.to.call{value: txn.value}(txn.data);
        require(success);
        emit TransactionExecuted(txId);
    }

    receive() external payable {}
}

콜드 스토리지 아키텍처

콜드 스토리지는 개인키를 완전히 오프라인으로 유지하여 원격 공격에 대한 최대 보안을 제공합니다. 적절한 콜드 스토리지 구현에는 신중한 프로토콜 설계가 필요합니다.

콜드 스토리지 원칙: 콜드 스토리지의 개인키는 인터넷에 연결된 장치에 절대 노출되어서는 안 됩니다. 모든 트랜잭션 서명은 에어갭 시스템에서 수행되며, 서명된 트랜잭션은 안전한 방법으로 전송됩니다.

콜드 스토리지 프로토콜

콜드 스토리지 트랜잭션 프로토콜:

1단계: 트랜잭션 준비 (온라인)
├── 핫 시스템에서 미서명 트랜잭션 생성
├── 목적지와 금액 확인
├── 서명을 위한 트랜잭션 해시 생성
└── QR 코드 또는 USB 장치로 내보내기

2단계: 에어갭 전송
├── 장치가 네트워크에 연결된 적 없음 확인
├── 미서명 트랜잭션 전송 방법:
│   ├── QR 코드 스캔 (권장)
│   ├── USB 장치 (일회용, 사용 후 포맷)
│   └── 수동 입력 (간단한 트랜잭션용)
└── 콜드 장치에서 트랜잭션 세부사항 확인

3단계: 오프라인 서명
├── 콜드 스토리지 장치에서 트랜잭션 검토
├── 모든 트랜잭션 매개변수 확인
├── PIN/암호 입력하여 서명키 잠금 해제
├── 트랜잭션 서명
└── 서명된 트랜잭션 내보내기

4단계: 브로드캐스트 (온라인)
├── 서명된 트랜잭션을 핫 시스템으로 전송
├── 서명된 트랜잭션 최종 확인
├── 블록체인 네트워크에 브로드캐스트
└── 확인 모니터링

보안 조치:
├── 물리적 접근 통제가 있는 콜드 스토리지 룸
├── 무선 신호 차단을 위한 패러데이 케이지
├── 서명을 위한 다중 증인 요구사항
├── 서명 세레모니 영상 기록
└── 변조 방지 씰이 있는 하드웨어 월렛

하드웨어 보안 모듈(HSM) 통합

HSM은 기관 사용을 위한 최고 수준의 키 보호를 제공하며, 변조 저항, 감사 로깅, 암호화 가속을 제공합니다.

HSM 구성

// 블록체인 서명을 위한 HSM 통합
import { HSMClient, KeyType, SignatureAlgorithm } from '@hsm/client';

interface HSMConfig {
    endpoint: string;
    partition: string;
    credentials: { type: 'certificate' | 'password'; value: string; };
    keyPolicy: { algorithm: string; quorumSize: number; };
}

class BlockchainHSMSigner {
    private client: HSMClient;

    constructor(config: HSMConfig) {
        this.client = new HSMClient(config);
    }

    async initialize(): Promise<void> {
        await this.client.connect();
        const status = await this.client.getStatus();
        if (status.tamperStatus !== 'CLEAR') throw new Error('HSM 변조 감지됨');
    }

    async generateKey(keyId: string): Promise<string> {
        const keyHandle = await this.client.generateKey({
            id: keyId, type: KeyType.ECDSA, curve: 'secp256k1'
        });
        return await this.client.getPublicKey(keyHandle);
    }

    async signTransaction(keyId: string, txHash: Buffer, auth: AuthToken[]): Promise<Signature> {
        if (auth.length < this.config.keyPolicy.quorumSize) {
            throw new Error('권한 부여 부족');
        }
        const sig = await this.client.sign({ keyId, data: txHash, authorization: auth });
        return { r: sig.r, s: sig.s, v: sig.recoveryParam };
    }
}

키 수명주기 관리

적절한 키 수명주기 관리는 키가 유용한 수명 동안 안전하게 유지되고 더 이상 필요하지 않을 때 적절히 폐기되도록 보장합니다.

키 수명주기 단계:

1. 생성
   ├── 암호학적으로 안전한 RNG 사용
   ├── 안전한 환경에서 생성 (HSM/에어갭)
   ├── 증인과 함께 키 세레모니 구현
   └── 생성 매개변수 문서화

2. 배포
   ├── 안전한 키 공유 배포 (Shamir)
   ├── 공유 무결성 확인
   ├── 보관 체인 수립
   └── 공유 보유자 문서화

3. 저장
   ├── 저장 시 키 암호화
   ├── 접근 통제 구현
   ├── 정기적 백업 확인
   └── 지리적 분산

4. 사용
   ├── 모든 키 사용 감사
   ├── 속도 제한 구현
   ├── 이상 징후 모니터링
   └── 권한 부여 정책 시행

5. 교체
   ├── 정기 교체 일정
   ├── 비상 교체 기능
   ├── 마이그레이션을 위한 중복 기간
   └── 모든 종속 시스템 업데이트

6. 폐기
   ├── 모든 시스템에서 안전한 삭제
   ├── 남은 접근 권한 철회
   ├── 감사 로그 보관
   └── 파기 문서화
요약: 키 저장소와 지갑 보안은 블록체인 자산 보호의 기본입니다. 멀티시그 지갑은 단일 장애 지점을 제거하고, 콜드 스토리지는 장기 보유에 대한 최대 보호를 제공하며, HSM은 기관급 보안을 제공합니다. 적절한 키 수명주기 관리를 구현하면 키의 유용한 수명 동안 보안이 보장됩니다.
