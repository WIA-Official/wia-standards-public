Chapter 02

Data Structures and Formats

Standardized Data Representation in Identity Protection Systems

2.1 Overview of Data Formats

The WIA-IDENTITY_THEFT_PREVENTION standard defines comprehensive data formats for identity profiles, authentication credentials, alerts, and security information. These standardized formats ensure interoperability across different systems and enable seamless integration of identity protection services.

2.1.1 Design Principles

  • Privacy First: All sensitive data includes encryption and anonymization requirements
  • Interoperability: JSON-based formats for maximum compatibility across platforms
  • Real-Time Ready: Optimized for streaming and instant alert delivery
  • Compliance Built-In: GDPR, CCPA, and PCI-DSS compliant by design
  • Extensibility: Version-aware schemas support future enhancements
  • Data Integrity: Checksum and digital signature requirements

2.2 Identity Profile Format

The identity profile is the core data structure containing personal information, contact details, and protection settings.

2.2.1 Base Identity Object

{ "identity": { "id": "uuid-v4", "version": "1.0", "timestamp": "2026-01-12T10:30:00Z", "profile": { "firstName": "John", "lastName": "Doe", "dateOfBirth": "1990-01-15", "ssn": "encrypted-string", "email": "john.doe@example.com", "phone": "+12025551234", "addresses": [ { "type": "home", "street": "123 Main St", "city": "New York", "state": "NY", "postalCode": "10001", "country": "US" } ] }, "verification": { "status": "verified", "method": "biometric", "verifiedAt": "2026-01-12T10:30:00Z", "confidence": 0.95 }, "metadata": { "createdAt": "2026-01-12T10:30:00Z", "updatedAt": "2026-01-12T10:30:00Z", "source": "user_registration", "checksum": "sha256-hash" } } }

2.2.2 Personal Information Structure

Personal information is stored in a structured format with multiple layers of detail:

{ "personalInformation": { "legalName": { "prefix": "Mr.", "firstName": "John", "middleName": "Michael", "lastName": "Doe", "suffix": "Jr.", "maiden": null }, "aliases": [ { "name": "Johnny Doe", "type": "nickname", "validFrom": "2010-01-01", "validTo": null } ], "demographics": { "dateOfBirth": "1990-01-15", "placeOfBirth": "New York, NY", "gender": "male", "nationality": "US", "citizenship": ["US"] }, "identifiers": { "ssn": { "value": "encrypted-string", "country": "US", "verifiedAt": "2026-01-12T10:30:00Z" }, "driverLicense": { "number": "encrypted-string", "state": "NY", "expiresAt": "2028-01-15" }, "passport": { "number": "encrypted-string", "country": "US", "expiresAt": "2030-01-15" } } } }
ℹ️ Encryption Note: All personally identifiable information (PII) such as SSN, driver's license numbers, and passport numbers must be encrypted using AES-256-GCM before storage or transmission. The encryption keys should be managed separately using a key management service (KMS).

Authentication credentials support multiple authentication factors including passwords, biometrics, and tokens.

2.3.1 Credential Structure

2.5 Biometric Data Format

Biometric templates are stored in standardized formats with strong encryption and anti-spoofing measures.

2.5.1 Biometric Template Structure

{ "biometric": { "id": "uuid-v4", "userId": "uuid-v4", "modality": "fingerprint", "version": "1.0", "template": { "format": "ISO-19794-2", "encoding": "base64", "data": "encrypted-base64-string", "hash": "sha256-hash", "compression": "none" }, "quality": { "score": 92, "nfiqScore": 2, "resolution": 500, "captureMethod": "optical" }, "enrollment": { "deviceId": "scanner-123", "deviceType": "optical_scanner", "enrolledAt": "2026-01-12T10:30:00Z", "enrolledBy": "user", "location": "New York, NY", "attempts": 3 }, "metadata": { "leftRight": "right", "fingerPosition": "index" }, "security": { "encryptionAlgorithm": "AES-256-GCM", "keyId": "key-uuid", "antiSpoofing": { "liveness": "passed", "method": "pulsation", "confidence": 0.98 } }, "lifecycle": { "status": "active", "expiresAt": "2027-01-12T10:30:00Z", "lastVerified": "2026-01-12T14:30:00Z", "verificationCount": 47 } } }
⚠️ Security Warning: Biometric templates must never be stored in reversible format. Always use one-way hashing or secure template formats that cannot be used to reconstruct the original biometric image. Implement liveness detection to prevent spoofing attacks using photos or replicas.

2.8 Encryption and Security

All sensitive data must be encrypted both at rest and in transit using industry-standard algorithms.

2.8.1 Encryption Requirements

Data Type Encryption Method Key Length Algorithm
PII (at rest) AES-GCM 256-bit AES-256-GCM
PII (in transit) TLS 256-bit TLS 1.3
Biometric templates AES-GCM 256-bit AES-256-GCM
Passwords Argon2id N/A Argon2id (m=64MB, t=3, p=4)
Authentication tokens HMAC 256-bit HMAC-SHA256

🎯 Key Takeaways

  • WIA-IDENTITY_THEFT_PREVENTION uses JSON-based standardized data formats for interoperability
  • All PII must be encrypted using AES-256-GCM with separate key management
  • Identity profiles include personal information, verification status, and metadata
  • Authentication credentials support multiple factors: passwords, biometrics, tokens, and MFA
  • Security alerts are structured with severity levels, threat details, and response actions
  • Biometric templates include quality scores, enrollment details, and anti-spoofing measures
  • Credit monitoring data follows standardized formats across all major bureaus
  • Strict validation rules ensure data integrity and consistency across systems
