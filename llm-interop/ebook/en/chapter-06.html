Chapter 6: Protocol Specifications

"A protocol is a promise - the foundation of trust between systems."

6.1 Protocol Overview

The WIA LLM Interoperability Standard supports three transport protocols:

Protocol Use Case Characteristics
REST API Standard operations Simple, stateless, widely supported
WebSocket Real-time streaming Bidirectional, persistent connection
gRPC High-performance Binary, efficient, strongly typed

6.3 WebSocket Protocol

Connection

WebSocket URL: wss://api.wia.org/ws/v1/llm

Connection Request:
GET /ws/v1/llm HTTP/1.1
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Protocol: wia-llm-v1

Handshake Sequence

1. Client connects to WebSocket endpoint
   CLIENT ---> SERVER: WebSocket upgrade request

2. Server accepts connection
   CLIENT <--- SERVER: 101 Switching Protocols

3. Client sends authentication
   CLIENT ---> SERVER: {
       "type": "AUTH",
       "token": "{did-auth-token}",
       "did": "did:wia:llm:anthropic:claude-3-opus"
   }

4. Server confirms authentication
   CLIENT <--- SERVER: {
       "type": "AUTH_ACK",
       "sessionId": "sess-123",
       "expiresIn": 3600
   }

5. Ready for messages

Message Types

TypeDirectionDescription
AUTHClient -> ServerAuthentication request
AUTH_ACKServer -> ClientAuthentication accepted
CAPABILITY_REGISTERClient -> ServerRegister capability
CAPABILITY_QUERYClient -> ServerQuery capabilities
MESSAGE_SENDClient -> ServerSend message
MESSAGE_RECEIVEServer -> ClientIncoming message
STREAM_STARTServer -> ClientStream beginning
STREAM_CHUNKServer -> ClientStream data chunk
STREAM_ENDServer -> ClientStream complete
PINGBothKeep-alive ping
PONGBothKeep-alive response

6.4 gRPC Protocol

Service Definitions (Protocol Buffers)

syntax = "proto3";
package wia.llm.v1;

// Capability Service
service LLMCapabilityService {
    rpc RegisterCapability(CapabilityRequest)
        returns (CapabilityResponse);

    rpc QueryCapabilities(QueryRequest)
        returns (stream CapabilityMatch);

    rpc GetCapability(GetCapabilityRequest)
        returns (CapabilityDocument);
}

// Message Service
service LLMMessageService {
    rpc SendMessage(MessageRequest)
        returns (MessageResponse);

    rpc StreamMessage(MessageRequest)
        returns (stream StreamChunk);

    rpc BidirectionalStream(stream MessageRequest)
        returns (stream MessageResponse);
}

// Federation Service
service LLMFederationService {
    rpc CreateFederation(FederationRequest)
        returns (FederationResponse);

    rpc JoinFederation(JoinRequest)
        returns (JoinResponse);

    rpc SubmitTask(TaskRequest)
        returns (stream TaskProgress);

    rpc InitiateConsensus(ConsensusRequest)
        returns (ConsensusResult);
}

6.5 Security Specifications

DID Authentication

Authentication Header Format:
Authorization: DID-Auth {signed-challenge}

Challenge-Response Flow:
1. Client requests challenge
   GET /auth/challenge?did={did}

2. Server returns challenge
   { "challenge": "random-string", "expiresIn": 300 }

3. Client signs challenge with DID private key
   signature = Sign(challenge, privateKey)

4. Client sends signed challenge
   POST /auth/verify
   { "did": "{did}", "challenge": "...", "signature": "..." }

5. Server verifies and returns token
   { "token": "{jwt}", "expiresIn": 3600 }

Message Signing

Signature Format:
{
    "algorithm": "Ed25519",
    "publicKey": "did:wia:llm:...#key-1",
    "value": "base64-encoded-signature"
}

Signing Process:
1. Serialize message to canonical JSON
2. Compute SHA-256 hash
3. Sign hash with Ed25519 private key
4. Encode signature as base64

Encryption

Encryption Format:
{
    "algorithm": "AES-256-GCM",
    "keyAgreement": "X25519",
    "recipientKey": "did:wia:llm:...#encryption-key-1",
    "ephemeralKey": "base64-public-key",
    "nonce": "base64-nonce",
    "ciphertext": "base64-encrypted-payload"
}

Encryption Process:
1. Generate ephemeral X25519 key pair
2. Derive shared secret with recipient public key
3. Derive AES key using HKDF
4. Encrypt payload with AES-256-GCM
5. Include ephemeral public key and nonce
