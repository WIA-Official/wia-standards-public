Chapter 3: AI-to-AI Messaging (WIA-LLM-MESSAGE)

"For AI models to collaborate, they must speak a common language."

3.1 The Need for AI Messaging

Once AI models know each other's capabilities, they need a way to communicate. The WIA-LLM-MESSAGE standard defines how AI models exchange information:

  • Send queries to other AI models
  • Delegate subtasks
  • Request verification of results
  • Stream responses in real-time
  • Handle errors gracefully

3.2 Message Envelope Structure

Every message is wrapped in a standardized envelope:

MessageEnvelope
├── @context           # JSON-LD context
├── id                 # Unique message ID
├── type               # "MessageEnvelope"
├── version            # Protocol version
├── header             # Routing and metadata
├── payload            # Message content
├── routing            # Delivery instructions
└── security           # Encryption and signing

3.3 Message Header

The header contains essential routing information:

header: {
    from: "did:wia:llm:anthropic:claude-3-opus",
    to: "did:wia:llm:openai:gpt-4",
    timestamp: "2024-01-15T10:30:00Z",
    messageType: "QUERY",
    correlationId: "corr-abc123",
    replyTo: "did:wia:llm:anthropic:claude-3-opus",
    ttl: 300,
    priority: "NORMAL"
}

Header Fields

FieldTypeDescription
fromDIDSender's identifier
toDIDRecipient's identifier
timestampISO 8601When message was created
messageTypeenumType of message
correlationIdstringLinks related messages
replyToDIDWhere to send response
ttlnumberTime-to-live in seconds
priorityenumMessage priority level

3.4 Message Types

The standard defines several message types:

Type Purpose Expected Response
QUERY Ask a question or request information RESPONSE
DELEGATE Assign a subtask to another AI RESPONSE or STREAM
COLLABORATE Request collaboration on a task ACCEPT/REJECT
VERIFY Request verification of results VERIFICATION
RESPONSE Reply to a query or delegation None (final)
STREAM Streaming response chunks More STREAM or END
ERROR Report an error condition None (final)

3.5 Query Payload

A query message requests information or processing:

{
    "header": {
        "from": "did:wia:llm:anthropic:claude-3-opus",
        "to": "did:wia:llm:local:codellama-34b",
        "messageType": "QUERY"
    },
    "payload": {
        "type": "query_payload",
        "content": "Analyze this code for security vulnerabilities",
        "context": {
            "code": "function login(user, pass) { ... }",
            "language": "javascript"
        },
        "parameters": {
            "maxTokens": 2048,
            "temperature": 0.3,
            "responseFormat": "structured"
        },
        "constraints": {
            "timeout": 30000,
            "requiredFields": ["vulnerabilities", "severity"]
        }
    }
}

3.6 Response Payload

A response message contains the result:

{
    "header": {
        "from": "did:wia:llm:local:codellama-34b",
        "to": "did:wia:llm:anthropic:claude-3-opus",
        "messageType": "RESPONSE",
        "correlationId": "corr-abc123"
    },
    "payload": {
        "type": "response_payload",
        "status": "SUCCESS",
        "content": {
            "vulnerabilities": [
                {
                    "type": "SQL_INJECTION",
                    "severity": "HIGH",
                    "line": 15,
                    "description": "User input not sanitized"
                }
            ],
            "summary": "Found 1 high-severity vulnerability"
        },
        "metadata": {
            "processingTime": 1250,
            "tokensUsed": 856,
            "confidence": 0.92
        }
    }
}

3.7 Delegation Workflow

When an AI delegates a subtask to another:

Step 1: Delegating AI sends DELEGATE message
+------------------+                    +------------------+
|   Claude 3       |  --- DELEGATE -->  |   Code Llama     |
|   (Orchestrator) |                    |   (Specialist)   |
+------------------+                    +------------------+

Step 2: Specialist processes and responds
+------------------+                    +------------------+
|   Claude 3       |  <-- RESPONSE ---  |   Code Llama     |
|                  |                    |                  |
+------------------+                    +------------------+

Step 3: Orchestrator continues with result

Delegation Message Example

{
    "header": {
        "messageType": "DELEGATE"
    },
    "payload": {
        "type": "delegate_payload",
        "task": "Analyze Python code for type errors",
        "input": {
            "code": "...",
            "requirements": ["mypy", "strict"]
        },
        "expectedOutput": {
            "format": "json",
            "schema": "TypeErrorReport"
        },
        "deadline": "2024-01-15T10:35:00Z"
    }
}

3.8 Streaming Responses

For long responses, streaming delivers content incrementally:

Stream Message 1:
{
    "header": { "messageType": "STREAM" },
    "payload": {
        "sequenceNumber": 1,
        "chunk": "The analysis reveals ",
        "done": false
    }
}

Stream Message 2:
{
    "header": { "messageType": "STREAM" },
    "payload": {
        "sequenceNumber": 2,
        "chunk": "several security concerns...",
        "done": false
    }
}

Stream Message 3 (Final):
{
    "header": { "messageType": "STREAM" },
    "payload": {
        "sequenceNumber": 3,
        "chunk": "",
        "done": true,
        "totalTokens": 1523
    }
}

3.9 Error Handling

When errors occur, the ERROR message type is used:

{
    "header": {
        "messageType": "ERROR",
        "correlationId": "corr-abc123"
    },
    "payload": {
        "type": "error_payload",
        "code": "CAPABILITY_MISMATCH",
        "message": "Task requires MEDICAL domain expertise",
        "details": {
            "required": ["MEDICAL"],
            "available": ["CODE", "MATH"]
        },
        "recoverable": true,
        "suggestion": "Route to a model with MEDICAL capability"
    }
}

Standard Error Codes

CodeDescriptionRecoverable
CAPABILITY_MISMATCHModel lacks required capabilityYes
RATE_LIMITEDToo many requestsYes
CONTEXT_OVERFLOWInput exceeds context windowYes
TIMEOUTProcessing took too longYes
CONTENT_POLICYRequest violates content policyNo
AUTHENTICATION_FAILEDInvalid credentialsYes
INTERNAL_ERRORUnexpected errorMaybe

3.10 Message Routing

Messages include routing instructions:

routing: {
    strategy: "CAPABILITY_BASED",
    hops: [],
    maxHops: 3,
    fallback: {
        strategy: "BROADCAST",
        scope: "FEDERATION"
    }
}

Routing Strategies

StrategyDescription
DIRECTSend directly to specified recipient
CAPABILITY_BASEDRoute based on capability match
LOAD_BALANCEDDistribute across available models
BROADCASTSend to all federation members
