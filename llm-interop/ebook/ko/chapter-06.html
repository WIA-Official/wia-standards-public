제6장: 프로토콜 명세

"프로토콜은 약속입니다 - 시스템 간 신뢰의 기반입니다."

6.1 프로토콜 개요

WIA LLM 상호운용 표준은 세 가지 전송 프로토콜을 지원합니다:

프로토콜 사용 사례 특성
REST API 표준 작업 단순, 무상태, 널리 지원됨
WebSocket 실시간 스트리밍 양방향, 지속 연결
gRPC 고성능 바이너리, 효율적, 강타입

6.2 REST API 명세

기본 URL

https://api.wia.org/v1/llm

공통 헤더

Content-Type: application/json
Authorization: Bearer {did-auth-token}
X-WIA-Version: 1.0
X-WIA-Request-ID: {uuid}

능력 엔드포인트

메서드엔드포인트설명
POST/capability능력 등록
GET/capability/{id}ID로 능력 조회
PUT/capability/{id}능력 업데이트
DELETE/capability/{id}능력 제거
POST/capability/query능력 쿼리

6.3 WebSocket 프로토콜

연결

WebSocket URL: wss://api.wia.org/ws/v1/llm

연결 요청:
GET /ws/v1/llm HTTP/1.1
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Protocol: wia-llm-v1

핸드셰이크 순서

1. 클라이언트가 WebSocket 엔드포인트에 연결
   CLIENT ---> SERVER: WebSocket 업그레이드 요청

2. 서버가 연결 수락
   CLIENT <--- SERVER: 101 Switching Protocols

3. 클라이언트가 인증 전송
   CLIENT ---> SERVER: {
       "type": "AUTH",
       "token": "{did-auth-token}",
       "did": "did:wia:llm:anthropic:claude-3-opus"
   }

4. 서버가 인증 확인
   CLIENT <--- SERVER: {
       "type": "AUTH_ACK",
       "sessionId": "sess-123",
       "expiresIn": 3600
   }

5. 메시지 준비 완료

메시지 유형

유형방향설명
AUTH클라이언트 -> 서버인증 요청
AUTH_ACK서버 -> 클라이언트인증 수락
CAPABILITY_REGISTER클라이언트 -> 서버능력 등록
CAPABILITY_QUERY클라이언트 -> 서버능력 쿼리
MESSAGE_SEND클라이언트 -> 서버메시지 전송
MESSAGE_RECEIVE서버 -> 클라이언트수신 메시지
STREAM_START서버 -> 클라이언트스트림 시작
STREAM_CHUNK서버 -> 클라이언트스트림 데이터 청크
STREAM_END서버 -> 클라이언트스트림 완료
PING양방향연결 유지 핑
PONG양방향연결 유지 응답

스트리밍 응답 예시

클라이언트가 쿼리 전송:

6.4 gRPC 프로토콜

서비스 정의 (Protocol Buffers)

syntax = "proto3";
package wia.llm.v1;

// 능력 서비스
service LLMCapabilityService {
    rpc RegisterCapability(CapabilityRequest)
        returns (CapabilityResponse);

    rpc QueryCapabilities(QueryRequest)
        returns (stream CapabilityMatch);

    rpc GetCapability(GetCapabilityRequest)
        returns (CapabilityDocument);
}

// 메시지 서비스
service LLMMessageService {
    rpc SendMessage(MessageRequest)
        returns (MessageResponse);

    rpc StreamMessage(MessageRequest)
        returns (stream StreamChunk);

    rpc BidirectionalStream(stream MessageRequest)
        returns (stream MessageResponse);
}

// 연합 서비스
service LLMFederationService {
    rpc CreateFederation(FederationRequest)
        returns (FederationResponse);

    rpc JoinFederation(JoinRequest)
        returns (JoinResponse);

    rpc SubmitTask(TaskRequest)
        returns (stream TaskProgress);

    rpc InitiateConsensus(ConsensusRequest)
        returns (ConsensusResult);
}

메시지 정의

message CapabilityDocument {
    string id = 1;
    ModelIdentity identity = 2;
    Capability capability = 3;
    Constraints constraints = 4;
    int64 timestamp = 5;
    bytes signature = 6;
}

message ModelIdentity {
    string did = 1;
    string name = 2;
    string provider = 3;
    string version = 4;
}

message Capability {
    int32 level = 1;
    repeated DomainExpertise domains = 2;
    repeated LanguageSupport languages = 3;
    repeated string modalities = 4;
    repeated ToolCapability tools = 5;
}

message MessageEnvelope {
    string id = 1;
    MessageHeader header = 2;
    bytes payload = 3;
    RoutingInfo routing = 4;
    SecurityInfo security = 5;
}

message TaskDocument {
    string id = 1;
    string federation_id = 2;
    string description = 3;
    Priority priority = 4;
    DecompositionConfig decomposition = 5;
    repeated Subtask subtasks = 6;
}

6.5 보안 명세

DID 인증

인증 헤더 형식:
Authorization: DID-Auth {signed-challenge}

챌린지-응답 흐름:
1. 클라이언트가 챌린지 요청
   GET /auth/challenge?did={did}

2. 서버가 챌린지 반환
   { "challenge": "random-string", "expiresIn": 300 }

3. 클라이언트가 DID 개인키로 챌린지 서명
   signature = Sign(challenge, privateKey)

4. 클라이언트가 서명된 챌린지 전송
   POST /auth/verify
   { "did": "{did}", "challenge": "...", "signature": "..." }

5. 서버가 검증하고 토큰 반환
   { "token": "{jwt}", "expiresIn": 3600 }

메시지 서명

서명 형식:
{
    "algorithm": "Ed25519",
    "publicKey": "did:wia:llm:...#key-1",
    "value": "base64-encoded-signature"
}

서명 과정:
1. 메시지를 정규 JSON으로 직렬화
2. SHA-256 해시 계산
3. Ed25519 개인키로 해시 서명
4. 서명을 base64로 인코딩

암호화

암호화 형식:
{
    "algorithm": "AES-256-GCM",
    "keyAgreement": "X25519",
    "recipientKey": "did:wia:llm:...#encryption-key-1",
    "ephemeralKey": "base64-public-key",
    "nonce": "base64-nonce",
    "ciphertext": "base64-encrypted-payload"
}

암호화 과정:
1. 임시 X25519 키 쌍 생성
2. 수신자 공개키로 공유 비밀 도출
3. HKDF를 사용하여 AES 키 도출
4. AES-256-GCM으로 페이로드 암호화
5. 임시 공개키와 nonce 포함

6.7 속도 제한

속도 제한 헤더:

X-RateLimit-Remaining: 95
X-RateLimit-Reset: 1705312200

기본 제한:

- 능력 쿼리: 100/분
- 메시지: 60/분
- 연합 작업: 30/분
