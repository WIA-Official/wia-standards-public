제3장: AI간 메시징 (WIA-LLM-MESSAGE)

"AI 모델들이 협력하려면, 공통의 언어로 말해야 합니다."

3.1 AI 메시징의 필요성

AI 모델들이 서로의 능력을 알게 되면, 통신하는 방법이 필요합니다. WIA-LLM-MESSAGE 표준은 AI 모델들이 정보를 교환하는 방법을 정의합니다:

  • 다른 AI 모델에 질의 전송
  • 하위 작업 위임
  • 결과 검증 요청
  • 실시간 응답 스트리밍
  • 오류 우아하게 처리

3.2 메시지 봉투 구조

모든 메시지는 표준화된 봉투로 감싸집니다:

MessageEnvelope
├── @context           # JSON-LD 컨텍스트
├── id                 # 고유 메시지 ID
├── type               # "MessageEnvelope"
├── version            # 프로토콜 버전
├── header             # 라우팅 및 메타데이터
├── payload            # 메시지 내용
├── routing            # 전달 지시
└── security           # 암호화 및 서명

3.3 메시지 헤더

헤더는 필수 라우팅 정보를 포함합니다:

header: {
    from: "did:wia:llm:anthropic:claude-3-opus",
    to: "did:wia:llm:openai:gpt-4",
    timestamp: "2024-01-15T10:30:00Z",
    messageType: "QUERY",
    correlationId: "corr-abc123",
    replyTo: "did:wia:llm:anthropic:claude-3-opus",
    ttl: 300,
    priority: "NORMAL"
}

헤더 필드

필드타입설명
fromDID발신자 식별자
toDID수신자 식별자
timestampISO 8601메시지 생성 시간
messageTypeenum메시지 유형
correlationIdstring관련 메시지 연결
replyToDID응답 보낼 곳
ttlnumber유효 시간(초)
priorityenum메시지 우선순위

3.4 메시지 유형

표준은 여러 메시지 유형을 정의합니다:

유형 목적 예상 응답
QUERY 질문하거나 정보 요청 RESPONSE
DELEGATE 다른 AI에 하위 작업 할당 RESPONSE 또는 STREAM
COLLABORATE 작업 협력 요청 ACCEPT/REJECT
VERIFY 결과 검증 요청 VERIFICATION
RESPONSE 질의나 위임에 대한 응답 없음 (최종)
STREAM 스트리밍 응답 청크 더 많은 STREAM 또는 END
ERROR 오류 상태 보고 없음 (최종)

3.5 질의 페이로드

질의 메시지는 정보나 처리를 요청합니다:

{
    "header": {
        "from": "did:wia:llm:anthropic:claude-3-opus",
        "to": "did:wia:llm:local:codellama-34b",
        "messageType": "QUERY"
    },
    "payload": {
        "type": "query_payload",
        "content": "이 코드의 보안 취약점을 분석해주세요",
        "context": {
            "code": "function login(user, pass) { ... }",
            "language": "javascript"
        },
        "parameters": {
            "maxTokens": 2048,
            "temperature": 0.3,
            "responseFormat": "structured"
        },
        "constraints": {
            "timeout": 30000,
            "requiredFields": ["vulnerabilities", "severity"]
        }
    }
}

3.6 응답 페이로드

응답 메시지는 결과를 포함합니다:

{
    "header": {
        "from": "did:wia:llm:local:codellama-34b",
        "to": "did:wia:llm:anthropic:claude-3-opus",
        "messageType": "RESPONSE",
        "correlationId": "corr-abc123"
    },
    "payload": {
        "type": "response_payload",
        "status": "SUCCESS",
        "content": {
            "vulnerabilities": [
                {
                    "type": "SQL_INJECTION",
                    "severity": "HIGH",
                    "line": 15,
                    "description": "사용자 입력이 살균되지 않음"
                }
            ],
            "summary": "1개의 고위험 취약점 발견"
        },
        "metadata": {
            "processingTime": 1250,
            "tokensUsed": 856,
            "confidence": 0.92
        }
    }
}

3.7 위임 워크플로우

AI가 다른 AI에 하위 작업을 위임할 때:

단계 1: 위임하는 AI가 DELEGATE 메시지 전송
+------------------+                    +------------------+
|   Claude 3       |  --- DELEGATE -->  |   Code Llama     |
|   (조정자)       |                    |   (전문가)       |
+------------------+                    +------------------+

단계 2: 전문가가 처리하고 응답
+------------------+                    +------------------+
|   Claude 3       |  <-- RESPONSE ---  |   Code Llama     |
|                  |                    |                  |
+------------------+                    +------------------+

단계 3: 조정자가 결과로 계속 진행

위임 메시지 예시

{
    "header": {
        "messageType": "DELEGATE"
    },
    "payload": {
        "type": "delegate_payload",
        "task": "Python 코드의 타입 오류 분석",
        "input": {
            "code": "...",
            "requirements": ["mypy", "strict"]
        },
        "expectedOutput": {
            "format": "json",
            "schema": "TypeErrorReport"
        },
        "deadline": "2024-01-15T10:35:00Z"
    }
}

3.8 스트리밍 응답

긴 응답의 경우, 스트리밍이 콘텐츠를 점진적으로 전달합니다:

스트림 메시지 1:
{
    "header": { "messageType": "STREAM" },
    "payload": {
        "sequenceNumber": 1,
        "chunk": "분석 결과 ",
        "done": false
    }
}

스트림 메시지 2:
{
    "header": { "messageType": "STREAM" },
    "payload": {
        "sequenceNumber": 2,
        "chunk": "여러 보안 문제가 발견되었습니다...",
        "done": false
    }
}

스트림 메시지 3 (최종):
{
    "header": { "messageType": "STREAM" },
    "payload": {
        "sequenceNumber": 3,
        "chunk": "",
        "done": true,
        "totalTokens": 1523
    }
}

3.9 오류 처리

오류 발생 시, ERROR 메시지 유형을 사용합니다:

{
    "header": {
        "messageType": "ERROR",
        "correlationId": "corr-abc123"
    },
    "payload": {
        "type": "error_payload",
        "code": "CAPABILITY_MISMATCH",
        "message": "작업에 MEDICAL 도메인 전문성이 필요합니다",
        "details": {
            "required": ["MEDICAL"],
            "available": ["CODE", "MATH"]
        },
        "recoverable": true,
        "suggestion": "MEDICAL 능력이 있는 모델로 라우팅"
    }
}

표준 오류 코드

코드설명복구 가능
CAPABILITY_MISMATCH모델에 필요한 능력 없음예
RATE_LIMITED너무 많은 요청예
CONTEXT_OVERFLOW입력이 컨텍스트 윈도우 초과예
TIMEOUT처리 시간 초과예
CONTENT_POLICY요청이 콘텐츠 정책 위반아니오
AUTHENTICATION_FAILED잘못된 자격 증명예
INTERNAL_ERROR예기치 않은 오류아마도

3.10 메시지 라우팅

메시지는 라우팅 지시를 포함합니다:

routing: {
    strategy: "CAPABILITY_BASED",
    hops: [],
    maxHops: 3,
    fallback: {
        strategy: "BROADCAST",
        scope: "FEDERATION"
    }
}

라우팅 전략

전략설명
DIRECT지정된 수신자에게 직접 전송
CAPABILITY_BASED능력 매칭 기반 라우팅
LOAD_BALANCED사용 가능한 모델에 분배
BROADCAST모든 연합 멤버에게 전송
