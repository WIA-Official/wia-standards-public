# 챕터 08 · Stable Diffusion · SDXL · Flux.1 · SD3.5 비교

## 학습 목표
- SD 계열 주요 모델(SDXL, Flux.1, SD3.5)의 아키텍처 차이를 설명할 수 있다
- 고문서 복원 프로젝트에 적합한 모델을 선택하는 기준을 세울 수 있다
- 각 모델의 파인튜닝 가능 여부와 VRAM 요구사항을 비교할 수 있다

## 이론

### 모델 계보 한 장 정리

확산 모델 실용화의 역사는 2022년 **Stable Diffusion 1.x** (Stability AI + CompVis) 공개로 시작됐다. 이후 2년간 아키텍처가 급격히 진화했으며, 현재 1인 개발자가 실무에서 다루는 주요 모델은 다음 4개다.

```
SD 1.5 (2022) → SDXL 1.0 (2023.07) → SD3.5 (2024.10) → Flux.1 (2024.08)
```

### SD 1.5 — 레거시이지만 여전히 유용

- **해상도**: 512×512 기본 (768×768까지 가능)
- **VRAM**: 4GB 이상 (FP16 기준)
- **특징**: LoRA 에코시스템이 가장 넓다. Civitai에 수십만 개 커뮤니티 모델 존재
- **파인튜닝**: 매우 성숙. DreamBooth, LoRA, Textual Inversion 전부 지원
- **고문서 복원**: 해상도 낮아 세밀한 문자 복원에 부적합. 사전 실험용으로만 활용

### SDXL 1.0 — 현재 기준 안정된 선택

- **해상도**: 1024×1024 기본 (최대 2048×2048)
- **VRAM**: 8GB 이상 (FP16). Base + Refiner 합치면 16GB 권장
- **구조**: Base 모델 + Refiner 모델 2단계 구조. 고해상도 디테일 개선
- **파인튜닝**: LoRA/QLoRA/DreamBooth 전부 지원. 학습 레시피 풍부
- **Inpainting**: `stabilityai/stable-diffusion-xl-inpaint` 공식 체크포인트 존재
- **고문서 복원**: **현재 가장 권장**. 1024px 해상도에서 문자 디테일 보존 우수

### Flux.1 — 2024년 신세대 모델

- **개발사**: Black Forest Labs (Stable Diffusion 핵심 개발진 독립)
- **구조**: Diffusion Transformer (DiT) 기반. 기존 U-Net 아키텍처 탈피
- **버전**:
  - `flux1-dev`: 비상업 연구용. 12B 파라미터
  - `flux1-schnell`: 4스텝 고속 생성. Apache 2.0 라이선스
- **VRAM**: FP8 양자화로 16GB GPU에서 실행 가능. 풀 정밀도는 24GB+
- **파인튜닝**: LoRA 지원. 단, SDXL 대비 레시피가 아직 적음
- **특징**: 텍스트 렌더링 능력 획기적 향상. 한글 생성도 시도 가능

### SD3.5 — 2024 최신 안정 모델

- **구조**: Multimodal Diffusion Transformer (MMDiT). CLIP + T5 인코더 혼합
- **버전**: Medium (2.5B), Large (8B), Large-Turbo (8B, 4스텝)
- **VRAM**: Medium 기준 6GB, Large 기준 16GB
- **파인튜닝**: Hugging Face diffusers로 LoRA 지원. 아직 에코시스템 성장 중
- **특징**: 지시문 따르기(instruction following)가 기존 대비 2~3배 향상

### 고문서 복원 모델 선택 기준

| 기준 | SD 1.5 | SDXL | Flux.1-dev | SD3.5-Medium |
|---|---|---|---|---|
| Inpainting 공식 지원 | ✅ | ✅ | 🔧 커뮤니티 | 🔧 커뮤니티 |
| 1024px 문자 선명도 | ❌ | ✅ | ✅ | ✅ |
| 16GB GPU 실행 | ✅ | ✅ | ✅ (FP8) | ✅ |
| LoRA 레시피 성숙도 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ |
| 추천 단계 | 사전 실험 | **Phase 1 실전** | Phase 2 고도화 | Phase 2 고도화 |

## 실전 코드

```python
# 모델별 파이프라인 로드 및 기본 Inpainting 비교
# 실행 환경: GPU 16GB 이상, diffusers>=0.28.0

import torch
from diffusers import (
    StableDiffusionInpaintPipeline,
    StableDiffusionXLInpaintPipeline,
)
from PIL import Image
import numpy as np


def load_sdxl_inpaint(device="cuda", dtype=torch.float16):
    """SDXL Inpainting 파이프라인 — 고문서 복원 1순위"""
    pipe = StableDiffusionXLInpaintPipeline.from_pretrained(
        "diffusers/stable-diffusion-xl-1.0-inpainting-0.1",
        torch_dtype=dtype,
    ).to(device)
    pipe.enable_attention_slicing()   # VRAM 절약
    pipe.enable_vae_slicing()         # 대형 이미지 처리
    return pipe


def load_sd15_inpaint(device="cuda", dtype=torch.float16):
    """SD 1.5 Inpainting — 사전 실험·저 VRAM 환경"""
    pipe = StableDiffusionInpaintPipeline.from_pretrained(
        "runwayml/stable-diffusion-inpainting",
        torch_dtype=dtype,
    ).to(device)
    return pipe


def create_damage_mask(image: Image.Image, threshold: int = 240) -> Image.Image:
    """
    간단한 손상 마스크 생성
    임계값 이상 밝기 픽셀 = 손상(흰색) 영역으로 마킹
    실제 고문서는 Otsu binarization 또는 SAM 사용 권장
    """
    gray = np.array(image.convert("L"))
    mask = (gray > threshold).astype(np.uint8) * 255
    return Image.fromarray(mask)


def inpaint_document(
    pipe,
    image_path: str,
    prompt: str = "historical Korean document, clear brushstroke text, aged paper texture, traditional ink",
    negative_prompt: str = "blurry, modern font, digital artifact, watermark",
    num_steps: int = 30,
    strength: float = 0.85,
    guidance_scale: float = 8.0,
    seed: int = 42,
) -> Image.Image:
    """
    고문서 이미지 복원 실행
    - strength: 0.7~0.9 범위 권장 (낮을수록 원본 보존)
    - guidance_scale: 7~9 범위 권장 (높을수록 프롬프트 엄격)
    """
    generator = torch.Generator("cuda").manual_seed(seed)
    
    image = Image.open(image_path).convert("RGB")
    
    # SDXL은 1024px 배수 선호
    w, h = image.size
    new_w = ((w + 63) // 64) * 64
    new_h = ((h + 63) // 64) * 64
    image = image.resize((new_w, new_h), Image.LANCZOS)
    
    # 손상 마스크 자동 생성 (실제론 수동 제작 또는 SAM 활용)
    mask = create_damage_mask(image)
    
    result = pipe(
        prompt=prompt,
        negative_prompt=negative_prompt,
        image=image,
        mask_image=mask,
        num_inference_steps=num_steps,
        strength=strength,
        guidance_scale=guidance_scale,
        generator=generator,
    ).images[0]
    
    return result


def compare_models_vram():
    """모델별 VRAM 사용량 출력 (GPU 있을 때만 정확)"""
    info = {
        "SD 1.5 Inpaint (FP16)":       "~4 GB",
        "SDXL Inpaint (FP16)":         "~8 GB",
        "SDXL Inpaint (BF16, VAE FP32)": "~10 GB",
        "Flux.1-dev (FP8)":            "~16 GB",
        "SD3.5 Medium (FP16)":         "~6 GB",
        "SD3.5 Large (FP16)":          "~16 GB",
    }
    print("모델별 최소 VRAM 가이드")
    print("-" * 40)
    for model, vram in info.items():
        print(f"{model:<35} {vram}")


if __name__ == "__main__":
    compare_models_vram()
    
    # SDXL 파이프라인 테스트
    # pipe = load_sdxl_inpaint()
    # result = inpaint_document(pipe, "sample_document.png")
    # result.save("restored_document.png")
    # print("복원 완료: restored_document.png")
```

## 함정 카드 — 자주 실수하는 5가지

1. **"SDXL Base와 Refiner를 항상 같이 써야 한다"** → Base만으로도 고품질 Inpainting 가능. Refiner는 최종 디테일 강화 단계. GPU VRAM이 부족하면 Base만 사용해도 충분하다.

2. **"Flux.1-dev를 상업적으로 사용한다"** → Flux.1-dev는 **비상업 라이선스**. 상업 제품에는 `flux1-schnell` (Apache 2.0) 또는 Flux Pro API를 써야 한다. 고문서 복원 서비스를 출시할 때 라이선스 확인 필수.

3. **"이미지 크기를 임의로 설정한다"** → SDXL은 64의 배수 해상도가 아니면 VAE 인코딩 오류 또는 품질 저하가 생긴다. 항상 `((w + 63) // 64) * 64`로 패딩.

4. **"다운로드 없이 파이프라인을 인스턴스화한다"** → `from_pretrained()`는 처음 실행 시 수 GB를 다운로드한다. 서버 환경에서는 먼저 `huggingface-cli download` 또는 `snapshot_download()`로 캐시 후 오프라인으로 실행.

5. **"SD3.5의 두 인코더를 무시한다"** → SD3.5는 CLIP + T5 두 텍스트 인코더를 사용한다. `prompt`(CLIP)와 `prompt_3`(T5) 파라미터를 모두 활용해야 최고 품질. T5 인코더만 메모리 6GB를 차지하므로 VRAM 관리 주의.

## 연습문제

**문제 1.** SDXL과 Flux.1-dev의 아키텍처 핵심 차이를 설명하라. U-Net과 DiT(Diffusion Transformer)의 구조적 차이는 무엇인가?

> **해설:** SDXL은 U-Net 기반 — 인코더-디코더 구조로 공간 피처를 계층적으로 처리. Flux.1은 DiT 기반 — 이미지를 패치로 분할 후 Transformer의 Self-Attention으로 처리. DiT는 텍스트-이미지 정렬이 더 강하고 확장성(파라미터 증가)이 유리하지만, 기존 LoRA 생태계가 U-Net 기준으로 쌓여 있어 현재는 SDXL이 파인튜닝 효율이 높다.

**문제 2.** VRAM 8GB GPU에서 SDXL Inpainting을 실행할 때 OOM(Out of Memory) 에러가 발생한다. 어떤 최적화를 순서대로 적용하겠는가?

> **해설:** ① `enable_attention_slicing()` — Attention을 슬라이스 단위로 처리. ② `enable_vae_slicing()` — VAE 디코딩을 분할 처리. ③ 입력 이미지를 768×768로 줄임. ④ `torch_dtype=torch.bfloat16` 적용. ⑤ 여전히 OOM이면 `pipe.enable_model_cpu_offload()`로 CPU-GPU 메모리 오프로드.

**문제 3.** 고문서 복원 서비스를 상업적으로 출시하려 할 때, Flux.1-dev, Flux.1-schnell, SDXL 중 어느 것을 선택하겠는가? 라이선스와 성능을 모두 고려하여 판단하라.

> **해설:** `Flux.1-schnell` (Apache 2.0) 또는 SDXL (CreativeML Open RAIL-M, 상업 허용)이 적합. Flux.1-dev는 비상업 전용으로 배제. 성능 측면에서 Flux.1-schnell은 4스텝 고속 생성으로 서버 비용 유리. SDXL은 Inpainting 체크포인트가 공식 지원되어 파인튜닝 후 고문서 특화가 용이. **권장: Phase 1은 SDXL, Phase 2 고도화 시 Flux.1-schnell 도입 검토**.
