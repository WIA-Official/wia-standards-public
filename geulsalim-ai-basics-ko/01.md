# 챕터 01 · 왜 1인이 파운데이션 모델을 만들 수 있는가

## 학습 목표
- AI 챔피언 대회 수준의 파운데이션 모델을 1인이 만들 수 있는 이유를 설명할 수 있다
- 오픈소스 생태계와 API 조합 전략의 차이를 이해한다
- 이 책 3권의 학습 로드맵 전체를 머릿속에 그릴 수 있다

## 이론

### "처음부터 만든다"는 착각

2022년 GPT-3가 공개됐을 때, 많은 사람이 이렇게 생각했다.

> "저걸 만들려면 수백억 원짜리 GPU 클러스터와 수백 명의 엔지니어가 필요하겠구나."

맞다. GPT-3를 **처음부터** 만들려면 그렇다. 하지만 지금 우리가 이야기하는 건 다른 길이다.

2024년 이후의 AI 개발 생태계는 완전히 바뀌었다. Stability AI가 Stable Diffusion을 오픈소스로 공개했고, Meta가 Llama 3.1을, Mistral이 Mixtral을 공개했다. 수천억 원을 들여 학습시킨 모델의 **가중치**가 무료로 풀려 있다.

이 가중치를 **내 목적에 맞게 조금만 조정**하는 것, 그게 파인튜닝이다. 그리고 그것은 A100 GPU 1장으로, 2~3주 안에, 1인이 할 수 있다.

### 홍익인간과 오픈소스

우리가 이 책을 쓰는 철학적 뿌리는 홍익인간(弘益人間)이다. **널리 인간을 이롭게 하라.**

오픈소스는 그 현대적 실천이다. 수천 명의 연구자가 수년간 쌓아 올린 지식을 공개하고, 다음 사람이 그 위에 올라서서 더 높은 곳을 바라볼 수 있게 한다. 뉴턴이 말했던 "거인의 어깨 위에 서서"가 AI 시대에는 오픈소스의 형태로 실현된다.

이 책은 그 어깨 위에 서는 방법을 가르친다.

### 1인 개발자의 현실적 전략

처음 AI 개발을 시작하는 1인 개발자에게 권하는 전략은 세 층위로 구성된다.

**1층: 오픈소스 파운데이션 모델 활용**
Stable Diffusion XL, Flux.1, Llama 3.1, Qwen2 등 이미 검증된 모델을 기반으로 삼는다. 이들은 수개월, 수억 원의 학습 비용이 들어간 결과물이다. 무료다.

**2층: API 조합**
Claude, GPT-4V, Gemini Vision 같은 유료 API는 강력한 보조 엔진이다. GPU 없이 호출만으로 이미지 설명, 텍스트 교정, 품질 평가를 처리할 수 있다. 월 수십 달러로 수억 원짜리 연구소 기능을 빌려 쓰는 셈이다.

**3층: 핵심 파인튜닝 1개**
내 도메인에 특화된 파인튜닝은 딱 한 곳에만 집중한다. 이미지 복원이면 SDXL Inpainting. 한국어 OCR이면 Tesseract + LLM 후처리. 모든 것을 학습시키려 하면 6개월 만에 연구비가 바닥난다.

### Claude Code의 등장

이 전략이 실현 가능해진 또 하나의 이유가 있다. Claude Code다.

2025년 이전까지 1인 개발자의 최대 병목은 **코드 작성 속도**였다. 파이프라인 하나 구축하는 데 일주일이 걸렸다. 이제는 자연어로 의도를 설명하면 Claude Code가 실행 가능한 코드를 즉시 생성한다. 디버깅, 아키텍처 설계, 문서화까지.

1인 개발자의 생산성이 10인 팀에 근접하는 시대다.

## 실전 코드

이 챕터의 코드는 환경 확인부터 시작한다. 이 책 전체를 따라가기 위한 최소 요구사항을 점검한다.

```python
# 환경 점검 스크립트 — 책 전체 실습 전에 먼저 실행
import sys
import subprocess

print(f"Python 버전: {sys.version}")

# 필수 패키지 설치 여부 확인
required = ["torch", "transformers", "diffusers", "datasets", "accelerate"]
missing = []

for pkg in required:
    try:
        __import__(pkg)
        print(f"  ✅ {pkg}")
    except ImportError:
        print(f"  ❌ {pkg} — 미설치")
        missing.append(pkg)

if missing:
    print("\n누락된 패키지 설치:")
    print(f"  pip install {' '.join(missing)}")
else:
    print("\n모든 패키지 준비 완료!")

# GPU 확인
try:
    import torch
    if torch.cuda.is_available():
        gpu = torch.cuda.get_device_name(0)
        vram = torch.cuda.get_device_properties(0).total_memory / 1e9
        print(f"\nGPU: {gpu}")
        print(f"VRAM: {vram:.1f} GB")
    else:
        print("\nGPU 없음 — CPU 모드 (학습은 클라우드 GPU 필요)")
except ImportError:
    pass
```

**실행 방법:**
```bash
pip install torch transformers diffusers datasets accelerate
python check_env.py
```

## 함정 카드 — 자주 실수하는 5가지

1. **"GPU가 없으면 시작도 못 한다"** → 거짓. 이론 학습, 코드 작성, 데이터 준비는 CPU로 충분하다. GPU는 실제 학습 때만 필요하고, 그때는 클라우드를 시간 단위로 빌리면 된다.

2. **"파이썬을 완벽히 알아야 한다"** → 기초면 충분하다. 함수 정의, 리스트, 딕셔너리, for문. 나머지는 Claude Code가 채워준다.

3. **"최신 모델을 써야 한다"** → 파인튜닝 목적이라면 검증된 구버전이 오히려 낫다. 최신 모델은 아직 파인튜닝 레시피가 없는 경우가 많다.

4. **"처음부터 직접 만들어야 진짜다"** → 오픈소스를 활용하는 것이 현재 AI 업계의 표준이다. OpenAI도 오픈소스 모델을 내부적으로 활용한다.

5. **"혼자서는 한계가 있다"** → 이 책을 쓴 사람은 Rust 언어를 전혀 몰랐다가 책 6권을 직접 썼다. Claude Code라는 파트너가 있다면 1인의 한계는 생각보다 훨씬 멀리 있다.

## 연습문제

**문제 1.** 오픈소스 파운데이션 모델 활용 전략의 핵심 3층 구조를 자신의 말로 설명하라. 각 층이 담당하는 역할을 명확히 구분할 것.

> **해설:** 1층은 기반(오픈소스 가중치), 2층은 보조 지능(API), 3층은 도메인 특화(파인튜닝 1개)다. 핵심은 "처음부터 만들지 않는다"는 원칙이다. 기존 자산 위에 내 필요만큼만 쌓는다.

**문제 2.** Lambda Labs, AWS g5, Paperspace, Vast.ai 중 비용이 가장 저렴한 옵션을 선택해야 한다면 어느 것을 선택하겠는가? 그 이유는? (정해진 정답 없음, 판단 근거를 서술)

> **해설:** 일반적으로 Vast.ai가 가장 저렴하나 신뢰성이 낮을 수 있다. 안정성 vs 비용의 트레이드오프를 이해하는 것이 목적이다. 중요한 학습에는 Lambda Labs나 AWS, 실험적 탐색에는 Vast.ai를 추천하는 것이 합리적이다.

**문제 3.** "파인튜닝은 딱 한 곳에만 집중하라"는 원칙이 왜 중요한가? 여러 도메인을 동시에 파인튜닝하려 할 때 발생하는 현실적 문제를 2가지 이상 서술하라.

> **해설:** ① 연구비(GPU 비용) 분산으로 어느 것도 완성하지 못할 수 있다 ② 각 도메인마다 데이터 수집·정제 시간이 따로 필요하다 ③ 파인튜닝된 모델 간 품질 비교 검증에 추가 시간이 소요된다. 선택과 집중이 1인 개발자의 생존 전략이다.
