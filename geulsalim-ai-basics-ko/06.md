## 이론

### 텍스트를 숫자로 만드는 여정

컴퓨터는 텍스트를 직접 이해하지 못한다. "고문서 복원"이라는 글자는 GPU에게는 의미 없다. 이 글자들을 숫자 배열로 바꾸는 작업이 토큰화(Tokenization)다.

```
"고문서 복원" → ["고문서", "복원"] → [15234, 8821] → 임베딩 벡터
```

이 과정이 얼마나 잘 되느냐가 모델 성능의 출발점을 결정한다.

### 세 가지 토크나이저

**BPE (Byte Pair Encoding)**: GPT 계열이 사용. 자주 함께 등장하는 바이트 쌍을 반복적으로 합친다. 한국어 처리 시 한 글자가 여러 토큰으로 쪼개질 수 있다.

**WordPiece**: BERT가 사용. "##" 접두사로 서브워드를 표시한다. `"먹었다"` → `["먹", "##었", "##다"]`

**SentencePiece**: LLaMA, T5가 사용. 언어에 독립적인 바이트 기반 처리. 한국어, 한자 등 비라틴 문자에 강하다.

### 임베딩 — 의미를 공간에 배치하기

토큰 ID는 단순한 숫자다. 15234와 15235가 비슷하다고 해서 그 단어가 비슷한 의미를 가지는 게 아니다.

임베딩은 이 숫자를 **고차원 벡터**(보통 768~4096차원)로 변환한다. 이 벡터 공간에서 의미가 비슷한 단어는 **거리가 가깝다**.

유명한 예시:
```
벡터("왕") - 벡터("남성") + 벡터("여성") ≈ 벡터("여왕")
```

이 임베딩이 학습을 통해 만들어진다는 것이 딥러닝의 핵심이다. 처음에는 랜덤하게 초기화되고, 학습 데이터를 보면서 의미 있는 위치로 조정된다.

## 실전 코드

```python
# 토크나이저와 임베딩 실전 탐구
from transformers import AutoTokenizer, AutoModel
import torch
import numpy as np

# ── 1. 토크나이저 비교 실험 ────────────────────────────
def compare_tokenizers():
    """동일 텍스트를 여러 토크나이저로 처리 비교"""
    
    models = {
        "GPT-2 (BPE)":       "gpt2",
        "BERT (WordPiece)":   "bert-base-multilingual-cased",
        "LLaMA (SentencePiece)": "huggyllama/llama-7b",  # 또는 오픈 모델
    }
    
    texts = [
        "안녕하세요, 고문서를 복원합니다.",
        "Hello, I restore historical documents.",
    ]
    
    for model_name, model_id in models.items():
        try:
            tokenizer = AutoTokenizer.from_pretrained(model_id)
            print(f"\n=== {model_name} ===")
            
            for text in texts:
                tokens = tokenizer.tokenize(text)
                ids = tokenizer.encode(text)
                print(f"  텍스트: {text}")
                print(f"  토큰: {tokens}")
                print(f"  ID: {ids}")
                print(f"  토큰 수: {len(tokens)}")
        except Exception as e:
            print(f"  로드 실패: {e}")


# ── 2. 실용적 토크나이저 사용법 ──────────────────────
def practical_tokenizer(model_name: str = "klue/bert-base"):
    """
    KLUE BERT: 한국어 특화 모델
    실제 파인튜닝에 사용할 토크나이저 패턴
    """
    tokenizer = AutoTokenizer.from_pretrained(model_name)
    
    texts = [
        "조선 왕조 실록의 한자를 현대 한국어로 번역한다.",
        "국립중앙도서관에서 공개한 고문서를 OCR로 처리한다.",
    ]
    
    # 배치 처리 (padding + truncation)
    encoded = tokenizer(
        texts,
        padding=True,           # 짧은 문장 패딩
        truncation=True,        # 최대 길이 초과 시 자르기
        max_length=128,
        return_tensors="pt"     # PyTorch 텐서 반환
    )
    
    print(f"입력 ID 형태: {encoded['input_ids'].shape}")
    print(f"어텐션 마스크: {encoded['attention_mask'].shape}")
    
    # 디코딩 확인
    for i, ids in enumerate(encoded['input_ids']):
        decoded = tokenizer.decode(ids, skip_special_tokens=True)
        print(f"\n문장 {i+1}: {decoded}")
    
    return encoded


# ── 3. 임베딩 추출 및 유사도 계산 ────────────────────
def get_embeddings(texts: list, model_name: str = "klue/bert-base"):
    """
    문장 임베딩 추출 (CLS 토큰 사용)
    문서 유사도 검색, 클러스터링에 활용
    """
    tokenizer = AutoTokenizer.from_pretrained(model_name)
    model = AutoModel.from_pretrained(model_name)
    model.eval()  # 추론 모드
    
    embeddings = []
    
    with torch.no_grad():  # 그래디언트 계산 불필요 (추론만)
        for text in texts:
            inputs = tokenizer(text, return_tensors="pt", 
                             padding=True, truncation=True, max_length=512)
            outputs = model(**inputs)
            
            # CLS 토큰의 hidden state = 문장 전체 표현
            cls_embedding = outputs.last_hidden_state[:, 0, :]  # [1, 768]
            embeddings.append(cls_embedding.squeeze().numpy())
    
    return np.array(embeddings)  # [n_texts, 768]


def cosine_similarity(a: np.ndarray, b: np.ndarray) -> float:
    """코사인 유사도: 방향이 같으면 1, 반대면 -1"""
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))


def find_similar_documents():
    """임베딩으로 유사 고문서 찾기 예시"""
    
    documents = [
        "조선 태조가 한양에 도읍을 정하다",           # 역사 문서
        "고려 말 왜구의 침입이 잦아지다",             # 역사 문서
        "세종대왕이 훈민정음을 창제하다",              # 역사 문서
        "파이썬으로 머신러닝 모델을 학습한다",         # IT 문서
        "GPU 클라우드에서 파인튜닝을 실행한다",        # IT 문서
    ]
    
    query = "조선 시대 문자 창제에 관한 내용"
    
    print(f"질의: {query}\n")
    
    # 임베딩 계산 (실제 환경에서 실행)
    # embeddings = get_embeddings(documents + [query])
    # query_emb = embeddings[-1]
    # doc_embs = embeddings[:-1]
    
    # 임시 데모 (랜덤 임베딩)
    np.random.seed(42)
    doc_embs = np.random.randn(len(documents), 768)
    query_emb = np.random.randn(768)
    
    similarities = [cosine_similarity(query_emb, d) for d in doc_embs]
    
    ranked = sorted(zip(similarities, documents), reverse=True)
    
    print("유사도 순위:")
    for score, doc in ranked:
        print(f"  {score:.4f} — {doc}")


if __name__ == "__main__":
    practical_tokenizer()
    find_similar_documents()
```

## 함정 카드 — 자주 실수하는 5가지

1. **"토큰 수 = 글자 수"** → 전혀 다르다. 한국어는 특히 1글자가 2~3토큰으로 쪼개질 수 있다. GPT-4 기준 한국어 텍스트는 영어 대비 약 2~3배 토큰을 더 소모한다. API 비용 계산 시 반드시 고려해야 한다.

2. **"train 시 `model.train()`, 추론 시 `model.eval()` 빠뜨린다"** → `eval()` 모드에서는 Dropout과 BatchNorm이 비활성화된다. 빠뜨리면 추론 결과가 매번 달라지고 재현성이 없다.

3. **"CLS 토큰과 평균 풀링을 혼동한다"** → BERT 스타일은 CLS 토큰을 문장 표현으로 사용. 하지만 Sentence-BERT 등 최신 임베딩 모델은 평균 풀링(모든 토큰 평균)이 더 정확하다. 사용하는 모델의 권장 방법을 확인해야 한다.

4. **"[PAD] 토큰을 어텐션에 포함시킨다"** → padding으로 추가된 [PAD] 토큰은 어텐션 마스크로 0 처리해야 한다. `attention_mask` 파라미터를 생략하면 패딩 토큰도 어텐션 계산에 포함되어 결과가 오염된다.

5. **"한국어 모델에 영어 모델을 쓴다"** → `bert-base-uncased`는 영어 전용이다. 한국어 처리에는 `klue/bert-base`, `snunlp/KR-ELECTRA-discriminator`, `beomi/kcbert-base` 중 목적에 맞는 것을 선택해야 한다.

## 연습문제

**문제 1.** `tokenizer.encode("안녕")` 과 `tokenizer.encode("安寧")` 의 결과가 다른 이유를 토크나이저 관점에서 설명하라. (한글과 한자 처리 차이)

> **해설:** 대부분의 토크나이저는 유니코드 코드포인트 기반으로 처리한다. "안녕"(한글 유니코드 U+C548, U+B155)과 "安寧"(한자 유니코드 U+5B89, U+5BF5)은 서로 다른 코드포인트다. 학습 데이터에 한자가 적은 모델은 한자를 더 작은 단위(바이트)로 쪼개어 처리한다.

**문제 2.** `get_embeddings()` 함수에서 `with torch.no_grad():` 블록을 제거하면 어떤 문제가 생기는가?

> **해설:** 그래디언트 계산 그래프가 생성되어 메모리 사용량이 급증한다. 추론 시에는 역전파가 필요 없으므로 `no_grad()`가 필수다. 대용량 문서 배치 처리 시 OOM이 발생할 수 있다.

**문제 3.** 코사인 유사도가 0.95인 두 문서와 0.3인 두 문서가 있다. 고문서 검색 시스템에서 어떤 임계값(threshold)을 사용하는 것이 적절한가? 너무 높거나 낮을 때의 문제를 각각 설명하라.

> **해설:** 보통 0.7~0.85 사이가 실용적이다. 너무 높으면(0.95+) 실질적으로 동일한 문서만 검색되어 다양성이 없다. 너무 낮으면(0.3 이하) 관련 없는 문서가 대거 포함된다. 도메인과 데이터에 따라 validation set으로 최적값을 찾는다.
