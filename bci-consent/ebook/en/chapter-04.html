4.2 Permission Types

The protocol defines 18 standardized permission types covering all potential uses of neural data...

  • dataCollection: Permission to collect neural signals
  • dataStorage: Permission to store collected data beyond session
  • realTimeProcessing: Process data in real-time for BCI control
  • offlineAnalysis: Analyze historical data
  • research: Use data in research studies
  • thirdPartySharing: Share data with external entities
  • aiTraining: Use data to train machine learning models
  • commercialUse: Use data for commercial purposes
  • dataLinking: Link neural data with other data sources
  • inferenceProcessing: Derive information beyond explicit measurements
  • longTermRetention: Keep data beyond minimum required period
  • cloudStorage: Store data in cloud (vs. local only)
  • crossBorderTransfer: Transfer data across national borders
  • publicationRights: Publish anonymized data or results
  • regulatorySharing: Share with regulatory authorities
  • emergencyAccess: Allow access in emergency situations
  • geneticLinking: Link neural data with genetic information
  • biometricUse: Use neural patterns for identification

4.3 Cryptographic Signatures

All consent records must be cryptographically signed to ensure authenticity and detect tampering. The protocol supports multiple signature algorithms...

4.3.1 Supported Algorithms

  • RSA-2048: Widely supported, good for most use cases
  • RSA-4096: Enhanced security for high-risk applications
  • ECDSA P-256: Elliptic curve, smaller signatures
  • Ed25519: Modern, fast, secure

4.3.2 Signature Verification Process

function verifyConsentSignature(consentRecord, signature, publicKey) {
    // 1. Extract all fields except signature
    const dataToVerify = { ...consentRecord };
    delete dataToVerify.signature;
    
    // 2. Canonicalize JSON (deterministic ordering)
    const canonical = JSON.stringify(dataToVerify, Object.keys(dataToVerify).sort());
    
    // 3. Hash the canonical representation
    const hash = sha256(canonical);
    
    // 4. Verify signature
    return crypto.verify(hash, signature, publicKey);
}

4.5 Validation Rules

The protocol includes comprehensive validation rules to ensure data integrity...

  • All required fields must be present and non-null
  • Date fields must be valid ISO 8601 timestamps
  • validUntil must be after consent.date
  • Capacity score must be 0-100
  • Jurisdiction must be recognized ISO 3166-1 alpha-2 code
  • Signature hash must match computed hash of record
  • Subject ID must be unique within organization
  • Consent ID must be globally unique (UUID recommended)

4.8 Data Format Evolution

The consent data format will evolve over time. The protocol uses semantic versioning to manage changes...

4.8.1 Versioning Scheme

  • Major Version (1.x.x): Breaking changes incompatible with previous versions
  • Minor Version (x.1.x): New features added in backwards-compatible manner
  • Patch Version (x.x.1): Backwards-compatible bug fixes

4.8.2 Backwards Compatibility

When adding new fields in minor versions:

  • New fields must be optional, not required
  • Existing field meanings must not change
  • Validators must ignore unknown fields in newer versions
  • Older clients can still process newer records (ignoring new fields)

4.8.3 Migration Strategies

When major version changes require migration:

  • Provide migration tools to convert old format to new
  • Support both formats during transition period (6-12 months)
  • Document all breaking changes clearly
  • Provide upgrade path for implementations
