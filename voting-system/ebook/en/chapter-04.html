CHAPTER 04

Phase 1: Ballot Data Format

← Previous Chapter Table of Contents Next Chapter →

The Foundation of Interoperability

Phase 1 of WIA-SOC-015 establishes the fundamental data structures that enable different voting systems to exchange information seamlessly. By standardizing how ballots, candidates, contests, and results are represented, the standard creates a common language for electoral technology regardless of vendor, jurisdiction, or implementation approach.

This chapter explores the ballot data format specification in detail, examining JSON schemas, field definitions, validation rules, and practical implementation considerations that form the foundation of WIA-SOC-015 compliance.

Core Data Model

Ballot Structure

The WIA-SOC-015 ballot data format uses JSON (JavaScript Object Notation) as its primary encoding format due to JSON's widespread support, human readability, and robust tooling ecosystem. The core ballot structure contains several key components:

{
  "ballotId": "BALLOT-2025-PRES-001",
  "version": "WIA-SOC-015-v1.0",
  "electionId": "ELECTION-2025-001",
  "electionName": {
    "en": "Presidential Election 2025",
    "ko": "2025년 대통령 선거",
    "es": "Elección Presidencial 2025"
  },
  "jurisdiction": {
    "country": "USA",
    "state": "California",
    "district": "District 12"
  },
  "metadata": {
    "created": "2025-01-15T10:00:00Z",
    "validFrom": "2025-03-01T00:00:00Z",
    "validUntil": "2025-03-01T20:00:00Z"
  },
  "contests": [],
  "accessibility": {},
  "security": {}
}

Contest Definition

    "en": "President of the United States",

  },

  "maxSelections": 1,
  "minSelections": 0,
  "writeInAllowed": true,
  "candidates": [
    {
      "candidateId": "CAND-A-2025",
      "name": "Jane Doe",
      "party": "Progressive Party",
      "photoUrl": "https://cdn.example.com/candidate-a.jpg",
      "biography": {
        "en": "Former Senator from California...",
        "ko": "캘리포니아 전직 상원의원..."

  "ballotStyle": {
    "layout": "vertical",
    "ordering": "alphabetical"
  }
}

Field Definitions

Required Fields

Field Type Description Example
ballotId string Unique identifier for ballot "BALLOT-2025-001"
version string WIA standard version "WIA-SOC-015-v1.0"
electionId string Unique election identifier "ELECTION-2025-001"
electionName i18n object Multilingual election name {"en": "...", "ko": "..."}
contests array List of contests on ballot [{...}, {...}]

Internationalization Support

WIA-SOC-015 mandates comprehensive multilingual support through internationalization (i18n) objects. Any user-facing text must be provided in multiple languages using ISO 639-1 language codes:

{
  "electionName": {
    "en": "Presidential Election 2025",
    "ko": "2025년 대통령 선거",
    "es": "Elección Presidencial 2025",
    "fr": "Élection présidentielle 2025",
    "zh": "2025年总统选举",
    "ar": "الانتخابات الرئاسية 2025",
    "ru": "Президентские выборы 2025",
    "de": "Präsidentschaftswahl 2025",
    "ja": "2025年大統領選挙"
  }
}

The standard requires support for at least English plus any languages spoken by more than 5% of the jurisdiction's population. Full compliance supports all 99 languages in the WIA language registry.

Visual Accommodations

Visual presentation preferences are encoded in the ballot data to ensure consistent accessibility:

  • High Contrast Mode: Alternative color schemes for low vision voters
  • Large Text Support: Scalable text sizes up to 36pt without layout breaking
  • Symbol Support: Icon representations of candidates and options
  • Color Blindness Modes: Accessible color palettes for various types of color vision deficiency

Validation Rules

Schema Validation

WIA-SOC-015 provides JSON Schema definitions for automated validation. All ballot data must pass schema validation before being accepted by compliant systems:

  • Type Checking: All fields must match specified data types (string, number, boolean, object, array)
  • Required Fields: Mandatory fields must be present and non-null
  • Format Validation: Dates, URLs, and identifiers must match specified formats
  • Range Validation: Numeric values must fall within allowed ranges
  • Enumeration Validation: Enumerated fields must contain only allowed values

Business Logic Validation

Beyond schema validation, the standard defines business rules that must be enforced:

Rule Description Error Code
Date Validity Election dates must be in the future at creation time VAL-001
Contest Consistency maxSelections must be >= minSelections VAL-002
Candidate Uniqueness No duplicate candidateId values within a contest VAL-003
Language Completeness All i18n objects must include at least 'en' key VAL-004
Security Metadata Digital signatures must be present and valid VAL-005

Aggregation Support

Results can be aggregated at multiple levels (precinct, district, state, national) while maintaining cryptographic verification of accuracy. The hierarchical structure enables drill-down analysis and independent verification at each level.

Implementation Best Practices

Data Storage

WIA-SOC-015 compliant systems should implement robust data storage practices:

  • Immutable Storage: Once created, ballot definitions should not be modified; create new versions instead
  • Encryption at Rest: All ballot and result data must be encrypted when stored
  • Backup and Recovery: Multiple redundant copies in geographically distributed locations
  • Access Logging: Complete audit trail of all data access and modifications
  • Version Control: Track all changes to ballot definitions with timestamps and authorship

Data Migration

Converting legacy ballot data to WIA-SOC-015 format requires careful mapping:

  1. Analyze existing data structures and identify equivalent WIA fields
  2. Create mapping rules for data transformation
  3. Implement validation to ensure migrated data meets WIA requirements
  4. Test migration with sample data before production conversion
  5. Maintain legacy data alongside WIA format during transition period
  6. Document any limitations or assumptions in the migration process
Migration Tools: The WIA provides open-source migration utilities for common legacy formats including Election Markup Language (EML), NIST SP 1500-100, and various proprietary formats. These tools accelerate adoption while ensuring data fidelity.

Extension Mechanisms

Custom Fields

While WIA-SOC-015 defines comprehensive standard fields, jurisdictions may require additional data. The standard supports extensions through a structured approach:

{
  "ballotId": "BALLOT-2025-001",
  "version": "WIA-SOC-015-v1.0",
  "extensions": {
    "jurisdiction-specific": {
      "customField1": "value1",
      "customField2": 123
    },
    "vendor-specific": {
      "vendorId": "ACME-VOTING-v2",
      "vendorData": {...}
    }
  },
  "...": "standard fields"
}

Extensions must not duplicate standard fields or contradict standard requirements. All core WIA-SOC-015 functionality must work without accessing extension data.

Performance Considerations

Large-Scale Elections

National elections with millions of voters require careful attention to data size and processing efficiency:

  • Compression: Use gzip or brotli compression for ballot transmission
  • Pagination: Implement cursor-based pagination for large result sets
  • Caching: Cache frequently accessed ballot definitions
  • CDN Distribution: Distribute static ballot content via content delivery networks
  • Database Optimization: Index key fields for efficient querying
