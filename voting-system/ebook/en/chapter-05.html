API-Driven Electoral Infrastructure

Phase 2 of WIA-SOC-015 defines standardized Application Programming Interfaces (APIs) that enable different voting system components to communicate effectively. By specifying RESTful endpoints, authentication mechanisms, request/response formats, and error handling protocols, the standard ensures that voter registration systems, ballot delivery platforms, vote tabulation engines, and audit tools can work together seamlessly regardless of vendor or implementation technology.

This chapter explores the API specification in detail, examining authentication, core endpoints, security considerations, and implementation patterns that enable robust, scalable voting infrastructure.

RESTful Architecture

Core Principles

WIA-SOC-015 APIs follow REpresentational State Transfer (REST) architectural principles:

  • Resource-Based: APIs expose resources (ballots, voters, results) rather than actions
  • Stateless: Each request contains all information needed for processing; no server-side session state
  • Cacheable: Responses indicate cacheability to improve performance
  • Uniform Interface: Consistent patterns for CRUD (Create, Read, Update, Delete) operations
  • Layered System: Clients cannot tell whether they're connected to end server or intermediary

Base URL Structure

All WIA-SOC-015 APIs use versioned URLs to ensure backward compatibility:

https://api.voting.example.com/v1/{resource}

The version number (v1) allows for API evolution while maintaining support for existing implementations. When breaking changes are necessary, a new version is introduced without disrupting existing integrations.

Authentication and Authorization

OAuth 2.0 Integration

WIA-SOC-015 mandates OAuth 2.0 for API authentication, providing secure, delegated access without sharing credentials:

Flow Type Use Case Token Type
Authorization Code Voter-facing applications Access + Refresh
Client Credentials Server-to-server communication Access Token
Device Code Voting terminals without keyboard Access Token

Token Format

Access tokens follow JSON Web Token (JWT) format with standardized claims:

{
  "iss": "https://auth.voting.example.com",
  "sub": "voter-12345",
  "aud": "voting-api",
  "exp": 1735689600,
  "iat": 1735686000,
  "scope": "vote.cast ballot.view",
  "wia": {
    "standard": "WIA-SOC-015-v1.0",
    "jurisdiction": "CA-DIST-12",
    "election": "ELECTION-2025-001"
  }
}

Role-Based Access Control

Different actors have different API access levels:

  • Voters: Can view their ballots, cast votes, verify submissions
  • Poll Workers: Can check in voters, handle provisional ballots, generate reports
  • Election Officials: Can manage ballot definitions, configure elections, access all data
  • Auditors: Can view audit trails, verify results, but cannot modify data
  • Observers: Can view public results and aggregate statistics

Audit Trail Endpoints

Get Audit Events

GET /v1/elections/{electionId}/audit

Retrieves audit trail events for an election.

Query Parameters:

  • startDate: Filter events after this date
  • endDate: Filter events before this date
  • eventType: Filter by event type
  • actor: Filter by actor ID
  • limit: Number of events to return (max 1000)
  • offset: Pagination offset

Error Handling

Standard Error Response

All errors follow a consistent format:

{
  "error": {
    "code": "AUTH_INVALID_TOKEN",
    "message": "The provided authentication token is invalid or expired",
    "details": "Token expired at 2025-01-15T10:00:00Z",
    "timestamp": "2025-01-15T10:05:30Z",
    "requestId": "req-abc123def456",
    "documentation": "https://docs.wia-soc-015.com/errors/AUTH_INVALID_TOKEN"
  }
}

HTTP Status Codes

Code Meaning Use Case
200 OK Successful GET request
201 Created Successful POST creating new resource
400 Bad Request Invalid request data
401 Unauthorized Missing or invalid authentication
403 Forbidden Authenticated but not authorized
404 Not Found Resource doesn't exist
429 Too Many Requests Rate limit exceeded
500 Internal Server Error Server-side failure

Rate Limiting

Limit Headers

All API responses include rate limit information:

X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 987
X-RateLimit-Reset: 1735689600

Limit Tiers

  • Voter Operations: 100 requests per hour per voter
  • Poll Worker Operations: 1,000 requests per hour per worker
  • Election Official Operations: 10,000 requests per hour
  • Audit Access: 10,000 requests per hour (public data)

Implementation Best Practices

Security

  • Always use HTTPS: Never transmit election data over unencrypted connections
  • Validate all inputs: Never trust client data; validate and sanitize all inputs
  • Implement CORS properly: Configure Cross-Origin Resource Sharing to prevent unauthorized access
  • Log security events: Record all authentication failures, authorization denials, and suspicious activity
  • Keep dependencies updated: Regularly update libraries and frameworks to patch vulnerabilities

Performance

  • Implement caching: Cache immutable data (ballot definitions) aggressively
  • Use pagination: Never return unbounded result sets
  • Compress responses: Enable gzip/brotli compression for all responses
  • Optimize database queries: Index frequently queried fields
  • Monitor performance: Track response times and error rates

Chapter Summary

Key Takeaways:

Phase 2 defines RESTful APIs following industry-standard patterns including OAuth 2.0 authentication and JWT tokens.
Standardized endpoints cover the complete voting lifecycle: registration, ballot delivery, vote submission, result tabulation, and audit access.
Role-based access control ensures voters, officials, and auditors have appropriate permissions without over-privileging any actor.
Comprehensive error handling and rate limiting protect system stability while providing clear feedback to API consumers.
Security best practices including HTTPS, input validation, and security logging are mandatory for WIA-SOC-015 compliance.
