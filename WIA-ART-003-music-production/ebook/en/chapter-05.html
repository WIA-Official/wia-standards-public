Java Example
import com.wia.standard.*;

public class WIAExample {
    public static void main(String[] args) {
        // Configuration using builder pattern
        WIAClient client = WIAClient.builder()
            .apiKey(System.getenv("WIA_API_KEY"))
            .environment(Environment.PRODUCTION)
            .connectionTimeout(Duration.ofSeconds(30))
            .build();

        try {
            // Create resource with validation
            Resource resource = Resource.builder()
                .name("Example Resource")
                .type("standard-compliant")
                .metadata(Map.of(
                    "version", "1.0",
                    "createdBy", "user@example.com"
                ))
                .build();

            Resource created = client.createResource(resource);
            System.out.println("Created: " + created.getId());

        } catch (ValidationException e) {
            System.err.println("Validation failed: " + e.getMessage());
        } finally {
            client.close();
        }
    }
}

5.1 Universal Plugin State

The WIA Plugin State specification enables plugin configurations to transfer between different host applications. Rather than replacing existing plugin formats, WIA-ART-003 wraps them with standardized metadata and parameter mapping.

State Structure

{
  "wia_plugin_state": {
    "version": "1.0",
    "plugin_id": {
      "vendor": "FabFilter",
      "name": "Pro-Q 3",
      "version": "3.21",
      "formats": {
        "vst3": "XXXXXXXX-XXXX-XXXX-XXXX",
        "au": "aufx:FabQ:FaFi",
        "aax": "com.fabfilter.pro-q-3"
      }
    },
    "semantic_params": [...],
    "native_state": {
      "format": "vst3",
      "blob": "base64..."
    }
  }
}
🔌 Format Translation

When transferring between hosts using different plugin formats:

  1. Semantic parameters provide human-readable mapping
  2. Native state preserves exact plugin configuration
  3. If native format unavailable, semantic params reconstruct settings

5.3 Plugin Discovery Protocol

The WIA Plugin Discovery Protocol enables hosts to locate equivalent plugins across format boundaries. This is essential for session portability when collaborators use different platforms.

// Plugin discovery request
const alternatives = await WIA.Plugin.findAlternatives({
  plugin: {
    vendor: "FabFilter",
    name: "Pro-Q 3"
  },
  requiredCapabilities: [
    "eq.parametric",
    "eq.bands.minimum.8"
  ]
});

// Response
{
  "exact_match": null,  // Original plugin not installed
  "alternatives": [
    {
      "plugin": {
        "vendor": "TDR",
        "name": "SlickEQ",
        "match_score": 0.85
      },
      "parameter_mapping": {...}
    }
  ]
}
🔍 Matching Criteria
  • Plugin category (EQ, compressor, reverb, etc.)
  • Supported parameter types and ranges
  • Processing quality characteristics
  • User preference weighting

5.4 Preset Exchange

WIA-ART-003 enables universal preset exchange between plugins. A preset created in one plugin can be translated to work with compatible plugins from different vendors.

Preset Package Format

WIA Preset Package (.wiap)
├── manifest.json
│   ├── name: "Warm Vocal"
│   ├── category: "vocal"
│   ├── creator: "Producer Name"
│   └── original_plugin: {...}
├── semantic_preset.json
│   └── (semantic parameter values)
├── native/
│   ├── fabfilter-pro-q-3.fxp
│   ├── waves-ssl-eq.xml
│   └── (other native formats)
└── preview.png

Translation Example

// Export preset from Plugin A
const preset = await WIA.Plugin.exportPreset(pluginInstance, {
  name: "Punchy Kick",
  category: "drums.kick",
  includeNative: true,
  generateSemantic: true
});

// Import to Plugin B (different vendor)
await WIA.Plugin.importPreset(differentPlugin, preset, {
  mappingStrategy: "semantic",
  preserveCharacter: true
});

5.5 SDK Implementation

The WIA Plugin SDK provides a TypeScript/JavaScript implementation for integrating plugin state management into DAW applications.

Core SDK Methods

import { WIAPlugin } from '@wia/music-sdk';

// Serialize plugin state
const state = await WIAPlugin.serialize(pluginInstance, {
  includeNative: true,
  includeSemantic: true,
  version: '1.0'
});

// Deserialize to new instance
await WIAPlugin.deserialize(newInstance, state, {
  fallbackToSemantic: true,
  tolerantMode: true
});

// Compare plugin states
const diff = WIAPlugin.compare(stateA, stateB, {
  ignoreMetadata: false,
  semanticTolerance: 0.01
});
