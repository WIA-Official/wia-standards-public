Chapter 3: Data Transformation

Cleaning, normalising, deriving — without losing the audit trail

Chapter overview: Five transformation types, when SQL beats code, when code beats SQL, a real-world windowed-aggregate example, the eight transformation patterns the standard recognises, and how to test transformations the way developers test functions.

3.1 What is transformation?

Transformation is the work between bronze (raw, faithful, unfiltered) and gold (curated, conformed, ready to consume). Every change must be reversible (the bronze layer is immutable) and traceable (column-level lineage to the input).

3.2 Five transformation types

TypeExampleTool fit
CleaningTrim whitespace, parse dates, drop NULLsSQL (dbt)
NormalisationCurrency to USD cents, units to SI, casing rulesSQL or Python
EnrichmentJoin with reference data, third-party feed, geocoder, ML modelSQL + UDF; Python for ML
AggregationHourly revenue, rolling P95 latency, churn cohortsSQL with window functions
FilterRemove fraudulent rows, isolate one cohort, GDPR redactionSQL

3.3 SQL-based transforms (dbt & SQLMesh)

SQL is the lingua franca of warehouses. dbt 1.9 + SQLMesh 0.115 add testing, documentation, and lineage on top.

-- models/marts/finance/daily_revenue.sql
{{ config(materialized='incremental', unique_key='date',
          on_schema_change='sync_all_columns',
          incremental_strategy='merge') }}

SELECT
  DATE(created_at)               AS date,
  currency,
  SUM(total_cents) / 100.0       AS revenue,
  COUNT(*)                       AS orders,
  COUNT(DISTINCT user_id)        AS unique_buyers
FROM {{ ref('orders_clean') }}
{% if is_incremental() %}
WHERE DATE(created_at) >= (SELECT MAX(date) - INTERVAL '3 days' FROM {{ this }})
{% endif %}
GROUP BY 1, 2

dbt tests (in schema.yml) attach to the model:

models:
  - name: daily_revenue
    columns:
      - name: date
        tests: [not_null, unique]
      - name: revenue
        tests:
          - not_null
          - dbt_utils.expression_is_true:
              expression: ">= 0"

3.4 Code-based transforms (PySpark, Polars, DuckDB)

Use code when SQL would be unreadable, when you need a non-trivial UDF (ML inference, geospatial), or when latency demands a streaming engine.

import polars as pl

orders = pl.scan_parquet("s3://bronze/orders/dt=*/part-*.parquet")

clean = (
    orders
    .filter(pl.col("total_cents") >= 0)
    .with_columns([
        pl.col("created_at").dt.truncate("1d").alias("date"),
        pl.col("currency").str.to_uppercase(),
    ])
    .group_by(["date","currency"])
    .agg([
        (pl.col("total_cents").sum() / 100).alias("revenue"),
        pl.col("order_id").n_unique().alias("orders"),
    ])
)

clean.sink_parquet("s3://silver/daily_revenue/", compression="zstd")

3.5 Complex example — sliding 7-day cohort retention

WITH user_first_seen AS (
  SELECT user_id, MIN(DATE(created_at)) AS cohort_date
  FROM {{ ref('orders_clean') }}
  GROUP BY user_id
),
activity AS (
  SELECT user_id, DATE(created_at) AS active_date
  FROM {{ ref('orders_clean') }}
  GROUP BY 1, 2
)
SELECT
  ufs.cohort_date,
  DATE_DIFF('day', ufs.cohort_date, a.active_date) AS day_offset,
  COUNT(DISTINCT a.user_id) AS active_users,
  ROUND(100.0 * COUNT(DISTINCT a.user_id) /
        NULLIF(MAX(COUNT(DISTINCT a.user_id))
               OVER (PARTITION BY ufs.cohort_date), 0), 2) AS retention_pct
FROM user_first_seen ufs
JOIN activity a USING (user_id)
WHERE DATE_DIFF('day', ufs.cohort_date, a.active_date) BETWEEN 0 AND 30
GROUP BY 1, 2
ORDER BY 1, 2;

3.6 Eight transformation patterns

  1. Slowly Changing Dimension (SCD) Type 2 — preserve history with valid_from / valid_to.
  2. Window aggregation — leading/lagging with ROWS / RANGE.
  3. Pivoting — long-to-wide with PIVOT or conditional SUM.
  4. Surrogate key — hash of natural keys (MD5) for stability across runs.
  5. Snapshotting — daily point-in-time copy for compliance.
  6. Deduplication — QUALIFY ROW_NUMBER() OVER (PARTITION BY id ORDER BY ts DESC) = 1.
  7. Conformed dimension — single calendar table joined by every fact.
  8. Type 6 hybrid — current value + history in one table.

3.7 Optimisation

  • Predicate pushdown — let Iceberg/Parquet skip files via min/max statistics.
  • Column pruning — read only what is needed (3-10× speed-up routine).
  • Broadcast small dimensions — under 100 MB usually beats shuffle joins.
  • Avoid SELECT * in incremental models — schema drift will silently change downstream.
  • Z-order on multi-key filters — for tables filtered by both date and user_id.

3.8 Testing transformations

Treat transforms like functions: inputs are fixtures, output is asserted. dbt tests, Soda CL contracts, and Great Expectations checkpoints share the same conformance role.

# tests/test_daily_revenue.sql
SELECT *
FROM {{ ref('daily_revenue') }}
WHERE revenue < 0
   OR orders < 0
   OR currency NOT IN ('USD','KRW','EUR','JPY','GBP')
-- The model passes if this query returns 0 rows.

Property-based testing (Hypothesis for Python, ScalaCheck for JVM) is recommended for non-trivial UDFs.
