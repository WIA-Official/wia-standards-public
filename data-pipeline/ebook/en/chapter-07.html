Chapter 7: Quality, Idempotency & Durability

Why re-running yesterday's load must never double-count revenue

Chapter overview: Why quality matters financially, how to validate at boundaries, idempotency in HTTP and pipeline form, exactly-once recipes, data contracts as governance, schema evolution, lineage, backpressure, and a quality-SLA scorecard you can ship next week.

7.1 Why data quality matters

A 2024 IBM/Forrester analysis put the cost of poor data quality at 15-25% of revenue for typical enterprises. The damage compounds: a wrong number in bronze becomes a wrong dashboard, becomes a wrong board decision, becomes a regulatory finding. Quality is not an after-thought; it is the boundary between trusted and untrusted data.

7.2 Validation at boundaries

Validate twice: at ingest (Pydantic / Avro schema) and at publish (Soda / Great Expectations). The first catches malformed records; the second catches statistical drift.

7.3 Idempotency

An operation is idempotent if applying it once or many times produces the same result. The standard requires:

  • Idempotency-Key on every mutating HTTP call (server caches response 24 h).
  • Iceberg MERGE INTO on every loader (transactionally upsert by primary key).
  • Deterministic surrogate keys (md5(natural_key)) so re-runs land on the same row.
  • Watermark-based extracts that can be replayed without duplicates.
-- The classic idempotent merge
MERGE INTO lake.orders t USING bronze.orders_in s ON t.order_id = s.order_id
WHEN MATCHED AND s.updated_at > t.updated_at THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *;

7.5 Exactly-once processing

End-to-end exactly-once requires three pieces working in concert:

  1. Idempotent producer — Kafka enable.idempotence=true, transactional id stable per worker.
  2. Transactional consumer — isolation.level=read_committed, offsets committed inside producer transaction.
  3. Idempotent sink — Iceberg MERGE, BigQuery Storage Write API committed_offset, or unique-key DELETE-INSERT.

7.7 Data contracts (Bitol / Open Data Contract Standard 1.1.0)

dataContractSpecification: "1.1.0"
id: orders-clean.v1
info:
  title: Orders (cleaned)
  version: 1.0.0
  owner: data-platform@company.com
schema:
  models:
    orders:
      type: table
      fields:
        order_id:    { type: long, required: true, primary: true }
        user_id:     { type: long, required: true }
        total_cents: { type: long, required: true }
        currency:    { type: string, required: true, format: ISO-4217 }
        created_at:  { type: timestamp, required: true }
quality:
  type: SodaCL
  specification: |
    checks for orders:
      - missing_count(order_id) = 0
      - duplicate_count(order_id) = 0
      - freshness(created_at) < 1d
servicelevels:
  freshness: { threshold: P1D }
  retention: { period: P7Y }

The contract is the agreement; the consumer can rely on the producer up to the SLA, and a violation is a contract incident, not a "what happened?" Slack thread.

7.8 Test-driven data

  1. Write the failing test first (an expected-row count, a forbidden value).
  2. Build / fix the transform until the test passes.
  3. Add the check to the data contract so it stays passed.
  4. Schedule the check on every run.

7.9 Referential integrity

Lakehouses and most warehouses do not enforce foreign keys. The standard recommends:

  • dbt's relationships test on every fact-to-dimension join.
  • Soda CL row_count sanity (target rows ≥ source rows × expected ratio).
  • Periodic EXCEPT queries between staging and curated layers.

7.10 Data lineage

Lineage is governance with teeth. Column-level lineage answers "if I change column X, what breaks?" before you change it. OpenLineage 1.21 events feed Marquez or DataHub; SQLLineage / sqlfluff parses dbt models; Iceberg tables expose column-id evolution.

7.11 Automated validation

# quality_gate.py — runs as the last task in every DAG
def quality_gate(snapshot_id: str) -> None:
    result = soda.scan(
        data_source="snowflake-prod",
        checks_files=["contracts/orders.yml"],
        variables={"snapshot_id": snapshot_id}
    )
    if result.has_failed_check_results():
        raise RuntimeError(f"Quality gate failed: {result.summary()}")
    publish_lineage_event(snapshot_id, "quality_pass")

7.13 Backpressure & flow control

Producers must slow down when sinks lag. Two mechanisms:

  • HTTP 429 with Retry-After on REST writers.
  • Reactive Streams / Project Reactor backpressure for in-process JVM pipelines.
  • Kafka quotas per consumer group to cap downstream stress.

7.14 Transactions & atomicity

Iceberg snapshots are atomic; the dataset is either at version N or N-1, never half-way. Multi-table transactions (e.g., orders + order_items) are coordinated through the orchestrator: load both into staging, then commit both in one merge transaction.

7.15 Reprocessing

Replays must be safe by default. The standard's three-step recipe:

  1. Roll back the watermark to the start of the bad window.
  2. Re-run the DAG with --dry-run-quality first; verify volume / distribution.
  3. Promote: real run, MERGE upserts overwrite the bad rows, lineage records the replay.
