Chapter 5: Orchestration

Turning isolated jobs into a reliable graph

Chapter overview: What orchestration solves, an Airflow 2.10 worked DAG, Dagster's software-defined assets, Prefect 3 deployments, scheduling strategy, dependency management, retries, resource governance, and the testing patterns that catch DAG breakage before merge.

5.1 What is orchestration?

Orchestration is the difference between "a script ran" and "the data is ready". The orchestrator owns: scheduling, dependency graph, retries, alerting, observability, parameterisation, and the boundaries between teams. Pick one orchestrator and let everything else plug in.

5.2 Apache Airflow 2.10

Still the de-facto open-source default. The TaskFlow API removes the boilerplate that made early Airflow miserable.

from airflow.decorators import dag, task
from datetime import datetime, timedelta

@dag(
    schedule="0 2 * * *",
    start_date=datetime(2026, 4, 25),
    catchup=False,
    max_active_runs=1,
    default_args={
        "retries": 3,
        "retry_delay": timedelta(minutes=5),
        "execution_timeout": timedelta(hours=1),
        "owner": "data-platform",
    },
    tags=["sales","daily","gold"],
)
def daily_sales_pipeline():

    @task
    def extract():
        return run_extractor("orders", since="{{ prev_data_interval_end_success }}")

    @task
    def transform(payload):
        return run_dbt_select(payload, models=["+orders_clean"])

    @task
    def load(snapshot_id):
        return iceberg_merge(target="lake.orders", snapshot=snapshot_id)

    @task
    def quality(snapshot_id):
        return run_soda("qg_orders", snapshot=snapshot_id)

    quality(load(transform(extract())))

daily_sales_pipeline()

5.3 Dagster & Prefect (alternatives)

Dagster 1.10 models the data graph as software-defined assets — assets, not tasks, are first-class. Asset checks and partition definitions are declarative.

import dagster as dg

@dg.asset
def orders_raw(context: dg.AssetExecutionContext): ...

@dg.asset(deps=[orders_raw])
def orders_clean(orders_raw): ...

@dg.asset_check(asset=orders_clean)
def orders_quality():
    return dg.AssetCheckResult(passed=row_count() > 0)

Prefect 3 takes a Python-first, dynamic-DAG approach with deployments. Useful when DAGs are determined at run time (e.g., per-customer pipelines).

5.5 Dependency management

Two anti-patterns produce 80% of orchestration incidents:

  • Implicit deps — task B reads a table that task A wrote, but the DAG does not encode the link. The orchestrator has no idea they belong together.
  • Cross-DAG hard deps — DAG A waits for DAG B's exact run. Use ExternalTaskSensor sparingly; prefer Datasets / asset materialisations.

Airflow Datasets (2.4+) and Dagster's auto-materialise close this loop:

from airflow.datasets import Dataset

orders_dataset = Dataset("iceberg://polaris-prod/lake.orders")

@dag(schedule=[orders_dataset])              # consumer
def daily_revenue():

5.6 Retries

  • Per-task retries — 3 with exponential back-off, default.
  • On-failure callbacks — page on-call only on the final failure, not every retry.
  • Non-retryable exceptions — schema break, validation failure, auth — fail fast, do not retry.
  • Backfills — separate trigger from the live schedule; resource-isolated workers.

5.8 Advanced DAG patterns

  • Dynamic task mapping — fan out one task per partition / customer.
  • Trigger rules — all_done, one_failed, none_skipped.
  • Task groups — collapse complex sub-graphs in the UI without losing dependencies.
  • Pool concurrency — cap how many tasks may hit a shared resource (Postgres, partner API).

5.10 Testing DAGs

  • DAG load test — pytest that imports every DAG file; catches syntax errors before merge.
  • Cycle detection — Airflow refuses to load cycles; CI step verifies.
  • Render test — render Jinja templates against frozen context.
  • Backfill simulation — run the DAG against a staging warehouse before promoting to prod.
def test_daily_sales_dag():
    from airflow.models import DagBag
    bag = DagBag(include_examples=False)
    dag = bag.get_dag("daily_sales_pipeline")
    assert dag is not None
    assert {t.task_id for t in dag.tasks} == {"extract","transform","load","quality"}
    assert not bag.import_errors

5.11 Failure recovery

  1. Idempotent steps mean a single task retry is safe.
  2. Atomic publishes (Iceberg snapshot commit) mean a partial run leaves no half-state.
  3. Quarantine quality failures instead of failing the whole DAG when blockers are non-blocking.
  4. Compaction job separately; do not couple curation latency to maintenance windows.
