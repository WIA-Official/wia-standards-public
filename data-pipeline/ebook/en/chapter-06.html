Chapter 6: Observability

Knowing whether the data is right, not just whether the job ran

Chapter overview: The three pillars (metrics / logs / traces) plus a fourth specific to data — lineage. What to count, when to alert, how to log structurally, how to monitor freshness, and how OpenTelemetry + OpenLineage stitch it all together.

6.1 What is data observability?

Application observability tells you whether the program is running. Data observability tells you whether what it produced is right. The four pillars: freshness (is the data current?), volume (did the expected number of rows arrive?), schema (is the structure intact?), and distribution (do the values look like they should?).

6.2 The three operational pillars

  • Metrics — counters, gauges, histograms (OTLP / Prometheus).
  • Logs — structured JSON with trace correlation IDs.
  • Traces — distributed spans across orchestrator → workers → warehouse.

Add lineage (OpenLineage 1.21) for data systems specifically — it ties operational signals to the dataset graph.

6.4 Alerting strategy

  • Alert on symptoms, not causes. "Daily revenue table is more than 6 hours stale" is a symptom; "the third Airflow worker had high CPU" is a cause that may not need waking anyone.
  • SLO-based alerting. Burn-rate alerts (Google SRE) give one page per real incident, not 47.
  • Severity ladder. P0 wakes a human; P1 emails on-call; P2 lands in Slack triage; P3 is a daily digest.
  • Mute the noisy. Quality warnings ride the daily digest until they trip a P1 threshold.

6.5 Dashboards (the three that matter)

  1. Operational health — DAG success rate, P95 duration, failed runs trend.
  2. Data freshness — last-success-at per dataset, lag-by-dataset heatmap.
  3. Quality scorecard — Soda CL pass / fail per dataset over time.

6.6 Logging best practices

import logging, json, sys

class JsonFormatter(logging.Formatter):
    def format(self, record):
        payload = {
            "timestamp": self.formatTime(record, "%Y-%m-%dT%H:%M:%S.%fZ"),
            "level": record.levelname,
            "message": record.getMessage(),
            "logger": record.name,
            "trace_id": getattr(record, "trace_id", None),
            "pipeline_id": getattr(record, "pipeline_id", None),
        }
        if record.exc_info:
            payload["exception"] = self.formatException(record.exc_info)
        return json.dumps(payload)

handler = logging.StreamHandler(sys.stdout)
handler.setFormatter(JsonFormatter())
logging.basicConfig(level=logging.INFO, handlers=[handler])

Three rules: no sensitive data in logs (masked at source); one event = one structured line; correlation IDs on every line.

6.7 Data quality monitoring

Soda CL, Great Expectations, and dbt tests are the three open-source pillars. The standard treats them as pluggable engines behind a single gate API.

# soda CL — orders_clean.yml
checks for orders_clean:
  - row_count between 10000 and 5000000
  - missing_count(user_id) = 0
  - duplicate_count(order_id) = 0
  - freshness(created_at) < 1d
  - schema:
      fail:
        when forbidden column present: [ssn, credit_card]
  - invalid_count(currency) = 0:
      valid values: [USD, KRW, EUR, JPY, GBP]

6.8 Cost monitoring

  • Tag every warehouse query with pipeline_id + run_id via session parameters.
  • Daily roll-up of compute cost / dataset / pipeline.
  • Alert when a single run's cost exceeds 2× its 30-day median.
  • Track storage tier hot / warm / cold and surface candidates for archival.

6.9 Distributed tracing

from opentelemetry import trace
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor

provider = TracerProvider()
provider.add_span_processor(BatchSpanProcessor(OTLPSpanExporter(endpoint="otel-collector:4317")))
trace.set_tracer_provider(provider)
tracer = trace.get_tracer("wia-data-pipeline")

with tracer.start_as_current_span("transform_orders") as span:
    span.set_attribute("pipeline.id", "pip_daily_sales")
    span.set_attribute("pipeline.run_id", "exec_5b9a")
    span.set_attribute("iceberg.snapshot_id", snapshot_id)
    do_transform()

6.10 Lineage with OpenLineage

Every job emits at minimum a START and a COMPLETE (or FAIL) event. The standard requires column-level lineage on outputs.

from openlineage.client import OpenLineageClient
from openlineage.client.event_v2 import Job, Run, RunEvent, RunState
from openlineage.client.facet_v2 import schema_dataset, column_lineage_dataset
from datetime import datetime

client = OpenLineageClient(url="https://api.wia-data-pipeline.org/v1")

client.emit(RunEvent(
    eventType=RunState.COMPLETE,
    eventTime=datetime.utcnow().isoformat()+"Z",
    run=Run(runId="5b9a..."),
    job=Job(namespace="pip.daily_sales", name="transform_orders"),
    inputs=[...],
    outputs=[{
        "namespace":"iceberg://polaris-prod","name":"lake.orders",
        "facets":{
            "schema": schema_dataset.SchemaDatasetFacet(fields=[...]),
            "columnLineage": column_lineage_dataset.ColumnLineageDatasetFacet(fields={...})
        }
    }],
    producer="https://wia-data-pipeline.org/sdk/1.1"
))

6.11 Freshness monitoring

Use updated_at max from the table itself; a freshness gate that simply checks if the latest row was within an SLA window catches 90 % of stale-data incidents.

SELECT
  TIMESTAMPDIFF(SECOND, MAX(updated_at), CURRENT_TIMESTAMP) AS lag_seconds
FROM lake.orders;

6.12 Profiling & performance

  • Snowflake Query History + Query Profile.
  • BigQuery INFORMATION_SCHEMA.JOBS_BY_PROJECT.
  • Iceberg snapshot summary — read time vs commit time.
  • Worker-level CPU / memory via Prometheus node-exporter.

Summary

  1. Four pillars: metrics, logs, traces, lineage. None is optional.
  2. Alert on symptoms (freshness, quality), not on causes (CPU spikes).
  3. Quality checks gate the publish — they are not after-thoughts.
  4. OpenLineage events tie operational signals to the dataset graph.
  5. Tag every query with run id; cost overruns become detectable.
