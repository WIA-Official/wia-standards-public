The world of data pipelines

Chapter overview: What a data pipeline is, why pipeline shape matters, the four generations that brought us here, the anatomy (Source / Pipeline / Destination), the canonical patterns (ETL, ELT, Lambda, Kappa), batch vs stream, and the six challenges WIA-DATA-004 sets out to solve.

1.1 What is a data pipeline?

A data pipeline is an automated process that moves data from one place to another while performing the transformations the receiving system needs. Just as a physical pipeline carries crude oil to a refinery, a data pipeline carries raw events, log lines, transactional rows, and sensor readings to a place where they are useful. When a pipeline is healthy, every dashboard, every model, every regulator's filing inherits that health. When it leaks, the whole system suffers вҖ” silently at first, then catastrophically.

1.3 Anatomy of a pipeline

Data Source

  • Operational databases: PostgreSQL, MySQL, MongoDB, Cassandra.
  • APIs: REST, GraphQL, gRPC, webhooks.
  • Files: Parquet, Avro, ORC, CSV, JSONL.
  • Streams: Apache Kafka, AWS Kinesis, Google Pub/Sub, NATS JetStream.
  • SaaS: Salesforce, HubSpot, Stripe, Google Analytics 4, Shopify.

Data Pipeline

  1. Ingestion вҖ” extract or capture data without dropping any.
  2. Validation вҖ” reject the malformed loudly, never silently.
  3. Transformation вҖ” clean, normalise, conform, derive.
  4. Enrichment вҖ” join with reference data, third-party feeds, geocoders, model inferences.
  5. Aggregation вҖ” daily / hourly rollups, windowed metrics.

Data Destination

  • Lakehouse: Iceberg / Delta / Hudi tables on object storage.
  • Warehouse: Snowflake, BigQuery, Redshift, Databricks SQL.
  • Lake: S3, ADLS Gen2, GCS, MinIO.
  • Operational caches: Redis, Memcached, DragonflyDB.
  • Reverse ETL targets: CRM, ad platforms, support tools (back into product).

Modern default. Cloud warehouses make in-place transforms cheap (dbt, SQLMesh). Raw data is preserved; transforms are versioned SQL.

Lambda & Kappa

Lambda runs a slow batch path beside a fast streaming path; Kappa unifies them with a single replayable log (Kafka). Modern systems pick Kappa unless legacy batch must coexist.

Reverse ETL

Pushes warehouse-curated data back into operational systems (Salesforce, Marketo). Tools: Hightouch, Census. Must respect downstream rate limits and deduplicate aggressively.

1.6 Six pipeline challenges

  1. Reliability under partial failure вҖ” what does "the pipeline ran" even mean if 3 of 100 batches failed?
  2. Schema drift вҖ” upstream renames a field on a Friday afternoon and silently breaks Monday's report.
  3. Idempotency вҖ” re-running yesterday's load must not double-count revenue.
  4. Observability вҖ” knowing not just that a job ran, but whether the data it produced was right.
  5. Lineage вҖ” answering "where did this number come from?" in minutes, not weeks.
  6. Cost вҖ” query bills, egress, idle clusters; the line you can not see is the line you can not control.

1.7 The WIA-DATA-004 answer

This standard binds the moving parts into one wire:

  • Standard formats вҖ” Parquet 2.10 / Avro 1.12 / Iceberg v2 with explicit page indexes, hidden partitioning, schema evolution by column id.
  • Open contracts вҖ” Bitol Data Contract Specification 1.1.0 between every producer and every consumer.
  • Idempotency by default вҖ” Idempotency-Key on every mutating call; merge-on-write Iceberg sinks; Kafka exactly-once with the three-rule recipe.
  • Observability built in вҖ” OpenLineage 1.21 events for every run, OTLP metrics, structured logs.
  • Quality as a gate вҖ” Soda CL / Great Expectations checks fail loud at the boundary, not three dashboards downstream.
  • Climate-justice fee вҖ” every byte costs energy. Bronze вҶ’ Silver вҶ’ Gold ZSTD tiers reduce footprint without manual tuning.
