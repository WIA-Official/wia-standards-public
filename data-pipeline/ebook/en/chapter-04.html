Chapter 4: Data Loading

Where data lands — and the failure modes you learn in production

Chapter overview: Full vs Incremental, the Upsert pattern with Iceberg MERGE, bulk loaders for warehouses, streaming sinks, destination-specific strategy (Snowflake / BigQuery / Redshift / Iceberg / Delta), partitioning, indexing, and the operational lessons that only appear under load.

4.1 What is loading?

Loading is the third L in (E)(T)(L). It is also where most production incidents originate — partial commits, primary key violations, throttled bulk endpoints, missing partitions, frozen warehouses. The standard's loading layer is engineered around idempotent, transactional, replay-safe writes.

4.3 Upsert / merge pattern

Iceberg v2's MERGE INTO is the canonical way to upsert into a lakehouse. It is transactional, supports schema evolution, and produces a snapshot per merge.

MERGE INTO lake.orders t
USING (
  SELECT order_id, user_id, total_cents, currency, created_at, updated_at
  FROM bronze.orders_incoming
  WHERE updated_at >= :since
) s
ON t.order_id = s.order_id
WHEN MATCHED AND s.updated_at > t.updated_at THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *;

For destinations without native MERGE (early Redshift, some MySQL): the canonical workaround is delete-then-insert inside one transaction.

4.5 Streaming loads

For sub-minute freshness, use streaming sinks:

  • Snowflake Snowpipe Streaming — millisecond inserts; per-row pricing.
  • BigQuery Storage Write API — exactly-once writes via committed_offset.
  • Iceberg streaming with Flink Iceberg sink — `commit-interval=30s` is a reasonable default.
  • Kafka Connect S3 Sink with rotation policy `flush.size=10000, rotate.interval.ms=60000`.

4.6 Destination-specific strategy

Snowflake

  • Use Snowpipe for files < 250 MB; Snowpipe Streaming for sub-minute freshness.
  • Cluster keys on (date, customer_id) when most queries filter by both.
  • Multi-cluster warehouses for concurrent ELT.

BigQuery

  • Partition by ingestion time or a date column; cluster on up to four columns.
  • External Iceberg tables via BigLake unify lake + warehouse.
  • Reservations control cost; on-demand for spiky workloads.

Redshift

  • Distribution key on the join column; sort key on the filter column.
  • VACUUM after large deletes; ANALYZE after large appends.
  • Spectrum (Redshift querying Parquet on S3) for cold archives.

Apache Iceberg lakehouse

  • Hidden partitioning eliminates `dt=` boilerplate.
  • `copy-on-write` for read-heavy curated layers; `merge-on-read` for write-heavy bronze.
  • Compact small files via `RewriteFiles` weekly; expire snapshots beyond retention.

4.8 Indexing strategy

Lakehouses do not have B-tree indexes; they rely on column statistics, page indexes, bloom filters, and z-order. Warehouses (Snowflake, Redshift, BigQuery) auto-maintain these. For OLTP destinations, build indexes on:

  • Primary key (always).
  • Foreign key columns used in joins.
  • Columns in WHERE clauses with high selectivity.

Avoid indexes on low-cardinality columns (status, country) unless used in selective filters; they cost write throughput.

4.9 Performance tactics

  • Right-size files. 128 MB ± 25 % for Parquet; smaller files multiply metadata cost.
  • Stage close to compute. Same region as the warehouse, or you pay egress and latency.
  • Compress before transfer. ZSTD-3 saves 60-80 % vs uncompressed at < 10 % CPU overhead.
  • Sort within partitions. Improves dictionary encoding and predicate pushdown.
  • Async commit. Iceberg snapshot commit is fast; the heavy file write should complete first, then commit.

4.10 Error handling at load

  1. Constraint violation — quarantine offending rows, do NOT abort the batch.
  2. Partial file write — Iceberg snapshot is atomic; partial files become orphans removed by `RemoveOrphanFiles`.
  3. Throttling — exponential back-off on `429 Too Many Requests`; respect `Retry-After`.
  4. Out-of-order events — Iceberg MERGE with `WHEN MATCHED AND s.updated_at > t.updated_at` resolves naturally.
  5. Disk full — pre-flight `du -h` on workers; alert at 80 % capacity.

Summary

  1. Idempotent merges (Iceberg MERGE) make replays safe.
  2. Bulk loaders win on throughput; streaming sinks win on freshness.
  3. Partitioning + sort order are the two most under-tuned levers.
  4. Quarantine constraint violators; never silently drop them.
  5. Right-size files (128 MB) and compress with ZSTD.
