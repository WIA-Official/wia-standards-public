Chapter 2: Detection Methods

Chapter Overview

This chapter explores the core detection methodologies employed by modern IDS/IPS systems. We'll examine signature-based detection, anomaly-based detection, protocol analysis, and hybrid approaches. Understanding these techniques is essential for selecting, configuring, and optimizing intrusion detection systems for maximum effectiveness.

2.1 Signature-Based Detection

Signature-based detection—also called pattern matching or misuse detection—is the most mature and widely deployed IDS methodology. It operates by comparing network traffic, system logs, or file contents against a database of known attack signatures.

How Signature-Based Detection Works

  1. Signature Database: IDS maintains a comprehensive database of attack signatures (rules), each describing unique characteristics of a specific exploit or malicious behavior
  2. Traffic Analysis: Each packet, log entry, or file is analyzed and compared against the signature database
  3. Pattern Matching: Multi-pattern matching algorithms (Aho-Corasick, Wu-Manber) enable efficient comparison against thousands of signatures
  4. Alert Generation: When a match occurs, IDS generates an alert containing signature ID, severity, source/destination details, and attack classification
  5. Regular Updates: Signature databases are continuously updated from threat intelligence feeds, vulnerability databases, and security research

Signature Components

Component Purpose Example
Signature ID (SID) Unique identifier for the signature SID: 2100498
Message Human-readable description of the attack "SQL Injection - UNION SELECT"
Protocol Network protocol being monitored TCP, UDP, ICMP, HTTP
Source/Destination IP addresses and ports to match any -> $HOME_NET:80
Content Patterns Byte patterns or strings to detect content:"UNION"; content:"SELECT";
Modifiers Additional matching conditions nocase, offset, depth, distance
Classification Attack category web-application-attack, trojan-activity
Example: Snort Signature for SQL Injection
alert tcp any any -> $HOME_NET 80 (
    msg:"WEB-ATTACKS SQL injection - UNION SELECT detected";
    flow:to_server,established;
    content:"UNION"; nocase; http_uri;
    content:"SELECT"; nocase; http_uri; distance:0;
    classtype:web-application-attack;
    sid:2100498; rev:3;
    reference:url,owasp.org/www-community/attacks/SQL_Injection;
)

# Breakdown:
# - alert tcp: Generate alert for TCP traffic
# - any any -> $HOME_NET 80: From any source to web servers (port 80)
# - flow:to_server,established: Only inspect established inbound connections
# - content:"UNION"; nocase; http_uri: Look for "UNION" (case-insensitive) in URI
# - content:"SELECT"; distance:0: "SELECT" must follow immediately
# - classtype: Categorizes as web application attack
# - sid:2100498: Unique signature identifier

Advanced Signature Techniques

Regular Expressions (PCRE): For complex pattern matching:

Example: PCRE for Email Validation in HTTP Traffic
alert tcp any any -> $HOME_NET 80 (
    msg:"Potential email harvesting detected";
    pcre:"/[\w\.-]+@[\w\.-]+\.\w{2,4}/i";
    classtype:policy-violation;
    sid:3000001;
)

# Matches email patterns like: user@example.com, admin@mail.domain.co.uk

Byte Jump and Byte Extract: For analyzing protocol structures:

Example: Detecting Malformed DNS Packets
alert udp any any -> any 53 (
    msg:"DNS query with suspicious length field";
    content:"|00 00 01|"; offset:4; depth:3;  # Standard query
    byte_extract:2,0,query_len,relative;        # Extract query length
    byte_test:2,>,512,0,relative;               # Test if > 512 bytes (DNS max)
    classtype:protocol-command-decode;
    sid:3000002;
)

Strengths of Signature-Based Detection

  • High Accuracy: >99% true positive rate for known attacks when signatures are well-tuned
  • Low False Positives: Typically <1% false positive rate in production environments
  • Fast Processing: Optimized algorithms process signatures in <1ms per packet
  • Specific Attribution: Alerts provide exact attack type, CVE reference, and remediation guidance
  • Compliance Friendly: Provides clear audit trail of detected threats
  • Actionable Intelligence: Security teams immediately understand what was detected and why

Limitations of Signature-Based Detection

  • Zero-Day Blind Spot: Cannot detect previously unknown attacks (no signature exists)
  • Signature Lag: Gap between vulnerability disclosure and signature availability (typically 24-72 hours)
  • Evasion Susceptibility: Attackers can obfuscate payloads (encoding, encryption, polymorphism) to evade signatures
  • Signature Bloat: Enterprise deployments may maintain 50,000-100,000+ signatures, impacting performance
  • Maintenance Overhead: Requires continuous signature updates and tuning to reduce false positives
  • Variant Vulnerability: Slight modifications to attacks can evade existing signatures

2.2 Anomaly-Based Detection

Anomaly-based detection identifies threats by recognizing deviations from established baselines of normal behavior. Unlike signature-based detection, this approach can detect novel attacks, insider threats, and zero-day exploits.

Anomaly Detection Process

Phase 1: Baseline Establishment (Training Period)

  • Duration: Typically 30-90 days in production environment
  • Data Collection: Monitor network traffic, user behavior, system activities
  • Feature Extraction: Calculate statistical properties (mean, standard deviation, distribution)
  • Profile Creation: Build models representing "normal" for networks, hosts, users, applications

Phase 2: Anomaly Scoring (Detection)

  • Real-Time Comparison: Compare current activity against baseline profile
  • Statistical Analysis: Calculate anomaly scores using statistical or ML models
  • Threshold Evaluation: Alert when anomaly score exceeds configured threshold
  • Contextual Analysis: Consider time of day, day of week, seasonal variations

Phase 3: Continuous Learning (Adaptation)

  • Baseline Updates: Periodically retrain models to adapt to legitimate changes
  • Feedback Loop: Incorporate analyst feedback on false positives/negatives
  • Drift Detection: Identify gradual changes in baseline behavior

Anomaly Detection Features

Feature Category Metrics Anomaly Indicators
Network Traffic Packets/sec, bytes/sec, connection rate, protocol distribution Sudden traffic spikes (DDoS), unusual protocol usage, abnormal packet sizes
Connection Patterns Connection duration, inter-arrival times, destination diversity Scanning behavior, beaconing (C2 callbacks), lateral movement
Application Behavior API calls, resource consumption, data access patterns Privilege escalation, data exfiltration, resource exhaustion
User Activity Login times, accessed resources, data transfer volume Account compromise, insider threats, credential stuffing
Time-Series Historical trends, seasonal patterns, cyclical behavior Off-hours activity, unusual access times, broken patterns

Statistical Methods for Anomaly Detection

Standard Deviation Method

Example: Detecting Traffic Anomalies
# Calculate baseline statistics
mean_traffic = average(historical_traffic_volume)
std_dev = standard_deviation(historical_traffic_volume)

# Define anomaly threshold (e.g., 3 standard deviations)
threshold_high = mean_traffic + (3 * std_dev)
threshold_low = mean_traffic - (3 * std_dev)

# Current traffic analysis
if current_traffic > threshold_high:
    alert("Abnormally high traffic detected - potential DDoS")
    anomaly_score = (current_traffic - mean_traffic) / std_dev
elif current_traffic < threshold_low:
    alert("Abnormally low traffic - potential network issue")

# Example values:
# Historical average: 10,000 packets/sec
# Standard deviation: 2,000 packets/sec
# Threshold high: 16,000 packets/sec (3σ)
# Current traffic: 25,000 packets/sec
# Anomaly score: 7.5σ (very high anomaly)

Clustering-Based Anomaly Detection

Example: K-Means Clustering for Network Flows
import numpy as np
from sklearn.cluster import KMeans

# Feature vector for each network flow
# [duration, bytes_sent, bytes_received, packets, unique_ports]
flows = [
    [10, 1500, 3000, 50, 1],      # Normal web browsing
    [120, 500000, 2000, 1000, 1], # Large upload (normal backup)
    [5, 100, 100, 10, 1],         # Normal API call
    [3600, 50, 1000000, 2000, 1], # ANOMALY: Large download
    [1, 50, 50, 1, 100]           # ANOMALY: Port scanning
]

# Train clustering model on normal traffic
kmeans = KMeans(n_clusters=3)
kmeans.fit(normal_flows)

# Calculate distance to nearest cluster for new flow
distance = kmeans.transform([new_flow])[0].min()

if distance > threshold:
    alert(f"Anomalous flow detected - distance: {distance}")

Strengths of Anomaly-Based Detection

  • Zero-Day Detection: Can identify previously unknown attacks
  • Insider Threat Detection: Detects malicious actions by authorized users
  • Adaptive: Automatically learns normal behavior for each environment
  • Policy Violation Detection: Identifies inappropriate use of systems
  • No Signature Updates: Doesn't rely on continuously updating signature databases

Limitations of Anomaly-Based Detection

  • High False Positive Rate: Typically 5-15% false positive rate, requiring significant analyst time
  • Training Requirements: Requires 30-90 days of clean data to establish accurate baseline
  • Baseline Pollution: If attackers are present during training, malicious behavior becomes "normal"
  • Concept Drift: Legitimate changes in network/application behavior can trigger false alerts
  • Explainability Gap: Difficult to explain why specific behavior was flagged as anomalous
  • Threshold Tuning: Balancing sensitivity (detecting threats) vs. specificity (avoiding false alarms) is challenging

2.3 Protocol Analysis

Protocol analysis—also called stateful protocol analysis or deep packet inspection (DPI)—validates network traffic against protocol specifications defined in RFCs. This method detects attacks that exploit protocol weaknesses, violations of protocol standards, or evasion techniques.

Protocol Layers Analyzed

Layer Protocols Analysis Performed
Layer 2 (Data Link) Ethernet, ARP, VLAN MAC address validation, ARP spoofing detection, VLAN hopping
Layer 3 (Network) IPv4, IPv6, ICMP IP header validation, fragmentation attacks, ICMP tunneling
Layer 4 (Transport) TCP, UDP, SCTP TCP state tracking, SYN floods, session hijacking, UDP floods
Layer 7 (Application) HTTP, DNS, TLS, FTP, SMTP, SMB Request/response validation, command injection, buffer overflows

TCP State Tracking

IDS maintains state machines for TCP connections to detect protocol violations:

TCP Connection State Machine
State: CLOSED
  ↓ (SYN received from client)
State: SYN_RECEIVED
  ↓ (SYN-ACK sent, ACK received)
State: ESTABLISHED
  ↓ (Data exchange)
State: ESTABLISHED
  ↓ (FIN received)
State: FIN_WAIT
  ↓ (FIN-ACK exchange complete)
State: CLOSED

# Violations Detected:
# - SYN-FIN packet (contradictory flags)
# - Data before handshake completion
# - ACK without prior SYN
# - Invalid sequence numbers
# - Connection reset attacks

HTTP Protocol Analysis

Example: HTTP Request Validation
# Valid HTTP Request
GET /api/users?id=123 HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Accept: application/json

# Violations Detected by IDS:
1. Missing Host header (required in HTTP/1.1)
2. Invalid HTTP method (e.g., "G3T" instead of "GET")
3. Oversized URI (> 8KB may indicate buffer overflow attempt)
4. Malformed headers (missing colon separator)
5. Multiple Host headers (HTTP request smuggling)
6. Invalid characters in URI (null bytes, control characters)
7. HTTP 0.9 request on HTTPS port (protocol downgrade)

# Example Detection:
alert tcp any any -> $HTTP_SERVERS 80 (
    msg:"HTTP request with multiple Host headers";
    flow:to_server,established;
    content:"Host|3a|"; http_header; offset:0;
    content:"Host|3a|"; http_header; distance:1;
    classtype:protocol-command-decode;
    sid:3000010;
)

DNS Protocol Analysis

Violation Attack Type Detection Method
Query length > 255 characters DNS tunneling, data exfiltration Length field validation
Excessive subdomain levels (>10) DNS tunneling Domain structure analysis
High entropy in domain name DGA (Domain Generation Algorithm) Statistical entropy calculation
TXT record > 255 bytes Data exfiltration via DNS Record size validation
Unusual query types (NULL, ANY) DNS amplification attack Query type frequency analysis

Evasion Detection

Protocol analysis identifies techniques attackers use to evade detection:

1. Packet Fragmentation

# Attack: Split malicious payload across multiple IP fragments
# to evade single-packet signature matching

Fragment 1: "UNION "
Fragment 2: "SELECT "
Fragment 3: "password FROM users"

# IDS Defense: IP reassembly before signature matching
# Snort preprocessor: frag3 reassembles fragments before detection engine

2. Encoding and Obfuscation

# Attack: URL encoding to bypass signature
Original:  /admin.php?cmd=cat /etc/passwd
Encoded:   /admin.php?cmd=cat%20%2Fetc%2Fpasswd
Double:    /admin.php?cmd=cat%2520%252Fetc%252Fpasswd
Unicode:   /admin.php?cmd=cat%20%u002f%u0065%u0074%u0063%u002f...

# IDS Defense: HTTP normalization preprocessor
# Decodes URL encoding, Unicode, and other obfuscation before matching

3. Session Splicing (Timing Attack)

# Attack: Insert delays between packets to evade stateful analysis
# Send "SELECT" at time T
# Wait 300 seconds (beyond IDS timeout)
# Send "* FROM users" at time T+300

# IDS Defense: Extended session tracking with configurable timeouts
# Maintain connection state for duration of session, not fixed time window

2.4 Hybrid Detection Approaches

Modern IDS/IPS systems combine multiple detection methodologies to maximize coverage while managing false positive rates:

Threat Type Primary Method Secondary Method Rationale
Known exploits (CVE-based) Signature-based Protocol analysis High confidence, low false positives
Zero-day attacks Anomaly-based Protocol analysis Detect novel attack patterns
Insider threats Anomaly-based Behavioral analysis Detect policy violations, data exfiltration
Protocol attacks (fuzzing) Protocol analysis Anomaly-based Validate protocol compliance
Evasion techniques Protocol analysis Signature-based Normalize traffic before matching

Layered Detection Example

Multi-Method Detection of SQL Injection
# Layer 1: Signature-Based Detection
# Fast, high-confidence detection of known SQL injection patterns
alert tcp any any -> $WEB_SERVERS 80 (
    msg:"SQL Injection - UNION SELECT";
    content:"UNION"; nocase; http_uri;
    content:"SELECT"; nocase; http_uri;
    sid:1000001;
)

# Layer 2: Protocol Analysis
# Detect protocol violations in HTTP requests
alert tcp any any -> $WEB_SERVERS 80 (
    msg:"HTTP URI exceeds maximum length";
    urilen:>8192;
    classtype:protocol-command-decode;
    sid:1000002;
)

# Layer 3: Anomaly Detection
# Detect unusual database query patterns
if (sql_query_rate > baseline_mean + 3*std_dev):
    alert("Abnormal database query rate detected")

if (parameter_length > typical_max_length):
    alert("Unusually long query parameter")

# Layer 4: Behavioral Analysis
# User typically accesses 5-10 records, suddenly accessing 10,000
if (user_data_access > user_baseline * 10):
    alert("Potential data exfiltration - abnormal access volume")

2.5 Performance Optimization

Balancing detection coverage with system performance requires optimization strategies:

Signature Optimization

  • Rule Ordering: Place frequently-matched signatures earlier in evaluation order
  • Fast Pattern Selection: Optimize content patterns for Aho-Corasick algorithm efficiency
  • Rule Pruning: Disable signatures irrelevant to environment (Windows rules on Linux network)
  • Threshold Tuning: Adjust detection thresholds to reduce false positive processing

Hardware Acceleration

  • Multi-Core Processing: Distribute packet analysis across CPU cores
  • GPU Acceleration: Offload pattern matching to graphics processors
  • FPGA/ASIC: Hardware-based signature matching for 100Gbps+ throughput
  • DMA (Direct Memory Access): Reduce CPU overhead in packet capture

2.6 Chapter Summary

  • Signature-based detection provides high accuracy (>99%) for known attacks but cannot detect zero-days
  • Anomaly-based detection identifies novel attacks and insider threats but has higher false positive rates (5-15%)
  • Protocol analysis validates traffic against RFCs, detecting evasion techniques and protocol violations
  • Hybrid approaches combine multiple methodologies to maximize detection coverage
  • Modern IDS uses multi-pattern matching (Aho-Corasick), regular expressions (PCRE), and statistical analysis
  • Protocol reassembly and normalization prevent evasion via fragmentation, encoding, and timing attacks
  • Performance optimization through rule tuning, hardware acceleration, and intelligent rule ordering is essential
