Chapter 7: Incident Response

Chapter Overview

This chapter explores how IDS/IPS integrates with incident response processes. We'll examine incident response frameworks, alert triage, investigation workflows, forensic analysis, containment strategies, remediation, and post-incident review to build effective security operations capabilities.

7.1 Incident Response Lifecycle

IDS/IPS plays a critical role throughout the incident response lifecycle defined by NIST SP 800-61:

Phase IDS/IPS Role Key Activities
1. Preparation Deploy and tune detection systems Sensor placement, signature updates, baseline creation, playbook development
2. Detection & Analysis Generate alerts on security events Alert generation, triage, validation, impact assessment
3. Containment Block malicious traffic (IPS mode) IP blocking, session termination, network isolation
4. Eradication Verify threat removal Confirm no ongoing malicious activity detected
5. Recovery Monitor for reinfection Enhanced monitoring of affected systems/networks
6. Post-Incident Improve detection capabilities New signatures, rule tuning, gap analysis

7.2 Alert Triage Process

Effective triage prioritizes critical threats from thousands of daily alerts:

Triage Workflow

SOC Alert Triage Decision Tree
STEP 1: Initial Classification
├─ Critical (P1): Active attack on critical system → Immediate escalation
├─ High (P2): Confirmed threat, non-critical system → Investigate within 1 hour
├─ Medium (P3): Suspicious activity, requires validation → Investigate within 4 hours
└─ Low (P4): Informational, policy violation → Queue for review

STEP 2: Enrichment
├─ Threat Intel: Is source IP malicious? (VirusTotal, OTX)
├─ Asset Context: What system was targeted? (CMDB)
├─ User Context: Is this normal for this user? (AD, IAM)
└─ Historical: Have we seen this before? (SIEM history)

STEP 3: Validation
├─ True Positive: Confirmed malicious activity → ESCALATE
├─ False Positive: Benign traffic misidentified → TUNE RULE
└─ Benign Suspicious: Legitimate but unusual → DOCUMENT

STEP 4: Severity Adjustment
Risk_Score = Base_Severity × Asset_Criticality × Threat_Confidence

Example:
Alert: SQL Injection (Severity: High = 8/10)
Target: Production DB (Criticality: Critical = 10/10)
Source IP: 15/90 VT detections (Confidence: Medium = 0.6)

Risk_Score = 8 × 10 × 0.6 = 48/100

If Risk_Score >= 70 → P1
If Risk_Score >= 50 → P2
If Risk_Score >= 30 → P3
Else → P4

Triage Metrics

Metric Definition Target
Mean Time to Detect (MTTD) Time from attack start to IDS alert <5 minutes
Mean Time to Triage (MTTT) Time from alert to initial classification <15 minutes
Mean Time to Respond (MTTR) Time from detection to containment action <1 hour (critical)
Alert-to-Incident Ratio Percentage of alerts that become incidents 5-10%
False Positive Rate Percentage of alerts that are false positives <5%

7.3 Investigation Workflow

Phase 1: Scoping

Investigation Checklist
# Initial Questions to Answer:
1. What happened?
   - Attack type, signature ID, MITRE ATT&CK technique
   - Timestamp of first and last detection

2. Who is affected?
   - Source IP/hostname (attacker)
   - Destination IP/hostname (victim)
   - User accounts involved

3. When did it happen?
   - First occurrence timestamp
   - Duration of attack
   - Frequency (single event vs. campaign)

4. How did it happen?
   - Attack vector (web, email, network, endpoint)
   - Vulnerability exploited (CVE-ID if applicable)
   - Evasion techniques used

5. Why was it successful?
   - Missing patches?
   - Misconfiguration?
   - Zero-day exploit?
   - Social engineering?

6. What is the impact?
   - Data accessed/exfiltrated?
   - Systems compromised?
   - Business operations affected?

Phase 2: Evidence Collection

  • IDS Alerts: Full alert details with signature, classification, priority
  • Packet Captures: PCAP files of suspicious traffic for deep analysis
  • Network Flows: NetFlow/IPFIX data showing connection patterns
  • System Logs: Authentication, application, system event logs
  • Endpoint Forensics: Memory dumps, disk images, process listings
  • Firewall Logs: Allowed/blocked connections
  • Proxy/Web Gateway: HTTP traffic, URLs visited
  • Email Gateway: Phishing emails, attachments
Example: Extracting PCAP from IDS Alert
#!/bin/bash
# Extract full packet capture for investigation

# From Suricata eve.json alert
ALERT_ID="12345678"
SRC_IP="192.168.1.100"
DST_IP="10.0.1.50"
TIMESTAMP="2025-01-15T14:23:15"

# Extract 5 minutes of traffic around alert time
START_TIME=$(date -d "${TIMESTAMP} -2 minutes" +"%Y-%m-%d %H:%M:%S")
END_TIME=$(date -d "${TIMESTAMP} +3 minutes" +"%Y-%m-%d %H:%M:%S")

# Use tcpdump to extract from full capture
tcpdump -r /var/log/suricata/full_capture.pcap \
    -w /forensics/incident_${ALERT_ID}.pcap \
    "((host ${SRC_IP} and host ${DST_IP}) and \
     (time >= '${START_TIME}' and time <= '${END_TIME}'))"

# Analyze in Wireshark or NetworkMiner
wireshark /forensics/incident_${ALERT_ID}.pcap &

# Export HTTP objects for malware analysis
tshark -r /forensics/incident_${ALERT_ID}.pcap \
    --export-objects http,/forensics/http_objects/

7.4 Containment Strategies

Short-Term Containment

Immediate actions to stop active attack:

Action Method Use Case
IP Blocking Firewall ACL, IPS block, null route Block external attacker IP addresses
DNS Sinkhole Redirect malicious domains to sinkhole Block C2 communications, malware downloads
Session Termination IPS sends TCP RST to kill connections Terminate active attack sessions
Account Disable Active Directory account lockout Compromised user credentials
Network Isolation VLAN isolation, EDR host containment Prevent lateral movement from compromised host
Service Shutdown Stop vulnerable service/application Active exploitation of critical vulnerability
Example: Automated Containment via SOAR
#!/usr/bin/env python3
# SOAR playbook: Contain malicious IP

import requests

# Alert details from IDS
malicious_ip = "185.x.x.x"
alert_severity = "critical"

# 1. Block at firewall
firewall_api = "https://firewall.company.com/api/v1/block"
response = requests.post(
    firewall_api,
    headers={"Authorization": "Bearer TOKEN"},
    json={"ip": malicious_ip, "duration": 86400}  # 24 hours
)

# 2. Add to threat intelligence
ti_api = "https://threatintel.company.com/api/v1/iocs"
requests.post(
    ti_api,
    json={
        "indicator": malicious_ip,
        "type": "ipv4-addr",
        "threat_type": "malicious_ip",
        "confidence": 90,
        "tags": ["intrusion_attempt", "sql_injection"]
    }
)

# 3. Create incident ticket
servicenow_api = "https://servicenow.company.com/api/now/table/incident"
requests.post(
    servicenow_api,
    auth=("admin", "password"),
    json={
        "short_description": f"IDS Alert: Attack from {malicious_ip}",
        "urgency": 1,
        "impact": 1,
        "assignment_group": "SOC Team"
    }
)

# 4. Notify SOC via Slack
slack_webhook = "https://hooks.slack.com/services/XXX/YYY/ZZZ"
requests.post(
    slack_webhook,
    json={
        "text": f"🚨 CRITICAL: Active attack from {malicious_ip} blocked at firewall"
    }
)

print(f"Containment actions completed for {malicious_ip}")

Long-Term Containment

  • Patch Deployment: Emergency patching of exploited vulnerabilities
  • Configuration Hardening: Disable unnecessary services, close ports
  • Segmentation: Implement micro-segmentation to limit blast radius
  • Access Control: Enforce least privilege, MFA for sensitive systems
  • Enhanced Monitoring: Deploy additional sensors on critical assets

7.5 Eradication and Recovery

Eradication Steps

  1. Malware Removal: Delete malicious files, remove persistence mechanisms
  2. Account Cleanup: Reset compromised credentials, revoke unauthorized access
  3. Backdoor Removal: Eliminate attacker-created accounts, SSH keys, firewall rules
  4. Vulnerability Patching: Apply security updates to prevent reinfection
  5. Configuration Restoration: Restore security settings to known-good state

Recovery Validation

Post-Eradication Verification Checklist
✓ IDS/IPS Monitoring
  - No alerts from previously affected systems (24-48 hours)
  - No connections to known-bad IPs/domains
  - No suspicious outbound traffic patterns

✓ Endpoint Verification
  - EDR shows no suspicious processes
  - File integrity scan shows no unauthorized changes
  - No scheduled tasks or startup items added
  - Registry monitored for changes

✓ Network Verification
  - No unexpected internal connections (lateral movement)
  - No DNS queries to suspicious domains
  - Bandwidth usage returned to baseline

✓ Authentication Verification
  - No failed login attempts from known attackers
  - All user accounts accounted for (no rogue accounts)
  - Privileged access logs show only authorized activity

✓ Data Integrity
  - Database integrity checks passed
  - No evidence of data modification or deletion
  - Backup verification (clean backups available)

7.6 Post-Incident Activities

Lessons Learned

Conduct post-incident review within 7 days of incident closure:

Topic Questions to Answer
Detection How was the incident detected? Could it have been detected earlier? Were IDS signatures effective?
Response Time What was MTTD, MTTT, MTTR? Were SLAs met? What caused delays?
Containment Were containment actions effective? Did any actions cause collateral damage?
Root Cause What vulnerability was exploited? Why did existing controls fail?
Improvements What signatures/rules should be added? What processes need improvement?

IDS Tuning Based on Incidents

Post-Incident IDS Improvements
# Incident: SQL Injection not detected by IDS

Root Cause Analysis:
- Attack used double URL encoding to evade signature
- IDS HTTP normalizer only decoded once
- Signature matched "UNION SELECT" but encoded payload was "%2555NION%2553ELECT"

Corrective Actions:

1. Update HTTP Preprocessor (Snort)
preprocessor http_inspect_server: \
    double_decode,           # ← NEW: Decode twice
    u_encode,                # URL encoding
    bare_byte,               # Bare byte encoding
    normalize_utf,           # UTF-8 normalization
    multi_slash,             # Multiple slashes
    iis_unicode,             # IIS Unicode
    apache_whitespace        # Apache whitespace

2. Create New Signature
alert tcp any any -> $HTTP_SERVERS $HTTP_PORTS (
    msg:"SQL Injection - Double Encoded UNION SELECT";
    flow:to_server,established;
    urilen:>100;                         # Unusually long URI
    pcre:"/%(?:25)+[0-9A-F]{2}/i";      # Double+ URL encoding pattern
    content:"UNION"; http_uri; nocase;
    content:"SELECT"; http_uri; nocase; distance:0;
    classtype:web-application-attack;
    sid:9000001; rev:1;
)

3. Implement Regression Testing
# Test new signature against historical PCAP
snort -c /etc/snort/snort.conf -r /forensics/incident_12345.pcap -A console
# Verify alert is now generated

Documentation

Maintain comprehensive incident records:

  • Incident Report: Executive summary, timeline, impact, resolution
  • Technical Analysis: Detailed forensics, IOCs, TTPs (MITRE ATT&CK mapping)
  • Evidence Archive: PCAPs, logs, memory dumps, screenshots
  • Action Items: Remediation tasks, signature updates, policy changes
  • Metrics: MTTD, MTTT, MTTR, financial impact

7.7 Incident Response Metrics

Metric Formula Industry Benchmark
Mean Time to Detect (MTTD) Avg(Incident_Detection_Time - Incident_Start_Time) <24 hours (IBM: 207 days avg for breaches)
Mean Time to Respond (MTTR) Avg(Containment_Time - Detection_Time) <1 hour (critical), <4 hours (high)
Mean Time to Resolve (MTTResolve) Avg(Incident_Close_Time - Incident_Start_Time) <72 hours (depends on severity)
Incident Volume Count of confirmed incidents per month Varies by org size; track trends
Containment Effectiveness (Incidents Contained / Total Incidents) × 100% >95%

7.8 Chapter Summary

  • IDS/IPS supports all six phases of the NIST incident response lifecycle
  • Effective alert triage uses enrichment, validation, and risk scoring to prioritize threats
  • Investigation requires scoping, evidence collection, and timeline reconstruction
  • Containment strategies include IP blocking, network isolation, and account disable
  • Eradication removes malware, backdoors, and patches vulnerabilities
  • Post-incident activities include lessons learned, IDS tuning, and documentation
  • Key metrics: MTTD, MTTT, MTTR, false positive rate, containment effectiveness
  • Continuous improvement through signature updates, rule tuning, and process refinement
