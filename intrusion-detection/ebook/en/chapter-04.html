Chapter 4: Host-Based IDS

Chapter Overview

This chapter examines Host-Based Intrusion Detection Systems (HIDS), which monitor individual systems for malicious activity. We'll explore file integrity monitoring, log analysis, process monitoring, rootkit detection, registry monitoring, and integration with endpoint security platforms.

4.1 HIDS vs NIDS: Complementary Approaches

While NIDS monitors network traffic, HIDS provides visibility into system-level activities that network monitoring cannot detect:

Aspect NIDS HIDS
Deployment Network chokepoints (perimeter, segments) Individual hosts (servers, workstations)
Visibility Network traffic, connections, protocols File changes, processes, logs, registry
Encrypted Traffic Cannot inspect encrypted payloads Sees traffic before encryption/after decryption
Local Attacks Cannot detect (no network traffic) Detects local privilege escalation, malware
Resource Impact Centralized, minimal per-host overhead Agent on each host consumes CPU/memory
Insider Threats Limited detection of authorized activities Monitors all user actions, data access
Compliance Network-level logging requirements System-level audit trails (PCI DSS, HIPAA)

4.2 File Integrity Monitoring (FIM)

FIM detects unauthorized changes to critical system files, configuration files, and application binaries by comparing current file states against known-good baselines:

FIM Workflow

  1. Baseline Creation: Calculate cryptographic hashes (SHA-256) of all monitored files during known-clean state
  2. Periodic Scanning: Recalculate hashes at regular intervals (hourly, daily) or in real-time using filesystem hooks
  3. Change Detection: Compare current hashes against baseline; flag any discrepancies
  4. Alert Generation: Report file changes with details: filename, timestamp, old/new hash, user responsible
  5. Baseline Updates: After authorized changes, update baseline to reflect legitimate modifications

Critical Files to Monitor

Platform File Categories Examples
Linux System binaries, configs, cron jobs, SSH keys /bin/, /sbin/, /etc/, /root/.ssh/, /var/spool/cron/
Windows System32, drivers, registry hives, startup items C:\Windows\System32\, C:\Windows\SysWOW64\, Registry
Web Server Web root, CGI scripts, server configs /var/www/, /etc/apache2/, IIS configuration
Database Data files, configs, stored procedures MySQL data/, PostgreSQL conf/, MSSQL databases
Application Binaries, libraries, config files /opt/app/, C:\Program Files\, .jar/.dll files

Real-Time vs Scheduled FIM

Approach Mechanism Pros Cons
Real-Time Filesystem hooks (inotify, FSEvents, NTFS USN) Immediate detection, no scan window Higher CPU usage, kernel module required
Scheduled Periodic hash calculation (cron/Task Scheduler) Lower resource usage, simpler deployment Detection delay (hours), scan performance impact

4.4 Process and Behavior Monitoring

HIDS monitors running processes for suspicious behavior patterns:

Process Attributes Monitored

Attribute Normal Behavior Suspicious Indicators
Process Name Known system/application processes Misspelled names (svch0st.exe vs svchost.exe)
Path Standard locations (C:\Windows\System32\) Unexpected paths (C:\Temp\, user Downloads/)
Parent-Child Expected relationships (explorer.exe → notepad.exe) Unusual spawning (cmd.exe → powershell.exe → WMI)
Command Line Typical arguments for each process Encoded commands, obfuscation, suspicious flags
Network Activity Expected connections (browser → Internet) Unexpected (calc.exe opening network connections)
Unsigned Code Digitally signed binaries Unsigned or self-signed executables
Example: Sysmon Process Monitoring (Windows)

      powershell.exe -NoP -NonI -W Hidden -Enc JABzAD0ATgBlAHcALQBPAGIAagBlAGMAdAA...

HIDS Alert Triggers:
✗ PowerShell with -Encoded flag (Base64 obfuscation)
✗ Hidden window (-W Hidden)
✗ Non-interactive (-NonI)
✗ Parent process is cmd.exe (not typical user launch)

Classification: Code execution via encoded PowerShell (T1059.001 - MITRE ATT&CK)
Action: Terminate process, quarantine host, alert SOC

Behavioral Baselines

HIDS establishes baselines for normal system behavior to detect anomalies:

  • Process Execution Patterns: Which processes typically run, when, and by whom
  • Resource Consumption: Normal CPU, memory, disk, network usage per process
  • File Access Patterns: Which users/processes access which files
  • Network Connections: Expected outbound destinations and ports
  • User Behavior: Login times, accessed resources, typical activities

4.5 Rootkit Detection

Rootkits hide malicious activity by manipulating system calls, kernel modules, or filesystem views. HIDS detects rootkits through cross-view analysis:

Detection Techniques

Technique Method What It Detects
Signature Scanning Compare binaries against known rootkit signatures Known rootkit families (Alureon, Rustock)
Integrity Checking Verify system binaries match known-good hashes Modified system files (trojanized ls, ps, netstat)
Cross-View Analysis Compare kernel-level vs user-space file listings Hidden files/processes visible to kernel but not users
Memory Analysis Scan memory for hidden processes/modules Direct Kernel Object Manipulation (DKOM) rootkits
System Call Hooking Detect modifications to syscall table Kernel-level API hooks used by rootkits
Example: chkrootkit Detection
#!/bin/bash
# chkrootkit - Check for rootkit signatures

# Check if common binaries are trojanized
for binary in ls ps netstat ifconfig who; do
    if [ -f /bin/$binary ]; then
        strings /bin/$binary | grep -q "INFECTED_SIGNATURE"
        if [ $? -eq 0 ]; then
            echo "WARNING: $binary appears to be trojanized!"
        fi
    fi
done

# Check for hidden processes (cross-view)
# Compare /proc listing with ps output
proc_count=$(ls -1 /proc | grep -c "^[0-9]")
ps_count=$(ps aux | wc -l)

if [ $proc_count -gt $((ps_count + 10)) ]; then
    echo "WARNING: Hidden processes detected (proc: $proc_count, ps: $ps_count)"
fi

# Check for kernel module rootkits
lsmod | grep -E "hide|rootkit|backdoor" && echo "WARNING: Suspicious kernel module"

# Verify system binaries integrity
md5sum /bin/ls | grep -q "a1b2c3d4e5f6..."  # Known-good hash
[ $? -ne 0 ] && echo "WARNING: /bin/ls has been modified"

4.8 Deployment Architecture

Centralized HIDS Architecture
                [HIDS Management Server]
                    |  (Receives alerts, manages agents)
                    |
      +-------------+-----------------+
      |             |                 |
   [Agent]       [Agent]           [Agent]
  Web Server   DB Server      Workstation
      |             |                 |
   - FIM         - FIM             - FIM
   - Logs        - Logs            - Logs
   - Process     - Process         - Process
   - Registry    - Registry        - Registry

Communication:
- Agents report to central server via TLS (port 1514)
- Server aggregates alerts, applies correlation rules
- Centralized signature/rule updates pushed to agents
- Syslog export to SIEM for broader correlation

Scalability:
- Single server: 1,000-5,000 agents
- Clustered: 10,000+ agents
- Agent resource usage: 50-200MB RAM, <5% CPU

Agent Deployment Best Practices

  • Prioritize Critical Systems: Deploy first on domain controllers, database servers, web servers
  • Baseline in Production: Establish baseline during normal operations before alerting
  • Tune for Environment: Disable checks not applicable (Windows rules on Linux)
  • Centralize Log Collection: Prevent local log tampering by attackers
  • Monitor Agent Health: Alert on agent disconnections or failures
  • Secure Agent Communication: Use TLS and agent authentication

4.10 Chapter Summary

  • HIDS provides system-level visibility that complements network-based detection
  • File Integrity Monitoring (FIM) detects unauthorized changes to critical files using cryptographic hashing
  • Log analysis monitors authentication, privilege escalation, and application security events
  • Process monitoring detects suspicious execution patterns and behavioral anomalies
  • Rootkit detection uses cross-view analysis and memory forensics to find hidden threats
  • Windows registry monitoring detects persistence mechanisms and policy changes
  • Popular platforms include OSSEC, Wazuh, Tripwire, and AIDE
  • Centralized architecture enables management of thousands of agents from single console
  • Integration with EDR, SIEM, and SOAR enhances detection and response capabilities
