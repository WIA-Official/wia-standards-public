제3장: 네트워크 기반 IDS (NIDS)

3.1 NIDS 개요

네트워크 기반 침입 탐지 시스템(Network-based Intrusion Detection System, NIDS)은 네트워크 세그먼트의 트래픽을 실시간으로 모니터링하여 악의적인 활동이나 정책 위반을 탐지합니다. NIDS는 네트워크 인프라의 전략적 지점에 배치되어 모든 호스트의 트래픽을 중앙에서 감시할 수 있다는 장점이 있습니다.

3.1.1 NIDS vs HIDS 비교

특성 NIDS (네트워크 기반) HIDS (호스트 기반)
모니터링 범위 전체 네트워크 세그먼트 단일 호스트
배치 위치 네트워크 인프라 (스위치, 라우터) 개별 서버/워크스테이션
탐지 대상 네트워크 공격, 스캔, DDoS 로컬 익스플로잇, 파일 변조
암호화 트래픽 분석 어려움 복호화 후 분석 가능
성능 영향 호스트에 영향 없음 호스트 리소스 사용
관리 복잡도 중앙 집중 관리 각 호스트별 관리
스위치드 네트워크 SPAN/TAP 필요 영향 없음
비용 초기 비용 높음, 운영 효율적 확장 시 비용 증가

3.2 NIDS 아키텍처

NIDS의 효과적인 운영을 위해서는 적절한 아키텍처 설계가 필수적입니다. 주요 구성 요소들이 유기적으로 연동되어 고성능 탐지를 제공해야 합니다.

3.2.2 고성능 아키텍처 패턴

  • 파이프라인 아키텍처: 각 처리 단계를 독립적인 스레드로 분리하여 병렬 처리
  • 워커 풀 패턴: 여러 워커 스레드가 패킷 큐에서 작업을 가져와 처리
  • 분산 처리: 센서를 분산 배치하고 중앙 관리 서버에서 통합 분석
  • 하드웨어 오프로딩: 패킷 필터링, 체크섬 검증 등을 NIC에서 처리

3.3 패킷 캡처 메커니즘

NIDS의 첫 번째 단계는 네트워크 트래픽을 효율적으로 캡처하는 것입니다. 패킷 손실 없이 고속 트래픽을 처리하기 위해 다양한 기술이 사용됩니다.

3.3.1 패킷 캡처 기술

기술 성능 장점 단점
libpcap/WinPcap ~1 Gbps 표준, 이식성 높음 커널-유저 컨텍스트 스위칭 오버헤드
PF_RING ~10 Gbps 제로카피, 멀티코어 지원 Linux 전용
AF_PACKET (PACKET_MMAP) ~10 Gbps 커널 내장, 메모리 맵 설정 복잡
DPDK (Data Plane Dev Kit) 40+ Gbps 커널 우회, 초고속 전용 NIC, 높은 복잡도
Netmap ~14 Gbps 간단한 API, 효율적 제한적인 하드웨어 지원

// 패킷 처리 콜백 함수
void packet_handler(u_char *user_data,
                   const struct pcap_pkthdr *pkthdr,
                   const u_char *packet) {
    // 이더넷 헤더 파싱
    struct ether_header *eth = (struct ether_header *)packet;

    // IP 헤더 파싱 (IPv4)
    if (ntohs(eth->ether_type) == ETHERTYPE_IP) {
        struct ip *ip_hdr = (struct ip *)(packet + sizeof(struct ether_header));
        char src_ip[INET_ADDRSTRLEN];
        char dst_ip[INET_ADDRSTRLEN];

        inet_ntop(AF_INET, &(ip_hdr->ip_src), src_ip, INET_ADDRSTRLEN);
        inet_ntop(AF_INET, &(ip_hdr->ip_dst), dst_ip, INET_ADDRSTRLEN);

        printf("패킷: %s -> %s, 길이: %d bytes\n",
               src_ip, dst_ip, pkthdr->len);

        // TCP 패킷인 경우
        if (ip_hdr->ip_p == IPPROTO_TCP) {
            struct tcphdr *tcp = (struct tcphdr *)
                (packet + sizeof(struct ether_header) + (ip_hdr->ip_hl << 2));
            printf("  TCP: %d -> %d\n",
                   ntohs(tcp->th_sport), ntohs(tcp->th_dport));
        }
    }
}

int main(int argc, char *argv[]) {
    char errbuf[PCAP_ERRBUF_SIZE];
    pcap_t *handle;

    // 네트워크 인터페이스 열기
    handle = pcap_open_live("eth0", BUFSIZ, 1, 1000, errbuf);
    if (handle == NULL) {
        fprintf(stderr, "장치를 열 수 없음: %s\n", errbuf);
        return 1;
    }

    // BPF 필터 설정 (TCP 트래픽만)
    struct bpf_program fp;
    if (pcap_compile(handle, &fp, "tcp", 0, PCAP_NETMASK_UNKNOWN) == -1) {
        fprintf(stderr, "필터 컴파일 실패\n");
        return 1;
    }

    if (pcap_setfilter(handle, &fp) == -1) {
        fprintf(stderr, "필터 설정 실패\n");
        return 1;
    }

    // 패킷 캡처 시작 (무한 루프)
    pcap_loop(handle, 0, packet_handler, NULL);

    pcap_close(handle);
    return 0;
}

3.3.3 BPF (Berkeley Packet Filter)

BPF는 커널 레벨에서 패킷을 필터링하여 불필요한 패킷을 조기에 제거함으로써 성능을 크게 향상시킵니다. 

# BPF 필터 표현식 예시

# HTTP 트래픽만 캡처
tcp port 80 or tcp port 443

# 특정 IP 대역에서 오는 SSH 트래픽
src net 192.168.1.0/24 and tcp port 22

# ICMP와 특정 호스트 간 트래픽
icmp or host 10.0.0.1

# SYN 플래그가 설정된 TCP 패킷 (포트 스캔 탐지)
tcp[tcpflags] & tcp-syn != 0 and tcp[tcpflags] & tcp-ack == 0

# 비정상적으로 작은 TCP 윈도우 (공격 의심)
tcp[14:2] < 1024

# DNS 쿼리 (UDP 53번 포트)
udp port 53 and udp[10] & 0x80 == 0

# 대용량 패킷 (Jumbo frames)
greater 1500

# 프래그먼트된 패킷
ip[6:2] & 0x1fff != 0

3.4 배치 전략

NIDS의 효과는 배치 위치에 크게 의존합니다. 조직의 네트워크 토폴로지와 보안 요구사항에 따라 최적의 배치 전략을 선택해야 합니다.

3.4.1 주요 배치 지점

배치 위치 모니터링 대상 탐지 가능 위협 고려사항
인터넷 게이트웨이 외부 ↔ 내부 트래픽 외부 공격, 악성코드 유입 높은 트래픽 볼륨
DMZ 경계 DMZ ↔ 내부망 웹 공격, DMZ 침해 후 확산 중간 트래픽

3.5 트래픽 미러링 방법

스위치드 네트워크 환경에서 NIDS가 트래픽을 볼 수 있도록 하려면 트래픽 미러링 메커니즘이 필요합니다.

3.5.1 SPAN (Switched Port Analyzer)

SPAN은 스위치의 한 포트 또는 VLAN의 트래픽을 다른 포트로 복사하는 기능입니다.

3.5.2 네트워크 TAP (Test Access Point)

TAP는 네트워크 링크에 물리적으로 삽입되어 트래픽을 복사하는 하드웨어 장치입니다.

특성 SPAN Network TAP
구현 방식 소프트웨어 (스위치 기능) 하드웨어 (물리적 장치)
비용 낮음 (기존 스위치 활용) 높음 (별도 장비 필요)
신뢰성 스위치 부하 시 패킷 드롭 가능 패킷 손실 없음 (Fail-open)
성능 영향 스위치 리소스 사용 스위치에 영향 없음
가시성 VLAN 간 트래픽만 (Layer 2+) 모든 패킷 (Layer 1)
유연성 설정 변경 용이 물리적 재배치 필요
에러 프레임 필터링됨 전달됨 (완전한 가시성)

3.6 패킷 재구성 및 정규화

공격자는 패킷 단편화, 인코딩, 난독화 등의 기법으로 IDS를 우회하려 시도합니다. 이를 방지하기 위해 패킷 재구성과 정규화가 필요합니다.

3.6.1 IP 단편화 재조합

IP 패킷은 MTU(Maximum Transmission Unit)보다 클 경우 여러 조각으로 분할됩니다. 공격자는 이를 악용하여 탐지를 회피할 수 있습니다. 

# IP 단편화 재조합 알고리즘
class FragmentReassembler:
    def __init__(self):
        self.fragments = {}  # {(src_ip, dst_ip, ip_id): [fragments]}
        self.timeout = 60    # 60초 타임아웃

    def process_fragment(self, packet):
        if not packet.is_fragment():
            return packet  # 단편화되지 않은 패킷은 그대로 반환

        key = (packet.src_ip, packet.dst_ip, packet.ip_id)

        if key not in self.fragments:
            self.fragments[key] = []

        self.fragments[key].append({
            'offset': packet.fragment_offset,
            'data': packet.payload,
            'more_fragments': packet.more_fragments,
            'timestamp': time.time()
        })

        # 모든 조각이 수신되었는지 확인
        if self._is_complete(key):
            return self._reassemble(key)

        return None  # 아직 불완전

    def _is_complete(self, key):
        fragments = sorted(self.fragments[key], key=lambda x: x['offset'])

        # 마지막 조각 확인 (more_fragments = False)
        if fragments[-1]['more_fragments']:
            return False

        # 연속성 확인
        expected_offset = 0
        for frag in fragments:
            if frag['offset'] != expected_offset:
                return False
            expected_offset += len(frag['data'])

        return True

    def _reassemble(self, key):
        fragments = sorted(self.fragments[key], key=lambda x: x['offset'])
        payload = b''.join([f['data'] for f in fragments])

        # 재조합된 패킷 생성
        reassembled = self._create_packet(key, payload)

        # 메모리에서 제거
        del self.fragments[key]

        return reassembled

3.6.2 TCP 스트림 재구성

TCP 세션의 페이로드를 올바르게 분석하기 위해서는 순서가 뒤바뀐 패킷을 재정렬하고, 재전송된 패킷을 처리해야 합니다. 

# TCP 스트림 재구성 예시
class TCPStreamReassembler:
    def __init__(self):
        self.streams = {}

    def process_packet(self, packet):
        key = (packet.src_ip, packet.src_port,
               packet.dst_ip, packet.dst_port)

        if key not in self.streams:
            self.streams[key] = {
                'next_seq': packet.seq + len(packet.payload),
                'buffer': {},  # {seq: data}
                'reassembled_data': b''
            }

        stream = self.streams[key]

        # 순서대로 도착한 경우
        if packet.seq == stream['next_seq']:
            stream['reassembled_data'] += packet.payload
            stream['next_seq'] += len(packet.payload)

            # 버퍼에 있는 후속 패킷도 처리
            while stream['next_seq'] in stream['buffer']:
                data = stream['buffer'][stream['next_seq']]
                stream['reassembled_data'] += data
                stream['next_seq'] += len(data)
                del stream['buffer'][stream['next_seq'] - len(data)]

        # 순서가 뒤바뀐 경우 버퍼에 저장
        elif packet.seq > stream['next_seq']:
            stream['buffer'][packet.seq] = packet.payload

        # 재전송 패킷은 무시
        elif packet.seq < stream['next_seq']:
            pass

        return stream['reassembled_data']

3.6.3 프로토콜 정규화

일반적인 회피 기술과 정규화:
  • 대소문자 혼합: "SeLeCt" → 모두 대문자로 정규화
  • URL 인코딩: "%2e%2e%2f" → "../"로 디코딩
  • 유니코드 인코딩: "\u002e\u002e\u002f" → 정규화
  • 과도한 공백: "SELECT FROM" → 단일 공백으로
  • 주석 삽입: "SEL/**/ECT" → 주석 제거
  • 경로 정규화: "/./dir/../file" → "/file"

3.7 세션 추적 및 상태 관리

효과적인 탐지를 위해서는 개별 패킷뿐만 아니라 전체 세션의 컨텍스트를 이해해야 합니다.

3.7.1 연결 상태 테이블

# 연결 상태 추적 구조
class ConnectionStateTable:
    def __init__(self):
        self.connections = {}
        self.max_connections = 1000000
        self.timeout = 3600  # 1시간

    def track_connection(self, packet):
        key = self._generate_key(packet)

        if key in self.connections:
            conn = self.connections[key]
            self._update_connection(conn, packet)
        else:
            if len(self.connections) >= self.max_connections:
                self._cleanup_old_connections()

            conn = {
                'state': 'NEW',
                'start_time': time.time(),
                'last_seen': time.time(),
                'packets_sent': 0,
                'packets_received': 0,
                'bytes_sent': 0,
                'bytes_received': 0,
                'flags_seen': set()
            }
            self.connections[key] = conn

        return conn

    def get_connection_info(self, packet):
        key = self._generate_key(packet)
        return self.connections.get(key)

    def _update_connection(self, conn, packet):
        conn['last_seen'] = time.time()

        if packet.direction == 'outbound':
            conn['packets_sent'] += 1
            conn['bytes_sent'] += len(packet)
        else:
            conn['packets_received'] += 1
            conn['bytes_received'] += len(packet)

        if packet.protocol == 'TCP':
            conn['flags_seen'].add(packet.tcp_flags)
            conn['state'] = self._determine_tcp_state(conn)

    def _determine_tcp_state(self, conn):
        flags = conn['flags_seen']

        if 'RST' in flags:
            return 'RESET'
        elif 'FIN' in flags:
            return 'CLOSING'
        elif 'SYN' in flags and 'ACK' in flags:
            return 'ESTABLISHED'
        elif 'SYN' in flags:
            return 'SYN_SENT'
        else:
            return 'UNKNOWN'

  set-cpu-affinity: yes    # CPU 친화성 설정
  cpu-affinity:
    - management-cpu-set:
        cpu: [ 0 ]         # 관리 스레드는 CPU 0 사용
    - receive-cpu-set:
        cpu: [ 1, 2 ]      # 패킷 수신은 CPU 1-2
    - worker-cpu-set:
        cpu: [ 3-7 ]       # 분석 워커는 CPU 3-7
  detect-thread-ratio: 1.0

# 워커 스레드 수 설정
max-pending-packets: 1024  # 대기 큐 크기
runmode: workers           # 워커 모드 사용

요약

네트워크 기반 IDS(NIDS)는 네트워크 세그먼트의 모든 트래픽을 중앙에서 모니터링할 수 있어 효율적인 보안 감시를 제공합니다. NIDS는 호스트 기반 IDS와 달리 개별 시스템에 에이전트를 설치할 필요 없이 네트워크 인프라의 전략적 지점에 배치됩니다.

NIDS의 핵심 아키텍처는 패킷 캡처 엔진, 디코더, 전처리기, 탐지 엔진, 후처리 모듈로 구성됩니다. 고성능 패킷 캡처를 위해 libpcap, PF_RING, DPDK 등 다양한 기술이 사용되며, BPF 필터를 통해 커널 레벨에서 조기 필터링을 수행합니다. 스위치드 네트워크에서는 SPAN 포트 또는 네트워크 TAP을 사용하여 트래픽을 미러링해야 합니다.

효과적인 탐지를 위해서는 IP 단편화 재조합, TCP 스트림 재구성, 프로토콜 정규화 등의 전처리가 필수적입니다. 이를 통해 공격자의 회피 기술을 무력화할 수 있습니다. 또한 세션 추적과 상태 관리를 통해 전체 연결의 컨텍스트를 이해하고 더 정확한 탐지를 수행할 수 있습니다. 고속 네트워크 환경에서는 멀티스레딩, 하드웨어 오프로딩, 규칙 최적화 등을 통한 성능 튜닝이 중요합니다.
