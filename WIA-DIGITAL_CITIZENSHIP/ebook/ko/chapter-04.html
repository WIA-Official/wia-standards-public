1. DID 문서 구조

1.1 기본 DID 문서

WIA-DIGITAL_CITIZENSHIP의 DID 문서는 W3C DID Core 1.0 표준을 따르며, 다음 필드로 구성됩니다:

필드명 타입 필수 설명
@context String | Array ✓ JSON-LD 컨텍스트 정의
id String (DID) ✓ DID 식별자 (예: did:wia:korea:abc123)
authentication Array ✓ 인증에 사용되는 공개키
assertionMethod Array 서명 검증용 공개키
service Array 연결된 서비스 엔드포인트
created DateTime ✓ DID 생성 시각
updated DateTime 마지막 수정 시각
proof Object ✓ 문서 무결성 증명

2. 검증 가능한 자격증명 (VC) 스키마

2.1 기본 VC 구조

필드명 타입 필수 설명
@context Array ✓ VC 표준 컨텍스트 + 커스텀 컨텍스트
id URI VC 고유 식별자
type Array ✓ ["VerifiableCredential", "SpecificType"]
issuer DID or Object ✓ 발급자 DID
issuanceDate DateTime ✓ 발급 일시
expirationDate DateTime 만료 일시
credentialSubject Object ✓ 자격증명 대상 및 내용
proof Object ✓ 암호학적 서명 증명

2.2 한국형 VC 타입 정의

WIA-DIGITAL_CITIZENSHIP에서 정의하는 한국 특화 자격증명 타입:

타입명 발급 기관 용도 유효기간
ResidentRegistrationCredential 행정안전부 주민등록 증명 (주민번호 대체) 평생
VotingEligibilityCredential 중앙선거관리위원회 투표권 증명 선거 기간
HealthInsuranceCredential 국민건강보험공단 건강보험 자격 1년
DriverLicenseCredential 경찰청 운전면허 증명 10년
EducationCredential 교육기관 학력/학위 증명 평생
EmploymentCredential 고용주 재직 증명 재직 기간

3.2 시맨틱 웹 활용

📚 시맨틱 웹의 장점

  • 상호운용성: 서로 다른 시스템이 데이터 의미를 정확히 이해
  • 확장성: 새로운 필드 추가 시 하위 호환성 유지
  • 국제화: 다국어 및 지역별 맥락 표현 가능
  • 검증 가능성: 스키마 검증을 통한 데이터 무결성 보장

8. 에러 코드 및 상태 메시지

8.1 표준 에러 코드

코드 의미 HTTP 상태 해결 방법
DID_NOT_FOUND DID가 블록체인에 존재하지 않음 404 DID 생성 또는 올바른 DID 확인
INVALID_SIGNATURE 서명 검증 실패 401 올바른 개인키로 재서명
CREDENTIAL_EXPIRED 자격증명 만료 403 새로운 자격증명 발급 요청
CREDENTIAL_REVOKED 자격증명 폐기됨 403 발급 기관에 문의
INVALID_FORMAT 데이터 형식 오류 400 스키마 확인 및 수정
UNTRUSTED_ISSUER 신뢰할 수 없는 발급자 403 공인된 발급자에게 재발급 요청

🎯 핵심 요점 정리

  • 표준 준수: W3C DID, VC 표준을 엄격히 따르며 한국 특화 확장을 추가했습니다.
  • 명확한 스키마: 모든 데이터 구조가 JSON-LD로 정의되어 검증 가능합니다.
  • 한국형 타입: 주민등록, 투표권, 건강보험 등 한국 특화 자격증명 타입을 정의했습니다.
  • 영지식 증명: 최소 정보 공개를 위한 ZKP 데이터 구조를 제공합니다.
  • 블록체인 연동: 모든 중요 작업이 블록체인 트랜잭션으로 기록됩니다.
  • API 표준화: RESTful API로 시스템 간 데이터 교환이 용이합니다.
