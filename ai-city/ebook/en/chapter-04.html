4.1 Why JSON-LD?

WIA AI-City uses JSON-LD (JSON for Linked Data) as its primary data format. This choice was deliberate—JSON-LD combines the simplicity of JSON with the power of semantic web technologies, enabling both human readability and machine understanding.

The Problem with Plain JSON

Consider a simple temperature reading in plain JSON:

{
  "temp": 23.5,
  "unit": "C"
}

This looks straightforward, but it's ambiguous. What does "C" mean? Celsius? Some other code? Where was this measured? When? What's the context? Different systems might interpret this data differently, leading to integration failures.

The JSON-LD Solution

The same reading in JSON-LD:

{
  "@context": "https://schema.wia.org/ai-city/v1",
  "@type": "TemperatureReading",
  "@id": "urn:wia:sensor:temp-001:reading:2024-12-24T10:30:00Z",
  "value": 23.5,
  "unit": "CEL",
  "measuredAt": {
    "@type": "GeoCoordinates",
    "latitude": 37.5665,
    "longitude": 126.9780
  },
  "timestamp": "2024-12-24T10:30:00Z",
  "sensor": {
    "@id": "urn:wia:sensor:temp-001",
    "@type": "TemperatureSensor",
    "accuracy": "±0.1"
  }
}

Now there's no ambiguity. The @context tells any system exactly how to interpret every field. The @type identifies what kind of data this is. The @id provides a globally unique identifier.

Key JSON-LD Concepts

Concept Purpose Example
@context Defines vocabulary and term mappings "https://schema.wia.org/ai-city/v1"
@type Specifies the type of data object "TrafficSensor", "CitizenConsent"
@id Globally unique identifier (IRI) "urn:wia:city:seoul:sensor:001"
@graph Contains multiple linked objects Array of related entities
@vocab Default vocabulary for terms "https://schema.wia.org/"

4.2 The WIA Context and Vocabulary

WIA provides a comprehensive context definition that all compliant systems must use. This context maps terms to precise semantic meanings.

WIA Context Structure

{
  "@context": {
    "@version": 1.1,
    "@vocab": "https://schema.wia.org/ai-city/",
    "wia": "https://schema.wia.org/",
    "schema": "https://schema.org/",
    "geo": "http://www.w3.org/2003/01/geo/wgs84_pos#",

    "AICityInstance": "wia:AICityInstance",
    "RustCore": "wia:RustCore",
    "SharingEngine": "wia:SharingEngine",
    "DataSovereignty": "wia:DataSovereignty",
    "AIDCLink": "wia:AIDCLink",

    "cityId": {"@id": "wia:cityId", "@type": "@id"},
    "population": {"@id": "schema:population", "@type": "xsd:integer"},
    "location": {"@id": "geo:location", "@type": "@id"},
    "timestamp": {"@id": "schema:dateCreated", "@type": "xsd:dateTime"},

    "classificationLevel": {
      "@id": "wia:classificationLevel",
      "@type": "@vocab",
      "@context": {
        "L1": "wia:IdentityData",
        "L2": "wia:LocationData",
        "L3": "wia:BehavioralData",
        "L4": "wia:PublicData"
      }
    }
  }
}

Embedding vs. Referencing Context

For production systems, WIA recommends referencing the context by URL rather than embedding it:

Referenced (Recommended) "@context": "https://schema.wia.org/ai-city/v1"

Smaller payload, always up-to-date
Embedded (Development) Full context object included in document

Self-contained, works offline

4.4 Data Classification Implementation

Every piece of data in WIA AI-City must be classified according to the four-level system. Classification determines how data is stored, transmitted, accessed, and eventually deleted.

Classification Requirements

Level Storage Transmission Access Control Retention
L1 - Identity Local only, encrypted at rest Never transmitted externally Citizen key required Until explicit deletion
L2 - Location Encrypted, segregated storage E2E encrypted, consent verified Purpose-specific consent Max 90 days default
L3 - Behavioral Standard encryption Encrypted, anonymization option Opt-in consent Max 1 year default
L4 - Public Standard storage Open transmission Public access Permanent archive

Classification in Practice

Every data object must include its classification level:

{
  "@type": "LocationReading",
  "classificationLevel": "L2",
  "consentRequired": true,
  "consentToken": "CST-A1B2C3D4E5F6G7H8",

  "data": {
    "latitude": 37.5665,
    "longitude": 126.9780,
    "accuracy": 10,
    "timestamp": "2024-12-24T10:30:00Z"
  },

  "dataSubject": {
    "@id": "urn:wia:citizen:KR:12345",
    "pseudonymized": true
  }
}

Automatic Classification Rules

Default Classification by Data Type:
  • Biometrics, health records, SSN: Always L1
  • Real-time location, travel history: L2 by default
  • Shopping, service usage, preferences: L3 by default
  • Aggregated statistics (10+ individuals): Can be L4
  • Environmental readings (no personal data): L4

4.6 AIDC-Link Message Format

All inter-system communication uses the standardized AIDC-Link message format. This ensures consistent message structure across all city systems.

Message Structure

{
  "@context": "https://schema.wia.org/ai-city/v1",
  "@type": "AIDCMessage",

  "header": {
    "messageId": "MSG-2024-ABC123DEF456",
    "version": "1.0",
    "layer": 2,
    "priority": "normal",
    "timestamp": "2024-12-24T10:30:00Z",
    "expiresAt": "2024-12-24T10:35:00Z"
  },

  "source": {
    "nodeId": "AIDC-NODE-001",
    "nodeType": "edge",
    "zoneId": "urn:wia:city:seoul:zone:gangnam",
    "location": {
      "latitude": 37.5665,
      "longitude": 126.9780
    }
  },

  "destination": {
    "nodeId": "AIDC-HUB-CENTRAL",
    "nodeType": "hub"
  },

  "payload": {
    "@type": "TrafficReading",
    "classificationLevel": "L4",
    "data": {
      "vehicleCount": 1247,
      "averageSpeed": 35.5,
      "congestionLevel": "moderate"
    }
  },

  "security": {
    "signature": "Ed25519:7f8e9d...",
    "encryption": "none",
    "integrity": "sha256:abc123..."
  }
}

Message Types

Type Direction Use Case
SENSOR_DATA Edge → Hub Telemetry from IoT devices
COMMAND Hub → Edge Control instructions to devices
QUERY Any → Any Data requests
RESPONSE Any → Any Query responses
EVENT Any → Subscribers Real-time notifications
ALERT Any → Emergency Critical safety events

4.8 Migration Strategies

Most cities have existing data in various formats. WIA provides guidance for migrating to compliant formats while maintaining operations.

Migration Approach

1
Inventory Existing Data

Catalog all data sources, formats, volumes, and owners. Identify which WIA schema each maps to.

2
Design Transformation Rules

Create mapping documents showing how existing fields translate to WIA properties.

3
Implement Adapters

Build adapter layers that transform data at ingestion points. Legacy systems continue unchanged.

4
Dual-Write Period

Write data in both old and new formats. Validate that transformations are correct.

5
Cutover and Validation

Switch consumers to new format. Retire legacy format after validation period.

Adapter Example

Converting legacy traffic sensor data to WIA format:

// Legacy format
{
  "sid": "TS001",
  "t": 1703412600,
  "vc": 1247,
  "spd": 35.5
}

// WIA adapter transformation
function transformToWIA(legacy) {
  return {
    "@context": "https://schema.wia.org/ai-city/v1",
    "@type": "TrafficReading",
    "@id": `urn:wia:city:seoul:traffic:${legacy.sid}:${legacy.t}`,

    "sensor": {
      "@id": `urn:wia:city:seoul:traffic:sensor:${legacy.sid}`
    },

    "timestamp": new Date(legacy.t * 1000).toISOString(),
    "vehicleCount": legacy.vc,
    "averageSpeed": legacy.spd,
    "classificationLevel": "L4"
  };
}

4.9 Validation and Testing

WIA provides tools and specifications for validating data format compliance.

Validation Layers

Syntax Validation Valid JSON, proper JSON-LD structure
Schema Validation Required fields present, correct types
Semantic Validation Consistent @context, valid references
Business Rules Classification logic, consent requirements

WIA Validator Tool

# Install WIA validation tools
npm install -g @wia/ai-city-validator

# Validate a single document
wia-validate document.json

# Validate with specific schema
wia-validate --schema TrafficReading sensor-data.json

# Batch validate directory
wia-validate --recursive ./data/

# Output formats
wia-validate --format json document.json
wia-validate --format report document.json

Common Validation Errors

Error Cause Solution
MISSING_CONTEXT No @context field Add WIA context URL
INVALID_TYPE Unknown @type value Use valid WIA type name
MISSING_ID No @id for identifiable entity Add URN identifier
CLASSIFICATION_REQUIRED No classificationLevel Add L1-L4 classification
CONSENT_MISSING L2/L3 data without consent Add consent token reference

4.10 Chapter Summary

Key Takeaways

  • JSON-LD Foundation: WIA uses JSON-LD for semantic interoperability. The @context, @type, and @id fields provide unambiguous data interpretation across all systems.
  • WIA Context: A comprehensive vocabulary defines all terms. Reference the context URL in production; embed only for development.
  • 12 Core Domains: Traffic, Environment, Utilities, Transit, Safety, Health, Citizen, Finance, Infrastructure, Governance, AI/ML, and Security cover all smart city needs.
  • Data Classification: Every data object must be classified L1-L4. Classification determines storage, transmission, access, and retention policies.
  • AI-City Instance: The master configuration document defines a city's complete WIA setup—Rust-Core, Sharing Engine, Data Sovereignty, and AIDC-Link settings.
  • AIDC-Link Messages: Standardized message format with header, source, destination, payload, and security fields enables consistent inter-system communication.
  • ID Patterns: Consistent identifier formats (URNs, typed IDs) ensure global uniqueness and easy parsing.
  • Migration Path: Adapter-based migration allows gradual transition from legacy formats without disrupting existing operations.
  • Validation: WIA provides tools and specifications for validating compliance at syntax, schema, semantic, and business rule levels.
