## 학습 목표

이 장을 완료하면 다음을 수행할 수 있습니다:

## 2.1 데이터 사일로 및 비호환성

### 2.1.1 바벨탑 문제

생태계 모니터링 데이터는 수백 가지의 호환되지 않는 형식으로 존재합니다:

```
형식 단편화 현황:
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  파일 형식 (30개 이상)                                                │
│  ├─ 스프레드시트: Excel (.xlsx, .xls), CSV, TSV                     │
│  ├─ 데이터베이스: Access (.mdb), SQLite, PostgreSQL 덤프             │
│  ├─ 통계: R data (.rda), SAS, SPSS                                 │
│  ├─ GIS: Shapefiles, GeoJSON, KML, GeoPackage                     │
│  ├─ 과학: NetCDF, HDF5, MATLAB                                     │
│  ├─ 문서: PDF 표, Word 문서                                         │
│  └─ 독점: 공급업체별 센서 형식                                       │
│                                                                     │
│  데이터 모델 (수백 개)                                                │
│  ├─ 동일한 개념에 대한 사용자 정의 필드 이름                           │
│  ├─ 다른 테이블 구조                                                 │
│  ├─ 일관성 없는 단위 및 척도                                          │
│  └─ 다양한 정밀도 및 유효 숫자                                        │
│                                                                     │
│  좌표계 (100개 이상)                                                  │
│  ├─ WGS84, NAD83, NAD27 (다른 기준)                                 │
│  ├─ UTM 구역 (60개의 다른 구역)                                      │
│  ├─ State Plane (미국만 124개 구역)                                 │
│  └─ 십진도 vs. 도-분-초                                             │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.1.2 실제 통합 예시

**시나리오:** 유역 관리자가 하천 건강을 평가해야 함

| 데이터 소스 | 형식 | 과제 |
|------------|------|------|
| USGS 하천 게이지 | 탭 구분 텍스트, 야드-파운드 단위 | 미터법으로 변환 필요, 사용자 정의 날짜 형식 파싱 |
| 주 수질 실험실 | 병합된 셀이 있는 Excel, 셀에 각주 | 가져오기 전 수동 정리 필요 |
| 대형무척추동물 조사 | Access 데이터베이스, 독점 분류 코드 | 번역 테이블 필요, 소프트웨어 더 이상 지원되지 않음 |
| 어류 조사 | PDF 보고서, 종 일반 이름만 | 수동 데이터 입력, 모호한 종 ID |
| 토지 이용 (GIS) | Shapefile, State Plane 좌표 | WGS84로 재투영, 공간 결합 필요 |
| 기후 데이터 | NetCDF, 다른 변수 이름 | 사용자 정의 파서, 표준 이름에 매핑 |

**결과:** 분석을 시작하기 전 3-4주의 데이터 전처리. 프로젝트 시간의 60-70%가 과학 대신 데이터 준비에 소비됨.

## 2.2 품질 관리 악몽

### 2.2.1 누락된 품질 플래그

대부분의 모니터링 데이터에는 데이터 품질 지표가 부족합니다:

```
데이터 품질 문제 (일반적인 데이터셋):
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  센서 오작동 (판독값의 5-15%)                                         │
│  ├─ 시간 경과에 따른 센서 드리프트                                    │
│  ├─ 보정 오류                                                       │
│  ├─ 측정 중 전원 장애                                                │
│  ├─ 오염 (센서의 조류 성장)                                          │
│  └─ 야생동물 간섭 (곰이 장비를 공격)                                  │
│  ⚠️  문제: QC 플래그 없이는 종종 감지 불가능                         │
│                                                                     │
│  데이터 입력 오류 (레코드의 1-3%)                                      │
│  ├─ 오타: 38.5°C가 385°C로 입력됨                                    │
│  ├─ 단위 혼동: 미터가 피트로 기록됨                                   │
│  ├─ 소수점 오류: 2.5가 25로 입력됨                                   │
│  ├─ 복사-붙여넣기 실수                                               │
│  └─ 잘못된 필드 (위도/경도가 바뀜)                                    │
│  ⚠️  문제: 이상치가 실제 극단 사건과 혼합됨                          │
│                                                                     │
│  잘못된 식별 (분류군에 따라 다름)                                      │
│  ├─ 맹금류: 10-15% 오류율 (시민 과학)                                │
│  ├─ 딱새류: 20-30% 오류율                                           │
│  ├─ 암호종: 40-60% 오류율                                           │
│  ├─ 곤충: 종종 속/과까지만 식별됨                                     │
│  └─ 식물 (꽃이 피지 않음): 높은 오류율                                │
│  ⚠️  문제: 신뢰도 점수가 기록되지 않음                               │
│                                                                     │
│  노력 편향 (체계적)                                                   │
│  ├─ 관찰자가 접근 가능한 지역만 조사                                  │
│  ├─ 좋은 날씨에만 조사                                               │
│  ├─ 대상 종을 찾으면 중지                                            │
│  ├─ 가변적인 검색 시간                                               │
│  └─ 노력에 대한 문서화 없음                                          │
│  ⚠️  문제: 부재를 조사 부족과 구별할 수 없음                         │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2.2 사례 연구: 온도 데이터 오염

연구팀이 기후 분석을 위해 3개 소스의 기상 관측소 데이터를 결합했습니다:

| 소스 | 레코드 | 발견된 문제 | 영향 |
|------|--------|-----------|------|
| 국립기상청 | 100,000 | 일부 관측소가 AC 배기구 근처 (도시 열섬) | 2-3°C 따뜻한 편향 |
| 연구 관측소 | 50,000 | 데이터의 15%가 보정되지 않은 센서에서 | 높은 산포 |
| 시민 기상 관측소 | 200,000 | 많은 실내, 직사광선, 열원 근처 | 심각한 오염 |

**정리 노력:**
- 품질 관리 알고리즘 개발에 6개월
- 시민 데이터의 30%가 사용 불가능으로 플래그 지정됨
- NWS 데이터의 10%가 보정 필요
- 최종 데이터셋이 원본보다 50% 작아짐
- 처음부터 품질 플래그가 있었다면 피할 수 있었음

## 2.3 메타데이터 격차 및 데이터 발견 가능성

### 2.3.1 보이지 않는 데이터 문제

방대한 양의 생태 데이터가 존재하지만 찾을 수 없습니다:

```
데이터 발견 장벽:
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  학술 "다크 데이터" (수집된 데이터의 50-80% 추정)                       │
│  ├─ 대학원생 논문 데이터 (미발표)                                     │
│  ├─ 실패한 실험 (null 결과 공유 안 됨)                                │
│  ├─ 예비 조사 (불완전한 것으로 간주)                                  │
│  ├─ 레거시 데이터 (구식 매체, CD, 플로피 디스크)                       │
│  └─ 은퇴한 PI 데이터 (보관 중인 하드 드라이브)                         │
│                                                                     │
│  기관 회색 데이터 (공적 자금 지원이지만 공개적으로 접근 불가)             │
│  ├─ 내부 보고서 (디지털화되지 않음)                                   │
│  ├─ 계약자 산출물 (파일에 묻혀 있음)                                  │
│  ├─ 허가에 필요한 모니터링 (집계되지 않음)                             │
│  └─ 오래된 조사 데이터 (보관소의 종이 기록)                            │
│                                                                     │
│  NGO 사일로 데이터 (조직 장벽)                                         │
│  ├─ 경쟁 우려 (기금 제공자가 독점 데이터를 원함)                       │
│  ├─ 개인정보 보호 우려 (민감한 종 위치)                                │
│  ├─ 데이터 공유 정책 없음                                            │
│  └─ 공유를 준비할 자원 부족                                           │
│                                                                     │
│  결과: 추정 100억 달러 이상의 모니터링 데이터가 사실상 손실됨            │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.3.2 부적절한 메타데이터

데이터가 공유될 때 종종 필수 컨텍스트가 부족합니다:

| 누락된 메타데이터 | 결과 | 예시 |
|-----------------|------|------|
| **샘플링 프로토콜** | 비교 가능성 평가 불가 | "조류 조사" - 지점 계수? 횡단면? 지속 시간? |
| **관찰자 신원** | 전문성 평가 불가 | 전문가 vs. 초보자의 종 ID? |
| **탐지 방법** | 편향 보정 불가 | 시각적? 음향? 카메라 트랩? 다른 탐지 확률 |
| **노력** | 탐지 확률 계산 불가 | 얼마나 오래 검색했는가? 날씨 조건? |
| **공간 정밀도** | 위치 정확도 평가 불가 | GPS가 10m 또는 1000m로 정확한가? |
| **분류학적 권위** | 이름 조정 불가 | 어떤 필드 가이드를 사용했는가? |
| **데이터 처리** | 결과 재현 불가 | 원시 데이터 또는 변환됨? 어떤 알고리즘? |
| **허가 및 윤리** | 법적 준수 확인 불가 | 현장 작업이 승인되었는가? |

## 2.4 기술 및 인프라 장벽

### 2.4.1 레거시 시스템 종속

조직은 구식 시스템에 투자하여 마이그레이션 과제에 직면합니다:

```
레거시 시스템 과제:
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  연방 기관 (예시)                                                     │
│  ├─ 시스템: 1998년에 구축된 Oracle 데이터베이스                       │
│  ├─ 데이터: 25년간의 모니터링 (100GB+)                               │
│  ├─ 사용자 정의: 데이터 입력/검증을 위한 독점 코드                     │
│  ├─ 사용자: 500+ 직원이 현재 시스템에 대해 교육받음                   │
│  └─ 예산: 현대화를 위한 자금 없음                                     │
│                                                                     │
│  마이그레이션 장벽:                                                   │
│  ├─ 비용: 완전한 점검을 위해 500만-1000만 달러 추정                   │
│  ├─ 위험: 마이그레이션 중 잠재적 데이터 손실                          │
│  ├─ 워크플로우: 진행 중인 모니터링을 방해할 수 있음                    │
│  ├─ 교육: 전체 인력을 재교육해야 함                                   │
│  ├─ 정치: 다른 부서가 다른 우선순위를 가짐                            │
```

### 2.4.2 센서 데이터 홍수

현대 센서 네트워크는 처리할 수 있는 것보다 더 빠르게 데이터를 생성합니다:

| 센서 네트워크 | 데이터 속도 | 연간 볼륨 | 과제 |
|-------------|-----------|---------|------|
| NEON (미국) | 10TB/월 | 120TB/년 | 실시간 QC 필요 |
| OzFlux (호주) | 100GB/월/사이트 | 연간 1.2TB (12개 사이트) | 스토리지 및 백업 비용 |
| 카메라 트랩 네트워크 | 10,000 이미지/일 | 연간 360만 이미지 | 이미지 처리 병목 |
| 음향 모니터링 | 24/7 녹음 | 연간 35TB/레코더 | 오디오 분석 지연 |
| IoT 기상 관측소 | 5분마다 1개 판독값/센서 | 연간 1억 개 판독값 (네트워크) | 데이터베이스 성능 |

**일반적인 문제:**
- 데이터 처리가 수집보다 몇 개월 뒤처짐
- 스토리지 비용이 모니터링 예산을 초과
- 자동화된 QC가 불충분하고, 수동 검토는 불가능
- 처리를 위한 표준화된 파이프라인 없음

## 2.7 주요 요점

| 과제 영역 | 주요 문제 | 영향 |
|----------|---------|------|
| 데이터 사일로 | 30개 이상의 형식, 호환되지 않는 스키마 | 시간의 60-70%가 데이터 전처리에 소비됨 |
| 품질 관리 | 품질 플래그 없음, 이상치 미감지 | 오염된 데이터셋, 신뢰 약화 |
| 메타데이터 격차 | 최소 문서화, 표준 없음 | 데이터 보이지 않거나 사용 불가 |
| 기술 | 레거시 시스템, 센서 홍수 | 처리 백로그, 높은 비용 |
| 경제 | 자금 부족, 잘못 정렬된 인센티브 | 제한된 공유, 낮은 품질 |
| 조정 | 다중 조직 경관 | 중복, 통합 실패 |

### 중요 통계
- **50-80%**: 공유되지 않은 생태 데이터 추정 ("다크 데이터")
- **60-70%**: 데이터 전처리에 소비되는 연구 프로젝트 시간 비율
- **30%**: 품질 관리 후 사용 불가능한 것으로 플래그 지정된 일반적인 데이터
- **10%**: 모니터링에 대한 평균 예산 배분 (종종 첫 번째로 삭감됨)
- **$100억+**: 전 세계적으로 "손실된" 모니터링 데이터의 가치
