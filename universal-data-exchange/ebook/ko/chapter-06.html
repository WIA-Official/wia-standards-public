챕터 요약

핵심 요점:

  1. RESTful API 설계는 상태 비저장, 균일한 인터페이스, 리소스 식별 및 HATEOAS(Hypermedia as the Engine of Application State)를 포함한 아키텍처 제약을 따라 확장 가능한 분산 시스템을 구현합니다.
  2. 리소스 지향 설계는 작업(동사)이 아닌 비즈니스 엔티티(명사)를 중심으로 API를 모델링하며, HTTP 메서드(GET, POST, PUT, PATCH, DELETE)를 사용하여 리소스에 대한 작업을 표현합니다.
  3. API 버전 관리 전략에는 URI 버전 관리(/v1/resources), 헤더 버전 관리(Accept-Version) 및 콘텐츠 협상이 포함되며, 각각 캐싱, 라우팅 및 클라이언트 구현 복잡성에 대한 트레이드오프가 있습니다.
  4. 페이지네이션, 필터링, 정렬 및 필드 선택은 대규모 컬렉션에 대한 효율적인 데이터 검색을 가능하게 하며, 커서 기반 또는 오프셋 기반 접근 방식을 통해 대역폭을 줄이고 응답 시간을 개선하며 사용자 경험을 향상시킵니다.
  5. OpenAPI(Swagger) 사양은 표준화된 YAML/JSON 설명을 통해 자동화된 문서, 클라이언트 SDK 생성, 검증 및 테스트를 가능하게 하는 기계 판독 가능한 API 정의를 제공합니다.
  6. API 설계는 개발자 경험, 성능, 유연성 및 진화 가능성의 균형을 맞춰야 하며 업계 표준 및 조직 규약과의 일관성을 유지해야 합니다.

복습 질문

  1. REST 아키텍처 제약(상태 비저장, 균일한 인터페이스, 캐시 가능성, 계층 시스템, 코드 온 디맨드)을 설명하십시오. 이러한 제약이 분산 시스템에서 확장성과 진화 가능성을 어떻게 가능하게 합니까?
  2. Books, Authors, Members 및 Loans를 위한 리소스가 있는 도서관 관리 시스템용 RESTful API를 설계하십시오. REST 원칙에 따라 URI 구조, HTTP 메서드 및 응답 형식을 정의하십시오.
  3. API 버전 관리를 위한 URI 버전 관리(/v1/), 헤더 버전 관리(Accept-Version: 1.0) 및 콘텐츠 협상을 비교하십시오. 캐싱, 라우팅, 문서화 및 클라이언트 복잡성에 대한 영향은 무엇입니까?
  4. 콘텐츠 협상 및 Accept 헤더가 유니버설 데이터 교환(WIA-CORE-003)에서 유연한 데이터 형식 선택을 어떻게 가능하게 합니까? JSON, XML 및 Protocol Buffer 협상을 보여주는 예를 제공하십시오.
  5. 대규모 결과 세트를 반환하는 REST API를 위한 페이지네이션 전략을 설계하십시오. 일관성, 성능 및 데이터 변경 처리를 위해 오프셋 기반(page/limit)과 커서 기반 접근 방식을 비교하십시오.
  6. HATEOAS(Hypermedia as the Engine of Application State)와 REST API에서의 역할을 설명하십시오. 하이퍼미디어 링크를 보여주는 응답 예를 제공하고 API 진화에 대한 이점을 논의하십시오.
  7. 카테고리, 가격 범위 및 가용성별 필터링을 지원하는 제품 API용 OpenAPI 사양을 작성하십시오. 요청 매개변수, 응답 스키마 및 오류 정의를 포함하십시오.
