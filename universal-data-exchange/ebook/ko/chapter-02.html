챕터 요약

핵심 요점:

  1. JSON은 사람이 읽을 수 있는 가독성, 광범위한 언어 지원 및 기본 JavaScript 통합으로 인해 웹 API의 지배적인 데이터 형식으로 남아 있지만 바이너리 형식에 비해 성능과 타입 안전성을 희생합니다.
  2. Protocol Buffers는 바이너리 인코딩과 필드 번호 지정을 통한 효율적인 스키마 진화를 통해 우수한 성능을 제공하여 높은 처리량의 마이크로서비스 통신 및 장기 데이터 저장에 이상적입니다.
  3. XML은 문서 중심 처리, XSLT 변환 및 XML 스키마 검증이 필요한 전문 분야, 특히 엔터프라이즈 통합 및 규제 준수 시나리오에서 계속 사용됩니다.
  4. 데이터 형식 선택은 성능 요구사항, 개발자 경험, 도구 생태계, 스키마 진화 요구 및 조직 경계를 넘어 상호운용성 제약 조건을 포함한 여러 요소의 균형을 필요로 합니다.
  5. 현대 API는 클라이언트가 Accept 헤더를 통해 선호하는 형식을 요청할 수 있도록 콘텐츠 협상을 구현하며, 서버는 동일한 기본 데이터 모델의 여러 직렬화(JSON, XML, Protobuf)를 지원합니다.
  6. Apache Avro, MessagePack 및 CBOR과 같은 새로운 형식은 스트리밍 데이터, IoT 장치 및 공간 제약 환경과 같은 특정 사용 사례에 대한 전문적인 이점을 제공합니다.

복습 질문

  1. JSON, Protocol Buffers 및 XML을 직렬화 성능, 스키마 진화, 사람이 읽을 수 있는 가독성, 언어 생태계 지원 및 도구 성숙도의 5가지 차원에서 비교하십시오. 공개 REST API에 어떤 형식을 선택하겠습니까? 그 이유는 무엇입니까?
  2. Protocol Buffers가 필드 번호 지정을 통해 하위 및 상위 호환성을 어떻게 달성하는지 설명하십시오. 이전 클라이언트가 인식하지 못하는 새 필드가 있는 메시지를 받으면 어떻게 됩니까?
  3. 링크드 데이터를 표현하기 위한 JSON-LD의 구조와 이점을 설명하십시오. @context 속성이 Schema.org 어휘와의 의미론적 상호운용성을 어떻게 가능하게 합니까?
  4. 스키마 없는 형식(예: JSON)과 스키마 적용 형식(예: Protocol Buffers 또는 XML 스키마) 간의 트레이드오프는 무엇입니까? 각 접근 방식이 어떤 시나리오에서 뛰어난 성과를 발휘합니까?
  5. HTTP API에서 콘텐츠 협상이 어떻게 작동합니까? JSON을 폴백으로 사용하여 Protocol Buffers를 요청하는 클라이언트를 보여주는 요청 헤더 예제를 작성하고 서버의 결정 논리를 설명하십시오.
  6. 데이터 형식과 유니버설 메타데이터(WIA-CORE-001) 간의 관계를 분석하십시오. 메타데이터가 기본 직렬화 형식과 관계없이 데이터 교환을 어떻게 향상시킬 수 있습니까?
  7. 하위 호환성을 유지하면서 기존 JSON API가 Protocol Buffers를 지원하도록 전환하는 마이그레이션 전략을 설계하십시오. 클라이언트 중단을 최소화하는 버전 관리 및 롤아웃 전략은 무엇입니까?
