Phase 2: API Interface

RESTful APIs, Authentication, SDK Integration, and Real-Time Data Exchange

5.1 Introduction: Programmatic Access to Soil Data

While standardized data formats (Phase 1) enable interoperability in theory, practical integration requires well-designed APIs (Application Programming Interfaces) that systems can use to exchange data programmatically. Phase 2 of the WIA-SOIL-MICROBIOME standard defines comprehensive RESTful APIs that enable laboratories, farmers, researchers, and carbon credit verifiers to submit, retrieve, analyze, and share soil microbiome data.

"弘益人間 (Hongik Ingan) - APIs democratize access to soil knowledge, enabling small farms and developing nations to participate equally in the regenerative agriculture revolution." — WIA Philosophy

The WIA API is designed according to REST (Representational State Transfer) principles, using standard HTTP methods, JSON payloads, and OpenAPI documentation. This approach ensures that developers can integrate WIA functionality into any modern software system without specialized knowledge of soil science—the API abstracts complexity while preserving scientific rigor.

5.1.1 API Design Principles

  • RESTful: Resource-oriented URLs, standard HTTP methods (GET, POST, PUT, DELETE)
  • Versioned: API version in URL path (e.g., /v1/) ensures backward compatibility
  • Stateless: Each request contains all necessary context, enabling horizontal scaling
  • Paginated: Large result sets use cursor-based pagination for efficiency
  • Documented: Complete OpenAPI 3.0 specification with interactive documentation
  • Secure: OAuth 2.0 + API keys, rate limiting, and comprehensive audit logging

5.1.2 Base URL and Versioning

Production:  https://api.wia.org/soil-microbiome/v1
Sandbox:     https://sandbox-api.wia.org/soil-microbiome/v1
EU Region:   https://eu-api.wia.org/soil-microbiome/v1
APAC Region: https://apac-api.wia.org/soil-microbiome/v1

The API is deployed across multiple regions for low latency and data sovereignty compliance. Requests are automatically routed to the nearest region, with optional geo-pinning for jurisdictions requiring data residency.

5.2 Authentication and Authorization

Security is paramount when dealing with agricultural and carbon credit data. The WIA API implements a multi-layered authentication system suitable for different use cases.

5.2.1 Authentication Methods

Method Use Case Security Level Example
API Key Server-to-server, scripts Medium Laboratory LIMS integration
OAuth 2.0 User-delegated access High Mobile apps, web dashboards
JWT Token Microservices, short-lived High Internal service communication
mTLS Enterprise integrations Very High Government soil databases

5.2.2 API Key Authentication

The simplest authentication method uses API keys passed in the Authorization header:

curl -X GET "https://api.wia.org/soil-microbiome/v1/samples" \
  -H "Authorization: Bearer wia_sk_live_abc123xyz789" \
  -H "Content-Type: application/json"

🔑 API Key Best Practices

  • Use separate keys for development, staging, and production
  • Rotate keys every 90 days
  • Store keys in environment variables or secret managers, never in code
  • Use read-only keys for analytics and reporting
  • Immediately revoke compromised keys through the dashboard

5.2.3 OAuth 2.0 Flow

For applications accessing data on behalf of users, OAuth 2.0 provides secure delegated authorization:

5.4 Real-Time Data Streaming

For applications requiring real-time updates—such as field sensors uploading continuous measurements or carbon credit platforms tracking sequestration in real time—the WIA API provides WebSocket and Server-Sent Events (SSE) endpoints.

5.4.1 WebSocket Connection

// Connect to WebSocket
const ws = new WebSocket('wss://stream.wia.org/soil-microbiome/v1');

// Authenticate
ws.send(JSON.stringify({
  type: 'auth',
  token: 'YOUR_API_KEY'
}));

// Subscribe to sample updates
ws.send(JSON.stringify({
  type: 'subscribe',
  channel: 'samples',
  filters: {
    projectId: 'proj_abc123',
    events: ['created', 'updated', 'analyzed']
  }
}));

// Receive real-time updates
ws.onmessage = (event) => {
  const data = JSON.parse(event.data);
  console.log('Sample update:', data);
  // {
  //   type: 'sample.analyzed',
  //   sampleId: 'urn:wia:sample:US-IA-2024-00001',
  //   healthScore: 78.5,
  //   timestamp: '2024-01-16T14:30:00Z'
  // }
};

    Ok(())
}

// TypeScript SDK automatically handles rate limits with exponential backoff
const client = new WIASoilClient({
  apiKey: process.env.WIA_API_KEY,
  retryOptions: {
    maxRetries: 3,
    backoffMultiplier: 2,
    respectRateLimits: true
  }

// Manual handling
try {
  const samples = await client.samples.list({ limit: 1000 });
} catch (error) {
  if (error.code === 'RATE_LIMIT_EXCEEDED') {
    const retryAfter = error.retryAfter; // seconds
    console.log(`Rate limited. Retry after ${retryAfter}s`);
    await sleep(retryAfter * 1000);
    // Retry request
  }
}

5.8.3 Webhook Verification

// Verify webhook signature to ensure authenticity
import crypto from 'crypto';

function verifyWebhook(payload, signature, secret) {
  const hmac = crypto.createHmac('sha256', secret);
  const digest = 'sha256=' + hmac.update(payload).digest('hex');
  return crypto.timingSafeEqual(
    Buffer.from(signature),
    Buffer.from(digest)
  );
}

app.post('/webhooks/wia', (req, res) => {
  const signature = req.headers['x-wia-signature'];
  const payload = JSON.stringify(req.body);

  if (!verifyWebhook(payload, signature, process.env.WIA_WEBHOOK_SECRET)) {
    return res.status(401).send('Invalid signature');
  }

  // Process webhook
  const event = req.body;
  if (event.type === 'sample.analyzed') {
    console.log('Sample analyzed:', event.data.sampleId);
    // Update your database, send notifications, etc.
  }

  res.status(200).send('OK');
});

📚 Chapter Summary: Key Takeaways
