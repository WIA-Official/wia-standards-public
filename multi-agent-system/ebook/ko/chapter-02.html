에이전트 통신 프로토콜

弘益人間 (홍익인간) · 널리 인간을 이롭게 하라

효과적인 통신은 협력의 기초입니다. 인간 사회가 공유된 언어와 이해를 통해 번영하는 것처럼, 다중 에이전트 시스템은 모두에게 이익이 되도록 에이전트가 조정하고, 협상하고, 협력할 수 있게 하는 표준화된 통신 프로토콜을 통해 집단 목표를 달성합니다.

에이전트 통신의 중요성

통신은 다중 에이전트 시스템의 초석입니다. 효과적인 통신 메커니즘이 없으면 에이전트는 행동을 조정하고, 정보를 공유하거나, 복잡한 문제를 해결하기 위해 협력할 수 없습니다. 에이전트 통신 언어(ACL)는 에이전트가 서로를 이해하는 데 필요한 표준화된 구문과 의미를 제공합니다.

핵심 통찰: 에이전트 통신은 단순한 데이터 교환이 아니라 공동 목표를 향한 조정된 행동을 가능하게 하는 의도, 신념, 약속을 공유하는 것입니다.

전통적인 프로세스 간 통신과 달리 에이전트 통신 프로토콜은 다음을 처리해야 합니다:

  • 메시지 내용의 의미적 이해
  • 발화 행위 및 수행어 (요청, 알림, 약속 등)
  • 도메인 개념의 공유 이해를 위한 온톨로지
  • 다중 메시지 상호작용을 위한 대화 프로토콜
  • 개방형 시스템에서의 신뢰 및 인증

FIPA-ACL: 지능형 물리적 에이전트 기반

FIPA-ACL(Foundation for Intelligent Physical Agents - Agent Communication Language)은 에이전트 통신을 위해 가장 널리 채택된 표준입니다. 1990년대 후반 FIPA에서 개발되었으며, 에이전트 상호작용을 위한 포괄적인 프레임워크를 제공합니다.

FIPA 메시지 구조

모든 FIPA-ACL 메시지는 통신 행위를 정의하는 여러 매개변수로 구성됩니다:

// FIPA-ACL 메시지 구조
interface FIPAMessage {
    performative: string;      // 통신 행위 유형
    sender: AgentIdentifier;   // 발신자 에이전트 ID
    receiver: AgentIdentifier[]; // 하나 이상의 수신자
    content: any;              // 메시지 내용
    language: string;          // 내용 언어 (예: 'fipa-sl')
    ontology: string;          // 도메인 온톨로지
    protocol: string;          // 상호작용 프로토콜
    conversationId: string;    // 대화 식별자
    replyWith: string;         // 응답 식별자
    inReplyTo: string;         // 이전 메시지 참조
    replyBy: Date;             // 응답 마감 시간
}

// FIPA-ACL 메시지 예제
const message: FIPAMessage = {
    performative: 'request',
    sender: { name: 'buyer-agent-001', address: 'http://mas.example.com/buyer001' },
    receiver: [{ name: 'seller-agent-042', address: 'http://mas.example.com/seller042' }],
    content: {
        action: 'provide-quote',
        item: 'laptop',
        quantity: 5,
        maxPrice: 1000
    },
    language: 'fipa-sl',
    ontology: 'ecommerce-ontology',
    protocol: 'fipa-request',
    conversationId: 'conv-12345',
    replyWith: 'req-67890',
    replyBy: new Date('2025-12-26T10:00:00Z')
};

FIPA 수행어

FIPA-ACL은 22개의 표준 수행어(통신 행위)를 정의합니다. 가장 일반적으로 사용되는 것은 다음과 같습니다:

수행어 목적 사용 예
INFORM 정보 공유 "온도는 25°C입니다"
REQUEST 행동 수행 요청 "상태 보고서를 보내주세요"
QUERY-IF 명제가 참인지 질의 "작업이 완료되었습니까?"
PROPOSE 제안하기 "50달러에 배송할 수 있습니다"
ACCEPT-PROPOSAL 제안 수락 "제안을 수락합니다"
REJECT-PROPOSAL 제안 거부 "가격이 너무 높습니다"
AGREE 행동 수행 동의 "오후 5시까지 완료하겠습니다"
REFUSE 행동 수행 거부 "요청을 완료할 수 없습니다"
FAILURE 행동 실패 보고 "오류로 인해 작업 실패"
CFP 제안 요청 "배송 견적 필요" 
class FIPAAgent {
    constructor(id) {
        this.id = id;
        this.messageHandler = new Map();
        this.setupHandlers();
    }

    setupHandlers() {
        // 다양한 수행어 처리
        this.messageHandler.set('inform', this.handleInform.bind(this));
        this.messageHandler.set('request', this.handleRequest.bind(this));
        this.messageHandler.set('query-if', this.handleQuery.bind(this));
        this.messageHandler.set('propose', this.handlePropose.bind(this));
        this.messageHandler.set('cfp', this.handleCFP.bind(this));
    }

    async sendInform(receiver, content) {
        return this.sendMessage({
            performative: 'inform',
            sender: this.id,
            receiver: [receiver],
            content: content,
            language: 'fipa-sl'
        });
    }

    async sendRequest(receiver, action) {
        return this.sendMessage({
            performative: 'request',
            sender: this.id,
            receiver: [receiver],
            content: { action: action },
            protocol: 'fipa-request',
            conversationId: this.generateConversationId()
        });
    }

    handleInform(message) {
        console.log(`정보 수신: ${JSON.stringify(message.content)}`);
        this.updateBeliefs(message.content);
    }

    async handleRequest(message) {
        const canDo = await this.canPerformAction(message.content.action);

        if (canDo) {
            await this.sendMessage({
                performative: 'agree',
                sender: this.id,
                receiver: [message.sender],
                inReplyTo: message.replyWith,
                content: { action: message.content.action }
            });

            const result = await this.performAction(message.content.action);

            await this.sendMessage({
                performative: result.success ? 'inform' : 'failure',
                sender: this.id,
                receiver: [message.sender],
                content: result
            });
        } else {
            await this.sendMessage({
                performative: 'refuse',
                sender: this.id,
                receiver: [message.sender],
                inReplyTo: message.replyWith,
                content: { reason: '요청된 작업을 수행할 수 없습니다' }
            });
        }
    }

    async handleCFP(message) {
        // 제안 요청 - 평가하고 제안으로 응답
        const proposal = await this.generateProposal(message.content);

        if (proposal) {
            await this.sendMessage({
                performative: 'propose',
                sender: this.id,
                receiver: [message.sender],
                inReplyTo: message.replyWith,
                content: proposal
            });
        }
    }
}

메시지 전송 및 프로토콜

에이전트 통신에는 신뢰할 수 있는 메시지 전송 메커니즘이 필요합니다. WIA-AI-016 표준은 여러 전송 프로토콜을 지원합니다:

HTTP/HTTPS

RESTful HTTP API는 에이전트 메시지를 위한 간단하고 널리 지원되는 전송 메커니즘을 제공합니다. 에이전트는 메시지를 수신하기 위한 엔드포인트를 노출하고 HTTP POST를 사용하여 메시지를 보냅니다.

// HTTP 기반 에이전트 메시지 전송
class HTTPAgentTransport {
    constructor(agentId, port) {
        this.agentId = agentId;
        this.port = port;
        this.endpoint = `http://localhost:${port}/messages`;
    }

    // HTTP POST를 통한 메시지 전송
    async send(message) {
        const response = await fetch(this.endpoint, {
            method: 'POST',
            headers: { 'Content-Type': 'application/json' },
            body: JSON.stringify(message)
        });

        return response.json();
    }

    // HTTP 서버를 통한 메시지 수신
    startServer(messageHandler) {
        const express = require('express');
        const app = express();

        app.use(express.json());

        app.post('/messages', async (req, res) => {
            const message = req.body;
            const response = await messageHandler(message);
            res.json(response);
        });

        app.listen(this.port, () => {
            console.log(`에이전트 ${this.agentId}가 포트 ${this.port}에서 수신 중`);
        });
    }
}

WebSocket

WebSocket은 실시간 에이전트 상호작용에 이상적인 양방향, 전이중 통신을 제공합니다. HTTP 폴링에 비해 지연 시간과 오버헤드를 줄입니다.

메시지 큐 시스템

대규모 시스템의 경우 RabbitMQ, Apache Kafka 또는 MQTT와 같은 메시지 큐 시스템이 보장된 전달, 메시지 지속성, pub/sub 패턴과 같은 기능을 제공하는 신뢰할 수 있고 확장 가능한 메시지 전달을 제공합니다.

대화 프로토콜

많은 에이전트 상호작용은 대화 프로토콜 또는 상호작용 프로토콜이라고 하는 구조화된 패턴을 따르는 여러 메시지 교환이 필요합니다.

요청 프로토콜

FIPA 요청 프로토콜은 한 에이전트가 다른 에이전트에게 작업을 수행하도록 요청하는 간단한 상호작용을 정의합니다:

  1. 개시자가 REQUEST를 보냄
  2. 참여자가 AGREE 또는 REFUSE로 응답
  3. 동의한 경우, 참여자는 나중에 INFORM(성공) 또는 FAILURE를 보냄

계약망 프로토콜

계약망 프로토콜은 입찰을 통한 작업 할당을 가능하게 합니다:

  1. 관리자가 잠재적 계약자에게 CFP(제안 요청)를 보냄
  2. 계약자가 PROPOSE 또는 REFUSE로 응답
  3. 관리자가 제안을 평가하고 승자에게 ACCEPT-PROPOSAL을 보냄
  4. 관리자가 다른 사람들에게 REJECT-PROPOSAL을 보냄
  5. 수락된 계약자가 작업을 수행하고 결과를 보고

온톨로지와 공유 이해

에이전트가 진정으로 서로를 이해하려면 공통 온톨로지, 즉 도메인의 개념, 관계 및 제약 조건의 공식적인 사양을 공유해야 합니다.

// 간단한 온톨로지 정의
const eCommerceOntology = {
    concepts: {
        Product: {
            properties: ['name', 'price', 'category', 'manufacturer'],
            relationships: ['hasReview', 'inCategory']
        },
        Order: {
            properties: ['orderId', 'totalAmount', 'status', 'timestamp'],
            relationships: ['containsProduct', 'placedBy', 'deliveredTo']
        },
        Customer: {
            properties: ['customerId', 'name', 'email', 'address'],
            relationships: ['placed', 'reviewed']
        }
    },

    relationships: {
        hasReview: { domain: 'Product', range: 'Review' },
        inCategory: { domain: 'Product', range: 'Category' },
        containsProduct: { domain: 'Order', range: 'Product' },
        placedBy: { domain: 'Order', range: 'Customer' }
    }
};

// 의미적 이해를 위한 온톨로지 사용
class SemanticAgent {
    constructor(ontology) {
        this.ontology = ontology;
    }

    validateMessage(message) {
        const concept = message.content.concept;

        if (!this.ontology.concepts[concept]) {
            throw new Error(`알 수 없는 개념: ${concept}`);
        }

        const requiredProps = this.ontology.concepts[concept].properties;
        const providedProps = Object.keys(message.content.data);

        return requiredProps.every(prop => providedProps.includes(prop));
    }

    interpretMessage(message) {
        if (!this.validateMessage(message)) {
            return null;
        }

        return {
            concept: message.content.concept,
            data: message.content.data,
            semantics: this.ontology.concepts[message.content.concept]
        };
    }
}

에이전트 통신의 보안

안전한 통신은 다중 에이전트 시스템, 특히 악의적인 에이전트가 존재할 수 있는 개방형 환경에서 중요합니다.

인증

에이전트는 디지털 인증서, API 키 또는 토큰 기반 인증을 사용하여 통신 파트너의 신원을 확인해야 합니다.

암호화

도청을 방지하기 위해 메시지 내용을 암호화해야 합니다. HTTP용 TLS/SSL과 메시지 큐용 암호화 레이어가 전송 보안을 제공합니다.

메시지 무결성

디지털 서명은 전송 중 메시지가 변조되지 않았음을 보장합니다.

성능 최적화

효율적인 통신은 확장 가능한 다중 에이전트 시스템에 필수적입니다:

  • 메시지 일괄 처리: 여러 작은 메시지를 결합하여 오버헤드 감소
  • 압축: 큰 메시지 페이로드 압축
  • 캐싱: 자주 액세스하는 정보를 캐시하여 쿼리 감소
  • 비동기 통신: 비차단 메시지 패턴 사용
  • 로컬 vs. 원격: 에이전트가 같은 위치에 있을 때 로컬 통신 최적화

요약

  • 에이전트 통신 프로토콜은 조정, 협상, 협력을 가능하게 함
  • FIPA-ACL은 다양한 통신 행위를 위한 22개의 수행어를 제공하는 포괄적인 표준
  • 여러 전송 메커니즘(HTTP, WebSocket, 메시지 큐)이 다양한 요구 사항을 지원
  • 대화 프로토콜은 계약망 및 경매와 같은 다중 메시지 상호작용을 구조화
  • 공유 온톨로지는 이질적 에이전트 간의 의미적 이해를 보장
  • 인증, 암호화, 서명을 포함한 보안 조치가 통신을 보호
  • 일괄 처리, 압축, 캐싱을 통한 성능 최적화가 확장성을 향상
  • 효과적인 통신은 에이전트가 집단 이익을 위해 함께 작업할 수 있게 하여 弘益人間을 구현
