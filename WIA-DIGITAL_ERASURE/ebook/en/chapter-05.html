CHAPTER 5

Phase 2: Impact Assessment

Analyzing Scope, Dependencies, and Regulatory Implications

弘益人間 - Benefit All Humanity

Phase 2 is where verified erasure requests are thoroughly analyzed before execution. This critical phase identifies all data to be deleted, evaluates legal obligations, assesses business impacts, and creates a detailed execution plan. Rushing through impact assessment can lead to incomplete deletions, legal violations, or operational disruptions.

Data Discovery and Mapping

The first step in impact assessment is comprehensive data discovery: identifying every location where the requester's personal data exists. This requires mature data governance practices and comprehensive mapping.

Data Inventory Framework

System Category Data Types Discovery Method Complexity
Production Databases User profiles, transactions, preferences Schema analysis, foreign key mapping Medium
Data Warehouses Analytics, business intelligence, reports ETL pipeline tracing, lineage tracking High
File Storage Documents, images, uploads, exports Metadata indexing, content scanning High
Backup Systems All production data snapshots Backup catalog analysis, restore testing Very High
Log Files Access logs, application logs, audit trails Log aggregation, pattern matching Medium
Cache Layers Session data, temporary objects Cache key enumeration, TTL analysis Low
Third-Party Systems CRM, email, analytics, advertising API queries, contract review High

Automated Discovery Tools

Manual data discovery doesn't scale. Organizations need automated tools that can:

  • Scan databases: Identify tables and columns containing personal data
  • Trace data lineage: Follow data transformations from source to destination
  • Map relationships: Understand foreign keys and data dependencies
  • Inventory files: Index document repositories and object storage
  • Query APIs: Check third-party systems for data presence
  • Generate reports: Create comprehensive data maps for review

Sample Discovery Query Workflow

  1. User Identifier Resolution: Find all identifiers (user_id, email, phone) associated with requester
  2. Primary Data Lookup: Query main user tables for profile data
  3. Relationship Traversal: Follow foreign keys to find related records (orders, reviews, messages)
  4. Secondary Identifier Search: Search for email, phone in all tables (many systems have denormalized data)
  5. Log File Scanning: Search log aggregation systems for user activity
  6. File System Scanning: Search file metadata and content for user information
  7. Third-Party Queries: Query each integrated service via API
  8. Compilation: Aggregate all findings into comprehensive inventory

Legal and Regulatory Analysis

Not all data can or should be immediately deleted. Impact assessment must evaluate legal obligations and determine what must be retained, what can be deleted, and what should be anonymized.

Retention Obligations Analysis

Data Category Retention Driver Typical Period Action
Financial Transactions Tax law, accounting standards 7 years Retain with pseudonymization
Contract Records Contract law, statute of limitations 6 years post-termination Retain minimized data
Employee Data Employment law, pension requirements 3-10 years Segregate, retain required fields
Health Records HIPAA, medical liability 6 years Retain with strict access controls
Communications Financial services regulations 3-7 years Retain archived, segregated
Website Activity No retention requirement N/A Delete immediately
Marketing Preferences No retention requirement N/A Delete immediately

Exemption Evaluation

For each data element, evaluate whether GDPR Article 17(3) exemptions apply:

Exemption Checklist

  • Is data needed for freedom of expression or information?
  • Is retention required by law (EU or Member State)?
  • Is data needed for public interest tasks (health, research)?
  • Is data subject to archiving/research/statistical purposes with safeguards?
  • Is data needed to establish, exercise, or defend legal claims?
  • Does any of above apply? Document justification and scope

Legal Review Required

Impact assessment for complex cases should involve legal counsel, particularly when:

  • Active or pending litigation involves the requester
  • Regulatory investigations are ongoing
  • Data spans multiple jurisdictions with conflicting requirements
  • Financial or health data is involved
  • Deletion would impact others (business partners, co-signers)

Legal review should be completed within 5 business days to maintain overall 30-day SLA.

Business Impact Analysis

Beyond legal requirements, assess operational impacts of deletion:

Operational Dependencies

  • Active Services: Can current services continue without this data?
  • Billing Integrity: Will deletion affect invoice accuracy or payment processing?
  • Analytics Impact: How will aggregate analytics be affected?
  • Machine Learning: Are ML models trained on this data? Do they need retraining?
  • Business Partner Impact: Will deletion affect joint ventures or B2B relationships?
  • Fraud Detection: Is data needed for security or fraud prevention?

Risk Assessment Matrix

Risk Factor Low Risk Medium Risk High Risk
Data Volume < 1GB, < 10K records 1-100GB, 10K-1M records > 100GB, > 1M records
System Complexity Single system 5-10 systems > 10 systems
Third-Party Count 0-2 vendors 3-5 vendors > 5 vendors
Account Status Inactive > 1 year Inactive 3-12 months Active account
Data Sensitivity Basic profile Financial or health Special category data

High-risk deletions require additional approval, extended testing, and enhanced verification.

Execution Planning

Based on discovery and analysis, create detailed execution plan:

Deletion Sequence Planning

Order matters. Delete data in correct sequence to avoid referential integrity errors or orphaned records:

  1. Child Records First: Start with tables that have foreign keys to other tables
  2. Dependent Services: Delete from downstream systems before upstream sources
  3. Transactional Consistency: Group related deletions in transactions where possible
  4. Third-Party Coordination: Initiate third-party deletions early (they have their own timelines)
  5. Cache Invalidation: Clear caches after database deletions
  6. Backup Handling: Plan backup strategy (cryptographic erasure, scheduled expiration)

Resource Allocation

Estimate resources needed for execution:

Resource Type Considerations Planning Factor
Compute Time Database query execution, batch processing Volume × system count × complexity factor
Storage Operations File deletions, object storage cleanup File count × average size
Network Bandwidth Third-party API calls, cross-region replication Data volume × replication factor
Human Time Manual reviews, legal approval, verification Risk level × complexity

Rollback Strategy

Always have a rollback plan in case issues are discovered mid-execution:

  • Pre-Deletion Snapshot: Create recoverable snapshot before deletion (deleted after verification)
  • Staged Rollout: Delete from non-critical systems first
  • Verification Gates: Verify each stage before proceeding to next
  • Emergency Stop: Ability to halt process if issues detected
  • Communication Plan: How to notify requester if problems occur

Documentation and Approval

Impact assessment must be thoroughly documented. Create Impact Assessment Report containing:

Impact Assessment Report Contents

  • Request summary (ID, user, date, scope)
  • Data inventory (all locations, volume, sensitivity)
  • Legal analysis (retention requirements, exemptions)
  • Business impact summary (risks, dependencies)
  • Execution plan (sequence, timeline, resources)
  • Approval signatures (legal, DPO, system owners)
  • Rollback strategy
  • Verification plan

For high-risk deletions, obtain formal approval from:

  • Data Protection Officer (DPO)
  • Legal counsel
  • System owners for affected systems
  • Business unit leaders if significant operational impact

Communication with Requester

If impact assessment reveals complications, communicate proactively:

  • Partial Retention: Explain what data must be retained and why
  • Timeline Extensions: If complexity requires more than 30 days, request extension (GDPR allows 2 additional months if justified)
  • Exemptions: If request is denied under Article 17(3), provide clear justification
  • Alternative Solutions: Offer anonymization or restricted processing where full deletion isn't possible

Key Takeaways

  • Comprehensive data discovery is essential—use automated tools to identify all locations where personal data exists
  • Legal analysis must evaluate retention obligations and GDPR Article 17(3) exemptions before deletion
  • Business impact assessment considers operational dependencies, risks, and resource requirements
  • Execution planning defines deletion sequence, resource allocation, and rollback strategies
  • Thorough documentation and appropriate approvals ensure accountability and compliance verification
