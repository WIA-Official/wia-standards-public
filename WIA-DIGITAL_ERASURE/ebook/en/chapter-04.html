CHAPTER 4

Phase 1: Request & Verification

Establishing Secure Erasure Request Processes

弘益人間 - Benefit All Humanity

Phase 1 establishes the foundation of the erasure process: receiving, authenticating, and validating requests from data subjects. This phase is critical—errors here can lead to unauthorized deletions or denial of legitimate requests. The goal is to create a secure, accessible, and user-friendly system that balances openness with rigorous identity verification.

Request Channels

Organizations must provide multiple channels for submitting erasure requests, ensuring accessibility for all users regardless of technical sophistication:

Channel Use Case Implementation Priority
Web Form Primary channel for most users Authenticated portal with guided workflow Required
API Programmatic submissions, integrations RESTful API with OAuth2/JWT authentication Required
Email Users without account access Dedicated privacy@domain.com address Required
Phone Accessibility, users needing assistance Dedicated privacy hotline with trained staff Recommended
Postal Mail Regulatory requirement in some jurisdictions Physical address with secure document handling Recommended
In-Person High-security scenarios, retail presence Physical locations with identity verification Optional

Web Form Implementation

The web form is the primary channel for most users. It should be:

  • Easy to Find: Linked from account settings, privacy policy, footer navigation
  • Simple to Use: Clear instructions, minimal required fields
  • Accessible: WCAG 2.1 AA compliant, screen reader compatible
  • Secure: HTTPS only, CSRF protection, rate limiting
  • Transparent: Clear explanation of what will happen

Best Practice: Guided Request Workflow

Implement a multi-step wizard that guides users through the process:

  1. Authentication: User logs in or verifies identity
  2. Request Type: Select erasure, access, or other rights
  3. Scope Selection: Choose what data to delete (all data, specific categories)
  4. Reason (Optional): Why are you requesting deletion?
  5. Confirmation: Review request details before submission
  6. Receipt: Immediate confirmation with tracking number

Identity Verification

Robust identity verification is essential to prevent unauthorized deletions. GDPR Recital 64 states that controllers should verify identity "where there are reasonable doubts concerning the identity of the natural person making the request."

Verification Methods

Method Security Level Use Case Implementation
Account Authentication Medium Active users with valid login Username/password + MFA
Email Verification Medium Users with registered email One-time verification link with expiry
SMS Verification Medium-High Users with registered phone number One-time code via SMS
Knowledge-Based Authentication Medium Users without current account access Security questions, previous transaction details
Document Verification High High-value accounts, suspicious requests Government ID, utility bill
Video Verification Very High Ultra-sensitive data, large-scale deletion Live video call with ID presentation

Security Warning: Preventing Abuse

Erasure requests can be weaponized. Malicious actors may attempt to delete someone else's data as a form of attack. Implement these safeguards:

  • Multi-Factor Verification: Require at least two verification methods for non-trivial accounts
  • Cooling-Off Period: 24-72 hour delay before execution for high-value accounts
  • Out-of-Band Confirmation: Send confirmation via registered email/phone before proceeding
  • Anomaly Detection: Flag requests from unusual locations, IP addresses, or user agents
  • Rate Limiting: Prevent bulk automated requests
  • Audit Logging: Record all verification attempts and failures

Verification Decision Tree

Not all requests require the same level of verification. Use a risk-based approach:

  • Low Risk: Dormant accounts, minimal data → Email verification
  • Medium Risk: Active accounts, standard data → Account auth + email verification
  • High Risk: Financial data, health records → MFA + document verification
  • Critical Risk: Major accounts, large-scale data → Video verification + legal review

Status Lifecycle

Requests progress through defined statuses:

  1. Received: Request submitted, awaiting initial review
  2. Pending Verification: Identity verification in progress
  3. Verified: Identity confirmed, awaiting processing
  4. In Progress: Deletion execution underway
  5. Completed: All data deleted, verified
  6. Rejected: Request denied (with documented reason)
  7. Cancelled: User withdrew request

Communication and Transparency

Acknowledgment

GDPR Article 12(3) requires acknowledgment within one month. Best practice is immediate acknowledgment:

Sample Acknowledgment Email

Subject: Erasure Request Received - Confirmation #REQ-20250108-A1B2C3

Dear User,

We have received your request to delete your personal data.

Request Details:
- Request ID: REQ-20250108-A1B2C3
- Submitted: January 8, 2025 at 2:30 PM UTC
- Expected Completion: Within 30 days (by February 7, 2025)

Next Steps:
1. We will verify your identity within 3 business days
2. Once verified, we will analyze the scope of data to be deleted
3. You will receive updates at key milestones
4. Final confirmation will be sent upon completion

You can track your request status at:
https://example.com/privacy/request/REQ-20250108-A1B2C3

If you have questions, reply to this email or contact privacy@example.com

Thank you,
Privacy Team

Status Updates

Proactive communication builds trust. Send updates at key milestones:

  • Request received (immediate)
  • Identity verified (within 3 days)
  • Deletion started (within 10 days)
  • Deletion completed (within 30 days)
  • Any delays or exceptions (as they occur)

Handling Edge Cases

Requests from Non-Users

What if someone requests deletion but isn't in your system? GDPR Article 12(2) allows organizations to request additional information if needed. Options:

  • No Data Found: Confirm no personal data exists, document the confirmation
  • Insufficient Information: Request additional details to locate data
  • Third-Party Data: If data came from partners, coordinate with source

Requests from Minors

GDPR Article 8 and COPPA have special provisions for children. For minors:

  • Verify age of requester
  • For children under 13 (or 16 in some EU countries), require parental consent
  • Expedite processing—children's data has heightened protection
  • Document compliance with age verification requirements

Requests During Active Litigation

If data is subject to legal hold or active litigation:

  • Consult legal counsel immediately
  • Document the legal obligation preventing deletion
  • Inform requester of the delay and reason (without disclosing privileged information)
  • Commit to deletion once legal hold is lifted

Database Schema

CREATE TABLE erasure_requests (
    request_id VARCHAR(50) PRIMARY KEY,
    user_id VARCHAR(100) NOT NULL,
    email VARCHAR(255),
    phone VARCHAR(50),
    submitted_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    channel VARCHAR(50) NOT NULL,
    status VARCHAR(50) NOT NULL,
    scope TEXT,
    reason TEXT,
    priority VARCHAR(20) DEFAULT 'normal',
    verified_at TIMESTAMP NULL,
    verification_method VARCHAR(50),
    assigned_to VARCHAR(100),
    sla_deadline TIMESTAMP NOT NULL,
    completed_at TIMESTAMP NULL,
    INDEX idx_user_id (user_id),
    INDEX idx_status (status),
    INDEX idx_sla_deadline (sla_deadline)
);

Key Takeaways

  • Provide multiple request channels (web form, API, email, phone) to ensure accessibility for all users
  • Implement robust identity verification using risk-based approach—higher-risk deletions require stronger verification
  • Track requests with comprehensive metadata including request ID, status, verification details, and SLA deadlines
  • Communicate proactively with immediate acknowledgment and milestone updates throughout the process
  • Handle edge cases carefully: non-users, minors, active litigation each require specific procedures and documentation
