Chapter 3: IoT Sensors & Field Devices

3.1 Sensor Types

Soil Moisture Sensors

Measure volumetric water content (0-100%). Critical for irrigation scheduling.

  • Capacitance sensors: Most common, affordable ($20-50)
  • TDR sensors: High accuracy, expensive ($200-500)
  • Installation: 3 sensors per hectare, depths of 15cm, 30cm, 60cm

Weather Stations

Measure temperature, humidity, wind, rainfall, solar radiation.

Weather Station Data:
{
  "temperature": {"value": 25.5, "unit": "°C"},
  "humidity": {"value": 70, "unit": "%"},
  "windSpeed": {"value": 12, "unit": "km/h"},
  "rainfall": {"value": 5.2, "unit": "mm"}
}

Cameras & Drones

Capture visual data for AI analysis. RGB cameras for disease detection, multispectral for NDVI.

3.2 Communication Protocols

MQTT for Sensors

const mqtt = require('mqtt');
const client = mqtt.connect('mqtt://crop-monitoring.wiastandards.com');

client.publish('wia/crop/FARM-001/soil-moisture/SM-001', JSON.stringify({
  sensorId: 'SM-001',
  timestamp: new Date().toISOString(),
  measurements: {
    soilMoisture: { value: 85, unit: '%' }
  }
}));

LoRaWAN for Remote Fields

Low-power, long-range (5-15 km) communication for remote areas without WiFi/cellular.

Power Management

Solar-powered sensors can run for years. Transmit every 5 minutes in normal mode, every 30 seconds during alerts, sleep mode at night to conserve power.

3.3 Installation Guide

  1. Site Survey: Map field, identify zones with different soil types
  2. Sensor Placement: 3 moisture sensors/hectare, weather station central
  3. Calibration: Test sensors in known moisture conditions
  4. Network Setup: Configure WiFi/LoRaWAN gateway
  5. Cloud Connection: Register sensors, verify data transmission

3.4 Real-Time Data Collection

# Python sensor reading script
import time
from sensors import SoilMoistureSensor
from mqtt import MQTTClient

sensor = SoilMoistureSensor(pin=17)
client = MQTTClient('mqtt://crop-monitoring.wiastandards.com')

while True:
    moisture = sensor.read_moisture()
    client.publish('wia/crop/FARM-001/soil', {
        'value': moisture,
        'timestamp': time.time()
    })
    time.sleep(300)  # Every 5 minutes

Reference Standards and Implementation Notes

This standard is designed to interoperate with established international frameworks. Implementations SHOULD treat the following as the single source of truth and use this PHASE document as the interoperability layer that maps existing data into a machine-readable, signed, and auditable shape.

  • OpenAPI Specification 3.1 — REST API definition
  • JSON Schema 2020-12 — schema validation
  • ISO/IEC 27001:2022 — information security management
  • ISO/IEC 27018:2019 — PII protection in public clouds
  • ISO/IEC 17065:2012 — product certification bodies
  • CycloneDX 1.5 / SPDX 2.3 — software bill of materials
  • Sigstore (DSSE envelope, Rekor transparency log) — signature anchoring
  • in-toto Attestation Framework 1.0 — build provenance
  • IETF RFC 9700 — OAuth 2.1
  • IETF RFC 9457 — Problem Details for HTTP APIs
  • IETF RFC 8615 — well-known URIs

Practical Implementation Checklist

  • Manifest integrity — every published manifest is signed via Sigstore DSSE. Tampering is detectable without trusting an intermediary. Updates are issued as new manifests; the prior manifest is preserved with a supersede marker.
  • Retention — keep signed evidence for at least seven years to satisfy ISO/IEC 17065:2012 audit expectations. Some regulators require longer retention; in that case the deployment policy MUST extend the window rather than relying on the defaults.
  • Versioning — follow Semantic Versioning 2.0.0. Major version bumps require at least a 90-day overlap with the prior major version on every WIA-published reference implementation.
  • Negative-result reporting — a skipped indicator without recorded justification is treated as a failed indicator at audit time.
  • Key rotation — manifest signing keys are rotated per the deployment policy. The previous key is preserved in a published key history so that auditors can verify historical manifests offline.

Reference Topologies

Implementations declare their operational topology in the manifest: single-tenant edge (one runtime per organization), multi-tenant gateway (shared runtime with header-based isolation), federated mesh (multiple runtimes peer to one another and publish manifests to a directory service), or air-gapped batch (signed evidence package transported out-of-band). Each topology ships with reference test vectors so that implementations can validate their deployment matches the canonical pattern.

Open Governance

Issues, errata, and proposals are tracked at github.com/WIA-Official/wia-standards/issues with the appropriate label. The WIA Standards working group reviews open issues at the start of every minor release cycle and publishes the resulting decision log alongside the release notes. Errata are issued as patch releases; new normative requirements trigger minor bumps; backwards-incompatible changes trigger major bumps.
