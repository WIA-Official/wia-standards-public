Phase 1: 데이터 형식과 보존 메타데이터

WIA-ART-012-art-preservation

4.1 Phase 1의 핵심: 표준화된 메타데이터

Phase 1 Data Format은 WIA-ART-012의 가장 기본적이면서도 가장 중요한 단계입니다. 특별한 소프트웨어나 API 없이도 디지털 예술 작품의 보존 정보를 구조화된 형태로 저장할 수 있습니다. 이 단계는 개인 아티스트부터 대규모 기관까지 누구나 즉시 채택할 수 있도록 설계되었습니다.

🎯 Phase 1의 목표
  • 즉시 사용 가능: 텍스트 에디터만 있으면 메타데이터 작성 가능
  • 인간 및 기계 가독성: JSON 형식으로 둘 다 쉽게 읽고 처리
  • 무결성 보장: 체크섬으로 파일 변조 및 손상 감지
  • 완전성 검증: JSON Schema로 필수 필드 누락 방지
  • 상호운용성: 기존 보존 표준(PREMIS, METS)과 매핑 가능

2009년 Yahoo! GeoCities가 종료되면서 3,800만 개의 웹사이트가 하루아침에 사라졌습니다. Internet Archive의 Jason Scott은 "역사상 가장 큰 문화적 재앙 중 하나"라고 평가했습니다. 이 사건은 적절한 메타데이터 없이는 디지털 콘텐츠를 보존할 수 없다는 교훈을 남겼습니다. Archive Team은 급히 크롤링을 시작했지만, 원본 작성자, 생성 날짜, 기술 환경 등의 메타데이터가 대부분 손실되었습니다.

4.2 WIA-ART-012 메타데이터 스키마

WIA-ART-012는 5개의 주요 섹션으로 구성된 메타데이터 구조를 정의합니다:

섹션 목적 필수 여부 주요 필드
wia_version 표준 버전 명시 MUST 1.0, 1.1, 2.0 등
artwork 작품 기본 정보 MUST id, title, artist, creation_date
preservation 보존 기술 정보 MUST formats, checksums, dependencies
provenance 출처 및 이력 추적 SHOULD blockchain, ipfs_hash, transfers
rights 저작권 및 라이선스 SHOULD license, copyright_holder, usage_terms
extensions 사용자 정의 확장 MAY vr-metadata, ai-generation 등

📦 실제 사례: Flashpoint Project

Flashpoint는 2018년부터 Adobe Flash 콘텐츠를 보존하는 커뮤니티 프로젝트입니다. 2024년 기준으로 200,000개 이상의 Flash 게임, 애니메이션, 웹 경험을 보존했습니다. Flashpoint의 메타데이터 구조는 WIA-ART-012와 유사한 원칙을 따릅니다:

  • 플랫폼 메타데이터: 원본 플랫폼, 발행일, 개발자
  • 기술 메타데이터: Flash 버전, 의존성, 파일 크기
  • 보존 메타데이터: 추가 날짜, 검증 상태, 에뮬레이터 설정
  • 출처 메타데이터: 원본 URL, Archive.org 링크, 수집 방법

Flashpoint는 WIA-ART-012 채택을 검토 중이며, 기존 메타데이터를 자동 변환하는 도구를 개발하고 있습니다.

4.3 보존 메타데이터 표준: PREMIS 통합

PREMIS (PREservation Metadata: Implementation Strategies)는 2005년 Library of Congress 주도로 개발된 디지털 보존 메타데이터 표준입니다. WIA-ART-012는 PREMIS의 핵심 개념을 채택하면서도, 예술 작품에 특화된 확장을 제공합니다.

4.3.1 PREMIS 데이터 모델

PREMIS는 5개의 핵심 엔티티를 정의합니다:

PREMIS 엔티티 설명 WIA-ART-012 매핑
Object 보존 대상 디지털 객체 artwork + preservation.files
Event 객체에 발생한 행동 (생성, 검증, 마이그레이션) provenance.chain_of_custody
Agent 이벤트와 연관된 사람/조직/소프트웨어 artwork.artist, preservation.preservation_agent
Rights 저작권, 라이선스, 접근 권한 rights 섹션 전체
Intellectual Entity 개념적 작품 (여러 표현 가능) artwork (작품의 본질)

4.4 체크섬 알고리즘과 무결성 검증

디지털 보존의 핵심은 파일 무결성 검증입니다. 하드 드라이브 오류, 전송 중 손상, 악의적 변조를 감지하기 위해 암호화 해시 함수를 사용합니다.

알고리즘 출력 크기 속도 보안 권장 사용
MD5 128bit (32자) 매우 빠름 ❌ 취약 레거시 호환성만
SHA-1 160bit (40자) 빠름 ⚠️ 약함 Git 호환성
SHA-256 256bit (64자) 중간 ✅ 강함 ⭐ 권장 (MUST)
SHA-512 512bit (128자) 중간 ✅ 매우 강함 고보안 환경
BLAKE3 256bit (64자) 매우 빠름 ✅ 강함 대용량 파일
⚠️ MD5의 위험성

2008년 연구자들이 MD5 충돌을 이용해 가짜 SSL 인증서를 생성했습니다. 2012년 Flame 멀웨어는 MD5 충돌로 Microsoft 서명을 위조했습니다. MD5는 파일 무결성 검증에 더 이상 안전하지 않으며, WIA-ART-012는 SHA-256을 최소 요구사항으로 규정합니다.

4.4.1 체크섬 생성 코드 예시

Python으로 SHA-256 생성

import hashlib
import json

def calculate_sha256(file_path):
    """파일의 SHA-256 체크섬 계산"""
    sha256_hash = hashlib.sha256()
    with open(file_path, "rb") as f:
        # 큰 파일도 메모리 효율적으로 처리 (64KB 청크)
        for byte_block in iter(lambda: f.read(65536), b""):
            sha256_hash.update(byte_block)
    return sha256_hash.hexdigest()

def create_wia_metadata(artwork_file):
    """WIA-ART-012 메타데이터 생성"""
    import os
    from datetime import datetime

    checksum = calculate_sha256(artwork_file)
    file_size = os.path.getsize(artwork_file)

    metadata = {
        "wia_version": "1.0",
        "artwork": {
            "id": "art-" + checksum[:16],
            "title": "Digital Artwork",
            "creation_date": datetime.now().isoformat()
        },
        "preservation": {
            "files": [{
                "path": os.path.basename(artwork_file),
                "size_bytes": file_size,
                "checksum_sha256": checksum,
                "preservation_date": datetime.utcnow().isoformat() + "Z"
            }]
        }
    }

    # JSON 파일로 저장
    metadata_file = artwork_file + ".wia.json"
    with open(metadata_file, 'w', encoding='utf-8') as f:
        json.dump(metadata, f, indent=2, ensure_ascii=False)

    print(f"✅ Metadata created: {metadata_file}")
    print(f"📊 SHA-256: {checksum}")
    return metadata

# 사용 예시
metadata = create_wia_metadata("my_artwork.png")

Node.js/TypeScript로 체크섬 검증

import * as crypto from 'crypto';
import * as fs from 'fs';

interface WIAFile {
  path: string;
  checksum_sha256: string;
  size_bytes: number;
}

async function verifyChecksum(file: WIAFile): Promise<boolean> {
  return new Promise((resolve, reject) => {
    const hash = crypto.createHash('sha256');
    const stream = fs.createReadStream(file.path);

    stream.on('data', (chunk) => hash.update(chunk));
    stream.on('end', () => {
      const calculated = hash.digest('hex');
      const isValid = calculated === file.checksum_sha256;

      if (isValid) {
        console.log(`✅ ${file.path}: Integrity verified`);
      } else {
        console.error(`❌ ${file.path}: CHECKSUM MISMATCH!`);
        console.error(`   Expected: ${file.checksum_sha256}`);
        console.error(`   Got:      ${calculated}`);
      }

      resolve(isValid);
    });
    stream.on('error', reject);
  });
}

// 사용 예시
const fileToVerify: WIAFile = {
  path: "artwork.png",
  checksum_sha256: "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855",
  size_bytes: 2048000
};

verifyChecksum(fileToVerify);

4.4.2 Bit Rot 감지

Bit rot (비트 부패)는 저장 매체의 물리적 노화로 데이터가 서서히 손상되는 현상입니다. 자기 테이프는 10-30년, 하드 드라이브는 3-5년, SSD는 5-10년 후 데이터 손실 위험이 증가합니다.

📊 Internet Archive의 Bit Rot 대응

Internet Archive는 500+ 페타바이트의 데이터를 보존하며, 다음 전략으로 bit rot을 방지합니다:

  • 정기 검증: 모든 파일의 체크섬을 월 1회 검증
  • 다중 복제: 각 파일을 최소 3개 지리적 위치에 저장
  • 자동 복구: 체크섬 불일치 시 정상 복사본에서 자동 복원
  • 마이그레이션: 오래된 테이프를 3년마다 새 매체로 복사
  • 다양한 매체: 하드 드라이브 + 테이프 + 클라우드 혼합 사용

2023년 보고서에 따르면, Internet Archive는 연간 약 0.02%의 파일에서 bit rot을 감지하고 자동 복구했습니다 (약 100TB).

4.5 METS (Metadata Encoding & Transmission Standard)

METS는 디지털 도서관 객체의 메타데이터를 XML로 인코딩하는 표준입니다. Library of Congress에서 개발했으며, 복잡한 디지털 객체(여러 파일, 계층 구조)를 기술하는 데 강력합니다.

4.5.1 METS 구조

METS 섹션 목적 WIA 매핑
metsHdr METS 문서 자체의 메타데이터 wia_version, preservation_date
dmdSec 기술적 메타데이터 (Descriptive) artwork 섹션
amdSec 관리 메타데이터 (Administrative) preservation, rights
fileSec 파일 목록 및 그룹 preservation.files
structMap 객체의 계층 구조 artwork.components
behaviorSec 실행 가능한 동작 preservation.runtime_environment

4.5.2 복잡한 디지털 예술의 METS 활용

인터랙티브 웹 아트는 수십 개의 파일(HTML, CSS, JS, 이미지, 오디오)로 구성됩니다. METS의 structMap은 이러한 파일들의 관계를 표현합니다: 

<mets:structMap TYPE="physical">
  <mets:div TYPE="interactive-artwork" LABEL="Digital Garden">
    <mets:div TYPE="webpage" ORDER="1">
      <mets:fptr FILEID="index_html"/>
      <mets:fptr FILEID="main_css"/>
      <mets:fptr FILEID="app_js"/>
    </mets:div>
    <mets:div TYPE="assets">
      <mets:div TYPE="images">
        <mets:fptr FILEID="bg_001"/>
        <mets:fptr FILEID="sprite_001"/>
      </mets:div>
      <mets:div TYPE="audio">
        <mets:fptr FILEID="ambient_music"/>
      </mets:div>
    </mets:div>
  </mets:div>
</mets:structMap>

WIA-ART-012는 동일한 구조를 더 간결한 JSON으로 표현합니다: 

{
  "artwork": {
    "id": "digital-garden-2024",
    "components": {
      "type": "interactive-artwork",
      "entry_point": "index.html",
      "structure": [
        {
          "type": "webpage",
          "files": ["index.html", "main.css", "app.js"]
        },
        {
          "type": "assets",
          "images": ["bg-001.png", "sprite-001.png"],
          "audio": ["ambient-music.mp3"]
        }
      ]
    }
  }
}

4.6 파일 형식 레지스트리

PRONOM은 영국 국립 아카이브(The National Archives)가 운영하는 파일 형식 레지스트리로, 30,000개 이상의 파일 형식을 문서화합니다. WIA-ART-012는 PRONOM PUID (Persistent Unique Identifier)를 지원하여 파일 형식을 명확히 식별합니다.

파일 형식 MIME Type PRONOM PUID 보존 위험
Adobe Flash SWF application/x-shockwave-flash fmt/109 🔴 높음 (EOL)
HTML5 text/html fmt/471 🟢 낮음
PNG image/png fmt/11 🟢 낮음
WebP image/webp fmt/567 🟡 중간
MP4 (H.264) video/mp4 fmt/199 🟢 낮음
Java Applet application/java-archive x-fmt/412 🔴 높음 (EOL)
Unity WebGL application/wasm fmt/1281 🟡 중간 
{
  "preservation": {
    "format": "Adobe Flash SWF",
    "format_version": "10.0",
    "pronom_puid": "fmt/109",
    "mime_type": "application/x-shockwave-flash",
    "format_registry_url": "https://www.nationalarchives.gov.uk/PRONOM/fmt/109",
    "obsolescence_risk": "high",
    "obsolescence_reason": "Adobe ended support on 2020-12-31",
    "migration_recommendation": "Convert to Ruffle emulator or HTML5 Canvas"
  }
}

4.7 JSON Schema 검증

WIA-ART-012는 JSON Schema를 사용하여 메타데이터의 유효성을 자동 검증합니다. 이는 필수 필드 누락, 잘못된 데이터 타입, 형식 오류를 사전에 방지합니다. 

{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "required": ["wia_version", "artwork", "preservation"],
  "properties": {
    "wia_version": {
      "type": "string",
      "pattern": "^\\d+\\.\\d+(\\.\\d+)?$",
      "description": "Semantic version (e.g., 1.0, 1.2.3)"
    },
    "artwork": {
      "type": "object",
      "required": ["id", "title", "artist", "creation_date"],
      "properties": {
        "id": {
          "type": "string",
          "minLength": 1,
          "description": "Unique identifier for the artwork"
        },
        "title": {
          "type": "string",
          "minLength": 1
        },
        "artist": {
          "type": "string",
          "minLength": 1
        },
        "creation_date": {
          "type": "string",
          "format": "date",
          "description": "ISO 8601 date (YYYY-MM-DD)"
        }
      }
    },
    "preservation": {
      "type": "object",
      "required": ["files"],
      "properties": {
        "files": {
          "type": "array",
          "minItems": 1,
          "items": {
            "type": "object",
            "required": ["path", "checksum_sha256"],
            "properties": {
              "path": {"type": "string"},
              "checksum_sha256": {
                "type": "string",
                "pattern": "^[a-f0-9]{64}$"
              },
              "size_bytes": {"type": "integer", "minimum": 0}
            }
          }
        }
      }
    }
  }
}

검증 코드 예시 (JavaScript)

const Ajv = require('ajv');
const addFormats = require('ajv-formats');
const fs = require('fs');

const ajv = new Ajv();
addFormats(ajv);

// JSON Schema 로드
const schema = JSON.parse(fs.readFileSync('wia-art-012-schema.json', 'utf8'));
const validate = ajv.compile(schema);

// 메타데이터 검증
const metadata = JSON.parse(fs.readFileSync('artwork-metadata.json', 'utf8'));
const valid = validate(metadata);

if (!valid) {
  console.error('❌ Validation failed:');
  validate.errors.forEach(err => {
    console.error(`  - ${err.instancePath}: ${err.message}`);
  });
  process.exit(1);
} else {
  console.log('✅ Metadata is valid!');
}

4.8 장 요약

✅ 핵심 요점
  1. 표준화된 메타데이터: JSON 형식으로 인간과 기계가 모두 읽고 처리 가능
  2. 5개 주요 섹션: wia_version, artwork, preservation, provenance, rights로 구조화
  3. PREMIS 통합: 기존 보존 표준과 완벽 호환, PREMIS ↔ WIA 양방향 변환
  4. 체크섬 필수: SHA-256을 최소 요구사항으로, MD5는 레거시 호환성만
  5. Bit Rot 대응: 정기 검증, 다중 복제, 자동 복구 전략
  6. METS 호환: 복잡한 계층 구조를 structMap으로 표현, JSON으로 간소화
  7. 파일 형식 식별: PRONOM PUID로 명확한 형식 정의, 노후화 위험 평가
  8. JSON Schema 검증: 자동 유효성 검사로 오류 사전 방지

4.9 복습 질문

  1. WIA-ART-012의 5개 주요 메타데이터 섹션(wia_version, artwork, preservation, provenance, rights)을 설명하고, 각 섹션의 필수 여부(MUST/SHOULD/MAY)를 논의하십시오.
  2. 체크섬 알고리즘(MD5, SHA-1, SHA-256, SHA-512, BLAKE3)을 비교하고, WIA-ART-012가 SHA-256을 최소 요구사항으로 규정하는 이유를 보안 관점에서 설명하십시오.
  3. Bit rot의 개념을 설명하고, Internet Archive가 500+ 페타바이트의 데이터를 보존하기 위해 사용하는 5가지 전략을 제시하십시오.
  4. PREMIS의 5개 핵심 엔티티(Object, Event, Agent, Rights, Intellectual Entity)를 WIA-ART-012의 해당 필드로 매핑하고, 양방향 변환의 이점을 논의하십시오.
  5. Flashpoint Project의 Flash 콘텐츠 보존 사례를 참고하여, Adobe Flash EOL(2020-12-31) 이후 200,000개 작품을 보존하기 위한 메타데이터 구조를 설계하십시오.
  6. JSON Schema 검증의 필요성을 설명하고, 필수 필드 누락, 잘못된 데이터 타입, 체크섬 형식 오류를 자동으로 감지하는 검증 규칙을 작성하십시오.
