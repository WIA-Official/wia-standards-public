# WIA-protein-dynamics PHASE 2 — API-INTERFACE Specification

**Standard:** WIA-protein-dynamics
**Phase:** 2 — API-INTERFACE
**Version:** 1.0
**Status:** Stable

This document defines the HTTP API contract that an accredited
protein-dynamics programme exposes for the records defined in PHASE-1.
The contract is consumed by collaborating laboratories, by reference
repositories that ingest published trajectories and observables, by
biobank governance services that verify consent provenance, and by
publication and citation tools that resolve a published result to its
underlying records.

## §1 Scope and Versioning

The API is JSON-over-HTTPS served on a domain published by the operating
programme. Versioning uses URL path segments (`/v1/`) and follows
Semantic Versioning 2.0.0 at the major-version level. The OpenAPI 3.1
document at `/v1/openapi.json` is the canonical machine-readable
description.

## §2 Root Discovery

```
GET /v1/
```

Response:

```json
{
  "standard": "WIA-protein-dynamics",
  "phase": "API-INTERFACE",
  "version": "1.0",
  "links": {
    "studies":       "/v1/studies",
    "preparations":  "/v1/preparations",
    "structures":    "/v1/structures",
    "simulations":   "/v1/simulations",
    "trajectories":  "/v1/trajectories",
    "observables":   "/v1/observables",
    "analyses":      "/v1/analyses",
    "validations":   "/v1/validations",
    "evidence":      "/v1/evidence",
    "openapi":       "/v1/openapi.json"
  }
}
```

## §3 Studies

```
POST   /v1/studies                    — create a study
GET    /v1/studies/{studyId}          — retrieve a study
PATCH  /v1/studies/{studyId}          — update mutable fields
GET    /v1/studies?domain={d}&since={ts}  — list studies
```

`POST /v1/studies` accepts the study record from PHASE-1 §2. The server
generates `studyId` server-side as a UUIDv7 and returns the canonical
record. Status transitions follow the schedule in PHASE-1 §2; invalid
transitions return `422 Unprocessable Entity` with type
`urn:wia:protein-dynamics:status-transition`.

## §4 Preparations and Structures

```
POST   /v1/studies/{studyId}/preparations    — register a preparation
GET    /v1/preparations/{prepId}             — retrieve preparation
POST   /v1/studies/{studyId}/structures      — register a starting structure
GET    /v1/structures/{structureId}          — retrieve structure
```

A preparation that derives from a clinical biospecimen carries the FHIR R5
`Specimen.id` per PHASE-1 §3. Servers MUST verify the consent chain via
the biobank governance service before accepting the preparation; failure
returns `403 Forbidden` with type
`urn:wia:protein-dynamics:consent-not-verified`.

## §5 Computational Trajectories

```
POST   /v1/studies/{studyId}/simulations         — submit a simulation
GET    /v1/simulations/{simId}                   — simulation record
GET    /v1/simulations/{simId}/trajectories      — list trajectories
GET    /v1/trajectories/{trajId}                 — trajectory metadata
GET    /v1/trajectories/{trajId}/archive         — fetch coordinate archive
```

Simulation submissions are asynchronous; the POST returns `202 Accepted`
with a `Location` header. The simulation transitions through
`pending` → `running` → `complete` or `failed`. A request whose engine,
force-field, or water-model combination is not in the programme's
supported matrix returns `422` with type
`urn:wia:protein-dynamics:engine-unsupported`.

## §6 Spectroscopic Observables

```
POST   /v1/studies/{studyId}/observables        — register an observable
GET    /v1/observables/{obsId}                   — observable record
GET    /v1/observables/{obsId}/raw               — fetch raw archive
GET    /v1/observables/{obsId}/reduced           — fetch reduced data
PATCH  /v1/observables/{obsId}/uncertainty       — append uncertainty
```

Raw archives are immutable resources delivered with content-addressed
URLs and `Cache-Control: max-age=31536000, immutable`. Reduced data MAY
be regenerated by re-running the analysis pipeline; the regenerated
artefact replaces the existing reduced artefact and the previous version
remains addressable through its prior content-address.

## §7 Analyses

```
POST   /v1/studies/{studyId}/analyses             — register an analysis
GET    /v1/analyses/{analysisId}                  — analysis record
GET    /v1/analyses/{analysisId}/outputs          — list outputs
GET    /v1/analyses/{analysisId}/environment      — locked dependency manifest
```

The locked dependency manifest is a content-addressed artefact that
captures the software environment used to produce the analysis outputs.
Programmes MUST refuse to publish an analysis whose environment manifest
cannot be re-pinned (i.e. one or more dependencies are no longer
resolvable); resolution failure returns `409 Conflict` with type
`urn:wia:protein-dynamics:environment-unresolvable`.

## §8 Cross-Validation

```
POST   /v1/studies/{studyId}/validations          — register a cross-validation
GET    /v1/validations/{vid}                      — validation record
```

Cross-validation records bind a computational analysis output to an
experimental analysis output and report the agreement metric per
PHASE-1 §8.

## §9 Biospecimen Governance Traversal

For preparations derived from clinical biospecimens, the API exposes a
read-only proxy that traverses the FHIR R5 `Specimen` and consent
resources at the source biobank. The proxy enforces the access scopes
of both the protein-dynamics programme and the source biobank, and
returns the authorisation decision as a Problem-Details extension when
access is denied.

## §10 Errors

All error responses are `application/problem+json` per RFC 9457. Defined
types under `urn:wia:protein-dynamics` include:

- `urn:wia:protein-dynamics:status-transition` — invalid status change.
- `urn:wia:protein-dynamics:consent-not-verified` — missing biospecimen
  consent.
- `urn:wia:protein-dynamics:engine-unsupported` — unsupported simulation
  engine combination.
- `urn:wia:protein-dynamics:environment-unresolvable` — dependency
  manifest cannot be re-pinned.
- `urn:wia:protein-dynamics:evidence-mismatch` — evidence package
  digest mismatch.

Each problem response carries `traceId` per W3C Trace Context.

## §11 Caching, ETag, and Conditional Requests

GET responses for stable resources (completed simulations, observables
with frozen reductions, evidence packages) are cacheable with
`Cache-Control: max-age=31536000, immutable`. Mutable resources (draft
studies, in-progress analyses) are cacheable for 60 seconds. ETags are
mandatory on every PATCH endpoint; conditional requests use `If-Match`.

## §12 Worked Example: Combined Computational and Experimental Study

The canonical positive vector exercises a study that combines a
molecular-dynamics campaign with single-molecule FRET measurements:

1. Create a study under `/v1/studies` with domain
   `"computational"` and a sibling study with domain `"single-molecule"`
   sharing the `studyId` prefix.
2. Register a preparation that carries a FHIR `Specimen.id`; the
   server verifies the consent chain via §9 before accepting.
3. Register the starting structure (e.g. a deposited PDB entry).
4. Submit a simulation; poll until `complete`; retrieve the trajectory.
5. Register the FRET observable on the same preparation; upload the
   raw archive and the reduced FRET-efficiency histogram.
6. Register an analysis on the trajectory that emits an FES output.
7. Register a cross-validation that compares the FES projection
   against the FRET histogram.
8. Request an evidence package; pin the manifest digest as the citation
   anchor.

A conformant server completes this flow without error for the canonical
positive vector under `tests/phase-vectors/phase-2-api-interface/`.

## §13 Audit and Observability

Every endpoint emits structured logs with `studyId`, `traceId`, the
issuing client certificate's subject, and the operator's clock skew vs
the reference NTP source. Audit logs are retained for at least seven
calendar years from the last access of the study.

## §14 Ensemble Endpoints

For studies that emit ensemble records (PHASE-1 §11), the following
endpoints expose the ensemble shape and the aggregated outputs:

```
POST   /v1/studies/{studyId}/ensembles            — register an ensemble
GET    /v1/ensembles/{eid}                        — ensemble record
GET    /v1/ensembles/{eid}/members                — list members
GET    /v1/ensembles/{eid}/aggregate?weight={w}   — server-rendered aggregate
                                                    output (e.g. weighted FES)
```

Server-rendered aggregates are convenience endpoints; consumers that
require full-fidelity per-member outputs MUST retrieve the constituent
analysis outputs and re-aggregate locally so that the aggregation
methodology is reproducible from the published artefacts alone.

## §15 Bulk Endpoints

Long campaigns that produce many simulations or observables (mutant
sweeps, ligand-titration series) are submitted as bulk operations. A
bulk operation accepts an array of simulation or observable
specifications and emits an operation resource that tracks the campaign
as a unit:

```
POST   /v1/bulk/simulations         — submit a bulk simulation campaign
POST   /v1/bulk/observables         — submit a bulk observable campaign
GET    /v1/bulk/{operationId}       — campaign status
DELETE /v1/bulk/{operationId}       — cancel (best-effort)
```

Bulk operations report aggregate progress and per-item terminal status.
Cancellation is best-effort; items already running continue to
completion and emit their results normally.

## §16 Pagination

List endpoints use cursor-based pagination via the `cursor` query
parameter and a `Link` header (RFC 8288) carrying `rel="next"` and
`rel="prev"` relations. Cursors are opaque to clients; servers MUST
persist enough state to resolve a cursor for at least 24 hours after
issuance.

## §17 Streaming Subscriptions and Heartbeat

Consumers that monitor long simulation campaigns subscribe to a
Server-Sent Events stream at `/v1/studies/{studyId}/events`. Events
emitted include simulation status changes, observable registrations,
analysis output availability, and evidence-package builds. Every
subscription includes a heartbeat every 30 seconds so that
intermediaries can detect stale connections and reconnect; replays from
the last known event identifier are supported via the
`Last-Event-ID` header (W3C EventSource semantics).

For studies that emit ensemble records the event stream includes
ensemble-aggregate events that fire when a member completes and the
ensemble's aggregate is recomputed; consumers that prefer pull-based
polling can disregard the events without loss of correctness because the
underlying records remain authoritative.

## §18 Conformance

A conformant server passes the test vectors published under
`tests/phase-vectors/phase-2-api-interface/`, emits an OpenAPI 3.1
document that round-trips the resource shapes defined in this PHASE,
and signs exported evidence packages per RFC 9421.
