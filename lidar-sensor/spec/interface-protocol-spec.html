WIA-SEMI-014: LiDAR Interface and Protocol Specification

1. Physical Interfaces

1.1 Connector Specifications

Automotive LiDAR sensors SHALL use waterproof, automotive-grade connectors:

Interface Connector Type Pinout
Power Deutsch DT series or equivalent (IP67) Pin 1: +12V/+24V
Pin 2: GND
Pin 3: GND (shield)
Ethernet M12 X-coded or Fakra (for Automotive Ethernet) Pin 1: TX+
Pin 2: RX+
Pin 3: TX-
Pin 4: RX-
CAN Bus M12 A-coded 5-pin Pin 1: CAN Shield
Pin 2: CAN_H
Pin 3: GND
Pin 4: CAN_L
Pin 5: NC

2. Ethernet Data Protocol

2.1 UDP Point Cloud Streaming

Point cloud data SHALL be transmitted via UDP for minimum latency:

UDP Packet Structure:
Header (42 bytes):
    - Ethernet Header (14 bytes)
    - IP Header (20 bytes)
    - UDP Header (8 bytes)

LiDAR Data Packet (variable length):
    - Packet Header (12 bytes):
        - Magic Number: 0xA5FF (2 bytes)
        - Packet Type: 0x01 (point cloud) (1 byte)
        - Sequence Number: (4 bytes, wraps at 2^32)
        - Timestamp: GPS time in microseconds (8 bytes)
        - Number of Points: N (2 bytes)
    
    - Point Data (N × point_size bytes):
        For each point:
            - X coordinate: float32 (4 bytes)
            - Y coordinate: float32 (4 bytes)
            - Z coordinate: float32 (4 bytes)
            - Intensity: uint8 or float32 (1 or 4 bytes)
            - Timestamp offset: uint32 nanoseconds (4 bytes)
            - Ring ID: uint8 (1 byte)
        
        Total: 17 or 21 bytes per point

2.2 ROS/ROS2 Integration

LiDAR drivers SHALL publish data using standard ROS message types:

sensor_msgs/PointCloud2:
    header:
        stamp: Time
        frame_id: "lidar_link"
    height: 1 (unordered cloud) or N_rings (organized)
    width: N_points
    fields:
        - name: "x", offset: 0, datatype: FLOAT32, count: 1
        - name: "y", offset: 4, datatype: FLOAT32, count: 1
        - name: "z", offset: 8, datatype: FLOAT32, count: 1
        - name: "intensity", offset: 12, datatype: FLOAT32, count: 1
        - name: "ring", offset: 16, datatype: UINT16, count: 1 (optional)
        - name: "time", offset: 18, datatype: FLOAT32, count: 1 (optional)
    is_bigendian: false
    point_step: 18 or 22 bytes
    row_step: width × point_step
    data: [binary point cloud data]
    is_dense: false (allows NaN/Inf for invalid points)

3. CAN Bus Protocol

3.1 CAN Message IDs

Message ID (Hex) Purpose Period DLC (bytes)
0x100 LiDAR Status 100ms 8
0x101 Diagnostics 1000ms 8
0x102 Configuration Command On-demand 8
0x103 Configuration Response On-demand 8

3.2 Status Message Format (ID 0x100)

Byte 0: Operational Mode
    0x00 = Initializing
    0x01 = Normal Operation
    0x02 = Degraded Mode (reduced performance)
    0x03 = Error Mode
    0x04 = Safe State

Byte 1: Health Status Flags (bitfield)
    Bit 0: Laser operational
    Bit 1: Detector operational
    Bit 2: Thermal warning
    Bit 3: Communication error
    Bit 4-7: Reserved

Byte 2-3: Frame Rate (uint16, 0.1 Hz units, e.g., 100 = 10.0 Hz)

Byte 4: Temperature (int8, °C, offset +50, e.g., 0x32 = 0°C)

Byte 5: Power Consumption (uint8, 0.5W units, e.g., 30 = 15W)

Byte 6-7: Uptime (uint16, hours)

LiDAR data MAY be exported in LAS 1.4 format for GIS and mapping applications. Refer to ASPRS LAS 1.4 specification for complete format details.

5. Time Synchronization

5.1 PPS (Pulse Per Second) Input

LiDAR sensors SHALL support GPS PPS input for multi-sensor time synchronization:

  • Input Signal: 3.3V or 5V logic, rising edge trigger
  • Pulse Width: 100ms ± 50ms
  • Accuracy: Timestamp alignment to GPS time within ±1ms
  • Connector: Dedicated pin on main connector or separate coax (SMA/MCX)

5.2 PTP (Precision Time Protocol) - IEEE 1588

LiDAR sensors MAY support PTP over Ethernet for sub-microsecond synchronization:

PTP Configuration:
    - Domain: 0 (default)
    - Transport: IEEE 802.3 (Layer 2) or UDP/IPv4
    - Sync Interval: 1 second
    - Delay Request Interval: 1 second
    - Expected Accuracy: <1 μs timestamp alignment

6. Configuration and Diagnostics API

6.1 HTTP REST API (Optional)

LiDAR sensors MAY provide HTTP REST API for configuration via Ethernet:

GET /api/v1/status
    Returns: JSON object with current sensor status

POST /api/v1/config
    Body: JSON configuration parameters
    {
        "frame_rate": 10,
        "range_mode": "standard",  // or "long_range", "short_range"
        "intensity_calibration": true,
        "return_mode": "strongest"  // or "last", "dual"
    }

GET /api/v1/diagnostics
    Returns: Detailed diagnostic data, error codes, logs

POST /api/v1/firmware_update
    Body: Firmware binary file
    Initiates firmware update process

7. Coordinate Systems and Transformations

7.1 Sensor Coordinate Frame

Point cloud data SHALL be expressed in right-hand Cartesian coordinate system:

  • X-axis: Forward (vehicle heading direction)
  • Y-axis: Left (lateral, perpendicular to heading)
  • Z-axis: Up (vertical, perpendicular to ground plane)
  • Origin: Sensor optical center

7.2 Calibration Parameters

Extrinsic calibration SHALL define transformation from LiDAR frame to vehicle frame:

Transformation Matrix (4×4 homogeneous):
    [R11  R12  R13  Tx]
    [R21  R22  R23  Ty]
    [R31  R32  R33  Tz]
    [0    0    0    1 ]

Where:
    R = 3×3 rotation matrix (roll, pitch, yaw)
    T = [Tx, Ty, Tz] translation vector (meters)

Typical Installation:
    - Tx: 0.5m to 2.0m forward from vehicle center
    - Ty: -0.3m to +0.3m lateral offset
    - Tz: 1.2m to 2.5m height above ground
    - Roll: ±2°, Pitch: ±5°, Yaw: ±10°
