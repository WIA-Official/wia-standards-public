Chapter Summary

Key Takeaways:

  1. Error code taxonomy in WIA-CORE-010 uses hierarchical namespace structure: Domain prefix (AUTH, DB, NET, BIZ), category identifier (1000-9999), and optional subcategory (01-99), enabling systematic error identification like AUTH-4001-01 for "Invalid JWT token signature".
  2. Severity classification maps to operational impact: CRITICAL (system down, data loss risk, revenue impact >$10K/hour), ERROR (feature broken, user workflow blocked), WARNING (degraded performance, approaching limits), INFO (expected behavior, audit events, successful operations).
  3. Error code registration prevents conflicts through centralized registry: Each domain team owns specific code ranges (AUTH: 4000-4999, DB: 5000-5999), with mandatory documentation including description, severity, retry guidance, and example scenarios before production deployment.
  4. Standardized error codes enable automated incident response: Sentry automatically routes AUTH-4003 (expired token) to low-priority queue with auto-refresh suggestion, while DB-5001 (connection pool exhausted) triggers immediate page to database team and scales connection pool.
  5. Compliance-driven error classification: GDPR requires separate tracking of data access errors (retention in audit logs: 6 years), HIPAA mandates specific codes for PHI access (minimum 256-bit encryption for storage), PCI DSS enforces distinct payment error codes (never log full card numbers).
  6. Universal Protocol integration allows error code forwarding across service boundaries with context preservation: Original error code (SERVICE-A-1001) wrapped in parent context (SERVICE-B-2001) creates causal chain for distributed debugging through Datadog's trace correlation.
  7. Error code analytics reveal system health patterns: 15% increase in NET-3001 (timeout errors) over 24 hours correlates with 99.5% to 99.2% availability drop, enabling predictive scaling before customer impact reaches SLA breach threshold.

Review Questions

  1. Design an error code taxonomy for a fintech platform with services: Authentication, Account Management, Payment Processing, Fraud Detection, Reporting. Define code ranges, naming conventions, and provide 3 example codes per service with complete documentation.
  2. A service returns error code PAY-6523. From the code alone, determine: (a) Which service domain owns it? (b) What severity level should it default to? (c) How should retry logic differ between PAY-6523-01 (transient network issue) and PAY-6523-02 (insufficient funds)?
  3. Implement an error code registry using JSON Schema. Include fields for: code, domain, severity, http_status, retry_policy, compliance_tags (GDPR/HIPAA/PCI), description, example, remediation_steps. Validate that AUTH codes map to 401/403 status codes only.
  4. Your system has 47 different "database error" codes across 12 microservices. Design a consolidation strategy: Which codes should be merged? How do you handle backward compatibility? What migration timeline balances risk and technical debt?
  5. Configure Datadog alerting rules based on error codes: Critical alerts (page immediately) for any CRITICAL severity codes, error budget consumption tracking for ERROR codes (alert when 90% budget consumed), and weekly digest for WARNING codes. How do you prevent alert fatigue?
  6. A HIPAA-compliant healthcare API logs error AUTH-4011 (patient record access denied). What information MUST be logged for compliance (audit trail), what MUST be excluded (PHI), and how do you balance debugging needs with regulatory requirements? Design the log structure.
  7. Trace a distributed transaction failure: Frontend gets generic error "Request failed" (ERR-9999). Using Universal Protocol correlation IDs and error code chain (API-GW-1001 → AUTH-4003 → DB-5012), reconstruct the failure. What tools (Datadog/Sentry) help visualize this? Design the trace view.
