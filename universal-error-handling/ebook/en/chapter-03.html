Chapter Summary

Key Takeaways:

  1. HTTP status code mapping in WIA-CORE-010 follows semantic precision: 400 (malformed request syntax), 401 (missing authentication), 403 (insufficient permissions), 404 (resource not found), 409 (conflict with current state), 422 (validation failed), 429 (rate limit exceeded), 500 (internal server error), 502 (upstream failure), 503 (service unavailable), 504 (gateway timeout).
  2. Client errors (4xx) indicate caller responsibility requiring request modification, never automatic retry without changes: 400/422 need data correction, 401 needs authentication, 403 needs authorization, 404 needs resource verification, 429 needs rate limiting backoff (Retry-After header specifies wait duration).
  3. Server errors (5xx) indicate provider responsibility suitable for retry with exponential backoff: 500 (retry after 1s, 2s, 4s), 502/503 (check upstream health, retry after 2s), 504 (increase timeout, retry after 5s), circuit breaker triggers after 5 consecutive 5xx preventing cascade failures.
  4. REST API error response structure standardizes troubleshooting: HTTP status code (transport layer), error code (business logic layer), message (human-readable), details (field-level validation), trace_id (distributed tracing), timestamp (Universal Timestamp format), help_url (documentation link for resolution).
  5. Content negotiation affects error format: Accept: application/json returns structured JSON errors, Accept: application/xml returns XML-formatted errors, Accept: text/html returns user-friendly error pages, API versioning (v1, v2) maintains backward compatibility for error response schemas.
  6. Rate limiting headers communicate capacity: X-RateLimit-Limit (maximum requests per window), X-RateLimit-Remaining (requests left), X-RateLimit-Reset (Unix timestamp when quota resets), 429 status with Retry-After (seconds to wait) enables intelligent client backoff preventing thundering herd problems.
  7. CORS error handling requires special treatment: Preflight OPTIONS requests must return 200 even for authentication errors (HTTP spec requirement), actual request returns appropriate 4xx/5xx with CORS headers (Access-Control-Allow-Origin), browser console shows CORS errors distinct from application errors requiring separate monitoring.

Review Questions

  1. A payment API receives invalid credit card number. Should it return 400 (Bad Request) or 422 (Unprocessable Entity)? Justify your choice based on RFC 7231 and PCI DSS requirements. What information can the error message safely include without violating PCI compliance?
  2. Design error responses for these scenarios: (a) User tries to delete another user's resource (b) API key is valid but has exceeded rate limit (c) Database connection pool is exhausted (d) Upstream payment processor is down. Specify HTTP status codes, error codes, retry guidance, and expected client behavior.
  3. Implement retry logic for HTTP client: 500 errors retry with exponential backoff (max 3 attempts), 502/503 retry with linear backoff (max 5 attempts), 504 increase timeout and retry once, 429 respect Retry-After header. Write pseudocode handling all cases with circuit breaker integration.
  4. An API returns 500 Internal Server Error with detailed stack trace including database credentials. Analyze security implications (GDPR, SOC 2, ISO 27001). Design a compliant error response that balances debugging needs with security requirements. How does Sentry's data scrubbing help?
  5. Configure API Gateway error transformation: Upstream service returns 200 with error payload {"success": false, "error": "Database timeout"}. Gateway must convert to proper HTTP status codes (503 for DB timeout) and WIA-CORE-010 format. Implement transformation rules using Kong or NGINX.
  6. A React SPA calls GraphQL API. GraphQL always returns 200 with errors array. How do you map GraphQL errors to HTTP status codes for monitoring? Design Datadog dashboard showing error rates by type (authentication, validation, system) when underlying protocol doesn't expose status codes.
  7. CORS preflight fails with 401 Unauthorized causing browser to block actual request. Explain why this violates HTTP specification. Fix the implementation: OPTIONS returns 200, subsequent POST with invalid auth returns 401. How does this affect security? Design test cases covering all CORS + auth combinations.
