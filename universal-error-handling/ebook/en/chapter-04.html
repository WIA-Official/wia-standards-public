Chapter Summary

Key Takeaways:

  1. Error propagation in microservices requires correlation IDs (Universal Protocol trace-id format: uuid-v4) flowing through all service calls, enabling distributed tracing from frontend request through 15+ backend services to pinpoint failure origin within seconds using Datadog APM or Jaeger visualization.
  2. Error context enrichment adds service-specific information while propagating upstream: Service-A enriches with user_id and session_id, Service-B adds transaction_id and payment_method, Service-C appends inventory_status, creating comprehensive error context without exposing internal implementation details to external callers.
  3. Synchronous vs asynchronous error handling patterns differ fundamentally: REST/gRPC propagate errors immediately via HTTP status codes requiring timeout management (default: 30s gateway, 5s inter-service), message queues use dead letter queues for poison messages with retry limits (3 attempts over 10 minutes) before manual intervention.
  4. Saga pattern for distributed transactions requires compensating actions: Order service creates order (SUCCESS), Payment service charges card (FAILURE), Order service must rollback with order cancellation, Shipping service never invoked. Error in any step triggers compensation chain recorded in saga log for audit compliance (GDPR right to explanation).
  5. Circuit breaker prevents cascade failures through state machine: CLOSED (normal operation, tracking failures), OPEN (immediately reject requests for 60s after 5 consecutive failures), HALF-OPEN (allow 3 probe requests to test recovery), metrics integration with Datadog triggers alerts when breakers open preventing revenue loss during incidents.
  6. Error boundaries in microservices architecture isolate blast radius: Authentication service down returns cached tokens for 5 minutes degrading to read-only mode, Recommendation service failures don't block checkout (graceful degradation), Payment processor timeout doesn't crash entire order workflow (queue for retry).
  7. Observability tracing with OpenTelemetry standardizes error propagation: Spans tagged with error=true, status_code (HTTP/gRPC), exception details (type, message, stack trace sanitized for PCI compliance), custom attributes (user_tier, feature_flags), enabling Sentry to group errors by root cause across 50+ microservices reducing MTTR from 45 minutes to 8 minutes.

Review Questions

  1. Design error propagation for e-commerce checkout spanning 7 microservices: API Gateway → Auth → Cart → Inventory → Pricing → Payment → Order. Define correlation ID format (Universal Protocol compliant), context enrichment at each hop, timeout values, and failure handling. What happens when Inventory is down?
  2. Implement saga pattern for hotel booking: Flight reservation → Hotel booking → Car rental. Each step can fail after confirmation. Design compensating transactions: What if hotel booking fails after flight is confirmed? How do you prevent double-charging during compensation? Include audit log structure for GDPR compliance.
  3. Configure circuit breaker for payment processor: 100 requests/second baseline, p99 latency normally 200ms. Design thresholds: (a) How many failures trigger OPEN? (b) How long to stay OPEN? (c) How many probes in HALF-OPEN? (d) What errors count vs. ignore (timeout vs. 400 validation)? Integrate with Datadog alerting.
  4. Message queue error handling: Order processing queue receives poison message causing repeated crashes. Design dead letter queue strategy: (a) Retry policy (exponential backoff) (b) Max retry attempts before DLQ (c) DLQ monitoring and alerting (d) Manual replay process (e) How does this satisfy HIPAA audit requirements for healthcare orders?
  5. Trace distributed failure using OpenTelemetry: User reports "checkout failed" (no error message shown). You have trace_id: "a1b2c3d4". Using Datadog APM, describe step-by-step investigation: (a) Which service failed? (b) What was the error code? (c) What upstream dependencies were called? (d) Which database query timeout caused failure? Design the trace view layout.
  6. Graceful degradation strategy: Recommendation service provides personalized product suggestions. Service dependencies: User profile (p99: 50ms), Product catalog (p99: 100ms), ML model (p99: 500ms). Design fallback chain when dependencies fail. How does this affect user experience? What metrics track degradation quality?
  7. Error context sanitization for security: Payment service error contains full credit card number and CVV in stack trace. Design sanitization pipeline: (a) Identify PII/PCI data patterns (b) Redact before logging (c) Preserve debugging value (d) Ensure Sentry receives compliant data. How do you test sanitization completeness?
