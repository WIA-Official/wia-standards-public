Chapter 4: Protocol Specification

Real-Time Conversion Pipeline and Avatar Streaming

Protocol Specification

4.1 Overview

The Voice-Sign protocol defines how audio data flows through the conversion pipeline, from capture to final avatar rendering. It supports both batch processing for pre-recorded audio and real-time streaming for live communication.

4.4 Chunking Strategy

Audio Chunking

Audio is divided into small chunks for continuous processing:

  • Chunk Size: 100-200ms of audio
  • Overlap: 50ms overlap between chunks for context
  • Buffering: 500ms buffer for ASR context window
  • VAD: Voice Activity Detection to skip silence

Transcript Chunking

Transcription results are delivered incrementally:

  • Partial Results: Low-confidence intermediate transcriptions
  • Final Results: High-confidence finalized text
  • Word Boundaries: Results sent at natural word/phrase boundaries

4.6 Caching and Optimization

Translation Cache

Frequently used phrases are cached to reduce latency:

  • Cache Key: Hash of (text + source_lang + target_lang)
  • TTL: 7 days for common phrases, 24 hours for others
  • Size: 10,000 most common phrases per language pair
  • Invalidation: Model updates trigger cache refresh

Pose Cache

Common sign sequences are pre-generated:

  • Common Signs: Top 1000 signs per language
  • Common Phrases: Greetings, questions, emergencies
  • Storage: Compressed pose data in CDN

4.7 Error Handling and Recovery

Retry Strategy

// Exponential backoff
retry_delays = [1s, 2s, 4s, 8s, 16s]

// Max retries
max_retries = 5

// Retry conditions
- Network errors (5xx)
- Timeout errors
- Rate limit errors (with backoff)

// No retry conditions
- Invalid input (4xx)
- Authentication errors
- Resource not found

Fallback Mechanisms

  • Model Fallback: Switch to backup ASR model if primary fails
  • Quality Degradation: Lower quality output if high quality unavailable
  • Text-Only Mode: Provide transcript if sign generation fails
  • Cached Response: Return cached result if available

4.8 Avatar Streaming Protocol

Streaming Format

// Frame-based streaming
{
  "type": "pose_frame",
  "frame": {
    "index": 42,
    "timestamp_ms": 1400,
    "joints": [
      {"joint_id": 0, "position": {"x": 0, "y": 1.5, "z": 0}, "rotation": {...}},
      {"joint_id": 1, "position": {"x": 0.1, "y": 1.4, "z": 0}, "rotation": {...}}
    ],
    "left_hand": {...},
    "right_hand": {...},
    "face": {...}
  },
  "interpolation": "linear",  // or "cubic", "slerp"
  "duration_ms": 33  // Time to next frame
}

Compression

Pose data is compressed for efficient streaming:

  • Delta Encoding: Only send changes from previous frame
  • Quantization: Reduce precision of position/rotation data
  • Joint Pruning: Omit static joints (e.g., legs for sign language)
  • Huffman Coding: Compress repeated patterns

4.9 Synchronization

Audio-Visual Sync

Ensure avatar movements match original audio timing:

  • Timestamps: All data includes precise timestamps
  • Clock Sync: NTP synchronization for distributed systems
  • Buffer Management: Jitter buffer to handle network variance
  • Drift Compensation: Periodic resynchronization

  • Pose Quality: Smoothness and naturalness metrics
  • End-to-End Quality: Overall translation quality score
