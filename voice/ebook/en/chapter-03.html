03.7 In Depth — Phase 1 — utterance and TTS request envelopes

The WIA Voice standard treats voice as a full-stack data flow: capture device → on-device pre-processing → wire envelope → server-side ASR or biometric → response envelope → client rendering. Each step has a published shape; the goal of this chapter, like every chapter in this book, is to make that shape and its conformance gate explicit so an integrator can build a Voice-conformant component without reading the entire specification first.

For the topic at hand — Phase 1 — utterance and TTS request envelopes — the relevant Phase artifacts are: a JSON envelope (Phase 1), a signed HTTP endpoint (Phase 2), a federation handshake when crossing organisational boundaries (Phase 3), and a bridge profile for non-WIA codecs and tools (Phase 4). Implementations are tested by the open conformance suite at https://github.com/WIA-Official/wia-voice-conformance, which exercises every shape mentioned here against a mock host. Hosts publishing bridge_profile=Full SHOULD additionally pass the suite's extension tests for at least one supported third-party voice platform (e.g., Azure Speech, Google Speech-to-Text, Whisper, Coqui TTS).

Two recurring design choices show up across every chapter and are worth flagging once here. First, every voice envelope carries an explicit consent_envelope_id pointer; voice data is biometric and the standard treats it as such. Second, every wire envelope is signed with Ed25519 (IETF RFC 8032) and the federation handshake reuses the same shape that WIA-SOCIAL Phase 3 §5 fixes; this keeps the cross-standard mental model uniform.

03.8 Reference Implementation Walk-Through

The reference container wia/voice-host:1.0.0 implements every Phase 2 endpoint and ships with the open conformance suite preconfigured. A typical local exploration looks like:

docker run -p 8080:8080 wia/voice-host:1.0.0
curl -X POST http://localhost:8080/voice/asr/recognise \
     -H "Content-Type: application/json" \
     -d '{ "wia_voice_version":"1.0.0", "type":"asr_utterance", "...":"..." }'

The container ships with mock voice data so an integrator can exercise every endpoint without recording any actual user voice. Once the integration is working against the mock, the operator swaps in their real ASR or TTS backend by setting the WIA_VOICE_ASR_BACKEND and WIA_VOICE_TTS_BACKEND environment variables.

03.9 Operational Considerations

Voice infrastructure has three operational considerations that integrators consistently underestimate. First, audio storage costs scale with retention; the standard makes retention a per-consent decision so an operator can size storage against the actual scope. Second, model versioning matters; an envelope MUST carry the model identifier (ASR model, TTS voice, voiceprint embedder) so a downstream consumer can detect when a model upgrade changes behaviour for a previously-stored utterance. Third, voice data is biometric and many jurisdictions impose stricter rules than for text data; the standard's consent envelope is designed to map cleanly onto those rules but operators MUST consult their local counsel before going to production.
