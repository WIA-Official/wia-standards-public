Chapter 05: Phase 2 — API Interface

5.1 Phase 2 in one paragraph

Phase 2 specifies the HTTPS surface for water scarcity data: publish, query, stream, federation handshake. Transport is TLS 1.3 (IETF RFC 8446) only; HTTP/2 RECOMMENDED.

5.2 Discovery

Every host publishes a discovery document at /.well-known/wia-water-scarcity-response listing endpoints, supported drought index kinds (SPI/SPEI/PDSI variants), supported jurisdictions, and rate limits.

5.3 Drought index endpoints

POST /wsr/drought                       (signed)
GET  /wsr/drought?basin_id={id}&from={ts}&to={ts}
GET  /wsr/drought/stream?basin_id={id}  (Server-Sent Events)

5.4 Allocation endpoints

POST /wsr/allocation                    (signed; jurisdictional authority required)
GET  /wsr/allocation/{allocation_id}
GET  /wsr/allocation?basin_id={id}&year={n}

5.5 Restriction endpoints

POST /wsr/restriction                   (signed; jurisdictional authority required)
GET  /wsr/restriction/{restriction_id}
GET  /wsr/restriction?jurisdiction_id={id}&active=true

5.6 Emergency endpoints

POST /wsr/emergency                     (signed; emergency-declaration authority required)
GET  /wsr/emergency/{emergency_id}
GET  /wsr/emergency/stream?basin_id={id}

Emergency stream is the primary mechanism for cross-jurisdiction early warning; reconnecting clients pass Last-Event-Id.

5.7 Authentication

Writes require HTTP Message Signatures per IETF RFC 9421. Public reads are anonymous (transparency); jurisdictional-detail reads (specific water-rights holders, individual restriction targets) require authentication.

Watershed-First Design

Every choice in this standard is filtered through: does this make the basin\x27s aggregate water-scarcity picture more visible to the people in the basin, or just more accessible to the bureaucratic centre? When the two conflict, basin-population visibility wins. That is why the standard mandates public read for aggregate basin data, refuses to centralise the data into a single coordinator, and requires every cross-jurisdiction read to be audit-logged so basin populations can verify what their water data is doing.

Implementer Notes

Three operational concerns recur in water-scarcity infrastructure and are baked into the standard. First, jurisdiction authority verification: writes to allocation/restriction/emergency endpoints require the writer to demonstrate authority for the jurisdiction (verified via WIA-OMNI-API jurisdictional_authority claims). Second, replay defence for emergency endpoints: a duplicated emergency declaration triggers duplicate response activations, so the seen-nonce cache MUST be persistent across host restarts. Third, public dashboard rate limits: drought-index streams attract dashboard traffic during dry seasons; hosts SHOULD provision read replicas to keep ingest paths responsive.
