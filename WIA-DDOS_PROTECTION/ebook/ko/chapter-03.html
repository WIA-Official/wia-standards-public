네트워크 수준 방어: 첫 번째 방어선

1. 다층 방어 전략 (Defense in Depth)

DDoS 방어는 단일 솔루션이 아닌 다층 방어가 필수입니다. 각 계층에서 공격을 탐지하고 차단합니다.

# 다층 방어 아키텍처 인터넷 ↓ [1. ISP/Transit Provider] ← BGP Flowspec, Blackholing ↓ [2. DDoS Mitigation Service] ← Cloudflare, AWS Shield ↓ [3. Edge Router/Firewall] ← ACL, Rate Limiting ↓ [4. Load Balancer] ← Connection Limiting, Health Check ↓ [5. Web Server] ← iptables, fail2ban ↓ [6. Application] ← Rate Limiting, CAPTCHA 각 계층이 독립적으로 방어 → 한 계층 돌파해도 다음 계층이 차단

2. Traffic Filtering (트래픽 필터링)

2.1 ACL (Access Control List)

가장 기본적인 방어로, IP/포트 기반으로 트래픽을 차단합니다.

# Cisco ACL 예시 ! 알려진 공격자 IP 차단 access-list 100 deny ip host 1.2.3.4 any access-list 100 deny ip 5.6.7.0 0.0.0.255 any ! 특정 국가 IP 대역 차단 (예: 북한) access-list 100 deny ip 175.45.176.0 0.0.15.255 any ! ICMP Flood 방어 (초당 100 패킷 제한) access-list 100 permit icmp any any echo rate-limit 100 ! 허용 access-list 100 permit ip any any interface GigabitEthernet0/0 ip access-group 100 in # Linux iptables ACL # 특정 IP 차단 iptables -A INPUT -s 1.2.3.4 -j DROP # IP 대역 차단 iptables -A INPUT -s 5.6.7.0/24 -j DROP # 지리적 차단 (GeoIP 모듈) iptables -A INPUT -m geoip --src-cc CN,RU -j DROP

2.2 Rate Limiting (속도 제한)

# iptables로 Rate Limiting # 초당 10개 SYN 패킷만 허용 iptables -A INPUT -p tcp --syn -m limit --limit 10/s --limit-burst 20 -j ACCEPT iptables -A INPUT -p tcp --syn -j DROP # 설명: # --limit 10/s: 초당 10개 # --limit-burst 20: 초기 버스트 20개 허용 # ICMP Rate Limiting iptables -A INPUT -p icmp --icmp-type echo-request -m limit --limit 1/s -j ACCEPT iptables -A INPUT -p icmp --icmp-type echo-request -j DROP # 연결당 Rate Limiting iptables -A INPUT -p tcp --dport 80 -m connlimit --connlimit-above 20 --connlimit-mask 32 -j REJECT # 한 IP당 최대 20개 연결

3. BGP Flowspec

BGP를 활용한 동적 트래픽 필터링으로, ISP 수준에서 공격 차단이 가능합니다.

# BGP Flowspec 예시 (ExaBGP) # UDP Flood 공격 차단 flow route { match { destination 203.0.113.0/24; # 공격받는 네트워크 protocol udp; destination-port =53; # DNS 포트 packet-length >1024; # 큰 패킷 } then { rate-limit 1000000; # 1 Mbps로 제한 # 또는 discard; # 완전 차단 } } # SYN Flood 차단 flow route { match { destination 203.0.113.10/32; protocol tcp; tcp-flags syn; packet-length <100; } then { discard; } } # 장점: # 1. ISP 라우터에서 차단 → 대역폭 절약 # 2. 동적 규칙 → 실시간 대응 # 3. 표준화 (RFC 8955)

4. Blackholing & Sinkholing

4.1 Blackhole Routing

공격 트래픽을 "블랙홀"로 보내 폐기합니다. 극단적이지만 효과적입니다.

# RTBH (Remotely Triggered Black Hole) # BGP를 통해 ISP에 블랙홀 요청 # Step 1: 공격받는 IP를 블랙홀 커뮤니티로 광고 router bgp 65000 network 203.0.113.10 mask 255.255.255.255 neighbor 10.0.0.1 send-community neighbor 10.0.0.1 route-map SET_BLACKHOLE out route-map SET_BLACKHOLE permit 10 match ip address prefix-list BLACKHOLE set community 65000:666 # 블랙홀 커뮤니티 # Step 2: ISP 라우터가 수신 후 Null0으로 라우팅 ip route 203.0.113.10 255.255.255.255 Null0 # 결과: 해당 IP로 가는 모든 트래픽 폐기 # 단점: 공격 트래픽뿐 아니라 정상 트래픽도 차단됨!

4.2 Sinkhole

공격 트래픽을 분석 서버로 리다이렉트하여 공격 패턴을 학습합니다.

# DNS Sinkhole (악성 도메인 차단) # 봇넷 C&C 도메인을 sinkhole 서버로 리다이렉트 # BIND DNS 설정 zone "malware-c2.com" { type master; file "/etc/bind/db.sinkhole"; }; # /etc/bind/db.sinkhole @ IN SOA ns.sinkhole.local. admin.sinkhole.local. (2024010101 3600 1800 604800 86400) @ IN NS ns.sinkhole.local. @ IN A 192.0.2.1 # Sinkhole 서버 IP * IN A 192.0.2.1 # 모든 서브도메인 # Sinkhole 서버에서 봇 IP 수집 및 차단

5. Scrubbing Center (트래픽 정화 센터)

공격 트래픽을 정화 센터로 우회하여 정상 트래픽만 오리진 서버로 전달합니다.

# Scrubbing Center 동작 원리 정상 상태: Client → Internet → Origin Server 공격 탐지 시: 1. BGP로 트래픽 Scrubbing Center로 우회 Client → Internet → Scrubbing Center 2. Scrubbing Center에서 공격 필터링 - DPI (Deep Packet Inspection) - 시그니처 매칭 - Rate Limiting 3. 정상 트래픽만 Origin으로 전달 Scrubbing Center → Origin Server (GRE 터널) # BGP Anycast로 가장 가까운 Scrubbing Center 선택 # 전 세계 10+ PoP (Point of Presence)

6. Anycast Routing

동일 IP를 여러 지역에 광고하여 트래픽을 분산하고, 공격 영향을 지역화합니다.

# Anycast 예시 # 동일 IP (203.0.113.1)를 전 세계 데이터센터에서 광고 서울 DC: 203.0.113.1/32 광고 도쿄 DC: 203.0.113.1/32 광고 싱가포르 DC: 203.0.113.1/32 광고 # 사용자는 가장 가까운 DC로 라우팅됨 한국 사용자 → 서울 DC 일본 사용자 → 도쿄 DC # DDoS 공격 시: 공격자(중국) → 싱가포르 DC로 트래픽 집중 → 서울/도쿄 DC는 정상 서비스 유지! # 장점: 1. 지역별 트래픽 분산 2. 공격 영향 지역화 3. 자동 Failover (한 DC 다운 시 다른 DC로) # Anycast DNS (Cloudflare 1.1.1.1) $ dig @1.1.1.1 example.com # 한국: 서울 서버 응답 # 미국: LA 서버 응답

7. SYN Cookies

SYN Flood 방어의 핵심 기술로, 상태 저장 없이 연결을 처리합니다.

# SYN Cookie 원리 정상 TCP: Client Server SYN → [상태 저장: SYN_RECEIVED] ← SYN-ACK ACK → [상태 변경: ESTABLISHED] SYN Flood: Attacker Server SYN(가짜IP)→ [상태 저장] ← 메모리 고갈! SYN(가짜IP)→ [상태 저장] ... SYN Cookie 방식: Attacker Server SYN → [상태 저장 안 함!] ← SYN-ACK (쿠키 포함) (응답 없음) [타임아웃, 메모리 낭비 없음] 정상 클라이언트: Client Server SYN → [상태 저장 안 함] ← SYN-ACK (쿠키) ACK(쿠키)→ [쿠키 검증 → 연결 성공] # Linux 활성화 sysctl -w net.ipv4.tcp_syncookies=1 # 영구 적용 echo "net.ipv4.tcp_syncookies = 1" >> /etc/sysctl.conf sysctl -p # 단점: # - TCP 옵션 일부 손실 (Window Scaling 등) # - 성능 오버헤드 약간 증가

8. Connection Tracking & Limiting

# conntrack (Linux Connection Tracking) # 연결 상태 추적 및 제한 # 최대 연결 수 증가 (기본: 65,536) sysctl -w net.netfilter.nf_conntrack_max=1048576 sysctl -w net.nf_conntrack_max=1048576 # 타임아웃 조정 (SYN_RECEIVED 상태) sysctl -w net.netfilter.nf_conntrack_tcp_timeout_syn_recv=10 # IP당 연결 제한 iptables -A INPUT -p tcp --dport 80 -m connlimit --connlimit-above 50 --connlimit-mask 32 -j REJECT --reject-with tcp-reset # 초당 신규 연결 제한 iptables -A INPUT -p tcp --dport 80 -m state --state NEW -m recent --set iptables -A INPUT -p tcp --dport 80 -m state --state NEW -m recent --update --seconds 1 --hitcount 10 -j DROP # 연결 상태 확인 conntrack -L | wc -l # 현재 연결 수 netstat -an | grep SYN_RECV | wc -l # SYN_RECV 상태 수

🎯 핵심 요점

  • 다층 방어 필수: ISP → Scrubbing → Edge → LB → Server → App
  • ACL: IP/포트 기반 기본 필터링
  • Rate Limiting: 초당 패킷/연결 수 제한
  • BGP Flowspec: ISP 수준 동적 필터링, RFC 8955
  • Blackholing: 극단적 차단 (정상 트래픽도 폐기)
  • Scrubbing Center: 공격 정화 후 정상 트래픽만 전달
  • Anycast: 트래픽 분산, 공격 영향 지역화
  • SYN Cookie: SYN Flood 방어, 상태 저장 없음
  • Connection Limiting: IP당/초당 연결 수 제한
  • 弘益人間: 네트워크 방어로 안정적인 서비스, 모두를 위한 인터넷 🌐
