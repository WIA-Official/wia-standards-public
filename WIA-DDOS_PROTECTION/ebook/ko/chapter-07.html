🚨 Chapter 07

사고 대응: DDoS 공격 대응 플레이북

1. DDoS 사고 대응 프로세스 (NIST 기반)

NIST (미국 표준기술연구소) SP 800-61 기준에 따른 사고 대응 프로세스는 4단계로 구성됩니다.

단계 영문명 주요 활동 소요 시간
1. 준비 Preparation 팀 구성, 도구 준비, 훈련, 플레이북 작성 사전 작업
2. 탐지/분석 Detection & Analysis 공격 탐지, 유형 분석, 영향 평가 0-10분
3. 억제/복구 Containment & Recovery 공격 차단, 트래픽 우회, 서비스 복구 10분-2시간
4. 사후 검토 Post-Incident Activity 근본 원인 분석, 재발 방지, 프로세스 개선 1-5일

2. 사고 대응 플레이북 (Playbook)

DDoS 공격 탐지 시 실행할 구체적인 절차를 문서화한 플레이북입니다.

🔴 Phase 1: 탐지 및 초기 대응 (0-5분)

  1. 알림 수신 확인
    • Slack/PagerDuty/SMS 알림 확인
    • 대시보드에서 이상 메트릭 확인 (대역폭, PPS, 에러율)
  2. 공격 유형 분류
    • Volumetric (대역폭 소진): UDP Flood, ICMP Flood, Amplification
    • Protocol (리소스 소진): SYN Flood, ACK Flood
    • Application (L7): HTTP Flood, Slowloris
  3. 사고 대응 팀 소집
    • 사고 대응 리더 (Incident Commander) 지정
    • 네트워크 엔지니어, 보안 팀, DevOps 소집
    • 전용 Slack 채널 생성 (#incident-ddos-20250115)

🟠 Phase 2: 즉시 대응 (5-15분)

  1. Rate Limiting 즉시 적용
  2. 상위 공격 IP 차단
  3. CDN/Scrubbing Center 활성화
  4. Auto-Scaling 활성화 (여유 있을 경우)
# Phase 2 자동화 스크립트 (emergency-response.sh) #!/bin/bash # DDoS 긴급 대응 스크립트 echo "[$(date)] 🚨 DDoS 긴급 대응 시작" # 1. Rate Limiting 적용 echo "1️⃣ Rate Limiting 적용 중..." iptables -A INPUT -p tcp --syn -m limit --limit 10/s --limit-burst 20 -j ACCEPT iptables -A INPUT -p tcp --syn -j DROP echo "✓ Rate Limiting 완료" # 2. Top 10 공격 IP 자동 차단 echo "2️⃣ Top 10 공격 IP 차단 중..." netstat -ntu | awk '{print $5}' | cut -d: -f1 | sort | uniq -c | sort -nr | head -10 | while read count ip; do if [ $count -gt 100 ]; then iptables -A INPUT -s $ip -j DROP echo " 차단: $ip ($count 연결)" fi done echo "✓ IP 차단 완료" # 3. Cloudflare 공격 모드 활성화 echo "3️⃣ Cloudflare 공격 모드 활성화 중..." curl -X PATCH "https://api.cloudflare.com/client/v4/zones/${ZONE_ID}/settings/security_level" \ -H "X-Auth-Email: ${CF_EMAIL}" \ -H "X-Auth-Key: ${CF_API_KEY}" \ -H "Content-Type: application/json" \ --data '{"value":"under_attack"}' echo "✓ Cloudflare 완료" # 4. 알림 전송 echo "4️⃣ 팀 알림 전송 중..." curl -X POST $SLACK_WEBHOOK_URL \ -H 'Content-Type: application/json' \ -d '{ "text": "🚨 DDoS 긴급 대응 완료", "blocks": [ {"type": "section", "text": {"type": "mrkdwn", "text": "*긴급 대응 조치 완료*\n✓ Rate Limiting\n✓ IP 차단\n✓ Cloudflare 공격 모드"}} ] }' echo "✓ 알림 완료" echo "[$(date)] ✅ 긴급 대응 완료"

🟡 Phase 3: 고급 대응 (15분-1시간)

  1. ISP/Cloud Provider 협력 요청
    • AWS Shield Advanced DRT 팀 활성화
    • ISP에 BGP Blackholing 요청
  2. 트래픽 분석 및 서명 생성
    • NetFlow/sFlow 데이터 분석
    • 공격 패턴 기반 방화벽 규칙 생성
  3. 대체 인프라 활성화
    • 백업 데이터센터 트래픽 분산
    • Anycast 네트워크 활용
  4. 고객/이해관계자 커뮤니케이션
    • 상태 페이지 업데이트 (status.example.com)
    • 소셜 미디어 공지

4. 사고 대응 체크리스트

✅ DDoS 대응 체크리스트

탐지 단계 (0-5분)
  • ☐ 알림 확인 및 사고 대응 팀 소집
  • ☐ 모니터링 대시보드에서 공격 유형 확인
  • ☐ 공격 규모 및 영향 범위 평가
  • ☐ 사고 대응 전용 채널 생성 (#incident-ddos-YYYYMMDD)
초기 대응 (5-15분)
  • ☐ Rate Limiting 적용 (iptables/nginx)
  • ☐ Top 공격 IP 차단
  • ☐ CDN/WAF 공격 모드 활성화
  • ☐ 긴급 대응 스크립트 실행
  • ☐ Auto-Scaling 활성화 (가능한 경우)
고급 대응 (15-60분)
  • ☐ ISP/Cloud Provider에 협력 요청
  • ☐ BGP Blackholing 또는 Scrubbing Center 활성화
  • ☐ 트래픽 분석 및 서명 기반 차단
  • ☐ 백업 인프라로 트래픽 분산
  • ☐ 상태 페이지 및 고객 공지
복구 및 검증 (1-2시간)
  • ☐ 서비스 정상 동작 확인
  • ☐ 응답 시간 및 에러율 모니터링
  • ☐ 공격 차단 규칙 검토 및 최적화
  • ☐ 로그 및 증거 수집 (법적 대응 준비)
사후 검토 (1-5일)
  • ☐ 사고 타임라인 작성
  • ☐ 근본 원인 분석 (RCA) 수행
  • ☐ 대응 프로세스 평가 및 개선
  • ☐ 재발 방지 계획 수립
  • ☐ 팀 교육 및 플레이북 업데이트

5. 사후 검토 및 개선 (Post-Mortem)

공격 종료 후 48시간 내에 사후 검토 회의를 개최하여 교훈을 정리합니다.

# DDoS 사후 검토 템플릿 ## 사고 개요 - **발생 일시**: 2025-01-15 14:23:00 KST - **종료 일시**: 2025-01-15 16:45:00 KST - **지속 시간**: 2시간 22분 - **사고 등급**: P1 (Critical) ## 공격 특성 - **공격 유형**: HTTP Flood + SYN Flood 복합 공격 - **최대 트래픽**: 120 Gbps, 35M PPS - **공격 소스**: 봇넷 (주로 IoT 디바이스), 67개 국가 - **타겟**: www.example.com, api.example.com ## 영향 범위 - **서비스 중단 시간**: 23분 (14:23-14:46) - **부분 장애 시간**: 1시간 59분 - **영향 받은 사용자**: 약 150,000명 - **매출 손실**: 약 $45,000 (추정) ## 타임라인 | 시각 | 이벤트 | |------|--------| | 14:23 | 공격 시작, 알림 발생 | | 14:25 | 사고 대응 팀 소집 | | 14:28 | 긴급 대응 스크립트 실행 | | 14:35 | Cloudflare 공격 모드 활성화 | | 14:46 | 서비스 부분 복구 (일부 사용자 접속 가능) | | 15:10 | AWS Shield DRT 팀 참여 | | 15:30 | ISP Scrubbing Center 활성화 | | 16:15 | 공격 트래픽 95% 감소 | | 16:45 | 서비스 완전 복구 | ## 잘된 점 (What Went Well) 1. ✅ 알림 시스템이 2분 내에 팀을 소집했습니다 2. ✅ 긴급 대응 스크립트가 효과적으로 작동했습니다 3. ✅ Cloudflare CDN이 트래픽의 70%를 차단했습니다 4. ✅ 고객 커뮤니케이션이 신속했습니다 (상태 페이지 15분 내 업데이트) ## 개선이 필요한 점 (What Can Be Improved) 1. ❌ Auto-Scaling이 제대로 동작하지 않았습니다 (설정 오류) 2. ❌ ISP 협력 요청에 25분 소요 (절차 복잡) 3. ❌ 백업 데이터센터로의 Failover가 수동이었습니다 4. ❌ 일부 팀원이 플레이북을 숙지하지 못했습니다 ## 액션 아이템 | 항목 | 담당자 | 기한 | 상태 | |------|--------|------|------| | Auto-Scaling 설정 검증 및 수정 | DevOps팀 | 2025-01-20 | 진행 중 | | ISP 핫라인 구축 (24/7 직통) | 네트워크팀 | 2025-01-25 | 미착수 | | 자동 Failover 스크립트 개발 | SRE팀 | 2025-02-01 | 미착수 | | 전 직원 DDoS 대응 훈련 실시 | 보안팀 | 2025-01-31 | 계획 중 | | 플레이북 업데이트 및 시뮬레이션 | 전체 | 2025-02-15 | 미착수 | ## 재발 방지 계획 1. 월 1회 DDoS 시뮬레이션 훈련 (Table-Top Exercise) 2. ⚠️ 법적 대응 고려사항

DDoS 공격은 대부분 국가에서 불법입니다. 공격 증거를 보존하여 법적 대응을 준비하세요.

  • 증거 수집: 로그, NetFlow 데이터, 패킷 캡처 (pcap) 보관
  • 신고: 사이버안전국, FBI IC3, Interpol 등에 신고
  • ISP 협력: 공격 소스 추적 요청
  • 보험 청구: Cyber Insurance 가입 시 피해액 청구

🎯 핵심 요점

  • NIST 4단계: 준비 → 탐지/분석 → 억제/복구 → 사후 검토
  • 플레이북 필수: 사전 문서화로 신속 대응
  • Phase 1 (0-5분): 팀 소집, 공격 유형 분류
  • Phase 2 (5-15분): Rate Limiting, IP 차단, CDN 활성화
  • Phase 3 (15-60분): ISP 협력, Scrubbing, 백업 인프라
  • 자동화: 긴급 대응 스크립트로 수동 작업 최소화
  • 사후 검토: 48시간 내 Post-Mortem, 재발 방지
  • 훈련 필수: 월 1회 시뮬레이션으로 대응 능력 유지
  • 弘益人間: 신속한 대응으로 사용자 피해 최소화 🛡️
