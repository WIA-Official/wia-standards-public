DDoS 공격 유형: OSI 7계층별 공격 분류

1. DDoS 공격의 3가지 주요 분류

DDoS 공격은 OSI 7계층 모델에서 공격하는 계층에 따라 크게 3가지로 분류됩니다.

🎯 DDoS 공격 분류

  • Volumetric Attacks (대역폭 소진): L3/L4 네트워크 계층, 대량 트래픽으로 대역폭 포화
  • Protocol Attacks (프로토콜 공격): L3/L4 프로토콜 취약점 악용, 서버 리소스 고갈
  • Application Layer Attacks (애플리케이션 공격): L7 HTTP/HTTPS, 정상 요청처럼 위장

2. Volumetric Attacks (대역폭 소진 공격)

대량의 트래픽을 보내 네트워크 대역폭을 포화시키는 공격입니다. 가장 흔하고 강력합니다.

2.1 UDP Flood

# UDP Flood 공격 원리 # UDP는 연결 없는 프로토콜 (Connectionless) 공격자 → 목표 서버 (53, 123, 161 등 UDP 포트) - UDP 패킷 초당 수백만 개 전송 - 서버는 모든 패킷 처리 시도 → CPU/대역폭 고갈 예시: DNS Amplification 1. 공격자: 가짜 출발지 IP (피해자 IP)로 DNS 쿼리 2. DNS 서버: 큰 응답을 피해자에게 전송 3. 증폭률: 1:50 (50바이트 요청 → 2500바이트 응답) # hping3로 UDP Flood 시뮬레이션 (테스트 환경에서만!) sudo hping3 --udp --flood --rand-source -p 53 TARGET_IP # --flood: 최대한 빠르게 # --rand-source: 무작위 출발지 IP # -p 53: DNS 포트

2.2 ICMP Flood (Ping Flood)

ICMP Echo Request (ping)을 대량으로 보내 대역폭과 CPU를 소진시킵니다.

# Ping Flood # 서버는 각 ping에 대해 Echo Reply 응답 필요 # 공격 명령 (테스트 환경에서만!) sudo hping3 --icmp --flood --rand-source TARGET_IP # Smurf Attack (증폭 공격) # 1. 공격자: 브로드캐스트 주소로 ICMP 전송 (출발지 IP = 피해자) # 2. 네트워크의 모든 장치: 피해자에게 ICMP 응답 # 3. 증폭률: 1:100+ 방어: ICMP Rate Limiting iptables -A INPUT -p icmp --icmp-type echo-request -m limit --limit 1/s -j ACCEPT iptables -A INPUT -p icmp --icmp-type echo-request -j DROP

2.3 DNS Amplification (DNS 증폭 공격)

DNS 프로토콜의 특성을 악용한 증폭 공격으로, 2013년 Spamhaus 공격(300Gbps)에 사용되었습니다.

단계행동데이터량
1. 요청공격자 → DNS (ANY 레코드)60 바이트
2. 응답DNS → 피해자 (전체 레코드)3000 바이트
증폭률1:50 (3000/60)

2.4 NTP Amplification (NTP 증폭 공격)

# NTP (Network Time Protocol) 취약점 # monlist 명령: 최근 600개 클라이언트 목록 반환 # 공격 흐름 공격자: ntpdc -c monlist -n NTP_SERVER (출발지 IP = 피해자) NTP 서버: 피해자에게 수백 KB 응답 전송 증폭률: 1:206 (90바이트 → 18,540바이트) # 방어: NTP 서버 설정 (ntp.conf) disable monitor # monlist 명령 비활성화 restrict default kod nomodify notrap nopeer noquery restrict -6 default kod nomodify notrap nopeer noquery

3. Protocol Attacks (프로토콜 공격)

TCP/IP 프로토콜의 취약점을 악용하여 서버의 상태 테이블이나 방화벽을 고갈시킵니다.

3.1 SYN Flood (가장 흔한 프로토콜 공격)

# TCP 3-Way Handshake 악용 # 정상: SYN → SYN-ACK → ACK # 공격: SYN → SYN-ACK → (응답 안 함) 정상 연결: Client Server | | |-------- SYN ----------->| (1. 연결 요청) |<------ SYN-ACK ---------| (2. 수락) |-------- ACK ----------->| (3. 확인) | | [연결 성공] SYN Flood 공격: Attacker Server | | |-- SYN (가짜 IP 1) ----->| 서버: 대기 (SYN_RECEIVED) |-- SYN (가짜 IP 2) ----->| 서버: 대기 |-- SYN (가짜 IP 3) ----->| 서버: 대기 ... ... (응답 없음) [Half-Open 연결 누적] [백로그 큐 가득 참] [정상 연결 거부!] # 공격 시뮬레이션 (테스트 환경) sudo hping3 -S --flood --rand-source -p 80 TARGET_IP # -S: SYN 플래그만 # 방어: SYN Cookies 활성화 (Linux) sysctl -w net.ipv4.tcp_syncookies=1 # SYN Cookie: 상태 저장 없이 SYN-ACK 생성

3.2 ACK Flood

ACK 패킷을 대량으로 전송하여 방화벽의 상태 테이블을 채웁니다.

# ACK Flood # 방화벽은 ACK 패킷을 기존 연결로 간주 → 상태 테이블 검색 sudo hping3 -A --flood --rand-source -p 80 TARGET_IP # 방어: Invalid ACK 차단 iptables -A INPUT -p tcp --tcp-flags ACK ACK -m state --state INVALID -j DROP

3.3 Fragmentation Attack (단편화 공격)

# IP 단편화를 악용한 공격 # 서버는 모든 조각을 모아 재조립해야 함 # Ping of Death # 최대 IP 패킷 크기 (65,535바이트) 초과 전송 # 재조립 시 버퍼 오버플로우 # Teardrop Attack # 조각 오프셋을 잘못 설정하여 재조립 실패 유도 # 방어: 단편화 패킷 차단 iptables -A INPUT -f -j DROP

4. Application Layer Attacks (L7 공격)

HTTP/HTTPS 프로토콜을 악용하여 정상 트래픽처럼 위장합니다. 탐지가 가장 어렵습니다.

4.1 HTTP Flood

# HTTP GET/POST Flood # 대량의 HTTP 요청으로 웹 서버 CPU/DB 고갈 # 공격 패턴 1: 무작위 URL GET /?random=12345678 HTTP/1.1 GET /?random=87654321 HTTP/1.1 # 캐시 우회 → 서버가 매번 처리 # 공격 패턴 2: 리소스 집약적 페이지 GET /search?q=expensive_query HTTP/1.1 POST /api/report (대량 데이터 처리) # 봇넷으로 분산 공격 (탐지 회피) Bot 1 (IP: 1.2.3.4): GET / (초당 10 요청) Bot 2 (IP: 5.6.7.8): GET / (초당 10 요청) ... Bot 10,000: 총 초당 10만 요청 # 방어: Rate Limiting (Nginx) limit_req_zone $binary_remote_addr zone=one:10m rate=10r/s; server { location / { limit_req zone=one burst=20; } }

4.2 Slowloris

느린 HTTP 요청으로 서버의 연결을 고갈시키는 교묘한 공격입니다.

# Slowloris 원리 # HTTP 요청을 완성하지 않고 계속 연결 유지 정상 요청: GET / HTTP/1.1\r\n Host: example.com\r\n \r\n ← 빈 줄 (헤더 끝) Slowloris 공격: GET / HTTP/1.1\r\n Host: example.com\r\n X-Header-1: value\r\n (10초 대기) X-Header-2: value\r\n (10초 대기) ... (헤더 끝을 보내지 않음!) # 서버는 계속 대기 → 최대 연결 수 도달 → 정상 요청 거부 # 공격 도구 python slowloris.py TARGET_IP -s 1000 # -s 1000: 1000개 동시 연결 # 방어: 요청 타임아웃 설정 (Apache) Timeout 30 RequestReadTimeout header=20-40,body=20-40 # 방어: 최소 데이터 전송률 강제 (Nginx) client_body_timeout 10s; client_header_timeout 10s;

4.3 Slow Read/Slow POST

# Slow Read Attack # TCP 윈도우 크기를 0으로 설정 → 서버가 데이터 전송 못함 Client → Server: GET /large_file.zip Server → Client: (데이터 전송 시작) Client → Server: TCP Window Size = 0 (더 받지 말라는 신호) Server: 연결 유지, 전송 대기... # Slow POST Attack # POST 데이터를 아주 천천히 전송 POST /upload HTTP/1.1 Content-Length: 1000000 \r\n (1바이트 전송) (10초 대기) (1바이트 전송) ... # 방어: 최소 전송률 강제 LimitRequestBody 10485760 # 10MB 제한 MinRate 500 # 최소 500 bytes/s

4.4 Cache-Busting Attack

캐시를 우회하여 오리진 서버에 직접 부하를 주는 공격입니다.

# 무작위 쿼리 파라미터로 캐시 우회 GET /page.html?v=12345 GET /page.html?v=67890 GET /page.html?v=11111 # 모두 같은 페이지지만 캐시 미스 # 방어: 쿼리 파라미터 무시 (Cloudflare) Cache Key: - URL Path: /page.html (포함) - Query String: 무시 # 또는 정규화 /page.html?v=* → /page.html (캐시 히트)

5. 신종 DDoS 공격

5.1 HTTP/2 Rapid Reset

2023년 발견된 최신 공격으로, Google 3.98억 rps 공격에 사용되었습니다.

# HTTP/2 Rapid Reset 공격 # HTTP/2의 스트림 취소(RST_STREAM) 악용 1. 클라이언트: 스트림 1 생성 (요청 전송) 2. 서버: 요청 처리 시작 3. 클라이언트: RST_STREAM (즉시 취소) 4. 반복 (초당 수백만 회) 서버 부하: - 요청 파싱 CPU 소모 - 취소 처리 CPU 소모 - 로그 기록 디스크 I/O # 방어: HTTP/2 스트림 제한 # Nginx http2_max_concurrent_streams 100; # Apache H2MaxSessionStreams 100

5.2 Carpet Bombing

특정 IP 대신 전체 서브넷을 공격하여 탐지를 회피합니다.

공격 방식설명방어 난이도
전통적 DDoS단일 IP (1.2.3.4) 집중 공격쉬움 (IP 차단)
Carpet Bombing서브넷 전체 (1.2.3.0/24 = 256 IP)어려움 (차단 시 정상 서비스도 차단)

🎯 핵심 요점

  • 3가지 주요 분류: Volumetric (대역폭), Protocol (프로토콜), Application (L7)
  • Volumetric: UDP/ICMP Flood, DNS/NTP Amplification (1:50~1:206 증폭)
  • Protocol: SYN Flood (가장 흔함), ACK Flood, Fragmentation
  • Application: HTTP Flood, Slowloris, Slow Read/POST, Cache-Busting
  • 증폭 공격: 작은 요청 → 큰 응답, 출발지 IP 위조
  • SYN Flood 방어: SYN Cookie (상태 저장 없이 연결)
  • Slowloris: 느린 요청으로 연결 고갈, 타임아웃 설정으로 방어
  • HTTP/2 Rapid Reset: 2023년 신종, 3.98억 rps, 스트림 제한으로 방어
  • 탐지 난이도: Volumetric < Protocol < Application Layer
