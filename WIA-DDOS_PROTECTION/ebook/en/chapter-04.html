Phase 1: Network-Level Protection

1. Traffic Filtering Techniques

Network-level protection forms the first line of defense against DDoS attacks, operating at OSI layers 3-4 (Network and Transport). These techniques filter malicious traffic before it reaches application servers, preserving bandwidth and server resources.

1.1 Access Control Lists (ACLs)

ACL Type Use Case Performance Impact Management Complexity
Standard ACL Block source IPs (known attackers) Low (<1% overhead) Low (simple rules)
Extended ACL Filter by IP, port, protocol Medium (2-5% overhead) Medium (moderate complexity)
Dynamic ACL Time-based, conditional filtering Medium (3-7% overhead) High (requires automation)
Reflexive ACL Stateful filtering (track sessions) High (5-10% overhead) Medium (auto-generated rules)

1.2 Blackhole and Sinkhole Routing

Technique Mechanism Pros Cons
Blackhole Routing Route attack traffic to null0 interface Instant mitigation, zero CPU cost Blocks all traffic (legitimate + attack)
Remotely Triggered Blackhole (RTBH) BGP community triggers ISP-level blackholing Upstream bandwidth preservation Entire /24 or /32 blackholed
Sinkhole Routing Redirect to analysis server for inspection Forensic data collection, traffic analysis Requires dedicated infrastructure

⚠️ Blackhole Routing: Emergency Use Only

Blackholing is a nuclear option that sacrifices availability to protect infrastructure. Use when:

  • Attack exceeds all mitigation capacity (>5 Tbps for most enterprises)
  • Collateral damage to shared infrastructure is imminent (data center meltdown)
  • Other mitigations ineffective and service degradation is 100% anyway
  • Coordinating with ISP/upstream providers to blackhole at their edge (saves your bandwidth)

Recovery: Remove blackhole routes immediately when attack subsides (typically 15-60 minutes).

2. Rate Limiting and Traffic Shaping

2.1 Rate Limiting Algorithms

Algorithm Description Use Case Burst Handling
Token Bucket Tokens added at fixed rate, consumed per packet General-purpose rate limiting Excellent (bucket size = burst capacity)
Leaky Bucket Queue packets, serve at constant rate Traffic smoothing, QoS Poor (enforces strict rate)
Fixed Window Count requests per time window (60s) Simple API rate limiting Poor (boundary gaming possible)
Sliding Window Rolling time window prevents boundary abuse Accurate API limiting Good (smoothed boundaries)
Adaptive Rate Limiting Dynamically adjust limits based on load Auto-scaling environments Excellent (AI-driven)
# Token Bucket Implementation (Python) import time from threading import Lock class TokenBucket: def __init__(self, capacity, fill_rate): self.capacity = capacity # Maximum tokens self.tokens = capacity # Current tokens self.fill_rate = fill_rate # Tokens added per second self.timestamp = time.time() self.lock = Lock() def consume(self, tokens=1): with self.lock: now = time.time() elapsed = now - self.timestamp self.tokens = min(self.capacity, self.tokens + elapsed * self.fill_rate) self.timestamp = now if self.tokens >= tokens: self.tokens -= tokens return True # Allow request return False # Rate limit exceeded # Usage rate_limiter = TokenBucket(capacity=1000, fill_rate=100) # 100 req/s for request in incoming_requests: if rate_limiter.consume(): process_request(request) else: reject_request(request, status=429)

3. BGP Flowspec

BGP Flowspec (RFC 5575) allows dynamic traffic filtering rules to be distributed via BGP, enabling rapid network-wide DDoS mitigation without manual router configuration.

3.1 Flowspec Capabilities

Match Criteria Actions Available Deployment Speed Scale
Destination IP prefix Drop, rate limit, redirect 10-60 seconds network-wide 100K+ rules
Source IP prefix Mark DSCP, police traffic
IP protocol (TCP/UDP/ICMP) Traffic shaping, sampling
Port (source/destination) Redirect to scrubbing center
TCP flags (SYN, ACK, RST) Community-based actions
Packet length Log for forensics
DSCP marking Divert to null interface

✅ BGP Flowspec Advantages

  • Rapid Deployment: Single command propagates rules to entire network in seconds
  • Centralized Management: Control DDoS mitigation from central controller or automation platform
  • ISP Cooperation: Share Flowspec rules with upstream providers for source-side filtering
  • Granular Filtering: Match specific attack patterns (e.g., "TCP SYN to port 80 from AS12345")
  • No Performance Penalty: Implemented in hardware (TCAM) on modern routers

4. Scrubbing Centers

DDoS scrubbing centers are specialized data centers that filter attack traffic, returning only clean traffic to the origin. Essential for Tbps-scale attacks exceeding on-premise capacity.

4.1 Scrubbing Center Architectures

Architecture Traffic Flow Latency Impact Cost Model
Always-On (Inline) All traffic permanently routes through scrubbing +5-15ms constant $10K-$100K/month fixed
On-Demand (BGP Divert) BGP announcement diverts traffic during attacks +20-50ms during attack $1K base + $5-$50/GB attack traffic
Hybrid Critical services always-on, others on-demand Variable (5-50ms) $5K-$30K/month + usage

🔧 Scrubbing Process Flow

Step 1: Traffic Diversion (2-5 minutes)

  • BGP announcement redirects traffic to scrubbing center
  • DNS updates point to scrubbing center IPs (for DNS-based)
  • Anycast routing automatically redirects (if pre-configured)

Step 2: Attack Traffic Filtering (continuous)

  • Multi-layer inspection: IP reputation, protocol validation, behavioral analysis
  • Challenge-response tests (SYN cookies, JavaScript challenges)
  • Rate limiting per source IP, ASN, geographic region
  • Signature-based blocking of known attack patterns

Step 3: Clean Traffic Return

  • Legitimate traffic forwarded via GRE tunnel to origin
  • Bandwidth typically reduced 80-99% (attack traffic filtered)
  • Connection state maintained for transparent user experience

Step 4: Traffic Restoration (1-2 minutes)

  • After attack subsides, BGP routing reverted to normal path
  • Gradual transition to prevent flash crowd from legitimate users

5. Anycast Routing

Anycast routing distributes traffic across multiple geographically dispersed data centers, each announcing the same IP prefix. Users automatically connect to the nearest server, distributing DDoS attack traffic across multiple locations.

5.1 Anycast DDoS Mitigation Benefits

Benefit Mechanism Attack Absorption
Geographic Distribution 10+ PoPs absorb attack at edge 100 Gbps attack → 10 Gbps per PoP
Automated Failover Overwhelmed PoP withdrawn from BGP Traffic automatically reroutes to healthy PoPs
Reduced Latency Users connect to nearest PoP Legitimate traffic unaffected during attack
Redundancy No single point of failure Total capacity = sum of all PoP bandwidth
# Anycast Configuration Example (BIRD BGP) # Define Anycast IP define ANYCAST_IP = 203.0.113.1/32; protocol static { route ANYCAST_IP via "lo"; # Announce from loopback } protocol bgp upstream_isp { local as 64500; neighbor 198.51.100.1 as 64501; export where proto = "static"; # Export anycast route import none; # Don't import routes # Advertise with specific communities export filter { if net = ANYCAST_IP then { bgp_community.add((64501, 1000)); # DDoS protection tag accept; } reject; } } # Health check: withdraw announcement if local service fails protocol bgp { # If health check fails, stop announcing ANYCAST_IP # Traffic automatically fails over to other PoPs }

🎓 Key Takeaways

  • Network-level protection operates at OSI L3-L4 using ACLs (4 types), blackhole routing, and traffic filtering with 1-10% performance overhead depending on complexity
  • Rate limiting algorithms (Token Bucket, Leaky Bucket, Sliding Window) control traffic flow; Token Bucket recommended for DDoS (excellent burst handling, capacity-based)
  • BGP Flowspec enables 10-60 second network-wide rule deployment, supporting 100K+ granular filtering rules without performance penalty via hardware TCAM implementation
  • Scrubbing centers provide Tbps-scale protection with three architectures: Always-On ($10K-$100K/month, +5-15ms latency), On-Demand ($1K base + usage, +20-50ms), Hybrid ($5K-$30K/month)
  • Anycast routing distributes attacks geographically (100 Gbps attack → 10 Gbps per PoP across 10 locations) with automated failover when individual PoPs overwhelmed
