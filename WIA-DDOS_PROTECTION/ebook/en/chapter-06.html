Phase 3: Monitoring and Detection

1. Traffic Analysis and Baselining

Effective DDoS detection begins with understanding normal traffic patterns. Baselining establishes the "normal" operational state against which anomalies are detected. This requires continuous monitoring of key performance indicators across network, application, and user behavior dimensions.

1.1 Key Metrics for Baseline

Metric Category Specific Metrics Baseline Method Alert Threshold
Network Volume Bandwidth (Gbps), Packets/sec 7-day moving average >3σ (99.7% confidence)
Connection Metrics New connections/sec, Concurrent connections Hourly averages (24h cycle) >2σ (95% confidence)
Protocol Distribution TCP/UDP/ICMP ratio, HTTP methods Weekly pattern recognition >20% deviation
Geographic Distribution Requests by country/ASN 30-day historical average >50% increase from single source
Application Metrics Request rate, Response time, Error rate Rolling 6-hour window >2.5σ (98% confidence)
User Behavior Session duration, Page depth, Bounce rate User cohort analysis >3σ for cohort
Resource Utilization CPU, Memory, Disk I/O, Network I/O 5-minute percentiles (P95, P99) P99 > historical P99 + 20%

📈 Statistical Anomaly Detection

Method 1: Standard Deviation (σ)

  • Calculate mean (μ) and standard deviation (σ) over baseline period (7-30 days)
  • Alert when current value > μ + 3σ (99.7% confidence) or < μ - 3σ
  • Works well for normally distributed metrics (e.g., request rate, latency)

Method 2: Percentile-Based

  • Track P50, P95, P99 percentiles over time
  • Alert when current P95 > historical P99 (indicates tail latency degradation)
  • Robust against outliers, works for skewed distributions

Method 3: Time-Series Forecasting

  • Use ARIMA, Prophet, or LSTM models to predict expected traffic
  • Alert when actual vs. predicted deviates >20%
  • Accounts for seasonality (daily, weekly patterns), holidays, events

✅ SIEM Best Practices for DDoS

  • Log Aggregation: Collect logs from firewalls, WAF, load balancers, CDN, servers (10K-1M events/sec)
  • Event Enrichment: Add geo-IP, threat intelligence, ASN data to raw logs for context
  • Correlation Rules: Define multi-condition rules (e.g., "high traffic + low session duration + specific geo = DDoS")
  • Dashboards: Real-time visualization of traffic rates, geo-distribution, protocol breakdown, error rates
  • Alerting: PagerDuty/Opsgenie integration for critical alerts (severity-based escalation)
  • Retention: Store 30 days of detailed logs, 1 year of aggregated metrics for forensics
  • Compliance: Meet SOC 2, PCI DSS, HIPAA log retention and monitoring requirements

# NetFlow Analysis for DDoS Detection (Python + nfdump) import subprocess import json # Analyze NetFlow data for top talkers (potential attack sources) def analyze_top_talkers(flow_file): cmd = f"nfdump -r {flow_file} -s srcip/bytes -n 20" result = subprocess.run(cmd, shell=True, capture_output=True, text=True) # Parse top source IPs by bytes top_talkers = [] for line in result.stdout.split('\n')[4:-3]: # Skip header/footer parts = line.split() if len(parts) >= 3: top_talkers.append({ 'src_ip': parts[0], 'bytes': int(parts[1]), 'percentage': float(parts[2].strip('%')) }) # Alert if single IP > 10% of total traffic for talker in top_talkers: if talker['percentage'] > 10: alert(f"DDoS suspected: {talker['src_ip']} = {talker['percentage']}% of traffic") return top_talkers # Detect SYN flood via flow analysis def detect_syn_flood(flow_file): cmd = f"nfdump -r {flow_file} 'flags S and not flags A' -s dstip/flows -n 10" result = subprocess.run(cmd, shell=True, capture_output=True, text=True) # Parse destinations receiving high SYN (no ACK) flows for line in result.stdout.split('\n')[4:-3]: parts = line.split() if len(parts) >= 2: dst_ip = parts[0] syn_flows = int(parts[1]) # Alert if > 100K SYN flows to single destination if syn_flows > 100000: alert(f"SYN flood detected: {dst_ip} receiving {syn_flows} SYN flows") # UDP flood detection def detect_udp_flood(flow_file): cmd = f"nfdump -r {flow_file} 'proto UDP' -s dstip/packets -n 10" result = subprocess.run(cmd, shell=True, capture_output=True, text=True) # Alert on high packet rate to single destination for line in result.stdout.split('\n')[4:-3]: parts = line.split() if len(parts) >= 2: dst_ip = parts[0] packets = int(parts[1]) # > 10M packets/min = likely DDoS if packets > 10_000_000: alert(f"UDP flood: {dst_ip} receiving {packets:,} packets/min")

⚠️ ML Model Challenges

Data Imbalance: DDoS attacks are rare events (0.01-1% of traffic). Use SMOTE, class weights, or focal loss to balance training.

Concept Drift: Attack patterns evolve. Retrain models monthly, use online learning for continuous adaptation.

Feature Engineering: Model quality depends on feature selection. Use domain knowledge to identify relevant metrics (protocol ratios, entropy, geo-distribution).

Adversarial Attacks: Attackers may poison training data or evade detection. Implement input validation, anomaly detection on features.

Explainability: Black-box models (neural networks) are hard to debug. Use SHAP values, LIME for interpretability.

🎓 Key Takeaways

  • Traffic baselining establishes normal operational state using 7-30 day historical data across 7 metric categories (network volume, connections, protocols, geography, application, user behavior, resources) with 2-3σ deviation thresholds triggering alerts
  • Anomaly detection approaches range from signature-based (instant detection, 1-3% FP) to deep learning (96-99% accuracy, 10-50ms latency) with trade-offs between speed, accuracy, and computational complexity
  • SIEM integration aggregates 10K-1M events/sec from firewalls, WAF, CDN, servers with event enrichment (geo-IP, threat intel, ASN), correlation rules, real-time dashboards, and severity-based alerting via PagerDuty/Opsgenie
  • NetFlow/sFlow provides flow-based monitoring with sampling rates 1:100 to 1:10000, enabling top talker analysis, SYN flood detection (>100K flows), and UDP flood identification (>10M packets/min) with minimal performance overhead
  • Machine learning models (Random Forest 92-96%, LSTM 96-99%, Autoencoder 93-96%) detect attacks with <1-50ms inference speed, requiring monthly retraining for concept drift and SMOTE/class weights to address data imbalance (0.01-1% attack traffic)
