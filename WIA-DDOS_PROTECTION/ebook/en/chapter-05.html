Phase 2: Application-Level Protection

1. Web Application Firewall (WAF)

Application-layer (L7) DDoS attacks target web applications, APIs, and databases with sophisticated request patterns that bypass network-level defenses. Web Application Firewalls provide deep packet inspection, behavioral analysis, and custom rule engines to identify and block malicious application traffic.

💡 Adaptive Rate Limiting

Dynamic adjustment of rate limits based on system load and user behavior:

  • Load-Based: Reduce limits by 50% when CPU >80%, database >90% capacity
  • Reputation-Based: Users with good history get 2x limits, suspicious users get 0.5x
  • Time-Based: Higher limits during off-peak hours (2AM-6AM), stricter during peak (12PM-2PM)
  • Geography-Based: Adjust limits per region based on attack patterns (e.g., reduce limits for ASN with recent abuse)
  • ML-Predicted: AI models predict and preemptively adjust limits before attacks materialize

✅ CDN DDoS Mitigation Workflow

Step 1: Traffic Routing

  • DNS points to CDN edge servers (not origin)
  • Anycast routing distributes traffic across global PoPs
  • Attack traffic absorbed at edge (never reaches origin)

Step 2: Edge Filtering

  • Layer 3/4 filtering: Rate limiting, protocol validation, geo-blocking
  • Layer 7 protection: WAF rules, bot detection, JavaScript challenges
  • Caching: 80-95% of requests served from cache (reduces origin load)

Step 3: Origin Shield

  • Single point of ingress to origin (hides origin IP)
  • Connection pooling: Reduces concurrent connections to origin
  • Request collapsing: Deduplicates identical requests

Step 4: Auto-Scaling

  • CDN capacity scales elastically (no manual intervention)
  • Traffic automatically fails over to other PoPs if one overwhelmed
  • Origin protected regardless of attack size (within CDN capacity)

5. SSL/TLS DDoS Protection

5.1 TLS Handshake Attack Mitigation

Attack Type Mechanism Mitigation Strategy Effectiveness
TLS Handshake Flood Incomplete handshakes exhaust server CPU SYN cookies, TLS session caching, hardware offload 85-95%
TLS Renegotiation Force expensive renegotiation (15x CPU cost) Disable renegotiation, rate limit attempts 95-99%
Certificate Validation Trigger OCSP/CRL lookups (network delay) OCSP stapling, cached responses 90-95%
THC-SSL-DOS Single machine exhausts server (legacy) Modern TLS stacks, connection limits 99%+

⚠️ HTTPS Flood: The Hidden Threat

HTTPS floods are 15-20x more expensive for servers than HTTP floods due to TLS decryption overhead. A 100,000 req/s HTTP flood becomes equivalent to 1.5-2 million req/s in terms of server CPU consumption.

Mitigation Approaches:

  • TLS Offloading: Dedicated hardware (F5, Citrix NetScaler) or CDN terminates TLS at edge
  • Session Resumption: Reuse TLS sessions (RFC 5077 tickets) to avoid full handshake (10x faster)
  • HTTP/2 Multiplexing: One TLS connection for multiple requests (reduces handshake overhead)
  • Certificate Optimization: Smaller certs (ECDSA 256-bit vs RSA 2048-bit), OCSP stapling

🎓 Key Takeaways

  • WAF provides 7 protection mechanisms (signature-based, behavioral, rate limiting, geo-blocking, bot detection, protocol validation, session security) with 1-10% false positive rates
  • Application-layer rate limiting varies by resource type: 30 req/min for anonymous users, 300 for authenticated, 10,000 for premium, with adaptive adjustment based on load and reputation
  • Challenge-response mechanisms range from reCAPTCHA v2 (90-95% bot block, medium friction) to Proof-of-Work (95-98%, no friction) with varying bypass costs ($0.50-$5 per 1000 solves)
  • CDN integration provides edge protection via global PoP distribution (80-450 PoPs), absorbing attacks before reaching origin with 80-95% cache hit rates reducing origin load
  • HTTPS floods consume 15-20x more server resources than HTTP; mitigation requires TLS offloading, session resumption, HTTP/2 multiplexing, and certificate optimization to neutralize performance impact
