7.1 ETL의 개념과 중요성

ETL(Extract, Transform, Load)은 데이터 웨어하우스의 핵심 프로세스로, 다양한 소스 시스템에서 데이터를 추출(Extract)하고, 비즈니스 규칙에 따라 변환(Transform)한 후, 대상 데이터 웨어하우스에 적재(Load)하는 일련의 과정을 의미합니다.

ETL이 중요한 이유

  • 데이터 통합: 여러 이질적인 소스 시스템의 데이터를 하나로 통합
  • 데이터 품질: 정제, 검증, 표준화를 통해 고품질 데이터 보장
  • 비즈니스 규칙 적용: 계산, 집계, 파생 속성 생성
  • 역사적 데이터 보존: SCD 및 이력 관리
  • 성능 최적화: 분석에 최적화된 형태로 데이터 구조화
  • 데이터 거버넌스: 데이터 계보, 감사 추적, 규제 준수

ETL vs ELT 비교

ETL과 ELT 접근법 비교
특성 ETL (Extract-Transform-Load) ELT (Extract-Load-Transform)
변환 위치 ETL 서버 (중간 계층) 데이터 웨어하우스 내부
처리 방식 행 단위 처리 (Row-by-row) 집합 기반 처리 (Set-based)
성능 ETL 서버 성능에 의존 DW의 MPP 성능 활용
확장성 ETL 서버 확장 필요 DW 확장으로 자동 개선
비용 ETL 도구 라이선스 비용 DW 컴퓨팅 비용
복잡한 변환 적합 SQL로 표현 어려울 수 있음
데이터 크기 중소 규모 대규모 (빅데이터)
적합한 플랫폼 전통적 RDBMS 클라우드 DW (Snowflake, BigQuery)
주요 도구 Informatica, Talend, SSIS dbt, Dataform, SQL
원본 데이터 보존 스테이징에만 영구 보존 가능

7.2 추출 (Extract) 전략

추출 단계는 소스 시스템에서 데이터를 가져오는 과정입니다. 운영 시스템의 성능에 최소한의 영향을 주면서 필요한 데이터를 효율적으로 추출하는 것이 핵심입니다.

추출 패턴

1. 전체 추출 (Full Extraction)

-- 전체 추출: 테이블의 모든 데이터를 가져옴
-- 장점: 단순, 데이터 일관성 보장
-- 단점: 느림, 소스 시스템 부하, 네트워크 대역폭

-- PostgreSQL 예시
CREATE TABLE staging.customer_full AS
SELECT *
FROM source_system.customers
WHERE 1=1;

-- 감사 컬럼 추가
ALTER TABLE staging.customer_full
ADD COLUMN etl_loaded_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
ADD COLUMN etl_batch_id BIGINT DEFAULT 0;

-- 적용 시나리오:
-- • 소규모 테이블 (< 100만 행)
-- • 초기 로드
-- • 변경 감지가 불가능한 소스
-- • 주기적 재구축 필요 시

2. 증분 추출 (Incremental Extraction)

-- 증분 추출: 마지막 추출 이후 변경된 데이터만 가져옴
-- 장점: 빠름, 소스 부하 최소화
-- 단점: 변경 추적 메커니즘 필요

-- 방법 1: 타임스탬프 기반
CREATE TABLE staging.customer_incremental AS
SELECT *
FROM source_system.customers
WHERE updated_timestamp > (
    SELECT MAX(etl_loaded_at)
    FROM staging.customer_incremental
)
OR updated_timestamp IS NULL;

-- 방법 2: 시퀀스 번호 기반
CREATE TABLE staging.order_incremental AS
SELECT *
FROM source_system.orders
WHERE order_id > (
    SELECT COALESCE(MAX(source_order_id), 0)
    FROM dwh.dim_order
);

-- 방법 3: 플래그 기반
CREATE TABLE staging.product_incremental AS
SELECT *
FROM source_system.products
WHERE is_exported = FALSE;

-- 소스 시스템에 플래그 업데이트
UPDATE source_system.products
SET is_exported = TRUE,
    exported_timestamp = CURRENT_TIMESTAMP
WHERE is_exported = FALSE;

-- 방법 4: 델타 테이블 사용
CREATE TABLE staging.customer_changes AS
SELECT *
FROM source_system.customer_changes_log
WHERE change_timestamp > :last_extraction_time;

3. CDC (Change Data Capture)

-- CDC: 데이터베이스 로그 기반 변경 추적
-- 장점: 실시간, 소스 부하 최소, 삭제 감지 가능
-- 단점: 설정 복잡, 도구 필요 (Debezium, Oracle GoldenGate)

-- PostgreSQL 논리적 복제 슬롯 사용
-- 1. 복제 슬롯 생성
SELECT * FROM pg_create_logical_replication_slot(
    'etl_slot',
    'pgoutput'
);

-- 2. 변경 사항 읽기
SELECT * FROM pg_logical_slot_get_changes(
    'etl_slot',
    NULL,
    NULL
);

-- Debezium (Kafka Connect) 설정 예시
{
  "name": "customer-cdc-connector",
  "config": {
    "connector.class": "io.debezium.connector.postgresql.PostgresConnector",
    "database.hostname": "source-db.example.com",
    "database.port": "5432",
    "database.user": "debezium",
    "database.password": "secret",
    "database.dbname": "production",
    "database.server.name": "source_system",
    "table.include.list": "public.customers,public.orders",
    "plugin.name": "pgoutput"
  }
}

-- 스테이징 테이블: CDC 이벤트 저장
CREATE TABLE staging.cdc_events (
    event_id BIGSERIAL PRIMARY KEY,
    source_table VARCHAR(100),
    operation VARCHAR(10),  -- INSERT, UPDATE, DELETE
    before_data JSONB,
    after_data JSONB,
    event_timestamp TIMESTAMP,
    captured_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- CDC 이벤트 처리
INSERT INTO staging.customer_updates
SELECT
    (after_data->>'customer_id')::INT as customer_id,
    after_data->>'customer_name' as customer_name,
    after_data->>'email' as email,
    operation
FROM staging.cdc_events
WHERE source_table = 'customers'
  AND captured_at > :last_processed_time;

추출 모범 사례

추출 전략별 권장 사용 시나리오
추출 방법 데이터 크기 변경 빈도 지연 시간 요구 권장 도구
전체 추출 소규모 (< 1M 행) 낮음 (일 1회) 시간/일 단위 SQL, JDBC, Python
타임스탬프 증분 중규모 (1M-100M) 중간 (시간당) 시간 단위 SQL, Airflow
시퀀스 증분 중대규모 높음 분 단위 SQL, Kafka
CDC 대규모 (> 100M) 매우 높음 실시간 (초 단위) Debezium, GoldenGate
델타 테이블 모든 크기 중간 시간 단위 SQL, dbt

추출 오류 처리

-- 재시도 로직이 포함된 추출 프로시저
CREATE OR REPLACE PROCEDURE extract_customers_with_retry(
    p_max_retries INT DEFAULT 3,
    p_retry_delay_seconds INT DEFAULT 60
)
LANGUAGE plpgsql
AS $$
DECLARE
    v_retry_count INT := 0;
    v_error_message TEXT;
    v_extraction_id BIGINT;
BEGIN
    -- 추출 작업 ID 생성
    INSERT INTO etl_metadata.extraction_log (
        source_table, status, start_time
    ) VALUES (
        'customers', 'RUNNING', CURRENT_TIMESTAMP
    ) RETURNING extraction_id INTO v_extraction_id;

    -- 재시도 루프
    WHILE v_retry_count < p_max_retries LOOP
        BEGIN
            -- 추출 실행
            TRUNCATE TABLE staging.customers;

            INSERT INTO staging.customers
            SELECT * FROM dblink(
                'source_connection',
                'SELECT * FROM customers WHERE updated_at > ' ||
                quote_literal(CURRENT_DATE - INTERVAL '1 day')
            ) AS t(
                customer_id INT,
                customer_name VARCHAR(200),
                email VARCHAR(100),
                updated_at TIMESTAMP
            );

            -- 성공 로그
            UPDATE etl_metadata.extraction_log
            SET status = 'SUCCESS',
                end_time = CURRENT_TIMESTAMP,
                rows_extracted = (SELECT COUNT(*) FROM staging.customers)
            WHERE extraction_id = v_extraction_id;

            RETURN;  -- 성공 시 종료

        EXCEPTION WHEN OTHERS THEN
            v_error_message := SQLERRM;
            v_retry_count := v_retry_count + 1;

            -- 오류 로그
            INSERT INTO etl_metadata.extraction_errors (
                extraction_id, retry_attempt, error_message, error_time
            ) VALUES (
                v_extraction_id, v_retry_count,
                v_error_message, CURRENT_TIMESTAMP
            );

            IF v_retry_count < p_max_retries THEN
                -- 대기 후 재시도
                PERFORM pg_sleep(p_retry_delay_seconds);
            ELSE
                -- 최대 재시도 초과
                UPDATE etl_metadata.extraction_log
                SET status = 'FAILED',
                    end_time = CURRENT_TIMESTAMP,
                    error_message = v_error_message
                WHERE extraction_id = v_extraction_id;

                RAISE EXCEPTION 'Extraction failed after % retries: %',
                    p_max_retries, v_error_message;
            END IF;
        END;
    END LOOP;
END;
$$;

7.3 변환 (Transform) 기법

변환 단계는 ETL의 핵심으로, 원본 데이터를 비즈니스 요구사항에 맞게 정제, 표준화, 강화하는 과정입니다.

1. 데이터 정제 (Data Cleansing)

-- 공백 제거 및 트림
UPDATE staging.customers
SET
    customer_name = TRIM(customer_name),
    email = LOWER(TRIM(email)),
    phone = REGEXP_REPLACE(phone, '[^0-9]', '', 'g');  -- 숫자만 남김

-- NULL 처리
UPDATE staging.customers
SET
    email = COALESCE(email, 'unknown@example.com'),
    city = COALESCE(city, 'Unknown'),
    country = COALESCE(country, 'Unknown');

-- 중복 제거
WITH duplicates AS (
    SELECT customer_id, email,
           ROW_NUMBER() OVER (
               PARTITION BY email
               ORDER BY updated_at DESC
           ) as rn
    FROM staging.customers
)
DELETE FROM staging.customers
WHERE customer_id IN (
    SELECT customer_id FROM duplicates WHERE rn > 1
);

-- 데이터 타입 변환
UPDATE staging.orders
SET
    order_amount = CASE
        WHEN order_amount ~ '^[0-9]+\.?[0-9]*$'
        THEN order_amount::DECIMAL(12,2)
        ELSE 0.00
    END;

-- 이상값 처리
UPDATE staging.products
SET list_price = (
    SELECT PERCENTILE_CONT(0.99) WITHIN GROUP (ORDER BY list_price)
    FROM staging.products
)
WHERE list_price > (
    SELECT PERCENTILE_CONT(0.99) WITHIN GROUP (ORDER BY list_price)
    FROM staging.products
) * 2;  -- 99 백분위수의 2배 이상은 캡핑

2. 데이터 검증 (Data Validation)

-- 데이터 품질 검증 프레임워크
CREATE TABLE etl_metadata.data_quality_rules (
    rule_id SERIAL PRIMARY KEY,
    table_name VARCHAR(100),
    rule_name VARCHAR(200),
    rule_type VARCHAR(50),  -- NOT_NULL, RANGE, FORMAT, UNIQUE, REFERENTIAL
    column_name VARCHAR(100),
    rule_sql TEXT,
    severity VARCHAR(20),   -- ERROR, WARNING, INFO
    is_active BOOLEAN DEFAULT TRUE
);

-- 검증 규칙 등록
INSERT INTO etl_metadata.data_quality_rules
    (table_name, rule_name, rule_type, column_name, rule_sql, severity)
VALUES
    ('staging.customers', 'Email Format', 'FORMAT', 'email',
     'email ~ ''^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z|a-z]{2,}$''',
     'ERROR'),

    ('staging.customers', 'Age Range', 'RANGE', 'age',
     'age BETWEEN 0 AND 120', 'ERROR'),

    ('staging.orders', 'Amount Positive', 'RANGE', 'order_amount',
     'order_amount > 0', 'ERROR'),

    ('staging.orders', 'Valid Customer', 'REFERENTIAL', 'customer_id',
     'EXISTS (SELECT 1 FROM staging.customers c WHERE c.customer_id = o.customer_id)',
     'ERROR');

-- 검증 실행 프로시저
CREATE OR REPLACE PROCEDURE validate_data_quality(
    p_table_name VARCHAR,
    p_fail_on_error BOOLEAN DEFAULT TRUE
)
LANGUAGE plpgsql
AS $$
DECLARE
    v_rule RECORD;
    v_violation_count INT;
    v_total_violations INT := 0;
    v_validation_id BIGINT;
BEGIN
    -- 검증 세션 시작
    INSERT INTO etl_metadata.validation_log (
        table_name, validation_start, status
    ) VALUES (
        p_table_name, CURRENT_TIMESTAMP, 'RUNNING'
    ) RETURNING validation_id INTO v_validation_id;

    -- 모든 활성 규칙 검증
    FOR v_rule IN
        SELECT * FROM etl_metadata.data_quality_rules
        WHERE table_name = p_table_name AND is_active = TRUE
    LOOP
        -- 위반 건수 확인
        EXECUTE format(
            'SELECT COUNT(*) FROM %I WHERE NOT (%s)',
            p_table_name, v_rule.rule_sql
        ) INTO v_violation_count;

        -- 위반 로그
        IF v_violation_count > 0 THEN
            INSERT INTO etl_metadata.validation_violations (
                validation_id, rule_id, violation_count, severity
            ) VALUES (
                v_validation_id, v_rule.rule_id,
                v_violation_count, v_rule.severity
            );

            IF v_rule.severity = 'ERROR' THEN
                v_total_violations := v_total_violations + v_violation_count;
            END IF;
        END IF;
    END LOOP;

    -- 검증 결과 업데이트
    UPDATE etl_metadata.validation_log
    SET validation_end = CURRENT_TIMESTAMP,
        status = CASE
            WHEN v_total_violations > 0 THEN 'FAILED'
            ELSE 'PASSED'
        END,
        total_violations = v_total_violations
    WHERE validation_id = v_validation_id;

    -- 오류 시 예외 발생
    IF p_fail_on_error AND v_total_violations > 0 THEN
        RAISE EXCEPTION 'Data quality validation failed: % violations',
            v_total_violations;
    END IF;
END;
$$;

-- 사용 예시
CALL validate_data_quality('staging.customers', TRUE);

3. 데이터 표준화 (Data Standardization)

-- 주소 표준화
CREATE OR REPLACE FUNCTION standardize_address(p_address TEXT)
RETURNS TEXT AS $$
BEGIN
    RETURN UPPER(TRIM(
        REGEXP_REPLACE(
            REGEXP_REPLACE(
                REGEXP_REPLACE(p_address, '\s+', ' ', 'g'),  -- 다중 공백 제거
                'Street', 'St', 'gi'),
            'Avenue', 'Ave', 'gi')
    ));
END;
$$ LANGUAGE plpgsql IMMUTABLE;

-- 전화번호 표준화
CREATE OR REPLACE FUNCTION standardize_phone(p_phone TEXT)
RETURNS TEXT AS $$
DECLARE
    v_digits TEXT;
BEGIN
    -- 숫자만 추출
    v_digits := REGEXP_REPLACE(p_phone, '[^0-9]', '', 'g');

    -- 포맷 적용
    RETURN CASE
        WHEN LENGTH(v_digits) = 10 THEN
            FORMAT('(%s) %s-%s',
                SUBSTRING(v_digits, 1, 3),
                SUBSTRING(v_digits, 4, 3),
                SUBSTRING(v_digits, 7, 4))
        WHEN LENGTH(v_digits) = 11 AND LEFT(v_digits, 1) = '1' THEN
            FORMAT('+1 (%s) %s-%s',
                SUBSTRING(v_digits, 2, 3),
                SUBSTRING(v_digits, 5, 3),
                SUBSTRING(v_digits, 8, 4))
        ELSE p_phone  -- 표준화 불가능한 경우 원본 반환
    END;
END;
$$ LANGUAGE plpgsql IMMUTABLE;

-- 날짜 표준화
CREATE OR REPLACE FUNCTION standardize_date(p_date_string TEXT)
RETURNS DATE AS $$
BEGIN
    RETURN CASE
        -- MM/DD/YYYY
        WHEN p_date_string ~ '^\d{1,2}/\d{1,2}/\d{4}$' THEN
            TO_DATE(p_date_string, 'MM/DD/YYYY')
        -- YYYY-MM-DD
        WHEN p_date_string ~ '^\d{4}-\d{2}-\d{2}$' THEN
            TO_DATE(p_date_string, 'YYYY-MM-DD')
        -- DD-Mon-YYYY
        WHEN p_date_string ~ '^\d{2}-[A-Za-z]{3}-\d{4}$' THEN
            TO_DATE(p_date_string, 'DD-Mon-YYYY')
        ELSE NULL
    END;
EXCEPTION WHEN OTHERS THEN
    RETURN NULL;
END;
$$ LANGUAGE plpgsql IMMUTABLE;

-- 표준화 적용
UPDATE staging.customers
SET
    address = standardize_address(address),
    phone = standardize_phone(phone),
    birth_date = standardize_date(birth_date_string);

4. 데이터 강화 (Data Enrichment)

-- 파생 속성 계산
UPDATE staging.customers
SET
    -- 연령 계산
    age = EXTRACT(YEAR FROM AGE(CURRENT_DATE, birth_date)),

    -- 고객 생애 가치 (LTV) 계산
    lifetime_value = (
        SELECT COALESCE(SUM(order_amount), 0)
        FROM staging.orders o
        WHERE o.customer_id = staging.customers.customer_id
    ),

    -- 마지막 구매 일자
    last_purchase_date = (
        SELECT MAX(order_date)
        FROM staging.orders o
        WHERE o.customer_id = staging.customers.customer_id
    ),

    -- 구매 빈도
    purchase_frequency = (
        SELECT COUNT(DISTINCT order_id)
        FROM staging.orders o
        WHERE o.customer_id = staging.customers.customer_id
          AND order_date >= CURRENT_DATE - INTERVAL '1 year'
    );

-- 세그먼트 할당
UPDATE staging.customers
SET customer_segment = CASE
    WHEN lifetime_value >= 10000 AND purchase_frequency >= 12 THEN 'Platinum'
    WHEN lifetime_value >= 5000 OR purchase_frequency >= 6 THEN 'Gold'
    WHEN lifetime_value >= 1000 OR purchase_frequency >= 2 THEN 'Silver'
    ELSE 'Bronze'
END;

-- RFM (Recency, Frequency, Monetary) 스코어링
WITH rfm_calc AS (
    SELECT
        customer_id,
        CURRENT_DATE - MAX(order_date) as recency_days,
        COUNT(DISTINCT order_id) as frequency,
        SUM(order_amount) as monetary
    FROM staging.orders
    WHERE order_date >= CURRENT_DATE - INTERVAL '1 year'
    GROUP BY customer_id
),
rfm_scores AS (
    SELECT
        customer_id,
        NTILE(5) OVER (ORDER BY recency_days ASC) as recency_score,
        NTILE(5) OVER (ORDER BY frequency DESC) as frequency_score,
        NTILE(5) OVER (ORDER BY monetary DESC) as monetary_score
    FROM rfm_calc
)
UPDATE staging.customers c
SET
    rfm_score = CONCAT(r.recency_score, r.frequency_score, r.monetary_score),
    rfm_segment = CASE
        WHEN r.recency_score >= 4 AND r.frequency_score >= 4 THEN 'Champions'
        WHEN r.recency_score >= 3 AND r.frequency_score >= 3 THEN 'Loyal Customers'
        WHEN r.recency_score >= 4 AND r.frequency_score <= 2 THEN 'Promising'
        WHEN r.recency_score <= 2 AND r.frequency_score >= 4 THEN 'At Risk'
        WHEN r.recency_score <= 2 AND r.frequency_score <= 2 THEN 'Lost'
        ELSE 'Others'
    END
FROM rfm_scores r
WHERE c.customer_id = r.customer_id;

5. 조인 및 통합

-- 여러 소스의 고객 데이터 통합
CREATE TABLE staging.customer_unified AS
SELECT
    COALESCE(web.email, crm.email, erp.customer_email) as email,
    COALESCE(crm.customer_name, web.full_name, erp.customer_name) as customer_name,

    -- 가장 신뢰할 수 있는 소스 선택
    COALESCE(crm.phone, web.phone) as phone,
    COALESCE(crm.address, erp.billing_address) as address,

    -- 소스 우선순위
    CASE
        WHEN crm.customer_id IS NOT NULL THEN 'CRM'
        WHEN web.user_id IS NOT NULL THEN 'WEB'
        WHEN erp.customer_code IS NOT NULL THEN 'ERP'
    END as primary_source,

    -- 모든 소스 ID 보존
    crm.customer_id as crm_id,
    web.user_id as web_id,
    erp.customer_code as erp_id,

    -- 마스터 데이터 품질 스코어
    (CASE WHEN crm.customer_name IS NOT NULL THEN 1 ELSE 0 END +
     CASE WHEN crm.email IS NOT NULL THEN 1 ELSE 0 END +
     CASE WHEN crm.phone IS NOT NULL THEN 1 ELSE 0 END +
     CASE WHEN crm.address IS NOT NULL THEN 1 ELSE 0 END
    ) as data_quality_score

FROM staging.crm_customers crm
FULL OUTER JOIN staging.web_users web
    ON LOWER(crm.email) = LOWER(web.email)
FULL OUTER JOIN staging.erp_customers erp
    ON LOWER(crm.email) = LOWER(erp.customer_email)
    OR crm.customer_id::TEXT = erp.customer_code;

-- 충돌 해결 (Conflict Resolution)
WITH duplicates AS (
    SELECT email,
           ROW_NUMBER() OVER (
               PARTITION BY LOWER(email)
               ORDER BY data_quality_score DESC,
                        CASE primary_source
                            WHEN 'CRM' THEN 1
                            WHEN 'ERP' THEN 2
                            WHEN 'WEB' THEN 3
                        END
           ) as rn
    FROM staging.customer_unified
)
DELETE FROM staging.customer_unified
WHERE email IN (
    SELECT email FROM duplicates WHERE rn > 1
);

-- Type 2 SCD 차원 로드 (MERGE 사용)
MERGE INTO dwh.dim_customer AS target
USING (
    SELECT
        customer_id,
        customer_name,
        email,
        customer_segment,
        MD5(CONCAT(
            COALESCE(customer_name, ''),
            COALESCE(customer_segment, '')
        )) as row_hash
    FROM staging.customer_unified
) AS source
ON target.customer_id = source.customer_id
   AND target.is_current = TRUE

-- Type 1 업데이트 (이름만)
WHEN MATCHED AND target.customer_name <> source.customer_name THEN
    UPDATE SET
        customer_name = source.customer_name,
        updated_timestamp = CURRENT_TIMESTAMP

-- Type 2 업데이트 (세그먼트 변경)
WHEN MATCHED AND target.row_hash <> source.row_hash THEN
    UPDATE SET
        is_current = FALSE,
        expiration_date = CURRENT_DATE - INTERVAL '1 day',
        updated_timestamp = CURRENT_TIMESTAMP

-- 신규 레코드 INSERT
WHEN NOT MATCHED THEN
    INSERT (
        customer_key, customer_id, customer_name,
        email, customer_segment, effective_date,
        is_current, row_hash
    ) VALUES (
        NEXTVAL('customer_key_seq'),
        source.customer_id,
        source.customer_name,
        source.email,
        source.customer_segment,
        CURRENT_DATE,
        TRUE,
        source.row_hash
    );

-- Type 2 새 버전 INSERT (세그먼트 변경된 경우)
INSERT INTO dwh.dim_customer (

    customer_segment, effective_date, is_current, row_hash

SELECT

    customer_key,
    product_key,
    store_key,
    quantity,
    unit_price,
    gross_amount,
    discount_amount,
    net_amount

SELECT
    -- 날짜 차원 조회
    dd.date_key,

    -- 고객 차원 조회 (시점 정확)
    dc.customer_key,

    -- 제품 차원 조회
    dp.product_key,

    -- 매장 차원 조회
    ds.store_key,

    -- 팩트 측정값
    so.quantity,
    so.unit_price,
    so.quantity * so.unit_price as gross_amount,
    COALESCE(so.discount_amount, 0) as discount_amount,
    so.quantity * so.unit_price - COALESCE(so.discount_amount, 0) as net_amount

FROM staging.sales_orders so

-- 필수 차원 조인
JOIN dwh.dim_date dd
    ON so.order_date = dd.full_date

JOIN dwh.dim_customer dc
    ON so.customer_id = dc.customer_id
    AND so.order_date BETWEEN dc.effective_date
        AND COALESCE(dc.expiration_date, '9999-12-31')

JOIN dwh.dim_product dp
    ON so.product_id = dp.product_id
    AND dp.is_current = TRUE

JOIN dwh.dim_store ds
    ON so.store_id = ds.store_id

-- 중복 방지
WHERE NOT EXISTS (
    SELECT 1 FROM dwh.fact_sales fs
    WHERE fs.date_key = dd.date_key
      AND fs.customer_key = dc.customer_key
      AND fs.product_key = dp.product_key
      AND fs.quantity = so.quantity

CREATE OR REPLACE PROCEDURE run_etl_job(
    p_job_name VARCHAR,
    p_parameters JSONB DEFAULT '{}'::JSONB
)
LANGUAGE plpgsql
AS $$
DECLARE
    v_execution_id BIGINT;
    v_step_id BIGINT;
    v_rows_extracted BIGINT := 0;
    v_rows_loaded BIGINT := 0;
BEGIN
    -- 실행 시작
    INSERT INTO etl_metadata.etl_execution_log (
        job_name, start_time, status, execution_parameters
    ) VALUES (
        p_job_name, CURRENT_TIMESTAMP, 'RUNNING', p_parameters
    ) RETURNING execution_id INTO v_execution_id;

    BEGIN
        -- STEP 1: 추출
        INSERT INTO etl_metadata.etl_step_log (
            execution_id, step_name, step_type, start_time, status
        ) VALUES (
            v_execution_id, 'Extract Customers', 'EXTRACT',
            CURRENT_TIMESTAMP, 'RUNNING'
        ) RETURNING step_id INTO v_step_id;

        CALL extract_customers_incremental();

        SELECT COUNT(*) INTO v_rows_extracted FROM staging.customers;

        UPDATE etl_metadata.etl_step_log
        SET end_time = CURRENT_TIMESTAMP,
            status = 'SUCCESS',
            rows_processed = v_rows_extracted
        WHERE step_id = v_step_id;

        -- STEP 2: 변환
        INSERT INTO etl_metadata.etl_step_log (
            execution_id, step_name, step_type, start_time, status
        ) VALUES (
            v_execution_id, 'Transform Customers', 'TRANSFORM',
            CURRENT_TIMESTAMP, 'RUNNING'
        ) RETURNING step_id INTO v_step_id;

        CALL transform_customers();

        UPDATE etl_metadata.etl_step_log
        SET end_time = CURRENT_TIMESTAMP,
            status = 'SUCCESS',
            rows_processed = v_rows_extracted
        WHERE step_id = v_step_id;

        -- STEP 3: 검증
        INSERT INTO etl_metadata.etl_step_log (
            execution_id, step_name, step_type, start_time, status
        ) VALUES (
            v_execution_id, 'Validate Customers', 'VALIDATE',
            CURRENT_TIMESTAMP, 'RUNNING'
        ) RETURNING step_id INTO v_step_id;

        CALL validate_data_quality('staging.customers', FALSE);

        UPDATE etl_metadata.etl_step_log
        SET end_time = CURRENT_TIMESTAMP,
            status = 'SUCCESS'
        WHERE step_id = v_step_id;

        -- STEP 4: 로드
        INSERT INTO etl_metadata.etl_step_log (
            execution_id, step_name, step_type, start_time, status
        ) VALUES (
            v_execution_id, 'Load Dimension', 'LOAD',
            CURRENT_TIMESTAMP, 'RUNNING'
        ) RETURNING step_id INTO v_step_id;

        CALL load_dim_customer();

        SELECT COUNT(*) INTO v_rows_loaded FROM dwh.dim_customer
        WHERE updated_timestamp >= CURRENT_TIMESTAMP - INTERVAL '1 minute';

        UPDATE etl_metadata.etl_step_log
        SET end_time = CURRENT_TIMESTAMP,
            status = 'SUCCESS',
            rows_processed = v_rows_loaded
        WHERE step_id = v_step_id;

        -- 실행 완료
        UPDATE etl_metadata.etl_execution_log
        SET end_time = CURRENT_TIMESTAMP,
            status = 'SUCCESS',
            rows_extracted = v_rows_extracted,
            rows_loaded = v_rows_loaded
        WHERE execution_id = v_execution_id;

        RAISE NOTICE 'ETL job % completed successfully', p_job_name;

    EXCEPTION WHEN OTHERS THEN
        -- 오류 처리
        UPDATE etl_metadata.etl_execution_log
        SET end_time = CURRENT_TIMESTAMP,
            status = 'FAILED',
            error_message = SQLERRM
        WHERE execution_id = v_execution_id;

        UPDATE etl_metadata.etl_step_log
        SET end_time = CURRENT_TIMESTAMP,
            status = 'FAILED',
            error_message = SQLERRM
        WHERE step_id = v_step_id;

        RAISE EXCEPTION 'ETL job % failed: %', p_job_name, SQLERRM;
    END;
END;
$$;

-- 재시작 가능한 ETL (Restartable ETL)
CREATE OR REPLACE PROCEDURE run_etl_job_restartable(
    p_job_name VARCHAR,
    p_execution_id BIGINT DEFAULT NULL  -- NULL이면 새 실행, 아니면 재시작
)
LANGUAGE plpgsql
AS $$
DECLARE
    v_execution_id BIGINT;
    v_last_completed_step VARCHAR(200);
BEGIN
    IF p_execution_id IS NULL THEN
        -- 새 실행 시작
        INSERT INTO etl_metadata.etl_execution_log (
            job_name, start_time, status
        ) VALUES (
            p_job_name, CURRENT_TIMESTAMP, 'RUNNING'
        ) RETURNING execution_id INTO v_execution_id;
    ELSE
        -- 기존 실행 재시작
        v_execution_id := p_execution_id;

        -- 마지막 완료 단계 확인
        SELECT step_name INTO v_last_completed_step
        FROM etl_metadata.etl_step_log
        WHERE execution_id = v_execution_id
          AND status = 'SUCCESS'
        ORDER BY step_id DESC
        LIMIT 1;

        RAISE NOTICE 'Restarting from step: %',
            COALESCE(v_last_completed_step, 'beginning');
    END IF;

    -- 단계별 실행 (체크포인트 기반)
    IF v_last_completed_step IS NULL OR
       v_last_completed_step < 'Extract' THEN
        PERFORM log_step(v_execution_id, 'Extract', 'EXTRACT');
        CALL extract_customers_incremental();
    END IF;

    IF v_last_completed_step IS NULL OR
       v_last_completed_step < 'Transform' THEN
        PERFORM log_step(v_execution_id, 'Transform', 'TRANSFORM');
        CALL transform_customers();
    END IF;

    IF v_last_completed_step IS NULL OR
       v_last_completed_step < 'Load' THEN
        PERFORM log_step(v_execution_id, 'Load', 'LOAD');
        CALL load_dim_customer();
    END IF;

    -- 완료
    UPDATE etl_metadata.etl_execution_log
    SET end_time = CURRENT_TIMESTAMP, status = 'SUCCESS'
    WHERE execution_id = v_execution_id;
END;
$$;

-- 워터마크 초기화
INSERT INTO etl_metadata.watermarks VALUES

-- 증분 추출 함수
CREATE OR REPLACE FUNCTION extract_incremental(
    p_source_table VARCHAR,
    p_target_table VARCHAR
) RETURNS BIGINT AS $$
DECLARE
    v_watermark_column VARCHAR(100);
    v_watermark_value VARCHAR(200);
    v_watermark_type VARCHAR(20);
    v_new_watermark VARCHAR(200);
    v_rows_extracted BIGINT;
    v_sql TEXT;
BEGIN
    -- 워터마크 조회
    SELECT watermark_column, watermark_value, watermark_type
    INTO v_watermark_column, v_watermark_value, v_watermark_type
    FROM etl_metadata.watermarks
    WHERE source_table = p_source_table;

    -- 증분 추출 SQL 생성
    v_sql := format(
        'INSERT INTO %I SELECT * FROM %I WHERE %I > %s',
        p_target_table,
        p_source_table,
        v_watermark_column,
        CASE v_watermark_type
            WHEN 'TIMESTAMP' THEN quote_literal(v_watermark_value::TIMESTAMP)
            WHEN 'INTEGER' THEN v_watermark_value
            WHEN 'STRING' THEN quote_literal(v_watermark_value)
        END
    );

    -- 실행
    EXECUTE v_sql;
    GET DIAGNOSTICS v_rows_extracted = ROW_COUNT;

    -- 새 워터마크 계산
    EXECUTE format(
        'SELECT MAX(%I)::TEXT FROM %I',
        v_watermark_column,
        p_target_table
    ) INTO v_new_watermark;

    -- 워터마크 업데이트
    IF v_new_watermark IS NOT NULL THEN
        UPDATE etl_metadata.watermarks
        SET watermark_value = v_new_watermark,
            last_updated = CURRENT_TIMESTAMP
        WHERE source_table = p_source_table;
    END IF;

    RETURN v_rows_extracted;
END;
$$ LANGUAGE plpgsql;

    'staging.customers'

델타 로드 패턴

-- 델타 처리: INSERT, UPDATE, DELETE 모두 처리
CREATE TABLE staging.customer_deltas (
    operation CHAR(1),  -- I=Insert, U=Update, D=Delete
    customer_id INT,
    customer_name VARCHAR(200),
    email VARCHAR(100),
    customer_segment VARCHAR(50),
    change_timestamp TIMESTAMP
);

-- 델타 적용 프로시저
CREATE OR REPLACE PROCEDURE apply_customer_deltas()
LANGUAGE plpgsql
AS $$
BEGIN
    -- DELETE 처리
    UPDATE dwh.dim_customer
    SET is_current = FALSE,
        expiration_date = d.change_timestamp::DATE,
        updated_timestamp = CURRENT_TIMESTAMP
    FROM staging.customer_deltas d
    WHERE dwh.dim_customer.customer_id = d.customer_id
      AND dwh.dim_customer.is_current = TRUE
      AND d.operation = 'D';

    -- UPDATE 처리 (Type 2 SCD)
    -- 기존 행 만료
    UPDATE dwh.dim_customer
    SET is_current = FALSE,
        expiration_date = d.change_timestamp::DATE - INTERVAL '1 day',
        updated_timestamp = CURRENT_TIMESTAMP
    FROM staging.customer_deltas d
    WHERE dwh.dim_customer.customer_id = d.customer_id
      AND dwh.dim_customer.is_current = TRUE
      AND d.operation = 'U';

    -- 새 버전 INSERT
    INSERT INTO dwh.dim_customer (
        customer_key, customer_id, customer_name, email,
        customer_segment, effective_date, is_current
    )
    SELECT
        NEXTVAL('customer_key_seq'),
        d.customer_id,
        d.customer_name,
        d.email,
        d.customer_segment,
        d.change_timestamp::DATE,
        TRUE
    FROM staging.customer_deltas d
    WHERE d.operation IN ('U', 'I');

    -- 델타 처리 완료 로그
    INSERT INTO etl_metadata.delta_processing_log (
        table_name, inserts, updates, deletes, processed_at
    )
    SELECT
        'dim_customer',
        COUNT(*) FILTER (WHERE operation = 'I'),
        COUNT(*) FILTER (WHERE operation = 'U'),
        COUNT(*) FILTER (WHERE operation = 'D'),
        CURRENT_TIMESTAMP
    FROM staging.customer_deltas;

    -- 스테이징 정리
    TRUNCATE TABLE staging.customer_deltas;
END;
$$;

7.8 ETL 모범 사례 및 성능 튜닝

모범 사례 체크리스트

  • 증분 로딩 우선: 전체 로드는 최소화
  • 병렬 처리: 독립적인 작업은 병렬 실행
  • 일괄 처리: 행 단위가 아닌 집합 기반 처리
  • 인덱스 관리: 로드 전 삭제, 로드 후 재생성
  • 파티셔닝 활용: 파티션 단위로 로드
  • 압축 사용: 네트워크 전송 시 압축
  • 재시작 가능성: 체크포인트 기반 재시작
  • 멱등성: 같은 데이터를 여러 번 실행해도 동일 결과
  • 모니터링: 실행 시간, 데이터 품질, 오류율 추적
  • 문서화: 데이터 계보, 변환 로직 문서화
  • 테스트: 단위 테스트, 통합 테스트, 데이터 품질 테스트
  • 버전 관리: ETL 코드 Git 관리

7.9 요약

ETL(Extract, Transform, Load)은 데이터 웨어하우스 구축의 핵심 프로세스입니다:

  • 추출(Extract)은 전체 추출, 증분 추출, CDC 등 다양한 전략을 사용하며, 소스 시스템 부하를 최소화하면서 효율적으로 데이터를 가져와야 합니다.
  • 변환(Transform)은 데이터 정제, 검증, 표준화, 강화를 통해 고품질의 분석 준비 데이터를 생성합니다. 비즈니스 규칙 적용과 SCD 처리가 핵심입니다.
  • 적재(Load)는 차원과 팩트를 데이터 웨어하우스에 로드하며, 성능 최적화를 위해 COPY/BULK INSERT, 병렬 처리, 인덱스 관리가 중요합니다.
  • ETL vs ELT: 전통적 ETL은 중간 계층에서 변환하지만, 현대 클라우드 DW에서는 ELT 패턴이 선호되어 웨어하우스의 MPP 성능을 활용합니다.
  • 오류 처리는 재시도 로직, 체크포인트 기반 재시작, 포괄적인 로깅을 통해 안정적인 ETL 파이프라인을 구축합니다.
  • 오케스트레이션은 Airflow, Talend, dbt 등 도구를 사용하여 복잡한 워크플로우를 관리하고 모니터링합니다.
  • 증분 로딩은 워터마크, 델타 테이블, CDC를 활용하여 변경된 데이터만 처리하여 성능을 크게 개선합니다.

성공적인 ETL은 성능, 안정성, 유지보수성의 균형을 맞추는 것입니다. 비즈니스 요구사항과 데이터 규모에 맞는 적절한 전략을 선택하고, 지속적인 모니터링과 최적화가 필요합니다.
