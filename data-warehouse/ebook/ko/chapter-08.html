클라우드 데이터 웨어하우스와 최적화

8.1 클라우드 데이터 웨어하우스 혁명

클라우드 데이터 웨어하우스는 전통적인 온프레미스 솔루션의 한계를 극복하고, 데이터 분석의 패러다임을 완전히 바꾸었습니다. 2010년대 중반 이후, Snowflake, Google BigQuery, Amazon Redshift, Azure Synapse와 같은 클라우드 네이티브 플랫폼이 등장하면서 데이터 웨어하우징은 새로운 시대를 맞이했습니다.

클라우드 DW의 주요 장점

  • 탄력성 (Elasticity): 몇 초 내에 컴퓨팅 리소스 확장/축소
  • 확장성 (Scalability): 거의 무제한 스토리지와 처리 능력
  • 비용 효율성: 사용한 만큼만 지불 (Pay-as-you-go)
  • 운영 간소화: 인프라 관리 불필요, 자동 백업/패치
  • 고성능: MPP(Massively Parallel Processing) 아키텍처
  • 글로벌 가용성: 전 세계 리전에서 접근 가능
  • 보안 및 컴플라이언스: 엔터프라이즈급 보안, 각종 인증
  • 빠른 배포: 몇 분 내에 완전한 DW 환경 구축

온프레미스 vs 클라우드 비교

온프레미스와 클라우드 데이터 웨어하우스 비교
특성 온프레미스 클라우드
초기 투자 매우 높음 ($수백만) 낮음 (무료 시작 가능)
배포 시간 몇 개월 ~ 년 몇 분 ~ 시간
확장성 하드웨어 구매 필요 즉시 확장 가능
운영 부담 높음 (DBA, 인프라 팀) 낮음 (관리형 서비스)
비용 모델 CapEx (자본 지출) OpEx (운영 비용)
성능 하드웨어 한계 거의 무제한 확장
재해 복구 별도 구축 필요 기본 제공
업그레이드 다운타임 필요 무중단 업그레이드
데이터 위치 온프레미스 (통제 높음) 클라우드 (리전 선택)
혁신 속도 느림 빠름 (지속적 개선)

클라우드 DW 시장 현황

2025년 현재, 클라우드 데이터 웨어하우스 시장은 다음과 같이 구성되어 있습니다:

  • Snowflake: 클라우드 네이티브 DW의 선구자, 멀티클라우드 지원
  • Google BigQuery: 서버리스, 빠른 쿼리, ML 통합
  • Amazon Redshift: AWS 생태계 통합, 성숙한 기능
  • Azure Synapse Analytics: 통합 분석 플랫폼, Power BI 연동
  • Databricks SQL: 레이크하우스 아키텍처
  • Firebolt: 극한 성능 최적화

8.2 Snowflake - 클라우드 네이티브의 선구자

Snowflake는 2012년 설립되어 클라우드 네이티브 데이터 웨어하우스의 표준을 정의했습니다. 독특한 아키텍처와 멀티클라우드 지원으로 빠르게 성장했습니다.

Snowflake 아키텍처

Snowflake의 혁신적인 아키텍처는 세 가지 계층으로 구성됩니다:

Snowflake 3-Layer Architecture:

┌─────────────────────────────────────┐
│     쿼리 처리 계층 (Compute)        │
│                                     │
│  Virtual Warehouse 1  VW 2  VW 3   │ ← 독립적으로 확장
│  (X-Small ~ 6X-Large)               │
│                                     │
│  각 웨어하우스:                      │
│  • 독립적 컴퓨팅 클러스터            │
│  • 자동 일시정지/재시작              │
│  • 초 단위 과금                      │
└─────────────────────────────────────┘
            ↕
┌─────────────────────────────────────┐
│      서비스 계층 (Services)          │
│                                     │
│  • 쿼리 최적화 및 실행 계획          │
│  • 메타데이터 관리                   │
│  • 트랜잭션 관리                     │
│  • 보안 및 접근 제어                 │
│  • 자동 클러스터링                   │
└─────────────────────────────────────┘
            ↕
┌─────────────────────────────────────┐
│     스토리지 계층 (Storage)          │
│                                     │
│  • S3/Azure Blob/GCS에 저장         │
│  • 자동 압축 (평균 4:1)              │
│  • 컬럼형 저장                       │
│  • 마이크로 파티션 (16MB)            │
│  • Time Travel (최대 90일)          │
└─────────────────────────────────────┘

Snowflake 주요 기능

-- 데이터베이스 및 웨어하우스 생성
CREATE DATABASE analytics_db;
CREATE SCHEMA analytics_db.sales;

-- Virtual Warehouse 생성
CREATE WAREHOUSE compute_wh
    WITH WAREHOUSE_SIZE = 'LARGE'
    AUTO_SUSPEND = 60              -- 60초 유휴 후 자동 일시정지
    AUTO_RESUME = TRUE             -- 쿼리 시 자동 재시작
    INITIALLY_SUSPENDED = TRUE;

-- 웨어하우스 사이즈 변경 (즉시 적용)
ALTER WAREHOUSE compute_wh SET WAREHOUSE_SIZE = 'X-LARGE';

-- 멀티 클러스터 웨어하우스 (동시 사용자 증가 시)
CREATE WAREHOUSE multi_cluster_wh
    WITH WAREHOUSE_SIZE = 'LARGE'
    MIN_CLUSTER_COUNT = 1
    MAX_CLUSTER_COUNT = 10
    SCALING_POLICY = 'STANDARD'    -- or 'ECONOMY'
    AUTO_SUSPEND = 60;

-- 테이블 생성
CREATE TABLE analytics_db.sales.fact_sales (
    sale_id NUMBER,
    sale_date DATE,
    customer_id NUMBER,
    product_id NUMBER,
    quantity NUMBER,
    amount DECIMAL(12,2)
);

-- 자동 클러스터링 (대규모 테이블 성능 최적화)
ALTER TABLE analytics_db.sales.fact_sales
    CLUSTER BY (sale_date, customer_id);

-- Time Travel (과거 데이터 조회)
-- 5분 전 데이터 조회
SELECT * FROM fact_sales
AT(OFFSET => -60*5);

-- 특정 시점 데이터 조회
SELECT * FROM fact_sales
AT(TIMESTAMP => '2025-01-01 00:00:00'::TIMESTAMP);

-- Time Travel로 실수 복구
CREATE TABLE fact_sales_backup AS
SELECT * FROM fact_sales
AT(OFFSET => -60*60);  -- 1시간 전

-- Zero-Copy Clone (즉시 복제, 추가 스토리지 없음)
CREATE TABLE fact_sales_dev CLONE fact_sales;
CREATE DATABASE analytics_db_dev CLONE analytics_db;

-- 외부 스테이지 생성 (S3/Azure/GCS)
CREATE STAGE s3_stage
    URL = 's3://my-bucket/data/'
    CREDENTIALS = (AWS_KEY_ID = 'xxx' AWS_SECRET_KEY = 'yyy');

-- COPY INTO로 대량 로드
COPY INTO fact_sales
FROM @s3_stage/sales/
FILE_FORMAT = (TYPE = CSV FIELD_DELIMITER = ',' SKIP_HEADER = 1)
ON_ERROR = 'CONTINUE';

-- Snowpipe (자동 데이터 수집)
CREATE PIPE sales_pipe
    AUTO_INGEST = TRUE
    AS
    COPY INTO fact_sales
    FROM @s3_stage/sales/
    FILE_FORMAT = (TYPE = JSON);

-- 데이터 공유 (다른 Snowflake 계정과)
CREATE SHARE sales_share;
GRANT USAGE ON DATABASE analytics_db TO SHARE sales_share;
GRANT SELECT ON TABLE fact_sales TO SHARE sales_share;
ALTER SHARE sales_share ADD ACCOUNTS = account1, account2;

Snowflake 최적화 기법

-- 웨어하우스 사이즈 선택 가이드
-- X-Small: 개발/테스트, 소규모 쿼리
-- Small-Medium: 일반 분석, BI 대시보드
-- Large-X-Large: 복잡한 분석, 대량 ETL
-- 2X-Large 이상: 매우 복잡한 쿼리, 빅데이터

-- 쿼리 성능 모니터링
SELECT
    query_id,
    query_text,
    warehouse_name,
    warehouse_size,
    execution_time / 1000 as execution_seconds,
    bytes_scanned / 1024 / 1024 / 1024 as gb_scanned,
    partitions_scanned,
    partitions_total
FROM TABLE(information_schema.query_history())
WHERE start_time >= DATEADD(hour, -1, CURRENT_TIMESTAMP())
ORDER BY execution_time DESC
LIMIT 10;

-- 클러스터링 키 효과 확인
SELECT SYSTEM$CLUSTERING_INFORMATION('fact_sales', '(sale_date)');

-- 파티션 프루닝 활용 (날짜 필터 필수)
-- 나쁜 쿼리 (모든 파티션 스캔)
SELECT * FROM fact_sales
WHERE customer_id = 12345;

-- 좋은 쿼리 (파티션 프루닝)
SELECT * FROM fact_sales
WHERE sale_date >= '2025-01-01'
  AND sale_date < '2025-02-01'
  AND customer_id = 12345;

-- Materialized View (자동 유지)
CREATE MATERIALIZED VIEW mv_monthly_sales AS
SELECT
    DATE_TRUNC('month', sale_date) as month,
    customer_id,
    SUM(amount) as total_sales,
    COUNT(*) as transaction_count
FROM fact_sales
GROUP BY 1, 2;

-- Result Cache 활용 (24시간 캐시)
-- 동일 쿼리는 즉시 반환 (비용 없음)

-- 비용 최적화: 웨어하우스 자동 일시정지
ALTER WAREHOUSE compute_wh SET AUTO_SUSPEND = 60;

-- 비용 모니터링
SELECT
    warehouse_name,
    SUM(credits_used) as total_credits,
    SUM(credits_used) * 3 as estimated_cost_usd  -- $3 per credit 가정
FROM TABLE(information_schema.warehouse_metering_history(
    DATE_RANGE_START => DATEADD(month, -1, CURRENT_DATE())
))
GROUP BY warehouse_name
ORDER BY total_credits DESC;

BigQuery 주요 기능

-- 데이터셋 및 테이블 생성
CREATE SCHEMA `project-id.analytics`;

CREATE TABLE `project-id.analytics.fact_sales`
(
    sale_id INT64,
    sale_date DATE,
    customer_id INT64,
    product_id INT64,
    quantity INT64,
    amount NUMERIC(12,2)
)
PARTITION BY sale_date
CLUSTER BY customer_id, product_id
OPTIONS(
    description = 'Sales fact table',
    require_partition_filter = TRUE  -- 파티션 필터 강제
);

-- 파티션 테이블 (일자별)
CREATE TABLE sales_partitioned
PARTITION BY DATE(sale_timestamp)
AS SELECT * FROM source_data;

-- 정수 범위 파티션
CREATE TABLE sales_by_customer_range
PARTITION BY RANGE_BUCKET(customer_id, GENERATE_ARRAY(0, 100000, 1000))
AS SELECT * FROM source_data;

-- 클러스터링 (파티션 내 정렬)
CREATE TABLE sales_clustered
PARTITION BY sale_date
CLUSTER BY customer_id, product_id
AS SELECT * FROM source_data;

-- 외부 테이블 (GCS, Drive, Bigtable)
CREATE EXTERNAL TABLE `project-id.analytics.external_sales`
OPTIONS (
    format = 'CSV',
    uris = ['gs://my-bucket/sales/*.csv'],
    skip_leading_rows = 1
);

-- 대량 데이터 로드 (무료)
LOAD DATA OVERWRITE analytics.fact_sales
FROM FILES (
    format = 'PARQUET',
    uris = ['gs://my-bucket/sales/*.parquet']
);

-- 스트리밍 INSERT (실시간 데이터)
INSERT INTO `project-id.analytics.fact_sales`
VALUES (1001, CURRENT_DATE(), 12345, 67890, 5, 999.99);

-- Materialized View (자동 갱신)
CREATE MATERIALIZED VIEW analytics.mv_daily_sales AS
SELECT
    sale_date,
    customer_id,
    SUM(amount) as total_sales,
    COUNT(*) as order_count
FROM analytics.fact_sales
GROUP BY sale_date, customer_id;

-- BI Engine (인메모리 분석)
CREATE BI_ENGINE_CAPACITY `project-id.us.my_capacity`
OPTIONS (size_gb = 100);

-- ML 모델 생성 (BigQuery ML)
CREATE OR REPLACE MODEL analytics.customer_churn_model
OPTIONS(
    model_type = 'LOGISTIC_REG',
    input_label_cols = ['churned']
) AS
SELECT
    customer_id,
    recency_days,
    frequency,
    monetary_value,
    churned
FROM analytics.customer_features;

-- 모델 예측
SELECT
    customer_id,
    predicted_churned,
    predicted_churned_probs[OFFSET(0)].prob as churn_probability
FROM ML.PREDICT(
    MODEL analytics.customer_churn_model,
    (SELECT * FROM analytics.customers_current)
)
WHERE predicted_churned = TRUE
ORDER BY churn_probability DESC;

-- 지리공간 분석
CREATE TABLE analytics.store_locations (
    store_id INT64,
    store_name STRING,
    location GEOGRAPHY
);

-- 반경 10km 내 매장 찾기
SELECT
    store_id,
    store_name,
    ST_DISTANCE(
        location,
        ST_GEOGPOINT(-122.419, 37.775)  -- San Francisco
    ) / 1000 as distance_km
FROM analytics.store_locations
WHERE ST_DWithin(
    location,
    ST_GEOGPOINT(-122.419, 37.775),
    10000  -- 10km in meters
)
ORDER BY distance_km;

BigQuery 최적화

-- 비용 최적화 1: SELECT * 금지
-- 나쁜 쿼리 (전체 컬럼 스캔)
SELECT * FROM analytics.fact_sales
WHERE sale_date = '2025-01-01';

-- 좋은 쿼리 (필요한 컬럼만)
SELECT customer_id, amount
FROM analytics.fact_sales
WHERE sale_date = '2025-01-01';

-- 비용 최적화 2: 파티션 필터 사용
-- 나쁜 쿼리 (전체 파티션 스캔)
SELECT SUM(amount)
FROM analytics.fact_sales
WHERE customer_id = 12345;

-- 좋은 쿼리 (파티션 프루닝)
SELECT SUM(amount)
FROM analytics.fact_sales
WHERE sale_date BETWEEN '2025-01-01' AND '2025-01-31'
  AND customer_id = 12345;

-- 비용 최적화 3: 클러스터링 활용
-- 클러스터링 키로 필터링하면 스캔 데이터 감소

-- 예상 비용 확인
SELECT
    SUM(size_bytes) / 1024 / 1024 / 1024 / 1024 as total_tb,
    SUM(size_bytes) / 1024 / 1024 / 1024 / 1024 * 6.25 as estimated_cost_usd
FROM `project-id.analytics.__TABLES__`;

-- 슬롯 사용량 모니터링
SELECT
    job_id,
    user_email,
    total_slot_ms / 1000 as slot_seconds,
    total_bytes_processed / 1024 / 1024 / 1024 as gb_processed,
    total_bytes_billed / 1024 / 1024 / 1024 as gb_billed,
    total_bytes_billed / 1024 / 1024 / 1024 / 1024 * 6.25 as cost_usd
FROM `region-us`.INFORMATION_SCHEMA.JOBS_BY_PROJECT
WHERE creation_time >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 1 DAY)
  AND job_type = 'QUERY'
  AND state = 'DONE'
ORDER BY total_bytes_billed DESC
LIMIT 10;

-- Flat-rate pricing (예측 가능한 비용)
-- 100 슬롯 = $2,000/월
-- 500 슬롯 = $10,000/월
-- 대량 쿼리 실행 시 On-Demand보다 저렴

8.4 Amazon Redshift - AWS 생태계의 중심

Amazon Redshift는 2013년 출시된 AWS의 데이터 웨어하우스 서비스로, AWS 생태계와의 긴밀한 통합이 강점입니다.

8.8 멀티클라우드 및 데이터 메시 전략

멀티클라우드 데이터 웨어하우스

대기업은 여러 클라우드를 동시에 사용하는 멀티클라우드 전략을 채택하고 있습니다.

-- Snowflake 멀티클라우드 데이터 공유

-- AWS 리전의 Snowflake 계정
USE WAREHOUSE compute_wh;
USE DATABASE sales_aws;

-- Azure 리전으로 데이터 복제
CREATE DATABASE sales_azure
  FROM SHARE aws_account.sales_share
  AS REPLICA OF sales_aws;

-- GCP 리전의 쿼리
-- 자동으로 가장 가까운 리전에서 데이터 접근

-- 크로스 클라우드 자동 페일오버
ALTER DATABASE sales_db
  ENABLE FAILOVER TO ACCOUNTS (azure_account, gcp_account);

-- 글로벌 쿼리 (Snowflake 유니버설 데이터 공유)
SELECT
    region,
    SUM(amount) as total_sales
FROM aws_account.sales.fact_sales
UNION ALL
SELECT
    region,
    SUM(amount) as total_sales
FROM azure_account.sales.fact_sales
UNION ALL
SELECT
    region,
    SUM(amount) as total_sales
FROM gcp_account.sales.fact_sales;

데이터 메시 아키텍처

데이터 메시는 중앙집중식 DW 대신 분산된 도메인 중심 데이터 아키텍처를 제안합니다.

Data Mesh 개념:

┌─────────────────────┐  ┌─────────────────────┐
│  Sales Domain       │  │  Marketing Domain   │
│                     │  │                     │
│  • Snowflake DW     │  │  • BigQuery DW      │
│  • 데이터 제품       │  │  • 데이터 제품       │
│  • API/공유         │  │  • API/공유         │
└─────────────────────┘  └─────────────────────┘
         ↓                        ↓
┌─────────────────────────────────────────────┐
│       Data Catalog & Governance             │
│       (메타데이터, 품질, 계보)                │
└─────────────────────────────────────────────┘
         ↓                        ↓
┌─────────────────────┐  ┌─────────────────────┐
│  Finance Domain     │  │  Operations Domain  │
│                     │  │                     │
│  • Redshift DW      │  │  • Synapse DW       │
│  • 데이터 제품       │  │  • 데이터 제품       │
│  • API/공유         │  │  • API/공유         │
└─────────────────────┘  └─────────────────────┘

각 도메인:
• 독립적인 DW 플랫폼 선택
• 자체 데이터 파이프라인
• 표준화된 데이터 제품 제공
• 중앙 거버넌스 준수

8.9 미래 트렌드

1. 데이터 레이크하우스

데이터 레이크의 유연성과 데이터 웨어하우스의 성능을 결합한 새로운 아키텍처입니다.

  • Databricks Lakehouse: Delta Lake 기반
  • Snowflake + Iceberg: 오픈 테이블 포맷
  • BigQuery + Biglake: 멀티클라우드 레이크

2. 실시간 분석

배치 처리에서 스트리밍 처리로의 전환이 가속화되고 있습니다.

  • Snowflake Streams: CDC 및 변경 추적
  • BigQuery Streaming: 초당 10만 행 INSERT
  • Redshift Streaming Ingestion: Kinesis 통합

3. AI/ML 네이티브 통합

데이터 웨어하우스에서 직접 ML 모델 학습 및 추론이 가능합니다.

  • BigQuery ML: SQL로 ML 모델 생성
  • Snowflake Snowpark ML: Python ML 파이프라인
  • Redshift ML: SageMaker 자동 통합

4. 서버리스 everywhere

모든 컴퓨팅이 서버리스로 전환되고 있습니다.

  • 자동 확장/축소
  • 초 단위 과금
  • Zero 관리 오버헤드

5. 데이터 거버넌스 자동화

규제 강화에 따라 데이터 거버넌스가 더욱 중요해지고 있습니다.

  • 자동 데이터 분류: PII, 민감 정보 자동 탐지
  • 동적 마스킹: 역할 기반 데이터 마스킹
  • 데이터 계보 추적: 자동 리니지 매핑

8.10 요약

클라우드 데이터 웨어하우스는 현대 데이터 분석의 핵심 인프라입니다:

  • Snowflake는 독특한 아키텍처와 멀티클라우드 지원으로 유연성과 확장성을 제공합니다. Zero-Copy Cloning, Time Travel, 데이터 공유 기능이 강력합니다.
  • BigQuery는 완전 서버리스 아키텍처로 페타바이트급 데이터를 빠르게 분석할 수 있으며, BigQuery ML을 통한 ML 통합이 우수합니다.
  • Redshift는 AWS 생태계와의 긴밀한 통합이 강점이며, RA3 노드와 Redshift Spectrum으로 유연성이 향상되었습니다.
  • Azure Synapse는 DW, 빅데이터, 데이터 통합을 하나의 플랫폼으로 통합하여 포괄적인 분석 환경을 제공합니다.
  • 플랫폼 선택은 기존 클라우드 투자, 비즈니스 요구사항, 팀 역량, 비용 모델을 종합적으로 고려해야 합니다.
  • 마이그레이션은 단계적 접근이 중요하며, PoC → 파일럿 → 점진적 이관 → 최적화 순서로 진행해야 합니다.
  • 미래 트렌드는 데이터 레이크하우스, 실시간 분석, AI/ML 통합, 서버리스, 자동화된 거버넌스로 나아가고 있습니다.

클라우드 데이터 웨어하우스는 계속 진화하고 있으며, 조직은 비즈니스 가치를 극대화하면서 비용을 최적화하는 전략이 필요합니다. 멀티클라우드와 데이터 메시 같은 현대적 아키텍처 패턴도 고려해야 합니다.

복습 문제

  1. Snowflake의 3계층 아키텍처(스토리지, 컴퓨팅, 서비스)를 설명하고, 이 아키텍처가 전통적인 온프레미스 DW와 비교하여 어떤 장점을 제공하는지 논하시오.
  2. BigQuery의 서버리스 아키텍처와 Redshift의 클러스터 기반 아키텍처를 비교하고, 각각 어떤 워크로드에 적합한지 설명하시오. 비용 모델의 차이도 포함하시오.
  3. 클라우드 DW 플랫폼(Snowflake, BigQuery, Redshift, Synapse) 중 하나를 선택할 때 고려해야 할 주요 요소 5가지를 제시하고, 의사결정 프로세스를 설명하시오.
  4. 온프레미스 Oracle DW에서 클라우드 DW로 마이그레이션하는 단계별 전략을 수립하시오. 평가, PoC, 파일럿, 이관, 최적화 각 단계의 핵심 활동과 산출물을 포함하시오.
  5. 각 클라우드 DW 플랫폼의 성능 최적화 기법을 비교하시오. Snowflake의 클러스터링, BigQuery의 파티셔닝, Redshift의 distribution key, Synapse의 분산 전략을 SQL 예제와 함께 설명하시오.
  6. 데이터 레이크하우스 아키텍처가 무엇이며, 전통적인 데이터 웨어하우스와 데이터 레이크의 한계를 어떻게 극복하는지 설명하시오. Databricks Lakehouse 또는 Snowflake + Iceberg를 예로 드시오.
