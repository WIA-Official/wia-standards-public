Phase 2: API 인터페이스 - 공동체 중심 접근 제어

WIA-HERITAGE-005: Traditional Knowledge Standard

05.1 전통적 API 설계의 근본적 문제

일반적인 API 설계는 "누가 이 데이터에 접근할 수 있는가?"라는 기술적 질문에 초점을 맞춥니다. 그러나 전통 지식의 경우, 진짜 질문은 "공동체가 이 접근을 승인했는가?", "사용 목적이 공동체의 가치와 일치하는가?", "이익이 공정하게 공유되는가?" 입니다.

기존 API의 한계

2010년대 초반, 여러 생물다양성 데이터베이스들이 원주민 전통 지식을 포함하는 RESTful API를 공개했습니다. 이들의 공통된 문제점:

  • API 키 기반 인증의 맹점: API 키만 있으면 누구나 대량으로 데이터를 수집할 수 있었고, 공동체는 누가 자신들의 지식을 사용하는지조차 몰랐습니다.
  • 목적 검증 없음: 제약 회사 연구자와 학부생 숙제가 동일한 접근 권한을 가졌고, 상업적 이용과 학술 연구가 구분되지 않았습니다.
  • 실시간 통보 부재: 민감한 의례 지식이 수백 번 조회되어도 공동체는 알 수 없었습니다.
  • 이익 추적 불가: 전통 지식이 신약 개발로 이어져도 원천 데이터와 연결할 방법이 없었습니다.
⚠️ 사례 연구: 남아프리카 후디아 사건의 API 측면

2001년, 남아프리카 CSIR(과학산업연구협의회)은 산(San)족의 후디아 식물 지식을 데이터베이스화하고 API로 공개했습니다. 제약 회사 Pfizer는 이 API를 통해 데이터를 수집하고, 산족의 동의 없이 식욕 억제제 특허를 출원했습니다.

문제점:

  • API 접근 로그에 Pfizer 접근 기록이 있었지만, 산족에게 통보되지 않음
  • API는 "학술 연구용"으로 표시되어 있었지만, 상업적 이용 검증 메커니즘 없음
  • 데이터에 출처가 표시되어 있었지만, 자동 이익 공유 장치 부재

결과: 6년간의 법적 분쟁 끝에 산족은 이익의 8%만 받았고, 이미 수억 달러의 연구 개발비가 지출된 후였습니다.

05.2 WIA-TK API 아키텍처: 3층 보안 모델

WIA-HERITAGE-005 API는 기술적 보안, 공동체 거버넌스, 블록체인 투명성의 3층 구조로 설계되었습니다.

3층 보안 아키텍처

계층목적기술책임 주체
Layer 1: 기술 인증신원 확인OAuth 2.0 + OpenID Connect시스템
Layer 2: 공동체 승인목적 및 윤리 검증FPIC 워크플로우 엔진공동체 대표
Layer 3: 블록체인 기록투명성 및 이익 추적Ethereum 스마트 계약분산 원장

API 엔드포인트 구조

WIA-TK API는 RESTful과 GraphQL 두 가지 인터페이스를 제공합니다:

Base URL: https://api.wia-heritage.org/v1/

# RESTful 엔드포인트
GET    /knowledge/{id}              # 단일 지식 항목 조회
GET    /knowledge/search            # 지식 검색
POST   /knowledge                   # 새 지식 기록 (공동체만)
PUT    /knowledge/{id}              # 지식 업데이트 (공동체만)
DELETE /knowledge/{id}              # 지식 삭제 (공동체만)

GET    /communities/{id}            # 공동체 정보
GET    /communities/{id}/knowledge  # 공동체별 지식 목록

POST   /consent/request             # FPIC 동의 요청
GET    /consent/status/{id}         # 동의 상태 확인
POST   /consent/revoke/{id}         # 동의 철회

GET    /access-log/{knowledge_id}   # 접근 이력 (블록체인)
GET    /benefit-tracking/{id}       # 이익 추적

# GraphQL 엔드포인트
POST   /graphql                     # GraphQL 쿼리/뮤테이션

05.3 Layer 1: OAuth 2.0 + 공동체 확장

WIA-TK API는 표준 OAuth 2.0 프로토콜을 사용하지만, 원주민 공동체의 특수한 요구사항을 반영하여 확장했습니다.

사용자 유형별 인증 흐름

사용자 유형인증 방식추가 요구사항
공동체 구성원공동체 ID + 생체인증장로 위원회 승인
학술 연구자대학 ORCID + 윤리 승인서IRB 증명, 연구 계획서
정부 기관정부 SSO공식 협력 협약
민간 기업기업 인증 + 법인 등록이익 공유 계약 사전 체결
일반 대중이메일/소셜 로그인Level 1 (공개) 데이터만

05.4 Layer 2: FPIC (사전 정보 동의) 자동화

FPIC (Free, Prior and Informed Consent)는 유엔 원주민 권리 선언(UNDRIP)의 핵심 원칙입니다. WIA-TK API는 이를 디지털 워크플로우로 구현합니다.

FPIC 워크플로우 단계

단계설명소요 시간주체
1. 정보 제공연구자가 목적, 방법, 예상 결과를 공동체 언어로 설명7일연구자
2. 공동체 숙의장로 위원회와 공동체 구성원이 내부 회의14-30일공동체
3. 질문 및 협상추가 정보 요청, 조건 협상7-14일양측
4. 결정공동체가 승인/거부/조건부 승인 결정1일공동체
5. 계약 체결이익 공유 조건을 스마트 계약으로 기록3일양측
6. 지속적 모니터링연구 진행 상황 보고, 동의 철회 가능연구 기간공동체

05.5 Layer 3: 블록체인 기반 투명성

모든 API 접근, 동의 기록, 이익 분배가 블록체인에 기록되어 변조 불가능하고 투명하게 추적됩니다.

스마트 계약 구조

Ethereum 기반 스마트 계약으로 자동 이익 분배를 구현합니다:

// Solidity 스마트 계약 (단순화된 예시)
contract TraditionalKnowledgeAccess {
    struct ConsentRecord {
        address researcher;
        address communityRepresentative;
        string knowledgeId;
        uint256 approvalDate;
        uint256 expirationDate;
        uint8 royaltyPercentage;
        bool isActive;
    }

    mapping(bytes32 => ConsentRecord) public consents;
    mapping(address => uint256) public communityBalances;

    event ConsentGranted(
        bytes32 consentId,
        address indexed researcher,
        address indexed community,
        string knowledgeId,
        uint256 expirationDate
    );

    event AccessLogged(
        bytes32 consentId,
        address indexed researcher,
        string knowledgeId,
        uint256 timestamp
    );

    event RoyaltyPaid(
        address indexed community,
        uint256 amount,
        string commercialProduct
    );

    function grantConsent(
        address _researcher,
        address _community,
        string memory _knowledgeId,
        uint256 _durationDays,
        uint8 _royaltyPercentage
    ) public returns (bytes32) {
        // FPIC 승인 시 호출
        bytes32 consentId = keccak256(
            abi.encodePacked(_researcher, _community, _knowledgeId, block.timestamp)
        );

        consents[consentId] = ConsentRecord({
            researcher: _researcher,
            communityRepresentative: _community,
            knowledgeId: _knowledgeId,
            approvalDate: block.timestamp,
            expirationDate: block.timestamp + (_durationDays * 1 days),
            royaltyPercentage: _royaltyPercentage,
            isActive: true
        });

        emit ConsentGranted(
            consentId,
            _researcher,
            _community,
            _knowledgeId,
            consents[consentId].expirationDate
        );

        return consentId;
    }

    function logAccess(bytes32 _consentId, string memory _knowledgeId) public {
        require(consents[_consentId].isActive, "Consent not active");
        require(block.timestamp <= consents[_consentId].expirationDate, "Consent expired");

        emit AccessLogged(
            _consentId,
            msg.sender,
            _knowledgeId,
            block.timestamp
        );
    }

    function payRoyalty(
        address _community,
        string memory _productName
    ) public payable {
        communityBalances[_community] += msg.value;

        emit RoyaltyPaid(_community, msg.value, _productName);
    }

    function withdrawRoyalty() public {
        uint256 balance = communityBalances[msg.sender];
        require(balance > 0, "No balance to withdraw");

        communityBalances[msg.sender] = 0;
        payable(msg.sender).transfer(balance);
    }
}

05.6 GraphQL API: 유연한 쿼리

복잡한 관계를 가진 전통 지식을 효율적으로 조회하기 위해 GraphQL 인터페이스를 제공합니다.

스키마 정의

type TraditionalKnowledge {
  id: ID!
  label: [MultilingualLabel!]!
  scientificName: String
  category: KnowledgeCategory!
  accessLevel: AccessLevel!

  # 관계
  treats: [MedicalCondition!]
  usedBy: [Community!]!
  taughtBy: [Elder!]
  relatedKnowledge: [TraditionalKnowledge!]

  # 메타데이터
  provenance: Provenance!
  consent: ConsentRecord
  accessHistory: [AccessLog!]

  # 멀티미디어
  images: [Image!]
  audioRecordings: [Audio!]
  videos: [Video!]
}

type Community {
  id: ID!
  name: String!
  language: String!
  location: GeoLocation!
  population: Int

  # 지식
  knowledgeItems: [TraditionalKnowledge!]!
  elders: [Elder!]!

  # 거버넌스
  representatives: [Person!]!
  consentRecords: [ConsentRecord!]!
}

type ConsentRecord {
  id: ID!
  community: Community!
  researcher: Person!
  knowledgeItems: [TraditionalKnowledge!]!
  purpose: String!
  approvalDate: DateTime!
  expirationDate: DateTime!
  conditions: ConsentConditions!
  blockchainTx: String!
  status: ConsentStatus!
}

type Query {
  # 지식 조회
  knowledge(id: ID!): TraditionalKnowledge
  searchKnowledge(
    query: String
    category: KnowledgeCategory
    community: ID
    accessibleBy: ID
  ): [TraditionalKnowledge!]!

  # 공동체 조회
  community(id: ID!): Community
  communities(region: String): [Community!]!

  # 동의 관리
  myConsents: [ConsentRecord!]!
  communityConsents(communityId: ID!): [ConsentRecord!]!
}

type Mutation {
  # FPIC 워크플로우
  requestConsent(input: ConsentRequestInput!): ConsentRequest!
  approveConsent(consentId: ID!, conditions: ConsentConditions): ConsentRecord!
  revokeConsent(consentId: ID!, reason: String!): Boolean!

  # 지식 기여 (공동체만)
  contributeKnowledge(input: KnowledgeInput!): TraditionalKnowledge!
  updateKnowledge(id: ID!, input: KnowledgeInput!): TraditionalKnowledge!
}

05.8 장 요약

✅ 핵심 내용
  1. 공동체 중심 설계: 기존 API의 기술 중심 접근과 달리, WIA-TK API는 "공동체의 승인"을 핵심으로 하며, 남아프리카 후디아 사건과 같은 생물해적행위를 방지하기 위한 실시간 통보 시스템을 구현합니다.
  2. 3층 보안 아키텍처: OAuth 2.0 기술 인증, FPIC 공동체 승인, 블록체인 투명성 기록의 3층 구조로 접근 제어와 이익 추적을 보장합니다. 각 층은 독립적으로 작동하면서 상호 보완합니다.
  3. FPIC 자동화: 유엔 원주민 권리 선언의 핵심인 사전 정보 동의를 6단계 디지털 워크플로우로 구현하며, 공동체 언어로 된 다국어 문서를 자동 생성하고 조건부 승인 메커니즘을 지원합니다.
  4. 블록체인 투명성: Ethereum 스마트 계약으로 모든 접근, 동의, 이익 분배를 변조 불가능하게 기록하며, 상업화 시 자동 로열티 분배가 가능합니다. 공동체는 실시간으로 수익을 추적하고 인출할 수 있습니다.
  5. GraphQL 유연성: 전통 지식의 복잡한 관계를 효율적으로 조회하기 위해 RESTful API 외에 GraphQL 인터페이스를 제공하며, 클라이언트가 필요한 데이터만 정확히 요청할 수 있습니다.
  6. 공정 사용 보호: 사용자 유형별 차등 속도 제한과 과도한 접근 탐지 시스템으로 대량 스크래핑을 방지하고, 의심스러운 패턴 발견 시 공동체에 자동 알림을 전송합니다.
弘益人間

API는 단순한 데이터 접근 인터페이스가 아니라, 공동체와 연구자 간의 신뢰 브리지입니다. 기술은 투명성을 보장하고, 공동체는 통제권을 유지하며, 인류는 지혜를 공유합니다.

05.9 복습 문제

  1. 남아프리카 후디아 사건에서 API의 어떤 결함이 생물해적행위를 가능하게 했는지 분석하고, WIA-TK API의 3층 보안 아키텍처가 이를 어떻게 방지하는지 각 층별로 설명하세요.
  2. OAuth 2.0 토큰의 표준 클레임과 WIA-TK 확장 클레임을 비교하고, 전통 지식 접근 제어에 특화된 5개 이상의 커스텀 클레임의 역할을 설명하세요.
  3. FPIC 워크플로우의 6단계를 상세히 설명하고, 각 단계에서 공동체의 자기결정권이 어떻게 보장되는지 논의하세요. 조건부 승인의 예시를 3가지 제시하세요.
  4. Ethereum 스마트 계약의 `grantConsent`, `logAccess`, `payRoyalty` 함수가 어떻게 작동하는지 설명하고, 제약 회사가 전통 지식 기반 신약을 출시했을 때 자동 로열티 분배 프로세스를 단계별로 서술하세요.
  5. GraphQL API가 RESTful API보다 전통 지식 조회에 적합한 이유를 설명하고, 제공된 스키마를 사용하여 "안데스 지역의 기후 변화 적응 농업 지식"을 조회하는 복잡한 쿼리를 작성하세요.
  6. 속도 제한 정책에서 공동체 구성원에게 무제한 접근을 허용하는 이유를 논의하고, 과도한 접근 탐지 시스템이 10분 내 127회 조회를 "자동화된 스크래핑"으로 판단하는 근거를 설명하세요.

05.10 다음 장 예고

제6장에서는 여러 공동체와 기관 간 전통 지식을 안전하게 교환하기 위한 프로토콜을 다룹니다. 종단 간 암호화(E2EE) 통신 체계, 장로 디지털 서명을 통한 지식 진위 검증, 문화적 금기를 자동으로 필터링하는 시스템, 동의 철회 시 분산 네트워크에서 데이터를 완전히 삭제하는 메커니즘, 그리고 국경을 넘는 원주민 공동체 간 지식 공유 프로토콜을 상세히 살펴봅니다. 실제 사례로 아마존 유역 7개국 원주민 연합의 의약 지식 교환 네트워크를 분석합니다.
