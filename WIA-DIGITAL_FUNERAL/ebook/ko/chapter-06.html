제6장: PHASE 3 - 프로토콜

弘益人間 (홍익인간)

"널리 인간을 이롭게 하라"

강력한 보안과 개인정보 보호는 모든 사람의 존엄과 추모의 권리를 지킵니다

6.1 보안 프로토콜

디지털 장례 서비스는 고인과 유족의 가장 민감한 정보를 다루기 때문에, 최고 수준의 보안이 요구됩니다. WIA-DIGITAL_FUNERAL 표준의 PHASE 3는 데이터 전송 보안, 저장 보안, 접근 제어, 감사 추적 등 다층 보안 아키텍처를 정의합니다. 이는 국제 보안 표준(ISO/IEC 27001, SOC 2)을 준수하며, 산업 최고 수준의 보안 관행을 구현합니다.

보안은 단순히 기술적 조치에 그치지 않고, 조직 문화, 프로세스, 사람의 역량을 모두 포괄하는 종합적인 접근이 필요합니다. 정기적인 보안 감사, 침투 테스트, 취약점 스캔, 직원 교육 등을 통해 지속적으로 보안 수준을 유지하고 개선해야 합니다.

6.1.1 전송 계층 보안 (Transport Security)

TLS 1.3 필수 요구사항

모든 네트워크 통신은 TLS 1.3 이상의 암호화를 사용해야 합니다. TLS 1.2 이하 버전은 보안 취약점이 발견되었으므로 사용을 금지합니다.

// Node.js에서 TLS 1.3 강제 설정
const https = require('https');
const fs = require('fs');

const serverOptions = {
    key: fs.readFileSync('/path/to/private-key.pem'),
    cert: fs.readFileSync('/path/to/certificate.pem'),
    ca: fs.readFileSync('/path/to/ca-certificate.pem'),

    // TLS 설정
    minVersion: 'TLSv1.3',
    maxVersion: 'TLSv1.3',

    // 강력한 암호 스위트만 허용
    ciphers: [
        'TLS_AES_256_GCM_SHA384',
        'TLS_CHACHA20_POLY1305_SHA256',
        'TLS_AES_128_GCM_SHA256'
    ].join(':'),

    // 완전 순방향 비밀성 (Perfect Forward Secrecy) 보장
    honorCipherOrder: true,

    // 세션 재사용 비활성화 (보안 강화)
    sessionTimeout: 0
};

const server = https.createServer(serverOptions, (req, res) => {
    // HSTS 헤더 추가 (브라우저가 항상 HTTPS 사용하도록 강제)
    res.setHeader('Strict-Transport-Security',
        'max-age=31536000; includeSubDomains; preload');

    // 기타 보안 헤더
    res.setHeader('X-Content-Type-Options', 'nosniff');
    res.setHeader('X-Frame-Options', 'DENY');
    res.setHeader('X-XSS-Protection', '1; mode=block');
    res.setHeader('Content-Security-Policy',
        "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'");

    // 요청 처리...
});

server.listen(443, () => {
    console.log('HTTPS 서버 시작 (TLS 1.3)');
});

6.1.2 데이터 암호화

데이터 상태 암호화 방식 키 길이 보안 수준
전송 중 (In Transit) TLS 1.3 with AES-256-GCM 256-bit Critical
저장 중 (At Rest) AES-256-GCM 256-bit Critical
데이터베이스 필드 레벨 암호화 (AES-256) 256-bit High
백업 AES-256-GCM + GPG 256-bit / 4096-bit RSA Critical
로그 파일 AES-256-CBC (민감 데이터만) 256-bit Medium 
// 민감 데이터 암호화 예시 (Node.js)
const crypto = require('crypto');

class DataEncryption {
    constructor(masterKey) {
        // 마스터 키는 환경 변수나 KMS에서 안전하게 로드
        this.algorithm = 'aes-256-gcm';
        this.masterKey = Buffer.from(masterKey, 'hex');
    }

    // 데이터 암호화
    encrypt(plaintext) {
        // 고유한 IV (Initialization Vector) 생성
        const iv = crypto.randomBytes(16);

        // 암호화 객체 생성
        const cipher = crypto.createCipheriv(
            this.algorithm,
            this.masterKey,
            iv
        );

        // 데이터 암호화
        let encrypted = cipher.update(plaintext, 'utf8', 'hex');
        encrypted += cipher.final('hex');

        // 인증 태그 추가 (데이터 무결성 보장)
        const authTag = cipher.getAuthTag();

        // IV, 암호문, 인증 태그를 함께 반환
        return {
            iv: iv.toString('hex'),
            encryptedData: encrypted,
            authTag: authTag.toString('hex')
        };
    }

    // 데이터 복호화
    decrypt(encryptedObj) {
        const decipher = crypto.createDecipheriv(
            this.algorithm,
            this.masterKey,
            Buffer.from(encryptedObj.iv, 'hex')
        );

        // 인증 태그 설정
        decipher.setAuthTag(Buffer.from(encryptedObj.authTag, 'hex'));

        // 복호화
        let decrypted = decipher.update(encryptedObj.encryptedData, 'hex', 'utf8');
        decrypted += decipher.final('utf8');

        return decrypted;
    }

    // 비밀번호 해싱 (bcrypt 사용)
    async hashPassword(password) {
        const bcrypt = require('bcrypt');
        const saltRounds = 12; // 계산 비용 (높을수록 안전하지만 느림)
        return await bcrypt.hash(password, saltRounds);
    }

    // 비밀번호 검증
    async verifyPassword(password, hash) {
        const bcrypt = require('bcrypt');
        return await bcrypt.compare(password, hash);
    }
}

// 사용 예시
const encryption = new DataEncryption(process.env.MASTER_ENCRYPTION_KEY);

// 민감 정보 암호화 (예: 주민등록번호, 주소)
const sensitiveData = '123-45-67890';
const encrypted = encryption.encrypt(sensitiveData);

// 데이터베이스에 저장
await db.memorials.update({
    id: memorialId,
    deceased: {
        name: '홍길동',
        ssn: encrypted.encryptedData,  // 암호화된 데이터
        ssn_iv: encrypted.iv,
        ssn_tag: encrypted.authTag
    }
});

// 필요 시 복호화
const decrypted = encryption.decrypt({
    encryptedData: record.deceased.ssn,
    iv: record.deceased.ssn_iv,
    authTag: record.deceased.ssn_tag
});

6.1.3 키 관리 (Key Management)

키 관리 모범 사례

  • 키 회전: 암호화 키는 최소 90일마다 교체되어야 합니다.
  • 키 분리: 개발, 스테이징, 프로덕션 환경은 각각 별도의 키를 사용합니다.
  • 하드웨어 보안 모듈(HSM): 중요한 키는 FIPS 140-2 Level 2 이상 인증 HSM에 저장합니다.
  • 키 백업: 키는 안전하게 암호화되어 지리적으로 분산된 위치에 백업됩니다.
  • 접근 제어: 키에 대한 접근은 최소 권한 원칙(Principle of Least Privilege)에 따라 엄격히 제한됩니다.
// AWS KMS를 사용한 키 관리 예시
const AWS = require('aws-sdk');
const kms = new AWS.KMS({ region: 'us-east-1' });

class KeyManagementService {
    constructor(cmkId) {
        this.cmkId = cmkId; // Customer Master Key ID
    }

    // 데이터 키 생성
    async generateDataKey() {
        const params = {
            KeyId: this.cmkId,
            KeySpec: 'AES_256'
        };

        const result = await kms.generateDataKey(params).promise();

        return {
            plaintextKey: result.Plaintext,
            encryptedKey: result.CiphertextBlob
        };
    }

    // 암호화된 데이터 키 복호화
    async decryptDataKey(encryptedKey) {
        const params = {
            CiphertextBlob: encryptedKey
        };

        const result = await kms.decrypt(params).promise();
        return result.Plaintext;
    }

    // 키 회전 수행
    async rotateKey() {
        const params = {
            KeyId: this.cmkId
        };

        await kms.enableKeyRotation(params).promise();
        console.log('자동 키 회전이 활성화되었습니다.');
    }
}

// 사용 예시
const kmsService = new KeyManagementService(
    'arn:aws:kms:us-east-1:123456789012:key/12345678-1234-1234-1234-123456789012'
);

// 새 데이터 키 생성
const { plaintextKey, encryptedKey } = await kmsService.generateDataKey();

// 평문 키로 데이터 암호화
const encryptedData = encryptDataWithKey(plaintextKey, sensitiveData);

// 암호화된 키와 데이터를 함께 저장
await db.save({
    data: encryptedData,
    dataKey: encryptedKey.toString('base64')
});

// 평문 키는 메모리에서 즉시 삭제
plaintextKey.fill(0);

6.2.2 동의 관리 (Consent Management)

// 개인정보 동의 관리 시스템
class ConsentManager {
    constructor(db) {
        this.db = db;
    }

    // 동의 기록
    async recordConsent(userId, consentData) {
        const consent = {
            userId: userId,
            timestamp: new Date().toISOString(),
            ipAddress: consentData.ipAddress,
            userAgent: consentData.userAgent,
            purposes: {
                essential: true,  // 필수: 서비스 제공
                marketing: consentData.marketing || false,
                analytics: consentData.analytics || false,
                thirdParty: consentData.thirdParty || false
            },
            version: '1.0',  // 개인정보 처리방침 버전
            method: consentData.method || 'web_form',
            explicitConsent: true,  // 명시적 동의 여부
            withdrawable: true  // 철회 가능 여부
        };

        // 감사를 위해 영구 보관
        await this.db.consents.insert(consent);

        // 사용자 프로필에도 최신 동의 상태 기록
        await this.db.users.update({
            id: userId,
            consent: consent
        });

        return consent;
    }

    // 동의 철회
    async withdrawConsent(userId, purposes) {
        const withdrawal = {
            userId: userId,
            timestamp: new Date().toISOString(),
            withdrawnPurposes: purposes,
            reason: '사용자 요청'
        };

        await this.db.consentWithdrawals.insert(withdrawal);

        // 동의 철회에 따른 데이터 처리 중단
        if (purposes.includes('marketing')) {
            await this.stopMarketingCommunications(userId);
        }

        if (purposes.includes('analytics')) {
            await this.anonymizeAnalyticsData(userId);
        }

        return withdrawal;
    }

    // 동의 상태 조회
    async getConsentStatus(userId) {
        const user = await this.db.users.findOne({ id: userId });
        return user.consent;
    }

    // GDPR 요구사항: 동의 입증
    async proveConsent(userId) {
        const consents = await this.db.consents.find({
            userId: userId
        }).sort({ timestamp: -1 });

        return {
            user: userId,
            consentHistory: consents,
            currentStatus: consents[0],
            isCompliant: this.validateConsent(consents[0])
        };
    }

    // 동의 유효성 검증
    validateConsent(consent) {
        // 명시적 동의 확인
        if (!consent.explicitConsent) return false;

        // 자유롭게 제공된 동의인지 확인
        if (consent.purposes.essential &&
            !consent.purposes.marketing &&
            consent.forced) {
            return false; // 필수가 아닌 것을 강제한 경우
        }

        // 구체적이고 명확한 동의인지 확인
        if (!consent.purposes || Object.keys(consent.purposes).length === 0) {
            return false;
        }

        return true;
    }

    // 동의서 갱신 (개인정보 처리방침 변경 시)
    async requestConsentUpdate(userId, newPolicyVersion) {
        await this.db.pendingConsentUpdates.insert({
            userId: userId,
            newVersion: newPolicyVersion,
            requestedAt: new Date().toISOString(),
            status: 'pending'
        });

        // 사용자에게 알림 발송
        await this.notifyConsentUpdateRequired(userId, newPolicyVersion);
    }
}

// 사용 예시
const consentManager = new ConsentManager(db);

// 회원가입 시 동의 기록
app.post('/api/register', async (req, res) => {
    const { userData, consent } = req.body;

    // 사용자 생성
    const user = await createUser(userData);

    // 동의 기록
    await consentManager.recordConsent(user.id, {
        marketing: consent.marketing,
        analytics: consent.analytics,
        thirdParty: consent.thirdParty,
        ipAddress: req.ip,
        userAgent: req.headers['user-agent'],
        method: 'web_form'
    });

    res.json({ success: true, userId: user.id });
});

// 마케팅 동의 철회
app.post('/api/consent/withdraw', async (req, res) => {
    const { userId, purposes } = req.body;

    await consentManager.withdrawConsent(userId, purposes);

    res.json({ success: true, message: '동의가 철회되었습니다.' });
});

6.2.3 개인정보 주체의 권리

GDPR이 보장하는 8가지 권리

  • 정보에 대한 권리 (Right to be Informed): 어떤 정보가 어떻게 사용되는지 알 권리
  • 접근권 (Right of Access): 자신의 개인정보를 열람할 권리
  • 정정권 (Right to Rectification): 부정확한 정보를 수정할 권리
  • 삭제권 (Right to Erasure, '잊힐 권리'): 특정 조건에서 정보 삭제를 요구할 권리
  • 처리 제한권 (Right to Restrict Processing): 특정 조건에서 처리를 제한할 권리
  • 데이터 이동권 (Right to Data Portability): 정보를 다른 서비스로 이동할 권리
  • 반대권 (Right to Object): 특정 처리에 반대할 권리
  • 자동화된 결정에 대한 권리: 자동화된 의사결정(AI 등)의 대상이 되지 않을 권리

6.3 접근성 (Accessibility)

弘益人間의 정신에 따라, 디지털 장례 서비스는 장애 여부, 나이, 기술 숙련도에 관계없이 모든 사람이 쉽게 이용할 수 있어야 합니다. WIA-DIGITAL_FUNERAL 표준은 WCAG 2.1 Level AA 이상의 접근성 기준을 준수하며, 다양한 보조 기술과의 호환성을 보장합니다.

6.4 품질 보증 (Quality Assurance)

디지털 장례 서비스의 품질은 기술적 완성도뿐 아니라 사용자 경험, 안정성, 성능 등 다양한 측면에서 평가되어야 합니다. WIA-DIGITAL_FUNERAL 표준은 포괄적인 품질 보증 프로세스를 정의합니다.

🎯 핵심 사항 요약

  • 다층 보안 아키텍처: TLS 1.3 전송 암호화, AES-256 저장 암호화, HSM 키 관리 등 산업 최고 수준의 보안을 구현합니다.
  • GDPR 완전 준수: 적법성, 목적 제한, 데이터 최소화 등 7가지 개인정보 처리 원칙을 철저히 준수하며, 동의 관리 시스템을 통해 투명성을 보장합니다.
  • 개인정보 주체 권리 보장: 접근권, 삭제권, 데이터 이동권 등 8가지 권리를 API로 구현하여 사용자가 자신의 정보를 완전히 통제할 수 있도록 합니다.
  • 보편적 접근성: WCAG 2.1 AA 기준을 준수하여 장애인, 고령자 등 모든 사람이 디지털 장례 서비스를 이용할 수 있도록 합니다.
  • 다국어 및 문화적 다양성: 최소 6개 언어 지원, RTL 언어 지원, 지역별 날짜/통화 형식 등 국제적 서비스를 위한 기반을 제공합니다.
  • 엄격한 품질 기준: 단위 테스트 80% 커버리지, 99.9% 가동 시간, p95 응답시간 500ms 미만 등 명확한 품질 목표를 설정합니다.
  • 지속적 개선: 정기적인 보안 감사, 성능 모니터링, 사용자 피드백을 통해 서비스 품질을 지속적으로 향상시킵니다.

📝 복습 질문

  1. TLS 1.3과 TLS 1.2의 주요 차이점을 설명하고, 디지털 장례 서비스에서 TLS 1.3 사용을 필수로 하는 이유를 보안 관점에서 논의하십시오.
  2. GDPR의 7가지 개인정보 처리 원칙을 각각 설명하고, 디지털 장례 서비스에서 각 원칙을 준수하기 위한 구체적인 기술적/운영적 조치를 제안하십시오.
  3. '잊힐 권리'(Right to Erasure)를 구현할 때 직면하는 기술적 도전과제를 분석하고, 추모관 콘텐츠의 영구성 요구와 개인정보 삭제 권리 사이의 균형을 어떻게 맞출 수 있는지 논의하십시오.
  4. WCAG 2.1의 4가지 원칙(인식 가능, 작동 가능, 이해 가능, 견고성)을 설명하고, 각 원칙이 디지털 장례 서비스 사용자 경험에 어떤 영향을 미치는지 구체적인 예시와 함께 분석하십시오.
  5. 키 관리에서 '키 회전'(Key Rotation)이 중요한 이유를 설명하고, 90일마다 키를 교체할 때 서비스 중단 없이 안전하게 전환하는 방법을 제안하십시오.
  6. 99.9% 가동 시간 SLA가 의미하는 바를 계산하고(월간 허용 다운타임), 이를 달성하기 위한 시스템 아키텍처 전략(이중화, 장애 조치 등)을 논의하십시오.
