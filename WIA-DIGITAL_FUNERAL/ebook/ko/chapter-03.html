WIA-DIGITAL_FUNERAL 제3장 v1.0.0

제3장: WIA-DIGITAL_FUNERAL 표준 개요

弘益人間 (홍익인간)

"널리 인간을 이롭게 하라"

표준화된 디지털 장례 서비스로 전 인류가 존엄하게 추모받을 수 있습니다

3.1 표준의 목적과 비전

WIA-DIGITAL_FUNERAL 표준은 전 세계 디지털 장례 서비스 산업에 일관된 기준과 모범 사례를 제공하기 위해 개발되었습니다. 디지털 장례 서비스는 급속히 성장하고 있지만, 서비스 제공자마다 다른 기술, 데이터 형식, 보안 수준을 사용하면서 상호운용성과 품질 보증에 어려움이 있었습니다. WIA-DIGITAL_FUNERAL 표준은 이러한 문제를 해결하고, 모든 인류가 기술의 혜택을 통해 존엄하게 추모받을 수 있는 환경을 조성하는 것을 목표로 합니다.

이 표준의 궁극적인 비전은 弘益人間(홍익인간)의 정신에 따라, 디지털 장례 서비스가 소수의 특권층만 누리는 혁신이 아니라 모든 사람에게 열려 있는 보편적 서비스가 되도록 하는 것입니다. 지리적 위치, 경제적 능력, 기술적 숙련도에 관계없이 누구나 사랑하는 이를 의미 있게 추모하고, 고인의 기억을 영원히 보존할 수 있어야 합니다.

3.1.1 표준의 핵심 원칙

원칙 설명 구현 요구사항 측정 지표
존엄성 (Dignity) 기술이 추모의 엄숙함과 존경을 높여야 함 고인에 대한 존중, 감정적 고려, 문화적 적절성 사용자 만족도, 문화 전문가 리뷰
접근성 (Accessibility) 기술 능력과 관계없이 모두가 사용 가능 WCAG 2.1 AA 준수, 다국어 지원, 단순한 UX 접근성 점수, 고령층 사용률
개인정보 보호 (Privacy) 민감한 정보를 최고 수준으로 보호 암호화, 접근 제어, GDPR/CCPA 준수 보안 감사 결과, 침해 사고 수
문화적 감수성 (Cultural Sensitivity) 다양한 전통과 신앙을 존중 커스터마이징 옵션, 문화별 템플릿 문화 다양성 지수, 맞춤화 사용률
지속가능성 (Sustainability) 장기적 데이터 보존과 서비스 지속성 표준 데이터 형식, 다중 백업, 이관 계획 데이터 보존율, 서비스 가동 시간
상호운용성 (Interoperability) 플랫폼 간 데이터 교환 가능 표준 API, 개방형 데이터 형식 API 호환성 테스트, 데이터 이식 성공률

3.1.2 표준 개발 과정

WIA-DIGITAL_FUNERAL 표준은 2년간의 연구와 협력을 통해 개발되었습니다. 전 세계 30개국 이상에서 장례 업계 전문가, 기술 개발자, 윤리학자, 종교 지도자, 법률 전문가, 그리고 유족들이 참여하여 다양한 관점과 요구사항을 반영했습니다. 특히 COVID-19 팬데믹 기간 동안 디지털 장례 서비스를 실제로 사용한 수천 명의 가족들로부터 피드백을 수집하여, 현장의 실제 요구를 표준에 담았습니다.

표준 개발 참여자

  • 장례 업계: 전통 장례식장, 디지털 장례 스타트업, 묘지 관리 기관
  • 기술 전문가: 소프트웨어 엔지니어, 보안 전문가, UX 디자이너
  • 학계: 사회학자, 심리학자, 인류학자, 윤리학자
  • 종교계: 불교, 기독교, 이슬람, 유대교, 힌두교 등 주요 종교 대표
  • 규제 당국: 개인정보 보호위원회, 의료 규제 기관
  • 사용자: 디지털 장례 경험자, 고령자 대표, 장애인 권익 단체

3.2 표준의 구조와 범위

WIA-DIGITAL_FUNERAL 표준은 4단계(PHASE) 구조로 구성되어 있으며, 각 단계는 디지털 장례 서비스의 특정 측면을 다룹니다. 이러한 단계적 접근 방식은 조직이 자신의 역량과 요구에 맞춰 점진적으로 표준을 구현할 수 있도록 설계되었습니다. 모든 단계를 한 번에 구현할 필요는 없으며, 우선순위에 따라 선택적으로 적용할 수 있습니다.

하지만 상호운용성을 위해서는 최소한 PHASE 1(데이터 형식)은 반드시 준수해야 합니다. PHASE 1을 준수하면 다른 표준 준수 플랫폼과 데이터를 교환할 수 있고, 향후 서비스 제공자를 변경할 때도 데이터 손실 없이 이관할 수 있습니다.

3.2.1 4단계 (4 PHASE) 개요

단계 명칭 핵심 내용 필수/선택 적용 대상
PHASE 1 데이터 형식 (Data Format) 추모 데이터 구조, JSON 스키마, 미디어 형식, 메타데이터 필수 모든 서비스 제공자
PHASE 2 API 인터페이스 (API Interface) RESTful API, GraphQL, 인증/권한, 이벤트 시스템 권장 통합을 제공하는 플랫폼
PHASE 3 프로토콜 (Protocol) 보안, 개인정보 보호, 접근성, 품질 보증, 감사 권장 규모 있는 서비스 제공자
PHASE 4 통합 (Integration) 레거시 시스템 통합, 타 표준 연동, 확장 메커니즘 선택 기존 시스템이 있는 조직

3.3 PHASE 1: 데이터 형식

PHASE 1 - 데이터 형식 상세

목적: 추모 정보의 표준화된 구조를 정의하여 플랫폼 간 데이터 교환 및 장기 보존을 가능하게 합니다.

필수 준수: 상호운용성을 위해 모든 WIA-DIGITAL_FUNERAL 준수 시스템은 이 단계를 반드시 구현해야 합니다.

PHASE 1은 디지털 장례 서비스에서 사용되는 모든 데이터의 구조와 형식을 정의합니다. 이는 고인 정보, 추모 콘텐츠, 사용자 데이터, 이벤트 정보 등을 포함합니다. 표준화된 데이터 형식을 사용하면 서로 다른 플랫폼 간에 데이터를 쉽게 교환할 수 있고, 10년, 20년 후에도 데이터를 정확하게 읽고 해석할 수 있습니다.

3.3.1 핵심 데이터 엔티티

엔티티 설명 필수 필드 선택 필드
Memorial 추모관의 최상위 객체 id, deceased, createdAt, standard customDomain, settings, features
Deceased 고인의 기본 정보 fullName, deathDate birthDate, biography, photos, epitaph
Tribute 추모 메시지와 헌화 id, author, message, postedAt media, visibility, moderationStatus
Ceremony 장례식 또는 추모식 정보 id, type, dateTime, location streamUrl, recording, attendees
Media 사진, 동영상, 음성 파일 id, type, url, uploadedAt caption, date, tags, metadata
User 추모관 관리자 및 기여자 id, name, email, role relationship, permissions, contactInfo

3.4 PHASE 2: API 인터페이스

PHASE 2 - API 인터페이스 상세

목적: 표준화된 API를 통해 외부 시스템이 디지털 장례 서비스와 통합할 수 있도록 합니다.

권장 구현: 타 시스템과의 통합을 제공하려는 플랫폼은 이 단계를 구현해야 합니다.

PHASE 2는 디지털 장례 서비스에 접근하기 위한 표준 API 인터페이스를 정의합니다. 이를 통해 장례식장 관리 시스템, 웹사이트 빌더, 모바일 앱, 타사 서비스 등이 표준화된 방식으로 추모관을 생성하고, 데이터를 조회하며, 업데이트할 수 있습니다. RESTful API와 GraphQL 두 가지 API 스타일을 모두 지원하여 개발자의 선호도와 사용 사례에 맞게 선택할 수 있습니다.

3.4.1 RESTful API 엔드포인트

엔드포인트 메서드 설명 인증
/api/v1/memorials POST 새 추모관 생성 API Key + OAuth 2.0
/api/v1/memorials/:id GET 추모관 정보 조회 선택 (공개 설정 따름)
/api/v1/memorials/:id PATCH 추모관 정보 수정 Owner/Admin 권한
/api/v1/memorials/:id/tributes POST 추모 메시지 제출 선택 (설정 따름)
/api/v1/memorials/:id/media POST 사진/동영상 업로드 인증된 사용자
/api/v1/ceremonies POST 장례식 일정 생성 API Key + OAuth 2.0
/api/v1/livestream/:id/start POST 라이브스트림 시작 Admin 권한

3.5 PHASE 3: 프로토콜

PHASE 3 - 프로토콜 상세

목적: 보안, 개인정보 보호, 접근성, 품질 보증 등 비기능적 요구사항을 정의합니다.

권장 구현: 전문적이고 규모 있는 서비스를 제공하는 조직은 이 단계를 구현해야 합니다.

PHASE 3은 기술적 구현을 넘어선 운영 및 거버넌스 측면의 요구사항을 다룹니다. 여기에는 데이터 보안, 개인정보 보호 규정 준수, 웹 접근성, 서비스 품질 보증, 감사 및 컴플라이언스 등이 포함됩니다. 이 단계를 준수하면 사용자의 신뢰를 얻고, 법적 리스크를 줄이며, 장기적으로 지속가능한 서비스를 제공할 수 있습니다.

3.6 PHASE 4: 통합

PHASE 4 - 통합 상세

목적: 기존 레거시 시스템 및 타 표준과의 통합 방법을 제공합니다.

선택적 구현: 기존 시스템이 있는 조직이나 다른 서비스와 연동이 필요한 경우 구현합니다.
