Chapter 5

실시간 처리 및 API

5.1 개요

Kafka, Flink, REST API를 사용하여 100ms 이하의 지연시간으로 실시간 사기 탐지 API를 구축하는 방법을 배웁니다. 이 장에서는 스트림 처리, 배치 분석, 웹훅, 초당 10,000개 이상의 거래로 확장하는 방법을 다룹니다.

5.2 주요 내용

  • Kafka 아키텍처: 토픽, 파티션, 소비자 그룹
  • Flink 스트림 처리: 윈도우 집계, 상태 저장 처리
  • REST API 설계: POST /fraud/analyze, GET /fraud/reports
  • 배치 처리: 단일 요청에서 최대 100개 거래 분석
  • 웹훅: 실시간 사기 알림
  • 속도 제한: API 남용 방지 (600 req/min)
  • 캐싱 전략: 다중 수준 캐싱 (L1: 앱, L2: Redis, L3: DB)

5.3 핵심 개념

실시간 처리는 세 가지 수준에서 작동합니다: (1) 데이터 수집 - Kafka는 다양한 소스(REST API, gRPC, 메시지 큐)에서 거래를 수집합니다. (2) 스트림 처리 - Flink는 특성을 계산하고 집계를 수행합니다(윈도우 카운트, 속도, 통계). (3) 추론 - ML 모델은 30ms 이내에 사기 점수를 예측합니다. 전체 파이프라인은 100ms(p95) 미만의 지연시간을 달성하여 결제 승인 흐름에서 인라인 사용을 가능하게 합니다.

핵심 요약

Kafka는 파티션당 10,000+ TPS로 이벤트 스트리밍을 제공합니다. Flink는 윈도우 집계와 상태 저장 처리를 수행합니다. REST API는 100ms(p95) 미만의 지연시간으로 실시간 사기 점수를 제공합니다. 배치 처리는 단일 요청에서 최대 100개 거래를 분석합니다. 웹훅은 실시간 사기 알림을 제공합니다. 다중 수준 캐싱(L1: 앱, L2: Redis, L3: DB)은 성능을 최적화합니다. 속도 제한(600 req/min)은 API 남용을 방지합니다.
