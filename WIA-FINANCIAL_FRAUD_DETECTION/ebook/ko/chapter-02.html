시스템 아키텍처 및 설계

확장 가능한 실시간 사기 탐지 시스템 구축

2.1 아키텍처 개요

프로덕션 사기 탐지 시스템은 하루 수백만 건의 트랜잭션을 100ms 미만의 지연 시간으로 처리하면서 99.99%의 가동 시간을 유지하기 위해 신중한 아키텍처 설계가 필요합니다. 이 장에서는 이러한 시스템을 구축하기 위한 아키텍처 패턴, 기술 선택 및 설계 원칙을 다룹니다.

2.2 핵심 구성 요소

2.2.1 트랜잭션 수집기

목적: 여러 소스에서 실시간으로 트랜잭션 데이터를 수집하고 전달을 보장합니다.

  • 다중 프로토콜 지원: REST API, gRPC, GraphQL, 메시지 큐
  • 스키마 검증 및 변환
  • 멱등성 키를 사용한 중복 제거
  • 최소 한 번 전달 의미론

2.2.2 특성 엔지니어링 파이프라인

원시 트랜잭션 데이터를 80개 이상의 ML 준비 특성으로 변환합니다:

  • 트랜잭션 특성 (15개): 금액, 통화, MCC, 시간대, 속도
  • 사용자 행동 특성 (25개): 과거 지출 패턴, 가맹점 선호도, 지리적 분포
  • 디바이스/네트워크 특성 (20개): 디바이스 지문, IP 지리적 위치, VPN 탐지
  • 그래프 특성 (10개): 소셜 네트워크 분석, 엔티티 해결
  • 외부 데이터 특성 (10개): 이메일 평판, 전화번호 검증, 주소 확인

2.2.3 ML 모델 앙상블

앙상블 전략: 여러 모델에 걸친 가중 투표

Fraud_Score = 0.40 × XGBoost
            + 0.30 × Random_Forest
            + 0.15 × DNN
            + 0.10 × Isolation_Forest
            + 0.05 × Autoencoder

2.2.4 규칙 엔진

Drools를 사용한 결정론적 사기 검사:

  • 블랙리스트/화이트리스트 규칙: 차단된 국가, IP, 이메일 도메인
  • 속도 규칙: 시간 윈도우당 최대 트랜잭션 수
  • 지리적 위치 규칙: 불가능한 이동 탐지
  • 행동 규칙: 첫 트랜잭션 임계값, 주소 불일치

2.3 기술 스택

2.3.1 스트림 처리

  • Apache Kafka: 이벤트 스트리밍 플랫폼 (파티션당 10K+ TPS)
  • Apache Flink: 상태 저장 스트림 처리 (윈도우 집계, 조인)
  • Kafka Streams: 경량 스트림 처리 (간단한 사용 사례용)

2.3.2 데이터 저장소

  • 트랜잭션: PostgreSQL 15 (Aurora), DynamoDB
  • 캐싱: Redis (특성 저장소), Elasticache
  • 시계열: InfluxDB, TimescaleDB
  • 그래프: Neo4j (네트워크 분석)
  • 데이터 레이크: S3, GCS (모델 학습용 과거 데이터)

2.3.3 ML 인프라

  • 학습: Python (XGBoost, scikit-learn, TensorFlow)
  • 특성 저장소: Feast, Tecton
  • 모델 서빙: TensorFlow Serving, Seldon Core
  • 실험 추적: MLflow, Weights & Biases

2.4 확장성 및 성능

2.4.1 수평 확장

Kubernetes 자동 확장 (HPA):

  • 최소 레플리카: 3, 최대 레플리카: 50
  • CPU 임계값: 70%, 메모리 임계값: 80%
  • 사용자 정의 메트릭: 요청 지연 시간 p95 < 100ms
  • 확장 정책: 분당 50%
  • 축소 대기 시간: 5분

2.4.2 데이터베이스 샤딩

샤드 키: customer_id (일관된 해싱)

def get_shard_id(customer_id: str, num_shards: int = 8) -> int:
    hash_value = int(hashlib.md5(customer_id.encode()).hexdigest(), 16)
    return hash_value % num_shards

2.4.3 캐싱 전략

다단계 캐싱:

  • L1 (애플리케이션): 인메모리 캐시 (60초 TTL)
  • L2 (분산): Redis (사용자 프로필에 대해 15분 TTL)
  • L3 (데이터베이스): 인덱스가 있는 PostgreSQL

2.5 고가용성

2.5.1 다중 리전 배포

  • 주 리전: us-east-1 (활성)
  • 보조 리전: eu-west-1 (활성)
  • 장애 조치: Route 53 지연 시간 기반 라우팅 + 상태 확인
  • RTO: < 5분
  • RPO: < 1분 (Aurora 지속적 복제)

2.5.2 회로 차단기 패턴

@circuit(failure_threshold=5, recovery_timeout=60)
def call_external_api(url, data):
    response = requests.post(url, json=data, timeout=2)
    response.raise_for_status()
    return response.json()

2.6 배포 패턴

2.6.1 블루-그린 배포

즉시 롤백이 가능한 주요 버전 업그레이드:

  • 기존 버전(블루)과 함께 새 버전(그린) 배포
  • 그린 환경에서 스모크 테스트 실행
  • 블루에서 그린으로 트래픽 전환 (Kubernetes Service를 통해)
  • 잠재적인 롤백을 위해 블루를 1시간 동안 실행 상태 유지

2.6.2 카나리 배포

위험 완화를 통한 점진적 롤아웃:

  • 0시간: 안정 100%, 카나리 0%
  • 1시간: 안정 95%, 카나리 5%
  • 4시간: 안정 75%, 카나리 25%
  • 8시간: 안정 50%, 카나리 50%
  • 16시간: 안정 0%, 카나리 100% (완전한 롤아웃)

핵심 요약

  • Kubernetes를 사용한 마이크로서비스 아키텍처로 10K+ TPS까지 수평 확장 가능
  • Kafka + Flink는 100ms 미만의 지연 시간으로 실시간 스트림 처리 제공
  • 특성 저장소(Feast)는 온라인/오프라인 시나리오를 위한 특성 관리를 중앙 집중화
  • Route 53을 사용한 다중 리전 배포로 99.99% 가동 시간 보장 (RTO < 5분)
  • customer_id에 의한 데이터베이스 샤딩으로 선형 확장성 제공
  • 카나리 배포로 잘못된 릴리스의 위험 감소
