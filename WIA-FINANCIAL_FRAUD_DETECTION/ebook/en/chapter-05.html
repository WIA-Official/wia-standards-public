Chapter 5

Real-Time Processing & APIs

5.1 Overview

Learn how to build real-time fraud detection APIs with sub-100ms latency using Kafka, Flink, and REST APIs. This chapter covers stream processing, batch analysis, webhooks, and scaling to 10,000+ transactions per second.

5.2 Key Topics

  • Kafka Architecture: Topics, partitions, consumer groups
  • Flink Stream Processing: Windowed aggregations, stateful processing
  • REST API Design: POST /fraud/analyze, GET /fraud/reports
  • Batch Processing: Analyze up to 100 transactions in single request
  • Webhooks: Real-time fraud alerts and notifications
  • Rate Limiting: Protect API from abuse (600 req/min)
  • Caching Strategy: Multi-level caching (L1: app, L2: Redis, L3: DB)
  • Performance Optimization: Achieve sub-100ms p95 latency

5.3 Implementation Example

# Real-time fraud detection API endpoint
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel

app = FastAPI()

class Transaction(BaseModel):
    id: str
    amount: float
    customer_id: str
    # ... other fields

@app.post("/fraud/analyze")
async def analyze_transaction(transaction: Transaction):
    # Extract features
    features = extract_features(transaction)
    
    # Get fraud score from model
    fraud_score = model.predict_proba([features])[0][1]
    
    # Make decision
    if fraud_score >= 0.90:
        decision = "block"
    elif fraud_score >= 0.70:
        decision = "review"
    elif fraud_score >= 0.50:
        decision = "challenge"
    else:
        decision = "approve"
    
    return {
        "transaction_id": transaction.id,
        "fraud_score": float(fraud_score),
        "decision": decision
    }

5.4 Best Practices

  • Use async/await for non-blocking I/O operations
  • Implement circuit breakers for external API calls
  • Cache frequently accessed data (user profiles, device fingerprints)
  • Use connection pooling for database connections
  • Implement rate limiting to prevent abuse
  • Monitor API latency (p50, p95, p99) and set alerts
