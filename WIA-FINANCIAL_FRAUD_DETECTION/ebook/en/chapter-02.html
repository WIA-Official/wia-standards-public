System Architecture & Design

Building scalable, real-time fraud detection systems

2.1 Architecture Overview

A production fraud detection system requires careful architectural design to handle millions of transactions per day with sub-100ms latency while maintaining 99.99% uptime. This chapter covers the architecture patterns, technology choices, and design principles for building such systems.

2.2 Core Components

2.2.1 Transaction Collector

Purpose: Ingest transaction data from multiple sources in real-time with guaranteed delivery.

  • Multi-protocol support: REST API, gRPC, GraphQL, message queues
  • Schema validation and transformation
  • Deduplication using idempotency keys
  • At-least-once delivery semantics

2.2.2 Feature Engineering Pipeline

Transforms raw transaction data into 80+ ML-ready features:

  • Transaction Features (15): Amount, currency, MCC, time of day, velocity
  • User Behavioral Features (25): Historical spending patterns, merchant affinity, geographic distribution
  • Device/Network Features (20): Device fingerprint, IP geolocation, VPN detection
  • Graph Features (10): Social network analysis, entity resolution
  • External Data Features (10): Email reputation, phone validation, address verification

2.2.3 ML Model Ensemble

Ensemble Strategy: Weighted voting across multiple models

Fraud_Score = 0.40 × XGBoost
            + 0.30 × Random_Forest
            + 0.15 × DNN
            + 0.10 × Isolation_Forest
            + 0.05 × Autoencoder

2.2.4 Rules Engine

Deterministic fraud checks using Drools:

  • Blacklist/Whitelist Rules: Blocked countries, IPs, email domains
  • Velocity Rules: Max transactions per time window
  • Geolocation Rules: Impossible travel detection
  • Behavioral Rules: First transaction thresholds, address mismatches

2.3 Technology Stack

2.3.1 Stream Processing

  • Apache Kafka: Event streaming platform (10K+ TPS per partition)
  • Apache Flink: Stateful stream processing (windowed aggregations, joins)
  • Kafka Streams: Lightweight stream processing (for simpler use cases)

2.3.2 Data Storage

  • Transactional: PostgreSQL 15 (Aurora), DynamoDB
  • Caching: Redis (feature store), Elasticache
  • Time-Series: InfluxDB, TimescaleDB
  • Graph: Neo4j (network analysis)
  • Data Lake: S3, GCS (historical data for model training)

2.3.3 ML Infrastructure

  • Training: Python (XGBoost, scikit-learn, TensorFlow)
  • Feature Store: Feast, Tecton
  • Model Serving: TensorFlow Serving, Seldon Core
  • Experiment Tracking: MLflow, Weights & Biases

2.4 Scalability & Performance

2.4.1 Horizontal Scaling

Kubernetes Auto-Scaling (HPA):

  • Min replicas: 3, Max replicas: 50
  • CPU threshold: 70%, Memory threshold: 80%
  • Custom metrics: Request latency p95 < 100ms
  • Scale-up policy: 50% per minute
  • Scale-down cooldown: 5 minutes

2.4.2 Database Sharding

Shard Key: customer_id (consistent hashing)

def get_shard_id(customer_id: str, num_shards: int = 8) -> int:
    hash_value = int(hashlib.md5(customer_id.encode()).hexdigest(), 16)
    return hash_value % num_shards

2.4.3 Caching Strategy

Multi-Level Caching:

  • L1 (Application): In-memory cache (60s TTL)
  • L2 (Distributed): Redis (15 min TTL for user profiles)
  • L3 (Database): PostgreSQL with indexes

2.5 High Availability

2.5.1 Multi-Region Deployment

  • Primary: us-east-1 (Active)
  • Secondary: eu-west-1 (Active)
  • Failover: Route 53 latency-based routing + health checks
  • RTO: < 5 minutes
  • RPO: < 1 minute (Aurora continuous replication)

2.5.2 Circuit Breaker Pattern

@circuit(failure_threshold=5, recovery_timeout=60)
def call_external_api(url, data):
    response = requests.post(url, json=data, timeout=2)
    response.raise_for_status()
    return response.json()

2.6 Deployment Patterns

2.6.1 Blue-Green Deployment

Major version upgrades with instant rollback:

  • Deploy new version (Green) alongside old version (Blue)
  • Run smoke tests on Green environment
  • Switch traffic from Blue to Green (via Kubernetes Service)
  • Keep Blue running for 1 hour for potential rollback

2.6.2 Canary Deployment

Gradual rollout with risk mitigation:

  • Hour 0: 100% stable, 0% canary
  • Hour 1: 95% stable, 5% canary
  • Hour 4: 75% stable, 25% canary
  • Hour 8: 50% stable, 50% canary
  • Hour 16: 0% stable, 100% canary (complete rollout)

Key Takeaways

  • Microservices architecture with Kubernetes enables horizontal scaling to 10K+ TPS
  • Kafka + Flink provide real-time stream processing with sub-100ms latency
  • Feature store (Feast) centralizes feature management for online/offline scenarios
  • Multi-region deployment with Route 53 ensures 99.99% uptime (RTO < 5 min)
  • Database sharding by customer_id enables linear scalability
  • Canary deployment reduces risk of bad releases
