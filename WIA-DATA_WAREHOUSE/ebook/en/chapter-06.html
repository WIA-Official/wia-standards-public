WIA-DATA_WAREHOUSE Standard - Chapter 6

Phase 3: Advanced Techniques

Introduction to Advanced Data Warehouse Techniques

Once you've mastered fundamental dimensional modeling and ETL processes, advanced techniques enable you to handle complex business scenarios, optimize for specific analytical requirements, and solve challenging data modeling problems. This chapter explores sophisticated patterns that separate basic data warehouses from enterprise-grade analytical platforms.

Slowly Changing Dimensions Deep Dive

While we introduced SCDs in Chapter 4, understanding all seven SCD types and knowing when to apply each is critical for accurate historical analysis. The choice of SCD strategy has profound implications for query complexity, storage requirements, and analytical capabilities.

Complete SCD Type Matrix

Type Strategy History Preserved Use Case
Type 0 Never change (fixed) N/A - no changes allowed Original values that must never change (e.g., birth date, original hire date)
Type 1 Overwrite No - history lost Corrections, irrelevant history (e.g., typo fixes, updated phone numbers)
Type 2 Add new row Full - all changes tracked Important historical changes (e.g., customer address moves, product price changes)
Type 3 Add new column Limited - one previous value Compare current vs. previous (e.g., current_tier vs. previous_tier for analysis)
Type 4 History table Full - in separate table Rapidly changing dimensions with many historical versions
Type 5 Type 4 + mini-dimension Full - optimized for common queries Large dimensions with subset of frequently analyzed attributes
Type 6 Hybrid (1+2+3) Full + current indicator Need both historical detail and easy access to current values
-- SCD Type 3 Example: Track current and previous customer tier CREATE TABLE dim_customer ( customer_key BIGSERIAL PRIMARY KEY, customer_id VARCHAR(50) UNIQUE NOT NULL, customer_name VARCHAR(255), -- Current tier information current_tier VARCHAR(50), current_tier_effective_date DATE, -- Previous tier information (Type 3) previous_tier VARCHAR(50), previous_tier_effective_date DATE, -- Standard attributes created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ); -- When a customer's tier changes, shift current to previous UPDATE dim_customer SET previous_tier = current_tier, previous_tier_effective_date = current_tier_effective_date, current_tier = 'Gold', current_tier_effective_date = CURRENT_DATE, updated_at = CURRENT_TIMESTAMP WHERE customer_id = 'CUST12345'; -- Easy analysis: customers who upgraded from Silver to Gold SELECT COUNT(*) FROM dim_customer WHERE previous_tier = 'Silver' AND current_tier = 'Gold';

Bridge Tables and Many-to-Many Relationships

Dimensional models traditionally handle one-to-many relationships easily: one customer places many orders, one product appears in many transactions. But what about many-to-many relationships? A patient can have multiple diagnoses, a bank account can have multiple owners, a project can have multiple team members.

Bridge tables (also called "bridging tables" or "many-to-many resolution tables") resolve these complex relationships while preserving the star schema structure.

-- Example: Patient Diagnoses (Many-to-Many) -- Dimension: Patient CREATE TABLE dim_patient ( patient_key BIGSERIAL PRIMARY KEY, patient_id VARCHAR(50) UNIQUE NOT NULL, patient_name VARCHAR(255), date_of_birth DATE, gender VARCHAR(10) ); -- Dimension: Diagnosis CREATE TABLE dim_diagnosis ( diagnosis_key BIGSERIAL PRIMARY KEY, diagnosis_code VARCHAR(20) UNIQUE NOT NULL, diagnosis_name VARCHAR(255), diagnosis_category VARCHAR(100), severity VARCHAR(50) ); -- Bridge Table: Resolves patient-to-diagnosis many-to-many CREATE TABLE bridge_patient_diagnosis ( patient_diagnosis_group_key BIGSERIAL PRIMARY KEY, patient_key BIGINT NOT NULL, diagnosis_key BIGINT NOT NULL, diagnosed_date DATE, weight_factor NUMERIC(5,4) DEFAULT 1.0, -- For weighted aggregation UNIQUE(patient_key, diagnosis_key, diagnosed_date) ); -- Fact Table: Hospital Encounters CREATE TABLE fact_encounter ( encounter_key BIGSERIAL PRIMARY KEY, patient_diagnosis_group_key BIGINT NOT NULL, -- References bridge table date_key INTEGER NOT NULL, facility_key BIGINT NOT NULL, procedure_count INTEGER, total_cost NUMERIC(12,2), length_of_stay_days INTEGER ); -- Query: Average cost per encounter for diabetes patients SELECT d.diagnosis_name, COUNT(DISTINCT f.encounter_key) as encounter_count, AVG(f.total_cost) as avg_cost_per_encounter FROM fact_encounter f JOIN bridge_patient_diagnosis b ON f.patient_diagnosis_group_key = b.patient_diagnosis_group_key JOIN dim_diagnosis d ON b.diagnosis_key = d.diagnosis_key WHERE d.diagnosis_category = 'Diabetes' GROUP BY d.diagnosis_name;

Weight Factors in Bridge Tables

When a fact has multiple dimension members (e.g., a patient with 3 diagnoses), aggregating facts can lead to over-counting. Weight factors solve this by dividing the impact proportionally. If a patient has 3 diagnoses, each gets a weight of 0.333 (1/3), ensuring that summing across all diagnoses yields the correct total.

Factless Fact Tables

Not all business events have measurable numeric facts. Factless fact tables track the occurrence of events or the coverage of relationships without numerical measurements. They answer questions like "Which students attended which classes?" or "Which products were on promotion, even if they didn't sell?"

Types of Factless Facts

1. Event-Tracking Factless Facts

-- Example: Student Class Attendance CREATE TABLE fact_class_attendance ( attendance_key BIGSERIAL PRIMARY KEY, date_key INTEGER NOT NULL, student_key BIGINT NOT NULL, class_key BIGINT NOT NULL, instructor_key BIGINT NOT NULL, -- No numeric measures - just tracking the event -- Can add optional flags was_absent BOOLEAN DEFAULT FALSE, was_tardy BOOLEAN DEFAULT FALSE, CONSTRAINT fk_date FOREIGN KEY (date_key) REFERENCES dim_date(date_key), CONSTRAINT fk_student FOREIGN KEY (student_key) REFERENCES dim_student(student_key), CONSTRAINT fk_class FOREIGN KEY (class_key) REFERENCES dim_class(class_key), CONSTRAINT fk_instructor FOREIGN KEY (instructor_key) REFERENCES dim_instructor(instructor_key) ); -- Query: How many students attended each class? SELECT c.class_name, c.semester, COUNT(*) as total_attendance_records, COUNT(*) FILTER (WHERE f.was_absent = FALSE) as present_count, COUNT(*) FILTER (WHERE f.was_absent = TRUE) as absent_count FROM fact_class_attendance f JOIN dim_class c ON f.class_key = c.class_key GROUP BY c.class_name, c.semester;

2. Coverage Factless Facts

-- Example: Product Promotion Coverage -- Tracks which products were eligible for which promotions -- Even if no sales occurred CREATE TABLE fact_promotion_coverage ( coverage_key BIGSERIAL PRIMARY KEY, date_key INTEGER NOT NULL, product_key BIGINT NOT NULL, promotion_key BIGINT NOT NULL, store_key BIGINT NOT NULL, -- No measures - tracking eligibility relationship -- Used to find what DIDN'T sell during promotions CONSTRAINT fk_date FOREIGN KEY (date_key) REFERENCES dim_date(date_key), CONSTRAINT fk_product FOREIGN KEY (product_key) REFERENCES dim_product(product_key), CONSTRAINT fk_promotion FOREIGN KEY (promotion_key) REFERENCES dim_promotion(promotion_key), CONSTRAINT fk_store FOREIGN KEY (store_key) REFERENCES dim_store(store_key) ); -- Query: Products on promotion but with zero sales (promotion failures) SELECT p.product_name, pr.promotion_name, COUNT(DISTINCT pc.date_key) as days_on_promotion FROM fact_promotion_coverage pc JOIN dim_product p ON pc.product_key = p.product_key JOIN dim_promotion pr ON pc.promotion_key = pr.promotion_key LEFT JOIN fact_sales fs ON pc.product_key = fs.product_key AND pc.date_key = fs.date_key AND pc.promotion_key = fs.promotion_key WHERE fs.sales_key IS NULL -- No sales during promotion GROUP BY p.product_name, pr.promotion_name ORDER BY days_on_promotion DESC;

Aggregate Tables and Summary Facts

Aggregate tables pre-compute summarized data at higher grains (daily, weekly, monthly instead of transaction-level). They dramatically improve query performance for common aggregations while trading storage space for speed.

Grain Level Row Count Example Storage Query Speed
Transaction Detail 1 billion rows 500 GB Slowest (full scan)
Daily Aggregate 10 million rows 5 GB 100x faster
Monthly Aggregate 300,000 rows 150 MB 1000x faster
-- Daily Sales Aggregate Table CREATE TABLE fact_sales_daily ( sales_daily_key BIGSERIAL PRIMARY KEY, date_key INTEGER NOT NULL, product_key BIGINT NOT NULL, store_key BIGINT NOT NULL, -- Aggregated measures transaction_count INTEGER, total_quantity NUMERIC(15,2), total_sales_amount NUMERIC(18,2), total_cost_amount NUMERIC(18,2), total_profit_amount NUMERIC(18,2), -- Statistical measures avg_transaction_amount NUMERIC(12,2), min_transaction_amount NUMERIC(12,2), max_transaction_amount NUMERIC(12,2), -- Unique customer count (approximate with HyperLogLog in some databases) distinct_customer_count INTEGER, updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ); -- Populate from transaction-level fact table INSERT INTO fact_sales_daily SELECT date_key, product_key, store_key, COUNT(*) as transaction_count, SUM(quantity) as total_quantity, SUM(net_sales_amount) as total_sales_amount, SUM(cost_amount) as total_cost_amount, SUM(profit_amount) as total_profit_amount, AVG(net_sales_amount) as avg_transaction_amount, MIN(net_sales_amount) as min_transaction_amount, MAX(net_sales_amount) as max_transaction_amount, COUNT(DISTINCT customer_key) as distinct_customer_count, CURRENT_TIMESTAMP FROM fact_sales WHERE date_key = 20250108 -- Yesterday's data GROUP BY date_key, product_key, store_key;

Surrogate Keys and Natural Keys

Surrogate keys are system-generated integer identifiers with no business meaning, while natural keys are business identifiers (customer ID, product SKU). The data warehousing best practice strongly favors surrogate keys for dimension primary keys.

Why Surrogate Keys?

Benefit Explanation
SCD Support Multiple dimension rows can share the same natural key with different effective dates
Performance Small integer keys are faster to join than long varchar natural keys
Integration Merge data from multiple sources with potentially conflicting natural keys
Null Handling Use special surrogate values (e.g., -1) for missing/unknown dimension members
Change Resilience If business changes natural key format, warehouse keys remain stable

Special Surrogate Key Values

Establish conventions for special dimension members:

  • -1: Unknown or missing dimension member
  • -2: Not applicable (dimension doesn't apply to this fact)
  • -3: Error / data quality issue
  • 0: Before the earliest meaningful date (for date dimensions)
  • 99999999: Future date / open-ended (for date dimensions)

Key Takeaways

  • Slowly Changing Dimensions have seven types (0-6), each suited to different business requirements. Type 2 (add new row) is most common, but understanding all types enables optimal historical tracking strategies.
  • Bridge tables resolve many-to-many relationships in dimensional models, enabling star schema simplicity while handling complex relationships like multi-diagnosis patients or multi-owner accounts. Weight factors prevent double-counting in aggregations.
  • Factless fact tables track event occurrence or relationship coverage without numeric measures. They answer questions about attendance, eligibility, and coverage, and enable powerful negative analysis (what didn't happen).
  • Aggregate tables pre-compute summaries at coarser grains (daily, monthly) to dramatically improve query performance. Trading storage for speed, they can deliver 100-1000x performance improvements for common reporting queries.
  • Surrogate keys (system-generated integers) are superior to natural keys (business identifiers) for dimension primary keys. They enable SCD, improve performance, simplify integration, and provide resilience to business key format changes.
