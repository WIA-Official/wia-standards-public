WIA-DATA_WAREHOUSE Standard - Chapter 4

Phase 1: Dimensional Modeling

Introduction to Dimensional Modeling

Dimensional modeling is the cornerstone of data warehouse design, providing a methodology for structuring data that is both intuitive for business users and optimized for analytical query performance. Pioneered by Ralph Kimball in the 1990s, dimensional modeling represents a paradigm shift from the highly normalized entity-relationship models used in transactional systems.

The fundamental concept is deceptively simple: organize data around business processes, with measurable facts surrounded by descriptive dimensions. This star-like structure (hence "star schema") mirrors how business users naturally think about their data, making reports and analyses more accessible while delivering exceptional query performance through denormalization and strategic indexing.

Fact Tables: The Core of Analysis

What Are Fact Tables?

Fact tables are the central tables in dimensional models, containing the quantitative measurements or metrics that drive business analysis. Each row in a fact table represents a business event or measurement at a specific grain (level of detail). Fact tables typically make up 90% or more of the data volume in a warehouse due to their granularity and historical accumulation.

Types of Facts

Fact Type Description Example Aggregation
Additive Can be summed across all dimensions Sales amount, quantity sold, costs SUM() works across all dimensions
Semi-Additive Can be summed across some dimensions but not others Account balance, inventory levels SUM() across products, AVG() across time
Non-Additive Cannot be meaningfully summed Ratios, percentages, unit prices Calculated from additive facts
Derived Calculated from other facts Profit (revenue - cost), margin % Formula-based computation

Fact Table Design: Grain Definition

The grain is the most critical design decision for a fact table. It defines what each row represents - the atomic level of measurement. Common grain statements include:

  • "One row per order line item" (retail sales)
  • "One row per account per day" (banking balances)
  • "One row per patient visit" (healthcare encounters)
  • "One row per web page view" (digital analytics)

Grain must be declared before designing the fact table. All facts must conform to the declared grain, and all dimensions must have a one-to-one or many-to-one relationship with the grain. Mixing grain levels within a single fact table leads to double-counting errors and analytical inconsistencies.

-- Example Fact Table: Sales Transactions CREATE TABLE fact_sales ( -- Surrogate key for the fact sales_key BIGSERIAL PRIMARY KEY, -- Foreign keys to dimensions date_key INTEGER NOT NULL, time_key INTEGER NOT NULL, product_key BIGINT NOT NULL, customer_key BIGINT NOT NULL, store_key BIGINT NOT NULL, promotion_key BIGINT, -- Additive facts quantity NUMERIC(10,2) NOT NULL, gross_sales_amount NUMERIC(12,2) NOT NULL, discount_amount NUMERIC(12,2) DEFAULT 0, tax_amount NUMERIC(12,2) NOT NULL, net_sales_amount NUMERIC(12,2) NOT NULL, cost_amount NUMERIC(12,2) NOT NULL, -- Derived facts (can be calculated, stored for performance) profit_amount NUMERIC(12,2) GENERATED ALWAYS AS (net_sales_amount - cost_amount) STORED, -- Degenerate dimensions (transaction identifiers) transaction_id VARCHAR(50) NOT NULL, -- Audit columns loaded_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, -- Foreign key constraints CONSTRAINT fk_date FOREIGN KEY (date_key) REFERENCES dim_date(date_key), CONSTRAINT fk_product FOREIGN KEY (product_key) REFERENCES dim_product(product_key), CONSTRAINT fk_customer FOREIGN KEY (customer_key) REFERENCES dim_customer(customer_key), CONSTRAINT fk_store FOREIGN KEY (store_key) REFERENCES dim_store(store_key) ); -- Indexes for query performance CREATE INDEX idx_fact_sales_date ON fact_sales(date_key); CREATE INDEX idx_fact_sales_product ON fact_sales(product_key); CREATE INDEX idx_fact_sales_customer ON fact_sales(customer_key); CREATE INDEX idx_fact_sales_transaction ON fact_sales(transaction_id);

Dimension Tables: Context for Analysis

What Are Dimension Tables?

Dimension tables provide the "who, what, where, when, why" context for facts. They contain descriptive attributes that enable filtering, grouping, and labeling in reports. Dimensions are typically denormalized, meaning related attributes are stored together rather than split across multiple normalized tables.

Key characteristics of dimension tables:

  • Wide: Many columns (20-100+ attributes not uncommon)
  • Shallow: Relatively few rows compared to facts
  • Denormalized: Redundancy accepted for query performance
  • Slowly changing: Updates infrequent compared to fact table inserts

Types of Dimensions

Dimension Type Characteristics Examples
Conformed Shared across multiple fact tables Date, customer, product, geography
Junk Combines low-cardinality flags and indicators Order status, payment method, shipping method
Degenerate Dimension attribute stored in fact table Transaction ID, order number, invoice number
Role-Playing Same dimension used multiple times with different meanings Date (order_date, ship_date, delivery_date)
Outrigger Secondary dimension referenced by another dimension Postal code dimension referenced by customer dimension
-- Example Dimension: Product CREATE TABLE dim_product ( -- Surrogate key product_key BIGSERIAL PRIMARY KEY, -- Natural key (business identifier) product_id VARCHAR(50) UNIQUE NOT NULL, -- Descriptive attributes product_name VARCHAR(255) NOT NULL, product_description TEXT, -- Hierarchical attributes (denormalized) brand VARCHAR(100), category VARCHAR(100), subcategory VARCHAR(100), department VARCHAR(100), -- Product attributes size VARCHAR(50), color VARCHAR(50), weight_kg NUMERIC(8,2), manufacturer VARCHAR(100), supplier VARCHAR(100), -- Pricing attributes standard_cost NUMERIC(10,2), list_price NUMERIC(10,2), -- Slowly Changing Dimension Type 2 columns effective_date TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP, expiration_date TIMESTAMP, is_current BOOLEAN DEFAULT TRUE, -- Audit columns created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ); CREATE INDEX idx_product_natural_key ON dim_product(product_id); CREATE INDEX idx_product_current ON dim_product(is_current) WHERE is_current = TRUE; CREATE INDEX idx_product_category ON dim_product(category, subcategory);

Star Schema Design

The star schema is the fundamental dimensional model pattern, consisting of a central fact table surrounded by denormalized dimension tables. The schema resembles a star when visualized, with the fact table at the center and dimension tables radiating outward.

Advantages of Star Schema

  • Simplicity: Easy for business users to understand and navigate
  • Query Performance: Fewer joins required (typically 3-7 vs. 15-30 in normalized models)
  • Aggregation: Straightforward GROUP BY operations on dimension attributes
  • BI Tool Compatibility: Optimal structure for most BI and OLAP tools

When to Use Star Schema

Star schemas work well for most data warehouse scenarios, particularly when:

  • Query performance is critical
  • Business users need self-service analytics
  • Dimension cardinality is manageable (thousands to millions of rows)
  • Storage costs are reasonable
  • Data redundancy is acceptable

Snowflake Schema Design

Snowflake schemas normalize dimension tables by extracting repeated attribute values into separate tables. For example, instead of storing category and subcategory in the product dimension, a snowflake schema creates separate category and subcategory dimension tables, with the product dimension referencing them via foreign keys.

Star vs. Snowflake Comparison

Aspect Star Schema Snowflake Schema
Complexity Simple, flat dimension tables Normalized, hierarchical dimensions
Query Performance Faster (fewer joins) Slower (more joins required)
Storage More storage (redundancy) Less storage (normalization)
Maintenance Easier ETL, simpler updates Complex ETL, referential integrity
User Experience Intuitive for end users Requires more technical knowledge
Best Use Case Most data warehouses Extremely large dimensions, constrained storage

Recommendation: Favor Star Schema

In modern data warehouses, storage is relatively inexpensive compared to compute and developer time. The WIA-DATA_WAREHOUSE standard recommends star schemas for most scenarios. The simplicity, performance, and user experience benefits far outweigh the minimal storage savings of snowflake schemas. Snowflaking should only be considered when dimension tables exceed tens of millions of rows or when working with legacy systems that require normalization.

Conformed Dimensions

Conformed dimensions are master dimensions shared across multiple fact tables, ensuring consistency in reporting and enabling integrated analysis across business processes. For example, a conformed customer dimension used by sales facts, service facts, and marketing facts ensures that customer attributes (name, segment, geography) are consistent across all reports.

Benefits of Conformed Dimensions

  • Consistency: Same attributes and values across all business processes
  • Integration: Enable drill-across queries spanning multiple fact tables
  • Efficiency: Build once, reuse many times
  • Governance: Single source of truth for master data

Common Conformed Dimensions

-- Conformed Date Dimension (used across all fact tables) CREATE TABLE dim_date ( date_key INTEGER PRIMARY KEY, -- Format: YYYYMMDD (e.g., 20250108) date_value DATE UNIQUE NOT NULL, -- Calendar attributes year INTEGER, quarter INTEGER, month INTEGER, month_name VARCHAR(20), week INTEGER, day_of_month INTEGER, day_of_week INTEGER, day_name VARCHAR(20), -- Flags is_weekend BOOLEAN, is_holiday BOOLEAN, is_business_day BOOLEAN, -- Fiscal period (if different from calendar) fiscal_year INTEGER, fiscal_quarter INTEGER, fiscal_month INTEGER, -- Relative periods (useful for analytics) is_current_day BOOLEAN, is_current_week BOOLEAN, is_current_month BOOLEAN, is_current_quarter BOOLEAN, is_current_year BOOLEAN ); -- Load date dimension with 10+ years of dates INSERT INTO dim_date (date_key, date_value, year, quarter, month, ...) SELECT TO_CHAR(d, 'YYYYMMDD')::INTEGER as date_key, d as date_value, EXTRACT(YEAR FROM d) as year, EXTRACT(QUARTER FROM d) as quarter, EXTRACT(MONTH FROM d) as month, ... FROM generate_series( '2020-01-01'::DATE, '2030-12-31'::DATE, '1 day'::INTERVAL ) as d;

Key Takeaways

  • Dimensional modeling organizes data around business processes with measurable facts surrounded by descriptive dimensions, creating intuitive star-like structures that mirror business thinking and deliver excellent query performance.
  • Fact tables contain quantitative measurements at a specific grain (level of detail). The grain declaration is the most critical design decision and must be established before designing the fact table structure.
  • Dimension tables provide context through descriptive attributes. They are typically wide (many columns), shallow (few rows), and denormalized for performance, accepting redundancy to minimize joins.
  • Star schemas (denormalized dimensions) outperform snowflake schemas (normalized dimensions) in most scenarios. The simplicity, performance, and user experience benefits justify the minimal storage overhead.
  • Conformed dimensions are master dimensions shared across multiple fact tables, ensuring consistency in reporting, enabling drill-across analysis, and providing a single source of truth for key business entities like customers, products, and time.
