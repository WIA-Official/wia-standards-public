WIA-DATA_WAREHOUSE Standard - Chapter 5

Phase 2: ETL/ELT Implementation

Introduction to Data Integration

Extract, Transform, Load (ETL) and Extract, Load, Transform (ELT) are the fundamental processes that move data from source systems into data warehouses. These processes are the lifeblood of any data warehouse, responsible for extracting data from disparate sources, transforming it to meet business requirements, and loading it into dimensional models for analysis.

The choice between ETL and ELT represents a fundamental architectural decision with significant implications for performance, cost, complexity, and maintainability. Understanding the strengths and weaknesses of each approach enables architects to select the optimal strategy for their specific requirements and constraints.

ETL vs. ELT: Understanding the Paradigm Shift

Aspect ETL (Traditional) ELT (Modern Cloud)
Transformation Location External processing engine (Informatica, Talend) Within the data warehouse (SQL, dbt)
Data Flow Extract → Transform → Load Extract → Load → Transform
Processing Power Limited by ETL server resources Leverages warehouse's MPP/distributed processing
Raw Data Not retained after transformation Preserved in staging/raw layer
Flexibility Re-transformation requires re-extraction Re-transformation uses persisted raw data
Latency Higher (transformation bottleneck) Lower (parallel transformation in warehouse)
Cost Model Expensive ETL tool licenses Pay for warehouse compute (elastic)
Best For On-premises, complex transformations, limited warehouse capacity Cloud data warehouses, agile development, large-scale data

When to Choose ETL

  • Working with on-premises data warehouses with limited processing capacity
  • Highly complex transformations requiring specialized processing engines
  • Security/compliance requirements prevent raw data from entering the warehouse
  • Source systems have strict connection time windows
  • Existing investments in ETL tools and expertise

When to Choose ELT

  • Using cloud-native data warehouses (Snowflake, BigQuery, Redshift)
  • Need to preserve raw data for auditing and re-processing
  • Agile development requiring rapid iteration on transformations
  • Large data volumes requiring distributed processing
  • Preference for SQL-based transformations over proprietary tools

The Extraction Phase

Full Extraction

Full extraction reads the entire dataset from source systems. This approach is simple and ensures completeness but can be resource-intensive and time-consuming for large datasets. Full extractions are appropriate for small tables, dimension tables that don't change frequently, or initial loads.

Incremental Extraction

Incremental extraction identifies and extracts only changed or new records since the last extraction. This dramatically reduces data volumes, extraction time, and resource consumption. Several strategies enable incremental extraction:

1. Timestamp-Based Extraction

Source tables include updated_at or modified_at timestamps. The ETL process tracks the last successful extraction timestamp and queries for records modified after that point.

-- Extract records modified since last successful load SELECT * FROM source_system.customers WHERE updated_at > '2025-01-07 23:00:00' AND updated_at <= '2025-01-08 23:00:00' ORDER BY updated_at; -- Store high water mark for next extraction UPDATE etl_metadata SET last_extraction_timestamp = '2025-01-08 23:00:00' WHERE source_table = 'customers';

2. Change Data Capture (CDC)

CDC technologies capture database changes (inserts, updates, deletes) from transaction logs in real-time or near-real-time. CDC provides comprehensive change tracking without impacting source system performance. Popular CDC tools include Debezium, Oracle GoldenGate, AWS DMS, and Qlik Replicate.

3. Trigger-Based Extraction

Database triggers log changes to shadow tables that the ETL process reads. While effective, triggers add overhead to source transactions and require careful management.

The Transformation Phase

Transformation is where raw source data becomes warehouse-ready dimensional data. Transformations implement business logic, cleanse data quality issues, and structure data according to dimensional models.

Common Transformation Types

Transformation Purpose Example
Cleansing Fix data quality issues Standardize phone formats, correct misspellings, handle nulls
Standardization Ensure consistent formats Convert all dates to YYYY-MM-DD, uppercase state codes
Deduplication Remove duplicate records Identify and merge duplicate customer records
Derivation Calculate new values profit = revenue - cost, age from birth_date
Aggregation Summarize detail data Daily totals from transaction records
Splitting Parse complex fields Split full_name into first_name, last_name
Joining Combine related data Enrich transactions with customer and product attributes
Filtering Remove irrelevant data Exclude test accounts, deleted records, invalid dates
Lookup Replace codes with descriptions Convert product_code to product_name

Slowly Changing Dimension (SCD) Processing

Implementing SCD Type 2 is one of the most common and important transformations in data warehousing. This strategy maintains full historical context by creating new dimension records when attributes change.

-- SCD Type 2 Implementation for Customer Dimension -- Step 1: Identify new and changed records WITH source_data AS ( SELECT customer_id, customer_name, email, segment, city, state, country FROM staging.customers ), existing_current AS ( SELECT * FROM dim_customer WHERE is_current = TRUE ), changes AS ( SELECT s.*, e.customer_key as existing_key, CASE WHEN e.customer_key IS NULL THEN 'INSERT' -- New customer WHEN s.customer_name != e.customer_name OR s.email != e.email OR s.segment != e.segment OR s.city != e.city OR s.state != e.state OR s.country != e.country THEN 'UPDATE' -- Changed attributes ELSE 'NOCHANGE' END as change_type FROM source_data s LEFT JOIN existing_current e ON s.customer_id = e.customer_id ) -- Step 2: Expire old records for updated customers UPDATE dim_customer SET is_current = FALSE, expiration_date = CURRENT_TIMESTAMP, updated_at = CURRENT_TIMESTAMP WHERE customer_key IN ( SELECT existing_key FROM changes WHERE change_type = 'UPDATE' ) AND is_current = TRUE; -- Step 3: Insert new records (both new customers and updated versions) INSERT INTO dim_customer ( customer_id, customer_name, email, segment, city, state, country, effective_date, is_current ) SELECT customer_id, customer_name, email, segment, city, state, country, CURRENT_TIMESTAMP as effective_date, TRUE as is_current FROM changes WHERE change_type IN ('INSERT', 'UPDATE');

The Loading Phase

Load Strategies

1. Full Refresh

Truncate the target table and load all data. Simple and ensures complete accuracy but requires reprocessing all data. Suitable for small dimension tables or when source data is completely replaced.

2. Incremental Append

Add new records to existing data without modifying existing rows. Used for fact tables where each row represents an immutable event (transactions, clicks, sensor readings).

3. Incremental Upsert (Merge)

Update existing records and insert new ones. Common for dimension tables and fact tables that require updates (order status changes, invoice adjustments).

-- Modern SQL MERGE statement for upsert operations MERGE INTO dim_product AS target USING staging.products AS source ON target.product_id = source.product_id AND target.is_current = TRUE -- Update existing records that have changed WHEN MATCHED AND ( target.product_name != source.product_name OR target.category != source.category OR target.list_price != source.list_price ) THEN UPDATE SET is_current = FALSE, expiration_date = CURRENT_TIMESTAMP -- Insert new records WHEN NOT MATCHED THEN INSERT (product_id, product_name, category, list_price, effective_date, is_current) VALUES (source.product_id, source.product_name, source.category, source.list_price, CURRENT_TIMESTAMP, TRUE);

Error Handling and Data Quality

Robust ETL processes must gracefully handle errors and data quality issues. Poor error handling leads to incomplete loads, data inconsistencies, and loss of trust in the warehouse.

Error Handling Strategies

  • Reject and Continue: Log invalid records to an error table and continue processing valid records
  • Abort on Error: Stop processing when errors exceed a threshold, preventing bad data from entering the warehouse
  • Default Values: Apply business rules to substitute missing or invalid values with defaults
  • Quarantine: Route suspicious records to a quarantine area for manual review and correction

ETL Monitoring and Logging

Every ETL job should log comprehensive metadata:

  • Job execution start/end times and duration
  • Records extracted, transformed, loaded, rejected
  • Data quality metrics (completeness, accuracy, consistency)
  • Error messages and stack traces
  • Source system and warehouse snapshots (for auditing)
  • Resource consumption (CPU, memory, I/O)

This metadata enables troubleshooting, performance optimization, and compliance reporting.

Modern ELT with dbt (data build tool)

dbt has emerged as the leading tool for ELT transformations in modern cloud data warehouses. dbt enables analytics engineers to transform data using SQL SELECT statements, applying software engineering best practices like version control, testing, and documentation to analytics code.

-- dbt model: models/marts/fct_daily_sales.sql -- Transforms raw order data into daily sales facts {{ config( materialized='incremental', unique_key='sales_date_key' ) }} WITH orders AS ( SELECT * FROM {{ ref('stg_orders') }} {% if is_incremental() %} WHERE order_date >= (SELECT MAX(sales_date) FROM {{ this }}) {% endif %} ), daily_aggregation AS ( SELECT TO_CHAR(order_date, 'YYYYMMDD')::INTEGER AS sales_date_key, COUNT(DISTINCT order_id) AS order_count, COUNT(DISTINCT customer_id) AS customer_count, SUM(order_amount) AS total_sales, SUM(cost_amount) AS total_cost, SUM(order_amount - cost_amount) AS total_profit, AVG(order_amount) AS avg_order_value FROM orders GROUP BY order_date ) SELECT * FROM daily_aggregation

Key Takeaways

  • ETL (Extract-Transform-Load) transforms data before loading, using external processing engines. ELT (Extract-Load-Transform) loads raw data first and transforms within the warehouse, leveraging distributed processing power of modern cloud data warehouses.
  • Incremental extraction strategies (timestamp-based, CDC, triggers) dramatically reduce data volumes and processing time compared to full extractions, enabling near-real-time data warehouses and minimizing source system impact.
  • Transformation types include cleansing (fixing quality issues), standardization (consistent formats), derivation (calculated fields), aggregation (summarization), and SCD processing (historical tracking), each serving specific business and technical requirements.
  • Load strategies vary by use case: full refresh for small dimensions, incremental append for immutable facts, and upsert/merge for dimensions and updatable facts. Modern SQL MERGE statements simplify complex upsert logic.
  • Robust error handling, comprehensive logging, and data quality monitoring are essential for production ETL systems. Errors should be logged, invalid records quarantined, and metadata tracked for troubleshooting and compliance.
