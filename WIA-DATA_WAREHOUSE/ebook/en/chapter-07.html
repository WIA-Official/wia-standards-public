WIA-DATA_WAREHOUSE Standard - Chapter 7

Phase 4: Performance & Optimization

Introduction to Performance Optimization

A well-designed dimensional model is only the starting point. As data volumes grow and user demands increase, performance optimization becomes critical. Queries that run in seconds with 100GB of data can take minutes or hours with 10TB. This chapter explores proven techniques to ensure your data warehouse delivers fast query response times even as it scales to petabytes.

Performance optimization is both art and science, requiring understanding of database internals, query execution plans, storage engines, and business requirements. The goal is not just raw speed, but cost-effective performance that delivers value to users.

Partitioning Strategies

Table partitioning divides large tables into smaller, more manageable pieces called partitions. Queries that filter on the partition key scan only relevant partitions, dramatically reducing I/O and improving performance. Partitioning also simplifies data lifecycle management, enabling efficient archiving and deletion of old partitions.

Partition Types

Type Description Use Case Example
Range Partition by value ranges Time-series data, dates, sequential IDs Monthly partitions: Jan 2025, Feb 2025, Mar 2025
List Partition by discrete values Geography, categories, status codes By region: North America, Europe, Asia-Pacific
Hash Partition by hash function Even distribution, load balancing customer_id % 16 creates 16 partitions
Composite Combine multiple strategies Multi-dimensional partitioning Range by year, sub-partitioned by hash on customer_id
-- Range Partitioning by Date (PostgreSQL example) CREATE TABLE fact_sales ( sales_key BIGSERIAL, date_key INTEGER NOT NULL, product_key BIGINT NOT NULL, customer_key BIGINT NOT NULL, quantity NUMERIC(10,2), amount NUMERIC(12,2), created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ) PARTITION BY RANGE (date_key); -- Create monthly partitions for 2025 CREATE TABLE fact_sales_2025_01 PARTITION OF fact_sales FOR VALUES FROM (20250101) TO (20250201); CREATE TABLE fact_sales_2025_02 PARTITION OF fact_sales FOR VALUES FROM (20250201) TO (20250301); CREATE TABLE fact_sales_2025_03 PARTITION OF fact_sales FOR VALUES FROM (20250301) TO (20250401); -- Query automatically uses partition pruning SELECT SUM(amount) FROM fact_sales WHERE date_key >= 20250101 AND date_key < 20250201; -- Only scans fact_sales_2025_01 partition! -- Drop old partitions for data lifecycle management DROP TABLE fact_sales_2023_01; -- Delete January 2023 data

Partitioning Best Practices

  • Choose the right partition key: Use columns frequently in WHERE clauses. Date/time is the most common choice for fact tables.
  • Partition size matters: Too many small partitions add overhead. Aim for 10-100 partitions, each containing millions of rows.
  • Align with business processes: Monthly partitions align with month-end reporting, daily partitions support daily ETL.
  • Index within partitions: Create indexes on each partition for commonly filtered columns.
  • Monitor partition pruning: Verify query execution plans actually skip irrelevant partitions.

Indexing Strategies

Indexes are database structures that speed up data retrieval at the cost of additional storage and slower writes. Choosing the right indexes requires understanding query patterns, cardinality, and database capabilities.

Index Type Best For Trade-offs
B-Tree Equality and range queries, high cardinality columns Default choice, balanced read/write performance
Bitmap Low cardinality columns (gender, status, boolean) Excellent for AND/OR queries, slower updates
Hash Equality queries only (exact matches) Very fast equality, cannot support ranges
Covering/Include Include non-indexed columns to avoid table lookup Larger index size, eliminates additional I/O
Partial Index subset of rows (WHERE condition) Smaller index for frequently filtered subsets
Composite Queries filtering on multiple columns Column order matters, supports prefix searches
-- B-Tree index on frequently filtered column CREATE INDEX idx_sales_customer ON fact_sales(customer_key); -- Composite index (order matters!) -- Supports: WHERE product_key = X AND date_key = Y -- Also supports: WHERE product_key = X -- Does NOT support: WHERE date_key = Y (alone) CREATE INDEX idx_sales_product_date ON fact_sales(product_key, date_key); -- Covering index includes additional columns -- Avoids going back to table for amount column CREATE INDEX idx_sales_customer_covering ON fact_sales(customer_key) INCLUDE (amount, quantity); -- Partial index (index only current records) CREATE INDEX idx_customer_current ON dim_customer(customer_id) WHERE is_current = TRUE; -- Bitmap index for low cardinality (Oracle, some columnar DBs) CREATE BITMAP INDEX idx_sales_status ON fact_sales(order_status);

Index Guidelines

  • Foreign keys: Always index foreign keys joining to dimensions
  • Filter columns: Index columns in WHERE clauses of frequent queries
  • Sort columns: Index columns in ORDER BY clauses
  • Join columns: Index columns used in JOIN conditions
  • Avoid over-indexing: Each index slows down inserts/updates. Monitor index usage and drop unused indexes.
  • Rebuild periodically: Indexes fragment over time. Schedule regular rebuilds.

Materialized Views

Materialized views pre-compute and store query results, similar to aggregate tables but managed by the database. They can be refreshed on-demand, on schedule, or incrementally, providing a balance between freshness and performance.

-- Create materialized view for monthly sales summary CREATE MATERIALIZED VIEW mv_monthly_sales AS SELECT d.year, d.month, p.category, p.subcategory, s.store_region, COUNT(DISTINCT f.customer_key) as customer_count, COUNT(*) as transaction_count, SUM(f.amount) as total_sales, SUM(f.quantity) as total_quantity, AVG(f.amount) as avg_transaction_amount FROM fact_sales f JOIN dim_date d ON f.date_key = d.date_key JOIN dim_product p ON f.product_key = p.product_key JOIN dim_store s ON f.store_key = s.store_key GROUP BY d.year, d.month, p.category, p.subcategory, s.store_region; -- Create index on materialized view CREATE INDEX idx_mv_monthly_year_month ON mv_monthly_sales(year, month); -- Refresh materialized view (full refresh) REFRESH MATERIALIZED VIEW mv_monthly_sales; -- Concurrent refresh (doesn't block reads) REFRESH MATERIALIZED VIEW CONCURRENTLY mv_monthly_sales; -- Schedule automatic refresh (using cron or scheduler) -- Refresh daily at 2 AM: -- 0 2 * * * psql -d warehouse -c "REFRESH MATERIALIZED VIEW mv_monthly_sales;"

Materialized View Strategies

Refresh Type Description When to Use
Complete Drop and rebuild entire view Small views, significant data changes
Incremental Update only changed rows Large views, small incremental changes (requires database support)
On-Demand Manual refresh when needed Ad-hoc analysis, infrequent queries
Scheduled Refresh at regular intervals Daily/hourly updates, batch processing

Columnar Storage and Compression

Traditional row-oriented databases store entire rows contiguously, optimized for transactional operations. Columnar databases store each column separately, dramatically improving analytical query performance and enabling higher compression ratios.

Row-Oriented vs. Column-Oriented Storage

Row-Oriented Storage

Row 1: [id=1, name="Alice", age=30, city="NYC", salary=75000]
Row 2: [id=2, name="Bob", age=25, city="LA", salary=65000]
Row 3: [id=3, name="Carol", age=35, city="NYC", salary=85000]

Query "SELECT AVG(salary) FROM employees WHERE city='NYC'" must read ALL columns for matching rows.

Column-Oriented Storage

ID column: [1, 2, 3]
Name column: ["Alice", "Bob", "Carol"]
Age column: [30, 25, 35]
City column: ["NYC", "LA", "NYC"]
Salary column: [75000, 65000, 85000]

Same query only reads City column (to filter) and Salary column (to aggregate). 60% less I/O!

Compression Benefits

Columnar storage enables superior compression because similar data types are stored together:

  • Run-Length Encoding: Repeated values compressed (e.g., 100 rows with "NYC" stored once)
  • Dictionary Encoding: Replace values with integer codes (e.g., "New York City" → 1)
  • Bit-Packing: Store small integers in fewer bits
  • Delta Encoding: Store differences between consecutive values for sorted data

Typical compression ratios range from 5:1 to 20:1, meaning 1TB of raw data might occupy only 50-200GB on disk. This reduces storage costs and I/O, further improving performance.

Query Optimization Techniques

1. Analyze Query Execution Plans

-- PostgreSQL: EXPLAIN ANALYZE shows actual execution plan EXPLAIN ANALYZE SELECT p.product_name, SUM(f.amount) as total_sales FROM fact_sales f JOIN dim_product p ON f.product_key = p.product_key WHERE f.date_key >= 20250101 AND f.date_key < 20250201 GROUP BY p.product_name ORDER BY total_sales DESC LIMIT 10; -- Look for: -- - Sequential scans (bad) vs. Index scans (good) -- - Hash joins (fast) vs. Nested loops (slow for large tables) -- - Partition pruning (reducing scanned data) -- - High row count estimates vs. actuals (statistics out of date)

2. Update Statistics Regularly

Database query optimizers rely on statistics about data distribution. Outdated statistics lead to poor execution plans. Schedule regular statistics updates, especially after large data loads.

-- PostgreSQL: Analyze tables to update statistics ANALYZE fact_sales; ANALYZE dim_product; -- Analyze all tables in database ANALYZE; -- Auto-vacuum daemon updates statistics automatically, -- but manual ANALYZE after large loads is recommended

3. Use Appropriate WHERE Clause Techniques

  • Avoid functions on indexed columns: WHERE YEAR(order_date) = 2025 can't use index. Use WHERE order_date >= '2025-01-01' AND order_date < '2026-01-01'
  • Use IN instead of OR for multiple values: WHERE status IN ('pending', 'processing') is faster than WHERE status = 'pending' OR status = 'processing'
  • Avoid leading wildcards in LIKE: WHERE product_name LIKE '%phone%' can't use index. WHERE product_name LIKE 'phone%' can.
  • Use EXISTS instead of IN for correlated subqueries: EXISTS can short-circuit after first match

Key Takeaways

  • Table partitioning divides large tables into manageable chunks, enabling partition pruning (scanning only relevant partitions), simplifying data lifecycle management, and dramatically improving query performance on time-series data.
  • Strategic indexing accelerates queries but comes at a cost: storage overhead and slower writes. Index foreign keys, filter columns, and join columns, but monitor index usage and drop unused indexes to avoid over-indexing.
  • Materialized views pre-compute expensive aggregations, trading storage and refresh overhead for query speed. Choose refresh strategies (complete, incremental, on-demand, scheduled) based on data volatility and freshness requirements.
  • Columnar storage stores each column separately rather than entire rows, dramatically reducing I/O for analytical queries (reading only needed columns) and enabling superior compression ratios (5:1 to 20:1) through encoding techniques.
  • Query optimization requires analyzing execution plans, maintaining current statistics, avoiding functions on indexed columns, and using appropriate SQL techniques. Small changes to WHERE clauses can yield order-of-magnitude performance improvements.
