클라우드 데이터 웨어하우스: Snowflake, BigQuery, Redshift

1. 클라우드 DW 혁명

2010년대 중반, 클라우드 데이터 웨어하우스는 DW 산업을 재정의했습니다. 전통적인 On-Premise DW (Oracle, Teradata)의 한계를 넘어선 새로운 패러다임을 제시했습니다.

1.1 On-Premise vs Cloud DW

항목 On-Premise (Oracle, Teradata) Cloud DW (Snowflake, BigQuery)
초기 비용 $500K - $5M (HW + SW) $0 (종량제)
구축 기간 6-12개월 1일 (즉시 시작)
확장성 수동 (HW 구매 필요) 자동 (초 단위)
유지보수 DBA 필수 (패치, 백업...) 완전 관리형
가용성 99.9% (자체 구축) 99.99% (SLA 보장)
재해 복구 복잡 (DR 센터 구축) 자동 (다중 리전)
성능 고정 (HW 한계) 무제한 (탄력적)
비용 효율 낮음 (24시간 가동) 높음 (사용량 기반)

🎯 클라우드 DW의 핵심 가치

  • Zero Administration: DBA 불필요, 패치/백업 자동
  • Elastic Scalability: 초 단위로 컴퓨팅 파워 조정
  • Pay-as-you-go: 사용한 만큼만 과금
  • Separation of Storage & Compute: 독립적 확장
  • Near-Unlimited Concurrency: 수천 명 동시 쿼리

2. Snowflake: 차세대 DW 아키텍처

Snowflake는 클라우드 네이티브로 처음부터 설계된 DW입니다. 2012년 창업, 2024년 시가총액 $50B (약 70조원)의 DW 리더입니다.

2.1 Snowflake 3-Layer 아키텍처

┌─────────────────────────────────────────────────────────┐
│          Layer 1: Database Storage                      │
│  ─────────────────────────────────────────────────────  │
│   S3 / Azure Blob / GCS (Object Storage)                │
│   - Micro-partitions (16MB 단위)                        │
│   - Columnar Format + Compression (10:1)                │
│   - Immutable (변경 불가)                               │
│   - Automatic Clustering                                │
└───────────────────┬─────────────────────────────────────┘
                    │
┌───────────────────▼─────────────────────────────────────┐
│          Layer 2: Query Processing (Virtual Warehouse)  │
│  ─────────────────────────────────────────────────────  │
│   ┌─────────┐   ┌─────────┐   ┌─────────┐              │
│   │ WH_ETL  │   │ WH_BI   │   │WH_DS    │  ← 독립적!  │
│   │ (XL)    │   │ (L)     │   │(XXL)    │              │
│   │ 8 nodes │   │ 4 nodes │   │16 nodes │              │
│   └─────────┘   └─────────┘   └─────────┘              │
│   - Massively Parallel Processing (MPP)                 │
│   - 자동 확장 (Auto-Scaling)                            │
│   - 자동 중지/재개 (Auto-Suspend/Resume)                │
└───────────────────┬─────────────────────────────────────┘
                    │
┌───────────────────▼─────────────────────────────────────┐
│          Layer 3: Cloud Services                        │
│  ─────────────────────────────────────────────────────  │
│   - 인증 (Authentication)                               │
│   - 메타데이터 관리 (Metadata Management)               │
│   - 쿼리 최적화 (Query Optimization)                    │
│   - 보안 (Encryption, Access Control)                   │
│   - Result Cache (24시간)                               │
└─────────────────────────────────────────────────────────┘

2.2 Snowflake의 혁신적 기능

Time Travel & Fail-safe

-- Time Travel: 과거 데이터 조회 (최대 90일) SELECT * FROM sales_table AT(TIMESTAMP => '2024-01-15 10:00:00'); SELECT * FROM sales_table BEFORE(STATEMENT => '01a70a63-0000-0001-0000-000000000001'); -- 실수로 삭제한 데이터 복구 CREATE TABLE sales_recovered AS SELECT * FROM sales_table BEFORE(STATEMENT => '01a70a63-0000-0001-0000-000000000001'); -- Fail-safe: 7일간 추가 보호 (Snowflake만 복구 가능) -- 총 97일 (90 + 7) 데이터 보호!

Zero-Copy Cloning

-- 테라바이트 테이블을 수 초만에 복제 (추가 스토리지 비용 없음!) CREATE TABLE sales_dev CLONE sales_prod; -- 데이터베이스 전체 복제 CREATE DATABASE analytics_dev CLONE analytics_prod; -- 원리: 메타데이터만 복사, 실제 데이터는 공유 -- 변경 시에만 새 Micro-partition 생성 (Copy-on-Write)

Data Sharing (실시간)

-- 데이터를 복사하지 않고 실시간 공유 CREATE SHARE sales_share; GRANT USAGE ON DATABASE sales_db TO SHARE sales_share; GRANT SELECT ON TABLE sales_db.public.sales_table TO SHARE sales_share; -- 파트너에게 공유 (계정 간) ALTER SHARE sales_share ADD ACCOUNTS = xy12345, ab67890; -- 파트너는 즉시 쿼리 가능 (복사 없음, 실시간 동기화!)

3. Google BigQuery: 서버리스 DW

BigQuery는 완전 서버리스 DW로, 가상 웨어하우스 개념이 없습니다. Google의 검색 인프라(Dremel)를 기반으로 페타바이트를 초 단위로 처리합니다.

3.1 BigQuery 아키텍처

┌────────────────────────────────────────────────────┐
│         Dremel (Query Engine)                      │
│  ────────────────────────────────────────────────  │
│   - Massively Parallel (수만 노드)                 │
│   - 동적 리소스 할당 (쿼리당 자동)                 │
│   - Columnar Execution                             │
└────────────────┬───────────────────────────────────┘
                 │
┌────────────────▼───────────────────────────────────┐
│         Colossus (Storage)                         │
│  ────────────────────────────────────────────────  │
│   - Capacitor (Columnar Format)                    │
│   - 자동 복제 (3 copies)                           │
│   - 압축 8:1                                       │
│   - 자동 최적화 (테이블 재구성)                    │
└────────────────┬───────────────────────────────────┘
                 │
┌────────────────▼───────────────────────────────────┐
│         Jupiter (Network)                          │
│  ────────────────────────────────────────────────  │
│   - Petabit/s 네트워크                             │
│   - 데이터센터 간 초고속 전송                      │
└────────────────────────────────────────────────────┘

3.2 BigQuery의 차별화 기능

BI Engine (인메모리 분석)

-- BI Engine 예약 (GB당 $0.06/시간) bq mk --capacity=100GB --location=us bi_engine_reservation -- 효과: -- 일반 쿼리: 5초 -- BI Engine: 0.1초 (50배 빠름!) -- 대시보드 새로고침이 즉각 반응

BigQuery ML (내장 머신러닝)

-- SQL만으로 머신러닝 모델 생성! CREATE OR REPLACE MODEL sales_forecast_model OPTIONS( model_type='linear_reg', input_label_cols=['total_sales'] ) AS SELECT date, day_of_week, is_holiday, total_sales FROM sales_daily WHERE date < '2024-01-01'; -- 예측 SELECT date, predicted_total_sales FROM ML.PREDICT(MODEL sales_forecast_model, ( SELECT * FROM sales_daily WHERE date >= '2024-01-01' )); -- 지원 모델: -- - Linear Regression, Logistic Regression -- - K-Means Clustering -- - Time Series (ARIMA+) -- - XGBoost, DNN, AutoML

GIS (지리공간 분석)

-- 위치 기반 쿼리 SELECT store_name, ST_DISTANCE( store_location, -- GEOGRAPHY 타입 ST_GEOGPOINT(127.0276, 37.4979) -- 강남역 ) / 1000 AS distance_km FROM stores WHERE ST_DISTANCE( store_location, ST_GEOGPOINT(127.0276, 37.4979) ) < 5000 -- 5km 이내 ORDER BY distance_km;

Capacity (Flat-rate) $2,000/월 (100 슬롯) 대규모, 지속적 쿼리

⚠️ BigQuery 비용 절감 팁

  • SELECT * 절대 금지! 필요한 컬럼만 → 90% 비용 절감
  • Partitioning 필수: WHERE에 파티션 컬럼 사용
  • Clustering 활용: 자주 필터링되는 컬럼
  • 쿼리 전 비용 확인: 우측 상단 "This query will process X TB"
  • Materialized View: 반복 쿼리 비용 제로

4. Amazon Redshift: AWS 생태계 DW

Redshift는 2013년 출시된 AWS의 DW 서비스로, PostgreSQL 기반입니다. AWS 서비스와의 완벽한 통합이 강점입니다.

4.2 Redshift RA3 vs DC2

항목 DC2 (Dense Compute) RA3 (Redshift Managed Storage)
스토리지 로컬 SSD (고정) S3 (무제한, 독립적 확장)
가격 $0.25/시간 (dc2.large) $1.086/시간 (ra3.xlplus)
캐싱 N/A 자동 (Hot Data를 SSD에)
적합 사례 < 10TB, 비용 민감 > 10TB, 확장성 필요

4.3 Redshift Spectrum (S3 직접 쿼리)

-- S3의 Parquet 파일을 Redshift에서 직접 쿼리! CREATE EXTERNAL SCHEMA spectrum_schema FROM DATA CATALOG DATABASE 'spectrumdb' IAM_ROLE 'arn:aws:iam::123456789012:role/SpectrumRole'; CREATE EXTERNAL TABLE spectrum_schema.sales_parquet ( sale_date DATE, customer_id INT, total_amount DECIMAL(10,2) ) STORED AS PARQUET LOCATION 's3://my-bucket/sales-data/'; -- Redshift 테이블 + S3 Parquet 조인 (하이브리드!) SELECT c.customer_name, SUM(s.total_amount) AS total_sales FROM spectrum_schema.sales_parquet s -- S3 JOIN customers c ON s.customer_id = c.customer_id -- Redshift WHERE s.sale_date >= '2024-01-01' GROUP BY c.customer_name; -- 장점: 데이터 이동 없이 Data Lake + DW 통합!

6. 선택 가이드

✅ Snowflake를 선택하세요

  • 멀티 클라우드 전략 (AWS + Azure + GCP)
  • 데이터 공유가 중요 (파트너, 고객과 실시간 공유)
  • 예측 가능한 비용 (시간당 고정)
  • Zero Administration 중시
  • Semi-structured 데이터 많음 (JSON, Avro, Parquet)

✅ BigQuery를 선택하세요

  • Google Cloud 생태계 (GKE, Dataflow, Looker)
  • 완전 서버리스 선호 (관리 제로)
  • 머신러닝 내장 필요 (BigQuery ML)
  • 지리공간 분석 (GIS)
  • 페타바이트급 데이터 (최고 확장성)
  • 비정기적 사용 (쿼리 없으면 비용 제로)

✅ Redshift를 선택하세요

  • AWS 올인 전략 (S3, Glue, Athena, QuickSight)
  • PostgreSQL 호환성 필요
  • Data Lake 통합 (Spectrum으로 S3 쿼리)
  • 비용 민감 (DC2는 가장 저렴)
  • Reserved Instance로 50% 할인

7. On-Premise에서 클라우드로 마이그레이션

7.1 마이그레이션 전략

🎯 3단계 마이그레이션

  1. Lift & Shift (Phase 1)
    • 현재 스키마 그대로 이전
    • ETL 코드 최소 수정
    • 기간: 2-3개월
  2. Optimize (Phase 2)
    • 클라우드 최적화 (Clustering, Partitioning)
    • ELT로 전환 (dbt 도입)
    • 기간: 3-6개월
  3. Innovate (Phase 3)
    • ML/AI 통합
    • 실시간 분석
    • Data Mesh 전환

7.2 마이그레이션 도구

-- Snowflake SnowSQL로 대량 데이터 적재 snowsql -a myaccount -u myuser -d mydb -s myschema -q " COPY INTO my_table FROM 's3://my-bucket/data/' FILE_FORMAT = (TYPE = 'CSV' COMPRESSION = 'GZIP') ON_ERROR = 'CONTINUE'; " -- BigQuery bq 명령어 bq load --source_format=CSV \ --skip_leading_rows=1 \ mydataset.mytable \ gs://my-bucket/data/*.csv \ schema.json -- Redshift COPY 명령 COPY sales_table FROM 's3://my-bucket/sales-data/' IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftRole' FORMAT AS PARQUET;

🎯 핵심 요점

  • 클라우드 DW = Zero Administration: DBA 불필요, 패치/백업 자동
  • Snowflake의 핵심: Storage & Compute 분리, Time Travel, Zero-Copy Clone
  • BigQuery의 강점: 완전 서버리스, BigQuery ML, GIS, 페타바이트 확장성
  • Redshift의 장점: AWS 통합, PostgreSQL 호환, Spectrum으로 Data Lake 연결
  • 가격 모델: Snowflake(시간), BigQuery(스캔량), Redshift(노드)
  • 멀티 클라우드 → Snowflake, GCP → BigQuery, AWS → Redshift
  • SELECT * 절대 금지 (특히 BigQuery): 필요한 컬럼만
  • Partitioning & Clustering 필수: 10-100배 성능 향상
  • 마이그레이션: Lift & Shift → Optimize → Innovate
  • 弘益人間: 클라우드로 민주화된 데이터 접근, 모두를 위한 인사이트 🌍
