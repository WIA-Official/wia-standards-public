Chapter 01: 데이터 웨어하우스 기초

OLTP vs OLAP, Inmon vs Kimball, 히스토리 관리, SCD 타입

1. 데이터 웨어하우스란?

데이터 웨어하우스(Data Warehouse, DW)는 Bill Inmon이 1990년에 정의한 개념으로, "의사결정 지원을 목적으로 주제 지향적(Subject-Oriented), 통합적(Integrated), 시간 변화(Time-Variant), 비휘발성(Non-Volatile) 데이터의 집합"입니다.

데이터 웨어하우스 4대 특징 (Bill Inmon, 1990)

1️⃣ 주제 지향적 (Subject-Oriented): 고객, 제품, 판매 등 비즈니스 주제 중심
2️⃣ 통합적 (Integrated): 여러 소스의 데이터를 일관된 형식으로 통합
3️⃣ 시간 변화 (Time-Variant): 과거 데이터 보존, 히스토리 추적
4️⃣ 비휘발성 (Non-Volatile): 삽입과 조회만 가능, 업데이트/삭제 불가

1.1 OLTP vs OLAP

데이터베이스 시스템은 크게 OLTP와 OLAP로 나뉩니다.

-- OLTP 쿼리 예시: 단일 레코드 CRUD -- 주문 생성 (10ms) INSERT INTO orders (customer_id, product_id, quantity, price) VALUES (12345, 67890, 2, 29.99); -- 재고 업데이트 (5ms) UPDATE inventory SET stock = stock - 2 WHERE product_id = 67890; -- OLAP 쿼리 예시: 집계 및 분석 -- 지난 12개월 월별 카테고리별 매출 TOP 10 (30초) SELECT DATE_TRUNC('month', order_date) AS month, p.category, SUM(o.quantity * o.price) AS total_revenue, COUNT(DISTINCT o.customer_id) AS unique_customers, AVG(o.quantity * o.price) AS avg_order_value FROM fact_sales o JOIN dim_product p ON o.product_key = p.product_key JOIN dim_date d ON o.date_key = d.date_key WHERE d.date BETWEEN '2024-01-01' AND '2024-12-31' GROUP BY month, p.category ORDER BY total_revenue DESC LIMIT 10;

2. 데이터 웨어하우스 아키텍처

2.1 Inmon 방법론 (Top-Down)

Bill Inmon의 기업 데이터 웨어하우스(Enterprise Data Warehouse, EDW) 접근법은 정규화된 중앙 DW를 먼저 구축한 후 부서별 데이터 마트를 파생시킵니다.

✅ Inmon 방법론 특징

아키텍처:
소스 시스템 → ETL → EDW (3NF 정규화) → 데이터 마트 (부서별)

장점:
• 단일 진실 공급원 (Single Source of Truth, SSOT)
• 데이터 일관성 보장
• 중복 최소화
• 장기적 유지보수 용이

단점:
• 초기 구축 비용 높음 (6개월 ~ 2년)
• 첫 ROI까지 시간 오래 걸림
• 유연성 낮음

2.2 Kimball 방법론 (Bottom-Up)

Ralph Kimball의 차원 모델링(Dimensional Modeling) 접근법은 비즈니스 프로세스별로 데이터 마트를 먼저 구축한 후 통합합니다.

✅ Kimball 방법론 특징

아키텍처:
소스 시스템 → ETL → 데이터 마트 (스타 스키마) → DW Bus (통합)

장점:
• 빠른 초기 ROI (2~3개월)
• 비즈니스 요구에 민첩하게 대응
• 쿼리 성능 우수
• 사용자 친화적 (비정규화)

단점:
• 데이터 중복 가능
• 일관성 유지 어려움
• 장기적으로 복잡도 증가

2.3 Inmon vs Kimball 비교

항목 Inmon (Top-Down) Kimball (Bottom-Up)
접근 방식 기업 전체 DW 먼저 부서별 마트 먼저
데이터 모델 3NF 정규화 (ER 모델) 스타/스노우플레이크 (차원 모델)
구축 기간 12~24개월 2~3개월 (마트당)
초기 비용 높음 ($500K ~ $5M) 낮음 ($50K ~ $500K)
쿼리 성능 느림 (정규화로 인한 JOIN) 빠름 (비정규화)
데이터 일관성 높음 (SSOT) 중간 (컨포멀 차원 필요)
유연성 낮음 높음
적합한 조직 대기업, 금융, 정부 중소기업, 스타트업

3. 히스토리 관리 및 SCD

데이터 웨어하우스의 핵심 기능 중 하나는 시간에 따른 데이터 변화를 추적하는 것입니다. 이를 위해 Slowly Changing Dimension (SCD) 기법을 사용합니다.

3.1 SCD Type 1: 덮어쓰기

변경 사항을 히스토리 없이 바로 덮어씁니다. 과거 데이터 손실됩니다.

-- SCD Type 1: 고객 주소 변경 -- BEFORE customer_id | name | address 1 | 홍길동 | 서울시 강남구 -- UPDATE UPDATE dim_customer SET address = '부산시 해운대구' WHERE customer_id = 1; -- AFTER (히스토리 없음) customer_id | name | address 1 | 홍길동 | 부산시 해운대구
⚠️ SCD Type 1 사용 사례
• 오타 수정 (이메일 주소, 전화번호)
• 히스토리가 중요하지 않은 속성
• 저장 공간 제약이 심한 경우

3.2 SCD Type 2: 새 행 추가

변경 사항을 새 행으로 추가하고 기존 행을 유지합니다. 완전한 히스토리 추적 가능.

-- SCD Type 2: 고객 등급 변경 (가장 많이 사용) CREATE TABLE dim_customer ( customer_key INT PRIMARY KEY, -- 서로게이트 키 (DW 내부 키) customer_id INT, -- 비즈니스 키 (자연 키) name VARCHAR(100), grade VARCHAR(20), effective_date DATE, -- 유효 시작일 expiration_date DATE, -- 유효 종료일 is_current BOOLEAN -- 현재 유효 여부 ); -- 초기 데이터 (2024-01-01) INSERT INTO dim_customer VALUES (1, 1001, '홍길동', 'Silver', '2024-01-01', '9999-12-31', TRUE); -- 등급 변경: Silver → Gold (2024-06-15) -- 1) 기존 행 만료 UPDATE dim_customer SET expiration_date = '2024-06-14', is_current = FALSE WHERE customer_key = 1; -- 2) 새 행 추가 INSERT INTO dim_customer VALUES (2, 1001, '홍길동', 'Gold', '2024-06-15', '9999-12-31', TRUE); -- 결과: 히스토리 완벽 추적 customer_key | customer_id | name | grade | effective_date | expiration_date | is_current 1 | 1001 | 홍길동 | Silver | 2024-01-01 | 2024-06-14 | FALSE 2 | 1001 | 홍길동 | Gold | 2024-06-15 | 9999-12-31 | TRUE
✅ SCD Type 2 장점
• 완전한 히스토리 추적
• 시점별 분석 가능 (2024년 1월에는 Silver 등급이었던 고객)
• 감사(Audit) 요구사항 충족

⚠️ SCD Type 2 단점
• 저장 공간 증가
• 쿼리 복잡도 증가 (JOIN 시 is_current 조건 필요)
• 업데이트 로직 복잡

3.3 SCD Type 3: 이전 값 컬럼 추가

변경 사항을 별도 컬럼에 저장합니다. 제한적 히스토리만 유지 (보통 1~2단계).

-- SCD Type 3: 부서 변경 (제한적 히스토리) CREATE TABLE dim_employee ( employee_id INT PRIMARY KEY, name VARCHAR(100), current_department VARCHAR(50), previous_department VARCHAR(50), department_change_date DATE ); -- 초기 데이터 INSERT INTO dim_employee VALUES (1, '김철수', 'IT', NULL, NULL); -- 부서 변경: IT → HR UPDATE dim_employee SET previous_department = current_department, current_department = 'HR', department_change_date = '2024-08-01' WHERE employee_id = 1; -- 결과 employee_id | name | current_dept | previous_dept | change_date 1 | 김철수 | HR | IT | 2024-08-01

3.4 SCD Type 4: 히스토리 테이블 분리

현재 데이터와 히스토리를 별도 테이블로 분리합니다.

-- SCD Type 4: 현재 + 히스토리 테이블 -- 현재 테이블 CREATE TABLE dim_product_current ( product_id INT PRIMARY KEY, name VARCHAR(100), price DECIMAL(10,2), category VARCHAR(50) ); -- 히스토리 테이블 CREATE TABLE dim_product_history ( product_id INT, name VARCHAR(100), price DECIMAL(10,2), category VARCHAR(50), effective_date DATE, expiration_date DATE, PRIMARY KEY (product_id, effective_date) );

3.5 SCD Type 6: Type 1 + 2 + 3 하이브리드

Type 2의 완전한 히스토리 + Type 3의 이전 값 컬럼을 조합합니다.

-- SCD Type 6: 하이브리드 CREATE TABLE dim_customer_type6 ( customer_key INT PRIMARY KEY, customer_id INT, name VARCHAR(100), grade VARCHAR(20), -- Type 2: 히스토리 current_grade VARCHAR(20), -- Type 3: 현재 값 previous_grade VARCHAR(20), -- Type 3: 이전 값 effective_date DATE, expiration_date DATE, is_current BOOLEAN );

4. 단일 진실 공급원 (SSOT)

Single Source of Truth (SSOT)는 데이터 웨어하우스의 핵심 가치입니다. 모든 부서가 동일한 정의, 동일한 계산 로직, 동일한 데이터를 사용하도록 보장합니다.

SSOT가 없을 때 발생하는 문제

시나리오: "작년 매출이 얼마였나요?"

• 영업팀: 50억 원 (계약 체결일 기준)
• 재무팀: 48억 원 (수금 완료일 기준)
• 경영진: 49억 원 (발주일 기준)

→ 회의에서 혼란 발생! 어느 숫자가 맞나?

SSOT 적용 후:
데이터 웨어하우스에서 "매출 = 발주일 기준, VAT 포함"으로 정의 통일
모든 부서가 동일한 쿼리 사용 → 49억 원으로 합의

핵심 요점

  • 데이터 웨어하우스 4대 특징: 주제 지향적, 통합적, 시간 변화, 비휘발성
  • OLTP vs OLAP: 업무 처리 vs 분석, 정규화 vs 비정규화, ms vs 초
  • Inmon (Top-Down): EDW 먼저, 3NF, 높은 일관성, 긴 구축 기간
  • Kimball (Bottom-Up): 데이터 마트 먼저, 스타 스키마, 빠른 ROI
  • SCD Type 1: 덮어쓰기, 히스토리 없음
  • SCD Type 2: 새 행 추가, 완전한 히스토리 (가장 많이 사용)
  • SCD Type 3: 이전 값 컬럼, 제한적 히스토리
  • SCD Type 4: 히스토리 테이블 분리
  • SSOT: 단일 진실 공급원으로 데이터 일관성 보장
  • 히스토리 관리: 시점별 분석 가능, 감사 요구사항 충족
