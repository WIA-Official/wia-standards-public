🧊 Chapter 04

OLAP 및 큐브: 다차원 분석의 세계

1. OLAP이란?

OLAP(Online Analytical Processing)은 다차원 데이터를 빠르게 분석하기 위한 기술입니다. E.F. Codd가 1993년에 정의했으며, 비즈니스 인텔리전스(BI)의 핵심 구성 요소입니다.

1.1 OLAP vs OLTP (심화)

비교 항목 OLTP (Transactional) OLAP (Analytical)
목적 일상적 비즈니스 운영 의사결정 지원, 분석
쿼리 패턴 단순, 빈번 (SELECT, INSERT, UPDATE) 복잡, 애드혹 (JOIN, GROUP BY, 집계)
데이터 범위 현재 데이터 (최근 몇 개월) 과거 데이터 (수년~수십년)
사용자 수천~수만명 (일반 직원) 수십~수백명 (분석가, 경영진)
응답 시간 밀리초 (즉각) 초~분 (허용)
데이터 모델 정규화 (3NF) 비정규화 (Star Schema)
읽기 vs 쓰기 균형 (50:50) 읽기 중심 (99:1)
예시 쿼리 "주문 #12345의 상태는?" "2024년 Q1 카테고리별 매출 추이는?"

🎯 Codd의 OLAP 12가지 규칙 (FASMI)

E.F. Codd는 OLAP을 5가지 특성으로 요약했습니다:

  • F (Fast): 대부분의 쿼리가 5초 이내 응답
  • A (Analysis): 복잡한 분석 쿼리 지원
  • S (Shared): 다수 사용자의 동시 접근
  • M (Multidimensional): 다차원 뷰 제공
  • I (Information): 필요한 모든 데이터 접근

2. OLAP 큐브 (Cube)

OLAP 큐브는 데이터를 다차원 구조로 표현한 것입니다. 3차원 큐브가 가장 직관적이지만, 실제로는 4차원 이상의 "하이퍼큐브"를 사용합니다.

2.1 3차원 큐브 시각화

                    시간 (Time) ↑
                         Q4 ┌─────────────────────────┐
                            │ $850K  │ $920K  │ $780K │
                         Q3 ├─────────────────────────┤
                            │ $800K  │ $880K  │ $750K │
                         Q2 ├─────────────────────────┤
                            │ $780K  │ $860K  │ $720K │
                         Q1 └─────────────────────────┘
                              전자   의류    식품
                              └───제품 카테고리 (Product)───┘
                            /
                           /
                          / 지역 (Geography)
                         /  서울, 부산, 대구...

큐브의 각 셀 = 특정 조합의 측정값
예: "2024 Q3, 전자제품, 서울 = $850K 매출"

2.2 큐브의 구성 요소

  • 차원(Dimension): 분석 축 (시간, 제품, 지역, 고객 등)
  • 계층(Hierarchy): 차원 내 레벨 (연도 → 분기 → 월 → 일)
  • 측정값(Measure): 분석 대상 숫자 (매출, 수량, 이익 등)
  • 셀(Cell): 차원 조합의 측정값 (예: 2024-Q1, 전자, 서울 = $850K)
-- SQL로 큐브 개념 이해 -- 3차원 큐브: 시간 x 제품 x 지역 SELECT d.year, d.quarter, p.category, s.region, SUM(f.total_amount) AS total_sales FROM fact_sales f JOIN dim_date d ON f.date_key = d.date_key JOIN dim_product p ON f.product_key = p.product_key JOIN dim_store s ON f.store_key = s.store_key WHERE d.year = 2024 GROUP BY d.year, d.quarter, p.category, s.region ORDER BY d.quarter, p.category, s.region; -- 결과: -- year | quarter | category | region | total_sales -- 2024 | Q1 | 전자 | 서울 | 850000 -- 2024 | Q1 | 전자 | 부산 | 620000 -- 2024 | Q1 | 의류 | 서울 | 780000 -- ... -- 이 결과를 3차원 공간에 배치하면 → OLAP 큐브!

3. MOLAP, ROLAP, HOLAP

OLAP 구현 방식은 데이터를 어디에 저장하느냐에 따라 세 가지로 나뉩니다.

3.1 MOLAP (Multidimensional OLAP)

데이터를 전용 다차원 DB에 저장합니다. 큐브를 물리적으로 미리 계산하여 저장합니다.

✅ MOLAP 장점

  • 최고의 성능: 미리 계산된 집계, 밀리초 응답
  • 최적화된 저장: 압축률 90% 이상
  • 복잡한 계산: 시계열 분석, 비율 계산 최적화

⚠️ MOLAP 단점

  • 차원의 저주: 차원이 많아지면 큐브 크기 폭발적 증가
  • 실시간성 부족: 큐브 재계산에 시간 소요
  • 저장 공간: 모든 조합을 미리 계산 (희소성 문제)
  • 전용 기술: 특수한 DB 필요 (Microsoft SSAS, Oracle Essbase)

3.2 ROLAP (Relational OLAP)

데이터를 관계형 DB (RDBMS)에 Star Schema로 저장하고, 쿼리 시 실시간 계산합니다.

✅ ROLAP 장점

  • 확장성: 대량 데이터 처리 (페타바이트급)
  • 실시간성: 최신 데이터 즉시 반영
  • 유연성: 새 차원 추가 용이
  • 표준 기술: SQL, 기존 RDBMS 활용

⚠️ ROLAP 단점

  • 느린 성능: 복잡한 쿼리는 분 단위 소요 가능
  • DB 부하: 동시 사용자 증가 시 성능 저하
  • 복잡한 쿼리: 다차원 분석을 SQL로 표현하기 어려움

3.3 HOLAP (Hybrid OLAP)

MOLAP과 ROLAP의 장점을 결합합니다. 집계 데이터는 MOLAP에, 상세 데이터는 ROLAP에 저장합니다.

🎯 HOLAP 전략

  • Hot Data (MOLAP): 최근 1년, 자주 조회되는 집계
  • Cold Data (ROLAP): 과거 데이터, 상세 트랜잭션
  • 자동 라우팅: 쿼리에 따라 MOLAP/ROLAP 선택
  • 예시: Microsoft SSAS의 기본 모드

3.4 MOLAP vs ROLAP vs HOLAP 비교

항목 MOLAP ROLAP HOLAP
저장소 전용 다차원 DB 관계형 DB (RDBMS) 둘 다 사용
쿼리 성능 ⚡⚡⚡ 매우 빠름 ⚡ 느림 ⚡⚡ 빠름
확장성 제한적 (TB급) 매우 좋음 (PB급) 좋음
실시간성 낮음 (큐브 재계산 필요) 높음 (즉시 반영) 중간
저장 공간 높은 압축 (적음) 중간 중간
복잡도 높음 (전문 지식 필요) 낮음 (SQL 사용) 매우 높음
주요 도구 SSAS, Essbase, Palo Snowflake, BigQuery, 모든 RDBMS SSAS (Hybrid 모드)
사용 사례 재무 분석, 예산 대규모 DW, 실시간 분석 중간 규모 DW

4. MDX (Multidimensional Expressions)

MDX는 OLAP 큐브를 쿼리하기 위한 전용 쿼리 언어입니다. SQL이 2차원 테이블을 쿼리한다면, MDX는 N차원 큐브를 쿼리합니다.

4.1 MDX 기본 구조

-- 기본 MDX 쿼리 구조 SELECT [측정값] ON COLUMNS, -- 열에 표시할 내용 [차원1].[계층1].Members ON ROWS -- 행에 표시할 내용 FROM [큐브명] WHERE ([차원2].[계층2].[멤버]) -- 필터 (슬라이서) -- 실전 예시 1: 2024년 제품 카테고리별 매출 SELECT [Measures].[Sales Amount] ON COLUMNS, [Product].[Category].Members ON ROWS FROM [Sales Cube] WHERE [Time].[Year].[2024] -- 결과: -- Sales Amount -- 전자제품 $3,500,000 -- 의류 $2,800,000 -- 식품 $1,900,000

4.2 MDX 핵심 함수

-- 1. Children: 직속 하위 레벨 SELECT [Measures].[Sales Amount] ON COLUMNS, [Time].[Year].[2024].Children ON ROWS -- Q1, Q2, Q3, Q4 FROM [Sales Cube] -- 2. Descendants: 모든 하위 레벨 SELECT [Measures].[Sales Amount] ON COLUMNS, Descendants([Geography].[Country].[Korea], [Geography].[City]) ON ROWS FROM [Sales Cube] -- 결과: 서울, 부산, 대구, 인천... -- 3. TopCount: 상위 N개 SELECT [Measures].[Sales Amount] ON COLUMNS, TopCount([Product].[Product Name].Members, 10, [Measures].[Sales Amount]) ON ROWS FROM [Sales Cube] -- 매출 상위 10개 제품 -- 4. CrossJoin: 두 차원의 조합 SELECT [Measures].[Sales Amount] ON COLUMNS, CrossJoin( [Time].[Quarter].[2024].Children, [Product].[Category].Members ) ON ROWS FROM [Sales Cube] -- 결과: (Q1, 전자), (Q1, 의류), (Q1, 식품), (Q2, 전자), ... -- 5. Filter: 조건 필터링 SELECT [Measures].[Sales Amount] ON COLUMNS, Filter( [Product].[Product Name].Members, [Measures].[Sales Amount] > 100000 ) ON ROWS FROM [Sales Cube] -- 매출 10만 이상 제품만 -- 6. ParallelPeriod: 전년 동기 비교 WITH MEMBER [Measures].[Sales Last Year] AS ([Measures].[Sales Amount], ParallelPeriod([Time].[Year], 1, [Time].[CurrentMember])) MEMBER [Measures].[YoY Growth] AS ([Measures].[Sales Amount] - [Measures].[Sales Last Year]) / [Measures].[Sales Last Year] SELECT {[Measures].[Sales Amount], [Measures].[Sales Last Year], [Measures].[YoY Growth]} ON COLUMNS, [Time].[Quarter].Members ON ROWS FROM [Sales Cube] WHERE [Time].[Year].[2024]

5. OLAP 연산: Slice, Dice, Drill, Pivot

OLAP의 강력함은 다양한 다차원 연산에서 나옵니다. 이를 통해 데이터를 다양한 각도에서 빠르게 탐색할 수 있습니다.

5.1 Slice (슬라이스)

하나의 차원을 특정 값으로 고정하여 차원을 줄입니다. 3D 큐브 → 2D 슬라이스

-- SQL로 Slice 표현 SELECT p.category, s.region, SUM(f.total_amount) AS sales FROM fact_sales f JOIN dim_date d ON f.date_key = d.date_key JOIN dim_product p ON f.product_key = p.product_key JOIN dim_store s ON f.store_key = s.store_key WHERE d.year = 2024 AND d.quarter = 1 -- Time 차원을 2024-Q1로 고정 (Slice!) GROUP BY p.category, s.region;

5.2 Dice (다이스)

여러 차원을 특정 범위로 제한하여 부분 큐브를 생성합니다.

-- SQL로 Dice 표현 SELECT d.month, p.brand, s.city, SUM(f.total_amount) AS sales FROM fact_sales f JOIN dim_date d ON f.date_key = d.date_key JOIN dim_product p ON f.product_key = p.product_key JOIN dim_store s ON f.store_key = s.store_key WHERE d.year = 2024 AND d.quarter IN (1, 2) -- Time 제한 AND p.category IN ('전자', '의류') -- Product 제한 AND s.region IN ('서울', '경기') -- Geography 제한 GROUP BY d.month, p.brand, s.city; -- 큐브의 작은 조각만 추출 (Dice!)

5.3 Drill-Down (드릴 다운)

계층의 상위에서 하위로 이동하여 더 세밀한 데이터를 봅니다. 연도 → 분기 → 월 → 일

-- 1단계: 연도별 매출 SELECT d.year, SUM(f.total_amount) AS sales FROM fact_sales f JOIN dim_date d ON f.date_key = d.date_key GROUP BY d.year; -- 결과: 2024 → $12,000,000 -- 2단계: Drill-Down → 분기별 매출 SELECT d.year, d.quarter, SUM(f.total_amount) AS sales FROM fact_sales f JOIN dim_date d ON f.date_key = d.date_key WHERE d.year = 2024 GROUP BY d.year, d.quarter; -- 결과: 2024-Q1 → $2,800,000, Q2 → $3,100,000... -- 3단계: Drill-Down → 월별 매출 SELECT d.year, d.quarter, d.month, SUM(f.total_amount) AS sales FROM fact_sales f JOIN dim_date d ON f.date_key = d.date_key WHERE d.year = 2024 AND d.quarter = 1 GROUP BY d.year, d.quarter, d.month; -- 결과: 2024-01 → $920K, 02 → $880K, 03 → $1,000K

5.4 Roll-Up (롤 업)

Drill-Down의 반대로, 계층의 하위에서 상위로 이동하여 집계합니다. 일 → 월 → 분기 → 연도

5.5 Pivot (피벗)

차원의 축을 회전하여 다른 관점에서 데이터를 봅니다. 행 ↔ 열 교환

-- 원래: 카테고리(행) x 지역(열) SELECT p.category, SUM(CASE WHEN s.region = '서울' THEN f.total_amount ELSE 0 END) AS seoul, SUM(CASE WHEN s.region = '부산' THEN f.total_amount ELSE 0 END) AS busan FROM fact_sales f JOIN dim_product p ON f.product_key = p.product_key JOIN dim_store s ON f.store_key = s.store_key GROUP BY p.category; -- Pivot 후: 지역(행) x 카테고리(열) SELECT s.region, SUM(CASE WHEN p.category = '전자' THEN f.total_amount ELSE 0 END) AS electronics, SUM(CASE WHEN p.category = '의류' THEN f.total_amount ELSE 0 END) AS apparel FROM fact_sales f JOIN dim_product p ON f.product_key = p.product_key JOIN dim_store s ON f.store_key = s.store_key GROUP BY s.region; 
OLAP 연산 시각화:

원본 큐브 (3D):
       Time
        ↑
        │    [2024-Q1]
        │   ┌─────────┐
        │   │ 전자 의류│
        │   └─────────┘
        └──→ Product
       /
      / Geography

1. Slice (2024-Q1 고정):
       Product
        ↑
        │   전자  의류
        │  ┌───┬───┐
        │  │850│780│ 서울
        │  ├───┼───┤
        │  │620│550│ 부산
        └──→ Geography

2. Drill-Down (Q1 → 월):
   2024-Q1 → [Jan, Feb, Mar]
   Jan → [Week1, Week2, Week3, Week4]

3. Pivot (행↔열 교환):
   Before: Category(행) x Region(열)
   After:  Region(행) x Category(열)

6. 주요 OLAP 도구

6.1 Microsoft SQL Server Analysis Services (SSAS)

가장 널리 사용되는 OLAP 플랫폼으로, MOLAP/ROLAP/HOLAP 모두 지원합니다.

  • Tabular Model: DAX 쿼리, 인메모리 VertiPaq 엔진, Power BI와 통합
  • Multidimensional Model: MDX 쿼리, 전통적인 큐브
  • 장점: Excel, Power BI 완벽 통합, 엔터프라이즈급 보안
  • 단점: Windows 종속, 라이선스 비용

6.2 Oracle Essbase

1992년부터 사용된 레거시 OLAP의 대명사입니다.

  • 강점: 재무 계획(FP&A), What-If 분석, Write-Back 지원
  • 사용자: Fortune 500 기업, 금융권
  • 단점: 구식 UI, 높은 학습 곡선, 비싼 라이선스

6.3 Apache Kylin

Hadoop 기반 오픈소스 OLAP 엔진으로, 빅데이터에 최적화되어 있습니다.

  • 아키텍처: Hive로 큐브 빌드, HBase에 저장, REST API 제공
  • 성능: TB급 데이터를 초 단위로 쿼리
  • 사용 사례: 중국 전자상거래 (JD.com, Meituan)
  • 단점: 큐브 빌드 시간이 김, 실시간 업데이트 어려움

6.4 ClickHouse

Yandex가 개발한 초고속 컬럼 지향 OLAP DB입니다.

  • 성능: 초당 10억 행 처리, 100PB급 데이터 지원
  • 특징: ROLAP 방식, SQL 지원, 실시간 삽입
  • 사용 사례: 로그 분석, 실시간 대시보드, IoT
  • 장점: 오픈소스, 수평 확장, 압축률 10:1
-- ClickHouse 예시: 10억 행 쿼리가 0.2초 SELECT toStartOfMonth(event_time) AS month, event_type, count() AS event_count, uniq(user_id) AS unique_users FROM events WHERE event_time >= '2024-01-01' GROUP BY month, event_type ORDER BY month DESC, event_count DESC LIMIT 100; -- Materialized View로 실시간 집계 CREATE MATERIALIZED VIEW daily_summary ENGINE = SummingMergeTree() PARTITION BY toYYYYMM(event_date) ORDER BY (event_date, event_type) AS SELECT toDate(event_time) AS event_date, event_type, count() AS event_count, sum(revenue) AS total_revenue FROM events GROUP BY event_date, event_type;

6.5 클라우드 OLAP: Snowflake, BigQuery, Redshift

현대 클라우드 DW는 ROLAP 방식으로 OLAP 기능을 제공합니다.

특징 Snowflake BigQuery Redshift
아키텍처 컴퓨팅/스토리지 분리 서버리스 MPP (Massively Parallel)
성능 ⚡⚡⚡ ⚡⚡⚡ ⚡⚡
확장성 자동 스케일링 무제한 수동 리사이징
가격 모델 스토리지 + 컴퓨팅 시간 스캔 데이터량 노드 시간
OLAP 기능 윈도우 함수, QUALIFY ROLLUP, CUBE, 머신러닝 윈도우 함수, Spectrum
-- BigQuery의 CUBE 연산 (자동 소계/총계) SELECT product_category, region, SUM(sales) AS total_sales FROM sales_data WHERE year = 2024 GROUP BY CUBE(product_category, region) ORDER BY product_category, region; -- 결과: -- category | region | total_sales -- 전자 | 서울 | 850000 (세부) -- 전자 | 부산 | 620000 -- 전자 | NULL | 1470000 (카테고리 소계) -- 의류 | 서울 | 780000 -- 의류 | 부산 | 550000 -- 의류 | NULL | 1330000 -- NULL | 서울 | 1630000 (지역 소계) -- NULL | 부산 | 1170000 -- NULL | NULL | 2800000 (총계)

  10. "차원의 저주"란 무엇이며, MOLAP에서 왜 문제가 되나요?

🎯 핵심 요점

  • OLAP은 다차원 분석: 시간, 제품, 지역 등 여러 축으로 데이터 탐색
  • MOLAP = 성능, ROLAP = 확장성, HOLAP = 균형
  • OLAP 큐브: 차원(축) x 측정값(셀) = N차원 하이퍼큐브
  • MDX는 큐브 전용 쿼리 언어: SQL보다 다차원 분석에 적합
  • Slice/Dice: 큐브 자르기, Drill-Down/Roll-Up: 계층 이동
  • 현대 트렌드: MOLAP → ROLAP (클라우드 DW의 성능 향상)
  • SSAS: 엔터프라이즈 표준, ClickHouse: 오픈소스 고성능
  • 클라우드 DW: Snowflake, BigQuery는 ROLAP 방식으로 OLAP 구현
  • CUBE 연산: 자동으로 모든 조합의 소계/총계 생성
  • 弘益人間: 빠른 분석으로 더 나은 의사결정, 더 나은 세상 🌍
