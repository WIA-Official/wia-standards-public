1. Data Lakehouse: Data Lake + Data Warehouse

Data Lakehouse는 Data Lake의 유연성과 Data Warehouse의 성능을 결합한 차세대 아키텍처입니다. Databricks가 2020년 제안한 개념으로, 단일 플랫폼에서 모든 데이터 워크로드를 처리합니다.

1.1 기존 아키텍처의 문제

전통적 Two-Tier 아키텍처:

┌─────────────────┐         ┌─────────────────┐
│   Data Lake     │         │  Data Warehouse │
│   (S3/HDFS)     │  ETL    │  (Snowflake)    │
│─────────────────│ ──────▶ │─────────────────│
│ • Raw 데이터    │         │ • 정제된 데이터 │
│ • Parquet/JSON  │         │ • Star Schema   │
│ • ML/AI         │         │ • BI/Reporting  │
│ • 저렴 ($)      │         │ • 비쌈 ($$$$)   │
│ • 느림 ⚡       │         │ • 빠름 ⚡⚡⚡     │
└─────────────────┘         └─────────────────┘

문제점:
1. 데이터 중복 (Lake → WH 복사)
2. ETL 복잡도 증가

4. 데이터 일관성 문제
5. 거버넌스 어려움 (두 시스템 관리)

1.3 Delta Lake (Databricks)

Delta Lake는 ACID 트랜잭션을 지원하는 오픈소스 스토리지 레이어입니다. Parquet 파일 위에 트랜잭션 로그를 추가합니다.

-- Delta Lake 테이블 생성 CREATE TABLE sales_delta ( sale_id BIGINT, customer_id INT, total_amount DECIMAL(10,2), sale_date DATE ) USING DELTA PARTITIONED BY (sale_date); -- ACID 트랜잭션 (동시 쓰기 지원!) -- Writer 1: INSERT INTO sales_delta VALUES (1, 100, 50.00, '2024-03-15'); -- Writer 2 (동시에): UPDATE sales_delta SET total_amount = 55.00 WHERE sale_id = 1; -- ✓ 충돌 없이 처리됨 (Optimistic Concurrency Control) -- Time Travel SELECT * FROM sales_delta VERSION AS OF 10; -- 버전 10으로 조회 SELECT * FROM sales_delta TIMESTAMP AS OF '2024-03-15 10:00:00'; -- Schema Evolution (컬럼 추가) ALTER TABLE sales_delta ADD COLUMN discount_amount DECIMAL(10,2); -- MERGE (Upsert) - SCD Type 2 간편 구현 MERGE INTO sales_delta AS target USING sales_updates AS source ON target.sale_id = source.sale_id WHEN MATCHED THEN UPDATE SET * WHEN NOT MATCHED THEN INSERT *; -- OPTIMIZE (파일 압축) OPTIMIZE sales_delta; -- VACUUM (오래된 파일 정리) VACUUM sales_delta RETAIN 168 HOURS; -- 7일치만 보관

1.4 Apache Iceberg (Netflix, Apple)

Iceberg는 엔진 독립적 테이블 포맷입니다. Spark, Flink, Trino, Presto 모두 지원합니다.

-- Iceberg 테이블 생성 (Spark) CREATE TABLE sales_iceberg ( sale_id bigint, customer_id int, total_amount decimal(10,2), sale_date date ) USING iceberg PARTITIONED BY (days(sale_date)); -- Hidden Partitioning (자동 파티션 관리!) -- 사용자는 sale_date만 필터링, Iceberg가 자동으로 파티션 선택 SELECT * FROM sales_iceberg WHERE sale_date BETWEEN '2024-01-01' AND '2024-01-31'; -- 내부적으로: WHERE days(sale_date) IN (19723, 19724, ..., 19753) -- Snapshot Isolation (동시성) -- Reader는 항상 일관된 스냅샷 읽기 -- Writer는 Lock 없이 쓰기 가능 -- Time Travel (Snapshot ID로) SELECT * FROM sales_iceberg.snapshots; -- 스냅샷 목록 SELECT * FROM sales_iceberg FOR SYSTEM_TIME AS OF '2024-03-15 10:00:00';

2. Data Mesh: 분산형 데이터 아키텍처

Data Mesh는 도메인 중심의 분산형 데이터 아키텍처입니다. Zhamak Dehghani (ThoughtWorks)가 2019년 제안한 패러다임으로, 중앙 집중식 DW의 한계를 극복합니다.

2.1 Data Mesh 4대 원칙

🎯 Data Mesh 핵심 원칙

  1. Domain Ownership (도메인 소유권)
    • 각 도메인 팀이 자신의 데이터 소유
    • 예: Marketing 팀 → 고객 데이터, Sales 팀 → 주문 데이터
  2. Data as a Product (제품으로서의 데이터)
    • 데이터를 API처럼 제공
    • 품질, 문서, SLA 보장
  3. Self-serve Data Platform (셀프 서비스 플랫폼)
    • 도메인 팀이 직접 데이터 제품 생성
    • 인프라 자동화 (Terraform, K8s)
  4. Federated Computational Governance (연합 거버넌스)
    • 중앙 정책 + 도메인 자율성
    • 자동화된 컴플라이언스 검증

3. Streaming Data Warehouse

전통적인 DW는 배치 처리만 지원했습니다. 하지만 현대 비즈니스는 실시간을 요구합니다. Streaming DW는 배치와 스트리밍을 통합합니다.

3.2 Apache Flink + Iceberg

-- Flink SQL로 실시간 데이터 → Iceberg 테이블 CREATE TABLE kafka_source ( sale_id BIGINT, customer_id INT, total_amount DECIMAL(10,2), event_time TIMESTAMP(3), WATERMARK FOR event_time AS event_time - INTERVAL '5' SECOND ) WITH ( 'connector' = 'kafka', 'topic' = 'sales-events', 'properties.bootstrap.servers' = 'kafka:9092', 'format' = 'json' ); CREATE TABLE iceberg_sink ( sale_id BIGINT, customer_id INT, total_amount DECIMAL(10,2), sale_hour TIMESTAMP(3), hourly_sales DECIMAL(10,2) ) WITH ( 'connector' = 'iceberg', 'catalog-name' = 'my_catalog', 'warehouse' = 's3://my-bucket/warehouse' ); -- 실시간 집계 (1시간 윈도우) INSERT INTO iceberg_sink SELECT sale_id, customer_id, total_amount, TUMBLE_START(event_time, INTERVAL '1' HOUR) AS sale_hour, SUM(total_amount) OVER ( PARTITION BY customer_id ORDER BY event_time RANGE BETWEEN INTERVAL '1' HOUR PRECEDING AND CURRENT ROW ) AS hourly_sales FROM kafka_source; -- 이제 Iceberg 테이블을 Spark, Trino, Presto로 즉시 쿼리 가능! -- 배치 + 스트리밍 통합 완료!

4. AI/ML 통합

4.1 Feature Store

Feature Store는 ML 피처를 중앙 관리하는 시스템입니다. DW에서 피처 생성 → Feature Store 저장 → ML 모델 서빙

-- Feast (오픈소스 Feature Store) from feast import FeatureStore store = FeatureStore(repo_path=".") # 온라인 피처 조회 (저지연) entity_rows = [{"customer_id": 12345}] features = store.get_online_features( features=[ "customer_features:age", "customer_features:lifetime_value", "customer_features:avg_order_value" ], entity_rows=entity_rows ).to_dict() # 결과 (밀리초 응답): # { # "customer_id": [12345], # "age": [35], # "lifetime_value": [5000.00], # "avg_order_value": [125.50] # } # 오프라인 피처 (학습용, Snowflake에서 생성) training_data = store.get_historical_features( entity_df=entity_df, features=[ "customer_features:age", "customer_features:lifetime_value", "order_features:last_30d_orders" ] ).to_df() # Snowflake에서 피처 생성 후 자동 동기화

5. 2030년 전망

5.1 예측 1: Lakehouse 표준화

✅ 2030년에는...

  • 전통적인 DW 시장 점유율 < 30% (현재 70%)
  • Lakehouse 점유율 > 60%
  • Delta Lake / Iceberg / Hudi 중 하나가 표준 (현재 경쟁 중)
  • AWS, Azure, GCP 모두 네이티브 Lakehouse 제공
  • On-Premise DW 거의 소멸 (< 5%)

5.2 예측 2: AI 네이티브 DW

🎯 AI가 DW를 운영

  • 자동 최적화: AI가 쿼리 패턴 학습 → 자동 인덱스/파티션 생성
  • 자연어 쿼리: "2024년 매출 상위 10개 제품" → 자동 SQL 생성 실행
  • 이상 탐지: 데이터 품질 자동 검증, 이상치 알림
  • 자동 스케일링: 워크로드 예측 → 사전 확장
  • 비용 최적화: AI가 가장 저렴한 실행 계획 선택

5.3 예측 3: Quantum Data Warehouse

양자 컴퓨팅이 상용화되면, DW 쿼리 성능이 1000배 향상될 것입니다.

  • 복잡한 JOIN: 현재 10분 → 양자 DW 0.6초
  • 암호화 검색: 동형 암호화로 암호화된 상태로 쿼리
  • 최적화 문제: NP-Hard 문제를 실시간 해결

5.4 예측 4: Edge Data Warehouse

IoT 시대에는 엣지에서 직접 분석이 필요합니다.

-- 2030년 Edge DW 시나리오 -- 자율주행차에서 직접 데이터 집계 -- 5G/6G로 클라우드와 동기화 SELECT vehicle_id, AVG(speed) AS avg_speed, MAX(acceleration) AS max_acceleration FROM vehicle_sensors WHERE timestamp > NOW() - INTERVAL '1 minute' GROUP BY vehicle_id; -- 엣지에서 실행 (밀리초) → 중요한 이벤트만 클라우드 전송

5.5 예측 5: Decentralized DW (블록체인 기반)

블록체인 기술로 탈중앙화 데이터 웨어하우스가 등장합니다.

  • 데이터 주권: 개인이 자신의 데이터 소유 및 통제
  • 토큰 경제: 데이터 제공 시 토큰 보상
  • 투명성: 모든 쿼리와 변경 이력 불변 기록
  • 프라이버시: Zero-Knowledge Proof로 정보 노출 없이 검증
