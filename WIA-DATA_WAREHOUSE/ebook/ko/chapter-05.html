데이터 마트 설계: 부서별 분석 환경 구축

1. 데이터 마트란?

데이터 마트(Data Mart)는 특정 부서나 비즈니스 영역에 최적화된 데이터의 부분집합입니다. 전사적 데이터 웨어하우스(EDW)가 "백화점"이라면, 데이터 마트는 "전문 매장"입니다.

🎯 데이터 마트의 특징

  • 부서 중심: 마케팅, 재무, 영업 등 특정 도메인에 집중
  • 빠른 구축: 전사 DW보다 2-3개월로 단축
  • 빠른 쿼리: 필요한 데이터만 포함하여 성능 최적화
  • 사용자 친화적: 해당 부서의 비즈니스 용어 사용
  • 낮은 비용: 전체 DW보다 저렴한 구축/운영 비용

1.1 EDW vs 데이터 마트

    ┌─────────────────────────────────────────────────┐

2. Dependent vs Independent 데이터 마트

데이터 마트는 데이터 소스에 따라 두 가지 유형으로 나뉩니다.

2.1 Dependent Data Mart (종속형)

EDW에서 데이터를 추출하여 생성합니다. Kimball의 접근 방식입니다. 

원천 시스템 → ETL → EDW → ETL → Dependent Mart
(CRM, ERP)         (통합)      (부서별 뷰)

✅ Dependent Mart 장점

  • 일관성: 모든 마트가 같은 정의 사용 (Single Source of Truth)
  • 재사용: EDW의 Conformed Dimensions 활용
  • 거버넌스: 중앙 집중식 데이터 품질 관리
  • 확장성: 새 마트 추가가 쉬움

2.2 Independent Data Mart (독립형)

원천 시스템에서 직접 데이터를 추출합니다. EDW 없이 구축 가능합니다. 

원천 시스템 → ETL → Independent Mart (Marketing)
(CRM)

원천 시스템 → ETL → Independent Mart (Finance)
(ERP)

⚠️ Independent Mart 단점

  • 데이터 사일로: 마트 간 정의 불일치 ("고객"의 정의가 다름)
  • 중복 노력: 각 마트마다 ETL 재구현
  • 통합 어려움: 크로스 부서 분석 불가
  • 유지보수 비용: N개 마트 = N배 관리

결론: 장기적으로는 Dependent Mart 권장!

3. Aggregate Tables (집계 테이블)

Aggregate Tables은 미리 계산된 요약 데이터를 저장하여 쿼리 성능을 극적으로 향상시킵니다. "쿼리 시간을 100배 단축"하는 마법의 기술입니다.

3.1 왜 Aggregate Tables이 필요한가?

팩트 테이블이 10억 행이라면, "2024년 월별 매출"을 계산하려면 10억 행을 스캔해야 합니다. 하지만 월별 집계 테이블이 있다면? 단 12행만 읽으면 됩니다!

-- 원본 팩트 테이블 (10억 행) CREATE TABLE fact_sales ( sale_key BIGINT, date_key INT, customer_key INT, product_key INT, quantity INT, total_amount DECIMAL(10,2) ); -- 쿼리: SELECT SUM(total_amount) FROM fact_sales WHERE year = 2024 -- → 10억 행 스캔, 30초 소요 -- Aggregate 테이블: 월별 집계 (12행) CREATE TABLE agg_sales_monthly ( year_month DATE, product_category VARCHAR(100), region VARCHAR(100), total_quantity BIGINT, total_amount DECIMAL(18,2), order_count INT, unique_customers INT ); -- 쿼리: SELECT SUM(total_amount) FROM agg_sales_monthly WHERE YEAR(year_month) = 2024 -- → 12행 스캔, 0.1초 소요! 300배 빠름!

3.2 Aggregate Navigation

BI 도구는 자동으로 적절한 Aggregate 테이블을 선택합니다. 

사용자 쿼리: "2024년 Q1 카테고리별 매출"

┌─────────────────────────────────────────┐
│  BI Tool (Tableau, Power BI)            │
│  Query Optimizer                        │
└──────────────┬──────────────────────────┘
               │
               ▼ Aggregate Navigation
    ┌──────────┴──────────┐
    │  적절한 테이블 선택  │
    └──────────┬──────────┘
               │
    ┌──────────┼──────────┐
    │          │          │
    ▼          ▼          ▼
fact_sales  agg_monthly  agg_quarterly
(10억행)    (1000행)    (100행)
            ✓ 선택!

결과: agg_monthly 사용 → 100배 빠름

3.3 Aggregate 테이블 설계 예시

-- 1. 일별 집계 (Daily Aggregate) CREATE TABLE agg_sales_daily AS SELECT d.full_date, d.year, d.month, p.category, p.brand, s.region, s.city, COUNT(DISTINCT f.customer_key) AS unique_customers, COUNT(*) AS order_count, SUM(f.quantity) AS total_quantity, SUM(f.total_amount) AS total_sales, SUM(f.profit_amount) AS total_profit, AVG(f.total_amount) AS avg_order_value FROM fact_sales f JOIN dim_date d ON f.date_key = d.date_key JOIN dim_product p ON f.product_key = p.product_key JOIN dim_store s ON f.store_key = s.store_key GROUP BY d.full_date, d.year, d.month, p.category, p.brand, s.region, s.city; -- 2. 월별 집계 (Monthly Aggregate) CREATE TABLE agg_sales_monthly AS SELECT year_month, category, region, SUM(unique_customers) AS total_unique_customers, SUM(order_count) AS total_orders, SUM(total_quantity) AS total_quantity, SUM(total_sales) AS total_sales, SUM(total_profit) AS total_profit FROM agg_sales_daily GROUP BY year_month, category, region; -- 3. 연도별 집계 (Yearly Aggregate) CREATE TABLE agg_sales_yearly AS SELECT year, category, SUM(total_orders) AS total_orders, SUM(total_sales) AS total_sales, SUM(total_profit) AS total_profit FROM agg_sales_monthly GROUP BY year, category;

🎯 Aggregate 설계 원칙

  1. 80/20 법칙: 80% 쿼리가 사용하는 집계 수준 식별
  2. 계층 구조: 일 → 월 → 분기 → 연도 순으로 구축
  3. 선택적 생성: 모든 조합이 아닌 자주 사용되는 것만
  4. 유지보수 자동화: dbt, Airflow로 자동 갱신
  5. 투명성: 사용자는 Aggregate 존재를 몰라도 됨 (BI 도구가 처리)

4. Bridge Tables (브릿지 테이블)

차원 모델링에서 다대다(Many-to-Many) 관계를 처리하기 위한 테이블입니다. 예: 한 환자가 여러 진단을 받고, 한 진단은 여러 환자에게 해당

4.1 문제 상황: 다대다 관계

-- 문제: 환자 한 명이 여러 질병 진단을 받음 -- Patient #123: 당뇨병, 고혈압, 고지혈증 -- 잘못된 설계 1: 팩트에 진단을 직접 저장 ❌ CREATE TABLE fact_patient_visit_wrong ( visit_key INT, patient_key INT, diagnosis VARCHAR(200), -- "당뇨병, 고혈압, 고지혈증" (파싱 어려움!) total_cost DECIMAL(10,2) ); -- 잘못된 설계 2: 각 진단마다 행 생성 ❌ -- 문제: SUM(total_cost)하면 중복 집계! visit_key | patient_key | diagnosis_key | total_cost 1 | 123 | 당뇨병 | 500000 1 | 123 | 고혈압 | 500000 (중복!) 1 | 123 | 고지혈증 | 500000 (중복!) -- SUM(total_cost) = 1,500,000 (실제는 500,000)

4.2 해결책: Bridge Table

-- 1. 팩트 테이블 (진단 없이) CREATE TABLE fact_patient_visit ( visit_key INT PRIMARY KEY, patient_key INT, date_key INT, hospital_key INT, total_cost DECIMAL(10,2), -- diagnosis는 여기에 없음! ); -- 2. Bridge Table (다대다 관계 연결) CREATE TABLE bridge_visit_diagnosis ( visit_key INT, diagnosis_key INT, diagnosis_weight DECIMAL(5,4), -- 각 진단의 가중치 (합=1.0) PRIMARY KEY (visit_key, diagnosis_key) ); -- 3. 차원 테이블 (진단 마스터) CREATE TABLE dim_diagnosis ( diagnosis_key INT PRIMARY KEY, diagnosis_code VARCHAR(20), diagnosis_name VARCHAR(200), category VARCHAR(100) ); -- 데이터 예시: -- fact_patient_visit: -- visit_key | patient_key | total_cost -- 1 | 123 | 500000 -- bridge_visit_diagnosis: -- visit_key | diagnosis_key | diagnosis_weight -- 1 | D001 (당뇨) | 0.333 (1/3) -- 1 | D002 (고혈압) | 0.333 -- 1 | D003 (고지혈) | 0.334 -- 올바른 쿼리: SELECT diag.diagnosis_name, SUM(f.total_cost * b.diagnosis_weight) AS weighted_cost -- 가중치 곱하기! FROM fact_patient_visit f JOIN bridge_visit_diagnosis b ON f.visit_key = b.visit_key JOIN dim_diagnosis diag ON b.diagnosis_key = diag.diagnosis_key WHERE f.patient_key = 123 GROUP BY diag.diagnosis_name; -- 결과: -- diagnosis_name | weighted_cost -- 당뇨병 | 166,667 (500000 * 0.333) -- 고혈압 | 166,667 -- 고지혈증 | 166,666 -- 합계: 500,000 ✓ 정확!

✅ Bridge Table 활용 사례

  • 의료: 환자 ↔ 진단, 환자 ↔ 처방
  • 교육: 학생 ↔ 수강 과목
  • 은행: 계좌 ↔ 공동 소유자
  • 소매: 주문 ↔ 프로모션 (여러 프로모션 동시 적용)
  • 보험: 보험 청구 ↔ 관련 사고

5. Junk Dimensions (잡동사니 차원)

저카디널리티 플래그들을 하나의 차원 테이블로 묶은 것입니다. 팩트 테이블의 외래 키 개수를 줄이고, 차원 폭발(Dimension Explosion)을 방지합니다.

5.1 문제 상황: 플래그 남용

-- 잘못된 설계: 팩트에 플래그 직접 저장 ❌ CREATE TABLE fact_sales_bad ( sale_key BIGINT, date_key INT, customer_key INT, product_key INT, -- 수많은 플래그들... is_weekend BOOLEAN, -- 2가지 is_holiday BOOLEAN, -- 2가지 is_promotion BOOLEAN, -- 2가지 payment_method VARCHAR(20), -- 5가지 (신용카드, 체크카드...) delivery_type VARCHAR(20), -- 3가지 (일반, 빠른, 익일) gift_wrap BOOLEAN, -- 2가지 total_amount DECIMAL(10,2) ); -- 문제점: -- 1. 팩트 테이블이 너무 넓어짐 (80+ 컬럼) -- 2. 플래그별 차원 테이블 만들기엔 너무 작음 -- 3. 쿼리 가독성 저하

5.2 해결책: Junk Dimension

-- Junk Dimension: 플래그 조합을 하나의 차원으로 CREATE TABLE dim_transaction_junk ( junk_key INT PRIMARY KEY, -- 플래그 조합 is_weekend BOOLEAN, is_holiday BOOLEAN, is_promotion BOOLEAN, payment_method VARCHAR(20), delivery_type VARCHAR(20), gift_wrap BOOLEAN, -- 편의를 위한 계산 컬럼 transaction_type_desc VARCHAR(100) -- "Weekend + Holiday + Promotion + ..." ); -- 모든 조합을 미리 생성 (2 * 2 * 2 * 5 * 3 * 2 = 240개 행) INSERT INTO dim_transaction_junk VALUES (1, FALSE, FALSE, FALSE, 'Credit', 'Standard', FALSE, 'Weekday Standard Credit'), (2, FALSE, FALSE, FALSE, 'Credit', 'Standard', TRUE, 'Weekday Standard Credit + Gift'), (3, FALSE, FALSE, FALSE, 'Credit', 'Express', FALSE, 'Weekday Express Credit'), -- ... 240개 조합 -- 개선된 팩트 테이블 CREATE TABLE fact_sales ( sale_key BIGINT, date_key INT, customer_key INT, product_key INT, junk_key INT, -- 6개 플래그를 1개 FK로 압축! total_amount DECIMAL(10,2) ); -- 쿼리 예시: SELECT j.payment_method, j.delivery_type, SUM(f.total_amount) AS total_sales FROM fact_sales f JOIN dim_transaction_junk j ON f.junk_key = j.junk_key WHERE j.is_promotion = TRUE AND j.is_weekend = FALSE GROUP BY j.payment_method, j.delivery_type;

🎯 Junk Dimension 설계 원칙

  • 저카디널리티만: 각 플래그가 2-10개 정도 값
  • 조합 수 제한: 총 조합이 10,000개 이하 (일반적으로 100-1000개)
  • 비즈니스 관련성: 서로 관련된 플래그끼리 묶기
  • 미리 생성: 모든 조합을 ETL 시 사전 생성
  • 설명 컬럼 추가: transaction_type_desc로 가독성 향상

6. Degenerate Dimensions (퇴화 차원)

차원 테이블 없이 팩트 테이블에 직접 저장되는 차원 속성입니다. 주로 트랜잭션 번호, 영수증 번호 등 비즈니스 키가 해당됩니다.

6.1 왜 Degenerate Dimension인가?

-- 예: 주문 번호 (Order Number) -- 잘못된 설계: 주문 번호만 있는 차원 테이블 ❌ CREATE TABLE dim_order ( order_key INT PRIMARY KEY, order_number VARCHAR(50) UNIQUE -- 다른 속성이 없음! (날짜, 고객 등은 별도 차원) ); -- 문제: 불필요한 조인, 저장 공간 낭비 -- 올바른 설계: 팩트에 직접 저장 ✅ CREATE TABLE fact_order_line ( order_line_key BIGINT PRIMARY KEY, -- 차원 FK order_date_key INT, customer_key INT, product_key INT, -- Degenerate Dimension (차원 테이블 없음!) order_number VARCHAR(50), -- DD invoice_number VARCHAR(50), -- DD tracking_number VARCHAR(100), -- DD -- 측정값 quantity INT, total_amount DECIMAL(10,2) ); -- 사용 예시: SELECT f.order_number, c.customer_name, SUM(f.total_amount) AS order_total FROM fact_order_line f JOIN dim_customer c ON f.customer_key = c.customer_key WHERE f.order_number = 'ORD-2024-123456' GROUP BY f.order_number, c.customer_name;

✅ Degenerate Dimension 활용 사례

  • 주문 번호 (Order Number)
  • 송장 번호 (Invoice Number)
  • 티켓 번호 (Ticket Number)
  • 배송 추적 번호 (Tracking Number)
  • POS 영수증 번호 (Receipt Number)
  • 거래 ID (Transaction ID)

7. 실전 데이터 마트 예시: 마케팅 마트

-- 마케팅 마트: 캠페인 효과 분석 CREATE TABLE mart_marketing_campaign AS SELECT -- 차원 d.full_date, d.year, d.quarter, d.month, c.customer_segment, -- 'VIP', 'Regular', 'New' c.age_group, c.region, camp.campaign_name, camp.campaign_type, -- 'Email', 'SMS', 'Push', 'Display' camp.campaign_budget, -- 측정값 COUNT(DISTINCT f.customer_key) AS reached_customers, COUNT(DISTINCT CASE WHEN f.clicked = TRUE THEN f.customer_key END) AS clicked_customers, COUNT(DISTINCT CASE WHEN f.converted = TRUE THEN f.customer_key END) AS converted_customers, -- KPI 계산 COUNT(DISTINCT CASE WHEN f.clicked = TRUE THEN f.customer_key END) * 1.0 / NULLIF(COUNT(DISTINCT f.customer_key), 0) AS click_through_rate, COUNT(DISTINCT CASE WHEN f.converted = TRUE THEN f.customer_key END) * 1.0 / NULLIF(COUNT(DISTINCT CASE WHEN f.clicked = TRUE THEN f.customer_key END), 0) AS conversion_rate, SUM(f.revenue) AS total_revenue, SUM(f.revenue) - camp.campaign_budget AS roi_amount, (SUM(f.revenue) - camp.campaign_budget) / NULLIF(camp.campaign_budget, 0) AS roi_percentage FROM fact_campaign_interaction f JOIN dim_date d ON f.date_key = d.date_key JOIN dim_customer c ON f.customer_key = c.customer_key JOIN dim_campaign camp ON f.campaign_key = camp.campaign_key GROUP BY d.full_date, d.year, d.quarter, d.month, c.customer_segment, c.age_group, c.region, camp.campaign_name, camp.campaign_type, camp.campaign_budget; -- 마케터의 질문에 즉시 답변: -- "2024년 Q1에 VIP 고객 대상 이메일 캠페인의 ROI는?" SELECT campaign_name, reached_customers, converted_customers, conversion_rate, roi_percentage FROM mart_marketing_campaign WHERE year = 2024 AND quarter = 1 AND customer_segment = 'VIP' AND campaign_type = 'Email' ORDER BY roi_percentage DESC;

(위의 쿼리와 동일) FROM {{ ref('fact_campaign_interaction') }} {% if is_incremental() %} WHERE date_key > (SELECT MAX(date_key) FROM {{ this }}) {% endif %} ) SELECT * FROM campaign_stats -- Airflow DAG로 스케줄링 # dags/marketing_mart_refresh.py refresh_marketing_mart = BashOperator( task_id='refresh_marketing_mart', bash_command='dbt run --models tag:marketing', schedule_interval='0 3 * * *', # 매일 새벽 3시 )

  10. 데이터 마트를 dbt로 관리하는 것의 장점은 무엇인가요?

🎯 핵심 요점

  • 데이터 마트 = 부서별 전문 매장: 빠른 구축, 빠른 쿼리, 낮은 비용
  • Dependent Mart 권장: EDW 기반, 일관성 보장, 확장 용이
  • Aggregate Tables로 100배 빠른 쿼리: 미리 계산된 요약 데이터
  • Bridge Table로 다대다 해결: diagnosis_weight로 중복 집계 방지
  • Junk Dimension으로 팩트 간소화: 저카디널리티 플래그 묶기
  • Degenerate Dimension: 차원 테이블 없이 팩트에 직접 저장 (주문번호 등)
  • Aggregate Navigation: BI 도구가 자동으로 최적 집계 테이블 선택
  • 마트 자동화: dbt + Airflow로 일일 갱신
  • 80/20 법칙: 80% 쿼리를 지원하는 집계만 생성
  • 弘益人間: 부서별 최적화된 데이터로 빠른 의사결정 지원 🌍
