5.1 REST API Endpoints

Phase 2 of the WIA-DATA_VISUALIZATION Standard defines programmatic interfaces for interacting with visualization services. The REST API provides HTTP endpoints for rendering charts, retrieving specifications, exporting visualizations, and managing visualization resources. This API enables applications to integrate visualization capabilities without implementing renderers themselves.

The REST API follows standard HTTP conventions: GET for retrieving resources, POST for creating resources, PUT for updating, and DELETE for removing. All endpoints accept and return JSON. Error responses follow RFC 7807 Problem Details format. Authentication uses standard Bearer tokens or API keys. CORS headers support cross-origin requests from web applications.

Endpoint Method Purpose Request Body
/api/v1/charts/render POST Render a chart specification Chart spec JSON
/api/v1/charts/{id} GET Retrieve saved chart specification None
/api/v1/charts POST Save a chart specification Chart spec JSON
/api/v1/charts/{id}/export POST Export chart to image/PDF Export options
/api/v1/validate POST Validate chart specification Chart spec JSON
/api/v1/data/query POST Execute data query Query specification

Rendering Endpoint: The primary endpoint accepts a chart specification and returns rendered output. Response formats include SVG (for high-quality scalable graphics), PNG (for raster images), HTML (for interactive visualizations), and JSON (for data-driven clients). The format is specified via Accept header or query parameter.

POST /api/v1/charts/render
Content-Type: application/json
Accept: image/svg+xml

{
  "$schema": "https://wiastandards.com/schema/visualization/v1.0",
  "type": "bar",
  "data": { "values": [...] },
  "encoding": { ... }
}

Response: 200 OK
Content-Type: image/svg+xml

Validation Endpoint: Before rendering, specifications can be validated against the schema. The validation endpoint checks syntax, required fields, type correctness, and semantic validity (like ensuring referenced data fields exist). It returns detailed error messages for invalid specifications, enabling developers to catch issues early.

Export Endpoint: The export endpoint generates static representations of visualizations for inclusion in reports, presentations, or publications. It supports various formats (PNG, PDF, SVG), resolutions (for print vs. screen), and sizing options. High-DPI rendering ensures quality on modern displays and in print.

5.2 JavaScript Rendering API

For client-side rendering in web browsers, the WIA standard defines a JavaScript API that provides fine-grained control over visualization creation and interaction. This API is implemented by compliant rendering libraries and provides a consistent interface regardless of the underlying implementation (D3, Canvas, WebGL, etc.).

import { WIARenderer } from '@wia/visualization';

const renderer = new WIARenderer({
  container: '#chart-container',
  width: 800,
  height: 600
});

const spec = {
  type: 'bar',
  data: { values: data },
  encoding: {
    x: { field: 'category', type: 'ordinal' },
    y: { field: 'value', type: 'quantitative' }
  }
};

// Render the chart
renderer.render(spec)
  .then(chart => {
    console.log('Chart rendered successfully');

    // Add interaction
    chart.on('click', (event, datum) => {
      console.log('Clicked:', datum);
    });
  })
  .catch(error => {
    console.error('Rendering failed:', error);
  });

The JavaScript API is promise-based, supporting both then/catch and async/await patterns. It provides lifecycle hooks for customization: beforeRender, afterRender, beforeUpdate, afterUpdate. These hooks enable analytics tracking, custom animations, and integration with application state management.

API Method Purpose Returns
render(spec) Render a chart from specification Promise<Chart>
update(spec) Update existing chart with new spec Promise<Chart>
updateData(data) Update only data (faster than full update) Promise<Chart>
export(format, options) Export chart to image/SVG/PDF Promise<Blob>
destroy() Clean up chart and remove from DOM void
on(event, handler) Register event listener void

Responsive Rendering: The JavaScript API handles responsive design automatically. Charts can be configured to resize with their containers. The renderer intelligently adjusts label placement, font sizes, and layout to maintain readability across screen sizes. A resize observer detects container size changes and triggers re-rendering.

Performance Optimization: The API includes performance optimizations for common scenarios. Data updates use efficient diffing to avoid full re-renders. Large datasets trigger automatic aggregation or sampling. Progressive rendering displays basic structure immediately while loading details. These optimizations happen transparently, requiring no developer intervention.

5.3 Language SDKs

Data visualization happens across many programming ecosystems. Data scientists work in Python and R. Backend developers use Java, Go, or C#. Analysts use SQL and spreadsheet tools. The WIA standard provides SDKs for major languages, enabling programmatic chart generation in any environment.

Python SDK: The Python SDK integrates with popular data science tools like Pandas, NumPy, and Matplotlib. It provides both a low-level API (direct specification construction) and a high-level API with fluent builder patterns. Integration with Jupyter notebooks enables inline visualization with minimal code.

from wia_visualization import Chart

# High-level fluent API
chart = (Chart()
    .bar()
    .data(dataframe)
    .x('category')
    .y('value')
    .color('region')
    .title('Sales by Category and Region')
    .render())

# Or low-level specification API
spec = {
    'type': 'bar',
    'data': {'values': data},
    'encoding': {
        'x': {'field': 'category', 'type': 'ordinal'},
        'y': {'field': 'value', 'type': 'quantitative'}
    }
}
chart = Chart(spec).render()

R SDK: R users benefit from integration with ggplot2 and the tidyverse. The SDK includes conversion utilities that translate ggplot specifications to WIA format, enabling R users to leverage existing ggplot skills while gaining WIA's interoperability benefits. Shiny integration supports interactive dashboards.

Language Package Name Key Features Integration

All SDKs share common design principles: idiomatic APIs for each language, comprehensive type definitions where applicable, integration with language-specific data structures, and consistent naming conventions. A Python developer and a Java developer working with the WIA SDK will encounter similar concepts even if the syntax differs.

5.4 Export and Static Generation

While interactive visualizations are powerful, many use cases require static exports: reports, presentations, email attachments, and print publications. The WIA standard defines comprehensive export capabilities that generate high-quality static representations while preserving design fidelity.

Image Export: Charts can be exported to PNG, JPEG, or WebP formats at configurable resolutions. For screen display, 1x or 2x resolution (for Retina displays) is typical. For print, 300 DPI ensures quality. The export API handles resolution scaling, ensuring that text remains readable and visual proportions are preserved at any size.

Vector Export: SVG and PDF exports provide resolution-independent representations suitable for professional publications. SVG files can be further edited in design tools like Adobe Illustrator or Inkscape. PDF exports can be incorporated directly into LaTeX documents or PowerPoint presentations without quality loss.

Data Export: Beyond visual representations, charts can export their underlying data in CSV, Excel, or JSON formats. This enables viewers to perform their own analysis or create custom visualizations. Accessibility guidelines encourage providing data tables alongside visualizations.

// Export to PNG at 2x resolution
chart.export('png', {
  scale: 2,
  width: 1600,
  height: 1200,
  quality: 0.95
}).then(blob => {
  // Save to file or upload to server
  saveAs(blob, 'chart.png');
});

// Export to PDF with custom page size
chart.export('pdf', {
  pageSize: 'A4',
  orientation: 'landscape',
  margins: { top: 20, right: 20, bottom: 20, left: 20 }
}).then(blob => {
  saveAs(blob, 'report.pdf');
});

5.5 Embedding and Integration Patterns

Visualizations rarely exist in isolation—they're embedded in dashboards, applications, reports, and workflows. The WIA standard defines patterns for embedding that balance flexibility with ease of use.

iframe Embedding: The simplest embedding approach uses iframes. A visualization server hosts rendered charts; external sites embed them via iframe tags. This approach provides strong isolation (the visualization can't interfere with the host page) and works without JavaScript. However, it limits communication between host and visualization.

JavaScript Widget: For tighter integration, the WIA standard defines a web component that can be embedded in any HTML page. The component loads the rendering library, renders the visualization, and provides an API for programmatic control. This enables two-way communication: the host can update the visualization, and the visualization can notify the host of user interactions.

<!-- Simple declarative embedding -->
<wia-chart
  spec-url="https://example.com/charts/revenue-analysis.json"
  width="800"
  height="600"
></wia-chart>

<!-- Programmatic control -->
<script>
const chart = document.querySelector('wia-chart');
chart.addEventListener('chart-click', (event) => {
  console.log('Clicked:', event.detail.data);
});

// Update data
chart.updateData(newData);
</script>

Framework Integration: The standard provides integration packages for popular frameworks. React hooks (useChart, useChartData) manage chart lifecycle within component trees. Vue components provide reactive bindings. Angular directives support dependency injection and change detection. These integrations make charts first-class citizens in modern web applications.

Framework Integration Package Key Features

Vue @wia/vue-visualization Reactive props, composition API
Angular @wia/angular-visualization Directives, services, RxJS integration

5.6 Authentication and Authorization

Enterprise deployments require secure access control for visualization APIs. The WIA standard supports standard authentication mechanisms and defines authorization patterns for controlling access to visualizations and underlying data.

Authentication: The API supports multiple authentication mechanisms: API keys for service-to-service communication, OAuth 2.0 for user-delegated access, and JWT tokens for stateless authentication. Organizations can choose the mechanism that fits their security requirements and infrastructure.

Authorization: Once authenticated, users' permissions determine what they can access. Role-based access control (RBAC) defines roles like viewer, analyst, and admin with different capabilities. Attribute-based access control (ABAC) provides finer granularity, enabling policies like "users can only view charts for their region."

Data Security: Visualizations often expose sensitive data. The standard defines row-level security patterns where data filtering based on user identity happens automatically. A sales dashboard might show global data to executives but regional data to regional managers—using the same visualization specification with different data filtering.

5.7 Caching and Performance

Rendering visualizations can be computationally expensive, especially for complex charts or large datasets. The WIA API standard includes caching specifications that enable performance optimization while ensuring data freshness.

HTTP Caching: Rendered visualizations include standard HTTP cache headers (Cache-Control, ETag, Last-Modified). Static visualizations can be cached aggressively. Dynamic visualizations specify appropriate TTL based on data update frequency. CDNs can cache rendered images, reducing load on origin servers.

Specification Caching: Chart specifications themselves are often reused. The API supports conditional requests (If-None-Match, If-Modified-Since) that avoid retransmitting unchanged specifications. Clients cache specifications locally and only re-fetch when versions change.

Data Caching: For visualizations backed by databases, query results can be cached. The standard defines cache key strategies based on query parameters, time windows, and user context. Intelligent cache invalidation ensures users see fresh data while minimizing database load.

Cache Type Scope Invalidation Performance Gain
Rendered image Per specification + data version Data changes, TTL expiry Very High (avoid rendering)
Specification Per chart ID Specification updates Medium (avoid network)
Query results Per query + parameters Data updates, time-based High (avoid database)
Client-side Browser/application memory Manual or automatic Very High (avoid everything)

5.8 Chapter Summary

✅ Key Takeaways:

  1. REST API enables server-side rendering: Standard HTTP endpoints for rendering, validation, and export support application integration without client-side rendering.
  2. JavaScript API provides client-side control: Promise-based API with lifecycle hooks enables fine-grained control over rendering and interaction in web browsers.
  3. Language SDKs support diverse ecosystems: Python, R, Java, Go, and other SDKs provide idiomatic APIs for programmatic chart generation in any environment.
  4. Export capabilities support static use cases: PNG, PDF, SVG, and data exports enable visualizations in reports, presentations, and publications.
  5. Security and performance are first-class concerns: Authentication, authorization, and intelligent caching ensure enterprise-grade deployments.
