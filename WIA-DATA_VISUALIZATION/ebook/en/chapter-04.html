📊 Chapter 4: Phase 1 - Data Format

4.1 Chart Specification Schema

Phase 1 of the WIA-DATA_VISUALIZATION Standard establishes the foundational data format for describing visualizations. At its core is the chart specification schema—a JSON-based declarative format that describes what should be visualized, how data maps to visual elements, and what styling should be applied. This specification is designed to be human-readable, machine-processable, and portable across different rendering engines.

The chart specification follows a hierarchical structure with several key sections. The top level includes metadata (version, title, description), the chart type, data sources, visual encodings (how data maps to visual channels), scales, axes, legends, and styling. This structure separates concerns while maintaining relationships, making specifications both modular and comprehensible.

Here's the basic structure of a WIA chart specification:

{
  "$schema": "https://wiastandards.com/schema/visualization/v1.0",
  "version": "1.0",
  "title": "Quarterly Revenue by Region",
  "description": "Revenue performance across regions for Q1-Q4 2024",
  "type": "bar",
  "data": { /* data source specification */ },
  "encoding": { /* visual mappings */ },
  "scales": { /* scale definitions */ },
  "axes": { /* axis configurations */ },
  "legend": { /* legend specification */ },
  "style": { /* visual styling */ },
  "accessibility": { /* accessibility metadata */ }
}

The schema is defined using JSON Schema, enabling automatic validation of chart specifications. Tools can check that required fields are present, that values are of correct types, and that relationships between fields are valid. This validation catches errors early and ensures that specifications are well-formed before rendering.

Schema Section Purpose Required
$schema Version identifier for the specification format Yes
type Chart type (bar, line, scatter, pie, etc.) Yes
data Data source (inline, URL, or reference) Yes
encoding Mappings from data fields to visual channels Yes
scales Scale definitions (linear, log, time, categorical) No (defaults provided)
axes Axis configurations and labels No (defaults provided)
style Colors, fonts, spacing No (defaults provided)
accessibility Alt text, descriptions, ARIA labels Yes (for compliance)

The specification schema supports extensive customization while providing sensible defaults. A minimal specification with just type, data, and encoding will render correctly using default scales, axes, and styling. Advanced users can override every aspect of the visualization's appearance and behavior.

4.2 Data Source Specifications

The data section of a chart specification defines where visualization data comes from and its structure. WIA-DATA_VISUALIZATION supports multiple data source types to accommodate different use cases, from inline data for simple charts to remote data sources for enterprise dashboards.

Inline Data: For simple visualizations or examples, data can be embedded directly in the specification as a JSON array of objects. Each object represents one data point with named fields. This approach makes specifications self-contained and portable but is impractical for large datasets.

"data": {
  "values": [
    { "quarter": "Q1", "region": "North", "revenue": 125000 },
    { "quarter": "Q1", "region": "South", "revenue": 98000 },
    { "quarter": "Q2", "region": "North", "revenue": 142000 },
    { "quarter": "Q2", "region": "South", "revenue": 108000 }
  ]
}

URL Data: For larger datasets, the specification can reference data by URL. The renderer fetches data from the specified endpoint. Formats include JSON, CSV, and TSV. URL-based data enables separation of specification from data, allowing the same chart to visualize different datasets.

"data": {
  "url": "https://api.example.com/revenue/quarterly",
  "format": { "type": "json" }
}

Named Data Sources: In dashboard contexts with multiple charts sharing data, named data sources avoid redundant fetching. Charts reference a shared data source by name. The dashboard framework handles fetching and caching, ensuring efficiency and consistency.

Data Format Use Case Limitations
Inline JSON Small datasets, examples, embedded charts Not suitable for large datasets (>1000 rows)
JSON URL Medium datasets, API-backed charts Network latency, requires CORS support
CSV/TSV URL Simple tabular data, data exports Limited type information, text-based parsing
Named reference Dashboard contexts, shared data Requires dashboard framework support
Database query Real-time enterprise data (Phase 2) Security considerations, query optimization

The data schema also supports type annotations that inform how data should be interpreted. Field types include quantitative (numeric values), ordinal (ordered categories), nominal (unordered categories), temporal (dates and times), and geographic (locations). Explicit typing enables automatic selection of appropriate scales and formats.

4.3 Visual Encoding Mappings

The encoding section defines how data fields map to visual channels—position, color, size, shape, and other visual properties. This mapping is the heart of the visualization specification, translating abstract data into concrete visual representations that humans can perceive and interpret.

WIA-DATA_VISUALIZATION supports a comprehensive set of visual channels, each suited for encoding different types of information. Position (x, y coordinates) is the most effective channel for quantitative comparisons. Color can encode categorical or quantitative data. Size works well for quantitative data but is less precise than position. Shape is effective for categorical distinctions but limited to small numbers of categories.

"encoding": {
  "x": {
    "field": "quarter",
    "type": "ordinal",
    "axis": { "title": "Quarter" }
  },
  "y": {
    "field": "revenue",
    "type": "quantitative",
    "axis": { "title": "Revenue ($)", "format": "$,.0f" }
  },
  "color": {
    "field": "region",
    "type": "nominal",
    "scale": { "scheme": "category10" }
  }
}
Visual Channel Best For Data Types Effectiveness
Position (x, y) Precise quantitative comparisons Quantitative, ordinal, temporal Highest
Color (hue) Categorical distinctions Nominal, ordinal High (limited categories)
Color (saturation/lightness) Quantitative intensity Quantitative Medium
Size Quantitative magnitude Quantitative Medium (less precise than position)
Shape Categorical distinctions Nominal (≤7 categories) Medium
Opacity Emphasis, density Quantitative, binary Low (subtle)

Each encoding specification includes the field name, data type, and optional customizations like scale overrides, axis properties, and legends. The renderer uses this information to construct appropriate scales (the functions that map data values to visual values) and render the corresponding visual elements.

Advanced encoding features include conditional encodings (change encoding based on data values), multi-field encodings (combine multiple fields), and calculated fields (derive new fields from existing data). These features enable sophisticated visualizations while maintaining declarative simplicity.

4.4 Scale Definitions

Scales are functions that map from data domains (the range of values in the data) to visual ranges (the range of visual values like pixel positions or colors). Understanding scales is essential for creating accurate, effective visualizations. The WIA standard defines several scale types, each optimized for different data characteristics.

Linear Scales: Map quantitative data to visual values using a linear function. A value halfway through the data range maps to a position halfway through the visual range. Linear scales are appropriate when equal differences in data should produce equal differences in visual representation. Most numeric data uses linear scales.

Logarithmic Scales: Map data using logarithmic transformation, compressing large values and expanding small values. Logarithmic scales are essential for data spanning multiple orders of magnitude (like population sizes from thousands to billions) where a linear scale would make small values invisible.

Time Scales: Specialized scales for temporal data that understand calendar structures. Time scales correctly handle irregular intervals (not all months have the same number of days), time zones, and human-friendly formatting. They enable visualizations like time series charts and timelines.

Ordinal Scales: Map discrete ordered categories to visual positions. Unlike linear scales, ordinal scales don't interpolate between values. They're used for data like "small, medium, large" or "Q1, Q2, Q3, Q4" where the ordering matters but numerical differences don't.

Scale Type Data Domain Use Cases Example
Linear Continuous quantitative Revenue, temperature, population Sales over time
Log Wide-range quantitative Country populations, earthquake magnitudes Log-scale scatter plot
Power/Sqrt Quantitative with outliers Area/volume representations Bubble chart sizing
Time Temporal (dates, times) Time series, timelines Stock price chart
Ordinal Ordered categories Size categories, quarters, grades Bar chart by quarter
Categorical Nominal categories Regions, product types Category color mapping

Color scales deserve special attention due to their perceptual complexity. Sequential color scales (light to dark of a single hue) work for ordered quantitative data. Diverging scales (two hues diverging from a neutral center) work for data with a meaningful midpoint like zero. Categorical scales assign distinct hues to categories. The WIA standard includes color scale specifications that ensure accessibility and perceptual uniformity.

4.5 Color Palettes and Accessibility

Color is one of the most powerful and most problematic aspects of visualization. The WIA standard takes color seriously, providing curated palettes that ensure both aesthetic appeal and accessibility. All standard palettes are tested for colorblind safety and perceptual uniformity.

Categorical Palettes: For distinguishing nominal categories, the standard provides palettes with maximally distinct hues. The default categorical palette includes 10 colors carefully chosen to be distinguishable even for colorblind viewers. When more than 10 categories are needed, the specification recommends using alternative encodings (faceting, filtering) rather than adding more colors.

Sequential Palettes: For ordered quantitative data, sequential palettes progress from light to dark (or vice versa) in a single hue. The WIA standard includes sequential palettes in purple (the standard's brand color), blue, green, and orange. These palettes are perceptually uniform—equal steps in data values produce equal perceptual steps in color.

Diverging Palettes: For data with a meaningful center point (like temperature anomalies or profit/loss), diverging palettes use two contrasting hues that diverge from a neutral center. The standard provides blue-to-red, purple-to-green, and brown-to-teal palettes. Critically, these avoid red-green combinations that colorblind users cannot distinguish.

Palette Name Type Colors Colorblind Safe
category10 Categorical 10 distinct hues Yes
purples Sequential 9 shades (light to dark) Yes
blues Sequential 9 shades (light to dark) Yes
blue-orange Diverging 11 colors (blue ← neutral → orange) Yes
purple-green Diverging 11 colors (purple ← neutral → green) Yes

Beyond named palettes, the standard supports custom color specifications with required accessibility validation. Custom colors must pass WCAG AAA contrast requirements for text and maintain distinguishability for common types of color vision deficiency. Tools implementing the standard include validators that check these requirements.

4.6 Typography and Text Rendering

Typography significantly impacts visualization readability and professionalism. The WIA standard establishes guidelines for fonts, sizes, weights, and text rendering to ensure consistency and accessibility across implementations.

Font Families: The standard recommends system font stacks that work across platforms without requiring custom font downloads. For body text and labels, sans-serif fonts (Helvetica, Arial, system-ui) provide clean readability. For code or data values requiring monospace alignment, Consolas or Monaco are recommended. The specification can override these with custom fonts when brand consistency requires it.

Text Sizing: Minimum text sizes ensure readability. Axis labels must be at least 11px. Chart titles should be 16-18px. Annotations should be 12-14px. These minimums apply to rendered output; the specification uses relative sizing (em or rem units) that scales appropriately based on context.

Text Contrast: All text must meet WCAG AAA contrast requirements (7:1 for body text, 4.5:1 for large text). The standard provides color combinations that meet these requirements. Dark text on light backgrounds or light text on dark backgrounds both work; what matters is sufficient contrast ratio.

Label Placement: The specification defines intelligent label placement rules. Axis labels rotate when horizontal labels would overlap. Pie chart labels use leader lines when slices are too small for internal labels. Bar chart labels position inside bars for large values and outside for small values. These rules ensure labels are always readable.

4.7 Accessibility Metadata

Accessibility is not optional in the WIA standard—it's required for compliance. Every chart specification must include accessibility metadata that enables screen reader users, keyboard navigators, and others with disabilities to access the information conveyed by the visualization.

"accessibility": {
  "title": "Quarterly Revenue by Region",
  "description": "Bar chart showing revenue performance across North and South regions for each quarter of 2024. North region consistently outperforms South, with both regions showing growth from Q1 to Q4.",
  "keyInsights": [
    "North region revenue increased 35% from Q1 ($125k) to Q4 ($168k)",
    "South region showed 28% growth from Q1 ($98k) to Q4 ($125k)",
    "Largest quarter-over-quarter growth was Q1-Q2 in North region (14%)"
  ],
  "dataTable": {
    "summary": "Quarterly revenue data by region",
    "headers": ["Quarter", "North Region", "South Region"],
    "includeInDom": true
  }
}

The accessibility section includes a descriptive title, a longer description explaining what the visualization shows, key insights that summarize the main takeaways, and specifications for an accessible data table representation. Renderers use this metadata to generate alt text, ARIA labels, and semantic HTML structures that assistive technologies can interpret.

Accessibility Feature Specification Field Purpose
Alt text accessibility.title Brief description for screen readers
Long description accessibility.description Detailed explanation of visualization
Key insights accessibility.keyInsights Main takeaways without visual interpretation
Data table accessibility.dataTable Tabular representation for screen readers
Keyboard navigation accessibility.keyboard Keyboard shortcuts and focus order

4.8 Chapter Summary

✅ Key Takeaways:

  1. Chart specifications are JSON-based and declarative: The schema defines what to visualize without prescribing how, enabling portability across renderers.
  2. Multiple data source types support different use cases: Inline data for simple charts, URL references for larger datasets, and named sources for dashboard contexts.
  3. Visual encodings map data to visual channels: Position, color, size, and other channels translate abstract data into perceivable visual representations.
  4. Scales handle different data types appropriately: Linear, logarithmic, time, ordinal, and categorical scales ensure accurate visual representation.
  5. Accessibility is mandatory, not optional: Every specification must include metadata enabling screen readers, keyboard navigation, and other assistive technologies.
