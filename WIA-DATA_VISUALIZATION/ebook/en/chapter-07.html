7.1 Dashboard Composition Frameworks

Phase 4 of the WIA-DATA_VISUALIZATION Standard addresses enterprise integration needs, starting with dashboard composition—the process of combining multiple visualizations into cohesive, interactive dashboards. Modern dashboards aren't just collections of charts; they're interactive applications where visualizations coordinate, share data, and respond to user interactions collectively.

The WIA dashboard framework defines how visualizations are arranged in layouts, how they share data sources, how interactions propagate between charts, and how dashboard state is managed and persisted. The framework is implementation-agnostic, working with web-based dashboards, native applications, and embedded analytics scenarios.

Layout System: Dashboards arrange visualizations in grids, flows, or custom layouts. The framework supports responsive layouts that adapt to screen sizes automatically. Visualizations can span multiple grid cells, be grouped into tabs or accordions, and be organized hierarchically. Layout specifications use familiar CSS Grid and Flexbox concepts, making them accessible to web developers.

{
  "dashboard": {
    "title": "Sales Performance Dashboard",
    "layout": {
      "type": "grid",
      "columns": 12,
      "gap": "16px",
      "areas": [
        { "id": "kpi-revenue", "row": "1", "col": "1 / 5" },
        { "id": "kpi-growth", "row": "1", "col": "5 / 9" },
        { "id": "kpi-customers", "row": "1", "col": "9 / 13" },
        { "id": "chart-trend", "row": "2", "col": "1 / 9" },
        { "id": "chart-region", "row": "2", "col": "9 / 13" },
        { "id": "table-details", "row": "3", "col": "1 / 13" }
      ]
    }
  }
}
Layout Type Use Case Characteristics
Grid Standard dashboards, reports Fixed rows/columns, responsive breakpoints
Flow Narrative dashboards, storytelling Vertical scroll, sequential presentation
Tabs Multi-page dashboards, categorized views Space-efficient, grouped content
Canvas Custom infographics, presentations Absolute positioning, design freedom
Masonry Mixed-size visualizations Pinterest-style, variable heights

Shared Data Sources: Rather than each visualization fetching data independently, dashboard-level data sources are defined once and referenced by multiple visualizations. This improves performance (data fetched once), ensures consistency (all charts show the same data version), and simplifies maintenance (update data source in one place).

Cross-Chart Filtering: The framework defines how selections and filters in one visualization affect others. When a user selects a region in a bar chart, line charts filter to that region, tables highlight relevant rows, and KPI cards recalculate values. These interactions are specified declaratively, without custom JavaScript code.

7.2 BI Tool Integration

Many organizations have significant investments in business intelligence platforms like Tableau, Power BI, Qlik, and Looker. The WIA standard provides integration paths that enable these platforms to consume and produce WIA-compliant visualizations, fostering interoperability without requiring migration.

Export from BI Tools: Integration plugins for major BI platforms enable exporting visualizations to WIA format. A Tableau workbook can be exported as a set of WIA chart specifications, enabling those visualizations to be rendered by any WIA-compliant renderer. This provides an escape hatch from vendor lock-in and enables sharing visualizations with stakeholders who don't have platform licenses.

Import into BI Tools: Conversely, WIA chart specifications can be imported into BI platforms where supported. A dashboard created with open-source WIA tools can be imported into Tableau for integration with enterprise authentication, scheduling, and data governance. This flexibility enables organizations to choose the right tool for each use case.

BI Platform Integration Method Export Support Import Support
Tableau Extension API, export plugin Partial (common chart types) Via custom data source
Power BI Custom visual, export utility Partial (common chart types) Via custom visual API
Looker Custom visualization API Good (LookML translation) Via custom visualizations
Qlik Sense Extension objects Partial (common chart types) Via extension framework
Apache Superset Plugin architecture Excellent (open source) Excellent (open source)

Limitations and Trade-offs: Not all features translate perfectly between platforms. Tableau's calculation language differs from WIA transforms. Power BI's DAX measures don't have direct WIA equivalents. The integration strategy accepts these limitations, focusing on common chart types and features that translate well while documenting limitations clearly.

Reference Implementations: The WIA project provides reference implementations for key integrations: a Tableau Web Data Connector that serves WIA data sources, a Power BI custom visual that renders WIA specifications, and a Looker custom visualization library. These reference implementations demonstrate integration patterns and provide starting points for custom integrations.

7.3 Application Embedding Patterns

Embedding analytics into applications—often called "embedded analytics"—is one of the fastest-growing visualization use cases. SaaS applications embed dashboards to provide customers with usage analytics. Internal applications embed visualizations to surface operational metrics. The WIA standard defines patterns that make embedding straightforward and secure.

White-Label Embedding: Embedded visualizations should match the host application's look and feel, not advertise the visualization platform. The WIA framework supports comprehensive theming that adapts visualizations to any brand. Themes control colors, fonts, spacing, borders, and every visual aspect. The embedding application provides a theme configuration, and all visualizations render consistently.

const wiaTheme = {
  colors: {
    primary: '#1E40AF',      // Application's primary color
    background: '#FFFFFF',
    text: '#1F2937',
    border: '#E5E7EB'
  },
  fonts: {
    body: 'Inter, sans-serif',
    headings: 'Poppins, sans-serif',
    monospace: 'JetBrains Mono, monospace'
  },
  spacing: {
    unit: 4,  // Base spacing unit in pixels
    chartPadding: 20,
    labelGap: 8
  }
};

Multi-Tenant Isolation: SaaS applications serve multiple customers (tenants) from a single instance. Embedded visualizations must enforce tenant isolation, ensuring Customer A can't see Customer B's data. The framework supports tenant-context injection, where the embedding application provides tenant identity and the visualization service automatically filters data accordingly.

Performance at Scale: Applications may embed hundreds or thousands of visualizations across many user sessions. The framework includes performance patterns: lazy loading (render only visible charts), virtualization (render only charts in viewport), caching strategies, and connection pooling. These patterns enable embedded analytics to scale to enterprise volumes.

7.4 Enterprise Authentication and SSO

Enterprise deployments require integration with corporate authentication systems. Employees shouldn't maintain separate credentials for visualization systems. The WIA standard supports standard enterprise authentication protocols, enabling seamless integration with existing identity infrastructure.

SAML 2.0 Integration: Security Assertion Markup Language (SAML) is widely used for enterprise single sign-on. Users authenticate to their corporate identity provider (IdP) like Okta, Azure AD, or Ping Identity. The IdP issues SAML assertions that the WIA visualization service validates. Users experience seamless access without additional login prompts.

OAuth 2.0 and OIDC: OpenID Connect (OIDC), built on OAuth 2.0, provides modern authentication and authorization. Applications redirect users to their OIDC provider for authentication, receive ID tokens that identify the user, and access tokens for API authorization. The WIA API validates these tokens and authorizes access based on scopes and claims.

LDAP/Active Directory: For on-premises deployments, integration with LDAP directories and Active Directory enables authentication using corporate credentials. The visualization service queries LDAP for user validation and group membership, enabling role-based access control based on organizational structure.

// OIDC authentication flow
1. User accesses dashboard → redirect to OIDC provider
2. User authenticates with corporate credentials
3. OIDC provider redirects back with authorization code
4. Application exchanges code for tokens:
   {
     "id_token": "eyJhbG...",      // User identity
     "access_token": "eyJhbG...",  // API authorization
     "refresh_token": "eyJhbG..."  // Token renewal
   }
5. Subsequent API requests include access token
6. Visualization service validates token and authorizes access

7.5 Role-Based Access Control

Not all users should access all visualizations or all data. Enterprise deployments require fine-grained access control that reflects organizational roles and responsibilities. The WIA framework defines role-based access control (RBAC) patterns for visualization resources.

Resource Hierarchy: Visualization resources are organized hierarchically: organizations contain workspaces, workspaces contain dashboards, dashboards contain visualizations. Permissions can be granted at any level and inherited downward. Granting workspace access implicitly grants access to all dashboards within that workspace.

Permission Model: The standard defines a simple permission model: View (see visualizations), Edit (modify specifications), Admin (manage access). These permissions apply to different resource types (dashboards, data sources, themes). Organizations can extend this model with custom permissions as needed.

Row-Level Security: Beyond controlling who can see which visualizations, row-level security controls what data users see within visualizations. A sales dashboard might show global data to executives but only regional data to regional managers. This filtering happens transparently based on user attributes, without requiring separate visualizations for different user groups.

// Permission specification
{
  "resource": "dashboard:sales-performance",
  "permissions": [
    {
      "principal": "role:executive",
      "actions": ["view", "edit", "admin"],
      "dataFilters": []  // No filtering - see all data
    },
    {
      "principal": "role:regional-manager",
      "actions": ["view"],
      "dataFilters": [
        { "field": "region", "operator": "equals", "value": "${user.region}" }
      ]  // See only their region
    },
    {
      "principal": "role:analyst",
      "actions": ["view", "edit"],
      "dataFilters": [
        { "field": "department", "operator": "equals", "value": "${user.department}" }
      ]  // See only their department
    }
  ]
}

7.6 Governance and Compliance

Enterprise visualization deployments must comply with data governance policies, audit requirements, and regulatory constraints. The WIA framework includes governance features that enable compliance without impeding analytics productivity.

Audit Logging: All operations are logged: who viewed which dashboard when, what data was accessed, what filters were applied, what exports were generated. Audit logs are immutable and retained according to compliance requirements. This enables security investigations, compliance reporting, and usage analytics.

Data Lineage: Visualizations display data from various sources, often transformed through multiple steps. Data lineage tracking documents where data came from, what transformations were applied, and when it was last updated. This transparency is essential for regulatory compliance and data quality management.

Governance Feature Purpose Implementation
Audit logging Track all access and operations Immutable log store, searchable UI
Data lineage Document data sources and transforms Metadata capture, visualization
Certification Mark trusted visualizations Approval workflow, badges
Data classification Mark sensitive data Labels, access restrictions
Retention policies Control data lifetime Automated archival/deletion

Certification Workflow: Not all visualizations are equally trustworthy. The framework supports certification workflows where designated approvers review visualizations for accuracy, compliance, and quality before they're published to broad audiences. Certified visualizations display a badge indicating they've been validated.

Data Sensitivity Marking: Data fields can be marked as sensitive (PII, financial, confidential) with appropriate access controls. The framework can automatically redact, mask, or restrict access to sensitive data based on user permissions. This reduces the risk of inadvertent data exposure.

7.7 Deployment Architectures

The WIA framework supports multiple deployment architectures to meet different organizational requirements: cloud-hosted SaaS, on-premises installations, hybrid configurations, and edge deployments.

Cloud SaaS: The simplest deployment model is cloud-hosted SaaS where a vendor manages infrastructure. Organizations configure authentication, connect data sources, and create visualizations without managing servers. This model minimizes operational overhead but may not meet data residency or security requirements.

On-Premises: Organizations with strict data governance requirements may deploy visualization services on-premises, within their own data centers. This provides maximum control over data and infrastructure but requires operational expertise. The WIA standard's container-based architecture simplifies on-premises deployment.

Hybrid: Hybrid architectures combine cloud services (for rendering, collaboration) with on-premises data (for security). Data never leaves the corporate network; instead, the on-premises component executes queries and sends aggregated results to cloud renderers. This balances convenience with security.

Architecture Data Location Operational Burden Best For
Cloud SaaS Cloud Low (vendor managed) Fast deployment, non-sensitive data
On-Premises On-prem High (self-managed) Strict compliance, air-gapped networks
Hybrid Mixed Medium Balance security and convenience
Edge Distributed Medium-High IoT, offline scenarios
