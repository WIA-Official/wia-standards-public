Chapter 2: Site Investigation per ISO 18400

Soil Restoration — WIA-ENV-015

ISO 18400 series sampling, ISO 18512 sample preservation, EPA SW-846 chain-of-custody, ASTM E1527-21 Phase I ESA, ASTM E1903-19 Phase II ESA, hydrogeological characterisation per ISO 11231.

This chapter elaborates on the Site Investigation per ISO 18400 portion of Soil Restoration (WIA-ENV-015). The soil restoration canonical envelope shape, API surface, and protocol exchanges are designed to interoperate across implementations published under the WIA Standards family. Each Phase file in `spec/` documents one architectural layer (envelopes, REST surface, federation, ecosystem) and the present chapter focuses on the site investigation per iso 18400 subset of that layering.

Implementations of site investigation per iso 18400 need to reconcile the per-standard expectations of ISO 11074:2015 (Soil quality vocabulary), ISO 17025 (Lab competence), ISO/TC 190 series, USDA Soil Taxonomy (12th Ed), EPA Method 9080 / 8260D / 8270E. The Phase 1 envelope class for Soil Restoration carries the per-host identifier and per-tenant identifier so that audit records can be reconstructed across hosts that federate through the Phase 3 protocol layer.

Operators stepping into site investigation per iso 18400 typically follow this trajectory: stand up the reference simulator at `simulator/index.html`, walk the canonical envelope per the CLI helper at `cli/soil-restoration.sh envelope`, then exercise the Phase 2 endpoints with curl/Postman against a host running the reference container at `wia/soil-restoration-host:1.0.0`. Once those steps complete, integrators wire the real backend one endpoint at a time per the Phase 1 §A.* envelope catalogue and the Phase 2 §B.* endpoint catalogue.

A common operational concern in the site investigation per iso 18400 space is observability. Every Phase 1 envelope SHOULD emit a structured log line at the host's audit transport with: timestamp per RFC 3339, host identifier, tenant identifier, envelope class, envelope identifier, operation outcome, and a W3C Trace Context `traceparent` propagated end-to-end. Phase 2 surfaces this trace identifier as the `X-WIA-Trace-Id` response header. Phase 3 protocol exchanges propagate the trace identifier inside the exchange envelope so that a federation crossing remains correlatable end-to-end.

The site investigation per iso 18400 portion of the standard composes with: WIA-OMNI-API for credential storage, WIA-AIR-SHIELD for runtime trust list, WIA-INTENT for workload intent declaration, and WIA Secure Enclave for sealed-data envelopes where personal data is processed. The composition lets one host running multiple WIA family standards reuse one identity, one signing-key chain, and one audit transport rather than maintaining N parallel per-standard implementations.

  5. Wire the conformance suite at https://github.com/WIA-Official/wia-soil-restoration-conformance.

Cross-standard composition recap

This chapter, like every other Phase 1-4 chapter in the Soil Restoration eBook, composes with the wider WIA Standards family. Implementations that adopt the canonical envelope reuse the cross-standard audit transport (W3C Trace Context plus OpenTelemetry semantic conventions), the cross-standard identity (WIA-OMNI-API), and the cross-standard runtime trust list (WIA-AIR-SHIELD) without per-standard re-implementation. The federation handshake described in Phase 3 §3 follows the same challenge-response pattern as the rest of the WIA Standards family so that one operator can federate across multiple standards using one signing-key chain and one audit transport.
