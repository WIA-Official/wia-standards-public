4.2 Apache Parquet Deep Dive

4.2.1 Why Parquet?

  • Columnar Storage: Only read columns needed for query
  • Efficient Compression: Similar data grouped together compresses better
  • Predicate Pushdown: Skip row groups based on statistics
  • Wide Ecosystem Support: Spark, Hive, Presto, Pandas, Arrow

4.2.2 Parquet File Structure

Parquet File Layout:
┌─────────────────────────────┐
│ 4-byte Magic: PAR1          │
├─────────────────────────────┤
│ Row Group 1                 │
│  ├─ Column Chunk A          │
│  │   ├─ Page 1 (data)       │
│  │   ├─ Page 2 (data)       │
│  │   └─ Statistics          │
│  ├─ Column Chunk B          │
│  └─ Column Chunk C          │
├─────────────────────────────┤
│ Row Group 2                 │
│  └─ ...                     │
├─────────────────────────────┤
│ File Metadata               │
│  ├─ Schema                  │
│  ├─ Row Group Metadata      │
│  └─ Column Statistics       │
├─────────────────────────────┤
│ 4-byte Footer Length        │
├─────────────────────────────┤
│ 4-byte Magic: PAR1          │
└─────────────────────────────┘

4.2.3 Parquet Configuration Best Practices

# Python (PySpark)
df.write.format("parquet") \
    .option("compression", "snappy") \
    .option("parquet.block.size", 134217728) \  # 128 MB
    .option("parquet.page.size", 1048576) \      # 1 MB
    .mode("append") \
    .save("s3://lake/data/")

# Key Parameters:
# - compression: snappy (default), gzip, zstd, none
# - block.size: Row group size (default 128 MB)
# - page.size: Page size within row group (default 1 MB)

4.3 Apache ORC Deep Dive

4.3.1 ORC vs Parquet

Feature ORC Parquet
Compression Slightly better Excellent
Read Performance Faster for Hive Faster for Spark
Write Performance Faster Good
Ecosystem Hive-native Broader support
Predicate Pushdown ✅ Excellent ✅ Excellent

Recommendation: Use Parquet for general-purpose analytics. Use ORC if deeply integrated with Hive ecosystem.

4.4 Partitioning Strategies

4.4.1 Date-Based Partitioning

s3://lake/events/
├── year=2025/
│   ├── month=01/
│   │   ├── day=15/
│   │   │   ├── part-00000.parquet
│   │   │   └── part-00001.parquet

# Benefits:
- Time-series queries optimized
- Easy to implement retention policies
- Natural data lifecycle management

# Query with partition pruning:
SELECT * FROM events
WHERE year = 2025 AND month = 01 AND day = 15;
-- Only scans year=2025/month=01/day=15/ partition

4.4.2 Hash-Based Partitioning

s3://lake/users/
├── hash_bucket=0/
├── hash_bucket=1/
├── hash_bucket=2/
...
├── hash_bucket=255/

# Benefits:
- Evenly distributed data
- Prevents hot partitions
- Good for high-cardinality keys

# Implementation:
hash_bucket = hash(user_id) % 256

4.4.3 Range-Based Partitioning

s3://lake/transactions/
├── amount_range=0-100/
├── amount_range=100-500/
├── amount_range=500-1000/
└── amount_range=1000-plus/

# Benefits:
- Logical data grouping
- Business-friendly partitions
- Optimized for range queries

4.5 Compression Codecs

4.5.1 Compression Comparison

Codec Compression Ratio Encode Speed Decode Speed CPU Usage
None 1.0x Fastest Fastest None
Snappy 2-3x Very Fast Very Fast Low
LZ4 2-3x Very Fast Fastest Very Low
ZSTD 3-5x Fast Fast Medium

WIA Recommendation: Use ZSTD for best balance of compression and performance.

4.6 Schema Design Best Practices

4.6.1 Column Order Optimization

# Good: Frequently filtered columns first
CREATE TABLE events (
  date        DATE,      -- Often filtered
  user_id     BIGINT,    -- Often filtered
  event_type  STRING,    -- Often filtered
  properties  MAP,       -- Rarely filtered
  metadata    STRUCT     -- Rarely filtered
);

# Bad: Random order
CREATE TABLE events (
  metadata    STRUCT,
  date        DATE,
  properties  MAP,
  event_type  STRING,
  user_id     BIGINT
);

4.6.2 Data Type Selection

Use Case ❌ Avoid ✅ Use Why
Timestamps STRING TIMESTAMP Native time operations, smaller size
IDs STRING BIGINT 8 bytes vs variable
Booleans STRING ('true'/'false') BOOLEAN 1 bit vs 4-5 bytes
Decimals DOUBLE DECIMAL(p,s) Precision guarantees

4.7 Format Migration Strategies

4.7.1 CSV to Parquet Migration

# PySpark migration
from pyspark.sql import SparkSession

spark = SparkSession.builder.getOrCreate()

# Read CSV
df = spark.read.format("csv") \
    .option("header", "true") \
    .option("inferSchema", "true") \
    .load("s3://lake/raw/data.csv")

# Write as Parquet
df.write.format("parquet") \
    .option("compression", "zstd") \
    .mode("overwrite") \
    .save("s3://lake/curated/data.parquet")

4.7.2 Schema Evolution Example

# Initial schema
CREATE TABLE users (
  id BIGINT,
  name STRING,
  email STRING
);

# Add column (safe, backward compatible)
ALTER TABLE users ADD COLUMN phone STRING;

# Rename column (requires rewriting in Parquet)
-- Use views for backward compatibility:
CREATE VIEW users_new AS
SELECT id, name AS full_name, email FROM users;
