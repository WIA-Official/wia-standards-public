[DATA-002] Chapter 2: Data Lake Architecture

Good architecture is invisible. Poor architecture is everywhere.

A well-designed data lake enables teams to move fast while maintaining quality and governance.

2.1 Three-Zone Architecture

The WIA Data Lake Standard recommends a three-zone architecture pattern that separates data by its level of processing and intended use:

┌─────────────────────────────────────────────────────────────┐
│                      DATA SOURCES                            │
│   IoT • APIs • Databases • Files • Streams • Applications    │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│                    RAW ZONE (Bronze)                         │
│  • Unprocessed data as-is                                    │
│  • Schema-on-read                                            │
│  • High durability, low cost storage                         │
│  • Append-only, immutable                                    │
└────────────────────┬────────────────────────────────────────┘
                     │ Data Quality • Cleansing • Enrichment
                     ▼
┌─────────────────────────────────────────────────────────────┐
│                  CURATED ZONE (Silver)                       │
│  • Cleaned, validated, transformed                           │
│  • Schema-on-write                                           │
│  • Business rules applied                                    │
│  • ACID transactions enabled                                 │
└────────────────────┬────────────────────────────────────────┘
                     │ Aggregation • Feature Engineering
                     ▼
┌─────────────────────────────────────────────────────────────┐
│                CONSUMPTION ZONE (Gold)                       │
│  • Analytics-ready datasets                                  │
│  • Optimized for query performance                           │
│  • Star/snowflake schemas                                    │
│  • BI-friendly views                                         │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│                    CONSUMERS                                 │
│  BI Tools • ML Pipelines • Reports • APIs • Applications     │
└─────────────────────────────────────────────────────────────┘

2.1.1 Raw Zone (Bronze Layer)

Characteristic Implementation Purpose
Data State Raw, unprocessed, as-received Preserve original source data
Schema Schema-on-read Flexibility to ingest any format
Storage S3 Standard, Azure Blob Hot Balance cost and access speed
Formats JSON, CSV, Avro, raw logs Native source formats
Retention Long-term (years) Regulatory compliance, reprocessing
Users Data engineers, data scientists Exploratory analysis, debugging

│   │   ├── month=01/
│   │   │   ├── day=15/
│   │   │   │   ├── hour=10/
│   │   │   │   │   └── sensors-20250115-1000.json.gz
├── api-logs/
│   ├── source=payment-api/
│   │   ├── date=2025-01-15/
│   │   │   └── logs-20250115.jsonl.gz
└── database-dumps/
    ├── postgres/
    │   ├── table=users/
    │   │   └── dump-20250115.parquet

2.1.2 Curated Zone (Silver Layer)

Characteristic Implementation Purpose
Data State Cleaned, validated, enriched High-quality business data
Schema Schema-on-write with enforcement Data quality guarantees
Storage Delta Lake, Iceberg, Hudi ACID transactions, versioning
Formats Parquet, ORC Columnar, compressed, optimized
Operations INSERT, UPDATE, DELETE, MERGE Full DML support
Users Analytics engineers, data scientists Feature engineering, modeling

2.1.3 Consumption Zone (Gold Layer)

Characteristic Implementation Purpose
Data State Aggregated, denormalized Ready for reporting and BI
Schema Star/snowflake schemas Optimized for analytical queries
Storage Materialized views, cubes Pre-computed aggregations
Performance Z-ordering, data skipping Sub-second query latency
Access SQL, REST APIs Business user friendly
Users Business analysts, executives Dashboards, reports, decisions

2.2 Schema-on-Read vs Schema-on-Write

One of the most important architectural decisions in data lake design:

Aspect Schema-on-Read Schema-on-Write
When Applied When data is queried When data is written
Flexibility High - can reinterpret data Low - schema changes require migration
Write Speed Fast - no validation Slower - schema enforcement
Read Speed Slower - schema interpretation Faster - schema known
Data Quality Variable - validation at query time Guaranteed - validation at write time
Use Case Exploration, raw data ingestion Production pipelines, reporting
Zone Raw (Bronze) Curated (Silver), Consumption (Gold)

Best Practice: Use schema-on-read for ingestion (Raw zone), then enforce schema-on-write as data progresses to Curated and Consumption zones.

2.3 Data Lakehouse Architecture

The data lakehouse combines the best attributes of data lakes and data warehouses:

2.3.1 Key Capabilities

  1. ACID Transactions
    • Atomic, Consistent, Isolated, Durable writes
    • Prevents data corruption in concurrent workloads
    • Enabled by Delta Lake, Iceberg, Hudi
  2. Schema Enforcement & Evolution
    • Prevent bad data from entering the lake
    • Safely add/remove columns without rewriting data
    • Backward and forward compatibility
  3. Time Travel
    • Access historical versions of data
    • Rollback erroneous updates
    • Audit and compliance
  4. Efficient Upserts & Deletes
    • UPDATE and DELETE operations on immutable storage
    • Critical for GDPR compliance (right to be forgotten)
    • CDC (Change Data Capture) support
  5. Unified Batch & Streaming
    • Same tables for real-time and batch processing
    • Lambda architecture without complexity
    • Simplified data pipelines

2.4 Storage Optimization Strategies

2.4.1 File Organization

Partitioning:

# Good: Balanced partitioning (1-10 GB per partition)
/year=2025/month=01/day=15/

# Bad: Over-partitioning (too many small files)
/year=2025/month=01/day=15/hour=10/minute=30/second=45/

# Bad: Under-partitioning (files too large)
/all_data/

File Sizing Best Practices:

  • Optimal file size: 128 MB - 1 GB (matches HDFS block size)
  • Avoid small files (< 10 MB): causes metadata overhead
  • Avoid huge files (> 10 GB): reduces parallelism
  • Use compaction to merge small files periodically

2.4.2 Compression

Codec Compression Ratio Speed Splittable Use Case
Snappy Good Very Fast No* Default for Parquet/ORC
GZIP Better Medium No Long-term storage
ZSTD Best Fast No* Modern default
LZ4 Good Fastest No Real-time streaming
Brotli Excellent Slow No Cold storage

* When used with Parquet/ORC, splittable at row group level

2.4.3 Data Layout Optimization

Z-Ordering (Multi-dimensional clustering):

-- Without Z-order: Linear scan
SELECT * FROM events WHERE user_id = 123 AND date = '2025-01-15'
-- Scans: 100% of files

-- With Z-order on (user_id, date)
OPTIMIZE events ZORDER BY (user_id, date)
-- Scans: ~5% of files (data skipping)

Column Ordering:

  • Place frequently filtered columns first
  • Group related columns together
  • Improves compression and query performance

2.5 Data Lifecycle Management

2.5.1 Tiered Storage Strategy

Tier Age Storage Class Cost Access Time
Hot 0-30 days S3 Standard, Azure Hot $$$ Milliseconds
Warm 30-90 days S3 Intelligent-Tiering $$ Milliseconds
Cold 90-365 days S3 Glacier Instant $ Milliseconds
Archive 1+ years S3 Glacier Deep Archive ¢ 12+ hours

Lifecycle Policy Example:

{
  "Rules": [
    {
      "Id": "RawDataLifecycle",
      "Prefix": "raw/",
      "Transitions": [
        {"Days": 30, "StorageClass": "INTELLIGENT_TIERING"},
        {"Days": 90, "StorageClass": "GLACIER_IR"},
        {"Days": 365, "StorageClass": "DEEP_ARCHIVE"}
      ]
    }
  ]
}

Summary

In this chapter, we covered:

  • Three-zone architecture: Raw (Bronze), Curated (Silver), Consumption (Gold)
  • Schema-on-read for flexibility, schema-on-write for quality
  • Data lakehouse pattern combines lake and warehouse benefits
  • Storage optimization through partitioning, compression, and Z-ordering
  • Tiered storage for cost-effective data lifecycle management
