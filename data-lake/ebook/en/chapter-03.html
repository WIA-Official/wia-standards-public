[DATA-002] Chapter 3: WIA Data Lake Standard Overview

Standards enable ecosystems. Ecosystems enable innovation.

The WIA Data Lake Standard provides the foundation for a thriving, interoperable data ecosystem.

3.1 Mission and Goals

3.1.1 Primary Mission

To provide an open, vendor-neutral standard for data lake implementations that enables interoperability, portability, and innovation across the entire data ecosystem.

3.1.2 Core Goals

Goal Description Success Metric
Interoperability Any tool can read/write standard format 100+ compatible tools
Portability Easy migration between platforms Zero vendor lock-in
Scalability Support petabyte+ data lakes Proven at exabyte scale
Performance Optimize for analytical workloads Sub-second queries on TB data
Accessibility Free, open specifications MIT license, no fees

3.2 Four-Phase Architecture

The WIA Data Lake Standard is organized into four progressive phases:

3.2.1 Phase 1: Data Format Standard

Scope: File formats, compression, partitioning

Key Specifications:

  • Apache Parquet as primary columnar format
  • Apache ORC for Hive/Spark ecosystems
  • Apache Avro for row-based, schema-evolution scenarios
  • JSON/JSONL for semi-structured data
  • CSV for legacy compatibility

Partitioning Strategies:

  • Date-based: year=YYYY/month=MM/day=DD
  • Hash-based: hash_bucket=0-255
  • Range-based: price_range=0-100, 100-500, 500+

3.2.2 Phase 2: API Interface Standard

Scope: Programming interfaces, SDKs

Core API Classes:

class WiaDataLake:
    def __init__(storage_path, catalog_uri)
    def ingest(source, target_zone, format)
    def query(sql, engine="spark")
    def get_metadata(table_name)
    def optimize(table_name, strategy="compact")
    def vacuum(table_name, retention_hours=168)

Supported Languages:

  • Python (primary)
  • Java/Scala
  • JavaScript/TypeScript
  • Rust (emerging)

3.2.3 Phase 3: Table Format Protocol

Scope: ACID transactions, schema evolution, time travel

Supported Table Formats:

Format Origin Best For WIA Status
Delta Lake Databricks Streaming + Batch ✅ Fully Supported
Apache Iceberg Netflix Large-scale Analytics ✅ Fully Supported
Apache Hudi Uber CDC, Incremental ✅ Fully Supported

3.2.4 Phase 4: Ecosystem Integration

Scope: Compute engines, catalogs, cloud platforms

Integration Points:

  • Compute Engines: Spark, Presto, Trino, Hive, Flink, Dremio
  • Catalogs: Hive Metastore, AWS Glue, Unity Catalog, Nessie
  • Cloud Platforms: AWS, Azure, GCP, Alibaba Cloud
  • Streaming: Kafka, Kinesis, Pulsar, Event Hubs
  • BI Tools: Tableau, Power BI, Looker, Metabase

3.3 Design Principles

3.3.1 Open by Default

All specifications are published under MIT license. No proprietary extensions required for compliance.

3.3.2 Cloud-Native

Designed for object storage (S3, Azure Blob, GCS) but works on-premises (HDFS, NFS).

3.3.3 Schema Evolution Support

Tables can safely add, remove, rename columns without breaking existing queries.

3.3.4 Time Travel

Access historical versions of data for auditing, debugging, and rollback.

3.3.5 Metadata-Driven

Rich metadata enables data discovery, lineage tracking, and governance.

3.3.6 Performance-Optimized

Columnar formats, predicate pushdown, data skipping, Z-ordering for fast queries.

3.4 Supported Table Formats Deep Dive

3.4.1 Delta Lake

Key Features:

  • ACID transactions via transaction log
  • Time travel with VERSION AS OF / TIMESTAMP AS OF
  • Schema enforcement and evolution
  • Efficient MERGE (upsert) operations
  • OPTIMIZE and VACUUM commands

Example Usage:

CREATE TABLE events
USING delta
LOCATION 's3://lake/curated/events';

INSERT INTO events VALUES (1, 'click', '2025-01-15');
UPDATE events SET event_type = 'purchase' WHERE id = 1;
DELETE FROM events WHERE created_at < '2024-01-01';

SELECT * FROM events VERSION AS OF 5;  -- Time travel
OPTIMIZE events ZORDER BY (user_id);   -- Performance
VACUUM events RETAIN 168 HOURS;         -- Cleanup

3.4.2 Apache Iceberg

Key Features:

  • Hidden partitioning (no need to specify partitions in queries)
  • Snapshot isolation for concurrent reads/writes
  • Partition evolution without rewriting data
  • Metadata-based file pruning

Example Usage:

CREATE TABLE analytics.users (
  id bigint,
  name string,
  created_at timestamp
)
USING iceberg
PARTITIONED BY (days(created_at));

-- Query without specifying partition
SELECT * FROM analytics.users WHERE created_at = '2025-01-15';
-- Iceberg automatically prunes files

3.4.3 Apache Hudi

Key Features:

  • Copy-On-Write (COW) and Merge-On-Read (MOR) tables
  • Incremental queries (only new data since last read)
  • Record-level indexing for fast updates
  • Timeline service for metadata

Example Usage:

-- Incremental read (only changes since last query)
SELECT * FROM hudi_table
WHERE _hoodie_commit_time > '20250115100000';

-- Upsert operation
INSERT INTO hudi_table
VALUES (1, 'updated', '2025-01-15')
ON DUPLICATE KEY UPDATE;

3.6 Reference Implementations

WIA provides reference implementations to demonstrate compliance:

3.6.1 Python Reference Implementation

pip install wia-datalake

from wia_datalake import WiaDataLake

lake = WiaDataLake(
    storage_path="s3://my-lake/",
    catalog_uri="glue://us-east-1"
)

# Ingest data
lake.ingest(
    source="s3://raw-data/events.json",
    target_zone="curated",
    format="delta"
)

# Query
results = lake.query("SELECT * FROM curated.events LIMIT 10")

3.6.2 Spark Integration

from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .config("spark.sql.extensions", "io.delta.sql.DeltaSparkSessionExtension") \
    .config("spark.sql.catalog.spark_catalog", "org.apache.spark.sql.delta.catalog.DeltaCatalog") \
    .getOrCreate()

df = spark.read.format("delta").load("s3://lake/curated/events")
df.write.format("delta").mode("append").save("s3://lake/curated/processed")
