Chapter 6: Phase 3 - Table Format Protocol

ACID on a data lake? That's the lakehouse revolution.

6.1 Why Table Formats Matter

Traditional data lakes lack ACID guarantees. Table formats solve this:

Challenge Without Table Format With Table Format
Concurrent Writes Data corruption ACID transactions
Updates/Deletes Impossible or slow Efficient MERGE/DELETE
Schema Changes Rewrite all data Schema evolution
Time Travel Manual snapshots Built-in versioning

6.2 Delta Lake Protocol

6.2.1 Transaction Log

Delta Lake Structure:
s3://lake/events/
├── _delta_log/
│   ├── 00000000000000000000.json  # Version 0
│   ├── 00000000000000000001.json  # Version 1
│   ├── 00000000000000000002.json  # Version 2
│   └── _last_checkpoint              # Optimization
├── part-00000-v1.parquet
├── part-00001-v1.parquet
└── part-00002-v2.parquet

6.2.2 Transaction Log Format

// Example transaction log entry
{
  "commitInfo": {
    "timestamp": 1705320000000,
    "operation": "WRITE",
    "operationMetrics": {
      "numFiles": "2",
      "numOutputRows": "1000000"
    }
  },
  "add": {
    "path": "part-00000-v1.parquet",
    "size": 134217728,
    "partitionValues": {"date": "2025-01-15"},
    "stats": "{\"numRecords\": 500000, \"minValues\": {...}, \"maxValues\": {...}}"
  }
}

6.2.3 Delta Lake Operations

-- Time Travel
SELECT * FROM events VERSION AS OF 10;
SELECT * FROM events TIMESTAMP AS OF '2025-01-15 10:00:00';

-- Describe History
DESCRIBE HISTORY events LIMIT 10;

-- MERGE (Upsert)
MERGE INTO target
USING source
ON target.id = source.id
WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *;

-- OPTIMIZE
OPTIMIZE events WHERE date = '2025-01-15';
OPTIMIZE events ZORDER BY (user_id, event_type);

-- VACUUM (Clean old files)
VACUUM events RETAIN 168 HOURS;  -- 7 days

6.3 Apache Iceberg Protocol

6.3.1 Metadata Hierarchy

├── metadata/
│   ├── v1.metadata.json
│   ├── v2.metadata.json
│   ├── snap-12345.avro
│   └── manifest-list-67890.avro
└── data/

    └── 00002-2-data-file.parquet

# Metadata JSON example

  "format-version": 2,

  "location": "s3://lake/events",
  "current-snapshot-id": 12345,

    "write.format.default": "parquet"
  },
  "schemas": [...],
  "partition-specs": [...]
}

6.3.2 Hidden Partitioning

-- Traditional partitioning: User must know partition columns
SELECT * FROM events WHERE year = 2025 AND month = 1 AND day = 15;

-- Iceberg hidden partitioning: Automatic
SELECT * FROM events WHERE timestamp = '2025-01-15';
-- Iceberg automatically maps timestamp to partitions

6.3.3 Partition Evolution

-- Start with daily partitions
CREATE TABLE events PARTITIONED BY (days(timestamp));

-- Later, change to hourly (without rewriting data!)
ALTER TABLE events SET PARTITION SPEC (hours(timestamp));

-- Old data stays in daily partitions, new data in hourly

6.4 Apache Hudi Protocol

6.3.1 Table Types

Type Write Latency Query Latency Use Case
Copy-On-Write (COW) Higher Lower Read-heavy workloads
Merge-On-Read (MOR) Lower Higher Write-heavy, real-time

6.4.2 Timeline and Commits

Hudi Structure:
s3://lake/events/
├── .hoodie/
│   ├── 20250115100000.commit
│   ├── 20250115103000.deltacommit
│   ├── 20250115110000.compaction
│   └── hoodie.properties
├── 2025/01/15/
│   ├── base-file-1.parquet          # Base data
│   └── log-file-1.log.1             # Delta logs (MOR only)

6.4.3 Incremental Queries

# Read only changed data since last read
df = spark.read.format("hudi") \
    .option("hoodie.datasource.query.type", "incremental") \
    .option("hoodie.datasource.read.begin.instanttime", "20250115100000") \
    .load("s3://lake/events/")

# Only new/updated records returned

WIA Recommendation:

  • Delta Lake: Best for Databricks users, streaming workloads
  • Iceberg: Best for multi-engine environments, partition evolution needs
  • Hudi: Best for incremental processing, CDC workloads

6.6 ACID Transaction Semantics

6.6.1 Atomicity

-- All-or-nothing guarantee
BEGIN TRANSACTION;
  INSERT INTO events VALUES (...);  -- 1 million rows
  UPDATE events SET status = 'processed' WHERE date = '2025-01-15';
COMMIT;

-- If any operation fails, entire transaction rolls back
-- No partial writes visible to readers

6.6.2 Consistency

-- Schema enforcement
INSERT INTO events (id, name, timestamp)
VALUES (1, 'event', '2025-01-15');
-- ✅ Succeeds

INSERT INTO events (id, name, timestamp)
VALUES (2, 123, 'invalid-date');
-- ❌ Fails: type mismatch, invalid timestamp

6.6.3 Isolation

-- Snapshot isolation: readers see consistent snapshot
-- Writer 1: Long-running transaction
UPDATE events SET status = 'processing' WHERE ...;

-- Reader 1 (concurrent): Sees pre-update snapshot
SELECT * FROM events;  -- No partial updates visible

-- Writer 1 commits
COMMIT;

-- Reader 2 (new query): Sees post-update snapshot
SELECT * FROM events;  -- All updates visible

6.6.4 Durability

-- Once committed, data persists despite failures
INSERT INTO events VALUES (...);
COMMIT;  -- Transaction log entry written

-- Even if Spark cluster crashes immediately after,
-- the commit is durable and will be visible on restart

6.7 Schema Evolution Patterns

6.7.1 Adding Columns

-- Initial schema
CREATE TABLE users (id BIGINT, name STRING);

-- Add column (backward compatible)
ALTER TABLE users ADD COLUMNS (email STRING);

-- Old queries still work
SELECT id, name FROM users;  -- ✅ No changes needed

-- New queries can use new column
SELECT id, name, email FROM users;  -- ✅

6.7.2 Renaming Columns

-- Iceberg supports true column renaming
ALTER TABLE users RENAME COLUMN name TO full_name;

-- Delta Lake requires creating views for backward compatibility
CREATE OR REPLACE VIEW users_legacy AS
  SELECT id, full_name AS name FROM users;

6.7.3 Changing Data Types

-- Safe: INT -> BIGINT
ALTER TABLE events ALTER COLUMN count SET DATA TYPE BIGINT;

-- Unsafe: STRING -> INT (requires data validation)
-- Use multi-step process:
1. ADD COLUMN count_int INT
2. UPDATE table SET count_int = CAST(count AS INT)
3. Verify all values converted
4. DROP COLUMN count
5. RENAME COLUMN count_int TO count

Summary

In this chapter, we learned:

  • Table formats enable ACID transactions on data lakes
  • Delta Lake: transaction log, time travel, Z-ordering
  • Iceberg: hidden partitioning, partition evolution, wide engine support
  • Hudi: incremental reads, COW/MOR tables, timeline service
  • ACID semantics: Atomicity, Consistency, Isolation, Durability
  • Schema evolution: adding, renaming, changing columns safely
