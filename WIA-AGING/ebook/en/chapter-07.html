🔗 Chapter 7: Phase 4 - Integration

7.1 Integration Architecture Patterns

The aging and longevity ecosystem is inherently complex, involving diverse stakeholders: hospitals and clinics, research institutions, wearable device manufacturers, pharmaceutical companies, insurance providers, and individual users. Each participant has their own systems, data formats, and integration requirements. The WIA-AGING standard addresses this complexity through flexible integration patterns that accommodate various architectural approaches while maintaining data integrity and security.

Integration patterns define how different systems connect, communicate, and coordinate. The WIA-AGING standard supports three primary integration patterns, each suited to different organizational needs and technical constraints:

7.1.1 Hub-and-Spoke Pattern

The hub-and-spoke pattern centralizes integration logic in a central hub that mediates all communication between systems. This is the most common pattern for healthcare organizations deploying WIA-AGING for internal integration.

┌─────────────┐        ┌──────────────────┐        ┌─────────────┐
│   Wearable  │───────▶│                  │───────▶│     EHR     │
│   Devices   │        │   WIA-AGING      │        │   System    │
└─────────────┘        │   Integration    │        └─────────────┘
                       │       Hub         │
┌─────────────┐        │                  │        ┌─────────────┐
│ Laboratory  │───────▶│                  │───────▶│  Research   │
│   Systems   │        │                  │        │  Database   │
└─────────────┘        └──────────────────┘        └─────────────┘

Advantages:

  • Single point of integration reduces complexity for each spoke system
  • Centralized governance, security, and monitoring
  • Easier to implement transformations and business rules
  • Simplified troubleshooting and audit trails

Considerations:

  • Hub becomes a single point of failure (mitigate with redundancy)
  • May introduce latency for high-volume real-time scenarios
  • Requires robust hub infrastructure for scalability

7.1.2 Federated Pattern

The federated pattern enables peer-to-peer communication while maintaining standardized protocols and data formats. This pattern is ideal for research consortia and multi-institutional collaborations where no single entity controls the infrastructure.

┌──────────────┐    WIA-AGING     ┌──────────────┐
│ Institution  │◀────Protocol────▶│ Institution  │
│      A       │                  │      B       │
└──────────────┘                  └──────────────┘
       ▲                                  ▲
       │                                  │
       │         ┌──────────────┐        │
       └────────▶│ Institution  │◀───────┘
                 │      C       │
                 └──────────────┘

Advantages:

  • No single point of failure or control
  • Each institution maintains sovereignty over their data
  • Scales naturally as institutions are added
  • Supports cross-institutional research without data centralization

7.1.3 Event-Driven Architecture

Event-driven architectures use publish-subscribe patterns where systems react to events (e.g., "new assessment completed") rather than polling for changes. This pattern is optimal for real-time responsiveness and loose coupling between systems.

Pattern Best For Complexity
Hub-and-Spoke Single organization, centralized control Medium
Federated Multi-institutional research, data sovereignty High
Event-Driven Real-time processing, microservices High

7.2 Healthcare System Adapters

Existing healthcare systems use a variety of standards and proprietary formats. WIA-AGING provides pre-built adapters for common healthcare IT systems, dramatically reducing integration time and cost.

7.2.1 Electronic Health Record (EHR) Adapter

EHR systems like Epic, Cerner, and Allscripts are the backbone of modern healthcare. The WIA-AGING EHR adapter supports both HL7 FHIR (Fast Healthcare Interoperability Resources) and legacy HL7 v2 messaging.

// FHIR-based integration example
import { WIAAgingEHRAdapter } from '@wia/aging-ehr-adapter';

const ehrAdapter = new WIAAgingEHRAdapter({
  ehrSystem: 'epic',
  fhirEndpoint: 'https://fhir.epic.example.com/api/FHIR/R4',
  authentication: {
    type: 'oauth2',
    clientId: process.env.EPIC_CLIENT_ID,
    clientSecret: process.env.EPIC_CLIENT_SECRET
  }
});

// Retrieve patient biomarker data from EHR
const patientData = await ehrAdapter.getPatientObservations({
  patientId: 'fhir-patient-123',
  observations: [
    'http://loinc.org/1920-8',  // Aspartate aminotransferase
    'http://loinc.org/1742-6',  // Alanine aminotransferase
    'http://loinc.org/2160-0',  // Creatinine
    'http://loinc.org/1751-7'   // Albumin
  ],
  dateRange: {
    start: '2024-01-01',
    end: '2025-12-31'
  }
});

// Transform FHIR observations to WIA-AGING biomarker format
const biomarkers = ehrAdapter.transformToWIAFormat(patientData);

// Create assessment using WIA-AGING API
const assessment = await wiaClient.assessments.create(userId, {
  biomarkers: biomarkers,
  source: 'ehr',
  source_system: 'epic'
});

7.3.2 Apple HealthKit Integration

// iOS app integration with HealthKit
import HealthKit

class WIAAgingHealthKitSync {
  let healthStore = HKHealthStore()

  func requestAuthorization() async throws {
    let biomarkerTypes: Set = [
      HKQuantityType.quantityType(forIdentifier: .heartRateVariabilitySDNN)!,
      HKQuantityType.quantityType(forIdentifier: .restingHeartRate)!,
      HKQuantityType.quantityType(forIdentifier: .vo2Max)!,
      HKQuantityType.categoryType(forIdentifier: .sleepAnalysis)!,
      HKQuantityType.quantityType(forIdentifier: .stepCount)!,
      HKQuantityType.quantityType(forIdentifier: .bodyTemperature)!
    ]

    try await healthStore.requestAuthorization(toShare: [], read: biomarkerTypes)
  }

  func syncRecentData() async throws {
    let calendar = Calendar.current
    let endDate = Date()
    let startDate = calendar.date(byAdding: .day, value: -7, to: endDate)!

    // Fetch HRV data
    let hrvSamples = try await fetchQuantitySamples(
      type: .heartRateVariabilitySDNN,
      start: startDate,
      end: endDate
    )

    // Transform to WIA-AGING format
    let biomarkerUpdates = hrvSamples.map { sample -> BiomarkerUpdate in
      return BiomarkerUpdate(
        biomarkerType: "heart_rate_variability",
        value: sample.quantity.doubleValue(for: HKUnit.secondUnit(with: .milli)),
        unit: "ms",
        timestamp: sample.startDate,
        source: "apple_healthkit",
        deviceModel: sample.device?.model ?? "unknown"
      )
    }

    // Send to WIA-AGING API
    try await wiaClient.biomarkers.batchUpload(biomarkerUpdates)
  }
}

7.3.3 Oura Ring Integration

// Oura API integration
import { WIAAgingOuraAdapter } from '@wia/aging-oura-adapter';

const ouraAdapter = new WIAAgingOuraAdapter({
  accessToken: process.env.OURA_ACCESS_TOKEN
});

// Fetch recent sleep data
const sleepData = await ouraAdapter.getSleepData({
  startDate: '2025-12-08',
  endDate: '2025-12-15'
});

// Extract biomarkers relevant to aging
const agingBiomarkers = sleepData.map(night => ({
  timestamp: night.bedtime_start,
  biomarkers: {
    sleep_total_minutes: night.total_sleep_duration / 60,
    sleep_efficiency: night.efficiency,
    deep_sleep_minutes: night.deep_sleep_duration / 60,
    rem_sleep_minutes: night.rem_sleep_duration / 60,
    hrv_average: night.hr_average,
    resting_heart_rate: night.hr_lowest,
    body_temperature_delta: night.temperature_delta
  },
  quality_score: night.score / 100
}));

// Create assessments incorporating wearable data
for (const day of agingBiomarkers) {
  await wiaClient.biomarkers.createFromWearable({
    userId: 'user_123',
    source: 'oura',
    timestamp: day.timestamp,
    biomarkers: day.biomarkers
  });
}

7.4 Legacy System Bridge

Many healthcare organizations rely on legacy systems that predate modern API architectures. The WIA-AGING Legacy System Bridge provides integration paths for older systems using file-based transfers, database replication, and messaging queues.

7.4.1 File-Based Integration

// sFTP-based integration for legacy lab systems
import { WIAAgingFileAdapter } from '@wia/aging-file-adapter';

const fileAdapter = new WIAAgingFileAdapter({
  transport: 'sftp',
  host: 'legacy-lab.example.com',
  port: 22,
  username: 'wia_integration',
  privateKeyPath: '/secure/keys/wia_sftp_key',
  watchDirectory: '/outbound/lab_results',
  archiveDirectory: '/archive/processed'
});

// Watch for new lab result files
fileAdapter.watch({
  pattern: /^lab_results_\d{8}_\d{6}\.csv$/,
  onNewFile: async (filePath) => {
    console.log(`Processing new file: ${filePath}`);

    // Parse CSV format
    const results = await fileAdapter.parseCSV(filePath, {
      columns: ['patient_id', 'test_date', 'loinc_code', 'value', 'unit', 'reference_range'],
      skipHeader: true
    });

    // Transform and upload to WIA-AGING
    for (const result of results) {
      await wiaClient.biomarkers.create({
        userId: mapPatientIdToWIAUserId(result.patient_id),
        biomarkerType: mapLOINCtoWIA(result.loinc_code),
        value: parseFloat(result.value),
        unit: result.unit,
        referenceRange: result.reference_range,
        source: 'legacy_lis',
        testDate: result.test_date
      });
    }

    // Move to archive
    await fileAdapter.archive(filePath);
  }
});

7.4.2 Database Replication

For legacy systems where file exports aren't feasible, direct database replication can be configured. The WIA-AGING DB Sync tool supports CDC (Change Data Capture) from major database platforms:

// Configure database sync
{
  "source_database": {
    "type": "oracle",
    "host": "legacy-db.example.com",
    "port": 1521,
    "service_name": "LAB_PROD",
    "credentials": {
      "username": "wia_readonly",
      "password_secret": "aws_secrets_manager:wia/legacy_db_password"
    }
  },
  "sync_configuration": {
    "mode": "cdc",  // Change Data Capture
    "tables": [
      {
        "name": "LAB_RESULTS",
        "primary_key": "RESULT_ID",
        "timestamp_column": "CREATED_DATE",
        "mapping": {
          "PATIENT_MRN": "patient_id",
          "TEST_CODE": "loinc_code",
          "RESULT_VALUE": "value",
          "RESULT_UNIT": "unit"
        }
      }
    ],
    "sync_interval": "5 minutes",
    "batch_size": 1000
  },
  "destination": {
    "type": "wia_aging_api",
    "endpoint": "https://api.wia-aging.org/v1/biomarkers/batch",
    "authentication": "oauth2"
  }
}

7.8 Performance Optimization

Large-scale integrations require careful performance optimization. The WIA-AGING platform implements several strategies to ensure high throughput and low latency even when processing millions of biomarker measurements daily.

7.8.1 Batch Processing

// Batch API for high-volume uploads
const batchResults = await wiaClient.biomarkers.batchCreate({
  biomarkers: [
    { userId: 'user_001', type: 'glucose', value: 95, timestamp: '2025-12-15T08:00:00Z' },
    { userId: 'user_002', type: 'glucose', value: 102, timestamp: '2025-12-15T08:00:00Z' },
    // ... up to 1000 records per batch
  ],
  options: {
    validateAsync: true,      // Validate in background
    deduplication: true,       // Skip duplicate timestamps
    partialSuccess: true       // Don't fail entire batch for single error
  }
});

console.log(`Processed: ${batchResults.successful_count}/${batchResults.total_count}`);
console.log(`Errors: ${batchResults.errors.length}`);

7.8.2 Caching Strategy

Reference data (biomarker definitions, unit conversions, normal ranges) is cached aggressively to reduce database load:

  • CDN Caching: Static reference data served via CloudFront with 24-hour TTL
  • Application Caching: Redis cache for frequently accessed user profiles and assessments
  • Client-Side Caching: ETags and conditional requests reduce bandwidth

7.9 Chapter Summary

✅ Key Takeaways:

  1. Flexible integration patterns: Hub-and-spoke for centralized control, federated for multi-institutional collaboration, event-driven for real-time responsiveness.
