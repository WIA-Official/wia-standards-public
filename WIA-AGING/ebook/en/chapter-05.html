5.1 API Design Philosophy

The WIA-AGING API is designed with a clear philosophy: simplicity, security, and scalability. In the aging and longevity ecosystem, APIs serve as the critical bridge between data sources (wearables, lab systems, EHRs) and applications (clinician dashboards, research platforms, consumer apps). Our design philosophy prioritizes developer experience while maintaining the highest standards for data privacy and regulatory compliance.

At the core of our API design is the principle of progressive disclosure. Developers should be able to get started quickly with simple use cases, such as retrieving a user's biological age estimate, while having access to advanced capabilities like batch processing, webhook subscriptions, and real-time data streaming as their needs evolve. This approach reduces time-to-first-API-call while supporting complex enterprise integrations.

The API adheres to REST architectural principles, making it familiar to developers worldwide. We use standard HTTP methods (GET, POST, PUT, DELETE), status codes, and headers. Resources are identified by clear, hierarchical URIs that reflect the logical structure of aging data. For example, /users/{userId}/assessments/{assessmentId} clearly indicates the relationship between users and their aging assessments.

Another fundamental principle is statelessness. Each API request contains all the information necessary to process it, without relying on server-side session state. This enables horizontal scalability, allowing the API to serve millions of concurrent users across distributed infrastructure. Authentication is handled via OAuth 2.0 and JWT tokens, ensuring secure, stateless authentication.

We also embrace API versioning from day one. The API version is included in the URL path (e.g., /v1/assessments), allowing us to introduce breaking changes in future versions while maintaining backward compatibility for existing integrations. This is critical in healthcare, where systems often have long upgrade cycles.

5.2 RESTful Endpoints Overview

The WIA-AGING API is organized into several resource collections, each representing a key entity in the aging ecosystem. Below is a comprehensive overview of the main endpoint categories and their purposes:

Resource Endpoint Pattern Purpose
Users /v1/users User profile management, demographics, consent tracking
Assessments /v1/users/{userId}/assessments Biological age assessments, biomarker data, historical tracking
Biomarkers /v1/biomarkers Individual biomarker values, reference ranges, trending
Interventions /v1/interventions Lifestyle interventions, prescriptions, tracking adherence
Devices /v1/devices Connected wearables, continuous monitoring devices
Research Studies /v1/studies Clinical trials, cohort studies, data contribution

5.2.1 User Management Endpoints

User endpoints handle profile creation, updating demographics, and managing consent preferences. These are foundational to all other API operations:

GET    /v1/users/{userId}           # Retrieve user profile
POST   /v1/users                     # Create new user
PUT    /v1/users/{userId}           # Update user profile
DELETE /v1/users/{userId}           # Delete user (GDPR compliance)
GET    /v1/users/{userId}/consent   # Get consent preferences
PUT    /v1/users/{userId}/consent   # Update consent preferences

5.2.2 Assessment Endpoints

Assessments are the core of the WIA-AGING API. They represent biological age calculations based on biomarker inputs:

GET    /v1/users/{userId}/assessments              # List all assessments
POST   /v1/users/{userId}/assessments              # Create new assessment
GET    /v1/users/{userId}/assessments/{id}         # Get specific assessment
PUT    /v1/users/{userId}/assessments/{id}         # Update assessment
DELETE /v1/users/{userId}/assessments/{id}         # Delete assessment
GET    /v1/users/{userId}/assessments/latest       # Get most recent assessment
GET    /v1/users/{userId}/assessments/trend        # Get trend analysis

5.3 Authentication & Authorization

Security is paramount when handling sensitive health data. The WIA-AGING API implements a multi-layered authentication and authorization strategy based on industry standards and healthcare regulations (HIPAA, GDPR, HITECH).

5.3.1 OAuth 2.0 Flow

The API supports multiple OAuth 2.0 grant types depending on the use case. For user-facing applications, we recommend the Authorization Code flow with PKCE (Proof Key for Code Exchange), which provides strong security even for public clients like mobile apps.

The authentication flow works as follows:

  1. Application redirects user to WIA authorization server: https://auth.wia-aging.org/authorize
  2. User authenticates and grants permissions (scopes)
  3. Authorization server redirects back with authorization code
  4. Application exchanges code for access token and refresh token
  5. Application uses access token to make API requests
# Step 1: Authorization Request
GET https://auth.wia-aging.org/authorize?
  response_type=code&
  client_id=YOUR_CLIENT_ID&
  redirect_uri=https://yourapp.com/callback&
  scope=aging:read aging:write&
  state=random_state_string&
  code_challenge=BASE64URL(SHA256(code_verifier))&
  code_challenge_method=S256

# Step 4: Token Exchange
POST https://auth.wia-aging.org/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code&
code=AUTHORIZATION_CODE&
redirect_uri=https://yourapp.com/callback&
client_id=YOUR_CLIENT_ID&
code_verifier=CODE_VERIFIER

5.3.2 JWT Token Structure

Access tokens are issued as JWTs (JSON Web Tokens), which are self-contained and can be validated without database lookups. Each JWT contains claims about the user and their permissions:

{
  "iss": "https://auth.wia-aging.org",
  "sub": "user_123456",
  "aud": "https://api.wia-aging.org",
  "exp": 1735689600,
  "iat": 1735686000,
  "scope": "aging:read aging:write biomarkers:read",
  "user_type": "individual",
  "certification_level": "gold",
  "region": "us-east"
}

5.3.3 Scope-Based Authorization

The API uses fine-grained scopes to control access to different resources and operations. Applications request specific scopes during authentication, and users explicitly grant these permissions:

Scope Description Access Level
aging:read Read biological age assessments Read-only
aging:write Create and update assessments Read-write
biomarkers:read Read individual biomarker values Read-only
biomarkers:write Submit new biomarker data Read-write
devices:manage Connect and manage devices Full control
research:contribute Contribute data to research studies Special permission

5.5 Error Handling

Robust error handling is critical for API reliability and developer experience. The WIA-AGING API uses standard HTTP status codes combined with detailed error response bodies to help developers quickly diagnose and fix issues.

5.5.1 Error Response Structure

All error responses follow a consistent JSON structure:

{
  "error": {
    "code": "INVALID_BIOMARKER_VALUE",
    "message": "Glucose value of 450 mg/dL exceeds maximum threshold of 400 mg/dL",
    "status": 400,
    "timestamp": "2025-12-15T10:30:00Z",
    "request_id": "req_abc123def456",
    "details": {
      "field": "biomarkers.glucose",
      "provided_value": 450,
      "max_allowed": 400,
      "unit": "mg/dL"
    },
    "documentation_url": "https://docs.wia-aging.org/errors/INVALID_BIOMARKER_VALUE"
  }
}

5.5.2 Common Error Codes

HTTP Status Error Code Description
400 INVALID_REQUEST Malformed JSON or missing required fields
401 UNAUTHORIZED Missing or invalid authentication token
403 INSUFFICIENT_SCOPE Token lacks required scope for this operation
404 RESOURCE_NOT_FOUND Requested assessment or user does not exist
409 DUPLICATE_ASSESSMENT Assessment for this date already exists
422 VALIDATION_ERROR Data failed business logic validation
429 RATE_LIMIT_EXCEEDED Too many requests, see rate limit headers
500 INTERNAL_SERVER_ERROR Unexpected server error, contact support

5.6 Rate Limiting & Quotas

To ensure fair usage and system stability, the WIA-AGING API implements rate limiting at multiple levels. Rate limits are designed to be generous for normal usage while preventing abuse and ensuring quality of service for all users.

5.6.1 Rate Limit Tiers

Tier Requests/Minute Requests/Day Concurrent Connections
Free 60 10,000 5
Professional 300 100,000 20
Enterprise 1,000 1,000,000 100
Research 2,000 5,000,000 500

5.6.2 Rate Limit Headers

Every API response includes headers that inform clients about their current rate limit status:

HTTP/1.1 200 OK
X-RateLimit-Limit: 300
X-RateLimit-Remaining: 285
X-RateLimit-Reset: 1735689600
X-RateLimit-Tier: professional
Retry-After: 45

5.6.3 Handling Rate Limits

When a client exceeds their rate limit, they receive a 429 status code. Best practices for handling this include:

// Example: Exponential backoff implementation
async function callApiWithRetry(url, options, maxRetries = 3) {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    const response = await fetch(url, options);

    if (response.status === 429) {
      const retryAfter = response.headers.get('Retry-After');
      const waitTime = retryAfter ? parseInt(retryAfter) : Math.pow(2, attempt) * 1000;

      console.log(`Rate limited. Waiting ${waitTime}ms before retry...`);
      await new Promise(resolve => setTimeout(resolve, waitTime));
      continue;
    }

    return response;
  }

  throw new Error('Max retries exceeded');
}

5.7 SDK Overview

While the REST API is fully documented and can be accessed directly via HTTP, we provide official SDKs in multiple languages to accelerate development and reduce boilerplate code. SDKs handle authentication, error handling, retries, and type safety automatically.

5.7.1 TypeScript/JavaScript SDK

import { WIAAgingClient } from '@wia/aging-sdk';

// Initialize client with OAuth credentials
const client = new WIAAgingClient({
  clientId: process.env.WIA_CLIENT_ID,
  clientSecret: process.env.WIA_CLIENT_SECRET,
  environment: 'production', // or 'sandbox'
});

// Create an assessment
const assessment = await client.assessments.create('user_123', {
  assessmentType: 'phenotypic_age',
  assessmentDate: new Date(),
  biomarkers: {
    chronologicalAge: 55,
    albumin: 4.2,
    creatinine: 1.0,
    glucose: 95,
    cReactiveProtein: 1.5,
    lymphocytePercent: 25,
    meanCellVolume: 90,
    redCellDistributionWidth: 13.2,
    alkalinePhosphatase: 70,
    whiteBloodCellCount: 6.5,
  },
});

console.log(`Biological age: ${assessment.results.biologicalAge}`);
console.log(`Age delta: ${assessment.results.ageDelta} years`);

// Get assessment trend
const trend = await client.assessments.getTrend('user_123', {
  startDate: '2024-01-01',
  endDate: '2025-12-31',
  interval: 'monthly',
});

trend.dataPoints.forEach(point => {
  console.log(`${point.date}: ${point.biologicalAge} years`);
});

5.9 API Versioning Strategy

The WIA-AGING API follows semantic versioning principles adapted for RESTful APIs. We maintain backward compatibility within major versions and provide clear migration paths when introducing breaking changes.

Version Support Policy:

  • Each major version is supported for minimum 24 months after release
  • Deprecated versions receive 12-month sunset period with advance notice
  • Security patches applied to all supported versions
  • New features added only to current major version

5.10 Chapter Summary

✅ Key Takeaways:

  1. RESTful design: The API follows REST principles with clear resource hierarchies, standard HTTP methods, and stateless architecture for optimal scalability.
  2. Security first: OAuth 2.0 with PKCE, JWT tokens, and fine-grained scope-based authorization protect sensitive health data while enabling flexible access control.
  3. Developer experience: Consistent JSON formatting, comprehensive error messages, official SDKs in multiple languages, and detailed documentation accelerate integration.
  4. Rate limiting: Tiered rate limits ensure fair usage and system stability while supporting diverse use cases from individual apps to large-scale research platforms.
  5. Event-driven: Webhook support enables real-time event notifications, eliminating polling overhead and enabling responsive applications.
