6.1 Protocol Architecture Overview

While the REST API (covered in Chapter 5) excels at request-response interactions, modern aging and longevity applications increasingly require real-time, bidirectional communication. Wearable devices stream continuous health data, remote monitoring systems need instant alerts, and collaborative research platforms demand synchronized data across multiple participants. The WIA-AGING Protocol Layer addresses these requirements with a comprehensive suite of real-time communication capabilities.

The protocol architecture is built on three foundational layers: the Transport Layer handles the physical transmission of data using technologies like WebSocket and gRPC; the Message Layer defines standardized message formats and serialization; and the Application Layer implements domain-specific logic for aging assessments, device integration, and research coordination.

This layered approach provides several key advantages. First, it enables protocol flexibility—different transport mechanisms can be used based on requirements without changing application logic. Second, it ensures forward compatibility—new message types can be added without breaking existing implementations. Third, it supports hybrid architectures where REST APIs and real-time protocols coexist seamlessly.

Layer Technology Purpose
Application WIA-AGING Protocol Domain-specific message types and workflows
Message Protocol Buffers, JSON Serialization, schema validation, versioning
Transport WebSocket, gRPC, MQTT Reliable, low-latency data transmission
Network TCP/IP, TLS 1.3 Secure network communication

6.2 Transport Layer: WebSocket vs gRPC

The WIA-AGING protocol supports multiple transport mechanisms, each optimized for different use cases. Understanding when to use each transport is critical for building efficient, scalable applications.

6.2.1 WebSocket Transport

WebSocket provides full-duplex communication over a single TCP connection, making it ideal for browser-based applications and scenarios requiring human-readable messages. The protocol upgrade from HTTP to WebSocket happens via a standard handshake:

GET /v1/stream/biomarkers HTTP/1.1
Host: stream.wia-aging.org
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Protocol: wia-aging-v1
Sec-WebSocket-Version: 13
Authorization: Bearer eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...

HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
Sec-WebSocket-Protocol: wia-aging-v1

Once established, both client and server can send messages at any time without the overhead of HTTP headers for each message. This dramatically reduces latency and bandwidth usage for high-frequency updates.

6.2.2 gRPC Transport

gRPC (Google Remote Procedure Call) uses HTTP/2 and Protocol Buffers for highly efficient, type-safe communication. It's particularly well-suited for server-to-server communication, mobile applications, and scenarios requiring strong typing and code generation:

// aging_service.proto
syntax = "proto3";

package wia.aging.v1;

service AgingStreamService {
  // Bidirectional streaming for real-time biomarker data
  rpc StreamBiomarkers(stream BiomarkerData) returns (stream Assessment);

  // Server streaming for continuous monitoring
  rpc MonitorAssessment(MonitorRequest) returns (stream AssessmentUpdate);

  // Client streaming for batch uploads
  rpc UploadBiomarkerBatch(stream BiomarkerData) returns (BatchUploadResponse);
}

message BiomarkerData {
  string user_id = 1;
  int64 timestamp = 2;
  string biomarker_type = 3;
  double value = 4;
  string unit = 5;
  map metadata = 6;
}

message Assessment {
  string assessment_id = 1;
  double biological_age = 2;
  double confidence_score = 3;
  repeated Recommendation recommendations = 4;
}

6.4.3 Server-Side Processing Pipeline

When the server receives streamed biomarker data, it flows through a processing pipeline:

  1. Validation: Check message format, user permissions, device certification
  2. Quality Filtering: Reject measurements below quality threshold
  3. Aggregation: Combine high-frequency data into meaningful intervals
  4. Storage: Persist to time-series database for historical analysis
  5. Analysis: Run real-time algorithms to detect anomalies
  6. Notification: Trigger alerts if thresholds exceeded
  7. Distribution: Forward to subscribed clients and research studies

6.5 Connection Management

Maintaining reliable connections over unreliable networks (especially mobile networks) requires sophisticated connection management strategies. The WIA-AGING protocol implements multiple mechanisms to ensure data integrity and user experience.

6.5.1 Heartbeat and Keep-Alive

// Client sends periodic heartbeat to detect disconnections early
const HEARTBEAT_INTERVAL = 30000;  // 30 seconds
let heartbeatTimer;

function startHeartbeat() {
  heartbeatTimer = setInterval(() => {
    if (ws.readyState === WebSocket.OPEN) {
      ws.send(JSON.stringify({
        type: 'heartbeat.ping',
        timestamp: Date.now()
      }));
    }
  }, HEARTBEAT_INTERVAL);
}

// Server responds with pong
ws.onmessage = (event) => {
  const message = JSON.parse(event.data);

  if (message.type === 'heartbeat.pong') {
    const latency = Date.now() - message.timestamp;
    console.log(`Connection latency: ${latency}ms`);
    updateConnectionQuality(latency);
  }
};

6.5.2 Automatic Reconnection

class ResilientWebSocket {
  constructor(url, options = {}) {
    this.url = url;
    this.options = options;
    this.reconnectDelay = options.reconnectDelay || 1000;
    this.maxReconnectDelay = options.maxReconnectDelay || 30000;
    this.reconnectAttempts = 0;
    this.messageQueue = [];
    this.connect();
  }

  connect() {
    this.ws = new WebSocket(this.url);

    this.ws.onopen = () => {
      console.log('WebSocket connected');
      this.reconnectAttempts = 0;
      this.reconnectDelay = this.options.reconnectDelay || 1000;
      this.flushMessageQueue();
      this.options.onopen?.();
    };

    this.ws.onclose = (event) => {
      if (!event.wasClean) {
        this.scheduleReconnect();
      }
      this.options.onclose?.(event);
    };

    this.ws.onerror = (error) => {
      console.error('WebSocket error:', error);
      this.options.onerror?.(error);
    };

    this.ws.onmessage = this.options.onmessage;
  }

  scheduleReconnect() {
    this.reconnectAttempts++;
    const delay = Math.min(
      this.reconnectDelay * Math.pow(2, this.reconnectAttempts),
      this.maxReconnectDelay
    );

    console.log(`Reconnecting in ${delay}ms (attempt ${this.reconnectAttempts})`);

    setTimeout(() => {
      console.log('Attempting to reconnect...');
      this.connect();
    }, delay);
  }

  send(data) {
    if (this.ws.readyState === WebSocket.OPEN) {
      this.ws.send(data);
    } else {
      // Queue messages during disconnection
      this.messageQueue.push(data);
    }
  }

  flushMessageQueue() {
    while (this.messageQueue.length > 0) {
      const message = this.messageQueue.shift();
      this.ws.send(message);
    }
  }
}

6.6 Security & Encryption

Protecting health data during transmission is non-negotiable. The WIA-AGING protocol implements multiple layers of security, from transport encryption to message-level authentication and end-to-end encryption for sensitive data.

6.6.1 Transport Layer Security (TLS 1.3)

All connections MUST use TLS 1.3, the latest version of the Transport Layer Security protocol. TLS 1.3 provides significant improvements over previous versions:

  • Faster handshakes: 0-RTT (zero round-trip time) resumption for repeat connections
  • Stronger encryption: Removed support for weak ciphers and compression
  • Forward secrecy: Perfect forward secrecy (PFS) is mandatory, ensuring past communications remain secure even if keys are compromised
  • Simplified protocol: Reduced complexity means fewer attack surfaces
Security Feature Implementation Purpose
Transport Encryption TLS 1.3 Encrypt all data in transit
Certificate Validation X.509 with OCSP Verify server identity
Message Signing HMAC-SHA256 Verify message integrity
End-to-End Encryption AES-256-GCM Encrypt sensitive payload data
Key Rotation Automatic every 24h Limit exposure window

6.6.2 Message-Level Authentication

// Every message includes HMAC signature for integrity verification
function signMessage(message, secretKey) {
  const payload = JSON.stringify(message.payload);
  const signature = crypto
    .createHmac('sha256', secretKey)
    .update(payload)
    .digest('hex');

  message.envelope.security = {
    signature: `sha256:${signature}`,
    algorithm: 'HMAC-SHA256'
  };

  return message;
}

// Server verifies signature before processing
function verifyMessage(message, secretKey) {
  const receivedSignature = message.envelope.security.signature.split(':')[1];
  const payload = JSON.stringify(message.payload);
  const computedSignature = crypto
    .createHmac('sha256', secretKey)
    .update(payload)
    .digest('hex');

  if (receivedSignature !== computedSignature) {
    throw new Error('Message signature verification failed');
  }

  return true;
}

6.6.3 End-to-End Encryption for Sensitive Data

For particularly sensitive biomarkers (genetic data, mental health assessments), the protocol supports end-to-end encryption where even the server cannot read the payload:

// Client encrypts sensitive data with user's public key
async function encryptSensitiveData(data, recipientPublicKey) {
  const encrypted = await crypto.subtle.encrypt(
    {
      name: 'RSA-OAEP',
      hash: 'SHA-256'
    },
    recipientPublicKey,
    new TextEncoder().encode(JSON.stringify(data))
  );

  return {
    encrypted: true,
    algorithm: 'RSA-OAEP-256',
    data: arrayBufferToBase64(encrypted)
  };
}

// Only the recipient can decrypt with their private key
async function decryptSensitiveData(encrypted, privateKey) {
  const decrypted = await crypto.subtle.decrypt(
    {
      name: 'RSA-OAEP',
      hash: 'SHA-256'
    },
    privateKey,
    base64ToArrayBuffer(encrypted.data)
  );

  return JSON.parse(new TextDecoder().decode(decrypted));
}

6.7 Quality of Service and Flow Control

In high-volume streaming scenarios, implementing quality of service (QoS) mechanisms prevents overwhelming clients or servers and ensures critical messages are prioritized.

6.7.1 Message Prioritization

// Messages include priority levels
const MessagePriority = {
  CRITICAL: 0,    // Life-threatening alerts
  HIGH: 1,        // Significant health changes
  NORMAL: 2,      // Regular updates
  LOW: 3          // Background sync, analytics
};

// Server processes high-priority messages first
class PriorityQueue {
  constructor() {
    this.queues = [[], [], [], []];  // One queue per priority level
  }

  enqueue(message, priority) {
    this.queues[priority].push(message);
  }

  dequeue() {
    for (let i = 0; i < this.queues.length; i++) {
      if (this.queues[i].length > 0) {
        return this.queues[i].shift();
      }
    }
    return null;
  }
}

6.7.2 Backpressure Handling

// Client implements backpressure when overwhelmed
class StreamProcessor {
  constructor(maxQueueSize = 1000) {
    this.maxQueueSize = maxQueueSize;
    this.queue = [];
    this.processing = false;
  }

  async onMessage(message) {
    if (this.queue.length >= this.maxQueueSize) {
      // Send backpressure signal to server
      this.sendBackpressure({
        reason: 'queue_full',
        current_size: this.queue.length,
        max_size: this.maxQueueSize
      });
      return;
    }

    this.queue.push(message);

    if (!this.processing) {
      this.processQueue();
    }
  }

  sendBackpressure(details) {
    ws.send(JSON.stringify({
      type: 'flow.backpressure',
      details: details
    }));
  }

  async processQueue() {
    this.processing = true;

    while (this.queue.length > 0) {
      const message = this.queue.shift();
      await this.processMessage(message);
    }

    this.processing = false;

    // Notify server that we're ready for more
    ws.send(JSON.stringify({ type: 'flow.ready' }));
  }
}
