📊 Chapter 4: Phase 1 - Data Format

4.1 Data Model Overview for Aging Assessment

The WIA-AGING Phase 1 Data Format establishes a comprehensive data model for representing all aspects of aging assessment and intervention. This model is designed to be both human-readable and machine-processable, using JSON (JavaScript Object Notation) as the serialization format due to its widespread support, flexibility, and ease of use.

The data model is organized around several core concepts that form the foundation of aging assessment:

  • Subject: The individual being assessed (with privacy-preserving identifiers)
  • Assessment: A comprehensive evaluation at a specific point in time
  • BiologicalAge: Estimates of biological age from various methods
  • Biomarker: Individual measurable indicators of aging processes
  • Intervention: Actions taken to influence aging trajectory
  • Outcome: Health events and functional status over time
  • Metadata: Information about data provenance, quality, and confidence

These concepts are related through a hierarchical structure: an Assessment contains multiple BiologicalAge estimates and Biomarker measurements for a Subject, potentially influenced by Interventions, with Metadata tracking provenance and quality throughout.

Core Data Type Purpose Key Fields Relationships
Subject Identify individual being assessed id, demographics, consent Has many Assessments
Assessment Comprehensive evaluation snapshot timestamp, biologicalAges, biomarkers Belongs to Subject, contains BiologicalAge & Biomarker
BiologicalAge Age estimate from specific method method, value, confidenceInterval Part of Assessment
Biomarker Individual measurable indicator type, value, unit, referenceRange Part of Assessment
Intervention Action to influence aging type, startDate, dosage, adherence Associated with Subject

4.2 JSON Schema Definition

The WIA-AGING Standard uses JSON Schema to formally define the structure, data types, and validation rules for all data types. JSON Schema provides several benefits:

  • Machine-Readable Validation: Automated tools can verify that data conforms to the standard
  • Documentation: Schema serves as formal documentation of data structures
  • Code Generation: Type definitions can be automatically generated for programming languages
  • Interoperability Testing: Implementations can be tested for conformance

The schema uses JSON Schema Draft 2020-12, the latest stable version, ensuring compatibility with modern tooling and validation libraries.

Why JSON Instead of XML or Protocol Buffers?

JSON was chosen for several reasons: it's human-readable (important for debugging and transparency), has excellent library support in all major programming languages, is lightweight compared to XML, and is the de facto standard for web APIs. While Protocol Buffers are more efficient for high-volume data transfer, JSON's accessibility and tooling ecosystem make it better suited for a standard that will be implemented by diverse organizations with varying technical sophistication.

4.3 Core Data Type: BiologicalAge

The BiologicalAge type represents an estimate of biological age calculated from a specific method. Different biological age clocks (Horvath, Hannum, PhenoAge, GrimAge, DunedinPACE, etc.) are supported, each with its own methodology and interpretation.

BiologicalAge Schema

{
  "$schema": "http://json-schema.org/draft/2020-12/schema",
  "$id": "https://aging.wiastandards.com/schemas/v1/BiologicalAge.json",
  "title": "BiologicalAge",
  "description": "Biological age estimate from a specific method",
  "type": "object",
  "required": ["method", "value", "unit", "timestamp"],
  "properties": {
    "method": {
      "type": "string",
      "enum": [
        "horvath",
        "hannum",
        "phenoage",
        "grimage",
        "dunedinpace",
        "telomere_length",
        "glycan_age",
        "phenotypic_age_levine",
        "custom"
      ],
      "description": "The biological age calculation method"
    },
    "value": {
      "type": "number",
      "minimum": 0,
      "maximum": 150,
      "description": "The biological age estimate in years"
    },
    "unit": {
      "type": "string",
      "const": "years",
      "description": "Unit of measurement (always years)"
    },
    "timestamp": {
      "type": "string",
      "format": "date-time",
      "description": "ISO 8601 timestamp when sample was collected"
    },
    "confidenceInterval": {
      "type": "object",
      "properties": {
        "lower": {"type": "number", "minimum": 0},
        "upper": {"type": "number", "maximum": 150},
        "confidence": {"type": "number", "minimum": 0, "maximum": 1}
      },
      "description": "Confidence interval for the estimate"
    },
    "accelerationVsChronological": {
      "type": "number",
      "description": "Years of age acceleration (positive) or deceleration (negative)"
    },
    "metadata": {
      "$ref": "#/definitions/Metadata"
    }
  }
}

BiologicalAge Example

{
  "method": "grimage",
  "value": 42.3,
  "unit": "years",
  "timestamp": "2024-12-15T09:30:00Z",
  "confidenceInterval": {
    "lower": 39.8,
    "upper": 44.8,
    "confidence": 0.95
  },
  "accelerationVsChronological": -3.7,
  "metadata": {
    "provider": "TruDiagnostic",
    "platform": "EPIC Array",
    "sampleType": "whole_blood",
    "processingDate": "2024-12-18T14:22:00Z",
    "qualityScore": 0.94
  }
}

In this example, a 46-year-old individual has a GrimAge biological age of 42.3 years, indicating age deceleration of 3.7 years. The 95% confidence interval ranges from 39.8 to 44.8 years.

4.4 Core Data Type: Biomarker

The Biomarker type represents individual measurable indicators of biological processes related to aging. This includes laboratory values (blood tests), physical measurements (grip strength, walking speed), imaging results, and functional assessments.

Biomarker Schema

{
  "$schema": "http://json-schema.org/draft/2020-12/schema",
  "$id": "https://aging.wiastandards.com/schemas/v1/Biomarker.json",
  "title": "Biomarker",
  "description": "Individual measurable indicator of aging",
  "type": "object",
  "required": ["type", "value", "unit", "timestamp"],
  "properties": {
    "type": {
      "type": "string",
      "description": "Biomarker type (preferably LOINC code)",
      "examples": ["crp", "glucose_fasting", "hba1c", "telomere_length"]
    },
    "loincCode": {
      "type": "string",
      "pattern": "^\\d{1,5}-\\d{1}$",
      "description": "LOINC code if applicable"
    },
    "value": {
      "type": "number",
      "description": "Measured value"
    },
    "unit": {
      "type": "string",
      "description": "Unit of measurement (UCUM preferred)",
      "examples": ["mg/L", "mg/dL", "mmol/L", "%"]
    },
    "timestamp": {
      "type": "string",
      "format": "date-time"
    },
    "referenceRange": {
      "type": "object",
      "properties": {
        "lower": {"type": "number"},
        "upper": {"type": "number"},
        "ageAdjusted": {"type": "boolean"},
        "population": {"type": "string"}
      }
    },
    "interpretation": {
      "type": "string",
      "enum": ["low", "normal", "high", "critical_low", "critical_high"]
    },
    "metadata": {
      "$ref": "#/definitions/Metadata"
    }
  }
}

4.6 Core Data Type: HealthMetric

The HealthMetric type captures continuous or frequent health measurements, typically from wearable devices or home monitoring equipment.

Metric Type Source Frequency Aging Relevance
Heart Rate Variability Wearables (Oura, Whoop) Continuous Autonomic nervous system health
Sleep Stages Wearables, sleep trackers Nightly Recovery, cognitive function
Glucose (CGM) Continuous glucose monitors Every 5-15 min Metabolic health
Blood Pressure Home monitors, smartwatches Daily to continuous Cardiovascular health
Activity/Steps Smartphones, fitness trackers Continuous Physical activity, frailty risk

4.8 Metadata Structure

Metadata is critical for establishing data provenance, quality, and trustworthiness. The WIA-AGING Standard requires comprehensive metadata for all data types.

{

4.9 Validation Rules

Beyond the structural validation provided by JSON Schema, the WIA-AGING Standard defines semantic validation rules that ensure data quality and consistency.

Validation Rule Type Description Severity
Biological age range Range check BiologicalAge.value must be 0-150 years Error
Timestamp consistency Temporal collectionDate ≤ processingDate ≤ current date Error
Confidence interval Logical lower ≤ value ≤ upper Error
Reference range Logical Biomarker reference range lower < upper Error
LOINC code format Format Matches pattern: \d{1,5}-\d{1} Warning
Unit consistency Semantic Unit matches biomarker type (e.g., glucose in mg/dL or mmol/L) Warning

Technical Note: Validation Levels

Error: Data fails validation and should be rejected by conformant systems

Warning: Data is processable but may have quality issues that should be flagged

Info: Best practice recommendations that don't affect validity

4.11 Migration from Legacy Formats

Many organizations have existing aging data in various formats. The WIA-AGING Standard provides migration guidance for common legacy formats:

From Spreadsheet Data

Most research labs and small providers store data in Excel or CSV files. Migration involves:

  1. Column Mapping: Map spreadsheet columns to WIA-AGING fields
  2. Data Cleaning: Standardize units, date formats, and terminology
  3. Metadata Addition: Add provenance information not in original data
  4. Validation: Run through WIA-AGING validators to catch issues

Migration Tools Available

The WIA-AGING Standard provides open-source migration tools for common formats:

  • csv-to-wia-aging: Convert CSV/Excel to WIA-AGING JSON
  • fhir-to-wia-aging: Extract aging data from FHIR resources
  • 23andme-to-wia-aging: Convert consumer genomics data
  • oura-to-wia-aging: Convert Oura Ring sleep and HRV data

All tools available at: github.com/WIA-Official/wia-aging-tools

From Proprietary APIs

For data currently accessible only through proprietary APIs, migration strategies include:

  • API Wrapper Development: Create adapters that fetch data via proprietary API and transform to WIA-AGING format
  • One-Time Export: Use bulk export features to download all data, then transform offline
  • Incremental Sync: Periodically fetch new data and append to WIA-AGING store
  • Vendor Engagement: Work with vendors to add native WIA-AGING export capability

4.12 Chapter Summary

✅ Key Takeaways:

  1. Comprehensive data model: The WIA-AGING Phase 1 data format covers all aspects of aging assessment through core types (Subject, Assessment, BiologicalAge, Biomarker, Intervention, Outcome) with rich metadata.
  2. JSON Schema foundation: Formal schemas enable machine validation, automatic code generation, documentation, and interoperability testing while maintaining human readability.
  3. Biological age support: Multiple biological age methods (Horvath, GrimAge, PhenoAge, DunedinPACE, telomere length, etc.) are supported with confidence intervals and acceleration metrics.
  4. Standard biomarker encoding: Biomarkers use LOINC codes where available, UCUM units, reference ranges, and interpretation fields for clinical integration.
  5. Migration path from legacy: Clear guidance and open-source tools help organizations migrate from spreadsheets, proprietary APIs, and other legacy formats to WIA-AGING standard.
