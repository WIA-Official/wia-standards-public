6.1 프로토콜 아키텍처 개요

제5장에서 다룬 REST API는 요청-응답 상호작용에서 뛰어나지만, 현대 노화 및 장수 애플리케이션은 점점 더 실시간 양방향 통신을 필요로 합니다. 웨어러블 기기는 연속적인 건강 데이터를 스트리밍하고, 원격 모니터링 시스템은 즉각적인 경고가 필요하며, 협업 연구 플랫폼은 여러 참가자 간에 동기화된 데이터를 요구합니다. WIA-AGING 프로토콜 계층은 포괄적인 실시간 통신 기능 모음으로 이러한 요구 사항을 해결합니다.

프로토콜 아키텍처는 세 가지 기본 계층으로 구축됩니다: 전송 계층은 WebSocket 및 gRPC와 같은 기술을 사용하여 데이터의 물리적 전송을 처리하고, 메시지 계층은 표준화된 메시지 형식과 직렬화를 정의하며, 애플리케이션 계층은 노화 평가, 기기 통합, 연구 조정을 위한 도메인별 로직을 구현합니다.

이러한 계층화된 접근 방식은 여러 주요 이점을 제공합니다. 첫째, 프로토콜 유연성을 가능하게 합니다 - 애플리케이션 로직을 변경하지 않고 요구 사항에 따라 다양한 전송 메커니즘을 사용할 수 있습니다. 둘째, 향후 호환성을 보장합니다 - 기존 구현을 깨뜨리지 않고 새로운 메시지 유형을 추가할 수 있습니다. 셋째, REST API와 실시간 프로토콜이 원활하게 공존하는 하이브리드 아키텍처를 지원합니다.

계층 기술 목적
애플리케이션 WIA-AGING 프로토콜 도메인별 메시지 유형 및 워크플로우
메시지 Protocol Buffers, JSON 직렬화, 스키마 검증, 버전 관리
전송 WebSocket, gRPC, MQTT 안정적이고 저지연 데이터 전송
네트워크 TCP/IP, TLS 1.3 안전한 네트워크 통신

6.2 전송 계층: WebSocket vs gRPC

WIA-AGING 프로토콜은 각각 다른 사용 사례에 최적화된 여러 전송 메커니즘을 지원합니다. 각 전송을 언제 사용할지 이해하는 것은 효율적이고 확장 가능한 애플리케이션을 구축하는 데 중요합니다.

6.2.1 WebSocket 전송

WebSocket은 단일 TCP 연결을 통해 전이중 통신을 제공하여 브라우저 기반 애플리케이션과 사람이 읽을 수 있는 메시지가 필요한 시나리오에 이상적입니다. HTTP에서 WebSocket으로의 프로토콜 업그레이드는 표준 핸드셰이크를 통해 발생합니다:

GET /v1/stream/biomarkers HTTP/1.1
Host: stream.wia-aging.org
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Protocol: wia-aging-v1
Sec-WebSocket-Version: 13
Authorization: Bearer eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...

HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
Sec-WebSocket-Protocol: wia-aging-v1

연결이 설정되면 클라이언트와 서버 모두 각 메시지에 대한 HTTP 헤더 오버헤드 없이 언제든지 메시지를 보낼 수 있습니다. 이는 고빈도 업데이트에 대한 지연 시간과 대역폭 사용을 크게 줄입니다.

6.2.2 gRPC 전송

gRPC(Google Remote Procedure Call)는 HTTP/2 및 Protocol Buffers를 사용하여 매우 효율적이고 타입 안전한 통신을 제공합니다. 서버 간 통신, 모바일 애플리케이션, 강력한 타이핑 및 코드 생성이 필요한 시나리오에 특히 적합합니다:

// aging_service.proto
syntax = "proto3";

package wia.aging.v1;

service AgingStreamService {
  // 실시간 바이오마커 데이터를 위한 양방향 스트리밍
  rpc StreamBiomarkers(stream BiomarkerData) returns (stream Assessment);

  // 연속 모니터링을 위한 서버 스트리밍
  rpc MonitorAssessment(MonitorRequest) returns (stream AssessmentUpdate);

  // 일괄 업로드를 위한 클라이언트 스트리밍
  rpc UploadBiomarkerBatch(stream BiomarkerData) returns (BatchUploadResponse);
}

message BiomarkerData {
  string user_id = 1;
  int64 timestamp = 2;
  string biomarker_type = 3;
  double value = 4;
  string unit = 5;
  map metadata = 6;
}

message Assessment {
  string assessment_id = 1;
  double biological_age = 2;
  double confidence_score = 3;
  repeated Recommendation recommendations = 4;
}

6.4 웨어러블 데이터를 위한 실시간 스트리밍

웨어러블 기기는 심박수, 활동 수준, 수면 단계, 혈중 산소 등 연속적인 건강 데이터 스트림을 생성합니다. WIA-AGING 프로토콜은 데이터 품질과 사용자 개인 정보를 유지하면서 이러한 데이터의 효율적인 스트리밍을 가능하게 합니다.

6.4.1 스트림 시작

// 클라이언트가 스트리밍 연결 시작

6.4.3 서버 측 처리 파이프라인

서버가 스트리밍된 바이오마커 데이터를 수신하면 처리 파이프라인을 통과합니다:

  1. 검증: 메시지 형식, 사용자 권한, 기기 인증 확인
  2. 품질 필터링: 품질 임계값 이하의 측정 거부
  3. 집계: 고빈도 데이터를 의미 있는 간격으로 결합
  4. 저장: 시계열 데이터베이스에 지속하여 이력 분석
  5. 분석: 실시간 알고리즘을 실행하여 이상 감지
  6. 알림: 임계값 초과 시 경고 트리거
  7. 배포: 구독된 클라이언트 및 연구에 전달

6.5 연결 관리

신뢰할 수 없는 네트워크(특히 모바일 네트워크)를 통해 안정적인 연결을 유지하려면 정교한 연결 관리 전략이 필요합니다. WIA-AGING 프로토콜은 데이터 무결성과 사용자 경험을 보장하기 위한 여러 메커니즘을 구현합니다.

6.5.2 자동 재연결

class ResilientWebSocket {
  constructor(url, options = {}) {
    this.url = url;
    this.options = options;
    this.reconnectDelay = options.reconnectDelay || 1000;
    this.maxReconnectDelay = options.maxReconnectDelay || 30000;
    this.reconnectAttempts = 0;
    this.messageQueue = [];
    this.connect();
  }

  connect() {
    this.ws = new WebSocket(this.url);

    this.ws.onopen = () => {
      console.log('WebSocket 연결됨');
      this.reconnectAttempts = 0;
      this.reconnectDelay = this.options.reconnectDelay || 1000;
      this.flushMessageQueue();
      this.options.onopen?.();
    };

    this.ws.onclose = (event) => {
      if (!event.wasClean) {
        this.scheduleReconnect();
      }
      this.options.onclose?.(event);
    };

    this.ws.onerror = (error) => {
      console.error('WebSocket 오류:', error);
      this.options.onerror?.(error);
    };

    this.ws.onmessage = this.options.onmessage;
  }

  scheduleReconnect() {
    this.reconnectAttempts++;
    const delay = Math.min(
      this.reconnectDelay * Math.pow(2, this.reconnectAttempts),
      this.maxReconnectDelay
    );

    console.log(`${delay}ms 후 재연결 (시도 ${this.reconnectAttempts})`);

    setTimeout(() => {
      console.log('재연결 시도 중...');
      this.connect();
    }, delay);
  }

  send(data) {
    if (this.ws.readyState === WebSocket.OPEN) {
      this.ws.send(data);
    } else {
      // 연결 끊김 중 메시지 대기열
      this.messageQueue.push(data);
    }
  }

  flushMessageQueue() {
    while (this.messageQueue.length > 0) {
      const message = this.messageQueue.shift();
      this.ws.send(message);
    }
  }
}

6.6 보안 및 암호화

전송 중 건강 데이터 보호는 타협할 수 없습니다. WIA-AGING 프로토콜은 전송 암호화에서 메시지 수준 인증 및 민감한 데이터에 대한 종단 간 암호화까지 여러 보안 계층을 구현합니다.

6.6.1 전송 계층 보안 (TLS 1.3)

모든 연결은 최신 버전의 전송 계층 보안 프로토콜인 TLS 1.3을 사용해야 합니다. TLS 1.3은 이전 버전에 비해 상당한 개선 사항을 제공합니다:

  • 더 빠른 핸드셰이크: 반복 연결을 위한 0-RTT(왕복 시간 0) 재개
  • 더 강력한 암호화: 약한 암호 및 압축에 대한 지원 제거
  • 순방향 비밀성: 완벽한 순방향 비밀성(PFS)이 필수이며, 키가 손상되어도 과거 통신이 안전하게 유지됨
  • 단순화된 프로토콜: 복잡성 감소로 공격 표면 감소
보안 기능 구현 목적
전송 암호화 TLS 1.3 전송 중인 모든 데이터 암호화
인증서 검증 OCSP가 있는 X.509 서버 신원 확인
메시지 서명 HMAC-SHA256 메시지 무결성 확인
종단 간 암호화 AES-256-GCM 민감한 페이로드 데이터 암호화
키 순환 24시간마다 자동 노출 기간 제한

6.6.2 메시지 수준 인증

// 모든 메시지에는 무결성 확인을 위한 HMAC 서명 포함
function signMessage(message, secretKey) {
  const payload = JSON.stringify(message.payload);
  const signature = crypto
    .createHmac('sha256', secretKey)
    .update(payload)
    .digest('hex');

  message.envelope.security = {
    signature: `sha256:${signature}`,
    algorithm: 'HMAC-SHA256'
  };

  return message;
}

// 서버가 처리 전 서명 확인
function verifyMessage(message, secretKey) {
  const receivedSignature = message.envelope.security.signature.split(':')[1];
  const payload = JSON.stringify(message.payload);
  const computedSignature = crypto
    .createHmac('sha256', secretKey)
    .update(payload)
    .digest('hex');

  if (receivedSignature !== computedSignature) {
    throw new Error('메시지 서명 확인 실패');
  }

  return true;
}

6.6.3 민감한 데이터에 대한 종단 간 암호화

특히 민감한 바이오마커(유전 데이터, 정신 건강 평가)의 경우, 프로토콜은 서버도 페이로드를 읽을 수 없는 종단 간 암호화를 지원합니다:

// 클라이언트가 사용자의 공개 키로 민감한 데이터 암호화
async function encryptSensitiveData(data, recipientPublicKey) {
  const encrypted = await crypto.subtle.encrypt(
    {
      name: 'RSA-OAEP',
      hash: 'SHA-256'
    },
    recipientPublicKey,
    new TextEncoder().encode(JSON.stringify(data))
  );

  return {
    encrypted: true,
    algorithm: 'RSA-OAEP-256',
    data: arrayBufferToBase64(encrypted)
  };
}

// 수신자만 개인 키로 복호화 가능
async function decryptSensitiveData(encrypted, privateKey) {
  const decrypted = await crypto.subtle.decrypt(
    {
      name: 'RSA-OAEP',
      hash: 'SHA-256'
    },
    privateKey,
    base64ToArrayBuffer(encrypted.data)
  );

  return JSON.parse(new TextDecoder().decode(decrypted));
}

6.7 서비스 품질 및 흐름 제어

대용량 스트리밍 시나리오에서 서비스 품질(QoS) 메커니즘을 구현하면 클라이언트나 서버를 압도하는 것을 방지하고 중요한 메시지가 우선 순위를 받도록 보장합니다.

6.7.1 메시지 우선 순위 지정

// 메시지에 우선 순위 수준 포함
const MessagePriority = {
  CRITICAL: 0,    // 생명을 위협하는 경고
  HIGH: 1,        // 중요한 건강 변화
  NORMAL: 2,      // 정기 업데이트
  LOW: 3          // 백그라운드 동기화, 분석
};

// 서버가 높은 우선 순위 메시지를 먼저 처리
class PriorityQueue {
  constructor() {
    this.queues = [[], [], [], []];  // 우선 순위 수준당 하나의 대기열
  }

  enqueue(message, priority) {
    this.queues[priority].push(message);
  }

  dequeue() {
    for (let i = 0; i < this.queues.length; i++) {
      if (this.queues[i].length > 0) {
        return this.queues[i].shift();
      }
    }
    return null;
  }
}

6.8 요약

✅ 핵심 내용:

  1. 다중 전송 지원: 브라우저 기반 실시간 통신을 위한 WebSocket, 효율적인 서버 간 및 모바일 통합을 위한 gRPC, 올바른 전송 선택을 위한 명확한 지침.
  2. 표준화된 메시징: 봉투-페이로드 패턴은 버전 관리, 라우팅, 상관 관계를 가능하게 하면서 다양한 바이오마커 유형과 사용 사례에 대한 유연성을 유지합니다.
  3. 실시간 스트리밍: 품질 필터링, 집계, 실시간 분석을 통한 웨어러블의 연속 데이터 스트리밍으로 사전 건강 모니터링 가능.
  4. 연결 복원력: 지수 백오프를 사용한 자동 재연결, 메시지 대기열, 하트비트 메커니즘으로 신뢰할 수 없는 네트워크에서도 안정적인 데이터 전송 보장.
  5. 보안 계층: 전송용 TLS 1.3, 무결성용 메시지 서명, 민감한 데이터용 선택적 종단 간 암호화로 건강 정보에 대한 심층 방어 제공.
