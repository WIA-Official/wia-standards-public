5.1 API 설계 철학

WIA-AGING API는 명확한 철학으로 설계되었습니다: 단순성, 보안성, 확장성. 노화 및 장수 생태계에서 API는 데이터 소스(웨어러블, 실험실 시스템, 전자건강기록)와 애플리케이션(임상의 대시보드, 연구 플랫폼, 소비자 앱) 간의 중요한 가교 역할을 합니다. 우리의 설계 철학은 데이터 프라이버시와 규제 준수에 대한 최고 수준의 표준을 유지하면서 개발자 경험을 우선시합니다.

API 설계의 핵심은 점진적 공개(progressive disclosure) 원칙입니다. 개발자는 사용자의 생물학적 나이 추정치를 검색하는 것과 같은 간단한 사용 사례로 빠르게 시작할 수 있어야 하며, 필요에 따라 일괄 처리, 웹훅 구독, 실시간 데이터 스트리밍과 같은 고급 기능에 접근할 수 있어야 합니다. 이 접근 방식은 첫 API 호출까지의 시간을 단축하면서 복잡한 엔터프라이즈 통합을 지원합니다.

API는 REST 아키텍처 원칙을 준수하여 전 세계 개발자에게 친숙합니다. 표준 HTTP 메서드(GET, POST, PUT, DELETE), 상태 코드, 헤더를 사용합니다. 리소스는 노화 데이터의 논리적 구조를 반영하는 명확하고 계층적인 URI로 식별됩니다. 예를 들어, /users/{userId}/assessments/{assessmentId}는 사용자와 노화 평가 간의 관계를 명확하게 나타냅니다.

또 다른 기본 원칙은 무상태성(statelessness)입니다. 각 API 요청은 서버 측 세션 상태에 의존하지 않고 처리하는 데 필요한 모든 정보를 포함합니다. 이를 통해 수평적 확장성이 가능하여 API가 분산 인프라에서 수백만 명의 동시 사용자에게 서비스를 제공할 수 있습니다. 인증은 OAuth 2.0 및 JWT 토큰을 통해 처리되어 안전하고 무상태적인 인증을 보장합니다.

또한 첫날부터 API 버전 관리를 수용합니다. API 버전은 URL 경로에 포함되어(예: /v1/assessments) 기존 통합에 대한 이전 버전과의 호환성을 유지하면서 향후 버전에서 중대한 변경 사항을 도입할 수 있습니다. 이는 시스템이 긴 업그레이드 주기를 갖는 의료 분야에서 매우 중요합니다.

5.2 RESTful 엔드포인트 개요

WIA-AGING API는 노화 생태계의 주요 엔터티를 나타내는 여러 리소스 컬렉션으로 구성됩니다. 다음은 주요 엔드포인트 카테고리와 그 목적에 대한 포괄적인 개요입니다:

리소스 엔드포인트 패턴 목적
사용자 /v1/users 사용자 프로필 관리, 인구통계, 동의 추적
평가 /v1/users/{userId}/assessments 생물학적 나이 평가, 바이오마커 데이터, 이력 추적
바이오마커 /v1/biomarkers 개별 바이오마커 값, 참조 범위, 추세
개입 /v1/interventions 생활 습관 개입, 처방, 준수도 추적
기기 /v1/devices 연결된 웨어러블, 연속 모니터링 기기
연구 /v1/studies 임상 시험, 코호트 연구, 데이터 기여

5.2.1 사용자 관리 엔드포인트

사용자 엔드포인트는 프로필 생성, 인구통계 업데이트, 동의 기본 설정 관리를 처리합니다. 이는 다른 모든 API 작업의 기초입니다:

GET    /v1/users/{userId}           # 사용자 프로필 검색
POST   /v1/users                     # 새 사용자 생성
PUT    /v1/users/{userId}           # 사용자 프로필 업데이트
DELETE /v1/users/{userId}           # 사용자 삭제 (GDPR 준수)
GET    /v1/users/{userId}/consent   # 동의 기본 설정 가져오기
PUT    /v1/users/{userId}/consent   # 동의 기본 설정 업데이트

5.2.2 평가 엔드포인트

평가는 WIA-AGING API의 핵심입니다. 바이오마커 입력을 기반으로 한 생물학적 나이 계산을 나타냅니다:

GET    /v1/users/{userId}/assessments              # 모든 평가 나열
POST   /v1/users/{userId}/assessments              # 새 평가 생성
GET    /v1/users/{userId}/assessments/{id}         # 특정 평가 가져오기
PUT    /v1/users/{userId}/assessments/{id}         # 평가 업데이트
DELETE /v1/users/{userId}/assessments/{id}         # 평가 삭제
GET    /v1/users/{userId}/assessments/latest       # 최근 평가 가져오기
GET    /v1/users/{userId}/assessments/trend        # 추세 분석 가져오기

5.3 인증 및 권한 부여

민감한 건강 데이터를 처리할 때 보안은 무엇보다 중요합니다. WIA-AGING API는 업계 표준 및 의료 규정(HIPAA, GDPR, HITECH)을 기반으로 한 다층 인증 및 권한 부여 전략을 구현합니다.

5.3.1 OAuth 2.0 플로우

API는 사용 사례에 따라 여러 OAuth 2.0 부여 유형을 지원합니다. 사용자 대면 애플리케이션의 경우 PKCE(Proof Key for Code Exchange)를 사용한 인증 코드 플로우를 권장합니다. 이는 모바일 앱과 같은 공개 클라이언트에도 강력한 보안을 제공합니다.

인증 플로우는 다음과 같이 작동합니다:

  1. 애플리케이션이 사용자를 WIA 인증 서버로 리디렉션: https://auth.wia-aging.org/authorize
  2. 사용자가 인증하고 권한(범위)을 부여합니다
  3. 인증 서버가 인증 코드와 함께 리디렉션합니다
  4. 애플리케이션이 코드를 액세스 토큰 및 갱신 토큰으로 교환합니다
  5. 애플리케이션이 액세스 토큰을 사용하여 API 요청을 합니다
# 1단계: 인증 요청
GET https://auth.wia-aging.org/authorize?
  response_type=code&
  client_id=YOUR_CLIENT_ID&
  redirect_uri=https://yourapp.com/callback&
  scope=aging:read aging:write&
  state=random_state_string&
  code_challenge=BASE64URL(SHA256(code_verifier))&
  code_challenge_method=S256

# 4단계: 토큰 교환
POST https://auth.wia-aging.org/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code&
code=AUTHORIZATION_CODE&
redirect_uri=https://yourapp.com/callback&
client_id=YOUR_CLIENT_ID&
code_verifier=CODE_VERIFIER

5.3.2 JWT 토큰 구조

액세스 토큰은 JWT(JSON Web Token)로 발급되며, 이는 자체 포함되어 있고 데이터베이스 조회 없이 검증할 수 있습니다. 각 JWT는 사용자와 권한에 대한 클레임을 포함합니다:

{
  "iss": "https://auth.wia-aging.org",
  "sub": "user_123456",
  "aud": "https://api.wia-aging.org",
  "exp": 1735689600,
  "iat": 1735686000,
  "scope": "aging:read aging:write biomarkers:read",
  "user_type": "individual",
  "certification_level": "gold",
  "region": "us-east"
}

5.3.3 범위 기반 권한 부여

API는 세밀한 범위를 사용하여 다양한 리소스 및 작업에 대한 액세스를 제어합니다. 애플리케이션은 인증 중에 특정 범위를 요청하고 사용자는 명시적으로 이러한 권한을 부여합니다:

범위 설명 액세스 수준
aging:read 생물학적 나이 평가 읽기 읽기 전용
aging:write 평가 생성 및 업데이트 읽기-쓰기
biomarkers:read 개별 바이오마커 값 읽기 읽기 전용
biomarkers:write 새 바이오마커 데이터 제출 읽기-쓰기
devices:manage 기기 연결 및 관리 전체 제어
research:contribute 연구에 데이터 기여 특별 권한

5.5 오류 처리

강력한 오류 처리는 API 안정성과 개발자 경험에 매우 중요합니다. WIA-AGING API는 표준 HTTP 상태 코드와 자세한 오류 응답 본문을 결합하여 개발자가 문제를 빠르게 진단하고 수정할 수 있도록 합니다.

5.5.1 오류 응답 구조

모든 오류 응답은 일관된 JSON 구조를 따릅니다:

{
  "error": {
    "code": "INVALID_BIOMARKER_VALUE",
    "message": "포도당 값 450 mg/dL이 최대 임계값 400 mg/dL을 초과합니다",
    "status": 400,
    "timestamp": "2025-12-15T10:30:00Z",
    "request_id": "req_abc123def456",
    "details": {
      "field": "biomarkers.glucose",
      "provided_value": 450,
      "max_allowed": 400,
      "unit": "mg/dL"
    },
    "documentation_url": "https://docs.wia-aging.org/errors/INVALID_BIOMARKER_VALUE"
  }
}

5.5.2 일반적인 오류 코드

HTTP 상태 오류 코드 설명
400 INVALID_REQUEST 잘못된 JSON 또는 필수 필드 누락
401 UNAUTHORIZED 인증 토큰 누락 또는 유효하지 않음
403 INSUFFICIENT_SCOPE 이 작업에 필요한 범위가 토큰에 없음

422 VALIDATION_ERROR 데이터가 비즈니스 로직 검증 실패
429 RATE_LIMIT_EXCEEDED 요청이 너무 많음, 속도 제한 헤더 참조
500 INTERNAL_SERVER_ERROR 예상치 못한 서버 오류, 지원팀에 문의

5.6 속도 제한 및 할당량

공정한 사용과 시스템 안정성을 보장하기 위해 WIA-AGING API는 여러 수준에서 속도 제한을 구현합니다. 속도 제한은 남용을 방지하고 모든 사용자에게 서비스 품질을 보장하면서 정상적인 사용에 대해 관대하게 설계되었습니다.

5.6.1 속도 제한 등급

등급 요청/분 요청/일 동시 연결
무료 60 10,000 5
프로페셔널 300 100,000 20
엔터프라이즈 1,000 1,000,000 100
연구 2,000 5,000,000 500

5.6.2 속도 제한 헤더

모든 API 응답에는 클라이언트가 현재 속도 제한 상태를 알 수 있는 헤더가 포함됩니다:

HTTP/1.1 200 OK
X-RateLimit-Limit: 300
X-RateLimit-Remaining: 285
X-RateLimit-Reset: 1735689600
X-RateLimit-Tier: professional
Retry-After: 45

5.6.3 속도 제한 처리

클라이언트가 속도 제한을 초과하면 429 상태 코드를 받습니다. 이를 처리하기 위한 모범 사례는 다음과 같습니다:

// 예제: 지수 백오프 구현
async function callApiWithRetry(url, options, maxRetries = 3) {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    const response = await fetch(url, options);

    if (response.status === 429) {
      const retryAfter = response.headers.get('Retry-After');
      const waitTime = retryAfter ? parseInt(retryAfter) : Math.pow(2, attempt) * 1000;

      console.log(`속도 제한됨. 재시도 전 ${waitTime}ms 대기 중...`);
      await new Promise(resolve => setTimeout(resolve, waitTime));
      continue;
    }

    return response;
  }

  throw new Error('최대 재시도 횟수 초과');
}

5.7 SDK 개요

REST API는 완전히 문서화되어 있고 HTTP를 통해 직접 액세스할 수 있지만, 개발을 가속화하고 보일러플레이트 코드를 줄이기 위해 여러 언어로 공식 SDK를 제공합니다. SDK는 인증, 오류 처리, 재시도, 타입 안전성을 자동으로 처리합니다.

5.7.1 TypeScript/JavaScript SDK

import { WIAAgingClient } from '@wia/aging-sdk';

// OAuth 자격 증명으로 클라이언트 초기화
const client = new WIAAgingClient({
  clientId: process.env.WIA_CLIENT_ID,
  clientSecret: process.env.WIA_CLIENT_SECRET,
  environment: 'production', // 또는 'sandbox'
});

// 평가 생성
const assessment = await client.assessments.create('user_123', {
  assessmentType: 'phenotypic_age',
  assessmentDate: new Date(),
  biomarkers: {
    chronologicalAge: 55,
    albumin: 4.2,
    creatinine: 1.0,
    glucose: 95,
    cReactiveProtein: 1.5,
    lymphocytePercent: 25,
    meanCellVolume: 90,
    redCellDistributionWidth: 13.2,
    alkalinePhosphatase: 70,
    whiteBloodCellCount: 6.5,
  },
});

console.log(`생물학적 나이: ${assessment.results.biologicalAge}`);
console.log(`나이 차이: ${assessment.results.ageDelta} 세`);

// 평가 추세 가져오기
const trend = await client.assessments.getTrend('user_123', {
  startDate: '2024-01-01',
  endDate: '2025-12-31',
  interval: 'monthly',
});

trend.dataPoints.forEach(point => {
  console.log(`${point.date}: ${point.biologicalAge} 세`);
});

users = ['user_123', 'user_456', 'user_789']
assessments = client.assessments.batch_create(users, biomarker_data)

5.8 웹훅 통합

이벤트 기반 아키텍처의 경우, WIA-AGING API는 특정 이벤트가 발생할 때 애플리케이션에 실시간 알림을 푸시하는 웹훅을 지원합니다. 이렇게 하면 폴링이 필요 없고 데이터 변경에 즉각 반응할 수 있습니다.

5.9 API 버전 관리 전략

WIA-AGING API는 RESTful API에 적응된 시맨틱 버전 관리 원칙을 따릅니다. 주요 버전 내에서 이전 버전과의 호환성을 유지하고 중대한 변경 사항을 도입할 때 명확한 마이그레이션 경로를 제공합니다.

버전 지원 정책:

  • 각 주요 버전은 릴리스 후 최소 24개월 동안 지원됩니다
  • 더 이상 사용되지 않는 버전은 사전 통지와 함께 12개월의 일몰 기간을 받습니다
  • 보안 패치는 지원되는 모든 버전에 적용됩니다
  • 새로운 기능은 현재 주요 버전에만 추가됩니다

5.10 요약

✅ 핵심 내용:

  1. RESTful 설계: API는 명확한 리소스 계층 구조, 표준 HTTP 메서드, 최적의 확장성을 위한 무상태 아키텍처로 REST 원칙을 따릅니다.
  2. 보안 우선: PKCE를 사용한 OAuth 2.0, JWT 토큰, 세밀한 범위 기반 권한 부여가 민감한 건강 데이터를 보호하면서 유연한 액세스 제어를 가능하게 합니다.
  3. 개발자 경험: 일관된 JSON 형식, 포괄적인 오류 메시지, 여러 언어의 공식 SDK, 상세한 문서화가 통합을 가속화합니다.
  4. 속도 제한: 계층화된 속도 제한은 공정한 사용과 시스템 안정성을 보장하면서 개별 앱부터 대규모 연구 플랫폼까지 다양한 사용 사례를 지원합니다.
  5. 이벤트 기반: 웹훅 지원으로 실시간 이벤트 알림이 가능하여 폴링 오버헤드를 제거하고 반응형 애플리케이션을 가능하게 합니다.
