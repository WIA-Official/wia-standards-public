Chapter 3: API Endpoints

Unified Gateway, Protocol Translation & Version Harmonization

3.1 Unified Gateway Architecture

WIA-OMNI-API provides a single entry point for all API requests, regardless of the underlying protocol. This unified gateway acts as the "mother" that welcomes all clients and routes their requests appropriately.

┌──────────────────────────────────────────────────┐ │ Single Entry Point (Gateway) │ │ https://api.example.com │ └─────────────────────┬────────────────────────────┘ │ ┌─────────────┼─────────────┐ │ │ │ ▼ ▼ ▼ REST Path GraphQL Path gRPC Path /api/v1 /graphql :50051 │ │ │ └─────────────┼─────────────┘ ▼ ┌─────────────────┐ │ Protocol Router │ └─────────────────┘

Single Endpoint Design

// All requests go to the same base URL
const gateway = "https://api.example.com/omni"

// REST-style
GET https://api.example.com/omni/users/123

// GraphQL-style
POST https://api.example.com/omni
{ "query": "{ user(id: 123) { name } }" }

// Intent-style (recommended)
POST https://api.example.com/omni
{ "intent": "get user 123" }

// Mother routes to optimal backend automatically

3.2 Protocol Translation Endpoints

The gateway automatically translates between protocols, allowing clients to use their preferred protocol while backends use theirs.

REST to GraphQL Bridge

// Client sends REST request
GET /api/users/123?fields=name,email,orders

// Mother translates to GraphQL internally
query {
  user(id: 123) {
    name
    email
    orders {
      id
      total
    }
  }
}

// Benefits:
// - No N+1 query problem
// - Precise field selection
// - Single network request
// - Client doesn't need to know GraphQL

GraphQL to REST Bridge

// Client sends GraphQL query
query {
  user(id: 123) {
    name
    orders {
      id
      total
    }
  }
}

// Mother decomposes to REST calls (if backend is REST)
1. GET /users/123 → { name }
2. GET /users/123/orders → [{ id, total }, ...]

// Mother executes in parallel and combines results
// Client receives unified GraphQL response

3.3 Version Harmonization

The Version Harmonizer ensures that API evolution never breaks existing clients. This is WIA-OMNI-API's sacred promise.

Never-Break Policy

Guarantee: Once an API contract is established, it is supported forever. Clients using version 1.0 will continue to work even when the API reaches version 10.0.

Version Detection

// Method 1: Explicit version header
GET /api/users/123
X-WIA-API-Version: 1.0

// Method 2: URL path versioning
GET /api/v1/users/123

// Method 3: Accept header
GET /api/users/123
Accept: application/vnd.api.v1+json

// Method 4: Auto-detection from request pattern
GET /api/users/123
// Mother infers version from request structure

// Method 5: No version specified → use latest compatible
GET /api/users/123
// Mother serves newest version compatible with client

Handling Schema Changes

Field Addition (Non-Breaking)

// Version 1.0 schema
{
  "id": 123,
  "name": "Kim"
}

// Version 2.0 schema (added fields)
{
  "id": 123,
  "name": "Kim",
  "email": "kim@example.com",  // New
  "phone": "+82-10-1234-5678"  // New
}

// Version 1.0 client receives v1 schema
// Version 2.0 client receives v2 schema
// Automatic and transparent

Field Removal (Handled Gracefully)

// Version 1.0 had "legacy_id" field
// Version 2.0 removed it

// Strategy: Keep serving to old clients
{
  "id": 123,
  "legacy_id": 123,  // Computed or mapped
  "_deprecated": {
    "fields": ["legacy_id"],
    "message": "Use 'id' instead",
    "sunset_date": null  // null = never forced
  }
}

Field Rename (Dual Support)

// Version 1.0: "user_name"
// Version 2.0: "name"

// Mother serves both
{
  "user_name": "Kim",  // Old field (alias)
  "name": "Kim",       // New field
  "_evolution": {
    "aliases": {
      "user_name": "name"
    },
    "preferred": "name"
  }
}

Type Change (Auto Conversion)

// Version 1.0: amount as string
{ "amount": "100.50" }

// Version 2.0: amount as number
{ "amount": 100.50 }

// Mother handles both:
// - Old clients get string
// - New clients get number
// - Conversion is lossless and documented

3.4 Intelligent Routing

The gateway routes requests to optimal backends based on multiple factors, learning and improving over time.

Routing Criteria

  • Client Capability: Browser vs. high-performance service
  • Network Condition: Bandwidth, latency, reliability
  • Request Type: Query vs. mutation vs. subscription
  • Performance Requirement: Speed vs. accuracy vs. cost
  • Data Freshness: Real-time vs. cacheable
  • Load Balancing: Backend availability and load

Smart Caching

// Mother learns caching patterns
// Frequently requested, rarely changed → cache aggressively
GET /api/users/123

Response Headers:
X-WIA-Cache-Strategy: aggressive
X-WIA-Cache-Hit: true
X-WIA-Cache-Age: 45s
X-WIA-Cache-TTL: 300s

// Frequently changing data → cache cautiously
GET /api/stock-prices

Response Headers:
X-WIA-Cache-Strategy: fresh
X-WIA-Cache-Hit: false
X-WIA-Cache-TTL: 5s

3.6 Batch and Bundling

Mother automatically detects opportunities for batching and bundling to optimize performance.

Automatic Batching

// Client sends multiple requests
GET /api/users/1
GET /api/users/2
GET /api/users/3

// Mother batches internally
GET /api/users?ids=1,2,3

// Returns individual responses to client
// Client doesn't know batching happened

Predictive Bundling

// Mother learns: 95% of requests for user data
// also request user's orders

// Client requests:
GET /api/users/123

// Mother proactively includes likely needed data:
{
  "user": { "id": 123, "name": "Kim" },
  "_prefetched": {
    "orders": [...],  // Included automatically
    "why": "Frequently requested together"
  }
}

// Next request is instant (already cached):
GET /api/users/123/orders  // Instant response!
Performance Boost: Automatic batching and predictive bundling can reduce network requests by 50-80% without any client changes.
