Chapter 2: Data Schema

2.1 Request/Response Normalization

One of OMNI-API's core functions is normalizing requests and responses across different protocols. Regardless of whether a client sends a REST call, GraphQL query, or gRPC invocation, the gateway transforms it into a unified internal format.

Unified Request Schema

omni-request.schema.json
{
  "omniapiVersion": "1.0",
  "requestId": "uuid-v4",
  "timestamp": "ISO-8601",
  "client": {
    "id": "client-identifier",
    "protocol": "REST|GraphQL|gRPC|WebSocket|MQTT|Intent",
    "version": "client-version",
    "capabilities": ["streaming", "compression", "..."]
  },
  "intent": {
    "action": "read|create|update|delete|execute",
    "resource": "resource-type",
    "resourceId": "optional-id",
    "parameters": {},
    "naturalLanguage": "optional human-readable intent"
  },
  "authentication": {
    "type": "Bearer|OAuth2|API-Key|JWT|Certificate",
    "credentials": "encrypted-credentials"
  },
  "preferences": {
    "responseFormat": "JSON|XML|Protobuf|MessagePack",
    "compression": true|false,
    "fields": ["selective", "field", "list"],
    "locale": "en-US|ko-KR|..."
  }
}

Unified Response Schema

omni-response.schema.json
{
  "omniapiVersion": "1.0",
  "requestId": "matching-request-id",
  "timestamp": "ISO-8601",
  "status": {
    "code": 200,
    "message": "Success",

    "type": "resource-type",
    "version": "schema-version",

  },
  "metadata": {
    "processingTime": "milliseconds",
    "cacheStatus": "hit|miss|bypass",
    "translationPath": ["REST", "Internal", "GraphQL"],
    "optimizations": ["bundled", "prefetched", "compressed"]
  },
  "links": {
    "self": "canonical-url",
    "related": ["related-resources"],
    "next": "optional-pagination"
  }
}

2.2 Protocol-Specific Transformations

The gateway performs intelligent transformations based on the source and target protocols.

REST to Internal Transformation

REST Element Internal Mapping Example
HTTP Method intent.action GET → "read", POST → "create"
URL Path intent.resource + resourceId /users/123 → resource:"user", id:"123"
Query Params intent.parameters ?filter=active → {filter: "active"}
Headers authentication + preferences Accept: application/json → format:"JSON"
Request Body data.content JSON payload → normalized object

GraphQL to Internal Transformation

GraphQL Query Example
query GetUserWithOrders {
  user(id: "123") {
    name
    email
    orders(limit: 10) {
      id
      total
      items {
        product
        quantity
      }
    }
  }
}
Internal Representation
{
  "intent": {
    "action": "read",
    "resource": "user",
    "resourceId": "123",
    "parameters": {
      "include": ["orders"],
      "orders.limit": 10
    }
  },
  "preferences": {
    "fields": ["name", "email", "orders.id", "orders.total", 
               "orders.items.product", "orders.items.quantity"]
  }
}

gRPC to Internal Transformation

gRPC Proto Definition
service UserService {
  rpc GetUser (UserRequest) returns (UserResponse);
}

message UserRequest {
  string id = 1;
  repeated string fields = 2;
}

message UserResponse {
  string id = 1;
  string name = 2;
  string email = 3;
}
Internal Representation
{
  "client": {
    "protocol": "gRPC",
    "service": "UserService",
    "method": "GetUser"
  },
  "intent": {
    "action": "read",
    "resource": "user",
    "resourceId": "from-UserRequest.id",
    "parameters": {}
  },
  "preferences": {
    "fields": "from-UserRequest.fields",
    "responseFormat": "Protobuf"
  }
}

2.3 Schema Evolution & Versioning

One of OMNI-API's sacred promises is absolute backward compatibility. This requires sophisticated schema versioning.

Schema Version Management

Version Strategy

  • Additive Changes: New fields can be added without version bump
  • Default Values: New required fields must have sensible defaults
  • Deprecation Tags: Old fields marked deprecated but never removed
  • Automatic Migration: Gateway translates between all versions

Multi-Version Support Example

Version 1.0 (2020) - User Schema
{
  "id": "string",
  "name": "string",
  "email": "string"
}
Version 2.0 (2023) - Extended Schema
{
  "id": "string",
  "name": "string",
  "email": "string",
  "avatar": "string",          // New field
  "createdAt": "timestamp",    // New field
  "preferences": {             // New nested object
    "language": "string",
    "timezone": "string"
  }
}
Version 3.0 (2025) - Current Schema
{
  "id": "string",
  "fullName": "string",        // Renamed from "name"
  "name": "string",            // Deprecated but still supported
  "email": "string",
  "avatar": "string",
  "createdAt": "timestamp",
  "updatedAt": "timestamp",    // New field
  "preferences": {
    "language": "string",
    "timezone": "string",
    "theme": "string"          // New nested field
  },
  "metadata": {                // New section
    "lastLogin": "timestamp",
    "loginCount": "integer"
  }
}
The Mother's Promise: A client using v1.0 schema from 2020 will continue to work perfectly in 2025, receiving only the fields it expects. A v3.0 client gets all the latest features.

2.4 Format-Agnostic Design

OMNI-API internally uses a format-agnostic representation that can be serialized to any format.

Supported Formats

Format Use Case Advantages Auto-Conversion
JSON Web APIs, JavaScript Human-readable, universal ✓
XML Legacy systems, SOAP Schema validation, widespread ✓
Protobuf gRPC, high-performance Compact, fast, type-safe ✓
MessagePack Binary efficiency Smaller than JSON, fast ✓
YAML Configuration, human editing Very readable, comments ✓
CBOR IoT, constrained devices Compact, low overhead ✓

Format Transformation Example

Same Data, Different Formats
// JSON
{
  "id": 123,
  "name": "John Doe",
  "active": true
}

// XML
<user>
  <id>123</id>
  <name>John Doe</name>
  <active>true</active>
</user>

// YAML
id: 123
name: John Doe
active: true

// All automatically converted by OMNI-API!

2.5 Nested Resources & Relationships

OMNI-API provides a standardized way to represent resource relationships and nested data.

Relationship Patterns

Pattern Description Example
Embedded Full nested object included user.orders[...full order objects]
Reference ID or URL reference only user.orderIds: ["o1", "o2"]
Hybrid Summary + reference user.orders: [{id, total, _link}]
Deferred Available on request _links.orders: "/users/123/orders"
Comprehensive Resource Example
{
  "id": "123",
  "name": "John Doe",
  "email": "john@example.com",
  
  // Embedded relationship
  "profile": {
    "bio": "Software engineer",
    "location": "Seoul, Korea"
  },
  
  // Reference relationship
  "companyId": "company-456",
  
  // Hybrid relationship (summary + reference)
  "recentOrders": [
    {
      "id": "order-789",
      "total": 99.99,
      "status": "shipped",
      "_link": "/orders/order-789"
    }
  ],
  
  // Deferred relationships
  "_links": {
    "self": "/users/123",
    "company": "/companies/company-456",
    "allOrders": "/users/123/orders",
    "payments": "/users/123/payments"
  }
}

2.6 Validation & Type Safety

OMNI-API enforces type safety and validation across all protocols, even those that don't natively support it.

Validation Rules

Type System

  • Primitive Types: string, number, integer, boolean, null
  • Complex Types: object, array, enum, union
  • Custom Types: email, url, uuid, timestamp, currency
  • Optional/Required: Explicit marking with defaults
Validation Schema Example
{
  "type": "object",
  "properties": {
    "email": {
      "type": "string",
      "format": "email",
      "required": true
    },
    "age": {
      "type": "integer",
      "minimum": 0,
      "maximum": 150
    },
    "role": {
      "type": "string",
      "enum": ["admin", "user", "guest"],
      "default": "user"
    },
    "preferences": {
      "type": "object",
      "properties": {
        "notifications": {
          "type": "boolean",
          "default": true
        }
      }
    }
  }
}
Validation Errors: When validation fails, OMNI-API returns clear, actionable error messages in the client's preferred format and language.
