제4장: 프로토콜 사양

프로토콜 수용자 & 의도 해석기

4.1 프로토콜 수용자

프로토콜 수용자는 WIA-OMNI-API의 "어머니" 철학의 핵심입니다. 현재와 미래의 모든 API 프로토콜을 받아들이고 이해하며, 각 클라이언트의 선호하는 통신 스타일에 적응합니다.

지원 프로토콜

REST 프로토콜 지원

rest_protocol:
  methods: [GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS]
  versions: [HTTP/1.0, HTTP/1.1, HTTP/2, HTTP/3]

  features:
    - resource_oriented: true
    - idempotent_operations: 강제됨
    - stateless: 선호됨
    - cacheable: 자동
    - hypermedia: HATEOAS 링크

GraphQL 프로토콜 지원

graphql_protocol:
  operations: [query, mutation, subscription]
  versions: [June2018, October2021, Draft]

  features:
    - introspection: 전체 스키마 검색
    - batching: 자동 쿼리 배치
    - deduplication: 동일 쿼리 병합
    - persisted_queries: APQ 지원

gRPC 프로토콜 지원

grpc_protocol:
  modes: [unary, server_stream, client_stream, bidi_stream]
  encoding: protobuf (Protocol Buffers)
  transport: HTTP/2

  features:
    - type_safety: 컴파일 시간 검사
    - streaming: 양방향
    - deadline: 타임아웃 지원
    - metadata: 사용자 정의 헤더

4.2 의도 해석기

의도 해석기는 클라이언트가 문자 그대로 요청하는 것이 아니라 달성하고자 하는 것을 이해합니다. 여러 수준의 의도 표현을 지원합니다.

의도 수준

수준 1: 프로토콜별 (전통적)

// REST
GET /api/users/123

// GraphQL
query { user(id: 123) { name email } }

// gRPC
UserService.GetUser({ user_id: 123 })

수준 2: 자연어

{
  "intent": "사용자 123의 프로필 정보 가져와"
}

// 어머니가 해석하고 최적 쿼리 실행

수준 3: WIA-INTENT (구조화된 의도)

{
  "intent": {
    "type": "wia-intent",
    "code": `
      intent GetUserProfile {
        given: user_id = 123
        want: user_profile
        constraints {
          include: [name, email, avatar]
          exclude: [password, internal_id]
          freshness: <= 5.minutes
        }
        certainty: >= 0.95
      }
    `
  }
}

4.3 프로토콜 자동 선택

클라이언트가 프로토콜 선호도를 지정하지 않으면, 어머니가 여러 요인을 기반으로 최적의 프로토콜을 선택합니다.

선택 기준

시나리오 최적 프로토콜 이유
브라우저, 간단한 쿼리 REST 광범위한 지원, 캐시 가능
복잡한 중첩 데이터 GraphQL 정확한 필드 선택
고성능 서비스 gRPC 바이너리 인코딩, 타입 안전
실시간 업데이트 WebSocket 양방향, 낮은 지연시간
IoT 장치, 제한된 대역폭 MQTT 경량, QoS 지원

4.4 프로토콜 상호 운용성

서로 다른 프로토콜이 어머니의 변환 계층을 통해 원활하게 상호 운용할 수 있습니다.

4.5 프로토콜 간 에러 처리

어머니는 모든 프로토콜에서 에러 응답을 정규화하여, 동정적이고 실행 가능한 에러 메시지를 제공합니다.

통합 에러 형식

{
  "success": false,
  "error": {
    "code": "USER_NOT_FOUND",
    "message": "ID 999 사용자가 존재하지 않습니다",
    "comfort": "걱정 마세요, 이런 일은 자주 일어나요.",
    "cause": "요청하신 사용자 ID가 시스템에 없습니다",
    "solution": "사용자 ID를 확인하거나 이름으로 검색해보세요",
    "recovery_options": [
      {
        "action": "search_similar",
        "description": "비슷한 이름의 사용자 검색",
        "endpoint": "/users/search",
        "auto_applicable": false,
        "confidence": 0.7
      }
    ]
  }
}
미래 보장: 새로운 프로토콜이 등장해도 API 통합이 구식이 되지 않습니다. 어머니가 적응하므로 여러분은 그럴 필요가 없습니다.
