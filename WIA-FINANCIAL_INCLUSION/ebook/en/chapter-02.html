CHAPTER 2

Core Data Structures for Digital Banking

弘益人間 (Benefit All Humanity)

Introduction to Financial Inclusion Data Architecture

Building inclusive digital banking systems requires carefully designed data structures that can serve billions of users, many of whom are accessing formal financial services for the first time. These data structures must be simple enough to work on basic mobile devices, secure enough to protect vulnerable users, flexible enough to accommodate diverse financial behaviors, and standardized enough to enable interoperability across different providers and platforms.

The WIA-FINANCIAL_INCLUSION standard defines a comprehensive set of data structures optimized for financial inclusion use cases. Unlike traditional banking systems designed for affluent customers with complex financial portfolios, these structures prioritize accessibility, simplicity, and efficiency while maintaining the security and reliability essential for financial services.

Account Types and Use Cases

The WIA-FINANCIAL_INCLUSION standard supports three primary account types, each designed for different user needs and regulatory contexts:

  • Basic Account: Entry-level account for unbanked users with minimal KYC requirements (Tier 1). Limited transaction amounts but immediate activation. Ideal for users opening their first formal financial account.
  • Savings Account: Standard account with full KYC (Tier 2). Higher transaction limits, interest accrual capability, and access to credit products. Suitable for regular users building financial history.
  • Business Account: Enhanced account for small businesses and merchants with comprehensive KYC (Tier 3). Highest transaction limits, merchant services integration, and detailed transaction reporting.

Alternative Data for Credit Scoring

Traditional credit scoring relies on formal credit history, which most unbanked individuals lack. The WIA-FINANCIAL_INCLUSION standard incorporates alternative data sources that can predict creditworthiness:

  • Mobile Money History: Frequency, volume, and regularity of mobile money transactions
  • Bill Payment Patterns: Consistency in paying utilities, rent, and other bills
  • Savings Behavior: Regular deposits and account balance trends
  • Social Networks: Size and quality of transaction networks and peer references
  • Digital Footprint: Length of digital financial service usage and diversity of services used

KYC Verification Data Structure

Data Security and Privacy Considerations

Financial inclusion serves vulnerable populations who may face additional risks from data breaches or privacy violations. The WIA standard mandates comprehensive security measures:

  • Encryption at Rest: All sensitive data (account numbers, PINs, biometric templates, national IDs) must be encrypted using AES-256 or stronger
  • Encryption in Transit: All network communications must use TLS 1.3 or newer
  • Data Minimization: Collect only data necessary for the service level being provided
  • Anonymization: Analytics and reporting must use anonymized data wherever possible
  • Access Controls: Role-based access control (RBAC) limiting employee access to user data
  • Audit Trails: Complete logging of all access to sensitive user data
  • User Consent: Explicit consent required for data collection and usage
  • Right to Deletion: Users can request deletion of personal data (subject to regulatory retention requirements)

Data Interoperability Standards

For financial inclusion to achieve maximum impact, different systems must be able to exchange data seamlessly. The WIA standard specifies data interchange formats based on widely adopted standards:

Interoperability Requirements

  • JSON Format: Primary data interchange format for APIs (ISO 21778)
  • ISO 20022: Support for standard financial messaging where applicable
  • E.164: Phone number format for international compatibility
  • ISO 4217: Currency codes for multi-currency support
  • ISO 3166: Country codes for international operations
  • ISO 8601: Date and time format to avoid ambiguity
  • UUID v4: Globally unique identifiers without centralized coordination

Key Takeaways

  • Account structures must balance simplicity for first-time users with functionality for growing financial needs through basic, savings, and business account types
  • Transaction data structures support diverse use cases from P2P transfers to cross-border remittances with comprehensive metadata for compliance and analytics
  • User profiles incorporate alternative data sources (mobile money history, bill payment patterns, social networks) essential for credit scoring without traditional credit history
  • Tiered KYC systems allow users to start with minimal documentation (Tier 1) and upgrade progressively as they provide more verification (Tiers 2-3)
  • Transaction limits correspond to KYC tiers, balancing financial inclusion access with regulatory compliance and fraud prevention
  • Data security measures including encryption, access controls, and audit trails protect vulnerable users from privacy violations and fraud
  • Standardized data formats using ISO standards (20022, 8601, 4217, 3166) enable interoperability across different financial service providers
  • Alternative credit scoring data captured in user profiles enables lending to unbanked populations without traditional credit bureau data
