# 제2장: 언어 데이터의 현재 과제

## 개요

> **弘益人間 (홍익인간)** - 널리 인간을 이롭게 하라

언어 데이터 분야는 급속한 발전을 이루었지만, 여전히 많은 도전 과제에 직면해 있습니다. 이 장에서는 언어 자원 구축, 관리, 활용 과정에서 마주치는 주요 문제들을 분석하고, WIA-LANG-010 표준이 이러한 과제를 어떻게 해결하고자 하는지 살펴봅니다.

## 1. 데이터 파편화 (Data Fragmentation)

### 문제 상황

현재 언어 데이터는 다양한 형식과 구조로 분산되어 있습니다:

| 문제 요소 | 현황 | 영향 |
|---------|------|-----|
| 포맷 불일치 | 100개 이상의 서로 다른 코퍼스 포맷 | 통합 활용 불가 |
| 주석 체계 차이 | 언어/프로젝트별 독자적 태그셋 | 비교 연구 어려움 |
| 메타데이터 부재 | 체계적 문서화 미흡 | 재사용성 저하 |
| 인코딩 문제 | 다양한 문자 인코딩 혼재 | 데이터 손상 위험 |
| 저장 위치 분산 | 개별 웹사이트, FTP, 개인 컴퓨터 | 접근성 제한 |

### 실제 사례

## 2. 리소스 불균형 (Resource Imbalance)

### 언어별 자원 격차

전 세계 언어의 디지털 자원은 극심한 불균형을 보입니다:

| 언어 등급 | 언어 수 | 사용자 수 | 자원 수준 | 예시 |
|---------|--------|---------|---------|-----|
| 초고자원 | ~10 | 20억+ | 10TB+ | 영어, 중국어 |
| 고자원 | ~50 | 1억~20억 | 1~10TB | 한국어, 일본어, 독일어 |
| 중자원 | ~200 | 100만~1억 | 100MB~1TB | 베트남어, 스와힐리어 |
| 저자원 | ~1,000 | 1만~100만 | 1~100MB | 체로키어, 나바호어 |
| 극저자원 | ~5,700 | 1만 미만 | <1MB | 대부분의 소멸위기 언어 |

## 3. 주석 일관성 문제 (Annotation Inconsistency)

### 주석 오류의 유형

| 오류 유형 | 설명 | 빈도 | 영향도 |
|---------|------|-----|-------|
| 주관적 판단 차이 | 주석자 간 해석 차이 | 높음 | 중간 |
| 가이드라인 모호성 | 불명확한 주석 기준 | 중간 | 높음 |
| 피로도 오류 | 장시간 작업으로 인한 실수 | 높음 | 낮음 |
| 경계 사례 | 명확하지 않은 언어 현상 | 중간 | 높음 |
| 도구 오류 | 주석 도구의 버그 | 낮음 | 중간 |

### 실제 주석 불일치 사례

```python
# 한국어 개체명 인식의 주석 불일치 예시

sentence = "삼성전자가 새로운 갤럭시를 출시했다."

# 주석자 A의 판단
annotation_A = [
    {"text": "삼성전자", "type": "ORG", "span": (0, 4)},
    {"text": "갤럭시", "type": "PRODUCT", "span": (11, 14)}
]

# 주석자 B의 판단
annotation_B = [
    {"text": "삼성전자", "type": "COMPANY", "span": (0, 4)},
    # "갤럭시"를 일반 명사로 판단하여 태그 안 함
]

# 주석자 C의 판단
annotation_C = [
    {"text": "삼성", "type": "ORG", "span": (0, 2)},  # "삼성"만 태그
    {"text": "전자", "type": "PRODUCT_TYPE", "span": (2, 4)},
    {"text": "갤럭시", "type": "BRAND", "span": (11, 14)}
]

# 불일치 분석
from wia_lang_010 import AnnotationAgreement

agreement = AnnotationAgreement.calculate([
    annotation_A,
    annotation_B,
    annotation_C
])

print(f"Kappa Score: {agreement.kappa}")  # 0.52 (moderate agreement)
print(f"F1 Score: {agreement.f1}")  # 0.67
print(f"Exact Match: {agreement.exact_match}")  # 0%
```

### 일관성 보장 방법

**1. 명확한 주석 가이드라인**

```markdown
# WIA-LANG-010 개체명 주석 가이드라인

## 조직명 (ORG) 주석 규칙

### 포함 대상
- 기업명: "삼성전자", "현대자동차"
- 정부 기관: "교육부", "환경청"
- 비영리 단체: "유네스코", "그린피스"

### 경계 설정
1. **법인 유형 표시 포함**
   - ✅ 올바름: "삼성전자주식회사" 전체를 ORG로 태그
   - ❌ 틀림: "삼성전자"만 태그하고 "주식회사" 제외

2. **복합 조직명**
   - ✅ 올바름: "한국과학기술원 전산학부" 전체를 ORG로 태그
   - ❌ 틀림: "한국과학기술원"과 "전산학부"를 분리하여 태그

3. **약어와 정식 명칭**
   - 둘 다 ORG로 태그: "KAIST"와 "한국과학기술원"
   - 동일 개체로 연결: coreference 관계 표시

### 애매한 사례
| 문장 | 판단 | 이유 |
|------|-----|------|
| "삼성의 신제품" | "삼성"을 ORG로 태그 | 조직을 지칭 |
| "갤럭시는 인기가 많다" | PRODUCT로 태그 | 제품명 |
| "아이폰과 갤럭시 중 무엇을 선택할까?" | 둘 다 PRODUCT | 제품 비교 문맥 |
```

**2. 주석자 간 일치도 측정**

```python
from wia_lang_010 import InterAnnotatorAgreement

# 여러 주석자의 작업 비교
iaa = InterAnnotatorAgreement()

# Cohen's Kappa (2명의 주석자)
kappa = iaa.cohens_kappa(annotator1, annotator2)
print(f"Cohen's Kappa: {kappa}")
# > 0.80: Excellent
# 0.60-0.80: Good
# < 0.60: Needs improvement

# Fleiss' Kappa (3명 이상의 주석자)
fleiss_kappa = iaa.fleiss_kappa([annotator1, annotator2, annotator3])

# Krippendorff's Alpha (다양한 주석 유형 지원)
alpha = iaa.krippendorffs_alpha(
    annotations=[ann1, ann2, ann3],
    metric="nominal"  # nominal, ordinal, interval, ratio
)

# 불일치 사례 분석
disagreements = iaa.find_disagreements()
for case in disagreements:
    print(f"Sentence: {case.sentence}")
    print(f"Annotator A: {case.annotation_a}")
    print(f"Annotator B: {case.annotation_b}")
    print(f"Difference: {case.difference_type}")
```

**3. 자동 일관성 검증**

```python
from wia_lang_010 import ConsistencyChecker

checker = ConsistencyChecker()

# 규칙 기반 검증
violations = checker.check_rules(corpus, rules={
    "pos_sequence": {
        # 불가능한 품사 연쇄 검출
        "invalid_sequences": [
            ("DET", "VERB"),  # 관형사 바로 뒤에 동사 불가
            ("AUX", "AUX")    # 조동사 연속 불가
        ]
    },
    "dependency": {
        # 의존 구조 제약
        "constraints": [
            {"head": "NOUN", "dependent": "DET", "relation": "det"},
            {"max_sentence_roots": 1}  # 문장 당 root는 하나만
        ]
    },
    "entity_boundary": {
        # 개체명 경계 제약
        "no_crossing": True,  # 개체명 경계 교차 금지
        "complete_phrases": True  # 완전한 구 단위로 태그
    }
})

# 통계적 이상치 탐지
outliers = checker.detect_outliers(corpus, features=[
    "token_pos_ratio",  # 토큰 당 품사 분포
    "sentence_length",   # 문장 길이
    "entity_density"     # 개체명 밀도
])

# 보고서 생성
report = checker.generate_report()
```

## 4. 메타데이터 부족 (Metadata Deficiency)

### 메타데이터의 중요성

적절한 메타데이터 없이는 언어 자원을 효과적으로 찾고, 평가하고, 재사용할 수 없습니다.

**문제 사례:**

```xml
<!-- 메타데이터 부족 사례 -->
<corpus>
  <!-- 언어가 무엇인지 불명확 -->
  <!-- 출처가 어디인지 알 수 없음 -->
  <!-- 라이선스 정보 없음 -->
  <!-- 버전 정보 없음 -->
  <sentence>This is a sample sentence.</sentence>
  <sentence>Another sentence here.</sentence>
</corpus>

<!-- 이런 질문에 답할 수 없음: -->
<!-- - 이 데이터를 상업적으로 사용할 수 있나? -->
<!-- - 어떤 도메인의 텍스트인가? -->
<!-- - 언제 수집되었나? -->
<!-- - 주석 품질은 어떤가? -->
```
