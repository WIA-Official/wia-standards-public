6.2 Common Pitfalls

❌ Never Do These

  • Reusing IVs/Nonces: Completely breaks security
  • Ignoring Auth Tags: Allows tampering attacks
  • Hardcoding Keys: Keys exposed in source control
  • Using ECB Mode: Reveals patterns in data
  • Custom Crypto: Use proven libraries
Pitfall Risk Solution
IV Reuse Complete compromise Random IV per encryption
No Auth Tag Data tampering Always verify tag
Hardcoded Keys Key exposure Use env vars or KMS
Timing Attacks Key recovery Constant-time compare

6.3 Testing Strategies

// Test: Encryption and Decryption
function testEncryption() {
  const plaintext = Buffer.from('Test message');
  const key = crypto.randomBytes(32);

  const encrypted = encryptAES(plaintext, key);
  const decrypted = decryptAES(encrypted, key);

  assert(plaintext.equals(decrypted));
  console.log('✓ Encryption test passed');
}

// Test: IV Uniqueness
function testIVUniqueness() {
  const plaintext = Buffer.from('Same message');
  const key = crypto.randomBytes(32);

  const enc1 = encryptAES(plaintext, key);
  const enc2 = encryptAES(plaintext, key);

  // Ciphertexts should be different
  assert(!enc1.ciphertext.equals(enc2.ciphertext));
  console.log('✓ IV uniqueness test passed');
}

// Test: Tamper Detection
function testTamperDetection() {
  const encrypted = encryptAES('message', key);
  encrypted.ciphertext[0] ^= 0xFF; // Tamper

  assert.throws(() => decryptAES(encrypted, key));
  console.log('✓ Tamper detection test passed');
}

6.4 Performance Optimization

6.4.1 Hardware Acceleration

  • AES-NI: Intel/AMD instruction set (10x+ speedup)
  • ARM Crypto Extensions: Mobile/embedded acceleration
  • GPU Acceleration: Parallel encryption for large datasets

6.4.2 Best Practices

  • Use streaming encryption for large files
  • Enable hardware acceleration when available
  • Buffer appropriately (16KB-64KB chunks)
  • Consider parallel encryption for independent data

6.5 Secure Coding Practices

  • Clear sensitive memory: Zero out keys after use
  • Constant-time operations: Prevent timing attacks
  • Input validation: Verify all inputs before processing
  • Error handling: Don't leak information in errors
  • Code review: Security-focused review for all crypto code

6.6 Chapter Summary

Key Takeaways

  • Design systems with end-to-end encryption and PFS
  • Never reuse IVs or ignore authentication tags
  • Test encryption, IV uniqueness, and tamper detection
  • Use hardware acceleration when available
  • Follow secure coding practices consistently
