4.1 양쪽의 장점 결합

하이브리드 암호화는 대칭 암호화의 효율성과 비대칭 암호화의 편리함을 결합합니다:

  1. 랜덤 대칭 키 생성 (AES-256)
  2. 대칭 키로 데이터 암호화 (빠름)
  3. 수신자의 공개키로 대칭 키 암호화
  4. 암호화된 데이터 + 암호화된 키 전송

실제 사례: HTTPS

HTTPS 웹사이트를 방문할 때 브라우저는 하이브리드 암호화를 사용합니다:

  • RSA/ECC로 초기 키 교환
  • AES로 실제 웹 트래픽 암호화
  • 이것이 TLS(Transport Layer Security)의 작동 방식

4.2 구현

function hybridEncrypt(data, recipientPublicKey) {
  // 1. 랜덤 AES 키 생성
  const aesKey = crypto.randomBytes(32);

  // 2. AES-GCM으로 데이터 암호화
  const { ciphertext, iv, authTag } = encryptAES(data, aesKey);

  // 3. RSA-OAEP로 AES 키 암호화
  const encryptedKey = crypto.publicEncrypt(
    {
      key: recipientPublicKey,
      padding: crypto.constants.RSA_PKCS1_OAEP_PADDING
    },
    aesKey
  );

  return { ciphertext, encryptedKey, iv, authTag };
}

function hybridDecrypt(encrypted, privateKey) {
  // 1. 개인키로 AES 키 복호화
  const aesKey = crypto.privateDecrypt(
    {
      key: privateKey,
      padding: crypto.constants.RSA_PKCS1_OAEP_PADDING
    },
    encrypted.encryptedKey
  );

  // 2. AES 키로 데이터 복호화
  return decryptAES(encrypted.ciphertext, aesKey,
                    encrypted.iv, encrypted.authTag);
}

4.3 성능 이점

1GB 데이터 암호화 시 비교:

방법 소요 시간 실용성
RSA-4096만 ~92시간 ❌ 비실용적
AES-256만 0.3초 ⚠️ 키 분배 문제
하이브리드 ~0.3초 ✅ 최적의 접근

4.4 TLS 핸드셰이크

TLS 핸드셰이크는 하이브리드 암호화의 완벽한 예시입니다:

// TLS 1.3 간소화된 핸드셰이크
1. 클라이언트 Hello
   - 지원 암호 스위트
   - 클라이언트 랜덤
   - 키 공유 (ECDHE)

2. 서버 Hello
   - 선택된 암호 스위트
   - 서버 랜덤
   - 키 공유 (ECDHE)
   - 인증서

3. 키 도출
   - ECDHE 공유 비밀
   - 대칭 키 도출

4. 암호화된 통신
   - 모든 트래픽에 AES-256-GCM

4.5 완전 순방향 비밀성 (PFS)

현대 하이브리드 시스템은 임시 키를 사용하여 완전 순방향 비밀성(PFS)을 달성합니다:

  • 각 세션마다 새로운 키 쌍 사용 (ECDHE)
  • 장기 키가 유출되어도 과거 세션은 안전
  • TLS 1.3은 모든 연결에 PFS 필수

PFS의 중요성

공격자가 오늘 암호화된 트래픽을 저장해두고, 나중에 개인키를 획득하더라도 과거 통신은 복호화할 수 없습니다.

4.6 ECIES (통합 암호화 체계)

구성요소 알고리즘 목적
키 합의 ECDH 공유 비밀 생성
키 도출 HKDF 암호화 키 도출
암호화 AES-GCM 데이터 암호화
인증 GCM/HMAC 무결성 보장

4.7 장 요약

핵심 내용

  • 하이브리드 암호화 = 대칭 속도 + 비대칭 편리함
  • TLS는 웹 트래픽에 하이브리드 암호화 사용
  • AES로 데이터 암호화, RSA/ECC로 AES 키 암호화
  • 성능은 순수 대칭과 거의 동일
  • 완전 순방향 비밀성이 과거 통신 보호
