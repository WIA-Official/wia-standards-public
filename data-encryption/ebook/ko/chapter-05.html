5.1 가장 취약한 연결고리

보안 원칙

암호화된 데이터의 보안은 키의 보안만큼만 강력합니다. 대부분의 암호화 침해는 알고리즘 약점이 아닌 키 관리 실패에서 발생합니다.

5.2 키 수명주기

단계 활동 모범 사례
생성 키 생성 CSRNG 사용, 충분한 엔트로피
배포 안전한 키 공유 비대칭 암호화, Diffie-Hellman
저장 저장 중 키 보호 HSM, KMS, 암호화된 키 저장소
사용 암호화에 키 사용 접근 제어, 로깅
순환 주기적 키 교체 대칭 90일, 비대칭 365일
파기 안전한 키 삭제 암호학적 삭제

5.3 키 도출 함수

KDF는 비밀번호나 다른 소스에서 암호화 키를 도출합니다:

5.3.1 PBKDF2

function deriveKeyFromPassword(password, salt) {
  return crypto.pbkdf2Sync(
    password,
    salt,
    100000,  // 반복 횟수
    32,      // 키 길이
    'sha256'
  );
}

5.3.2 Argon2 (권장)

Password Hashing Competition 우승자인 Argon2가 더 안전합니다:

  • 메모리 하드: GPU/ASIC 공격에 저항
  • 구성 가능: 시간 및 메모리 비용 조절 가능
  • 세 가지 변형: Argon2d, Argon2i, Argon2id (권장)

5.4 키 저장소 솔루션

솔루션 보안 수준 사용 사례
HSM (하드웨어 보안 모듈) 최고 금융, 정부
클라우드 KMS 높음 클라우드 애플리케이션
시크릿 매니저 중상 애플리케이션 시크릿
암호화된 파일 중간 개발, 테스트

5.5 엔벨로프 암호화

데이터를 데이터 암호화 키(DEK)로 암호화한 다음, DEK를 키 암호화 키(KEK)로 암호화합니다:

// 엔벨로프 암호화 패턴
{
  "encrypted_data": "AES 암호화된 콘텐츠...",
  "encrypted_dek": "RSA 암호화된 DEK...",
  "kek_id": "master-key-v2",
  "algorithm": "AES-256-GCM"
}

이점:

  • 데이터 재암호화 없이 키 순환 가능
  • KEK는 HSM 또는 KMS에 저장
  • DEK는 암호화된 데이터와 함께 저장
  • AWS, Azure, Google Cloud에서 사용

5.6 키 순환 정책

키 유형 권장 주기 고려 사항
대칭 (데이터) 90일 자동화 권장
비대칭 (서명) 1-2년 인증서 수명과 연동
마스터 키 (KEK) 1년 HSM에서 관리
세션 키 세션마다 PFS 구현

5.7 장 요약

핵심 내용

  • 키 관리는 대부분 시스템의 가장 취약한 연결고리
  • 키 수명주기: 생성, 배포, 저장, 사용, 순환, 파기
  • 비밀번호 기반 키 도출에는 Argon2id 사용
  • 프로덕션에서는 HSM 또는 클라우드 KMS에 키 저장
  • 엔벨로프 암호화로 효율적인 키 순환 가능
