Chapter 2: Audit Architecture

Components and Design Patterns

WIA-SEC-017 Security Audit Standard

2.1 Architecture Overview

The WIA-SEC-017 audit architecture consists of several interconnected components that work together to collect, process, store, and analyze security events. Understanding this architecture is essential for effective implementation.

2.2 Audit Sources

Audit events originate from multiple sources across the IT infrastructure:

Source Type Event Examples Collection Method
Applications User logins, data access, transactions SDK integration
Databases Query execution, schema changes, admin operations Native auditing + connector
Operating Systems Process execution, file access, privilege changes Agent-based
Network Devices Firewall rules, routing changes, traffic logs Syslog forwarding
Cloud Platforms IAM changes, resource provisioning, API calls Cloud trail integration
Security Tools Malware detection, intrusion attempts, scans API integration

2.3 Collection Layer

SDK Integration

For applications, the WIA-SEC-017 SDK provides the easiest integration path:

// JavaScript/TypeScript SDK
import { AuditClient } from '@wia/sec-017-audit';

const audit = new AuditClient({
  endpoint: 'https://audit.company.com',
  apiKey: process.env.AUDIT_API_KEY,
  bufferSize: 100,      // Batch events for efficiency
  flushInterval: 5000   // Flush every 5 seconds
});

// Log authentication event
await audit.logEvent({
  type: 'AUTHENTICATION',
  action: 'LOGIN',
  actor: { id: 'user@example.com', type: 'USER' },
  outcome: 'SUCCESS',
  metadata: { mfa: true, ip: '192.168.1.100' }
});

Agent-Based Collection

For systems that cannot integrate an SDK, lightweight agents collect events locally and forward them to the central audit system. Agents provide:

  • File-based log collection and parsing
  • System-level event monitoring (auditd integration)
  • Local buffering for network reliability
  • Compression and encryption in transit

API Endpoints

The audit system exposes REST and gRPC APIs for direct event submission from external systems and custom integrations.

2.4 Processing Layer

Event Validation

Every incoming event is validated against the WIA-SEC-017 schema:

  • Required fields present (timestamp, actor, action, resource)
  • Data types correct (ISO 8601 timestamps, valid URNs)
  • Enumerated values within allowed sets
  • Size limits respected (maximum payload size)

Enrichment

Events are automatically enriched with additional context:

  • Geo-location: IP addresses mapped to locations
  • Identity resolution: User IDs linked to full profiles
  • Asset mapping: Resource IDs linked to asset inventory
  • Threat intelligence: IPs/domains checked against threat feeds

Chain Verification

Tamper-Proof Chain Structure

Each event includes a hash of the previous event, creating a chain:

Event N:
{
  "id": "evt-12345",
  "timestamp": "2025-01-07T10:30:00Z",
  "data": { ... },
  "previousHash": "a3f2b1...",   // Hash of Event N-1
  "hash": "d4e5f6..."           // SHA-256(data + previousHash)
}

Any modification to a historical event invalidates all subsequent hashes, making tampering immediately detectable.

2.5 Storage Layer

Tiered Storage Strategy

Tier Duration Use Case Technology
Hot 0-30 days Real-time queries, active investigation Elasticsearch, ClickHouse
Warm 30-90 days Recent historical analysis Object storage with indexing
Cold 90 days - 1 year Compliance queries, audits Compressed object storage
Archive 1-7 years Long-term retention, legal hold WORM storage, tape

Write-Once Storage

For maximum integrity, archive storage should use WORM (Write Once Read Many) technology that physically prevents modification or deletion of stored data.

2.6 Analysis Layer

Real-Time Processing

  • Rule Engine: Evaluate events against detection rules
  • Correlation: Connect related events across systems
  • Anomaly Detection: ML-based behavioral analysis
  • Alerting: Notify security teams of critical events

Reporting and Visualization

  • Dashboards: Real-time security posture visualization
  • Compliance Reports: Pre-built reports for SOC2, ISO27001, etc.
  • Ad-hoc Queries: Full-text search and SQL-like querying
  • Forensic Tools: Timeline reconstruction for investigations

2.7 Chapter Summary

Key Takeaways

  • WIA-SEC-017 architecture has four layers: Collection, Processing, Storage, Analysis
  • Multiple collection methods: SDK, agents, API, syslog integration
  • Processing includes validation, enrichment, and chain verification
  • Tiered storage balances performance, cost, and compliance
  • Analysis layer provides real-time detection and reporting
  • Chain structure ensures tamper-proof audit trails
