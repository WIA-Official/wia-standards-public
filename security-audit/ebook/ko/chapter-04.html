제4장: 1단계 - 데이터 형식

표준화된 감사 이벤트 스키마

4.1 감사 이벤트 스키마

WIA-SEC-017은 모든 감사 이벤트에 대해 표준화된 JSON 스키마를 정의합니다. 이 스키마는 6가지 핵심 질문에 답합니다: 누가, 무엇을, 언제, 어디서, 어떻게, 결과는?

{
  "eventId": "evt_01HQMXK9J2N4P5R6S7T8V9W0",
  "timestamp": "2025-01-15T09:30:45.123Z",
  "version": "1.0",
  "actor": {
    "type": "user",
    "id": "user_123",
    "email": "admin@example.com",
    "ip": "192.168.1.100",
    "userAgent": "Mozilla/5.0..."
  },
  "action": {
    "type": "DATA_ACCESS",
    "operation": "READ",
    "resource": "/api/customers/456"
  },
  "context": {
    "service": "customer-api",
    "environment": "production",
    "requestId": "req_abc123"
  },
  "outcome": {
    "status": "SUCCESS",
    "statusCode": 200
  },
  "metadata": {
    "classification": "CONFIDENTIAL",
    "compliance": ["SOC2", "GDPR"]
  }
}

4.2 필수 필드

필드유형설명예시
eventIdstring전역 고유 식별자 (ULID)evt_01HQ...
timestampstringISO 8601 형식 (UTC)2025-01-15T09:30:45.123Z
versionstring스키마 버전1.0
actorobject작업 수행자 정보사용자, 시스템, API 키
actionobject수행된 작업타입, 작업, 리소스
outcomeobject작업 결과성공/실패, 상태 코드

4.3 Actor 객체

Actor는 작업을 수행한 주체를 식별합니다:

Actor 유형

유형설명필수 필드
user인간 사용자id, email
service서비스 계정id, serviceName
api_keyAPI 키id, keyId
system시스템 프로세스id, processName
 
// 사용자 Actor
"actor": {
  "type": "user",
  "id": "user_123",
  "email": "user@example.com",
  "ip": "192.168.1.100",
  "userAgent": "Chrome/120.0",
  "sessionId": "sess_abc",
  "mfaVerified": true
}

// 서비스 Actor
"actor": {
  "type": "service",
  "id": "svc_payment",
  "serviceName": "payment-service",
  "version": "2.1.0",
  "environment": "production"
}

4.4 Action 객체

Action은 수행된 작업을 설명합니다:

Action 유형

유형설명예시 작업
AUTHENTICATION인증 이벤트LOGIN, LOGOUT, MFA_VERIFY
AUTHORIZATION권한 결정GRANT, DENY, ESCALATE
DATA_ACCESS데이터 액세스READ, WRITE, DELETE
CONFIGURATION구성 변경CREATE, UPDATE, DELETE
SECURITY보안 이벤트BLOCK, ALERT, QUARANTINE
 
"action": {
  "type": "DATA_ACCESS",
  "operation": "READ",
  "resource": "/api/customers/456",
  "resourceType": "customer",
  "resourceId": "456",
  "query": "fields=name,email",
  "recordCount": 1
}

4.5 체인 메타데이터

변조 방지를 위한 체인 관련 필드:

{
  "chain": {
    "sequence": 12345,
    "previousHash": "sha256:abc123...",
    "hash": "sha256:def456...",
    "signature": "sig:xyz789...",
    "signedAt": "2025-01-15T09:30:45.500Z",
    "signedBy": "audit-server-1"
  }
}

체인 필드 설명

  • sequence: 연속적인 순서 번호
  • previousHash: 이전 항목의 해시
  • hash: 현재 항목의 해시 (이벤트 + previousHash)
  • signature: 해시의 디지털 서명
  • signedAt: 서명 시간
  • signedBy: 서명한 서버 식별자

4.6 데이터 분류

감사 이벤트의 민감도에 따른 분류:

분류설명보존 기간액세스 제한
PUBLIC공개 정보1년제한 없음
INTERNAL내부 정보3년직원만
CONFIDENTIAL기밀 정보5년권한 필요
RESTRICTED극비 정보7년특별 승인

PII 처리

개인 식별 정보(PII)가 포함된 이벤트:

  • 자동 마스킹 또는 해싱 적용
  • GDPR/HIPAA 규정 준수 태그 추가
  • 보존 기간 후 자동 익명화

4.7 스키마 검증

모든 이벤트는 제출 전에 스키마 검증을 거칩니다:

// JSON Schema 검증 예시
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "required": ["eventId", "timestamp", "version", "actor", "action", "outcome"],
  "properties": {
    "eventId": {
      "type": "string",
      "pattern": "^evt_[A-Z0-9]{26}$"
    },
    "timestamp": {
      "type": "string",
      "format": "date-time"
    },
    "actor": {
      "type": "object",
      "required": ["type", "id"]
    }
  }
}

4.8 장 요약

핵심 내용

  • 표준화된 JSON 스키마로 일관성 보장
  • 6가지 질문 답변: 누가, 무엇을, 언제, 어디서, 어떻게, 결과
  • Actor 유형: user, service, api_key, system
  • Action 유형: 인증, 권한, 데이터, 구성, 보안
  • 체인 메타데이터로 변조 방지
  • 데이터 분류로 민감도 관리
