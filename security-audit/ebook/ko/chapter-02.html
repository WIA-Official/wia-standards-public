2.2 변조 방지 체인 구조

보안 감사의 핵심 과제는 로그 자체가 공격자나 악의적 내부자에 의해 변조되지 않도록 보장하는 것입니다. WIA-SEC-017은 블록체인에서 영감을 받은 해시 체인 구조를 사용합니다.

해시 체인 구조:
┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│    항목 1       │    │    항목 2       │    │    항목 3       │
│  seq: 1         │───►│  seq: 2         │───►│  seq: 3         │
│  data: {...}    │    │  data: {...}    │    │  data: {...}    │
│  prev: null     │    │  prev: hash_1   │    │  prev: hash_2   │
│  hash: hash_1   │    │  hash: hash_2   │    │  hash: hash_3   │
│  sig: sig_1     │    │  sig: sig_2     │    │  sig: sig_3     │
└─────────────────┘    └─────────────────┘    └─────────────────┘

hash_n = SHA256(data + prev_hash)
sig_n = Sign(hash_n, private_key)

해시 체인의 작동 원리

  1. 각 감사 항목에는 고유한 순서 번호(sequence)가 할당됩니다
  2. 이벤트 데이터와 이전 항목의 해시가 결합되어 현재 해시가 계산됩니다
  3. 해시는 개인 키로 디지털 서명됩니다
  4. 항목이 수정되면 모든 후속 해시가 무효화됩니다

변조 탐지

공격자가 중간에 있는 항목을 수정하면:

  • 해당 항목의 해시가 변경됨
  • 다음 항목의 prev_hash와 불일치 발생
  • 체인 검증 시 즉시 변조 감지

2.3 저장소 계층

감사 데이터는 접근 빈도와 규정 준수 요구 사항에 따라 계층화된 저장소에 저장됩니다:

계층기간저장소 유형특징
Hot0-7일SSD/메모리실시간 쿼리, 고성능
Warm7-90일HDD/객체 저장소빠른 검색, 중간 비용
Cold90일-1년객체 저장소압축 저장, 저비용
Archive1년+Glacier/테이프규정 준수, 최저 비용

WORM 저장소

Write Once Read Many(WORM) 저장소는 감사 로그에 이상적입니다:

  • 한 번 기록된 데이터는 수정/삭제 불가
  • 법적 증거로 사용 가능
  • 규정 준수 요구 사항 충족
  • 클라우드 서비스: AWS S3 Object Lock, Azure Immutable Blob

2.4 보안 고려 사항

키 관리

  • HSM: 하드웨어 보안 모듈로 서명 키 보호
  • 키 순환: 정기적인 서명 키 교체
  • 분리: 서명 키와 암호화 키 분리

액세스 제어

직무 분리 원칙

  • 쓰기 전용 계정: 감사 이벤트 제출만 가능
  • 읽기 전용 계정: 로그 조회만 가능
  • 관리자 계정: 구성 변경, 별도 MFA 필요
  • 감사자 계정: 검증 및 보고서 생성

암호화

상태방법권장 사항
전송 중TLSTLS 1.3, AEAD 암호
저장 시AESAES-256-GCM
키 관리KMSHSM 또는 클라우드 KMS

2.5 장 요약

핵심 내용

  • 4계층 아키텍처: 수집, 처리, 저장, 분석
  • 해시 체인으로 변조 방지 보장
  • 디지털 서명으로 부인 방지 제공
  • 계층화된 저장소로 비용 최적화
  • WORM 저장소로 불변성 보장
  • 직무 분리로 보안 강화
