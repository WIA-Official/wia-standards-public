Sanctions Screening Overview

Sanctions screening is the process of checking customers, transactions, and business partners against government-issued sanctions lists to ensure compliance with economic and trade sanctions programs. Violations can result in severe penalties including hundreds of millions in fines and criminal prosecution.

Major Sanctions Programs

Authority List/Program Scope Prohibition
OFAC (US) SDN List ~10,000 individuals and entities All transactions blocked, assets frozen

PEP Screening

Politically Exposed Persons (PEPs) require Enhanced Due Diligence. Key categories:

  • Foreign PEPs: Heads of state, senior politicians, military, judges, state enterprise executives
  • Domestic PEPs: Prominent national figures (requirements vary by jurisdiction)
  • International Organization PEPs: Senior UN, IMF, World Bank officials
  • RCAs: Relatives and Close Associates of PEPs

const end = Math.min(i + matchDistance + 1, len2); for (let j = start; j < end; j++) { if (str2Matches[j] || str1[i] !== str2[j]) continue; str1Matches[i] = true; str2Matches[j] = true; matches++; break; } } if (matches === 0) return 0.0; let k = 0; for (let i = 0; i < len1; i++) { if (!str1Matches[i]) continue; while (!str2Matches[k]) k++; if (str1[i] !== str2[k]) transpositions++; k++; } return ( (matches / len1 + matches / len2 + (matches - transpositions / 2) / matches) / 3.0 ); // Exact match if (normalizedInput === normalizedRef) { return { score: 100, matchType: 'exact', algorithm: 'exact', normalized: { input: normalizedInput, reference: normalizedRef } }; } // Try multiple algorithms and take best score const levenScore = 100 - ( this.levenshteinDistance(normalizedInput, normalizedRef) / Math.max(normalizedInput.length, normalizedRef.length) * 100 ); When Account opening, transaction processing Nightly/periodic rescreening of customer base
Response Time <2 seconds required Hours acceptable
Purpose Prevent prohibited transactions Catch list updates, ongoing monitoring

Match Resolution Workflow

Not all screening matches are true hits. Effective resolution process:

  1. Automated filtering: Eliminate obvious false positives based on date of birth, address, nationality
  2. Analyst review: Human review of remaining matches
  3. Enhanced verification: Gather additional customer information
  4. Escalation: Borderline cases to senior compliance
  5. Disposition: Confirm match, clear as false positive, or escalate
  6. Action: Block/reject if confirmed hit, or proceed if cleared

Practical Implementation Checklist

  • Manifest integrity — every published manifest is signed via Sigstore DSSE. Tampering is detectable without trusting an intermediary. Updates are issued as new manifests; the prior manifest is preserved with a supersede marker.
  • Retention — keep signed evidence for at least seven years to satisfy ISO/IEC 17065:2012 audit expectations. Some regulators require longer retention; in that case the deployment policy MUST extend the window rather than relying on the defaults.
  • Versioning — follow Semantic Versioning 2.0.0. Major version bumps require at least a 90-day overlap with the prior major version on every WIA-published reference implementation.
  • Negative-result reporting — a skipped indicator without recorded justification is treated as a failed indicator at audit time.
  • Key rotation — manifest signing keys are rotated per the deployment policy. The previous key is preserved in a published key history so that auditors can verify historical manifests offline.

Reference Topologies

Implementations declare their operational topology in the manifest: single-tenant edge (one runtime per organization), multi-tenant gateway (shared runtime with header-based isolation), federated mesh (multiple runtimes peer to one another and publish manifests to a directory service), or air-gapped batch (signed evidence package transported out-of-band). Each topology ships with reference test vectors so that implementations can validate their deployment matches the canonical pattern.
