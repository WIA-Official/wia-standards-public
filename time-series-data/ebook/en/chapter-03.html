Chapter 03: Storage Engines and Compression

3.1 Storage Engines

WIA-DATA-014 does not mandate a particular storage engine. Reference deployments commonly use one of: InfluxDB (the canonical industrial-IoT engine), Prometheus (operationally focused, pull model), TimescaleDB (Postgres-based, full SQL), VictoriaMetrics (Prometheus-compatible with better cardinality handling), Apache IoTDB (designed for Apache Hadoop integration). The standard\x27s wire format maps cleanly to all five.

3.2 Compression Trade-offs

The dominant compression algorithms in time-series databases are Gorilla (Facebook\x27s float compression with delta-of-delta on timestamps) and various delta-encoding schemes for integer points. WIA-DATA-014 does not require a specific algorithm; the host advertises supported compression formats in the discovery document and the client negotiates via Accept-Encoding.

3.3 Index Design

Indexes are the difference between a query taking 50 milliseconds and 50 minutes. Two indexing strategies dominate: (1) per-series time-sorted (each series_id has its own time-sorted block, ideal when most queries scope to a single series); (2) per-label inverted (each label value points to the list of series carrying it, ideal when queries filter by label). Production hosts often build both.

3.4 Sharding

Beyond a few terabytes per node, sharding is unavoidable. Two common strategies: (1) shard by series_id hash (predictable distribution, but range queries fan out across shards); (2) shard by time window (range queries stay local, but series with very different cadences create hot shards). The standard does not pick a winner; the host\x27s operator picks based on the query mix.

3.5 Cold Storage

Long-tail data (5+ years old) typically migrates to cold storage (S3, GCS, Azure Blob) with an index that the live host queries on demand. The standard\x27s archive endpoint (Phase 2 §archive) supports this pattern by emitting Parquet files with deterministic naming so that any cold-storage backend can host them without WIA-specific code.

Summary

Storage engine choice is operator-dependent; the standard\x27s wire format is engine-agnostic. The implementer\x27s job is to map the WIA envelope to the chosen engine\x27s native format and to manage cardinality, retention, and sharding within the operational budget.

Implementer Notes

Three operational concerns recur across time-series implementations and are baked into the standard. First, cardinality management: every label dimension multiplies the number of distinct series the host must index, and beyond a few million distinct combinations index lookup latency degrades sharply. The standard recommends declaring per-series cardinality budgets and refusing writes that would push past them. Second, late-arriving points: mobile and edge sensors frequently emit measurements with timestamps in the past after a network outage; the standard requires hosts to accept late points up to a configurable bound (default 24 hours) and trigger a re-rollup. Third, retention policy: every series declares explicit retention bounds and downsampling rules, so capacity planning is not magic and operators can reason about long-term storage cost from the moment they create a series.
