Chapter 04: Phase 1 Data Format Walk-Through

4.1 Phase 1 in one paragraph

Phase 1 specifies the JSON envelopes for time-series data: series definitions, individual points, query envelopes, and rollup requests. Every envelope is signed with Ed25519 (IETF RFC 8032) so that signed payloads survive intermediate processing.

4.2 Series Envelope

The series envelope is the registration record. It declares the series name, unit (UCUM code), label set, retention policy, and downsampling rules. Hosts use the envelope to provision storage with the right retention bounds before any points arrive.

{
  "wia_time_series_data_version": "1.0.0",
  "type": "series",
  "series_id": "ts_temperature_kitchen_sensor_A",
  "unit": "Cel",
  "labels": { "env": "production", "region": "us-east-1" },
  "retention_seconds": 2592000,
  "downsampling_policy": [
    { "from_resolution_seconds": 1, "after_seconds": 3600,
      "to_resolution_seconds": 60 },
    { "from_resolution_seconds": 60, "after_seconds": 604800,
      "to_resolution_seconds": 3600 }
  ],
  "signature": { "alg": "Ed25519", "value": "..." }
}

4.3 Point Envelope

One observation. Required fields: series_id, captured_at, value, value_type. The captured_at timestamp is RFC 3339 UTC with millisecond precision (sub-second precision required for streams at ≥ 1 Hz).

4.4 Query Envelope

A query envelope expresses a structured request: which series, which time window, which aggregation, which resolution. Hosts execute the query against their internal index; the response is a stream of points or a single aggregate value depending on the requested resolution.

4.5 Rollup Request Envelope

Operators trigger explicit rollups via the rollup request envelope. The host computes the rollup over the requested time window and stores the result at the target resolution. Rollups are idempotent at the host level — re-running the same rollup yields the same result, allowing operators to safely retry.

Summary

Phase 1 envelopes are the source of truth for the wire format. JSON Schema 2020-12 documents at /spec/schemas/ are the authoritative validation reference.

Implementer Notes

Three operational concerns recur across time-series implementations and are baked into the standard. First, cardinality management: every label dimension multiplies the number of distinct series the host must index, and beyond a few million distinct combinations index lookup latency degrades sharply. The standard recommends declaring per-series cardinality budgets and refusing writes that would push past them. Second, late-arriving points: mobile and edge sensors frequently emit measurements with timestamps in the past after a network outage; the standard requires hosts to accept late points up to a configurable bound (default 24 hours) and trigger a re-rollup. Third, retention policy: every series declares explicit retention bounds and downsampling rules, so capacity planning is not magic and operators can reason about long-term storage cost from the moment they create a series.
