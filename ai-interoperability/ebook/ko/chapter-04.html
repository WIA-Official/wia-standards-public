🔄제4장: 1단계 - 모델 교환 형식

弘益人間 (홍익인간)
"표준화된 AI 모델 교환을 통해 모든 인류에게 이익을 - 모델이 플랫폼 간에 자유롭게 이동하면서 기능과 의도를 보존할 수 있도록 합니다."

WIA AI 상호운용성 표준의 첫 번째 단계에서는 AI 모델을 다양한 플랫폼과 환경에서 교환, 공유 및 배포할 수 있는 기본 데이터 형식을 설정합니다. 이 장에서는 상세한 JSON 사양, 검증 규칙 및 실용적인 구현 예제를 포함하여 WIA 모델 교환 스키마를 포괄적으로 다룹니다.

4.1 데이터 모델 개요

WIA 모델 교환 형식은 기존 모델 형식(ONNX, TensorFlow SavedModel, PyTorch 등)을 풍부한 메타데이터와 표준화된 구조로 래핑하는 플랫폼 독립적이고 확장 가능한 스키마로 설계되었습니다. 기존 모델 형식을 대체하는 대신, WIA는 모델이 다양한 AI 플랫폼에서 일관되게 이해, 검증 및 배포될 수 있도록 보장하는 범용 봉투를 제공합니다.

핵심 설계 원칙

  • 형식 독립적: 모든 주요 ML 프레임워크(TensorFlow, PyTorch, ONNX, JAX 등) 지원
  • 풍부한 메타데이터: 출처, 기능, 요구 사항 및 성능 특성을 포함한 포괄적인 모델 정보
  • 검증 우선: 데이터 무결성 및 호환성을 보장하는 내장 스키마 검증
  • 버전 관리된 진화: 하위 호환성을 위한 의미론적 버전 관리 지원
  • 보안 인식: 암호화 서명, 체크섬 및 보안 메타데이터
  • 사람이 읽을 수 있음: 기계 파싱 및 사람의 검사가 모두 가능한 JSON 기반 형식

모델 교환 수명주기

WIA 모델 교환 형식은 완전한 모델 수명주기를 지원합니다:

  1. 내보내기: 소스 프레임워크에서 WIA 메타데이터와 함께 모델이 패키징됩니다
  2. 검증: 스키마 검증 및 무결성 검사
  3. 전송: API, 레지스트리 또는 파일 시스템을 통해 모델 패키지 전송
  4. 발견: 메타데이터가 모델 검색 및 기능 매칭을 가능하게 함
  5. 가져오기: 대상 플랫폼이 모델을 검증하고 언팩합니다
  6. 배포: 적절한 런타임 구성으로 모델 인스턴스화
핵심 통찰: WIA 형식은 모델 바이너리를 페이로드로 취급하면서 메타데이터를 일급 관심사로 격상시킵니다. 이를 통해 플랫폼이 프레임워크별 모델 파일을 파싱할 필요 없이 지능적인 모델 관리, 발견 및 오케스트레이션이 가능합니다.

4.3 지원되는 모델 형식

WIA 모델 교환 형식은 모든 주요 ML 프레임워크의 모델 래핑을 지원합니다. modelFormat 섹션은 기본 모델 형식을 지정하여 수신 시스템이 모델 바이너리를 적절하게 처리할 수 있도록 합니다.

프레임워크 형식 파일 확장자 WIA 지원
ONNX onnx .onnx ✅ 완전
TensorFlow saved_model /saved_model/ ✅ 완전
TensorFlow keras_h5 .h5, .keras ✅ 완전
PyTorch torch_script .pt, .pth ✅ 완전
PyTorch state_dict .pth ✅ 완전
JAX flax .msgpack ✅ 완전

scikit-learn pickle .pkl ⚠️ 제한적

LightGBM lightgbm .txt ✅ 완전

프레임워크별 고려사항

ONNX (권장)

ONNX는 최대 상호운용성을 위한 선호 형식입니다. 다른 형식의 모델은 가능한 경우 ONNX로 변환되어야 합니다. ONNX는 표준화된 연산자를 제공하며 모든 주요 추론 프레임워크에서 지원됩니다.

TensorFlow SavedModel

SavedModel 형식은 완전한 계산 그래프, 변수 및 자산을 포함합니다. WIA 패키지는 루트 디렉토리를 가리키는 메타데이터와 함께 전체 SavedModel 디렉토리 구조를 포함해야 합니다.

PyTorch TorchScript

TorchScript는 추적 및 스크립팅 모드를 모두 제공합니다. WIA 메타데이터는 사용된 모드와 추적/스크립팅된 모델의 알려진 제한 사항을 지정해야 합니다.

Hugging Face Transformers

Transformers 모델에는 모델 가중치, 토크나이저 및 구성 파일이 포함됩니다. WIA 패키지는 모든 구성요소를 포함하고 모델 아키텍처(BERT, GPT, T5 등)를 지정해야 합니다.

4.5 검증 규칙

WIA 모델 교환 패키지는 수신 시스템에서 수락되기 전에 포괄적인 검증을 통과해야 합니다. 검증은 여러 수준에서 발생합니다:

스키마 검증 규칙

규칙 카테고리 검증 심각도
필수 필드 모든 필수 필드가 존재해야 함 ERROR
타입 검사 모든 필드가 지정된 타입과 일치해야 함 ERROR
형식 검증 UUID, 타임스탬프, URL이 유효해야 함 ERROR
열거형 검증 열거형 값이 허용된 집합에 있어야 함 ERROR
Semver 검증 버전 문자열이 semver를 따라야 함 ERROR
체크섬 검증 아티팩트 체크섬이 검증 가능해야 함 ERROR
서명 검증 암호화 서명이 유효해야 함 ERROR
종속성 해결 종속성이 해결 가능해야 함 WARNING
하드웨어 호환성 대상 시스템이 요구 사항을 충족해야 함 WARNING
라이선스 호환성 라이선스가 의도된 사용을 허용해야 함 WARNING

의미론적 검증 규칙

스키마 검증 외에도 의미론적 규칙은 논리적 일관성을 보장합니다:

  • 타임스탬프 일관성: modified는 created 이상이어야 함
  • 버전 일관성: 모델 버전은 모든 상위 모델 버전 이상이어야 함
  • 아티팩트 크기: 선언된 아티팩트 크기가 실제 파일 크기와 일치해야 함
  • 입/출력 호환성: 모델 서명이 선언된 기능과 호환되어야 함
  • 프레임워크 버전: 프레임워크 버전이 형식과 호환되어야 함
  • 양자화 일관성: 정밀도 및 양자화 설정이 호환되어야 함

보안 검증 규칙

보안 중요: 모든 WIA 모델 패키지는 암호화 방식으로 서명되어야 합니다. 수신 시스템은 프로덕션 환경에서 서명되지 않은 패키지를 거부해야 합니다.
  • 서명 필수: 프로덕션 배포에는 유효한 서명이 있어야 함
  • 체크섬 검증: 모든 아티팩트에 검증된 체크섬이 있어야 함(최소 SHA-256)
  • 취약점 스캔: 모델에 알려진 취약점이 스캔되어야 함
  • 라이선스 검증: 라이선스가 SPDX 호환 식별자여야 함
  • 종속성 안전성: 모든 종속성이 신뢰할 수 있는 소스에서 제공되어야 함

검증 구현

JSON 스키마를 사용한 Python 검증 예제:

import jsonschema
import hashlib
import json

def validate_wia_model(model_json, model_files):
    """
    WIA 모델 패키지 검증.

    Args:
        model_json: 파싱된 WIA 메타데이터 JSON
        model_files: 아티팩트 경로를 파일 내용에 매핑하는 딕셔너리

    Returns:
        오류 및 경고가 포함된 ValidationResult
    """
    errors = []
    warnings = []

    # 1. 스키마 검증
    try:
        jsonschema.validate(
            instance=model_json,
            schema=WIA_SCHEMA
        )
    except jsonschema.ValidationError as e:
        errors.append(f"스키마 검증 실패: {e.message}")
        return ValidationResult(errors, warnings)

    # 2. 체크섬 검증
    for artifact in model_json.get("artifacts", []):
        path = artifact["path"]
        expected_checksum = artifact["checksum"]["value"]
        algorithm = artifact["checksum"]["algorithm"]

        if path not in model_files:
            errors.append(f"아티팩트 누락: {path}")
            continue

        if algorithm == "sha256":
            actual = hashlib.sha256(model_files[path]).hexdigest()
        elif algorithm == "sha512":
            actual = hashlib.sha512(model_files[path]).hexdigest()
        else:
            errors.append(f"지원되지 않는 해시: {algorithm}")
            continue

        if actual != expected_checksum:
            errors.append(
                f"{path}의 체크섬 불일치: "
                f"예상 {expected_checksum}, 실제 {actual}"
            )

    # 3. 의미론적 검증
    created = model_json["modelMetadata"]["created"]
    modified = model_json["modelMetadata"]["modified"]

    if modified < created:
        errors.append(
            "수정 타임스탬프가 생성 타임스탬프보다 앞설 수 없습니다"
        )

    # 4. 하드웨어 요구 사항 확인
    reqs = model_json.get("performance", {}).get(
        "hardwareRequirements", {}
    )

    if not check_hardware_compatibility(reqs):
        warnings.append(
            "모델 하드웨어 요구 사항이 시스템 기능을 초과합니다"
        )

    return ValidationResult(errors, warnings)

4.7 레거시 형식에서 마이그레이션

기존 모델 관리 접근 방식에서 WIA 모델 교환 형식으로 전환하는 조직에는 명확한 마이그레이션 전략이 필요합니다. 이 섹션에서는 일반적인 마이그레이션 시나리오에 대한 지침을 제공합니다.

마이그레이션 패턴

레거시 형식 마이그레이션 전략 복잡도
MLflow 모델 MLmodel 파일에서 메타데이터 추출, WIA 스키마에 매핑 낮음
Hugging Face Hub 모델 카드를 WIA 메타데이터로 변환, 아티팩트 보존 낮음
TensorFlow Hub SavedModel 메타데이터 추출, WIA 봉투 추가 중간
사용자 정의 형식 수동 메타데이터 작성, 아티팩트 패키징 높음
모델 레지스트리 레지스트리 메타데이터 내보내기, WIA 스키마로 변환 중간

MLflow에서 WIA로 마이그레이션 예제

# MLflow 모델
mlflow_model/
├── MLmodel
├── conda.yaml
├── python_env.yaml
└── model/
    └── model.pkl

# WIA로 변환
wia_package/
├── wia-model.json          # WIA 메타데이터
├── artifacts/
│   └── model.pkl
├── mlflow/                 # 보존된 MLflow 메타데이터
│   ├── MLmodel
│   ├── conda.yaml
│   └── python_env.yaml
└── signature.txt

Python 마이그레이션 스크립트:

import mlflow
import json
from datetime import datetime

def migrate_mlflow_to_wia(mlflow_model_uri, output_path):
    """MLflow 모델을 WIA 형식으로 변환."""

    # MLflow 모델 로드
    model = mlflow.pyfunc.load_model(mlflow_model_uri)
    mlflow_metadata = model.metadata

    # WIA 메타데이터 생성
    wia_metadata = {
        "wiaVersion": "1.0.0",
        "schemaVersion": "1.0.0",
        "modelMetadata": {
            "id": str(uuid.uuid4()),
            "name": mlflow_metadata.run_id,
            "version": "1.0.0",
            "description": mlflow_metadata.signature.to_dict(),
            "created": datetime.utcnow().isoformat() + "Z",
            "modified": datetime.utcnow().isoformat() + "Z",
            "license": "Unknown",
            "tags": []
        },
        "modelFormat": {
            "framework": mlflow_metadata.flavors.get("python_function", {}).get("loader_module", "unknown"),
            "format": "mlflow"
        },
        "artifacts": extract_artifacts(mlflow_model_uri)
    }

    # WIA 패키지 작성
    with open(f"{output_path}/wia-model.json", "w") as f:
        json.dump(wia_metadata, f, indent=2)

    # 아티팩트 복사
    copy_artifacts(mlflow_model_uri, f"{output_path}/artifacts")

    return wia_metadata

하위 호환성

WIA 패키지는 가능한 경우 원래 모델 메타데이터를 보존하여 하위 호환성을 가능하게 해야 합니다:

  • 원본 파일 보존: 패키지에 원래 메타데이터 파일(MLmodel, 모델 카드 등) 포함
  • 이중 형식 지원: 시스템은 전환 중에 WIA 메타데이터와 레거시 형식을 모두 읽을 수 있음
  • 점진적 마이그레이션: WIA 인식 및 레거시 시스템이 공존하는 단계적 롤아웃 지원
  • 메타데이터 매핑: 레거시 및 WIA 스키마 간의 필드 매핑 문서화
마이그레이션 팁: extensions 필드를 사용하여 WIA 스키마에 직접 매핑되지 않는 레거시 메타데이터를 보존하세요. 예: "extensions": {"mlflow": {...}, "custom": {...}}

4.8 장 요약

이 장에서는 모델 교환 형식을 통해 WIA AI 상호운용성의 기초를 확립했습니다:

  • 포괄적인 스키마: 식별, 기능, 성능, 보안 및 출처를 다루는 풍부한 메타데이터가 포함된 JSON 기반 형식
  • 프레임워크 독립적: 모든 주요 ML 프레임워크(ONNX, TensorFlow, PyTorch, Hugging Face, XGBoost 등) 지원
  • 검증 우선: 스키마 준수, 의미론적 일관성 및 보안을 보장하는 다층 검증
  • 프로덕션 준비: NLP, 컴퓨터 비전 및 표 형식 작업에 대한 실제 모델 패키징을 보여주는 완전한 예제
  • 마이그레이션 경로: MLflow 및 Hugging Face와 같은 레거시 형식에서 전환하기 위한 명확한 전략

WIA 모델 교환 형식은 모든 상위 수준 상호운용성 기능을 가능하게 하는 데이터 기반을 제공합니다. 모델이 패키징되고 설명되는 방식을 표준화함으로써 다음을 가능하게 합니다:

  • 자동화된 모델 발견 및 선택
  • 크로스 플랫폼 모델 배포
  • 지능적인 모델 라우팅 및 오케스트레이션
  • 포괄적인 모델 거버넌스 및 감사
  • 조직 경계를 넘어 안전한 모델 공유
핵심 요점: 모델 교환 형식은 단순한 기술 사양이 아닙니다. 모델이 플랫폼 간에 자유롭게 이동하면서 정체성, 기능 및 의도를 보존하는 개방적이고 상호운용 가능한 AI 생태계의 기초입니다.

4.9 복습 질문

  1. WIA 모델 교환 형식의 6가지 핵심 설계 원칙은 무엇이며, 각각이 광범위한 AI 상호운용성을 달성하는 데 중요한 이유는 무엇입니까?
  2. 스키마 검증과 의미론적 검증의 차이점을 설명하세요. 의미론적 검증에는 실패하지만 스키마 검증에는 통과하는 검증 규칙의 예를 제공하세요.
  3. WIA가 최대 상호운용성을 위해 ONNX를 선호 모델 형식으로 권장하는 이유는 무엇입니까? TensorFlow SavedModel 또는 PyTorch TorchScript와 같은 프레임워크 네이티브 형식과 비교할 때 절충점은 무엇입니까?
  4. 각 주요 메타데이터 섹션(modelMetadata, modelCapabilities, performance, provenance, security)의 목적을 설명하세요. 이 메타데이터가 어떻게 지능적인 모델 발견 및 배포를 가능하게 합니까?
  5. 프로덕션 환경에서 WIA 모델 패키지에 필요한 보안 조치는 무엇입니까? 암호화 서명이 필수적인 이유는 무엇이며, 어떤 공격을 방지합니까?
  6. 레거시 MLflow 모델 레지스트리를 WIA 형식으로 마이그레이션하고 있습니다. 어떤 메타데이터가 보존되고, 어떤 메타데이터가 변환되며, 어떤 새로운 메타데이터를 추가해야 합니까? 마이그레이션 전략과 예상되는 과제를 설명하세요.

4.10 다음 장 미리보기

모델 교환 형식이 확립되었으므로 이제 AI 모델을 패키징하고 설명하는 표준화된 방법이 생겼습니다. 그러나 모델은 배포되고 호출될 수 있을 때만 유용합니다. 제5장에서는 2단계: 프로토콜 사양을 살펴보며 다음을 다룹니다:

  • WIA API 프로토콜: 모델 호출을 위한 RESTful 및 gRPC API
  • 요청/응답 형식: 표준화된 추론 페이로드
  • 스트리밍 지원: 생성형 AI를 위한 실시간 모델 출력
  • 배치 처리: 효율적인 대량 추론 프로토콜
  • 오류 처리: 포괄적인 오류 코드 및 복구 전략
  • 인증 및 권한 부여: 보안 API 액세스 패턴

프로토콜 레이어는 모델에 생명을 불어넣어 애플리케이션이 이기종 플랫폼에서 AI 기능을 발견, 호출 및 오케스트레이션할 수 있도록 합니다. 모델 교환 형식과 프로토콜 사양이 함께 범용 AI 상호운용성에 대한 WIA의 비전의 기술적 기반을 형성합니다.

다음 장 미리보기: 제5장에서는 WIA 형식으로 패키징된 모델을 표준화된 API를 사용하여 호출하는 방법을 배웁니다. LLM용 스트리밍 응답, 동기 및 비동기 호출 패턴, 프로덕션 API 설계 모범 사례를 다룹니다.
