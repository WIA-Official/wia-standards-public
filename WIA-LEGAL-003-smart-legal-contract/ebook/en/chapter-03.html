Chapter 3: Phase 1 - Data Format Specifications

3.1 Introduction to Data Format Standards

Phase 1 of the WIA-LEGAL-003 standard establishes the foundational data structures and formats that enable Smart Legal Contracts to be created, stored, exchanged, and interpreted consistently across different platforms, tools, and jurisdictions. Just as HTML provides a standard format for web pages and JSON provides a standard for data interchange, WIA-LEGAL-003 Phase 1 defines how Smart Legal Contracts are structured and represented.

Standardized data formats solve critical interoperability problems. Without common formats, each platform implements SLCs differently, creating silos that prevent contracts from moving between systems. Legal professionals cannot review contracts created on unfamiliar platforms; developers cannot build tools that work across ecosystems; and regulators cannot effectively audit or supervise SLC deployments. Phase 1 addresses these challenges through comprehensive specifications for contract structure, metadata, code-law linkages, and versioning.

This chapter details the technical specifications of Phase 1, explaining how each component works and why specific design decisions were made. Understanding these foundations is essential for implementing WIA-LEGAL-003-compliant systems and for appreciating how subsequent phases build upon these base formats.

3.2 Core Contract Structure

3.3 Party Identification and Authentication

3.3.1 Legal Identity Representation

Smart Legal Contracts must bridge the gap between blockchain addresses (pseudonymous identifiers) and legal identities (real-world persons and organizations). The standard defines structured party representations that link blockchain identities to legally recognizable entities while preserving privacy where appropriate.

Each party object includes multiple identification layers:

  • Legal Name: Official registered name of person or organization
  • Jurisdiction: Primary legal jurisdiction using ISO 3166-2 codes
  • Legal Identifier: Official registration numbers (EIN, VAT, Corporate Registration, etc.)
  • Blockchain Address: Public key address that controls contract execution
  • Verification Credentials: Optional cryptographic proofs of identity (W3C Verifiable Credentials)

3.4 Code-Law Linkage Mechanisms

3.4.1 Bidirectional Mapping Format

The most critical challenge in Smart Legal Contracts is establishing clear, verifiable connections between legal prose and executable code. When disputes arise or when parties need to understand contract behavior, they must be able to trace which code implements which legal provisions. The standard defines a bidirectional mapping format that links contract clauses to code functions and vice versa.

3.4.2 Mapping Relationship Types

Not all code-law relationships are simple implementations. The standard defines multiple relationship types to capture nuanced connections:

Table 3.3: Code-Law Relationship Types
Relationship Description Legal Effect Example
implements Code directly executes the legal provision Code execution satisfies legal obligation Payment function implements payment clause
enforces Code prevents violations of legal provision Code prevents legally prohibited actions Access control enforces authorization requirements
monitors Code tracks compliance without direct enforcement Provides evidence; doesn't prevent violations Event logging monitors delivery milestones
facilitates Code enables but doesn't mandate legal action Provides mechanism; doesn't create obligation Dispute resolution voting mechanism
calculates Code performs numerical computations specified in law Computation result binds parties Interest calculation per legal formula
conflicts Code behavior contradicts legal provision Legal text prevails over code Bug causes incorrect execution

3.5 Versioning and Amendment Tracking

3.5.1 Semantic Versioning for Contracts

Smart Legal Contracts evolve over time through amendments, bug fixes, and regulatory updates. The standard adopts semantic versioning (SemVer) principles adapted for legal contracts, where version numbers convey meaningful information about the nature of changes.

Version Format: MAJOR.MINOR.PATCH[-PRERELEASE][+BUILD]

  • MAJOR: Incremented for substantive changes to parties' rights and obligations (e.g., 1.0.0 → 2.0.0)
  • MINOR: Incremented for backward-compatible additions or clarifications (e.g., 1.0.0 → 1.1.0)
  • PATCH: Incremented for bug fixes and technical corrections that don't alter intent (e.g., 1.0.0 → 1.0.1)
  • PRERELEASE: Optional identifier for draft versions (e.g., 1.0.0-draft.3)
  • BUILD: Optional build metadata (e.g., 1.0.0+20251227)

3.6 Content Addressing and Integrity Verification

3.6.1 IPFS Integration for Document Storage

Storing large legal documents directly on blockchain is prohibitively expensive. The standard uses content-addressed storage systems like IPFS (InterPlanetary File System) for document storage while maintaining blockchain records of document hashes to ensure integrity.

Content Addressing Benefits:

  • Immutability: IPFS addresses are cryptographic hashes of content; changing content changes address
  • Deduplication: Identical documents share the same address, saving storage
  • Distributed Availability: Documents can be retrieved from any IPFS node that pins the content
  • Verification: Anyone can verify document authenticity by comparing hash to IPFS address

3.9 Implementation Best Practices

3.9.1 Validation and Schema Compliance

All WIA-LEGAL-003 Phase 1 implementations must validate contract data against official JSON Schema definitions. This ensures interoperability and catches formatting errors before contracts are deployed.

// Example validation using JSON Schema
const Ajv = require('ajv');
const ajv = new Ajv();

// Load official WIA-LEGAL-003 schema
const schema = require('@wia/legal-003-schema-v1');

// Validate SLC data
function validateSLC(slcData) {
  const validate = ajv.compile(schema);
  const valid = validate(slcData);

  if (!valid) {
    console.error('Validation errors:', validate.errors);
    return false;
  }

  return true;
}

// Additional semantic validation
function semanticValidation(slcData) {
  // Check that all parties have blockchain addresses
  const partiesValid = slcData.parties.every(
    party => party.blockchainAddress
  );

  // Verify code-law mappings reference existing components
  const mappingsValid = slcData.codeLawMappings.every(
    mapping => {
      const legalExists = documentExists(mapping.legalClause.documentId);
      const codeExists = contractExists(mapping.codeImplementation.contractAddress);
      return legalExists && codeExists;
    }
  );

  return partiesValid && mappingsValid;
}
