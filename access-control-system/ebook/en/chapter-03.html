Chapter 3: WIA Standard Overview

The WIA Access Control System (WIA-ACS) standard provides a comprehensive, unified approach to physical and digital access control. Designed to address the challenges outlined in Chapter 2, WIA-ACS offers standardized data formats, APIs, protocols, and integration patterns that enable interoperability, security, and scalability. This chapter provides an overview of the WIA-ACS standard, its design principles, architecture, and four-phase implementation model.

Design Philosophy and Principles

WIA-ACS is built on the foundational philosophy of 弘益人間 (Hongik Ingan)—broadly benefiting humanity. This philosophy manifests in several core design principles:

1. Openness and Interoperability

WIA-ACS uses open, well-documented standards rather than proprietary protocols. All specifications are publicly available, enabling any vendor to implement WIA-ACS-compliant systems without licensing fees or vendor lock-in.

2. Security by Design

Security is not an afterthought but built into every layer:

  • All communications encrypted by default (TLS 1.3+)
  • Multi-factor authentication required for privileged operations
  • Zero Trust principles applied throughout
  • Comprehensive audit logging with cryptographic integrity
  • Regular security updates and vulnerability disclosure process

3. Privacy Protection

WIA-ACS incorporates privacy-preserving techniques:

  • Data minimization—collect only necessary information
  • Purpose limitation—use data only for intended purposes
  • Consent management for optional data collection
  • Right to access, rectification, and deletion (GDPR-compliant)
  • Pseudonymization options for analytics and reporting

4. Scalability and Performance

The standard supports deployments from small businesses to global enterprises:

  • Horizontal scaling through distributed architectures
  • Edge computing for low-latency local decisions
  • Efficient data structures minimizing storage and bandwidth
  • Caching strategies for high-frequency operations
  • Asynchronous processing for non-critical operations

5. Accessibility and Usability

WIA-ACS prioritizes user experience for both end users and administrators:

  • Unified credential management across physical and digital systems
  • Self-service portals for common operations
  • Intuitive administrative interfaces
  • Comprehensive documentation and training materials
  • Support for 99+ languages and regional variants

Application Layer

User-facing applications and integrations consume WIA-ACS services through standardized APIs. This layer includes:

  • Administrative Portal: Web-based management console
  • Self-Service Portal: User credential management
  • Mobile Applications: iOS and Android apps for mobile credentials
  • Third-Party Integrations: HR systems, visitor management, etc.

Service Layer

Core business logic implemented as microservices:

Service Responsibility Key Functions
Authentication Verify user identity Credential validation, MFA, session management
Authorization Grant/deny access RBAC/ABAC evaluation, policy enforcement
Credential Manage credentials Provisioning, revocation, renewal, encoding
Audit Log all events Event collection, integrity verification, reporting
User User lifecycle Provisioning, deprovisioning, profile management
Device Hardware management Reader/controller registration, health monitoring

Data Layer

Persistent storage with strong consistency guarantees and encryption at rest. WIA-ACS supports multiple database backends while defining standard schemas for interoperability.

Integration Layer

Adapters and connectors enable WIA-ACS to integrate with existing systems. The standard defines common integration patterns for:

  • Directory services (LDAP, Active Directory)
  • Physical access control systems (PACS)
  • Identity providers (SAML, OAuth, OpenID Connect)
  • Biometric devices and databases
  • Smart card and RFID systems

Four-Phase Implementation Model

WIA-ACS uses a phased approach to implementation, allowing organizations to adopt the standard incrementally while achieving immediate benefits at each phase.

Phase 1: Data Format

Standardize data schemas for users, credentials, permissions, and audit events. This foundational phase enables data exchange and reporting across systems even without full integration.

Deliverables:

  • JSON schemas for all core entities
  • Data validation rules and constraints
  • Import/export templates
  • Migration guides from legacy formats

Benefits: Unified reporting, easier migrations, data portability

Phase 2: API Interface

Define RESTful and gRPC APIs for authentication, authorization, credential management, and auditing. Applications can now programmatically interact with access control systems.

Deliverables:

  • OpenAPI 3.0 specifications
  • gRPC protocol buffer definitions
  • SDK libraries (TypeScript, Python, Go, Java)
  • API reference documentation

Benefits: Automation, integration flexibility, self-service capabilities

Phase 3: Protocol

Establish secure communication protocols using TLS 1.3, OAuth 2.0, OpenID Connect, and SAML 2.0. This ensures end-to-end encryption and federated identity support.

Deliverables:

  • Protocol specifications and flow diagrams
  • Certificate management guidelines
  • Federation configuration guides
  • Security testing tools

Benefits: Strong security, single sign-on, compliance readiness

Phase 4: Integration

Deploy reference implementations and integration adapters for common enterprise systems. Organizations can connect WIA-ACS to existing LDAP, PACS, biometric, and IoT infrastructure.

Deliverables:

  • Reference implementation (open source)
  • Integration adapters for major vendors
  • Deployment guides and best practices
  • Certification program for compliant products

Benefits: Rapid deployment, vendor interoperability, future-proof architecture

WIA-ACS vs. Existing Standards

WIA-ACS complements and extends existing standards rather than replacing them:

Standard Focus WIA-ACS Relationship
OSDP Reader-controller communication WIA-ACS supports OSDP as a transport protocol
OAuth 2.0 Authorization framework WIA-ACS uses OAuth for API authorization
OpenID Connect Authentication layer on OAuth WIA-ACS supports OIDC for user authentication
SAML 2.0 Enterprise SSO WIA-ACS provides SAML integration for enterprise IdPs
LDAP Directory services WIA-ACS integrates with LDAP/AD as user source
XACML Policy language WIA-ACS can express policies in XACML

The key differentiator is that WIA-ACS provides a complete, end-to-end standard covering the full access control lifecycle—from credential provisioning to audit reporting—while existing standards typically address only specific aspects.

Certification and Compliance

WIA-ACS includes a certification program to verify compliance and interoperability:

Certification Levels

Level Requirements Suitable For
Bronze Phase 1 (Data Format) compliance Basic interoperability, reporting tools
Silver Phase 1 + 2 (API Interface) Integrated applications, automation
Gold Phase 1 + 2 + 3 (Protocol) Enterprise deployments, high security
Platinum All phases + reference integrations Complete systems, multi-vendor environments

Certification Process

  1. Self-Assessment: Vendor completes compliance checklist
  2. Documentation Review: Submit technical documentation and architecture
  3. Conformance Testing: Automated test suite validates API compliance
  4. Interoperability Testing: Cross-vendor integration verification
  5. Security Audit: Third-party security assessment
  6. Certification Granted: Public listing in WIA-ACS certified products directory

Open Source Reference Implementation

To accelerate adoption and ensure standard quality, WIA provides an open-source reference implementation covering all four phases. This implementation serves multiple purposes:

  • Validation: Proves the standard is implementable and performant
  • Education: Demonstrates best practices and design patterns
  • Rapid Deployment: Organizations can deploy immediately without vendor selection
  • Integration Testing: Vendors can test against a known-good implementation
  • Community Development: Contributions improve the standard itself

Governance and Evolution

WIA-ACS is governed by an open, transparent process:

Technical Steering Committee

Representatives from vendors, integrators, end users, and security researchers guide standard development.

Working Groups

  • Security Working Group: Threat modeling, vulnerability response
  • Privacy Working Group: GDPR compliance, privacy-enhancing technologies
  • Integration Working Group: Vendor adapters, interoperability testing
  • Accessibility Working Group: Inclusive design, internationalization

Version Management

WIA-ACS follows semantic versioning (MAJOR.MINOR.PATCH):

  • Major: Breaking changes (e.g., 1.x → 2.x)
  • Minor: Backward-compatible features (e.g., 1.0 → 1.1)
  • Patch: Bug fixes and clarifications (e.g., 1.0.1 → 1.0.2)

Major versions maintained for 5 years, minor versions for 2 years, ensuring long-term stability.
