Chapter 4: Phase 1 - Data Format

Phase 1 of the WIA-ACS standard establishes standardized data formats for all entities in an access control system. By defining common schemas for users, credentials, permissions, access rules, and audit events, this phase enables data exchange, reporting, and migration between systems—even without full API integration. This chapter details the data models, schemas, and validation rules that form the foundation of WIA-ACS interoperability.

User Entity Schema

The User entity represents an individual person who may be granted access to resources. WIA-ACS user schema balances completeness with privacy, collecting only necessary information.

Required vs. Optional Fields

Field Required Format Purpose
user_id Yes String (UUID) Unique identifier across all WIA-ACS systems
external_id No String ID in source system (HR, LDAP, etc.)
status Yes Enum active, suspended, terminated
identity.* Partial Various Minimum: given_name, family_name, email
roles Yes Array At least one role required for access
attributes No Object Custom attributes for ABAC policies

Data Validation and Constraints

All WIA-ACS schemas include JSON Schema validation rules to ensure data integrity: 

Example Validation Rules:

user_id:
  - Pattern: ^usr-[0-9]{8}-[a-z0-9]{6}$
  - Uniqueness: Global unique constraint
  - Immutable: Cannot be changed after creation

email:
  - Format: Valid RFC 5322 email address
  - Maximum length: 255 characters
  - Must be verified before use in authentication

status:
  - Enum: ["active", "suspended", "terminated"]
  - Transition rules:
      active → suspended (reversible)
      active → terminated (irreversible)
      suspended → active (with approval)

roles:
  - Type: Array of role IDs
  - Minimum items: 1 (at least one role required)
  - Each role must exist in role database
  - Circular dependencies detected and prevented

timestamp fields:
  - Format: ISO 8601 with timezone (RFC 3339)
  - Validation: Future timestamps rejected for created_at
  - Precision: Milliseconds

expires_at:
  - Must be after issued_at
  - Maximum validity: Configurable (default 2 years)
  - Warning period: 30 days before expiration

Data Migration and Import/Export

WIA-ACS provides standardized formats for data migration from legacy systems:

CSV Import Format

user_id,external_id,status,given_name,family_name,email,department,roles
usr-001,E-12345,active,John,Doe,john.doe@example.com,Engineering,"role-employee,role-engineering"
usr-002,E-12346,active,Jane,Smith,jane.smith@example.com,Security,"role-employee,role-security"

Validation:
- Header row required
- UTF-8 encoding
- Quote strings containing commas
- Array fields pipe-delimited alternative: "role-employee|role-engineering"

Bulk Export Format (NDJSON)

Newline-Delimited JSON for efficient streaming:

{"user_id":"usr-001","status":"active","identity":{"given_name":"John"...}}
{"user_id":"usr-002","status":"active","identity":{"given_name":"Jane"...}}
{"user_id":"usr-003","status":"terminated","identity":{"given_name":"Bob"...}}

Benefits:
- Stream processing of large datasets
- Fault tolerance (corrupted line doesn't break entire file)
- Easy to parallelize processing
- Compatible with tools like jq, BigQuery, Athena

Chapter Summary

This chapter detailed Phase 1 of WIA-ACS: standardized data formats. We explored schemas for users, credentials, roles, permissions, access rules, and audit events. These JSON-based schemas enable data exchange between systems, support both RBAC and ABAC models, include comprehensive validation rules, and provide migration tools for legacy systems.

Key Takeaways

  1. Standardized schemas enable interoperability even without full system integration
  2. User and credential schemas balance security requirements with privacy protection
  3. Permission models support both simple RBAC and complex ABAC authorization
  4. Audit event schema includes cryptographic integrity for tamper-proof logging
  5. Data validation rules ensure consistency and prevent common errors across implementations
