✅ Chapter 7: Phase 4 - Integration

7.1 Overview of Phase 4

Phase 4 of the WIA-DATA_QUALITY Standard addresses the challenge of embedding data quality throughout enterprise data architectures. While earlier phases define formats, APIs, and protocols, Phase 4 provides the patterns, connectors, and best practices for integrating quality into existing systems. This phase transforms data quality from a standalone capability into a core component of the data platform.

Enterprise data environments are complex, typically including multiple databases, data warehouses, data lakes, streaming platforms, and applications. Data flows through ETL/ELT pipelines, undergoes transformations, and is consumed by analytics and AI applications. Phase 4 ensures that quality is assessed and maintained at every stage of this journey, not just at rest in storage systems.

The integration phase takes a "quality-as-code" approach, defining quality rules and thresholds alongside the pipelines and applications that process data. This enables quality checks to be version-controlled, tested, and deployed through the same CI/CD processes used for application code. Quality becomes a first-class citizen in the development lifecycle.

Phase 4 also addresses organizational integration, providing connectors to data governance platforms, data catalogs, and workflow systems. Quality assessments flow into governance dashboards. Quality metadata enriches catalog entries. Quality alerts trigger remediation workflows. This integration ensures quality information reaches all stakeholders who need it.

7.3 ETL/ELT Pipeline Integration

Data quality must be integrated into data pipelines to catch issues before they propagate downstream. Phase 4 defines patterns for integrating quality checks into popular orchestration frameworks including Apache Airflow, dbt, Prefect, and Dagster.

Airflow Integration

# WIA Data Quality operator for Airflow
from wia_quality.airflow import WIAQualityOperator, WIAQualityBranchOperator

with DAG('customer_pipeline', schedule_interval='@daily') as dag:

    extract = PythonOperator(
        task_id='extract_data',
        python_callable=extract_customers
    )

    # Quality check after extraction
    quality_check = WIAQualityOperator(
        task_id='quality_check',
        source={'type': 'table', 'name': 'staging.customers'},
        profile='standard',
        thresholds={
            'completeness': 95,
            'validity': 98
        },
        fail_on_breach=True
    )

    # Conditional routing based on quality
    route = WIAQualityBranchOperator(
        task_id='quality_route',
        quality_task_id='quality_check',
        good_path='transform',
        bad_path='quarantine'
    )

    transform = PythonOperator(
        task_id='transform',
        python_callable=transform_customers
    )

    quarantine = PythonOperator(
        task_id='quarantine',
        python_callable=quarantine_bad_data
    )

    extract >> quality_check >> route >> [transform, quarantine]

7.5 Data Governance Integration

Data quality is a core component of data governance. Phase 4 defines patterns for integrating quality with governance processes including policy management, stewardship workflows, and compliance reporting.

Stewardship Workflow Integration

// Create stewardship task for quality issue
const governanceClient = new GovernanceClient({
  endpoint: 'https://governance.example.com/api'
});

// When quality issue is detected
qualityService.on('quality.issue.detected', async (event) => {
  const issue = event.data;

  if (issue.severity === 'critical' || issue.severity === 'high') {
    // Create stewardship task
    await governanceClient.createTask({
      type: 'DATA_QUALITY_REMEDIATION',
      title: `Quality issue: ${issue.description}`,
      description: `
        Dataset: ${issue.target.identifier}
        Dimension: ${issue.dimension}
        Severity: ${issue.severity}
        Affected Records: ${issue.context.recordsAffected}
      `,
      assignee: await governanceClient.getDataSteward(issue.target.identifier),
      priority: issue.severity === 'critical' ? 'P1' : 'P2',
      dueDate: addBusinessDays(new Date(), issue.severity === 'critical' ? 1 : 5),
      metadata: {
        qualityIssueId: issue.id,
        assessmentId: issue.assessmentId
      }
    });
  }
});

7.6 CI/CD Integration

Quality-as-code means quality rules should be version-controlled and tested like application code. Phase 4 defines patterns for integrating quality into CI/CD pipelines.

# GitHub Actions workflow for data quality
name: Data Quality CI

on:
  push:
    paths:
      - 'dbt/**'
      - 'quality-rules/**'
  pull_request:
    paths:
      - 'dbt/**'
      - 'quality-rules/**'

jobs:
  quality-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Setup WIA Quality CLI
        run: |
          npm install -g @wia/data-quality-cli

      - name: Validate Quality Rules
        run: |
          wia-dq rules validate ./quality-rules/

      - name: Run Quality Tests on Sample Data
        run: |
          wia-dq test ./tests/quality-tests.yaml \
            --data ./tests/sample-data/ \
            --report junit

      - name: Publish Test Results
        uses: EnricoMi/publish-unit-test-result-action@v2
        with:
          files: '**/quality-test-results.xml'

      - name: Quality Gate Check
        run: |
          wia-dq gate check ./quality-gates.yaml
        env:
          WIA_API_KEY: ${{ secrets.WIA_API_KEY }}

7.8 Enterprise Architecture Patterns

Phase 4 defines architectural patterns for embedding quality in different data architectures including data mesh, data fabric, and traditional centralized approaches.

Architecture Quality Pattern Key Considerations
Data Mesh Federated quality with domain ownership Domain teams own quality, federated governance
Data Fabric Active metadata-driven quality Quality rules from metadata, automated enforcement
Centralized DW Layered quality (bronze/silver/gold) Quality gates between layers, centralized team
Lambda Architecture Batch + streaming quality Different rules for batch vs real-time
