✅ Chapter 4: Phase 1 - Data Format

4.1 Overview of Phase 1

Phase 1 of the WIA-DATA_QUALITY Standard establishes the foundational data formats for representing quality-related information. This phase is essential because without standardized formats, organizations cannot exchange quality data, compare quality across systems, or build interoperable quality tools. Phase 1 provides the common language that makes all subsequent phases possible.

The data formats defined in Phase 1 cover three primary areas: Quality Records that capture assessments for individual data elements, Metadata Schemas that describe data structures and their quality characteristics, and Profile Formats that summarize quality across datasets. Together, these formats provide a complete picture of data quality at all levels of granularity.

All formats in Phase 1 are defined using JSON Schema, which provides a standard way to describe the structure and validation rules for JSON documents. While JSON is the primary serialization format, the standard also supports Protocol Buffers for high-performance scenarios. This flexibility ensures the standard can be used in both human-readable contexts and high-throughput data pipelines.

Phase 1 can be implemented independently, providing immediate value even without implementing subsequent phases. Organizations that adopt Phase 1 gain the ability to represent quality data consistently, exchange quality information with partners, and establish baselines for quality improvement. This makes Phase 1 an excellent starting point for organizations new to the WIA-DATA_QUALITY Standard.

4.2 Quality Record Schema

The Quality Record is the fundamental unit of quality data in the WIA-DATA_QUALITY Standard. Each Quality Record captures the quality assessment for a specific data element, record, or dataset. The schema is designed to be comprehensive yet flexible, capturing all relevant quality information while allowing for extension.

{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "type": "object",
  "required": ["type", "version", "id", "timestamp", "target", "assessment"],
  "properties": {
    "type": {
      "type": "string",
      "const": "WIA-DATA_QUALITY-Record"
    },
    "version": {
      "type": "string",
      "pattern": "^\\d+\\.\\d+$"
    },
    "id": {
      "type": "string",
      "format": "uuid"
    },
    "timestamp": {
      "type": "string",
      "format": "date-time"
    },
    "target": {
      "type": "object",
      "properties": {
        "type": { "enum": ["field", "record", "dataset", "system"] },
        "identifier": { "type": "string" },
        "path": { "type": "string" }
      }
    },
    "assessment": {
      "type": "object",
      "properties": {
        "dimensions": { "$ref": "#/$defs/dimensions" },
        "overallScore": { "type": "number", "minimum": 0, "maximum": 100 },
        "issues": { "type": "array", "items": { "$ref": "#/$defs/issue" } }
      }
    }
  }
}

The target object identifies what the quality assessment applies to. The type field indicates the granularity (field, record, dataset, or system), the identifier provides a unique reference, and the path shows the location within a data structure. This flexibility allows quality records to be attached at any level of the data hierarchy.

The assessment object contains the actual quality measurements. It includes dimension-specific scores, an aggregated overall score, and a list of specific issues found. This structure supports both summary reporting (looking at overall scores) and detailed analysis (examining individual issues).

4.3 Dimension Definitions

Each quality dimension has a standardized definition that ensures consistent measurement across implementations. The dimension definitions specify the calculation method, required inputs, and valid ranges for each metric.

Dimension Data Type Range Required Inputs
completeness number 0-100 requiredFields, populatedFields
accuracy number 0-100 verifiedRecords, matchingRecords
consistency number 0-100 crossSystemRecords, matchingRecords
timeliness number 0-100 dataAge, freshnessRequirement
validity number 0-100 totalRecords, validRecords
uniqueness number 0-100 totalRecords, uniqueRecords

4.4 Issue Format

Quality issues are represented in a standardized format that captures both the nature of the problem and sufficient context for remediation. The issue format is designed to support both human review and automated processing.

{
  "$defs": {
    "issue": {
      "type": "object",
      "required": ["id", "severity", "dimension", "description"],
      "properties": {
        "id": { "type": "string" },
        "severity": { "enum": ["critical", "high", "medium", "low", "info"] },
        "dimension": { "enum": ["completeness", "accuracy", "consistency", "timeliness", "validity", "uniqueness"] },
        "description": { "type": "string" },
        "location": {
          "type": "object",
          "properties": {
            "recordId": { "type": "string" },
            "field": { "type": "string" },
            "value": { "type": "string" }
          }
        },
        "remediation": {
          "type": "object",
          "properties": {
            "action": { "enum": ["manual-review", "auto-correct", "delete", "flag"] },
            "suggestedValue": { "type": "string" },
            "confidence": { "type": "number" }
          }
        }
      }
    }
  }
}

The severity classification helps prioritize remediation efforts. Critical issues may prevent data from being used at all, while low-severity issues may be acceptable in some contexts. The standard defines guidelines for classifying severity based on business impact and data criticality.

The remediation object provides actionable guidance for addressing the issue. It can suggest specific corrections (with confidence scores for automated suggestions) or indicate that manual review is required. This structure supports both automated cleansing pipelines and human data steward workflows.

4.5 Metadata Schema

The Metadata Schema describes data structures and their quality characteristics. This schema enables documentation of quality requirements, historical quality trends, and relationships between data elements. It provides the context needed to interpret quality assessments.

{
  "type": "WIA-DATA_QUALITY-Metadata",
  "version": "1.0",
  "dataset": {
    "name": "customer_master",
    "description": "Master customer data from CRM system",
    "owner": "sales-ops-team",
    "classification": "confidential",
    "updateFrequency": "daily"
  },
  "fields": [
    {
      "name": "customer_id",
      "type": "string",
      "description": "Unique customer identifier",
      "qualityRequirements": {
        "completeness": {"required": true, "threshold": 100},
        "uniqueness": {"required": true, "threshold": 100}
      }
    },
    {
      "name": "email",
      "type": "string",
      "description": "Customer email address",
      "qualityRequirements": {
        "completeness": {"required": true, "threshold": 95},
        "validity": {"required": true, "threshold": 99, "rules": ["email-format"]}
      }
    }
  ],
  "qualityHistory": [
    {"date": "2025-01-01", "overallScore": 94.5},
    {"date": "2025-01-08", "overallScore": 96.2}
  ]
}

4.6 Profile Format

The Profile Format provides summary statistics for datasets, enabling quick assessment of data quality without examining individual records. Profiles are generated through data profiling and serve as the basis for quality dashboards and trend analysis.

4.7 Validation Rules

Phase 1 includes a library of standard validation rules that can be referenced in metadata and used in quality assessments. These rules cover common validation scenarios and provide consistent behavior across implementations.

Rule ID Description Parameters
not-null Value must not be null or empty None
email-format Value must match email format None
phone-format Value must match phone format region: string
date-range Date must fall within range min: date, max: date
numeric-range Number must fall within range min: number, max: number

4.8 Extension Mechanisms

The Phase 1 formats support extension for organization-specific needs. Custom dimensions, rules, and metadata can be added using designated extension points without breaking compatibility with standard tooling.

{
  "type": "WIA-DATA_QUALITY-Record",
  "version": "1.0",
  "id": "...",
  "assessment": {
    "dimensions": {
      "completeness": { "score": 97.5 },
      "accuracy": { "score": 99.2 }
    }
  },
  "extensions": {
    "x-industry-specific": {
      "regulatoryCompliance": 100,
      "sectorRequirements": ["HIPAA", "SOX"]
    },
    "x-organization": {
      "internalAuditScore": 95.5,
      "businessCriticality": "high"
    }
  }
}
