Chapter 3: Decision Logging Architecture

Designing Comprehensive Audit Capture Systems

Foundations of Decision Logging

Effective audit systems begin with comprehensive decision logging—capturing every critical piece of information needed to understand, verify, and analyze AI decisions. Poor logging design creates audit gaps, making compliance verification impossible and hiding potential problems until they cause harm.

The principle of 弘益人間 demands that we design logging systems serving all stakeholders: regulators need compliance evidence, affected individuals need explanations, data scientists need debugging information, and society needs transparency. Well-designed logging architecture satisfies all these needs efficiently.

Core Data Schema Design

A robust audit log schema captures essential decision attributes while remaining extensible for diverse use cases:

interface DecisionAuditLog {
  // Unique identification
  decision_id: string;
  correlation_id?: string;  // Links related decisions

  // Temporal information
  timestamp: string;  // ISO 8601 format
  timezone: string;
  processing_duration_ms: number;

  // System information
  system: {
    name: string;
    version: string;
    environment: "production" | "staging" | "development";
    deployment_id: string;
  };

  // Model information
  model: {
    name: string;
    version: string;
    type: "neural_network" | "decision_tree" | "ensemble" | "rule_based";
    training_date: string;
    training_dataset_id: string;
  };

  // Input data
  input: {
    raw_data: any;
    preprocessed_data: any;
    feature_vector?: number[];
    data_sources: string[];
  };

  // Decision output
  output: {
    decision: any;
    confidence: number;
    alternatives?: Array<{decision: any; confidence: number}>;
    flags: string[];  // "high_risk", "low_confidence", etc.
  };

  // Reasoning and explainability
  reasoning: {
    explanation: string;
    key_features: Array<{name: string; importance: number; value: any}>;
    decision_path?: string;  // For tree models
    attention_weights?: number[];  // For neural networks
    counterfactuals?: any[];
  };

  // Context
  context: {
    user_id?: string;
    session_id?: string;
    request_id?: string;
    geographic_location?: string;
    decision_type: string;
    business_impact: "low" | "medium" | "high" | "critical";
  };

  // Compliance metadata
  compliance: {
    data_subject_consent: boolean;
    purpose: string[];
    legal_basis: string;
    automated: boolean;
    human_review_required: boolean;
    human_reviewer_id?: string;
  };

  // Audit metadata
  audit: {
    log_version: string;
    integrity_hash: string;
    previous_hash?: string;  // For hash chaining
    signature?: string;  // Cryptographic signature
  };
}

What to Log: The Five Essential Categories

1. Input Data and Context

Capture all information that influenced the decision. This includes:

  • Raw input data: The original request or data before any processing
  • Preprocessed data: Data after cleaning, normalization, or transformation
  • Feature engineering: Derived features used by the model
  • External data sources: APIs, databases, or services consulted
  • Environmental context: Time, location, user session information

2. Decision Output and Alternatives

Document not just what was decided, but what could have been decided:

  • Primary decision: The actual output
  • Confidence scores: Model certainty in the decision
  • Alternative outcomes: Other plausible decisions with their probabilities
  • Decision flags: Markers for unusual or notable characteristics
// Example: Credit decision with alternatives
{
  "output": {
    "decision": "APPROVED",
    "credit_limit": 50000,
    "confidence": 0.87,
    "alternatives": [
      {"decision": "APPROVED", "credit_limit": 40000, "confidence": 0.92},
      {"decision": "MANUAL_REVIEW", "confidence": 0.11},
      {"decision": "DENIED", "confidence": 0.02}
    ],
    "flags": ["near_threshold", "income_verification_recommended"]
  }
}

3. Reasoning and Explainability

Provide interpretable explanations for the decision:

  • Natural language explanation: Human-readable reasoning
  • Feature importance: Which inputs most influenced the decision
  • Decision path: Logical flow from input to output
  • Counterfactuals: What changes would alter the decision
// Feature importance example
{
  "reasoning": {
    "explanation": "Application approved based on strong credit history and stable income.",
    "key_features": [
      {"name": "credit_score", "importance": 0.42, "value": 750},
      {"name": "debt_to_income_ratio", "importance": 0.28, "value": 0.25},
      {"name": "employment_years", "importance": 0.18, "value": 8},
      {"name": "account_age_years", "importance": 0.12, "value": 12}
    ],
    "counterfactuals": [
      "If credit_score were below 650, decision would be MANUAL_REVIEW",
      "If debt_to_income_ratio exceeded 0.43, decision would be DENIED"
    ]
  }
}

4. Model and System Metadata

Track which version of which model made the decision:

  • Model identification: Name, version, type
  • Training provenance: When and on what data the model was trained
  • System environment: Production, staging, or development
  • Performance characteristics: Processing time, resource usage

5. Compliance and Governance Information

Document compliance with regulatory requirements:

  • Consent and authorization: Was proper consent obtained?
  • Purpose limitation: What purpose justifies this processing?
  • Human oversight: Was human review performed or required?
  • Impact assessment: Risk level and business impact classification

Synchronous vs. Asynchronous Logging

Logging strategy must balance completeness with performance:

Synchronous Logging

Decision logging occurs in the request path before returning results:

async function makeDecision(input: any): Promise {
  const decision = await model.predict(input);

  // Synchronous logging - blocks until logged
  await auditLogger.log({
    input,
    decision,
    timestamp: new Date().toISOString()
  });

  return decision;
}

Advantages: Guaranteed logging, no lost decisions, immediate availability

Disadvantages: Adds latency, can become bottleneck, failure impacts decision delivery

Asynchronous Logging

Decision logging happens in background after returning results:

async function makeDecision(input: any): Promise {
  const decision = await model.predict(input);

  // Asynchronous logging - doesn't block
  auditLogger.logAsync({
    input,
    decision,
    timestamp: new Date().toISOString()
  }).catch(err => {
    // Handle logging failures separately
    errorReporter.report(err);
  });

  return decision;
}

Advantages: Minimal latency impact, better throughput, resilient to logging failures

Disadvantages: Potential log loss on system crash, delayed availability

Hybrid Approach

Many systems use a hybrid strategy:

  • Critical metadata logged synchronously (lightweight)
  • Detailed context and reasoning logged asynchronously
  • In-memory buffer provides durability guarantee
  • Batch processing optimizes throughput
class HybridAuditLogger {
  private buffer: DecisionAuditLog[] = [];
  private flushInterval = 5000; // 5 seconds

  async logDecision(decision: Decision): Promise {
    // Synchronous: minimal critical data
    const quickLog = {
      decision_id: generateId(),
      timestamp: new Date().toISOString(),
      decision: decision.outcome,
      confidence: decision.confidence
    };

    await this.persistCritical(quickLog);

    // Asynchronous: detailed data
    const fullLog = this.buildFullLog(decision);
    this.buffer.push(fullLog);

    if (this.buffer.length >= 100) {
      this.flush();  // Batch insert
    }
  }

  private async flush(): Promise {
    const logs = this.buffer.splice(0);
    await this.persistBatch(logs);
  }
}

Privacy-Preserving Logging

Audit logs often contain sensitive personal data. Privacy-preserving techniques balance auditability with data protection:

Pseudonymization

Replace direct identifiers with pseudonyms:

function pseudonymize(log: DecisionAuditLog): DecisionAuditLog {
  return {
    ...log,
    context: {
      ...log.context,
      user_id: hashWithSalt(log.context.user_id, SECRET_SALT)
    },
    input: {
      ...log.input,
      raw_data: pseudonymizeFields(log.input.raw_data, ['name', 'email', 'phone'])
    }
  };
}

Encryption at Rest

Encrypt sensitive fields before storage:

async function encryptSensitiveFields(log: DecisionAuditLog): Promise {
  const encryptedInput = await encrypt(
    JSON.stringify(log.input),
    ENCRYPTION_KEY
  );

  return {
    ...log,
    input: {
      encrypted: true,
      data: encryptedInput
    }
  };
}

Differential Privacy

Add carefully calibrated noise to protect individual privacy while maintaining statistical utility:

function addDifferentialPrivacy(value: number, epsilon: number): number {
  const sensitivity = 1.0;
  const scale = sensitivity / epsilon;
  const noise = laplacian(0, scale);
  return value + noise;
}

Structured vs. Unstructured Logging

Structured Logging (Recommended)

Use consistent schema with typed fields:

{
  "decision_id": "DEC-2024-001247",
  "timestamp": "2024-01-15T14:32:18.123Z",
  "model": {"name": "credit-v3", "version": "3.2.1"},
  "input": {"credit_score": 720},
  "output": {"decision": "APPROVED", "confidence": 0.92}
}

Benefits: Queryable, parseable, indexable, type-safe

Unstructured Logging (Avoid)

"2024-01-15 14:32:18 - Credit decision: APPROVED for user with score 720 (confidence: 92%)"

Limitations: Difficult to query, requires parsing, error-prone, limited analysis capability

Performance Optimization Strategies

Batching

Group multiple log entries and write together:

class BatchedLogger {
  private batchSize = 100;
  private batchTimeoutMs = 5000;

  async logBatch(logs: DecisionAuditLog[]): Promise {
    // Single bulk insert instead of N individual inserts
    await database.bulkInsert('audit_logs', logs);
  }
}

Compression

Compress log data before storage:

import { gzip } from 'zlib';

async function compressLog(log: DecisionAuditLog): Promise {
  const json = JSON.stringify(log);
  return gzip(json);
}

Sampling for High-Volume Systems

For extremely high-volume systems, intelligent sampling reduces load:

function shouldLog(decision: Decision): boolean {
  // Always log high-risk decisions
  if (decision.risk_level === 'high') return true;

  // Always log decisions near confidence threshold
  if (decision.confidence < 0.7 || decision.confidence > 0.95) return true;

  // Sample 10% of routine decisions
  if (Math.random() < 0.1) return true;

  return false;
}
⚠️ Warning: Sampling must comply with regulatory requirements. Some regulations mandate logging ALL decisions affecting individuals. Always check applicable regulations before implementing sampling.

Multi-Model Pipeline Logging

Complex AI systems often chain multiple models. Audit logging must capture the entire pipeline:

class PipelineLogger {
  async logPipeline(request: any): Promise {
    const pipelineId = generateId();

    // Step 1: Preprocessing
    const preprocessed = await preprocessor.process(request);
    await this.logStep({
      pipeline_id: pipelineId,
      step: 'preprocessing',
      input: request,
      output: preprocessed
    });

    // Step 2: Risk scoring
    const riskScore = await riskModel.predict(preprocessed);
    await this.logStep({
      pipeline_id: pipelineId,
      step: 'risk_scoring',
      input: preprocessed,
      output: riskScore
    });

    // Step 3: Final decision
    const decision = await decisionModel.predict({
      ...preprocessed,
      risk_score: riskScore
    });
    await this.logStep({
      pipeline_id: pipelineId,
      step: 'decision',
      input: { preprocessed, riskScore },
      output: decision
    });
  }
}

Chapter Summary

  • Comprehensive decision logging captures inputs, outputs, reasoning, metadata, and compliance information
  • Core schema design should include unique identification, temporal data, system/model info, decision details, and audit metadata
  • Five essential logging categories: input data, decision output, reasoning, model metadata, and compliance information
  • Synchronous logging guarantees completeness but adds latency; asynchronous logging optimizes performance but risks log loss
  • Hybrid approaches combine critical synchronous logging with detailed asynchronous logging
  • Privacy-preserving techniques include pseudonymization, encryption, and differential privacy
  • Structured logging with consistent schemas enables efficient querying and analysis
  • Performance optimization strategies include batching, compression, and intelligent sampling
  • Multi-model pipelines require comprehensive logging across all stages with shared correlation IDs
