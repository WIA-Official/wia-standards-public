Production Deployment Patterns

Real deployments rarely match the textbook architecture. This chapter walks four patterns observed across regulated cloud deployments and discusses where each pattern thrives and where each one breaks down.

6.1 Pattern A: Single-cloud confidential zone

The simplest pattern: one cloud provider, one confidential zone, one trust anchor. The cloud's first-party verifier handles all attestation; the cloud's KMS handles all sealing. This pattern minimises operational complexity but couples the tenant tightly to the cloud — any sealing-key rotation that crosses cloud boundaries will require a re-architecture.

Best for: regulated workloads with a single cloud provider mandate, prototypes, and small teams.

6.2 Pattern B: Multi-cloud with per-cloud sealing

The workload runs in multiple clouds; sealed data is sealed per cloud and never crosses cloud boundaries. A higher-level orchestrator routes data to the cloud where the relevant sealed key resides. Cross-cloud disaster recovery becomes a workload- migration event (Phase 3 §4.3) rather than a key transfer.

Best for: regulated workloads where data residency is strict and clouds are independent failure domains.

6.3 Pattern C: Federated verifier with single cloud

One cloud provider, but the workload's policy engine accepts verdicts from both the cloud's first-party verifier and an independent third-party verifier. Unanimous agreement is required for high-trust transactions; the cloud first-party verdict alone is enough for operational telemetry.

Best for: workloads under audit by an external party (regulator, customer) where the cloud's first-party verdict is not sufficient evidence.

6.4 Pattern D: On-premises with HSM-backed sealing

The workload runs on-premises on dedicated TEE-equipped hardware. Sealing keys are wrapped by an HSM under the tenant's control. The verifier is also on-premises; trust anchors are refreshed manually from a documented vendor endpoint. This pattern maximises tenant control at the cost of operational overhead.

Best for: defence, intelligence, and the most sensitive financial workloads where cloud trust assumptions are unacceptable.

6.5 Pattern selection guidance

The standard does not endorse a single pattern. The selection criteria are: (a) regulatory residency, (b) audit requirements, (c) operational team size, (d) recovery time objectives. A workload's chosen pattern is documented in the deployment manifest envelope so that downstream consumers (tenant security team, auditor, regulator) can verify the workload is operating under the expected pattern.

Implementer notes

Three operational concerns recur across TEE implementations and are baked into the standard. First, attestation collateral freshness — verifiers MUST refresh ahead of expiry and emit warning envelopes when seven days from expiry, because operations teams routinely miss month-out collateral expiry warnings buried in cloud-provider dashboards. Second, encrypted-memory budgeting — EPC and TDX private memory are finite and the runtime MUST refuse over-budget allocations rather than silently swap to host memory, which would defeat the confidentiality guarantee that motivates the entire deployment. Third, side-channel mitigation reporting — the runtime MUST report applied mitigations as part of the launch attestation so a downstream verifier can refuse pods running on unmitigated hardware, even if the operations team forgot to patch.

Worked example

Consider a regulated bank running confidential workloads across two clouds. The bank requires per-tenant nodes (no co-residency with other tenants), HSM-backed key wrapping, and federation of verifiers across both clouds plus an independent third-party verifier. Without this standard, the bank reimplements attestation glue per cloud, maintains three separate verifier libraries, and accepts that a workload migration between clouds is a multi-week project. With this standard, the bank publishes one capacity envelope per cloud, registers three verifiers in the federation envelope, and migration becomes a single signed event with audit-grade evidence on both sides.
