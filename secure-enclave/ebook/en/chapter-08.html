Chapter 8 · WIA Secure Enclave v1.0.0

Roadmap, Conformance, Closing

This closing chapter sets out the conformance regime, the forward roadmap, and the governance model under which the standard evolves.

8.1 Conformance levels

The standard publishes three conformance levels:

  • Minimal — Phase 1 envelopes, single TEE family, single cloud. Suitable for prototypes and small-team deployments.
  • Core — Phase 1 + 2 + 3, multiple TEE families supported, federated verifier. Suitable for production cloud deployments.
  • Full — Phase 1 + 2 + 3 + 4, post-quantum hybrid signatures, formal-verification metadata, HSM-backed sealing. Suitable for regulated workloads under external audit.

8.2 Conformance test suite

A black-box conformance suite at https://github.com/WIA-Official/wia-secure-enclave-conformance walks every Phase 1 envelope, Phase 2 confidential-container launch, Phase 3 cross-platform attestation flow, and Phase 4 post-quantum hybrid verification. Hosts publishing conformance_level=Full SHOULD additionally pass the suite's HSM-backed-sealing extension tests.

8.3 Reference container

The wia/secure-enclave-host:1.0.0 container implements every Phase 2 endpoint with mock TEE evidence so integrators can exercise their bridge against it before going to production. The container ships with mock SGX, TDX, SEV-SNP, and Nitro evidence generators.

8.5 Governance

The standard is maintained by the WIA Standards Committee. Change proposals follow the WIA RFC process: anyone may submit a proposal; the Committee reviews quarterly; accepted proposals enter an open comment period before merging into a minor-version release. Breaking changes require a two-thirds Committee vote plus a 12-month deprecation window per IETF RFC 8594 and RFC 9745.

8.6 Closing

Trusted Execution Environments are no longer a research curiosity. They are load-bearing for regulated cloud, confidential AI, and identity attestation across every major public cloud and a growing set of on-premises deployments. The dominant pattern today is per-vendor silos that prevent a unified operational view. WIA-SEC-013 exists to close that gap with an open, federation-aware, audit-logged standard that lets operators, tenants, and auditors share evidence on terms each can verify.

Implementer notes

Three operational concerns recur across TEE implementations and are baked into the standard. First, attestation collateral freshness — verifiers MUST refresh ahead of expiry and emit warning envelopes when seven days from expiry, because operations teams routinely miss month-out collateral expiry warnings buried in cloud-provider dashboards. Second, encrypted-memory budgeting — EPC and TDX private memory are finite and the runtime MUST refuse over-budget allocations rather than silently swap to host memory, which would defeat the confidentiality guarantee that motivates the entire deployment. Third, side-channel mitigation reporting — the runtime MUST report applied mitigations as part of the launch attestation so a downstream verifier can refuse pods running on unmitigated hardware, even if the operations team forgot to patch.

Worked example

Consider a regulated bank running confidential workloads across two clouds. The bank requires per-tenant nodes (no co-residency with other tenants), HSM-backed key wrapping, and federation of verifiers across both clouds plus an independent third-party verifier. Without this standard, the bank reimplements attestation glue per cloud, maintains three separate verifier libraries, and accepts that a workload migration between clouds is a multi-week project. With this standard, the bank publishes one capacity envelope per cloud, registers three verifiers in the federation envelope, and migration becomes a single signed event with audit-grade evidence on both sides.
