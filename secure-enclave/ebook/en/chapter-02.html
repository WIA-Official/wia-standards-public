Chapter 2 · WIA Secure Enclave v1.0.0

Phase 1 Envelopes — Identity, Attestation, Sealing

Phase 1 specifies the canonical JSON envelopes that downstream phases build on: enclave identity, attestation request, attestation quote, sealed-data record, and key-derivation parameters. Every envelope is signed with Ed25519 (IETF RFC 8032) over the canonical JSON form (RFC 8785 JCS).

2.1 Enclave identity envelope

The enclave identity is derived from the platform measurement — MRENCLAVE on SGX, MRTD on TDX, the launch measurement on SEV-SNP — combined with the tenant identifier. The envelope is re-issued at every measured boot.

{
  "wia_secure_enclave_version": "1.0.0",
  "type": "enclave_identity",
  "enclave_id": "encl_01HXY...",
  "tenant_id": "did:wia:tenant:...",
  "tee_family": "sgx" | "tdx" | "sev-snp" | "trustzone" | "nitro",
  "platform_measurement": "0x...",
  "signer_measurement": "0x...",
  "issued_at": "RFC 3339",
  "signature": { "alg": "Ed25519", "value": "..." }
}

2.2 Attestation request envelope

A relying party requests attestation by sending a nonce and the expected enclave identity. The challenge nonce defends against replay; the expected identity lets the relying party fail fast if the wrong enclave answers.

2.3 Attestation quote envelope

The quote envelope wraps the native vendor evidence. The native blob is opaque to the envelope schema; the envelope provides enough metadata for a verifier to dispatch.

2.4 Sealed-data record

Sealed data is encrypted with a key derived from the enclave identity. The record carries the wrapped key, the AEAD ciphertext, the AAD that binds the ciphertext to a context, and a sealing policy that names the identities allowed to unseal.

{
  "type": "sealed_data",
  "record_id": "seal_01HXY...",
  "wrapped_key": "",
  "ciphertext": "",
  "aad": "",
  "sealing_policy": {
    "allowed_identities": ["did:wia:enclave:..."],
    "policy_kind": "exact" | "mrsigner" | "tenant"
  }
}

2.5 Key-derivation parameters

The standard mandates HKDF-SHA-384 (FIPS 180-4) with a domain-separation context that includes the tenant ID, the enclave identity, and the sealing-policy kind. The derivation produces a 256-bit key suitable for AES-256-GCM (NIST SP 800-38D).

Implementer notes

Three operational concerns recur across TEE implementations and are baked into the standard. First, attestation collateral freshness — verifiers MUST refresh ahead of expiry and emit warning envelopes when seven days from expiry, because operations teams routinely miss month-out collateral expiry warnings buried in cloud-provider dashboards. Second, encrypted-memory budgeting — EPC and TDX private memory are finite and the runtime MUST refuse over-budget allocations rather than silently swap to host memory, which would defeat the confidentiality guarantee that motivates the entire deployment. Third, side-channel mitigation reporting — the runtime MUST report applied mitigations as part of the launch attestation so a downstream verifier can refuse pods running on unmitigated hardware, even if the operations team forgot to patch.

Worked example

Consider a regulated bank running confidential workloads across two clouds. The bank requires per-tenant nodes (no co-residency with other tenants), HSM-backed key wrapping, and federation of verifiers across both clouds plus an independent third-party verifier. Without this standard, the bank reimplements attestation glue per cloud, maintains three separate verifier libraries, and accepts that a workload migration between clouds is a multi-week project. With this standard, the bank publishes one capacity envelope per cloud, registers three verifiers in the federation envelope, and migration becomes a single signed event with audit-grade evidence on both sides.
