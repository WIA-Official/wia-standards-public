Chapter 1 · WIA Secure Enclave v1.0.0

Why a Secure-Enclave Standard

Trusted Execution Environments (TEEs) have become a load-bearing primitive for confidential cloud workloads, regulated machine learning, and identity attestation. Yet every silicon vendor ships a slightly different model — Intel SGX and TDX, AMD SEV-SNP, ARM TrustZone with Realm Management Extensions, AWS Nitro Enclaves, Apple Secure Enclave — and every cloud provider wraps these in its own attestation service. The result is a fragmented landscape where a workload that runs perfectly on AWS Nitro must be ported, re-attested, and re-audited to run on Azure Confidential Computing or GCP Confidential Space.

1.1 The portability problem

The portability problem has three concrete symptoms. First, attestation evidence comes in incompatible blobs: an SGX Quote, a TDX Quote, a SEV-SNP Attestation Report, a Nitro COSE_Sign1 document. Second, sealing primitives bind data to platform identities that do not survive cross-cloud migration. Third, operational tooling — fleet attestation dashboards, drift detection, collateral refresh — has to be re-implemented per vendor.

This standard does not replace the native attestation evidence. It defines a vendor-neutral envelope that carries the native blob plus enough metadata for a verifier to dispatch correctly, and a unified verdict envelope that downstream policy engines can consume without parsing platform-specific formats.

1.2 Threat model

The standard adopts the classical TEE threat model: the adversary controls the host operating system, hypervisor, and physical machine, but cannot subvert the silicon root of trust. The adversary may attempt replay attacks against attestation quotes, side-channel attacks against in-enclave secrets, and collateral-staleness attacks against verifiers. The standard specifies replay defence (96-bit nonce + 300-second skew window), constant-time signature verification, and mandatory collateral refresh cadence.

1.3 What this book covers

The remaining seven chapters walk through each Phase of the standard: Phase 1 envelopes (Chapter 2), Phase 2 confidential containers and ML inference (Chapter 3), Phase 3 cross-platform attestation (Chapter 4), Phase 4 post-quantum readiness and operational hardening (Chapter 5). Chapters 6–8 cover production deployment patterns, incident response, and forward roadmap.

1.4 Audience

This book is written for security engineers building or operating TEE-backed workloads, platform engineers integrating TEE primitives into Kubernetes-class orchestrators, and auditors verifying the integrity of confidential computing pipelines. A reader is assumed to be comfortable with public-key cryptography and the basics of x86 / ARM virtualisation.

Implementer notes

Three operational concerns recur across TEE implementations and are baked into the standard. First, attestation collateral freshness — verifiers MUST refresh ahead of expiry and emit warning envelopes when seven days from expiry, because operations teams routinely miss month-out collateral expiry warnings buried in cloud-provider dashboards. Second, encrypted-memory budgeting — EPC and TDX private memory are finite and the runtime MUST refuse over-budget allocations rather than silently swap to host memory, which would defeat the confidentiality guarantee that motivates the entire deployment. Third, side-channel mitigation reporting — the runtime MUST report applied mitigations as part of the launch attestation so a downstream verifier can refuse pods running on unmitigated hardware, even if the operations team forgot to patch.

Worked example

Consider a regulated bank running confidential workloads across two clouds. The bank requires per-tenant nodes (no co-residency with other tenants), HSM-backed key wrapping, and federation of verifiers across both clouds plus an independent third-party verifier. Without this standard, the bank reimplements attestation glue per cloud, maintains three separate verifier libraries, and accepts that a workload migration between clouds is a multi-week project. With this standard, the bank publishes one capacity envelope per cloud, registers three verifiers in the federation envelope, and migration becomes a single signed event with audit-grade evidence on both sides.
