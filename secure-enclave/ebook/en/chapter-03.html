Phase 2 — Confidential Containers and ML Inference

Phase 2 extends Phase 1 to long-lived workloads. The two canonical Phase 2 deployments are confidential containers (Kata Containers in TDX mode, Confidential Containers, AMD SEV-SNP CVMs) and ML inference (running a quantised model inside an enclave so both the model weights and the inference inputs are confidential).

3.1 Confidential containers

A confidential container deployment runs every container image inside a TEE-backed pod. The pod's launch measurement is verified against a signed image-digest envelope before the container starts; unsigned or unverified images are refused with a cc_image_refused event.

The five operational requirements specified in Phase 2 §2.4 are: image attestation, sealed secret injection, encrypted-memory budgeting, sealed crash dumps, and side-channel mitigation reporting. Each requirement maps to a specific event class so a fleet's confidential-computing health is visible at the dashboard level.

3.2 ML inference inside an enclave

The ML-inference profile mandates quantised model weights (INT8 or INT4) so the model fits in encrypted memory, streaming inputs in encrypted chunks rather than wholesale loads, and output redaction for inference outputs that risk leaking input content (gradients, attention maps).

The threat model is two-sided: the model owner trusts the enclave to compute over inputs without revealing them, and the input owner trusts the enclave to compute over the model without revealing it. Both trust assumptions rest on the attestation chain plus a published model card describing the quantisation delta vs. FP32.

3.3 Memory accounting

EPC (SGX) and private memory (SEV-SNP, TDX) are finite. The standard requires the runtime to track per-tenant memory consumption and to refuse over-budget allocations rather than silently swap to host memory (which would defeat confidentiality). Capacity envelopes (Phase 2 §2.8) declare a tenant's reserved budget at onboarding so capacity is deterministic rather than discovered at workload-launch time.

3.4 Multi-tenant isolation

A confidential-computing node typically runs workloads from multiple tenants. The standard requires either per-tenant dedicated nodes (simplest) or per-tenant TEE instances with side-channel mitigations applied (more efficient but harder to audit). The chosen mode is declared in the node's attestation report so a tenant's policy engine can refuse mixed-tenant nodes when its policy demands isolation.

Implementer notes

Three operational concerns recur across TEE implementations and are baked into the standard. First, attestation collateral freshness — verifiers MUST refresh ahead of expiry and emit warning envelopes when seven days from expiry, because operations teams routinely miss month-out collateral expiry warnings buried in cloud-provider dashboards. Second, encrypted-memory budgeting — EPC and TDX private memory are finite and the runtime MUST refuse over-budget allocations rather than silently swap to host memory, which would defeat the confidentiality guarantee that motivates the entire deployment. Third, side-channel mitigation reporting — the runtime MUST report applied mitigations as part of the launch attestation so a downstream verifier can refuse pods running on unmitigated hardware, even if the operations team forgot to patch.

Worked example

Consider a regulated bank running confidential workloads across two clouds. The bank requires per-tenant nodes (no co-residency with other tenants), HSM-backed key wrapping, and federation of verifiers across both clouds plus an independent third-party verifier. Without this standard, the bank reimplements attestation glue per cloud, maintains three separate verifier libraries, and accepts that a workload migration between clouds is a multi-week project. With this standard, the bank publishes one capacity envelope per cloud, registers three verifiers in the federation envelope, and migration becomes a single signed event with audit-grade evidence on both sides.
