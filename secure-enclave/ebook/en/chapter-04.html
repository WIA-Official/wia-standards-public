Phase 3 — Cross-Platform Attestation

Phase 3 lets workloads cross trust boundaries — between cloud providers, between geographies, between first-party and third-party verifiers. The cross-platform attestation envelope (Phase 3 §3.4) wraps native evidence and lets a verifier dispatch to the correct family-specific verification path without parsing the native blob.

4.1 The five TEE families

The standard enumerates five TEE families and their native evidence formats: Intel SGX (Quote signed by a Quoting Enclave), Intel TDX (Quote signed via DCAP), AMD SEV-SNP (Attestation Report signed by the AMD Secure Processor), ARM TrustZone (Realm attestation), and AWS Nitro (COSE_Sign1-wrapped attestation document). Each has its own trust anchor maintained by the platform vendor.

4.2 Trust anchor lifecycle

Every conformant verifier fetches trust anchors over an authenticated channel from the vendor endpoint, refreshes on a documented cadence (≤ 30 days), caches anchors signed by the vendor for the cache lifetime, and refuses attestations whose chain root is not present in the active anchor set.

The most common failure mode here is stale collateral. Verifiers MUST refresh ahead of expiry and emit a warning envelope when ≤ 7 days from expiry so an operator can intervene before live workloads start failing attestation.

4.3 Cross-cloud workload migration

A workload migrating between cloud providers re-attests on arrival. The new attestation is verified against the new provider's trust anchors; the workload's sealing key is re-derived from the new attestation identity and a tenant-supplied key-wrapping key. Sealed data from the prior provider is unsealed using the prior identity's key, re-sealed with the new identity's key, and the prior identity's key is destroyed.

4.4 Multi-vendor verification policies

Production policy engines accept multi-vendor evidence and emit a unified verdict. The verdict envelope carries a policy_id and policy_version so a verdict consumer can verify which policy was applied and whether the policy itself has been updated. Policy tightening (e.g. raising the minimum SGX SVN level) deploys via a deprecation window: the verifier emits warnings before refusing attestations that satisfy only the older policy.

4.5 Federation of verifiers

High-trust workloads submit one attestation to multiple verifiers — typically the cloud's first-party verifier plus an independent third-party verifier — and require unanimous agreement. The federation envelope (Phase 3 §3.9) lets the workload submit once and receive a bundle of signed verdicts.

Implementer notes

Three operational concerns recur across TEE implementations and are baked into the standard. First, attestation collateral freshness — verifiers MUST refresh ahead of expiry and emit warning envelopes when seven days from expiry, because operations teams routinely miss month-out collateral expiry warnings buried in cloud-provider dashboards. Second, encrypted-memory budgeting — EPC and TDX private memory are finite and the runtime MUST refuse over-budget allocations rather than silently swap to host memory, which would defeat the confidentiality guarantee that motivates the entire deployment. Third, side-channel mitigation reporting — the runtime MUST report applied mitigations as part of the launch attestation so a downstream verifier can refuse pods running on unmitigated hardware, even if the operations team forgot to patch.

Worked example

Consider a regulated bank running confidential workloads across two clouds. The bank requires per-tenant nodes (no co-residency with other tenants), HSM-backed key wrapping, and federation of verifiers across both clouds plus an independent third-party verifier. Without this standard, the bank reimplements attestation glue per cloud, maintains three separate verifier libraries, and accepts that a workload migration between clouds is a multi-week project. With this standard, the bank publishes one capacity envelope per cloud, registers three verifiers in the federation envelope, and migration becomes a single signed event with audit-grade evidence on both sides.
