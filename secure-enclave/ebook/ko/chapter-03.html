페이즈 2 — 기밀 컨테이너와 ML 추론

페이즈 2는 페이즈 1을 장기 워크로드로 확장합니다. 두 가지 표준 페이즈 2 배포는 기밀 컨테이너(TDX 모드의 Kata Containers, Confidential Containers, AMD SEV-SNP CVMs)와 ML 추론(엔클레이브 내부에서 양자화된 모델을 실행하여 모델 가중치와 추론 입력 모두 기밀로 유지)입니다.

3.1 기밀 컨테이너

기밀 컨테이너 배포는 모든 컨테이너 이미지를 TEE 기반 파드 내부에서 실행합니다. 파드의 launch measurement는 컨테이너 시작 전에 서명된 이미지 다이제스트 봉투에 대해 검증됩니다. 서명되지 않거나 검증되지 않은 이미지는 cc_image_refused 이벤트와 함께 거부됩니다.

페이즈 2 §2.4에 명시된 다섯 가지 운영 요건은 — 이미지 어테스테이션, 봉인된 비밀 주입, 암호화 메모리 예산, 봉인된 크래시 덤프, 사이드채널 완화 보고. 각 요건은 특정 이벤트 클래스에 매핑되어 플릿의 기밀 컴퓨팅 건강 상태가 대시보드 수준에서 보입니다.

3.2 엔클레이브 내부 ML 추론

ML 추론 프로파일은 양자화된 모델 가중치(INT8 또는 INT4)로 모델이 암호화 메모리에 들어가도록 강제하며, 한꺼번에 로드하지 않고 암호화된 청크로 입력을 스트리밍하며, 입력 콘텐츠 누출 위험이 있는 추론 출력(그래디언트, 어텐션 맵)에 대한 출력 검열을 의무화합니다.

위협 모델은 양면입니다 — 모델 소유자는 엔클레이브가 입력을 드러내지 않고 입력에 대해 계산함을 신뢰하고, 입력 소유자는 엔클레이브가 모델을 드러내지 않고 모델에 대해 계산함을 신뢰합니다. 두 신뢰 가정 모두 어테스테이션 체인과 FP32 대비 양자화 차이를 설명하는 게시된 모델 카드에 의존합니다.

3.3 메모리 회계

EPC(SGX)와 사설 메모리(SEV-SNP, TDX)는 유한합니다. 표준은 런타임이 테넌트별 메모리 소비를 추적하고, 호스트 메모리로 조용히 스왑하지 않고 예산 초과 할당을 거부할 것을 요구합니다(스왑은 기밀성을 무력화합니다). 용량 봉투(페이즈 2 §2.8)는 온보딩 시 테넌트의 예약 예산을 선언하여 워크로드 실행 시점에 발견하는 대신 결정론적으로 용량을 확보합니다.

3.4 멀티테넌트 격리

기밀 컴퓨팅 노드는 일반적으로 여러 테넌트의 워크로드를 실행합니다. 표준은 테넌트별 전용 노드(가장 단순) 또는 사이드채널 완화가 적용된 테넌트별 TEE 인스턴스(더 효율적이지만 감사 어려움)를 요구합니다. 선택된 모드는 노드의 어테스테이션 보고서에 선언되어 테넌트의 정책 엔진이 격리를 요구하는 경우 혼합 테넌트 노드를 거부할 수 있습니다.

장 간 연결

이 장은 WIA 보안 엔클레이브 8장 시리즈에 속합니다. 1장은 맥락을 설정합니다. 2-5장은 네 페이즈를 살핍니다. 6장은 프로덕션 배포 패턴을 다룹니다. 7장은 인시던트 대응과 포렌식을 다룹니다. 8장은 적합성 체제, 향후 로드맵, 거버넌스로 마무리합니다.

  • 적합성 테스트 스위트 — https://github.com/WIA-Official/wia-secure-enclave-conformance는 최소, 코어, 풀 수준에서 적합성을 주장하기 위해 구현이 통과해야 하는 것의 단일 진실 소스입니다.
  • 참조 컨테이너 — wia/secure-enclave-host:1.0.0은 모든 페이즈 2 엔드포인트를 모의 TEE 증거로 구현하여 통합자가 프로덕션 하드웨어 이전에 자신의 브리지를 시험할 수 있게 합니다.
  • 컴패니언 CLI — cli/secure-enclave.sh는 샘플 봉투 생성기(validate, identity, attest, seal, verify, info 서브커맨드)를 출시하여 구현자가 JSON을 직접 작성하지 않고 적합한 페이로드를 만들 수 있게 합니다.

구현자 노트

세 가지 운영 관심사가 TEE 구현에서 반복적으로 나타나며 표준에 박혀 있습니다. 첫째, 어테스테이션 부속자료 신선도 — 검증자는 만료 전에 새로고침해야 하며 만료까지 7일이 남으면 경고 봉투를 발행해야 합니다. 운영 팀은 클라우드 제공자 대시보드에 묻힌 한 달 전 부속자료 만료 경고를 일상적으로 놓치기 때문입니다. 둘째, 암호화 메모리 예산 — EPC와 TDX 사설 메모리는 유한하며 런타임은 호스트 메모리로 조용히 스왑하지 않고 예산 초과 할당을 거부해야 합니다. 스왑은 전체 배포의 동기인 기밀성 보장을 무력화합니다. 셋째, 사이드채널 완화 보고 — 런타임은 적용된 완화를 시작 어테스테이션의 일부로 보고해야 운영 팀이 패치를 잊었더라도 다운스트림 검증자가 미완화 하드웨어에서 실행되는 파드를 거부할 수 있습니다.

워크 예제

두 클라우드에서 기밀 워크로드를 실행하는 규제 은행을 가정합니다. 은행은 테넌트별 노드(다른 테넌트와의 공동 거주 없음), HSM 기반 키 래핑, 두 클라우드 모두와 독립 3자 검증자에 걸친 검증자 연합을 요구합니다. 이 표준 없이는 은행이 클라우드별로 어테스테이션 글루를 재구현하고, 세 개의 별도 검증자 라이브러리를 유지하며, 클라우드 간 워크로드 마이그레이션이 다주짜리 프로젝트임을 받아들입니다. 이 표준으로 은행은 클라우드별 용량 봉투 하나를 게시하고, 연합 봉투에 세 검증자를 등록하며, 마이그레이션은 양쪽에 감사 등급 증거가 있는 단일 서명 이벤트가 됩니다.
