2장 · WIA 보안 엔클레이브 v1.0.0

페이즈 1 봉투 — 신원, 어테스테이션, 봉인

페이즈 1은 다운스트림 페이즈가 기반으로 삼는 표준 JSON 봉투를 명시합니다 — 엔클레이브 신원, 어테스테이션 요청, 어테스테이션 quote, 봉인 데이터 레코드, 키 도출 매개변수. 모든 봉투는 표준 JSON 형식(RFC 8785 JCS)에 대해 Ed25519(IETF RFC 8032)로 서명됩니다.

2.1 엔클레이브 신원 봉투

엔클레이브 신원은 플랫폼 측정값(SGX의 MRENCLAVE, TDX의 MRTD, SEV-SNP의 launch measurement)과 테넌트 식별자의 조합에서 도출됩니다. 봉투는 측정 부팅마다 재발급됩니다.

{
  "wia_secure_enclave_version": "1.0.0",
  "type": "enclave_identity",
  "enclave_id": "encl_01HXY...",
  "tenant_id": "did:wia:tenant:...",
  "tee_family": "sgx" | "tdx" | "sev-snp" | "trustzone" | "nitro",
  "platform_measurement": "0x...",
  "signer_measurement": "0x...",
  "issued_at": "RFC 3339",
  "signature": { "alg": "Ed25519", "value": "..." }
}

2.2 어테스테이션 요청 봉투

의존 당사자는 nonce와 예상 엔클레이브 신원을 보내 어테스테이션을 요청합니다. 챌린지 nonce는 재생을 방어하고, 예상 신원은 잘못된 엔클레이브가 응답할 경우 의존 당사자가 빠르게 실패할 수 있게 합니다.

2.3 어테스테이션 quote 봉투

quote 봉투는 네이티브 벤더 증거를 감쌉니다. 네이티브 블롭은 봉투 스키마에 불투명하며, 봉투는 검증자가 디스패치하기에 충분한 메타데이터를 제공합니다.

2.4 봉인 데이터 레코드

봉인 데이터는 엔클레이브 신원에서 도출된 키로 암호화됩니다. 레코드는 래핑된 키, AEAD 암호문, 암호문을 컨텍스트에 바인드하는 AAD, 봉인 해제를 허용하는 신원을 명시한 봉인 정책을 담습니다.

{
  "type": "sealed_data",
  "record_id": "seal_01HXY...",
  "wrapped_key": "",
  "ciphertext": "",
  "aad": "",
  "sealing_policy": {
    "allowed_identities": ["did:wia:enclave:..."],
    "policy_kind": "exact" | "mrsigner" | "tenant"
  }
}

2.5 키 도출 매개변수

표준은 테넌트 ID, 엔클레이브 신원, 봉인 정책 종류를 포함하는 도메인 분리 컨텍스트와 함께 HKDF-SHA-384(FIPS 180-4)를 명시합니다. 도출은 AES-256-GCM(NIST SP 800-38D)에 적합한 256-bit 키를 생성합니다.

장 간 연결

이 장은 WIA 보안 엔클레이브 8장 시리즈에 속합니다. 1장은 맥락을 설정합니다. 2-5장은 네 페이즈를 살핍니다. 6장은 프로덕션 배포 패턴을 다룹니다. 7장은 인시던트 대응과 포렌식을 다룹니다. 8장은 적합성 체제, 향후 로드맵, 거버넌스로 마무리합니다.

  • 적합성 테스트 스위트 — https://github.com/WIA-Official/wia-secure-enclave-conformance는 최소, 코어, 풀 수준에서 적합성을 주장하기 위해 구현이 통과해야 하는 것의 단일 진실 소스입니다.
  • 참조 컨테이너 — wia/secure-enclave-host:1.0.0은 모든 페이즈 2 엔드포인트를 모의 TEE 증거로 구현하여 통합자가 프로덕션 하드웨어 이전에 자신의 브리지를 시험할 수 있게 합니다.
  • 컴패니언 CLI — cli/secure-enclave.sh는 샘플 봉투 생성기(validate, identity, attest, seal, verify, info 서브커맨드)를 출시하여 구현자가 JSON을 직접 작성하지 않고 적합한 페이로드를 만들 수 있게 합니다.

구현자 노트

세 가지 운영 관심사가 TEE 구현에서 반복적으로 나타나며 표준에 박혀 있습니다. 첫째, 어테스테이션 부속자료 신선도 — 검증자는 만료 전에 새로고침해야 하며 만료까지 7일이 남으면 경고 봉투를 발행해야 합니다. 운영 팀은 클라우드 제공자 대시보드에 묻힌 한 달 전 부속자료 만료 경고를 일상적으로 놓치기 때문입니다. 둘째, 암호화 메모리 예산 — EPC와 TDX 사설 메모리는 유한하며 런타임은 호스트 메모리로 조용히 스왑하지 않고 예산 초과 할당을 거부해야 합니다. 스왑은 전체 배포의 동기인 기밀성 보장을 무력화합니다. 셋째, 사이드채널 완화 보고 — 런타임은 적용된 완화를 시작 어테스테이션의 일부로 보고해야 운영 팀이 패치를 잊었더라도 다운스트림 검증자가 미완화 하드웨어에서 실행되는 파드를 거부할 수 있습니다.

워크 예제

두 클라우드에서 기밀 워크로드를 실행하는 규제 은행을 가정합니다. 은행은 테넌트별 노드(다른 테넌트와의 공동 거주 없음), HSM 기반 키 래핑, 두 클라우드 모두와 독립 3자 검증자에 걸친 검증자 연합을 요구합니다. 이 표준 없이는 은행이 클라우드별로 어테스테이션 글루를 재구현하고, 세 개의 별도 검증자 라이브러리를 유지하며, 클라우드 간 워크로드 마이그레이션이 다주짜리 프로젝트임을 받아들입니다. 이 표준으로 은행은 클라우드별 용량 봉투 하나를 게시하고, 연합 봉투에 세 검증자를 등록하며, 마이그레이션은 양쪽에 감사 등급 증거가 있는 단일 서명 이벤트가 됩니다.
