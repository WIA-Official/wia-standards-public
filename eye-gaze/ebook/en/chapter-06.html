Chapter 6: Security & Privacy

6.1 Overview

Eye tracking data is highly personal and sensitive. This chapter covers security measures, privacy protections, and consent management for WIA-compliant systems.

Why Eye Data Is Sensitive

Eye tracking data can reveal:

  • Reading Patterns - What text you've read and comprehended
  • Attention & Interest - What captures your attention
  • Cognitive State - Mental workload, fatigue, stress levels
  • Emotional Responses - Arousal and emotional reactions via pupil changes
  • Biometric Identity - Eye movement patterns can identify individuals
  • Health Information - Certain eye behaviors correlate with medical conditions

6.2 Data Classification

Sensitivity Levels

Data Type Sensitivity Examples Protection Level
Gaze Coordinates High Screen positions, fixations Encryption, access control
Pupil Data Very High Pupil diameter, changes Encryption, consent required
Eye Images Critical Camera images of eyes Never stored, consent + encryption
Aggregated Stats Medium Average accuracy, session duration Anonymization
Device Metadata Low Device model, firmware version Standard protection

6.3 Consent Management

Informed Consent Requirements

Before collecting eye tracking data, applications must obtain explicit consent:

interface ConsentRequest {
  /** Application requesting consent */
  applicationId: string;
  applicationName: string;

  /** What data will be collected */
  dataTypes: DataType[];

  /** Purpose of data collection */
  purpose: string;

  /** How long data will be retained */
  retentionPeriod: string;

  /** Will data be shared with third parties */
  sharingPolicy: SharingPolicy;

  /** User rights (access, deletion, etc.) */
  userRights: string[];
}

type DataType =
  | 'gaze_coordinates'
  | 'pupil_diameter'
  | 'eye_images'
  | 'fixation_events'
  | 'screen_content';

interface SharingPolicy {
  shared: boolean;
  recipients?: string[];
  purpose?: string;
}

Granular Consent

Users should be able to consent to specific data types:

interface ConsentResponse {
  granted: boolean;

6.4 Data Encryption

Transport Encryption

All network communication must use TLS 1.3+:

// WebSocket with TLS
const ws = new WebSocket('wss://tracker.local:8443/stream', {
  rejectUnauthorized: true,
  minVersion: 'TLSv1.3'
});

// Verify server certificate
ws.on('upgrade', (response) => {
  const cert = response.socket.getPeerCertificate();
  if (!isValidCertificate(cert)) {
    ws.close();
    throw new Error('Invalid server certificate');
  }
});

At-Rest Encryption

Stored eye tracking data must be encrypted:

// AES-256-GCM encryption
interface EncryptedGazeData {
  version: '1.0';
  algorithm: 'AES-256-GCM';
  iv: string;           // Base64 initialization vector
  ciphertext: string;   // Base64 encrypted data
  authTag: string;      // Base64 authentication tag
  metadata?: {
    timestamp: number;
    recordCount: number;
  };
}

End-to-End Encryption

For multi-user scenarios (e.g., research data collection):

// Researcher's public key encrypts data
{
  "encryptedSymmetricKey": "...",  // RSA-OAEP encrypted AES key
  "encryptedData": "...",           // AES-GCM encrypted gaze data
  "recipientKeyId": "researcher-001"
}

// Only researcher with private key can decrypt

6.6 Data Minimization

Principles

  • Collect Only What's Needed - Don't request pupil data if only gaze coordinates are needed
  • Reduce Granularity - Aggregate or round data when high precision isn't necessary
  • Delete Early - Implement automatic deletion of temporary data
  • Anonymize - Remove identifying information when possible

Data Retention Policies

interface RetentionPolicy {
  /** Data category */
  dataType: string;

  /** How long to keep data */
  retentionPeriod: number; // milliseconds

  /** What happens after retention period */
  action: 'delete' | 'anonymize' | 'archive';

  /** Exceptions to retention period */
  exceptions?: {
    legalHold?: boolean;
    userRequested?: boolean;
  };
}

// Example policies
const policies = [
  {
    dataType: 'session_data',
    retentionPeriod: 0,           // Delete immediately on session end
    action: 'delete'
  },
  {
    dataType: 'calibration_profile',
    retentionPeriod: 90 * 86400000, // 90 days
    action: 'delete'
  },
  {
    dataType: 'research_data',
    retentionPeriod: 365 * 86400000, // 1 year
    action: 'anonymize'
  }
];

6.7 Privacy by Design

Privacy-Preserving Techniques

1. Differential Privacy

Add controlled noise to aggregate statistics:

function addDifferentialPrivacy(value, epsilon = 1.0) {
  // Laplace mechanism
  const scale = 1.0 / epsilon;
  const noise = laplaceSample(0, scale);
  return value + noise;
}

// Example: Average fixation duration
const trueAverage = 245.3; // ms
const privateAverage = addDifferentialPrivacy(trueAverage);
// Result: 247.1 ms (statistically similar, but not exact)

2. k-Anonymity

Ensure data cannot be linked to fewer than k individuals:

// Generalize gaze coordinates to regions
function anonymizeGazeData(gazePoints, k = 5) {
  // Group points into grid cells
  const gridSize = 0.1; // 10% of screen
  const cells = groupIntoGrid(gazePoints, gridSize);

  // Only return cells with k+ points
  return cells.filter(cell => cell.points.length >= k);
}

3. Homomorphic Encryption

Compute on encrypted data without decrypting:

// Research scenario: Compute average fixation duration
// without seeing individual data points
const encryptedDurations = fixations.map(f =>
  encrypt(f.duration, publicKey)
);

const encryptedSum = homomorphicSum(encryptedDurations);
const encryptedAvg = homomorphicDivide(
  encryptedSum,
  fixations.length
);

// Researcher can decrypt only the average, not individual durations

6.9 Regulatory Compliance

GDPR Compliance (EU)

GDPR Requirements

  • Right to Access - Users can view their data
  • Right to Rectification - Users can correct inaccurate data
  • Right to Erasure - "Right to be forgotten"
  • Right to Data Portability - Export data in machine-readable format
  • Right to Object - Refuse certain processing
  • Data Breach Notification - Report breaches within 72 hours

HIPAA Compliance (US Healthcare)

For medical AAC applications:

  • PHI Protection - Eye data linked to medical records is Protected Health Information
  • BAA Required - Business Associate Agreement with service providers
  • Encryption Mandatory - Both in transit and at rest
  • Access Logging - Detailed audit trails required
  • Breach Notification - Notify affected individuals

COPPA Compliance (Children's Privacy)

For applications used by children under 13:

  • Parental Consent - Verifiable parental consent required
  • Data Minimization - Collect only necessary data
  • No Profiling - Don't build behavioral profiles of children
  • Transparency - Clear, child-appropriate privacy notices

6.10 Security Best Practices

For Device Manufacturers

  • Implement secure boot and firmware signing
  • Regular security updates and patching
  • Hardware-based encryption keys (TPM/Secure Enclave)
  • No default passwords or credentials
  • Secure development lifecycle (SDL)

For Application Developers

  • Request minimal permissions needed
  • Implement consent management UI
  • Never log sensitive data
  • Use secure credential storage
  • Regular security audits and penetration testing

For Users

  • Review and understand consent requests
  • Only install trusted applications
  • Regularly review access logs
  • Revoke unused application permissions
  • Keep devices and software updated
