Chapter 4: Protocol Specification

4.1 Overview

The WIA Eye Gaze standard supports multiple communication protocols optimized for different use cases: WebSocket for network communication, IPC for local ultra-low latency, and binary formats for high-frequency data transmission.

4.2 WebSocket Protocol

Connection Establishment

// Client connects to WebSocket endpoint
const ws = new WebSocket('ws://localhost:8080/stream');

ws.onopen = () => {
  // Send handshake
  ws.send(JSON.stringify({
    type: 'handshake',
    version: '1.0.0',
    clientId: 'aac-app-001',
    capabilities: ['GAZE_POINT', 'EVENTS']
  }));
};

Server Handshake Response

{
  "type": "handshake_ack",
  "sessionId": "session_2025_001",
  "protocolVersion": "1.0.0",
  "serverCapabilities": [
    "GAZE_POINT",
    "BINOCULAR",
    "EVENTS",
    "BINARY_FORMAT"
  ]
}

Message Format

All WebSocket messages follow this envelope format:

{
  "type": "message_type",     // Message type identifier
  "timestamp": 1704067200000, // Server timestamp
  "sequenceId": 12345,        // Sequence number
  "data": { ... }             // Message payload
}

Message Types

Type Direction Description
handshake Client → Server Initial connection handshake
handshake_ack Server → Client Handshake acknowledgment
gaze_point Server → Client Single gaze data point
gaze_batch Server → Client Multiple gaze points
event Server → Client Gaze event notification
command Client → Server Device control command
heartbeat Bidirectional Keep-alive ping
error Server → Client Error notification

Heartbeat & Keep-Alive

Clients must send heartbeat every 5 seconds:

// Client heartbeat
setInterval(() => {
  ws.send(JSON.stringify({
    type: 'heartbeat',
    timestamp: Date.now()
  }));
}, 5000);

// Server responds
{
  "type": "heartbeat_ack",
  "timestamp": 1704067200000
}

Connection Timeout

Server disconnects clients missing 3 consecutive heartbeats (15 seconds).

4.3 Inter-Process Communication (IPC)

For local applications requiring ultra-low latency (<1ms), IPC provides direct shared memory access.

Platform-Specific Transports

Platform Transport Path/Name
Linux Unix Domain Socket /tmp/wia-eye-gaze.sock
macOS Unix Domain Socket /tmp/wia-eye-gaze.sock
Windows Named Pipe \\.\pipe\wia-eye-gaze

IPC Frame Structure

Frame Header (20 bytes):
┌────────────────────────────────────────────────┐
│ Magic (2)  │ Version(1) │ Type (1)  │ Flags(1)│
│ 0x57 0x49  │ 0x01       │           │         │
├────────────┴────────────┴───────────┴─────────┤
│ Reserved (1)          │ Payload Length (2)    │
├───────────────────────┴───────────────────────┤
│ Timestamp (8 bytes) - microseconds            │
├───────────────────────────────────────────────┤
│ Sequence Number (4 bytes)                     │
├───────────────────────────────────────────────┤
│ Payload (variable)                            │
└───────────────────────────────────────────────┘

IPC Message Types

Type Code Description
GAZE_POINT 0x01 Single gaze point (21 bytes)
GAZE_BATCH 0x02 Multiple gaze points
GAZE_EVENT 0x03 Event notification
CONTROL 0x04 Control command
HANDSHAKE 0x10 Connection handshake

Compact Binary GazePoint (21 bytes)

┌──────────────────────────────────────────────┐
│ TimestampOffset(2) │ X(4)    │ Y(4)    │ Conf(2) │
│ uint16 ms          │ float32 │ float32 │ uint16  │
├────────────────────┴─────────┴─────────┴─────────┤
│ Flags(1) │ LeftPupil(2) │ RightPupil(2)          │
├──────────┴──────────────┴────────────────────────┤
│ LeftOpen(1) │ RightOpen(1) │ FixationID(2)       │
└─────────────┴──────────────┴─────────────────────┘

Total: 21 bytes per sample

4.4 Binary Format Specification

When to Use Binary Format

  • High Sampling Rates: 120Hz and above
  • Bandwidth Constrained: Mobile or remote connections
  • Storage: Long-term data recording
  • Performance Critical: Real-time gaming or VR

Binary vs JSON Size

Format Size per Point 120Hz/second 1 hour
JSON (pretty) ~450 bytes 54 KB 194 MB
JSON (compact) ~180 bytes 22 KB 78 MB
MessagePack ~60 bytes 7 KB 26 MB
WIA Binary 21 bytes 2.5 KB 9 MB

4.5 Latency Requirements

End-to-End Latency Budget

Latency Breakdown for AAC

Eye → Tracker Processing:     3-10ms
Tracker → Computer Transfer:  1-3ms
API Processing:               1-2ms
Network/IPC:                  0.1-5ms
Application Processing:       5-10ms
Screen Rendering:            16ms (60Hz)
─────────────────────────────────────
Total AAC Latency:           26-46ms (acceptable)
Gaming Target:               <20ms total

Latency Optimization Techniques

  • Use IPC for local applications instead of WebSocket
  • Binary Format reduces serialization overhead
  • Direct Rendering skip intermediate buffers
  • Predictive Smoothing compensate for system latency

4.6 Data Quality & Error Handling

Quality Monitoring

{
  "type": "quality_report",
  "data": {
    "timestamp": 1704067200000,
    "trackingQuality": "good",
    "dataLossRate": 0.02,      // 2% loss
    "averageConfidence": 0.91,
    "validSampleRate": 0.98,   // 98% valid
    "recommendation": "continue"
  }
}

Quality States

State Confidence Valid Rate Action
Excellent > 0.9 > 95% Continue
Good 0.7 - 0.9 85-95% Continue
Fair 0.5 - 0.7 70-85% Monitor closely
Poor < 0.5 < 70% Recalibrate or adjust

Error Recovery

// Automatic reconnection
ws.onclose = (event) => {
  if (!event.wasClean) {
    console.log('Connection lost, reconnecting...');
    setTimeout(() => {
      reconnect();
    }, Math.min(1000 * Math.pow(2, reconnectAttempts), 30000));
  }
};

4.7 Synchronization & Timestamping

Timestamp Types

  • Device Timestamp: Hardware clock (microsecond precision)
  • System Timestamp: OS clock (millisecond precision)
  • Server Timestamp: API server clock
  • Client Timestamp: Application receive time

Clock Synchronization

// NTP-style clock sync
{
  "type": "time_sync",
  "clientSendTime": 1704067200000,
  "serverReceiveTime": 1704067200002,
  "serverSendTime": 1704067200003,
  "clientReceiveTime": 1704067200005
}

// Calculate offset
offset = ((serverReceiveTime - clientSendTime) +
          (serverSendTime - clientReceiveTime)) / 2
// Result: ~2ms offset

4.8 Bandwidth Optimization

Adaptive Sampling

Dynamically adjust sampling rate based on gaze activity:

{
  "type": "config_update",
  "adaptiveSampling": {
    "enabled": true,
    "fixationRate": 30,    // Hz during fixation
    "saccadeRate": 120,    // Hz during saccade
    "idleRate": 10         // Hz when no activity
  }
}

Delta Encoding

Send only changes from previous sample:

// Full sample
{ "x": 0.5234, "y": 0.3891, "confidence": 0.95 }

// Delta (only x changed)
{ "dx": 0.0012 }  // 75% size reduction

Compression Support

Algorithm Compression Latency Impact CPU Usage
None 1.0x 0ms Low
LZ4 2-3x +0.5ms Low
Zstandard 3-5x +1-2ms Medium
gzip 4-6x +3-5ms High

Compression Recommendation

Use LZ4 for real-time AAC/gaming (best latency/compression tradeoff). Use Zstandard for data logging/recording.

4.9 Security Considerations

TLS/SSL for WebSocket

// Secure WebSocket connection
const ws = new WebSocket('wss://eye-tracker.local:8443/stream');

// Certificate validation
ws.addEventListener('error', (event) => {
  if (event.message.includes('certificate')) {
    console.error('Invalid server certificate');
  }
});

IPC Security

  • File Permissions: Socket/pipe restricted to owner and group (0660)
  • Process Validation: Verify connecting process identity
  • Message Authentication: HMAC signatures for sensitive commands

Authentication

// Token-based authentication
{
  "type": "handshake",
  "clientId": "aac-app-001",
  "authToken": "eyJhbGciOiJIUzI1NiIs...",
  "capabilities": ["GAZE_POINT"]
}
