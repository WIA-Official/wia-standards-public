Chapter 2: Data Schema

2.1 Overview

The WIA Eye Gaze data schema defines standardized structures for representing gaze tracking data, ensuring interoperability across all devices and applications. This chapter covers the core data types, coordinate systems, and event structures.

2.2 Base Types

Vector2D

Represents 2D coordinates in normalized or pixel space.

interface Vector2D {
  x: number;  // X coordinate
  y: number;  // Y coordinate
}

Vector3D

Represents 3D coordinates for spatial gaze tracking.

interface Vector3D {
  x: number;  // X coordinate (mm)
  y: number;  // Y coordinate (mm)
  z: number;  // Z coordinate (mm, distance from device)
}

BoundingBox

Defines rectangular regions on screen.

interface BoundingBox {
  x: number;       // Top-left X (normalized)
  y: number;       // Top-left Y (normalized)
  width: number;   // Width (normalized)
  height: number;  // Height (normalized)
}

2.3 GazePoint - The Core Data Structure

GazePoint is the fundamental unit of eye tracking data, representing where the user is looking at a specific moment in time.

Complete GazePoint Structure

interface GazePoint {
  // === Required Fields ===

  /** Unix timestamp in milliseconds */
  timestamp: number;

  /** Screen X coordinate (0.0 = left, 1.0 = right) */
  x: number;

  /** Screen Y coordinate (0.0 = top, 1.0 = bottom) */
  y: number;

  /** Data confidence (0.0 - 1.0) */
  confidence: number;

  /** Data validity flag */
  valid: boolean;

  // === Optional Fields ===

  /** Left eye data */
  leftEye?: EyeData;

  /** Right eye data */
  rightEye?: EyeData;

  /** Is currently in fixation */
  fixation?: boolean;

  /** Is currently in saccade */
  saccade?: boolean;

  /** Fixation ID (groups fixation points) */
  fixationId?: string;

  /** Device-specific timestamp */
  deviceTimestamp?: number;

  /** Additional metadata */
  metadata?: Record<string, unknown>;
}

Example GazePoint

{
  "timestamp": 1704067200000,
  "x": 0.5234,
  "y": 0.3891,
  "confidence": 0.95,
  "valid": true,
  "leftEye": {
    "gaze": { "x": 0.5198, "y": 0.3876 },
    "valid": true,
    "pupilDiameter": 4.2,
    "eyeOpenness": 0.92
  },
  "rightEye": {
    "gaze": { "x": 0.5270, "y": 0.3906 },
    "valid": true,
    "pupilDiameter": 4.1,
    "eyeOpenness": 0.91
  },
  "fixation": true,
  "fixationId": "fix_001"
}

2.4 EyeData - Individual Eye Information

Detailed data for left and right eyes individually.

interface EyeData {
  // === Basic Gaze Data ===

  /** Gaze coordinates for this eye */
  gaze: Vector2D;

  /** Data validity */
  valid: boolean;

  // === Pupil Data ===

  /** Pupil diameter in millimeters */
  pupilDiameter?: number;

  /** Pupil center in camera image (normalized) */
  pupilCenter?: Vector2D;

  // === 3D Data ===

  /** Eye position in 3D space (mm from tracker) */
  gazeOrigin?: Vector3D;

  /** Gaze direction as unit vector */
  gazeDirection?: Vector3D;

  // === Eye State ===

  /** Eye openness (0.0 = closed, 1.0 = fully open) */
  eyeOpenness?: number;

  /** Eye opening in millimeters */
  eyeOpennessMm?: number;
}

Pupil Diameter Applications

Pupil diameter is useful for:

  • Cognitive Load - Larger pupils indicate higher mental effort
  • Emotional Response - Pupil changes correlate with arousal and interest
  • Lighting Adaptation - Tracking pupil response to brightness changes
  • Fatigue Detection - Pupil dynamics change with tiredness

2.5 Coordinate System

Screen Coordinates (Normalized)

All gaze coordinates use normalized values from 0.0 to 1.0:

Screen Layout:

(0.0, 0.0) ──────────────────────► X (1.0, 0.0)
     │
     │     ┌─────────────────────┐
     │     │                     │
     │     │      Screen         │
     │     │                     │
     │     │   (0.5, 0.5)        │
     │     │     Center          │
     │     │                     │
     │     └─────────────────────┘
     │
     ▼
Y (0.0, 1.0)                    (1.0, 1.0)

Coordinate Conversion

Converting between normalized and pixel coordinates:

// Normalized to Pixel
function toPixelCoordinates(
  normalizedPoint: Vector2D,
  screenWidth: number,
  screenHeight: number
): { px: number; py: number } {
  return {
    px: Math.round(normalizedPoint.x * screenWidth),
    py: Math.round(normalizedPoint.y * screenHeight)
  };
}

// Example: 1920x1080 screen
// (0.5, 0.5) → (960, 540)

// Pixel to Normalized
function toNormalizedCoordinates(
  px: number,
  py: number,
  screenWidth: number,
  screenHeight: number
): Vector2D {
  return {
    x: px / screenWidth,
    y: py / screenHeight
  };
}

3D Coordinate System

For devices supporting 3D gaze tracking:

3D Space (Eye Tracker as Origin):

             Y (up)
             │
             │
             │
             ┼──────────► X (right)
            /
           /
          /
         Z (toward user)

* Origin: Eye tracker center
* Unit: Millimeters (mm)
* Right-handed coordinate system

2.6 Data Quality Indicators

Confidence Values

Range Quality Recommended Action
0.9 - 1.0 Excellent Normal use
0.7 - 0.9 Good Normal use
0.5 - 0.7 Fair Monitor quality
0.3 - 0.5 Poor Consider recalibration
0.0 - 0.3 Very Poor Data may be unreliable

Invalid Data Handling

// Invalid data example (during blink)
{
  "timestamp": 1704067200000,
  "x": 0,
  "y": 0,
  "confidence": 0,
  "valid": false,
  "invalidReason": "blink"
}

Invalid Reasons

  • "blink" - Eye blink detected
  • "tracking_lost" - Eyes not detected by tracker
  • "out_of_range" - User outside tracking area
  • "calibration_required" - Device needs calibration
  • "unknown" - Unspecified reason

2.7 Event System

Event Categories

Category Events Use Cases
Eye Movement Fixation, Saccade, Smooth Pursuit Reading analysis, attention tracking
Eye State Blink, Wink, Double Blink Alternative input, fatigue detection
Interaction Dwell, Gaze Enter/Leave AAC selection, UI interaction
System Calibration, Tracking Status Device management

Fixation Events

Fixations occur when the eyes remain relatively still, indicating visual attention.

interface FixationEvent {
  type: 'fixation_start' | 'fixation_update' | 'fixation_end';
  timestamp: number;
  eventId: string;
  fixationId: string;

  /** Fixation center point */
  centroid: Vector2D;

  /** Fixation radius (normalized) */
  radius?: number;

  /** Angular dispersion (degrees) */
  dispersion?: number;

  /** Duration (ms, on fixation_end) */
  duration?: number;
}

Fixation Detection Parameters

  • Minimum Duration: 100ms (configurable)
  • Maximum Dispersion: 1.0° (configurable)
  • Merge Distance: 0.5° (nearby fixations merged)

Saccade Events

Rapid eye movements between fixations.

interface SaccadeEvent {
  type: 'saccade_start' | 'saccade_end';
  timestamp: number;
  eventId: string;
  saccadeId: string;

  /** Start position */
  startPosition: Vector2D;

  /** End position (on saccade_end) */
  endPosition?: Vector2D;

  /** Movement distance (degrees) */
  amplitude?: number;

  /** Peak velocity (degrees/second) */
  peakVelocity?: number;

  /** Duration (ms) */
  duration?: number;
}

Blink Events

Eye closure detection for input and fatigue monitoring.

interface BlinkEvent {
  type: 'blink_start' | 'blink_end' | 'blink' | 'double_blink';
  timestamp: number;
  eventId: string;
  blinkId: string;

  /** Which eye blinked */
  eye: 'left' | 'right' | 'both';

  /** Blink duration (ms) */
  duration?: number;

  /** Interval for double blink (ms) */
  interval?: number;
}

Intentional vs. Natural Blinks

  • Natural Blink: 100-400ms duration
  • Intentional Blink: 400ms+ duration (configurable threshold)

AAC applications can use intentional blinks as a selection mechanism.

Dwell Events

Critical for AAC - gaze-based selection by looking at a target.

interface DwellEvent {
  type: 'dwell_start' | 'dwell_progress' | 'dwell_complete' | 'dwell_cancel';
  timestamp: number;
  eventId: string;
  dwellId: string;

  /** Target element */
  target: GazeTarget;

  /** Configured dwell time (ms) */
  dwellThreshold: number;

  /** Elapsed time (ms) */
  elapsed: number;

  /** Progress (0.0 - 1.0) */
  progress: number;

  /** Cancel reason (if cancelled) */
  cancelReason?: 'gaze_left' | 'timeout' | 'user_cancel';
}

Dwell Interaction Flow

User looks at button
    │
    ▼
gaze_enter {target: button_1}
    │
    ▼
dwell_start {target: button_1, threshold: 800ms}
    │
    ├─ dwell_progress {progress: 0.25}
    ├─ dwell_progress {progress: 0.50}
    ├─ dwell_progress {progress: 0.75}
    ├─ dwell_progress {progress: 1.00}
    │      │
    │      ▼
    │  dwell_complete {target: button_1}
    │      ↓
    │  [Action Triggered]
    │
    └─ (if gaze leaves)
       dwell_cancel {reason: 'gaze_left'}
           │
           ▼
       gaze_leave {target: button_1}

2.8 Batch Data Format

For efficient transmission of multiple samples:

interface GazeBatch {
  version: string;
  deviceId: string;
  sessionId: string;
  startTimestamp: number;
  endTimestamp: number;
  samplingRate: number;
  samples: GazePoint[];
}

{
  "version": "1.0.0",
  "deviceId": "tobii-pro-fusion-001",
