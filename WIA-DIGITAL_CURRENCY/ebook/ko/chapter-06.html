📡 제6장: Phase 3 - 프로토콜

6.1 프로토콜 아키텍처

WIA-DIGITAL_CURRENCY 표준의 프로토콜 레이어는 실시간 통신, 크로스 체인 거래, 보안 프로토콜을 정의합니다. 이 레이어는 다양한 디지털 화폐 시스템 간의 원활한 상호작용을 가능하게 합니다.

6.1.1 핵심 구성 요소

  • 전송 계층: TCP/IP, WebSocket, gRPC
  • 메시지 계층: 표준화된 메시지 형식
  • 세션 관리: 연결 상태 및 세션 유지
  • 보안 계층: 암호화, 인증, 서명

6.2 전송 계층

전송 계층은 네트워크 노드 간의 데이터 전달을 담당합니다. WIA-DIGITAL_CURRENCY는 다양한 사용 사례에 맞춰 여러 전송 프로토콜을 지원합니다.

프로토콜 사용 사례 특성 지연시간
HTTPS REST API 요청-응답 50-200ms
WebSocket 실시간 스트리밍 양방향, 지속 연결 10-50ms
gRPC 고성능 서비스 간 통신 바이너리, 스트리밍 5-20ms
QUIC 차세대 전송 멀티플렉싱, 0-RTT 3-15ms

6.2.1 프로토콜 선택 가이드

  • HTTPS: 단순 조회, 일회성 거래 제출에 적합
  • WebSocket: 실시간 가격, 거래 알림, 대시보드
  • gRPC: 마이크로서비스 간 통신, 배치 처리
  • QUIC: 모바일 환경, 불안정한 네트워크

6.4 실시간 스트리밍

WebSocket을 통한 실시간 이벤트 스트리밍을 지원합니다.

// WebSocket 연결 예시
const ws = new WebSocket('wss://api.wia-dc.org/v1/stream');

ws.onopen = () => {
  ws.send(JSON.stringify({
    action: 'subscribe',
    channels: ['transactions', 'blocks']
  }));
};

ws.onmessage = (event) => {
  const data = JSON.parse(event.data);
  console.log('Received:', data);
};

6.4.1 이벤트 유형

이벤트 설명
transaction.pending 거래 생성됨
transaction.confirmed 거래 확인됨
block.new 새 블록 생성
balance.changed 잔액 변동

6.5 크로스 체인 프로토콜

WIA 표준은 다양한 블록체인 간의 거래를 지원하는 크로스 체인 프로토콜을 정의합니다. 이를 통해 사용자는 서로 다른 네트워크의 디지털 자산을 중개자 없이 교환할 수 있습니다.

6.5.1 원자 스왑

해시 타임락 컨트랙트(HTLC)를 사용한 신뢰 불필요 교환을 지원합니다.

// HTLC 스왑 흐름
1. Alice: HTLC 생성 (Chain A)
   - hash(secret) 사용
   - 24시간 타임락

2. Bob: HTLC 생성 (Chain B)
   - 동일한 hash 사용
   - 12시간 타임락

3. Alice: Chain B에서 secret으로 청구
4. Bob: Chain A에서 공개된 secret으로 청구

6.5.2 브릿지 프로토콜

브릿지 유형 보안 모델 속도 비용
Lock & Mint 다중서명 검증 10-30분 중간
Burn & Mint SPV 증명 30-60분 낮음
유동성 풀 AMM 기반 1-5분 높음
ZK 롤업 영지식 증명 5-15분 중간

6.5.3 IBC (Inter-Blockchain Communication)

Cosmos SDK 기반 체인 간 표준 통신 프로토콜을 지원합니다:

  • Light Client: 상대 체인의 상태 검증
  • Connection: 두 체인 간 신뢰 채널 설정
  • Channel: 특정 모듈 간 통신 경로
  • Packet: 실제 데이터 전송 단위

6.6 보안 프로토콜

6.6.1 암호화

  • 전송 암호화: TLS 1.3 필수
  • 메시지 암호화: AES-256-GCM
  • 키 교환: ECDH (P-256 또는 X25519)

6.6.2 서명 알고리즘

알고리즘 용도
Ed25519 거래 서명
ECDSA (secp256k1) 이더리움 호환
Schnorr 다중 서명, 집계

6.7 연결 관리

프로토콜은 다음 연결 관리 기능을 제공합니다:

  • 하트비트: 30초 간격 연결 확인
  • 자동 재연결: 지수 백오프 전략
  • 세션 복구: 마지막 메시지부터 재개

6.7.1 연결 상태 관리

// 연결 상태 머신
enum ConnectionState {
  DISCONNECTED = 'disconnected',
  CONNECTING = 'connecting',
  CONNECTED = 'connected',
  RECONNECTING = 'reconnecting',
  FAILED = 'failed'
}

// 연결 관리자 구현
class ConnectionManager {
  private state: ConnectionState = ConnectionState.DISCONNECTED;
  private reconnectAttempts = 0;
  private maxReconnectAttempts = 10;
  private heartbeatInterval: number = 30000; // 30초

  async connect(): Promise<void> {
    this.state = ConnectionState.CONNECTING;
    try {
      await this.establishConnection();
      this.state = ConnectionState.CONNECTED;
      this.startHeartbeat();
      this.reconnectAttempts = 0;
    } catch (error) {
      await this.handleConnectionError(error);
    }
  }

  private async handleConnectionError(error: Error): Promise<void> {
    if (this.reconnectAttempts < this.maxReconnectAttempts) {
      this.state = ConnectionState.RECONNECTING;
      const delay = Math.min(1000 * Math.pow(2, this.reconnectAttempts), 30000);
      await this.wait(delay);
      this.reconnectAttempts++;
      return this.connect();
    }
    this.state = ConnectionState.FAILED;
    throw new Error('Max reconnection attempts exceeded');
  }
}

6.7.2 세션 복구 메커니즘

복구 시나리오 처리 방법 데이터 손실
일시적 네트워크 단절 세션 ID로 재연결, 마지막 시퀀스부터 재개 없음
서버 재시작 세션 복원 후 미전송 메시지 재전송 최소 (버퍼 내)
클라이언트 크래시 새 세션 생성, 마지막 확인 시점부터 폴링 가능 (미확인 메시지)
장기 단절 (>1시간) 새 세션 생성, 전체 동기화 없음 (재동기화)

6.9 성능 최적화

6.9.1 배치 처리

대량의 트랜잭션을 효율적으로 처리하기 위한 배치 프로토콜을 지원합니다:

// 배치 요청
POST /v1/transactions/batch
{
  "batch_id": "batch_2025_001",
  "transactions": [
    { "id": "tx_001", "type": "TRANSFER", ... },
    { "id": "tx_002", "type": "TRANSFER", ... },
    // ... 최대 1000개
  ],
  "options": {
    "atomic": false,  // 개별 처리 (일부 실패 허용)
    "priority": "normal",
    "callback_url": "https://example.com/callback"
  }
}

// 배치 응답
{
  "batch_id": "batch_2025_001",
  "status": "processing",
  "total": 1000,
  "accepted": 998,
  "rejected": 2,
  "rejected_details": [
    { "id": "tx_050", "error": "INSUFFICIENT_BALANCE" },
    { "id": "tx_789", "error": "INVALID_SIGNATURE" }
  ]
}

6.10 장 요약

✓ 핵심 요점:

  1. 다중 전송 계층: HTTP, WebSocket, gRPC, QUIC 지원
  2. 표준 메시지: 일관된 헤더/바디 구조와 확장 가능한 메타데이터
  3. 실시간 스트리밍: WebSocket 기반 이벤트 구독과 세션 복구
  4. 크로스 체인: HTLC 기반 원자 스왑과 다양한 브릿지 프로토콜
  5. 강력한 보안: TLS 1.3, AES-256, Ed25519, Schnorr
  6. 성능 최적화: 배치 처리, 프로토콜 협상, 압축 지원

  1. WIA 프로토콜의 네 가지 핵심 구성 요소를 설명하시오.
    • 힌트: 전송, 메시지, 세션, 보안 계층을 고려하세요.
  2. HTTP, WebSocket, gRPC, QUIC의 사용 사례와 지연시간 특성을 비교하시오.
    • 힌트: 각 프로토콜의 연결 방식과 적합한 시나리오를 분석하세요.
  3. 연결 상태 머신의 5가지 상태와 상태 전이를 설명하시오.
    • 힌트: DISCONNECTED에서 CONNECTED까지의 흐름을 추적하세요.
  4. 프로토콜 협상 핸드셰이크의 HELLO/WELCOME 메시지 구조를 설명하시오.
    • 힌트: 클라이언트와 서버 간의 기능 협상 과정을 설명하세요.
  5. 배치 처리의 atomic 옵션이 true일 때와 false일 때의 차이를 설명하시오.
    • 힌트: 일부 트랜잭션 실패 시 전체 처리에 미치는 영향을 고려하세요.
  6. 세션 복구 메커니즘에서 장기 단절과 일시적 단절의 처리 방식 차이를 설명하시오.
    • 힌트: 세션 ID 유지 여부와 동기화 범위를 비교하세요.
