Chapter 06

영지식 증명 (Zero-Knowledge Proofs)

Proving Without Revealing - The Art of Cryptographic Privacy

6.1 영지식 증명의 개념

영지식 증명(Zero-Knowledge Proof, ZKP)은 1985년 Goldwasser, Micali, Rackoff에 의해 제안된 암호학적 프로토콜로, 증명자(Prover)가 검증자(Verifier)에게 특정 명제가 참임을 증명하되, 그 명제가 참이라는 사실 외에는 어떤 정보도 공개하지 않는 것을 보장합니다.

영지식 증명의 핵심 아이디어

"나는 비밀을 알고 있다. 하지만 비밀이 무엇인지는 말하지 않겠다.
그래도 내가 비밀을 알고 있다는 것은 증명할 수 있다."

6.1.1 동굴 비유 (Ali Baba's Cave)

영지식 증명을 직관적으로 이해하기 위해 유명한 "동굴 비유"를 살펴봅니다.

알리바바의 동굴

원형 동굴이 있고, 중간에 마법의 문이 있습니다. 문을 열려면 비밀 주문을 알아야 합니다.

  1. Alice(증명자)는 동굴 안으로 들어가 왼쪽 또는 오른쪽 경로를 선택합니다.
  2. Bob(검증자)은 동굴 입구에서 "왼쪽으로 나와!" 또는 "오른쪽으로 나와!"라고 외칩니다.
  3. Alice가 비밀 주문을 안다면, 어느 경로로 들어갔든 Bob이 원하는 쪽으로 나올 수 있습니다.
  4. 이를 여러 번 반복하면, Bob은 Alice가 주문을 안다고 확신하게 됩니다.
  5. 하지만 Bob은 주문 자체는 전혀 알지 못합니다.

6.1.2 영지식 증명의 세 가지 속성

속성 정의 의미
완전성 (Completeness) 참인 명제는 정직한 증명자가 증명 가능 진실을 말하면 반드시 증명됨
건전성 (Soundness) 거짓 명제는 증명 불가 거짓말쟁이는 검증자를 속일 수 없음
영지식성 (Zero-Knowledge) 명제의 참 외에 다른 정보 누출 없음 비밀 자체는 절대 공개되지 않음

6.2 영지식 증명의 유형

6.2.1 대화형 vs 비대화형

💬

대화형 ZKP (Interactive)

증명자와 검증자가 여러 라운드의 메시지를 주고받습니다. 검증자가 무작위 챌린지를 보내고 증명자가 응답합니다.

  • 실시간 상호작용 필요
  • 특정 검증자만 확신
  • 원본 ZKP 형태
📄

비대화형 ZKP (Non-Interactive)

단일 메시지로 증명이 완료됩니다. 누구나 증명을 검증할 수 있습니다. 블록체인에서 주로 사용됩니다.

  • 한 번의 증명 생성
  • 공개적으로 검증 가능
  • Fiat-Shamir 변환

6.3 zk-SNARKs

zk-SNARK(Zero-Knowledge Succinct Non-interactive ARgument of Knowledge)는 가장 널리 사용되는 영지식 증명 시스템입니다. Zcash, Ethereum 등 주요 블록체인에서 활용됩니다.

zk-SNARKs 특성
S - Succinct: 증명 크기가 작고 검증이 빠름
N - Non-interactive: 단일 메시지
AR - ARgument: 계산적 건전성
K - of Knowledge: 지식 존재 증명

6.3.1 zk-SNARKs 작동 원리

  1. 산술 회로 변환: 증명할 계산을 산술 회로로 표현
  2. R1CS 변환: 회로를 Rank-1 Constraint System으로 변환
  3. QAP 변환: R1CS를 Quadratic Arithmetic Program으로 변환
  4. 다항식 커밋먼트: 타원 곡선 페어링을 이용한 증명 생성
# zk-SNARK 개념적 흐름 (의사 코드)
class ZkSnark:
    """zk-SNARK 시스템의 개념적 구현"""

    def __init__(self):
        self.proving_key = None
        self.verification_key = None

    def setup(self, circuit):
        """
        신뢰 설정 (Trusted Setup)
        - 독성 폐기물(toxic waste)이 생성됨
        - 이것이 유출되면 가짜 증명 생성 가능
        """
        # 무작위 비밀 s 생성 (나중에 폐기해야 함)
        s = random_field_element()

        # 증명 키와 검증 키 생성
        self.proving_key = self._generate_proving_key(circuit, s)
        self.verification_key = self._generate_verification_key(circuit, s)

        # 비밀 s 폐기 (중요!)
        del s

        return self.proving_key, self.verification_key

    def prove(self, proving_key, public_input, private_witness):
        """
        증명 생성

        Parameters:
        - proving_key: 설정 단계에서 생성된 증명 키
        - public_input: 공개 입력 (검증자도 알고 있음)
        - private_witness: 비밀 입력 (증명자만 알고 있음)

        Returns:
        - proof: 간결한 증명 (약 200 바이트)
        """
        # 1. witness로 산술 회로 계산
        wire_values = self._evaluate_circuit(public_input, private_witness)

        # 2. 다항식 구성
        polynomials = self._construct_polynomials(wire_values)

        # 3. 타원 곡선 연산으로 증명 생성
        proof = self._generate_proof(proving_key, polynomials)

        return proof

    def verify(self, verification_key, public_input, proof):
        """
        증명 검증

        Parameters:
        - verification_key: 설정 단계에서 생성된 검증 키
        - public_input: 공개 입력
        - proof: 증명자가 생성한 증명

        Returns:
        - bool: 증명이 유효하면 True
        """
        # 페어링 검사를 통한 검증
        # e(A, B) = e(C, D) 형태의 등식 확인
        return self._pairing_check(verification_key, public_input, proof)

6.3.2 Trusted Setup 문제

Trusted Setup의 위험성

zk-SNARKs는 초기 설정 단계에서 "독성 폐기물(toxic waste)"이라 불리는 비밀 값이 생성됩니다. 이 값이 유출되면 누구나 가짜 증명을 생성할 수 있습니다.

  • 해결책 1: 다자간 의식(MPC Ceremony)으로 분산 생성
  • 해결책 2: 범용 설정(Universal Setup) 사용
  • 해결책 3: zk-STARKs처럼 투명한 설정 사용

6.4 zk-STARKs

zk-STARK(Zero-Knowledge Scalable Transparent ARgument of Knowledge)는 Trusted Setup이 필요 없고 양자 컴퓨터에도 안전한 증명 시스템입니다.

특성 zk-SNARKs zk-STARKs
Trusted Setup 필요 불필요 (투명)
양자 안전성 아니오 (타원 곡선 기반) 예 (해시 기반)
증명 크기 ~200 bytes ~50-200 KB
검증 시간 ~10ms ~100ms
증명 생성 시간 상대적으로 빠름 상대적으로 느림
스케일링 증명 크기 일정 로그 스케일링

6.4.1 zk-STARKs의 핵심 기술

📊

AIR (Algebraic Intermediate Representation)

계산을 다항식 제약 조건으로 표현합니다. 실행 추적(execution trace)을 테이블 형태로 만들고 다항식으로 인코딩합니다.

🔍

FRI (Fast Reed-Solomon IOP)

다항식이 낮은 차수임을 증명하는 프로토콜입니다. 해시 함수만 사용하여 투명한 설정을 가능하게 합니다.

6.5 Bulletproofs

Bulletproofs는 Trusted Setup 없이 짧은 증명을 생성하는 시스템으로, 특히 범위 증명(range proof)에 최적화되어 있습니다. Monero 암호화폐에서 거래 금액을 숨기는 데 사용됩니다.

범위 증명 (Range Proof)

"나는 이 암호화된 값이 0과 2^64 사이에 있다는 것을 증명하지만, 실제 값이 무엇인지는 공개하지 않는다."

금융 거래에서 금액이 유효한 범위 내에 있음을 증명하면서 실제 금액은 비밀로 유지하는 데 사용됩니다.

6.6.2 신원 및 인증

🪪

연령 증명

"나는 18세 이상이다"라는 것을 증명하면서 실제 생년월일이나 신분증 정보는 공개하지 않습니다. 주류 판매, 성인 콘텐츠 접근 등에 활용됩니다.

🎓

자격 증명

"나는 특정 대학의 졸업생이다"를 증명하면서 학번, 졸업 연도 등 세부 정보는 공개하지 않습니다. 채용 과정에서 프라이버시 보호에 유용합니다.

💳

신용 증명

"나의 신용 점수는 700 이상이다"를 증명하면서 실제 점수나 재정 상태는 공개하지 않습니다. 대출 심사에서 최소한의 정보만 공유합니다.

6.6.3 의료 분야

# 의료 자격 증명 ZKP 예시 (개념적)
class MedicalCredentialZKP:
    """
    환자가 특정 의료 조건을 만족함을 증명하는 ZKP
    실제 의료 기록은 공개하지 않음
    """

    def prove_vaccination_status(self, vaccination_record, required_vaccine):
        """
        예방접종 완료를 증명

        증명 내용: "나는 COVID-19 백신을 완전히 접종했다"
        공개하지 않는 내용: 접종 날짜, 백신 제조사, 의료기관 등
        """
        # 1. 공개 입력: 필요한 백신 종류
        public_input = {
            'required_vaccine': required_vaccine,
            'verification_authority': 'CDC'
        }

        # 2. 비밀 입력: 실제 접종 기록
        private_witness = {
            'vaccination_date': vaccination_record.date,
            'vaccine_manufacturer': vaccination_record.manufacturer,
            'lot_number': vaccination_record.lot_number,
            'healthcare_provider': vaccination_record.provider,
            'digital_signature': vaccination_record.signature
        }

        # 3. 증명 생성: "비밀 입력이 공개 조건을 만족함"
        proof = self.zkp_system.prove(
            circuit=self.vaccination_circuit,
            public_input=public_input,
            private_witness=private_witness
        )

        return proof

    def prove_medical_condition(self, diagnosis_record, condition_category):
        """
        특정 의료 조건에 해당함을 증명

        예: "나는 장애인 복지법상 장애인이다" (등급, 세부 진단 미공개)
        예: "나는 희귀질환자이다" (구체적 질환명 미공개)
        """
        public_input = {
            'condition_category': condition_category,
            'valid_until': '2027-01-01'
        }

        private_witness = {
            'diagnosis_code': diagnosis_record.icd_code,
            'diagnosis_date': diagnosis_record.date,
            'certifying_physician': diagnosis_record.physician,
            'medical_institution': diagnosis_record.institution
        }

        proof = self.zkp_system.prove(
            circuit=self.medical_condition_circuit,
            public_input=public_input,
            private_witness=private_witness
        )

        return proof

6.8.1 Circom 예시

// Circom으로 작성한 간단한 ZKP 회로
// 목적: 해시의 사전 이미지를 알고 있음을 증명

pragma circom 2.0.0;

include "circomlib/poseidon.circom";

template PreimageProof() {
    // 비밀 입력: 해시의 사전 이미지
    signal private input preimage;

    // 공개 입력: 해시 값
    signal input hash;

    // Poseidon 해시 계산
    component hasher = Poseidon(1);
    hasher.inputs[0] <== preimage;

    // 제약 조건: 계산된 해시 == 공개 해시
    hash === hasher.out;
}

component main {public [hash]} = PreimageProof();

/*
사용 방법:
1. 컴파일: circom preimage.circom --r1cs --wasm
2. 설정: snarkjs groth16 setup preimage.r1cs pot_final.ptau
3. 증명 생성: snarkjs groth16 prove preimage.zkey witness.wtns proof.json
4. 검증: snarkjs groth16 verify verification_key.json public.json proof.json
*/
