6.1 Introduction to Zero-Knowledge Proofs

Zero-Knowledge Proofs (ZKPs) are cryptographic protocols that allow one party (the prover) to convince another party (the verifier) that a statement is true, without revealing any information beyond the validity of the statement itself. This seemingly paradoxical capability has profound implications for privacy, authentication, and blockchain technology.

The concept was introduced in 1985 by Shafi Goldwasser, Silvio Micali, and Charles Rackoff in their seminal paper "The Knowledge Complexity of Interactive Proof Systems." Since then, ZKPs have evolved from theoretical constructs to practical tools deployed in cryptocurrency systems, authentication protocols, and privacy-preserving applications.

Zero-Knowledge Proof
A protocol between a prover P and a verifier V satisfying three properties:

Completeness: If the statement is true, an honest prover can convince an honest verifier
Soundness: If the statement is false, no cheating prover can convince the verifier (except with negligible probability)
Zero-Knowledge: The verifier learns nothing beyond the validity of the statement

6.3 Interactive vs. Non-Interactive Proofs

6.3.1 Interactive Zero-Knowledge Proofs

Interactive proofs require multiple rounds of communication between prover and verifier. The verifier sends random challenges, and the prover responds.

import hashlib
import random

class SchnorrIdentification:
    """
    Schnorr's interactive zero-knowledge identification protocol.
    Proves knowledge of discrete logarithm x where y = g^x mod p.
    """

    def __init__(self, p, g, y, x=None):
        """
        Args:
            p: Prime modulus
            g: Generator
            y: Public key (g^x mod p)
            x: Private key (secret, only prover knows)
        """
        self.p = p
        self.g = g
        self.y = y
        self.x = x  # Prover's secret

    def prover_commitment(self):
        """Step 1: Prover generates random commitment."""
        self.r = random.randint(1, self.p - 2)  # Random nonce
        self.t = pow(self.g, self.r, self.p)    # Commitment
        return self.t

    def verifier_challenge(self):
        """Step 2: Verifier sends random challenge."""
        self.c = random.randint(1, self.p - 2)
        return self.c

    def prover_response(self, c):
        """Step 3: Prover computes response."""
        # s = r + c*x mod (p-1)
        self.s = (self.r + c * self.x) % (self.p - 1)
        return self.s

    def verifier_check(self, t, c, s):
        """Step 4: Verifier checks the proof."""
        # Check: g^s == t * y^c mod p
        lhs = pow(self.g, s, self.p)
        rhs = (t * pow(self.y, c, self.p)) % self.p
        return lhs == rhs


# Example: Proving knowledge of discrete log
# Small example (use much larger primes in practice!)
p = 23  # Prime
g = 5   # Generator
x = 7   # Secret key
y = pow(g, x, p)  # Public key: g^x mod p = 17

# Prover knows x, verifier knows only (p, g, y)
prover = SchnorrIdentification(p, g, y, x=x)
verifier = SchnorrIdentification(p, g, y, x=None)

# Interactive protocol
print("Schnorr Identification Protocol")
print(f"Public: p={p}, g={g}, y={y}")
print(f"Secret: x={x} (only prover knows)")

# Run multiple rounds to reduce cheating probability
for round_num in range(5):
    # Step 1: Prover commitment
    t = prover.prover_commitment()

    # Step 2: Verifier challenge
    c = verifier.verifier_challenge()

    # Step 3: Prover response
    s = prover.prover_response(c)

    # Step 4: Verifier check
    valid = verifier.verifier_check(t, c, s)
    print(f"Round {round_num + 1}: t={t}, c={c}, s={s}, Valid={valid}")

6.3.2 Non-Interactive Zero-Knowledge (NIZK)

Non-interactive proofs require only one message from prover to verifier. This is achieved using the Fiat-Shamir heuristic, which replaces the verifier's random challenge with a hash of the prover's commitment.

import hashlib
import random

class SchnorrNIZK:
    """
    Non-interactive Schnorr proof using Fiat-Shamir heuristic.
    Proves knowledge of x where y = g^x mod p.
    """

    def __init__(self, p, g):
        self.p = p
        self.g = g

    def prove(self, x, y, message=b""):
        """
        Generate non-interactive proof of knowledge of x.

        Args:
            x: Secret (discrete log)
            y: Public value (g^x mod p)
            message: Optional message to bind to proof

        Returns:
            Proof tuple (t, s)
        """
        # Generate random commitment
        r = random.randint(1, self.p - 2)
        t = pow(self.g, r, self.p)

        # Fiat-Shamir: Hash commitment (and message) to get challenge
        hash_input = f"{t}{y}{message.decode() if message else ''}".encode()
        c = int(hashlib.sha256(hash_input).hexdigest(), 16) % (self.p - 1)

        # Compute response
        s = (r + c * x) % (self.p - 1)

        return (t, s)

    def verify(self, y, proof, message=b""):
        """
        Verify non-interactive proof.

        Args:
            y: Public value (claimed g^x mod p)
            proof: Proof tuple (t, s)
            message: Optional bound message

        Returns:
            True if proof is valid
        """
        t, s = proof

        # Recompute challenge using same hash
        hash_input = f"{t}{y}{message.decode() if message else ''}".encode()
        c = int(hashlib.sha256(hash_input).hexdigest(), 16) % (self.p - 1)

        # Verify: g^s == t * y^c mod p
        lhs = pow(self.g, s, self.p)
        rhs = (t * pow(y, c, self.p)) % self.p

        return lhs == rhs


# Example: Non-interactive proof
p = 2**256 - 2**32 - 977  # secp256k1-like prime (simplified)
p = 104729  # Use smaller prime for demo
g = 2

# Prover's secret and public key
x = 12345
y = pow(g, x, p)

nizk = SchnorrNIZK(p, g)

# Generate proof
proof = nizk.prove(x, y, message=b"Verify my identity")
print(f"Proof: t={proof[0]}, s={proof[1]}")

# Anyone can verify (without knowing x)
is_valid = nizk.verify(y, proof, message=b"Verify my identity")
print(f"Proof valid: {is_valid}")

# Tampering with message invalidates proof
is_valid_tampered = nizk.verify(y, proof, message=b"Different message")
print(f"Tampered message valid: {is_valid_tampered}")

6.4 zk-SNARKs

zk-SNARKs (Zero-Knowledge Succinct Non-Interactive Arguments of Knowledge) are a revolutionary advancement in zero-knowledge proofs. They provide extremely short proofs that can be verified quickly, regardless of the complexity of the statement being proved.

zk-SNARK Properties
Zero-Knowledge + Succinct + Non-Interactive + ARgument of Knowledge

Succinct: Proof size is O(1) or O(log n), verification is O(1)
Non-Interactive: Single message from prover to verifier
Argument: Computationally sound (vs. statistically sound)

6.5 zk-STARKs

zk-STARKs (Zero-Knowledge Scalable Transparent ARguments of Knowledge) address some limitations of SNARKs, particularly the trusted setup requirement and quantum vulnerability.

zk-STARK Advantages

  • No trusted setup: Transparent, no toxic waste
  • Post-quantum secure: Based on collision-resistant hashes
  • Scalable prover: Proving time scales quasi-linearly
  • Publicly verifiable: Anyone can verify without setup

Trade-offs

  • Larger proofs: ~100 KB vs. ~200 bytes for SNARKs
  • Slower verification: ~100ms vs. ~10ms
Property zk-SNARKs zk-STARKs
Trusted Setup Required (most) Not required
Proof Size ~200 bytes ~100 KB
Verification Time ~10ms ~100ms
Prover Time Seconds-minutes Seconds-minutes
Quantum Resistant No Yes
Assumptions Elliptic curves, pairings Hash functions only

6.6.2 Anonymous Credentials

// Anonymous age verification
class AnonymousAgeVerification {
    /*
    Goal: Prove "I am over 18" without revealing actual age or identity

    Setup:
    - Government issues signed credential: Sign(age, name, birthdate)
    - User holds credential

    Verification:
    - User generates ZKP proving:
      1. I have a valid government signature
      2. The age in credential >= 18
    - Verifier checks proof, learns nothing about actual age/identity
    */

    function proveAgeOver18(credential, governmentPublicKey) {
        // Extract age from credential (private)
        const age = credential.age;

        // Public inputs
        const minAge = 18;
        const govPubKey = governmentPublicKey;

        // Generate ZKP
        const proof = generateProof({
            private: {
                credential: credential,
                age: age
            },
            public: {
                minAge: minAge,
                govPubKey: govPubKey
            },
            constraints: [
                "verifySignature(credential, govPubKey) == true",
                "age >= minAge"
            ]
        });

        return proof;
    }

    function verify(proof, minAge, govPubKey) {
        // Verifier learns: user is over 18 with valid credential
        // Verifier does NOT learn: exact age, name, or any other info
        return verifyProof(proof, { minAge, govPubKey });
    }
}

6.8 Security Considerations

ZKP Security Warnings

  • Trusted Setup: If compromised, fake proofs can be created (SNARKs)
  • Circuit Bugs: Incorrect constraints allow invalid proofs
  • Soundness Errors: Implementation bugs can break security
  • Side Channels: Timing attacks may leak witness information
  • Quantum Threat: Most SNARKs are not post-quantum secure

6.9 Summary

Zero-knowledge proofs enable proving statements without revealing underlying information. From interactive protocols to non-interactive SNARKs and STARKs, ZKPs have evolved into practical tools transforming privacy, authentication, and blockchain technology.

Key Takeaways

  • ZKPs prove knowledge without revealing information
  • Three properties: Completeness, Soundness, Zero-Knowledge
  • SNARKs: Small proofs, fast verification, needs trusted setup
  • STARKs: No trusted setup, quantum-resistant, larger proofs
  • Applications: Private transactions, anonymous credentials, zkRollups
  • Tools: Circom, snarkjs, arkworks, ZoKrates, Cairo, Noir
  • Security: Careful with trusted setup, circuit design, implementation
