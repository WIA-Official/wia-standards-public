Chapter 03

Identity & Access Management

3.1 Introduction to Identity and Access Management

Identity and Access Management (IAM) is a framework of policies, processes, and technologies that enable organizations to manage digital identities and control user access to critical information. In an era of cloud computing, remote work, and sophisticated cyber threats, effective IAM has become a cornerstone of enterprise security.

The fundamental principle of IAM is simple: ensure that the right individuals have access to the right resources at the right times for the right reasons. However, implementing this principle at enterprise scale across diverse systems, applications, and user populations is extraordinarily complex.

3.1.1 The Identity Challenge

Modern organizations face unprecedented identity management challenges:

  • Identity Proliferation: Users maintain dozens of accounts across enterprise systems, cloud applications, and partner platforms
  • Diverse Populations: Employees, contractors, partners, customers, and machines all require managed identities
  • Hybrid Environments: Identity must span on-premises systems, multiple clouds, and SaaS applications
  • Dynamic Workforce: Remote work, job changes, and turnover require rapid provisioning and deprovisioning
  • Regulatory Requirements: Compliance mandates require auditability, segregation of duties, and access reviews

The Cost of Poor IAM

According to industry research, compromised credentials are involved in over 80% of data breaches. Orphaned accounts, excessive privileges, and weak authentication create significant security vulnerabilities that attackers readily exploit.

3.2 Identity Lifecycle Management

Identity lifecycle management encompasses all processes for creating, maintaining, and removing digital identities throughout their existence within an organization.

3.2.1 Identity Lifecycle Stages

1

Joiner (Onboarding)

When a new employee, contractor, or partner joins, their identity is created in authoritative sources, provisioned to required systems, and granted baseline access appropriate for their role.

2

Mover (Role Change)

When users change roles, departments, or locations, their access must be updated to reflect new responsibilities while revoking access no longer required. This is often the most challenging phase.

3

Leaver (Offboarding)

When users leave the organization, all access must be promptly revoked, accounts disabled or deleted, and access tokens invalidated across all connected systems.

3.2.2 Provisioning and Deprovisioning

Automated provisioning ensures that identity changes flow from authoritative sources (HR systems, identity governance platforms) to all connected applications and systems:

Automated Provisioning Benefits

  • Reduced time to productivity for new hires
  • Consistent application of access policies
  • Elimination of manual errors
  • Comprehensive audit trail
  • Rapid response to role changes

Provisioning Capabilities

  • Role-based account creation
  • Group membership management
  • License assignment
  • Resource provisioning
  • Workflow approvals

Deprovisioning Risk

Orphaned accounts - accounts belonging to former employees that remain active - represent significant security risks. Attackers often target these accounts as they may go unmonitored. The WIA-SEC-001 standard requires deprovisioning within 24 hours of separation for standard users and immediately for privileged accounts.

3.2.3 Identity Data Quality

The effectiveness of IAM depends on the quality of underlying identity data. Organizations must maintain:

  • Authoritative Sources: Define which systems are authoritative for each identity attribute
  • Data Synchronization: Ensure identity data is consistent across all systems
  • Attribute Standards: Define naming conventions and data formats
  • Data Validation: Implement validation rules to prevent erroneous data entry
  • Regular Audits: Periodically verify identity data accuracy

3.3 Authentication Mechanisms

Authentication is the process of verifying that users are who they claim to be. Modern authentication goes far beyond simple username and password combinations.

3.3.1 Authentication Factors

Factor Type Description Examples Strengths/Weaknesses
Knowledge Something you know Passwords, PINs, security questions Easy to implement; vulnerable to theft/guessing
Possession Something you have Smart cards, tokens, mobile devices Physical requirement; can be lost/stolen
Inherence Something you are Fingerprint, face, iris, voice Unique to individual; privacy concerns
Location Somewhere you are GPS, network location, IP address Context-aware; can be spoofed
Behavior Something you do Typing patterns, gait, usage patterns Continuous verification; requires baseline

3.3.2 Multi-Factor Authentication (MFA)

MFA requires users to present two or more authentication factors from different categories. This dramatically reduces the risk of unauthorized access even if one factor is compromised.

WIA-SEC-001 MFA Requirements

The WIA-SEC-001 standard mandates MFA for all privileged access, remote access, and access to sensitive data or systems. For Bronze certification, MFA is required for privileged users. Gold and Platinum levels require MFA for all users accessing enterprise resources.

3.3.3 Modern Authentication Methods

FIDO2/WebAuthn

Passwordless authentication using public key cryptography. Users authenticate with biometrics or PINs on their devices, with private keys never leaving the device.

Push Notifications

Authentication requests sent to registered mobile devices. Users approve or deny access with a single tap, providing convenience with security.

TOTP/HOTP

Time-based and HMAC-based One-Time Passwords generated by authenticator apps. Widely supported and doesn't require network connectivity.

Hardware Tokens

Physical devices that generate one-time codes or support cryptographic authentication. Provide highest security but require distribution and management.

3.3.4 Password Security

Despite the push toward passwordless authentication, passwords remain prevalent. Organizations must implement strong password policies:

WIA-SEC-001 Password Requirements: =================================== Minimum Length: 12 characters (14 for privileged accounts) Complexity: Mix of uppercase, lowercase, numbers, and symbols History: Prevent reuse of last 24 passwords Maximum Age: 90 days (or risk-based with continuous auth) Lockout: 5 failed attempts triggers 30-minute lockout Additional Controls: • Check passwords against breach databases • Implement password strength meters • Prohibit common passwords and patterns • Support secure password managers • Enable self-service password reset with MFA

3.4 Authorization and Access Control

While authentication verifies identity, authorization determines what authenticated users are permitted to do. Effective authorization ensures users can access only the resources necessary for their roles.

3.4.1 Access Control Models

Role-Based Access Control (RBAC)

Access permissions are assigned to roles, and users are assigned to roles based on their job functions. This simplifies management but can lead to role explosion in complex environments.

Attribute-Based Access Control (ABAC)

Access decisions are based on attributes of users, resources, actions, and environment. Provides fine-grained control but requires comprehensive attribute management.

Policy-Based Access Control (PBAC)

Access decisions are driven by policies that can incorporate multiple factors including roles, attributes, time, location, and risk scores.

Risk-Based Access Control

Access decisions factor in real-time risk assessment, requiring additional verification or denying access when risk indicators are elevated.

3.4.2 Principle of Least Privilege

The principle of least privilege dictates that users should have only the minimum access necessary to perform their job functions. Implementing this principle requires:

  • Role Analysis: Define roles based on actual job requirements, not historical precedent
  • Just-In-Time Access: Grant elevated access only when needed and for limited duration
  • Access Certification: Regular review and recertification of user access rights
  • Entitlement Reviews: Periodic review of what access each role or permission grants
  • Segregation of Duties: Prevent any single user from having conflicting permissions

3.5 Single Sign-On and Federation

Single Sign-On (SSO) allows users to authenticate once and gain access to multiple applications without re-entering credentials. Federation extends this concept across organizational boundaries.

3.5.1 SSO Benefits

  • User Experience: Reduced password fatigue and friction
  • Security: Fewer passwords to manage and potentially compromise
  • Productivity: Less time spent on authentication
  • Help Desk Reduction: Fewer password reset requests
  • Centralized Control: Single point for authentication policy enforcement

3.5.2 Federation Protocols

Protocol Primary Use Key Features
SAML 2.0 Enterprise SSO XML-based, mature, widely supported in enterprise apps
OAuth 2.0 API Authorization Token-based, delegated authorization, not authentication
OpenID Connect Consumer/Modern SSO Built on OAuth 2.0, adds identity layer, JSON-based
WS-Federation Microsoft Environments SOAP-based, common in Microsoft-centric environments

3.5.3 Identity Provider Architecture

A centralized Identity Provider (IdP) serves as the authoritative source for authentication:

SSO Authentication Flow (SAML): ================================ 1. User accesses Service Provider (SP) application 2. SP generates SAML AuthnRequest 3. User redirected to Identity Provider (IdP) 4. IdP authenticates user (if not already authenticated) 5. IdP generates SAML Response with Assertion 6. User redirected back to SP with SAML Response 7. SP validates Assertion signature and claims 8. SP creates session and grants access

3.6 Privileged Access Management

Privileged Access Management (PAM) focuses on securing, controlling, and monitoring access to privileged accounts and credentials. These accounts - administrator, root, service accounts - pose the greatest risk if compromised.

3.6.1 Types of Privileged Accounts

Administrative Accounts

Domain admins, system administrators, database admins with elevated privileges on enterprise systems.

Service Accounts

Non-human accounts used by applications to access resources. Often have extensive privileges and rarely-changed passwords.

Emergency Accounts

Break-glass accounts for emergency access when normal authentication fails. Must be heavily controlled and audited.

Third-Party Access

Vendor and contractor accounts with privileged access for system maintenance and support.

3.6.2 PAM Controls

  • Credential Vaulting: Store privileged credentials in encrypted vaults with controlled access
  • Password Rotation: Automatically rotate privileged passwords on schedule or after use
  • Session Management: Record and monitor privileged sessions for audit and forensics
  • Just-In-Time Access: Grant privileged access only when needed, with automatic expiration
  • Approval Workflows: Require manager or security approval for privileged access
  • Command Filtering: Restrict which commands can be executed during privileged sessions

Privileged Account Risks

Privileged accounts are primary targets for attackers. A single compromised admin credential can enable complete network takeover. The WIA-SEC-001 standard requires comprehensive PAM controls for Gold and Platinum certification levels.

3.6.3 Zero Standing Privileges

The zero standing privileges model eliminates persistent privileged access. Users request elevated access for specific tasks, receive temporary credentials, and access is automatically revoked upon completion or timeout.

PAM Best Practices

Implement a privileged access workstation (PAW) strategy where privileged tasks are performed only from dedicated, hardened systems. Never use privileged accounts for everyday activities like email or web browsing, and implement network segmentation to protect PAM infrastructure.

3.7 Identity Governance

Identity Governance ensures that access rights are appropriate, compliant, and regularly reviewed. It provides visibility into who has access to what and whether that access is justified.

3.7.1 Access Certification

Access certification campaigns require managers and resource owners to periodically review and confirm user access rights:

  • Manager-based reviews for direct reports
  • Application owner reviews for specific systems
  • Data owner reviews for sensitive information access
  • Risk-based reviews focused on high-risk access
  • Continuous certification with real-time decisions

3.7.2 Segregation of Duties

SoD controls prevent individuals from having conflicting access that could enable fraud or errors:

Common SoD Conflicts: ====================== • Create Vendor + Approve Payments • Create Purchase Order + Receive Goods • Modify Financial Records + Approve Financial Reports • Create User Accounts + Assign Privileges • Develop Code + Deploy to Production • Request Access + Approve Access

3.7.3 Access Request and Approval

Structured access request processes ensure appropriate access is granted with proper authorization:

  1. User submits access request through self-service portal
  2. System checks request against policies (SoD, role requirements)
  3. Request routed to appropriate approvers based on access type
  4. Approvers review and approve/deny with justification
  5. Approved access automatically provisioned to target systems
  6. Access logged and available for audit review
