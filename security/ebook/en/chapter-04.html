Chapter Overview

Identity and Access Management (IAM) is the discipline that ensures the right individuals access the right resources at the right times for the right reasons. In today's complex IT environments spanning on-premises, cloud, and hybrid infrastructures, robust IAM is essential for both security and operational efficiency.

This chapter covers the core concepts of IAM, modern authentication methods, authorization frameworks, and the emerging Zero Trust security model. We will explore practical implementations including Single Sign-On (SSO), Multi-Factor Authentication (MFA), and Privileged Access Management (PAM).

Zero Trust Architecture

Zero Trust is a security model based on the principle "never trust, always verify." Unlike traditional perimeter-based security, Zero Trust assumes that threats exist both outside and inside the network, requiring continuous verification of every user, device, and application attempting to access resources.

Core Zero Trust Principles

🔒
Verify Explicitly

Always authenticate and authorize based on all available data points

🎯
Least Privilege

Limit user access with just-in-time and just-enough-access

💥
Assume Breach

Minimize blast radius and segment access, verify end-to-end encryption

📊
Continuous Validation

Monitor and validate security posture continuously

Zero Trust Implementation Steps

  1. Define the Protect Surface: Identify critical data, applications, assets, and services (DAAS).
  2. Map Transaction Flows: Understand how traffic moves across the network to access protected resources.
  3. Build Zero Trust Architecture: Design the architecture around the protect surface.
  4. Create Zero Trust Policies: Develop policies using the Kipling method (who, what, when, where, why, how).
  5. Monitor and Maintain: Continuously monitor and improve the Zero Trust environment.

Zero Trust Maturity Model

Organizations typically progress through stages: Traditional (perimeter-based), Advanced (some Zero Trust concepts), and Optimal (full Zero Trust implementation). The journey requires incremental improvements across all pillars simultaneously.

Authentication Methods

Authentication is the process of verifying that users are who they claim to be. Modern authentication combines multiple factors to provide strong assurance of identity while balancing security with user experience.

Authentication Factors

🔑

Something You Know

Passwords, PINs, security questions. The most common but also most vulnerable factor due to phishing and credential theft.

📱

Something You Have

Security tokens, smart cards, mobile devices. Provides stronger security when combined with other factors.

👆

Something You Are

Biometrics including fingerprint, facial recognition, iris scan. Cannot be lost or forgotten but raises privacy concerns.

📍

Somewhere You Are

Geographic location, IP address, network location. Used as a contextual signal for risk-based authentication.

⏰

Something You Do

Behavioral biometrics including typing patterns, mouse movements, and usage patterns.

👥

Someone You Know

Social verification through trusted contacts, used primarily for account recovery scenarios.

Multi-Factor Authentication (MFA)

MFA requires users to provide two or more verification factors, dramatically reducing the risk of unauthorized access even when one factor is compromised.

MFA Method Security Level User Experience Phishing Resistant
SMS OTP Low (SIM swap risk) Good No
Email OTP Low Good No
TOTP Authenticator Medium Good No
Push Notification Medium-High Excellent Partial (with number matching)
Hardware Token (FIDO2) Very High Good Yes
Passkeys Very High Excellent Yes

Phishing-Resistant MFA

NIST and security experts recommend phishing-resistant MFA methods like FIDO2/WebAuthn hardware tokens and passkeys. These methods use cryptographic binding to the legitimate site, preventing credential interception by phishing sites.

Passwordless Authentication

Passwordless authentication eliminates passwords entirely, using stronger factors like biometrics, hardware tokens, or magic links. This approach improves both security and user experience. 

// FIDO2/WebAuthn Authentication Flow

Registration {
  1. User initiates registration on website
  2. Server generates challenge with user info
  3. Browser calls navigator.credentials.create()
  4. Authenticator creates key pair
  5. User verifies with biometric/PIN
  6. Public key sent to server for storage
  7. Private key stays securely on device
}

Authentication {
  1. User initiates login
  2. Server sends challenge with credential IDs
  3. Browser calls navigator.credentials.get()
  4. Authenticator finds matching credential
  5. User verifies with biometric/PIN
  6. Authenticator signs challenge with private key
  7. Server verifies signature with stored public key
  8. User authenticated - phishing impossible!
}

Authorization Frameworks

Authorization determines what authenticated users are allowed to do. Effective authorization frameworks enforce the principle of least privilege, ensuring users have only the access necessary to perform their duties.

Access Control Models

Role-Based Access Control (RBAC)

RBAC assigns permissions to roles, and users are assigned to roles. This simplifies administration by grouping permissions that typically go together. 

# RBAC Example Configuration

roles:
  security_analyst:
    permissions:
      - "view:security_events"
      - "analyze:alerts"
      - "create:incidents"
      - "update:incidents"
      - "view:threat_intelligence"
    data_scope: "all_events"
    
  security_manager:
    inherits: "security_analyst"
    additional_permissions:
      - "manage:team_members"
      - "approve:incident_closures"
      - "configure:alert_rules"
      - "view:audit_logs"
      
  soc_administrator:
    inherits: "security_manager"
    additional_permissions:
      - "manage:integrations"
      - "configure:system_settings"
      - "manage:user_accounts"

Attribute-Based Access Control (ABAC)

ABAC makes access decisions based on attributes of the user, resource, action, and environment. This provides fine-grained, context-aware access control. 

# ABAC Policy Example

policy "financial_data_access" {
  description = "Control access to financial reports"
  
  condition {
    # User attributes
    user.department == "finance" OR user.department == "executive"
    user.clearance_level >= resource.classification_level
    user.training_completed.includes("financial_data_handling")
    
    # Environmental attributes
    request.time BETWEEN "08:00" AND "18:00"
    request.location IN allowed_locations
    device.compliance_status == "compliant"
    
    # Action constraints
    action IN ["read", "analyze"]
    OR (action == "export" AND user.role == "controller")
  }
  
  effect = "allow"
}

Policy-Based Access Control

Modern systems often use policy languages like Open Policy Agent (OPA) Rego to define complex access control rules that can be evaluated consistently across systems.

Single Sign-On (SSO)

SSO allows users to authenticate once and gain access to multiple applications without re-entering credentials. This improves user experience while centralizing authentication control and audit logging.

SSO Protocols

Protocol Use Case Token Format Key Features
SAML 2.0 Enterprise web apps XML assertions Mature, widely supported, complex
OAuth 2.0 API authorization Access tokens Delegated authorization, flexible
OpenID Connect Modern web/mobile apps JWT ID tokens Authentication layer on OAuth 2.0
Kerberos Windows domain Tickets On-premises, Windows integration

OIDC Authentication Flow

1
User Initiates Login

User clicks "Sign in with SSO" on the application

2
Redirect to Identity Provider

3
User Authenticates

4
Authorization Code Returned

5
Token Exchange

6
Session Established

Application creates session and grants access to user

Privileged Access Management (PAM)

PAM focuses on securing, managing, and monitoring access to privileged accounts - those with elevated permissions that can make significant changes to systems. Compromised privileged accounts are among the most damaging security incidents.

PAM Components

🔐

Password Vault

Secure storage for privileged credentials with automatic rotation and check-out/check-in workflows.

⏱️

Just-in-Time Access

Grant privileged access only when needed, for limited duration, with automatic revocation.

📹

Session Recording

Record all privileged sessions for audit, forensics, and compliance requirements.

🚫

Least Privilege Enforcement

Remove standing privileges and implement just-enough-access for all operations.

🔍

Access Analytics

Monitor privileged access patterns to detect anomalies and potential threats.

🤖

Service Account Management

Discover, manage, and rotate credentials for non-human privileged identities.

# PAM Policy Configuration

privileged_access_policy:
  name: "Production Server Access"
  
  access_request:
    approval_required: true
    approvers: ["team_lead", "security_team"]
    justification_required: true
    ticket_required: true
    
  session_controls:
    max_duration: "4h"
    session_recording: true
    keystroke_logging: true
    clipboard_control: "blocked"
    file_transfer: "logged"
    
  credential_management:
    rotation_frequency: "after_each_use"
    complexity_requirements: "very_high"
    vaulting: required
    
  monitoring:
    real_time_alerts: true
    anomaly_detection: enabled
    audit_retention: "7_years"

PAM Best Practices

Implement the "Four Eyes" principle for critical operations, require MFA for all privileged access, eliminate shared accounts, and regularly review and certify privileged access rights.

Identity Lifecycle Management

Identity lifecycle management ensures that identities are properly created, maintained, and removed throughout their existence. Automated lifecycle management reduces security risks from orphaned accounts and excessive permissions.

Lifecycle Stages

// Identity Lifecycle Stages

JOINER (Onboarding)
├── HR creates employee record
├── Identity automatically provisioned
├── Base access granted from role template
├── Manager notified for additional access requests
├── Training requirements assigned
└── Welcome email with account activation

MOVER (Role Change)
├── HR updates employee record
├── Previous role access removed
├── New role access granted
├── Access certification triggered
├── Training requirements updated
└── Manager approval for sensitive access

LEAVER (Offboarding)
├── HR initiates termination
├── All access disabled immediately
├── Mailbox converted or archived
├── Data ownership transferred
├── Accounts deprovisioned after retention period
└── Audit trail preserved

Access Certification

Regular access certification (or access reviews) ensures that users retain only the access they need. Managers and application owners periodically review and certify that access assignments remain appropriate.

Chapter Summary

Identity and Access Management is fundamental to modern security programs. Key takeaways from this chapter include:

  • Zero Trust architecture assumes no implicit trust, requiring continuous verification of users, devices, and applications.
  • Multi-factor authentication significantly reduces unauthorized access risk; phishing-resistant methods like FIDO2 provide the strongest protection.
  • Passwordless authentication improves both security and user experience by eliminating vulnerable passwords.
  • Authorization frameworks (RBAC, ABAC) enforce least privilege by controlling what authenticated users can do.
  • Single Sign-On centralizes authentication while improving user experience across multiple applications.
  • Privileged Access Management protects the most sensitive accounts through vaulting, just-in-time access, and session recording.
  • Identity lifecycle management automates provisioning, changes, and deprovisioning to prevent orphaned accounts.
