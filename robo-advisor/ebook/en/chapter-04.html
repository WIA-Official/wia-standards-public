4.2 System Architecture

4.2.1 Microservices Architecture

Modern robo-advisor platforms use microservices for scalability and maintainability:

  • User Service: Authentication, profile management, KYC/AML
  • Portfolio Service: Portfolio creation, tracking, performance calculation
  • Risk Service: Risk assessment, questionnaires, profiling
  • Optimization Service: Portfolio optimization, rebalancing logic
  • Trading Service: Order generation, execution, settlement
  • Tax Service: Tax-loss harvesting, tax reporting, lot tracking
  • Market Data Service: Real-time prices, historical data, corporate actions
  • Notification Service: Email, SMS, push notifications
  • Reporting Service: Performance reports, statements, tax documents

4.2.2 Data Flow

  1. User completes risk assessment → Risk Service
  2. Risk Service calculates profile → Portfolio Service
  3. Portfolio Service requests optimal allocation → Optimization Service
  4. Optimization Service returns target portfolio → Portfolio Service
  5. Portfolio Service generates orders → Trading Service
  6. Trading Service executes trades → Broker API
  7. Settlement data flows back → Portfolio Service
  8. Performance calculated and displayed → User interface

4.3 Database Schema Design

Core Tables

-- Users
CREATE TABLE users (
    user_id UUID PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,
    created_at TIMESTAMP NOT NULL,
    kyc_status VARCHAR(50),
    tax_residency VARCHAR(2)
);

-- Portfolios
CREATE TABLE portfolios (
    portfolio_id UUID PRIMARY KEY,
    user_id UUID REFERENCES users(user_id),
    account_type VARCHAR(50),  -- taxable, ira, 401k
    created_at TIMESTAMP NOT NULL,
    status VARCHAR(20) NOT NULL,
    target_allocation JSONB,
    current_value DECIMAL(15,2)
);

-- Holdings
CREATE TABLE holdings (
    holding_id UUID PRIMARY KEY,
    portfolio_id UUID REFERENCES portfolios(portfolio_id),
    symbol VARCHAR(20) NOT NULL,
    shares DECIMAL(15,6) NOT NULL,
    cost_basis DECIMAL(10,2),
    acquisition_date DATE,
    current_price DECIMAL(10,2),
    last_updated TIMESTAMP
);

-- Transactions
CREATE TABLE transactions (
    transaction_id UUID PRIMARY KEY,
    portfolio_id UUID REFERENCES portfolios(portfolio_id),
    type VARCHAR(20),  -- buy, sell, dividend, rebalance
    symbol VARCHAR(20),
    shares DECIMAL(15,6),
    price DECIMAL(10,2),
    executed_at TIMESTAMP,
    settlement_date DATE,
    tax_lot_id UUID
);

-- Rebalance Events
CREATE TABLE rebalance_events (
    rebalance_id UUID PRIMARY KEY,
    portfolio_id UUID REFERENCES portfolios(portfolio_id),
    triggered_at TIMESTAMP,
    executed_at TIMESTAMP,
    trigger_type VARCHAR(30),
    status VARCHAR(20),
    transactions JSONB,
    impact_analysis JSONB
);

4.5 Security and Encryption

Data Protection

  • At Rest: AES-256 encryption for sensitive PII and financial data
  • In Transit: TLS 1.3 for all API communications
  • Authentication: OAuth 2.0 with JWT tokens
  • Authorization: Role-based access control (RBAC)
  • Audit Logging: Immutable audit trails for all transactions

4.6 Scalability Considerations

Horizontal Scaling

  • Stateless services behind load balancers
  • Database read replicas for query distribution
  • Caching layer (Redis) for frequently accessed data
  • Message queues (Kafka, RabbitMQ) for async processing

Performance Optimization

  • Portfolio value calculations cached for 1-minute TTL
  • Market data cached and updated every 15 seconds
  • Batch processing for rebalancing across portfolios
  • CDN for static assets and historical data

4.7 Disaster Recovery and High Availability

  • Multi-region deployment: Active-active across 3+ AWS regions
  • Database replication: Synchronous replication with automatic failover
  • Backup strategy: Continuous backups with point-in-time recovery
  • RTO/RPO: Recovery Time Objective < 5 minutes, Recovery Point Objective < 1 minute
