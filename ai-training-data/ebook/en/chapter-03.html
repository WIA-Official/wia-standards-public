Chapter 3: Dataset Management and Versioning

Controlling and Tracking Training Data Evolution

The Importance of Version Control for Datasets

Just as software requires version control for reproducibility and collaboration, training datasets need systematic versioning to track changes, enable rollbacks, and ensure reproducible experiments. Without proper version control, it becomes impossible to determine which data version was used to train a specific model, making debugging and improvement challenging.

Dataset versioning goes beyond simple file naming. It encompasses tracking content changes, metadata updates, schema evolution, and the relationships between different versions. The WIA-AI-007 standard provides a comprehensive framework for dataset version management that integrates with modern development workflows.

Semantic Versioning for Datasets

WIA-AI-007 adopts semantic versioning (SemVer) principles adapted for datasets. Version numbers follow the format MAJOR.MINOR.PATCH:

MAJOR Version Changes

Increment the MAJOR version when making incompatible changes that break existing models or workflows:

  • Schema modifications (changing feature names, types, or structure)
  • Label encoding changes (modifying class definitions or hierarchies)
  • Significant data distribution shifts
  • Format changes (switching from Parquet to TFRecord)
  • Splitting strategy changes (modifying train/val/test splits)

MINOR Version Changes

Increment the MINOR version for backward-compatible additions:

  • Adding new samples to existing classes
  • Adding new optional features
  • Expanding label taxonomy with new classes
  • Adding supplementary metadata
  • Improving annotation quality for existing samples

PATCH Version Changes

Increment the PATCH version for backward-compatible fixes:

  • Correcting labeling errors
  • Fixing metadata errors
  • Removing duplicate samples
  • Updating documentation
  • Minor quality improvements
# Example version progression
1.0.0 → Initial release (10,000 samples, 5 classes)
1.1.0 → Added 2,000 new samples (backward-compatible)
1.1.1 → Fixed 50 labeling errors (patch)
1.2.0 → Added "patient_age" feature (backward-compatible)
2.0.0 → Changed label encoding from one-hot to multi-label (breaking change)

Version Control Systems for Data

DVC (Data Version Control)

DVC is a popular open-source tool that brings Git-like version control to datasets. It stores metadata in Git while keeping large data files in remote storage (S3, GCS, Azure, etc.). 

# Initialize DVC in your project
dvc init

# Add dataset to version control
dvc add data/training_dataset.parquet

# Commit the metadata (not the data itself)
git add data/training_dataset.parquet.dvc .gitignore
git commit -m "Add training dataset v1.0.0"

# Push data to remote storage
dvc push

# Later, retrieve specific version
git checkout v1.0.0
dvc pull

Git LFS (Large File Storage)

Git LFS is suitable for smaller datasets (< 1GB) where you want to keep data versioned alongside code. It replaces large files with text pointers while storing the actual files on a remote server.

Lakefs

LakeFS provides Git-like operations over data lakes, offering branches, commits, and merges for large-scale datasets. It's designed for big data scenarios and integrates with data lake technologies like S3 and Delta Lake.

Pachyderm

Pachyderm offers data versioning combined with data pipeline orchestration. It automatically tracks data lineage and ensures reproducibility of data transformations.

Branch-Based Dataset Development

Similar to software development, datasets can benefit from branch-based workflows that enable parallel experimentation and safe integration of changes: 

# Create experimental branch for testing new labeling approach
dvc branch create experimental-labels

# Make changes to labels
python relabel_samples.py --method=active_learning

# Add and commit changes
dvc add data/dataset.parquet
git commit -m "Experiment: Active learning labels"

# If experiment successful, merge to main
dvc branch merge experimental-labels main

# If unsuccessful, discard
dvc branch delete experimental-labels

Snapshot and Checkpoint Management

Regular snapshots capture the state of datasets at specific points in time, enabling point-in-time recovery and historical analysis:

  • Daily snapshots: Automated daily captures for active datasets
  • Release snapshots: Immutable snapshots for each official version
  • Experiment snapshots: Lightweight snapshots before major experiments
  • Milestone snapshots: Snapshots at significant project milestones

Data Deduplication and Diff

Efficient storage requires deduplication across versions. WIA-AI-007 recommends content-addressable storage where identical data blocks are stored only once: 

# View differences between versions
dvc diff v1.0.0 v2.0.0

# Output:
# Added: 5000 new samples
# Modified: 1200 samples (label corrections)
# Removed: 300 duplicate samples
# Schema changes: Added "patient_age" feature
# Total size increase: +2.3 GB (after deduplication)

Dataset Migration Strategies

When upgrading to new major versions, provide migration guides and tools: 

# Migration script from v1.x to v2.0
from wia_ai_007 import DatasetMigration

migration = DatasetMigration(
    source_version="1.2.0",
    target_version="2.0.0"
)

# Automatic migration with validation
migration.migrate(
    input_path="medical-images-v1.2.0",
    output_path="medical-images-v2.0.0",
    preserve_splits=True,
    validate=True,
    backup=True
)

# Generate migration report
report = migration.generate_report()
print(f"Migrated {report['samples_migrated']} samples")
print(f"Warnings: {report['warnings']}")

Continuous Integration for Datasets

Integrate dataset validation into CI/CD pipelines to catch issues early: 

# .github/workflows/dataset-ci.yml
name: Dataset Quality Checks

on:
  push:
    paths:
      - 'data/**'

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2

      - name: Validate Schema
        run: |
          python -m wia_ai_007.validate \
            --schema schema.json \
            --data data/dataset.parquet

      - name: Quality Checks
        run: |
          python -m wia_ai_007.quality \
            --data data/dataset.parquet \
            --min-score 0.95

      - name: Bias Detection
        run: |
          python -m wia_ai_007.bias \
            --data data/dataset.parquet \
            --report bias_report.json

      - name: Generate Report
        run: |
          python -m wia_ai_007.report \
            --output dataset_report.html

Immutable Version Storage

Released versions should be immutable to ensure reproducibility. Use object storage features to enforce immutability:

  • S3 Object Lock: Prevents deletion and modification of objects
  • GCS retention policies: Enforces minimum retention periods
  • Azure immutable blob storage: WORM (Write Once Read Many) capability

Best Practices for Dataset Versioning

  • Version everything: Data, schema, code, and documentation together
  • Immutable releases: Never modify released versions
  • Document changes: Maintain detailed changelogs
  • Automate validation: Use CI/CD for quality checks
  • Track lineage: Record complete provenance information
  • Use branches: Experiment safely in separate branches
  • Regular snapshots: Create restore points for active datasets
  • Access control: Implement role-based permissions
  • Plan migrations: Provide tools and guides for major upgrades
  • Archive old versions: Clean up but preserve historical data

Chapter Summary

This chapter covered dataset management and versioning strategies essential for reproducible AI development. We explored semantic versioning adapted for datasets, various version control systems (DVC, Git LFS, LakeFS, Pachyderm), and lineage tracking mechanisms.

Proper version control enables teams to track dataset evolution, experiment safely through branching, maintain reproducibility through immutable releases, and collaborate effectively through centralized registries. The WIA-AI-007 standard provides comprehensive schemas for capturing version history, provenance, and lifecycle management.

Following these practices ensures that datasets serve the philosophy of 弘益人間 by promoting transparency, reproducibility, and collaborative improvement of AI training data across the community.
