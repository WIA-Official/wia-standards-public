Chapter 2: Data Formats and Schemas

Standardized Formats for AI Training Data

Introduction to Data Formats

The choice of data format significantly impacts the efficiency, portability, and usability of training datasets. WIA-AI-007 supports multiple standard formats while providing guidelines for format selection based on data type, use case, and performance requirements. This chapter explores the various format options and provides schemas for standardized metadata management.

Common Training Data Formats

JSON (JavaScript Object Notation)

JSON is a lightweight, human-readable format ideal for structured and semi-structured data. It's widely supported across programming languages and platforms, making it excellent for metadata, configuration, and smaller datasets. JSON supports nested structures and is self-describing, though it can be verbose for large datasets. 

{
  "dataset": {
    "name": "medical-images-v1",
    "version": "1.0.0",
    "type": "image",
    "samples": 50000,
    "created": "2025-01-15T10:30:00Z",
    "license": "CC-BY-4.0",
    "philosophy": "弘益人間"
  },
  "schema": {
    "features": {
      "image": {"type": "binary", "format": "png"},
      "metadata": {"type": "object"}
    },
    "labels": {
      "diagnosis": {"type": "categorical", "classes": ["normal", "abnormal"]}
    }
  }
}

Parquet

Apache Parquet is a columnar storage format optimized for analytical queries and large-scale data processing. It provides efficient compression, supports complex nested data structures, and enables column pruning for faster reads. Parquet is ideal for tabular data and is well-integrated with big data frameworks like Spark and Hadoop.

TFRecord

TFRecord is TensorFlow's native format, designed specifically for machine learning workflows. It uses Protocol Buffers for serialization, providing compact binary representation with efficient sequential reading. TFRecord is particularly effective for training pipelines in TensorFlow but requires framework-specific tools for manipulation.

HDF5 (Hierarchical Data Format)

HDF5 is designed for storing large numerical arrays and supports hierarchical organization of datasets. It enables efficient partial loading of large files, supports metadata at multiple levels, and provides good compression. HDF5 is popular in scientific computing and works well for image, video, and sensor data.

Arrow

Apache Arrow provides a language-independent columnar memory format for flat and hierarchical data. It enables zero-copy reads and supports efficient inter-process communication. Arrow is becoming increasingly popular for data exchange between different systems and languages.

Format Selection Guidelines

Use Case Recommended Format Rationale
Metadata & Configuration JSON Human-readable, widely supported, easy to edit
Tabular Data (< 1M rows) CSV, JSON Simple, compatible, easy to inspect
Tabular Data (> 1M rows) Parquet Efficient compression, columnar storage, fast queries
Images (TensorFlow) TFRecord Native integration, efficient pipeline
Images (PyTorch) HDF5, ZIP Random access, memory mapping support
Scientific Arrays HDF5, Zarr Partial loading, metadata support, compression
Cross-language Exchange Arrow Zero-copy, language-agnostic, efficient

WIA-AI-007 Dataset Schema

All datasets following the WIA-AI-007 standard must include a metadata file that conforms to the standard schema. This ensures consistent documentation, provenance tracking, and quality metrics across all datasets.

Core Metadata Fields

{
  "wia-ai-007": "1.0",
  "dataset": {
    "id": "unique-dataset-identifier",
    "name": "dataset-name",
    "version": "1.0.0",
    "created": "2025-01-15T10:30:00Z",
    "updated": "2025-01-15T10:30:00Z",
    "description": "Detailed dataset description",
    "type": "image|text|audio|video|tabular|multimodal",
    "format": "json|parquet|tfrecord|hdf5|arrow",
    "size": {
      "samples": 100000,

      "train": 0.8,

Versioning Strategy

WIA-AI-007 adopts semantic versioning (MAJOR.MINOR.PATCH) for datasets:

  • MAJOR: Incompatible changes (schema changes, significant data modifications)
  • MINOR: Backward-compatible additions (new samples, new features)
  • PATCH: Backward-compatible fixes (error corrections, metadata updates)
{
  "version_history": [
    {
      "version": "1.0.0",

      "changes": ["Initial release"],
      "breaking_changes": false

Schema Validation

WIA-AI-007 provides JSON Schema definitions for validating dataset metadata. This ensures all required fields are present and correctly formatted: 

{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "title": "WIA-AI-007 Dataset Metadata",
  "type": "object",
  "required": ["wia-ai-007", "dataset", "schema", "provenance", "quality", "license"],
  "properties": {
    "wia-ai-007": {
      "type": "string",
      "pattern": "^[0-9]+\\.[0-9]+$"
    },
    "dataset": {
      "type": "object",
      "required": ["id", "name", "version", "type", "format"],
      "properties": {
        "version": {
          "type": "string",
          "pattern": "^[0-9]+\\.[0-9]+\\.[0-9]+$"
        }
      }
    }
  }
}

Multi-Modal Dataset Organization

For datasets containing multiple modalities (e.g., images with text descriptions), WIA-AI-007 recommends organizing data by samples rather than by modality: 

dataset/
├── metadata.json          # WIA-AI-007 metadata
├── schema.json           # Schema validation
├── samples/
│   ├── 000001/
│   │   ├── image.png
│   │   ├── text.txt
│   │   └── labels.json
│   ├── 000002/
│   │   ├── image.png
│   │   ├── text.txt
│   │   └── labels.json
│   └── ...
└── splits/
    ├── train.txt         # List of training sample IDs
    ├── val.txt
    └── test.txt

Best Practices for Schema Design

  • Be explicit: Define types, ranges, and constraints clearly
  • Document thoroughly: Include descriptions for all fields
  • Version schemas: Track schema changes alongside data versions
  • Validate early: Check schema compliance during data ingestion
  • Plan for evolution: Design schemas that can accommodate future additions
  • Include provenance: Always document data origin and transformations
  • Capture statistics: Record distributions and summary statistics
  • Embed ethics: Include consent, anonymization, and usage information

Format Conversion Tools

WIA-AI-007 encourages the development of conversion tools to translate between formats while preserving metadata. Example conversion workflow: 

# Convert Parquet to TFRecord with metadata preservation
from wia_ai_007 import DatasetConverter

converter = DatasetConverter()
converter.convert(
    source="dataset.parquet",
    target="dataset.tfrecord",
    metadata="metadata.json",
    preserve_splits=True,
    validate_schema=True
)

Chapter Summary

This chapter covered the data formats and schemas used in the WIA-AI-007 standard. We explored various format options including JSON, Parquet, TFRecord, HDF5, and Arrow, along with guidelines for selecting the appropriate format based on use case and scale.

The WIA-AI-007 metadata schema provides a comprehensive structure for documenting datasets, including core metadata, schema definitions, provenance tracking, quality metrics, licensing information, and statistical summaries. This standardized approach ensures consistency, reproducibility, and interoperability across the AI ecosystem.

Proper schema design and validation are fundamental to creating high-quality, trustworthy training datasets that embody the philosophy of 弘益人間—benefiting all humanity through transparent, well-documented, and ethically-sourced AI training data.
