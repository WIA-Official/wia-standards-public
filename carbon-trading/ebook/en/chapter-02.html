Exchange Engine Design

The exchange engine is the heart of the trading platform, coordinating order flow, matching, and execution. It must handle high throughput while maintaining strict ordering guarantees and fair treatment of all market participants.

Order Types and Lifecycle

WIA-CARBON_TRADING supports a comprehensive set of order types to accommodate different trading strategies and risk management needs:

  • Market Orders: Execute immediately at best available price
  • Limit Orders: Execute only at specified price or better
  • Stop Orders: Trigger when price reaches specified level
  • Iceberg Orders: Display only portion of total quantity
  • Good-Till-Cancelled: Remain active until filled or cancelled
  • Fill-or-Kill: Execute entirely immediately or cancel
  • Auction Orders: Participate in scheduled auctions only

// Order Lifecycle Management
class OrderManager {
    private orders: Map<string, TradeOrder> = new Map();
    private orderBook: OrderBook;

    async submitOrder(order: TradeOrder): Promise<OrderResult> {
        // Validate order parameters
        const validation = this.validateOrder(order);
        if (!validation.valid) {
            return { status: 'rejected', reason: validation.reason };
        }

        // Pre-trade risk checks
        const riskCheck = await this.riskEngine.checkOrder(order);
        if (!riskCheck.approved) {
            return { status: 'rejected', reason: riskCheck.reason };
        }

        // Submit to order book
        order.status = 'active';
        order.timestamp = new Date();
        this.orders.set(order.orderId, order);

        // Send to matching engine
        const matchResult = await this.matchingEngine.processOrder(order);

        return matchResult;
    }
}

Matching Engine Architecture

The matching engine is responsible for pairing buy and sell orders according to price-time priority rules. Performance is critical, as matching latency directly impacts market fairness and efficiency.

Performance Target: WIA-CARBON_TRADING matching engine achieves sub-millisecond latency for order matching, processing over 100,000 orders per second during peak trading periods.

Price-Time Priority Algorithm

The standard matching algorithm follows price-time priority: orders at the best price are matched first, with earlier orders at the same price taking precedence. This ensures fair and deterministic order execution.

// Matching Engine Implementation
class MatchingEngine {
    private orderBooks: Map<string, OrderBook> = new Map();

    processOrder(incomingOrder: TradeOrder): MatchResult {
        const book = this.getOrderBook(incomingOrder.instrument);
        const matches: Match[] = [];

        if (incomingOrder.side === 'buy') {
            return this.matchBuyOrder(incomingOrder, book, matches);
        } else {
            return this.matchSellOrder(incomingOrder, book, matches);
        }
    }

    private matchBuyOrder(
        buyOrder: TradeOrder,
        book: OrderBook,
        matches: Match[]
    ): MatchResult {
        let remainingQuantity = buyOrder.quantity;

        // Iterate through sell orders at best prices
        while (remainingQuantity > 0 && book.asks.length > 0) {
            const bestAsk = book.asks[0];

            // Check price compatibility
            if (buyOrder.orderType === 'limit' && bestAsk.price > buyOrder.price) {
                break; // No more matches possible
            }

            // Calculate match quantity
            const matchQuantity = Math.min(remainingQuantity, bestAsk.remainingQuantity);

            // Create match record
            matches.push({
                matchId: generateMatchId(),
                buyOrderId: buyOrder.orderId,
                sellOrderId: bestAsk.orderId,
                quantity: matchQuantity,
                price: bestAsk.price,
                timestamp: new Date()
            });

            // Update quantities
            remainingQuantity -= matchQuantity;
            bestAsk.remainingQuantity -= matchQuantity;

            // Remove filled orders
            if (bestAsk.remainingQuantity === 0) {
                book.asks.shift();
                bestAsk.status = 'filled';
            }
        }

        // Handle remaining quantity
        if (remainingQuantity > 0) {
            if (buyOrder.orderType === 'limit') {
                buyOrder.remainingQuantity = remainingQuantity;
                this.insertOrder(buyOrder, book.bids);
            }
        }

        return {
            status: matches.length > 0 ? 'matched' : 'pending',
            matches,
            remainingQuantity
        };
    }
}

Settlement System

Settlement is the process of transferring carbon credits from seller to buyer and funds from buyer to seller following a successful trade. WIA-CARBON_TRADING supports multiple settlement modes to accommodate different market requirements.

Settlement Mode Timing Use Case
T+0 (Same Day) End of trading day Spot market transactions
T+1 Next business day Standard spot trades
T+2 Two business days International transfers
Real-Time Gross Immediate Pre-funded accounts 
// Settlement Engine
class SettlementEngine {
    private pendingSettlements: Settlement[] = [];
    private registry: CarbonRegistry;
    private paymentGateway: PaymentGateway;

    async processSettlement(trade: Trade): Promise<SettlementResult> {
        const settlement: Settlement = {
            settlementId: generateSettlementId(),
            tradeId: trade.tradeId,
            buyer: trade.buyerParticipantId,
            seller: trade.sellerParticipantId,
            creditType: trade.instrument.creditType,
            quantity: trade.quantity,
            amount: trade.quantity * trade.price,
            status: 'pending',
            settlementDate: this.calculateSettlementDate(trade)
        };

        try {
            // Step 1: Reserve credits in seller account
            await this.registry.reserveCredits(
                settlement.seller,
                settlement.creditType,
                settlement.quantity
            );

            // Step 2: Reserve funds in buyer account
            await this.paymentGateway.reserveFunds(
                settlement.buyer,
                settlement.amount
            );

            // Step 3: Execute atomic transfer
            await this.executeAtomicTransfer(settlement);

            settlement.status = 'completed';
            settlement.completedAt = new Date();

            return { success: true, settlement };

        } catch (error) {
            // Rollback reservations
            await this.rollbackSettlement(settlement);
            settlement.status = 'failed';
            settlement.failureReason = error.message;

            return { success: false, settlement, error };
        }
    }

    private async executeAtomicTransfer(settlement: Settlement): Promise<void> {
        // Use distributed transaction for atomicity
        const transaction = await this.beginTransaction();

        try {
            // Transfer credits
            await this.registry.transferCredits(
                settlement.seller,
                settlement.buyer,
                settlement.creditType,
                settlement.quantity,
                transaction
            );

            // Transfer funds
            await this.paymentGateway.transferFunds(
                settlement.buyer,
                settlement.seller,
                settlement.amount,
                transaction
            );

            await transaction.commit();
        } catch (error) {
            await transaction.rollback();
            throw error;
        }
    }
}

Data Architecture

The platform employs a multi-database architecture optimized for different data access patterns. Time-series data, transactional data, and reference data each have specialized storage solutions.

Data Integrity: All critical data is replicated across multiple availability zones with synchronous writes. Transaction logs enable point-in-time recovery and complete audit trail reconstruction.
