제6장: 3단계 - 통신 프로토콜

WIA 경비 로봇 표준 - ROB-012

6.1 실시간 통신의 필요성

2단계 REST API는 요청-응답 상호 작용에 탁월하지만 경비 로봇 시스템에는 HTTP의 풀 기반 특성이 비효율적이거나 실용적이지 않은 몇 가지 시나리오가 있습니다. 라이브 비디오 스트리밍, 지속적인 센서 원격 측정 및 즉각적인 경고 알림은 양방향 실시간 통신 프로토콜이 필요합니다.

3단계는 HTTP REST API를 보완하여 두 가지 핵심 프로토콜을 추가합니다: MQTT(Message Queuing Telemetry Transport)는 게시-구독 메시징용이고 WebSocket은 웹 브라우저에서 직접 접근 가능한 양방향 통신용입니다. 이러한 프로토콜은 낮은 지연 시간, 효율적인 대역폭 사용 및 푸시 기반 업데이트를 가능하게 합니다.

REST vs 실시간 프로토콜

특성 HTTP REST API MQTT WebSocket
통신 패턴 요청-응답 (풀) 게시-구독 (푸시) 양방향 (푸시/풀)
연결 모델 상태 비저장, 단기 상태 유지, 장기 상태 유지, 장기
오버헤드 높음 (HTTP 헤더) 매우 낮음 낮음
지연 시간 100-500 ms 1-50 ms 1-50 ms
최적 사용 사례 명령, 구성, 조회 센서 데이터, 경고 비디오 스트림, 대시보드
브라우저 지원 보편적 라이브러리 필요 네이티브

6.2 MQTT 프로토콜 사양

MQTT는 제약된 장치 및 신뢰할 수 없는 네트워크용으로 설계된 경량 게시-구독 메시징 프로토콜입니다. 낮은 대역폭 요구 사항, 작은 코드 공간 및 네트워크 불안정성에 대한 탄력성으로 IoT 및 로봇 공학 애플리케이션에서 널리 채택되었습니다.

MQTT 토픽 구조

ROB-012는 계층적 토픽 네임스페이스를 정의합니다:

wia/robot/{robot_id}/{category}/{subcategory}/{data_type}

예시:
wia/robot/ROB-SEC-001/sensors/camera/front/detections
wia/robot/ROB-SEC-001/status/battery/level
wia/robot/ROB-SEC-001/alerts/intrusion/high
wia/robot/ROB-SEC-001/telemetry/location/gps
wia/robot/ALL/commands/emergency_stop

토픽 카테고리

카테고리 토픽 패턴 게시 빈도 QoS 수준
센서 데이터 wia/robot/{id}/sensors/# 1-30 Hz 0 (최대 1회)
상태 업데이트 wia/robot/{id}/status/# 0.1-1 Hz 1 (최소 1회)
경고 wia/robot/{id}/alerts/# 이벤트 기반 2 (정확히 1회)
원격 측정 wia/robot/{id}/telemetry/# 1-10 Hz 0 (최대 1회)
명령 wia/robot/{id}/commands/# 이벤트 기반 2 (정확히 1회)

QoS (서비스 품질) 수준

MQTT는 메시지 전달 보장에 대한 3가지 QoS 수준을 제공합니다:

  • QoS 0 (최대 1회): "Fire and forget" - 메시지는 최대 1회 전달됩니다. 데이터 손실이 허용되는 고빈도 센서 데이터에 사용됩니다
  • QoS 1 (최소 1회): 메시지는 최소 1회 전달되도록 보장되며 중복이 발생할 수 있습니다. 상태 업데이트에 사용됩니다
  • QoS 2 (정확히 1회): 메시지는 정확히 1회 전달됩니다. 경고 및 명령과 같은 중요한 메시지에 사용됩니다

MQTT 메시지 예시

// 센서 데이터 게시 (QoS 0)
토픽: wia/robot/ROB-SEC-001/sensors/camera/front/snapshot
페이로드: {
  "timestamp": "2025-12-27T14:30:45.123456Z",
  "image_url": "/media/snapshots/20251227-143045-front.jpg",
  "detections": [
    {"class": "person", "confidence": 0.94, "bbox": [450, 320, 180, 380]}
  ]
}

// 경고 게시 (QoS 2)
토픽: wia/robot/ROB-SEC-001/alerts/intrusion/high
페이로드: {
  "alert_id": "ALT-2025-001234",
  "severity": "high",
  "type": "intrusion_detected",
  "timestamp": "2025-12-27T14:28:15Z",
  "location": {"latitude": 37.5666, "longitude": 126.9781},
  "description": "Person detected in restricted area"
}

// 명령 수신 (QoS 2)
토픽: wia/robot/ROB-SEC-001/commands/navigate
페이로드: {
  "command_id": "CMD-5678",
  "destination": {"latitude": 37.5670, "longitude": 126.9785},
  "urgency": "high"
}

6.3 WebSocket 프로토콜 사양

WebSocket은 단일 TCP 연결을 통한 전이중 통신을 제공하는 프로토콜입니다. HTTP 요청으로 시작하여 양방향 데이터 스트리밍을 허용하는 WebSocket 연결로 "업그레이드"됩니다. 웹 브라우저에 대한 네이티브 지원으로 플러그인 없이 실시간 웹 애플리케이션을 구축할 수 있습니다.

WebSocket 연결 설정

// 클라이언트에서 WebSocket 연결 시작
GET /api/v1/ws/stream HTTP/1.1
Host: robot-001.local
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

// 서버 응답
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

// 이제 WebSocket 연결이 설정됨

WebSocket 메시지 유형

ROB-012는 JSON 기반 메시지 프로토콜을 정의합니다:

// 구독 요청 (클라이언트 → 서버)
{
  "type": "subscribe",
  "channels": [
    "status",
    "alerts",
    "sensors.camera.front"
  ]
}

// 구독 확인 (서버 → 클라이언트)
{
  "type": "subscribed",
  "channels": ["status", "alerts", "sensors.camera.front"],
  "subscription_id": "SUB-12345"
}

// 데이터 스트림 (서버 → 클라이언트)
{
  "type": "data",
  "channel": "sensors.camera.front",
  "timestamp": "2025-12-27T14:30:45.123456Z",
  "payload": {
    "image_url": "/media/live/frame-143045.jpg",
    "detections": [...]
  }
}

// 핑/퐁 (연결 유지)
{
  "type": "ping"
}

{
  "type": "pong",
  "timestamp": "2025-12-27T14:30:46.000000Z"
}

6.4 비디오 스트리밍

라이브 비디오 피드는 경비 로봇의 핵심 기능입니다. ROB-012는 다양한 대역폭 조건 및 클라이언트 기능을 수용하기 위해 여러 비디오 스트리밍 프로토콜을 지원합니다.

프로토콜 전송 지연 시간 브라우저 지원 사용 사례
RTSP/RTP UDP/TCP 매우 낮음 (50-200 ms) 플러그인 필요 전용 모니터링 스테이션
WebRTC UDP (ICE/STUN/TURN) 매우 낮음 (50-300 ms) 네이티브 실시간 웹 모니터링
HLS (HTTP Live Streaming) HTTP 중간 (2-10초) 네이티브 모바일 앱, 확장 가능한 배포
MJPEG over HTTP HTTP 낮음 (100-500 ms) 네이티브 간단한 통합, 레거시 시스템

WebRTC 신호 프로토콜

// 비디오 스트림 요청 (클라이언트 → 로봇)
{
  "type": "webrtc_offer",
  "camera_id": "cam_front_01",
  "sdp": "v=0\r\no=- 123456789 2 IN IP4 0.0.0.0\r\n..."
}

// 비디오 스트림 응답 (로봇 → 클라이언트)
{
  "type": "webrtc_answer",
  "camera_id": "cam_front_01",
  "sdp": "v=0\r\no=- 987654321 2 IN IP4 192.168.1.100\r\n..."
}

// ICE 후보 교환
{
  "type": "ice_candidate",
  "camera_id": "cam_front_01",
  "candidate": "candidate:1 1 UDP 2130706431 192.168.1.100 54321 typ host"
}

6.5 보안 고려 사항

실시간 프로토콜은 추가적인 보안 과제를 제시합니다. ROB-012는 여러 보안 계층을 의무화합니다:

TLS 암호화

  • MQTT over TLS (MQTTS): 포트 8883, TLS 1.2 이상 필요
  • WebSocket Secure (WSS): TLS를 통한 WebSocket, HTTP와 동일한 인증서 사용
  • WebRTC DTLS: 미디어 스트림에 대한 데이터그램 TLS

인증 메커니즘

프로토콜 인증 방법 재인증 간격
MQTT 사용자 이름/비밀번호, 클라이언트 인증서 연결당 1회
WebSocket 초기 HTTP 업그레이드 중 Bearer 토큰 토큰 만료 시 (일반적으로 1-24시간)
WebRTC 신호 중 토큰, DTLS 핸드셰이크 세션당 1회

핵심 요점

  • 3단계는 HTTP REST API를 보완하여 라이브 비디오, 센서 스트리밍 및 즉각적인 경고를 위한 실시간 프로토콜을 추가합니다
  • MQTT는 게시-구독 메시징을 제공하며 낮은 오버헤드, QoS 수준 및 네트워크 불안정성에 대한 탄력성을 제공합니다
  • WebSocket은 웹 브라우저에서 양방향 실시간 통신을 가능하게 하여 플러그인 없이 실시간 대시보드를 지원합니다
  • 여러 비디오 스트리밍 프로토콜(RTSP, WebRTC, HLS, MJPEG)은 지연 시간 요구 사항 및 클라이언트 기능이 다양한 시나리오를 지원합니다
  • TLS 암호화(MQTTS, WSS, DTLS)와 강력한 인증 메커니즘은 실시간 통신의 보안을 보장합니다
  • 계층적 MQTT 토픽 네임스페이스와 QoS 수준은 효율적인 메시지 라우팅 및 전달 보장을 제공합니다

복습 문제

  1. HTTP REST API와 실시간 프로토콜(MQTT, WebSocket)을 비교하고 각 통신 패턴이 적절한 경비 로봇 시나리오를 식별하십시오.
  2. MQTT QoS 수준(0, 1, 2)을 설명하고 센서 데이터, 상태 업데이트 및 경고에 대해 다양한 수준을 사용하는 이유를 논의하십시오.
  3. ROB-012 MQTT 토픽 네임스페이스(wia/robot/{id}/{category}/{subcategory})가 어떻게 체계적인 메시지 구성 및 효율적인 구독을 가능하게 하는지 분석하십시오.
  4. WebSocket 연결 설정 프로세스(HTTP 업그레이드, 핸드셰이크)와 양방향 통신이 설정된 후 작동 방식을 설명하십시오.
  5. 4가지 비디오 스트리밍 프로토콜(RTSP, WebRTC, HLS, MJPEG)의 지연 시간, 브라우저 지원 및 사용 사례를 비교하십시오.
  6. 각 실시간 프로토콜(MQTTS, WSS, WebRTC DTLS)의 TLS 암호화 및 인증 메커니즘이 데이터 프라이버시와 시스템 보안을 어떻게 보장하는지 평가하십시오.
  7. WebRTC 신호 프로토콜(제공/응답, ICE 후보)이 NAT 통과 및 최적의 미디어 경로 설정에 어떻게 기여하는지 논의하십시오.
