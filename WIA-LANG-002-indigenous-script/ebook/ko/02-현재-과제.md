# 제2장: 원주민 문자 디지털화의 현재 과제

## 복합적 장벽에 직면한 문자 유산

원주민 문자의 디지털화는 기술적, 사회적, 정치적 요인이 복잡하게 얽혀 있는 도전입니다. 이 장에서는 문자 디지털화 과정에서 직면하는 주요 과제들을 체계적으로 분석합니다.

## 유니코드 등재의 복잡성

### 제안 과정의 장벽

유니코드 컨소시엄에 새로운 문자를 등재하려면 다음 단계를 거쳐야 합니다:

| 단계 | 소요 시간 | 필요 사항 |
|------|----------|----------|
| 1. 제안서 작성 | 6-12개월 | 문자 목록, 역사, 사용 현황 |
| 2. 기술 검토 | 12-24개월 | 인코딩 모델, 코드 포인트 할당 |
| 3. 투표 | 6개월 | 위원회 승인 |
| 4. 출판 | 6-12개월 | 유니코드 표준에 포함 |
| **총합** | **2-5년** | |

### 제안서 작성의 어려움

**기술적 요구사항**:
- 완전한 문자 목록 (모든 변형 포함)
- 인코딩 모델 제안
- 기존 문자와의 관계 분석
- 정렬 규칙 정의

**문서화 요구사항**:
- 역사적 배경
- 현재 사용 현황
- 대표적 문헌 샘플
- 전문가 검토 의견

```
제안서 거부 사유 (일반적):
├── 불완전한 문자 목록
├── 기존 문자와의 중복
├── 불충분한 사용 증거
├── 커뮤니티 합의 부재
└── 기술적 문제 (인코딩 충돌 등)
```

### 대기 중인 문자들

현재 유니코드 등재를 기다리는 주요 문자:

| 문자 | 언어 | 지역 | 상태 |
|------|------|------|------|
| 코카 (Kōka) | 오제이베 | 북미 | 제안 준비 중 |

| 아드람 (Adlam) | 풀라어 | 서아프리카 | 등재 완료 (2016) |
| 오스마냐 (Osmanya) | 소말리어 | 동아프리카 | 등재 완료 (2004) |

## 폰트 개발의 과제

### 기술적 복잡성

**복합 문자 처리**:
많은 원주민 문자는 복합 문자 형성 규칙을 가집니다:

```
복합 문자 유형:
├── 조합형 (한글형)
│   └── 자음 + 모음 → 음절
├── 변형형 (아랍형)
│   └── 글자가 위치에 따라 형태 변화
├── 스택형 (티베트형)
│   └── 글자가 수직으로 쌓임
└── 합자형 (인도계 문자)
    └── 자음 연속 → 합자 형성
```

**OpenType 기능 구현**:

```
필요한 OpenType 기능:
├── GSUB (Glyph Substitution)
│   ├── liga: 합자
│   ├── calt: 문맥 대체
│   └── rlig: 필수 합자
├── GPOS (Glyph Positioning)
│   ├── mark: 마크 배치
│   └── kern: 자간 조정
└── 스크립트별 규칙
    └── 언어별 세부 규칙
```

### 디자인 일관성 문제

**역사적 형태 변이**:
- 시대별 서체 차이
- 지역별 변형
- 필기체 vs 인쇄체

**현대적 적응**:
- 화면 가독성
- 다양한 크기 대응
- 굵기 변형 (웨이트)

### 비용 문제

전문 폰트 개발 비용:

| 항목 | 비용 (USD) | 비고 |
|------|-----------|------|
| 기본 디자인 | $10,000-50,000 | 글리프 수에 따라 |
| OpenType 개발 | $5,000-20,000 | 복잡도에 따라 |
| 힌팅 | $5,000-15,000 | 화면 최적화 |
| 테스트 | $3,000-10,000 | 다양한 환경 |
| **총합** | $23,000-95,000 | |

## 입력 시스템의 부재

### 키보드 레이아웃

**물리적 키보드의 한계**:
- 표준 키보드는 라틴 문자 기준
- 복잡한 문자는 다중 키 입력 필요
- 기존 레이아웃과의 충돌

**가상 키보드의 과제**:
- 글리프 수가 많은 문자의 배치
- 터치 타겟 크기
- 스크롤/페이지 네비게이션

### IME (입력기) 개발

```python
# 간단한 IME 로직 예시
class IndigenousScriptIME:
    def __init__(self, script_rules):
        self.rules = script_rules
        self.buffer = ""

    def process_key(self, key):
        self.buffer += key

        # 변환 규칙 적용
        for pattern, replacement in self.rules:
            if self.buffer.endswith(pattern):
                self.buffer = self.buffer[:-len(pattern)] + replacement
                break

        return self.buffer

# 예: 올치키 문자 IME
olchiki_rules = [
    ("a", "ᱚ"),   # a → ᱚ
    ("aa", "ᱛ"),  # aa → ᱛ
    ("i", "ᱤ"),   # i → ᱤ
    # ...
]
```

### 음성 및 필기 입력

**음성 입력의 한계**:
- 원주민 언어 음성 인식 모델 부재
- 훈련 데이터 부족
- 다양한 방언 처리

**필기 입력의 과제**:
- 필기체 변이 인식
- 스트로크 순서 정규화
- 실시간 처리 성능

## 역사적 문서 디지털화

### OCR의 한계

**기술적 과제**:

| 과제 | 설명 | 영향 |
|------|------|------|
| 학습 데이터 | 레이블링된 데이터 부족 | 정확도 저하 |
| 문서 상태 | 열화, 손상, 오염 | 인식 실패 |
| 형태 변이 | 필사자별 스타일 차이 | 일관성 부족 |
| 레이아웃 | 비표준 배치 | 순서 인식 오류 |

**정확도 비교**:

```
OCR 정확도 (문자 단위):
├── 라틴 문자 (인쇄): 99%+
├── 한자 (인쇄): 95-98%
├── 아랍 문자 (인쇄): 90-95%
├── 원주민 문자 (인쇄): 70-90%
└── 원주민 문자 (필사): 50-80%
```

### 메타데이터 부재

**일반적 문제**:
- 문서 출처 불명
- 저자/시기 미상
- 문맥 정보 손실

**카탈로그 표준화**:
- 기존 도서관 표준과의 호환
- 원주민 문자 전용 필드
- 다국어 기술(記述)

## 커뮤니티 관련 과제

### 표준화 논쟁

**공동체 내 갈등**:
- 정자(正字) vs 약자(略字)
- 역사적 형태 vs 현대적 형태
- 발음 vs 전통 표기

**사례: 올치키 문자**
- 창제자의 원형 vs 현대 변형
- 지역별 선호 차이
- 학교 교육 표준

### 문화적 민감성

**고려 사항**:
- 신성한 텍스트의 디지털화 동의
- 접근 제한이 필요한 자료
- 문화적 맥락의 보존

**윤리적 가이드라인**:
```
디지털화 동의 체크리스트:
☐ 커뮤니티 대표자 동의
☐ 사용 목적 명시
☐ 접근 권한 설정
☐ 수익 분배 합의
☐ 철회 조건 명시
```
