2장: 접근 제어 모델과 아키텍처

2.1 접근 제어 모델 개요

접근 제어 모델은 시스템이 접근 결정을 내리는 방식을 정의하는 이론적 프레임워크입니다. 각 모델은 서로 다른 보안 요구사항과 운영 환경에 최적화되어 있으며, 현대 시스템에서는 여러 모델을 조합하여 사용하는 경우가 많습니다.

모델 선택의 중요성: 잘못된 접근 제어 모델의 선택은 보안 취약점을 만들거나, 과도한 제한으로 업무 효율성을 저해할 수 있습니다. 조직의 규모, 산업 특성, 규제 요구사항을 고려하여 적절한 모델을 선택해야 합니다.

2.2 임의적 접근 제어 (DAC)

임의적 접근 제어(Discretionary Access Control)는 리소스 소유자가 다른 사용자에게 접근 권한을 부여하거나 제거할 수 있는 모델입니다. 유닉스/리눅스 파일 시스템이 대표적인 예입니다.

2.2.1 DAC의 특징

  • 리소스 소유자에게 권한 관리 권한 위임
  • 유연하고 관리가 간단
  • 사용자 간 권한 전이 가능
  • 정보 유출 위험 존재
// DAC 구현 예시 (Unix 스타일)
interface DACPermission {
    owner: string;
    group: string;
    permissions: {
        owner: { read: boolean; write: boolean; execute: boolean };
        group: { read: boolean; write: boolean; execute: boolean };
        others: { read: boolean; write: boolean; execute: boolean };
    };
}

class DACController {
    checkAccess(user: User, resource: Resource, action: Action): boolean {
        const perm = resource.permissions;

        if (user.id === perm.owner) {
            return perm.permissions.owner[action];
        } else if (user.groups.includes(perm.group)) {
            return perm.permissions.group[action];
        } else {
            return perm.permissions.others[action];
        }
    }

    // 소유자가 권한 변경 가능
    changePermissions(owner: User, resource: Resource, newPerm: DACPermission): void {
        if (resource.permissions.owner !== owner.id) {
            throw new Error('소유자만 권한을 변경할 수 있습니다');
        }
        resource.permissions = newPerm;
    }
}

2.3 강제적 접근 제어 (MAC)

강제적 접근 제어(Mandatory Access Control)는 중앙 기관이 정의한 보안 정책에 따라 접근이 결정됩니다. 사용자나 리소스 소유자가 정책을 변경할 수 없습니다.

2.3.1 Bell-LaPadula 모델

기밀성을 중시하는 군사/정부 환경에서 사용되는 모델입니다.

규칙 설명
No Read Up (Simple Security) 낮은 보안 등급의 주체는 높은 등급의 객체를 읽을 수 없음
No Write Down (*-Property) 높은 보안 등급의 주체는 낮은 등급의 객체에 쓸 수 없음 
// Bell-LaPadula 모델 구현
enum SecurityLevel {
    UNCLASSIFIED = 0,
    CONFIDENTIAL = 1,
    SECRET = 2,
    TOP_SECRET = 3
}

class BellLaPadula {
    canRead(subject: Subject, object: Resource): boolean {
        // No Read Up: 주체의 클리어런스가 객체의 분류 등급 이상이어야 함
        return subject.clearance >= object.classification;
    }

    canWrite(subject: Subject, object: Resource): boolean {
        // No Write Down: 주체의 클리어런스가 객체의 분류 등급 이하여야 함
        return subject.clearance <= object.classification;
    }

    checkAccess(subject: Subject, object: Resource, action: Action): boolean {
        if (action === 'read') {
            return this.canRead(subject, object);
        } else if (action === 'write') {
            return this.canWrite(subject, object);
        }
        return false;
    }
}

2.3.2 Biba 무결성 모델

데이터 무결성을 중시하는 상업 환경에 적합한 모델입니다.

Biba의 규칙: Bell-LaPadula와 반대로 "No Read Down" (신뢰할 수 없는 데이터 읽기 금지)과 "No Write Up" (높은 무결성 등급 데이터 수정 금지) 규칙을 적용합니다.

2.4 역할 기반 접근 제어 (RBAC)

역할 기반 접근 제어(Role-Based Access Control)는 개별 사용자가 아닌 역할에 권한을 할당하고, 사용자를 역할에 매핑하는 모델입니다. 대규모 조직에서 권한 관리를 단순화합니다.

2.4.1 RBAC 핵심 구성요소

// RBAC 핵심 구조
interface RBACSystem {
    // 사용자 집합
    users: Set<User>;

    // 역할 집합
    roles: Set<Role>;

    // 권한 집합
    permissions: Set<Permission>;

    // 사용자-역할 매핑
    userRoleAssignment: Map<User, Set<Role>>;

    // 역할-권한 매핑
    rolePermissionAssignment: Map<Role, Set<Permission>>;

    // 역할 계층 (선택적)
    roleHierarchy: Map<Role, Set<Role>>;
}

class RBACController {
    // 사용자의 유효 권한 계산
    getEffectivePermissions(user: User): Set<Permission> {
        const permissions = new Set<Permission>();
        const userRoles = this.userRoleAssignment.get(user) || new Set();

        for (const role of userRoles) {
            // 직접 할당된 권한
            const rolePerms = this.rolePermissionAssignment.get(role);
            rolePerms?.forEach(p => permissions.add(p));

            // 역할 계층을 통한 상속된 권한
            const inheritedRoles = this.getInheritedRoles(role);
            for (const inheritedRole of inheritedRoles) {
                const inheritedPerms = this.rolePermissionAssignment.get(inheritedRole);
                inheritedPerms?.forEach(p => permissions.add(p));
            }
        }

        return permissions;
    }

    checkAccess(user: User, permission: Permission): boolean {
        const effectivePerms = this.getEffectivePermissions(user);
        return effectivePerms.has(permission);
    }
}

2.5 속성 기반 접근 제어 (ABAC)

속성 기반 접근 제어(Attribute-Based Access Control)는 주체, 객체, 환경의 속성을 기반으로 동적인 접근 결정을 내립니다. 가장 유연하고 강력한 모델입니다.

// ABAC 정책 구현
interface ABACPolicy {
    id: string;
    description: string;
    target: {
        subjectAttributes?: AttributeCondition[];
        resourceAttributes?: AttributeCondition[];
        actionAttributes?: AttributeCondition[];
    };
    condition: Expression;
    effect: 'permit' | 'deny';
}

class ABACEngine {
    policies: ABACPolicy[];

    evaluateAccess(request: AccessRequest): AccessDecision {
        const context = {
            subject: request.subject.attributes,
            resource: request.resource.attributes,
            action: request.action,
            environment: {
                currentTime: new Date(),
                ipAddress: request.clientIP,
                deviceType: request.deviceType,
                location: request.geoLocation
            }
        };

        // 모든 적용 가능한 정책 평가
        const applicablePolicies = this.policies.filter(p =>
            this.matchesTarget(p.target, context)
        );

        // 정책 결합 알고리즘 적용
        return this.combineDecisions(applicablePolicies, context);
    }

    private evaluateCondition(condition: Expression, context: Context): boolean {
        // 조건식 평가 (예: subject.department === 'engineering' && time.hour >= 9)
        return this.expressionEvaluator.evaluate(condition, context);
    }
}
ABAC의 장점: ABAC는 "오전 9시부터 오후 6시 사이에 회사 네트워크에서 접속한 정규직 엔지니어만 소스 코드에 접근 가능"과 같은 복잡하고 세분화된 정책을 표현할 수 있습니다.

2.6 관계 기반 접근 제어 (ReBAC)

관계 기반 접근 제어(Relationship-Based Access Control)는 엔티티 간의 관계를 기반으로 접근을 결정합니다. 소셜 미디어, 협업 도구에서 널리 사용됩니다.

// ReBAC 구현 (Google Zanzibar 스타일)
interface Tuple {
    object: string;      // 리소스 (예: "document:budget-2024")
    relation: string;    // 관계 (예: "viewer", "editor", "owner")
    subject: string;     // 사용자 또는 그룹 (예: "user:alice")
}

class ReBACEngine {
    tuples: Tuple[];

    // 직접 관계 확인
    check(object: string, relation: string, subject: string): boolean {
        // 직접 매칭
        if (this.tuples.some(t =>
            t.object === object &&
            t.relation === relation &&
            t.subject === subject
        )) {
            return true;
        }

        // 간접 관계 (그룹 멤버십, 관계 상속 등) 확인
        return this.checkIndirect(object, relation, subject);
    }

    // 예: viewer는 editor를 포함, editor는 owner를 포함
    private checkIndirect(object: string, relation: string, subject: string): boolean {
        const relationHierarchy = {
            'viewer': ['editor', 'owner'],
            'editor': ['owner']
        };

        const higherRelations = relationHierarchy[relation] || [];
        return higherRelations.some(r => this.check(object, r, subject));
    }
}
