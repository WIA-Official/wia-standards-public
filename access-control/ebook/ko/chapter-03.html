3장: 인증 메커니즘

3.1 인증의 중요성

인증(Authentication)은 접근 제어의 첫 번째 관문입니다. 사용자, 시스템, 또는 서비스의 신원을 확인하는 과정으로, "당신이 주장하는 그 사람이 맞습니까?"라는 질문에 답합니다. 강력한 인증 없이는 어떤 인가 시스템도 효과적으로 작동할 수 없습니다.

인증의 세 가지 요소: 현대 보안에서는 단일 요소 인증으로는 충분하지 않습니다. 다중 요소 인증(MFA)은 지식, 소유, 생체 중 두 가지 이상을 결합하여 보안을 강화합니다.

3.2 비밀번호 기반 인증

가장 오래되고 널리 사용되는 인증 방식이지만, 여러 취약점이 있습니다:

3.2.1 비밀번호 보안 구현

// 안전한 비밀번호 처리
import { scrypt, randomBytes } from 'crypto';

class PasswordManager {
    private readonly SALT_LENGTH = 32;
    private readonly KEY_LENGTH = 64;
    private readonly SCRYPT_PARAMS = { N: 16384, r: 8, p: 1 };

    async hashPassword(password: string): Promise<string> {
        const salt = randomBytes(this.SALT_LENGTH);
        const hash = await this.deriveKey(password, salt);

        // 솔트와 해시를 함께 저장
        return `${salt.toString('hex')}:${hash.toString('hex')}`;
    }

    async verifyPassword(password: string, stored: string): Promise<boolean> {
        const [saltHex, hashHex] = stored.split(':');
        const salt = Buffer.from(saltHex, 'hex');
        const storedHash = Buffer.from(hashHex, 'hex');

        const hash = await this.deriveKey(password, salt);

        // 타이밍 공격 방지를 위한 상수 시간 비교
        return crypto.timingSafeEqual(hash, storedHash);
    }

    private deriveKey(password: string, salt: Buffer): Promise<Buffer> {
        return new Promise((resolve, reject) => {
            scrypt(password, salt, this.KEY_LENGTH, this.SCRYPT_PARAMS,
                (err, key) => err ? reject(err) : resolve(key));
        });
    }
}

3.3 다중 요소 인증 (MFA)

MFA는 두 가지 이상의 독립적인 인증 요소를 결합하여 보안을 강화합니다:

// MFA 구현 예시
interface MFAConfig {
    requiredFactors: number;  // 최소 요구 요소 수
    availableMethods: AuthMethod[];
    riskBasedEnforcement: boolean;
}

class MFAController {
    private config: MFAConfig;
    private verifiedFactors: Map<string, AuthFactor[]> = new Map();

    async initiateAuth(userId: string): Promise<AuthChallenge> {
        const user = await this.userService.getUser(userId);
        const riskScore = await this.riskEngine.calculateRisk(user);

        // 위험 수준에 따라 요구 요소 수 조정
        const requiredFactors = riskScore > 0.7
            ? this.config.requiredFactors + 1
            : this.config.requiredFactors;

        return {
            sessionId: this.generateSessionId(),
            requiredFactors,
            availableMethods: user.enabledMethods
        };
    }

    async verifyFactor(sessionId: string, factor: AuthFactor, proof: any): Promise<VerifyResult> {
        let verified = false;

        switch (factor.type) {
            case 'password':
                verified = await this.verifyPassword(factor.userId, proof);
                break;
            case 'totp':
                verified = this.verifyTOTP(factor.secret, proof);
                break;
            case 'webauthn':
                verified = await this.verifyWebAuthn(factor, proof);
                break;
            case 'sms':
                verified = await this.verifySMSCode(sessionId, proof);
                break;
        }

        if (verified) {
            const factors = this.verifiedFactors.get(sessionId) || [];
            factors.push(factor);
            this.verifiedFactors.set(sessionId, factors);
        }

        return {
            verified,
            remainingFactors: this.getRemainingFactors(sessionId)
        };
    }
}

3.4 TOTP 기반 인증

시간 기반 일회용 비밀번호(TOTP)는 공유된 비밀과 현재 시간을 기반으로 일회용 코드를 생성합니다:

// TOTP 구현 (RFC 6238)
import { createHmac } from 'crypto';

class TOTPGenerator {
    private readonly PERIOD = 30;  // 30초 간격
    private readonly DIGITS = 6;   // 6자리 코드

    generate(secret: Buffer, time: number = Date.now()): string {
        const counter = Math.floor(time / 1000 / this.PERIOD);
        const counterBuffer = Buffer.alloc(8);
        counterBuffer.writeBigInt64BE(BigInt(counter));

        const hmac = createHmac('sha1', secret);
        hmac.update(counterBuffer);
        const hash = hmac.digest();

        // Dynamic truncation
        const offset = hash[hash.length - 1] & 0xf;
        const binary =
            ((hash[offset] & 0x7f) << 24) |
            ((hash[offset + 1] & 0xff) << 16) |
            ((hash[offset + 2] & 0xff) << 8) |
            (hash[offset + 3] & 0xff);

        const otp = binary % Math.pow(10, this.DIGITS);
        return otp.toString().padStart(this.DIGITS, '0');
    }

    verify(secret: Buffer, token: string, window: number = 1): boolean {
        const now = Date.now();
        for (let i = -window; i <= window; i++) {
            const time = now + (i * this.PERIOD * 1000);
            if (this.generate(secret, time) === token) {
                return true;
            }
        }
        return false;
    }
}

3.5 생체 인식 인증

생체 인식은 사용자의 고유한 신체적 또는 행동적 특성을 사용합니다:

3.5.1 생체 인식 유형

유형 특성 FAR/FRR
지문 고유한 융선 패턴 매우 낮음
얼굴 얼굴 구조 및 특징 낮음-중간
홍채 홍채 패턴 극히 낮음
음성 음성 패턴 중간
행동 타이핑 패턴, 걸음걸이 중간-높음
생체 인식 보안: 생체 정보는 변경할 수 없으므로 유출 시 심각한 문제가 됩니다. 생체 템플릿은 원본 데이터가 아닌 수학적 표현으로 저장하고, 암호화하여 보호해야 합니다.

3.6 FIDO2/WebAuthn

FIDO2는 비밀번호 없는 인증을 위한 현대적 표준입니다:

// WebAuthn 등록 및 인증 흐름
class WebAuthnService {
    // 등록 옵션 생성
    async generateRegistrationOptions(user: User): Promise<PublicKeyCredentialCreationOptions> {
        return {
            challenge: this.generateChallenge(),
            rp: {
                name: "WIA Access Control",
                id: "example.com"
            },
            user: {
                id: Buffer.from(user.id),
                name: user.email,
                displayName: user.name
            },
            pubKeyCredParams: [
                { alg: -7, type: "public-key" },   // ES256
                { alg: -257, type: "public-key" }  // RS256
            ],
            timeout: 60000,
            attestation: "direct",
            authenticatorSelection: {
                authenticatorAttachment: "platform",
                requireResidentKey: false,
                userVerification: "preferred"
            }
        };
    }

    // 등록 응답 검증
    async verifyRegistration(
        response: AuthenticatorAttestationResponse,
        expectedChallenge: Buffer
    ): Promise<CredentialRecord> {
        // 1. clientDataJSON 파싱 및 검증
        const clientData = JSON.parse(
            Buffer.from(response.clientDataJSON).toString()
        );

        if (clientData.type !== 'webauthn.create') {
            throw new Error('Invalid response type');
        }

        // 2. 챌린지 검증
        const receivedChallenge = Buffer.from(clientData.challenge, 'base64');
        if (!receivedChallenge.equals(expectedChallenge)) {
            throw new Error('Challenge mismatch');
        }

        // 3. attestationObject 파싱 및 공개 키 추출
        const attestation = this.parseAttestationObject(response.attestationObject);

        return {
            credentialId: attestation.credentialId,
            publicKey: attestation.publicKey,
            signCount: attestation.signCount
        };
    }
}

3.7 인증서 기반 인증

X.509 인증서를 사용한 상호 TLS(mTLS) 인증:

// mTLS 인증 설정
import { createServer } from 'https';
import { readFileSync } from 'fs';

const server = createServer({
    key: readFileSync('server-key.pem'),
    cert: readFileSync('server-cert.pem'),
    ca: readFileSync('ca-cert.pem'),
    requestCert: true,
    rejectUnauthorized: true
}, (req, res) => {
    const clientCert = req.socket.getPeerCertificate();

    if (req.client.authorized) {
        const subject = clientCert.subject;
        console.log(`인증된 클라이언트: ${subject.CN}`);

        // 인증서 속성 기반 추가 검증
        if (!this.validateCertificateAttributes(clientCert)) {
            res.writeHead(403);
            res.end('Certificate validation failed');
            return;
        }

        // 정상 처리
        res.writeHead(200);
        res.end(`Welcome, ${subject.CN}`);
    } else {
        res.writeHead(401);
        res.end('Client certificate required');
    }
});

server.listen(443);

3.8 위험 기반 인증

상황에 따라 인증 요구사항을 동적으로 조정합니다:

// 위험 기반 인증 엔진
class RiskBasedAuth {
    async calculateRiskScore(context: AuthContext): Promise<number> {
        let score = 0;

        // 위치 위험
        if (!this.isKnownLocation(context.ipAddress)) {
            score += 0.2;
        }
        if (this.isHighRiskCountry(context.country)) {
            score += 0.3;
        }

        // 기기 위험
        if (!this.isKnownDevice(context.deviceFingerprint)) {
            score += 0.2;
        }

        // 시간 위험
        if (!this.isNormalAccessTime(context.userId, context.timestamp)) {
            score += 0.1;
        }

        // 행동 위험
        const behaviorScore = await this.analyzeBehavior(context);
        score += behaviorScore;

        return Math.min(score, 1.0);
    }

    determineAuthRequirements(riskScore: number): AuthRequirements {
        if (riskScore < 0.3) {
            return { factors: 1, methods: ['password'] };
        } else if (riskScore < 0.6) {
            return { factors: 2, methods: ['password', 'totp', 'push'] };
        } else {
            return {
                factors: 2,
                methods: ['password', 'webauthn'],
                additionalVerification: true
            };
        }
    }
}
적응형 인증: 위험 기반 인증은 보안과 사용자 경험 사이의 균형을 맞춥니다. 위험이 낮은 상황에서는 간단한 인증을, 위험이 높은 상황에서는 강화된 인증을 요구합니다.
