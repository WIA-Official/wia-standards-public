Chapter 2: Architecture and Access Models

The architecture of an access control system determines its scalability, performance, security, and manageability. This chapter explores the architectural patterns and access models defined by the WIA-ACCESS_CONTROL standard, with particular focus on Role-Based Access Control (RBAC) and Attribute-Based Access Control (ABAC).

WIA-ACCESS_CONTROL Reference Architecture

The WIA-ACCESS_CONTROL standard defines a modular, extensible architecture that can be adapted to various organizational needs. The reference architecture separates concerns into distinct components that communicate through well-defined interfaces.

Core Architectural Components

Policy Enforcement Point (PEP)

The Policy Enforcement Point serves as the gateway for all access requests. It intercepts requests from applications, forwards them to the PDP for evaluation, and enforces the resulting decisions. The PEP must be positioned strategically to ensure no bypass is possible.

class PolicyEnforcementPoint {
    private pdp: PolicyDecisionPoint;
    private cache: DecisionCache;
    private auditor: AuditLogger;

    constructor(config: PEPConfig) {
        this.pdp = new PolicyDecisionPoint(config.pdpEndpoint);
        this.cache = new DecisionCache(config.cacheTTL);
        this.auditor = new AuditLogger(config.auditConfig);
    }

    async enforceAccess(request: AccessRequest): Promise {
        // Check cache first for performance
        const cachedDecision = this.cache.get(request.getCacheKey());
        if (cachedDecision && !cachedDecision.expired) {
            this.auditor.logCachedDecision(request, cachedDecision);
            return this.enforce(cachedDecision);
        }

        // Request decision from PDP
        const decision = await this.pdp.evaluate(request);

        // Cache the decision
        this.cache.set(request.getCacheKey(), decision);

        // Log the decision
        this.auditor.logDecision(request, decision);

        // Enforce the decision
        return this.enforce(decision);
    }

    private enforce(decision: PolicyDecision): EnforcementResult {
        if (decision.effect === 'Permit') {
            return { allowed: true, obligations: decision.obligations };
        } else {
            return { allowed: false, reason: decision.reason };
        }
    }
}

Policy Decision Point (PDP)

The Policy Decision Point is the brain of the access control system. It evaluates access requests against policies and renders authorization decisions. The PDP must handle complex policy logic efficiently while maintaining consistency.

Performance Consideration: The PDP is often the bottleneck in access control systems. WIA-ACCESS_CONTROL recommends implementing decision caching, policy indexing, and load balancing to ensure responsive authorization decisions even under heavy load.

Policy Administration Point (PAP)

The Policy Administration Point provides interfaces for creating, editing, testing, and deploying access control policies. It should include workflow capabilities for policy approval, version control, and rollback mechanisms.

Policy Information Point (PIP)

The Policy Information Point retrieves attribute values needed for policy evaluation. It interfaces with identity providers, HR systems, asset databases, and other authoritative sources to gather subject, object, and environmental attributes.

Role-Based Access Control (RBAC)

RBAC is the most widely deployed access control model in enterprise environments. It simplifies administration by assigning permissions to roles rather than directly to users. The WIA-ACCESS_CONTROL standard defines a hierarchical RBAC model with support for constraints.

RBAC Components

  • Users: Human individuals or automated processes that need access
  • Roles: Named collections of permissions that reflect job functions
  • Permissions: Approved operations on specific resources
  • Sessions: Mappings between users and activated roles
// WIA-ACCESS_CONTROL RBAC Model Implementation
interface RBACModel {
    // Core RBAC
    users: Set<User>;
    roles: Set<Role>;
    permissions: Set<Permission>;

    // User-Role Assignment
    userRoleAssignment: Map<User, Set<Role>>;

    // Permission-Role Assignment
    permissionRoleAssignment: Map<Role, Set<Permission>>;

    // Hierarchical RBAC
    roleHierarchy: Map<Role, Set<Role>>; // Senior roles inherit from junior roles

    // Constrained RBAC
    staticSeparationOfDuty: SSDConstraint[];
    dynamicSeparationOfDuty: DSDConstraint[];
}

// Role Definition
interface Role {
    id: string;
    name: string;
    description: string;
    inheritsFrom: Role[];
    permissions: Permission[];
    constraints: RoleConstraint[];
}

RBAC Constraints

Constraints add security controls to the RBAC model by limiting role assignments and activations.

Separation of Duties: Static Separation of Duty (SSD) prevents users from being assigned conflicting roles (e.g., a user cannot be both an auditor and an administrator). Dynamic Separation of Duty (DSD) prevents users from activating conflicting roles in the same session. 
// Separation of Duty Constraints
class SeparationOfDutyEngine {
    private ssdConstraints: SSDConstraint[] = [];
    private dsdConstraints: DSDConstraint[] = [];

    addSSDConstraint(constraint: SSDConstraint): void {
        // Example: User cannot be both 'accountant' and 'auditor'
        this.ssdConstraints.push(constraint);
    }

    addDSDConstraint(constraint: DSDConstraint): void {
        // Example: User cannot activate 'purchaser' and 'approver' in same session
        this.dsdConstraints.push(constraint);
    }

    checkSSD(user: User, newRole: Role): boolean {
        const userRoles = this.getUserRoles(user);

        for (const constraint of this.ssdConstraints) {
            const conflictingRoles = constraint.roles;
            const cardinality = constraint.cardinality;

            const matchingRoles = userRoles.filter(r =>
                conflictingRoles.includes(r) || r === newRole
            );

            if (matchingRoles.length >= cardinality) {
                return false; // Constraint violated
            }
        }
        return true;
    }

    checkDSD(session: Session, roleToActivate: Role): boolean {
        const activeRoles = session.getActiveRoles();

        for (const constraint of this.dsdConstraints) {
            const conflictingRoles = constraint.roles;
            const cardinality = constraint.cardinality;

            const matchingRoles = activeRoles.filter(r =>
                conflictingRoles.includes(r) || r === roleToActivate
            );

            if (matchingRoles.length >= cardinality) {
                return false; // Constraint violated
            }
        }
        return true;
    }
}

Attribute-Based Access Control (ABAC)

ABAC provides fine-grained access control by evaluating attributes of subjects, objects, actions, and the environment. This model is more flexible than RBAC and can express complex, context-aware policies.

ABAC Components

Component Description Examples
Subject Attributes Properties of the requesting entity Role, department, clearance, location
Object Attributes Properties of the target resource Classification, owner, type, sensitivity
Action Attributes Properties of the requested operation Read, write, delete, approve
Environment Attributes Contextual conditions Time, location, threat level, device trust 
// WIA-ACCESS_CONTROL ABAC Policy Language
interface ABACPolicy {
    id: string;
    name: string;
    description: string;
    target: PolicyTarget;
    rules: PolicyRule[];
    combiningAlgorithm: CombiningAlgorithm;
}

interface PolicyRule {
    id: string;
    effect: 'Permit' | 'Deny';
    condition: Expression;
    obligations?: Obligation[];
}

// Example ABAC Policy
const documentAccessPolicy: ABACPolicy = {
    id: 'policy-doc-001',
    name: 'Confidential Document Access',
    description: 'Controls access to confidential documents',
    target: {
        resources: { type: 'document', classification: 'confidential' }
    },
    combiningAlgorithm: 'deny-unless-permit',
    rules: [
        {
            id: 'rule-1',
            effect: 'Permit',
            condition: {
                and: [
                    { attr: 'subject.clearance', op: '>=', value: 'confidential' },
                    { attr: 'subject.department', op: '==', value: 'object.owner_department' },
                    { attr: 'environment.network', op: 'in', value: ['corporate', 'vpn'] },
                    { attr: 'environment.time', op: 'between', value: ['08:00', '18:00'] }
                ]
            },
            obligations: [
                { type: 'log', params: { level: 'info', message: 'Confidential access granted' } },
                { type: 'notify', params: { recipient: 'security-team', condition: 'after-hours' } }
            ]
        },
        {
            id: 'rule-2-emergency',
            effect: 'Permit',
            condition: {
                and: [
                    { attr: 'subject.role', op: 'in', value: ['executive', 'security-admin'] },
                    { attr: 'request.justification', op: 'exists' }
                ]
            },
            obligations: [
                { type: 'log', params: { level: 'alert', message: 'Emergency access invoked' } }
            ]
        }
    ]
};

Hybrid RBAC-ABAC Model

The WIA-ACCESS_CONTROL standard recommends a hybrid approach that combines the administrative simplicity of RBAC with the fine-grained flexibility of ABAC. This model uses RBAC for coarse-grained access control and ABAC for context-aware refinements.

Best Practice: Start with RBAC for basic access control structure, then layer ABAC policies for scenarios requiring dynamic, context-aware decisions. This approach balances manageability with flexibility. 
// Hybrid RBAC-ABAC Implementation
class HybridAccessControl {
    private rbacEngine: RBACEngine;
    private abacEngine: ABACEngine;

    async checkAccess(request: AccessRequest): Promise<AccessDecision> {
        // Step 1: RBAC check - does the user's role permit this action?
        const rbacDecision = await this.rbacEngine.evaluate({
            subject: request.subject,
            resource: request.resource,
            action: request.action
        });

        if (rbacDecision.effect === 'Deny') {
            return {
                effect: 'Deny',
                reason: 'RBAC: Insufficient role permissions',
                evaluatedBy: 'RBAC'
            };
        }

        // Step 2: ABAC refinement - check contextual constraints
        const abacDecision = await this.abacEngine.evaluate({
            subject: request.subject,
            resource: request.resource,
            action: request.action,
            environment: request.environment
        });

        if (abacDecision.effect === 'Deny') {
            return {
                effect: 'Deny',
                reason: `ABAC: ${abacDecision.reason}`,
                evaluatedBy: 'ABAC'
            };
        }

        // Both RBAC and ABAC permit - access granted
        return {
            effect: 'Permit',
            obligations: [...rbacDecision.obligations, ...abacDecision.obligations],
            evaluatedBy: 'Hybrid'
        };
    }
}

Deployment Architectures

Centralized Architecture

All access control decisions are made by a central PDP. This simplifies policy management but can become a bottleneck and single point of failure.

Distributed Architecture

Multiple PDPs are deployed across the infrastructure. Policies are synchronized across instances. This improves performance and availability but increases complexity.

Edge Architecture

Lightweight PEPs with local decision-making capabilities are deployed at the edge. Complex decisions are escalated to central PDPs. This optimizes latency for common cases.

Summary

This chapter has explored the architectural foundations of access control systems according to the WIA-ACCESS_CONTROL standard. We examined the reference architecture with its core components (PEP, PDP, PAP, PIP, PRP), dove deep into RBAC and ABAC models, and discussed hybrid approaches and deployment patterns.

Key Takeaways:
  • The WIA-ACCESS_CONTROL architecture separates concerns into distinct, well-defined components
  • RBAC provides administrative simplicity through role-based permission assignment
  • ABAC enables fine-grained, context-aware access control through attribute evaluation
  • Hybrid models combine RBAC and ABAC for optimal balance of manageability and flexibility
  • Deployment architecture should match organizational scale and performance requirements
