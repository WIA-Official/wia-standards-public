Chapter 4: Authorization Policies

Authorization determines what authenticated subjects are permitted to do. This chapter explores the policy languages, permission models, and role structures that form the core of the WIA-ACCESS_CONTROL authorization framework. Effective authorization policies balance security requirements with operational needs while remaining manageable at scale.

The Authorization Decision Process

When an authenticated subject requests access to a resource, the authorization system must determine whether to permit or deny the request. This decision process involves evaluating the request against applicable policies, combining multiple policy results, and returning a final decision with any associated obligations.

WIA Policy Language (WPL)

The WIA-ACCESS_CONTROL standard defines a policy language called WPL (WIA Policy Language) that enables precise expression of authorization rules. WPL supports conditions, obligations, and advice while remaining human-readable.

Policy Structure

// WIA Policy Language (WPL) Structure
interface WPLPolicy {
    id: string;
    version: string;
    name: string;
    description: string;

    // Target: When does this policy apply?
    target: {
        subjects?: SubjectMatch[];
        resources?: ResourceMatch[];
        actions?: ActionMatch[];
        environments?: EnvironmentMatch[];
    };

    // Rules: What decisions does the policy contain?
    rules: WPLRule[];

    // Combining Algorithm: How are rule results combined?
    ruleCombining: CombiningAlgorithm;

    // Obligations: What must happen if policy permits/denies?
    obligations?: Obligation[];

    // Advice: What should happen (non-mandatory)?
    advice?: Advice[];
}

interface WPLRule {
    id: string;
    effect: 'Permit' | 'Deny';
    description?: string;
    condition?: Expression;
    obligations?: Obligation[];
}

type CombiningAlgorithm =
    | 'deny-overrides'      // Any Deny wins
    | 'permit-overrides'    // Any Permit wins
    | 'first-applicable'    // First matching rule wins
    | 'only-one-applicable' // Error if multiple apply
    | 'deny-unless-permit'  // Default Deny
    | 'permit-unless-deny'; // Default Permit

Policy Examples

// Example: Document Access Policy
const documentAccessPolicy: WPLPolicy = {
    id: 'pol-doc-access-001',
    version: '1.0.0',
    name: 'Document Access Control',
    description: 'Controls access to organizational documents',

    target: {
        resources: [{ type: 'document' }]
    },

    ruleCombining: 'deny-overrides',

    rules: [
        // Rule 1: Owners have full access
        {
            id: 'rule-owner-full',
            effect: 'Permit',
            description: 'Document owners have full access',
            condition: {
                equals: [
                    { attribute: 'subject.id' },
                    { attribute: 'resource.owner' }
                ]
            }
        },

        // Rule 2: Department members can read
        {
            id: 'rule-dept-read',
            effect: 'Permit',
            description: 'Same department can read',
            condition: {
                and: [
                    {
                        equals: [
                            { attribute: 'subject.department' },
                            { attribute: 'resource.department' }
                        ]
                    },
                    {
                        in: [
                            { attribute: 'action' },
                            { value: ['read', 'list'] }
                        ]
                    }
                ]
            }
        },

        // Rule 3: Classified documents require clearance
        {
            id: 'rule-classified',
            effect: 'Deny',
            description: 'Deny if insufficient clearance',
            condition: {
                and: [
                    {
                        greaterThan: [
                            { attribute: 'resource.classificationLevel' },
                            { attribute: 'subject.clearanceLevel' }
                        ]
                    }
                ]
            }
        }
    ],

    obligations: [
        {
            id: 'obl-audit',
            fulfillOn: 'Permit',
            action: 'log',
            parameters: {
                level: 'info',
                includeAttributes: ['subject.id', 'resource.id', 'action']
            }
        }
    ]
};

Permission Models

Permissions define what actions subjects can perform on resources. The WIA-ACCESS_CONTROL standard supports multiple permission models to accommodate different organizational needs.

Explicit Permission Model

In the explicit model, permissions are directly assigned to subjects or roles. This model is straightforward but can become unwieldy with many subjects and resources.

// Explicit Permission Assignment
interface ExplicitPermission {
    subject: string;
    resource: string;
    actions: string[];
    constraints?: Constraint[];
    validFrom?: Date;
    validUntil?: Date;
}

const explicitPermissions: ExplicitPermission[] = [
    {
        subject: 'user:alice',
        resource: 'document:budget-2025',
        actions: ['read', 'write', 'delete'],
        constraints: [
            { type: 'time', startHour: 9, endHour: 17 }
        ]
    },
    {
        subject: 'role:finance-manager',
        resource: 'database:financial-records',
        actions: ['read', 'write'],
        validFrom: new Date('2025-01-01'),
        validUntil: new Date('2025-12-31')
    }
];

Hierarchical Permission Model

The hierarchical model organizes resources and permissions in a tree structure. Permissions on parent nodes cascade to children unless explicitly overridden.

Permission Inheritance: In a hierarchical model, granting read access to /projects automatically grants read access to /projects/alpha, /projects/beta, and all descendant resources. This simplifies administration but requires careful planning to avoid over-permissioning. 
// Hierarchical Resource Permissions
class HierarchicalPermissionEngine {
    private resourceTree: ResourceTree;

    async checkPermission(
        subject: Subject,
        resource: Resource,
        action: Action
    ): Promise<PermissionResult> {
        // Get the resource path
        const path = this.resourceTree.getPath(resource);

        // Check permissions from root to leaf
        let effectivePermission: Permission | null = null;

        for (const node of path) {
            const permission = await this.getPermissionAtNode(subject, node, action);

            if (permission) {
                if (permission.override === 'explicit') {
                    // Explicit override stops inheritance
                    return { permitted: permission.effect === 'allow' };
                }
                effectivePermission = permission;
            }
        }

        // Check for deny-by-default
        if (!effectivePermission) {
            return { permitted: false, reason: 'No applicable permission found' };
        }

        return { permitted: effectivePermission.effect === 'allow' };
    }
}

// Example resource hierarchy
const resourceHierarchy = {
    '/': {
        permissions: [{ role: 'authenticated', actions: ['list'] }]
    },
    '/projects': {
        permissions: [{ role: 'employee', actions: ['read', 'list'] }]
    },
    '/projects/alpha': {
        permissions: [
            { role: 'alpha-team', actions: ['read', 'write', 'delete'] },
            { role: 'employee', actions: ['read'], override: 'explicit' }
        ]
    },
    '/projects/alpha/confidential': {
        permissions: [
            { role: 'alpha-lead', actions: ['read', 'write'] }
            // Employees cannot read even though they can read parent
        ]
    }
};

Role Design Patterns

Effective role design is crucial for manageable access control. The WIA-ACCESS_CONTROL standard recommends several patterns for organizing roles.

Job Function Roles

Job function roles correspond to organizational positions. They group permissions needed to perform specific job duties.

Role Description Key Permissions
software-engineer Develops software code.read, code.write, deploy.staging
qa-engineer Tests software code.read, tests.run, bugs.create
devops-engineer Manages infrastructure infra.manage, deploy.all, monitoring.admin
security-analyst Security operations logs.read, alerts.manage, audit.run

Privilege Level Roles

Privilege level roles define escalating access levels, typically used with the principle of least privilege.

Composite Roles

Composite roles combine multiple base roles for users with cross-functional responsibilities.

Best Practice: Keep base roles atomic and focused. Create composite roles by combining base roles rather than duplicating permissions. This reduces maintenance burden and ensures consistency. 
// Composite Role Definition
interface CompositeRole {
    id: string;
    name: string;
    baseRoles: string[];
    additionalPermissions?: Permission[];
    constraints?: Constraint[];

Dynamic Authorization

Static role assignments are insufficient for modern, dynamic environments. WIA-ACCESS_CONTROL supports dynamic authorization based on runtime context.

// Dynamic Authorization Engine
class DynamicAuthorizationEngine {
    async authorize(request: AuthorizationRequest): Promise<AuthorizationDecision> {
        // Gather dynamic context
        const context = await this.gatherContext(request);

        // Evaluate dynamic policies
        const decisions = await Promise.all([
            this.evaluateTimeBasedPolicies(request, context),
            this.evaluateLocationBasedPolicies(request, context),
            this.evaluateRiskBasedPolicies(request, context),
            this.evaluateRelationshipBasedPolicies(request, context)
        ]);

        // Combine decisions
        return this.combineDecisions(decisions);
    }

    private async evaluateRiskBasedPolicies(
        request: AuthorizationRequest,
        context: AuthorizationContext
    ): Promise<PolicyDecision> {
        const riskScore = await this.riskEngine.calculate({
            subject: request.subject,
            resource: request.resource,
            action: request.action,
            context: context
        });

        // High-risk actions require additional controls
        if (riskScore > 0.7) {
            if (!context.mfaVerified) {
                return {
                    effect: 'Deny',
                    reason: 'High-risk action requires MFA verification'
                };
            }

            if (request.action === 'delete' && !context.approvalPresent) {
                return {
                    effect: 'Deny',
                    reason: 'High-risk deletion requires approval',
                    obligations: [
                        { action: 'request-approval', params: { approvers: 'managers' } }
                    ]
                };
            }
        }

        return { effect: 'NotApplicable' };
    }

    private async evaluateRelationshipBasedPolicies(
        request: AuthorizationRequest,
        context: AuthorizationContext
    ): Promise<PolicyDecision> {
        // Check if subject has relationship to resource
        const relationships = await this.relationshipService.getRelationships(
            request.subject.id,
            request.resource.id
        );

        for (const rel of relationships) {
            if (rel.type === 'owner') {
                return { effect: 'Permit', reason: 'Owner access' };
            }
            if (rel.type === 'team-member' && request.action === 'read') {
                return { effect: 'Permit', reason: 'Team member read access' };
            }
        }

        return { effect: 'NotApplicable' };
    }
}

Policy Testing and Validation

Authorization policies must be thoroughly tested before deployment. Errors in policies can lead to security breaches or operational disruptions.

Testing Imperative: Always test policies in a non-production environment with comprehensive test cases covering both permitted and denied scenarios. Policy errors can have severe consequences. 
// Policy Testing Framework
class PolicyTestSuite {
    private policyEngine: PolicyEngine;

    async runTests(policy: WPLPolicy, testCases: TestCase[]): Promise<TestReport> {
        const results: TestResult[] = [];

        for (const testCase of testCases) {
            const actual = await this.policyEngine.evaluate({
                subject: testCase.subject,
                resource: testCase.resource,
                action: testCase.action,
                environment: testCase.environment
            });

            const passed = actual.effect === testCase.expectedEffect;

            results.push({
                testCase: testCase,
                actual: actual,
                passed: passed,
                message: passed ? 'PASS' :
                    `Expected ${testCase.expectedEffect}, got ${actual.effect}`
            });
        }

        return this.generateReport(policy, results);
    }
}
