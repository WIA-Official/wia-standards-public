CHAPTER 2

Mathematical Foundations

Understanding the Mathematics Behind HE

Homomorphic encryption may seem like magic, but it's grounded in rigorous mathematics. Understanding these foundations is essential for properly implementing, optimizing, and securing HE systems. This chapter explores the mathematical structures that make homomorphic encryption possible, from basic algebra to advanced lattice theory.

Don't worry if you're not a mathematician—we'll build up concepts gradually, providing intuition alongside formal definitions. By the end, you'll understand why HE is secure, how operations work, and what parameters mean.

Algebraic Structures: The Foundation

Groups, Rings, and Fields

Homomorphic encryption operates within algebraic structures that define how numbers interact:

Group

A set with one operation (like addition) that satisfies:

  • Closure: a + b is in the group
  • Associativity: (a + b) + c = a + (b + c)
  • Identity: There exists 0 where a + 0 = a
  • Inverse: For each a, exists -a where a + (-a) = 0

Example: Integers under addition (ℤ, +)

Ring

A set with two operations (addition and multiplication) where:

  • Forms a group under addition
  • Multiplication is associative and distributive over addition
  • Has multiplicative identity (1)

Example: Integers (ℤ, +, ×)

Field

A ring where every non-zero element has a multiplicative inverse.

Example: Rational numbers (ℚ), Real numbers (ℝ)

Modular Arithmetic

HE schemes work in finite structures, using modular arithmetic—arithmetic "with wraparound":

a ≡ b (mod n) means n divides (a - b)

Example with mod 7:

8 ≡ 1 (mod 7)     because 8 - 1 = 7
15 ≡ 1 (mod 7)    because 15 - 1 = 14 = 2 × 7
3 + 5 ≡ 1 (mod 7) because 8 ≡ 1 (mod 7)

Modular arithmetic creates finite rings like ℤn = {0, 1, 2, ..., n-1} with addition and multiplication mod n.

Polynomial Rings: The Core Structure

What is a Polynomial Ring?

Modern HE schemes don't encrypt single numbers—they encrypt polynomials. This provides massive efficiency gains through SIMD-style batching.

R = ℤ[x] / (f(x))

This notation means: polynomials with integer coefficients, where f(x) = 0. For HE, we typically use:

R = ℤ[x] / (xn + 1)

Where n is a power of 2 (e.g., 4096, 8192, 16384).

Polynomial Operations

Let's see how polynomials work in this ring with n = 4:

R = ℤ[x] / (x⁴ + 1)

Polynomials in R:
a(x) = 5 + 3x + 7x² + 2x³
b(x) = 1 + 4x + 6x² + 8x³

Addition (coefficient-wise):
a(x) + b(x) = 6 + 7x + 13x² + 10x³

Multiplication (with reduction by x⁴ + 1):
Step 1: Multiply normally
a(x) × b(x) = 5 + 23x + 61x² + 96x³ + 74x⁴ + 86x⁵ + 68x⁶ + 16x⁷

Step 2: Reduce using x⁴ = -1:
x⁴ → -1
x⁵ → -x
x⁶ → -x²
x⁷ → -x³

Result: (5 - 74) + (23 - 86)x + (61 - 68)x² + (96 - 16)x³
      = -69 + (-63)x + (-7)x² + 80x³

Why This Structure?

The polynomial ring R = ℤ[x]/(xn + 1) has special properties:

  • Batching: A single polynomial can encode n/2 values using Chinese Remainder Theorem
  • FFT-friendly: Polynomial multiplication uses Fast Fourier Transform, making it O(n log n) instead of O(n²)
  • Security: The structure supports hard lattice problems (RLWE)
  • Homomorphism: Operations on polynomials translate to operations on encoded values

Ring Learning With Errors (RLWE)

The Security Foundation

Modern HE schemes base security on the Ring Learning With Errors problem—a hard problem even for quantum computers:

RLWE Problem

Given: Pairs (ai, bi) where:

  • ai ∈ Rq is uniformly random
  • bi = ai · s + ei (mod q)
  • s ∈ Rq is a secret polynomial
  • ei is small error from a Gaussian distribution

Find: The secret polynomial s

This problem is believed to be as hard as worst-case lattice problems, providing strong security guarantees.

Error Distributions

The "error" e is sampled from a discrete Gaussian distribution χ with standard deviation σ:

P(e = k) ∝ exp(-k²/2σ²)

Typical values: σ = 3.2, giving errors mostly in range [-10, 10].

Standard Deviation (σ) Typical Error Range Security Impact
3.2 [-10, 10] Standard security
6.4 [-20, 20] Higher security
1.6 [-5, 5] Lower security (faster)

BFV Scheme: Detailed Mathematics

Parameter Setup

The BFV (Brakerski-Fan-Vercauteren) scheme operates with these parameters:

  • n: Polynomial degree (power of 2, e.g., 8192)
  • q: Ciphertext modulus (large integer, e.g., 2²¹⁸)
  • t: Plaintext modulus (smaller integer, e.g., 1024)
  • χ: Error distribution (discrete Gaussian, σ = 3.2)
  • Δ = ⌊q/t⌋: Scaling factor

Key Generation

1. Sample secret key:
   s ← χ (polynomial with small coefficients)

2. Sample random polynomial:
   a ← R_q (uniformly random)

3. Sample error:
   e ← χ

4. Compute public key element:
   b = -a·s - e (mod q)

5. Public key: pk = (b, a)
   Secret key: sk = s

Encryption

To encrypt a message m ∈ Rt:

1. Sample small random polynomial:
   r ← χ

2. Sample small errors:
   e₀, e₁ ← χ

3. Compute ciphertext components:
   c₀ = b·r + e₀ + Δ·m (mod q)
   c₁ = a·r + e₁ (mod q)

4. Output ciphertext: ct = (c₀, c₁)

Decryption

1. Compute with secret key:
   noisy = c₀ + c₁·s (mod q)

2. Observe what happens:
   noisy = (b·r + e₀ + Δ·m) + (a·r + e₁)·s
         = (-a·s - e)·r + e₀ + Δ·m + a·r·s + e₁·s
         = -a·r·s - e·r + e₀ + Δ·m + a·r·s + e₁·s
         = Δ·m + (e₀ + e₁·s - e·r)
         = Δ·m + noise

3. Scale down and round:
   scaled = ⌊(t/q)·noisy⌉ (mod t)
         ≈ ⌊t·(Δ·m)/q⌉
         ≈ ⌊m + small_error⌉
         = m (if noise is small enough)

The Magic of Noise Cancellation

Notice how a·r·s terms cancel! This is the beautiful algebraic structure that makes decryption work. The remaining noise (e₀ + e₁·s - e·r) is small because all error terms are small.

Homomorphic Operations

Addition

Adding two ciphertexts is simple component-wise addition:

ct₁ = (c₀⁽¹⁾, c₁⁽¹⁾) encrypts m₁
ct₂ = (c₀⁽²⁾, c₁⁽²⁾) encrypts m₂

ct_sum = (c₀⁽¹⁾ + c₀⁽²⁾, c₁⁽¹⁾ + c₁⁽²⁾)

Why it works:
(c₀⁽¹⁾ + c₀⁽²⁾) + (c₁⁽¹⁾ + c₁⁽²⁾)·s
= (c₀⁽¹⁾ + c₁⁽¹⁾·s) + (c₀⁽²⁾ + c₁⁽²⁾·s)
= Δ·m₁ + noise₁ + Δ·m₂ + noise₂
= Δ·(m₁ + m₂) + (noise₁ + noise₂)

Noise grows additively—slowly!

Multiplication

Multiplication is more involved:

ct₁ = (c₀⁽¹⁾, c₁⁽¹⁾) encrypts m₁
ct₂ = (c₀⁽²⁾, c₁⁽²⁾) encrypts m₂

Compute:
d₀ = c₀⁽¹⁾·c₀⁽²⁾
d₁ = c₀⁽¹⁾·c₁⁽²⁾ + c₁⁽¹⁾·c₀⁽²⁾
d₂ = c₁⁽¹⁾·c₁⁽²⁾

Result: ct_mult = (d₀, d₁, d₂) has 3 components!

To decrypt: d₀ + d₁·s + d₂·s² ≈ Δ²·m₁·m₂/q

This creates two problems: 3 components (instead of 2) and wrong scaling (Δ² instead of Δ).

Relinearization

Relinearization converts 3-component ciphertext back to 2 components using an evaluation key:

Evaluation key: evk encrypts s² under s

Apply relinearization:
ct' = (d₀, d₁) + Decrypt(evk, d₂)

Result: 2-component ciphertext encrypting m₁·m₂

Rescaling

After multiplication, scaling is Δ² instead of Δ. Rescaling fixes this:

ct_rescaled = ⌊(t/q)·ct_mult⌉ (mod q')

where q' = q/p for some prime p

Noise Budget Analysis

Tracking Noise Growth

Every ciphertext has a "noise budget" B—how much noise it can tolerate:

Operation Noise Growth Budget Impact
Fresh encryption σ√n Initial: ~log₂(q/t) bits
Addition max(noise₁, noise₂) + small -1 bit typically
Multiplication noise₁ × noise₂ Budget halves
Relinearization +small noise -few bits

Multiplication Depth

The maximum multiplication depth is limited by noise budget:

Initial budget: B₀ = log₂(q/t) bits

After k multiplications:
B_k ≈ B₀ - k·log₂(B₀) - k·log₂(n) - overhead

Example with B₀ = 150 bits, n = 8192:
Depth 1: ~137 bits
Depth 2: ~124 bits
Depth 3: ~111 bits
Depth 4: ~98 bits
Depth 5: ~85 bits (getting risky)
Depth 6: ~72 bits (may fail)

Parameter Selection

Security Level

Parameters must provide sufficient security against known attacks. Security level λ in bits means 2λ operations to break:

Security Level Typical n Typical log₂(q) Use Case
128-bit 4096-8192 109-218 Standard applications
192-bit 16384 438-881 High security
256-bit 32768 ~1700 Extreme security (slow)

Balancing Performance and Security

Parameter selection involves trade-offs:

Parameter Trade-offs

  • Larger n: Better security, more batching slots, but slower operations
  • Larger q: More noise budget (deeper computation), but slower and less secure
  • Larger t: Larger message space, but less noise budget
  • Smaller σ: Faster operations, but weaker security

Chinese Remainder Theorem and Batching

SIMD-Style Parallelism

The Chinese Remainder Theorem (CRT) allows encoding multiple values in a single ciphertext:

One polynomial encrypts n/2 values simultaneously

This provides massive efficiency—operations on one ciphertext affect all batched values in parallel.

How Batching Works

1. Choose plaintext modulus t = 1 (mod 2n)

2. Factor x^n + 1 into irreducible polynomials mod t:
   x^n + 1 = F₁(x) · F₂(x) · ... · F_k(x) (mod t)

3. Use CRT to map:
   m(x) ↔ (m₁, m₂, ..., m_k)

4. Each slot holds independent value

Example with n = 8, t = 17:
Can encode 4 values per ciphertext
Addition/multiplication work component-wise

Chapter Summary

  • HE operates in polynomial rings R = ℤ[x]/(xn + 1) with modular arithmetic
  • Security based on Ring-LWE problem: finding secret from noisy linear combinations
  • BFV encryption adds noise-masked message Δ·m to RLWE instance
  • Decryption cancels algebraic terms, leaves Δ·m + small noise
  • Addition grows noise additively (slow), multiplication multiplicatively (fast)
  • Relinearization reduces ciphertext size after multiplication
  • Noise budget limits multiplication depth before decryption fails
  • Parameter selection balances security, performance, and computation depth
  • Chinese Remainder Theorem enables SIMD batching of n/2 values
  • Typical parameters: n = 8192, log₂(q) = 218, t = 1024 for 128-bit security
