CHAPTER 5

Performance Optimization

The Performance Challenge

Homomorphic encryption's primary barrier to adoption has always been performance. HE operations can be 100-10,000x slower than plaintext equivalents. However, through careful optimization, proper parameter selection, and hardware acceleration, HE can achieve practical performance for many real-world applications. This chapter explores proven techniques to maximize HE performance.

Batching and SIMD

CRT Batching in BFV/BGV

Process multiple values simultaneously using the Chinese Remainder Theorem:

// Without batching: encrypt 1000 values → 1000 ciphertexts
// With batching: encrypt 1000 values → 1 ciphertext (if n/2 ≥ 1000)

// SEAL batching example
BatchEncoder encoder(context);
vector values(poly_modulus_degree / 2, 0);
values[0] = 1;
values[1] = 2;
// ... fill all slots ...
values[4095] = 4096;

Plaintext plain;
encoder.encode(values, plain);
Ciphertext cipher;
encryptor.encrypt(plain, cipher);

// One operation affects ALL values
evaluator.square(cipher, cipher);  // Squares all 4096 values!

SIMD Operations

Design algorithms to exploit batching:

Example: Encrypted Database Sum

// Naive: n ciphertexts, n-1 additions
total = ct[0]
for i in 1..n:
    total = add(total, ct[i])

// Optimized: batch into fewer ciphertexts, tree reduction
// If n=4096 values, batch into 1 ciphertext
// Use rotations and additions to sum
result = ct[0]
for shift in [1, 2, 4, 8, ..., 2048]:
    rotated = rotate(ct[0], shift)
    result = add(result, rotated)
// Now result[0] contains sum of all values!
// Reduction: log₂(4096) = 12 operations vs 4095

Batching Strategies

  • Horizontal batching: Same operation on multiple data points (e.g., matrix row)
  • Vertical batching: Different operations packed together
  • Hybrid batching: Combine both for maximum efficiency
  • Padding: Fill unused slots with zeros (minimal overhead)

Number Theoretic Transform (NTT)

Why NTT Matters

Polynomial multiplication is the heart of HE. NTT reduces complexity from O(n²) to O(n log n):

c(x) = a(x) × b(x) mod (xn + 1) 
// Naive multiplication: O(n²)
for i in 0..n:
    for j in 0..n:
        result[(i+j) mod n] += a[i] * b[j]

// NTT multiplication: O(n log n)
A = NTT(a)
B = NTT(b)
C = A ⊙ B  // Point-wise multiplication O(n)
c = INTT(C)

NTT Optimization Techniques

  • Precomputation: Cache twiddle factors (roots of unity)
  • Lazy reduction: Delay modular reduction until necessary
  • Specialized primes: Choose q for fast modular arithmetic
  • SIMD instructions: Vectorize NTT butterfly operations
// Optimized NTT with AVX2
__m256i ntt_butterfly_avx2(__m256i a, __m256i b, __m256i w, __m256i q) {
    __m256i t = _mm256_mul_epu32(b, w);
    t = _mm256_rem_epu32(t, q);  // Montgomery reduction
    __m256i u = _mm256_add_epi64(a, t);
    __m256i v = _mm256_sub_epi64(a, t);
    return _mm256_blend_epi32(u, v, 0xAA);
}

Algorithmic Optimizations

Minimize Multiplicative Depth

Multiplications are expensive. Restructure algorithms to reduce depth:

Example: Polynomial Evaluation

// Naive: f(x) = a₀ + a₁x + a₂x² + a₃x³ + a₄x⁴
// Depth: 4 (need x, x², x³, x⁴)
result = a₀ + a₁*x + a₂*(x*x) + a₃*(x*x*x) + a₄*(x*x*x*x)

// Horner's method: f(x) = a₀ + x(a₁ + x(a₂ + x(a₃ + x·a₄)))
// Depth: 4 but sequential (can't parallelize)
result = a₀
for i in [1, 2, 3, 4]:
    result = a_i + x * result

// Paterson-Stockmeyer: balanced tree
// Depth: 2 (optimal!)
y = x²
result = (a₀ + a₁*x) + y*(a₂ + a₃*x) + y²*a₄

Plaintext-Ciphertext Operations

When possible, keep one operand as plaintext:

// Slow: encrypt constants, multiply ciphertexts
ct_5 = encrypt(5)
result = multiply(ct_x, ct_5)  // Full multiplication

// Fast: multiply ciphertext by plaintext constant
result = multiply_plain(ct_x, 5)  // 10x faster!

Approximation Techniques

For CKKS and approximate computation:

  • Low-degree polynomials: Approximate activation functions
  • Iterative refinement: Start with low precision, refine if needed
  • Truncation: Drop least significant bits early
  • Quantization: Reduce bit width of values

Hardware Acceleration

CPU Optimization

SIMD Instructions

  • AVX2: 256-bit vectors, process 4 64-bit integers simultaneously
  • AVX-512: 512-bit vectors, 8 integers simultaneously
  • ARM NEON: SIMD for ARM processors

Typical speedup: 2-4x for NTT operations

// CUDA kernel for NTT butterfly
__global__ void ntt_layer_kernel(uint64_t* data, uint64_t* twiddles,
                                  uint64_t q, int layer) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    int m = 1 << layer;
    int k = idx / m;
    int j = idx % m;

    uint64_t w = twiddles[m + j];
    uint64_t u = data[2*k*m + j];
    uint64_t v = mulmod(data[2*k*m + j + m], w, q);

    data[2*k*m + j] = addmod(u, v, q);
    data[2*k*m + j + m] = submod(u, v, q);
}

Memory Optimization

Ciphertext Compression

// Standard ciphertext: 2 polynomials × n × log q bits
// Compressed: 1 polynomial + seed
// Size reduction: ~50%

// Seed-based randomness (for public key)
Ciphertext compress(Ciphertext ct, PublicKey pk) {
    // Keep c₀, compress c₁ using seed
    seed = extract_seed(ct.c1, pk);
    return CompressedCiphertext(ct.c0, seed);
}

Ciphertext decompress(CompressedCiphertext cct, PublicKey pk) {
    c1 = regenerate_from_seed(cct.seed, pk);
    return Ciphertext(cct.c0, c1);
}

Lazy Evaluation

Defer expensive operations until results are needed:

// Build computation graph, optimize before executing
graph = ComputationGraph()
x = graph.encrypt(data)
y = graph.add(x, x)
z = graph.multiply(y, y)
result = graph.add(z, x)

// Optimize graph
optimized = graph.optimize()  // Combine operations, reorder
output = optimized.execute()  // Execute efficiently

Cache Optimization

  • Data layout: Arrange polynomials for cache-friendly access
  • Blocking: Process data in cache-sized chunks
  • Prefetching: Load data before needed
  • Key reuse: Keep frequently-used keys in cache

Parallel Execution

Multi-threading

// Process multiple ciphertexts in parallel
#pragma omp parallel for
for (int i = 0; i < num_ciphertexts; i++) {
    results[i] = evaluate_circuit(ciphertexts[i]);
}

// Pipeline stages
Thread 1: Encrypt batch 1
Thread 2: Compute batch 0 (while Thread 1 encrypts)
Thread 3: Decrypt batch -1 (while others work)

Distributed Computing

Scale across multiple machines:

Map-Reduce Pattern for HE

// Map: Distribute ciphertexts to workers
workers = [Worker1, Worker2, ..., WorkerN]
for i, worker in enumerate(workers):
    worker.send(ciphertexts[i::N])  // Every Nth ciphertext

// Compute: Each worker processes its share
results = parallel_map(lambda ct: compute(ct), ciphertexts)

// Reduce: Aggregate results homomorphically
final = tree_reduce(lambda a, b: add(a, b), results)

Chapter Summary

  • HE performance bottlenecks: polynomial multiplication (NTT), relinearization, bootstrapping
  • Batching via CRT enables SIMD-style operations on n/2 values simultaneously
  • NTT reduces polynomial multiplication from O(n²) to O(n log n)
  • Parameter optimization crucial: balance security, depth, and performance
  • Algorithmic optimization: minimize multiplicative depth, use Horner's method, exploit plaintext-ciphertext operations
  • Hardware acceleration: SIMD (2-4x), GPU (10-30x), FPGA (5-10x), ASIC (100x potential)
  • Memory optimization: ciphertext compression, lazy evaluation, cache-friendly layouts
  • Parallelism: multi-threading for multiple ciphertexts, distributed computing for massive scale
  • Real-world speedups: 100-1000x achievable through combined optimizations
  • Modern HE is practical for many applications with proper optimization
