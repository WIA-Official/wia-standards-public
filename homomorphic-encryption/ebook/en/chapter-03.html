CHAPTER 3

Types of HE Schemes

The HE Landscape: Choosing the Right Scheme

Not all homomorphic encryption schemes are created equal. Each scheme offers different trade-offs between functionality, performance, and complexity. Understanding these differences is crucial for selecting the right tool for your application. This chapter explores the major HE scheme families, their characteristics, and when to use each one.

Partially Homomorphic Encryption (PHE)

RSA: Multiplicative Homomorphism

RSA, the famous public-key cryptosystem from 1978, has an interesting property: it's multiplicatively homomorphic.

Enc(m₁) × Enc(m₂) = Enc(m₁ × m₂ mod N)

How it works:

Public key: (N, e) where N = p·q (large primes)
Private key: (d) where e·d ≡ 1 (mod φ(N))

Encryption: c = m^e (mod N)
Decryption: m = c^d (mod N)

Homomorphic multiplication:
c₁ × c₂ = m₁^e × m₂^e = (m₁ × m₂)^e (mod N)

RSA Use Cases

  • Blind signatures: Sign documents without seeing content
  • Private information retrieval: Query databases privately
  • E-voting: Multiply encrypted votes to get product

Paillier: Additive Homomorphism

The Paillier cryptosystem (1999) provides additive homomorphism with excellent performance:

Enc(m₁) × Enc(m₂) = Enc(m₁ + m₂ mod N)

How it works:

Public key: (N, g) where N = p·q, g is generator
Private key: (λ, μ) where λ = lcm(p-1, q-1)

Encryption: c = g^m · r^N (mod N²)
            (r is random)

Decryption: m = L(c^λ mod N²) · μ (mod N)
            where L(x) = (x-1)/N

Homomorphic addition:
c₁ × c₂ = g^m₁·r₁^N × g^m₂·r₂^N
        = g^(m₁+m₂) × (r₁·r₂)^N (mod N²)
        = Enc(m₁ + m₂)

Paillier Applications

Electronic Voting: Each vote is encrypted. Homomorphically add all votes to get encrypted total. Decrypt once to get result—no individual votes revealed.

Private Statistics: Compute encrypted sums, averages, and variance without seeing individual data points.

Secure Auctions: Bidders submit encrypted bids. Auctioneer computes highest bid without seeing individual bids.

ElGamal: Multiplicative Homomorphism

ElGamal provides multiplicative homomorphism with perfect semantic security:

Setup: Choose prime p, generator g
Public key: (p, g, h) where h = g^x
Private key: x

Encryption: (c₁, c₂) = (g^r, m·h^r)
Decryption: m = c₂ / c₁^x

Homomorphic multiplication:
(c₁⁽¹⁾, c₂⁽¹⁾) × (c₁⁽²⁾, c₂⁽²⁾)
= (c₁⁽¹⁾·c₁⁽²⁾, c₂⁽¹⁾·c₂⁽²⁾)
= (g^(r₁+r₂), (m₁·m₂)·h^(r₁+r₂))
= Enc(m₁ × m₂)

Fully Homomorphic Encryption: BFV

BFV Overview

Brakerski-Fan-Vercauteren (BFV) is optimized for exact integer arithmetic. It's the workhorse of HE for most applications requiring precise calculations.

Key Characteristics

Feature Details
Plaintext Space Integers mod t (e.g., ℤ₁₀₂₄)
Operations Exact addition and multiplication
Batching n/2 slots per ciphertext via CRT
Best For Database queries, logic circuits, counting
Performance Addition: ~0.1ms, Multiplication: ~10ms

BFV Parameter Levels

BFV uses modulus chain for depth management:

Modulus chain: q = q₀ · q₁ · q₂ · ... · q_L

Each multiplication consumes one level:
Level L: q = q₀·q₁·...·q_L (fresh, full modulus)
Level L-1: q = q₀·q₁·...·q_(L-1) (after 1 mult)
Level L-2: q = q₀·q₁·...·q_(L-2) (after 2 mults)
...
Level 0: q = q₀ (minimal modulus, decrypt here)

Example for depth-5 computation:
q₀ = 60 bits (final layer)
q₁ = q₂ = q₃ = q₄ = q₅ = 40 bits each
Total: log₂(q) = 260 bits

When to Use BFV

  • Database Queries: WHERE clauses, GROUP BY, exact counting
  • Comparison Operations: Equality, greater-than using binary circuits
  • Genomic Analysis: DNA sequence matching, variant calling
  • Secure Voting: Tally exact vote counts
  • Financial Calculations: When rounding errors are unacceptable

Fully Homomorphic Encryption: CKKS

CKKS Overview

Cheon-Kim-Kim-Song (CKKS) revolutionized HE by embracing approximation. Instead of exact arithmetic, CKKS works with real/complex numbers to a certain precision.

The Approximation Insight

Many real-world applications don't need perfect precision. Machine learning works with floating-point. Signal processing tolerates noise. CKKS exploits this to achieve better performance than exact schemes for these applications.

Key Characteristics

Feature Details
Plaintext Space Real/complex numbers (approximate)
Operations Approximate addition and multiplication
Precision Configurable (e.g., 32-bit float precision)
Batching n/2 complex numbers via DFT
Best For Machine learning, signal processing, statistics

CKKS Encoding

CKKS uses a clever encoding that treats decryption error as "controlled noise":

Encode real vector z = [z₁, z₂, ..., z_(n/2)]:

1. Apply inverse DFT:
   π⁻¹(z) → polynomial m(x) with complex coefficients

2. Scale and round to integers:
   M(x) = ⌊Δ · m(x)⌉
   (Δ is scaling factor, e.g., 2⁴⁰)

3. Encrypt M(x) as usual

4. Decrypt and decode:
   M'(x) ≈ M(x) (small error from HE noise)
   m'(x) = M'(x) / Δ
   z' = π(m'(x)) ≈ z (controlled approximation)

CKKS Rescaling

After multiplication, scale doubles. Rescaling maintains precision:

Before multiplication:
Enc(m₁) at scale Δ
Enc(m₂) at scale Δ

After multiplication:
Enc(m₁ × m₂) at scale Δ²

Rescaling:
Divide by Δ, switch to smaller modulus
Result: Enc(m₁ × m₂) at scale Δ

When to Use CKKS

  • Neural Networks: Forward propagation, gradient descent
  • Logistic Regression: Training and inference on encrypted features
  • Signal Processing: FFT, filtering, compression
  • Statistical Analysis: Mean, variance, correlation
  • Scientific Computing: Simulations where approximation is acceptable

TFHE: Fast Bootstrapping

TFHE Overview

Torus FHE (TFHE) takes a different approach: optimize bootstrapping at all costs. While BFV and CKKS avoid bootstrapping due to overhead, TFHE makes it so fast that you can bootstrap after every gate.

Key Innovations

The Torus

TFHE uses the torus 𝕋 = ℝ/ℤ (real numbers modulo 1) as its message space. This enables efficient approximate arithmetic and fast bootstrapping.

Feature Details
Message Space Binary or small integers
Bootstrapping ~10-100ms (1000x faster than BFV/CKKS)
Ciphertext Size Very small (~KB)
Gate Types Any boolean gate (NAND, AND, OR, XOR)
Best For Logic circuits, comparison, decision trees

TFHE Gate Bootstrapping

TFHE can evaluate any boolean gate with bootstrapping:

NAND gate evaluation:
Input: Enc(b₁), Enc(b₂)
Output: Enc(NOT(b₁ AND b₂))

Process:
1. Compute noisy NAND homomorphically
2. Bootstrap to refresh and reduce noise
3. Result: fresh ciphertext

Time: ~13ms on modern CPU
Compare: BFV bootstrap ~10+ seconds!

When to Use TFHE

  • Comparison Operations: Greater-than, equality checks
  • Decision Trees: Encrypted classification
  • Boolean Circuits: Any logic computation
  • Encrypted Control Flow: Branching on encrypted conditions
  • Small Message Space: Binary or ternary data

BGV: The Original Leveled FHE

BGV Overview

Brakerski-Gentry-Vaikuntanathan (BGV) was one of the first practical FHE schemes. Very similar to BFV but with different noise management.

BGV vs BFV

Aspect BGV BFV
Noise Location Noise in low-order bits Noise in high-order bits
Multiplication Modulus switching after mult Rescaling integrated in mult
Performance Slightly slower Slightly faster
Implementation More complex Simpler
Use Today Less common Standard for integers

Scheme Comparison and Selection Guide

Performance Comparison

Operation Paillier BFV CKKS TFHE
Key Generation 10ms 100ms 100ms 50ms
Encryption 1ms 5ms 5ms 0.1ms
Addition 0.01ms 0.1ms 0.1ms N/A
Multiplication N/A 10ms 8ms N/A
Bootstrapping N/A ~10s ~10s ~13ms
Ciphertext Size 4KB 100KB 100KB 2KB

Selection Decision Tree

Choose Your Scheme

  1. Do you need only addition OR only multiplication?
    • Addition only → Paillier (fast, simple)
    • Multiplication only → RSA or ElGamal
  2. Do you need both addition and multiplication?
    • Exact integers → BFV
    • Approximate reals → CKKS
    • Binary logic → TFHE
  3. What's your computation depth?
    • Shallow (1-5 mults) → BFV or CKKS without bootstrap
    • Medium (5-10 mults) → BFV or CKKS with careful parameters
    • Deep (10+ mults) → TFHE or BFV/CKKS with bootstrap
  4. What's your performance requirement?
    • Milliseconds → Paillier or TFHE for simple ops
    • Seconds → BFV or CKKS
    • Minutes acceptable → Any scheme with bootstrap

Hybrid Approaches

Combining Multiple Schemes

Real applications often benefit from combining schemes:

Example: Private ML Prediction

1. Client encrypts input with CKKS
   (good for neural network arithmetic)

2. Server performs NN inference
   (matrix multiplications, activations)

3. For comparison/argmax:
   Convert CKKS → TFHE ciphertexts
   (TFHE better for comparisons)

4. TFHE computes argmax via comparison tree
   (fast bootstrapping enables deep circuit)

5. Return encrypted result to client

PHE + FHE Hybrid

Use fast PHE where possible, FHE only when needed:

E-voting example:
1. Use Paillier for vote tallying (just addition)
2. Use BFV for threshold checks (needs comparison)
3. Combine results securely

Chapter Summary

  • PHE schemes (RSA, Paillier, ElGamal) support one operation type, very fast
  • BFV optimized for exact integer arithmetic, ideal for databases and logic
  • CKKS supports approximate real/complex numbers, perfect for ML and signal processing
  • TFHE features extremely fast bootstrapping, best for binary circuits and comparisons
  • BGV similar to BFV but less commonly used today
  • Performance trade-offs: Paillier fastest, TFHE best for deep circuits, BFV/CKKS balanced
  • Scheme selection depends on operation types, depth, precision needs, and performance requirements
  • Hybrid approaches combining multiple schemes often provide best results
  • No single "best" scheme—choice depends on application requirements
