6.1 물리 계층 사양

물리 계층은 도시 전체 통신의 기반을 형성합니다. WIA는 연결성, 중복성, 미래 대비를 위한 최소 요구사항을 명시합니다.

양자 준비 인프라

완전한 양자 네트워크는 아직 실용적이지 않지만, WIA는 인프라가 "양자 준비" 상태일 것을 요구합니다:

  • 양자 키 분배(QKD)가 가능한 광섬유
  • 양자 이후 암호화 알고리즘 구현
  • 양자 네트워킹을 위한 업그레이드 경로 문서화

6.2 메시지 프로토콜

AIDC-Link는 노드 간 통신을 위한 효율적인 바이너리 및 텍스트 메시지 형식을 정의합니다.

바이너리 메시지 형식

┌─────────────────────────────────────────────────────────────┐
│ Magic (4 bytes) │ Version │ Type │ Length │ Flags            │
├─────────────────────────────────────────────────────────────┤
│ Message ID (16 bytes UUID)                                   │
├─────────────────────────────────────────────────────────────┤
│ Source Node ID (8 bytes)                                     │
├─────────────────────────────────────────────────────────────┤
│ Destination Node ID (8 bytes)                                │
├─────────────────────────────────────────────────────────────┤
│ Timestamp (8 bytes)                                          │
├─────────────────────────────────────────────────────────────┤
│ Payload (가변, zstd 압축)                                     │
├─────────────────────────────────────────────────────────────┤
│ Signature (64 bytes Ed25519)                                 │
└─────────────────────────────────────────────────────────────┘

메시지 유형

코드 유형 설명
0x01 SENSOR_DATA 센서 원격 측정 데이터
0x02 COMMAND 제어 명령
0x03 QUERY 데이터 요청
0x04 RESPONSE 쿼리 응답
0x05 EVENT 이벤트 알림
0x06 ALERT 긴급 경보
0x07 HEARTBEAT 노드 상태 확인
0x08 SYNC 상태 동기화

메시지 페이로드 예시

// 센서 데이터 메시지 (JSON 페이로드, 압축 전)
{
  "@context": "https://schema.wia.org/ai-city/v1",
  "@type": "TrafficSensorReading",
  "sensorId": "TRF-2024-SEOUL-GN-001",
  "timestamp": "2024-12-24T10:30:00Z",
  "vehicleCount": 145,
  "averageSpeed": 32.5,
  "congestionLevel": 0.62,
  "classifications": {
    "cars": 98,
    "trucks": 12,
    "buses": 8,
    "motorcycles": 27
  }
}

압축 및 성능

Zstandard 압축: 모든 페이로드는 Zstandard(zstd)로 압축되어 대역폭을 70-80% 절감합니다. 압축 레벨은 메시지 우선순위에 따라 동적으로 조정됩니다:

  • 긴급 (ALERT): 레벨 1 (가장 빠름, ~5μs 압축)
  • 실시간 (SENSOR_DATA): 레벨 3 (균형, ~50μs)
  • 배치 (SYNC): 레벨 9 (최대 압축, ~500μs)

6.3 엣지 AI 배포

엣지 노드는 지연시간이 중요한 의사결정, 프라이버시 보존, 대역폭 효율성을 위해 데이터를 로컬에서 처리합니다.

엣지 노드 사양

구성요소 최소 권장
CPU 16코어 32코어
RAM 64 GB ECC 128 GB ECC
GPU/NPU 1x NVIDIA L4 2x NVIDIA L4 / H100
스토리지 2 TB NVMe 4 TB NVMe RAID
네트워크 10 Gbps 25 Gbps

처리 카테고리

엣지 처리 (<50ms 필수)
  • 실시간 신호 조정: 교통 신호등은 즉각적인 조정을 위해 로컬 AI 모델을 사용합니다
  • 긴급 감지: 사고, 화재, 의료 응급상황의 즉각적인 식별
  • 프라이버시 필터링: 민감한 데이터는 엣지에서 필터링되어 절대 업로드되지 않음
  • 비디오 분석: 얼굴/번호판 감지는 로컬에서 수행되며, 집계된 통계만 전송
중앙 처리 (지연시간 허용)
  • 도시 전체 최적화: 글로벌 교통 흐름 모델 및 예측
  • 장기 추세 분석: 역사적 패턴 및 계절적 변화
  • 구역 간 조정: 여러 구역에 걸친 리소스 할당
  • 모델 훈련: 엣지 배포를 위한 새로운 AI 모델 개발

엣지 AI 모델 배포

// Rust 엣지 노드 예시
use wia_ai_city::edge::{EdgeNode, ModelRegistry};
use tokio;

#[tokio::main]
async fn main() -> Result<()> {
    let node = EdgeNode::new("edge-gangnam-01").await?;

    // AI 모델 로드
    let traffic_model = ModelRegistry::load("traffic-signal-v3.2")
        .with_device("cuda:0")
        .with_batch_size(32)
        .build()?;

    // 센서 데이터 스트림 구독
    let mut stream = node.subscribe("sensors/traffic/*").await?;

    while let Some(reading) = stream.next().await {
        // 로컬에서 추론 실행
        let prediction = traffic_model.predict(&reading).await?;

        // 지연시간 <50ms인 경우 직접 조치
        if prediction.confidence > 0.95 {
            node.send_command(
                "traffic-signal-001",
                SignalCommand::AdjustTiming(prediction.optimal_timing)
            ).await?;
        }
    }

    Ok(())
}

6.4 보안 강화

암호화 표준

용도 알고리즘 키 크기
저장 데이터 AES-256-GCM 256비트
전송 데이터 TLS 1.3 / mTLS 256비트
키 교환 X25519 / Kyber (양자 이후) 256비트+
서명 Ed25519 / Dilithium (양자 이후) 256비트+

노드 인증

// 노드 간 상호 TLS 핸드셰이크
노드 A ────────── Client Hello ────────────► 노드 B
노드 A ◄────────── Server Hello, Cert ─────── 노드 B
노드 A ────────── Client Cert, Verify ──────► 노드 B
노드 A ◄────────── Session Established ────── 노드 B
노드 A ◄══════════ 암호화된 트래픽 ═══════════► 노드 B

인증서 관리

자동 인증서 순환: 모든 노드 인증서는 Let's Encrypt 스타일 자동 갱신을 사용하여 90일마다 순환됩니다.

# 인증서 계층 구조
루트 CA (오프라인, 10년 수명)
  └── 중간 CA (온라인, 2년 수명)
      ├── 중앙 허브 인증서 (90일)
      ├── 엣지 노드 인증서 (90일)
      └── 센서 인증서 (90일)

침입 탐지

모니터링되는 위협:

  • 무단 접근 시도: 실패한 인증 시도는 로그되며 3회 실패 후 차단
  • 이상 트래픽 패턴: ML 모델이 비정상적인 네트워크 동작 감지
  • 센서 스푸핑: 물리적으로 불가능한 판독값(예: 1초 만에 100km 이동) 거부
  • 중간자 공격: 인증서 피닝 및 mTLS로 방지
  • DDoS 공격: 속도 제한 및 지리적 필터링

제로 트러스트 아키텍처

핵심 원칙: "절대 신뢰하지 말고 항상 검증하라"

  • 네트워크 위치에 관계없이 모든 통신이 인증됨
  • 최소 권한 접근: 노드는 역할에 필요한 것만 접근
  • 마이크로 세그멘테이션: 각 구역이 격리된 네트워크 세그먼트
  • 지속적인 모니터링: 모든 접근이 실시간으로 로그되고 분석됨

6.5 연합 프로토콜

도시 간 연합은 데이터 주권을 존중하면서 WIA 준수 도시 간 조정을 가능하게 합니다.

연합 계층

계층 공유 데이터 사용 사례
공개 (L4) 집계된 통계 지역 교통, 날씨
보호됨 익명화된 패턴 긴급 조정
양자 맞춤 계약 경계 구역 관리

도시 간 통신 예시

// 서울 ↔ 인천 대기질 데이터 공유
{
  "@context": "https://schema.wia.org/ai-city/federation/v1",
  "@type": "FederatedQuery",
  "fromCity": "AIC-2024-SEOUL1",
  "toCity": "AIC-2024-INCHEON1",
  "query": {
    "type": "environment.air_quality",
    "zones": ["border-zone-01", "border-zone-02"],
    "timeRange": {
      "start": "2024-12-24T00:00:00Z",
      "end": "2024-12-24T23:59:59Z"
    }
  },
  "dataClassification": "L4",  // 공개 데이터만
  "purpose": "cross-city-air-quality-monitoring",
  "signature": "..."
}

연합 동의 메커니즘

도시 간 데이터 공유 규칙:

  • 기본적으로 L4(공개) 데이터만 자동으로 공유됨
  • L3 데이터는 양 도시 관리자의 명시적 승인 필요
  • L2/L1 데이터는 절대 도시 경계를 넘지 않음
  • 모든 연합 쿼리는 감사 로그에 기록됨

// 단일 배치 메시지로 100개 센서 판독값 전송

  "readings": [
    { "sensorId": "TRF-001", "vehicleCount": 45, "timestamp": "..." },
    { "sensorId": "TRF-002", "vehicleCount": 52, "timestamp": "..." },

  ]
}

6.7 요약

핵심 요점

  • 물리 계층: 100 Gbps 광섬유 백본, 5G/6G 메시, 양자 준비 인프라로 미래 대비.
  • 메시지 프로토콜: 압축, 서명, 암호화를 갖춘 효율적인 바이너리 형식으로 대역폭 70-80% 절감.
  • 엣지 AI: <50ms 지연시간이 중요한 의사결정을 위한 로컬 처리로 프라이버시 보존 및 대역폭 절약.
  • 보안: mTLS, AES-256, Ed25519 서명, 제로 트러스트 아키텍처를 통한 심층 방어.
  • 연합: 데이터 주권을 존중하는 도시 간 조정, L4 데이터만 기본 공유.
  • 성능: 배칭 및 적응형 샘플링으로 최대 98% 오버헤드 감소.
