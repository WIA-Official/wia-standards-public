4.1 왜 JSON-LD인가?

WIA AI-City는 주요 데이터 형식으로 JSON-LD (JSON for Linked Data)를 사용합니다. 이 선택은 의도적이었습니다—JSON-LD는 JSON의 단순성과 시맨틱 웹 기술의 강력함을 결합하여 인간 가독성과 기계 이해를 모두 가능하게 합니다.

일반 JSON의 문제점

일반 JSON으로 작성된 간단한 온도 측정값을 고려해보세요:

{
  "temp": 23.5,
  "unit": "C"
}

간단해 보이지만 모호합니다. "C"는 무엇을 의미할까요? 섭씨? 다른 코드? 어디서 측정되었나요? 언제? 맥락이 무엇인가요? 다른 시스템은 이 데이터를 다르게 해석할 수 있어 통합 실패로 이어집니다.

JSON-LD 솔루션

JSON-LD로 작성된 동일한 측정값:

{
  "@context": "https://schema.wia.org/ai-city/v1",
  "@type": "TemperatureReading",
  "@id": "urn:wia:sensor:temp-001:reading:2024-12-24T10:30:00Z",
  "value": 23.5,
  "unit": "CEL",
  "measuredAt": {
    "@type": "GeoCoordinates",
    "latitude": 37.5665,
    "longitude": 126.9780
  },
  "timestamp": "2024-12-24T10:30:00Z",
  "sensor": {
    "@id": "urn:wia:sensor:temp-001",
    "@type": "TemperatureSensor",
    "accuracy": "±0.1"
  }
}

이제 모호함이 없습니다. @context는 모든 시스템에 각 필드를 정확히 어떻게 해석해야 하는지 알려줍니다. @type은 이것이 어떤 종류의 데이터인지 식별합니다. @id는 전역적으로 고유한 식별자를 제공합니다.

주요 JSON-LD 개념

개념 목적 예시
@context 어휘 및 용어 매핑 정의 "https://schema.wia.org/ai-city/v1"
@type 데이터 객체의 유형 지정 "TrafficSensor", "CitizenConsent"
@id 전역적으로 고유한 식별자 (IRI) "urn:wia:city:seoul:sensor:001"
@graph 여러 연결된 객체 포함 관련 엔티티의 배열
@vocab 용어의 기본 어휘 "https://schema.wia.org/"

4.2 WIA Context와 어휘

WIA는 모든 호환 시스템이 사용해야 하는 포괄적인 컨텍스트 정의를 제공합니다. 이 컨텍스트는 용어를 정확한 시맨틱 의미로 매핑합니다.

WIA Context 구조

{
  "@context": {
    "@version": 1.1,
    "@vocab": "https://schema.wia.org/ai-city/",
    "wia": "https://schema.wia.org/",
    "schema": "https://schema.org/",
    "geo": "http://www.w3.org/2003/01/geo/wgs84_pos#",

    "AICityInstance": "wia:AICityInstance",
    "RustCore": "wia:RustCore",
    "SharingEngine": "wia:SharingEngine",
    "DataSovereignty": "wia:DataSovereignty",
    "AIDCLink": "wia:AIDCLink",

    "cityId": {"@id": "wia:cityId", "@type": "@id"},
    "population": {"@id": "schema:population", "@type": "xsd:integer"},
    "location": {"@id": "geo:location", "@type": "@id"},
    "timestamp": {"@id": "schema:dateCreated", "@type": "xsd:dateTime"},

    "classificationLevel": {
      "@id": "wia:classificationLevel",
      "@type": "@vocab",
      "@context": {
        "L1": "wia:IdentityData",
        "L2": "wia:LocationData",
        "L3": "wia:BehavioralData",
        "L4": "wia:PublicData"
      }
    }
  }
}

Context 임베딩 vs. 참조

프로덕션 시스템의 경우, WIA는 임베딩보다 URL로 컨텍스트를 참조할 것을 권장합니다:

참조 방식 (권장) "@context": "https://schema.wia.org/ai-city/v1"

더 작은 페이로드, 항상 최신 상태
임베딩 방식 (개발) 문서에 전체 컨텍스트 객체 포함

자체 포함, 오프라인에서 작동

4.4 데이터 분류 구현

WIA AI-City의 모든 데이터는 4단계 시스템에 따라 분류되어야 합니다. 분류는 데이터가 저장, 전송, 접근, 결국 삭제되는 방식을 결정합니다.

분류 요구사항

레벨 저장 전송 접근 제어 보존
L1 - 신원 로컬만, 저장 시 암호화 외부 전송 안 함 시민 키 필요 명시적 삭제까지
L2 - 위치 암호화, 격리 저장 E2E 암호화, 동의 확인 목적별 동의 기본 최대 90일
L3 - 행동 표준 암호화 암호화, 익명화 옵션 옵트인 동의 기본 최대 1년
L4 - 공개 표준 저장 개방 전송 공개 접근 영구 아카이브

실제 분류

모든 데이터 객체는 분류 레벨을 포함해야 합니다:

{
  "@type": "LocationReading",
  "classificationLevel": "L2",
  "consentRequired": true,
  "consentToken": "CST-A1B2C3D4E5F6G7H8",

  "data": {
    "latitude": 37.5665,
    "longitude": 126.9780,
    "accuracy": 10,
    "timestamp": "2024-12-24T10:30:00Z"
  },

  "dataSubject": {
    "@id": "urn:wia:citizen:KR:12345",
    "pseudonymized": true
  }
}

자동 분류 규칙

데이터 타입별 기본 분류:
  • 생체인식, 건강 기록, 주민등록번호: 항상 L1
  • 실시간 위치, 이동 이력: 기본 L2
  • 쇼핑, 서비스 사용, 선호도: 기본 L3
  • 집계 통계 (10명 이상): L4 가능
  • 환경 측정값 (개인 데이터 없음): L4

모든 시스템 간 통신은 표준화된 AIDC-Link 메시지 형식을 사용합니다. 이것은 모든 도시 시스템에서 일관된 메시지 구조를 보장합니다.

4.8 마이그레이션 전략

대부분의 도시는 다양한 형식의 기존 데이터를 가지고 있습니다. WIA는 운영을 유지하면서 호환 형식으로 마이그레이션하기 위한 지침을 제공합니다.

마이그레이션 접근 방식

1
기존 데이터 목록 작성

모든 데이터 소스, 형식, 볼륨, 소유자를 카탈로그화합니다. 각각이 어떤 WIA 스키마에 매핑되는지 식별합니다.

2
변환 규칙 설계

기존 필드가 WIA 속성으로 어떻게 변환되는지 보여주는 매핑 문서를 작성합니다.

3
어댑터 구현

데이터 수집 지점에서 데이터를 변환하는 어댑터 계층을 구축합니다. 레거시 시스템은 변경 없이 계속됩니다.

4
이중 쓰기 기간

기존 형식과 새 형식 모두로 데이터를 씁니다. 변환이 올바른지 검증합니다.

5
전환 및 검증

소비자를 새 형식으로 전환합니다. 검증 기간 후 레거시 형식을 폐기합니다.

어댑터 예시

레거시 교통 센서 데이터를 WIA 형식으로 변환:

// 레거시 형식
{
  "sid": "TS001",
  "t": 1703412600,
  "vc": 1247,
  "spd": 35.5
}

// WIA 어댑터 변환
function transformToWIA(legacy) {
  return {
    "@context": "https://schema.wia.org/ai-city/v1",
    "@type": "TrafficReading",
    "@id": `urn:wia:city:seoul:traffic:${legacy.sid}:${legacy.t}`,

    "sensor": {
      "@id": `urn:wia:city:seoul:traffic:sensor:${legacy.sid}`
    },

    "timestamp": new Date(legacy.t * 1000).toISOString(),
    "vehicleCount": legacy.vc,
    "averageSpeed": legacy.spd,
    "classificationLevel": "L4"
  };
}

4.9 검증 및 테스트

WIA는 데이터 형식 준수를 검증하기 위한 도구와 사양을 제공합니다.

검증 계층

구문 검증 유효한 JSON, 적절한 JSON-LD 구조
스키마 검증 필수 필드 존재, 올바른 타입
시맨틱 검증 일관된 @context, 유효한 참조
비즈니스 규칙 분류 로직, 동의 요구사항

WIA Validator 도구

# WIA 검증 도구 설치
npm install -g @wia/ai-city-validator

# 단일 문서 검증
wia-validate document.json

# 특정 스키마로 검증
wia-validate --schema TrafficReading sensor-data.json

# 디렉토리 일괄 검증
wia-validate --recursive ./data/

# 출력 형식
wia-validate --format json document.json
wia-validate --format report document.json

일반적인 검증 오류

오류 원인 해결책
MISSING_CONTEXT @context 필드 없음 WIA 컨텍스트 URL 추가
INVALID_TYPE 알 수 없는 @type 값 유효한 WIA 타입 이름 사용
MISSING_ID 식별 가능한 엔티티에 @id 없음 URN 식별자 추가
CLASSIFICATION_REQUIRED classificationLevel 없음 L1-L4 분류 추가
CONSENT_MISSING 동의 없는 L2/L3 데이터 동의 토큰 참조 추가

4.10 챕터 요약

핵심 요점

  • JSON-LD 기초: WIA는 시맨틱 상호운용성을 위해 JSON-LD를 사용합니다. @context, @type, @id 필드는 모든 시스템에서 명확한 데이터 해석을 제공합니다.
  • WIA Context: 포괄적인 어휘가 모든 용어를 정의합니다. 프로덕션에서는 컨텍스트 URL을 참조하고, 개발에서만 임베딩하세요.
  • 12개 핵심 도메인: 교통, 환경, 유틸리티, 대중교통, 안전, 보건, 시민, 재무, 인프라, 거버넌스, AI/ML, 보안이 모든 스마트 시티 요구를 다룹니다.
  • 데이터 분류: 모든 데이터 객체는 L1-L4로 분류되어야 합니다. 분류는 저장, 전송, 접근, 보존 정책을 결정합니다.
  • AI-City 인스턴스: 마스터 구성 문서는 도시의 완전한 WIA 설정을 정의합니다—Rust-Core, 나눔 엔진, 데이터 주권, AIDC-Link 설정.
  • AIDC-Link 메시지: 헤더, 소스, 대상, 페이로드, 보안 필드가 있는 표준화된 메시지 형식은 일관된 시스템 간 통신을 가능하게 합니다.
  • ID 패턴: 일관된 식별자 형식 (URN, 타입별 ID)은 전역 고유성과 쉬운 파싱을 보장합니다.
  • 마이그레이션 경로: 어댑터 기반 마이그레이션은 기존 운영을 방해하지 않고 레거시 형식에서 점진적 전환을 허용합니다.
  • 검증: WIA는 구문, 스키마, 시맨틱, 비즈니스 규칙 수준에서 준수를 검증하기 위한 도구와 사양을 제공합니다.

4.11 복습 질문

이해도 테스트

  1. 일반 JSON과 JSON-LD의 차이점을 설명하세요. 왜 JSON-LD가 스마트 시티 데이터 통합에 더 적합한가요?
  2. 센서 측정값이 시민의 실시간 위치를 포함합니다. 어떤 분류 레벨을 가져야 하며, 어떤 기술적 요구사항이 적용되나요?
  3. 대기질 센서 측정값에 대한 JSON-LD 문서를 작성하세요. 적절한 @context, @type, @id, 분류, 측정 데이터를 포함하세요.
  4. 귀 도시에는 다양한 부서의 50개 레거시 데이터 형식이 있습니다. 중단을 최소화하는 마이그레이션 전략을 설명하세요.
  5. 들어오는 AIDC-Link 메시지를 처리하기 전에 어떤 검증 확인을 수행해야 하나요?
  6. L2 및 L3 데이터에 대해 동의 토큰을 참조해야 하는 이유는 무엇인가요? 이것이 데이터 주권 기둥과 어떻게 관련되나요?
  7. 새 센서 타입인 환경 소음 모니터를 위한 ID 패턴을 설계하세요. 도시, 구역, 센서별 구성요소를 포함하세요.

4.12 앞으로의 전망

다음 내용: 5장 - Phase 2: API 인터페이스 계층

데이터 형식이 표준화되면, Phase 2는 도시 서비스를 노출하는 API 계층을 구축합니다. 5장은 다음을 다룹니다:

  • REST API 설계: WIA 규칙 및 모범 사례를 따르는 RESTful 엔드포인트.
  • GraphQL 인터페이스: 복잡한 데이터 관계 및 실시간 구독을 위한 유연한 쿼리.
  • 인증: 안전한 API 접근을 위한 OAuth 2.0 / OpenID Connect 구현.
  • 시민 포털 API: 동의 관리, 데이터 접근, 프라이버시 제어 엔드포인트.
  • 개발자 SDK: WIA AI-City에서 애플리케이션을 구축하기 위한 클라이언트 라이브러리 및 도구.

데이터는 공통 언어를 사용합니다. 이제 모든 사람이 접근할 수 있도록 하는 인터페이스를 구축해봅시다.
