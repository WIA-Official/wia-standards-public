API Specifications and Integration

5.1 Moderation Service API Overview

A well-designed moderation API enables developers to integrate content safety into their applications seamlessly. The API should provide comprehensive coverage across content types (text, image, video, audio), offer flexible deployment models (real-time, batch, streaming), and deliver actionable results with clear confidence scores and explanations. This chapter presents a complete API specification for content moderation services following RESTful principles and industry best practices.

The API design prioritizes developer experience through consistent request/response formats, predictable error handling, comprehensive documentation, and language-specific SDKs. It balances simplicity for basic use cases with power for advanced scenarios requiring custom rules, confidence thresholds, and integration with existing workflows.

class ContentModerationClient {
    private config: ModerationConfig;
    private baseUrl: string;

    constructor(config: ModerationConfig) {
        this.config = config;
        this.baseUrl = config.baseUrl || 'https://api.moderation.example.com';
    }

    /**
     * Moderate text content
     */
    async moderateText(request: TextModerationRequest): Promise {
        return this.makeRequest('/v1/moderate/text', {
            method: 'POST',
            body: request
        });
    }

    /**
     * Moderate image content
     */
    async moderateImage(request: ImageModerationRequest): Promise {
        return this.makeRequest('/v1/moderate/image', {
            method: 'POST',
            body: request
        });
    }

    /**
     * Moderate video content (async operation)
     */
    async moderateVideo(request: VideoModerationRequest): Promise {
        return this.makeRequest('/v1/moderate/video', {
            method: 'POST',
            body: request
        });
    }

    /**
     * Moderate audio content
     */
    async moderateAudio(request: AudioModerationRequest): Promise {
        return this.makeRequest('/v1/moderate/audio', {
            method: 'POST',
            body: request
        });
    }

    /**
     * Batch moderation for multiple items
     */
    async moderateBatch(requests: BatchModerationRequest): Promise {
        return this.makeRequest('/v1/moderate/batch', {
            method: 'POST',
            body: requests
        });
    }

    /**
     * Get results for async moderation job
     */
    async getResults(jobId: string): Promise {
        return this.makeRequest(`/v1/results/${jobId}`, {
            method: 'GET'
        });
    }

    /**
     * Submit feedback on moderation decision
     */
    async submitFeedback(feedback: ModerationFeedback): Promise {
        return this.makeRequest('/v1/feedback', {
            method: 'POST',
            body: feedback
        });
    }

    /**
     * Create webhook subscription for real-time notifications
     */
    async createWebhook(webhook: WebhookConfig): Promise {
        return this.makeRequest('/v1/webhooks', {
            method: 'POST',
            body: webhook
        });
    }

    private async makeRequest(
        endpoint: string,
        options: RequestOptions
    ): Promise {
        const url = `${this.baseUrl}${endpoint}`;
        const headers = {
            'Authorization': `Bearer ${this.config.apiKey}`,
            'Content-Type': 'application/json',
            'User-Agent': 'ContentModerationSDK/1.0'
        };

        let attempts = 0;
        const maxAttempts = this.config.retryAttempts || 3;

        while (attempts < maxAttempts) {
            try {
                const response = await fetch(url, {
                    method: options.method,
                    headers,
                    body: options.body ? JSON.stringify(options.body) : undefined,
                    signal: AbortSignal.timeout(this.config.timeout || 30000)
                });

                if (!response.ok) {
                    const error = await response.json();
                    throw new ModerationAPIError(
                        error.message || 'API request failed',
                        response.status,
                        error.code
                    );
                }

                return await response.json();
            } catch (error) {
                attempts++;
                if (attempts >= maxAttempts) {
                    throw error;
                }
                // Exponential backoff
                await this.sleep(Math.pow(2, attempts) * 1000);
            }
        }

        throw new Error('Max retry attempts exceeded');
    }

    private sleep(ms: number): Promise {
        return new Promise(resolve => setTimeout(resolve, ms));
    }
}

class ModerationAPIError extends Error {

}

5.3 Webhook Integration

For asynchronous operations like video moderation or batch processing, webhooks provide real-time notifications when results are ready. Instead of polling the API repeatedly, your application receives HTTP POST requests to a configured endpoint when moderation completes. This reduces latency, saves API calls, and enables event-driven architectures. 

// TypeScript: Webhook Handler

import { createHmac } from 'crypto';

interface WebhookPayload {
    event: 'moderation.completed' | 'moderation.failed';
    timestamp: number;
    data: {
        jobId: string;
        results?: ModerationResults;
        error?: string;
    };
}

class WebhookHandler {
    private webhookSecret: string;

    constructor(secret: string) {
        this.webhookSecret = secret;
    }

    /**
     * Verify webhook signature to ensure authenticity
     */
    verifySignature(payload: string, signature: string): boolean {
        const expectedSignature = createHmac('sha256', this.webhookSecret)
            .update(payload)
            .digest('hex');
        
        return signature === `sha256=${expectedSignature}`;
    }

    /**
     * Handle incoming webhook
     */
    async handleWebhook(
        rawPayload: string,
        signature: string
    ): Promise {
        // Verify signature
        if (!this.verifySignature(rawPayload, signature)) {
            throw new Error('Invalid webhook signature');
        }

        const payload: WebhookPayload = JSON.parse(rawPayload);

        switch (payload.event) {
            case 'moderation.completed':
                await this.handleModerationCompleted(payload.data);
                break;
            case 'moderation.failed':
                await this.handleModerationFailed(payload.data);
                break;
            default:
                console.warn(`Unknown webhook event: ${payload.event}`);
        }
    }

    private async handleModerationCompleted(data: any): Promise {
        console.log(`Moderation completed for job ${data.jobId}`);
        // Process results, update database, notify users, etc.
    }

    private async handleModerationFailed(data: any): Promise {
        console.error(`Moderation failed for job ${data.jobId}:`, data.error);
        // Handle failure, retry, alert monitoring, etc.
    }
}

// Express.js webhook endpoint example
import express from 'express';

const app = express();
const webhookHandler = new WebhookHandler(process.env.WEBHOOK_SECRET!);

app.post('/webhooks/moderation', 
    express.raw({ type: 'application/json' }),
    async (req, res) => {
        try {
            const signature = req.headers['x-moderation-signature'] as string;
            const payload = req.body.toString();
            
            await webhookHandler.handleWebhook(payload, signature);
            
            res.status(200).send('OK');
        } catch (error) {
            console.error('Webhook processing error:', error);
            res.status(400).send('Bad Request');
        }
    }
);

5.4 Rate Limiting and Authentication

All API endpoints implement rate limiting to ensure fair usage and system stability. Rate limits are applied per API key and vary by endpoint based on computational cost. Exceeding rate limits returns HTTP 429 responses with headers indicating when the limit resets. For high-volume applications, contact support for increased limits.

HTTP Header Description Example Value
X-RateLimit-Limit Maximum requests per window 1000
X-RateLimit-Remaining Requests remaining in current window 847
X-RateLimit-Reset Unix timestamp when limit resets 1640995200
Retry-After Seconds until retry allowed (429 responses) 45
Security Best Practices: Never expose API keys in client-side code. Always make moderation API calls from your backend servers. Rotate API keys periodically and immediately if compromised. Use environment variables or secure vaults for key storage. Monitor API usage for anomalies indicating unauthorized access.

Key Takeaways

  • RESTful Design: The API follows REST principles with clear resource-oriented endpoints, standard HTTP methods, and predictable response formats
  • Multi-Content Support: Dedicated endpoints for text, image, video, and audio moderation with content-type-specific request/response schemas
  • Async Operations: Video moderation and batch processing use async patterns with job IDs and webhook notifications for completed results
  • SDK Convenience: Client SDKs abstract HTTP details, provide automatic retries, handle authentication, and offer type-safe interfaces
  • Webhook Integration: Real-time notifications via webhooks enable event-driven architectures and reduce polling overhead
  • Rate Limiting: Per-endpoint rate limits ensure fair usage with clear headers indicating quota status and reset times
  • Security First: Bearer token authentication, webhook signature verification, and HTTPS-only communication protect API access

Reference Standards and Implementation Notes

This standard is designed to interoperate with established international frameworks. Implementations SHOULD treat the following as the single source of truth and use this PHASE document as the interoperability layer that maps existing data into a machine-readable, signed, and auditable shape.

  • OpenAPI Specification 3.1 — REST API definition
  • JSON Schema 2020-12 — schema validation
  • ISO/IEC 27001:2022 — information security management
  • ISO/IEC 27018:2019 — PII protection in public clouds
  • ISO/IEC 17065:2012 — product certification bodies
  • CycloneDX 1.5 / SPDX 2.3 — software bill of materials
  • Sigstore (DSSE envelope, Rekor transparency log) — signature anchoring
  • in-toto Attestation Framework 1.0 — build provenance
  • IETF RFC 9700 — OAuth 2.1
  • IETF RFC 9457 — Problem Details for HTTP APIs
  • IETF RFC 8615 — well-known URIs

Practical Implementation Checklist

  • Manifest integrity — every published manifest is signed via Sigstore DSSE. Tampering is detectable without trusting an intermediary. Updates are issued as new manifests; the prior manifest is preserved with a supersede marker.
  • Retention — keep signed evidence for at least seven years to satisfy ISO/IEC 17065:2012 audit expectations. Some regulators require longer retention; in that case the deployment policy MUST extend the window rather than relying on the defaults.
  • Versioning — follow Semantic Versioning 2.0.0. Major version bumps require at least a 90-day overlap with the prior major version on every WIA-published reference implementation.
  • Negative-result reporting — a skipped indicator without recorded justification is treated as a failed indicator at audit time.
  • Key rotation — manifest signing keys are rotated per the deployment policy. The previous key is preserved in a published key history so that auditors can verify historical manifests offline.

Reference Topologies

Implementations declare their operational topology in the manifest: single-tenant edge (one runtime per organization), multi-tenant gateway (shared runtime with header-based isolation), federated mesh (multiple runtimes peer to one another and publish manifests to a directory service), or air-gapped batch (signed evidence package transported out-of-band). Each topology ships with reference test vectors so that implementations can validate their deployment matches the canonical pattern.

Open Governance

Issues, errata, and proposals are tracked at github.com/WIA-Official/wia-standards/issues with the appropriate label. The WIA Standards working group reviews open issues at the start of every minor release cycle and publishes the resulting decision log alongside the release notes. Errata are issued as patch releases; new normative requirements trigger minor bumps; backwards-incompatible changes trigger major bumps.
