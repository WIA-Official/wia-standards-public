Chapter 06: Phase 3 - 프로토콜

Protocol Specification - 분산 실행과 통신 프로토콜

학습 목표

  • WIA-DIGITAL_EXECUTOR의 분산 실행 프로토콜을 이해한다
  • 작업 분배와 상태 동기화 메커니즘을 학습한다
  • 이벤트 전달 보장 방식을 파악한다
  • 분산 잠금과 합의 프로토콜을 익힌다

1. 분산 실행 아키텍처

WIA-DIGITAL_EXECUTOR는 수평 확장 가능한 분산 아키텍처를 채택합니다. 작업은 여러 워커 노드에 분배되어 병렬로 실행되며, 중앙 코디네이터가 상태를 관리합니다.

컴포넌트 역할 확장 방식
Coordinator 워크플로우 스케줄링, 상태 관리 Active-Passive HA
Worker 실제 작업 실행 수평 확장 (N개)
State Store 실행 상태 영구 저장 복제 + 샤딩
Message Queue 작업 분배, 이벤트 전달 클러스터링

2. 작업 분배 프로토콜 (TDP)

Task Distribution Protocol(TDP)은 코디네이터에서 워커로 작업을 분배하는 프로토콜입니다.

작업 분배 흐름

Coordinator → Task Queue → Worker → Result
// TDP 메시지 형식 { "protocol": "TDP", "version": "1.0", "messageType": "TASK_DISPATCH", "messageId": "msg-abc123", "timestamp": "2025-01-20T10:35:00.123Z", "payload": { "executionId": "exec-xyz789", "taskId": "process-payment", "workflowId": "wf-order-processing", "input": { "amount": 150000, "currency": "KRW", "method": "card" }, "timeout": "PT30S", "retryPolicy": { "maxAttempts": 3, "backoff": "exponential" } }, "routing": { "targetQueue": "worker.payment", "priority": "high", "affinity": "payment-service" } }

2.1 작업 할당 전략

전략 설명 사용 시나리오
Round Robin 워커에 순차 배분 균등한 작업 부하
Least Loaded 부하 가장 적은 워커에 배분 작업 시간 불균등
Affinity 특정 워커 그룹에 배분 특수 리소스 필요
Sticky 같은 실행의 작업을 같은 워커에 로컬 캐시 활용

3. 상태 동기화 프로토콜 (SSP)

State Synchronization Protocol(SSP)은 분산된 컴포넌트 간에 실행 상태를 일관되게 유지합니다.

// SSP 상태 업데이트 메시지 { "protocol": "SSP", "version": "1.0", "messageType": "STATE_UPDATE", "executionId": "exec-xyz789", "sequenceNumber": 42, "previousHash": "sha256:abc123", "update": { "taskId": "process-payment", "transition": { "from": "running", "to": "completed" }, "data": { "output": { "transactionId": "TXN-123456", "status": "approved" }, "duration": "PT2.5S", "completedAt": "2025-01-20T10:35:02.500Z" } }, "checkpoint": { "hash": "sha256:def456", "timestamp": "2025-01-20T10:35:02.500Z" } }

상태 일관성 보장

  • 시퀀스 번호: 순서 보장을 위한 단조 증가 번호
  • 해시 체인: 상태 무결성 검증을 위한 해시 연결
  • 체크포인트: 복구 지점 저장
  • 낙관적 동시성: 충돌 감지 및 해결

4. 이벤트 전달 프로토콜 (EDP)

Event Delivery Protocol(EDP)은 시스템 내 이벤트의 신뢰성 있는 전달을 보장합니다.

4.1 전달 보장 수준

수준 보장 메커니즘
At-Most-Once 최대 1회 전달 Fire-and-forget
At-Least-Once 최소 1회 전달 ACK + 재전송
Exactly-Once 정확히 1회 전달 중복 제거 + 트랜잭션
// EDP 이벤트 메시지 { "protocol": "EDP", "version": "1.0", "eventId": "evt-abc123", "eventType": "task.completed", "source": "worker-01", "timestamp": "2025-01-20T10:35:02.500Z", "delivery": { "guarantee": "exactly-once", "idempotencyKey": "exec-xyz789:process-payment:completed", "ackRequired": true, "ackTimeout": "PT5S" }, "data": { "executionId": "exec-xyz789", "taskId": "process-payment", "status": "completed", "output": { ... } } } // ACK 응답 { "protocol": "EDP", "messageType": "ACK", "eventId": "evt-abc123", "receiver": "coordinator-01", "timestamp": "2025-01-20T10:35:02.510Z" }

5. 분산 잠금 프로토콜 (DLP)

Distributed Lock Protocol(DLP)은 분산 환경에서 동시 실행을 제어합니다.

// 잠금 획득 요청 { "protocol": "DLP", "version": "1.0", "operation": "ACQUIRE", "lockId": "exec-xyz789:process-payment", "owner": "worker-01", "ttl": "PT30S", "waitTimeout": "PT10S" } // 잠금 획득 응답 { "protocol": "DLP", "operation": "ACQUIRED", "lockId": "exec-xyz789:process-payment", "fenceToken": 12345, "expiresAt": "2025-01-20T10:35:30.000Z" } // 잠금 해제 { "protocol": "DLP", "operation": "RELEASE", "lockId": "exec-xyz789:process-payment", "fenceToken": 12345 }

분산 잠금 주의사항

  • TTL 필수: 네트워크 분할 시 잠금 해제 보장
  • 펜스 토큰: 좀비 프로세스의 잘못된 접근 방지
  • 클라이언트 타임아웃: 잠금 대기 시간 제한
  • 갱신: 장기 작업 시 TTL 연장 필요

6. 하트비트 프로토콜

워커와 코디네이터 간의 생존 확인을 위한 하트비트 프로토콜입니다.

// 워커 → 코디네이터 하트비트 { "protocol": "HBP", "messageType": "HEARTBEAT", "workerId": "worker-01", "timestamp": "2025-01-20T10:35:00.000Z", "status": { "load": 0.65, "activeTaskCount": 5, "queuedTaskCount": 3, "memoryUsage": 0.72, "cpuUsage": 0.45 }, "capabilities": ["payment", "notification"], "interval": "PT5S" } // 코디네이터 → 워커 응답 { "protocol": "HBP", "messageType": "HEARTBEAT_ACK", "workerId": "worker-01", "timestamp": "2025-01-20T10:35:00.100Z", "commands": [] // 명령이 있을 수 있음 }

7. 장애 감지와 복구

분산 시스템의 장애를 감지하고 자동 복구하는 메커니즘입니다.

장애 유형 감지 방법 복구 전략
워커 실패 하트비트 타임아웃 작업 재할당
네트워크 분할 다수결 쿼럼 소수 파티션 중단
코디네이터 실패 리더 선출 타임아웃 자동 페일오버
작업 타임아웃 데드라인 초과 취소 + 재시도/보상

복구 절차

  1. 장애 감지 및 격리
  2. 영향받은 작업 식별
  3. 체크포인트에서 상태 복원
  4. 작업 재할당 또는 보상 실행
  5. 정상 운영 재개

8. 프로토콜 버전 관리

프로토콜의 하위 호환성을 유지하면서 진화할 수 있도록 버전 관리합니다.

// 버전 협상 { "protocol": "VERSION_NEGOTIATION", "supportedVersions": ["1.0", "1.1", "2.0"], "preferredVersion": "2.0" } // 협상 결과 { "protocol": "VERSION_NEGOTIATION", "selectedVersion": "1.1", "deprecationWarnings": [ "1.0 버전은 2025-06-01에 지원 종료됩니다" ] }

핵심 요점

  • TDP: 코디네이터에서 워커로 작업 분배
  • SSP: 분산 컴포넌트 간 상태 일관성 유지
  • EDP: 신뢰성 있는 이벤트 전달 (Exactly-Once 지원)
  • DLP: 분산 환경 동시성 제어
  • 하트비트: 워커 생존 확인과 부하 모니터링
  • 장애 복구: 자동 감지와 체크포인트 기반 복구
  • 버전 협상: 하위 호환성 유지

복습 문제

  1. TDP에서 작업 할당 전략 4가지를 설명하시오.
  2. SSP의 상태 일관성 보장 메커니즘을 설명하시오.
  3. At-Least-Once와 Exactly-Once 전달의 차이를 설명하시오.
  4. 분산 잠금에서 펜스 토큰의 역할을 설명하시오.
  5. 워커 실패 시 복구 절차를 순서대로 나열하시오.
  6. 프로토콜 버전 협상이 필요한 이유를 설명하시오.
