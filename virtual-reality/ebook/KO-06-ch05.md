# 5장: 기술 표준 및 사양

## 5.1 OpenXR 표준

### 상호 운용성의 기초

OpenXR는 VR/AR 플랫폼 파편화에 대한 업계의 답변입니다. OpenGL, Vulkan, WebGL을 만든 Khronos Group에서 개발한 OpenXR는 XR 애플리케이션 및 장치를 위한 로열티 프리 오픈 표준 API를 제공합니다.

**역사**:
- **2017년**: OpenXR 작업 그룹 형성 (창립 멤버: Valve, Microsoft, Epic, Unity, Meta, Google, HTC)
- **2019년 7월**: OpenXR 1.0 사양 출시
- **2021년 7월**: 중요한 확장 기능이 포함된 OpenXR 1.0.17
- **2023-2025년**: 업계 전반에서 거의 보편적인 채택
- **2026년**: OpenXR 1.1 향상된 기능 추가 (동적 중심 와 렌더링, 전신 추적 확장)

### API 아키텍처

**핵심 개념**: "한 번 작성하면 어디서나 실행"

개발자가 OpenXR API로 작성 → 런타임이 장치별 구현으로 변환

```
[OpenXR API를 사용하는 애플리케이션]
         ↓
[OpenXR 런타임] ← 플랫폼별 (Meta, Valve, Microsoft 등)
         ↓
[VR/AR 하드웨어]
```

**주요 구성 요소**:

**1. 인스턴스 및 시스템**:
```cpp
XrInstance instance;  // OpenXR 런타임에 대한 연결
XrSystemId system;    // 특정 XR 시스템 (헤드셋)
```

**2. 세션**:
```cpp
XrSession session;    // 활성 VR/AR 세션
XrSessionState state; // 실행 중, 표시, 포커스 등
```

**3. 공간**:
```cpp
XrSpace space;        // 좌표계 (월드, 스테이지, 뷰)
```

**4. 액션**:
```cpp
XrActionSet actionSet;  // 사용자 입력 그룹
XrAction grabAction;    // 특정 액션 (버튼, 트리거)
```

**5. 구성 레이어**:
```cpp
XrCompositionLayerProjection projLayer; // 렌더링된 콘텐츠 레이어
XrCompositionLayerQuad quadLayer;       // 2D UI 오버레이
```

### 런타임 구현

**주요 OpenXR 런타임 (2026년)**:

**Meta Quest OpenXR 런타임**:
- 전체 OpenXR 1.0 준수
- 확장: 손 추적, 패스스루, 눈 추적 (Quest Pro)
- 현재 Quest 앱의 90%가 OpenXR 사용 (독점 Oculus SDK에서 전환)

**SteamVR OpenXR 런타임**:
- 모든 SteamVR 호환 헤드셋 지원
- Index, Vive, Link를 통한 Quest, Windows MR 등과 함께 작동
- 가장 성숙하고 기능이 완전한 런타임

**Windows Mixed Reality OpenXR 런타임**:
- 더 이상 사용되지 않지만 여전히 작동 (2026년)
- 이전 WMR 헤드셋 지원 (HP Reverb G2 등)

**Varjo OpenXR 런타임**:
- 전문 기능 (고해상도, 비디오 패스스루)
- 기업 사용 사례를 위한 확장

**PlayStation VR2**:
- 2024년 OpenXR 지원 추가
- 크로스 플랫폼 PC VR 콘텐츠 가능

**Apple visionOS**:
- Apple 특정 확장 기능이 있는 맞춤형 구현
- 2025년 발표된 전체 OpenXR 지원 (이전에는 ARKit 중심)

### 확장 시스템

OpenXR의 확장성을 통해 하드웨어 공급업체는 고유한 기능을 노출할 수 있습니다:

**핵심 확장 (광범위하게 지원됨)**:
- `XR_EXT_hand_tracking`: 손 골격 추적
- `XR_EXT_eye_gaze_interaction`: 시선 입력을 위한 눈 추적
- `XR_FB_passthrough`: MR을 위한 카메라 패스스루
- `XR_FB_display_refresh_rate`: 새로 고침 빈도 쿼리/설정
- `XR_KHR_composition_layer_depth`: 더 나은 재투영을 위한 깊이 제출

**공급업체별 확장**:
- `XR_FB_foveation`: Meta의 중심 와 렌더링
- `XR_VARJO_quad_views`: Varjo의 포커스 디스플레이 + 주변 뷰
- `XR_MSFT_spatial_anchor`: Microsoft의 영구 공간 앵커

**미래 확장 (2026년 개발 중)**:
- `XR_EXT_full_body_tracking`: 표준화된 전신 골격
- `XR_KHR_facial_expression`: 얼굴 추적 상호 운용성
- `XR_EXT_force_feedback`: 고급 햅틱

### 업계 채택 현황 (2026년)

**엔진**:
- Unity: 전체 OpenXR 지원 (2021년+)
- Unreal Engine: 네이티브 OpenXR 지원 (UE 4.26+, 2021년)
- Godot: 커뮤니티 OpenXR 플러그인

**플랫폼**:
- Meta Quest: 새 앱의 90%가 OpenXR 사용
- SteamVR: 85% 호환성
- PSVR2: 채택 증가 (PC 모드는 필요)
- Vision Pro: 60% (여전히 ARKit 사용이 많음)

**실현된 이점**:
- 개발 시간: 멀티 플랫폼의 경우 30-50% 감소
- 버그 감소: 플랫폼별 코드 최소화
- 미래 지향적: 새 장치가 자동으로 호환

**WIA-VR OpenXR 요구 사항**:
- **표준 계층**: OpenXR 1.0 핵심 준수 필수
- **프리미엄 계층**: 주요 확장 기능이 있는 OpenXR 1.1+
- **인증**: OpenXR 적합성 테스트 스위트 통과 필수

## 5.2 WebXR Device API

### 오픈 웹의 VR/AR

**WebXR**는 몰입형 경험을 웹 브라우저에 제공하며 앱 설치가 필요하지 않습니다.

**W3C 사양**:
- WebXR Device API (후보 권장 사항, 2020년)
- WebVR의 후속 (2019년 더 이상 사용되지 않음)
- VR 및 AR 모드 모두 지원

### API 개요

**JavaScript 기반 API**:

```javascript
// XR 지원 확인
if (navigator.xr) {
  // VR 세션 요청
  navigator.xr.requestSession('immersive-vr').then(session => {
    // VR 세션 시작
    session.requestAnimationFrame(onXRFrame);
  });
}

// 렌더 루프
function onXRFrame(time, frame) {
  // 포즈 가져오기
  let pose = frame.getViewerPose(referenceSpace);

  // 각 눈에 대해 렌더링
  for (let view of pose.views) {
    // 이 뷰에 대한 장면 렌더링
    renderEye(view);
  }

  // 다음 프레임 요청
  session.requestAnimationFrame(onXRFrame);
}
```

**세션 유형**:
- `inline`: 비몰입형 (웹페이지의 3D)
- `immersive-vr`: 전체 VR 모드 (디스플레이 인수)
- `immersive-ar`: AR 모드 (카메라 패스스루)

### 브라우저 지원 (2026년)

**Chrome/Edge (Chromium)**:
- 전체 WebXR VR 지원 (2020년+)
- Android의 WebXR AR (ARCore 장치)
- Quest 브라우저: 전체 VR 지원
- 전반적으로 최고의 지원

**Firefox**:
- WebXR VR 지원
- AR 제한적
- Chrome보다 최적화 덜 됨

**Safari (iOS/iPadOS)**:
- WebXR AR 지원 추가 (2023년)
- 제한된 VR 지원 (헤드셋 하드웨어 없음)
- Vision Pro: 공간 웹 경험 (맞춤형 구현)

**Meta Quest 브라우저**:
- Chromium 기반
- 전체 WebXR 지원
- 손 추적 지원
- 큰 사용자 기반 (헤드셋과 함께 번들)

### 프레임워크 및 라이브러리

**Three.js**:
- 웹용 가장 인기 있는 3D 라이브러리
- 내장된 WebXR 지원
- 광범위한 예제 및 커뮤니티

**Babylon.js**:
- Microsoft 지원
- 강력한 WebXR 구현
- 성능 및 도구에 중점

**A-Frame**:
- Mozilla 프로젝트 (커뮤니티 유지 관리)
- 선언적 HTML 기반 VR
- 웹 개발자가 배우기 쉬움

**PlayCanvas**:
- 클라우드 기반 게임 엔진
- WebGL/WebXR 배포
- 비주얼 에디터

### 사용 사례 및 제한 사항

**이상적인 용도**:
- 제품 시각화 (가구, 자동차)
- 전자 상거래 ("구매 전 체험")
- 교육 콘텐츠
- 교육 시뮬레이션 (브라우저를 통해 액세스 가능)
- 데모 및 프로토타입
- 크로스 플랫폼 경험

**제한 사항**:
- **성능**: 네이티브 앱 성능의 60-70%
- **기능**: 고급 하드웨어 기능에 대한 제한된 액세스
- **배포**: 사용자가 URL을 찾아야 함 (앱 스토어 없음)
- **지속성**: 브라우저 보안이 로컬 저장소 제한
- **대역폭**: 큰 자산을 다운로드해야 함 (앱 설치 대비)

**향후 개발**:
- WebAssembly + WebGPU: 네이티브에 가까운 성능
- WebXR Layers API: 향상된 UI 렌더링
- WebXR Hit Test API: 더 나은 AR 배치
- WebXR Anchors API: 영구 AR 콘텐츠

**WIA-VR WebXR 표준**:
- **표준 계층**: 플랫폼 브라우저에서 WebXR Device API 지원
- **프리미엄 계층**: 손 추적, AR을 포함한 전체 WebXR 기능 세트
- **모범 사례**: 콘텐츠는 브라우저 전반에서 우아하게 저하되어야 함

## 5.3 디스플레이 및 성능 표준

### 해상도 요구 사항

**WIA-VR 해상도 계층**:

**기본 계층** (입문 수준 허용):
- 최소: 눈당 1440×1600 (2.3 MP/눈)
- PPD: 15-18 (보이는 픽셀, 일반 사용에 허용)
- 예: Quest 2, Pico Neo 3

**표준 계층** (현재 주류):
- 최소: 눈당 1832×1920 (3.5 MP/눈)
- PPD: 20-25 (대부분의 콘텐츠에 편안함)
- 예: Quest 3, Pico 4

**프리미엄 계층** (고급):
- 최소: 눈당 2160×2160 (4.7 MP/눈)
- PPD: 28-35 (읽기, 세부 작업에 적합)
- 예: Vision Pro, Varjo, Bigscreen Beyond

**울트라 계층** (미래, 망막 해상도):
- 목표: 눈당 4000×4000+ (16 MP/눈)
- PPD: 60+ (현실과 구별할 수 없음)
- 상태: 중심 와 렌더링 필요, 아직 상용화되지 않음

**해상도 테스트**:
- 표준 시력 검사표 테스트 (거리에서 문자 읽기)
- 스크린 도어 효과 가시성 (픽셀 사이의 검은 그리드)
- 다양한 크기의 텍스트 가독성
- 세부 사항 인식 (가장자리의 앨리어싱)

### 재생률 사양

**WIA-VR 재생률 표준**:

**표준 이하** (허용되지 않음):
- 60Hz: 구식, 높은 멀미 위험
- 72Hz: 최소 모바일 VR (2019-2020년), 현재 표준 이하

**기본 계층**:
- 90Hz: 최소 허용 (프레임당 11.1ms)
- 대부분의 사용자가 대부분의 콘텐츠에서 부드러움
- 입문 수준 독립형 VR

**표준 계층**:
- 90-120Hz: 권장 범위
- 120Hz 선호 (프레임당 8.3ms)
- 눈에 띄게 더 부드러운 머리 추적
- 현재 주류

**프리미엄 계층**:
- 120-144Hz: 고성능
- 거의 인지할 수 없는 새로 고침에 접근
- 상당한 GPU 성능 필요

**가변 재생률**:
- 일부 헤드셋에서 지원 (모니터용 FreeSync/G-Sync와 유사)
- 프레임 속도가 변동할 때 끊김 제거
- 프리미엄 계층+ 권장

**재생률 테스트**:
- 실제 새로 고침 전달의 광도계 측정
- 프레임 시간 일관성 (지터 테스트)
- 모션 선명도 테스트 (움직이는 패턴, 스크롤 텍스트)

### 색 정확도 및 색역

**WIA-VR 색상 표준**:

**기본 계층**:
- 최소 sRGB 색 영역 (가장 일반적인 표준)
- 6비트+FRC 또는 8비트 색 깊이
- 게임 및 엔터테인먼트에 적합

**표준 계층**:
- DCI-P3 색 영역 (시네마 표준, sRGB보다 ~25% 넓음)
- 최소 8비트 색 깊이
- Delta E <5 색 정확도
- 현재 주류 기대

**프리미엄 계층**:
- DCI-P3 이상 (Rec. 2020 목표)
- 10비트 색 깊이 (HDR 지원)
- Delta E <3 색 정확도
- 전문가/창작 작업 품질

**테스트 요구 사항**:
- 색도계 측정
- 업계 표준 테스트 패턴
- 보정 전/후 결과

### 잔상 및 모션 블러

**낮은 잔상 디스플레이**:
최신 VR 디스플레이는 "낮은 잔상"을 사용합니다 (픽셀이 각 프레임의 일부에만 켜진 다음 어두워짐):

**중요한 이유**:
- 기존 디스플레이: 픽셀이 다음 프레임까지 켜진 상태 유지 (샘플 앤 홀드)
- 샘플 앤 홀드 중 머리 움직임 = 모션 블러
- 낮은 잔상: 픽셀이 짧게 깜박임 (스트로브처럼), 흐림 제거

**WIA-VR 잔상 표준**:
- **최대**: 3ms 잔상 (픽셀 온 타임)
- **목표**: <2ms (더 나은 선명도)
- **테스트**: 고속 카메라 캡처, 추적 카메라 방법론

## 5.4 추적 정확도 표준

### 위치 추적 정밀도

**WIA-VR 위치 정확도 계층**:

**기본 계층**:
- 정확도: ±10mm (1cm) 95백분위수
- 업데이트 속도: 최소 60Hz
- 볼륨: 1.5m × 1.5m (착석/서 있음)

**표준 계층**:
- 정확도: ±5mm (0.5cm) 95백분위수
- 업데이트 속도: 90Hz
- 볼륨: 3m × 3m (룸 스케일)
- 지터: 프레임 간 <2mm

**프리미엄 계층**:
- 정확도: ±2mm 95백분위수
- 업데이트 속도: 120Hz+
- 볼륨: 10m × 10m (창고 규모 가능)
- 지터: 프레임 간 <1mm
- 예측 추적 (효과적인 대기 시간 감소)

**테스트 방법론**:
- 알려진 위치가 있는 로봇 팔 (실측값)
- 통계 분석 (평균 오류, 표준 편차, 백분위수)
- 엣지 케이스 테스트 (빠른 움직임, 폐색)

### 회전 추적 정밀도

**WIA-VR 회전 정확도 계층**:

**기본 계층**:
- 정확도: 축당 ±2°
- 업데이트 속도: 최소 60Hz

**표준 계층**:
- 정확도: 축당 ±1°
- 업데이트 속도: 90Hz
- 시간 경과에 따른 최소 드리프트

**프리미엄 계층**:
- 정확도: 축당 ±0.5°
- 업데이트 속도: 120Hz+
- 드리프트 보정 알고리즘

**IMU 센서 요구 사항**:
- 가속도계: ±2g 범위, <0.01g 노이즈
- 자이로스코프: ±2000°/s 범위, <0.1°/s 드리프트
- 자력계: 선택 사항 (드리프트 보정용)
- 센서 융합: 칼만 필터 또는 동등

### 손 추적 사양

**표준 수준**:

**테스트**:

- 표준화된 제스처 시퀀스
- 폐색 시나리오

## 5.5 안전 및 편안함 표준

### IPD 조정 범위

**동공 간 거리 (IPD)**: 동공 사이의 거리

**인간 IPD 분포**:
- 평균 성인: 63mm
- 범위 (5-95백분위수): 58-72mm
- 어린이: 50-60mm
- 좁은 IPD (<58mm): 성인의 ~20%, 대부분의 어린이
- 넓은 IPD (>72mm): 성인의 ~5%

**WIA-VR IPD 표준**:

**기본 계층**:
- 고정 IPD 또는 2-3개 사전 설정 위치
- 60-68mm 범위 수용 필수
- 비용 제약 장치에 허용

**표준 계층**:
- 연속 IPD 조정: 58-72mm 범위
- 물리적 렌즈 이동 (디지털 조정 아님)
- 사용자 프로필별로 저장된 IPD 설정

**프리미엄 계층**:
- 연속 조정: 56-75mm 범위
- 자동 IPD 감지 (눈 추적)
- 애플리케이션별 IPD 미세 조정

**어린이 장치**:
- 확장 범위: 50-72mm
- 연령에 적합한 콘텐츠 제한
- 내장된 사용 시간 제한

### 무게 분배 및 편안함

**WIA-VR 무게 및 균형 표준**:

**무게 제한**:
- **허용**: 머리 착용 질량에 대해 총 <600g
- **목표**: <500g (장기 사용에 편안함)
- **이상**: <400g (안경에 접근)
- **미래**: <200g (AR 안경 목표)

**균형 요구 사항**:
- 전방 무거움: 목 긴장, 얼굴 압력 유발
- 후방 균형추: 총 무게를 추가하지만 하중 균형
- 상단 스트랩 ("할로" 디자인): 머리 상단에 무게 분산
- 최적: 50-60% 전방 무게, 40-50% 후방

**접촉 지점**:
- 이마: 주요 하중 지지 (패딩, 넓은 접촉)
- 광대뼈: 보조 지원 (최소 압력)
- 머리 상단: 선택 사항 (할로 스트랩)
- **피해야 할 것**: 코다리 (통증 유발, 안경 간섭)

**안면 인터페이스**:
- 재료: 의료용 실리콘 또는 폼
- 세척 가능/교체 가능
- 저자극성
- 통풍 (김 서림, 열 축적 최소화)
- 여러 크기 (좁은/표준/넓은 얼굴 크기)

**테스트**:
- 장기 착용 연구 (2+ 시간 세션)
- 압력 매핑 (얼굴 접촉 지점의 센서)
- 사용자 편안함 설문 조사 (다양한 인구 통계)
- 빨간 자국/들뜸 평가

### 열 관리

**열원**:
- 프로세서/GPU: 주요 열 발생기 (독립형 VR에서 10-15W)
- 디스플레이 백라이트: LCD 헤드셋에서 중요
- 배터리: 방전 중 열 발생
- 사용자 얼굴: 밀폐된 공간에 50-100W 열 부하 추가

**WIA-VR 열 표준**:

**표면 온도**:
- 얼굴 접촉 표면: 최대 <40°C (104°F)
- 외부 표면: 최대 <45°C (113°F)
- 권장: <37°C 얼굴 접촉 (체온)

**냉각 전략**:
- 수동 냉각: 히트 싱크, 히트 파이프, 열 패드
- 능동 냉각: 팬 (소음 <30dB 요구 사항)
- 환기: 공기 간격, 흡입/배기 경로
- 열 스로틀링: 과열 전에 성능 감소

**테스트**:
- 열화상 카메라 매핑
- 장기간 고부하 시나리오
- 주변 온도 변화 (20-35°C 환경)

### 청색광 노출

**우려 사항**:
- 눈에서 인치 떨어진 디스플레이가 청색광 방출
- 장기 노출은 수면 주기에 영향을 미칠 수 있음 (일주기 리듬)
- 극단적인 노출로 잠재적인 망막 손상 (논쟁 중)

**WIA-VR 청색광 표준**:
- 청색광 위험 위험 그룹: RG0 또는 RG1이어야 함 (IEC 62471 표준)
- RG0: 위험 면제 (가장 안전)
- RG1: 낮은 위험 (일반 사용에 허용)
- RG2/RG3: 소비자 VR에서 허용되지 않음

**완화 기능**:
- 청색광 필터링 모드 (청색 스펙트럼 감소, 따뜻한 이미지)
- 사용 시간 알림
- 저녁 사용을 위한 "야간 모드"

### 멀미 완화

**VR 멀미의 원인**:
- 감각 충돌 (눈은 움직임을 보지만 내이는 느끼지 못함)
- 낮은 프레임 속도 또는 일관성 없는 프레임 시간
- 대기 시간 (머리 움직임 → 디스플레이 업데이트 지연)
- 인공 이동 (특히 부드러운 이동)
- 시야 조작

**WIA-VR 멀미 예방 표준**:

**기술 요구 사항**:
- 프레임 속도: 최소 90Hz, 120Hz 권장
- 대기 시간: <20ms 모션 투 포톤
- 프레임 시간 일관성: 프레임 간 <5% 변화

**필수 편안함 설정**:
- **이동 옵션**:
  - 텔레포트 (가장 편안함)
  - 비네팅이 있는 부드러운 이동
  - 스냅 회전 (부드러운 회전 대비)
- **시야 축소**: 이동 중 FOV를 좁히는 옵션
- **편안함 수준 라벨링**: 편안함 / 보통 / 강렬함

**처음 사용자 경험**:
- 튜토리얼 강조: 휴식을 취하고 불편하면 중지
- 점진적 온보딩 (정지 상태로 시작, 움직임으로 진행)
- "VR 레그" 교육 (시간이 지남에 따라 내성이 쌓임)

**콘텐츠 등급 시스템**:
- **편안함**: 정지 또는 최소 움직임, 낮은 멀미 위험
- **보통**: 일부 이동, 때때로 빠른 움직임
- **강렬함**: 빠른 속도, 자유 이동, 롤러코스터 경험

## 5.6 콘텐츠 표준

### 3D 자산 형식

**WIA-VR 지원 자산 형식**:

**glTF 2.0** (기본 표준):
- Khronos Group 오픈 표준
- "3D의 JPEG" - 범용 전송 형식
- JSON + 바이너리 지오메트리 + 텍스처
- PBR (물리 기반 렌더링) 재료
- 애니메이션 지원
- 확장: Draco 압축, KTX 텍스처

**USD (Universal Scene Description)**:
- Pixar 오픈 표준
- 복잡한 장면 구성
- 크고 다중 자산 장면에 이상적
- AR/VR에 점점 더 채택
- Vision Pro용 Apple의 선호 형식

**FBX**:
- Autodesk 독점 (하지만 광범위하게 지원됨)
- 게임 엔진용 업계 표준
- 애니메이션, 리깅, 복잡한 계층 구조
- 큰 파일 크기 (웹 배포에 덜 이상적)

**OBJ**:
- 레거시 형식, 지오메트리만
- 재료, 애니메이션 또는 PBR 지원 없음
- 간단한 모델에만 허용

**권장 접근 방식**:
- 개발: 네이티브 도구 형식 (Blender, Maya 등)
- 교환: glTF 2.0 (필요에 따라 확장 포함)
- 런타임: 엔진별 최적화 형식

**WIA-VR 요구 사항**:
- **표준 계층**: glTF 2.0 가져오기/내보내기 지원
- **프리미엄 계층**: glTF 2.0 + USD 지원

### 비디오 형식

**360° / 3D 비디오 표준**:

**모노스코픽 360°**:
- 해상도: 최소 4K (3840×2160), 8K 권장
- 형식: 정방형 투영
- 사용 사례: 간단한 360° 비디오, 저비용 VR 비디오

**스테레오스코픽 360°** (3D):
- 해상도: 눈당 5K+ (나란히 또는 위아래)
- 형식: 정방형 스테레오
- 사용 사례: 깊이가 있는 몰입형 VR 비디오

**볼륨메트릭 비디오** (신흥):
- 포인트 클라우드 또는 메시 시퀀스
- 시청자가 장면 내에서 이동할 수 있음 (6DOF)
- 대규모 데이터 요구 사항 (Gbps 스트림)
- 상태: 전문 사용 사례 (스포츠, 콘서트)

**코덱**:
- **H.264/AVC**: 범용 지원, 보통 효율성
- **H.265/HEVC**: 50% 더 나은 압축, 라이선스 비용, 좋은 지원
- **AV1**: 로열티 프리, 우수한 압축, 증가하는 지원 (2024년+)
- **공간 비디오** (Apple): Vision Pro용 H.265 기반 MV-HEVC

**WIA-VR 비디오 표준**:
- **기본**: H.264, 4K 360° 모노, 30fps
- **표준**: H.265, 5K 360° 스테레오, 60fps
- **프리미엄**: AV1 또는 H.265, 8K+ 스테레오, 60fps, 공간 오디오
- **HDR**: 선택 사항이지만 권장 (프리미엄 계층)

- 임상 검증 연구
- 환자 안전 프로토콜

### 준수 테스트 절차

**1단계: 자체 평가**:
- 제조업체가 상세한 설문지 작성
- 기술 사양 문서화
- 제조업체 실험실의 테스트 결과
- 개인 정보 보호/보안 문서

**2단계: 문서 검토**:
- WIA 표준 위원회가 제출 검토
- 예비 평가 (통과/명확화/실패)
- 필요한 경우 후속 질문

**3단계: 독립 실험실 테스트**:
- 인증된 타사 테스트 시설
- 하드웨어 테스트: 디스플레이, 추적, 대기 시간, 열
- 소프트웨어 테스트: OpenXR 적합성, 충돌 테스트
- 안전 테스트: 청색광, 열 제한, 기계적
- 기간: 2-4주

**4단계: 보안 감사** (전문가+):
- 타사 보안 회사
- 침투 테스트
- 개인 정보 보호 정책 검토
- 데이터 흐름 분석
- 소스 코드 검토 (높은 보안 계층의 경우)

**5단계: 사용자 테스트** (전문가+):
- 다양한 사용자 패널 (나이, 크기, 능력, 경험)
- 편안함 평가 (2+ 시간 세션)
- 접근성 검증
- 멀미율
- 사용성 점수

**6단계: 인증 결정**:
- WIA 표준 위원회 최종 검토
- 인증 부여 / 조건부 (사소한 수정) / 거부
- 공개 인증 목록
- 배지 및 마케팅 자료 제공

### 지속적인 준수

**연간 갱신**:
- 변경되지 않은 모델에 대한 축약 테스트
- 중요한 업데이트에 대한 전체 테스트
- 지원 사고 및 사용자 불만 검토

**모니터링**:
- 무작위 시장 감사
- 사용자 피드백 검토
- 보안 사고 대응

**취소**:
- 주요 안전 사고 → 즉시 검토, 가능한 취소
- 개인 정보 보호 위반 → 조사, 잠재적 취소
- 허위 인증 주장 → 취소 및 처벌
