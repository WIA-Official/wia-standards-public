# Chapter 5: Technical Standards & Specifications

## 5.1 OpenXR Standard

### The Foundation of Interoperability

OpenXR is the industry's answer to VR/AR platform fragmentation. Developed by the Khronos Group (same organization behind OpenGL, Vulkan, WebGL), OpenXR provides a royalty-free, open standard API for XR applications and devices.

**History**:
- **2017**: OpenXR working group formed (founding members: Valve, Microsoft, Epic, Unity, Meta, Google, HTC)
- **July 2019**: OpenXR 1.0 specification released
- **July 2021**: OpenXR 1.0.17 with significant extensions
- **2023-2025**: Near-universal adoption across industry
- **2026**: OpenXR 1.1 adds enhanced features (dynamic foveated rendering, full-body tracking extensions)

### API Architecture

**Core Concept**: "Write once, run everywhere"

Developers write to OpenXR API → Runtime translates to device-specific implementation

```
[Application using OpenXR API]
         ↓
[OpenXR Runtime] ← Platform-specific (Meta, Valve, Microsoft, etc.)
         ↓
[VR/AR Hardware]
```

**Key Components**:

**1. Instance & System**:
```cpp
XrInstance instance;  // Connection to OpenXR runtime
XrSystemId system;    // Specific XR system (headset)
```

**2. Session**:
```cpp
XrSession session;    // Active VR/AR session
XrSessionState state; // Running, visible, focused, etc.
```

**3. Spaces**:
```cpp
XrSpace space;        // Coordinate systems (world, stage, view)
```

**4. Actions**:
```cpp
XrActionSet actionSet;  // Group of user inputs
XrAction grabAction;    // Specific action (button, trigger)
```

**5. Composition Layers**:
```cpp
XrCompositionLayerProjection projLayer; // Rendered content layers
XrCompositionLayerQuad quadLayer;       // 2D UI overlays
```

### Runtime Implementations

**Major OpenXR Runtimes (2026)**:

**Meta Quest OpenXR Runtime**:
- Full OpenXR 1.0 compliance
- Extensions: Hand tracking, passthrough, eye tracking (Quest Pro)
- 90% of Quest apps now use OpenXR (transition from proprietary Oculus SDK)

**SteamVR OpenXR Runtime**:
- Supports any SteamVR-compatible headset
- Works with Index, Vive, Quest via Link, Windows MR, etc.
- Most mature and feature-complete runtime

**Windows Mixed Reality OpenXR Runtime**:
- Deprecated but still functional (2026)
- Supports older WMR headsets (HP Reverb G2, etc.)

**Varjo OpenXR Runtime**:
- Professional features (high-resolution, video pass-through)
- Extensions for enterprise use cases

**PlayStation VR2**:
- OpenXR support added 2024
- Enables cross-platform PC VR content

**Apple visionOS**:
- Custom implementation with Apple-specific extensions
- Full OpenXR support announced 2025 (previously ARKit-focused)

### Extension System

OpenXR's extensibility allows hardware vendors to expose unique features:

**Core Extensions (widely supported)**:
- `XR_EXT_hand_tracking`: Hand skeleton tracking
- `XR_EXT_eye_gaze_interaction`: Eye tracking for gaze input
- `XR_FB_passthrough`: Camera passthrough for MR
- `XR_FB_display_refresh_rate`: Query/set refresh rate
- `XR_KHR_composition_layer_depth`: Depth submission for better reprojection

**Vendor-Specific Extensions**:
- `XR_FB_foveation`: Meta's foveated rendering
- `XR_VARJO_quad_views`: Varjo's focus display + peripheral views
- `XR_MSFT_spatial_anchor`: Microsoft's persistent spatial anchors

**Future Extensions (In Development 2026)**:
- `XR_EXT_full_body_tracking`: Standardized full-body skeleton
- `XR_KHR_facial_expression`: Facial tracking interoperability
- `XR_EXT_force_feedback`: Advanced haptics

### Industry Adoption Status (2026)

**Engines**:
- Unity: Full OpenXR support (2021+)
- Unreal Engine: Native OpenXR support (UE 4.26+, 2021)
- Godot: Community OpenXR plugin

**Platforms**:
- Meta Quest: 90% of new apps use OpenXR
- SteamVR: 85% compatibility
- PSVR2: Growing adoption (PC mode requires it)
- Vision Pro: 60% (still heavy ARKit usage)

**Benefits Realized**:
- Development time: 30-50% reduction for multi-platform
- Bug reduction: Platform-specific code minimized
- Future-proofing: New devices automatically compatible

**WIA-VR OpenXR Requirements**:
- **Standard Tier**: OpenXR 1.0 core compliance mandatory
- **Premium Tier**: OpenXR 1.1+ with major extensions
- **Certification**: OpenXR conformance test suite must pass

## 5.2 WebXR Device API

### VR/AR on the Open Web

**WebXR** brings immersive experiences to web browsers, no app installation required.

**W3C Specification**:
- WebXR Device API (Candidate Recommendation, 2020)
- Successor to WebVR (deprecated 2019)
- Supports both VR and AR modes

### API Overview

**JavaScript-based API**:

```javascript
// Check for XR support
if (navigator.xr) {
  // Request VR session
  navigator.xr.requestSession('immersive-vr').then(session => {
    // VR session started
    session.requestAnimationFrame(onXRFrame);
  });
}

// Render loop
function onXRFrame(time, frame) {
  // Get pose
  let pose = frame.getViewerPose(referenceSpace);

  // Render for each eye
  for (let view of pose.views) {
    // Render scene for this view
    renderEye(view);
  }

  // Request next frame
  session.requestAnimationFrame(onXRFrame);
}
```

**Session Types**:
- `inline`: Non-immersive (3D in webpage)
- `immersive-vr`: Full VR mode (takes over display)
- `immersive-ar`: AR mode (camera passthrough)

### Browser Support (2026)

**Chrome/Edge (Chromium)**:
- Full WebXR VR support (2020+)
- WebXR AR on Android (ARCore devices)
- Quest browser: Full VR support
- Best overall support

**Firefox**:
- WebXR VR support
- AR limited
- Less optimized than Chrome

**Safari (iOS/iPadOS)**:
- WebXR AR support added (2023)
- Limited VR support (no headset hardware)
- Vision Pro: Spatial web experiences (custom implementation)

**Meta Quest Browser**:
- Chromium-based
- Full WebXR support
- Hand tracking support
- Large user base (bundled with headset)

### Frameworks & Libraries

**Three.js**:
- Most popular 3D library for web
- WebXR support built-in
- Extensive examples and community

**Babylon.js**:
- Microsoft-backed
- Strong WebXR implementation
- Focus on performance and tooling

**A-Frame**:
- Mozilla project (community-maintained)
- Declarative HTML-based VR
- Easy for web developers to learn

**PlayCanvas**:
- Cloud-based game engine
- WebGL/WebXR deployment
- Visual editor

### Use Cases & Limitations

**Ideal For**:
- Product visualization (furniture, cars)
- E-commerce ("try before you buy")
- Educational content
- Training simulations (accessible via browser)
- Demos and prototypes
- Cross-platform experiences

**Limitations**:
- **Performance**: 60-70% of native app performance
- **Features**: Limited access to advanced hardware features
- **Distribution**: Users must discover URL (no app store)
- **Persistence**: Browser security limits local storage
- **Bandwidth**: Large assets must download (vs app installs)

**Future Developments**:
- WebAssembly + WebGPU: Near-native performance
- WebXR Layers API: Improved UI rendering
- WebXR Hit Test API: Better AR placement
- WebXR Anchors API: Persistent AR content

**WIA-VR WebXR Standards**:
- **Standard Tier**: WebXR Device API support in platform browser
- **Premium Tier**: Full WebXR feature set including hand tracking, AR
- **Best Practices**: Content should gracefully degrade across browsers

## 5.3 Display & Performance Standards

### Resolution Requirements

**WIA-VR Resolution Tiers**:

**Basic Tier** (Entry-level acceptable):
- Minimum: 1440×1600 per eye (2.3 MP/eye)
- PPD: 15-18 (visible pixels, acceptable for casual use)
- Examples: Quest 2, Pico Neo 3

**Standard Tier** (Current mainstream):
- Minimum: 1832×1920 per eye (3.5 MP/eye)
- PPD: 20-25 (comfortable for most content)
- Examples: Quest 3, Pico 4

**Premium Tier** (High-end):
- Minimum: 2160×2160 per eye (4.7 MP/eye)
- PPD: 28-35 (suitable for reading, detailed work)
- Examples: Vision Pro, Varjo, Bigscreen Beyond

**Ultra Tier** (Future, retinal resolution):
- Target: 4000×4000+ per eye (16 MP/eye)
- PPD: 60+ (indistinguishable from reality)
- Status: Requires foveated rendering, not yet commercialized

**Resolution Testing**:
- Standard eye chart tests (read letters at distance)
- Screen door effect visibility (black grid between pixels)
- Text readability at various sizes
- Fine detail perception (aliasing on edges)

### Refresh Rate Specifications

**WIA-VR Refresh Rate Standards**:

**Below Standard** (Not acceptable):
- 60Hz: Obsolete, high motion sickness risk
- 72Hz: Minimum mobile VR (2019-2020), now below standard

**Basic Tier**:
- 90Hz: Minimum acceptable (11.1ms per frame)
- Smooth for most users in most content
- Entry-level standalone VR

**Standard Tier**:
- 90-120Hz: Recommended range
- 120Hz preferred (8.3ms per frame)
- Noticeably smoother head tracking
- Current mainstream

**Premium Tier**:
- 120-144Hz: High-performance
- Approaching imperceptible refresh
- Requires significant GPU power

**Variable Refresh Rate**:
- Supported on some headsets (like FreeSync/G-Sync for monitors)
- Eliminates stuttering when frame rate fluctuates
- Recommended for Premium Tier+

**Refresh Rate Testing**:
- Photometer measurements of actual refresh delivery
- Frame time consistency (jitter testing)
- Motion clarity tests (moving patterns, scrolling text)

### Color Accuracy & Gamut

**WIA-VR Color Standards**:

**Basic Tier**:
- sRGB color gamut minimum (most common standard)
- 6-bit+FRC or 8-bit color depth
- Adequate for gaming and entertainment

**Standard Tier**:
- DCI-P3 color gamut (cinema standard, ~25% wider than sRGB)
- 8-bit color depth minimum
- Delta E <5 color accuracy
- Current mainstream expectation

**Premium Tier**:
- DCI-P3 or wider (Rec. 2020 target)
- 10-bit color depth (HDR support)
- Delta E <3 color accuracy
- Professional/creative work quality

**Testing Requirements**:
- Colorimeter measurements
- Industry standard test patterns
- Before/after calibration results

### Persistence & Motion Blur

**Low Persistence Displays**:
Modern VR displays use "low persistence" (pixels lit for only part of each frame, then go dark):

**Why it matters**:
- Traditional displays: Pixels stay lit until next frame (sample-and-hold)
- Moving head during sample-and-hold = motion blur
- Low persistence: Pixels flash briefly (like strobe), eliminating blur

**WIA-VR Persistence Standards**:
- **Maximum**: 3ms persistence (pixel on-time)
- **Target**: <2ms (better clarity)
- **Testing**: High-speed camera captures, pursuit camera methodology

## 5.4 Tracking Accuracy Standards

### Positional Tracking Precision

**WIA-VR Position Accuracy Tiers**:

**Basic Tier**:
- Accuracy: ±10mm (1cm) 95th percentile
- Update rate: 60Hz minimum
- Volume: 1.5m × 1.5m (seated/standing)

**Standard Tier**:
- Accuracy: ±5mm (0.5cm) 95th percentile
- Update rate: 90Hz
- Volume: 3m × 3m (room-scale)
- Jitter: <2mm frame-to-frame

**Premium Tier**:
- Accuracy: ±2mm 95th percentile
- Update rate: 120Hz+
- Volume: 10m × 10m (warehouse-scale possible)
- Jitter: <1mm frame-to-frame
- Predictive tracking (reduces effective latency)

**Testing Methodology**:
- Robotic arm with known positions (ground truth)
- Statistical analysis (mean error, standard deviation, percentiles)
- Edge case testing (fast movements, occlusion)

### Rotational Tracking Precision

**WIA-VR Rotation Accuracy Tiers**:

**Basic Tier**:
- Accuracy: ±2° per axis
- Update rate: 60Hz minimum

**Standard Tier**:
- Accuracy: ±1° per axis
- Update rate: 90Hz
- Minimal drift over time

**Premium Tier**:
- Accuracy: ±0.5° per axis
- Update rate: 120Hz+
- Drift correction algorithms

**IMU Sensor Requirements**:
- Accelerometer: ±2g range, <0.01g noise
- Gyroscope: ±2000°/s range, <0.1°/s drift
- Magnetometer: Optional (for drift correction)
- Sensor fusion: Kalman filter or equivalent

### Hand Tracking Specifications

**WIA-VR Hand Tracking Tiers** (for devices supporting hand tracking):

**Basic Level**:
- 21-point hand skeleton (minimum)
- Accuracy: ±15mm position
- Update rate: 30Hz
- Gestures: Pinch, grab, point

**Standard Level**:
- 26-point hand skeleton (with fingertips)
- Accuracy: ±10mm position
- Update rate: 60Hz
- Occlusion: Maintain tracking when hands overlap briefly
- Gestures: Extended set (thumb up, peace sign, etc.)

**Premium Level**:
- 26+ point skeleton (including metacarpals)
- Accuracy: ±5mm position
- Update rate: 90Hz
- Robust occlusion handling
- Finger orientation accuracy: ±5°
- Gesture recognition: ML-based custom gestures

**Testing**:
- Mocap ground truth comparison
- Standardized gesture sequences
- Occlusion scenarios
- Diverse hand sizes and skin tones

### Eye Tracking Requirements

**WIA-VR Eye Tracking Tiers** (for Premium+ devices):

**Basic Eye Tracking**:
- Accuracy: ±3-5° gaze direction
- Update rate: 60Hz
- Calibration required
- Use case: Gaze-based UI, basic foveated rendering

**Standard Eye Tracking**:
- Accuracy: ±2° gaze direction
- Update rate: 90Hz
- Minimal calibration (or calibration-free)
- Vergence detection (focus distance)
- Use case: Advanced foveated rendering, social presence

**Premium Eye Tracking**:
- Accuracy: ±1° gaze direction
- Update rate: 120Hz
- No calibration required
- Pupil diameter measurement
- Eye openness detection (blink tracking)
- Use case: All applications, medical research

**Privacy Requirements**:
- Raw eye images never leave device
- Processed eye data encrypted
- User consent for eye tracking features
- Clear indicators when eye tracking active
- Ability to disable per application

## 5.5 Safety & Comfort Standards

### IPD Adjustment Range

**Inter-Pupillary Distance (IPD)**: Distance between pupils

**Human IPD Distribution**:
- Average adult: 63mm
- Range (5th-95th percentile): 58-72mm
- Children: 50-60mm
- Narrow IPD (<58mm): ~20% of adults, most children
- Wide IPD (>72mm): ~5% of adults

**WIA-VR IPD Standards**:

**Basic Tier**:
- Fixed IPD or 2-3 preset positions
- Must accommodate 60-68mm range
- Acceptable for cost-constrained devices

**Standard Tier**:
- Continuous IPD adjustment: 58-72mm range
- Physical lens movement (not digital adjustment)
- IPD setting saved per user profile

**Premium Tier**:
- Continuous adjustment: 56-75mm range
- Automatic IPD detection (eye tracking)
- Per-application IPD fine-tuning

**Children's Devices**:
- Extended range: 50-72mm
- Age-appropriate content restrictions
- Usage time limits built-in

### Weight Distribution & Comfort

**WIA-VR Weight & Balance Standards**:

**Weight Limits**:
- **Acceptable**: <600g total for head-worn mass
- **Target**: <500g (comfortable for extended use)
- **Ideal**: <400g (approaching eyeglasses)
- **Future**: <200g (AR glasses target)

**Balance Requirements**:
- Front-heavy: Causes neck strain, face pressure
- Rear counterweight: Adds total weight but balances load
- Top strap ("halo" design): Distributes weight to top of head
- Optimal: 50-60% weight forward, 40-50% rear

**Contact Points**:
- Forehead: Primary load-bearing (padded, broad contact)
- Cheekbones: Secondary support (minimal pressure)
- Top of head: Optional (halo straps)
- **Avoid**: Bridge of nose (causes pain, glasses interference)

**Facial Interface**:
- Material: Medical-grade silicone or foam
- Washable/replaceable
- Hypoallergenic
- Ventilation (minimize fogging, heat buildup)
- Multiple sizes (narrow/standard/wide face sizes)

**Testing**:
- Extended wear studies (2+ hour sessions)
- Pressure mapping (sensors on face contact points)
- User comfort surveys (diverse demographics)
- Red mark/indentation assessment

### Thermal Management

**Heat Sources**:
- Processor/GPU: Primary heat generator (10-15W in standalone VR)
- Display backlight: Significant in LCD headsets
- Battery: Generates heat during discharge
- User's face: Adds 50-100W heat load in enclosed space

**WIA-VR Thermal Standards**:

**Surface Temperatures**:
- Face-contact surfaces: <40°C maximum (104°F)
- External surfaces: <45°C maximum (113°F)
- Recommended: <37°C face contact (body temperature)

**Cooling Strategies**:
- Passive cooling: Heat sinks, heat pipes, thermal pads
- Active cooling: Fans (noise <30dB requirement)
- Ventilation: Air gaps, intake/exhaust paths
- Thermal throttling: Reduce performance before overheating

**Testing**:
- Thermal camera mapping
- Prolonged high-load scenarios
- Ambient temperature variations (20-35°C environments)

### Blue Light Exposure

**Concerns**:
- Displays inches from eyes emit blue light
- Extended exposure may affect sleep cycles (circadian rhythm)
- Potential retinal damage with extreme exposure (debated)

**WIA-VR Blue Light Standards**:
- Blue light hazard risk group: Must be RG0 or RG1 (IEC 62471 standard)
- RG0: Exempt from risk (safest)
- RG1: Low risk (acceptable for general use)
- RG2/RG3: Not permitted in consumer VR

**Mitigation Features**:
- Blue light filtering mode (reduce blue spectrum, warmer image)
- Usage time reminders
- "Night mode" for evening use

### Motion Sickness Mitigation

**Causes of VR Motion Sickness**:
- Sensory conflict (eyes see movement, inner ear doesn't feel it)
- Low frame rate or inconsistent frame times
- Latency (head movement → display update delay)
- Artificial locomotion (especially smooth locomotion)
- Field of view manipulation

**WIA-VR Motion Sickness Prevention Standards**:

**Technical Requirements**:
- Frame rate: 90Hz minimum, 120Hz recommended
- Latency: <20ms motion-to-photon
- Frame time consistency: <5% variation frame-to-frame

**Comfort Settings Required**:
- **Locomotion options**:
  - Teleportation (most comfortable)
  - Smooth locomotion with vignetting
  - Snap turning (vs smooth turning)
- **Field of view reduction**: Option to narrow FOV during movement
- **Comfort level labeling**: Comfortable / Moderate / Intense

**First-Time User Experience**:
- Tutorial emphasizing: Take breaks, stop if uncomfortable
- Gradual onboarding (start with stationary, progress to movement)
- "VR legs" education (tolerance builds over time)

**Content Rating System**:
- **Comfortable**: Stationary or minimal movement, low motion sickness risk
- **Moderate**: Some locomotion, occasional fast movements
- **Intense**: Fast-paced, free movement, roller coaster experiences

## 5.6 Content Standards

### 3D Asset Formats

**WIA-VR Supported Asset Formats**:

**glTF 2.0** (Primary Standard):
- Khronos Group open standard
- "JPEG of 3D" - universal transmission format
- JSON + binary geometry + textures
- PBR (Physically Based Rendering) materials
- Animation support
- Extensions: Draco compression, KTX textures

**USD (Universal Scene Description)**:
- Pixar open standard
- Complex scene composition
- Ideal for large, multi-asset scenes
- Increasingly adopted for AR/VR
- Apple's preferred format for Vision Pro

**FBX**:
- Autodesk proprietary (but widely supported)
- Industry standard for game engines
- Animation, rigging, complex hierarchies
- Large file sizes (less ideal for web distribution)

**OBJ**:
- Legacy format, geometry only
- No materials, animations, or PBR support
- Acceptable for simple models only

**Recommended Approach**:
- Development: Native tool formats (Blender, Maya, etc.)
- Exchange: glTF 2.0 (with extensions as needed)
- Runtime: Engine-specific optimized formats

**WIA-VR Requirements**:
- **Standard Tier**: glTF 2.0 import/export support
- **Premium Tier**: glTF 2.0 + USD support

### Video Formats

**360° / 3D Video Standards**:

**Monoscopic 360°**:
- Resolution: 4K minimum (3840×2160), 8K recommended
- Format: Equirectangular projection
- Use case: Simple 360° videos, low-cost VR video

**Stereoscopic 360°** (3D):
- Resolution: 5K+ per eye (side-by-side or top-bottom)
- Format: Equirectangular stereo
- Use case: Immersive VR video with depth

**Volumetric Video** (Emerging):
- Point clouds or mesh sequences
- Allows viewer to move within scene (6DOF)
- Massive data requirements (Gbps streams)
- Status: Specialized use cases (sports, concerts)

**Codecs**:
- **H.264/AVC**: Universal support, moderate efficiency
- **H.265/HEVC**: 50% better compression, licensing fees, good support
- **AV1**: Royalty-free, excellent compression, growing support (2024+)
- **Spatial video** (Apple): H.265-based MV-HEVC for Vision Pro

**WIA-VR Video Standards**:
- **Basic**: H.264, 4K 360° mono, 30fps
- **Standard**: H.265, 5K 360° stereo, 60fps
- **Premium**: AV1 or H.265, 8K+ stereo, 60fps, spatial audio
- **HDR**: Optional but recommended (Premium Tier)

### Spatial Audio Formats

**WIA-VR Audio Standards**:

**Minimum (Basic Tier)**:
- Stereo audio with HRTF spatialization
- Binaural audio support

**Standard Tier**:
- First-order ambisonics (B-format, 4 channels)
- Object-based audio (individual sound sources in 3D)
- Head tracking integration

**Premium Tier**:
- Higher-order ambisonics (9-25+ channels)
- Dolby Atmos or DTS:X support
- Spatial audio authoring tools

**Recommended Formats**:
- TBE (Two Big Ears) Spatial Workstation format
- Dolby Atmos ADM
- Ambisonics (AmbiX or FuMa)

### Accessibility Requirements

**WIA-VR Accessibility Mandates**:

**Visual Accessibility**:
- Colorblind modes (deuteranopia, protanopia, tritanopia simulations)
- High-contrast mode
- Subtitles/captions for all spoken content (customizable size, position)
- Audio descriptions for visual-only content

**Auditory Accessibility**:
- Visual indicators for audio cues (gunshot direction, alarms)
- Closed captions
- Visual alternatives for audio-only information

**Motor Accessibility**:
- Seated mode (all content playable seated)
- One-handed mode (for users with single hand mobility)
- Customizable control mapping
- Voice command alternatives
- Head-gaze-only interaction mode (for users unable to use hands)

**Cognitive Accessibility**:
- Adjustable difficulty
- Removal of flashing/strobing effects (photosensitivity)
- Simplified UI mode
- Tutorial/practice modes

**Age Ratings**:
- IARC (International Age Rating Coalition) integration
- Age-appropriate content filtering
- Parental controls

## 5.7 WIA-VR Certification Criteria

### Certification Tiers Summary

**WIA-VR Certified (Basic Tier)**:
- Display: 1440×1600/eye min, 90Hz
- Tracking: 6DOF, ±10mm position accuracy
- OpenXR 1.0 core compliance
- Basic safety standards met
- Privacy policy compliant

**WIA-VR Professional (Standard Tier)**:
- Display: 1832×1920/eye min, 90-120Hz
- Tracking: ±5mm position accuracy
- OpenXR 1.1 with major extensions
- IPD adjustment 58-72mm
- Advanced safety features
- Accessibility features

**WIA-VR Enterprise (Business Tier)**:
- Professional certification +
- Device management APIs
- SSO integration
- Commercial support SLA
- GDPR/CCPA/HIPAA documentation
- Enterprise security audit

**WIA-VR Medical (Healthcare Tier)**:
- Professional certification +
- Medical device regulations (where applicable)
- HIPAA compliance (US)
- Clinical validation studies
- Patient safety protocols

### Compliance Testing Procedures

**Phase 1: Self-Assessment**:
- Manufacturer completes detailed questionnaire
- Technical specifications documented
- Test results from manufacturer labs
- Privacy/security documentation

**Phase 2: Documentation Review**:
- WIA standards committee reviews submission
- Preliminary assessment (pass/clarify/fail)
- Follow-up questions if needed

**Phase 3: Independent Lab Testing**:
- Certified third-party testing facility
- Hardware tests: Display, tracking, latency, thermal
- Software tests: OpenXR conformance, crash testing
- Safety tests: Blue light, thermal limits, mechanical
- Duration: 2-4 weeks

**Phase 4: Security Audit** (Professional+):
- Third-party security firm
- Penetration testing
- Privacy policy review
- Data flow analysis
- Source code review (for high-security tiers)

**Phase 5: User Testing** (Professional+):
- Diverse user panel (age, size, ability, experience)
- Comfort assessment (2+ hour sessions)
- Accessibility validation
- Motion sickness rates
- Usability scoring

**Phase 6: Certification Decision**:
- WIA standards committee final review
- Certification granted / Conditional (minor fixes) / Denied
- Public certification listing
- Badge and marketing materials provided

### Ongoing Compliance

**Annual Renewal**:
- Abbreviated testing for unchanged models
- Full testing for significant updates
- Review of support incidents and user complaints

**Monitoring**:
- Random marketplace audits
- User feedback review
- Security incident response

**Revocation**:
- Major safety incident → Immediate review, possible revocation
- Privacy violation → Investigation, potential revocation
- False certification claims → Revocation and penalties
