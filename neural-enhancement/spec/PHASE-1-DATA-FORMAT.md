# WIA-neural-enhancement PHASE 1 ‚Äî Data Format Specification

**Standard:** WIA-neural-enhancement
**Phase:** 1 ‚Äî Data Format
**Version:** 1.0
**Status:** Stable

This PHASE defines the canonical data format for neural-
enhancement device operations: implant or wearable device
registry, neural-recording session records, stimulation
session records, decoded-intent and effector-output records,
clinical-event and adverse-event records, calibration
records, consent records, and the cross-references binding
recording to stimulation, intent decode, and effector
output. The shape interoperates with established neural-
data formats (BIDS, NWB) and clinical-research frameworks
(HL7 FHIR R5, IEC 62366-1 usability engineering, ISO 14971
risk management) so deployments do not require parallel
data models.

- Brain Imaging Data Structure (BIDS) ‚Äî including the iEEG
  and physio extensions
- Neurodata Without Borders (NWB) 2.x
- HL7 FHIR R5 ‚Äî for clinical-event resources

## ¬ß1 Scope

This PHASE applies to the data shape generated by
neural-enhancement systems: invasive (intracortical
microelectrode arrays, ECoG grids), minimally invasive
(stentrodes), and non-invasive (EEG, fNIRS, transcranial
stimulation) devices used for restoration of motor or
sensory function and for cognitive-augmentation research.
It covers recording, stimulation, decoding/effector-output,
calibration, consent, and adverse-event capture; transport
protocols are in PHASE 3 and integration with clinical
systems is in PHASE 4.

The standard is consent-aware: every record references the
governing consent record and the participant's
pseudonymised identifier; PII binding is held in a
separately access-controlled service. Any operation that
would re-identify a participant requires an explicit
elevated authorisation captured in the audit chain.

In scope: device registry, recording session, stimulation
session, decoded-intent record, effector-output record,
calibration record, consent record, adverse-event record,
clinical-context note. Out of scope: device firmware update
distribution (cross-domain to medical-device update
standards), specific neural-decoding algorithms (carried
opaquely as algorithm-version references), and clinical
trial protocol management (cross-domain to clinical-research
standards).

## ¬ß2 Device registry record

Every fielded device carries a stable registry record:

| Field             | Source / Binding                                       |
|-------------------|--------------------------------------------------------|
| `deviceRef`       | URN of form `urn:wia:nenh:device:<authority>:<id>`     |
| `udi`             | Unique Device Identifier per regulatory authority      |
| `deviceClass`     | `intracortical-array`, `ecog-grid`, `stentrode`, `eeg-headset`, `fnirs-headset`, `tdcs-tms-stimulator` |
| `vendorRef`       | URN of manufacturer                                    |
| `model`           | model designation                                      |
| `firmwareVersion` | semantic version of installed firmware                 |
| `regulatoryClearances[]` | jurisdiction-specific clearances (FDA 510(k), |
|                   | CE-MDR, KMFDS, etc.)                                   |
| `intendedUse`     | declared intended use per regulatory dossier           |
| `riskClass`       | per ISO 14971 risk classification                      |
| `installedAt`     | RFC 3339 of implantation or first-fit                  |
| `lastServiceAt`   | RFC 3339                                               |

Registry mutations are signed by the responsible clinician
and audit-logged. Firmware updates emit a follow-up
`device-firmware-updated` audit-chain entry.

## ¬ß3 Recording session record

Each recording session emits a record:

- `recordingId` ‚Äî URN
- `deviceRef` ‚Äî URN
- `participantPseudonym` ‚Äî privacy-preserving pseudonym
  (PHASE 3 ¬ß8 binding)
- `consentRef` ‚Äî URN of the governing consent record
- `protocolRef` ‚Äî URN of the active research/clinical protocol
- `sessionStart`, `sessionEnd` ‚Äî RFC 3339 with offset
- `samplingRateHz` ‚Äî declared sampling rate
- `channels[]` ‚Äî per-channel records: channel ID, anatomical
  region (if known), impedance at session start, filter
  settings
- `bidsBundleRef` ‚Äî URI of the BIDS-formatted data bundle
  (or NWB equivalent)
- `notes` ‚Äî free-text clinician notes

Recording sessions without a valid `consentRef` are refused
at boundary intake. Channel impedance values outside the
device's specification flag the channel `provisional` until
re-validated.

## ¬ß4 Stimulation session record

Each stimulation session emits a record:

- `stimulationId` ‚Äî URN
- `deviceRef` ‚Äî URN
- `participantPseudonym` ‚Äî pseudonym
- `consentRef` ‚Äî URN
- `protocolRef` ‚Äî URN
- `sessionStart`, `sessionEnd` ‚Äî RFC 3339
- `stimulationModality` ‚Äî `electrical`, `magnetic`,
  `optical`, `ultrasonic`, `chemical-pump`
- `targetRegion` ‚Äî anatomical region per the device's
  declared targeting capability
- `programParameters` ‚Äî modality-specific parameter set
  (amplitude, frequency, pulse-width, phase, duty-cycle)
- `programVersion` ‚Äî version of the stimulation program
- `safetyEnvelopeRef` ‚Äî URN of the active safety envelope
  (PHASE 3 ¬ß3 enforcement)
- `signatures[]` ‚Äî clinician + device signatures

A stimulation session that exceeds its `safetyEnvelopeRef`
limits is automatically interrupted and emits a follow-up
`stimulation-interrupted` audit-chain entry.

## ¬ß5 Decoded-intent and effector-output records

For BCI applications, the decoded intent and effector output
are recorded together with the contributing recording:

- `decodeId` ‚Äî URN
- `recordingRef` ‚Äî URN of the contributing recording
- `decoderModelRef` ‚Äî URN of the decoder model version
- `intentClass` ‚Äî declared intent semantics (movement
  direction, cursor position, communication symbol, robot
  end-effector command, etc.)
- `confidence` ‚Äî declared confidence band
- `effectorOutputRef` ‚Äî URN of the resulting effector
  command (cursor move, prosthetic actuation, communication
  output)
- `safetyVerdict` ‚Äî `permitted`, `deferred`, `refused` ‚Äî
  applied by the safety envelope before effector dispatch

Decoded-intent records do not by themselves indicate device
malfunction or safety issues; they feed into PHASE 4 ¬ß5
clinical-review workflows.

## ¬ß6 Calibration record

Per device, calibration records:

- `calibrationId` ‚Äî URN
- `deviceRef` ‚Äî URN
- `calibrationDate` ‚Äî RFC 3339
- `clinicianRef` ‚Äî URN
- `methodology` ‚Äî declared methodology (vendor procedure,
  research protocol, or hybrid)
- `outcomeMetrics` ‚Äî modality-specific (channel impedance
  distribution, decoder calibration set accuracy, stimulus
  threshold per region)
- `reCalibrationDueAt` ‚Äî RFC 3339

Devices past `reCalibrationDueAt` are refused for
research-use sessions; clinical-care sessions may continue
under documented exception with elevated monitoring.

## ¬ß7 Adverse-event record

Adverse events are first-class records:

- `aeId` ‚Äî URN
- `participantPseudonym`
- `deviceRef`
- `relatedSessionRefs[]` ‚Äî recording or stimulation session URNs
- `onsetAt` ‚Äî RFC 3339
- `severity` ‚Äî `mild`, `moderate`, `severe`, `life-threatening`
- `narrativeRef` ‚Äî URI of the AE narrative (clinical note,
  imaging report, etc.)
- `causalityAssessment` ‚Äî `unrelated`, `unlikely`, `possible`,
  `probable`, `definite` (per the deployment's declared
  framework)
- `reportingObligationsRef` ‚Äî URN of the relevant regulatory
  reporting obligation
- `reportedAt[]` ‚Äî list of authorities and the times of
  reporting

AE records trigger PHASE 4 ¬ß6 regulatory-reporting workflow
and may pause stimulation for the affected participant.

## ¬ß8 Consent record

Consent records bind a participant to authorised operations:

- `consentId` ‚Äî URN
- `participantPseudonym` ‚Äî pseudonym
- `protocolRef` ‚Äî URN of the protocol the consent governs
- `signedAt` ‚Äî RFC 3339
- `validUntil` ‚Äî RFC 3339
- `scope[]` ‚Äî closed enum: `recording`, `stimulation`,
  `data-sharing`, `device-update`, `re-identification`
- `withdrawalAllowed` ‚Äî boolean (typically true)
- `participantSignatureRef` ‚Äî URN of the participant's
  signature artifact

A consent record marked withdrawn revokes outstanding
permissions; in-flight sessions complete only the safe
exit path.

## ¬ß9 Clinical-context note record

Clinical context (e.g., medication changes, sleep state,
acute illness) at the time of a session:

- `contextNoteId` ‚Äî URN
- `participantPseudonym`
- `relatedSessionRefs[]`
- `at` ‚Äî RFC 3339
- `noteText` ‚Äî free-text clinical note
- `clinicianRef` ‚Äî URN

Clinical-context notes feed downstream analysis where
medication or state may confound results.

## Annex B ‚Äî Conformance disclosure

Sections ¬ß2, ¬ß3, ¬ß4, ¬ß6, ¬ß7, ¬ß8 are mandatory; ¬ß5 is
mandatory for any BCI deployment with a connected effector;
¬ß9 is mandatory for any clinical-care deployment. A
deployment `partial` or `excluded` on ¬ß7 (Adverse events)
or ¬ß8 (Consent) is non-conformant overall.

## Annex C ‚Äî Versioning and deprecation

Versioning follows SemVer 2.0.0. Decoder models and
stimulation programs carry their own versions; mutations
are versioned and audit-chained.

## Annex E ‚Äî Vendor extensions

Device vendors may extend records with `x-vendor-*` fields.
Extensions MUST NOT contradict canonical fields and MUST
NOT be required for core conformance.

## Annex F ‚Äî Time discipline cross-reference

Time fields use the discipline of PHASE 3 ¬ß6 (UTC, leap-
second handling). Session-recording timestamps are
disciplined to GPS-based time references at the recording
site; drift outside declared bound tags the session
`provisional` until backfilled from device-internal clock
records.

## Annex G ‚Äî Conformance level

Implementations declare conformance level (Surface / Verified
/ Anchored). Anchored requires a continuous evidence package
plus an annual ISO 13485 / IEC 62304 audit covering the
integration contracts in PHASE 4.

## Annex H ‚Äî BIDS / NWB bundle cross-walk

Recording sessions' `bidsBundleRef` may point to a BIDS
dataset, an NWB file, or a hybrid container. The deployment
declares its preferred container format in the capability
document; cross-format conversion (e.g., BIDS ‚Üí NWB) is a
boundary-internal concern and does not affect downstream
consumers.

## Annex I ‚Äî Safety-envelope structure (cross-reference)

The `safetyEnvelopeRef` on stimulation records points to a
versioned envelope record (PHASE 3 ¬ß3, Annex K). The
envelope encodes:

- Per-electrode cumulative charge limit
- Per-pulse charge limit
- Peak amplitude limit
- Session duration limit
- Detection-and-interrupt rules (e.g., on after-discharge)
- Responsible-clinician and risk-officer counter-signatures

Recording-side after-discharge monitors operate within the
same envelope's rules; a stimulation session whose envelope
disables the monitor is refused at boundary intake.

## Annex J ‚Äî Decoded-intent semantic register

Decoded-intent `intentClass` values are registered per
deployment in a semantic register published in the capability
document. Common entries:

- `cursor-2d-direction-rad`
- `cursor-3d-direction-rad`
- `prosthetic-grasp-force-band`
- `prosthetic-finger-flex-set`
- `communication-symbol-utf8`
- `communication-character-set-icd10`
- `assistive-device-command`

Intent classes outside the registered list are refused at
boundary intake to avoid uncontrolled effector dispatch.
