Phase 2: API 인터페이스와 검색 시스템

WIA-HERITAGE-008: Folklore Archive Standard

05.1 RESTful API 설계 철학

Phase 2는 Phase 1의 데이터를 실제로 활용할 수 있게 하는 프로그래밍 인터페이스를 제공합니다. WIA-HERITAGE-008 API는 REpresentational State Transfer (REST) 아키텍처 스타일을 따르며, 직관적이고 예측 가능한 엔드포인트를 제공합니다.

REST 원칙 적용

  • 리소스 중심: URL은 동사가 아닌 명사로 표현 (/getStory가 아닌 /stories/:id)
  • HTTP 메서드 활용: GET(조회), POST(생성), PUT(전체 수정), PATCH(부분 수정), DELETE(삭제)
  • 무상태성: 각 요청은 독립적이며 서버는 세션 상태를 저장하지 않음
  • 계층화: 클라이언트는 중간 프록시/캐시 존재를 알 필요 없음
  • 캐시 가능: 응답에 캐시 정책 명시하여 성능 최적화
엔드포인트메서드기능권한

05.2 GraphQL 대안 제공

REST의 over-fetching(불필요한 데이터 전송)과 under-fetching(여러 요청 필요) 문제를 해결하기 위해 GraphQL도 지원합니다. 클라이언트가 정확히 필요한 데이터만 요청할 수 있습니다.

GraphQL 쿼리 예시

query GetStoryWithNarrator($id: ID!) {
  story(id: $id) {
    title
    dateRecorded
    language
    narrator {
      name
      birthYear
      community {
        name
        location
      }
    }
    audio {
      url
      duration
    }
    motifs {
      code
      title
    }
  }
}

REST로 같은 데이터를 얻으려면 최소 3번의 요청(/story, /narrator, /community)이 필요하지만, GraphQL은 1번으로 충분합니다.

특징RESTGraphQL

05.3 고급 검색 알고리즘

민속 자료 검색은 일반 문서 검색과 다른 특수성이 있습니다. 같은 이야기의 변형 찾기, 유사 모티프 발견, 지리적 분포 분석 등이 중요합니다.

6가지 검색 모드

🔍 전문 검색 (Full-Text Search)

Elasticsearch를 사용하여 68개 언어의 형태소 분석을 지원합니다. 한국어는 은전한닢(mecab-ko), 영어는 Snowball, 중국어는 IK Analyzer를 사용합니다.

GET /api/v1/search?q=설문대할망&lang=kor&analyzer=mecab
🎯 퍼지 검색 (Fuzzy Search)

철자 오류, 방언 변형을 허용합니다. Levenshtein distance 2 이내의 단어를 매칭합니다.

GET /api/v1/search?q=선녀와나무군&fuzzy=2
// "선녀와 나무꾼" 찾음 (오타 허용)
🧠 시맨틱 검색 (Semantic Search)

GPT-4 임베딩으로 의미적 유사도를 계산합니다. "날개옷"을 검색하면 "깃옷", "천의무봉"도 찾습니다.

POST /api/v1/search/semantic
{
  "query": "하늘에서 내려온 여인",
  "threshold": 0.75
}

모티프 검색

ATU 색인으로 서사 구조 패턴을 매칭합니다. "ATU 400" (잃어버린 아내 찾기)를 검색하면 세계 각국의 비슷한 이야기를 찾습니다.

ATU 510A콩쥐 팥쥐叶限灰かぶりCinderella

05.4 인증과 권한 관리

문화적으로 민감한 자료를 보호하기 위해 OAuth 2.0과 JWT(JSON Web Token) 기반 인증 시스템을 구현합니다.

인증 흐름

  1. 클라이언트가 /auth/login에 사용자명/비밀번호 전송
  2. 서버가 검증 후 JWT 토큰 발급 (유효기간 24시간)
  3. 클라이언트가 이후 모든 요청에 Authorization 헤더로 토큰 전송
  4. 서버가 토큰 검증 후 사용자 권한 확인
  5. 리프레시 토큰으로 만료 전 갱신
POST /auth/login
{
  "username": "kim.younghee@jeju.kr",
  "password": "********"
}

Response:
{
  "accessToken": "eyJhbGc...",
  "refreshToken": "dGhpc2lz...",
  "expiresIn": 86400,
  "user": {
    "id": "narrator:kr:kim-younghee",
    "roles": ["narrator", "community:jeju"]
  }
}

05.5 페이지네이션과 성능 최적화

대규모 아카이브에서는 수백만 개의 이야기가 있을 수 있습니다. 효율적인 페이지네이션이 필수입니다.

3가지 페이지네이션 방식

1. Offset 기반: 전통적 방식, 구현 간단하지만 대규모 데이터에서 느림

GET /api/v1/stories?offset=100&limit=20
// 101~120번째 이야기

2. Cursor 기반: 대규모 데이터에 최적, 일관성 보장

GET /api/v1/stories?cursor=eyJpZCI6MTIzfQ&limit=20
// cursor 다음 20개

3. 무한 스크롤: 모바일 앱에 적합

GET /api/v1/stories?after=story:kr:jeju:123&limit=20

캐싱 전략

  • CDN 캐싱: 오디오/비디오 파일을 Cloudflare/Fastly에 캐시 (TTL 7일)
  • Redis 캐싱: 인기 이야기 메타데이터 (TTL 1시간)
  • 브라우저 캐싱: Cache-Control 헤더로 클라이언트 캐시 (max-age=3600)
  • ETags: 변경 감지로 불필요한 전송 방지

05.6 지리적 검색과 매핑

이야기의 지리적 분포를 검색하고 시각화합니다. PostGIS를 사용하여 공간 쿼리를 처리합니다.

지리 검색 API

// 제주도 중심 반경 50km 내 이야기
GET /api/v1/geo/search?lat=33.4996&lon=126.5312&radius=50

// 특정 행정구역 내 이야기
GET /api/v1/geo/search?region=kr-jeju

// 두 지역 간 이야기 전파 경로
GET /api/v1/geo/diffusion?from=kr-jeju&to=kr-ulleung

GeoJSON 응답

표준 GeoJSON 형식으로 응답하여 Leaflet, Mapbox, Google Maps에 바로 표시 가능합니다.
