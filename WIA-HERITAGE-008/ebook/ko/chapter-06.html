06.1 오디오 스트리밍 프로토콜

Phase 3는 Phase 2의 정적 데이터 API를 실시간 스트리밍과 협업 기능으로 확장합니다. 고음질 오디오 스트리밍은 구전 전통 보존의 핵심입니다. 단순한 재생이 아니라 다층적 상호작용을 가능하게 해야 합니다.

적응형 비트레이트 스트리밍 (Adaptive Bitrate Streaming)

네트워크 대역폭이 변하는 환경에서 끊김 없는 재생을 보장합니다.

네트워크 속도권장 코덱비트레이트지연
저속 (2G)AAC-LC64 kbps모노
모바일 (4G)HE-AAC128-192 kbps스테레오
WiFi (10-50Mbps)AAC or Opus256-320 kbps스테레오
광대역 (100Mbps+)FLAC 또는 WAV1411 kbps무손실
🎵 HLS 스트리밍 매니페스트 예시
#EXTM3U
#EXT-X-VERSION:3
#EXT-X-TARGETDURATION:10
#EXTINF:10.0,
segment-00001.m4s
#EXTINF:10.0,
segment-00002.m4s
#EXT-X-ENDLIST

HLS(HTTP Live Streaming)를 통해 모든 플랫폼에서 호환 가능합니다. 자동 재연결, 대역폭 감지, 자막 동기화를 포함합니다.

06.2 협업 주석 시스템

여러 전문가가 동시에 같은 이야기에 주석을 추가할 수 있는 시스템입니다. 문화 해석자, 언어학자, 음악가, 역사가가 각자의 관점으로 기여합니다.

계층화된 주석 구조

계층담당자내용권한
Level 0구연자원본 이야기 및 메타데이터소유자 통제
Level 1음성학자음운 전사 (IPA)권역별 검토
Level 2언어학자문법 분석, 방언 특성학술 검증
Level 3민속학자모티프 분류, 문화적 맥락학술 검증
Level 4공동체공동체 해석, 현대적 의미공동체 평의

실시간 협업 편집 - CRDT 알고리즘

여러 사용자가 동시에 편집할 때 충돌을 자동으로 해결합니다. CRDT(Conflict-free Replicated Data Type)는 최종적 일관성을 보장합니다.

06.3 크라우드소싱 전사 시스템

수백만 시간의 오디오 컨텐츠를 전문가 혼자서 전사할 수 없습니다. 커뮤니티 참여를 통해 대규모 전사를 가능하게 합니다.

전사 워크플로우

  1. 태스크 분할: 30분 녹음을 5분 단위로 분할
  2. 참여자 매칭: 언어 능력과 방언 지식 기준으로 할당
  3. 이중 검증: 2명 이상이 독립적으로 전사 후 비교
  4. 전문가 검토: 품질 점수 80% 이상만 수락
  5. 보상 및 인증: 기여도에 따라 디지털 배지 및 인증서 발급

품질 관리 메트릭

메트릭측정 방법합격선보상
WER (Word Error Rate)정답과의 편집거리<10%5 크레딧
일관성 점수다른 전사자와의 동의도>85%보너스 2 크레딧
완성도공백/생략 정도<5%기본
시간 준수기한 내 제출100%상훈
👥 참여자 레벨 시스템
  • 견습생 (0-20시간): 교육 중, 검토 필수
  • 숙련공 (20-100시간): 80% 이상 WER 달성, 자동 승인 가능
  • 장인 (100+ 시간): 전문 검수 역할, 보너스 2배
  • 마스터 (500+ 시간): 품질 관리 위원회 참여, 특별 존경

06.4 P2P 데이터 분산 배포

중앙 서버 의존도를 줄이기 위해 IPFS와 BitTorrent 기반 P2P 분산을 지원합니다. 이는 검열 저항성과 지역 자율성을 보장합니다.

IPFS 아카이브 구조

/ipfs/QmFolklorearchive
  /metadata.json
  /stories
    /korean
      /jeju
        /story-001
          /audio.flac
          /transcript.txt
          /annotations.json
    /african
      /griots
        /story-520/

콘텐츠 주소 지정(content-addressed) 방식으로 변조 불가능하며, 네트워크의 여러 노드에서 자동 복제됩니다.

하이브리드 배포 전략

계층기술용도지역
1차CDN (Cloudflare)전 세계 고속 접근글로벌
2차IPFS검열 저항, 오프라인 접근지역 노드
3차BitTorrent대용량 벌크 전송지역 커뮤니티
로컬USB 오프라인 패키지인터넷 없는 지역마을 도서관

06.5 오프라인 동기화

인터넷이 없는 지역에서도 아카이브를 사용할 수 있도록 설계합니다. 온라인이 되면 자동으로 변경사항을 동기화합니다.

오프라인 우선(Offline-First) 아키텍처

📱 로컬 우선 데이터 흐름
  1. 사용자가 로컬 데이터베이스(IndexedDB)에서 작업
  2. 변경사항은 로컬 큐에 저장
  3. 인터넷 연결 시 서버와 동기화
  4. 충돌 발생 시 사용자 선택 또는 자동 병합
  5. 완료 후 로컬 변경 이력 유지

마을 도서관의 기준봉은 매달 한 번 도시 센터에 가서 클라우드와 동기화하면 됩니다. 그 사이에 수백 시간의 오디오를 편집하고 주석을 달 수 있습니다.

06.6 CRDT 충돌 해결

여러 사용자가 같은 텍스트를 동시에 편집할 때 데이터 일관성을 보장하는 핵심 기술입니다.

Operational Transformation vs CRDT

특성Operational TransformCRDT
구현 복잡도매우 높음중간
중앙 서버 필수네, 필수아니오, 선택
네트워크 지연 허용제한적무제한
메모리 오버헤드낮음높음
P2P 호환성어려움네이티브

JSON CRDT 예시

// Yjs 라이브러리 사용
const yDoc = new Y.Doc();
const yText = yDoc.getText("shared-transcript");

yText.observe(event => {
  console.log("변경:", event.changes);
});

// 사용자 A
yText.insert(0, "설문대할망은");

// 사용자 B (동시)
yText.insert(0, "[여신]");

// 결과: "[여신]설문대할망은" (자동 병합)

06.8 장 요약

✅ 핵심 내용
  1. 적응형 스트리밍: HLS를 통해 2Mbps부터 100Mbps까지 모든 네트워크 환경 지원
  2. 계층화 주석: 구연자-음성학자-언어학자-민속학자-공동체의 5단계 협업
  3. 크라우드소싱: 전사 품질을 WER 10% 이하로 유지하면서 규모 확대
  4. P2P 분산: CDN+IPFS+BitTorrent 하이브리드로 검열 저항성 및 지역 자율성
  5. 오프라인 우선: IndexedDB + CRDT로 인터넷 없이 작업 가능
  6. CRDT 기술: 동시 편집 시 자동 병합으로 충돌 해결
弘益人間

기술은 모두를 포함하기 위해 존재합니다. 연결과 오프라인 모두를 지원합니다

06.9 복습 문제

  1. 적응형 비트레이트 스트리밍이 HLS를 사용하는 이유를 설명하고, 실제 모바일 환경에서의 구현 방식을 논하세요.
  2. 협업 주석의 5단계 계층 구조와 각 계층의 권한 관리 방식을 설명하세요.
  3. 크라우드소싱 전사에서 WER 메트릭이 중요한 이유와 이중 검증 프로세스의 효과를 분석하세요.
  4. P2P 분산 배포에서 CDN, IPFS, BitTorrent를 조합하는 이유를 네트워크 복원력 측면에서 논하세요.
  5. 오프라인 우선 아키텍처에서 CRDT가 필수인 이유와 다른 대안들과의 비교를 하세요.
  6. 품질 보증 시스템에서 자동 검사와 수동 검토의 역할 분담이 어떻게 비용 효율성을 달성하는지 설명하세요.
