  7. 실제 애플리케이션에서의 코드 예제 활용

4.1 WIA 데이터 형식의 철학

弘益人間 (홍익인간) - 널리 인간을 이롭게 하라

표준화된 데이터 형식은 모든 제조업체와 개발자가 같은 언어로 소통할 수 있게 합니다. 이는 기술 장벽을 낮추고, 혁신을 촉진하며, 궁극적으로 사용자에게 이익을 가져다줍니다.

4.1.1 설계 원칙

WIA Phase 1 데이터 형식은 다음 원칙을 따릅니다:

  • 인간 가독성 (Human Readability): JSON 형식 채택으로 개발자가 쉽게 이해하고 디버깅
  • 기계 효율성 (Machine Efficiency): 바이너리 변환 가능, 최소 오버헤드
  • 확장성 (Extensibility): 미래 기능 추가 시 하위 호환성 유지
  • 언어 중립성 (Language Neutrality): 모든 점자 코드 지원 (영어, 한국어, 일본어, 아랍어 등)
  • 검증 가능성 (Verifiability): 스키마 기반 자동 검증
  • 상호운용성 (Interoperability): 기존 유니코드 점자 표준과 완벽 호환

4.1.2 왜 JSON인가?

형식 장점 단점 WIA 선택
바이너리 컴팩트, 빠름 사람이 읽을 수 없음, 디버깅 어려움 ✗ (레이어 2에서 사용)
XML 구조화, 검증 가능 장황함, 파싱 오버헤드 ✗
JSON 가독성, 범용성, 가벼움 바이너리보다 큼 ✓ (최적의 균형)
Protocol Buffers 매우 컴팩트, 성능 스키마 의존, 가독성 낮음 ✗

결론: JSON은 개발 용이성과 성능의 최적 균형을 제공합니다. 실시간 성능이 필요한 경우, Phase 2에서 바이너리 직렬화로 변환 가능합니다.

4.2 점자 셀 데이터 모델

4.2.1 기본 BrailleCell 구조

WIA 표준의 가장 기본적인 단위는 BrailleCell입니다:

{
  "type": "BrailleCell",
  "dots": 6,           // 6 또는 8
  "pattern": "⠓",     // 유니코드 점자 문자
  "dotPattern": [1, 2, 5],  // 활성화된 점의 번호 (1-6 또는 1-8)
  "decimal": 27,       // 10진수 표현 (0-63 또는 0-255)
  "hex": "0x1B",       // 16진수 표현
  "binary": "00011011" // 2진수 표현 (점 8-7-6-5-4-3-2-1)
}

4.2.2 점 번호 체계

점자 점은 국제 표준에 따라 번호가 매겨집니다:

6점 점자 8점 점자 
1 ⚫ ⚫ 4
2 ⚫ ⚫ 5
3 ⚫ ⚫ 6
                    
1 ⚫ ⚫ 4
2 ⚫ ⚫ 5
3 ⚫ ⚫ 6
7 ⚫ ⚫ 8

4.2.3 비트 매핑

각 점은 비트로 표현됩니다 (점 1 = LSB):

문자 유니코드 활성 점 비트 패턴 10진수 16진수
A ⠁ (U+2801) 1 00000001 1 0x01
B ⠃ (U+2803) 1, 2 00000011 3 0x03
C ⠉ (U+2809) 1, 4 00001001 9 0x09
H ⠓ (U+2813) 1, 2, 5 00010011 19 0x13
Z ⠵ (U+2835) 1, 3, 5, 6 00110101 53 0x35

계산 공식 (6점):

decimal = (점1 ? 1 : 0) + (점2 ? 2 : 0) + (점3 ? 4 : 0) +
          (점4 ? 8 : 0) + (점5 ? 16 : 0) + (점6 ? 32 : 0)

유니코드 = U+2800 + decimal

계산 공식 (8점):

decimal = (점1~6 계산) + (점7 ? 64 : 0) + (점8 ? 128 : 0)

유니코드 = U+2800 + decimal

4.6 데이터 유효성 검사 규칙

4.6.1 구조적 검증

규칙 설명 검증 방법
Type Validation type 필드가 유효한 값 JSON Schema enum 체크
Dots Consistency dots=6이면 점 7,8 사용 불가 dotPattern 배열 최대값 체크
Unicode Range pattern이 U+2800~U+28FF 범위 정규식 ^[⠀-⣿]$
Decimal Range decimal이 0~63(6점) 또는 0~255(8점) 조건부 범위 체크
Pattern Consistency pattern, decimal, dotPattern 일치 크로스 필드 검증

4.6.2 의미적 검증

// TypeScript 검증 함수
function validateBrailleCell(cell: BrailleCell): ValidationResult {
  const errors: string[] = [];

  // 1. Dots 일관성 체크
  if (cell.dots === 6 && cell.decimal > 63) {
    errors.push("6점 셀은 decimal이 0-63 범위여야 합니다");
  }

  // 2. dotPattern과 decimal 일치 체크
  const calculatedDecimal = cell.dotPattern.reduce(
    (sum, dot) => sum + Math.pow(2, dot - 1),
    0
  );
  if (calculatedDecimal !== cell.decimal) {
    errors.push(`dotPattern [${cell.dotPattern}]에서 계산된 값 ${calculatedDecimal}이(가) decimal ${cell.decimal}과(와) 불일치`);
  }

  // 3. Unicode 일치 체크
  const expectedUnicode = String.fromCharCode(0x2800 + cell.decimal);
  if (cell.pattern !== expectedUnicode) {
    errors.push(`pattern '${cell.pattern}'이(가) decimal ${cell.decimal}에 대응하는 '${expectedUnicode}'와(과) 불일치`);
  }

  // 4. Binary 일치 체크
  if (cell.binary) {
    const binaryDecimal = parseInt(cell.binary, 2);
    if (binaryDecimal !== cell.decimal) {
      errors.push(`binary '${cell.binary}'이(가) decimal ${cell.decimal}과(와) 불일치`);
    }
  }

  return {
    valid: errors.length === 0,
    errors: errors
  };
}

4.7 실용적인 코드 예제

4.7.1 텍스트를 점자로 변환 (JavaScript)

// 1급 영어 점자 변환 (간단한 예제)
const grade1Map = {
  'a': 0x01, 'b': 0x03, 'c': 0x09, 'd': 0x19, 'e': 0x11,
  'f': 0x0B, 'g': 0x1B, 'h': 0x13, 'i': 0x0A, 'j': 0x1A,
  'k': 0x05, 'l': 0x07, 'm': 0x0D, 'n': 0x1D, 'o': 0x15,
  'p': 0x0F, 'q': 0x1F, 'r': 0x17, 's': 0x0E, 't': 0x1E,
  'u': 0x25, 'v': 0x27, 'w': 0x3A, 'x': 0x2D, 'y': 0x3D, 'z': 0x35,
  ' ': 0x00  // 빈 셀
};

function textToBraille(text) {
  const lowerText = text.toLowerCase();
  const cells = [];

  for (let char of lowerText) {
    const decimal = grade1Map[char];
    if (decimal !== undefined) {
      cells.push({
        type: "BrailleCell",
        dots: 6,
        pattern: String.fromCharCode(0x2800 + decimal),
        decimal: decimal,
        hex: "0x" + decimal.toString(16).toUpperCase().padStart(2, '0')
      });
    }
  }

  return {
    type: "BrailleLine",
    cells: cells,
    length: cells.length,
    encoding: "grade1",
    language: "en-US"
  };
}

grade1_reverse_map = {
    0x01: 'a', 0x03: 'b', 0x09: 'c', 0x19: 'd', 0x11: 'e',
    0x0B: 'f', 0x1B: 'g', 0x13: 'h', 0x0A: 'i', 0x1A: 'j',
    0x05: 'k', 0x07: 'l', 0x0D: 'm', 0x1D: 'n', 0x15: 'o',
    0x0F: 'p', 0x1F: 'q', 0x17: 'r', 0x0E: 's', 0x1E: 't',
    0x25: 'u', 0x27: 'v', 0x3A: 'w', 0x2D: 'x', 0x3D: 'y', 0x35: 'z',
    0x00: ' '
}

def braille_to_text(braille_line):
    text = ""
    for cell in braille_line['cells']:
        decimal = cell['decimal']
        char = grade1_reverse_map.get(decimal, '?')
        text += char
    return text

  console.log(`  - 언어: ${doc.metadata.language}`);

}

  const decimal = parseInt(req.params.decimal);

  if (decimal < 0 || decimal > 255) {
    return res.status(400).json({
      error: "decimal은 0-255 범위여야 합니다"
    });
  }

  const unicode = String.fromCharCode(0x2800 + decimal);
  const codePoint = "U+" + (0x2800 + decimal).toString(16).toUpperCase().padStart(4, '0');

  res.json({
    decimal: decimal,
    hex: "0x" + decimal.toString(16).toUpperCase(),
    unicode: unicode,
    codePoint: codePoint
  });
});

4.8.1 기본 편집기 클래스

class BrailleEditor {
  private document: BrailleDocument;
  private currentLine: number;

  constructor() {
    this.document = this.createEmptyDocument();
    this.currentLine = 0;
  }

  createEmptyDocument(): BrailleDocument {
    return {
      type: "BrailleDocument",
      version: "1.0",
      metadata: {
        title: "새 문서",
        author: "",
        date: new Date().toISOString().split('T')[0],
        language: "en-US",
        brailleCode: "UEB",
        dots: 6,
        encoding: "grade1"
      },
      lines: [],
      totalLines: 0,
      totalCells: 0
    };
  }

  insertText(text: string): void {
    const brailleLine = textToBraille(text);
    this.document.lines.push({
      lineNumber: this.document.lines.length + 1,
      cells: brailleLine.cells
    });
    this.document.totalLines++;
    this.document.totalCells += brailleLine.cells.length;
  }

  deleteLine(lineNumber: number): void {
    const index = this.document.lines.findIndex(
      l => l.lineNumber === lineNumber
    );
    if (index !== -1) {
      const removedCells = this.document.lines[index].cells.length;
      this.document.lines.splice(index, 1);
      this.document.totalLines--;
      this.document.totalCells -= removedCells;

      // 줄 번호 재정렬
      this.document.lines.forEach((line, i) => {
        line.lineNumber = i + 1;
      });
    }
  }

  save(filename: string): void {
    saveBrailleDocument(this.document, filename);
  }

  load(filename: string): void {
    this.document = loadBrailleDocument(filename);
  }

  export(): string {
    return JSON.stringify(this.document, null, 2);
  }
}

// 사용 예
const editor = new BrailleEditor();
editor.insertText("Hello World");
editor.insertText("WIA Standards");
editor.save("my-document.braille.json");

4.9 챕터 요약: 5가지 핵심 포인트

  1. JSON 기반 데이터 형식은 개발 용이성과 성능의 균형

    WIA Phase 1은 JSON을 채택하여 인간 가독성과 기계 효율성을 동시에 달성했습니다. BrailleCell, BrailleLine, BrailleDocument의 계층 구조로 복잡한 점자 데이터를 명확하게 표현합니다.

  2. 유니코드 점자 블록(U+2800~U+28FF)과 완벽 호환

    256개 코드 포인트로 6점(64개) 및 8점(192개) 점자를 모두 표현합니다. 각 점자 셀은 decimal, hex, binary, dotPattern, unicode 등 다양한 형식으로 상호 변환 가능합니다.

  3. JSON Schema로 자동 검증 가능

    타입, 범위, 일관성을 자동으로 검증하여 개발 오류를 사전 방지합니다. Ajv 등의 도구로 런타임 검증을 쉽게 구현할 수 있습니다.

  4. 다국어 지원은 변환 테이블로 확장

    영어 UEB, 한국어, 일본어, 중국어, 아랍어 등 각 언어의 점자 규칙을 JSON 변환 테이블로 정의합니다. language 및 encoding 필드로 적절한 테이블을 선택합니다.

  5. 실용적인 구현 예제로 즉시 활용 가능

    JavaScript, TypeScript, Python 코드 예제로 텍스트-점자 변환, 파일 I/O, REST API, 편집기 구현을 쉽게 시작할 수 있습니다. 弘익人間의 철학처럼, 표준은 모두에게 접근 가능해야 합니다.

4.10 복습 문제

  1. BrailleCell JSON 객체의 필수 필드 3가지는 무엇이며, 각각의 역할은 무엇입니까?
    답안 보기

    필수 필드는 (1) type: "BrailleCell"로 객체 유형 식별, (2) dots: 6 또는 8로 점자 셀 크기 정의, (3) pattern: 유니코드 점자 문자(U+2800~U+28FF)로 실제 점자 패턴 표현. 이 3개 필드만으로 점자 셀의 핵심 정보를 전달할 수 있으며, dotPattern, decimal, hex, binary는 선택적 보조 정보입니다.

  2. 유니코드 점자 문자 "⠓" (H)의 decimal, hex, dotPattern, binary 값을 계산하세요.
    답안 보기

    "⠓"는 점 1-2-5가 활성화된 패턴입니다.
    dotPattern: [1, 2, 5]
    decimal: 2^0 + 2^1 + 2^4 = 1 + 2 + 16 = 19
    hex: 0x13
    binary: 00010011 (오른쪽부터 점1, 점2, ..., 점8)
    유니코드: U+2800 + 19 = U+2813 = "⠓"

  3. 6점 점자와 8점 점자의 decimal 범위 차이는 무엇이며, 이것이 실무에서 왜 중요합니까?
    답안 보기

    6점 점자는 decimal 범위가 0-63 (2^6=64개 조합), 8점 점자는 0-255 (2^8=256개 조합)입니다. 실무에서 중요한 이유: (1) 검증 - dots=6일 때 decimal > 63이면 오류, (2) 호환성 - 6점 전용 디스플레이에 8점 데이터 전송 시 하위 6점만 사용, (3) 용도 구분 - 일반 텍스트는 6점, 프로그래밍/특수문자는 8점 필요. 올바른 범위 체크로 런타임 오류를 방지합니다.

  4. WIA가 XML이나 Protocol Buffers 대신 JSON을 선택한 주요 이유 3가지는 무엇입니까?
    답안 보기

    (1) 인간 가독성: 개발자가 파일을 직접 읽고 편집할 수 있어 디버깅과 학습이 쉬움. XML은 너무 장황하고 Protocol Buffers는 바이너리라 읽기 불가. (2) 범용성: 모든 프로그래밍 언어에 JSON 파서가 내장되어 있어 즉시 사용 가능. (3) 성능과 크기의 균형: XML보다 가볍고 빠르며, Protocol Buffers만큼 복잡하지 않음. Phase 2에서 바이너리 최적화 가능하므로 Phase 1은 개발 편의성 우선.

  5. 다국어 점자를 지원하기 위해 BrailleLine 객체에 포함되어야 할 필드는 무엇이며, 한국어 "안녕"을 점역할 때 필요한 정보는?
    답안 보기

    필수 필드: (1) language: "ko-KR" (언어 코드), (2) encoding: "korean-standard" (점자 체계). 한국어 "안녕" 점역 시 필요한 정보: (1) 한글 분해 규칙 (초성 ㅇ, 중성 ㅏ, 종성 ㄴ 등), (2) 각 자모의 점자 패턴 (ㅇ=⠈, ㅏ=⠣, ㄴ=⠴ 등), (3) 초성/중성/종성 위치에 따른 변환 규칙, (4) 한국어 점자 변환 테이블. 이를 통해 "안녕" = [⠈, ⠣, ⠴, ⠒, ⠬, ⠁] 등으로 변환됩니다.

  6. 점자 데이터의 구조적 검증과 의미적 검증의 차이를 설명하고, 각각의 예를 드세요.
    답안 보기

    구조적 검증: JSON 스키마로 타입, 필드 존재, 값 범위를 체크. 예: type이 "BrailleCell"인지, dots가 6 또는 8인지, pattern이 유니코드 점자 문자(⠀-⣿)인지 확인. 의미적 검증: 필드 간 일관성과 논리적 정합성을 체크. 예: dotPattern [1,2,5]에서 계산한 decimal (19)가 실제 decimal 필드 값과 일치하는지, pattern "⠓"가 U+2800+19=U+2813와 일치하는지 검증. 구조적 검증은 자동화 가능하지만, 의미적 검증은 도메인 지식과 커스텀 로직이 필요합니다.
