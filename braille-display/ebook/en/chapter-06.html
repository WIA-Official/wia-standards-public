⠿ Chapter 6: Phase 3 - Communication Protocol

Part III: Technical Specification

Introduction

Phase 3 defines the low-level communication protocols that enable the API (Phase 2) to interact with physical Braille display hardware. This chapter covers the technical details of data transmission over USB HID, Bluetooth LE, and serial connections, as well as integration with existing systems like BRLTTY.

Understanding these protocols is essential for device manufacturers, driver developers, and those implementing the WIA standard at the system level. Application developers using the high-level API typically don't need this knowledge, but it provides valuable insight into how the system works under the hood.

Protocol Goal: Achieve sub-100ms latency for display updates with robust error recovery, while maintaining compatibility with diverse hardware and operating systems.

6.1 Protocol Architecture Overview

6.1.1 Layered Protocol Stack

The WIA Braille Display protocol uses a layered architecture:

Layer Purpose Examples
Application Layer High-level API calls display.write(), display.clear()
Command Layer Protocol messages WRITE_CELLS, READ_KEYS, GET_INFO
Transport Layer Data framing and sequencing Packet assembly, checksums, ACK/NAK
Physical Layer Hardware communication USB HID, Bluetooth LE GATT, Serial UART

6.1.2 Transport Options

WIA supports three primary transport mechanisms:

  1. USB HID - Universal Serial Bus Human Interface Device (preferred for desktop/laptop)
  2. Bluetooth LE - Bluetooth Low Energy with GATT services (preferred for mobile/wireless)
  3. Serial - RS-232/UART for legacy devices and embedded systems

All transports use the same command layer protocol, ensuring consistent behavior regardless of connection type.

6.1.3 Design Requirements

Requirement Target Critical?
Update Latency < 100ms Yes
Key Event Latency < 50ms Yes
Packet Loss Recovery < 200ms Yes
Connection Establishment < 2s No
Battery Life (Bluetooth) > 8 hours Yes
Concurrent Connections 1 device Yes

6.2 USB HID Protocol

6.2.1 HID Overview

USB HID (Human Interface Device) is the preferred protocol for wired connections. Benefits include:

  • No driver installation required (OS provides generic HID driver)
  • Low latency (~1-10ms)
  • High reliability (USB error correction)
  • Power delivery (bus-powered devices possible)
  • Plug-and-play device discovery

6.2.2 HID Device Descriptor

WIA Braille displays identify themselves with specific USB descriptors:

USB Device Descriptor:
  bcdUSB:             2.00
  bDeviceClass:       0x00  (Interface-specific)
  bDeviceSubClass:    0x00
  bDeviceProtocol:    0x00
  idVendor:           0x####  (Manufacturer assigned)
  idProduct:          0x####  (Product specific)
  bcdDevice:          1.00
  iManufacturer:      1  "Manufacturer Name"
  iProduct:           2  "WIA Braille Display Model"
  iSerialNumber:      3  "SERIAL123456"
  bNumConfigurations: 1

USB HID Report Descriptor:
  Usage Page:         0x41  (Braille Display)
  Usage:              0x01  (Braille Display)
  Collection:         0x01  (Application)
    Report ID:        0x01
    Input (Data):     64 bytes  (keys, routing)
    Output (Data):    64 bytes  (cell patterns)
    Feature (Data):   64 bytes  (device info, config)

6.2.3 HID Report Structure

WIA uses three types of HID reports:

Input Report (Device → Host)

Byte 0:     Report ID (0x01 = keys, 0x02 = routing)
Byte 1:     Sequence number (0-255, wraps)
Byte 2-3:   Timestamp (milliseconds, low/high byte)
Byte 4-63:  Payload (key states, routing button states)

Example - Key Press Report:
[01] [42] [A3 12] [15 00 00 00 ...]
 |    |    |  |    └─ Key bitmap (bit 0 = key 0, etc.)
 |    |    └──┴─ Timestamp: 0x12A3 = 4771ms
 |    └─ Sequence: 66
 └─ Report ID: 0x01 (key event)

Output Report (Host → Device)

Byte 0:     Report ID (0x10 = write cells, 0x11 = set cursor)
Byte 1:     Sequence number
Byte 2:     Command flags
Byte 3:     Payload length
Byte 4-63:  Payload (cell patterns, configuration)

Example - Write Cells:
[10] [55] [00] [28] [00 00 20 10 08 04 02 01 ...]
 |    |    |    |    └─ Cell patterns (8-bit dot patterns)
 |    |    |    └─ Length: 40 cells
 |    |    └─ Flags: 0x00 (normal write)
 |    └─ Sequence: 85
 └─ Report ID: 0x10 (write cells)

Feature Report (Bidirectional)

Byte 0:     Report ID (0x20 = device info, 0x21 = config)
Byte 1:     Request/Response code
Byte 2-63:  Data

Example - Device Info Request:
[20] [01] [00 00 00 ...]
 |    └─ Request: 0x01 (get device info)
 └─ Report ID: 0x20 (device info)

Example - Device Info Response:
[20] [81] [28 00 08 01 ...]
 |    |    |  |  |  └─ Feature flags
 |    |    |  |  └─ Dot format: 0x08 (8-dot)
 |    |    |  └─ Reserved
 |    |    └─ Cell count: 40
 |    └─ Response: 0x81 (device info response)
 └─ Report ID: 0x20

6.2.4 Command Set

Report ID Command Direction Description
0x10 WRITE_CELLS Host → Device Update Braille cell patterns
0x11 SET_CURSOR Host → Device Set cursor position and state
0x12 CLEAR_DISPLAY Host → Device Clear all cells
0x13 SET_CONFIG Host → Device Configure device settings
0x01 KEY_EVENT Device → Host Navigation/function key press
0x02 ROUTING_EVENT Device → Host Routing button press
0x03 PERKIN_EVENT Device → Host Perkin key (Braille input)
0x20 DEVICE_INFO Bidirectional Request/response device capabilities
0x21 FIRMWARE_UPDATE Host → Device Firmware update packets

6.2.5 Error Handling

USB HID provides built-in error detection (CRC), but the protocol adds additional checks:

// Sequence number validation
if (report.sequence != (lastSequence + 1) % 256) {
    // Packet lost or out of order
    requestRetransmit(lastSequence + 1);
}

// Timestamp validation (detect stale packets)
const age = currentTime - report.timestamp;
if (age > 1000) {  // More than 1 second old
    logWarning('Stale packet received');
    discard();
}

// Length validation
if (report.payloadLength > MAX_PAYLOAD) {
    sendError(ERROR_INVALID_LENGTH);
    discard();
}

6.2.6 Performance Optimization

USB HID polling interval affects latency:

Polling Interval Latency CPU Usage Recommendation
1ms ~1-2ms High High-performance systems
8ms ~8-16ms Medium Recommended default
16ms ~16-32ms Low Battery-powered hosts

WIA recommends 8ms polling (125Hz) as optimal balance between latency and efficiency.

6.3 Bluetooth LE Protocol

6.3.1 Bluetooth LE Overview

Bluetooth Low Energy provides wireless connectivity optimized for low power consumption:

  • Range: 10-50 meters (depending on environment)
  • Power: < 15mA average current (8+ hours on 400mAh battery)
  • Latency: 7.5ms - 4000ms (connection interval dependent)
  • Throughput: ~5-10 KB/s typical

6.3.2 GATT Service Definition

WIA Braille displays expose a custom GATT (Generic Attribute Profile) service:

Service: WIA Braille Display Service
UUID: 0000FEB0-0000-1000-8000-00805F9B34FB

Characteristics:

1. Device Information (Read)
   UUID: 0000FEB1-0000-1000-8000-00805F9B34FB
   Properties: Read
   Value: Device capabilities (cell count, features, etc.)

2. Cell Data (Write, Notify)
   UUID: 0000FEB2-0000-1000-8000-00805F9B34FB
   Properties: Write, Notify
   Value: Braille cell patterns (up to 244 bytes per packet)

3. Key Events (Notify)
   UUID: 0000FEB3-0000-1000-8000-00805F9B34FB
   Properties: Notify
   Value: Key press events

4. Routing Events (Notify)
   UUID: 0000FEB4-0000-1000-8000-00805F9B34FB
   Properties: Notify
   Value: Routing button events

5. Control (Write, Read, Notify)
   UUID: 0000FEB5-0000-1000-8000-00805F9B34FB
   Properties: Write, Read, Notify
   Value: Commands and status

6. Configuration (Write, Read)
   UUID: 0000FEB6-0000-1000-8000-00805F9B34FB
   Properties: Write, Read
   Value: Device configuration

Parameter Value Rationale

Slave Latency 0 Minimize key event delay
Supervision Timeout 2000ms Detect disconnection quickly

6.4 Serial Communication

6.4.1 Serial Overview

Serial communication (RS-232/UART) supports legacy devices and embedded systems:

Parameter Value
Baud Rate 115200 bps (default), 9600-921600 supported
Data Bits 8
Parity None
Stop Bits 1
Flow Control RTS/CTS (hardware) or XON/XOFF (software)

6.4.2 Packet Framing

Serial communication requires explicit framing (unlike USB/Bluetooth):

Packet Structure:
┌────────┬──────────┬─────────┬─────────┬──────────┬─────────┐
│ SYNC   │ LENGTH   │ SEQ     │ COMMAND │ PAYLOAD  │ CHECKSUM│
│ 2 bytes│ 1 byte   │ 1 byte  │ 1 byte  │ 0-255 B  │ 2 bytes │
└────────┴──────────┴─────────┴─────────┴──────────┴─────────┘

SYNC:      0xAA 0x55 (packet start marker)
LENGTH:    Payload length (0-255)
SEQ:       Sequence number (0-255, wraps)
COMMAND:   Command code
PAYLOAD:   Command-specific data
CHECKSUM:  CRC-16/CCITT

Example - Write Cells Command:
AA 55 2A 00 10 00 28 [40 cell patterns...] [CRC-16]
└─┬─┘ |  |  |  └─┬─┘
SYNC  |  |  |    └─ Start cell: 0, Count: 40
      |  |  └─ Command: 0x10 (WRITE_CELLS)
      |  └─ Sequence: 0
      └─ Length: 42 bytes (2 header + 40 cells)

6.4.3 Error Detection

// CRC-16/CCITT implementation
function calculateCRC16(data) {
    let crc = 0xFFFF;

    for (let byte of data) {
        crc ^= byte << 8;
        for (let bit = 0; bit < 8; bit++) {
            if (crc & 0x8000) {
                crc = (crc << 1) ^ 0x1021;
            } else {
                crc = crc << 1;
            }
        }
    }

    return crc & 0xFFFF;
}

// Verify received packet
function verifyPacket(packet) {
    const receivedCRC = (packet[packet.length - 2] << 8) |
                        packet[packet.length - 1];
    const calculatedCRC = calculateCRC16(packet.slice(0, -2));

    return receivedCRC === calculatedCRC;
}

6.4.4 Flow Control

Hardware flow control prevents buffer overruns:

// RTS/CTS flow control
void configureSerial() {
    serial.setBaudRate(115200);
    serial.setFlowControl(FLOW_CONTROL_RTS_CTS);

    // Device asserts CTS when ready to receive
    // Device deasserts CTS when buffer nearly full
    // Host monitors CTS and stops sending when deasserted
}

// Software flow control (if hardware unavailable)
void handleXonXoff() {
    if (receivedByte == XON) {   // 0x11
        resumeTransmission();
    }
    if (receivedByte == XOFF) {  // 0x13
        pauseTransmission();
    }
}

static int brl_construct(BrailleDisplay *brl, char **parameters) {
    // Initialize WIA SDK
    WiaBrailleDisplay *display = wia_braille_display_new();

    // Discover and connect

static void brl_destruct(BrailleDisplay *brl) {

    wia_braille_display_disconnect(display);
    wia_braille_display_free(display);

static int brl_writeWindow(BrailleDisplay *brl, const wchar_t *text) {
    WiaBrailleDisplay *display = brl->data;

    // Convert text to Braille
    uint8_t cells[brl->textColumns];
    for (int i = 0; i < brl->textColumns; i++) {
        cells[i] = convertToBraille(text[i]);
    }

    // Write to display
    return wia_braille_display_write_cells(display, cells, brl->textColumns);
}

static int brl_readCommand(BrailleDisplay *brl, KeyTableCommandContext context) {
    WiaBrailleDisplay *display = brl->data;

    // Poll for key events
    WiaKeyEvent *event = wia_braille_display_read_key(display, 100);  // 100ms timeout
    if (!event) {
        return EOF;  // No key pressed
    }

    // Map WIA key to BRLTTY command
    return mapKeyToCommand(event);
}

6.7 Security Considerations

6.7.1 Threat Model

Threat Risk Mitigation
Eavesdropping Medium Bluetooth encryption, USB physical security
Unauthorized Control Low Pairing required, OS permissions
Malicious Firmware High Signed firmware updates, secure boot
Denial of Service Low Rate limiting, watchdog timers
Privacy Leakage Medium No logging of displayed content

6.7.2 Firmware Update Security

// Signed firmware update process
1. Host verifies firmware signature
   - RSA-2048 or ECDSA-256
   - Signed by manufacturer

2. Device enters bootloader mode
   - Only accepts signed updates
   - Validates signature before flashing

3. Cryptographic hash verification
   - SHA-256 of firmware image
   - Prevents corrupted updates

4. Rollback protection
   - Version number must increase
   - Prevent downgrade attacks

Example:
{
    "firmwareVersion": "2.1.0",
    "signatureAlgorithm": "RSA-2048",
    "signature": "base64_encoded_signature",
    "sha256": "abcdef123456...",
    "minBootloaderVersion": "1.0.0"
}

6.7.3 Privacy Protection

// Privacy-preserving guidelines

1. No content logging
   - Devices MUST NOT log displayed content
   - Temporary buffer only (cleared on disconnect)

2. Anonymous telemetry only
   - Usage statistics: session duration, key presses (count only)
   - NO content of what was displayed
   - User opt-in required

3. Secure disposal
   - Clear all buffers on power-off
   - Factory reset clears Bluetooth bonding keys

6.8.2 Latency Breakdown (USB HID)

Total Latency: API call → Cell update

1. API Processing:        ~1-5ms
   - Input validation
   - Format conversion

2. USB Transfer:          ~1-8ms
   - Polling interval dependent
   - 8ms interval = worst case 8ms

3. Device Processing:     ~5-20ms
   - Firmware parsing
   - Piezo actuator control

4. Physical Update:       ~20-50ms
   - Mechanical pin movement
   - Varies by hardware

Total: 27-83ms (well under 100ms target)

6.8.3 Latency Optimization Techniques

// Batch updates
function optimizedUpdate() {
    // Bad: Multiple individual updates
    for (let i = 0; i < 40; i++) {
        await display.writeCell(i, pattern[i]);  // 40 round trips!
    }

    // Good: Single batched update
    await display.writeCells(pattern);  // 1 round trip
}

// Differential updates
function updateOnlyChanged(oldCells, newCells) {
    const changes = [];
    for (let i = 0; i < newCells.length; i++) {
        if (oldCells[i] !== newCells[i]) {
            changes.push({ index: i, value: newCells[i] });
        }
    }

    // Only send changed cells
    await display.writeCellsPartial(changes);
}

// Predictive pre-loading
function preloadNextPage(content, currentPage) {
    // While user reads current page, prepare next
    const nextPage = content.getPage(currentPage + 1);
    // Buffer but don't display yet
    display.bufferCells(nextPage);

    // On "next" key, instant display (already buffered)
}

Chapter Summary

Phase 3 of the WIA Braille Display Standard defines the communication protocols that bridge the API layer with physical hardware. Key points:

  • USB HID - Preferred wired protocol with low latency, no driver installation, and robust error handling
  • Bluetooth LE - Wireless protocol optimized for low power with GATT services and secure pairing
  • Serial - Legacy support for older devices with explicit packet framing and flow control
  • BRLTTY Integration - Seamless compatibility with Linux systems through driver implementation
  • Message Formats - Comprehensive command set for device control and status reporting
  • Security - Encryption, signed firmware updates, and privacy protection measures
  • Latency - Sub-100ms update targets with optimization techniques for real-time responsiveness

Understanding these protocols enables manufacturers to build compliant devices and developers to troubleshoot low-level issues.
