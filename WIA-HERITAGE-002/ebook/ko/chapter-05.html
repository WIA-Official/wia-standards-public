Phase 2: API 인터페이스

WIA-HERITAGE-002: Archaeological Data Standards

05.1 API 인터페이스의 역할과 중요성

WIA-HERITAGE-002의 Phase 2는 고고학 데이터를 프로그래밍 방식으로 접근하고 조작할 수 있는 표준화된 API(Application Programming Interface)를 정의합니다. Phase 1이 데이터 형식의 표준화에 집중했다면, Phase 2는 이러한 데이터를 어떻게 전송하고, 검색하고, 업데이트할 것인지에 대한 명확한 규칙을 제공합니다.

API는 현대 디지털 생태계의 핵심 구성 요소입니다. 박물관, 대학 연구소, 정부 기관, 민간 기업들이 각자의 고고학 데이터베이스를 운영하면서도 서로 원활하게 데이터를 교환할 수 있으려면 공통의 언어가 필요합니다. WIA-HERITAGE-002 API는 바로 이 공통 언어를 제공합니다.

🔌 API의 핵심 기능
  • 데이터 조회: 유물, 발굴지, 3D 모델 등의 메타데이터 검색
  • 데이터 업로드: 새로운 발굴 데이터와 파일 제출
  • 인증 및 권한 관리: 연구자별 접근 권한 제어
  • 버전 관리: 데이터 변경 이력 추적
  • 실시간 알림: 관심 분야의 새로운 발견 통지
  • 일괄 처리: 대량 데이터 다운로드 및 분석

API 아키텍처의 설계 원칙

WIA-HERITAGE-002 API는 RESTful 아키텍처 스타일을 채택했습니다. REST(Representational State Transfer)는 웹의 확장성과 단순성을 최대한 활용하는 설계 패턴으로, 다음과 같은 원칙을 따릅니다:

  • 무상태성(Stateless): 각 요청은 독립적이며 서버는 세션 정보를 저장하지 않음
  • 캐시 가능성(Cacheable): 응답에 캐시 정책이 명시되어 네트워크 효율 향상
  • 계층화된 시스템: 로드 밸런서, 프록시, 게이트웨이 등 중간 계층 허용
  • 균일한 인터페이스: 일관된 URL 구조와 HTTP 메서드 사용

05.2 RESTful 엔드포인트 설계

WIA-HERITAGE-002는 7개의 핵심 리소스 타입을 정의하며, 각각에 대해 표준화된 엔드포인트를 제공합니다. 모든 엔드포인트는 https://api.heritage.wia.org/v1/을 베이스 URL로 사용합니다.

리소스엔드포인트설명예시
유물/artifacts고고학 유물 데이터청동기 검, 토기 파편
발굴지/sites발굴 현장 정보경주 월성, 아테네 아고라
3D 모델/modelsglTF 형식 3D 스캔조각상, 건축물
이미지/images사진 및 멀티스펙트럴발굴 과정, 유물 상세
분석/analyses과학적 분석 결과방사성탄소 연대, DNA
컬렉션/collections큐레이션된 데이터셋신라 금관 특별전
연구자/researchers사용자 프로필소속, 전문 분야

HTTP 메서드와 CRUD 작업

각 리소스는 표준 HTTP 메서드를 통해 조작됩니다. 이는 데이터베이스의 CRUD(Create, Read, Update, Delete) 작업과 직접 대응됩니다:

GET    /artifacts        # 유물 목록 조회 (필터링, 페이징 지원)
GET    /artifacts/{id}   # 특정 유물 상세 조회
POST   /artifacts        # 새 유물 등록
PUT    /artifacts/{id}   # 유물 정보 전체 수정
PATCH  /artifacts/{id}   # 유물 정보 부분 수정
DELETE /artifacts/{id}   # 유물 삭제 (소프트 삭제, 관리자 전용)

쿼리 파라미터와 고급 검색

WIA-HERITAGE-002 API는 강력한 검색 기능을 제공합니다. URL 쿼리 파라미터를 통해 복잡한 필터링, 정렬, 페이징이 가능합니다:

# 경주에서 발견된 신라시대 금속 유물, 연대순 정렬, 20개씩
GET /artifacts?site=Gyeongju&period=Silla&material=metal&sort=date&limit=20

# 3D 모델이 있는 석조 건축물, 해상도 5mm 이상
GET /sites?has3d=true&type=architecture&material=stone&resolution_min=5mm

# 2024년에 업로드된 멀티스펙트럴 이미지
GET /images?type=multispectral&uploaded_after=2024-01-01

지원되는 주요 쿼리 파라미터는 다음과 같습니다:

  • q: 전체 텍스트 검색 (메타데이터 전체 대상)
  • geo: 지리적 바운딩 박스 (위도/경도)
  • date_range: 발굴 또는 제작 연대 범위
  • bbox: 공간적 범위 (GIS 좌표)
  • license: 데이터 라이선스 필터 (CC0, CC-BY 등)
  • format: 응답 형식 (json, xml, csv)

05.3 인증 및 권한 관리

고고학 데이터는 민감한 정보를 포함할 수 있습니다. 발굴 전 유적지의 정확한 위치는 도굴꾼으로부터 보호되어야 하며, 원주민 공동체의 성스러운 유물은 문화적 프로토콜에 따라 접근이 제한될 수 있습니다. WIA-HERITAGE-002는 이러한 요구사항을 충족하는 세분화된 권한 관리 시스템을 제공합니다.

OAuth 2.0 기반 인증

표준은 OAuth 2.0 프로토콜을 채택하여 안전하고 유연한 인증을 구현합니다. 연구자는 자신의 기관 계정(대학, 박물관 등)으로 로그인하여 API 액세스 토큰을 받습니다:

# 1단계: 인증 요청
GET https://api.heritage.wia.org/v1/oauth/authorize
  ?client_id=your_app_id
  &redirect_uri=https://your-app.com/callback
  &scope=read_artifacts write_artifacts read_sites
  &response_type=code

# 2단계: 액세스 토큰 발급
POST https://api.heritage.wia.org/v1/oauth/token
Content-Type: application/json
{
  "grant_type": "authorization_code",
  "code": "AUTH_CODE_FROM_STEP1",
  "client_id": "your_app_id",
  "client_secret": "your_secret"
}

# 응답: 액세스 토큰
{
  "access_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "refresh_token_here",
  "scope": "read_artifacts write_artifacts read_sites"
}

계층적 권한 모델

WIA-HERITAGE-002는 4단계 권한 레벨을 정의합니다:

레벨권한대상 사용자제한사항
Public공개 데이터 읽기누구나민감 정보 제외
Researcher전체 읽기, 제한적 쓰기검증된 연구자자신의 데이터만 수정
Curator전체 읽기/쓰기, 검토박물관 큐레이터기관 데이터 관리

API 키 방식 (간편 인증)

간단한 읽기 전용 접근을 위해 API 키 방식도 지원됩니다. 연구자는 웹 포털에서 API 키를 생성하고 요청 헤더에 포함시킵니다:

GET /artifacts/12345
Authorization: ApiKey your-api-key-here

Python SDK 사용 예제

Python SDK는 가장 인기 있는 라이브러리로, 데이터 과학자와 고고학자들이 즐겨 사용합니다:

from wia_heritage import HeritageClient

# 클라이언트 초기화
client = HeritageClient(api_key='your-api-key')

# 유물 검색
artifacts = client.artifacts.search(
    site='Gyeongju',
    period='Silla',
    material='gold',
    limit=10

# 결과 출력
for artifact in artifacts:
    print(f"{artifact.name} - {artifact.date_range}")

    print(f"  3D 모델: {artifact.has_3d_model}")

# 새 유물 등록
new_artifact = client.artifacts.create({
    'name': '금동관',
    'site_id': 'gyeongju-147',
    'period': 'Silla',
    'material': ['gold', 'bronze'],
    'dimensions': {'height': 27.3, 'width': 14.1},
    'discovery_date': '2024-03-15',

# 3D 모델 업로드
with open('crown_scan.glb', 'rb') as f:

05.5 데이터 페이징과 성능 최적화

고고학 데이터베이스는 수백만 개의 레코드를 포함할 수 있습니다. 모든 데이터를 한 번에 전송하는 것은 비효율적이므로 WIA-HERITAGE-002는 커서 기반 페이징(cursor-based pagination)을 권장합니다.

커서 페이징의 작동 원리

# 첫 페이지 요청
GET /artifacts?limit=100

# 응답
{
  "data": [ /* 100개 유물 */ ],
  "pagination": {
    "next_cursor": "eyJpZCI6MTIzNDUsImRhdGUiOiIyMDI0LTAzLTE1In0=",
    "has_more": true
  }
}

# 다음 페이지 요청
GET /artifacts?limit=100&cursor=eyJpZCI6MTIzNDUsImRhdGUiOiIyMDI0LTAzLTE1In0=

이 방식은 오프셋 기반 페이징(?page=2&limit=100)에 비해 다음과 같은 장점이 있습니다:

  • 데이터가 추가/삭제되어도 일관된 결과 보장
  • 대규모 데이터셋에서 빠른 성능 (인덱스 스캔만 필요)
  • 중복 또는 누락 없이 안정적인 순회

필드 선택과 부분 응답

네트워크 대역폭을 절약하기 위해 필요한 필드만 요청할 수 있습니다:

# 기본 정보만 필요한 경우
GET /artifacts/12345?fields=id,name,date,thumbnail

# 응답 (3D 모델 등 대용량 데이터 제외)
{

  "date": "6세기 중반",

# 변경 있으면 200 OK (새 데이터 전송)

명확한 에러 처리는 개발자 경험의 핵심입니다. WIA-HERITAGE-002는 표준 HTTP 상태 코드를 의미론적으로 사용하며, 모든 에러에 대해 상세한 JSON 응답을 제공합니다.

Rate Limiting

공정한 리소스 사용을 위해 API는 요청 빈도를 제한합니다. 각 응답 헤더에 현재 상태가 포함됩니다:

X-RateLimit-Limit: 1000        # 시간당 최대 요청 수
X-RateLimit-Remaining: 847     # 남은 요청 수
X-RateLimit-Reset: 1709654400  # 리셋 시각 (Unix timestamp)

제한을 초과하면 429 상태 코드와 함께 대기 시간이 안내됩니다:

{
  "error": {
    "code": "RATE_LIMIT_EXCEEDED",
    "message": "시간당 요청 한도를 초과했습니다",
    "retry_after": 1847  # 초 단위
  }
}

05.7 웹훅과 이벤트 스트림

실시간 데이터 동기화를 위해 WIA-HERITAGE-002는 웹훅(Webhook)과 Server-Sent Events(SSE)를 지원합니다. 이를 통해 새로운 발굴 데이터가 등록되면 즉시 알림을 받을 수 있습니다.

웹훅 등록

관심 있는 이벤트에 대해 콜백 URL을 등록하면, 해당 이벤트 발생 시 POST 요청을 받습니다:

# 웹훅 생성
POST /webhooks
{
  "url": "https://your-app.com/webhook/heritage",
  "events": ["artifact.created", "site.updated", "model.uploaded"],
  "filters": {
    "site": "Gyeongju",
    "period": "Silla"
  },
  "secret": "your-webhook-secret"
}

# 이벤트 발생 시 수신되는 페이로드
POST https://your-app.com/webhook/heritage
X-WIA-Signature: sha256=abc123...
{
  "event": "artifact.created",
  "timestamp": "2024-03-15T14:32:11Z",
  "data": {
    "id": "artifact-98765",
    "name": "금제 귀걸이",
    "site": "Gyeongju",
    "period": "Silla",
    "discoverer": "김고고 교수"
  }
}

서명 검증

웹훅 요청이 실제로 WIA 서버에서 왔는지 확인하기 위해 HMAC 서명을 검증합니다:

import hmac
import hashlib

def verify_webhook(payload, signature, secret):
    computed = hmac.new(
        secret.encode(),
        payload.encode(),
        hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(f"sha256={computed}", signature)

Server-Sent Events (SSE)

웹 브라우저에서 실시간 알림을 받으려면 SSE를 사용합니다:

// JavaScript
const eventSource = new EventSource(
  'https://api.heritage.wia.org/v1/events?filter=site:Gyeongju'
);

eventSource.addEventListener('artifact.created', (event) => {
  const artifact = JSON.parse(event.data);
  showNotification(`새 발견: ${artifact.name}`);
});

  1. RESTful API 아키텍처를 통해 7개 핵심 리소스(유물, 발굴지, 3D 모델 등)에 표준화된 방식으로 접근합니다.
  2. OAuth 2.0 인증과 4단계 권한 모델로 민감한 고고학 데이터를 안전하게 보호합니다.
  3. 6개 언어의 공식 SDK가 제공되어 Python, JavaScript 등으로 쉽게 API를 활용할 수 있습니다.
  4. 커서 기반 페이징, 필드 선택, HTTP 캐싱을 통해 대규모 데이터셋을 효율적으로 처리합니다.
  5. 표준 HTTP 상태 코드와 상세한 에러 메시지로 명확한 디버깅이 가능합니다.
  6. 웹훅과 SSE를 통해 실시간 데이터 동기화와 알림을 구현할 수 있습니다.

05.9 복습 문제

  1. RESTful API의 4가지 핵심 설계 원칙을 설명하고, 각각이 고고학 데이터 시스템에서 중요한 이유를 논하세요.
  2. OAuth 2.0 인증 흐름의 주요 단계를 나열하고, API 키 방식과 비교하여 장단점을 설명하세요.
  3. 커서 기반 페이징이 오프셋 기반 페이징보다 대규모 데이터셋에서 유리한 이유는 무엇인가요?
  4. 웹훅 페이로드의 서명 검증이 왜 중요한지 설명하고, 검증하지 않을 경우 발생할 수 있는 보안 문제를 제시하세요.
  5. Rate Limiting이 필요한 이유와 이를 우회하려는 클라이언트에 대한 적절한 대응 방법을 논하세요.
  6. Python SDK를 사용하여 "Gyeongju" 발굴지의 "Silla" 시대 금제 유물을 검색하는 코드를 작성하세요. (의사코드 가능)
