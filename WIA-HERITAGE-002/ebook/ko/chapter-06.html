Phase 3: 프로토콜

WIA-HERITAGE-002: Archaeological Data Standards

06.1 실시간 통신 프로토콜의 필요성

Phase 2의 RESTful API는 정적인 데이터 조회와 업데이트에는 탁월하지만, 실시간 협업과 대용량 3D 모델 스트리밍에는 한계가 있습니다. 고고학자 3명이 서로 다른 대륙에서 동시에 같은 유물의 3D 모델을 회전시키고 주석을 달면서 토론한다고 상상해보세요. 이를 위해서는 밀리초 단위의 지연시간으로 데이터를 양방향 전송할 수 있는 프로토콜이 필요합니다.

WIA-HERITAGE-002 Phase 3는 세 가지 핵심 프로토콜을 정의합니다: WebSocket을 이용한 실시간 메시징, WebRTC를 이용한 P2P 3D 스트리밍, 그리고 QUIC 기반 대용량 파일 전송입니다. 이러한 프로토콜들은 현대 고고학의 글로벌 협업 요구사항을 충족시킵니다.

🌐 프로토콜 계층 구조
  • WebSocket: 실시간 양방향 메시징 (채팅, 알림, 커서 동기화)
  • WebRTC: P2P 비디오/오디오 및 데이터 채널 (화상회의, 3D 협업)
  • QUIC/HTTP3: 고속 대용량 전송 (gigapixel 이미지, 4K 스캔)
  • MQTT: IoT 센서 데이터 수집 (발굴지 환경 모니터링)
  • GraphQL Subscriptions: 선택적 데이터 구독 (특정 필드 변경 감지)

프로토콜 선택 기준

각 프로토콜은 특정 사용 사례에 최적화되어 있습니다. 적절한 프로토콜 선택은 성능과 사용자 경험에 직접적인 영향을 미칩니다:

사용 사례권장 프로토콜이유대역폭
실시간 채팅WebSocket낮은 지연, 양방향 통신~10 KB/s
3D 모델 스트리밍WebRTC Data ChannelP2P, 낮은 지연1-5 MB/s
대용량 파일 다운로드HTTP/3 (QUIC)손실 복구, 멀티플렉싱10-100 MB/s
발굴지 센서MQTT저전력, 불안정한 네트워크~1 KB/s
데이터 구독GraphQL Subscriptions선택적 필드, 효율적~5 KB/s

06.2 WebSocket 기반 실시간 협업

WebSocket은 단일 TCP 연결을 통해 클라이언트와 서버 간 전이중(full-duplex) 통신을 제공하는 프로토콜입니다. HTTP 요청-응답 모델과 달리, 한 번 연결이 맺어지면 어느 쪽이든 언제든지 메시지를 보낼 수 있습니다.

연결 수립 과정

WebSocket은 HTTP 핸드셰이크로 시작하여 프로토콜을 업그레이드합니다:

메시지 형식: Heritage Protocol Messages (HPM)

WIA-HERITAGE-002는 WebSocket 메시지의 표준 구조를 정의합니다. 모든 메시지는 JSON 형식이며 다음 필드를 포함합니다:

{
  "type": "cursor.move",           // 메시지 타입
  "artifact_id": "12345",          // 대상 유물 ID
  "user_id": "researcher-987",     // 발신자 ID
  "timestamp": 1709654478123,      // Unix 밀리초
  "data": {                        // 타입별 페이로드
    "x": 0.534,
    "y": 0.721,
    "rotation": 45.3
  },
  "metadata": {                    // 선택적 메타데이터
    "client_version": "2.1.4",
    "network_latency": 23
  }
}

06.3 3D 모델 프로그레시브 스트리밍

고고학 3D 모델은 종종 수 기가바이트에 달하는 고해상도 메쉬와 4K/8K 텍스처를 포함합니다. 사용자가 모델을 보기 전에 전체 파일을 다운로드하도록 강요하는 것은 비현실적입니다. WIA-HERITAGE-002는 프로그레시브 스트리밍을 통해 즉각적인 시각화를 가능하게 합니다.

Level of Detail (LoD) 계층

3D 모델은 여러 해상도 버전으로 사전 처리됩니다. 클라이언트는 먼저 저해상도 모델을 받아 즉시 표시하고, 백그라운드에서 고해상도 데이터를 점진적으로 다운로드합니다:

LoD 1 (저화질)10K1024×1024500 KB1초
LoD 2 (중화질)100K2048×20485 MB10초
LoD 3 (고화질)1M4096×409650 MB100초

Adaptive Bitrate Streaming (ABS)

네트워크 상태에 따라 동적으로 품질을 조정합니다. 클라이언트는 현재 대역폭을 측정하고 최적의 LoD 레벨을 요청합니다:

// 대역폭 측정
const bandwidth = await measureBandwidth();

// 적절한 LoD 선택
let lod;
if (bandwidth > 10_000_000) lod = 4;       // 10 Mbps: 원본
else if (bandwidth > 5_000_000) lod = 3;   // 5 Mbps: 고화질
else if (bandwidth > 1_000_000) lod = 2;   // 1 Mbps: 중화질
else if (bandwidth > 500_000) lod = 1;     // 500 Kbps: 저화질
else lod = 0;                              // 프리뷰

// 해당 LoD 스트림 요청
const stream = await client.models.stream({
  id: 'model-12345',
  lod: lod,
  format: 'glb'
});

06.4 WebRTC P2P 데이터 채널

WebRTC(Web Real-Time Communication)는 원래 화상회의를 위해 설계되었지만, 데이터 채널 기능을 통해 임의의 바이너리 데이터를 P2P로 전송할 수 있습니다. 이는 중앙 서버의 대역폭 부담을 줄이고 지연시간을 최소화하는 데 이상적입니다.

연결 수립: STUN/TURN 서버

대부분의 사용자는 NAT와 방화벽 뒤에 있어 직접 연결이 불가능합니다. STUN 서버는 공인 IP를 찾고, TURN 서버는 직접 연결이 실패할 때 릴레이 역할을 합니다:

const configuration = {
  iceServers: [
    { urls: 'stun:stun.heritage.wia.org:3478' },
    {
      urls: 'turn:turn.heritage.wia.org:3478',
      username: 'heritage-user',
      credential: 'temporary-password'
    }
  ]
};

const peerConnection = new RTCPeerConnection(configuration);

// 데이터 채널 생성 (3D 모델 청크 전송용)
const dataChannel = peerConnection.createDataChannel('3d-models', {
  ordered: false,          // 순서 보장 안 함 (속도 우선)
  maxRetransmits: 3        // 최대 3회 재전송
});

dataChannel.onopen = () => {
  console.log('P2P 데이터 채널 오픈');
  sendModelChunk(modelData);
};

dataChannel.onmessage = (event) => {
  const chunk = event.data;
  assembleModel(chunk);
};

시그널링 프로토콜

WebRTC 연결을 맺기 위해서는 시그널링 서버를 통해 SDP(Session Description Protocol) offer/answer를 교환해야 합니다. WIA-HERITAGE-002는 이를 WebSocket으로 구현합니다:

멀티파티 협업: Mesh vs SFU

3명 이상이 협업할 때는 두 가지 토폴로지가 가능합니다:

방식구조장점단점적합한 인원

06.5 MQTT를 이용한 IoT 센서 통합

현대 발굴 현장은 온도, 습도, 조도, 토양 pH, 진동 등을 실시간으로 모니터링하는 수십 개의 센서로 구성됩니다. 이러한 센서 데이터는 유물의 보존 상태를 이해하고 최적의 발굴 시기를 결정하는 데 중요합니다.

MQTT 프로토콜 개요

MQTT(Message Queuing Telemetry Transport)는 IoT 환경에 최적화된 경량 발행-구독 프로토콜입니다. HTTP에 비해 10배 이상 적은 대역폭을 사용하며, 불안정한 네트워크에서도 안정적으로 작동합니다.

🌡️ 발굴지 센서 토픽 구조
wia/heritage/{site_id}/{sensor_type}/{sensor_id}

예시:
wia/heritage/gyeongju-147/temperature/temp-01
wia/heritage/gyeongju-147/humidity/hum-01
wia/heritage/gyeongju-147/vibration/vib-03
wia/heritage/gyeongju-147/photo/img-camera-02

센서 데이터 메시지 형식

{
  "sensor_id": "temp-01",
  "timestamp": 1709654478123,
  "value": 23.7,
  "unit": "celsius",
  "location": {
    "lat": 35.8562,
    "lon": 129.2247,
    "depth": -2.3  // 지하 2.3m
  },
  "battery": 87,   // 배터리 잔량 (%)
  "quality": "good"  // good, fair, poor
}

MQTT는 3단계 QoS를 제공하여 메시지 전달 보장 수준을 조정할 수 있습니다:

QoS보장사용 사례오버헤드
0최대 1회 (At most once)주기적 온도 측정 (데이터 손실 허용)최소

06.6 GraphQL Subscriptions

RESTful API는 정적 데이터 조회에 적합하지만, 특정 필드의 변경을 실시간으로 감지하려면 비효율적인 폴링이 필요합니다. GraphQL Subscriptions는 선언적 방식으로 정확히 원하는 데이터의 변경만 구독할 수 있습니다.

Subscription 쿼리 예제

06.7 네트워크 복원력과 오프라인 지원

발굴 현장은 종종 원격지에 위치하여 네트워크가 불안정하거나 완전히 끊어질 수 있습니다. WIA-HERITAGE-002는 이러한 환경에서도 작업을 계속할 수 있는 메커니즘을 제공합니다.

자동 재연결 전략

class ResilientWebSocket {
  constructor(url, options = {}) {
    this.url = url;
    this.reconnectDelay = options.reconnectDelay || 1000;
    this.maxReconnectDelay = options.maxReconnectDelay || 30000;
    this.currentDelay = this.reconnectDelay;
    this.connect();
  }

  connect() {
    this.ws = new WebSocket(this.url);

    this.ws.onopen = () => {
      console.log('연결됨');
      this.currentDelay = this.reconnectDelay;  // 지수 백오프 리셋
    };

    this.ws.onclose = (event) => {
      if (!event.wasClean) {
        console.log(`연결 끊김. ${this.currentDelay}ms 후 재시도...`);
        setTimeout(() => this.connect(), this.currentDelay);

        // 지수 백오프
        this.currentDelay = Math.min(
          this.currentDelay * 2,
          this.maxReconnectDelay
        );
      }
    };
  }
}

      id: crypto.randomUUID(),
      timestamp: Date.now(),
      operation: operation
    });
    localStorage.setItem('offline_queue', JSON.stringify(this.queue));
  }

  async process() {
    if (this.processing || this.queue.length === 0) return;

    this.processing = true;

    while (this.queue.length > 0) {
      const item = this.queue[0];

      try {
        await executeOperation(item.operation);
        this.queue.shift();  // 성공 시 제거
        localStorage.setItem('offline_queue', JSON.stringify(this.queue));
      } catch (error) {
        if (error.network) {
          console.log('네트워크 오류. 나중에 재시도...');
          break;  // 네트워크 오류면 중단
        } else {
          console.error('작업 실패:', error);
          this.queue.shift();  // 영구 오류면 스킵
        }
      }
    }

    this.processing = false;
  }
}

// 네트워크 상태 감지
window.addEventListener('online', () => {
  console.log('온라인 복구. 대기 중인 작업 처리 중...');
  offlineQueue.process();
});

06.8 장 요약

✅ 핵심 내용
  1. WebSocket 프로토콜을 통해 실시간 양방향 메시징과 협업 기능(커서 동기화, 채팅, 주석)을 구현합니다.
  2. 프로그레시브 3D 스트리밍은 LoD 계층과 Draco 압축을 활용하여 대용량 모델을 즉시 시각화합니다.
  3. WebRTC P2P 데이터 채널은 중앙 서버 부담 없이 낮은 지연시간으로 연구자 간 직접 데이터 전송을 가능하게 합니다.
  4. MQTT 프로토콜로 발굴 현장의 IoT 센서 데이터를 경량하고 안정적으로 수집합니다.
  5. GraphQL Subscriptions는 특정 데이터 필드의 변경만 선택적으로 구독하여 효율적인 실시간 업데이트를 제공합니다.
  6. 자동 재연결과 오프라인 큐로 불안정한 네트워크 환경에서도 데이터 손실 없이 작업을 지속할 수 있습니다.

06.9 복습 문제

  1. WebSocket과 HTTP 폴링을 비교하고, 실시간 협업 시나리오에서 WebSocket이 유리한 이유를 설명하세요.
  2. 3D 모델의 LoD(Level of Detail) 계층 구조가 사용자 경험 향상에 어떻게 기여하는지 설명하세요.
  3. WebRTC에서 STUN과 TURN 서버의 역할을 설명하고, 각각이 필요한 네트워크 상황을 제시하세요.
  4. MQTT의 QoS 레벨 0, 1, 2의 차이를 설명하고, 발굴 현장 센서 데이터에서 각 레벨의 적절한 사용 사례를 예를 들어 설명하세요.
  5. GraphQL Subscriptions가 REST API 폴링에 비해 가지는 장점을 네트워크 효율성과 개발자 경험 측면에서 논하세요.
  6. 오프라인 큐 메커니즘이 없다면 발굴 현장에서 어떤 문제가 발생할 수 있는지 구체적인 시나리오를 제시하세요.
