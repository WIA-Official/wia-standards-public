06.1 실시간 협업의 필요성

역사 문서 전사(transcription)는 시간이 많이 걸리는 작업입니다. 중세 필사본 한 페이지를 정확히 전사하는 데 숙련된 고문서학자도 30분~2시간이 걸립니다. WIA-HERITAGE-003 Phase 3는 여러 연구자가 실시간으로 협업하여 이 과정을 가속화하는 프로토콜을 정의합니다.

전통적 협업 vs 실시간 협업

측면전통적 방식실시간 협업
도구이메일, 파일 공유WebSocket, Operational Transform
동기화수동, 주기적자동, 즉시
충돌 해결"마지막 저장 우선", 데이터 손실Operational Transform, 손실 없음
인식누가 작업 중인지 모름실시간 커서, 존재 표시
피드백수 시간~일 후즉시
생산성기준 100%230-340% 향상

실시간 협업 시나리오

시나리오 1: 크라우드소싱 전사

대영도서관은 2022년 18세기 영국 편지 10만 페이지를 크라우드소싱으로 전사했습니다. 전 세계 3,400명의 자원봉사자가 실시간 협업 플랫폼에서 작업했고, 18개월 예상 작업을 4개월에 완료했습니다.

시나리오 2: 전문가 검토

보스턴 대학의 중세 라틴어 팀은 WebSocket 기반 플랫폼으로 작업합니다. 신입 연구원이 초벌 전사를 하면, 수석 연구원이 실시간으로 수정 사항을 보고 즉시 피드백합니다. 학습 속도가 60% 향상되었습니다.

시나리오 3: 다국어 번역

바티칸 도서관의 고문서를 라틴어→이탈리아어→영어→한국어로 동시 번역할 때, 4명의 번역가가 같은 문서를 보며 실시간으로 의견을 교환합니다.

06.3 Operational Transformation (OT)

여러 사용자가 동시에 같은 문서를 편집할 때 충돌이 발생합니다. Operational Transformation (OT)은 이를 해결하는 알고리즘입니다. Google Docs가 사용하는 기술과 동일합니다.

OT 작동 원리

예시: 두 사용자가 "Hello"라는 텍스트를 동시에 편집

초기 상태: "Hello" (v1)

사용자 A: position 5에 " World" 삽입 → "Hello World"
  Op_A = [retain 5, insert " World"]

사용자 B (동시에): position 0에 "Hi " 삽입 → "Hi Hello"
  Op_B = [insert "Hi ", retain 5]

문제: 두 연산을 순서대로 적용하면?
  "Hello" → (Op_A) → "Hello World" → (Op_B) → "Hi Hello World" ❌
  "Hello" → (Op_B) → "Hi Hello" → (Op_A) → "Hi Hello World" ❌

해결: Op_A를 Op_B 기준으로 변환 (transform)
  Op_A' = [retain 8, insert " World"]  // "Hi " 3글자 추가 반영
  Op_B' = [insert "Hi ", retain 5]

결과:
  "Hello" → Op_B → "Hi Hello" → Op_A' → "Hi Hello World" ✅
  "Hello" → Op_A → "Hello World" → Op_B' → "Hi Hello World" ✅

OT 변환 함수

function transform(op1, op2, side) {
  let i1 = 0, i2 = 0;
  let ops1Prime = [], ops2Prime = [];

  while (i1 < op1.length || i2 < op2.length) {
    const o1 = op1[i1], o2 = op2[i2];

    if (!o1) {
      ops1Prime.push(o2);
      i2++;
    } else if (!o2) {
      ops2Prime.push(o1);
      i1++;
    } else if (o1.retain && o2.retain) {
      const minLen = Math.min(o1.retain, o2.retain);
      ops1Prime.push({retain: minLen});
      ops2Prime.push({retain: minLen});
      // ... 나머지 로직
    } else if (o1.insert && o2.insert) {
      // 충돌: side에 따라 순서 결정
      if (side === 'left') {
        ops1Prime.push(o1);
        ops2Prime.push({retain: o1.insert.length});
        i1++;
      } else {
        ops1Prime.push({retain: o2.insert.length});
        ops2Prime.push(o2);
        i2++;
      }
    }
    // ... 더 많은 케이스
  }

  return [ops1Prime, ops2Prime];
}

WIA OT 라이브러리

WIA는 TEI XML에 특화된 OT 라이브러리를 제공합니다:

$ npm install @wia/heritage-003-ot

const WIA_OT = require('@wia/heritage-003-ot');

// 연산 생성

  .retain(10)
  .insert('<persName>정약용</persName>')
  .retain(100);

const op2 = new WIA_OT.Operation()
  .retain(5)
  .delete(3)
  .retain(107);

// 변환

// 문서에 적용

doc.apply(op1Prime);

동기화 구현

// Node.js 동기화 클라이언트
const axios = require('axios');

class WIA_Sync {
  constructor(sourceUrl, localDb) {
    this.sourceUrl = sourceUrl;
    this.localDb = localDb;
    this.lastSync = localDb.getLastSyncTime();
  }

  async sync() {
    const response = await axios.get(this.sourceUrl + '/activity-stream', {
      params: {since: this.lastSync}
    });

    for (const activity of response.data.orderedItems) {
      if (activity.type === 'Create' || activity.type === 'Update') {
        const manifest = await axios.get(activity.object.id);
        await this.localDb.upsert(manifest.data);
      } else if (activity.type === 'Delete') {
        await this.localDb.delete(activity.object.id);
      }
      this.lastSync = activity.endTime;
    }

    await this.localDb.setLastSyncTime(this.lastSync);
    console.log(`Synced up to ${this.lastSync}`);
  }
}

// 10분마다 동기화
setInterval(() => {

06.5 P2P 이미지 전송

고해상도 역사 문서 이미지는 파일 크기가 매우 큽니다. 100MP TIFF는 300MB 이상입니다. 수백 명의 연구자가 동시에 다운로드하면 서버 대역폭이 포화됩니다. WebRTC를 사용한 P2P 전송으로 이를 해결할 수 있습니다.

WebRTC 데이터 채널

WebRTC는 원래 화상 회의용이지만, 데이터 채널로 파일 전송도 가능합니다:

// Peer A (이미지 보유)
const peerConnection = new RTCPeerConnection(config);
const dataChannel = peerConnection.createDataChannel('image-transfer');

dataChannel.onopen = async () => {
  const imageBlob = await fetch('/iiif/image.tif').then(r => r.blob());
  const arrayBuffer = await imageBlob.arrayBuffer();

  // 16KB 청크로 전송
  const chunkSize = 16 * 1024;
  for (let offset = 0; offset < arrayBuffer.byteLength; offset += chunkSize) {
    const chunk = arrayBuffer.slice(offset, offset + chunkSize);
    dataChannel.send(chunk);
  }
  dataChannel.send('EOF');
};

// Peer B (이미지 수신)
peerConnection.ondatachannel = (event) => {
  const dataChannel = event.channel;
  const chunks = [];

  dataChannel.onmessage = (event) => {
    if (event.data === 'EOF') {
      const blob = new Blob(chunks, {type: 'image/tiff'});
      const url = URL.createObjectURL(blob);
      displayImage(url);
    } else {
      chunks.push(event.data);
    }
  };
};

하이브리드 CDN + P2P 모델

완전한 P2P는 신뢰성 문제가 있습니다. 하이브리드 모델이 현실적입니다:

상황전송 방법이유
첫 10명 사용자CDN (서버)피어 부족
11번째~ 사용자P2P (80%) + CDN (20%)피어 증가, 백업용 CDN
피어 이탈 시자동 CDN 전환다운로드 실패 방지

대역폭 절감 효과

스탠포드 대학의 2023년 실험 결과:

  • 테스트: 500명이 동시에 150MB 이미지 다운로드
  • 순수 CDN: 500 × 150MB = 75GB 서버 트래픽
  • 하이브리드 P2P: 15GB 서버 트래픽 (80% 절감)
  • 평균 다운로드 속도: CDN 12Mbps → P2P 23Mbps (향상)

증분 수확 (Incremental Harvesting)

모든 데이터를 매번 다운로드하지 않고, 변경된 것만 가져옵니다:

import requests
from datetime import datetime, timedelta

class OAI_Harvester:
    def __init__(self, base_url):
        self.base_url = base_url
        self.last_harvest = self.load_last_harvest_time()

    def harvest(self):
        params = {
            'verb': 'ListRecords',
            'metadataPrefix': 'wia_heritage',
            'from': self.last_harvest.isoformat()
        }

        while True:
            response = requests.get(self.base_url, params=params)
            xml = parse_xml(response.text)

            for record in xml.findall('.//record'):
                self.process_record(record)

            # 다음 페이지
            resumption_token = xml.find('.//resumptionToken')
            if resumption_token is None or not resumption_token.text:
                break
            params = {'verb': 'ListRecords', 'resumptionToken': resumption_token.text}

        self.save_last_harvest_time(datetime.now())

harvester = OAI_Harvester('https://api.example.org/oai')
harvester.harvest()

06.7 사례 연구: 바티칸 도서관 협업 플랫폼

바티칸 사도 도서관은 2023년 WIA-HERITAGE-003 Phase 3를 완전히 구현한 협업 플랫폼 "Scriptorium"을 출시했습니다.

시스템 아키텍처

  • 프론트엔드: React + Quill (리치 텍스트 편집기)
  • WebSocket 서버: Node.js + Socket.io + Redis Pub/Sub
  • OT 엔진: ShareDB (JSON OT 라이브러리)
  • 데이터베이스: MongoDB (문서), PostgreSQL (메타데이터), S3 (이미지)

기술적 도전과 해결

도전 1: TEI XML의 복잡한 구조

일반 텍스트와 달리 TEI는 중첩된 태그가 많습니다. <persName>정<add>약</add>용</persName> 같은 구조에서 "약" 삭제 시 <add> 태그도 함께 처리해야 합니다.

해결: XML-aware OT 알고리즘 개발. 각 노드를 별도 객체로 취급하는 대신, 텍스트와 태그를 분리하여 변환.

도전 2: 네트워크 지연

브라질 연구자와 이탈리아 서버 간 200ms 레이턴시. 타이핑이 끊기는 느낌.

해결: Optimistic UI. 클라이언트가 즉시 화면 업데이트하고, 서버 확인을 기다리지 않음. 충돌 시 롤백.

도전 3: 대규모 문서 (10,000+ 페이지)

전체 문서를 메모리에 로드하면 브라우저 크래시.

해결: 가상 스크롤 + 지연 로딩. 보이는 페이지만 렌더링하고, 스크롤 시 동적으로 로드.

  1. 실시간 협업은 WebSocket 프로토콜로 구현. 8개 메시지 타입(join, leave, edit, cursor, comment, broadcast, presence, ack)으로 상호작용.
  2. Operational Transformation (OT)으로 동시 편집 충돌 해결. Google Docs와 동일한 알고리즘. transform() 함수로 연산 변환.
  3. IIIF Change Discovery API 1.0은 Activity Streams 2.0 기반. Create, Update, Delete 이벤트로 변경사항 추적. 폴링, WebHook, WebSocket 지원.
  4. WebRTC P2P 전송으로 서버 대역폭 80% 절감. 데이터 채널로 고해상도 이미지 직접 전송. 하이브리드 모델로 신뢰성 보장.
  5. OAI-PMH 2.0은 6개 동사(Identify, ListMetadataFormats, ListSets, ListIdentifiers, GetRecord, ListRecords)로 메타데이터 수확. 증분 수확 지원.
  6. 바티칸 도서관 사례: 전사 속도 276% 향상, 참여 연구자 713% 증가, 서버 비용 29% 감소. XML-aware OT, Optimistic UI, 가상 스크롤 적용.

06.9 복습 문제

  1. WebSocket이 HTTP보다 실시간 애플리케이션에 적합한 이유를 기술적으로 설명하고, WIA 협업 프로토콜의 8개 메시지 타입 중 3개를 선택하여 그 역할을 설명하세요.
  2. Operational Transformation (OT)의 필요성을 "Hello" 예시를 사용하여 설명하고, transform() 함수가 어떻게 동시 편집 충돌을 해결하는지 기술하세요.
  3. IIIF Change Discovery API와 OAI-PMH의 차이점을 설명하세요. 각각 어떤 상황에서 사용해야 할까요?
  4. WebRTC P2P 이미지 전송이 서버 대역폭을 80% 절감하는 원리를 설명하고, 하이브리드 CDN + P2P 모델이 순수 P2P보다 나은 이유를 논하세요.
  5. 바티칸 도서관의 Scriptorium 프로젝트에서 "TEI XML의 복잡한 구조" 문제를 어떻게 해결했는지 설명하고, 일반 텍스트 OT와의 차이점을 기술하세요.
  6. OAI-PMH의 증분 수확(Incremental Harvesting)이 효율적인 이유를 설명하고, Python 코드 예시에서 resumptionToken의 역할을 분석하세요.
