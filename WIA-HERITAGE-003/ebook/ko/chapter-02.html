02.1 OCR과 필기 인식의 한계

광학 문자 인식(OCR, Optical Character Recognition)은 역사 문서 디지털화의 핵심 기술입니다. 그러나 현대 인쇄물과 달리 역사 문서는 손글씨, 퇴색된 잉크, 불규칙한 레이아웃, 다양한 문자 체계로 인해 OCR 정확도가 크게 떨어집니다.

문서 유형OCR 정확도주요 문제처리 비용
현대 인쇄물99.9%거의 없음$0.001/페이지
20세기 타자기98.5%불균일한 잉크 농도$0.005/페이지
19세기 인쇄본92.7%고딕체, 프락투어체$0.02/페이지
18세기 필사본76.3%필기체, 약자, 리가처$0.15/페이지
중세 라틴 필사본61.2%축약어, 장식 대문자$0.45/페이지
한문 목판 인쇄83.1%한자 이체자, 목판 왜곡$0.08/페이지
아랍어 필사본68.9%모음 부호, 연결 문자$0.22/페이지

고문자 인식의 특수 문제들

팔림프세스트(Palimpsest): 양피지가 비쌌던 시대에는 기존 텍스트를 지우고 다시 쓰곤 했습니다. 아르키메데스 팔림프세스트가 유명한 사례인데, 10세기 비잔틴 기도서 밑에서 아르키메데스의 잃어버린 수학 논문이 발견되었습니다. 멀티스펙트럴 이미징으로 지워진 텍스트를 복원할 수 있지만, 두 층의 텍스트를 분리하는 AI 알고리즘 개발은 여전히 도전과제입니다.

축약어와 약자: 중세 라틴어 필사본은 공간과 시간을 절약하기 위해 수백 가지 축약 기호를 사용했습니다. 예를 들어 "q̃" = "quam", "ꝰ" = "-us" 어미. 일반 OCR은 이를 인식하지 못하며, 전문 훈련 데이터가 필요합니다.

가변 필기체: 같은 사람이 쓴 글자도 위치와 연결에 따라 형태가 바뀝니다. 아랍어, 우르두어는 단어 내 위치(초성, 중성, 종성)에 따라 문자 형태가 다르며, 영어 필기체도 "s"가 단어 중간과 끝에서 다릅니다(중간: ſ, 끝: s).

🤖 현대 AI HTR(Handwritten Text Recognition)의 돌파구
  • Transkribus: 유럽 READ 프로젝트의 AI 엔진. 15,000개 이상의 필기체 모델 제공. 사용자가 100-200페이지만 수동 전사하면 자신의 문서에 특화된 모델 훈련 가능.
  • Google Cloud Vision API: 100개 이상 언어 손글씨 인식. 다만 현대 필기체에 최적화되어 있어 역사 문서는 정확도 떨어짐.
  • Kraken: 오픈소스 OCR 엔진. PyTorch 기반으로 사용자 정의 모델 훈련 가능. 고대 그리스어, 콥트어, 아람어 모델 존재.
  • Calamari: Tesseract의 후속. LSTM 신경망으로 문맥 이해. 독일어 프락투어체 95%+ 정확도.

02.2 메타데이터 표준의 정글

역사 문서를 설명하는 메타데이터 표준은 너무 많습니다. 각 커뮤니티(도서관, 기록보관소, 박물관)가 자신만의 표준을 개발했고, 상호 변환이 어렵습니다.

Dublin Core: 가장 단순하지만 제한적

Dublin Core는 1995년 더블린, 오하이오에서 열린 워크숍에서 탄생했습니다. 15개의 핵심 요소만으로 모든 디지털 리소스를 설명하려는 야심찬 목표였습니다:

  • Title, Creator, Subject, Description, Publisher
  • Contributor, Date, Type, Format, Identifier
  • Source, Language, Relation, Coverage, Rights

단순함이 장점이지만, 역사 문서의 복잡한 맥락을 담기에는 부족합니다. "Creator"에 필사자를 쓸까요, 원저자를 쓸까요? 13세기에 씌어진 5세기 성 아우구스티누스 설교 필사본이라면?

MARC21: 강력하지만 복잡

MARC(Machine-Readable Cataloging)는 1960년대부터 도서관계의 표준입니다. 엄청나게 상세하지만(200+ 필드), 학습 곡선이 가파릅니다. 예를 들어:

245 10 $a Die Leiden des jungen Werthers / $c von Johann Wolfgang von Goethe.
260    $a Leipzig : $b Weygandsche Buchhandlung, $c 1774.
300    $a 2 v. ; $c 18 cm.
500    $a Epistolary novel.

이 코드를 이해하려면 몇 주간의 교육이 필요합니다. 새로운 기관이 채택하기 어렵습니다.

EAD: 계층 구조의 힘

Encoded Archival Description은 기록물의 계층적 구조를 표현하는 데 탁월합니다. 예를 들어:

<archdesc level="collection">
  <did>
    <unittitle>조선왕조실록</unittitle>
    <unitdate>1392-1863</unitdate>
  </did>
  <dsc>
    <c01 level="series">
      <did><unittitle>태조실록</unittitle></did>
      <c02 level="file">
        <did><unittitle>태조 원년 1월</unittitle></did>
      </c02>
    </c01>
  </dsc>
</archdesc>

그러나 145개 요소를 올바르게 사용하려면 전문 아키비스트가 필요합니다.

02.3 텍스트 인코딩: TEI의 야심과 현실

Text Encoding Initiative(TEI)는 1987년부터 인문학 텍스트의 디지털 인코딩을 표준화하려는 국제 프로젝트입니다. XML 기반으로 텍스트의 구조, 의미, 시각적 특징을 모두 포착할 수 있습니다.

TEI의 강력함

TEI로 인코딩하면 다음을 표현할 수 있습니다:

<text>
  <body>
    <div type="letter">
      <opener>
        <dateline><placeName>파리</placeName> <date when="1796-04-03">1796년 4월 3일</date></dateline>
        <salute>사랑하는 <persName ref="#josephine">조세핀</persName>에게</salute>
      </opener>
      <p>나는 매일 밤 당신을 꿈꿉니다. <del>하지만</del> <add place="above">그러나</add>
      당신은 <hi rend="underline">결코</hi> 답장하지 않는군요.</p>
      <closer>
        <signed><persName ref="#napoleon">나폴레옹</persName></signed>
      </closer>
    </div>
  </body>
</text>

이 인코딩은 다음 정보를 담고 있습니다:

  • 문서 유형(편지), 날짜(1796-04-03), 장소(파리)
  • 발신자(나폴레옹), 수신자(조세핀), 실제 인물 ID 참조
  • 작성 과정(삭제된 "하지만", 추가된 "그러나")
  • 강조("결코" 밑줄)

TEI의 복잡성 문제

TEI 가이드라인은 5,000페이지가 넘습니다. 600개 이상의 XML 요소를 정의하며, 각각 수십 개의 속성이 있습니다. 역사학자가 이를 모두 마스터하기는 불가능에 가깝습니다.

TEI 프로필요소 수사용 사례학습 시간
TEI Lite~100개간단한 텍스트2주
TEI Tite~50개신문, 잡지1주
TEI Simplest~20개산문, 시2일
TEI All600+개중세 필사본, 희곡6개월+

2023년 영국 대학 조사에 따르면, TEI를 배운 대학원생의 67%가 "너무 복잡하다"고 응답했고, 43%는 졸업 후 TEI 사용을 중단했습니다.

02.4 이미지 제공: IIIF의 등장

International Image Interoperability Framework(IIIF, "트리플-아이-에프"로 발음)는 역사 문서 이미지 제공의 게임 체인저입니다. 2011년 스탠포드, 옥스포드, 대영도서관 등이 협력하여 개발했습니다.

IIIF 이전의 문제

각 기관이 자체 이미지 뷰어를 개발했습니다:

  • Gallica(프랑스 국립도서관): 자체 Flash 뷰어
  • Internet Archive: BookReader
  • Google Books: 자체 뷰어
  • HathiTrust: PageTurner

문제는? 한 도서관의 이미지를 다른 도서관의 뷰어로 볼 수 없습니다. 주석을 추가해도 이동 불가. 비교 보기 불가능.

IIIF가 제공하는 것

IIIF는 4개의 API를 정의합니다:

📐 IIIF API
  • Image API: 이미지의 특정 영역을 원하는 크기와 형식으로 요청. 예: 2000x3000픽셀 이미지의 왼쪽 위 500x500 영역을 300x300 JPEG로 요청
  • Presentation API: 이미지들의 순서, 메타데이터, 구조 정의. JSON-LD 형식의 "매니페스트"
  • Search API: 문서 내 텍스트 검색. OCR 결과와 연동
  • Auth API: 인증 및 접근 제어. 저작권 보호 자료 처리

IIIF 매니페스트 예시

{
  "@context": "http://iiif.io/api/presentation/3/context.json",
  "id": "https://example.org/manuscript/123/manifest",
  "type": "Manifest",
  "label": { "ko": ["조선왕조실록 태조 1권"] },
  "metadata": [
    { "label": { "ko": ["저자"] }, "value": { "ko": ["춘추관"] } },
    { "label": { "ko": ["날짜"] }, "value": { "ko": ["1413년"] } }
  ],
  "items": [
    {
      "id": "https://example.org/manuscript/123/canvas/1",
      "type": "Canvas",
      "width": 5000,
      "height": 7000,
      "items": [{
        "type": "AnnotationPage",
        "items": [{
          "type": "Annotation",
          "motivation": "painting",
          "body": {
            "id": "https://example.org/iiif/image/123/page1/full/max/0/default.jpg",
            "type": "Image"
          }
        }]
      }]
    }
  ]
}

이 매니페스트를 Mirador, Universal Viewer, Diva.js 등 어떤 IIIF 뷰어로도 볼 수 있습니다. 한번 만들면 모든 곳에서 작동합니다.

02.5 저작권과 접근권의 딜레마

역사 문서 디지털화의 가장 큰 비기술적 장벽은 저작권입니다. 문서 자체는 수백 년 전 것이지만, 디지털 이미지는 누구 소유일까요?

국가별 저작권 상황

국가보호 기간공공 도메인 정책디지털 이미지 권리
미국저자 사후 70년1928년 이전 저작물은 PD충실한 재현은 새 저작권 없음
EU저자 사후 70년국가별 상이Bridgeman Art Library 판례 적용
영국저자 사후 70년1949년 이전 대부분 PD박물관이 이미지 권리 주장 가능
한국저자 사후 70년1955년 이전 대부분 PD명확한 판례 없음
중국저자 사후 50년1972년 이전 대부분 PD정부 소유 원칙

Bridgeman vs. Corel 판례

1999년 미국 연방법원은 혁명적인 판결을 내렸습니다. 공공 도메인 그림을 충실하게 재현한 사진은 새로운 저작권을 생성하지 않는다는 것입니다. 즉, 모나리자 원본이 공공 도메인이면, 그것을 스캔한 이미지도 공공 도메인입니다.

이는 박물관의 비즈니스 모델을 위협했습니다. 많은 박물관이 이미지 라이선스 판매로 수익을 올렸기 때문입니다. 결과적으로 일부 박물관은 고해상도 이미지 공개를 거부했습니다.

오픈 액세스 운동

반대로 일부 선구적인 기관들은 완전 오픈 액세스를 선택했습니다:

  • Rijksmuseum(네덜란드): 2013년 렘브란트, 베르메르 등 70만 점 고해상도 이미지 CC0(퍼블릭 도메인) 공개
  • 메트로폴리탄 미술관(미국): 2017년 37만 점 CC0 공개. 연간 다운로드 1,200만 건
  • 스미소니언 박물관(미국): 2020년 280만 점 이미지 오픈 액세스. 3D 스캔 포함
  • 국립중앙박물관(한국): 2021년 5만 점 고해상도 이미지 CC BY 공개

오픈 액세스 정책을 채택한 기관들은 수익 감소를 우려했지만, 실제로는 인지도 상승, 방문객 증가, 연구 협력 확대 등 긍정적 효과가 더 컸다고 보고합니다.

02.6 장기 보존 형식의 선택

디지털 파일은 물리적 문서보다 취약할 수 있습니다. 파일 형식이 구식이 되거나, 소프트웨어가 더 이상 지원되지 않거나, 저장 매체가 고장 나면 데이터가 영구 손실됩니다.

형식장점단점예상 수명권장 용도
TIFF 무압축보편적 지원, 무손실파일 크기 거대100년+마스터 아카이브

3-2-1 백업 규칙

디지털 보존의 황금률은 "3-2-1 규칙"입니다:

  • 3개의 복사본: 원본 + 2개 백업
  • 2가지 매체: 예) 서버 + 테이프
  • 1개는 오프사이트: 다른 도시/국가

바티칸 도서관은 더 나아가 "5-3-2" 규칙을 사용합니다: 5개 복사본, 3가지 매체(SSD, 테이프, 클라우드), 2개 대륙(유럽, 북미).

비트 부패(Bit Rot) 문제

하드 드라이브, SSD, 심지어 DVD도 시간이 지나면 자연스럽게 데이터가 손상됩니다. 이를 "비트 부패"라고 합니다. 연구에 따르면:

  • HDD: 연간 0.5-2% 비트 에러율
  • SSD: 5년 후 10% 데이터 손실 가능
  • CD-R: 5-10년 수명(저품질 제품)
  • DVD-R: 10-25년 수명
  • M-DISC: 1,000년 수명 주장(미검증)

해결책은 정기적인 "체크섬 검증"입니다. 파일의 SHA-256 해시를 계산하여 저장하고, 매년 재계산하여 비교합니다. 변경이 감지되면 백업에서 복구합니다.

02.7 문화적 민감성과 원주민 지식

역사 문서 디지털화는 단순한 기술 문제가 아닙니다. 일부 문서는 문화적으로 민감하거나 원주민 공동체의 신성한 지식을 담고 있습니다.

사례: 호주 원주민 문화

호주 원주민 문화에서는 죽은 사람의 이름, 이미지, 목소리를 언급하는 것이 금기시됩니다. 따라서 역사 사진, 녹음, 문서를 무분별하게 온라인 공개하면 원주민 공동체에 정신적 고통을 줍니다.

해결책으로 AIATSIS(호주 원주민·토레스 해협 섬주민 연구소)는 "문화적 주의" 경고 시스템을 개발했습니다:

⚠️ 문화적 주의 경고

"이 자료는 고인의 이미지, 이름 또는 목소리를 포함할 수 있습니다. 원주민 및 토레스 해협 섬주민 시청자는 주의하시기 바랍니다."

추가 옵션: 이미지 흐림, 클릭해야 표시, 커뮤니티 허가 필요

사례: 북미 원주민 신성한 지식

나바호족, 호피족 같은 북미 원주민 부족은 특정 의식, 노래, 기도가 부족 외부인에게 공개되어서는 안 된다고 믿습니다. 20세기 초 인류학자들이 기록한 이러한 자료들이 디지털 아카이브에 있습니다.

Local Contexts 프로젝트는 "전통 지식(TK) 라벨" 시스템을 개발했습니다:

  • TK 제한적 사용(TK Restricted): 커뮤니티 외부 사용 금지
  • TK 비상업적(TK Non-Commercial): 상업적 사용 금지
  • TK 계절적(TK Seasonal): 특정 계절에만 접근 가능
  • TK 남성/여성 전용: 성별 제한

유럽의 식민지 문서

유럽 국가들이 아프리카, 아시아 식민지에서 가져온 문서들도 논쟁적입니다. 예를 들어 영국 도서관은 인도, 케냐, 말레이시아의 역사 문서 수백만 점을 보유하고 있습니다. 이를 디지털화하여 공개해야 할까요, 아니면 원소유국에 반환해야 할까요?

절충안으로 "디지털 반환" 개념이 등장했습니다. 물리적 원본은 유지하되, 고해상도 디지털 복사본을 무료로 원소유국에 제공하는 것입니다.

02.8 장 요약

✅ 핵심 내용
  1. OCR 정확도는 문서 유형에 따라 61-99%로 크게 차이. 중세 필사본은 61.2%, 현대 인쇄물은 99.9%. AI HTR(Transkribus, Kraken)이 돌파구 제공.
  2. 메타데이터 표준은 Dublin Core(단순), MARC21(복잡), EAD(계층), TEI(텍스트)로 분열. 각각 장단점 있으며 상호운용성 부족.
  3. IIIF는 이미지 제공의 게임 체인저. 770+ 기관 채택, 5억+ 이미지 제공. Image API, Presentation API, Search API로 구성.
  4. 저작권 딜레마: 공공 도메인 문서도 이미지 권리 논쟁. Bridgeman 판례로 충실한 재현은 새 저작권 없음 확립. 오픈 액세스 운동 확산 중.
  5. 장기 보존: TIFF, PDF/A가 100년+ 수명. 3-2-1 백업 규칙(3복사본, 2매체, 1오프사이트) 필수. 비트 부패 대응 위해 체크섬 검증.
  6. 문화적 민감성: 원주민 지식, 식민지 문서, 고인 이미지 처리에 신중해야. TK 라벨, 문화적 주의 경고, 디지털 반환 개념 등장.

02.9 복습 문제

  1. 19세기 인쇄본과 18세기 필사본의 OCR 정확도 차이가 16.4%p인 이유를 설명하고, 필사본 인식의 3가지 주요 문제점을 나열하세요.
  2. Dublin Core, MARC21, EAD, TEI 중 두 가지를 선택하여 장단점을 비교하세요. 어떤 상황에서 각각을 사용해야 할까요?
  3. IIIF의 4가지 API(Image, Presentation, Search, Auth)를 설명하고, IIIF가 해결하는 핵심 문제를 기술하세요.
  4. Bridgeman vs. Corel 판례가 박물관과 디지털 아카이브에 미친 영향을 설명하세요. 왜 일부 박물관은 이에 반발했을까요?
  5. 장기 디지털 보존을 위한 "3-2-1 규칙"을 설명하고, 비트 부패(Bit Rot)를 방지하기 위한 체크섬 검증 방법을 기술하세요.
  6. 호주 원주민 문서의 문화적 민감성 문제를 예로 들어, 역사 문서 디지털화에서 고려해야 할 윤리적 측면을 논하세요. TK 라벨 시스템의 역할은 무엇인가요?
