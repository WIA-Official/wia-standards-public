WIA-HERITAGE-003 표준 개요

WIA-HERITAGE-003: Historical Document Standards

03.1 표준의 전체 아키텍처

WIA-HERITAGE-003은 역사 문서 디지털화를 위한 완전한 생태계를 제공합니다. 단순히 파일 형식을 정의하는 것을 넘어, 데이터 생성부터 장기 보존, 접근, 분석까지 전체 생명주기를 아우릅니다.

4단계 레이어드 아키텍처

표준은 OSI 7계층 모델에서 영감을 받아, 역사 문서 시스템을 4개의 계층으로 분리합니다. 각 계층은 명확한 책임을 가지며, 상위 계층은 하위 계층의 세부사항을 알 필요가 없습니다.

계층명칭역할핵심 기술독립 구현 가능
Layer 1데이터 형식메타데이터, 텍스트, 이미지 표현JSON, TEI XML, IIIF✅ 예
Layer 2API 인터페이스CRUD 연산, 검색, 업로드REST, GraphQL✅ 예
Layer 3프로토콜협업, 스트리밍, 동기화WebSocket, WebRTC⚠️ Layer 2 필요
Layer 4통합AI, 블록체인, 외부 시스템ML Pipeline, Ethereum⚠️ Layer 1-2 필요
🏗️ 레이어드 아키텍처의 장점
  • 점진적 채택: Layer 1만 구현해도 즉시 가치 제공. 나중에 Layer 2, 3, 4 추가 가능
  • 기술 독립성: Layer 2에서 REST 대신 GraphQL 사용 가능. Layer 1은 영향 없음
  • 테스트 용이: 각 계층을 독립적으로 테스트하고 검증
  • 유지보수성: 한 계층의 버그가 다른 계층에 파급되지 않음
  • 확장성: 필요한 계층만 스케일 아웃 (예: API 서버만 10대로 증설)

03.2 핵심 설계 원칙

WIA-HERITAGE-003 표준은 다음 8가지 원칙을 기반으로 설계되었습니다. 이 원칙들은 때로 상충되기도 하며, 실제 구현 시 균형을 잡아야 합니다.

1. 웹 우선 (Web-First)

모든 기능은 웹 브라우저에서 별도 플러그인 없이 작동해야 합니다. 이는 접근성을 극대화하고, 운영체제나 디바이스에 독립적입니다.

  • 긍정적 효과: 전 세계 누구나 Chrome, Firefox, Safari로 접근 가능
  • 기술적 제약: WebAssembly, Service Worker, IndexedDB 등 최신 웹 표준 활용
  • 실제 적용: IIIF 뷰어가 JavaScript만으로 수만 페이지 문서 렌더링

2. API 우선 (API-First)

UI를 만들기 전에 API를 먼저 설계합니다. 이는 다양한 클라이언트(웹, 모바일, CLI, 다른 서버)가 같은 기능을 사용할 수 있게 합니다.

// 잘못된 접근: UI에서 직접 DB 접근
async function displayDocument(docId) {
  const doc = await database.documents.findById(docId); // ❌
  renderHTML(doc);
}

// 올바른 접근: API를 통한 접근
async function displayDocument(docId) {
  const doc = await fetch(`/api/documents/${docId}`).then(r => r.json()); // ✅
  renderHTML(doc);
}

3. 데이터 주권 (Data Sovereignty)

각 기관은 자신의 데이터를 완전히 통제합니다. 중앙 서버에 의존하지 않으며, 원하면 완전히 독립적으로 운영 가능합니다.

모델데이터 위치장점단점
중앙집중형단일 클라우드 서버간단한 관리단일 장애점, 통제권 상실
연합형(Federation)각 기관 서버통제권 유지, 프라이버시복잡한 동기화
분산형(P2P)여러 노드에 복제높은 가용성일관성 문제, 매우 복잡

WIA-HERITAGE-003은 연합형 모델을 채택합니다. 각 기관이 자신의 서버를 운영하되, 표준 프로토콜로 다른 기관과 협력합니다.

4. 후방 호환성 (Backward Compatibility)

새 버전은 항상 이전 버전과 호환됩니다. WIA-HERITAGE-003 v2.0이 나와도 v1.0 데이터는 그대로 사용 가능합니다.

5. 확장 가능성 (Extensibility)

표준에 정의되지 않은 새로운 필드나 기능을 추가할 수 있습니다. JSON의 `extensions` 필드나 XML의 커스텀 네임스페이스를 통해 구현합니다.

6. 다국어/다문자 (Multilingual/Multiscript)

모든 텍스트 필드는 여러 언어와 문자 체계를 지원합니다. 아랍어, 한자, 데바나가리, 키릴 문자 모두 완벽히 지원합니다.

7. 오프라인 우선 (Offline-First)

네트워크 연결 없이도 기본 기능(보기, 검색, 주석)이 작동해야 합니다. Progressive Web App (PWA) 기술 활용.

8. 홍익인간 (Benefit All Humanity)

역사 문서는 인류의 공동 유산입니다. 표준은 특정 기업, 국가, 기관의 이익이 아닌 전 인류의 이익을 우선합니다.

03.3 기존 표준과의 관계

WIA-HERITAGE-003은 기존의 검증된 표준들을 대체하지 않고 통합합니다. "바퀴를 재발명하지 않는다"는 원칙입니다.

Dublin Core 통합

WIA-HERITAGE-003 JSON 메타데이터는 Dublin Core 15개 요소를 모두 포함하며, 추가 필드로 확장합니다:

{
  "@context": "https://wia.org/heritage-003/context.json",
  "id": "https://example.org/documents/goryeo-daejanggyeong-vol1",

  // Dublin Core 호환 필드
  "title": {"ko": "고려대장경 제1권", "en": "Tripitaka Koreana Vol. 1"},
  "creator": {"ko": "수기 스님 등", "en": "Monk Sugi et al."},
  "date": "1251",
  "type": "manuscript",
  "format": "application/tei+xml",
  "language": ["hanja", "ko"],
  "rights": "CC BY 4.0",

  // WIA-HERITAGE-003 확장 필드
  "physicalDescription": {
    "material": "목판",
    "dimensions": {"width": "69.5cm", "height": "23.9cm"},
    "condition": "excellent"
  },
  "digitizationInfo": {
    "scanDate": "2023-05-12",
    "resolution": 600,
    "equipment": "Bookeye 4 V3"
  }
}

TEI XML 통합

전사(transcription) 텍스트는 TEI P5 호환 XML로 저장합니다. WIA는 TEI의 프로필을 정의하여 사용할 요소를 제한합니다:

TEI 프로필사용 요소용도
WIA-Heritage-Simple<text>, <body>, <div>, <p>, <lb>간단한 전사
WIA-Heritage-Standard+<persName>, <placeName>, <date>, <note>개체명 인식
WIA-Heritage-Critical+<del>, <add>, <choice>, <app>비평적 편집

IIIF 통합

모든 이미지는 IIIF Image API 3.0과 Presentation API 3.0을 따릅니다. WIA-HERITAGE-003은 IIIF 매니페스트에 추가 메타데이터를 확장합니다.

03.6 표준 거버넌스와 진화

WIA-HERITAGE-003은 살아있는 표준입니다. 커뮤니티의 피드백을 받아 지속적으로 진화합니다.

거버넌스 구조

  • Steering Committee (7명): Gold 인증 기관 대표. 전략적 방향 결정.
  • Technical Working Group (15명): 기술 전문가. 사양 작성 및 개정.
  • Community Forum (전체 참여): GitHub Discussions. 누구나 제안 가능.
  • Annual Conference: 연 1회, 주요 개선사항 승인.

버전 관리 정책

Semantic Versioning 2.0 사용: MAJOR.MINOR.PATCH (예: 1.3.2)

  • MAJOR: 호환성 깨지는 변경. 예: 필수 필드 제거 (1.0 → 2.0)
  • MINOR: 새 기능 추가, 호환성 유지. 예: 새 API 엔드포인트 (1.0 → 1.1)
  • PATCH: 버그 수정, 문서 개선. 예: 오타 수정 (1.0.0 → 1.0.1)

MAJOR 버전은 3년에 한 번, MINOR는 6개월마다, PATCH는 필요 시 수시로 릴리스합니다.

03.8 장 요약

✅ 핵심 내용
  1. 4단계 레이어드 아키텍처: 데이터 형식, API, 프로토콜, 통합 계층으로 구성. 각 계층 독립적 구현 가능.
  2. 8가지 설계 원칙: 웹 우선, API 우선, 데이터 주권, 후방 호환성, 확장 가능성, 다국어, 오프라인 우선, 홍익인간.

  5. 3단계 인증: Bronze(무료), Silver($2K), Gold($10K). 2024년 228개 기관 인증.
  6. 거버넌스: Steering Committee, Technical Working Group, Community Forum. Semantic Versioning으로 체계적 진화.

  6. Semantic Versioning (MAJOR.MINOR.PATCH)의 개념을 설명하고, WIA-HERITAGE-003의 버전 관리 정책이 표준의 안정성과 혁신 사이의 균형을 어떻게 맞추는지 논하세요.
