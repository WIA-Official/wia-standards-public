WIA-HERITAGE-003: Historical Document Standards

05.1 RESTful API 설계 원칙

WIA-HERITAGE-003 Phase 2는 역사 문서에 프로그래밍 방식으로 접근할 수 있는 RESTful API를 정의합니다. API는 웹 애플리케이션, 모바일 앱, 연구 도구, 다른 디지털 도서관이 문서 컬렉션과 상호작용하는 표준 방법입니다.

REST 아키텍처 제약조건

WIA API는 Roy Fielding의 REST 원칙을 따릅니다:

🔧 REST 핵심 원칙
  • 클라이언트-서버 분리: UI와 데이터 저장소가 독립적으로 진화
  • 무상태성(Stateless): 각 요청은 완전한 정보 포함, 서버는 세션 저장 안 함
  • 캐시 가능성: 응답에 캐시 가능 여부 명시, 네트워크 효율 향상
  • 균일한 인터페이스: 모든 리소스는 동일한 HTTP 메서드로 조작
  • 계층화 시스템: 클라이언트는 중간 서버(프록시, 로드밸런서) 인식 불필요

리소스 설계

API는 다음 주요 리소스를 노출합니다:

05.3 검색 API

강력한 검색 기능은 디지털 도서관의 핵심입니다. WIA API는 메타데이터 검색과 전체 텍스트 검색을 모두 지원합니다.

간단한 검색 (GET /search)

GET /heritage/v1/search?q=정약용&fields=title,creator,description

{
  "query": "정약용",
  "totalResults": 47,
  "results": [
    {
      "id": "https://api.example.org/heritage/v1/documents/dasan-gyeongse-yupyo",
      "title": {"ko": "경세유표"},
      "creator": {"name": {"ko": "정약용"}},
      "date": "1817",
      "score": 0.95,
      "highlights": {
        "description": "다산 정약용이 강진 유배 시절 저술한..."
      }
    }
  ]
}

고급 검색 (POST /search)

복잡한 쿼리는 POST로 JSON 페이로드 전송:

POST /heritage/v1/search
Content-Type: application/json

{
  "query": {
    "bool": {
      "must": [
        {"match": {"text": "종묘"}},
        {"range": {"date": {"gte": "1400", "lte": "1500"}}}
      ],
      "should": [
        {"match": {"creator.name.ko": "춘추관"}}
      ],
      "filter": [
        {"term": {"type": "manuscript"}},
        {"term": {"language": "hanja"}}
      ]
    }
  },
  "sort": [
    {"date": "asc"},
    {"_score": "desc"}
  ],
  "from": 0,
  "size": 20
}

이 쿼리는 Elasticsearch Query DSL과 유사하며, 다음을 의미합니다:

  • 텍스트에 "종묘"가 반드시 포함 (must)
  • 날짜가 1400-1500년 사이 (must)
  • 작성자가 "춘추관"이면 점수 상승 (should)
  • 필사본(manuscript)만 (filter)
  • 한자(hanja) 언어만 (filter)
  • 날짜 오름차순, 점수 내림차순 정렬

패싯 검색 (Faceted Search)

05.4 IIIF Image API 통합

모든 이미지는 IIIF Image API 3.0을 따릅니다. 이는 WIA API와 별도의 엔드포인트이지만 긴밀히 통합됩니다.

IIIF Image API URL 구조

{scheme}://{server}/{prefix}/{identifier}/{region}/{size}/{rotation}/{quality}.{format}

예시:

https://iiif.example.org/image/joseon-annals-vol127-page1/full/max/0/default.jpg
https://iiif.example.org/image/joseon-annals-vol127-page1/0,0,500,500/300,/0/default.jpg
https://iiif.example.org/image/joseon-annals-vol127-page1/pct:10,10,80,80/!800,800/90/gray.png

파라미터 설명

파라미터값설명
regionfull, x,y,w,h, pct:x,y,w,h이미지의 특정 영역
sizemax, w,h, !w,h, pct:n출력 크기
rotation0, 90, 180, 270회전 각도
qualitydefault, color, gray, bitonal색상 품질
formatjpg, png, webp, tif파일 형식

05.5 인증과 권한 관리

일부 역사 문서는 저작권, 문화적 민감성, 보존 상태로 인해 접근 제한이 필요합니다. WIA API는 OAuth 2.0과 JWT를 사용합니다.

OAuth 2.0 흐름

1. 클라이언트가 인증 URL로 사용자 리디렉션
   GET https://auth.example.org/oauth/authorize?
       client_id=abc123&
       redirect_uri=https://myapp.com/callback&
       response_type=code&
       scope=read:documents write:annotations

2. 사용자 로그인 후 인증 코드 받음
   https://myapp.com/callback?code=xyz789

3. 인증 코드로 액세스 토큰 교환
   POST https://auth.example.org/oauth/token
   {
     "grant_type": "authorization_code",
     "code": "xyz789",
     "client_id": "abc123",
     "client_secret": "secret456"
   }

4. 액세스 토큰 획득
   {
     "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
     "token_type": "Bearer",
     "expires_in": 3600,
     "refresh_token": "refresh_xyz",
     "scope": "read:documents write:annotations"
   }

5. API 요청에 토큰 사용
   GET /heritage/v1/documents/restricted-doc-123
   Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

권한 스코프

스코프설명권한
read:documents문서 읽기GET /documents, /search
write:documents문서 작성/수정POST, PUT /documents
delete:documents문서 삭제DELETE /documents
read:annotations주석 읽기GET /annotations
write:annotations주석 작성POST /annotations
admin관리자 권한모든 API

05.6 GraphQL 대안

REST API 외에도 WIA-HERITAGE-003은 선택적으로 GraphQL 엔드포인트를 제공할 수 있습니다. GraphQL은 클라이언트가 필요한 데이터만 정확히 요청할 수 있게 합니다.

GraphQL 스키마

type Document {
  id: ID!
  title: MultilingualString!
  date: String
  creator: Person
  language: [String!]!
  type: DocumentType!
  description: MultilingualString
  images: [Image!]!
  text: TEIText
  annotations: [Annotation!]!
}

type MultilingualString {
  ko: String
  en: String
  ja: String
}

type Person {
  name: MultilingualString!
  birth: String
  death: String
  wikidataId: String
}

enum DocumentType {
  MANUSCRIPT
  LETTER
  OFFICIAL
  DIARY
  COMMERCIAL
}

type Query {
  document(id: ID!): Document
  documents(
    page: Int = 1,
    limit: Int = 50,
    type: DocumentType,
    language: String,
    dateFrom: String,
    dateTo: String
  ): DocumentConnection!
  search(query: String!, limit: Int = 20): [Document!]!
}

type Mutation {
  createDocument(input: CreateDocumentInput!): Document!
  updateDocument(id: ID!, input: UpdateDocumentInput!): Document!
  deleteDocument(id: ID!): Boolean!
}

05.7 실전 구현 가이드

세 가지 인기 있는 백엔드 프레임워크로 API 구현 예시를 제공합니다.

Express.js (Node.js) 구현

const express = require('express');
const app = express();

// 문서 조회
app.get('/heritage/v1/documents/:id', async (req, res) => {
  try {
    const doc = await database.documents.findById(req.params.id);
    if (!doc) {
      return res.status(404).json({error: 'Document not found'});
    }
    res.set('Cache-Control', 'public, max-age=3600');
    res.set('ETag', doc.etag);
    res.json(doc);
  } catch (err) {
    res.status(500).json({error: 'Internal server error'});
  }
});

// 문서 검색
app.get('/heritage/v1/search', async (req, res) => {
  const {q, page = 1, limit = 50} = req.query;
  const results = await searchEngine.search(q, {
    from: (page - 1) * limit,
    size: limit
  });
  res.json({
    totalResults: results.total,
    page: parseInt(page),
    perPage: parseInt(limit),
    items: results.hits
  });
});

app.listen(3000, () => console.log('API running on port 3000'));

Flask (Python) 구현

from flask import Flask, jsonify, request
from flask_cors import CORS

app = Flask(__name__)
CORS(app)

@app.route('/heritage/v1/documents/<doc_id>')
def get_document(doc_id):

    if not doc:
        return jsonify({'error': 'Document not found'}), 404

@app.route('/heritage/v1/search')

    query = request.args.get('q', '')

Spring Boot (Java) 구현

@RestController
@RequestMapping("/heritage/v1")
public class DocumentController {

    @Autowired
    private DocumentRepository documentRepo;

    @GetMapping("/documents/{id}")
    public ResponseEntity<Document> getDocument(@PathVariable String id) {
        Document doc = documentRepo.findById(id)
            .orElseThrow(() -> new DocumentNotFoundException(id));

        return ResponseEntity.ok()
            .cacheControl(CacheControl.maxAge(1, TimeUnit.HOURS).cachePublic())
            .eTag(doc.getEtag())
            .body(doc);
    }

    @GetMapping("/search")
    public SearchResponse search(
        @RequestParam String q,
        @RequestParam(defaultValue = "1") int page,
        @RequestParam(defaultValue = "50") int limit
    ) {
        SearchResults results = searchService.search(q, page, limit);
        return new SearchResponse(results);
    }
}

05.8 장 요약

✅ 핵심 내용
  1. RESTful API는 6개 주요 리소스(/documents, /iiif, /text, /annotations, /collections, /search)를 HTTP 메서드로 조작합니다.
  2. CRUD 연산: GET(조회), POST(생성), PUT(수정), DELETE(삭제). 페이지네이션, 필터링, ETag 기반 조건부 요청 지원.
  3. 검색 API는 간단한 쿼리 파라미터와 Elasticsearch 스타일의 고급 쿼리 DSL을 모두 지원. 패싯 검색으로 탐색 가능.
  4. IIIF Image API 3.0 통합으로 region, size, rotation, quality, format 파라미터로 이미지 변환 요청 가능.
  5. OAuth 2.0 + JWT 인증. 6개 권한 스코프(read:documents, write:documents 등). IIIF Auth API로 이미지 접근 제어.
  6. GraphQL 옵션 제공. 오버페칭/언더페칭 해결하지만 캐싱 복잡. REST와 병행 가능.
  7. 실전 구현: Express.js, Flask, Spring Boot 코드 예시. CORS, 에러 처리, 캐싱 모범 사례 포함.

05.9 복습 문제

  1. REST 아키텍처의 5가지 핵심 제약조건을 나열하고, "무상태성(Stateless)"이 API 확장성에 미치는 영향을 설명하세요.
  2. IIIF Image API URL 구조에서 다음 요청의 의미를 설명하세요: /image/doc123/pct:10,10,80,80/!800,800/90/gray.png
  3. OAuth 2.0 Authorization Code 흐름의 5단계를 순서대로 설명하고, 왜 클라이언트 시크릿이 서버에서만 사용되어야 하는지 보안 관점에서 논하세요.
  4. 패싯 검색(Faceted Search)이 사용자 경험에 어떻게 기여하는지 설명하고, WIA API의 패싯 응답 예시를 분석하세요.
  5. GraphQL이 REST API의 "오버페칭"과 "언더페칭" 문제를 어떻게 해결하는지 구체적인 예를 들어 설명하세요. 반면 GraphQL의 단점은 무엇인가요?
  6. ETag 헤더를 사용한 조건부 요청(If-Match)이 동시성 제어(Concurrency Control)에서 중요한 이유를 설명하고, PUT 요청 시나리오를 가정하여 설명하세요.
