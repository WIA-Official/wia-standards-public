📊 Chapter 4: Phase 1 - Data Format

Phase 1 of the WIA-AMR standard establishes the foundational data formats that enable interoperability at the most fundamental level—the structure and semantics of data exchanged between AMR systems. This chapter provides complete specifications for every data type, JSON schemas, validation rules, and practical implementation guidance.

4.1 Data Model Overview

The WIA-AMR data model organizes information into logical domains that reflect the core aspects of AMR operation. Each domain has defined schemas, required and optional fields, and clear semantics that ensure consistent interpretation across implementations.

Core Data Domains

Domain Description Key Data Types
Identity Robot identification and metadata RobotInfo, Manufacturer, Model
Pose Position and orientation Position, Orientation, Pose
Kinematics Motion state Velocity, Acceleration
Power Energy state BatteryState, ChargingInfo
Sensors Perception data LidarScan, CameraImage, ProximitySensor
Navigation Path and goal information Path, Waypoint, Goal, Map
Tasks Work assignments Task, Action, Order
Safety Safety status SafetyState, EmergencyStop, Obstacle

4.3 Position and Orientation

Spatial data is fundamental to AMR operation. The WIA-AMR standard defines precise formats for position and orientation that eliminate ambiguity:

Position Schema

Field Type Unit Required Description
x number meters ✅ X coordinate in reference frame
y number meters ✅ Y coordinate in reference frame
z number meters Optional Z coordinate (default 0)

Orientation Schema

Orientation can be specified in two formats—theta (simple) or quaternion (complete). Systems must support theta and should support quaternion:

// Simple orientation (2D rotation)
"orientation": {
  "theta": 1.5708  // radians, counterclockwise from X-axis
}

// Full quaternion (3D rotation)
"orientation": {
  "quaternion": {
    "x": 0.0,
    "y": 0.0,
    "z": 0.7071,
    "w": 0.7071
  }
}

Reference Frames

The standard defines reference frames to ensure consistent interpretation of coordinates:

Frame Origin Use Case
map Map origin (0,0) Global positioning within facility
odom Robot start position Odometry-based localization
base_link Robot center Robot-relative coordinates
sensor_* Sensor mount point Sensor-specific data

4.4 Battery and Power Data

Accurate battery information is critical for fleet management. The WIA-AMR standard provides comprehensive power state data:

{
  "battery": {
    "stateOfCharge": 85.5,       // Percentage (0-100)
    "voltage": 48.2,             // Volts
    "current": -8.5,             // Amps (negative = discharging)
    "temperature": 32.1,         // Celsius
    "health": 98,                // Percentage (0-100)
    "cycleCount": 245,           // Complete charge cycles
    "estimatedRuntime": 180,     // Minutes remaining
    "charging": false,           // Currently charging
    "chargingRate": null,        // W when charging
    "timeToFull": null           // Minutes when charging
  }
}

Battery Field Semantics

Field Range Notes
stateOfCharge 0-100 Must be calculated using coulomb counting or equivalent
health 0-100 Capacity compared to new battery
estimatedRuntime ≥0 Based on current load, not maximum

4.5 Sensor Data Formats

AMRs employ various sensors for perception. The WIA-AMR standard defines formats for common sensor types:

LiDAR Scan Data

{
  "lidarScan": {
    "sensorId": "front_lidar",
    "timestamp": "2025-01-15T14:30:45.100Z",
    "frame": "sensor_lidar_front",
    "angleMin": -3.14159,        // radians
    "angleMax": 3.14159,
    "angleIncrement": 0.00175,   // ~0.1 degree
    "rangeMin": 0.05,            // meters
    "rangeMax": 25.0,
    "ranges": [2.45, 2.46, 2.48, ...],  // distance per angle
    "intensities": [45, 48, 52, ...]    // optional reflectivity
  }
}

Camera Image Reference

Full image data is typically not included in telemetry due to bandwidth. Instead, references to image streams or snapshots are provided:

{
  "cameraCapture": {
    "sensorId": "front_camera",
    "timestamp": "2025-01-15T14:30:45.100Z",
    "frame": "sensor_camera_front",
    "imageUrl": "https://fleet.example.com/images/AMR-001/front/1705330245100.jpg",
    "width": 1920,
    "height": 1080,
    "encoding": "jpeg",
    "compressionQuality": 85
  }
}

4.6 Navigation Data Structures

Map Format

Maps are represented as occupancy grids with metadata:

{
  "map": {
    "mapId": "warehouse-main-v3",
    "name": "Main Warehouse Floor",
    "version": "3.2.0",
    "created": "2025-01-10T08:00:00Z",
    "resolution": 0.05,           // meters per cell
    "origin": {
      "position": { "x": -25.0, "y": -15.0 },
      "orientation": { "theta": 0.0 }
    },
    "dimensions": {
      "width": 1000,              // cells
      "height": 600               // cells
    },
    "dataUrl": "https://maps.example.com/warehouse-main-v3.pgm",
    "dataFormat": "pgm",          // portable graymap
    "occupancyThreshold": {
      "free": 25,                 // 0-threshold = free
      "occupied": 65              // threshold-255 = occupied
    },
    "nodes": [
      { "nodeId": "DOCK-01", "position": { "x": 2.5, "y": 3.0 }, "type": "dock" },
      { "nodeId": "PICK-15", "position": { "x": 25.0, "y": 30.0 }, "type": "pick" }
    ],
    "edges": [
      { "edgeId": "DOCK-01_PICK-15", "start": "DOCK-01", "end": "PICK-15", "bidirectional": true }
    ],
    "zones": [
      { "zoneId": "charging-area", "type": "charging", "polygon": [...] },
      { "zoneId": "high-traffic", "type": "speed-limit", "maxSpeed": 0.5 }
    ]
  }
}

4.7 Task and Order Structures

Tasks represent work assignments for AMRs. The WIA-AMR task format is compatible with VDA 5050 orders while providing additional structure:

{
  "task": {
    "taskId": "TASK-2025-00001",
    "orderId": "ORD-2025-15472",
    "priority": 5,
    "created": "2025-01-15T14:25:00Z",
    "deadline": "2025-01-15T15:00:00Z",
    "status": "assigned",
    "assignedRobot": "AMR-001",
    "taskType": "transport",
    "source": {
      "nodeId": "PICK-15",
      "position": { "x": 25.0, "y": 30.0 },
      "actions": [
        {
          "actionType": "pick",
          "actionId": "pick-001",
          "parameters": {
            "itemId": "SKU-12345",
            "quantity": 1
          }
        }
      ]

        {
          "actionType": "drop",

    "constraints": {
      "maxDuration": 600,
      "requiredCapabilities": ["transport", "lift"],

4.8 Safety Data Structures

Safety information is mandatory in all telemetry messages. The safety schema ensures consistent reporting of safety-critical states:

{
  "safety": {
    "emergencyStop": false,          // E-stop activated
    "protectiveStop": false,         // Safety sensor triggered
    "obstacleDetected": false,       // Obstacle in path
    "safetyZone": "normal",          // normal | warning | protective
    "fieldViolation": false,         // Safety field penetrated
    "lastSafetyEvent": {
      "type": "obstacle_detected",
      "timestamp": "2025-01-15T14:28:30Z",
      "duration": 2500,              // ms
      "resolved": true
    },
    "safetySystemStatus": {
      "lidarSafety": "operational",
      "bumperSensors": "operational",
      "emergencyStopCircuit": "operational"
    }
  }
}

Safety Zone Definitions

Zone Response Typical Distance
normal Full speed operation >2m from obstacles
warning Reduced speed 1-2m from obstacles
protective Immediate stop <1m from obstacles

4.9 Validation and Schema Registry

All WIA-AMR data formats are defined as JSON Schema documents, enabling automated validation:

// Example: Validating telemetry message
import Ajv from 'ajv';
import { telemetrySchema } from '@wia/amr-schemas';

const ajv = new Ajv({ allErrors: true });
const validate = ajv.compile(telemetrySchema);

const telemetryMessage = { /* ... */ };
const valid = validate(telemetryMessage);

if (!valid) {
  console.error('Validation errors:', validate.errors);
} else {
  console.log('Message is WIA-AMR compliant');
}

Schema Registry

The official WIA-AMR schema registry is available at:

  • Base URL: https://schemas.wiastandards.com/amr/v1/
  • Telemetry: telemetry-v1.0.json
  • Task: task-v1.0.json
  • Map: map-v1.0.json
  • Complete: wia-amr-v1.0.json (all schemas bundled)

📌 Chapter Summary: Key Takeaways

  1. Data Domains: WIA-AMR organizes data into logical domains (Identity, Pose, Kinematics, Power, Sensors, Navigation, Tasks, Safety)
  2. Telemetry Schema: Comprehensive JSON structure combining all robot state information in a single message
  3. Position Format: Meters as standard unit, with defined reference frames (map, odom, base_link)
  4. Battery Data: Complete power state including SoC, voltage, current, health, and runtime estimates
  5. Navigation Structures: Maps as occupancy grids with nodes, edges, and zones; paths as waypoint sequences
  6. Safety is Mandatory: Every telemetry message must include safety state information
