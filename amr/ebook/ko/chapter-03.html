🏗️ WIA-AMR 표준 개요

WIA-AMR 표준의 전체 아키텍처와 4단계 접근법, 그리고 표준이 구축된 철학적 기반을 살펴봅니다.

3.1 WIA-AMR 표준의 비전

WIA-AMR 표준은 자율이동로봇 생태계의 근본적인 변화를 목표로 합니다. 단순히 기술 명세를 정의하는 것을 넘어, 산업 전체의 협력과 혁신을 촉진하는 기반을 구축하고자 합니다.

弘益人間
"널리 인간을 이롭게 하라" - WIA 표준의 핵심 철학

WIA-AMR 표준의 비전은 세 가지 핵심 요소로 구성됩니다:

보편적 상호운용성: 어떤 제조사의 AMR이든, 어떤 산업에서 사용되든, 모두 함께 작동할 수 있어야 합니다. 개별 시스템이 아닌 통합된 생태계로서 AMR 기술이 발전해야 합니다.

민주화된 접근성: 대기업뿐 아니라 중소기업, 스타트업, 연구기관 모두가 AMR 기술을 도입하고 개발할 수 있어야 합니다. 표준은 무료로 공개되며, 참조 구현은 오픈소스로 제공됩니다.

지속적 진화: 기술은 빠르게 변화합니다. 표준은 새로운 기술과 요구사항을 수용할 수 있도록 유연하게 설계되며, 커뮤니티 주도로 지속적으로 발전합니다.

3.1.1 표준의 범위

WIA-AMR 표준은 AMR 운영에 필요한 전체 기술 스택을 다룹니다. 단일 계층이 아닌 데이터부터 시스템 통합까지 모든 수준을 포괄합니다.

계층 범위 표준화 대상
데이터 계층 정보 표현 및 교환 JSON 스키마, 데이터 타입, 단위 체계
API 계층 시스템 간 인터페이스 REST API, 인증, 에러 처리
프로토콜 계층 실시간 통신 MQTT/WebSocket, 메시지 형식, VDA 5050
통합 계층 기업 시스템 연동 WMS/ERP 통합 패턴, 이벤트 처리

3.1.2 표준이 다루지 않는 영역

WIA-AMR 표준은 상호운용성에 집중하며, 다음 영역은 의도적으로 범위에서 제외됩니다:

하드웨어 사양: 센서 종류, 구동 방식, 배터리 용량 등 물리적 하드웨어 요구사항은 표준화하지 않습니다. 제조사의 혁신 여지를 남겨두기 위함입니다.

내부 알고리즘: SLAM, 경로 계획, 장애물 회피 등 내부 알고리즘의 구현 방식은 제조사 재량입니다. 표준은 입력과 출력만 정의합니다.

비즈니스 프로세스: 작업 할당 전략, 충전 스케줄링 등 비즈니스 로직은 사용자 환경에 따라 다르므로 표준화하지 않습니다.

3.2 설계 원칙

WIA-AMR 표준은 명확한 설계 원칙에 기반하여 개발되었습니다. 이 원칙들은 모든 기술적 결정의 기준이 됩니다.

3.2.1 단순성 우선

복잡한 표준은 채택되기 어렵습니다. WIA-AMR은 최소한의 필수 요소만 표준화하고, 선택적 확장을 허용하는 방식을 채택합니다.

핵심 사양의 경량화: 필수 구현 요소는 최소화하여 초기 도입 장벽을 낮춥니다. 기본적인 상호운용성은 적은 노력으로 달성할 수 있습니다.

점진적 복잡도: 고급 기능은 선택적 모듈로 제공됩니다. 필요에 따라 추가 기능을 도입할 수 있으며, 모든 기능을 한꺼번에 구현할 필요가 없습니다.

명확한 문서화: 모든 사양은 예제와 함께 명확하게 문서화됩니다. 모호한 해석의 여지를 최소화합니다.

3.2.2 실용성 중심

이론적 완벽함보다 현장 적용 가능성을 우선합니다.

기존 기술 활용: JSON, REST, MQTT 등 널리 검증된 기술을 기반으로 합니다. 새로운 기술을 발명하기보다 기존 기술을 조합합니다.

점진적 마이그레이션: 기존 시스템을 한꺼번에 교체하지 않고도 표준을 도입할 수 있습니다. 어댑터 패턴과 호환 레이어를 통한 단계적 전환을 지원합니다.

현장 검증: 모든 사양은 실제 환경에서 테스트됩니다. 이론상으로만 동작하는 사양은 포함되지 않습니다.

3.2.3 확장성 내재화

미래의 요구사항과 기술 변화를 수용할 수 있도록 설계합니다.

버전 관리: 명확한 버전 체계를 통해 호환성을 유지하면서 발전합니다. 하위 호환성 파괴는 메이저 버전 변경에서만 허용됩니다.

확장 포인트: 사양 곳곳에 확장 포인트가 정의되어 있습니다. 벤더 특화 기능이나 산업별 요구사항을 표준 범위 내에서 추가할 수 있습니다.

스키마 진화: 데이터 스키마는 새 필드 추가를 허용하며, 알려지지 않은 필드는 무시하는 방식으로 처리됩니다.

💡 설계 원칙 요약

• 단순성: 최소 필수 요소 + 선택적 확장

• 실용성: 검증된 기술 기반 + 점진적 도입

• 확장성: 명확한 버전 관리 + 확장 포인트

• 개방성: 무료 공개 + 오픈소스 참조 구현

• 호환성: VDA 5050 등 기존 표준과 연동

3.3 4단계 아키텍처

WIA-AMR 표준은 4개의 계층적 단계(Phase)로 구성됩니다. 각 단계는 명확한 목적을 가지며, 하위 단계가 상위 단계의 기반이 됩니다.

1
데이터 형식
표준 데이터 구조와 JSON 스키마
2
API 인터페이스
RESTful API와 인증/인가
3
프로토콜
실시간 통신과 VDA 5050
4
통합
WMS/ERP 연동 패턴

3.3.1 1단계: 데이터 형식 (Data Format)

모든 상호운용성의 기초는 데이터입니다. 시스템 간에 교환되는 정보가 동일하게 해석되어야 유의미한 통신이 가능합니다.

1단계에서는 AMR 시스템에서 사용되는 모든 데이터의 구조와 형식을 정의합니다. JSON Schema를 사용하여 엄격한 검증이 가능하며, 각 필드의 의미와 단위가 명확히 정의됩니다.

주요 데이터 타입:

  • 로봇 상태(RobotState): 위치, 방향, 속도, 배터리 등 로봇의 현재 상태
  • 작업(Task): 이동, 피킹, 도킹 등 로봇이 수행하는 작업
  • 지도(Map): 환경 지도, 경로, 노드, 엣지 정보
  • 이벤트(Event): 상태 변화, 오류, 경고 등 비동기 알림
  • 명령(Command): 로봇 제어를 위한 명령 메시지
// RobotState 스키마 예시 { "$schema": "http://json-schema.org/draft-07/schema#", "type": "object", "properties": { "robotId": { "type": "string", "description": "로봇 고유 식별자" }, "position": { "type": "object", "properties": { "x": { "type": "number", "description": "X 좌표 (미터)" }, "y": { "type": "number", "description": "Y 좌표 (미터)" }, "theta": { "type": "number", "description": "방향 (라디안)" } }, "required": ["x", "y", "theta"] }, "velocity": { "type": "object", "properties": { "linear": { "type": "number", "description": "선속도 (m/s)" }, "angular": { "type": "number", "description": "각속도 (rad/s)" } } }, "batteryLevel": { "type": "number", "minimum": 0, "maximum": 100, "description": "배터리 잔량 (%)" }, "operatingState": { "type": "string", "enum": ["IDLE", "NAVIGATING", "CHARGING", "ERROR"], "description": "운영 상태" }, "timestamp": { "type": "string", "format": "date-time", "description": "ISO 8601 타임스탬프" } }, "required": ["robotId", "position", "operatingState", "timestamp"] }

3.3.2 2단계: API 인터페이스 (API Interface)

데이터 형식이 정의되면, 이를 기반으로 시스템 간 인터페이스를 구축합니다. 2단계에서는 RESTful API 설계, 인증/인가 메커니즘, 에러 처리 패턴을 정의합니다.

API 설계 원칙:

  • 자원 중심: /robots, /tasks, /maps 등 자원 기반 URL 구조
  • 표준 HTTP 메서드: GET(조회), POST(생성), PUT(수정), DELETE(삭제)
  • 일관된 응답 형식: 성공/실패에 관계없이 동일한 구조
  • 페이지네이션: 대량 데이터 조회를 위한 표준 방식

인증/인가: OAuth 2.0과 API 키 방식을 모두 지원합니다. JWT(JSON Web Token)를 통한 무상태 인증을 권장하며, 역할 기반 접근 제어(RBAC)를 지원합니다.

엔드포인트 메서드 설명
/api/v1/robots GET 모든 로봇 목록 조회
/api/v1/robots/{id} GET 특정 로봇 상세 정보
/api/v1/robots/{id}/state GET 로봇 현재 상태
/api/v1/robots/{id}/commands POST 로봇에 명령 전송
/api/v1/tasks POST 새 작업 생성
/api/v1/tasks/{id} GET 작업 상태 조회

3.3.3 3단계: 프로토콜 (Protocol)

REST API는 요청-응답 패턴에 적합하지만, AMR 시스템에서는 실시간 양방향 통신이 필수적입니다. 3단계에서는 실시간 메시징 프로토콜과 VDA 5050 호환 통신을 정의합니다.

통신 패턴:

  • MQTT: 경량 발행/구독 메시징, IoT 환경에 최적화
  • WebSocket: 브라우저 기반 실시간 통신, 대시보드용
  • VDA 5050: 자동차 산업 표준, 마스터 컨트롤-AGV/AMR 통신

VDA 5050 호환성: WIA-AMR 표준은 VDA 5050과 완전 호환됩니다. VDA 5050 메시지를 WIA-AMR 형식으로 변환하거나 그 반대가 가능하며, 기존 VDA 5050 시스템과의 통합이 원활합니다.

3.3.4 4단계: 통합 (Integration)

마지막 단계는 AMR 시스템을 기존 기업 시스템과 통합하는 것입니다. WMS, ERP, MES 등과의 연동 패턴과 모범 사례를 정의합니다.

통합 패턴:

  • 이벤트 기반 통합: 시스템 간 느슨한 결합을 위한 이벤트 버스 활용
  • API 게이트웨이: 다중 시스템 접근을 위한 단일 진입점
  • 데이터 동기화: 마스터 데이터의 일관성 유지
  • 워크플로우 오케스트레이션: 복잡한 비즈니스 프로세스 조율

WMS 통합 참조 아키텍처: 작업 생성, 상태 동기화, 재고 업데이트 등 핵심 통합 시나리오에 대한 상세 가이드를 제공합니다.

3.4 기존 표준과의 관계

WIA-AMR은 기존 표준들과 경쟁하지 않고 보완합니다. 주요 표준들과의 관계를 살펴봅니다.

3.4.1 VDA 5050과의 호환성

VDA 5050은 독일 자동차산업협회가 개발한 AGV/AMR 통신 표준입니다. WIA-AMR은 VDA 5050을 프로토콜 계층의 핵심으로 채택합니다.

상호 매핑: VDA 5050의 Order, State, Visualization 등 핵심 메시지 타입이 WIA-AMR 스키마와 1:1로 매핑됩니다.

확장 영역: WIA-AMR은 VDA 5050이 다루지 않는 영역(WMS 통합, 분석, 멀티테넌시 등)을 확장합니다.

VDA 5050 WIA-AMR 관계
Order 메시지 Task 스키마 호환 + 확장
State 메시지 RobotState 스키마 1:1 매핑
Visualization Telemetry API 포함
Connection MQTT/WebSocket 동일 프로토콜
미정의 WMS 통합 WIA-AMR 확장
미정의 분석/대시보드 WIA-AMR 확장

3.4.2 ROS/ROS 2와의 관계

ROS(Robot Operating System)는 로봇 소프트웨어 개발의 사실상 표준입니다. WIA-AMR은 ROS와 경쟁하지 않습니다.

ROS는 개발 프레임워크: 로봇 내부 소프트웨어 개발에 사용됩니다. 센서 드라이버, 알고리즘 구현, 시뮬레이션 등에 활용됩니다.

WIA-AMR은 상호운용 표준: 로봇 간, 로봇-시스템 간 통신에 사용됩니다. ROS 기반 로봇이든 아니든 WIA-AMR을 통해 통신할 수 있습니다.

ROS 브리지: WIA-AMR 참조 구현에는 ROS 2 브리지가 포함됩니다. ROS 토픽을 WIA-AMR 메시지로 변환하거나 그 반대가 가능합니다.

3.4.3 ISO 안전 표준과의 관계

ISO 3691-4(무인 산업차량 안전), ISO 13482(개인용 서비스 로봇) 등 안전 표준은 WIA-AMR과 독립적입니다.

WIA-AMR은 안전 요구사항을 직접 정의하지 않지만, 안전 관련 데이터(비상 정지 상태, 안전 영역 정보 등)의 교환 형식을 표준화합니다. 이를 통해 다중 벤더 환경에서도 일관된 안전 모니터링이 가능합니다.

3.5 거버넌스 및 발전

표준의 장기적 성공은 효과적인 거버넌스에 달려 있습니다. WIA-AMR은 개방적이고 투명한 거버넌스 구조를 채택합니다.

3.5.1 거버넌스 구조

WIA 표준 위원회: 표준의 전략적 방향을 결정하는 최상위 의사결정 기구입니다. 산업계, 학계, 정부 대표로 구성됩니다.

기술 실무 그룹: 각 단계(Phase)별로 기술 실무 그룹이 운영됩니다. 구체적인 사양 개발과 검토를 담당합니다.

커뮤니티 기여: GitHub를 통해 누구나 이슈 제기, 개선 제안, 풀 리퀘스트를 제출할 수 있습니다. 모든 논의는 공개적으로 이루어집니다.

3.5.2 버전 관리

WIA-AMR은 시맨틱 버전 관리(Semantic Versioning)를 따릅니다: MAJOR.MINOR.PATCH

MAJOR: 하위 호환성을 깨는 변경. 기존 구현이 수정 없이 작동하지 않을 수 있음.

MINOR: 하위 호환 가능한 기능 추가. 기존 구현은 계속 작동함.

PATCH: 버그 수정, 문서 개선 등. 기능 변경 없음.

✅ 핵심 요약

• WIA-AMR은 보편적 상호운용성, 민주화된 접근성, 지속적 진화를 비전으로 함

• 4단계 아키텍처: 데이터 형식 → API 인터페이스 → 프로토콜 → 통합

• 단순성, 실용성, 확장성을 핵심 설계 원칙으로 채택

• VDA 5050과 완전 호환되며, ROS 2 브리지 제공

• 개방적 거버넌스와 시맨틱 버전 관리로 지속 발전
