🔌 2단계: API 인터페이스

RESTful API 설계, 인증/인가 메커니즘, 에러 처리, 페이지네이션 등 시스템 간 인터페이스 표준을 다룹니다.

5.1 RESTful API 설계 원칙

WIA-AMR의 API 인터페이스는 REST(Representational State Transfer) 아키텍처 스타일을 따릅니다. REST는 HTTP 프로토콜의 특성을 최대한 활용하여 직관적이고 확장 가능한 API를 설계하는 방법론입니다.

5.1.1 자원 중심 설계

REST API에서 모든 것은 자원(Resource)입니다. 로봇, 작업, 지도, 이벤트 등이 각각 하나의 자원이며, URI를 통해 고유하게 식별됩니다.

자원 명명 규칙:

  • 복수형 명사 사용: /robots, /tasks, /maps
  • 소문자와 하이픈 사용: /robot-states, /task-histories
  • 동사 사용 금지: /getRobots ❌, /robots ✅
  • 계층 구조 표현: /robots/{robotId}/tasks

💡 URI 설계 모범 사례

• /api/v1/robots - 모든 로봇 목록

• /api/v1/robots/amr-001 - 특정 로봇

• /api/v1/robots/amr-001/state - 로봇의 현재 상태

• /api/v1/robots/amr-001/tasks - 로봇에 할당된 작업들

• /api/v1/maps/warehouse-01/nodes - 지도의 노드 목록

5.1.2 HTTP 메서드 활용

HTTP 메서드는 자원에 대한 행위를 표현합니다. WIA-AMR은 다음과 같이 표준 HTTP 메서드를 사용합니다:

메서드 행위 멱등성 안전성 예시
GET 조회 ✅ ✅ 로봇 상태 조회
POST 생성 ❌ ❌ 새 작업 생성
PUT 전체 수정 ✅ ❌ 작업 정보 전체 갱신
PATCH 부분 수정 ❌ ❌ 작업 우선순위만 변경
DELETE 삭제 ✅ ❌ 작업 취소

멱등성(Idempotency): 동일한 요청을 여러 번 수행해도 결과가 같은 것. GET, PUT, DELETE는 멱등해야 합니다.

안전성(Safety): 요청이 서버 상태를 변경하지 않는 것. GET만 안전합니다.

5.2 핵심 API 엔드포인트

WIA-AMR 표준은 다음과 같은 핵심 API 엔드포인트를 정의합니다. 모든 엔드포인트는 `/api/v1` 접두사를 사용합니다.

5.2.1 로봇 관리 API

GET /api/v1/robots

등록된 모든 로봇 목록을 조회합니다.

// 요청 GET /api/v1/robots?status=IDLE&limit=10 HTTP/1.1 Authorization: Bearer {token} Accept: application/json // 응답 { "data": [ { "robotId": "amr-001", "name": "창고 로봇 1", "manufacturer": "WIA Robotics", "model": "WIA-AMR-500", "status": "IDLE", "position": { "x": 10.5, "y": 20.3, "theta": 0 }, "batteryLevel": 85, "lastSeen": "2025-01-15T10:30:00Z" }, // ... 더 많은 로봇 ], "pagination": { "total": 50, "limit": 10, "offset": 0, "hasMore": true } }
GET /api/v1/robots/{robotId}

특정 로봇의 상세 정보를 조회합니다.

GET /api/v1/robots/{robotId}/state

로봇의 현재 실시간 상태를 조회합니다.

POST /api/v1/robots/{robotId}/commands

로봇에 명령을 전송합니다.

// 이동 명령 요청 POST /api/v1/robots/amr-001/commands HTTP/1.1 Authorization: Bearer {token} Content-Type: application/json { "commandType": "NAVIGATE", "destination": { "x": 50.0, "y": 30.0, "theta": 1.57 }, "parameters": { "maxSpeed": 1.5, "avoidanceMode": "DYNAMIC" } } // 응답 { "commandId": "cmd-12345", "status": "ACCEPTED", "estimatedDuration": 45, "message": "명령이 수락되었습니다" }

5.2.2 작업 관리 API

POST /api/v1/tasks

새로운 작업을 생성합니다.

// 작업 생성 요청 POST /api/v1/tasks HTTP/1.1 Authorization: Bearer {token} Content-Type: application/json { "taskType": "NAVIGATE", "priority": 80, "destination": { "x": 100.0, "y": 50.0, "theta": 0 }, "actions": [ { "actionType": "PICK", "parameters": { "itemId": "SKU-12345", "quantity": 1 } } ], "constraints": { "deadline": "2025-01-15T12:00:00Z", "requiredCapabilities": ["LIFT", "SCAN"] } } // 응답 { "taskId": "task-67890", "status": "PENDING", "createdAt": "2025-01-15T10:30:00Z", "estimatedStartTime": "2025-01-15T10:32:00Z", "assignedRobotId": null }
GET /api/v1/tasks/{taskId}

특정 작업의 상태와 세부 정보를 조회합니다.

PUT /api/v1/tasks/{taskId}

작업 정보를 수정합니다 (상태가 PENDING인 경우에만).

DELETE /api/v1/tasks/{taskId}

작업을 취소합니다.

5.2.3 지도 관리 API

메서드 엔드포인트 설명
GET /api/v1/maps 모든 지도 목록 조회
GET /api/v1/maps/{mapId} 특정 지도 상세 정보
POST /api/v1/maps 새 지도 업로드
GET /api/v1/maps/{mapId}/nodes 지도의 노드 목록
GET /api/v1/maps/{mapId}/edges 지도의 엣지 목록
GET /api/v1/maps/{mapId}/zones 지도의 영역 목록

5.3 인증 및 인가

API 보안을 위해 WIA-AMR은 두 가지 인증 방식을 지원합니다: OAuth 2.0과 API Key.

5.3.1 OAuth 2.0

OAuth 2.0은 표준화된 인증 프레임워크로, 특히 사용자 기반 접근 제어가 필요한 경우에 적합합니다.

지원 그랜트 타입:

  • Client Credentials: 서비스 간 통신 (권장)
  • Authorization Code: 사용자 대신 접근 시
  • Refresh Token: 토큰 갱신
// Client Credentials 그랜트로 토큰 획득 POST /oauth/token HTTP/1.1 Content-Type: application/x-www-form-urlencoded grant_type=client_credentials &client_id=your-client-id &client_secret=your-client-secret &scope=robots:read tasks:write // 응답 { "access_token": "eyJhbGciOiJSUzI1NiIs...", "token_type": "Bearer", "expires_in": 3600, "scope": "robots:read tasks:write" } // API 호출 시 토큰 사용 GET /api/v1/robots HTTP/1.1 Authorization: Bearer eyJhbGciOiJSUzI1NiIs...

5.3.2 API Key

API Key는 단순한 인증이 필요한 경우에 적합합니다. 개발 환경이나 내부 서비스 통신에 주로 사용됩니다.

// 헤더를 통한 API Key 전달 GET /api/v1/robots HTTP/1.1 X-API-Key: wia_amr_key_abc123def456 // 또는 쿼리 파라미터 (권장하지 않음) GET /api/v1/robots?api_key=wia_amr_key_abc123def456

⚠️ API Key 보안 주의사항

• API Key는 반드시 HTTPS를 통해 전송해야 합니다

• 클라이언트 측 코드에 API Key를 노출하지 마세요

• 정기적으로 API Key를 갱신하세요

• IP 화이트리스트와 함께 사용하세요

5.3.3 역할 기반 접근 제어 (RBAC)

WIA-AMR은 역할 기반 접근 제어를 정의합니다:

역할 권한 사용 시나리오
admin 모든 권한 시스템 관리자
operator 로봇 제어, 작업 관리 현장 운영자
viewer 읽기 전용 모니터링, 대시보드
system 시스템 간 통신 WMS, ERP 연동

5.4 에러 처리

일관된 에러 처리는 API 사용성에 매우 중요합니다. WIA-AMR은 표준화된 에러 응답 형식을 정의합니다.

5.4.1 HTTP 상태 코드

코드 의미 사용 시나리오
200 OK 요청 성공
201 Created 리소스 생성 성공
204 No Content 삭제 성공
400 Bad Request 잘못된 요청 형식
401 Unauthorized 인증 실패
403 Forbidden 권한 없음
404 Not Found 리소스 없음
409 Conflict 상태 충돌
422 Unprocessable Entity 유효성 검증 실패
429 Too Many Requests 속도 제한 초과
500 Internal Server Error 서버 내부 오류
503 Service Unavailable 서비스 일시 중단

5.4.2 에러 응답 형식

{ "error": { "code": "ROBOT_NOT_FOUND", "message": "로봇 'amr-999'를 찾을 수 없습니다", "details": { "robotId": "amr-999", "suggestion": "로봇 ID를 확인하거나 /api/v1/robots에서 사용 가능한 로봇을 조회하세요" }, "requestId": "req-abc123", "timestamp": "2025-01-15T10:30:00Z" } }

에러 코드 체계: WIA-AMR은 계층적 에러 코드 체계를 사용합니다:

  • ROBOT_*: 로봇 관련 에러
  • TASK_*: 작업 관련 에러
  • AUTH_*: 인증/인가 에러
  • VALIDATION_*: 유효성 검증 에러
  • SYSTEM_*: 시스템 에러

5.5 페이지네이션 및 필터링

대량의 데이터를 효율적으로 조회하기 위한 표준 방식을 정의합니다.

5.5.1 오프셋 기반 페이지네이션

// 요청 GET /api/v1/tasks?limit=20&offset=40 // 응답 { "data": [...], "pagination": { "total": 150, "limit": 20, "offset": 40, "hasMore": true, "links": { "first": "/api/v1/tasks?limit=20&offset=0", "prev": "/api/v1/tasks?limit=20&offset=20", "next": "/api/v1/tasks?limit=20&offset=60", "last": "/api/v1/tasks?limit=20&offset=140" } } }

5.5.2 필터링 및 정렬

// 상태 필터링 GET /api/v1/tasks?status=IN_PROGRESS // 다중 조건 GET /api/v1/tasks?status=PENDING,ASSIGNED&priority[gte]=50 // 날짜 범위 GET /api/v1/tasks?createdAt[gte]=2025-01-01&createdAt[lt]=2025-02-01 // 정렬 GET /api/v1/tasks?sort=-priority,createdAt // 필드 선택 GET /api/v1/robots?fields=robotId,status,position

✅ 핵심 요약

• REST 아키텍처 스타일을 따르며, 자원 중심 URI 설계

• HTTP 메서드를 의미에 맞게 사용 (GET, POST, PUT, DELETE)

• OAuth 2.0과 API Key 두 가지 인증 방식 지원

• 일관된 에러 응답 형식과 HTTP 상태 코드 사용

• 오프셋 기반 페이지네이션과 유연한 필터링 지원
