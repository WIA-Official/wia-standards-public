📊 1단계: 데이터 형식

JSON Schema 기반의 표준 데이터 구조, 핵심 스키마, 단위 체계, 검증 방법을 상세히 다룹니다.

4.1 데이터 형식의 중요성

상호운용성의 가장 기본적인 요소는 데이터입니다. 서로 다른 시스템이 동일한 데이터를 동일하게 해석할 수 있어야 유의미한 통신이 가능합니다. WIA-AMR 표준의 1단계는 이러한 데이터 형식의 표준화에 집중합니다.

데이터 형식 표준화가 중요한 이유는 다음과 같습니다:

의미의 일관성: "위치"라는 데이터가 있을 때, 모든 시스템이 이를 동일한 방식으로 해석해야 합니다. 단위(미터 vs 밀리미터), 좌표계(원점 위치), 정밀도 등이 명확히 정의되어야 합니다.

자동 검증 가능성: 스키마가 명확히 정의되면, 데이터의 유효성을 자동으로 검증할 수 있습니다. 잘못된 형식의 데이터가 시스템에 유입되는 것을 사전에 방지합니다.

문서화와 코드 생성: 스키마로부터 자동으로 문서와 타입 정의를 생성할 수 있습니다. 개발 생산성이 향상되고 오류가 감소합니다.

4.1.1 JSON Schema 선택 이유

WIA-AMR은 JSON Schema를 데이터 형식 정의의 기반으로 선택했습니다. 여러 대안(Protocol Buffers, Avro, XML Schema 등) 중 JSON Schema를 선택한 이유는 다음과 같습니다:

범용성: JSON은 거의 모든 프로그래밍 언어와 플랫폼에서 지원됩니다. 별도의 컴파일러나 도구 없이 바로 사용 가능합니다.

가독성: 사람이 읽고 쓰기 쉬운 텍스트 형식입니다. 디버깅과 문제 해결이 용이합니다.

웹 친화성: HTTP 기반 REST API와 자연스럽게 연동됩니다. 추가적인 직렬화/역직렬화 계층이 필요 없습니다.

풍부한 도구 생태계: 검증 라이브러리, 문서 생성기, 코드 생성기 등 다양한 도구가 존재합니다.

형식 장점 단점 적합성
JSON Schema 범용성, 가독성, 웹 친화 성능(바이너리 대비) ✅ 선택
Protocol Buffers 고성능, 타입 안전 컴파일 필요, 가독성 낮음 ⚠️ 옵션
Avro 스키마 진화, 압축 도구 제한적 ❌ 부적합
XML Schema 표현력, 검증 복잡성, 용량 ❌ 부적합

4.2 핵심 데이터 타입

WIA-AMR 표준은 AMR 시스템에서 사용되는 모든 데이터를 몇 가지 핵심 타입으로 분류합니다. 각 타입은 명확한 JSON Schema로 정의됩니다.

4.2.3 Map (지도)

로봇이 내비게이션에 사용하는 환경 지도를 정의합니다. 노드-엣지 기반의 토폴로지 맵과 그리드 기반의 점유 맵을 모두 지원합니다.

필드 타입 설명
mapId string 지도 고유 식별자
name string 지도 이름
version string 지도 버전
origin Position 지도 원점 (실제 세계 기준)
resolution number 그리드 해상도 (m/cell)
nodes Node[] 내비게이션 노드 목록
edges Edge[] 노드 간 연결 엣지
zones Zone[] 특수 영역 (충전소, 금지구역 등)
occupancyGrid string (base64) 점유 그리드 데이터

4.3 단위 체계 및 좌표계

데이터의 올바른 해석을 위해 단위 체계와 좌표계를 명확히 정의합니다.

4.3.1 표준 단위

WIA-AMR은 SI 단위계를 기본으로 사용합니다. 모든 데이터는 별도 명시가 없는 한 다음 단위를 사용합니다:

물리량 단위 기호 비고
거리 미터 m 소수점 이하 밀리미터 정밀도 권장
각도 라디안 rad -π ~ π 범위
시간 초 s 타임스탬프는 ISO 8601
속도 미터/초 m/s
각속도 라디안/초 rad/s
가속도 미터/초² m/s²
질량 킬로그램 kg
온도 섭씨 °C
전압 볼트 V
전류 암페어 A

4.3.2 좌표계 규약

WIA-AMR은 오른손 좌표계를 사용합니다:

X축: 전방 (로봇 이동 방향)

Y축: 좌측 (X축 기준)

Z축: 상방 (수직 상향)

Theta (θ): Z축 기준 반시계 방향 양수, X축(전방)이 0

⚠️ 좌표계 변환 주의사항

기존 시스템이 다른 좌표계를 사용하는 경우, WIA-AMR 어댑터에서 변환을 수행해야 합니다. 일반적인 변환:

• 밀리미터 → 미터: mm / 1000

• 도(degree) → 라디안: deg × π / 180

• 시계방향 양수 → 반시계방향 양수: -theta

4.4 스키마 검증

데이터의 유효성을 보장하기 위해 스키마 검증을 수행합니다. 모든 데이터는 수신 시점에 검증되어야 합니다.

4.4.1 검증 라이브러리

언어별 JSON Schema 검증 라이브러리:

언어 라이브러리 설치
TypeScript/JavaScript ajv npm install ajv
Python jsonschema pip install jsonschema
Java everit-json-schema Maven dependency
Go gojsonschema go get github.com/xeipuuv/gojsonschema
C# Newtonsoft.Json.Schema NuGet package

4.4.2 TypeScript 검증 예제

import Ajv from 'ajv'; import addFormats from 'ajv-formats'; // AJV 인스턴스 생성 const ajv = new Ajv({ allErrors: true }); addFormats(ajv); // RobotState 스키마 로드 const robotStateSchema = require('./schemas/robot-state.json'); const validateRobotState = ajv.compile(robotStateSchema); // 검증 함수 function validateAndParse(data: unknown): RobotState { const valid = validateRobotState(data); if (!valid) { const errors = validateRobotState.errors?.map(e => `${e.instancePath} ${e.message}` ).join(', '); throw new Error(`검증 실패: ${errors}`); } return data as RobotState; } // 사용 예시 try { const robotState = validateAndParse({ robotId: "amr-001", position: { x: 10.5, y: 20.3, theta: 1.57 }, operatingState: "NAVIGATING", timestamp: "2025-01-15T10:30:00Z" }); console.log('검증 성공:', robotState); } catch (error) { console.error('검증 실패:', error.message); }

4.5 확장성과 커스텀 필드

표준 스키마를 유지하면서 벤더별 확장을 허용하는 방법을 정의합니다.

4.5.1 확장 포인트

모든 핵심 스키마에는 `extensions` 필드가 정의되어 있습니다. 이 필드를 통해 표준에 정의되지 않은 추가 데이터를 포함할 수 있습니다.

{ "robotId": "amr-001", "position": { "x": 10.5, "y": 20.3, "theta": 1.57 }, "operatingState": "NAVIGATING", "timestamp": "2025-01-15T10:30:00Z", // 표준 확장 필드 "extensions": { "vendor:acme": { "internalTemperature": 45.2, "motorCurrent": [1.2, 1.3, 1.1, 1.4], "customSensorData": { ... } } } }

확장 필드 명명 규칙:

  • 벤더 접두사 필수: `vendor:회사명:필드명`
  • 소문자와 언더스코어 사용
  • 표준 필드와 충돌 금지

✅ 핵심 요약

• JSON Schema를 사용하여 모든 데이터 형식을 정의

• 핵심 데이터 타입: RobotState, Task, Map, Telemetry

• SI 단위계 사용 (미터, 라디안, 초)

• 오른손 좌표계: X-전방, Y-좌측, Z-상방

• extensions 필드를 통한 벤더 확장 지원
