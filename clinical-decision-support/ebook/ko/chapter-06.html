제6장: EHR 통합 및 데이터 표준

"상호운용성 없이는 진정한 CDSS는 불가능합니다. 표준 기반 데이터 교환은 의료 시스템 간 원활한 정보 흐름을 가능하게 합니다."

6.1 HL7 FHIR 표준

FHIR (Fast Healthcare Interoperability Resources)는 현대 의료 데이터 교환의 표준입니다.

🎯 FHIR의 핵심 개념

개념 설명
Resources Patient, Observation, Medication 등 데이터 단위
RESTful API HTTP 기반 표준 API (GET, POST, PUT, DELETE)
JSON/XML 웹 친화적 데이터 형식
Extensions 표준 확장 메커니즘

📋 주요 FHIR Resources

// Patient Resource 예시 (JSON)
{
  "resourceType": "Patient",
  "id": "example",
  "name": [{
    "family": "김",
    "given": ["민수"]
  }],
  "gender": "male",
  "birthDate": "1985-03-15",
  "address": [{
    "city": "서울",
    "postalCode": "06234"
  }]
}

// Observation Resource (혈압)
{
  "resourceType": "Observation",
  "status": "final",
  "category": "vital-signs",
  "code": {
    "coding": [{
      "system": "http://loinc.org",
      "code": "85354-9",
      "display": "Blood pressure"
    }]
  },
  "subject": {"reference": "Patient/example"},
  "effectiveDateTime": "2025-01-15T09:30:00Z",
  "component": [
    {
      "code": {"text": "Systolic"},
      "valueQuantity": {"value": 140, "unit": "mmHg"}
    },
    {
      "code": {"text": "Diastolic"},
      "valueQuantity": {"value": 90, "unit": "mmHg"}
    }
  ]
}

🔗 FHIR API 작업

# 환자 조회
GET /Patient/12345

# 환자의 최근 혈압 관찰
GET /Observation?patient=12345&code=85354-9&_sort=-date&_count=10

# 환자의 활성 약물
GET /MedicationStatement?patient=12345&status=active

# 새 처방 생성
POST /MedicationRequest
{
  "resourceType": "MedicationRequest",
  "status": "active",
  "intent": "order",
  "subject": {"reference": "Patient/12345"},
  "medicationCodeableConcept": {
    "coding": [{
      "system": "http://www.nlm.nih.gov/research/umls/rxnorm",
      "code": "197361",
      "display": "Lisinopril 10 MG Oral Tablet"
    }]
  },
  "dosageInstruction": [{
    "text": "하루 1회 10mg 복용"
  }]
}

6.2 의료 용어 표준

일관된 의료 데이터 교환을 위해 표준화된 용어 체계가 필수입니다.

📚 주요 용어 체계

표준 용도 예시
SNOMED CT 진단, 증상, 처치 44054006 = 당뇨병
LOINC 검사 및 관찰 2345-7 = 혈당
RxNorm 약물 197361 = Lisinopril 10mg
ICD-10 진단 코드 (청구) E11.9 = 제2형 당뇨병
CPT 처치 코드 (청구) 99213 = 외래 진찰
UCUM 단위 mg/dL, mmHg

🌐 한국 표준

  • KCD (한국표준질병사인분류) — ICD-10 기반
  • EDI 코드 — 보험 청구
  • ATC 코드 — 약물 분류
  • 의약품 표준 코드 — 식약처 지정

6.3 CDS Hooks 표준

CDS Hooks는 EHR 워크플로우에 CDSS를 통합하는 표준입니다.

⚡ CDS Hooks 흐름

1. EHR에서 이벤트 발생
   ↓
2. Hook 트리거 (예: "medication-prescribe")
   ↓
3. EHR → CDS 서비스 요청
   {
     "hook": "medication-prescribe",
     "context": {
       "patientId": "12345",
       "medications": [...]
     }
   }
   ↓
4. CDS 서비스 분석 수행
   - DDI 검사
   - 알레르기 확인
   - 용량 검증
   ↓
5. CDS 서비스 → EHR 응답 (Cards)
   {
     "cards": [
       {
         "summary": "⚠️ Warfarin-Aspirin 상호작용",
         "indicator": "warning",
         "detail": "출혈 위험 증가...",
         "suggestions": [
           {"label": "Clopidogrel로 대체"}
         ]
       }
     ]
   }
   ↓
6. EHR이 카드를 UI에 표시
   ↓
7. 의사 결정 (승인/대안 선택/무시)

🔧 주요 Hook 유형

Hook 트리거 시점
patient-view 환자 차트 열람 시
medication-prescribe 약물 처방 시
order-select 검사/처치 오더 시
order-sign 오더 서명 (확정) 시
encounter-discharge 퇴원 처리 시

6.4 데이터 품질 관리

✅ 데이터 품질 차원

차원 설명 검증 방법
완전성 필수 데이터 누락 없음 필수 필드 체크
정확성 데이터가 현실 반영 범위 검증, 논리 확인
일관성 모순 없음 교차 검증
적시성 최신 데이터 타임스탬프 확인
유효성 형식/표준 준수 스키마 검증

🔍 데이터 검증 예시

// 혈압 데이터 검증
{
  "systolic": 140,
  "diastolic": 90
}

검증 규칙:
✅ 완전성: systolic AND diastolic 존재
✅ 범위: 50 < systolic < 300
✅ 범위: 30 < diastolic < 200
✅ 논리: systolic > diastolic
⚠️ 경고: systolic >= 140 (고혈압)

// 약물 데이터 검증
{
  "medication": "Lisinopril",
  "dose": "10",
  "unit": "mg",
  "route": "PO"
}

검증:
✅ RxNorm 코드 매칭
✅ 용량 범위 (2.5-40 mg)
✅ 투여 경로 유효성
✅ 빈도 형식

6.5 개인정보 보호 및 보안

🔒 규정 준수

규정 지역 핵심 요구사항
HIPAA 미국 PHI 보호, 암호화, 감사 로그
GDPR EU 동의, 접근권, 삭제권
PIPL 중국 개인정보 보호, 데이터 현지화
개인정보보호법 한국 수집 동의, 안전조치, 유출 신고

🛡️ 보안 기술

  • 암호화 — 전송: TLS 1.3, 저장: AES-256
  • 접근 제어 — RBAC, ABAC, OAuth 2.0
  • 감사 로그 — 모든 접근 기록
  • 익명화 — 분석 시 식별정보 제거
  • 백업 및 복구 — 재해 복구 계획
