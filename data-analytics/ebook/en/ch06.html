Chapter 6: Real-time Analytics Architecture

6.1 Introduction to Real-time Analytics

Real-time analytics processes data as it arrives, providing immediate insights and enabling instant decision-making. Unlike batch processing that analyzes historical data, real-time analytics operates on streaming data with milliseconds to seconds latency. This capability is crucial for use cases like fraud detection, IoT monitoring, stock trading, and personalized recommendations.

6.2 Batch vs. Stream Processing

Data ScopeHistorical, bounded datasetsContinuous, unbounded streams
LatencyMinutes to hoursMilliseconds to seconds

ComplexitySimpler architectureMore complex, requires distributed systems
ExamplesApache Hadoop, Spark batchApache Kafka, Flink, Spark Streaming

6.3 Stream Processing Architectures

6.3.1 Lambda Architecture

Combines batch and stream processing to balance accuracy with speed.

Components:
  • Batch Layer: Processes complete datasets for accuracy
  • Speed Layer: Processes real-time data for low latency
  • Serving Layer: Merges results from both layers
Pros: Combines benefits of batch and stream
Cons: Complex to maintain, code duplication

6.3.2 Kappa Architecture

Simplifies Lambda by using only stream processing.

Key Idea: Treat everything as a stream, reprocess historical data as needed
Pros: Simpler, single codebase
Cons: Requires replayable message queue

6.5 Apache Kafka Fundamentals

Kafka is a distributed event streaming platform used for building real-time data pipelines.

6.5.1 Kafka Components

# Kafka Producer Example (Python)
from kafka import KafkaProducer
import json

producer = KafkaProducer(
    bootstrap_servers=['localhost:9092'],
    value_serializer=lambda v: json.dumps(v).encode('utf-8')
)

# Send messages
for i in range(100):
    data = {
        'user_id': i,
        'action': 'click',
        'timestamp': time.time()
    }
    producer.send('user-events', value=data)

producer.flush()
producer.close()

# Kafka Consumer Example
from kafka import KafkaConsumer
import json

consumer = KafkaConsumer(
    'user-events',
    bootstrap_servers=['localhost:9092'],
    value_deserializer=lambda m: json.loads(m.decode('utf-8')),
    auto_offset_reset='earliest',
    group_id='analytics-group'
)

for message in consumer:
    event = message.value
    # Process event
    print(f"User {event['user_id']} performed {event['action']}")

6.7 State Management

Maintaining state in distributed stream processing is crucial for complex analytics.

Stateful Operations:
  • Aggregations: Running sums, counts, averages
  • Joins: Combining streams based on keys
  • Pattern Detection: Complex event processing
  • Deduplication: Tracking seen events

6.8 Exactly-Once Semantics

Guaranteeing each message is processed exactly once, even in case of failures.

SemanticsGuaranteeTrade-offs
At-Most-OnceMay lose messagesHighest performance, lowest reliability
At-Least-OnceMay duplicate messagesGood performance, needs idempotent processing
Exactly-OnceNo loss, no duplicationLower performance, highest reliability

6.10 Implementing Real-time Analytics with WIA-DATA-012

# WIA-DATA-012 Real-time Analytics
from wia_analytics import RealtimeAnalytics

# Initialize
realtime = RealtimeAnalytics(api_key="your_api_key")

# Create stream processor
stream = realtime.create_stream(
    name="sales-events",
    source={
        "type": "kafka",
        "brokers": ["localhost:9092"],
        "topic": "transactions"
    },
    window={
        "type": "tumbling",
        "duration": "5m"
    },
    aggregations=[
        {"metric": "count", "field": "*"},
        {"metric": "sum", "field": "amount"},
        {"metric": "avg", "field": "amount"}
    ]
)

# Define anomaly detection
stream.add_anomaly_detector(
    algorithm="isolation_forest",
    sensitivity=0.95,
    features=["amount", "frequency"]
)

# Set up alerts
stream.add_alert(
    condition="sum(amount) > 100000",
    action="notify",
    channels=["email", "slack"]
)

# Start processing
stream.start()

# Monitor metrics
metrics = stream.get_metrics()
print(f"Events/sec: {metrics['throughput']}")
print(f"Avg latency: {metrics['latency_ms']}ms")
print(f"Anomalies: {metrics['anomalies_detected']}")

6.11 Performance Optimization

Best Practices:
  • Partitioning: Distribute load across partitions
  • Parallelism: Process multiple partitions concurrently
  • Compression: Reduce network and storage overhead
  • Batching: Group small messages for efficiency
  • Caching: Store frequently accessed data in memory
  • Backpressure: Handle overload gracefully
