## Theory

### Why Cloud GPU?

The first place I got stuck when starting this project was this very question: "Should I buy a GPU or rent one?" At the time, a used A100 80GB was asking 15 million won, and to run it properly, I'd need separate provisions for electricity, cooling, and space. Annual maintenance costs added up to 20-30% of the purchase price. For a solo developer, this number wasn't reality.

So I went the cloud route. Rent by the hour, shut it down when experiments finish, turn it back on when needed. The total amount I actually spent to complete all the hands-on practice in this book on cloud GPU was **approximately 300,000 to 800,000 won**. Every time a learning cycle failed, 20,000 to 50,000 won would disappear, so during the first month I spent a lot of time staring at receipts.

### Comparison of Four Services

| Service | A100 80GB per hour | Characteristics | Recommended Use |
|---|---|---|---|
| **Lambda Labs** | $1.99/hr | Stable, frequent stock shortages | Important long-term training |
| **AWS g5.2xlarge** | $1.21/hr | A10G 24GB, highest stability | Experimentation and development |
| **Paperspace** | $2.30/hr | GUI notebook, easy setup | Beginners, visualization |
| **Vast.ai** | $0.4~0.8/hr | Cheapest, lower reliability | Short-term experiments |

**Recommended Strategy:**
- Initial environment testing → Paperspace (easy with GUI)
- Main training → Lambda Labs (stability)
- Cost-critical experiments → Vast.ai
- If you have AWS credits → g5 series

I've used all four. This table is the summary of those experiences. The first few weeks, I built fundamentals with GUI on Paperspace, then moved to Lambda Labs for serious learning, and ran parallel experiments like hyperparameter sweeps on Vast.ai to stick to budget. No single option is the answer—**different tools fit different learning stages**. I once decided to use only Lambda from the start and lost 3 days to stock shortages. Knowing multiple providers frees you from such obstacles.

### Understanding GPU Specs

In fine-tuning, VRAM size determines almost everything.

| VRAM | Possible Tasks |
|---|---|
| 8GB (RTX 3070) | LoRA rank 4, batch 1, 512px |
| 16GB (T4, V100) | LoRA rank 16, batch 2, 768px |
| 24GB (A10G, RTX 3090) | QLoRA Full, batch 4, 1024px |
| 40GB (A100 40GB) | Full fine-tune small models |
| 80GB (A100 80GB) | Full fine-tune large models |

This book's hands-on practice is designed around **24GB (A10G)** as the baseline. AWS g5.2xlarge, equipped with A10G 24GB, offers optimal value.

Early on, I believed "you need A100 80GB to learn properly" and kept holding onto expensive instances, then saw the month's receipt and came to my senses. In reality, most of the 60 chapters in this book can be completed with **a single A10G 24GB**. Fine-tuning is not just about "using expensive GPUs well," but first about **"not wasting with the GPU you have"**. A pipeline designed to run on 24GB is easy to move to 80GB. The reverse is not.

```python
# Real-time GPU usage monitoring script
# Run in a separate terminal during training

import time
import subprocess

def get_gpu_stats():
    """Check GPU status with nvidia-smi"""
    result = subprocess.run(
        ['nvidia-smi', '--query-gpu=utilization.gpu,memory.used,memory.total,temperature.gpu',
         '--format=csv,noheader,nounits'],
        capture_output=True, text=True
    )
    if result.returncode != 0:
        return None
    
    values = result.stdout.strip().split(', ')
    return {
        'gpu_util': int(values[0]),
        'mem_used': int(values[1]),
        'mem_total': int(values[2]),
        'temp': int(values[3])
    }

def monitor(interval: int = 5):
    """Print GPU status every interval seconds"""
    print("Starting GPU monitoring (Ctrl+C to exit)")
    print(f"{'Time':12} {'GPU%':6} {'VRAM':14} {'Temp':6}")
    print("-" * 45)
    
    while True:
        stats = get_gpu_stats()
        if stats:
            vram_bar = "█" * (stats['mem_used'] // (stats['mem_total'] // 20))
            print(
                f"{time.strftime('%H:%M:%S'):12} "
                f"{stats['gpu_util']:5}% "
                f"{stats['mem_used']:5}/{stats['mem_total']}MB "
                f"{stats['temp']:4}°C"
            )
        time.sleep(interval)

if __name__ == "__main__":
    monitor()
```

## Pitfall Cards — Five Things I Learned the Hard Way

1. **"Leaving instances running and forgetting them"** → The most expensive mistake. One Friday evening I started training and forgot about it over the weekend. Monday morning, 300,000 won had evaporated. After that, I embed an `auto_shutdown()` function at the end of Lambda Labs training scripts, and always set **max_runtime** when creating Vast.ai instances.

2. **"Losing SSH keys"** → Lambda Labs and AWS only allow SSH key download once at the beginning. In my first week, not knowing this, I'd issue new keys every time I created an instance. It's convenient to safely back them up in the `~/.ssh/` folder and copy them once more to tools like 1Password.

3. **"CUDA version and PyTorch version mismatch"** → Where I lost my first half day. Running just `pip install torch` once installed PyTorch CPU build for CUDA 11.8. Check your CUDA first with `nvcc --version`, then explicitly install the PyTorch wheel matching that version. The above script is based on CUDA 12.1.

4. **"Running training without tmux"** → I once ran training over SSH on public Wi-Fi, and when the connection dropped, 3 hours of training vanished. After that day, the first command is always `tmux new -s name`. `screen` serves the same purpose.

5. **"Not backing up model files locally"** → A Vast.ai instance shutdown once took my checkpoint with it. After that, I put code by default to **periodically upload to S3 or Hugging Face Hub every 10 minutes** in my training loop. GPU costs are expensive to think about, but **the time to recreate a lost checkpoint costs even more**.

## Practice Problems

**Problem 1.** Choose the optimal GPU cloud for the following conditions and explain why: "Budget 200,000 won, completion within 24 hours, SDXL LoRA fine-tuning."

> **Answer:** Vast.ai ($0.8/hr × 24hr ≈ $19 ≈ 27,000 won) or Lambda Labs ($1.99/hr × 24hr ≈ $48 ≈ 70,000 won). For maximum budget savings, use Vast.ai; if stability matters, use Lambda Labs. For 24 hours, tmux is essential.

**Problem 2.** Modify the `get_gpu_stats()` function to return statistics for all GPUs as a list when multiple GPUs are present (multi-GPU setup).

> **Answer:** Remove `--id=0,1,2` from `nvidia-smi`, or split the output with `splitlines()` to parse each line. Change the return type to `List[dict]`.

**Problem 3.** Write a command to verify the actual installed package versions after running the `pip install` commands in the setup script above. How should you respond if unexpected versions are installed?

> **Answer:** Check with `pip show torch transformers diffusers | grep Version`. If versions differ, force reinstall with `pip install "torch==2.1.0" --force-reinstall`. In case of dependency conflicts, use `pip install package --no-deps` to install without dependencies, then verify manually.
