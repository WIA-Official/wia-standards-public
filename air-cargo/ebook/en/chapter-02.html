Regulatory Framework

IATA Regulations, Customs Procedures, and Compliance Requirements

Introduction to Air Cargo Regulations

Air cargo operates within one of the most heavily regulated environments in global logistics. Multiple international organizations, national governments, and industry bodies establish rules governing safety, security, customs clearance, and operational procedures. Understanding this regulatory framework is essential for compliance and efficient operations.

The regulatory landscape encompasses three primary domains: operational standards (IATA, ICAO), customs and trade (WCO, national customs), and safety/security (TSA, EU regulations, dangerous goods rules). Each domain has specific requirements that must be reflected in cargo management systems.

IATA Standards and Regulations

International Air Transport Association (IATA) Role

IATA establishes global standards for air cargo operations, including documentation formats, operational procedures, and industry best practices. Key IATA standards include:

Standard/Manual Purpose Key Requirements Update Frequency
Dangerous Goods Regulations (DGR) Safe transport of hazardous materials Classification, packaging, labeling, documentation Annual
Cargo Services Conference (CSC) Standard cargo practices and procedures AWB formats, handling codes, ULD standards Continuous
Temperature Control Regulations (TCR) Perishable and pharmaceutical handling Temperature ranges, packaging, monitoring Annual
Live Animals Regulations (LAR) Humane transport of live animals Container specifications, veterinary docs Annual
IATA Cargo-XML Electronic data interchange standards Message formats, data elements Version releases
ONE Record Digital standard for data sharing RESTful API, data model, logistics objects Continuous

Air Waybill (AWB) Standards

The Air Waybill is the most critical document in air cargo, serving as a contract of carriage, receipt, and customs declaration. IATA defines two formats:

  • Master Air Waybill (MAWB): Issued by airline to freight forwarder for consolidated shipments
  • House Air Waybill (HAWB): Issued by freight forwarder to individual shippers
  • Electronic Air Waybill (e-AWB): Paperless version achieving 70%+ adoption

AWB Number Structure:

Format: XXX-XXXXXXXX

  • First 3 digits: IATA airline prefix (e.g., 020 for Lufthansa, 180 for Korean Air)
  • Next 7 digits: Serial number assigned by airline
  • Last digit: Check digit calculated using mod 7 algorithm

// AWB Validator and Generator
class AWBManager {
  // Validate AWB number format and check digit
  static validateAWB(awbNumber: string): boolean {
    // Remove hyphen if present
    const cleanAWB = awbNumber.replace("-", "");

    if (cleanAWB.length !== 11) {
      return false;
    }

    const prefix = cleanAWB.substring(0, 3);
    const serial = cleanAWB.substring(3, 10);
    const checkDigit = parseInt(cleanAWB.substring(10, 11));

    // Validate numeric
    if (!/^\d{11}$/.test(cleanAWB)) {
      return false;
    }

    // Calculate check digit using mod 7
    const calculatedCheckDigit = this.calculateCheckDigit(prefix + serial);

    return calculatedCheckDigit === checkDigit;
  }

  // Calculate check digit using IATA mod 7 algorithm
  static calculateCheckDigit(awbNumberWithoutCheck: string): number {
    const number = BigInt(awbNumberWithoutCheck);
    const checkDigit = Number(number % 7n);
    return checkDigit;
  }

  // Generate next AWB number for given prefix
  static generateAWBNumber(prefix: string, lastSerial: number): string {
    if (prefix.length !== 3 || !/^\d{3}$/.test(prefix)) {
      throw new Error("Prefix must be 3 digits");
    }

    const nextSerial = (lastSerial + 1).toString().padStart(7, "0");
    const awbWithoutCheck = prefix + nextSerial;
    const checkDigit = this.calculateCheckDigit(awbWithoutCheck);

    return `${prefix}-${nextSerial}${checkDigit}`;
  }

  // Format AWB for display
  static formatAWB(awbNumber: string): string {
    const clean = awbNumber.replace(/\D/g, "");
    if (clean.length !== 11) {
      return awbNumber;
    }

    return `${clean.substring(0, 3)}-${clean.substring(3)}`;
  }

  // Parse AWB components
  static parseAWB(awbNumber: string): {
    prefix: string;
    serial: string;
    checkDigit: number;
  } | null {
    const clean = awbNumber.replace("-", "");

    if (clean.length !== 11 || !/^\d{11}$/.test(clean)) {
      return null;
    }

    return {
      prefix: clean.substring(0, 3),
      serial: clean.substring(3, 10),
      checkDigit: parseInt(clean.substring(10, 11))
    };
  }
}

Dangerous Goods Regulations

Classification System

The IATA Dangerous Goods Regulations (DGR) classify hazardous materials into 9 classes based on their primary hazard:

Class Hazard Type Examples Special Requirements
Class 1 Explosives Fireworks, ammunition, flares Forbidden on passenger aircraft
Class 2 Gases Compressed gas, aerosols, refrigerants Pressure testing, special containers
Class 3 Flammable liquids Paints, solvents, perfumes Flash point documentation required
Class 4 Flammable solids Matches, sulfur, metal powders Keep away from heat sources
Class 5 Oxidizing substances Hydrogen peroxide, fertilizers Segregation from flammables
Class 6 Toxic and infectious Pesticides, medical waste, clinical samples UN3373 packaging for Category B
Class 7 Radioactive materials Medical isotopes, industrial equipment Special permits, radiation monitoring
Class 8 Corrosive substances Acids, batteries, cleaning agents Special packaging to prevent leakage
Class 9 Miscellaneous dangerous goods Lithium batteries, dry ice, magnets Varies by substance

⚠️ Critical Compliance Requirements:

  • All dangerous goods must be declared and properly documented
  • Shippers must complete dangerous goods training (recurrent every 2 years)
  • Packaging must meet UN specification standards
  • Labels and markings must be applied per IATA DGR
  • Shipper's Declaration for Dangerous Goods (DGD) required
  • Some substances forbidden on passenger aircraft (Cargo Aircraft Only)
  • Non-compliance can result in fines up to $500,000 and criminal prosecution

Lithium Battery Regulations

Lithium batteries are Class 9 dangerous goods with specific packaging and documentation requirements. These are among the most commonly shipped dangerous goods in air cargo:

Battery Type UN Number Packaging Instruction Maximum per Package
Lithium ion cells UN3480 PI 965 (Section IA, IB, II) Varies by section
Lithium ion with equipment UN3481 PI 966 5kg net weight
Lithium ion in equipment UN3481 PI 967 5kg net weight
Lithium metal cells UN3090 PI 968 2.5kg net weight

Customs Procedures and Trade Compliance

Customs Documentation Requirements

International air cargo shipments must clear customs in both export and import countries. Required documentation typically includes:

  • Commercial Invoice: Details of goods, value, and transaction terms
  • Packing List: Description of package contents, weights, and dimensions
  • Certificate of Origin: Establishes country of manufacture for tariff purposes
  • Export/Import Licenses: Required for controlled goods
  • Customs Entry/Declaration: Formal customs filing
  • Harmonized Tariff Code: Classification for duty assessment

Incoterms and Cargo Responsibilities

International Commercial Terms (Incoterms) define responsibilities between buyers and sellers:

Incoterm Full Name Seller Responsibility Common Usage
EXW Ex Works Make goods available at seller's facility Buyer controls entire logistics
FCA Free Carrier Deliver to carrier nominated by buyer Most common for air cargo
CPT Carriage Paid To Pay freight to named destination Seller arranges transport
CIP Carriage and Insurance Paid To Pay freight and insurance Higher seller responsibility
DAP Delivered At Place Deliver to named place, ready for unloading Seller bears most risk
DDP Delivered Duty Paid Deliver with all duties paid Maximum seller obligation

// Customs Declaration Manager
class CustomsManager {
  // Calculate total duties and taxes
  calculateDuties(declaration: CustomsDeclaration): {
    totalDuties: number;
    totalTaxes: number;
    totalPayable: number;
  } {
    let totalDuties = 0;
    let totalTaxes = 0;

    for (const pkg of declaration.packages) {
      // Calculate duty
      const duty = pkg.totalValue.amount * (pkg.dutyRate / 100);
      totalDuties += duty;

      // Calculate VAT (example: 20%)
      const vatRate = 0.20;
      const tax = (pkg.totalValue.amount + duty) * vatRate;
      totalTaxes += tax;
    }

    return {
      totalDuties,
      totalTaxes,
      totalPayable: totalDuties + totalTaxes
    };
  }

  // Validate HS Code format
  static validateHSCode(hsCode: string): boolean {
    // HS codes are 6-10 digits
    const hsPattern = /^\d{4}\.\d{2}(\.\d{2}(\.\d{2})?)?$/;
    return hsPattern.test(hsCode);
  }

  // Determine duty rate based on HS code (simplified)
  static getDutyRate(hsCode: string, countryCode: string): number {
    // In real implementation, this would query a tariff database
    // This is a simplified example
    const dutyRates: Map<string, number> = new Map([
      ["8471", 0],        // Computers - duty free in many countries
      ["6203", 16.5],     // Men's apparel
      ["9403", 0],        // Furniture
      ["8517", 0],        // Telecommunications equipment
      ["3004", 0],        // Pharmaceutical products
      ["8528", 5.0],      // Monitors and TVs
    ]);

    const primaryCode = hsCode.substring(0, 4);
    return dutyRates.get(primaryCode) || 5.0;  // Default 5%
  }

  // Generate customs declaration
  createDeclaration(shipment: CargoShipment, packages: CustomsPackage[]): CustomsDeclaration {
    const declarationNumber = this.generateDeclarationNumber();

    return {
      declarationNumber,
      declarationType: "IMPORT",
      declarationDate: new Date(),

      awbNumber: shipment.awbNumber,
      commercialInvoiceNumber: `INV-${Date.now()}`,

      exporter: {
        name: shipment.shipper.name,
        taxId: "CN-TAX-123456",
        address: shipment.shipper.address
      },

      importer: {
        name: shipment.consignee.name,
        taxId: "US-EIN-987654",
        address: shipment.consignee.address
      },

      declarant: {
        name: "Global Customs Brokers Inc",
        taxId: "US-EIN-555555",
        customsBrokerLicense: "CB-12345",
        address: {
          street: "200 Broker Street",
          city: "Los Angeles",
          state: "CA",
          postalCode: "90001",
          country: "US"
        }
      },

      packages,
      totalValue: { amount: packages.reduce((sum, p) => sum + p.totalValue.amount, 0), currency: "USD" },
      totalNetWeight: shipment.weight,
      totalGrossWeight: { value: shipment.weight.value * 1.1, unit: shipment.weight.unit },

      incoterms: shipment.incoterms,
      currencyOfPayment: "USD",
      countryOfOrigin: "CN",
      countryOfExport: "CN",
      countryOfDestination: "US",

      harmonizedCodes: packages.map(p => p.harmonizedCode),

      estimatedDuties: { amount: 0, currency: "USD" },
      estimatedTaxes: { amount: 0, currency: "USD" },

      attachments: []
    };
  }

  private generateDeclarationNumber(): string {
    const timestamp = Date.now();
    const random = Math.floor(Math.random() * 10000);
    return `CD-${timestamp}-${random}`;
  }
}

Security Regulations

Aviation Security Requirements

Air cargo security has intensified significantly since 2010. Key security programs include:

  • Known Shipper Programs: Pre-vetted shippers with established security protocols
  • Regulated Agent Programs: Freight forwarders authorized to handle secure cargo
  • Account Consignee: Pre-cleared importers for expedited clearance
  • Cargo Screening: 100% screening requirement for passenger aircraft
  • Secure Supply Chain: Chain of custody documentation from origin to destination

Key Takeaways

  • IATA establishes global standards for air cargo including AWB formats, dangerous goods handling, and electronic data interchange
  • AWB numbers follow XXX-XXXXXXXX format with mod 7 check digit validation for data integrity
  • Dangerous goods are classified into 9 classes, each with specific packaging, labeling, and documentation requirements
  • Lithium batteries (Class 9) are among the most commonly shipped dangerous goods with strict regulations
  • Customs declarations require harmonized codes, proper valuation, and complete documentation for duty assessment
  • Incoterms define buyer/seller responsibilities for freight costs, insurance, and customs clearance
  • Security regulations require 100% cargo screening for passenger aircraft and established chain of custody
  • TypeScript implementations enable automated validation of AWB numbers, HS codes, and customs calculations

  • CycloneDX 1.5 / SPDX 2.3 — software bill of materials
  • Sigstore (DSSE envelope, Rekor transparency log) — signature anchoring
  • in-toto Attestation Framework 1.0 — build provenance
  • IETF RFC 9700 — OAuth 2.1
  • IETF RFC 9457 — Problem Details for HTTP APIs
  • IETF RFC 8615 — well-known URIs

Practical Implementation Checklist

  • Manifest integrity — every published manifest is signed via Sigstore DSSE. Tampering is detectable without trusting an intermediary. Updates are issued as new manifests; the prior manifest is preserved with a supersede marker.
  • Retention — keep signed evidence for at least seven years to satisfy ISO/IEC 17065:2012 audit expectations. Some regulators require longer retention; in that case the deployment policy MUST extend the window rather than relying on the defaults.
  • Versioning — follow Semantic Versioning 2.0.0. Major version bumps require at least a 90-day overlap with the prior major version on every WIA-published reference implementation.
  • Negative-result reporting — a skipped indicator without recorded justification is treated as a failed indicator at audit time.
  • Key rotation — manifest signing keys are rotated per the deployment policy. The previous key is preserved in a published key history so that auditors can verify historical manifests offline.

Reference Topologies

Implementations declare their operational topology in the manifest: single-tenant edge (one runtime per organization), multi-tenant gateway (shared runtime with header-based isolation), federated mesh (multiple runtimes peer to one another and publish manifests to a directory service), or air-gapped batch (signed evidence package transported out-of-band). Each topology ships with reference test vectors so that implementations can validate their deployment matches the canonical pattern.

Open Governance

Issues, errata, and proposals are tracked at github.com/WIA-Official/wia-standards/issues with the appropriate label. The WIA Standards working group reviews open issues at the start of every minor release cycle and publishes the resulting decision log alongside the release notes. Errata are issued as patch releases; new normative requirements trigger minor bumps; backwards-incompatible changes trigger major bumps.
