Electronic Air Waybill (e-AWB)

E-AWB Implementation

The e-AWB eliminates paper Air Waybill documents, transmitting shipment data electronically between all parties. IATA's e-AWB program has achieved over 75% penetration globally, with benefits including faster processing, reduced errors, and environmental sustainability.

Aspect Paper AWB Electronic AWB Benefits
Processing Time 30-45 minutes 5-10 minutes 70% faster
Error Rate 15-20% 2-3% 85% reduction
Cost per Shipment $25-30 $5-8 $20+ savings
Environmental Impact High (paper, printing) Minimal Sustainable
Tracking Visibility Limited, manual updates Real-time, automated Full transparency
Data Quality Variable, manual entry High, validated Better decisions

E-AWB Data Requirements

E-AWB implementations must transmit standardized data elements via IATA Cargo-XML or Cargo-IMP messaging:

Mandatory E-AWB Data Elements:

  • AWB Number: Unique identifier (XXX-XXXXXXXX)
  • Shipper/Consignee: Complete party information with addresses
  • Origin/Destination: IATA airport codes
  • Routing: Flight details and transfer points
  • Pieces/Weight: Total pieces, gross/chargeable weight
  • Commodity Description: Nature of goods
  • Special Handling: SHC codes (PER, VAL, etc.)
  • Accounting Information: Charges and payment terms
  • Customs Data: HS codes, value declaration

class EAWBProcessor {
  // Generate FWB (Freight Waybill) message
  generateFWB(shipment: CargoShipment): EAWBMessage {
    return {
      messageType: "FWB",
      version: "16",
      timestamp: new Date(),

            city: shipment.shipper.address.city,
            state: shipment.shipper.address.state,

  // Generate FSU (Freight Status Update) message
  generateFSU(
    awbNumber: string,
    statusCode: string,
    location: string,
    pieces: number,
    weight: Weight
  ): EAWBMessage {
    return {
      messageType: "FSU",
      version: "16",
      timestamp: new Date(),
      sender: "HANDLER",
      recipient: "ALL",

      awbData: {
        prefix: awbNumber.substring(0, 3),
        serialNumber: awbNumber.substring(4, 12),
        origin: "",
        destination: "",
        pieces,
        weight
      },

      routing: [],

      shipmentDetails: {
        shipper: { name: "", address: { street: "", city: "", country: "" } },
        consignee: { name: "", address: { street: "", city: "", country: "" } },
        description: "",
        specialHandling: [],
        harmCodes: []
      },

      statusUpdates: [{
        statusCode,
        location,
        timestamp: new Date(),
        pieces,
        weight
      }]
    };
  }

  // Validate E-AWB message
  validateMessage(message: EAWBMessage): {
    valid: boolean;
    errors: string[];
  } {
    const errors: string[] = [];

    // Validate AWB number format
    const awbNumber = `${message.awbData.prefix}-${message.awbData.serialNumber}`;
    if (!AWBManager.validateAWB(awbNumber)) {
      errors.push("Invalid AWB number format");
    }

    // Validate airport codes
    const validAirports = ["JFK", "LAX", "ORD", "ATL", "DFW", "LHR", "CDG", "FRA",
                           "HKG", "SIN", "NRT", "ICN", "PVG", "DXB", "AMS"];

    if (!validAirports.includes(message.awbData.origin)) {
      errors.push(`Invalid origin airport: ${message.awbData.origin}`);
    }

    if (!validAirports.includes(message.awbData.destination)) {
      errors.push(`Invalid destination airport: ${message.awbData.destination}`);
    }

    // Validate weight
    if (message.awbData.weight.value <= 0) {
      errors.push("Weight must be greater than zero");
    }

    // Validate pieces
    if (message.awbData.pieces <= 0) {
      errors.push("Pieces must be greater than zero");
    }

    return {
      valid: errors.length === 0,
      errors
    };
  }

// IoT Tracking Manager
class IoTTrackingManager {
  private devices: Map<string, IoTDevice>;
  private readings: Map<string, SensorReading[]>;
  private alerts: Map<string, Alert>;

  constructor() {
    this.devices = new Map();
    this.readings = new Map();
    this.alerts = new Map();
  }

  // Register IoT device
  registerDevice(device: IoTDevice): void {
    this.devices.set(device.deviceId, device);
    this.readings.set(device.deviceId, []);
    console.log(`Device ${device.deviceId} registered`);
  }

  // Assign device to shipment
  assignDevice(deviceId: string, awbNumber: string): void {
    const device = this.devices.get(deviceId);

    if (!device) {
      throw new Error(`Device ${deviceId} not found`);
    }

    device.assignedToAWB = awbNumber;
    device.assignmentDate = new Date();
    device.isActive = true;

    console.log(`Device ${deviceId} assigned to AWB ${awbNumber}`);
  }

  // Process sensor reading
  processSensorReading(reading: SensorReading): void {
    // Store reading
    const deviceReadings = this.readings.get(reading.deviceId) || [];
    deviceReadings.push(reading);
    this.readings.set(reading.deviceId, deviceReadings);

    // Update device last report time
    const device = this.devices.get(reading.deviceId);
    if (device) {
      device.lastReportTime = reading.timestamp;
      device.batteryLevel = reading.batteryLevel;
      device.signalStrength = reading.signalStrength;
    }

    // Check thresholds
    this.checkThresholds(reading);
  }

  // Check if reading exceeds thresholds
  private checkThresholds(reading: SensorReading): void {
    const device = this.devices.get(reading.deviceId);

    if (!device) return;

    for (const threshold of device.alertThresholds) {
      let value: number | undefined;
      let exceeded = false;

      switch (threshold.parameter) {
        case "TEMPERATURE":
          value = reading.temperature;
          break;
        case "HUMIDITY":
          value = reading.humidity;
          break;
        case "SHOCK":
          value = reading.shock;
          break;
        case "LIGHT":
          value = reading.light;
          break;
        case "TILT":
          value = reading.tilt;
          break;
      }

      if (value !== undefined) {
        if (threshold.minValue !== undefined && value < threshold.minValue) {
          exceeded = true;
        }
        if (threshold.maxValue !== undefined && value > threshold.maxValue) {
          exceeded = true;
        }

        if (exceeded) {
          this.createAlert({
            alertId: `ALERT-${Date.now()}-${Math.random().toString(36).substr(2, 9)}`,
            deviceId: reading.deviceId,
            awbNumber: device.assignedToAWB,
            alertType: "THRESHOLD_EXCEEDED",
            severity: "WARNING",
            timestamp: reading.timestamp,
            parameter: threshold.parameter,
            value,
            threshold: threshold.maxValue || threshold.minValue,
            message: `${threshold.parameter} ${value}${threshold.unit} exceeds threshold`,
            acknowledged: false
          });
        }
      }
    }

    // Check battery level
    if (reading.batteryLevel < 20) {
      this.createAlert({
        alertId: `ALERT-${Date.now()}-${Math.random().toString(36).substr(2, 9)}`,
        deviceId: reading.deviceId,
        awbNumber: device.assignedToAWB,
        alertType: "BATTERY_LOW",
        severity: "WARNING",
        timestamp: reading.timestamp,
        message: `Device battery low: ${reading.batteryLevel}%`,
        acknowledged: false
      });
    }
  }

  // Create alert
  private createAlert(alert: Alert): void {
    this.alerts.set(alert.alertId, alert);
    console.log(`⚠️  ALERT: ${alert.message} (AWB: ${alert.awbNumber})`);
  }

  // Get shipment tracking history
  getShipmentTracking(awbNumber: string): {
    currentLocation?: GPSCoordinate;
    readings: SensorReading[];
    alerts: Alert[];
  } {
    // Find device assigned to this AWB
    const device = Array.from(this.devices.values()).find(d => d.assignedToAWB === awbNumber);

    if (!device) {
      return { readings: [], alerts: [] };
    }

    const readings = this.readings.get(device.deviceId) || [];
    const shipmentAlerts = Array.from(this.alerts.values())
      .filter(a => a.awbNumber === awbNumber);

    // Get most recent location
    const latestReading = readings[readings.length - 1];
    const currentLocation = latestReading?.location;

    return {
      currentLocation,
      readings,
      alerts: shipmentAlerts
    };
  }

  // Calculate temperature excursion duration
  calculateExcursionDuration(
    deviceId: string,
    minTemp: number,
    maxTemp: number
  ): number {
    const readings = this.readings.get(deviceId) || [];
    let excursionMinutes = 0;

    for (let i = 1; i < readings.length; i++) {
      const temp = readings[i].temperature;
      if (temp !== undefined && (temp < minTemp || temp > maxTemp)) {
        const timeDiff = readings[i].timestamp.getTime() - readings[i - 1].timestamp.getTime();
        excursionMinutes += timeDiff / (1000 * 60);
      }
    }

    return Math.round(excursionMinutes);
  }
}

Reference Standards and Implementation Notes

This standard is designed to interoperate with established international frameworks. Implementations SHOULD treat the following as the single source of truth and use this PHASE document as the interoperability layer that maps existing data into a machine-readable, signed, and auditable shape.

  • OpenAPI Specification 3.1 — REST API definition
  • JSON Schema 2020-12 — schema validation
  • ISO/IEC 27001:2022 — information security management
  • ISO/IEC 27018:2019 — PII protection in public clouds
  • ISO/IEC 17065:2012 — product certification bodies
  • CycloneDX 1.5 / SPDX 2.3 — software bill of materials
  • Sigstore (DSSE envelope, Rekor transparency log) — signature anchoring
  • in-toto Attestation Framework 1.0 — build provenance
  • IETF RFC 9700 — OAuth 2.1
  • IETF RFC 9457 — Problem Details for HTTP APIs
  • IETF RFC 8615 — well-known URIs

Practical Implementation Checklist

  • Manifest integrity — every published manifest is signed via Sigstore DSSE. Tampering is detectable without trusting an intermediary. Updates are issued as new manifests; the prior manifest is preserved with a supersede marker.
  • Retention — keep signed evidence for at least seven years to satisfy ISO/IEC 17065:2012 audit expectations. Some regulators require longer retention; in that case the deployment policy MUST extend the window rather than relying on the defaults.
  • Versioning — follow Semantic Versioning 2.0.0. Major version bumps require at least a 90-day overlap with the prior major version on every WIA-published reference implementation.
  • Negative-result reporting — a skipped indicator without recorded justification is treated as a failed indicator at audit time.
  • Key rotation — manifest signing keys are rotated per the deployment policy. The previous key is preserved in a published key history so that auditors can verify historical manifests offline.

Reference Topologies

Implementations declare their operational topology in the manifest: single-tenant edge (one runtime per organization), multi-tenant gateway (shared runtime with header-based isolation), federated mesh (multiple runtimes peer to one another and publish manifests to a directory service), or air-gapped batch (signed evidence package transported out-of-band). Each topology ships with reference test vectors so that implementations can validate their deployment matches the canonical pattern.

Open Governance

Issues, errata, and proposals are tracked at github.com/WIA-Official/wia-standards/issues with the appropriate label. The WIA Standards working group reviews open issues at the start of every minor release cycle and publishes the resulting decision log alongside the release notes. Errata are issued as patch releases; new normative requirements trigger minor bumps; backwards-incompatible changes trigger major bumps.
