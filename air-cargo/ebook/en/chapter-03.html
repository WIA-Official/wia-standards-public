Cargo Handling

ULD Operations, Warehouse Management, and Ground Operations

Introduction to Cargo Handling

Cargo handling encompasses all ground operations from receiving cargo at the origin terminal to delivering it at the destination. Efficient handling is critical for maintaining cargo integrity, meeting service commitments, and optimizing aircraft utilization. Modern cargo terminals process thousands of tons daily, requiring sophisticated systems for tracking, storage, and movement.

The cargo handling process involves multiple specialized operations: acceptance and documentation, screening and security clearance, storage and warehousing, build-up into Unit Load Devices (ULDs) or bulk loading, aircraft loading/unloading, break-down, and delivery preparation. Each step must be coordinated precisely to avoid delays and damage.

Unit Load Devices (ULD)

ULD Types and Specifications

ULDs are containers and pallets designed to fit specific aircraft types. They enable efficient loading, maximize aircraft space utilization, and protect cargo during transport. The IATA ULD Technical Manual defines over 100 different ULD types.

ULD Type Description Internal Dimensions Max Weight Compatible Aircraft
AKE Contoured container 156 x 153 x 163 cm 1,588 kg B737, A320, B757
LD3 Contoured container 156 x 153 x 163 cm 1,588 kg B777, B787, A330
LD7 Refrigerated container 317 x 153 x 163 cm 4,536 kg Widebody aircraft
AMA Main deck container 244 x 244 x 244 cm 6,804 kg B747F, B777F
AMJ Main deck container 244 x 244 x 318 cm 6,804 kg B747F, MD-11F
PLA Pallet with net 224 x 318 x 244 cm 4,626 kg Widebody aircraft
PMC Pallet with net 244 x 318 x 244 cm 6,804 kg B747F, B777F
RKN Cool Dolly (temp controlled) 156 x 153 x 163 cm 1,588 kg Narrow body

ULD Identification and Tracking

Each ULD has a unique identifier consisting of three components:

ULD Number Structure: ABC 12345 XY

  • ABC: ULD type code (e.g., AKE, LD3, PMC)
  • 12345: Serial number (up to 5 digits)
  • XY: Owner code (2 letters representing airline or lessor)

Example: AKE 98765 DL = Delta Airlines contoured container #98765

interface ULD {

// ULD Management System
class ULDManager {
  private ulds: Map<string, ULD>;
  private uldTypes: Map<string, ULDType>;
  private buildUps: Map<string, ULDBuildUp>;

  constructor() {
    this.ulds = new Map();
    this.uldTypes = new Map();
    this.buildUps = new Map();
    this.initializeULDTypes();
  }

  // Initialize standard ULD types
  private initializeULDTypes(): void {
    const uldTypes: ULDType[] = [
      {
        code: "AKE",
        description: "Contoured Container",
        category: "CONTAINER",
        isContoured: true,
        maxWeight: { value: 1588, unit: "KG" },
        internalDimensions: { length: 156, width: 153, height: 163, unit: "CM" },
        compatibleAircraft: ["B737", "A320", "B757"]
      },
      {
        code: "LD3",
        description: "LD3 Contoured Container",
        category: "CONTAINER",
        isContoured: true,
        maxWeight: { value: 1588, unit: "KG" },
        internalDimensions: { length: 156, width: 153, height: 163, unit: "CM" },
        compatibleAircraft: ["B777", "B787", "A330", "A350"]
      },
      {
        code: "AMA",
        description: "Main Deck Container",
        category: "CONTAINER",
        isContoured: false,
        maxWeight: { value: 6804, unit: "KG" },
        internalDimensions: { length: 244, width: 244, height: 244, unit: "CM" },
        compatibleAircraft: ["B747F", "B777F", "B767F"]
      },
      {
        code: "PMC",
        description: "Main Deck Pallet",
        category: "PALLET",
        isContoured: false,
        maxWeight: { value: 6804, unit: "KG" },
        internalDimensions: { length: 244, width: 318, height: 244, unit: "CM" },
        compatibleAircraft: ["B747F", "B777F"]
      }
    ];

    uldTypes.forEach(type => {
      this.uldTypes.set(type.code, type);
    });
  }

  // Register new ULD
  registerULD(uldData: Omit<ULD, "movements" | "damageReports">): ULD {
    const uld: ULD = {
      ...uldData,
      movements: [{
        timestamp: new Date(),
        location: uldData.currentLocation,
        movementType: "RECEIVED",
        handledBy: "SYSTEM"
      }],
      damageReports: []
    };

    this.ulds.set(uld.uldNumber, uld);
    console.log(`ULD ${uld.uldNumber} registered successfully`);

    return uld;
  }

  // Update ULD location
  updateULDLocation(
    uldNumber: string,
    newLocation: string,
    movementType: ULDMovement["movementType"],
    handledBy: string,
    flightNumber?: string
  ): void {
    const uld = this.ulds.get(uldNumber);

    if (!uld) {
      throw new Error(`ULD ${uldNumber} not found`);
    }

    uld.currentLocation = newLocation;
    uld.movements.push({
      timestamp: new Date(),
      location: newLocation,
      movementType,
      flightNumber,
      handledBy
    });

    console.log(`ULD ${uldNumber} moved to ${newLocation}`);
  }

  // Build up ULD
  buildUpULD(buildUpData: ULDBuildUp): void {
    const uld = this.ulds.get(buildUpData.uldNumber);

    if (!uld) {
      throw new Error(`ULD ${buildUpData.uldNumber} not found`);
    }

    // Validate weight
    if (buildUpData.totalWeight.value > uld.maxGrossWeight.value) {
      throw new Error(`Build-up weight exceeds ULD maximum capacity`);
    }

    // Update ULD status
    uld.currentStatus = ULDStatus.BUILT_UP;
    uld.assignedAWB = buildUpData.awbNumbers;

    // Record movement
    this.updateULDLocation(
      buildUpData.uldNumber,
      uld.currentLocation,
      "BUILT_UP",
      buildUpData.builtBy
    );

    // Store build-up record
    this.buildUps.set(buildUpData.uldNumber, buildUpData);

    console.log(`ULD ${buildUpData.uldNumber} built up with ${buildUpData.totalPieces} pieces`);
  }

  // Calculate ULD utilization
  calculateUtilization(uldNumber: string): {
    weightUtilization: number;
    volumeUtilization: number;
  } {
    const buildUp = this.buildUps.get(uldNumber);
    const uld = this.ulds.get(uldNumber);

    if (!buildUp || !uld) {
      return { weightUtilization: 0, volumeUtilization: 0 };
    }

    const weightUtilization = (buildUp.totalWeight.value / uld.maxGrossWeight.value) * 100;

    // Simplified volume calculation
    const totalCargoVolume = buildUp.cargoItems.reduce((sum, item) => {
      const itemVolume = item.dimensions.length * item.dimensions.width * item.dimensions.height;
      return sum + (itemVolume * item.pieces);
    }, 0);

    const uldVolume = uld.internalDimensions.length *
                      uld.internalDimensions.width *
                      uld.internalDimensions.height;

    const volumeUtilization = (totalCargoVolume / uldVolume) * 100;

    return {
      weightUtilization: Math.round(weightUtilization * 100) / 100,
      volumeUtilization: Math.round(volumeUtilization * 100) / 100
    };
  }

  // Get available ULDs by type
  getAvailableULDs(uldTypeCode: string, location: string): ULD[] {
    return Array.from(this.ulds.values()).filter(uld =>
      uld.uldType.code === uldTypeCode &&
      uld.currentLocation === location &&
      uld.currentStatus === ULDStatus.AVAILABLE &&
      uld.condition === "SERVICEABLE"
    );
  }

  // Report ULD damage
  reportDamage(
    uldNumber: string,
    damageReport: Omit<DamageReport, "reportDate" | "repaired">
  ): void {
    const uld = this.ulds.get(uldNumber);

    if (!uld) {
      throw new Error(`ULD ${uldNumber} not found`);
    }

    const report: DamageReport = {
      ...damageReport,
      reportDate: new Date(),
      repaired: false
    };

    uld.damageReports.push(report);

    if (report.severity === "MAJOR" || report.severity === "CRITICAL") {
      uld.condition = "DAMAGED";
      uld.currentStatus = ULDStatus.UNDER_REPAIR;
    }

    console.log(`Damage reported for ULD ${uldNumber}: ${damageReport.damageType}`);
  }

  // Parse ULD number
  static parseULDNumber(uldNumber: string): {
    typeCode: string;
    serialNumber: string;
    ownerCode: string;
  } | null {
    // Format: AAABBBBBCC or AAA-BBBBB-CC
    const cleaned = uldNumber.replace(/-/g, "");

    if (cleaned.length < 8) {
      return null;
    }

    return {
      typeCode: cleaned.substring(0, 3),
      serialNumber: cleaned.substring(3, cleaned.length - 2),
      ownerCode: cleaned.substring(cleaned.length - 2)
    };
  }
}

Warehouse Operations

Cargo Terminal Layout

Modern cargo terminals are designed for efficient flow from landside to airside. Key functional areas include:

  • Acceptance Area: Cargo receipt, documentation verification, and initial screening
  • Export Storage: Temperature-controlled and general cargo storage zones
  • Build-up Area: ULD assembly and pallet preparation stations
  • Screening Zone: X-ray and physical inspection facilities
  • Airside Transfer: Dolly staging and aircraft loading positions
  • Import Break-down: ULD disassembly and cargo sorting
  • Import Storage: Customs clearance and delivery preparation
  • Special Cargo: Dedicated areas for perishables, valuables, and live animals

Loading and Build-up Procedures

Load Planning Process

Proper load planning ensures aircraft balance, maximizes capacity utilization, and maintains cargo integrity:

Load Planning Steps:

  1. Flight Analysis: Review aircraft type, weight limits, and compartment configurations
  2. Cargo Prioritization: Sort by destination, transfer connections, and special handling
  3. Weight Distribution: Calculate center of gravity and load distribution
  4. ULD Selection: Choose appropriate ULD types for cargo and aircraft
  5. Build-up Assignment: Assign cargo to specific ULDs with proper stacking
  6. Loading Sequence: Determine ULD loading order for aircraft balance
  7. Documentation: Generate load plan, ULD tags, and loading instructions
  8. Verification: Confirm actual weights match planned weights

Best Practices for Cargo Build-up

  • Heavy items on bottom: Place dense cargo at ULD base for stability
  • Distribute weight evenly: Avoid concentrated loads on one side
  • Protect corners: Use edge protectors and cardboard to prevent damage
  • Secure properly: Use nets, straps, and void fillers to prevent shifting
  • Label clearly: Mark fragile items, orientation arrows, and special handling
  • Segregate incompatibles: Keep dangerous goods separate per regulations
  • Optimize space: Fill voids with soft cargo or dunnage
  • Document thoroughly: Record all AWBs, piece counts, and weights

Key Takeaways

  • Unit Load Devices (ULDs) enable efficient containerization with over 850,000 units in global circulation
  • ULD numbering follows a standardized format: Type Code + Serial Number + Owner Code (e.g., AKE12345DL)
  • Major ULD types include AKE, LD3 for narrow-body aircraft and AMA, PMC for wide-body freighters
  • Warehouse operations require zone-based storage, real-time inventory tracking, and integration with carrier systems
  • Ground support equipment includes main deck loaders, lower deck loaders, dollies, and specialized handling equipment
  • Load planning must consider aircraft balance, weight distribution, and cargo compatibility
  • Proper build-up procedures include weight distribution, securing methods, and thorough documentation
  • TypeScript implementations enable ULD tracking, utilization calculations, and automated build-up management

Reference Standards and Implementation Notes

This standard is designed to interoperate with established international frameworks. Implementations SHOULD treat the following as the single source of truth and use this PHASE document as the interoperability layer that maps existing data into a machine-readable, signed, and auditable shape.

Practical Implementation Checklist

  • Manifest integrity — every published manifest is signed via Sigstore DSSE. Tampering is detectable without trusting an intermediary. Updates are issued as new manifests; the prior manifest is preserved with a supersede marker.
  • Retention — keep signed evidence for at least seven years to satisfy ISO/IEC 17065:2012 audit expectations. Some regulators require longer retention; in that case the deployment policy MUST extend the window rather than relying on the defaults.
  • Versioning — follow Semantic Versioning 2.0.0. Major version bumps require at least a 90-day overlap with the prior major version on every WIA-published reference implementation.
  • Negative-result reporting — a skipped indicator without recorded justification is treated as a failed indicator at audit time.
  • Key rotation — manifest signing keys are rotated per the deployment policy. The previous key is preserved in a published key history so that auditors can verify historical manifests offline.

Reference Topologies

Implementations declare their operational topology in the manifest: single-tenant edge (one runtime per organization), multi-tenant gateway (shared runtime with header-based isolation), federated mesh (multiple runtimes peer to one another and publish manifests to a directory service), or air-gapped batch (signed evidence package transported out-of-band). Each topology ships with reference test vectors so that implementations can validate their deployment matches the canonical pattern.

Open Governance

Issues, errata, and proposals are tracked at github.com/WIA-Official/wia-standards/issues with the appropriate label. The WIA Standards working group reviews open issues at the start of every minor release cycle and publishes the resulting decision log alongside the release notes. Errata are issued as patch releases; new normative requirements trigger minor bumps; backwards-incompatible changes trigger major bumps.
