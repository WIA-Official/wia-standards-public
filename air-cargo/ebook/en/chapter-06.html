Cargo Screening, Known Shipper Programs, and Supply Chain Security

Introduction to Air Cargo Security

Air cargo security has become a critical priority following several high-profile incidents. Regulatory authorities worldwide have implemented comprehensive security programs requiring 100% screening of cargo on passenger aircraft, validation of supply chain participants, and strict chain of custody documentation.

The air cargo security framework encompasses physical screening, entity validation (Known Shipper, Regulated Agent programs), cybersecurity measures, and continuous monitoring of the supply chain. Compliance is mandatory, with severe penalties for violations including fines, criminal prosecution, and loss of operating authority.

Cargo Screening Requirements

Screening Methods and Technologies

Aviation security regulations require all cargo to be screened before loading on aircraft. Multiple screening methods are approved, each with different capabilities and limitations:

Screening Method Technology Throughput Detection Capability Limitations
X-ray Screening Dual-energy X-ray systems 200-300 pieces/hour Explosives, weapons, prohibited items Limited penetration for dense cargo
ETD (Explosive Trace Detection) Chemical analysis of swabs 30-50 packages/hour Explosive residue, chemical agents Labor-intensive, sampling only
EDS (Explosive Detection System) CT scanning technology 400-600 pieces/hour 3D imaging, high accuracy High cost, size limitations
Canine Teams Trained explosive detection dogs 100-150 packages/hour Wide range of explosives Requires handlers, shift limits
Physical Search Manual inspection 10-20 packages/hour Visual/tactile inspection Very slow, intrusive

Screening Exemptions and Alternatives

Certain cargo may qualify for screening exemptions or alternative procedures:

  • Known Shipper Program: Pre-vetted shippers with established security protocols
  • Secure Supply Chain: Cargo secured from origin throughout chain of custody
  • Account Consignee: Pre-cleared regular importers
  • Mail and Diplomatic Pouches: Handled under separate security frameworks
  • Transfer/Transit Cargo: May rely on origin screening if security standards met

⚠️ Critical Security Requirements:

  • 100% screening mandatory for all cargo on passenger aircraft
  • Cargo must be screened by TSA-approved methods or originated from Known Shipper
  • Chain of custody must be maintained from screening to aircraft loading
  • Screened cargo must be protected in secure area until loading
  • Security incidents must be reported immediately to authorities
  • Screening records must be retained for minimum 180 days
  • Personnel performing screening must be properly trained and certified

Known Shipper and Regulated Agent Programs

Known Shipper Program Overview

The Known Shipper program allows pre-validated shippers to tender cargo without additional screening, provided they maintain approved security protocols. This program significantly streamlines operations for established shippers.

Program Aspect Requirements Validation Process Maintenance
Application Business verification, security plan submission Background checks, site inspection Annual revalidation required
Security Procedures Access control, employee screening, cargo protection TSA/CAA review and approval Continuous compliance monitoring
Employee Vetting Background checks for all cargo-handling personnel Criminal history, employment verification New hire screening, periodic rechecks
Facility Security Secured storage areas, access logs, CCTV Physical security assessment Regular security audits
Documentation Cargo manifest, shipper declaration, chain of custody Process review and approval Record retention, audit compliance

Regulated Agent Framework

Regulated Agents are freight forwarders and ground handlers authorized to accept cargo from Known Shippers and consolidate it for airline transport. They serve as critical links in the secure supply chain.

Regulated Agent Responsibilities:

  • Verify Known Shipper status before accepting cargo
  • Maintain secure storage and handling facilities
  • Implement access control and employee vetting programs
  • Conduct security screening when required
  • Document chain of custody for all cargo
  • Report security incidents to authorities and airlines
  • Undergo regular security audits and inspections
  • Maintain security training programs for all employees

interface SecurityEntity {

  securityProgram: SecurityProgram;

// Security Compliance Manager
class SecurityComplianceManager {
  private entities: Map<string, SecurityEntity>;
  private screeningRecords: Map<string, ScreeningRecord[]>;
  private incidents: Map<string, SecurityIncident>;

  constructor() {
    this.entities = new Map();
    this.screeningRecords = new Map();
    this.incidents = new Map();
  }

  // Register security entity
  registerEntity(entity: SecurityEntity): void {
    this.entities.set(entity.entityId, entity);
    console.log(`Entity ${entity.companyName} registered as ${entity.entityType}`);
  }

  // Validate Known Shipper status
  validateKnownShipper(entityId: string): {
    valid: boolean;
    status: string;
    expiryDate?: Date;
    message: string;
  } {
    const entity = this.entities.get(entityId);

    if (!entity) {
      return {
        valid: false,
        status: "NOT_FOUND",
        message: "Entity not found in database"
      };
    }

    if (entity.entityType !== "KNOWN_SHIPPER") {
      return {
        valid: false,
        status: "WRONG_TYPE",
        message: "Entity is not a Known Shipper"
      };
    }

    if (entity.approvalStatus !== "APPROVED") {
      return {
        valid: false,
        status: entity.approvalStatus,
        message: `Known Shipper status is ${entity.approvalStatus}`
      };
    }

    const now = new Date();
    if (entity.expiryDate < now) {
      return {
        valid: false,
        status: "EXPIRED",
        expiryDate: entity.expiryDate,
        message: "Known Shipper approval has expired"
      };
    }

    return {
      valid: true,
      status: "APPROVED",
      expiryDate: entity.expiryDate,
      message: "Known Shipper validation successful"
    };
  }

  // Record cargo screening
  recordScreening(record: ScreeningRecord): void {
    const awbRecords = this.screeningRecords.get(record.awbNumber) || [];
    awbRecords.push(record);
    this.screeningRecords.set(record.awbNumber, awbRecords);

    console.log(`Screening recorded for AWB ${record.awbNumber}: ${record.screeningResult}`);

    // Alert if suspicious or prohibited item found
    if (record.screeningResult === "SUSPICIOUS" || record.screeningResult === "PROHIBITED_ITEM") {
      this.createSecurityIncident({
        incidentId: `INC-${Date.now()}`,
        incidentDate: record.screeningDate,
        reportedDate: new Date(),
        incidentType: "SUSPICIOUS_ITEM",
        severity: record.screeningResult === "PROHIBITED_ITEM" ? "CRITICAL" : "HIGH",
        description: `Screening alert for AWB ${record.awbNumber}: ${record.findings}`,
        location: record.screeningLocation,
        reportedBy: record.screenedBy,
        investigationStatus: "REPORTED",
        authoritiesNotified: ["TSA", "LOCAL_POLICE"]
      });
    }
  }

  // Verify screening status for shipment
  verifyScreeningStatus(awbNumber: string): {
    screened: boolean;
    method?: string;
    screenedDate?: Date;
    cleared: boolean;
  } {
    const records = this.screeningRecords.get(awbNumber);

    if (!records || records.length === 0) {
      return {
        screened: false,
        cleared: false
      };
    }

    const latestRecord = records[records.length - 1];

    return {
      screened: true,
      method: latestRecord.screeningMethod,
      screenedDate: latestRecord.screeningDate,
      cleared: latestRecord.screeningResult === "CLEARED"
    };
  }

  // Create security incident
  createSecurityIncident(incident: SecurityIncident): void {
    this.incidents.set(incident.incidentId, incident);

    console.log(`🚨 SECURITY INCIDENT: ${incident.incidentType}`);
    console.log(`   Severity: ${incident.severity}`);
    console.log(`   Location: ${incident.location}`);
    console.log(`   Description: ${incident.description}`);

    if (incident.authoritiesNotified.length > 0) {
      console.log(`   Authorities notified: ${incident.authoritiesNotified.join(", ")}`);
    }
  }

  // Conduct security audit
  conductAudit(
    entityId: string,
    auditType: SecurityAudit["auditType"],
    findings: AuditFinding[]
  ): SecurityAudit {
    const entity = this.entities.get(entityId);

    if (!entity) {
      throw new Error(`Entity ${entityId} not found`);
    }

    // Determine overall rating based on findings
    const hasCritical = findings.some(f => f.severity === "CRITICAL");
    const hasMajor = findings.some(f => f.severity === "MAJOR");

    let overallRating: SecurityAudit["overallRating"];
    if (hasCritical) {
      overallRating = "NON_COMPLIANT";
    } else if (hasMajor) {
      overallRating = "MAJOR_ISSUES";
    } else if (findings.length > 0) {
      overallRating = "MINOR_ISSUES";
    } else {
      overallRating = "COMPLIANT";
    }

    const audit: SecurityAudit = {
      auditId: `AUDIT-${Date.now()}`,
      auditDate: new Date(),
      auditType,
      auditor: "SECURITY_TEAM",
      findings,
      overallRating,
      correctiveActions: []
    };

    entity.audits.push(audit);

    console.log(`Security audit completed for ${entity.companyName}`);
    console.log(`Rating: ${overallRating}, Findings: ${findings.length}`);

    // Suspend entity if non-compliant
    if (overallRating === "NON_COMPLIANT") {
      entity.approvalStatus = "SUSPENDED";
      console.log(`⚠️  Entity ${entity.companyName} SUSPENDED due to non-compliance`);
    }

    return audit;
  }

  // Get expiring approvals (within 30 days)
  getExpiringApprovals(daysAhead: number = 30): SecurityEntity[] {
    const cutoffDate = new Date();
    cutoffDate.setDate(cutoffDate.getDate() + daysAhead);

    return Array.from(this.entities.values()).filter(entity =>
      entity.approvalStatus === "APPROVED" &&
      entity.expiryDate <= cutoffDate &&
      entity.expiryDate >= new Date()
    );
  }
}

Cybersecurity Measures

Data Protection and System Security

Air cargo systems process sensitive commercial and personal information requiring robust cybersecurity measures:

  • Data Encryption: TLS 1.3 for data in transit, AES-256 for data at rest
  • Access Control: Role-based access control (RBAC) with least privilege principle
  • Authentication: Multi-factor authentication for all system access
  • Audit Logging: Comprehensive logging of all system access and data modifications
  • Vulnerability Management: Regular security assessments and penetration testing
  • Incident Response: 24/7 security operations center with incident response plan
  • Data Privacy: GDPR, CCPA compliance for personal data handling

Key Takeaways

  • 100% screening is mandatory for all cargo on passenger aircraft using approved methods
  • Known Shipper programs enable pre-vetted shippers to tender cargo with streamlined procedures
  • Regulated Agents serve as critical links in the secure supply chain, validating shippers and consolidating cargo
  • Chain of custody must be maintained from screening through aircraft loading with documented transfers
  • Security incidents must be reported immediately to authorities with potential fines up to $500,000
  • Regular security audits validate compliance with approval conditions and identify corrective actions
  • Cybersecurity measures protect sensitive cargo data including encryption, access control, and audit logging
  • TypeScript implementations enable automated compliance validation, screening tracking, and incident management

Reference Standards and Implementation Notes

This standard is designed to interoperate with established international frameworks. Implementations SHOULD treat the following as the single source of truth and use this PHASE document as the interoperability layer that maps existing data into a machine-readable, signed, and auditable shape.

  • OpenAPI Specification 3.1 — REST API definition
  • JSON Schema 2020-12 — schema validation
  • ISO/IEC 27001:2022 — information security management
  • ISO/IEC 27018:2019 — PII protection in public clouds
  • ISO/IEC 17065:2012 — product certification bodies
  • CycloneDX 1.5 / SPDX 2.3 — software bill of materials
  • Sigstore (DSSE envelope, Rekor transparency log) — signature anchoring
  • in-toto Attestation Framework 1.0 — build provenance
  • IETF RFC 9700 — OAuth 2.1
  • IETF RFC 9457 — Problem Details for HTTP APIs
  • IETF RFC 8615 — well-known URIs

Practical Implementation Checklist

  • Manifest integrity — every published manifest is signed via Sigstore DSSE. Tampering is detectable without trusting an intermediary. Updates are issued as new manifests; the prior manifest is preserved with a supersede marker.
  • Retention — keep signed evidence for at least seven years to satisfy ISO/IEC 17065:2012 audit expectations. Some regulators require longer retention; in that case the deployment policy MUST extend the window rather than relying on the defaults.
  • Versioning — follow Semantic Versioning 2.0.0. Major version bumps require at least a 90-day overlap with the prior major version on every WIA-published reference implementation.
  • Negative-result reporting — a skipped indicator without recorded justification is treated as a failed indicator at audit time.
  • Key rotation — manifest signing keys are rotated per the deployment policy. The previous key is preserved in a published key history so that auditors can verify historical manifests offline.

Reference Topologies

Implementations declare their operational topology in the manifest: single-tenant edge (one runtime per organization), multi-tenant gateway (shared runtime with header-based isolation), federated mesh (multiple runtimes peer to one another and publish manifests to a directory service), or air-gapped batch (signed evidence package transported out-of-band). Each topology ships with reference test vectors so that implementations can validate their deployment matches the canonical pattern.

Open Governance

Issues, errata, and proposals are tracked at github.com/WIA-Official/wia-standards/issues with the appropriate label. The WIA Standards working group reviews open issues at the start of every minor release cycle and publishes the resulting decision log alongside the release notes. Errata are issued as patch releases; new normative requirements trigger minor bumps; backwards-incompatible changes trigger major bumps.
