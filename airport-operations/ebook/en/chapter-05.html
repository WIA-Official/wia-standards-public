Chapter 5

API Specifications

This chapter provides comprehensive API specifications for airport operations systems, including Airport Operational Database (AODB) integration, flight information systems, resource management APIs, and real-time data exchange protocols that enable seamless interoperability across the airport ecosystem.

1. Introduction to Airport APIs

Modern airports operate through a complex web of interconnected systems that must share data in real-time to coordinate operations. APIs (Application Programming Interfaces) provide the standardized interfaces that enable this data exchange. The WIA Airport Operations standard defines a comprehensive API framework covering flight operations, resource management, baggage handling, passenger processing, and commercial operations.

Airport APIs must handle high transaction volumes (thousands of requests per second), provide sub-second response times, ensure data consistency across distributed systems, and maintain high availability (>99.9% uptime). They must also support multiple data formats (JSON, XML, IATA Type B), authentication protocols, and versioning strategies to accommodate legacy and modern systems.

2. Airport Operational Database (AODB) API

The AODB is the central data repository for all flight and operational information. It serves as the single source of truth for flight schedules, gate assignments, delays, cancellations, and resource allocations.

2.1 AODB Core Entities

Entity Description Update Frequency
Flight Flight schedule and status Real-time (1-60 seconds)
Aircraft Aircraft registration and type Per flight update
Gate Gate availability and assignment Real-time
Stand Aircraft parking position Real-time
Resource Equipment and personnel On allocation
Passenger Passenger manifests Per check-in event

2.2 Flight Information API

// WIA Airport Operations - Flight API Specification import { WIAAirportOperationsSDK } from '@wia/airport-operations'; // Initialize SDK const sdk = new WIAAirportOperationsSDK({ apiKey: process.env.WIA_API_KEY, baseUrl: 'https://api.airport.example', timeout: 5000, retryAttempts: 3 }); // Flight Information Interfaces interface Flight { id: string; flightNumber: string; airline: { iataCode: string; icaoCode: string; name: string; }; aircraft: { registration: string; type: string; // ICAO aircraft type category: 'A' | 'B' | 'C' | 'D' | 'E' | 'F'; }; schedule: { std: Date; // Scheduled Time of Departure sta: Date; // Scheduled Time of Arrival etd: Date; // Estimated Time of Departure eta: Date; // Estimated Time of Arrival atd?: Date; // Actual Time of Departure ata?: Date; // Actual Time of Arrival }; route: { origin: string; // IATA airport code destination: string; via?: string[]; // Technical stops }; terminal: string; gate?: string; stand?: string; carousel?: string; // For arrivals status: 'SCHEDULED' | 'BOARDING' | 'DEPARTED' | 'ARRIVED' | 'DELAYED' | 'CANCELLED'; delays: Delay[]; passengers: { booked: number; checkedIn: number; boarded: number; }; baggage: { expected: number; loaded: number; }; } interface Delay { code: string; // IATA delay code description: string; minutes: number; category: 'AIRLINE' | 'AIRPORT' | 'WEATHER' | 'ATC' | 'TECHNICAL'; timestamp: Date; } // API Methods class FlightAPI { // Get flight by flight number and date async getFlight( flightNumber: string, date: Date ): Promise { return await sdk.request({ method: 'GET', endpoint: `/flights/${flightNumber}`, params: { date: date.toISOString().split('T')[0] } }); } // Get all flights for a date range async getFlights( startDate: Date, endDate: Date, filters?: FlightFilters ): Promise { return await sdk.request({ method: 'GET', endpoint: '/flights', params: { startDate: startDate.toISOString(), endDate: endDate.toISOString(), ...filters } }); } // Update flight status async updateFlightStatus( flightId: string, update: FlightStatusUpdate ): Promise { return await sdk.request({ method: 'PATCH', endpoint: `/flights/${flightId}/status`, body: update }); } // Add flight delay async addDelay( flightId: string, delay: Delay ): Promise { return await sdk.request({ method: 'POST', endpoint: `/flights/${flightId}/delays`, body: delay }); } // Get real-time flight updates via WebSocket subscribeToFlightUpdates( callback: (flight: Flight) => void ): WebSocket { const ws = sdk.createWebSocket('/ws/flights'); ws.onmessage = (event) => { const flight: Flight = JSON.parse(event.data); callback(flight); }; return ws; } } // Example Usage async function monitorFlight(flightNumber: string) { const flightAPI = new FlightAPI(); // Get initial flight info const flight = await flightAPI.getFlight( flightNumber, new Date() ); console.log(`Flight ${flight.flightNumber}: ${flight.status}`); console.log(`Gate: ${flight.gate}, Stand: ${flight.stand}`); console.log(`ETD: ${flight.schedule.etd}`); // Subscribe to real-time updates flightAPI.subscribeToFlightUpdates((updatedFlight) => { if (updatedFlight.id === flight.id) { console.log(`Status changed: ${updatedFlight.status}`); if (updatedFlight.delays.length > 0) { const latestDelay = updatedFlight.delays[updatedFlight.delays.length - 1]; console.log(`Delay: ${latestDelay.minutes} min - ${latestDelay.description}`); } } }); }

3. Resource Management API

Resource management APIs handle allocation and tracking of airport resources including gates, stands, check-in counters, baggage carousels, and ground service equipment.

3.1 Gate Allocation API

// Gate Allocation API interface Gate { id: string; number: string; terminal: string; type: 'CONTACT' | 'REMOTE'; aircraftCategory: ('A' | 'B' | 'C' | 'D' | 'E' | 'F')[]; facilities: { jetway: boolean; doubleJetway: boolean; groundPower: boolean; preconditionedAir: boolean; }; status: 'AVAILABLE' | 'OCCUPIED' | 'RESERVED' | 'MAINTENANCE'; currentFlight?: string; nextAvailable?: Date; } interface GateAllocationRequest { flightId: string; aircraftType: string; passengerCount: number; sta: Date; std: Date; preferences?: { terminal?: string; proximity?: string[]; // Proximity to other gates requiresJetway?: boolean; }; constraints?: { wheelchairPassengers?: number; reducedMobility?: number; }; } interface GateAllocation { gateId: string; flightId: string; allocatedAt: Date; occupancyStart: Date; occupancyEnd: Date; score: number; // Allocation quality score alternatives: string[]; // Alternative gate options } class GateAllocationAPI { async allocateGate( request: GateAllocationRequest ): Promise { // Allocation algorithm considers: // 1. Aircraft size compatibility // 2. Time availability // 3. Terminal assignment // 4. Passenger requirements // 5. Operational efficiency const eligibleGates = await this.findEligibleGates(request); const scoredGates = await this.scoreGates(eligibleGates, request); const bestGate = scoredGates[0]; const allocation: GateAllocation = { gateId: bestGate.id, flightId: request.flightId, allocatedAt: new Date(), occupancyStart: request.sta, occupancyEnd: request.std, score: bestGate.score, alternatives: scoredGates.slice(1, 4).map(g => g.id) }; // Reserve gate await this.reserveGate(bestGate.id, allocation); return allocation; } private async findEligibleGates( request: GateAllocationRequest ): Promise { const gates = await sdk.request({ method: 'GET', endpoint: '/resources/gates', params: { status: 'AVAILABLE', terminal: request.preferences?.terminal } }); return gates.filter(gate => { // Check aircraft compatibility const aircraftCat = this.getAircraftCategory(request.aircraftType); if (!gate.aircraftCategory.includes(aircraftCat)) { return false; } // Check jetway requirement if (request.preferences?.requiresJetway && !gate.facilities.jetway) { return false; } // Check time availability if (gate.nextAvailable && gate.nextAvailable > request.sta) { return false; } return true; }); } private async scoreGates( gates: Gate[], request: GateAllocationRequest ): Promise> { const scored = gates.map(gate => { let score = 100; // Prefer gates in requested terminal if (request.preferences?.terminal === gate.terminal) { score += 20; } // Prefer contact gates for high passenger counts if (gate.type === 'CONTACT' && request.passengerCount > 100) { score += 15; } // Penalty for oversized gates (inefficient use) const aircraftCat = this.getAircraftCategory(request.aircraftType); const maxGateCat = gate.aircraftCategory[gate.aircraftCategory.length - 1]; if (maxGateCat > aircraftCat) { score -= 10; } return { ...gate, score }; }); return scored.sort((a, b) => b.score - a.score); } private getAircraftCategory(aircraftType: string): 'A' | 'B' | 'C' | 'D' | 'E' | 'F' { // ICAO aircraft category based on wingspan const categories: Record = { 'A320': 'C', 'B737': 'C', 'B777': 'E', 'B747': 'E', 'A380': 'F' }; return categories[aircraftType] || 'C'; } private async reserveGate( gateId: string, allocation: GateAllocation ): Promise { await sdk.request({ method: 'POST', endpoint: `/resources/gates/${gateId}/reserve`, body: allocation }); } }

6. Security and Authentication

Airport APIs implement multi-layered security including OAuth 2.0, API keys, IP whitelisting, and rate limiting to protect sensitive operational data.

Key Takeaways

  • AODB serves as the central data repository and single source of truth for flight operations
  • Flight APIs provide real-time access to schedules, status, delays, and resource assignments
  • Resource management APIs optimize allocation of gates, stands, and equipment
  • Baggage APIs enable end-to-end tracking and reconciliation
  • Multiple data formats (JSON, XML, Type B) ensure interoperability with diverse systems
  • WebSocket support enables real-time data streaming for dynamic operations

Reference Standards and Implementation Notes

This standard is designed to interoperate with established international frameworks. Implementations SHOULD treat the following as the single source of truth and use this PHASE document as the interoperability layer that maps existing data into a machine-readable, signed, and auditable shape.

  • OpenAPI Specification 3.1 — REST API definition
  • JSON Schema 2020-12 — schema validation
  • ISO/IEC 27001:2022 — information security management
  • ISO/IEC 27018:2019 — PII protection in public clouds
  • ISO/IEC 17065:2012 — product certification bodies
  • CycloneDX 1.5 / SPDX 2.3 — software bill of materials
  • Sigstore (DSSE envelope, Rekor transparency log) — signature anchoring
  • in-toto Attestation Framework 1.0 — build provenance
  • IETF RFC 9700 — OAuth 2.1
  • IETF RFC 9457 — Problem Details for HTTP APIs
  • IETF RFC 8615 — well-known URIs

Practical Implementation Checklist

  • Manifest integrity — every published manifest is signed via Sigstore DSSE. Tampering is detectable without trusting an intermediary. Updates are issued as new manifests; the prior manifest is preserved with a supersede marker.
  • Retention — keep signed evidence for at least seven years to satisfy ISO/IEC 17065:2012 audit expectations. Some regulators require longer retention; in that case the deployment policy MUST extend the window rather than relying on the defaults.
  • Versioning — follow Semantic Versioning 2.0.0. Major version bumps require at least a 90-day overlap with the prior major version on every WIA-published reference implementation.
  • Negative-result reporting — a skipped indicator without recorded justification is treated as a failed indicator at audit time.
  • Key rotation — manifest signing keys are rotated per the deployment policy. The previous key is preserved in a published key history so that auditors can verify historical manifests offline.

Reference Topologies

Implementations declare their operational topology in the manifest: single-tenant edge (one runtime per organization), multi-tenant gateway (shared runtime with header-based isolation), federated mesh (multiple runtimes peer to one another and publish manifests to a directory service), or air-gapped batch (signed evidence package transported out-of-band). Each topology ships with reference test vectors so that implementations can validate their deployment matches the canonical pattern.

Open Governance

Issues, errata, and proposals are tracked at github.com/WIA-Official/wia-standards/issues with the appropriate label. The WIA Standards working group reviews open issues at the start of every minor release cycle and publishes the resulting decision log alongside the release notes. Errata are issued as patch releases; new normative requirements trigger minor bumps; backwards-incompatible changes trigger major bumps.
