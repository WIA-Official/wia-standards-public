Chapter 7: Security, Privacy, and Data Management

7.1 The Sensitivity of AAC Data

AAC systems handle some of the most sensitive data imaginable: they capture every word a person communicates. This data reveals intimate details about users' lives, relationships, health, and thoughts. The WIA-AAC standard recognizes the profound responsibility this entails and establishes comprehensive security and privacy requirements.

Beyond message content, AAC systems store voice banking recordings, calibration data that may reveal medical conditions, location data, and detailed usage patterns. All of this information requires the highest levels of protection.

Privacy First Principle: AAC users must have complete control over their communication data. The default state must always be maximum privacy. Data collection should be minimal, transparent, and user-controlled.

7.2 Data Classification

WIA-AAC defines data sensitivity levels to guide protection requirements:

Level Data Types Protection Requirements
Critical Message content, voice recordings, health-related vocabulary usage Encryption at rest and in transit, access logging, user consent required
High User profile, calibration data, personal symbols, contacts Encryption at rest, authentication required
Medium General settings, vocabulary structure, usage statistics Authentication required, optional encryption
Low Application preferences, display settings, general documentation Standard protection

7.3 Encryption Standards

All sensitive AAC data must be protected with strong encryption:

7.3.1 Encryption Requirements

// WIA-AAC Encryption Standards
const encryptionRequirements = {
    // Data at rest
    atRest: {
        algorithm: 'AES-256-GCM',
        keyDerivation: 'PBKDF2',
        keyIterations: 100000,
        saltLength: 16
    },

    // Data in transit
    inTransit: {
        protocol: 'TLS 1.3',
        minimumProtocol: 'TLS 1.2',
        cipherSuites: [
            'TLS_AES_256_GCM_SHA384',
            'TLS_CHACHA20_POLY1305_SHA256'
        ],
        certificateValidation: 'required'
    },

    // Cloud synchronization
    cloudSync: {
        endToEndEncryption: 'required',
        keyManagement: 'client-side',
        zeroKnowledge: 'recommended'
    }
};

// Implementation example
class SecureStorage {
    private encryptionKey: CryptoKey;

    async encryptData(data: any): Promise<EncryptedData> {
        const iv = crypto.getRandomValues(new Uint8Array(12));
        const encoded = new TextEncoder().encode(JSON.stringify(data));

        const encrypted = await crypto.subtle.encrypt(
            { name: 'AES-GCM', iv },
            this.encryptionKey,
            encoded
        );

        return {
            iv: this.arrayToBase64(iv),
            data: this.arrayToBase64(new Uint8Array(encrypted)),
            algorithm: 'AES-256-GCM'
        };
    }

    async decryptData(encrypted: EncryptedData): Promise<any> {
        const iv = this.base64ToArray(encrypted.iv);
        const data = this.base64ToArray(encrypted.data);

        const decrypted = await crypto.subtle.decrypt(
            { name: 'AES-GCM', iv },
            this.encryptionKey,
            data
        );

        return JSON.parse(new TextDecoder().decode(decrypted));
    }
}

7.3.2 Key Management

  • Encryption keys derived from user-controlled credentials
  • Key rotation supported without data loss
  • Secure key storage using platform capabilities (Keychain, Keystore)
  • Recovery mechanisms that do not compromise security
Zero Knowledge Design: Cloud service providers should not have access to unencrypted user data. End-to-end encryption with client-side key management ensures that even if servers are compromised, message content remains protected.

7.4 Authentication and Access Control

Controlling access to AAC systems and data:

7.4.1 User Authentication

// Authentication options for AAC systems
interface AuthenticationConfig {
    // Primary authentication
    primary: AuthMethod;

    // Fallback methods (accessibility considerations)
    fallbacks: AuthMethod[];

    // Timeout settings
    lockTimeout: number;  // Minutes before requiring re-auth
    requireAuthForSettings: boolean;

    // Delegate access
    delegateAccess: DelegateConfig[];
}

type AuthMethod =
    | { type: 'pin'; length: number }
    | { type: 'pattern'; gridSize: number }
    | { type: 'biometric'; modality: 'fingerprint' | 'face' }
    | { type: 'switch-pattern'; sequence: string }
    | { type: 'voice'; passphrase: string }
    | { type: 'device-trust'; deviceId: string };

// Accessible authentication example
const accessibleAuth: AuthenticationConfig = {
    primary: { type: 'pin', length: 4 },
    fallbacks: [
        { type: 'switch-pattern', sequence: 'long-short-long' },
        { type: 'device-trust', deviceId: 'trusted-ipad' }
    ],
    lockTimeout: 30,
    requireAuthForSettings: true,
    delegateAccess: [
        {
            name: 'Parent',
            permissions: ['settings', 'vocabulary-edit'],
            authMethod: { type: 'pin', length: 6 }
        }
    ]
};

7.5.2 Data Subject Rights

WIA-AAC systems must support user rights under privacy regulations:

  • Right to Access: Users can view all data collected about them
  • Right to Portability: Export personal data in standard formats
  • Right to Correction: Modify inaccurate data
  • Right to Deletion: Request complete data removal
  • Right to Restriction: Limit data processing
// Data rights implementation
interface DataRights {
    // Access all personal data
    exportAllData(): Promise<PersonalDataExport>;

    // Delete specific data categories
    deleteData(categories: DataCategory[]): Promise<DeletionResult>;

    // Complete account deletion
    deleteAccount(): Promise<void>;

    // View data processing activities
    getProcessingActivities(): ProcessingActivity[];

    // Manage consent
    getConsents(): ConsentRecord[];
    updateConsent(consentId: string, granted: boolean): Promise<void>;
}
Regulatory Compliance: WIA-AAC systems must comply with applicable privacy regulations including GDPR, CCPA, HIPAA (when applicable), and accessibility regulations. The standard provides a framework that meets or exceeds these requirements.

class SecureHttpClient {
    private pinnedCerts: Set<string>;

    async fetch(url: string, options?: RequestInit): Promise<Response> {
        const response = await fetch(url, {
            ...options,
            // Custom certificate validation
            // (implementation varies by platform)
        });

        if (!this.validateCertificate(response)) {
            throw new SecurityError('Certificate validation failed');
        }

        return response;
    }
}

7.9 Incident Response

Procedures for handling security incidents:

7.9.1 Incident Categories

Category Examples Response
Data Breach Unauthorized access to user data User notification, password reset, investigation
Device Theft AAC device lost or stolen Remote wipe, account lockout, backup restore
Malware Malicious software on device Isolation, clean reinstall, credential change
Unauthorized Access Unknown person accessing system Session termination, audit review, access change

7.9.2 Remote Security Commands

// Emergency security commands (require strong authentication)
interface RemoteSecurity {
    // Lock device immediately
    remoteLock(deviceId: string): Promise<void>;

    // Wipe sensitive data
    remoteWipe(deviceId: string, level: WipeLevel): Promise<void>;

    // Revoke all sessions
    revokeAllSessions(userId: string): Promise<void>;

    // Disable cloud access
    disableCloudAccess(userId: string): Promise<void>;
}

type WipeLevel = 'messages-only' | 'user-data' | 'complete';
User Empowerment: Users or their authorized representatives must have the ability to remotely secure or wipe a lost device. These capabilities must be accessible even if the primary AAC device is unavailable.

7.10 Compliance and Certification

WIA-AAC security requirements align with major frameworks:

  • GDPR (General Data Protection Regulation)
  • HIPAA (for healthcare contexts)
  • COPPA (for child users)
  • SOC 2 Type II (for cloud services)
  • ISO 27001 (information security management)

Vendors seeking WIA-AAC compliance must demonstrate adherence to these security and privacy requirements through third-party audit or self-certification with documentation.
