3.1 Mission and Goals

3.1.1 The WIA AAC Mission

The WIA AAC Standard aims to create a universal communication bridge between all assistive input devices and all communication software, ensuring that no person is ever locked out of communication due to technology incompatibility.

3.1.2 Specific Goals

Goal Description Benefit
Sensor Standardization Unified output format for all AAC sensors Any sensor works with any software
API Unification Standard interface for software developers Reduced development time, broader support
Protocol Specification Standardized communication methods Reliable, predictable connectivity
Ecosystem Integration Connection to WIA standards family AAC to sign language, braille, speech

3.4 Design Principles

3.4.1 Core Principles

  1. Universal Compatibility

    Any compliant sensor must work with any compliant software without modifications.

  2. Graceful Degradation

    Systems should function with partial data; optional fields enhance but don't require functionality.

  3. Real-Time Performance

    Signal processing must meet real-time requirements (typically <50ms latency).

  4. Extensibility

    New sensor types and data fields can be added without breaking existing implementations.

  5. Human-Centered Design

    All decisions prioritize user communication needs over technical convenience.

3.4.2 Technical Decisions

Decision Choice Rationale
Data Format JSON (primary), Protocol Buffers (optional) Human readable, widely supported, efficient option available
Coordinate System Normalized 0.0-1.0 Resolution independent, simple transformation
Timestamps Unix milliseconds (UTC) Universal, precise, timezone independent
Confidence Scores 0.0-1.0 float Intuitive, consistent across sensor types
Versioning Semantic versioning (X.Y.Z) Clear compatibility expectations

3.5 Development Phases

3.5.1 Phase Overview

Development Timeline:

Phase 1: Signal Format Standard
  Duration: 3-4 months
  Focus: Define JSON schemas for all sensor types
  Deliverables:
    - Signal format specification
    - JSON Schema files
    - Validation tools
    - Example datasets

Phase 2: API Interface Standard
  Duration: 4-5 months
  Focus: Standard programming interfaces
  Deliverables:
    - API specification document
    - Reference implementation (Rust)
    - SDKs (TypeScript, Python)
    - Developer documentation

Phase 3: Communication Protocol
  Duration: 3-4 months
  Focus: Transport layer specifications
  Deliverables:
    - Bluetooth LE GATT profile
    - USB HID report descriptors
    - WiFi/TCP message format
    - Serial protocol definition

Phase 4: Ecosystem Integration
  Duration: 4-5 months
  Focus: Connect to WIA family
  Deliverables:
    - ISP (sign language) integration
    - WIA Braille integration
    - WIA Talk (TTS) integration
    - Complete pipeline demonstration

3.5.2 Implementation Priority

Priority Sensor Type Reason
1 Eye Tracker Largest market, most complex signals
2 Switch Most common, foundational access method
3 Head Movement Growing adoption, clear use cases
4 EMG Important for progressive conditions
5 Breath Specialized but critical population
6 BCI/EEG Emerging technology, complex signals

// Example: Complete Communication Pipeline

// 1. User input via AAC
const aac = new WiaAAC();
await aac.connect({ type: 'eye_tracker' });

aac.on('text', async (text) => {
    // text = "Hello, my name is John"

    // 2. Output to speech
    await wiaTalk.speak(text);

    // 3. Output to sign language
    await wiaISP.animate(text);

    // 4. Output to braille display
    await wiaBraille.display(text);

    // All outputs happen simultaneously
    // User's message reaches everyone
});
