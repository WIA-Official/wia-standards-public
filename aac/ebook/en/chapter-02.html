Chapter 2: System Architecture

2.1 Architectural Overview

The WIA-AAC standard defines a layered architecture that separates concerns and enables flexibility while maintaining interoperability. This architecture allows different components to be developed, updated, and replaced independently while ensuring that all parts work together seamlessly.

The architecture follows a modular design philosophy where each layer has well-defined responsibilities and interfaces. This approach enables manufacturers to specialize in specific components while ensuring compatibility with the broader AAC ecosystem.

Design Principle: The WIA-AAC architecture prioritizes user continuity. Users should be able to switch devices, apps, or services without losing their vocabulary, customizations, or communication history.

2.2 The Five-Layer Model

WIA-AAC systems are organized into five distinct layers:

Layer Name Responsibility
5 Application Layer User interface, communication displays, settings
4 Communication Engine Layer Message composition, prediction, language processing
3 Content Layer Symbols, vocabulary, voice data, user profiles
2 Access Layer Input methods, selection techniques, calibration
1 Platform Layer Hardware, OS, device capabilities 
// WIA-AAC Layer Architecture Implementation
interface WIAAACSystem {
    // Layer 1: Platform
    platform: {
        deviceType: 'dedicated' | 'tablet' | 'phone' | 'computer';
        operatingSystem: string;
        screenSize: { width: number; height: number };
        capabilities: DeviceCapabilities;
    };

    // Layer 2: Access
    accessManager: {
        availableMethods: AccessMethod[];
        activeMethod: AccessMethod;
        calibrationData: CalibrationProfile;
    };

    // Layer 3: Content
    contentRepository: {
        symbolSets: SymbolSet[];
        vocabularies: VocabularySet[];
        userProfiles: UserProfile[];
        voiceData: VoiceBank[];
    };

    // Layer 4: Communication Engine
    communicationEngine: {
        messageComposer: MessageComposer;
        predictor: PredictionEngine;
        languageProcessor: LanguageProcessor;
    };

    // Layer 5: Application
    application: {
        displays: CommunicationDisplay[];
        settings: SystemSettings;
        uiTheme: UITheme;
    };
}

2.3 Layer 1: Platform Layer

The Platform Layer represents the physical hardware and operating system on which the AAC system runs. WIA-AAC is designed to be platform-agnostic, supporting:

2.3.1 Dedicated AAC Devices

Purpose-built hardware optimized for AAC use, featuring:

  • Ruggedized construction for durability
  • High-quality speakers for speech output
  • Long battery life for all-day use
  • Integrated mounting options
  • Environmental controls integration

2.3.2 Consumer Devices

Mainstream tablets, phones, and computers running AAC applications:

  • iOS devices (iPad, iPhone)
  • Android tablets and phones
  • Windows computers
  • macOS devices
  • Chrome OS devices

2.3.3 Platform Abstraction

// Platform capability detection
class PlatformAbstraction {
    static detectCapabilities(): DeviceCapabilities {
        return {
            touchScreen: this.hasTouchScreen(),
            eyeTracker: this.hasEyeTracker(),
            switchPorts: this.detectSwitchPorts(),
            speakers: this.detectAudioOutput(),
            microphone: this.hasMicrophone(),
            camera: this.hasCamera(),
            bluetooth: this.hasBluetooth(),
            wifi: this.hasWifi(),
            cellularData: this.hasCellular(),
            gps: this.hasGPS(),
            accelerometer: this.hasAccelerometer()
        };
    }

    static getOptimalConfiguration(): AccessRecommendation {
        const caps = this.detectCapabilities();
        // Recommend access methods based on available hardware
        return this.analyzeCapabilities(caps);
    }
}

2.4 Layer 2: Access Layer

The Access Layer manages how users physically interact with the AAC system. It abstracts various input methods into a unified interface that higher layers can use without knowing the specific input mechanism.

2.4.1 Direct Selection Methods

  • Touch: Finger or stylus contact with touchscreen
  • Eye Gaze: Visual fixation detected by eye tracking camera
  • Head Tracking: Head movement controlling cursor position
  • Mouse/Joystick: Traditional pointing devices

2.4.2 Scanning Methods

  • Automatic Scanning: Items highlight sequentially at timed intervals
  • Step Scanning: User controls advancement between items
  • Group-Item Scanning: Hierarchical selection reduces selections needed
  • Inverse Scanning: Items cycle while switch is held; release to select

2.4.3 Access Method Interface

// Unified access method interface
interface AccessMethod {
    readonly type: 'direct' | 'scanning' | 'encoding';
    readonly name: string;

    initialize(): Promise<void>;
    calibrate(): Promise<CalibrationResult>;

    // Unified selection event
    onSelection(callback: (target: SelectionTarget) => void): void;

    // For direct selection methods
    onPointerMove?(callback: (position: Point) => void): void;

    // For scanning methods
    onScanProgress?(callback: (current: ScanPosition) => void): void;

    configure(settings: AccessSettings): void;
    getStatus(): AccessStatus;
    dispose(): void;
}

// Example: Eye gaze implementation
class EyeGazeAccess implements AccessMethod {
    readonly type = 'direct';
    readonly name = 'Eye Gaze';

    private tracker: EyeTracker;
    private dwellTime: number = 800; // ms

    async initialize() {
        this.tracker = await EyeTracker.connect();
        await this.tracker.startTracking();
    }

    async calibrate() {
        return await this.tracker.runCalibration({
            points: 9,
            animation: 'shrinking-circle'
        });
    }

    configure(settings: AccessSettings) {
        this.dwellTime = settings.dwellTime ?? 800;
        this.tracker.setSmoothing(settings.smoothing ?? 'medium');
    }
}
Accessibility Requirement: All WIA-AAC systems MUST support at least two distinct access methods to ensure users have alternatives if their primary method becomes unavailable.

2.5 Layer 3: Content Layer

The Content Layer manages all the linguistic and personalization resources used by the AAC system:

2.5.1 Symbol Management

Symbols are stored in standardized packages that include:

  • Vector and raster images at multiple resolutions
  • Semantic metadata linking symbols to concepts
  • Multilingual labels and descriptions
  • Usage statistics and frequency data
  • Licensing and attribution information

2.5.2 Vocabulary Organization

    background?: BackgroundImage;
}

2.5.3 User Profile Storage

User profiles contain personalized settings and data:

  • Access method configurations and calibration data
  • Vocabulary customizations and personal words
  • Voice preferences and voice bank data
  • Usage patterns for prediction optimization
  • Communication history (optional, with privacy controls)

2.6 Layer 4: Communication Engine Layer

The Communication Engine is the intelligent core that transforms user selections into meaningful communication:

2.6.1 Message Composition

The message composer manages the construction of utterances:

  • Concatenating symbols into phrases
  • Managing message buffer and history
  • Handling grammar and morphology
  • Supporting multiple composition strategies

2.6.2 Prediction Engine

// Prediction engine interface
interface PredictionEngine {
    // Word/symbol prediction
    predictNext(context: MessageContext): Prediction[];

    // Sentence completion
    completeSentence(partial: string): string[];

    // Learn from user patterns
    learn(selection: SelectionEvent): void;

    // Configure prediction behavior
    configure(settings: PredictionSettings): void;
}

interface PredictionSettings {
    maxPredictions: number;
    includeRecent: boolean;
    includeFrequent: boolean;
    useLanguageModel: boolean;
    personalWeight: number;  // 0-1, weight of personalized vs general predictions
    contextWindow: number;   // Number of previous selections to consider
}

// Example prediction result
interface Prediction {
    symbol: Symbol;
    confidence: number;
    source: 'language-model' | 'personal-history' | 'recency' | 'frequency';
}

2.6.3 Language Processing

The language processor handles linguistic transformations:

  • Morphological processing (plurals, tenses, conjugations)
  • Syntactic arrangement of symbols
  • Acronym and abbreviation expansion
  • Rate enhancement techniques

2.7 Layer 5: Application Layer

The Application Layer provides the user-facing interface:

2.7.1 Communication Displays

Multiple display types are supported:

  • Grid Display: Traditional rows and columns of symbols
  • Visual Scene Display: Interactive photographs or illustrations
  • Keyboard Display: Letter-by-letter spelling with prediction
  • Hybrid Display: Combination of multiple display types

2.7.2 User Interface Components

UI Requirement: The Application Layer MUST adapt to the capabilities of the underlying platform while maintaining consistent functionality. A WIA-AAC compliant app should provide equivalent features whether running on a dedicated device or a consumer tablet.

2.8 Inter-Layer Communication

Layers communicate through well-defined interfaces and events:

2.8.1 Event Flow

A typical selection flows through the system as follows:

  1. Platform Layer detects raw input (touch, switch press, gaze fixation)
  2. Access Layer interprets input and identifies selected target
  3. Application Layer updates visual feedback
  4. Communication Engine processes selection
  5. Content Layer provides symbol and vocabulary data
  6. Communication Engine updates message and predictions
  7. Application Layer renders updated display and speaks if configured

2.8.2 Event Bus Architecture

// Central event bus for inter-layer communication
class AACEventBus {
    private listeners: Map<string, Set<EventHandler>> = new Map();

    emit(event: AACEvent): void {
        const handlers = this.listeners.get(event.type);
        if (handlers) {
            handlers.forEach(handler => handler(event));
        }
    }

    on(eventType: string, handler: EventHandler): Unsubscribe {
        if (!this.listeners.has(eventType)) {
            this.listeners.set(eventType, new Set());
        }
        this.listeners.get(eventType)!.add(handler);
        return () => this.listeners.get(eventType)?.delete(handler);
    }
}

// Event types
type AACEvent =
    | { type: 'selection'; target: SelectionTarget }
    | { type: 'message-update'; message: Message }
    | { type: 'speak'; text: string }
    | { type: 'prediction-update'; predictions: Prediction[] }
    | { type: 'page-change'; page: VocabularyPage }
    | { type: 'access-change'; method: AccessMethod };

2.9 Extensibility Points

The WIA-AAC architecture provides several extensibility points:

  • Custom Access Methods: New input technologies can implement the AccessMethod interface
  • Symbol Set Plugins: Additional symbol systems can be added via the content layer
  • Prediction Engines: Alternative prediction algorithms can be swapped in
  • Voice Engines: Different TTS engines can be integrated
  • Display Templates: Custom UI layouts can be created

This layered, extensible architecture ensures that WIA-AAC systems can evolve with advancing technology while maintaining compatibility and user continuity.
