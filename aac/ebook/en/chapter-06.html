Chapter 6: Integration and Interoperability

6.1 The Connected AAC Ecosystem

Modern AAC systems do not operate in isolation. They must integrate seamlessly with the broader technology ecosystem including smart devices, communication platforms, productivity tools, and environmental control systems. The WIA-AAC standard defines protocols and APIs that enable this integration while maintaining security and user control.

Integration serves multiple purposes: it extends the communicative reach of AAC users, enables participation in digital life, provides independence through environmental control, and ensures AAC data can move freely between systems as users' needs change.

Interoperability Mandate: WIA-AAC compliant systems must support standardized data exchange formats and APIs. Users must never be locked into a single vendor's ecosystem. Data portability is a fundamental right.

6.2 User Profile Portability

User profiles contain personalized settings that represent significant time investment. These must be portable:

6.2.1 Profile Export Format

// WIA-AAC User Profile Exchange Format
interface UserProfileExport {
    format: 'wia-aac-profile';
    version: '1.0';
    exportDate: string;
    exportedBy: string;  // Application identifier

    // Core user data
    user: {
        id: string;
        displayName: string;
        languages: string[];
    };

    // Access configuration
    access: {
        methods: AccessMethodConfig[];
        activeMethod: string;
        calibrationData: Map<string, CalibrationData>;
    };

    // Voice preferences
    voice: {
        selectedVoice: string;
        settings: VoiceSettings;
        pronunciationDictionary: PronunciationEntry[];
        bankedVoice?: VoiceBankReference;
    };

    // Vocabulary customizations
    vocabulary: {
        baseVocabulary: string;
        customPages: VocabularyPage[];
        customSymbols: Symbol[];
        hiddenItems: string[];
        buttonCustomizations: ButtonCustomization[];
    };

    // Usage data (optional, privacy-controlled)
    usage?: {
        frequencyData: WordFrequency[];
        predictionModel?: PredictionModelData;
        communicationStats?: UsageStatistics;
    };

    // Digital signature for integrity
    signature?: string;
}

6.2.2 Profile Import/Export API

// Profile portability operations
interface ProfilePortability {
    // Export user profile
    exportProfile(userId: string, options: ExportOptions): Promise<ProfileExport>;

    // Import profile to system
    importProfile(profile: ProfileExport, options: ImportOptions): Promise<ImportResult>;

    // Merge profiles (e.g., combining from multiple devices)
    mergeProfiles(primary: ProfileExport, secondary: ProfileExport): Promise<MergeResult>;

    // Validate profile before import
    validateProfile(profile: ProfileExport): ValidationResult;
}

interface ExportOptions {
    includeUsageData: boolean;
    includeVoiceBank: boolean;
    includeCalibration: boolean;
    encryptionKey?: string;  // For sensitive data protection
}

interface ImportOptions {
    conflictResolution: 'keep-existing' | 'use-imported' | 'merge' | 'ask';
    recalibrationRequired: boolean;
    verifySignature: boolean;
}

                   'fan' | 'outlet' | 'speaker' | 'camera' | 'bed' | 'phone';

// AAC vocabulary integration
interface ECUVocabularyPage {
    generateControlButtons(devices: SmartDevice[]): VocabularyPage {
        const page: VocabularyPage = {
            id: 'ecu-controls',
            name: 'Home Control',
            type: 'ecu',
            rows: 4,
            columns: 6,
            cells: []
        };

        for (const device of devices) {
            page.cells.push({
                symbol: this.getDeviceSymbol(device.type),
                label: device.name,
                action: {
                    type: 'ecu-control',
                    deviceId: device.id,
                    showSubmenu: device.capabilities.length > 2
                }
            });
        }

        return page;
    }
}

6.5 Cloud Synchronization

Cloud services enable synchronization across devices and backup protection:

6.5.1 Sync Architecture

// Cloud synchronization service
interface CloudSyncService {
    // Connection management
    connect(credentials: CloudCredentials): Promise<void>;
    getConnectionStatus(): ConnectionStatus;

    // Synchronization
    syncProfile(): Promise<SyncResult>;
    syncVocabulary(): Promise<SyncResult>;
    syncSettings(): Promise<SyncResult>;

    // Conflict resolution
    resolveConflict(conflict: SyncConflict, resolution: ConflictResolution): Promise<void>;

    // Backup and restore
    createBackup(name: string): Promise<BackupInfo>;
    listBackups(): Promise<BackupInfo[]>;
    restoreBackup(backupId: string): Promise<RestoreResult>;

    // Real-time sync (for multi-device scenarios)
    enableRealtimeSync(options: RealtimeSyncOptions): void;
    onRemoteChange(callback: (change: RemoteChange) => void): void;
}

interface SyncResult {
    success: boolean;
    itemsSynced: number;
    conflicts: SyncConflict[];
    timestamp: Date;
}

// Multi-device scenario: Changes sync across devices
class MultiDeviceSync {
    async handleVocabularyChange(change: VocabularyChange) {
        // Local change made
        await this.syncService.pushChange(change);

        // Notify other devices
        this.syncService.broadcastChange(change);
    }

    // Receive change from another device
    onRemoteVocabularyChange(change: VocabularyChange) {
        this.vocabulary.applyRemoteChange(change);
        this.ui.refreshDisplay();
    }
}

6.5.2 Offline-First Architecture

AAC systems must function fully offline with synchronization when connectivity returns:

  • All core functionality available offline
  • Changes queued during offline periods
  • Automatic sync when connection restored
  • Conflict resolution for concurrent edits
  • Bandwidth-efficient delta synchronization
Reliability Requirement: Communication cannot wait for internet connectivity. WIA-AAC systems must maintain full functionality offline, with cloud features enhancing rather than enabling core communication capabilities.

6.6.2 WebSocket Events

// Real-time event stream for integrations
// Connect: wss://api.aac-device.local/v1/events

interface AACEvent {
    type: EventType;
    timestamp: string;
    data: EventData;
}

type EventType =
    | 'message.spoken'        // Message was spoken
    | 'message.queued'        // Message added to queue
    | 'selection.made'        // User made a selection
    | 'page.changed'          // Vocabulary page changed
    | 'access.changed'        // Access method changed
    | 'profile.updated'       // Profile was modified
    | 'ecu.device.status';    // Smart device status changed

// Subscribing to events
const ws = new WebSocket('wss://api.aac-device.local/v1/events');
ws.onmessage = (event) => {
    const aacEvent = JSON.parse(event.data);
    if (aacEvent.type === 'message.spoken') {
        console.log('User said:', aacEvent.data.text);
    }
};

6.7 Data Exchange Formats

Standard formats for exchanging AAC-related data:

6.7.1 OBZ/OBF Format

Open Board Format for vocabulary exchange:

  • JSON-based specification for communication boards
  • Widely adopted across AAC applications
  • Supports symbols, pages, and navigation
  • WIA-AAC extends OBZ with additional metadata

6.7.2 Symbol Exchange Format

// Symbol package exchange format
interface SymbolExchange {
    format: 'wia-symbol-pack';
    version: '1.0';

    metadata: {
        name: string;
        symbolSet: string;
        symbolCount: number;
        languages: string[];
        license: string;
    };

    symbols: ExportedSymbol[];

    // Relationships and categories
    categories: Category[];
    conceptMappings: ConceptMapping[];
}

interface ExportedSymbol {
    id: string;
    images: {
        format: 'svg' | 'png';
        data: string;  // Base64 or relative path
    }[];
    labels: {
        language: string;
        text: string;
        spoken?: string;
    }[];
    semantics: {
        partOfSpeech: string;
        categories: string[];
    };
}

6.9 Integration Security

Security considerations for integrated systems:

  • Authentication: OAuth 2.0 for third-party app access
  • Authorization: Granular permission scopes
  • Encryption: TLS for all network communication
  • Local API Security: Token-based access for local integrations
  • Audit Logging: Track integration access and actions
