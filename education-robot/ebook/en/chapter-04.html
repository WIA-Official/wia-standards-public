Chapter 4: Phase 1 - Data Format Specifications

4.1 Introduction to Data Formats

Phase 1 of WIA-ROB-015 establishes standardized data formats for all information generated, stored, and exchanged within educational robotics ecosystems. This foundational layer enables interoperability at the most fundamental level—when different systems speak the same data language, they can share student progress, exchange curriculum materials, aggregate learning analytics, and preserve educational investments across platform transitions.

The standard specifies JSON (JavaScript Object Notation) as the primary data format due to its human readability, machine parse ability, widespread support across programming languages, self-documenting structure, and extensibility for future additions. JSON Schema provides formal validation ensuring data quality and consistency.

All data formats incorporate privacy-by-design principles, specifying minimal necessary data collection, anonymization requirements, consent mechanisms, retention policies, and secure storage specifications. This ensures student data protection is built into the standard's foundation, not bolted on as an afterthought.

4.2 Student Progress Data Format

Student progress tracking is central to educational robotics. The standard defines a comprehensive yet flexible format for recording what students learn, how they learn it, and evidence of their mastery.

Core Progress Schema

The student progress format includes student identifier (anonymized/pseudonymized per privacy requirements), session metadata (timestamp, duration, platform, robot model), skills and competencies attempted and mastered, learning objectives addressed, projects completed, errors encountered and resolved, collaborative activities and peer interactions, and teacher observations and assessments.

{
  "studentId": "anonymous-uuid-here",
  "sessionId": "session-12345",
  "timestamp": "2025-01-15T14:30:00Z",
  "duration": 2700,
  "platform": {
    "robotModel": "LEGO-SPIKE-PRIME",
    "softwareVersion": "3.4.1",
    "wiaCompliance": "Silver-1.0"
  },
  "learningObjectives": [
    {
      "id": "CSTA-1B-AP-10",
      "description": "Modularize computational artifacts",
      "status": "mastered",
      "evidence": ["project-maze-solver-v3"]
    }
  ],
  "skills": [
    {
      "skill": "loop-while",
      "attempts": 8,
      "mastery": 0.85,
      "firstAttempt": "2025-01-15T14:35:00Z",
      "masteryAchieved": "2025-01-15T15:20:00Z"
    }
  ],
  "projects": [
    {
      "projectId": "maze-solver-v3",
      "type": "autonomous-navigation",
      "complexity": "intermediate",
      "completionStatus": "success",
      "iterations": 12,
      "timeSpent": 1800
    }
  ]
}

Computational Thinking Assessment

The standard includes specific formats for assessing computational thinking dimensions: decomposition (breaking problems into parts), pattern recognition (identifying similarities), abstraction (focusing on essential features), algorithm design (step-by-step solutions), and debugging (systematic error correction).

Each dimension includes rubric-based assessment levels, evidence collection methods, progression indicators, and connection to specific activities. This enables tracking how robotics activities develop computational thinking over time, providing valuable research data and instructional feedback.

Privacy-Preserving Identifiers

Student identifiers use pseudonymization—replacing personally identifiable information with pseudonyms that enable tracking individuals across sessions while protecting identity. The mapping between real identities and pseudonyms is stored separately with strict access controls, encryption at rest and in transit, audit logging of all access, and automatic expiration based on retention policies.

4.3 Robot Sensor Data Format

Educational robots generate diverse sensor data—motor encoders, distance sensors, color sensors, gyroscopes, accelerometers, touch sensors, light sensors, sound sensors, temperature sensors, and custom add-on sensors. The standard provides unified formats for all sensor types while enabling extensibility for new sensors.

Unified Sensor Message Format

Each sensor reading follows a common structure: sensor identification (type, model, location on robot), timestamp (high-precision for real-time applications), value (type-specific reading with units), quality indicators (confidence, accuracy, calibration status), and metadata (environmental conditions, error flags).

{
  "sensorType": "ultrasonic-distance",
  "sensorId": "us-front-001",
  "timestamp": "2025-01-15T14:35:22.347Z",
  "value": 45.3,
  "unit": "centimeters",
  "quality": {
    "confidence": 0.95,
    "calibrationStatus": "valid",
    "lastCalibration": "2025-01-15T14:30:00Z"
  },
  "metadata": {
    "temperature": 22.5,
    "ambientLight": "moderate",
    "surfaceType": "flat"
  }
}

Motor and Encoder Data

Motor control and encoder feedback are fundamental to educational robotics. The standard specifies formats for command (target speed/position, acceleration limits, duration) and state information (current speed/position, encoder counts, power consumption, temperature, error conditions).

This enables detailed analysis of robot behavior, troubleshooting unexpected movements, comparing student programs' efficiency, and analyzing mechanical design effectiveness—all valuable for both learning and research.

Sensor Fusion and Derived Data

Advanced applications combine multiple sensors to derive higher-level information—pose estimation from encoders and gyroscopes, object recognition from cameras and distance sensors, or environmental mapping from multiple sensor types. The standard includes formats for this derived data, maintaining traceability to source sensors for validation and debugging.

4.4 Curriculum Metadata Format

Standardized curriculum metadata enables discovering, sharing, and reusing educational robotics activities across platforms. This dramatically reduces teachers' workload and improves educational quality by enabling community sharing of effective curricula.

Learning Resource Metadata

Each curriculum resource includes descriptive information (title, description, author, version), educational metadata (grade levels, subject areas, learning objectives, standards alignment), technical requirements (required robot models, sensors, programming environment), pedagogical information (instructional approach, group size, time estimate, prerequisite skills), and assessment resources (rubrics, success criteria, sample solutions).

Standards Alignment

The format includes explicit alignment with educational standards: CSTA K-12 Computer Science Standards, NGSS (Next Generation Science Standards), CCSS Mathematics Standards, ISTE Standards for Students, and regional/national standards (flexible extension system).

Each alignment specifies the standard identifier, grade band, specific performance indicator, how the activity addresses the standard, and assessment evidence showing student mastery. This explicit alignment helps teachers justify robotics in curriculum, ensures coverage of required content, supports assessment and accountability, and enables data aggregation showing robotics' educational impact.

Difficulty Levels and Progression

Curriculum metadata includes difficulty assessments across multiple dimensions: programming complexity (beginner/intermediate/advanced), mechanical building complexity, problem-solving challenge level, collaboration requirements, and time commitment. This multi-dimensional difficulty model enables better matching of activities to students' current capabilities and supports progressive skill development.

4.5 Learning Analytics Data Format

Educational robotics generates rich data about student learning processes—not just final outcomes but how students approach problems, what strategies they try, where they struggle, and how they improve. Learning analytics formats capture this process data for research and instructional improvement.

xAPI Integration

WIA-ROB-015 builds on the Experience API (xAPI, also called Tin Can API) for learning analytics, extending it with robotics-specific verbs and objects. xAPI's actor-verb-object structure (e.g., "Student programmed robot") provides flexible, standardized event tracking.

{
  "actor": {
    "objectType": "Student",
    "account": {
      "homePage": "https://school.edu",
      "name": "anonymous-student-uuid"
    }
  },
  "verb": {
    "id": "http://wia.org/verbs/debugged",
    "display": {"en-US": "debugged"}
  },
  "object": {
    "id": "http://wia.org/activities/maze-solver",
    "objectType": "Activity",
    "definition": {
      "type": "http://wia.org/activity-types/robot-program"
    }
  },
  "result": {
    "success": true,
    "duration": "PT45M",
    "extensions": {
      "http://wia.org/extensions/errors-fixed": 3,
      "http://wia.org/extensions/iterations": 7
    }
  }
}

Intervention and Support Data

The format captures when and how teachers or systems provide support: hint types and timing, direct instruction interventions, peer collaboration facilitation, and assessment feedback. Analyzing intervention patterns helps identify where students commonly struggle and which support strategies are most effective.

Affective and Engagement Measures

Beyond cognitive measures, learning analytics include affective dimensions: time-on-task and persistence indicators, frustration signals (repeated errors, help-seeking), engagement patterns (focused work vs. distraction), and self-reported confidence and enjoyment. While more challenging to measure reliably, these dimensions significantly impact learning and are valuable for understanding robotics education's full impact.

4.6 Assessment Data Format

Assessment in educational robotics extends beyond traditional tests to include performance assessments, portfolios, peer evaluations, and self-assessments. The standard provides formats supporting this diversity while maintaining consistency for aggregation and analysis.

Rubric-Based Assessment

The standard format for rubric-based assessments includes rubric definition (criteria, performance levels, descriptors), assessment instance (which student, which project, when), scores for each criterion, evidence supporting scores (photos, code snippets, videos, observations), and assessor information (teacher, peer, self-assessment).

Portfolio Artifacts

Student portfolios in robotics include diverse artifacts—program code, robot design documentation, project videos, reflection journals, and presentation materials. The format provides rich metadata for each artifact: creation date and version, project context, learning objectives demonstrated, student reflection on learning, teacher commentary, and peer feedback.

Portfolio formats use standard MIME types for various media while providing educational metadata linking artifacts to learning objectives, standards, and skill development. This enables both showcasing student work and tracking learning progression over time.

Competency-Based Assessment

The standard supports competency-based education approaches tracking mastery of discrete skills and knowledge elements. Competency data includes competency definition (what is being assessed), current mastery level (typically 0-100% or categorical levels), evidence of mastery (artifacts, observations, assessments), and learning progression (how mastery developed over time).

Competency frameworks enable students to progress at their own pace, teachers to identify specific gaps needing attention, parents to understand concrete skill development, and researchers to analyze learning progressions across many students.

Assessment Type Data Captured Primary Use Frequency
Rubric Assessment Multi-criteria scores, evidence, feedback Project evaluation, summative assessment Per project
Skills Checklist Binary or scaled mastery per skill Progress tracking, differentiation Continuous
Portfolio Artifact Work sample, metadata, reflection Demonstrating growth, showcasing work Ongoing collection

4.7 Classroom Management Data

Effective educational robotics requires managing shared resources, coordinating group work, and tracking equipment. The standard includes formats for robot inventory and assignment, group compositions and roles, equipment checkout and return, maintenance and repair tracking, and safety incident reporting.

Equipment and Resource Management

Schools need to track which robots are assigned to which students or groups, equipment condition and maintenance needs, battery status and charging schedules, replacement parts inventory, and usage statistics for planning purchases. Standardized formats enable schools to use common management tools across different robot platforms rather than maintaining separate systems for each.

Group Management and Collaboration

The format tracks group compositions (which students work together), role assignments (programmer, builder, tester, documenter), task division and coordination, and collaborative artifacts (shared programs, group designs). This information supports research on effective collaboration, enables teachers to assess teamwork skills, and helps ensure equitable participation within groups.

4.8 Data Privacy and Security

All data formats incorporate privacy and security specifications: data minimization (collect only necessary information), pseudonymization of student identifiers, consent tracking and management, retention periods and deletion procedures, encryption requirements for sensitive data, access control specifications, and audit logging requirements.

The standard provides clear guidance on what constitutes personally identifiable information (PII) in robotics contexts, how to de-identify data for research, consent requirements for different data types and student ages, and complying with FERPA, COPPA, GDPR and other privacy regulations.

Data Portability and Ownership

Students and parents have rights to access, correct, and delete personal data. The standard requires export formats enabling data portability—students can take their robotics learning records when changing schools or platforms. Standard formats make this portability practical, preventing data from becoming trapped in proprietary systems.

Privacy Best Practices in Data Formats

  • Anonymization by Default: Student identifiers are pseudonymous unless specific consent for identified data exists
  • Granular Consent: Separate consent for different data uses (education, research, improvement)
  • Age-Appropriate Controls: Simplified interfaces for younger students, parental controls for children under 13
  • Transparency: Human-readable data formats allow students/parents to understand what's collected
  • Minimal Collection: Schemas specify only necessary data elements, discouraging "collect everything" approaches
  • Secure by Design: Encryption, access controls, and audit logging built into format specifications

Key Takeaways

  • JSON as Universal Format: JSON provides human-readable, machine-parseable data interchange enabling interoperability across diverse educational robotics platforms and tools.
  • Comprehensive Progress Tracking: Student progress data captures not just outcomes but learning processes, enabling rich analytics and instructional improvement.
  • Sensor Data Standardization: Unified sensor formats enable comparison across platforms, support troubleshooting, and facilitate research on robot behavior and student programming.
  • Rich Curriculum Metadata: Standardized curriculum formats with standards alignment, difficulty levels, and pedagogical information enable discovery, sharing, and reuse of effective educational resources.
  • xAPI Integration for Analytics: Building on established xAPI standards provides familiar frameworks extended with robotics-specific elements for learning analytics.
  • Multi-Modal Assessment: Formats support rubrics, portfolios, competencies, peer evaluation, and self-assessment, reflecting the diversity of meaningful assessment in robotics education.
  • Privacy by Design: Privacy protections, data minimization, anonymization, and consent management are built into data format specifications from the ground up.
  • Data Portability: Standard formats enable students to take their learning records across platforms and schools, preventing vendor lock-in at the data level.
  • Classroom Management Integration: Equipment tracking, group management, and resource allocation formats support practical classroom operations.

Reference Standards and Implementation Notes

This standard is designed to interoperate with established international frameworks. Implementations SHOULD treat the following as the single source of truth and use this PHASE document as the interoperability layer that maps existing data into a machine-readable, signed, and auditable shape.

  • OpenAPI Specification 3.1 — REST API definition
  • JSON Schema 2020-12 — schema validation
  • ISO/IEC 27001:2022 — information security management
  • ISO/IEC 27018:2019 — PII protection in public clouds
  • ISO/IEC 17065:2012 — product certification bodies
  • CycloneDX 1.5 / SPDX 2.3 — software bill of materials
  • Sigstore (DSSE envelope, Rekor transparency log) — signature anchoring
  • in-toto Attestation Framework 1.0 — build provenance
  • IETF RFC 9700 — OAuth 2.1
  • IETF RFC 9457 — Problem Details for HTTP APIs
  • IETF RFC 8615 — well-known URIs

Practical Implementation Checklist

  • Manifest integrity — every published manifest is signed via Sigstore DSSE. Tampering is detectable without trusting an intermediary. Updates are issued as new manifests; the prior manifest is preserved with a supersede marker.
  • Retention — keep signed evidence for at least seven years to satisfy ISO/IEC 17065:2012 audit expectations. Some regulators require longer retention; in that case the deployment policy MUST extend the window rather than relying on the defaults.
  • Versioning — follow Semantic Versioning 2.0.0. Major version bumps require at least a 90-day overlap with the prior major version on every WIA-published reference implementation.
  • Negative-result reporting — a skipped indicator without recorded justification is treated as a failed indicator at audit time.
  • Key rotation — manifest signing keys are rotated per the deployment policy. The previous key is preserved in a published key history so that auditors can verify historical manifests offline.

Reference Topologies

Implementations declare their operational topology in the manifest: single-tenant edge (one runtime per organization), multi-tenant gateway (shared runtime with header-based isolation), federated mesh (multiple runtimes peer to one another and publish manifests to a directory service), or air-gapped batch (signed evidence package transported out-of-band). Each topology ships with reference test vectors so that implementations can validate their deployment matches the canonical pattern.

Open Governance

Issues, errata, and proposals are tracked at github.com/WIA-Official/wia-standards/issues with the appropriate label. The WIA Standards working group reviews open issues at the start of every minor release cycle and publishes the resulting decision log alongside the release notes. Errata are issued as patch releases; new normative requirements trigger minor bumps; backwards-incompatible changes trigger major bumps.
