Chapter 5: Phase 2 - API Interface Standards

5.1 API Architecture Overview

Phase 2 of WIA-ROB-015 defines Application Programming Interfaces (APIs) that enable software to control educational robots, access sensor data, manage learning sessions, and integrate with educational systems. These APIs provide the programmable interface that makes educational robotics accessible to students at all levels while supporting advanced capabilities for experienced programmers.

The standard specifies multiple API types serving different needs: REST APIs for general robot control and management, WebSocket connections for real-time sensor streaming, block-based programming interfaces for visual programming environments, Python SDK for intermediate and advanced students, JavaScript SDK for web-based applications and beginners, and LMS integration APIs for connecting with learning management systems.

All APIs follow consistent design principles including age-appropriate complexity levels, safety-first operation with automatic limits, comprehensive error handling and educational feedback, backward compatibility guarantees, and extensive documentation with examples.

5.2 REST API for Robot Control

The WIA-ROB-015 REST API provides HTTP-based access to core robot functions. This enables remote control, web-based programming environments, server-side orchestration of multi-robot activities, and integration with enterprise systems.

Core Endpoints

The REST API includes endpoints for robot discovery and connection, motor control (speed, position, duration), sensor reading (current values, streaming configuration), program management (upload, execute, pause, stop), status and diagnostics, configuration and calibration, and emergency stop and safety controls.

Example endpoint structure demonstrates the API design:

POST /api/v1/robots/{robotId}/motors/{motorId}/move
{
  "speed": 50,
  "duration": 3000,
  "acceleration": "moderate",
  "safetyLimits": {
    "maxSpeed": 75,
    "workspaceRestriction": "desk-area-1"
  }
}

Response:
{
  "status": "executing",
  "commandId": "cmd-12345",
  "estimatedCompletion": "2025-01-15T14:35:25Z",
  "safety": {
    "limitsApplied": true,
    "workspaceClear": true
  }
}

Authentication and Security

API authentication uses OAuth 2.0 for teacher/administrator access and classroom-specific access tokens for student access with age-appropriate permissions. All communications use TLS encryption, rate limiting prevents abuse, and audit logs track all robot control actions for safety and accountability.

Student API access includes automatic safety limits: maximum speeds appropriate for educational settings, workspace restrictions preventing robots leaving designated areas, time limits for continuous operation, emergency stop accessible at all times, and teacher override capabilities.

Error Handling and Educational Feedback

API errors provide educational feedback beyond simple error codes. Responses include clear explanations of what went wrong, suggestions for correction, links to relevant documentation or tutorials, severity levels (warning vs. critical error), and recovery procedures.

For example, a motor overload error provides: "Motor A cannot reach target speed. The robot may be stuck or carrying too much weight. Try reducing speed to 50% or checking for obstacles." This transforms debugging from frustrating technical troubleshooting into learning opportunities about robot limitations and problem-solving strategies.

5.3 Block-Based Programming Interface Specification

Block-based visual programming makes coding accessible to young learners and beginners. WIA-ROB-015 specifies how educational robots should integrate with block-based environments like Scratch, Blockly, and proprietary visual programming systems.

Standard Block Categories

The specification defines standard block categories present across all compliant platforms: Motion blocks (move forward/backward, turn, stop, speed control), Sensing blocks (read sensor values, wait until condition), Control blocks (loops, conditions, wait commands), Variables and data blocks, Event blocks (when program starts, when sensor triggered), Operators (mathematics, logic, comparison), and Advanced blocks (functions, multi-threading for older students).

Each block category includes age-appropriate variations. Elementary blocks use simple, concrete language and limited parameters. Middle school blocks introduce variables and more complex logic. High school blocks expose advanced features including custom functions and event-driven programming.

Block-to-Code Translation

A key feature supports showing generated code alongside blocks, helping students transition from visual to text programming. Students see how their blocks translate to Python, JavaScript, or other languages, building understanding of text syntax while maintaining visual scaffolding.

The specification requires bidirectional translation—students can edit code and see blocks update, or modify blocks and see code change. This flexibility supports diverse learning styles and smooth progression from visual to text programming.

Custom Block Extensions

While standard blocks ensure baseline compatibility, platforms can add custom blocks for unique features. Extension specifications ensure custom blocks: use clear naming conventions, provide tooltips and documentation, maintain visual design consistency, don't conflict with standard blocks, and include metadata for filtering (e.g., "advanced only").

5.4 Python SDK for Education

Python is widely used in education for its readable syntax and extensive libraries. The WIA-ROB-015 Python SDK provides idiomatic Python interfaces to robot control suitable for middle school through university students.

SDK Architecture

The Python SDK follows object-oriented design with intuitive class hierarchies: Robot class (main interface for robot control), Motor class (individual motor control), SensorGroup class (accessing all sensors), Vision class (camera and image processing), Navigation class (mapping and pathfinding), and Utilities (helpers for common tasks like PID control).

from wia_robot import Robot, Motor

# Connect to robot
robot = Robot.connect("spike-prime-01")

# Move motors
robot.motor_a.run_for_degrees(360, speed=50)
robot.motor_b.run_for_degrees(360, speed=50)

# Read sensor
distance = robot.distance_sensor.get_distance()
if distance < 10:
    robot.stop()
    robot.speak("Obstacle detected!")

# Advanced: PID line following
follower = robot.create_line_follower(
    color_sensor=robot.color_sensor_1,
    motor_left=robot.motor_a,
    motor_right=robot.motor_b,
    target_light=50
)
follower.follow_line(duration=10)  # Follow for 10 seconds

Educational Features

The SDK includes educational features beyond basic robot control: interactive help and documentation, example programs for common tasks, debugging tools with visual feedback, simulation mode for testing without hardware, performance profiling for optimization challenges, and integration with Jupyter notebooks for data science education.

Error messages are pedagogically designed. Instead of cryptic exceptions, students see contextual explanations with suggestions: "Motor A didn't complete movement in expected time. The robot may be stuck. Try checking for obstacles or reducing the target distance."

Advanced Capabilities

For advanced students and university courses, the SDK exposes: computer vision with OpenCV integration, machine learning with scikit-learn/TensorFlow, custom sensor data processing, low-level motor control for PID tuning, network programming for multi-robot systems, and real-time operating system features where available.

5.5 JavaScript SDK for Web Applications

JavaScript enables browser-based robot programming without installation, making it accessible for Chromebooks and shared devices common in schools. The WIA-ROB-015 JavaScript SDK supports both beginner-friendly and advanced web development approaches.

Web-Based Programming Environments

The JavaScript SDK enables fully browser-based IDEs: code editor with syntax highlighting and autocomplete, visual debugger with breakpoints and variable inspection, real-time sensor visualization, program sharing and collaboration features, integration with Google Classroom/Microsoft Teams, and offline support with service workers.

import { Robot } from '@wia/robot-sdk';

// Connect to robot via WebSocket
const robot = new Robot();
await robot.connect('ws://robot-spike-01.local');

// Event-driven programming
robot.on('sensor-update', (data) => {
    if (data.distance < 15) {
        robot.motors.stop();
        console.log('Too close to obstacle!');
    }
});

// Async/await for sequential actions
async function performDance() {
    await robot.motors.rotate(360, { speed: 75 });
    await robot.lights.color('green');
    await robot.sound.play('success.wav');
}

performDance();

Progressive Web App Support

The SDK supports Progressive Web App (PWA) features enabling offline robot programming, installation as standalone apps, background synchronization for cloud-saved programs, push notifications for multi-user collaboration, and responsive design for tablets and smartphones.

5.6 Learning Management System Integration

Integrating educational robotics with Learning Management Systems (LMS) like Moodle, Canvas, Google Classroom, and Blackboard ensures robotics activities fit seamlessly into broader coursework. The WIA-ROB-015 LMS integration APIs enable this critical connection.

LTI (Learning Tools Interoperability)

The standard implements IMS Global LTI 1.3 for deep LMS integration: single sign-on (students use LMS credentials), roster sync (automatic class enrollment), grade passback (robot project scores appear in LMS gradebook), content integration (activities embedded in LMS pages), and deep linking (easy content selection for teachers).

LTI integration means teachers don't manage separate accounts for robotics platforms. Students access robotics activities through familiar LMS interfaces. Grades flow automatically to institutional systems. Robotics becomes part of unified digital learning environments rather than disconnected tools.

Learning Analytics Integration

Beyond basic gradebook integration, the LMS APIs support rich learning analytics: xAPI statements for detailed activity tracking, Caliper events for learning analytics platforms, assignment submission with robot programs and videos, discussion forum integration for peer learning, and peer review workflows for collaborative assessment.

5.7 Real-Time Communication APIs

Educational robotics often requires real-time interaction—live sensor feedback during programming, collaborative multi-robot activities, or competitions with live scoring. WebSocket APIs enable these real-time capabilities.

WebSocket Sensor Streaming

Real-time sensor streaming uses WebSocket connections for low-latency bidirectional communication: sensor value subscriptions (client specifies which sensors to stream), configurable update rates (balance responsiveness with bandwidth), multiplexed streams (multiple sensors over single connection), and automatic reconnection (handling network interruptions gracefully).

Multi-Robot Coordination

Real-time APIs support coordinated activities with multiple robots: robot state synchronization, collision avoidance communication, cooperative task coordination, competition scoring and monitoring, and spectator viewing interfaces.

5.8 Safety and Rate Limiting

All APIs implement comprehensive safety controls: speed and acceleration limits appropriate for educational settings, workspace restrictions preventing robots leaving designated areas, emergency stop mechanisms with hardware and software redundancy, timeout protections preventing runaway programs, and rate limiting preventing API abuse.

Age-Appropriate Safety Profiles

Safety settings adapt to student age and experience: Elementary (ages 5-11): maximum 30% motor speed, 2-minute continuous operation, desktop workspace only. Middle School (ages 12-14): maximum 60% speed, 5-minute operation, larger workspace with marked boundaries. High School (ages 15-18): maximum 85% speed, 10-minute operation, flexible workspace. University: configurable limits with instructor approval.

5.9 API Documentation and Developer Support

Comprehensive documentation is mandatory for all WIA-ROB-015 compliant APIs: interactive API reference with live examples, quick-start guides for common tasks, video tutorials for visual learners, code samples in multiple languages, troubleshooting guides for common errors, and community forums for peer support.

Documentation uses clear, educational language avoiding unnecessary jargon. Examples progress from simple to complex, supporting differentiated learning. All code samples are tested and maintained to ensure accuracy.

Key Takeaways

  • Multi-Level API Design: WIA-ROB-015 provides REST APIs for general control, WebSockets for real-time interaction, block-based interfaces for beginners, and Python/JavaScript SDKs for text programming.
  • Safety-First Architecture: All APIs implement age-appropriate safety limits, workspace restrictions, emergency stops, and rate limiting to ensure student safety.
  • Educational Error Handling: API errors provide pedagogical feedback with explanations, suggestions, and learning resources rather than cryptic technical messages.
  • Block-to-Code Bridges: Bidirectional translation between visual blocks and text code supports smooth progression from beginner to advanced programming.
  • LMS Integration: Deep integration with learning management systems via LTI and platform-specific APIs ensures robotics fits seamlessly into school digital ecosystems.
  • Real-Time Capabilities: WebSocket APIs enable live sensor streaming, multi-robot coordination, and interactive programming experiences with low latency.
  • Progressive Disclosure: APIs expose simple interfaces initially with progressive revelation of complexity as student skills advance.
  • Comprehensive Documentation: Interactive references, tutorials, code samples, and community support make APIs accessible to students and teachers.
  • Cross-Platform SDKs: Python and JavaScript SDKs work across all compliant robots, enabling platform-independent curriculum development.

Reference Standards and Implementation Notes

This standard is designed to interoperate with established international frameworks. Implementations SHOULD treat the following as the single source of truth and use this PHASE document as the interoperability layer that maps existing data into a machine-readable, signed, and auditable shape.

  • OpenAPI Specification 3.1 — REST API definition
  • JSON Schema 2020-12 — schema validation
  • ISO/IEC 27001:2022 — information security management
  • ISO/IEC 27018:2019 — PII protection in public clouds
  • ISO/IEC 17065:2012 — product certification bodies
  • CycloneDX 1.5 / SPDX 2.3 — software bill of materials
  • Sigstore (DSSE envelope, Rekor transparency log) — signature anchoring
  • in-toto Attestation Framework 1.0 — build provenance
  • IETF RFC 9700 — OAuth 2.1
  • IETF RFC 9457 — Problem Details for HTTP APIs
  • IETF RFC 8615 — well-known URIs

Practical Implementation Checklist

  • Manifest integrity — every published manifest is signed via Sigstore DSSE. Tampering is detectable without trusting an intermediary. Updates are issued as new manifests; the prior manifest is preserved with a supersede marker.
  • Retention — keep signed evidence for at least seven years to satisfy ISO/IEC 17065:2012 audit expectations. Some regulators require longer retention; in that case the deployment policy MUST extend the window rather than relying on the defaults.
  • Versioning — follow Semantic Versioning 2.0.0. Major version bumps require at least a 90-day overlap with the prior major version on every WIA-published reference implementation.
  • Negative-result reporting — a skipped indicator without recorded justification is treated as a failed indicator at audit time.
  • Key rotation — manifest signing keys are rotated per the deployment policy. The previous key is preserved in a published key history so that auditors can verify historical manifests offline.

Reference Topologies

Implementations declare their operational topology in the manifest: single-tenant edge (one runtime per organization), multi-tenant gateway (shared runtime with header-based isolation), federated mesh (multiple runtimes peer to one another and publish manifests to a directory service), or air-gapped batch (signed evidence package transported out-of-band). Each topology ships with reference test vectors so that implementations can validate their deployment matches the canonical pattern.

Open Governance

Issues, errata, and proposals are tracked at github.com/WIA-Official/wia-standards/issues with the appropriate label. The WIA Standards working group reviews open issues at the start of every minor release cycle and publishes the resulting decision log alongside the release notes. Errata are issued as patch releases; new normative requirements trigger minor bumps; backwards-incompatible changes trigger major bumps.
