Chapter 5

Phase 2: API Interface Standards

WIA-SEMI-001 Standard | Reading Time: ~35 minutes
Chapter Overview: Phase 2 builds on the data formats from Phase 1 to define standardized software interfaces for semiconductor control and monitoring. This chapter covers RESTful APIs, driver architectures, authentication mechanisms, and SDK design patterns that enable unified chip interaction.

5.1 API Design Principles

The WIA-SEMI-001 API interface standards follow industry best practices and modern design principles:

  • RESTful Architecture: HTTP-based APIs using standard methods (GET, POST, PUT, DELETE) for resource manipulation
  • Statelessness: Each request contains all necessary information, enabling scalability and reliability
  • Resource-Oriented: APIs expose chip capabilities as resources with consistent naming
  • Versioned: API versions in URL path enable evolution without breaking clients
  • Self-Documenting: OpenAPI/Swagger specifications provide machine-readable documentation
  • Security-First: Authentication, authorization, and encryption built in from the start

5.3 WebSocket Streaming API

For high-frequency monitoring, WIA-SEMI-001 supports WebSocket streaming: 

// WebSocket connection for real-time telemetry
const ws = new WebSocket('wss://chipname.local/api/v1/stream/telemetry');

ws.onopen = () => {
  // Subscribe to specific metrics
  ws.send(JSON.stringify({
    action: 'subscribe',
    metrics: ['power', 'temperature', 'frequency'],
    sampleRate: 100 // Hz

5.4 Driver Architecture

WIA-SEMI-001 defines a standardized driver architecture for major operating systems:

5.4.1 Linux Driver Interface

struct wia_semi_chip_info {
    const char *manufacturer;

// Power management operations
struct wia_semi_pm_ops {
    int (*set_power_mode)(struct device *dev, enum wia_power_mode mode);

// Register driver
int wia_semi_register_driver(struct wia_semi_driver *driver);
void wia_semi_unregister_driver(struct wia_semi_driver *driver);

5.5 SDK Design

WIA provides official SDKs for multiple languages, all following consistent patterns:

5.5.1 TypeScript/JavaScript SDK

import { WIASemiChip, PowerMode } from '@wia/semi-sdk';

// Connect to chip
const chip = new WIASemiChip({
  host: 'chipname.local',
  apiKey: process.env.WIA_API_KEY
});

// Get chip information
const info = await chip.getInfo();
console.log(`Connected to ${info.metadata.productName}`);

// Set power mode
await chip.config.setPowerMode(PowerMode.Performance);

// Monitor real-time telemetry
chip.monitor.on('telemetry', (data) => {
  console.log(`Power: ${data.power.value}W, Temp: ${data.temperature.value}°C`);
});

// Start streaming at 100Hz
await chip.monitor.startStreaming({
  metrics: ['power', 'temperature', 'frequency'],
  sampleRate: 100
});

5.5.2 Python SDK

from wia_semi import WIASemiChip, PowerMode

# Connect to chip
chip = WIASemiChip(
    host='chipname.local',
    api_key=os.environ['WIA_API_KEY']
)

# Get chip information
info = chip.get_info()
print(f"Connected to {info['metadata']['productName']}")

# Set power mode
chip.config.set_power_mode(PowerMode.PERFORMANCE)

# Get current temperature
temp = chip.monitor.get_temperature()
for sensor in temp['sensors']:
    print(f"{sensor['location']}: {sensor['temperature']['value']}°C")

# Stream telemetry
for telemetry in chip.monitor.stream_telemetry(sample_rate=100):
    print(f"Power: {telemetry['power']['value']}W")

5.6 Authentication and Security

WIA-SEMI-001 mandates secure authentication for all API access:

5.6.1 Authentication Methods

Method Use Case Security Level Token Lifetime
API Key Development, testing Medium No expiration
OAuth 2.0 Third-party applications High 1 hour (refreshable)
JWT Bearer Token Production systems High 15 minutes (refreshable)
Mutual TLS Critical infrastructure Very High Certificate lifetime

5.6.2 Authorization Scopes

Fine-grained permissions control what operations clients can perform:

  • chip:read - Read chip information and specifications
  • chip:monitor - Access real-time monitoring data
  • chip:config - Modify configuration parameters
  • chip:debug - Access debugging and diagnostic features
  • chip:admin - Full administrative access
// Example: OAuth 2.0 token request
POST /oauth/token HTTP/1.1
Host: auth.wia.org
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&client_id=your_client_id
&client_secret=your_client_secret
&scope=chip:read chip:monitor chip:config

HTTP/1.1 200 OK
Content-Type: application/json

{
  "access_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "scope": "chip:read chip:monitor chip:config"
}

5.7 Error Handling

Standardized error responses enable consistent error handling:

HTTP Status Error Code Description Example
400 INVALID_PARAMETER Invalid request parameter Frequency out of range
401 UNAUTHORIZED Missing or invalid authentication Expired token
403 FORBIDDEN Insufficient permissions Config requires admin scope
404 NOT_FOUND Resource doesn't exist Invalid sensor ID
429 RATE_LIMIT_EXCEEDED Too many requests Monitor API rate limit
503 THERMAL_THROTTLED Operation unavailable due to thermal state Can't increase frequency

Chapter Summary

Key Takeaways:

  1. Phase 2 defines RESTful APIs for chip information, configuration, and monitoring, providing unified software interfaces across all compliant chips.
  2. WebSocket streaming enables high-frequency real-time telemetry for power, temperature, and performance monitoring.
  3. Standardized driver interfaces for Linux and Windows simplify OS integration and enable consistent hardware abstraction.
  4. Official SDKs in multiple languages (TypeScript, Python, Java, C++) accelerate application development with consistent APIs.
  5. Comprehensive security through OAuth 2.0, JWT tokens, and mutual TLS protects chip access with fine-grained authorization scopes.
