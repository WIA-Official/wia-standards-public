제6장: 약국 관리 및 CPOE

"투약 오류는 환자 안전의 가장 큰 위협입니다. CPOE와 약국 시스템은 이를 70% 이상 감소시킵니다."

6.1 전산 처방 입력 (CPOE)

💊 CPOE란?

Computerized Physician Order Entry - 의사가 직접 전자 시스템으로 처방하는 시스템:

  • 약물 처방 — 약명, 용량, 용법, 투여 경로
  • 검사 처방 — 혈액, 영상, 병리 검사
  • 치료 처방 — 물리치료, 영양 상담 등
  • 간호 지시 — 활력징후 측정, 체위 변경 등

✅ CPOE의 장점

문제 종이 처방 CPOE
글씨 판독 필기체 오독 20% 오독 0%
용량 오류 10배 오류 빈번 자동 검증
알레르기 수동 확인 자동 경고
약물 상호작용 의사 기억 의존 실시간 체크

⚠️ 실시간 안전 체크

처방 시 자동으로 검증되는 항목들:

// 1. 알레르기 체크
if (약물.성분 in 환자.알레르기) {
    alert("🚫 알레르기 약물입니다! 페니실린 알레르기");
    return BLOCK;
}

// 2. 약물 상호작용
if (현재복용중.contains("와파린") && 신규처방 == "아스피린") {
    alert("⚠️ 출혈 위험 증가! 용량 조절 필요");
    return WARNING;
}

// 3. 용량 범위 체크
if (용량 > 최대권장용량) {
    alert("⚠️ 최대 권장 용량 초과: " + 최대권장용량);
    return WARNING;
}

// 4. 신장 기능 기반 용량 조절
if (GFR < 30 && 약물.신장배설) {
    자동권장용량 = 원용량 * 0.5;
    alert("ℹ️ 신기능 저하로 용량 조절 권장: " + 자동권장용량);
}

// 5. 중복 처방
if (동일성분약물이미처방) {
    alert("⚠️ 중복 처방 확인: 타이레놀 + 게보린 (아세트아미노펜)");
}

6.2 약국 정보 시스템

🏥 병원 약국 워크플로우

1. CPOE 처방 → 약국 시스템 전송 (HL7)
   ↓
2. 약사 검증 (상호작용, 용량 재확인)
   ↓
3. 조제 지시서 출력
   ↓
4. 자동 조제 or 수동 조제
   ↓
5. 조제 완료 → 바코드 부착
   ↓
6. 환자 인계 (바코드 스캔으로 환자 확인)
   ↓
7. 복약 지도
   ↓
8. HIS로 조제 완료 전송

🤖 자동 조제 시스템

로봇 조제 시스템의 장점:

  • 정확도 — 조제 오류 99.9% 감소
  • 속도 — 처방당 평균 30초 (수동은 3-5분)
  • 재고 관리 — 실시간 재고 추적
  • 추적성 — 약품별 로트 번호, 유효기한 관리

📦 재고 관리

Just-In-Time 재고 시스템:

약품 현재고 안전재고 상태
타이레놀 500mg 5,000정 2,000정 ✓ 정상
인슐린 주사 150개 300개 ⚠️ 발주 필요
항생제 주사 50개 200개 🚨 긴급 발주

💰 마약류 관리

특별 관리 약품 추적:

  • 이중 잠금 — 마약 금고 별도 보관
  • 처방 제한 — 마약류 처방 면허 확인
  • 투여 기록 — 투여 시간, 투여자 기록
  • 폐기 절차 — 2인 입회 하에 폐기 및 기록
  • 정부 보고 — 식약처에 월별 사용량 보고

6.3 바코드 및 RFID

📊 5-Rights 확인

바코드 시스템으로 투약 오류 방지:

  1. Right Patient — 환자 팔찌 바코드 스캔
  2. Right Drug — 약품 바코드 스캔
  3. Right Dose — 용량 자동 확인
  4. Right Route — 투여 경로 (경구/주사) 확인
  5. Right Time — 투약 시간 확인
간호사 투약 프로세스:
1. 환자 팔찌 스캔 → *삐* (환자 ID: P123456)
2. 약품 바코드 스캔 → *삐* (타이레놀 500mg)
3. 시스템 자동 확인:
   ✓ 올바른 환자
   ✓ 올바른 약물
   ✓ 올바른 시간 (14:00 투약 예정)
   ✓ 올바른 용량 (500mg)
   
   → "투약 가능" 표시

만약 불일치:
   🚫 "경고! 다른 환자의 약입니다"
   🚫 "경고! 투약 시간이 아닙니다 (2시간 후)"

참조 표준 및 적용 지침

본 표준은 다음 ALLOW 출처를 단일 진실원으로 삼아 데이터 교환·서명· 감사 증적 인터페이스를 제공합니다. 구현체는 이들 원전을 대체하지 않고, 데이터 교환·감사 증적·자동 검증을 위한 상호운용 계층을 정의합니다.

  • OpenAPI Specification 3.1 — REST API 정의
  • JSON Schema 2020-12 — 스키마 검증
  • ISO/IEC 27001:2022 — 정보보안 경영시스템
  • ISO/IEC 27018:2019 — 공용 클라우드의 PII 보호
  • ISO/IEC 17065:2012 — 제품 인증 기관
  • CycloneDX 1.5 / SPDX 2.3 — 소프트웨어 BOM
  • Sigstore (DSSE 봉투, Rekor 투명성 로그) — 서명 앵커링
  • in-toto Attestation Framework 1.0 — 빌드 출처 검증
  • IETF RFC 9700 — OAuth 2.1
  • IETF RFC 9457 — Problem Details for HTTP APIs
  • IETF RFC 8615 — well-known URIs

실무 적용 체크리스트

  • 매니페스트 무결성: 모든 발급은 SHA-256 또는 Sigstore 서명을 부착해 변조를 차단합니다. 발급 후 변경은 새 매니페스트로 처리 되며 이전 매니페스트는 supersede 마킹과 함께 보존됩니다.
  • 보관 기간: ISO/IEC 17065:2012 감사 기대치를 충족하기 위해 7년 이상의 보관 정책을 명시합니다. 관할 규정이 더 길면 그것을 따릅니다.
  • 버전 관리: 정정·확장은 시맨틱 버저닝(SemVer 2.0.0)을 따릅니다. 필수 요건의 추가는 마이너, 호환 단절은 메이저 버전 갱신을 트리거합니다. 메이저 변경은 최소 90일 병행 호환 기간을 거치며 그 기간 동안 신·구 두 버전 모두 WIA 등록부에 Stable로 표기 됩니다.
  • 부정결과 보고: 누락된 지표는 사유와 함께 매니페스트에 기록 되어야 합니다. 사유가 없는 누락은 감사 시 실패로 간주됩니다.
  • 키 회전: 매니페스트 서명 키는 배포 정책에 따라 회전됩니다. 이전 키는 공개된 키 이력에 보존되어 과거 매니페스트의 사후 검증을 보장합니다.

참조 토폴로지

구현체의 운용 토폴로지는 다음 4가지 중 하나로 매니페스트에 선언됩니다: 단일 테넌트 엣지(조직당 1개 런타임), 멀티 테넌트 게이트웨이(헤더 기반 격리로 다중 테넌트 서비스), 연합 메시(다수 런타임이 디렉토리 서비스 통해 상호 인식), 에어갭 배치(라이브 연결 없이 서명된 패키지를 out-of-band 채널로 전송). 각 토폴로지는 참조 테스트 벡터를 포함하며 구현체가 자신의 배포가 표준 패턴과 일치하는지 검증할 수 있도록 합니다.
