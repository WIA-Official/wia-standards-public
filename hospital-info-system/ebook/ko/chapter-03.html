제3장: 환자 관리 시스템 (PMS)

"환자의 병원 여정은 등록에서 시작됩니다. 첫 인상이 전체 경험을 좌우하며, 효율적인 PMS는 대기 시간을 줄이고 환자 만족도를 높입니다."

— 환자 중심 설계 원칙

3.1 환자 등록 (Patient Registration)

📋 주요 기능

환자 관리 시스템의 첫 번째 접점:

  • 신규 환자 등록 — 인구통계학적 정보 수집
  • MRN 발급 — 고유 의무기록번호 자동 생성
  • 보험 정보 — 건강보험, 실손보험 등록
  • 중복 환자 확인 — 이름, 생년월일, 주민번호로 기존 환자 검색
  • 신분증 스캔 — OCR로 자동 정보 입력

💳 환자 식별자

식별자 형식 예시
MRN 병원코드 + 8자리 SMC-12345678
건강보험번호 13자리 1234567890123
주민등록번호 13자리 (암호화) 850312-1******
바코드/RFID 환자 팔찌 *MRN12345678*

🔍 중복 환자 감지 알고리즘

동명이인 또는 오타로 인한 중복 등록을 방지합니다:

def find_duplicate_patients(new_patient):
    # 1. 주민번호 완전 일치 확인
    if exact_match_by_ssn(new_patient.ssn):
        return "EXACT_MATCH"

    # 2. 이름 + 생년월일 일치
    candidates = find_by_name_dob(
        new_patient.name,
        new_patient.birth_date
    )

    # 3. Fuzzy matching (오타 허용)
    for candidate in candidates:
        similarity = calculate_similarity(new_patient, candidate)
        # 이름 유사도 + 생년월일 일치 + 전화번호/주소 유사
        if similarity > 0.85:
            return "POTENTIAL_DUPLICATE", candidate

    return "NO_MATCH"

3.2 예약 시스템 (Appointment Scheduling)

📅 예약 유형

  • 외래 진료 — 일반 진료, 전문 클리닉
  • 검사 — 영상 검사, 내시경, 심전도 등
  • 시술/수술 — 수술실 스케줄링
  • 재활 치료 — 물리 치료, 작업 치료

🔔 자동 알림

No-show(예약 불이행) 감소 전략:

  • SMS/카카오톡 — 예약 1일 전, 2시간 전 알림
  • 이메일 — 예약 확인서 발송
  • 모바일 앱 푸시 — 실시간 알림
  • 자동 전화 — IVR 시스템으로 음성 안내

실제 효과: 자동 알림 시스템 도입 후 No-show율이 25%에서 8%로 감소했습니다.

🤖 AI 기반 예약 최적화

머신러닝으로 최적의 스케줄링:

  • 대기 시간 예측 — 과거 데이터로 진료 시간 예측
  • No-show 확률 — 환자별 불이행 가능성 예측
  • Overbooking — 예측 기반으로 적절한 초과 예약
  • 자원 할당 — 진료실, 의료진 최적 배치

3.3 입원/퇴원 관리 (ADT - Admission, Discharge, Transfer)

🏥 입원 프로세스

환자의 입원부터 퇴원까지 전체 여정 관리:

  1. 입원 접수
    • 진료과에서 입원 오더 발행
    • 병상 예약 및 배정
    • 입원 동의서 작성
  2. 병상 관리
    • 실시간 병상 가용성 모니터링
    • 환자 중증도별 병상 배정
    • 격리실 관리 (감염 환자)
  3. 전동 (Transfer)
    • 병동 간 이동 (일반 → ICU)
    • 병실 변경
    • 전동 이력 추적
  4. 퇴원
    • 퇴원 처방 및 약 처방
    • 퇴원 요약지 작성
    • 퇴원 후 관리 계획

🛏️ 병상 관리 대시보드

병동 총 병상 재원 가용 점유율
ICU 20 19 1 95%
내과 60 48 12 80%
외과 50 32 18 64%

📊 평균 재원 일수 (ALOS)

병상 회전율 최적화를 위한 지표:

ALOS = 총 재원 일수 / 퇴원 환자 수

예시:
- 1월 총 재원 일수: 1,800일
- 1월 퇴원 환자 수: 300명
- ALOS = 1,800 / 300 = 6일

목표:
- 일반 병동: 5-7일
- 중환자실: 3-5일
- 수술 후: 7-10일

   ↓

3.5 응급실 관리 (Emergency Department)

🚑 트리아지 (Triage)

환자 중증도 분류 시스템 (KTAS - Korean Triage and Acuity Scale):

등급 색상 상태 목표 시간
1단계 빨강 소생 (심정지) 즉시
2단계 주황 응급 (심근경색) 10분 이내
3단계 노랑 긴급 (골절) 30분 이내
4단계 녹색 준응급 (복통) 1시간 이내
5단계 파랑 비응급 (감기) 2시간 이내

🏃 Fast Track

경증 환자를 위한 신속 진료:

  • 대상 — 4-5단계 환자 (비응급)
  • 진료 — 전담 의사/간호사 배치
  • 목표 — 1시간 이내 진료 완료
  • 효과 — 응급실 혼잡도 30% 감소

3.6 환자 포털 (Patient Portal)

📱 모바일 앱 기능

환자가 직접 의료 정보를 관리합니다:

  • 나의 건강 기록
    • 진료 이력, 처방 내역
    • 검사 결과 조회
    • 영상 검사 이미지 (DICOM)
    • 예방 접종 기록
  • 예약 관리
    • 온라인 예약/변경/취소
    • 대기 순서 확인
    • 진료 알림
  • 원격 진료
    • 화상 상담
    • 채팅 상담
    • 전자 처방전 발급
  • 건강 관리
    • 혈압, 혈당 기록
    • 복약 알림
    • 건강 목표 설정

🔐 보안 및 프라이버시

환자 포털의 보안 요구사항:

  • 본인 인증 — 공동 인증서, 생체 인증
  • 데이터 암호화 — TLS 1.3, AES-256
  • 접근 로그 — 모든 조회 기록 저장
  • 타임아웃 — 5분 비활성 시 자동 로그아웃

환자 참여 효과: 포털 사용 환자는 예약 준수율 95%, 만성질환 관리 개선 40%, 응급실 방문 25% 감소

참조 표준 및 적용 지침

본 표준은 다음 ALLOW 출처를 단일 진실원으로 삼아 데이터 교환·서명· 감사 증적 인터페이스를 제공합니다. 구현체는 이들 원전을 대체하지 않고, 데이터 교환·감사 증적·자동 검증을 위한 상호운용 계층을 정의합니다.

  • OpenAPI Specification 3.1 — REST API 정의
  • JSON Schema 2020-12 — 스키마 검증
  • ISO/IEC 27001:2022 — 정보보안 경영시스템
  • ISO/IEC 27018:2019 — 공용 클라우드의 PII 보호
  • ISO/IEC 17065:2012 — 제품 인증 기관
  • CycloneDX 1.5 / SPDX 2.3 — 소프트웨어 BOM
  • Sigstore (DSSE 봉투, Rekor 투명성 로그) — 서명 앵커링
  • in-toto Attestation Framework 1.0 — 빌드 출처 검증
  • IETF RFC 9700 — OAuth 2.1
  • IETF RFC 9457 — Problem Details for HTTP APIs
  • IETF RFC 8615 — well-known URIs

실무 적용 체크리스트

  • 매니페스트 무결성: 모든 발급은 SHA-256 또는 Sigstore 서명을 부착해 변조를 차단합니다. 발급 후 변경은 새 매니페스트로 처리 되며 이전 매니페스트는 supersede 마킹과 함께 보존됩니다.
  • 보관 기간: ISO/IEC 17065:2012 감사 기대치를 충족하기 위해 7년 이상의 보관 정책을 명시합니다. 관할 규정이 더 길면 그것을 따릅니다.
  • 버전 관리: 정정·확장은 시맨틱 버저닝(SemVer 2.0.0)을 따릅니다. 필수 요건의 추가는 마이너, 호환 단절은 메이저 버전 갱신을 트리거합니다. 메이저 변경은 최소 90일 병행 호환 기간을 거치며 그 기간 동안 신·구 두 버전 모두 WIA 등록부에 Stable로 표기 됩니다.
  • 부정결과 보고: 누락된 지표는 사유와 함께 매니페스트에 기록 되어야 합니다. 사유가 없는 누락은 감사 시 실패로 간주됩니다.
  • 키 회전: 매니페스트 서명 키는 배포 정책에 따라 회전됩니다. 이전 키는 공개된 키 이력에 보존되어 과거 매니페스트의 사후 검증을 보장합니다.

참조 토폴로지

구현체의 운용 토폴로지는 다음 4가지 중 하나로 매니페스트에 선언됩니다: 단일 테넌트 엣지(조직당 1개 런타임), 멀티 테넌트 게이트웨이(헤더 기반 격리로 다중 테넌트 서비스), 연합 메시(다수 런타임이 디렉토리 서비스 통해 상호 인식), 에어갭 배치(라이브 연결 없이 서명된 패키지를 out-of-band 채널로 전송). 각 토폴로지는 참조 테스트 벡터를 포함하며 구현체가 자신의 배포가 표준 패턴과 일치하는지 검증할 수 있도록 합니다.
