제4장: 전자 건강 기록 (EHR) 통합

"완벽한 EHR 시스템은 환자의 전체 의료 여정을 하나의 타임라인으로 보여줍니다. 과거의 모든 진료 기록이 현재의 의사 결정을 돕습니다."

— EHR 설계 철학

4.1 EHR vs EMR: 차이점 이해

📊 비교표

특성 EMR EHR
범위 단일 병원/클리닉 여러 의료 기관 통합
데이터 공유 내부에서만 사용 기관 간 교환 가능
환자 접근 제한적 환자 포털 제공
표준화 병원 자체 코드 HL7 FHIR, CDA
목적 진료 기록 관리 전체 건강 관리

현대 병원 정보 시스템은 EMR을 핵심으로 하되, EHR 기능을 통해 외부와 연결됩니다.

🏷️ 의료 용어 표준화

상호 운용성을 위한 필수 코드 시스템:

  • ICD-10 — 질병 및 진단 코드
    • 예: I10 (본태성 고혈압), E11 (2형 당뇨병)
  • SNOMED CT — 임상 용어 체계
    • 73211009: 당뇨병
    • 38341003: 고혈압
  • LOINC — 검사 및 관찰 코드
    • 2345-7: 혈당 [mg/dL]
    • 718-7: 헤모글로빈
  • RxNorm — 약물 코드
    • 1191: 아스피린
    • 4917: 메트포르민

4.3 SOAP 노트 및 임상 문서

📝 SOAP 형식

표준화된 진료 기록 방식:

항목 설명 예시
S (Subjective) 환자의 호소 "3일 전부터 기침과 발열"
O (Objective) 객관적 소견 체온 38.5°C, 폐 청진 수포음
A (Assessment) 진단 및 평가 지역사회 획득 폐렴 의심
P (Plan) 치료 계획 흉부 X-ray, 항생제 투여

✍️ 임상 문서 아키텍처 (CDA)

HL7 CDA는 XML 기반의 표준 문서 형식입니다:

<ClinicalDocument xmlns="urn:hl7-org:v3">
  <typeId root="2.16.840.1.113883.1.3" extension="POCD_HD000040"/>
  <templateId root="2.16.840.1.113883.10.20.22.1.1"/>
  <id root="1.2.410.200060.1.1.1" extension="DOC_2024_001"/>
  <code code="34133-9" codeSystem="2.16.840.1.113883.6.1" 
        displayName="Summarization of Episode Note"/>
  <title>퇴원 요약지</title>
  <recordTarget>
    <patientRole>
      <id extension="P123456"/>
      <patient>
        <name>김민지</name>
        <birthTime value="19850312"/>
      </patient>
    </patientRole>
  </recordTarget>
  <!-- 진단, 처치, 처방 등 -->
</ClinicalDocument>

4.4 상호 운용성 (Interoperability)

🔗 HL7 FHIR (Fast Healthcare Interoperability Resources)

현대 EHR 통합의 표준:

GET /fhir/Patient/P123456

Response:
{
  "resourceType": "Patient",
  "id": "P123456",
  "identifier": [
    {
      "system": "http://hospital.org/mrn",
      "value": "SMC-12345678"
    }
  ],
  "name": [
    {
      "family": "김",
      "given": ["민지"]
    }
  ],
  "birthDate": "1985-03-12",
  "gender": "female"

🌐 HIE (Health Information Exchange)

지역 또는 국가 단위 의료 정보 교환:

  • 한국 - 차세대 전자의무기록 교류시스템
    • 진료 정보 교류 (EDI)
    • 응급 의료 정보 공유
    • 건강검진 결과 조회
  • 미국 - CommonWell, Carequality
    • 병원 간 EHR 교환
    • 환자 매칭 알고리즘

🔄 데이터 교환 시나리오

환자가 병원 A에서 병원 B로 전원 시:

1. 병원 A → HIE: FHIR Bundle 전송
   - Patient resource
   - Condition resources (진단)
   - MedicationStatement (투약)
   - Observation (검사 결과)

2. HIE → 병원 B: 환자 동의 확인 후 데이터 전송

3. 병원 B: 수신한 데이터를 로컬 EMR에 통합
   - 자동 매핑 (FHIR → 내부 DB)
   - 중복 확인
   - 의료진 검토 및 승인

실제 효과: HIE 도입 후 전원 환자의 중복 검사 50% 감소, 진료 시간 30분 단축

4.5 임상 의사 결정 지원 (CDSS)

🧠 CDSS 유형

유형 설명 예시
지식 기반 규칙 엔진 약물 상호작용 경고
비지식 기반 머신러닝 패혈증 조기 예측

⚠️ 실시간 경고 시스템

// 약물 처방 시 자동 체크
if (약물 == "워파린" && 환자.알레르기.contains("와파린")) {
    alert("⚠️ 알레르기 약물입니다!");
}

if (약물 == "NSAID" && 환자.만성질환.contains("위궤양")) {
    alert("⚠️ 금기 약물입니다. 위장 출혈 위험!");
}

if (신장기능 < 30 && 약물.requires_renal_adjustment) {
    alert("⚠️ 용량 조절이 필요합니다. 권장 용량: " + 조정용량);
}

📊 예측 분석

AI 기반 위험 예측:

  • 재입원 예측 — 퇴원 후 30일 내 재입원 확률 계산
  • 욕창 위험 — Braden Scale 자동 계산
  • 낙상 위험 — 나이, 약물, 보행 능력 분석
  • 패혈증 조기 경보 — 바이탈 사인 및 검사 결과 모니터링

4.6 데이터 품질 및 무결성

✅ 데이터 검증

입력 시점의 품질 보장:

  • 필수 필드 — 환자 ID, 진료 날짜, 진단 코드 필수 입력
  • 범위 검증 — 혈압 300/200은 입력 불가 (재확인 요청)
  • 형식 검증 — 주민번호 13자리, 전화번호 형식
  • 중복 방지 — 동일 검사 중복 오더 경고

🔍 데이터 품질 지표

지표 목표 측정 방법
완전성 > 95% 필수 필드 입력율
정확성 > 98% 오류율 (감사)
일관성 > 90% 표준 용어 사용율
적시성 < 24시간 진료 후 기록 완료

다음 장 예고

제5장에서는 검사실 정보 시스템(LIS)과 영상 정보 시스템(RIS/PACS)을 다룹니다. 검체 관리, 자동 분석 장비 연동, DICOM 표준, 영상 저장 및 전송을 학습합니다.

참조 표준 및 적용 지침

본 표준은 다음 ALLOW 출처를 단일 진실원으로 삼아 데이터 교환·서명· 감사 증적 인터페이스를 제공합니다. 구현체는 이들 원전을 대체하지 않고, 데이터 교환·감사 증적·자동 검증을 위한 상호운용 계층을 정의합니다.

  • OpenAPI Specification 3.1 — REST API 정의
  • JSON Schema 2020-12 — 스키마 검증
  • ISO/IEC 27001:2022 — 정보보안 경영시스템
  • ISO/IEC 27018:2019 — 공용 클라우드의 PII 보호
  • ISO/IEC 17065:2012 — 제품 인증 기관
  • CycloneDX 1.5 / SPDX 2.3 — 소프트웨어 BOM
  • Sigstore (DSSE 봉투, Rekor 투명성 로그) — 서명 앵커링
  • in-toto Attestation Framework 1.0 — 빌드 출처 검증
  • IETF RFC 9700 — OAuth 2.1
  • IETF RFC 9457 — Problem Details for HTTP APIs
  • IETF RFC 8615 — well-known URIs

실무 적용 체크리스트

  • 매니페스트 무결성: 모든 발급은 SHA-256 또는 Sigstore 서명을 부착해 변조를 차단합니다. 발급 후 변경은 새 매니페스트로 처리 되며 이전 매니페스트는 supersede 마킹과 함께 보존됩니다.
  • 보관 기간: ISO/IEC 17065:2012 감사 기대치를 충족하기 위해 7년 이상의 보관 정책을 명시합니다. 관할 규정이 더 길면 그것을 따릅니다.
  • 버전 관리: 정정·확장은 시맨틱 버저닝(SemVer 2.0.0)을 따릅니다. 필수 요건의 추가는 마이너, 호환 단절은 메이저 버전 갱신을 트리거합니다. 메이저 변경은 최소 90일 병행 호환 기간을 거치며 그 기간 동안 신·구 두 버전 모두 WIA 등록부에 Stable로 표기 됩니다.
  • 부정결과 보고: 누락된 지표는 사유와 함께 매니페스트에 기록 되어야 합니다. 사유가 없는 누락은 감사 시 실패로 간주됩니다.
  • 키 회전: 매니페스트 서명 키는 배포 정책에 따라 회전됩니다. 이전 키는 공개된 키 이력에 보존되어 과거 매니페스트의 사후 검증을 보장합니다.

참조 토폴로지

구현체의 운용 토폴로지는 다음 4가지 중 하나로 매니페스트에 선언됩니다: 단일 테넌트 엣지(조직당 1개 런타임), 멀티 테넌트 게이트웨이(헤더 기반 격리로 다중 테넌트 서비스), 연합 메시(다수 런타임이 디렉토리 서비스 통해 상호 인식), 에어갭 배치(라이브 연결 없이 서명된 패키지를 out-of-band 채널로 전송). 각 토폴로지는 참조 테스트 벡터를 포함하며 구현체가 자신의 배포가 표준 패턴과 일치하는지 검증할 수 있도록 합니다.
