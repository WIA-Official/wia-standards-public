2.1 HIS 아키텍처 패턴

🏗️ 3-Tier 아키�ekstur

전통적인 병원 정보 시스템은 3계층 구조를 따릅니다:

┌─────────────────────────────────────┐
│   Presentation Layer (프레젠테이션)   │
│   - 웹 브라우저, 모바일 앱              │
│   - 데스크톱 클라이언트                 │
│   - 의료 기기 인터페이스                │
└─────────────────────────────────────┘
              ↕
┌─────────────────────────────────────┐
│   Application Layer (애플리케이션)    │
│   - 비즈니스 로직                      │
│   - API 서버 (REST/FHIR)              │
│   - 임상 의사 결정 지원                │
└─────────────────────────────────────┘
              ↕
┌─────────────────────────────────────┐
│   Data Layer (데이터)                 │
│   - RDBMS (PostgreSQL, Oracle)       │
│   - NoSQL (MongoDB, Cassandra)       │
│   - Data Warehouse (BI/Analytics)    │
└─────────────────────────────────────┘

☁️ 마이크로서비스 아키텍처

현대적인 HIS는 마이크로서비스 패턴을 채택합니다:

  • Patient Service — 환자 등록, 인구통계학적 정보
  • Appointment Service — 예약 관리, 스케줄링
  • EHR Service — 전자 건강 기록, 진료 노트
  • Order Service — CPOE, 검사/처방 오더
  • Lab Service — 검사실 결과, LIS 통합
  • Imaging Service — RIS/PACS 통합
  • Pharmacy Service — 조제, 재고, 약물 관리
  • Billing Service — 청구, 수납, 보험 처리

🎯 장점

특성 Monolithic Microservices
배포 전체 시스템 다운타임 서비스별 독립 배포
확장성 수직 확장만 가능 서비스별 수평 확장
장애 격리 전체 시스템 영향 서비스별 격리
기술 스택 단일 기술 서비스별 최적 기술

2.2 데이터베이스 설계

🗃️ 관계형 데이터베이스 (RDBMS)

구조화된 의료 데이터를 위한 주요 테이블 설계:

-- 환자 테이블
CREATE TABLE patients (
    patient_id VARCHAR(20) PRIMARY KEY,
    mrn VARCHAR(20) UNIQUE NOT NULL,  -- Medical Record Number
    first_name VARCHAR(100) NOT NULL,
    last_name VARCHAR(100) NOT NULL,
    birth_date DATE NOT NULL,
    gender VARCHAR(10),
    ssn VARCHAR(11) ENCRYPTED,  -- 주민등록번호 암호화
    phone VARCHAR(20),
    email VARCHAR(100),
    address TEXT,
    emergency_contact JSONB,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

-- 진료 기록 테이블
CREATE TABLE encounters (
    encounter_id VARCHAR(20) PRIMARY KEY,
    patient_id VARCHAR(20) REFERENCES patients(patient_id),
    encounter_type VARCHAR(50),  -- inpatient, outpatient, emergency
    admission_date TIMESTAMP,
    discharge_date TIMESTAMP,
    department VARCHAR(100),
    attending_physician VARCHAR(20),
    chief_complaint TEXT,
    diagnosis_codes VARCHAR(50)[],  -- ICD-10 codes
    status VARCHAR(20),
    created_at TIMESTAMP DEFAULT NOW()
);

-- 처방 테이블
CREATE TABLE orders (
    order_id VARCHAR(20) PRIMARY KEY,
    encounter_id VARCHAR(20) REFERENCES encounters(encounter_id),
    order_type VARCHAR(20),  -- medication, lab, imaging, procedure
    order_code VARCHAR(50),
    order_name TEXT,
    ordered_by VARCHAR(20),
    order_datetime TIMESTAMP,
    priority VARCHAR(10),  -- stat, urgent, routine
    status VARCHAR(20),  -- pending, active, completed, cancelled
    details JSONB
);

2.4 클라우드 vs 온프레미스

☁️ 클라우드 배포

AWS, Azure, GCP를 활용한 HIS 구축:

  • 장점
    • 초기 자본 지출 절감 (CAPEX → OPEX)
    • 탄력적 확장성 (피크 시간 대응)
    • 재해 복구 용이 (다중 리전 백업)
    • 자동 패치 및 업데이트
  • 단점
    • 장기 운영 비용 높을 수 있음
    • 데이터 주권 문제 (해외 서버 저장)
    • 인터넷 의존성 (연결 끊김 시 마비)
    • 규제 준수 복잡성

🏢 온프레미스 배포

병원 내부 데이터센터에서 운영:

  • 장점
    • 완전한 데이터 통제권
    • 규제 준수 용이 (데이터 국내 보관)
    • 네트워크 지연 최소화
    • 장기적으로 비용 효율적
  • 단점
    • 높은 초기 투자 비용
    • 하드웨어 관리 및 유지보수 부담
    • 확장성 제한
    • 재해 복구 계획 복잡

🔀 하이브리드 접근

많은 병원이 하이브리드 모델을 채택합니다:

  • 온프레미스 — 핵심 EHR, PACS (저지연 필요)
  • 클라우드 — 백업, 아카이브, 분석, 환자 포털
  • 엣지 — 의료 기기 데이터 (실시간 처리)

실제 사례: 삼성서울병원은 온프레미스로 핵심 HIS를 운영하면서, AWS를 활용하여 의료 영상 아카이브 및 빅데이터 분석을 수행합니다.

2.6 보안 아키텍처

🛡️ 다층 보안 (Defense in Depth)

Layer 7: 사용자 교육 및 정책
Layer 6: 애플리케이션 보안 (OWASP Top 10 방어)
Layer 5: 데이터 보안 (암호화, 토큰화)
Layer 4: 네트워크 보안 (방화벽, IDS/IPS)
Layer 3: 호스트 보안 (안티바이러스, EDR)
Layer 2: 물리적 보안 (출입 통제, CCTV)
Layer 1: 정책 및 절차 (HIPAA, ISO 27001)

🔐 인증 및 권한 부여

  • MFA (다중 인증) — 비밀번호 + OTP/생체인증
  • SSO (Single Sign-On) — SAML 2.0, OAuth 2.0
  • RBAC (역할 기반) — 의사, 간호사, 행정직별 권한
  • ABAC (속성 기반) — 시간, 위치, 환자 관계 기반 접근

🔒 데이터 암호화

상태 암호화 방법 표준
저장 시 (At Rest) AES-256, TDE FIPS 140-2
전송 중 (In Transit) TLS 1.3 HTTPS, FTPS
사용 중 (In Use) 메모리 암호화 Intel SGX

2.7 재해 복구 및 비즈니스 연속성

참조 표준 및 적용 지침

본 표준은 다음 ALLOW 출처를 단일 진실원으로 삼아 데이터 교환·서명· 감사 증적 인터페이스를 제공합니다. 구현체는 이들 원전을 대체하지 않고, 데이터 교환·감사 증적·자동 검증을 위한 상호운용 계층을 정의합니다.

  • OpenAPI Specification 3.1 — REST API 정의
  • JSON Schema 2020-12 — 스키마 검증
  • ISO/IEC 27001:2022 — 정보보안 경영시스템
  • ISO/IEC 27018:2019 — 공용 클라우드의 PII 보호
  • ISO/IEC 17065:2012 — 제품 인증 기관
  • CycloneDX 1.5 / SPDX 2.3 — 소프트웨어 BOM
  • Sigstore (DSSE 봉투, Rekor 투명성 로그) — 서명 앵커링
  • in-toto Attestation Framework 1.0 — 빌드 출처 검증
  • IETF RFC 9700 — OAuth 2.1
  • IETF RFC 9457 — Problem Details for HTTP APIs
  • IETF RFC 8615 — well-known URIs

실무 적용 체크리스트

  • 매니페스트 무결성: 모든 발급은 SHA-256 또는 Sigstore 서명을 부착해 변조를 차단합니다. 발급 후 변경은 새 매니페스트로 처리 되며 이전 매니페스트는 supersede 마킹과 함께 보존됩니다.
  • 보관 기간: ISO/IEC 17065:2012 감사 기대치를 충족하기 위해 7년 이상의 보관 정책을 명시합니다. 관할 규정이 더 길면 그것을 따릅니다.
  • 버전 관리: 정정·확장은 시맨틱 버저닝(SemVer 2.0.0)을 따릅니다. 필수 요건의 추가는 마이너, 호환 단절은 메이저 버전 갱신을 트리거합니다. 메이저 변경은 최소 90일 병행 호환 기간을 거치며 그 기간 동안 신·구 두 버전 모두 WIA 등록부에 Stable로 표기 됩니다.
  • 부정결과 보고: 누락된 지표는 사유와 함께 매니페스트에 기록 되어야 합니다. 사유가 없는 누락은 감사 시 실패로 간주됩니다.
  • 키 회전: 매니페스트 서명 키는 배포 정책에 따라 회전됩니다. 이전 키는 공개된 키 이력에 보존되어 과거 매니페스트의 사후 검증을 보장합니다.

참조 토폴로지

구현체의 운용 토폴로지는 다음 4가지 중 하나로 매니페스트에 선언됩니다: 단일 테넌트 엣지(조직당 1개 런타임), 멀티 테넌트 게이트웨이(헤더 기반 격리로 다중 테넌트 서비스), 연합 메시(다수 런타임이 디렉토리 서비스 통해 상호 인식), 에어갭 배치(라이브 연결 없이 서명된 패키지를 out-of-band 채널로 전송). 각 토폴로지는 참조 테스트 벡터를 포함하며 구현체가 자신의 배포가 표준 패턴과 일치하는지 검증할 수 있도록 합니다.

거버넌스 및 변경 관리
