[MED-003] 제6장: Phase 3 - 통신 프로토콜

"보안 없는 통신은 위험한 통신이다"

Phase 3은 생체신호 데이터의 안전하고 효율적인 전송을 보장하는 통신 프로토콜을 정의합니다.

6.1 Bluetooth LE 프로토콜

6.1.1 GATT 서비스 정의

WIA-MED-003 전용 Bluetooth LE GATT 서비스:

Service UUID: 00001820-0000-1000-8000-00805f9b34fb
Service Name: WIA-MED-003 Vital Sign Service

Characteristics:

1. 기기 정보 (Read)
   UUID: 00002a1e-0000-1000-8000-00805f9b34fb
   - 제조사, 모델, 펌웨어 버전

2. 신호 구성 (Read/Write)
   UUID: 00002a1f-0000-1000-8000-00805f9b34fb
   - 샘플링 레이트, 필터 설정

3. 데이터 스트림 (Notify)
   UUID: 00002a20-0000-1000-8000-00805f9b34fb
   - 실시간 생체신호 데이터

4. 제어 명령 (Write)
   UUID: 00002a21-0000-1000-8000-00805f9b34fb
   - START, STOP, PAUSE, RESUME

5. 상태 (Notify)
   UUID: 00002a22-0000-1000-8000-00805f9b34fb
   - 배터리, 신호 품질, 오류

6.1.2 데이터 패킷 구조

BLE 패킷 (최대 20 bytes):

[Header 4 bytes][Payload N bytes][CRC 2 bytes]

Header:
- Byte 0: 패킷 유형 (0x01=ECG, 0x02=SpO2, etc.)
- Byte 1: 시퀀스 번호
- Byte 2-3: 페이로드 길이

Payload:
- 신호 데이터 (압축 또는 원본)

CRC:
- CRC-16 체크섬

예제 (ECG 데이터):
01 3A 00 0C 05 12 18 0A 15 1C 09 14 1B 08 A3 4F
│  │  │  │  └─────────────────────┘        └──┘
│  │  │  │       ECG 샘플 (12 bytes)       CRC
│  │  │  └─ 길이 (12)
│  │  └─ 길이 MSB
│  └─ 시퀀스 #58
└─ ECG (0x01)

6.1.3 연결 파라미터 최적화

파라미터 최소 최대 권장 용도

6.3 MQTT 프로토콜

6.3.1 토픽 구조

WIA-MED-003 MQTT 토픽 계층:

wia-med-003/
  ├─ {deviceId}/
  │   ├─ info                   # 기기 정보 (retained)
  │   ├─ status                 # 상태 업데이트
  │   ├─ data/
  │   │   ├─ ecg               # ECG 데이터
  │   │   ├─ spo2              # SpO2 데이터
  │   │   ├─ bp                # 혈압 데이터
  │   │   └─ hrv               # HRV 데이터
  │   ├─ alerts                # 알림
  │   └─ control               # 제어 명령
  └─ broadcast/
      └─ discovery             # 기기 발견

예제:
발행: wia-med-003/ECG-12345/data/ecg
구독: wia-med-003/+/data/ecg (모든 기기의 ECG)

6.3.2 QoS 레벨 선택

토픽 QoS 이유
data/* 0 실시간 데이터, 다음 샘플이 곧 도착
alerts 1 중요하지만 중복 허용
control 2 정확히 한 번 실행 필요
info 1 + retained 새 구독자가 즉시 확인 필요

// 서버 (Node.js + ws)
const WebSocket = require('ws');

const wss = new WebSocket.Server({
  port: 8080,
  verifyClient: (info, callback) => {
    // JWT 토큰 검증
    const token = new URL(info.req.url, 'ws://base').searchParams.get('token');
    jwt.verify(token, SECRET_KEY, (err, decoded) => {
      callback(!err, 200, 'OK');
    });
  }
});

wss.on('connection', (ws, req) => {
  const deviceId = new URL(req.url, 'ws://base').searchParams.get('deviceId');

  // 기기 스트림 시작
  const stream = startDeviceStream(deviceId);

  stream.on('data', (data) => {
    if (ws.readyState === WebSocket.OPEN) {
      ws.send(JSON.stringify(data));
    }
  });

  ws.on('message', (message) => {
    const command = JSON.parse(message);
    handleCommand(stream, command);
  });

  ws.on('close', () => {
    stream.stop();
  });
});

6.6 보안 및 암호화

6.6.1 종단간 암호화

// AES-256-GCM 암호화
const crypto = require('crypto');

function encryptData(data, key) {
  const iv = crypto.randomBytes(12);
  const cipher = crypto.createCipheriv('aes-256-gcm', key, iv);

  let encrypted = cipher.update(JSON.stringify(data), 'utf8', 'hex');
  encrypted += cipher.final('hex');

  const authTag = cipher.getAuthTag();

  return {
    iv: iv.toString('hex'),
    data: encrypted,
    authTag: authTag.toString('hex')
  };
}

function decryptData(encryptedData, key) {
  const decipher = crypto.createDecipheriv(
    'aes-256-gcm',
    key,
    Buffer.from(encryptedData.iv, 'hex')
  );

  decipher.setAuthTag(Buffer.from(encryptedData.authTag, 'hex'));

  let decrypted = decipher.update(encryptedData.data, 'hex', 'utf8');
  decrypted += decipher.final('utf8');

  return JSON.parse(decrypted);
}

// JWT 토큰 검증
function verifyToken(token) {
  try {
    const decoded = jwt.verify(token, PUBLIC_KEY, {
      algorithms: ['RS256'],
      issuer: 'https://auth.wia.live',
      audience: 'wia-med-003'
    });

    return {
      valid: true,
      deviceId: decoded.sub,
      permissions: decoded.scope.split(' ')
    };
  } catch (error) {
    return { valid: false, error: error.message };
  }
}
