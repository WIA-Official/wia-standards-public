Chapter 5: Phase 2 - API Interface Standard

5.1 VitalSignStream Class

class VitalSignStream {
  // Device Management
  async discoverDevices(filter?)
  async connect(deviceId)
  async disconnect(deviceId)
  
  // Data Streaming
  async startStream(deviceId, config?)
  async stopStream(streamId)
  onData(callback)
  
  // Data Storage
  async saveData(data, storage?)
  async loadData(query)
  async exportData(format, query)
  
  // Alerts
  setAlert(config)
  removeAlert(alertId)
  onAlert(callback)
}

5.2 TypeScript Example

import { VitalSignStream } from 'wia-med-003';

const stream = new VitalSignStream({
  apiKey: 'your-api-key'
});

// Discover devices
const devices = await stream.discoverDevices({
  signalType: 'ECG'
});

// Connect and stream
await stream.connect(devices[0].id);
await stream.startStream(devices[0].id);

stream.onData((data) => {
  console.log('ECG:', data.analysis?.heartRate, 'bpm');
});

// Set alert
stream.setAlert({
  signalType: 'ECG',
  condition: 'heartRate > 100 OR heartRate < 60',
  severity: 'high'
});

5.4 HTTP Surface — Discovery

In addition to the in-process SDK, every WIA Vital Sign Streaming host exposes an HTTP surface so that non-SDK consumers (clinical dashboards, EHR bridges, conformance test runners) can subscribe and replay without depending on a vendor SDK.

GET https://<host>/.well-known/wia-vital-sign-streaming

{
  "wia_vital_sign_streaming_version": "1.0.0",

    "snapshot": "https://stream.example/vss/snapshot",

5.5 Stream Subscribe (Server-Sent Events)

GET /vss/stream/{stream_id}?since={cursor}
Accept: text/event-stream
Authorization: WIA-Sig keyid="...",signature="..."

Each event payload is a Phase 1 frame envelope. Reconnecting subscribers pass the most recent event id as Last-Event-Id per WHATWG HTML §9.2.

5.6 Snapshot

GET /vss/snapshot/{stream_id}

Returns the most recent value per channel for low-rate clinical dashboards that do not need full waveform.

5.7 Replay

GET /vss/replay/{stream_id}?from={ts}&to={ts}
Accept: application/json

Returns historical frames within the requested window. Hosts MUST return a cursor when the window is too large for one response, with next_cursor in the body.

5.8 Alert Routing

POST /vss/alert
Content-Type: application/json
Authorization: WIA-Sig keyid="...",signature="..."

Body is a signed alert envelope; the host fans out to subscribed consumers (nurse station, on-call mobile, EHR notification queue).

5.9 Authentication and Errors

Writes (alert submissions, replay if private) require HTTP Message Signatures per IETF RFC 9421. Reads MAY be anonymous subject to the discovery rate limit; PHI-bearing reads MUST be authenticated. Errors follow IETF RFC 9457 problem details.

5.10 Conformance

A Phase 2 conformant host MUST publish discovery, implement stream subscribe over SSE with Last-Event-Id recovery, implement snapshot and replay endpoints, honour Idempotency-Key for alert submissions, and emit problem-detail JSON for 4xx/5xx.

5.11 Reference Subscribe / Reconnect Walkthrough

The most common failure case for SSE-based clinical streams is the wandering Wi-Fi link. The standard's design accommodates it with cursor-based recovery; the receiver does not lose data as long as the host honours its replay window.

GET /vss/stream/stream-001?since=fr_AAAA HTTP/1.1
Host: stream.example
Accept: text/event-stream
Authorization: WIA-Sig keyid="...",signature="..."

HTTP/1.1 200 OK
Content-Type: text/event-stream
Cache-Control: no-store

id: fr_AAAB
event: frame
data: { "wia_vital_sign_streaming_version":"1.0.0", "type":"frame", "channel":"ecg", "..." }

id: fr_AAAC
event: frame
data: { "wia_vital_sign_streaming_version":"1.0.0", "type":"frame", "channel":"ecg", "..." }

If the link drops after fr_AAAC, the receiver reconnects with Last-Event-Id: fr_AAAC. The host resumes streaming from fr_AAAD if available, or from the closest later cursor it still holds and emits a gap_marker for the missing range.
