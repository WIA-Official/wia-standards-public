1장: 데이터 통합 개요 및 패러다임

홍익인간 (弘益人間)

"널리 인간을 이롭게 하라"

데이터 통합은 조직의 데이터 사일로를 제거하고, 정보의 자유로운 흐름을 가능하게 하며, 모든 이해관계자가 더 나은 결정을 내릴 수 있도록 지원합니다. WIA-DATA-010 표준은 이러한 역량을 모든 규모의 조직이 활용할 수 있도록 보장합니다.

1.1 데이터 통합이란?

데이터 통합(Data Integration)은 여러 개의 서로 다른 데이터 소스에서 데이터를 추출하여 통합된 일관성 있는 뷰로 결합하는 프로세스입니다. 이를 통해 조직은 더 나은 의사결정, 분석, 운영 효율성을 달성할 수 있습니다.

[i] 정의: 데이터 통합은 이기종 데이터 소스를 연결하고, 데이터 품질을 보장하며, 일관성을 유지하고, 조직 전체에서 통합된 정보에 대한 시기적절한 접근을 제공하는 기술, 프로세스 및 모범 사례를 포괄합니다.

1.1.1 역사적 진화

데이터 통합은 수십 년에 걸쳐 크게 발전해왔습니다:

  • 1980년대-1990년대: 파일 전송, 스프레드시트, 사용자 정의 스크립트를 사용한 수동 데이터 통합. 데이터 웨어하우스가 중앙 집중식 저장소로 등장
  • 2000년대: 엔터프라이즈 애플리케이션 통합(EAI)과 ETL 도구로 통합 자동화. 서비스 지향 아키텍처(SOA)가 API 기반 통합 가능
  • 2010년대: 클라우드 컴퓨팅이 SaaS 플랫폼으로 통합을 혁신. 빅데이터 기술(Hadoop, Spark)이 대규모 통합 가능
  • 2020년대: 실시간 스트리밍, 데이터 메시 아키텍처, AI 기반 통합. 클라우드 네이티브 도구와 서버리스 플랫폼 우세

1.1.2 데이터 통합의 중요성

조직이 직면한 중요한 과제들을 데이터 통합이 해결합니다:

  • 데이터 사일로: 격리된 시스템에 갇힌 정보가 전체적인 분석과 의사결정을 방해
  • 일관성 없는 데이터: 시스템 간 동일한 정보의 다른 표현이 오류와 혼란 초래
  • 수동 프로세스: 시간 소모적인 복사-붙여넣기 작업이 리소스를 낭비하고 오류 발생
  • 실시간 요구사항: 비즈니스는 어제의 배치 보고서가 아닌 현재 데이터에 즉시 접근 필요
  • 규정 준수: 규제 요구사항이 정확하고 감사 가능한 데이터 계보와 거버넌스 요구
  • 확장성: 증가하는 데이터 양과 새로운 소스가 레거시 통합 방식 압도
  • 혁신: 고급 분석, AI/ML, 데이터 제품이 통합되고 고품질 데이터 필요

1.2 핵심 통합 패러다임

현대 데이터 통합은 서로 다른 사용 사례와 요구사항에 적합한 여러 가지 뚜렷한 패러다임을 사용합니다:

1.2.1 ETL (Extract, Transform, Load)

ETL은 데이터를 소스에서 추출하고, 별도의 처리 계층에서 변환한 후, 대상 시스템에 로드하는 전통적인 접근 방식입니다.

단계 설명 주요 활동 도구
추출(Extract) 소스 시스템에서 데이터 가져오기 데이터베이스, API, 파일 연결; 증분 또는 전체 데이터 읽기 JDBC/ODBC 커넥터, API 클라이언트, CDC 도구
변환(Transform) 데이터를 대상 형식으로 변환 정제, 검증, 중복 제거, 집계, 조인, 데이터 강화 SQL, Python, Spark, 사용자 정의 로직
로드(Load) 대상에 데이터 쓰기 대량 삽입, upsert, 대상 테이블에 병합 데이터베이스 드라이버, 클라우드 API, 파일 작성기

ETL 장점:

  • 로드하기 전에 변환이 발생하여 깨끗한 데이터만 대상에 도달
  • 대상 시스템을 처리 오버헤드로부터 보호
  • 잘 확립된 패턴과 성숙한 도구
  • 전문 ETL 서버로 복잡한 변환 처리 가능

ETL 한계:

  • 사전에 상당한 스키마 설계와 변환 로직 필요
  • 유연성 부족 - 변경 시 ETL 작업 업데이트 필요
  • ETL 서버가 대용량 데이터에 대한 병목 현상 발생
  • 변환이 로드 전에 발생하여 통찰 시간 느림

1.2.2 ELT (Extract, Load, Transform)

ELT는 원시 데이터를 먼저 대상 시스템에 로드한 후, 대상의 처리 능력을 사용하여 변환하는 현대적인 클라우드 기반 접근 방식입니다.

측면 ETL ELT
변환 위치 별도 ETL 서버/미들웨어 대상 데이터 웨어하우스 내부
적합한 대상 온프레미스, 레거시 시스템, 중간 데이터 볼륨 클라우드 데이터 웨어하우스 (Snowflake, BigQuery, Redshift)
성능 ETL 서버 용량에 제한 데이터 웨어하우스의 대규모 병렬 처리 활용
유연성 사전에 정의된 고정 변환 필요에 따라 변환, 동일한 원시 데이터에서 여러 뷰
데이터 보존 일반적으로 변환된 데이터만 저장 원시 데이터 보존, 언제든지 재변환 가능
통찰 시간 느림 - 변환을 기다려야 함 빠름 - 로드 즉시 데이터 사용 가능
비용 모델 ETL 서버 인프라 비용 클라우드 스토리지 + 컴퓨트 비용

ELT 장점:

  • 강력한 클라우드 데이터 웨어하우스 엔진 활용 (Snowflake, BigQuery, Redshift)
  • 더 빠른 데이터 가용성 - 즉시 로드, 나중에 변환
  • 여러 변환과 재처리를 위한 원시 데이터 보존
  • 더 유연함 - 재추출 없이 새로운 변환 생성 가능
  • 클라우드 인프라로 손쉬운 확장

1.2.3 실시간 스트리밍

스트리밍 통합은 이벤트가 발생할 때 데이터를 지속적으로 처리하여 실시간 분석과 즉각적인 대응을 가능하게 합니다.

주요 스트리밍 기술:

  • Apache Kafka: 고처리량, 내결함성 데이터 파이프라인을 위한 분산 이벤트 스트리밍 플랫폼
  • Apache Flink: 상태 유지 계산과 이벤트 시간 처리를 갖춘 스트림 처리 프레임워크
  • Apache Pulsar: 멀티 테넌시와 지리적 복제를 갖춘 클라우드 네이티브 메시징
  • AWS Kinesis: 실시간 데이터 수집 및 처리를 위한 관리형 스트리밍 서비스
  • Google Pub/Sub: 이벤트 기반 아키텍처를 위한 글로벌 메시징 서비스

스트리밍 사용 사례:

  • 금융 거래에서 실시간 사기 탐지
  • IoT 센서 데이터 처리 및 알림
  • 실시간 사용자 활동 추적 및 개인화
  • 실시간 재고 및 공급망 모니터링
  • 지속적으로 업데이트되는 데이터 웨어하우스를 위한 스트리밍 ETL
  • 이벤트 기반 마이크로서비스 통신

1.2.4 API 통합

API 기반 통합은 REST, GraphQL 또는 gRPC API를 통해 시스템을 직접 연결하여 동기식 데이터 교환을 수행합니다.

API 유형 프로토콜 최적 사용 예시
REST HTTP/JSON 표준 웹 서비스, CRUD 작업 Salesforce, Stripe, Shopify API
GraphQL HTTP/GraphQL 유연한 쿼리, 모바일 앱, 프론트엔드 데이터 가져오기 GitHub, Shopify, Contentful API
gRPC HTTP/2 + Protobuf 마이크로서비스, 고성능 내부 API Google Cloud 서비스, 내부 서비스
SOAP XML/HTTP 레거시 엔터프라이즈 시스템, 공식 계약 은행, 의료, 레거시 엔터프라이즈
Webhook HTTP 콜백 이벤트 알림, 실시간 업데이트 결제 확인, CI/CD 트리거

1.2.5 변경 데이터 캡처 (CDC)

CDC는 트랜잭션 로그를 읽어 실시간으로 데이터베이스 변경 사항(삽입, 업데이트, 삭제)을 캡처하여 효율적인 증분 동기화를 가능하게 합니다.

CDC 접근 방식:

  • 로그 기반 CDC: 데이터베이스 트랜잭션 로그 읽기 (MySQL binlog, PostgreSQL WAL)
  • 트리거 기반 CDC: 데이터베이스 트리거가 섀도 테이블에 변경 사항 캡처
  • 타임스탬프 기반 CDC: last_modified 타임스탬프 컬럼 기반 쿼리
  • 스냅샷 + 증분: 전체 스냅샷 후 증분 변경 사항

CDC 장점:

  • 최소한의 소스 시스템 영향으로 거의 실시간 데이터 복제
  • 삭제 포함 모든 변경 사항 캡처 (타임스탬프 기반 접근 방식과 비교)
  • 이벤트 기반 아키텍처와 데이터 스트리밍 가능
  • 대규모 데이터셋에 대해 전체 테이블 스캔보다 효율적

CDC 도구:

  • Debezium: 다양한 데이터베이스를 위한 오픈 소스 CDC 플랫폼
  • Maxwell: Kafka/Kinesis로의 MySQL CDC
  • Oracle GoldenGate: Oracle 및 이기종 데이터베이스를 위한 엔터프라이즈 CDC
  • AWS DMS: CDC 기능이 있는 데이터베이스 마이그레이션 서비스
  • Striim: CDC를 사용한 실시간 데이터 통합

1.3 현대 통합 아키텍처

1.3.1 데이터 메시 아키텍처

데이터 메시(Data Mesh)는 데이터를 제품으로 취급하고, 도메인 지향 소유권과 연합 거버넌스를 갖춘 분산형 접근 방식입니다.

데이터 메시의 네 가지 원칙:

  1. 도메인 지향 분산화: 중앙 IT가 아닌 비즈니스 도메인(영업, 마케팅, 재무)별 데이터 소유권
  2. 제품으로서의 데이터: 도메인 팀이 SLA, 문서, 품질 표준을 갖춘 제품으로 데이터 취급
  3. 셀프 서비스 데이터 인프라: 플랫폼 팀이 도메인 팀이 데이터를 관리할 수 있는 도구와 인프라 제공
  4. 연합 컴퓨테이셔널 거버넌스: 자동화된 정책이 도메인 자율성을 허용하면서 표준 시행

1.3.2 데이터 패브릭

데이터 패브릭(Data Fabric)은 메타데이터와 AI를 사용하여 분산 환경 전체에서 데이터와 연결 프로세스의 통합 계층을 생성합니다.

주요 구성 요소:

  • 능동적 메타데이터 관리 및 지식 그래프
  • AI/ML 기반 데이터 통합 및 권장 사항
  • 모든 소스에 대한 통합 데이터 액세스 계층
  • 자동화된 데이터 파이프라인 생성 및 최적화

1.3.3 람다 아키텍처

람다 아키텍처(Lambda Architecture)는 포괄적인 데이터 통합을 위해 배치 및 스트림 처리를 결합합니다.

세 가지 계층:

  • 배치 계층: 완전한 히스토리 데이터 처리, 배치 뷰 생성
  • 속도 계층: 즉각적인 통찰을 위해 실시간 데이터 스트림 처리
  • 서빙 계층: 쿼리를 위해 배치 및 실시간 뷰 병합

1.3.4 카파 아키텍처

카파 아키텍처(Kappa Architecture)는 실시간 및 히스토리 데이터 모두에 대해 스트림 처리만 사용하여 람다를 단순화합니다.

람다 대비 장점:

  • 두 개(배치 + 스트림) 대신 단일 코드베이스(스트림 처리)
  • 더 간단한 아키텍처 및 유지 관리
  • 더 빠른 개발 및 반복

1.4 데이터 통합 과제

조직은 데이터 통합을 구현할 때 수많은 과제에 직면합니다:

과제 설명 영향 완화 전략
데이터 품질 불완전하거나 부정확하거나 일관성 없는 데이터 잘못된 결정, 실패한 분석, 규정 준수 문제 데이터 프로파일링, 검증 규칙, 품질 메트릭
스키마 진화 시간이 지남에 따라 소스 시스템 스키마 변경 파이프라인 실패, 데이터 손실, 다운스트림 오류 스키마 버전 관리, 자동 감지, 유연한 스키마
성능 증가하는 데이터 볼륨으로 느린 통합 지연된 통찰, 증가된 비용, SLA 위반 증분 로드, 병렬 처리, 최적화
데이터 보안 통합 중 민감한 데이터 보호 침해, 규정 준수 위반, 법적 책임 암호화, 토큰화, 접근 제어, 감사
복잡성 수백 개의 소스 및 대상 관리 높은 유지 관리 비용, 기술 요구 사항, 취약성 표준화, 자동화, 관리형 서비스
비용 인프라, 도구 및 운영 비용 예산 초과, ROI 우려, 리소스 제약 클라우드 최적화, 오픈 소스, 점진적 접근

1.5 WIA-DATA-010 표준의 이점

WIA-DATA-010 표준은 데이터 통합을 위한 포괄적인 프레임워크를 제공합니다:

  • ✅ 상호 운용성: 표준 형식과 프로토콜이 도구 및 플랫폼 간 호환성 보장
  • ✅ 모범 사례: ETL, ELT, 스트리밍 및 API 통합을 위한 검증된 패턴
  • ✅ 보안: 내장된 암호화, 인증 및 권한 부여 사양
  • ✅ 거버넌스: 데이터 계보, 품질 및 규정 준수 프레임워크
  • ✅ 확장성: 메가바이트에서 페타바이트까지 데이터를 처리하는 아키텍처
  • ✅ 신뢰성: 오류 처리, 재시도 로직, 모니터링 및 알림 표준
  • ✅ 비용 효율성: 최적화된 리소스 사용 및 클라우드 비용 관리
  • ✅ 개방형 표준: 개방형 프로토콜과 형식으로 벤더 종속 방지
  • ✅ 접근성: 모든 규모의 조직이 고급 데이터 통합을 이용 가능

1.6 데이터 통합 시작하기

데이터 통합 구현은 구조화된 접근 방식을 따릅니다:

  1. 현재 상태 평가: 데이터 소스 목록 작성, 데이터 흐름 이해, 문제점 식별
  2. 요구사항 정의: 비즈니스 목표, 지연 시간 요구, 데이터 볼륨, 규정 준수 요구사항
  3. 아키텍처 선택: 요구사항에 따라 적절한 패턴(ETL, ELT, 스트리밍) 선택
  4. 도구 선택: 통합 도구 및 플랫폼 평가 및 선택
  5. 파이프라인 설계: 소스-대상 스키마 매핑, 변환 정의, 오류 처리 계획
  6. 구현: WIA-DATA-010 표준을 따라 파이프라인 구축
  7. 테스트: 현실적인 볼륨으로 데이터 품질, 성능, 오류 처리 검증
  8. 배포: 모니터링 및 알림과 함께 프로덕션에 롤아웃
  9. 모니터링 및 최적화: 메트릭 추적, 성능 최적화, 필요에 따라 개선
  10. 반복: 새 소스 추가, 변환 개선, 인프라 확장

요약

데이터 통합은 현대 조직이 사일로를 제거하고 분석을 활성화하며 혁신을 주도하는 데 필수적입니다. 이 장에서 다룬 내용:

  • 핵심 통합 패러다임: ETL, ELT, 스트리밍, API 및 CDC
  • 현대 아키텍처: 데이터 메시, 데이터 패브릭, 람다 및 카파
  • 주요 과제: 데이터 품질, 스키마 진화, 성능, 보안, 복잡성 및 비용
  • WIA-DATA-010 표준의 이점 및 구현 접근 방식

올바른 아키텍처는 조직 규모, 기술 성숙도, 데이터 볼륨, 지연 시간 요구사항 및 팀 역량에 따라 달라집니다. 많은 조직이 여러 패턴을 결합한 하이브리드 접근 방식을 사용합니다.
