홍익인간 (弘益人間)

"널리 인간을 이롭게 하라"

실시간 데이터 통합을 통해 즉각적인 인사이트를 얻고 빠른 의사결정을 지원하여 조직의 경쟁력을 강화합니다.

1. 실시간 데이터 통합 개요

실시간 데이터 통합은 데이터가 생성되는 즉시 처리하고 분석할 수 있도록 하는 아키텍처와 기술의 집합입니다. 전통적인 배치 처리 방식과 달리, 실시간 통합은 밀리초에서 초 단위의 지연시간으로 데이터를 처리합니다.

실시간 통합의 필요성

  • 즉각적인 비즈니스 인사이트: 실시간 데이터를 통해 현재 상황을 즉시 파악
  • 고객 경험 향상: 실시간 개인화 및 즉각적인 응답
  • 사기 탐지: 이상 거래를 실시간으로 감지하고 차단
  • 운영 효율성: 실시간 모니터링을 통한 문제 조기 발견
  • 경쟁 우위: 빠른 의사결정으로 시장 선점

배치 처리 vs 실시간 처리

비교 항목 배치 처리 실시간 처리
지연시간 분~시간 단위 밀리초~초 단위
데이터 크기 대용량 일괄 처리 작은 단위의 연속 처리
복잡도 상대적으로 단순 높은 복잡도
비용 상대적으로 저렴 높은 인프라 비용
사용 사례 일일 리포트, 월간 분석 사기 탐지, 실시간 추천
장애 복구 재처리 용이 복잡한 상태 관리 필요

2. Apache Kafka를 활용한 실시간 스트리밍

Apache Kafka는 실시간 데이터 파이프라인 구축을 위한 분산 스트리밍 플랫폼입니다. 높은 처리량, 낮은 지연시간, 내결함성을 제공합니다.

Kafka 아키텍처 핵심 개념

  • Producer: 메시지를 Kafka 토픽에 발행
  • Consumer: 토픽에서 메시지를 구독하고 처리
  • Topic: 메시지의 논리적 카테고리
  • Partition: 토픽의 물리적 분할 단위 (병렬 처리)
  • Broker: Kafka 서버 인스턴스
  • ZooKeeper/KRaft: 클러스터 메타데이터 관리

Kafka Producer 예제

from kafka import KafkaProducer
import json
from datetime import datetime

# Producer 설정
producer = KafkaProducer(
    bootstrap_servers=['localhost:9092'],
    value_serializer=lambda v: json.dumps(v).encode('utf-8'),
    acks='all',  # 모든 복제본이 확인할 때까지 대기
    retries=3,
    max_in_flight_requests_per_connection=1  # 순서 보장
)

# 사용자 이벤트 데이터 전송
def send_user_event(user_id, event_type, metadata):
    message = {
        'user_id': user_id,
        'event_type': event_type,
        'timestamp': datetime.utcnow().isoformat(),
        'metadata': metadata
    }

    # 파티션 키로 user_id 사용 (같은 사용자 이벤트는 같은 파티션)
    producer.send(
        'user-events',
        key=str(user_id).encode('utf-8'),
        value=message
    )
    producer.flush()
    print(f"Event sent: {message}")

# 예제 사용
send_user_event(
    user_id=12345,
    event_type='purchase',
    metadata={'product_id': 'P-5678', 'amount': 99.99}
)

    'production.inventory.orders',
    bootstrap_servers=['kafka:9092'],
    value_deserializer=lambda m: json.loads(m.decode('utf-8'))
)

for message in consumer:
    event = message.value

    # CDC 이벤트 구조
    operation = event['payload']['op']  # c: create, u: update, d: delete

        # 새로운 주문 생성

        order_data = event['payload']['before']
        handle_order_deletion(order_data)

4. 스트리밍 ETL 파이프라인

스트리밍 ETL은 전통적인 배치 ETL 패턴을 실시간으로 적용한 것입니다. Extract-Transform-Load를 연속적인 스트림으로 처리합니다.

Apache Flink를 활용한 스트리밍 ETL

from pyflink.datastream import StreamExecutionEnvironment
from pyflink.table import StreamTableEnvironment, DataTypes
from pyflink.table.descriptors import Schema, Kafka, Json

env = StreamExecutionEnvironment.get_execution_environment()
env.set_parallelism(4)
t_env = StreamTableEnvironment.create(env)

# 소스 테이블 정의 (Kafka)
t_env.execute_sql("""
    CREATE TABLE user_events (
        user_id BIGINT,
        event_type STRING,
        product_id STRING,
        amount DECIMAL(10, 2),
        event_time TIMESTAMP(3),
        WATERMARK FOR event_time AS event_time - INTERVAL '5' SECOND
    ) WITH (
        'connector' = 'kafka',
        'topic' = 'user-events',
        'properties.bootstrap.servers' = 'kafka:9092',
        'properties.group.id' = 'flink-consumer',
        'format' = 'json',
        'scan.startup.mode' = 'latest-offset'
    )
""")

# 싱크 테이블 정의 (PostgreSQL)
t_env.execute_sql("""
    CREATE TABLE user_activity_summary (
        user_id BIGINT,
        window_start TIMESTAMP(3),
        window_end TIMESTAMP(3),
        total_events BIGINT,
        total_amount DECIMAL(10, 2),
        PRIMARY KEY (user_id, window_start) NOT ENFORCED
    ) WITH (
        'connector' = 'jdbc',
        'url' = 'jdbc:postgresql://postgres:5432/analytics',
        'table-name' = 'user_activity_summary',
        'username' = 'admin',
        'password' = 'password'
    )
""")

# 스트리밍 변환 및 집계
t_env.execute_sql("""
    INSERT INTO user_activity_summary
    SELECT
        user_id,
        TUMBLE_START(event_time, INTERVAL '1' HOUR) as window_start,
        TUMBLE_END(event_time, INTERVAL '1' HOUR) as window_end,
        COUNT(*) as total_events,
        SUM(amount) as total_amount
    FROM user_events
    WHERE event_type = 'purchase'
    GROUP BY
        user_id,
        TUMBLE(event_time, INTERVAL '1' HOUR)
""")

Kafka Streams를 활용한 실시간 변환

Properties props = new Properties();
props.put(StreamsConfig.APPLICATION_ID_CONFIG, "order-enrichment");
props.put(StreamsConfig.BOOTSTRAP_SERVERS_CONFIG, "kafka:9092");

StreamsBuilder builder = new StreamsBuilder();

// 주문 스트림
KStream orders = builder.stream("orders");

// 고객 정보 테이블 (GlobalKTable)
GlobalKTable customers = builder.globalTable("customers");

// 주문을 고객 정보로 enrichment
KStream enrichedOrders = orders.join(
    customers,
    (orderId, order) -> order.getCustomerId(),
    (order, customer) -> new EnrichedOrder(
        order,
        customer.getName(),
        customer.getEmail(),
        customer.getSegment()
    )
);

// 고객 세그먼트별로 라우팅
enrichedOrders
    .filter((k, v) -> v.getCustomerSegment().equals("VIP"))
    .to("vip-orders");

enrichedOrders
    .filter((k, v) -> v.getCustomerSegment().equals("STANDARD"))
    .to("standard-orders");

KafkaStreams streams = new KafkaStreams(builder.build(), props);
streams.start();

5. 실시간 파이프라인 아키텍처 패턴

Lambda 아키텍처

Lambda 아키텍처는 배치 레이어와 속도 레이어를 결합하여 정확성과 실시간성을 모두 보장합니다.

  • 배치 레이어: 전체 데이터셋에 대한 정확한 계산
  • 속도 레이어: 실시간 데이터 처리 (근사치)
  • 서빙 레이어: 두 레이어의 결과를 병합하여 제공

Kappa 아키텍처

Kappa는 Lambda의 복잡성을 줄이고 스트림 처리만으로 모든 데이터를 처리합니다.

  • 단일 파이프라인: 모든 데이터를 스트림으로 처리
  • 재처리: 필요시 이벤트 로그를 재생하여 재계산
  • 간단한 운영: 하나의 코드베이스로 관리

아키텍처 비교

특징 Lambda Kappa
복잡도 높음 (두 개의 파이프라인) 낮음 (단일 파이프라인)
정확성 매우 높음 높음
지연시간 속도 레이어는 낮음 낮음
운영 비용 높음 중간
재처리 배치 레이어로 재계산 스트림 재생
적합한 경우 정확성이 매우 중요한 경우 운영 단순성이 중요한 경우

6. 실시간 데이터 품질 관리

실시간 파이프라인에서도 데이터 품질을 보장해야 합니다. 잘못된 데이터가 시스템에 들어오는 즉시 탐지하고 처리해야 합니다.

실시간 데이터 검증

from pyspark.sql import SparkSession
from pyspark.sql.functions import *

spark = SparkSession.builder \
    .appName("RealTimeValidation") \
    .getOrCreate()

# Kafka 스트림 읽기
df = spark \
    .readStream \
    .format("kafka") \
    .option("kafka.bootstrap.servers", "kafka:9092") \
    .option("subscribe", "orders") \
    .load()

# JSON 파싱 및 검증
from pyspark.sql.types import *

schema = StructType([
    StructField("order_id", StringType(), False),
    StructField("user_id", LongType(), False),
    StructField("amount", DoubleType(), False),
    StructField("timestamp", TimestampType(), False)
])

parsed_df = df.select(
    from_json(col("value").cast("string"), schema).alias("data")
).select("data.*")

# 데이터 품질 검증 규칙
valid_df = parsed_df.filter(
    (col("order_id").isNotNull()) &
    (col("user_id") > 0) &
    (col("amount") > 0) &
    (col("amount") < 100000) &  # 비정상 금액 필터링
    (col("timestamp").isNotNull())
)

# 유효하지 않은 데이터는 별도 토픽으로 전송
invalid_df = parsed_df.join(valid_df, "order_id", "left_anti")

# 유효 데이터 처리
valid_query = valid_df \
    .selectExpr("CAST(order_id AS STRING) as key", "to_json(struct(*)) as value") \
    .writeStream \
    .format("kafka") \
    .option("kafka.bootstrap.servers", "kafka:9092") \
    .option("topic", "validated-orders") \
    .option("checkpointLocation", "/tmp/checkpoint/valid") \
    .start()

# 무효 데이터 로깅
invalid_query = invalid_df \
    .selectExpr("CAST(order_id AS STRING) as key", "to_json(struct(*)) as value") \
    .writeStream \
    .format("kafka") \
    .option("kafka.bootstrap.servers", "kafka:9092") \
    .option("topic", "invalid-orders") \
    .option("checkpointLocation", "/tmp/checkpoint/invalid") \
    .start()

spark.streams.awaitAnyTermination()

    # 오프셋 커밋 (consumer-producer 패턴)
    # producer.send_offsets_to_transaction(...)

    # 트랜잭션 커밋

except Exception as e:

# 외부 체크포인트 유지 (작업 취소 시에도 보존)
from pyflink.datastream.checkpoint_config import ExternalizedCheckpointCleanup
env.get_checkpoint_config().enable_externalized_checkpoints(
    ExternalizedCheckpointCleanup.RETAIN_ON_CANCELLATION
)

# 상태 백엔드 설정 (RocksDB - 큰 상태에 적합)
from pyflink.datastream.state_backend import RocksDBStateBackend
env.set_state_backend(RocksDBStateBackend("hdfs://namenode:8020/flink/checkpoints"))

핵심 요점

  • 실시간 처리의 필요성: 즉각적인 인사이트와 빠른 의사결정을 위해 실시간 데이터 통합이 필수적입니다.
  • Kafka 생태계: Apache Kafka는 실시간 스트리밍의 핵심 플랫폼으로, 높은 처리량과 내결함성을 제공합니다.
  • CDC 기술: Debezium 등의 CDC 도구로 데이터베이스 변경사항을 실시간으로 캡처할 수 있습니다.
  • 스트리밍 처리: Flink, Kafka Streams 등으로 실시간 ETL과 복잡한 이벤트 처리가 가능합니다.
  • 아키텍처 선택: Lambda는 정확성, Kappa는 운영 단순성에 중점을 둡니다.
  • 데이터 품질: 실시간 환경에서도 데이터 검증과 품질 관리는 필수입니다.
  • 모니터링: 처리량, 지연시간, Consumer Lag 등을 지속적으로 모니터링해야 합니다.
  • 성능 최적화: 파티셔닝, 압축, 배치 처리 등으로 성능을 향상시킬 수 있습니다.
  • 장애 복구: Exactly-Once 시맨틱과 체크포인팅으로 데이터 일관성을 보장합니다.
  • 운영 복잡도: 실시간 파이프라인은 배치보다 운영이 복잡하므로 충분한 준비가 필요합니다.

요약

이 장에서는 실시간 데이터 통합의 핵심 개념과 구현 방법을 살펴보았습니다. Apache Kafka를 중심으로 한 스트리밍 아키텍처, CDC를 통한 데이터베이스 실시간 동기화, Flink와 Kafka Streams를 활용한 스트리밍 ETL, 그리고 Lambda/Kappa 아키텍처 패턴을 학습했습니다.

실시간 데이터 통합은 배치 처리보다 복잡하지만, 즉각적인 비즈니스 인사이트와 빠른 의사결정을 가능하게 합니다. 적절한 아키텍처 선택, 철저한 모니터링, 그리고 장애 복구 전략을 통해 안정적이고 확장 가능한 실시간 파이프라인을 구축할 수 있습니다.
