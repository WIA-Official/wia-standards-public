APIs democratize data access, enabling organizations to connect systems, share information, and build innovative solutions. Well-designed API integration empowers teams to move fast while maintaining security and reliability.

4.1 API Integration Fundamentals

API-based integration connects systems through Application Programming Interfaces, enabling real-time, synchronous data exchange.

4.1.1 Why API Integration?

APIs provide advantages over traditional file-based or database integration:

  • Real-Time: Immediate data access, no waiting for batch processing
  • Standardized: Well-defined contracts (OpenAPI, GraphQL schemas)
  • Secure: Authentication, authorization, rate limiting built-in
  • Scalable: Load balancers, caching, horizontal scaling
  • Language-Agnostic: Any language can call HTTP APIs
  • Versioned: API versions allow backward compatibility
  • Self-Service: Developers can integrate without database access

4.2 REST APIs

REST (Representational State Transfer) is the dominant API style for web services.

4.2.1 REST Principles

Principle Description Implementation
Client-Server Separation of concerns Frontend and backend evolve independently
Stateless No session state on server Each request contains all needed context (auth token)
Cacheable Responses can be cached Cache-Control headers, ETags for conditional requests
Uniform Interface Standardized interactions HTTP methods (GET, POST, PUT, DELETE), resource URIs
Layered System Intermediary layers allowed Load balancers, caches, proxies transparent to client

4.2.2 HTTP Methods for CRUD

HTTP Method CRUD Operation Idempotent Example
GET Read Yes GET /customers/123 - Retrieve customer
POST Create No POST /customers - Create new customer
PUT Update (full replace) Yes PUT /customers/123 - Replace customer
PATCH Update (partial) No PATCH /customers/123 - Update fields
DELETE Delete Yes DELETE /customers/123 - Remove customer

4.2.3 REST API Best Practices

  • Versioning: /v1/customers, /v2/customers for breaking changes
  • Pagination: limit/offset or cursor-based for large datasets
  • Filtering: /customers?status=active&created_after=2025-01-01
  • Sorting: /customers?sort=-created_at (desc by created_at)
  • Field Selection: /customers?fields=id,name,email (sparse fieldsets)
  • HATEOAS: Include links to related resources in responses
  • Error Handling: Consistent error format with HTTP status codes
  • Rate Limiting: Protect API with rate limits (X-RateLimit headers)

4.2.4 REST API Integration Example

import requests
import time

class SalesforceIntegration:
    def __init__(self, instance_url, access_token):
        self.base_url = f"{instance_url}/services/data/v55.0"
        self.headers = {
            'Authorization': f'Bearer {access_token}',
            'Content-Type': 'application/json'
        }

    def get_accounts(self, limit=200):
        """Fetch accounts with pagination"""
        accounts = []
        url = f"{self.base_url}/query"
        query = f"SELECT Id, Name, Industry FROM Account LIMIT {limit}"

        params = {'q': query}

        while url:
            response = requests.get(url, headers=self.headers, params=params)
            response.raise_for_status()

            data = response.json()
            accounts.extend(data['records'])

            # Handle pagination
            url = data.get('nextRecordsUrl')
            if url:
                url = self.base_url.replace('/services/data/v55.0', '') + url
                params = None  # params already in nextRecordsUrl

        return accounts

    def create_account(self, account_data):
        """Create new account"""
        url = f"{self.base_url}/sobjects/Account"
        response = requests.post(url, headers=self.headers, json=account_data)
        response.raise_for_status()
        return response.json()

    def sync_accounts_to_warehouse(self, db_connection):
        """Sync Salesforce accounts to data warehouse"""
        accounts = self.get_accounts()

        # Transform and load to warehouse
        for account in accounts:
            db_connection.upsert('accounts', {
                'salesforce_id': account['Id'],
                'name': account['Name'],
                'industry': account.get('Industry'),
                'synced_at': time.time()
            })

        print(f"Synced {len(accounts)} accounts")

4.3 GraphQL APIs

GraphQL is a query language for APIs enabling clients to request exactly the data they need.

4.3.1 GraphQL vs REST

Aspect REST GraphQL
Data Fetching Multiple endpoints, fixed responses Single endpoint, client specifies fields
Over-fetching Common (get more data than needed) None (request exactly what you need)
Under-fetching Common (need multiple requests) Rare (get related data in one query)
Versioning URL versioning (/v1, /v2) Field deprecation, no versions
Caching HTTP caching works well Requires custom caching
Learning Curve Low (familiar HTTP) Medium (new concepts)
Tooling Mature, widespread Growing, GraphiQL explorer

4.3.3 GraphQL Integration with Python

from gql import gql, Client
from gql.transport.requests import RequestsHTTPTransport

class GraphQLIntegration:
    def __init__(self, url, api_key):
        transport = RequestsHTTPTransport(
            url=url,
            headers={'Authorization': f'Bearer {api_key}'}
        )
        self.client = Client(transport=transport)

    def fetch_customers_with_orders(self):
        query = gql('''
            query {
                customers(first: 100) {
                    edges {
                        node {
                            id
                            name
                            email
                            orders(first: 50) {
                                edges {
                                    node {
                                        id
                                        total
                                        createdAt
                                    }
                                }
                            }
                        }
                    }
                }
            }
        ''')

        result = self.client.execute(query)
        return result['customers']['edges']

4.4 Webhooks

Webhooks enable real-time event notifications via HTTP callbacks.

4.4.1 Webhook Workflow

  1. Consumer registers webhook URL with provider
  2. Event occurs in provider system (e.g., payment received)
  3. Provider sends HTTP POST to webhook URL with event data
  4. Consumer processes event and returns 200 OK
  5. Provider retries if non-200 response

4.4.2 Webhook Best Practices

  • Idempotency: Handle duplicate webhook deliveries
  • Signature Verification: Verify webhook authenticity (HMAC signatures)
  • Quick Response: Return 200 OK quickly, process async
  • Retry Logic: Implement exponential backoff for retries
  • Dead Letter Queue: Store failed webhooks for investigation
  • Monitoring: Track webhook delivery success rates

4.4.3 Webhook Handler Example

from flask import Flask, request, jsonify
import hmac
import hashlib
import json

app = Flask(__name__)

@app.route('/webhooks/stripe', methods=['POST'])
def stripe_webhook():
    payload = request.data
    sig_header = request.headers.get('Stripe-Signature')

    # Verify webhook signature
    if not verify_stripe_signature(payload, sig_header):
        return jsonify({'error': 'Invalid signature'}), 401

    event = json.loads(payload)

    # Process different event types
    if event['type'] == 'payment_intent.succeeded':
        payment_intent = event['data']['object']
        handle_successful_payment(payment_intent)

    elif event['type'] == 'customer.subscription.created':
        subscription = event['data']['object']
        handle_new_subscription(subscription)

    # Return 200 quickly
    return jsonify({'status': 'received'}), 200

def verify_stripe_signature(payload, sig_header):
    webhook_secret = 'whsec_...'  # From Stripe
    expected_sig = hmac.new(
        webhook_secret.encode(),
        payload,
        hashlib.sha256
    ).hexdigest()

    received_sig = sig_header.split(',')[1].split('=')[1]
    return hmac.compare_digest(expected_sig, received_sig)

4.5 gRPC APIs

gRPC is a high-performance RPC framework using HTTP/2 and Protocol Buffers.

4.5.1 gRPC Benefits

  • Performance: Binary protocol (Protobuf) vs. JSON, HTTP/2 multiplexing
  • Streaming: Bi-directional streaming support
  • Type Safety: Strongly-typed Protobuf schemas
  • Code Generation: Auto-generate client/server code
  • Multi-Language: SDKs for 10+ languages

4.5.2 When to Use gRPC

Scenario REST gRPC
Internal microservices ✓ ✓✓ (better performance)
Public-facing API ✓✓ (better for browsers) ✗ (limited browser support)
Real-time streaming ✗ (WebSockets needed) ✓✓ (native support)
Mobile apps ✓ ✓✓ (lower latency/bandwidth)
IoT devices ✓ ✓✓ (efficient binary)

4.6 API Integration Patterns

4.6.1 Polling vs. Webhooks

Aspect Polling Webhooks
How it Works Periodically check for updates Provider pushes updates to you
Latency Delay up to poll interval Near real-time
Efficiency Wasteful (many empty checks) Efficient (only when events occur)
Implementation Simple (just schedule API calls) Complex (endpoint setup, security)
Best For Batch updates, simple integrations Real-time events, notifications

4.6.2 API Gateway Pattern

An API Gateway sits between clients and microservices, providing:

  • Single entry point for all client requests
  • Request routing to appropriate microservices
  • Authentication and authorization
  • Rate limiting and throttling
  • Response caching
  • Request/response transformation
  • Monitoring and logging

Popular API Gateways: Kong, AWS API Gateway, Azure API Management, Apigee

4.7 API Security

4.7.1 Authentication Methods

Method How It Works Use Cases
API Keys Static secret in header/query Simple service-to-service auth
OAuth 2.0 Token-based authorization User-delegated access (Google, Facebook)
JWT Self-contained signed tokens Stateless authentication
mTLS Mutual TLS certificates High-security service-to-service

4.7.2 API Security Best Practices

  • HTTPS Only: Encrypt all API traffic with TLS
  • Rate Limiting: Prevent abuse with per-client limits
  • Input Validation: Validate all inputs, prevent injection attacks
  • Least Privilege: Grant minimal necessary permissions
  • Logging & Monitoring: Track API usage, detect anomalies
  • API Versioning: Deprecate old versions gracefully
  • Secrets Management: Never hardcode API keys, use vaults

4.8 API Monitoring & Reliability

Critical metrics for API integration:

  • Uptime: API availability (target: 99.9%+)
  • Latency: Response times (p50, p95, p99)
  • Error Rate: 4xx and 5xx errors percentage
  • Throughput: Requests per second handled
  • Rate Limit Usage: How close to limits

Summary

API-based integration provides real-time, secure, and scalable data exchange:

  • REST: Dominant standard for web APIs, HTTP + JSON
  • GraphQL: Flexible querying, request exactly what you need
  • Webhooks: Real-time event notifications
  • gRPC: High-performance for microservices

Choose the right API style based on your needs: REST for public APIs, GraphQL for flexible clients, gRPC for internal services, webhooks for events. Implement robust security, monitoring, and error handling for production reliability.
