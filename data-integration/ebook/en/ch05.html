5.1 Introduction to Real-time Integration

Real-time data integration processes data continuously as events occur, enabling immediate insights and actions rather than waiting for batch processing windows.

5.1.1 Real-time vs Batch Processing

Aspect Batch Processing Real-time Processing
Latency Hours to days Seconds to milliseconds
Data Volume Process entire datasets Process events as they arrive
Complexity Lower - scheduled jobs Higher - continuous processing
Resource Usage Periodic spikes Consistent steady state
Cost Lower for high volumes Higher infrastructure costs
Use Cases Reporting, analytics, data warehousing Fraud detection, IoT, personalization

5.2 Streaming Platforms

5.2.1 Apache Kafka

Apache Kafka is the dominant distributed event streaming platform for building real-time data pipelines and streaming applications.

Core Concepts:

  • Topics: Named streams of events (e.g., "user-clicks", "orders")
  • Producers: Applications that publish events to topics
  • Consumers: Applications that subscribe to topics and process events
  • Partitions: Topics divided into partitions for parallelism
  • Consumer Groups: Multiple consumers sharing topic processing
  • Retention: Events stored for configured time period

Kafka Advantages:

  • High throughput: millions of events per second
  • Fault tolerance: replicated across multiple brokers
  • Scalability: add brokers to scale horizontally
  • Durability: events persisted to disk
  • Replay: consumers can re-read historical events
  • Ecosystem: Kafka Connect, Kafka Streams, ksqlDB

5.3 Stream Processing Frameworks

5.3.1 Apache Flink

Apache Flink is a stream processing framework for stateful computations over unbounded and bounded data streams.

Flink Features:

  • True Streaming: Event-at-a-time processing, not micro-batching
  • Stateful: Maintain state across events (counters, aggregations)
  • Event Time: Process based on event timestamps, not processing time
  • Exactly-Once: Guarantees with checkpointing and recovery
  • Low Latency: Millisecond processing latencies
  • Windowing: Tumbling, sliding, session windows

5.3.2 Kafka Streams

Kafka Streams is a lightweight library for building streaming applications on top of Kafka.

Benefits:

  • Simple: Java/Scala library, no separate cluster
  • Kafka-native: Deeply integrated with Kafka
  • Stateful processing: Local state stores with changelog
  • Exactly-once: Leverages Kafka transactions
  • Interactive queries: Query state stores

5.3.3 ksqlDB

ksqlDB provides SQL interface for stream processing on Kafka.

Example - Continuous Query:

-- Create stream from Kafka topic
CREATE STREAM user_clicks (
    user_id VARCHAR,
    page VARCHAR,
    timestamp BIGINT
) WITH (
    kafka_topic='clicks',
    value_format='json'
);

-- Real-time aggregation - clicks per user per hour
CREATE TABLE clicks_per_hour AS
SELECT 
    user_id,
    COUNT(*) AS click_count,
    WINDOWSTART AS window_start
FROM user_clicks
WINDOW TUMBLING (SIZE 1 HOUR)
GROUP BY user_id
EMIT CHANGES;

5.4 Change Data Capture (CDC)

CDC captures database changes in real-time by reading transaction logs, enabling efficient event-driven architectures.

5.4.1 CDC Approaches

Approach How It Works Pros Cons
Log-Based CDC Read database transaction logs Low overhead, captures all changes including deletes Requires log access, database-specific
Trigger-Based Database triggers write to shadow tables Works on any database Performance impact, requires schema changes
Timestamp-Based Query by last_modified timestamp Simple, universal Misses deletes, clock issues, polling overhead

5.4.2 Debezium for CDC

Debezium is an open-source CDC platform supporting MySQL, PostgreSQL, MongoDB, SQL Server, and more.

Debezium publishes change events to Kafka topics, which can be consumed by downstream applications.

5.5 Real-time Integration Patterns

5.5.1 Event Sourcing

Event Sourcing stores all changes as a sequence of immutable events.

Benefits:

  • Complete audit trail of all state changes
  • Temporal queries: "what was state at time T?"
  • Replayability: rebuild state by replaying events
  • Event-driven: naturally integrates with streaming

5.5.2 CQRS (Command Query Responsibility Segregation)

CQRS separates write (command) and read (query) models.

Pattern:

  1. Commands (writes) go to event store
  2. Events published to stream (Kafka)
  3. Read models (materialized views) built from events
  4. Queries read from optimized read models

5.5.3 Outbox Pattern

Outbox Pattern ensures reliable event publishing with database changes.

Steps:

  1. Application writes business data + event to outbox table in same transaction
  2. CDC tool reads outbox table
  3. Publishes events to Kafka
  4. Deletes processed events from outbox

This guarantees atomic database update + event publishing (no dual writes problem).

5.8 Challenges and Best Practices

5.8.1 Challenges

  • Complexity: More moving parts than batch
  • Debugging: Harder to debug distributed stream processing
  • State Management: Maintaining state across failures
  • Ordering: Preserving event order at scale
  • Late Events: Handling events that arrive out of order
  • Backpressure: Handling downstream slow consumers

5.8.2 Best Practices

  • Idempotent Processing: Handle duplicate events gracefully
  • Schema Evolution: Use schema registry (Confluent Schema Registry)
  • Monitoring: Track lag, throughput, error rates
  • Backpressure Handling: Implement rate limiting and buffering
  • Dead Letter Queues: Capture poison messages
  • Watermarks: Handle late events with event time watermarks
  • Testing: Test with realistic event volumes and patterns

Summary

Real-time data integration enables immediate insights and actions:

  • Streaming Platforms: Kafka, Kinesis, Pub/Sub for event transport
  • Stream Processing: Flink, Kafka Streams, ksqlDB for transformations
  • CDC: Debezium for real-time database change capture
  • Patterns: Event Sourcing, CQRS, Outbox for reliability

Real-time integration is more complex than batch but essential for modern use cases like fraud detection, personalization, IoT, and operational analytics.
