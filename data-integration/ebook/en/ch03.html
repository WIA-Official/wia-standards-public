Chapter 3: Data Integration Architecture Patterns

Hongik Ingan (弘益人間)

"Benefit All Humanity"

Architecture patterns are the foundation of scalable, maintainable data integration systems. Choosing the right pattern ensures your data infrastructure serves your organization's needs today and adapts to tomorrow's challenges.

3.1 Layered Data Architecture

Modern data architectures organize data into distinct layers, each serving specific purposes:

3.1.1 Bronze-Silver-Gold (Medallion Architecture)

The Medallion Architecture organizes data into three progressive quality layers:

Layer Purpose Data State Transformations Consumers
Bronze (Raw) Landing zone for raw data Exactly as received from source None - preserve original state Data engineers, debugging, reprocessing
Silver (Clean) Cleaned, validated, conformed Standardized formats, validated, deduplicated Data quality, type conversion, deduplication Analytics engineers, data scientists
Gold (Curated) Business-ready analytics Aggregated, joined, business logic applied Business calculations, aggregations, joins BI tools, business users, ML models

Benefits of Medallion Architecture:

  • Clear separation of concerns: raw preservation, quality improvement, business value
  • Incremental quality improvement: data gets cleaner as it moves through layers
  • Auditability: Can trace back to raw data for debugging or compliance
  • Reprocessing: Can rebuild Silver/Gold layers from Bronze if business logic changes
  • Performance: Each layer optimized for its consumers (raw, clean, aggregated)

3.1.2 Staging-ODS-Data Warehouse

Traditional enterprise data architecture with three tiers:

  • Staging Area: Temporary landing zone for extraction, minimal transformation
  • ODS (Operational Data Store): Integrated, current operational data for real-time reporting
  • Data Warehouse: Historical, analytical data warehouse with dimensional models

3.2 Data Mesh Architecture

Data Mesh is a paradigm shift from centralized data platforms to decentralized, domain-oriented data ownership.

3.2.1 Four Principles of Data Mesh

Principle Description Implementation
1. Domain Ownership Domain teams own their data end-to-end Sales team owns customer data, Finance owns transactions
2. Data as a Product Treat data like a product with SLAs, docs, quality Documented schemas, quality metrics, consumer support
3. Self-Serve Platform Platform team provides tools, not data pipelines Infrastructure for domains to build their own products
4. Federated Governance Automated policies, not central control Data contracts, automated quality checks, compliance tests

3.2.2 Data Mesh vs. Centralized Data Platform

Aspect Centralized Data Platform Data Mesh
Ownership Central data team owns all pipelines Domain teams own their data products
Scaling Central team becomes bottleneck Scales with organization (N teams)
Domain Knowledge Data team learns all domains Domain experts build their data
Agility Slow (central queue of requests) Fast (domains move independently)
Complexity Central platform complexity Distributed complexity

3.3 Lambda Architecture

Lambda Architecture combines batch and real-time processing for comprehensive data integration.

3.3.1 Three Layers

Layer Purpose Technologies Characteristics
Batch Layer Process complete historical data Hadoop, Spark, Hive High latency, accurate, complete
Speed Layer Real-time processing of new data Kafka Streams, Flink, Storm Low latency, approximate, incremental
Serving Layer Merge batch and real-time views Cassandra, HBase, Druid Query interface, unified view

3.3.2 Lambda Architecture Benefits and Challenges

Benefits:

  • Best of both worlds: complete batch processing + real-time updates
  • Fault tolerance: Batch layer can recover from speed layer failures
  • Accurate results: Batch layer corrects approximate real-time results

Challenges:

  • Complexity: Maintain two separate codebases (batch + stream)
  • Consistency: Ensure batch and speed layers produce same results
  • Operational overhead: Monitor and maintain two processing systems

3.4 Kappa Architecture

Kappa Architecture simplifies Lambda by using only stream processing.

3.4.1 Core Concept

Everything is a stream:

  • Real-time data: Events processed as they arrive
  • Historical data: "Replay" historical events through same stream processing
  • Single codebase: Same processing logic for both real-time and batch

3.3.2 Kappa vs. Lambda

Aspect Lambda Kappa
Processing Paradigms Batch + Stream Stream only
Codebase Two separate codebases Single codebase
Complexity Higher Lower
Historical Processing Batch layer Replay stream
Best For Complex batch + real-time needs Stream-first organizations

3.5 Data Fabric Architecture

Data Fabric creates an integrated layer across distributed data environments using metadata and AI.

3.5.1 Core Components

  • Active Metadata Management: Automated metadata collection, lineage tracking, knowledge graphs
  • Data Catalog: Unified search and discovery across all data sources
  • Data Virtualization: Query across sources without moving data
  • Data Integration: Unified pipelines for ETL, ELT, streaming
  • AI/ML Automation: Intelligent recommendations, auto-mapping, quality detection

3.5.2 Data Fabric vs. Data Mesh

Aspect Data Fabric Data Mesh
Approach Technology-driven integration layer Organizational and architectural paradigm
Focus Unified access through metadata and automation Decentralized ownership with federated governance
Implementation Central platform with AI/ML capabilities Domain teams build data products on shared infrastructure
Data Movement Often virtualized (query in place) Domains publish data products

3.6 Modern Data Stack Architecture

The Modern Data Stack is a cloud-native, composable set of tools for data integration and analytics.

3.6.1 Modern Data Stack Components

Layer Purpose Popular Tools
Data Sources Production databases, SaaS apps, APIs PostgreSQL, Salesforce, Stripe, Google Analytics
Ingestion Extract and load data (ELT) Fivetran, Airbyte, Stitch, Meltano
Storage Cloud data warehouse Snowflake, BigQuery, Redshift, Databricks
Transformation SQL-based transformations dbt, Dataform, Matillion
Orchestration Schedule and monitor workflows Airflow, Prefect, Dagster, dbt Cloud
BI & Analytics Dashboards and reporting Looker, Tableau, Power BI, Metabase
Reverse ETL Push warehouse data back to SaaS tools Hightouch, Census, Polytomic

3.6.2 Modern Data Stack Principles

  • Cloud-Native: SaaS tools, no infrastructure management
  • ELT Over ETL: Leverage warehouse compute power
  • SQL-First: Transformations in SQL, not Python/Java
  • Modularity: Best-of-breed tools, not monolithic suites
  • Version Control: Git for transformation code (dbt models)
  • Self-Service: Analysts can build pipelines without engineering

3.7 Event-Driven Architecture

Event-Driven Architecture (EDA) integrates systems through asynchronous event streams.

3.7.1 Event-Driven Patterns

Pattern Description Use Cases
Event Notification Notify other systems when something happens User signup → send welcome email
Event-Carried State Transfer Event contains all state needed by consumers Order placed event with full order details
Event Sourcing Store all changes as immutable events Banking transactions, audit logs
CQRS Separate read and write models Write to event store, read from projections

3.7.2 Event Streaming Platforms

  • Apache Kafka: Distributed log for event streaming, high throughput
  • Amazon Kinesis: Managed streaming on AWS
  • Google Pub/Sub: Global messaging for event-driven systems
  • Azure Event Hubs: Big data streaming on Azure
  • Apache Pulsar: Multi-tenant, geo-replicated streaming

3.8 Choosing the Right Architecture

Decision matrix for selecting data integration architecture:

Your Situation Recommended Architecture
Small team, cloud data warehouse, SaaS sources Modern Data Stack (Fivetran + Snowflake + dbt)
Large organization with multiple domains Data Mesh
Real-time analytics requirements Kappa Architecture with Kafka + Flink
Both batch and real-time needs Lambda Architecture or Lakehouse
Multiple cloud and on-premise sources Data Fabric
Microservices with event-driven flows Event-Driven Architecture with Kafka
Progressive data quality improvement Medallion Architecture (Bronze-Silver-Gold)

Summary

Modern data integration leverages diverse architectural patterns:

  • Layered architectures: Medallion (Bronze-Silver-Gold) for progressive quality
  • Data Mesh: Decentralized, domain-oriented data ownership
  • Lambda/Kappa: Batch and/or stream processing
  • Data Fabric: Unified access through metadata and AI
  • Modern Data Stack: Cloud-native, composable tools
  • Event-Driven: Asynchronous integration through events

The right architecture depends on your organization size, technical maturity, data volumes, latency requirements, and team capabilities. Many organizations use hybrid approaches combining multiple patterns.
