--------------------------------------------------------------------------------
7.1 후견인 네트워크 개요
--------------------------------------------------------------------------------

후견인은 신원 복구 결정에 참여할 권한이 있는 신뢰 엔터티입니다.
잘 구성된 후견인 네트워크는 수십 년의 보존 후에도
신원 복원이 가능하도록 보장합니다.

[ID] 후견인 유형:

    PERSON           -> 대상자가 신뢰하는 개인
    ORGANIZATION     -> 법인 (로펌, 신탁 회사)
    SMART_CONTRACT   -> 자율적 온체인 후견인

[ID] 후견인 기록:

    Guardian {
      guardianId:          "GUARD-2025-001"
      guardianType:        "PERSON"
      name:                "홍길순"
      contactInfo:         "[암호화됨]"
      threshold:           1       // 복구 시 가중치
      relationship:        "배우자"
      status:              "ACTIVE"
      createdAt:           "2025-01-01"
      lastVerified:        "2025-03-01"
    }

[ID] 후견인 선정 기준:

    - 장기간 연락 가능성
    - 신뢰성과 관계의 깊이
    - 기술적 이해도
    - 지리적 분산 (다양한 위치)


--------------------------------------------------------------------------------
7.2 임계값 구성
--------------------------------------------------------------------------------

임계값 복구는 최소 수의 후견인 승인을 요구합니다.

[ID] 임계값 모델:

    2-of-3:   3명 중 2명의 후견인 필요
    3-of-5:   5명 중 3명의 후견인 필요
    4-of-7:   7명 중 4명의 후견인 필요

[ID] 구성 예시:

    RecoveryConfig {
      identityId:       "CRYOID-2025-001"
      threshold:        3
      totalGuardians:   5
      waitingPeriod:    72      // 실행 전 대기 시간 (시간)
      emergencyBypass:  false   // 우회 불허
    }

[ID] 권장 구성:

    +---------------------------------------------------+
    | 보존 기간      | 권장 모델    | 후견인 유형       |
    +---------------------------------------------------+
    | 10년 미만      | 2-of-3       | 개인 위주         |
    | 10-50년        | 3-of-5       | 개인 + 기관       |
    | 50년 이상      | 4-of-7       | 기관 + 스마트계약 |
    +---------------------------------------------------+


--------------------------------------------------------------------------------
7.3 복구 시작
--------------------------------------------------------------------------------

[ID] 복구 요청:

    RecoveryRequest {
      requestId:    "RECOVERY-2025-001"
      identityId:   "CRYOID-2025-001"
      requester:    "홍명동"
      reason:       "부활 준비를 위한 신원 검증"
      status:       "PENDING"
      threshold:    3
      approvals:    []
      rejections:   []
      initiatedAt:  "2025-06-01T10:00:00Z"
      expiresAt:    "2025-06-08T10:00:00Z"  // 7일
    }

[ID] 시작 워크플로:

    1. 요청자가 복구 요청 제출
    2. 시스템이 요청자 신원 검증
    3. 모든 후견인에게 등록된 채널로 알림
    4. 대기 기간 시작 (선택적)
    5. 후견인들이 결정 제출

[ID] 요청자 검증:

    RequesterVerification {
      requesterName:     "홍명동"
      relationship:      "자녀"
      verificationMethod: ["신분증", "가족관계증명서"]
      verifiedBy:        "WIA 검증 서비스"
      verifiedAt:        "2025-06-01T09:00:00Z"
    }


--------------------------------------------------------------------------------
7.4 후견인 결정 프로세스
--------------------------------------------------------------------------------

[ID] 후견인 결정:

    GuardianDecision {
      requestId:         "RECOVERY-2025-001"
      guardianId:        "GUARD-2025-001"
      decision:          "APPROVE"  // APPROVE | REJECT
      timestamp:         "2025-06-02T14:30:00Z"
      verificationProof: "0x7f8e9a..."
      notes:             "화상 통화로 요청자 신원 확인"
    }

[ID] 결정 요구사항:

    - 후견인의 다중 인증
    - 결정에 대한 암호화 서명
    - 선택적: 검증 증명 (영상, 대면)
    - 시간 기록 및 불변 로깅

[ID] 결정 타임라인:

    시작          1일차         3일차         7일차
      |-------------|-------------|-------------|
      |  알림 발송  |             |             |
      |             |  첫 승인    |             |
      |             |             |  임계값도달 |
      |             |             |             | 만료
      +-------------+-------------+-------------+


--------------------------------------------------------------------------------
7.5 복구 실행
--------------------------------------------------------------------------------

[ID] 복구 상태 전환:

    PENDING   --> 후견인 알림 발송
        |
        v
    AWAITING  --> 후견인 결정 수집 중
        |
        +----> APPROVED (임계값 도달)
        |         |
        |         v
        |      EXECUTING --> 자격증명 발급
        |         |
        |         v
        |      COMPLETED --> 접근 허용
        |
        +----> REJECTED (임계값 도달 불가)
        |
        +----> EXPIRED (시간 제한 초과)

[ID] 복구 자격증명:

    RecoveryCredentials {
      requestId:    "RECOVERY-2025-001"
      identityId:   "CRYOID-2025-001"
      issuedTo:     "홍명동"
      issuedAt:     "2025-06-03T10:00:00Z"
      validFor:     "24h"
      permissions:  ["IDENTITY_ACCESS", "CREDENTIAL_RECOVERY"]
      auditLog:     [/* 후견인 결정 내역 */]
    }

[ID] 복구 완료 알림:

    RecoveryNotification {
      type:        "RECOVERY_COMPLETED"
      identityId:  "CRYOID-2025-001"
      requestId:   "RECOVERY-2025-001"
      timestamp:   "2025-06-03T10:00:00Z"
      recipients:  ["모든 후견인", "보존 시설"]
    }


--------------------------------------------------------------------------------
7.6 후견인 유지관리
--------------------------------------------------------------------------------

[ID] 유지관리 작업:

    - 연간 후견인 검증
    - 연락처 정보 업데이트
    - 후견인 교체 절차
    - 임계값 조정 (다중 후견인 승인 필요)

[ID] 후견인 교체:

    후견인을 교체하려면:
    1. 기존 후견인들이 교체에 투표
    2. 임계값 승인 필요
    3. 새 후견인 온보딩
    4. 이전 후견인 제거
    5. 모든 변경사항 로깅

    GuardianReplacement {
      oldGuardianId:  "GUARD-2025-002"
      newGuardianId:  "GUARD-2025-006"
      reason:         "기존 후견인 사망"
      approvedBy:     ["GUARD-001", "GUARD-003", "GUARD-004"]
      effectiveDate:  "2025-07-01"
    }

[!] 후견인 실패 시나리오:

    - 후견인 사망: 교체 절차 시작
    - 후견인 연락 불가: 타임아웃 후 남은 후견인으로 진행
    - 후견인 침해: 임계값에 의한 긴급 철회
