제6장: Phase 3 - 통신 프로토콜

弘益人間 · 널리 인간을 이롭게 하라

개요

2021년, 도쿄의 teamLab Borderless 미술관에서 기술적 문제가 발생했습니다. 50개의 인터랙티브 작품이 네트워크로 연결되어 있었는데, 센서 데이터를 전송하던 중 일부 작품에서 200ms 이상의 지연이 발생했습니다. 관람객이 움직여도 작품이 늦게 반응하면서 몰입감이 깨졌고, 일부 작품은 아예 동기화가 깨져서 서로 다른 시간대의 데이터를 표시했습니다. 문제는 각 작품이 서로 다른 통신 프로토콜(UDP, TCP, WebSocket)을 사용하고 있었고, 시간 동기화 메커니즘도 통일되지 않았다는 것이었습니다.

WIA-ART-010의 Phase 3는 바로 이러한 통신과 동기화 문제를 해결합니다. OSC(Open Sound Control), WebSocket, MQTT 등 검증된 프로토콜을 기반으로 실시간 센서 데이터 전송 표준을 정의하고, NTP 기반 시간 동기화, 네트워크 토폴로지, 에러 처리 전략을 제시합니다. 이 장에서는 대규모 분산 인터랙티브 아트 시스템의 통신 아키텍처를 실제 구현 예제와 함께 상세히 살펴봅니다.

1. WIA-ART-010 통신 프로토콜 개요

1.1 프로토콜 선택 기준

WIA-ART-010은 세 가지 주요 통신 프로토콜을 지원하며, 각각 특정 사용 사례에 최적화되어 있습니다:

프로토콜 전송 계층 지연시간 신뢰성 주요 용도
OSC UDP < 5ms 낮음 실시간 센서/오디오
WebSocket TCP < 20ms 높음 브라우저 통합, 양방향 통신
MQTT TCP < 50ms 매우 높음 IoT 센서, 원격 모니터링
HTTP/REST TCP < 100ms 높음 설정 변경, 상태 조회

1.2 프로토콜 선택 가이드

작품의 요구사항에 따라 적절한 프로토콜을 선택합니다:

  • OSC: 깊이 카메라, 모션 센서 등 초저지연이 필요한 실시간 센서 데이터
  • WebSocket: 웹 기반 인터랙티브 작품, 실시간 대시보드, 모바일 앱 연동
  • MQTT: 대규모 센서 네트워크, 신뢰성이 중요한 환경, IoT 디바이스
  • HTTP/REST: 작품 설정 변경, 메타데이터 조회, 비실시간 제어

2. OSC (Open Sound Control) 프로토콜

2.1 OSC 기본 구조

OSC는 음악 공연과 멀티미디어 아트를 위해 설계된 경량 통신 프로토콜입니다. UDP 기반으로 초저지연을 제공하며, 주소 기반 메시지 라우팅을 지원합니다.

WIA-ART-010 OSC 메시지 형식

// 센서 데이터 전송
Address: /wia/art-010/sensor/sensor-001/skeleton
Arguments:
  - person_id (string): "person-1"
  - joint_name (string): "head"
  - x (float): 0.5
  - y (float): 1.7
  - z (float): 3.2
  - confidence (float): 0.95

// OSC Bundle (다중 메시지)
Bundle timestamp: 1705478400523
Messages:
  /wia/art-010/sensor/sensor-001/skeleton "person-1" "head" 0.5 1.7 3.2 0.95
  /wia/art-010/sensor/sensor-001/skeleton "person-1" "left_hand" 0.3 1.2 3.1 0.92
  /wia/art-010/sensor/sensor-001/skeleton "person-1" "right_hand" 0.7 1.2 3.1 0.93

// 출력 장치 제어
Address: /wia/art-010/output/led-strip-001/color
Arguments:
  - hue (int): 120
  - saturation (int): 100
  - value (int): 80

2.2 OSC 주소 체계

WIA-ART-010은 계층적 OSC 주소 구조를 정의합니다: 

/wia/art-010/{category}/{device-id}/{data-type}/{parameter}

예시:
/wia/art-010/sensor/sensor-001/skeleton/head/x
/wia/art-010/sensor/sensor-001/gesture/type
/wia/art-010/sensor/sensor-001/crowd/count
/wia/art-010/output/led-strip-001/brightness
/wia/art-010/output/led-strip-001/color/hue
/wia/art-010/interaction/mode-001/active

2.3 Node.js OSC 구현 예제

import { Client, Server } from 'node-osc';

// OSC 서버 (센서 데이터 수신)
const oscServer = new Server(7400, '0.0.0.0');

oscServer.on('message', (msg) => {
  const [address, ...args] = msg;

  // WIA-ART-010 센서 메시지 파싱
  if (address.startsWith('/wia/art-010/sensor/')) {
    const parts = address.split('/');
    const sensorId = parts[4];
    const dataType = parts[5];

    if (dataType === 'skeleton') {
      const [personId, jointName, x, y, z, confidence] = args;
      console.log(`Sensor ${sensorId}: ${jointName} at (${x}, ${y}, ${z})`);

      // 손 높이로 LED 색상 제어
      if (jointName === 'left_hand') {
        const hue = mapRange(y, 0.5, 2.5, 0, 360);
        sendOSC('/wia/art-010/output/led-strip-001/color', [hue, 100, 100]);
      }
    }
  }
});

// OSC 클라이언트 (출력 장치 제어)
const oscClient = new Client('192.168.1.100', 7500);

function sendOSC(address, args) {
  oscClient.send(address, ...args, (err) => {
    if (err) console.error('OSC send error:', err);
  });
}

function mapRange(value, inMin, inMax, outMin, outMax) {
  return (value - inMin) * (outMax - outMin) / (inMax - inMin) + outMin;
}

console.log('WIA-ART-010 OSC Server listening on port 7400');

3. WebSocket 프로토콜

3.2 Node.js WebSocket 서버 구현

import { WebSocketServer } from 'ws';

const wss = new WebSocketServer({ port: 8080 });

// 연결된 모든 클라이언트 추적
const clients = new Set();

wss.on('connection', (ws, req) => {
  const clientId = req.headers['sec-websocket-key'].substring(0, 8);
  console.log(`Client ${clientId} connected`);

  clients.add(ws);

  // 클라이언트로부터 메시지 수신
  ws.on('message', (data) => {
    try {
      const message = JSON.parse(data);

      // WIA-ART-010 프로토콜 검증
      if (message.protocol !== 'wia-art-010-websocket') {
        ws.send(JSON.stringify({
          type: 'error',
          message: 'Invalid protocol'
        }));
        return;
      }

      // 메시지 타입별 처리
      switch (message.type) {
        case 'output_command':
          handleOutputCommand(message);
          break;

        case 'subscribe_sensor':
          // 센서 구독 등록
          ws.sensorSubscriptions = ws.sensorSubscriptions || [];
          ws.sensorSubscriptions.push(message.sensor_id);
          break;
      }
    } catch (err) {
      console.error('Message parsing error:', err);
    }
  });

  // 연결 종료
  ws.on('close', () => {
    clients.delete(ws);
    console.log(`Client ${clientId} disconnected`);
  });
});

// 센서 데이터를 구독 중인 클라이언트에게 브로드캐스트
function broadcastSensorData(sensorId, data) {
  const message = {
    protocol: 'wia-art-010-websocket',
    version: '1.0',
    type: 'sensor_data',
    sensor_id: sensorId,
    timestamp: Date.now() / 1000,
    data: data
  };

  const messageStr = JSON.stringify(message);

  clients.forEach(client => {
    if (client.readyState === WebSocket.OPEN) {
      if (!client.sensorSubscriptions ||
          client.sensorSubscriptions.includes(sensorId)) {
        client.send(messageStr);
      }
    }
  });
}

// 출력 장치 제어 처리
function handleOutputCommand(message) {
  const { output_id, command, params } = message;

  // 실제 하드웨어 제어 로직
  console.log(`Controlling ${output_id}: ${command}`, params);

  // 확인 응답 전송
  broadcastToAll({
    type: 'output_status',
    output_id: output_id,
    status: 'success'
  });
}

function broadcastToAll(message) {
  const messageStr = JSON.stringify(message);
  clients.forEach(client => {
    if (client.readyState === WebSocket.OPEN) {
      client.send(messageStr);
    }
  });
}

console.log('WIA-ART-010 WebSocket server running on ws://localhost:8080');

4. MQTT 프로토콜

4.1 MQTT 토픽 구조

MQTT는 Publish/Subscribe 패턴으로 IoT 환경에 최적화되어 있습니다. WIA-ART-010은 계층적 토픽 구조를 사용합니다: 

wia/art-010/{artwork-id}/{category}/{device-id}/{data-type}

예시:
wia/art-010/interactive-2024-001/sensor/sensor-001/skeleton
wia/art-010/interactive-2024-001/sensor/sensor-001/gesture
wia/art-010/interactive-2024-001/output/led-strip-001/status
wia/art-010/interactive-2024-001/system/health
wia/art-010/interactive-2024-001/system/errors

4.2 MQTT QoS 레벨

데이터 중요도에 따라 적절한 QoS(Quality of Service) 레벨을 선택합니다:

QoS 레벨 보장 수준 오버헤드 사용 예
QoS 0 최대 1회 전달 (Fire and Forget) 최소 실시간 센서 스트림 (일부 손실 허용)
QoS 1 최소 1회 전달 (중복 가능) 중간 제스처 이벤트, 상태 업데이트
QoS 2 정확히 1회 전달 (중복 없음) 최대 설정 변경, 중요한 제어 명령

4.3 Node.js MQTT 구현

import mqtt from 'mqtt';

const ARTWORK_ID = 'interactive-2024-001';
const BROKER_URL = 'mqtt://mqtt.artcenter.kr';

// MQTT 클라이언트 연결
const client = mqtt.connect(BROKER_URL, {
  clientId: `wia-art-010-${Math.random().toString(16).substr(2, 8)}`,
  clean: true,
  reconnectPeriod: 1000
});

client.on('connect', () => {
  console.log('Connected to MQTT broker');

  // 센서 토픽 구독 (QoS 0: 실시간 스트림)
  client.subscribe(`wia/art-010/${ARTWORK_ID}/sensor/+/skeleton`, { qos: 0 });

  // 시스템 상태 구독 (QoS 1: 중요한 알림)
  client.subscribe(`wia/art-010/${ARTWORK_ID}/system/#`, { qos: 1 });
});

client.on('message', (topic, message) => {
  const parts = topic.split('/');
  const category = parts[3];  // sensor, output, system
  const deviceId = parts[4];
  const dataType = parts[5];

  if (category === 'sensor' && dataType === 'skeleton') {
    const data = JSON.parse(message.toString());

    // 손 위치로 LED 제어
    if (data.joint === 'left_hand') {
      const hue = mapRange(data.y, 0.5, 2.5, 0, 360);

      // LED 제어 명령 발행 (QoS 1)
      client.publish(
        `wia/art-010/${ARTWORK_ID}/output/led-strip-001/command`,
        JSON.stringify({
          action: 'setColor',
          h: hue,
          s: 100,
          v: 100,
          timestamp: Date.now() / 1000
        }),
        { qos: 1 }
      );
    }
  }
});

// 센서 데이터 발행 (예: Kinect에서 읽은 데이터)
function publishSkeletonData(sensorId, personId, joint, position) {
  const topic = `wia/art-010/${ARTWORK_ID}/sensor/${sensorId}/skeleton`;
  const payload = {
    person_id: personId,
    joint: joint,
    x: position.x,
    y: position.y,
    z: position.z,
    confidence: position.confidence,
    timestamp: Date.now() / 1000
  };

  // QoS 0: 고속 스트리밍, 일부 패킷 손실 허용
  client.publish(topic, JSON.stringify(payload), { qos: 0 });
}

function mapRange(value, inMin, inMax, outMin, outMax) {
  return (value - inMin) * (outMax - outMin) / (inMax - inMin) + outMin;
}

5. 시간 동기화

5.1 NTP 기반 동기화

분산 시스템에서 여러 작품의 시간을 동기화하는 것은 매우 중요합니다. WIA-ART-010은 NTP(Network Time Protocol)를 사용하여 마이크로초 단위의 정확한 시간 동기화를 제공합니다. 

import { NTPClient } from '@wia/art-010';

// NTP 서버와 동기화
const ntpClient = new NTPClient({
  server: 'ntp.artcenter.kr',
  port: 123,
  syncInterval: 60000  // 1분마다 재동기화
});

await ntpClient.sync();

// 동기화된 타임스탬프 얻기
function getNowTimestamp() {
  return ntpClient.now();  // 서버 시간 기준 Unix timestamp
}

// 센서 데이터에 정확한 타임스탬프 추가
function publishSensorData(sensorId, data) {
  const message = {
    sensor_id: sensorId,
    timestamp: getNowTimestamp(),  // NTP 동기화된 시간
    data: data
  };

  oscClient.send('/wia/art-010/sensor/' + sensorId, message);
}

// 타임스탬프 검증 (지연 감지)
function validateTimestamp(receivedTimestamp) {
  const now = getNowTimestamp();
  const latency = now - receivedTimestamp;

  if (latency > 0.1) {  // 100ms 초과
    console.warn(`High latency detected: ${latency * 1000}ms`);
  }

  return latency;
}

5.2 타임코드 기반 동기화

비디오나 오디오와 동기화가 필요한 경우 SMPTE 타임코드를 사용합니다:

타임코드 형식 프레임율 정밀도 사용 예
SMPTE 24 24 fps ~42ms 영화 프로젝션 동기화
SMPTE 30 30 fps ~33ms 비디오 아트 동기화
SMPTE 60 60 fps ~17ms 고속 비주얼 동기화

핵심 요점

  • 다중 프로토콜 지원: OSC (초저지연 <5ms), WebSocket (웹 통합), MQTT (IoT 신뢰성), HTTP/REST (설정 관리)를 요구사항에 맞게 선택할 수 있습니다.
  • 표준화된 메시지 형식: OSC 주소 체계, WebSocket JSON 구조, MQTT 토픽 계층을 통일하여 플랫폼 간 상호운용성을 보장합니다.
  • NTP 시간 동기화: 분산 시스템의 모든 노드가 마이크로초 단위로 동기화되어 여러 작품 간 정확한 타이밍을 유지합니다.
  • QoS 레벨 선택: MQTT의 QoS 0/1/2를 데이터 중요도에 따라 선택하여 성능과 신뢰성의 균형을 맞춥니다.
  • TouchDesigner 통합: OSC와 WebSocket을 통해 TouchDesigner, Processing, openFrameworks 등 주요 크리에이티브 플랫폼과 원활하게 연동됩니다.
  • 확장 가능한 토폴로지: Star, Mesh, Hybrid 네트워크 구조를 지원하여 소규모 설치부터 teamLab 규모의 대형 전시까지 대응합니다.

복습 질문

  1. OSC, WebSocket, MQTT 세 가지 프로토콜의 전송 계층(UDP/TCP), 지연시간, 신뢰성을 비교하고, teamLab Borderless처럼 50개 작품이 연동된 대규모 전시에서 각 프로토콜이 적합한 사용 사례를 제시하십시오. 왜 OSC는 실시간 센서에, MQTT는 원격 모니터링에 더 적합한지 기술적으로 설명하십시오.
  2. WIA-ART-010 OSC 주소 체계 "/wia/art-010/{category}/{device-id}/{data-type}/{parameter}"의 설계 원칙을 분석하십시오. 이 계층적 구조가 OSC의 패턴 매칭 기능과 어떻게 결합되어 효율적인 메시지 라우팅을 가능하게 하는지, 그리고 "/wia/art-010/sensor/*/skeleton/head/*"와 같은 와일드카드 구독의 장점을 논의하십시오.
  3. WebSocket 서버 구현 예제에서 broadcastSensorData() 함수가 client.sensorSubscriptions를 확인하여 선택적 브로드캐스팅을 수행합니다. 만약 100명의 클라이언트가 연결되어 있고 각각 다른 센서를 구독한다면, 전체 브로드캐스트 대비 이 최적화가 네트워크 트래픽과 CPU 사용량에 미치는 영향을 정량적으로 분석하십시오.
  4. MQTT의 QoS 0, 1, 2 레벨의 메시지 전달 보장 수준과 네트워크 오버헤드를 비교하고, Rafael Lozano-Hemmer의 "Pulse Room"처럼 생체 신호를 측정하는 작품에서 어떤 QoS 레벨이 적합한지 판단하십시오. 심박수 데이터 손실이 관람객 경험에 미치는 영향을 고려하여 논의하십시오.
  5. NTP 기반 시간 동기화의 필요성과 구현 방법을 설명하십시오. 제공된 코드에서 validateTimestamp() 함수가 100ms 이상의 지연을 경고하는데, 왜 이 임계값을 선택했는지, 그리고 30 FPS 센서 데이터 스트림에서 100ms 지연이 몇 프레임에 해당하며 관람객이 인지 가능한 수준인지 분석하십시오.
  6. Star 토폴로지 네트워크 구조의 장단점을 분석하고, 중앙 서버가 Single Point of Failure가 될 위험을 어떻게 완화할 수 있는지 논의하십시오. teamLab의 대규모 전시에서 중앙 서버 장애 시 개별 작품이 독립 모드로 동작하도록 하는 Failover 전략을 제안하십시오.

다음 장 예고
