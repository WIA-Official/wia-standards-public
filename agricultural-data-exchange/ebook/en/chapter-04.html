Chapter 4

Communication Protocols

This chapter explores the various communication protocols supported by WIA-AGRI-020, including HTTPS, MQTT, WebSocket, and CoAP. Learn when to use each protocol and how to optimize for different agricultural IoT scenarios.

Understanding Protocol Selection

Agricultural data exchange involves diverse scenarios with different requirements. A weather station with reliable power and broadband internet has very different needs from a battery-powered soil sensor in a remote field. WIA-AGRI-020 supports multiple protocols to accommodate this diversity.

Protocol Selection Criteria

  • Power Availability: Battery-powered sensors need low-power protocols like MQTT or CoAP.
  • Network Reliability: Cellular or satellite connections work better with protocols designed for intermittent connectivity.
  • Latency Requirements: Real-time applications like autonomous equipment need WebSocket or MQTT.
  • Data Volume: High-frequency sensors benefit from efficient binary protocols.
  • Implementation Complexity: Simple applications may prefer straightforward HTTPS REST APIs.

HTTPS (RESTful APIs)

HTTPS is the most widely supported protocol, making it ideal for web applications, mobile apps, and systems with reliable internet connectivity.

When to Use HTTPS

  • Web dashboards and farm management applications
  • Mobile apps for farmers and agronomists
  • Batch data uploads from desktop software
  • Integration with existing enterprise systems
  • Manual data entry and verification workflows

HTTPS Best Practices

  • Use TLS 1.3: Ensures strongest encryption and best performance.
  • Enable Compression: GZIP reduces bandwidth by 70-80% for JSON payloads.
  • Implement Retry Logic: Network issues are common in rural areas; retry with exponential backoff.
  • Connection Pooling: Reuse TCP connections to reduce overhead for multiple requests.
  • Validate Certificates: Always verify SSL certificates to prevent man-in-the-middle attacks.

MQTT (Message Queuing Telemetry Transport)

MQTT is designed for IoT devices with limited power and bandwidth. It's the preferred protocol for agricultural sensors, especially those running on batteries or solar power.

Why MQTT for Agriculture

  • Low Bandwidth: Minimal packet overhead, ideal for cellular connections
  • Low Power: Optimized for battery-powered devices
  • Quality of Service: Three QoS levels ensure reliable delivery
  • Last Will: Automatic notifications when devices go offline
  • Retained Messages: New subscribers receive last known state

MQTT Broker Configuration

protocol: mqtt
broker: mqtt.wia-agri.org
port: 1883 (TCP) / 8883 (TLS)
qos: 1
keepAlive: 60
cleanSession: false
username: farmId
password: api_token

Topic Structure

MQTT uses hierarchical topics for organizing messages:

wia-agri/{farmId}/sensors/soil/{deviceId}/data
wia-agri/{farmId}/sensors/weather/{deviceId}/data
wia-agri/{farmId}/equipment/{equipmentId}/telemetry
wia-agri/{farmId}/alerts/{alertType}

Publishing Data

Python example for publishing soil sensor data:

import paho.mqtt.client as mqtt
import json

client = mqtt.Client(client_id="SENSOR-SOIL-001")
client.username_pw_set("FARM-2025-001", "api_token")
client.tls_set()  # Enable TLS encryption
client.connect("mqtt.wia-agri.org", 8883, 60)

topic = "wia-agri/FARM-2025-001/sensors/soil/SENSOR-SOIL-001/data"
payload = {
    "timestamp": "2025-12-26T10:30:00.000Z",
    "readings": {"soilMoisture": {"value": 45.5}}
}

client.publish(topic, json.dumps(payload), qos=1)
client.disconnect()

Subscribing to Data

def on_message(client, userdata, msg):
    data = json.loads(msg.payload)
    print(f"Received from {msg.topic}: {data}")

client.subscribe("wia-agri/FARM-2025-001/sensors/#")
client.on_message = on_message
client.loop_forever()

WebSocket (Real-Time Bidirectional)

WebSocket provides full-duplex communication over a single TCP connection, perfect for real-time dashboards and interactive applications.

WebSocket Use Cases

  • Live farm monitoring dashboards
  • Real-time alerts and notifications
  • Interactive crop health visualization
  • Collaborative farm planning tools
  • Live equipment tracking

Establishing Connection

const ws = new WebSocket('wss://stream.wia-agri.org/v1/stream');

ws.onopen = () => {
  // Authenticate
  ws.send(JSON.stringify({
    action: 'authenticate',
    token: 'wia_live_abc123def456ghi789'
  }));

  // Subscribe to farm data
  ws.send(JSON.stringify({
    action: 'subscribe',
    farmId: 'FARM-2025-001',
    channels: ['sensors', 'alerts', 'equipment']
  }));
};

ws.onmessage = (event) => {
  const message = JSON.parse(event.data);

  switch(message.type) {
    case 'data':
      updateDashboard(message.payload);
      break;
    case 'alert':
      showNotification(message);
      break;
    case 'ping':
      ws.send(JSON.stringify({type: 'pong'}));
      break;
  }
};

ws.onerror = (error) => {
  console.error('WebSocket error:', error);
};

ws.onclose = () => {
  console.log('Connection closed, reconnecting...');
  setTimeout(reconnect, 5000);
};

CoAP (Constrained Application Protocol)

CoAP is designed for extremely resource-constrained devices. It's ideal for battery-powered sensors with minimal processing power.

CoAP Characteristics

  • UDP-based (lightweight)
  • RESTful design like HTTP
  • Built-in discovery
  • Multicast support
  • 4-byte header overhead

CoAP Request Example

Offline Operation

Store-and-Forward Pattern

Agricultural sensors often operate in areas with intermittent connectivity. The store-and-forward pattern ensures no data is lost:

class OfflineDataManager {
  constructor() {
    this.queue = [];
    this.maxQueueSize = 10000;
  }

  async collectData(reading) {
    this.queue.push(reading);

    if (this.queue.length >= this.maxQueueSize) {
      this.queue.shift(); // Remove oldest
    }

    await this.trySync();
  }

  async trySync() {
    if (!this.isOnline()) return;

    while (this.queue.length > 0) {
      const batch = this.queue.splice(0, 100);

      try {
        await this.sendBatch(batch);
      } catch (error) {
        this.queue.unshift(...batch);
        break; // Stop on failure
      }
    }
  }

  isOnline() {
    return navigator.onLine;
  }
}

Connection Management

Reconnection Strategy

Implement exponential backoff for reliable reconnection:

function reconnectWithBackoff(maxAttempts = 10) {
  let attempt = 0;
  let delay = 1000; // 1 second

  const tryConnect = async () => {
    try {
      await connect();
      console.log('Connected successfully');
      return true;
    } catch (error) {
      attempt++;

      if (attempt >= maxAttempts) {
        console.error('Max reconnection attempts reached');
        return false;
      }

      const waitTime = Math.min(delay * Math.pow(2, attempt), 300000);
      console.log(`Reconnecting in ${waitTime}ms (attempt ${attempt})`);

      await sleep(waitTime);
      return tryConnect();
    }
  };

  return tryConnect();
}

🎯 Best Practices

  • Choose protocol based on device capabilities and network conditions
  • Always use TLS encryption for production deployments
  • Implement keep-alive mechanisms to detect connection failures
  • Use QoS 1 for MQTT sensor data (at-least-once delivery)
  • Batch data submissions when possible to reduce overhead
  • Monitor connection health with heartbeat messages
  • Implement exponential backoff for reconnection attempts
  • Store data locally during offline periods

Chapter Summary

WIA-AGRI-020 supports four main protocols: HTTPS for web applications, MQTT for IoT sensors, WebSocket for real-time dashboards, and CoAP for ultra-low-power devices. Each protocol is optimized for different scenarios, and proper selection based on power availability, network reliability, and latency requirements ensures efficient agricultural data exchange.
