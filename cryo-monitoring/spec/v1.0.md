## 1. Overview

The WIA CRYO-008 standard defines data formats and protocols for real-time monitoring of cryopreserved biological materials. This specification focuses on **Phase 1: Data Format**, establishing the foundational schema for all monitoring data exchange.

### 1.1 Objectives

- **Interoperability:** Enable seamless data exchange between monitoring systems
- **Reliability:** Ensure data integrity through comprehensive validation
- **Extensibility:** Support future sensor types and monitoring modalities
- **Auditability:** Provide complete traceability for all monitoring events

### 1.2 Normative References

- ISO 8601: Date and time format
- JSON Schema Draft 2020-12
- IEEE 754: Floating-point arithmetic
- NIST SP 800-175B: Cryptographic hash functions

## 2. Scope

This specification covers:

- ✅ Monitoring data record format (JSON schema)
- ✅ Sensor reading data structures
- ✅ Alert and event encoding

- ❌ Communication protocols (see v1.2)

## 3. Data Format Specification

### 3.1 Core Data Model

All monitoring data MUST be encoded in JSON format using UTF-8 encoding.

#### 3.1.1 MonitoringRecord Object

```json
{
  "monitoringId": "string",
  "subjectId": "string",
  "timestamp": "ISO8601",
  "facilityId": "string",
  "sensors": [SensorReading],
  "alerts": [Alert],
  "metadata": Metadata,
  "signature": "string"
}
```

#### 3.1.2 Field Definitions

| Field | Type | Required | Description | Constraints |
|-------|------|----------|-------------|-------------|
| `monitoringId` | string | Yes | Unique monitoring session ID | Format: `MON-YYYY-NNNNNN` |
| `subjectId` | string | Yes | Cryopreserved subject ID | Format: `CRYO-YYYY-NNNN` |
| `timestamp` | string | Yes | UTC timestamp | ISO 8601 with timezone |
| `facilityId` | string | Yes | Facility identifier | Format: `FAC-CC-RR-NNN` |
| `sensors` | array | Yes | Sensor readings | Min 1 element |
| `alerts` | array | No | Active alerts | Can be empty |
| `metadata` | object | Yes | Record metadata | See §3.2 |
| `signature` | string | Yes | Cryptographic signature | SHA-256 hash |

---

### 3.2 SensorReading Object

```json
{
  "sensorId": "string",
  "type": "enum",
  "value": number,
  "unit": "string",
  "location": "string",
  "status": "enum",
  "confidence": number,
  "calibrationDate": "ISO8601",
  "nextCalibration": "ISO8601"
}
```

#### 3.2.1 Sensor Type Enumeration

| Type | Description | Units | Valid Range |
|------|-------------|-------|-------------|
| `temperature` | Temperature sensor | celsius, fahrenheit, kelvin | -273.15 to 1000 |
| `pressure` | Pressure sensor | pascal, bar, psi | 0 to 10000 |
| `oxygen` | O2 level sensor | percent, ppm | 0 to 100 |
| `nitrogen` | N2 level sensor | percent, ppm | 0 to 100 |
| `eeg` | Electroencephalogram | microvolt | -500 to 500 |
| `ecg` | Electrocardiogram | millivolt | -10 to 10 |
| `vibration` | Vibration sensor | g-force | 0 to 100 |
| `humidity` | Humidity sensor | percent | 0 to 100 |

#### 3.2.2 Status Enumeration

- `ok` - Normal operation
- `warning` - Parameter deviation detected
- `critical` - Critical threshold exceeded
- `error` - Sensor malfunction
- `offline` - Sensor not responding
- `calibrating` - Under calibration

---

### 3.3 Alert Object

```json
{
  "alertId": "string",
  "severity": "enum",
  "type": "string",
  "message": "string",
  "triggeredAt": "ISO8601",
  "acknowledgedAt": "ISO8601",
  "acknowledgedBy": "string",
  "relatedSensors": ["string"]
}
```

#### 3.3.1 Alert Severity Levels

| Severity | Code | Response Time | Escalation |
|----------|------|---------------|------------|
| `info` | 0 | No action required | None |
| `warning` | 1 | Within 15 minutes | Email + SMS |
| `urgent` | 2 | Within 5 minutes | Multi-channel |
| `critical` | 3 | Immediate | All channels + emergency protocol |

## 4. Sensor Requirements

### 4.1 Temperature Sensors

**Primary Specification:**
- Type: Platinum RTD (PT100 or PT1000)
- Accuracy: ±0.1°C at -196°C
- Response Time: T90 < 10 seconds
- Operating Range: -200°C to +50°C
- Calibration: Annually against NIST-traceable standard

**Backup Specification:**
- Type: Thermocouple Type K
- Accuracy: ±0.5°C at -196°C
- Response Time: T90 < 30 seconds

### 4.2 Liquid Level Sensors

- Type: Capacitive or ultrasonic
- Resolution: ±1 cm
- Operating Temperature: -196°C to +50°C
- Update Rate: 0.1 Hz minimum

### 4.3 Sensor Redundancy

Critical monitoring points MUST implement:
- **Triple Redundancy:** 3 independent sensors
- **Voting Logic:** 2-out-of-3 agreement for status
- **Automatic Failover:** Switch to backup within 5 seconds

## 5. Monitoring Parameters

### 5.1 Temperature Monitoring

| Parameter | Target | Acceptable | Warning | Critical |
|-----------|--------|------------|---------|----------|
| Subject Core | -196°C | ±0.5°C | ±1.0°C | ±2.0°C |
| Dewar Interior | -196°C | ±1.0°C | ±2.0°C | ±5.0°C |
| LN2 Bath | -196°C | ±0.2°C | ±0.5°C | ±1.0°C |
| Ambient Room | 20°C | 18-25°C | 26-30°C | >30°C |

### 5.2 Sampling Rates

| Sensor Type | Minimum | Recommended | Maximum |
|-------------|---------|-------------|---------|
| Temperature | 0.1 Hz | 1 Hz | 10 Hz |
| Pressure | 0.01 Hz | 0.1 Hz | 1 Hz |
| Level | 0.01 Hz | 0.1 Hz | 1 Hz |
| Vital Signs (EEG/ECG) | 100 Hz | 250 Hz | 1000 Hz |

## 6. Data Validation Rules

### 6.1 Required Field Validation

```typescript
interface ValidationRules {
  monitoringId: /^MON-\d{4}-\d{6}$/,
  subjectId: /^CRYO-\d{4}-\d{4}$/,
  facilityId: /^FAC-[A-Z]{2}-[A-Z]{2}-\d{3}$/,
  timestamp: ISO8601_REGEX,
  sensors: minLength(1),
  signature: /^[a-f0-9]{64}$/  // SHA-256
}
```

### 6.2 Range Validation

All sensor values MUST be validated against physically possible ranges:

```javascript
function validateTemperature(value, unit) {
  if (unit === 'celsius') {
    return value >= -273.15 && value <= 1000;
  }
  // Similar for other units
}
```

### 6.3 Temporal Validation

- Timestamps MUST NOT be in the future
- Timestamps MUST NOT be older than 48 hours (configurable)
- Alert `triggeredAt` MUST be <= `acknowledgedAt`

## 8. Conformance

### 8.1 Conformance Levels

**Level 1 (Basic):**
- ✓ Implements core MonitoringRecord schema
- ✓ Validates all required fields
- ✓ Supports temperature sensors

**Level 2 (Standard):**
- ✓ All Level 1 requirements
- ✓ Supports all sensor types
- ✓ Implements alert mechanism
- ✓ Triple sensor redundancy

**Level 3 (Full):**
- ✓ All Level 2 requirements
- ✓ Cryptographic signatures
- ✓ Blockchain audit logging
- ✓ Real-time validation

### 8.2 Testing Requirements

Conformance testing MUST include:
1. JSON schema validation against official schema
2. Range validation for all sensor types
3. Timestamp format verification
4. Signature verification
5. Load testing (1000+ records/second)
