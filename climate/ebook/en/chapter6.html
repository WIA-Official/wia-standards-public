6.2 Authentication Methods

6.2.1 API Key Authentication

For: Sensors, embedded devices, server-to-server

Format: wia_climate_live_1A2B3C4D5E... (64 hex chars)

curl -H "Authorization: Bearer wia_climate_live_..." \
     https://api.wia.live/climate/v1/data

6.2.2 JWT Token Authentication

For: User sessions, dashboards, mobile apps

Expiry: 1 hour (access token), 30 days (refresh token)

{
  "alg": "RS256",
  "typ": "JWT"
}.{
  "sub": "user_12345",
  "iss": "api.wia.live",
  "aud": "climate",
  "exp": 1702483200,
  "scope": ["read:data", "write:data"],
  "device_id": "ORCA-001"
}

6.2.3 mTLS (Mutual TLS)

For: High-security deployments, government installations

Requires: Client certificate signed by WIA CA

6.3.2 Data Isolation

  • Public Data: Climate model outputs, aggregated statistics (read by anyone)
  • Organization Data: Device telemetry, operational metrics (read/write by org members)
  • Private Data: Proprietary research, competitive intelligence (owner only)

6.4 Data Integrity

6.4.1 Message Signing

Optional HMAC-SHA256 signatures for high-integrity applications:

{
  "version": "1.0.0",
  "type": "carbon_capture",
  // ... message content ...
  "meta": {
    "signature": {
      "algorithm": "HMAC-SHA256",
      "value": "a1b2c3d4...",
      "key_id": "key_123"
    }
  }
}

Signature Computation:

import hmac
import hashlib
import json

def sign_message(message, secret_key):
    # Create canonical JSON (sorted keys, no whitespace)
    canonical = json.dumps(message, sort_keys=True, separators=(',', ':'))

    # Compute HMAC-SHA256
    signature = hmac.new(
        secret_key.encode(),
        canonical.encode(),
        hashlib.sha256
    ).hexdigest()

    return signature

6.4.2 Validation Pipeline

  1. Schema Validation: Ensure message conforms to JSON Schema
  2. Range Validation: Check values are within physical limits
  3. Temporal Validation: Reject messages with future timestamps or excessive clock skew (>5 minutes)
  4. Geospatial Validation: Verify coordinates are valid and match registered device location (within tolerance)
  5. Device Authorization: Confirm device is registered and authorized to send this data type

6.5 Encryption

6.5.1 Transport Encryption

Protocol Encryption Minimum Version
HTTPS TLS 1.3 (fallback: 1.2)
WebSocket WSS (TLS) 1.3
MQTT MQTTS (TLS) 1.2

6.5.2 At-Rest Encryption

  • Database: AES-256 encryption for sensitive fields
  • Backups: Encrypted with rotating keys
  • Logs: PII and secrets redacted/encrypted

6.6 Sensor Network Security

6.6.1 IoT Device Hardening

Best Practices:

  • Store API keys in secure element or TPM
  • Disable unused network services
  • Use read-only filesystem where possible
  • Implement secure boot
  • Regular firmware updates via signed OTA

6.6.2 Edge Gateway Security

#!/bin/bash
# Secure edge gateway setup

# 1. Firewall rules (only HTTPS/MQTTS outbound)
sudo ufw default deny incoming
sudo ufw default deny outgoing
sudo ufw allow out 443/tcp  # HTTPS
sudo ufw allow out 8883/tcp # MQTTS
sudo ufw enable

# 2. Fail2ban for SSH
sudo apt install fail2ban

# 3. Automatic security updates
sudo apt install unattended-upgrades

# 4. Secure API key storage
echo "WIA_API_KEY=wia_climate_..." | sudo tee /etc/wia/credentials
sudo chmod 600 /etc/wia/credentials
sudo chown root:root /etc/wia/credentials

6.7 Privacy Considerations

6.7.1 Data Minimization

Principle: Only collect and transmit data necessary for climate monitoring.

  • Avoid including Personally Identifiable Information (PII)
  • Use device IDs instead of operator names
  • Aggregate fine-grained location data when possible
  • Redact proprietary algorithms or trade secrets

6.7.2 GDPR Compliance (if applicable)

  • Right to Access: Users can query all data associated with their devices
  • Right to Deletion: API endpoint to delete historical data
  • Data Portability: Export data in standard JSON format
  • Consent: Required for any human-related climate data (e.g., indoor vertical farm worker exposure)

6.8 Incident Response

6.8.1 Security Event Categories

Severity Examples Response Time
Critical Data breach, API compromise, ransomware < 1 hour
High Unauthorized access attempt, DDoS attack < 4 hours
Medium Suspicious activity, rate limit abuse < 24 hours
Low Failed login attempts, invalid data < 1 week

6.8.2 Incident Response Playbook

  1. Detect: Automated monitoring alerts security team
  2. Contain: Isolate affected systems, revoke compromised credentials
  3. Investigate: Analyze logs, identify root cause and scope
  4. Remediate: Patch vulnerabilities, restore from clean backups
  5. Notify: Inform affected users within 72 hours (GDPR requirement)
  6. Review: Post-mortem analysis, update security procedures

6.10 Security Checklist

✓ Implementation Checklist

  • ☐ TLS 1.3 enabled on all endpoints
  • ☐ API keys stored in secrets manager (not hardcoded)
  • ☐ Certificate pinning for critical connections
  • ☐ Input validation on all API endpoints
  • ☐ Rate limiting configured (per IP and per API key)
  • ☐ Logging enabled with log retention policy
  • ☐ Security headers configured (HSTS, CSP, etc.)
  • ☐ Regular security audits scheduled
  • ☐ Incident response plan documented
  • ☐ Dependency scanning for vulnerabilities
