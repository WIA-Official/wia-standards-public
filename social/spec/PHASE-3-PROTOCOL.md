# WIA-SOCIAL — Phase 3: Federation Protocol Specification

**Standard**: WIA-SOCIAL
**Phase**: 3 of 4 — Protocol
**Version**: 1.0.0
**Status**: Draft

---

## 1. Scope

Phase 3 specifies how independent WIA-SOCIAL hosts establish trust, exchange
identity bundles and activities, detect replays, recover from key
compromise, and revoke stale data. Phase 1 fixed shapes; Phase 2 fixed HTTP
verbs; this phase fixes the *behaviour over time*.

---

## 2. Roles

| Role | Description |
|------|-------------|
| **Subject** | A natural or legal person identified by an identity bundle |
| **Origin host** | The host that publishes a subject's bundle |
| **Peer host** | Any host that has completed a federation handshake with the origin |
| **Bridge** | A host that translates WIA-SOCIAL ↔ legacy network APIs (Phase 4) |

A single host MAY play multiple roles. Trust flows along the directed edge
*peer → origin → subject*; reverse trust requires a separate handshake in
the opposite direction.

---

## 3. State Machine — Federation Handshake

```
   ┌──────────┐
   │   IDLE   │
   └────┬─────┘
        │ peer issues handshake (Phase 2 §7)
        ▼
   ┌──────────┐
   │ PENDING  │  origin verifies signature, fetches peer discovery doc
   └────┬─────┘
        │
        ├── invalid signature → REJECTED ───┐
        │                                   │
        │ valid                              │
        ▼                                   ▼
   ┌──────────┐                       ┌──────────┐
   │ACCEPTED  │── periodic refill ──▶│REVOKED   │
   └────┬─────┘                       └──────────┘
        │
        │ peer publishes revocation
        ▼
   ┌──────────┐
   │REVOKED   │
   └──────────┘
```

* `PENDING` MUST resolve within 30 seconds; otherwise the origin MUST
  return a problem of type `…/handshake-timeout`.
* `ACCEPTED` is durable; both sides MUST persist the receipt (Phase 1 §5).
* `REVOKED` is terminal; a fresh handshake creates a new state machine with
  a new receipt id.

## 4. Bundle Replication

### 4.1 Pull Model

A peer MAY pull a subject's bundle on-demand:

```
GET https://origin/wia-social/bundle/acct:alice@example.org
```

The peer MUST verify the bundle's detached signature and the origin host's
HTTP message signature before relying on the result.

### 4.2 Push Model

The origin MAY push bundle changes to subscribed peers via WebSub callbacks
(Phase 2 §6.2). Each push MUST be signed by the origin's host key.

### 4.3 Cache Invariants

A peer that caches a bundle MUST honour:

1. `expires_at` if present.
2. The maximum age advertised by the origin's `Cache-Control` header.
3. A revocation message received from the origin host (any model).
4. The subject's `previous_signature` chain — a peer MUST NOT serve a
   bundle whose `previous_signature` does not match the head it has stored.

A peer MAY serve a stale bundle marked `STALE` for up to 24 hours when the
origin is unreachable, but MUST emit a `Warning: 110` header per IETF RFC
9111.

---

## 5. Activity Fan-Out

The actor publishes an activity to its origin (Phase 2 §5.1). The origin
fans out to:

1. **Subscribers** of `feed/{subject}` (WebSub callbacks).
2. **Mentioned subjects' origin hosts** for in-thread notifications.
3. **Bridges** that handle network claims listed in `fanout`.

Fan-out MUST be at-least-once. Each peer MUST ignore duplicates by
`activity_id`. Maximum fan-out latency target is 5 seconds at p99 over a
30-day window (operational target, not a normative requirement).

### 5.1 Reaction & Comment Threading

A reply activity carries `in_reply_to` set to the parent `activity_id`.
The peer holding the parent MUST:

1. Emit a `notification` to the parent's actor.
2. Increment its locally-cached reply counter.
3. Re-fan to its own subscribers if the parent's audience permits.

### 5.2 Edit & Delete

`edit` and `delete` verbs reference an existing `activity_id`. They MUST be
signed by the original actor's current key. A peer that has cached the
target activity MUST update or tombstone it within 60 seconds of receipt.

## 6. Replay Defence

Each signed envelope (handshake, bundle write, activity, receipt) carries:

| Field | Purpose |
|-------|---------|
| `nonce` | 96-bit random value, base64url |
| `timestamp` | RFC 3339 UTC, generated by the signer |

A receiver MUST:

1. Reject envelopes with `timestamp` skew greater than ±300 seconds from
   the receiver's clock (RFC 8915 NTS recommended for synchronisation).
2. Reject envelopes whose `(signer, nonce)` tuple has been seen within the
   last 600 seconds.
3. Maintain the seen-nonce cache for at least 600 seconds.

These bounds make replay attacks on signed envelopes impractical without
also breaking the underlying signature.

---

## 7. Key Rotation

A subject rotates keys by publishing a new identity bundle whose:

* `key.public` differs from the prior bundle.
* `previous_signature` equals the prior bundle's `signature.value`.
* Detached signature is computed under the **new** key.

The origin MUST verify the prior bundle's signature with the **old** key
before accepting the rotation. Once accepted, the old key is retired.

Peers MUST follow the chain when a rotation happens; an outdated cache MUST
be updated within 24 hours.

### 7.1 Emergency Revocation

If the subject loses control of their signing key, the subject's recovery
contact (a claim of type `recovery_email` or a designated peer host) MAY
publish a `revocation_receipt` of type `key_compromise`. After such a
revocation, the subject MUST establish a new identity bundle from scratch
— there is no in-band link between the old and new identity.

---

## 8. Block, Mute, Unfollow

* `block` is a directed edge from actor A to subject B. Once published, A's
  origin MUST refuse to fan-out activities authored by B to A's feed and
  MUST refuse delivery of B-authored notifications to A.
* `mute` is local — it is not federated. A host MAY store mutes in the
  user's preferences without publishing them.
* `unfollow` removes a subscription edge. Origins MUST tear down any
  WebSub subscriptions associated with the unfollow within 60 seconds.

---

## 9. Privacy Boundaries

Phase 1 §4.3 defines per-claim audiences. The protocol enforcement is:

| Audience | Enforcement |
|----------|-------------|
| `public` | None — caches MAY be served to any client |
| `friends` | Origin MUST verify that the requesting peer's actor has a `follow` from the subject |
| `verified` | As above, plus the requesting peer MUST present a verifiable claim of address ownership |
| `private` | Never federated — origin returns `403 Forbidden` to all peers |

A peer that downgrades enforcement (e.g. caches a `friends` bundle and
serves it publicly) violates the standard and MUST be removed from the
discovery `peer_allowlist` of any conformant host.

## 10. Cryptographic Suite

| Use | Algorithm | Reference |
|-----|-----------|-----------|
| Identity signing | Ed25519 | IETF RFC 8032 |
| Optional alt | ECDSA P-256 | FIPS 186-5 |
| HTTP message signing | Ed25519 over RFC 9421 covered components | RFC 9421 |
| Bundle hashing | SHA-256 | FIPS 180-4 |
| Transport | TLS 1.3 | IETF RFC 8446 |
| Future PQC | NIST PQC suites | NIST FIPS 203/204/205 once standardised |

Migration to post-quantum signatures will follow a separate Phase 3.x
addendum after the PQC standards stabilise.

---

## 11. Error Recovery

| Failure | Recovery |
|---------|----------|
| Origin unreachable | Peers serve `STALE` per §4.3 with `Warning: 110` |
| Signature verification fails | Reject and log; never silently accept |
| Clock skew exceeds 300 s | Reject envelope; surface NTS sync issue to operator |
| Storage corruption | Re-fetch from origin; if origin gone, mark subject as `unrecoverable` |
| Bridge outage | Activity remains queued for the bridge target; user notified after 1 hour |

---

## 12. Conformance Checklist

A Phase 3 conformant implementation MUST:

1. Implement the handshake state machine (§3) with all four states.
2. Honour replay defence bounds (§6).
3. Honour key rotation chain (§7).
4. Enforce per-claim audiences for federated reads (§9).
5. Tombstone deletes within 60 seconds (§5.2).
6. Maintain a seen-nonce cache for at least 600 seconds.

The simulator at `simulator/index.html` provides interactive walkthroughs of
the handshake, fan-out and rotation flows.

## Appendix A — Worked Handshake Trace

```
α = origin (alpha.example), β = peer (beta.example)
S(x, k) = Ed25519 signature of bytes x with key k
Verify(x, sig, pk) = boolean

1. β → α : POST /wia-social/handshake
            { from_peer: β, to_peer: α, ephemeral_pk: epk_β,
              nonce: n1, ts: t1, signature: S(env, sk_β) }

2. α      : Verify(env, signature, host_pk(β))
            check |t1 − now| ≤ 300 s
            check (β, n1) not in seen_nonces
            insert (β, n1) with TTL 600 s

3. α → β : 200 OK
            { type: federation_receipt,
              accepted_subject: β,
              by_peer: α,
              receipt_id: rcpt_01HXY…,
              signature: S(receipt, host_sk(α)) }

4. β      : Verify(receipt, signature, host_pk(α))
            persist receipt under (peer = α)

5. Both transition β↔α to ACCEPTED.
```

A failed verification at step 2 yields `401 Unauthorized` with a problem
document of type `…/handshake-bad-signature`. A skew failure yields
`…/handshake-clock-skew` with `Date:` populated for the peer's diagnosis.

## Appendix B — Replay Cache Sizing

For a busy host receiving 1 000 envelopes per second, the seen-nonce cache
must hold roughly `1 000 × 600 = 600 000` entries to enforce §6's
600-second window. With 16-byte nonce keys and a 4-byte timestamp, the
cache footprint is approximately `600 000 × 24 ≈ 15 MiB`. Hosts SHOULD
provision at least double this to absorb traffic bursts.

## Appendix C — Privacy Threats and Mitigations

| Threat | Mitigation |
|--------|------------|
| Bundle correlation across peers | Subjects publish per-relationship bundles; each relationship holds a separately-signed sub-bundle |
| Side-channel timing on signature verify | Implementations MUST use constant-time Ed25519 (libsodium, ring) |
| Compromised peer leaks `friends` claims | Origin removes the peer from the trust list; peer's cached data is treated as compromised |
| Long-term metadata accumulation | Subjects may publish a `data_retention` claim hinting at desired retention; conformant peers SHOULD honour it |
| Mass-scrape of public bundles by hostile crawler | Hosts apply per-source rate limits and MAY require an opaque proof-of-work header for high-rate anonymous reads |
| Bundle exfiltration via compromised origin | Subjects retain their signing key out-of-band; a fresh origin can re-publish the bundle and peers automatically follow the chain via §7 |

## Appendix D — Operational Notes for Origin Hosts

* Origins SHOULD persist receipts in a write-once, append-only store
  (e.g. an object store with object-lock semantics). Receipts that
  silently disappear break the auditability promise of §3.
* Origins SHOULD expose a `peers` introspection endpoint, returning the
  current trust list with last-handshake timestamps, for operator
  dashboards. The endpoint MUST be authenticated.
* When draining a host for maintenance, origins SHOULD publish a `notice`
  envelope to all subscribed peers with an estimated `restore_at`
  timestamp; peers display this to subjects to set expectations.
