Chapter 3: Authentication Methods

Chapter 3: Authentication Methods

Authentication is the process of verifying that someone or something is who they claim to be. It's the first line of defense in identity security and the foundation upon which all access control decisions are made. As threats evolve and technology advances, authentication methods have become increasingly sophisticated, moving from simple passwords to multi-factor approaches combining multiple verification techniques.

3.1 Authentication Fundamentals

Authentication relies on one or more of three fundamental factors that can verify identity:

  • Something You Know: Knowledge-based factors like passwords, PINs, security questions
  • Something You Have: Possession-based factors like tokens, smart cards, mobile devices
  • Something You Are: Inherence-based factors like biometrics (fingerprints, facial recognition)

Additionally, modern authentication may consider contextual factors:

  • Somewhere You Are: Location-based verification (geolocation, network location)
  • Something You Do: Behavioral biometrics (typing patterns, mouse movements)

3.1.1 Single-Factor vs Multi-Factor Authentication

Single-Factor Authentication (SFA) relies on only one authentication factor, typically a password. While convenient, SFA is vulnerable to various attacks including phishing, credential stuffing, and password guessing.

Multi-Factor Authentication (MFA) requires two or more different types of factors. Even if one factor is compromised (e.g., password stolen), the additional factors prevent unauthorized access. MFA dramatically improves security - Microsoft reports that MFA blocks 99.9% of automated account compromise attacks.

3.2 Knowledge-Based Authentication

Knowledge-based authentication relies on information that only the legitimate user should know. While widely used, knowledge factors have significant security limitations.

3.2.1 Passwords and Passphrases

Passwords remain the most common authentication method despite well-documented weaknesses. The challenge is balancing security (complex, unique passwords) with usability (memorable passwords).

Password Policy Traditional Approach Modern Best Practice (NIST)
Minimum Length 8-10 characters 12-15 characters minimum
Complexity Require special chars, numbers Length more important than complexity
Expiration 90-day forced changes No periodic changes unless compromised
Reuse Prevent last 5-10 passwords Check against breach databases
Security Questions Use for password reset Avoid - easily guessed/researched

Passphrases vs Passwords

Passphrases are longer phrases that are easier to remember but harder to crack. For example, "BlueDolphin$SwimsFast2025!" is both memorable and strong. Four random words like "correct-horse-battery-staple" provide excellent security while remaining memorable. Length matters more than complexity - a 20-character password of all lowercase letters is stronger than an 8-character password with mixed case and symbols.

3.2.2 Password Security Best Practices

  • Hash and Salt: Never store passwords in plain text. Use strong hashing algorithms (bcrypt, Argon2, PBKDF2) with unique salts per user.
  • Breach Detection: Check new passwords against databases of compromised credentials (Have I Been Pwned API).
  • Rate Limiting: Implement account lockout or exponential backoff after failed attempts.
  • Password Managers: Encourage use of password managers to enable unique, strong passwords.
  • Secure Transmission: Always use TLS/SSL for password submission.
  • No Hints: Don't provide password hints or security questions with easily researched answers.

3.2.3 PIN and Pattern Authentication

Personal Identification Numbers (PINs) and pattern locks are common on mobile devices. They offer convenience but provide limited security due to small keyspaces (10,000 possibilities for 4-digit PINs). They work best combined with device possession as a second factor.

3.3 Possession-Based Authentication

Possession factors verify identity by confirming the user has a specific physical or digital object. These factors are harder to steal remotely than knowledge factors.

3.3.1 Hardware Tokens

Hardware tokens generate one-time passwords (OTPs) or cryptographic responses:

Token Type How It Works Pros Cons
TOTP Tokens Time-based OTP, synced with server No internet needed, strong security Can desync, physical device required
Challenge-Response Generate response to server challenge Resistant to replay attacks More complex user interaction
Smart Cards Cryptographic chip with PKI Very secure, multi-purpose Requires reader, higher cost
Security Keys (FIDO2) USB/NFC keys with public key crypto Phishing-resistant, standard Can be lost, requires physical access

3.3.2 Software Tokens and Mobile Apps

Software tokens installed on smartphones provide similar functionality to hardware tokens without additional hardware:

  • Authenticator Apps: Google Authenticator, Microsoft Authenticator, Authy generate time-based OTPs
  • Push Notifications: Apps send approval requests that users accept/deny
  • SMS OTP: Codes sent via text message (less secure due to SIM swapping risks)
  • Email OTP: Codes sent to registered email addresses
Security Warning: SMS-Based Authentication Risks

While convenient, SMS-based authentication is vulnerable to SIM swapping attacks where attackers convince mobile carriers to transfer a victim's phone number to a SIM card they control. NIST deprecated SMS as a secure MFA method. Prefer authenticator apps, push notifications, or hardware tokens for sensitive applications.

3.3.3 Certificate-Based Authentication

Digital certificates using Public Key Infrastructure (PKI) provide strong authentication:

  • User possesses private key (on smart card, TPM, or device storage)
  • Server verifies possession by challenging user to sign data with private key
  • Certificate authority validates identity before issuing certificates
  • Certificates can be revoked if compromised

3.4 Biometric Authentication

Biometric authentication uses unique physical or behavioral characteristics to verify identity. Biometrics provide convenience (nothing to remember or carry) but raise privacy concerns and face technical challenges.

3.4.1 Physical Biometrics

Biometric Type How It Works Accuracy Considerations
Fingerprint Scan and match fingerprint patterns High (FAR 0.001-0.1%) Can be spoofed, affected by injuries
Facial Recognition Analyze facial features and geometry Medium-High Lighting sensitive, mask challenges
Iris Scan Analyze iris patterns Very High (FAR 0.0001%) Requires specialized hardware
Voice Recognition Analyze voice characteristics Medium Affected by illness, background noise
Retina Scan Scan blood vessel patterns in retina Very High Intrusive, expensive equipment

FAR = False Acceptance Rate (probability of incorrectly accepting an impostor)

3.4.2 Behavioral Biometrics

Behavioral biometrics analyze how users interact with systems:

  • Keystroke Dynamics: Typing speed, rhythm, pressure patterns
  • Mouse Movement: Speed, acceleration, movement patterns
  • Gait Analysis: Walking patterns detected by smartphone sensors
  • Voice Dynamics: Speech patterns, cadence, vocabulary

Behavioral biometrics work well for continuous authentication, detecting when an authenticated session is taken over by a different person.

3.4.3 Biometric Challenges and Privacy

The Irrevocability Problem

Unlike passwords that can be changed if compromised, you can't change your fingerprints or face. Once biometric data is stolen, it's permanently compromised. Best practices include:

  • Store biometric templates, not raw biometric data
  • Use one-way hashing and encryption for biometric storage
  • Store biometrics locally on devices (e.g., Apple Secure Enclave) rather than centrally
  • Implement liveness detection to prevent spoofing
  • Consider biometrics as convenient MFA rather than sole authentication

Privacy regulations (GDPR, BIPA) classify biometric data as sensitive personal information requiring explicit consent, data protection impact assessments, and strong security controls.

3.5 Multi-Factor Authentication (MFA)

MFA combines two or more independent authentication factors. The key word is "independent" - using password plus security question is not true MFA since both are knowledge factors.

3.5.1 Common MFA Combinations

Combination Example Security Level User Experience
Know + Have Password + TOTP app High Good
Know + Are Password + Fingerprint High Excellent
Have + Are Security key + Fingerprint Very High Good
Know + Have + Are Password + Token + Fingerprint Very High Fair (more friction)

Risk factors considered include:

  • Device Trust: Known vs unknown devices, managed vs unmanaged
  • Location: Geolocation, impossible travel detection
  • Network: Corporate network vs public WiFi vs Tor
  • Behavior: Deviations from normal access patterns
  • Time: Access during unusual hours
  • Resource Sensitivity: Higher authentication for sensitive resources

3.6 Passwordless Authentication

Passwordless authentication eliminates passwords entirely, using other factors for authentication. This addresses password weaknesses (reuse, phishing, credential stuffing) while often improving user experience.

3.6.1 WebAuthn and FIDO2

The FIDO2 framework (comprising WebAuthn and CTAP protocols) enables passwordless authentication using public key cryptography:

  1. Registration: User registers a security key or biometric authenticator with the service
  2. Credential Storage: Private key stored securely on user's device, public key stored by service
  3. Authentication: Service sends challenge, device signs with private key, service verifies with public key
  4. Phishing Resistance: Authentication tied to specific origin, prevents phishing
Platform Authenticators

Modern devices have built-in platform authenticators: Windows Hello (fingerprint, facial recognition, PIN), Apple Touch ID/Face ID, Android biometrics. These provide convenient, secure passwordless authentication without additional hardware.

3.6.2 Magic Links and Email Authentication

Email-based authentication sends time-limited, single-use links that authenticate users when clicked. Security relies on email account security, making this suitable for low-to-medium security scenarios.

3.6.3 Biometric-Only Authentication

Some implementations use biometrics as the sole authentication factor. While convenient, this has limitations:

  • Biometrics can't be revoked if compromised
  • Privacy concerns with biometric data storage
  • False reject rates impact legitimate users
  • Best used with device possession as implicit second factor

3.7 Authentication Protocols

Various protocols standardize authentication across systems and enable federation.

3.7.1 Kerberos

Kerberos provides network authentication using symmetric key cryptography and trusted third-party Key Distribution Centers (KDCs). Widely used in enterprise environments (Active Directory uses Kerberos).

3.7.2 LDAP Authentication

Lightweight Directory Access Protocol (LDAP) enables applications to authenticate users against directory services. Simple Bind sends credentials in clear text (use TLS), while SASL provides secure authentication mechanisms.

3.7.3 OAuth 2.0 and OpenID Connect

OAuth 2.0 is an authorization framework often used for authentication. OpenID Connect (OIDC) extends OAuth 2.0 specifically for authentication, providing standardized identity tokens (JWTs).

3.7.4 SAML 2.0

Security Assertion Markup Language (SAML) enables enterprise single sign-on by exchanging XML-based authentication and authorization data between identity providers and service providers.

3.8 Session Management

After successful authentication, session management maintains authenticated state:

3.8.1 Session Security Best Practices

  • Session Tokens: Use cryptographically random, long tokens (128+ bits entropy)
  • Secure Storage: HttpOnly, Secure, SameSite cookie flags
  • Timeouts: Idle timeout (15-30 min), absolute timeout (8-12 hours)
  • Logout: Properly invalidate sessions on all devices
  • Binding: Bind sessions to IP address, user agent (with caution for mobile users)
  • Rotation: Rotate session IDs after privilege escalation

3.9 WIA-SEC-009 Authentication Requirements

The WIA-SEC-009 standard mandates specific authentication controls based on risk level:

Risk Level Minimum Authentication Additional Requirements
Low Single factor (password) Password policy: 12+ chars, breach detection
Medium MFA (2 factors) TOTP or better, risk-based step-up
High Strong MFA Hardware token or biometric, context awareness
Critical Phishing-resistant MFA FIDO2/WebAuthn, PKI certificates, continuous auth

Summary

Authentication methods have evolved from simple passwords to sophisticated multi-factor and passwordless approaches. Knowledge factors (passwords) remain common but have well-known weaknesses. Possession factors (tokens, mobile devices) and inherence factors (biometrics) provide stronger security, especially when combined in MFA. Modern authentication leverages risk-based adaptive approaches and passwordless methods like FIDO2/WebAuthn. The WIA-SEC-009 standard recommends authentication strength aligned with risk levels, emphasizing MFA for most scenarios and phishing-resistant authentication for high-risk applications.
