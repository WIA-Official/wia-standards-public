Chapter 4: Phase 1 - Data Format Standardization

弘益人間 (Benefit All Humanity)

4.1 The Foundation of Interoperability

Data format standardization forms the bedrock of the WIA-ROB-014 specification. Before devices can communicate effectively, they must speak a common language—not just in terms of communication protocols, but in how they structure, encode, and interpret information about pets, their activities, and their care.

Phase 1 establishes comprehensive data schemas covering every aspect of pet care: from basic identification and health profiles to feeding schedules, activity logs, behavioral patterns, and health metrics. These schemas ensure that a feeding record created by one device can be understood, validated, and utilized by any other compliant system.

4.2.2 Species Taxonomy

The standard uses scientific nomenclature for species identification, ensuring precision across languages and regions. Common names are supported as metadata but never used as primary identifiers.

felis_catus Domestic Cat Mammal - Carnivore 12-18 years
psittacus_erithacus African Grey Parrot Bird - Psittaciformes 40-60 years
oryctolagus_cuniculus Domestic Rabbit Mammal - Herbivore 8-12 years

4.3.3 Food Inventory Tracking

Inventory management prevents unexpected food shortages and enables automated reordering. The schema tracks multiple food types, batch information, and expiration dates.

Field Type Purpose Required
foodId UUID Unique identifier for food product Yes
currentLevel Number + Unit Remaining quantity Yes
capacity Number + Unit Maximum storage capacity Yes
reorderThreshold Number + Unit Trigger point for low-food alert Yes
expirationDate ISO 8601 Date Food freshness limit No
batchNumber String Manufacturing batch for recalls No

4.5.2 Health Trend Analysis

Aggregated health data over time reveals trends invisible in individual measurements. Weight changes, activity pattern shifts, and gradual vital sign deviations can indicate developing health issues.

Metric Aggregation Period Clinical Significance
Weight Weekly average >5% change warrants investigation
Activity Level 7-day rolling average >20% decrease may indicate illness
Heart Rate Variability Daily measurement Stress and autonomic function indicator
Sleep Duration Nightly total Changes correlate with health status
Water Consumption Daily total Increase may indicate diabetes or kidney issues

4.6.2 Severity Levels and Response

The standard defines five severity levels with corresponding response protocols.

Severity Description Response Time Examples
Critical Immediate danger to pet Instant notification, emergency contact Feeder jam during multi-day absence, critical vitals
High Requires prompt attention Immediate notification Missed feeding, persistent barking, health anomaly
Medium Noteworthy, monitor closely Standard notification Activity decrease, food running low
Low Informational Batched notification New behavior detected, minor schedule deviation
Info Non-urgent information Daily summary only Successful feeding, normal activity day

4.8 Data Validation and Quality

4.8.1 JSON Schema Validation

Every data schema has a corresponding JSON Schema definition enabling automated validation. Devices and applications can verify data integrity before processing, preventing errors and ensuring compliance.

4.8.2 Data Quality Indicators

Measurements include quality indicators reflecting confidence and accuracy. This allows downstream systems to weight data appropriately in analysis and decision-making.

Quality Level Definition Usage Recommendation
Excellent Calibrated sensor, ideal conditions Use for clinical decisions
Good Normal operation, minor interference Use for most purposes
Fair Suboptimal conditions, estimate Use for trends only
Poor High uncertainty, significant interference Informational only
Invalid Sensor error or impossible value Discard, do not use

4.9 Internationalization and Localization

4.9.1 Multi-Language Support

Text fields support Unicode and accept language tags (ISO 639-1). User-facing strings are separated from structured data, enabling translation without modifying core records.

4.9.2 Regional Adaptations

Units, date formats, and measurement preferences adapt to regional settings while maintaining standardized storage format. Display localization occurs at the application layer, not data layer.

4.10 Implementation Considerations

4.10.1 Backward Compatibility

Schemas use semantic versioning. Minor version updates add optional fields; major versions may restructure data. Parsers must gracefully handle unknown fields and support multiple schema versions.

4.10.2 Storage Efficiency

While JSON provides human readability, implementations may use Protocol Buffers or similar binary formats for storage and transmission efficiency. The standard provides transformation specifications between formats.

Key Takeaways

  • Data format standardization enables devices from different manufacturers to share information
  • Pet profiles accommodate diverse species while maintaining consistent structure
  • Feeding schemas capture both scheduled intentions and actual events with variance tracking
  • Activity and behavior data supports health monitoring and trend analysis
  • Vital signs include quality indicators and species-specific normal ranges
  • Alert classification prevents notification fatigue while ensuring urgent issues receive attention
  • Device registration enables capability discovery and proper feature negotiation
  • JSON Schema validation ensures data integrity across the ecosystem
  • Internationalization support enables global deployment without data fragmentation
