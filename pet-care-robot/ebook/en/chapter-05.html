Chapter 5: Phase 2 - API Interface Design

弘益人間 (Benefit All Humanity)

5.1 Building on Standardized Data

With data formats established in Phase 1, Phase 2 introduces the communication layer—standardized APIs that enable control, configuration, and data exchange across the pet care ecosystem. These RESTful interfaces provide the common language through which applications, devices, and services interact regardless of manufacturer or implementation details.

The API design philosophy prioritizes simplicity, consistency, and developer experience. Every endpoint follows predictable patterns, uses standard HTTP methods appropriately, and provides comprehensive error handling. Authentication and authorization protect sensitive operations while maintaining ease of integration.

5.2 API Architecture Principles

5.2.1 RESTful Design

The standard adopts REST architectural constraints: stateless communication, resource-based URIs, standard HTTP methods, and hypermedia-driven state transitions. This proven approach ensures compatibility with existing tools, frameworks, and developer knowledge.

Core REST Principles Applied:
  • Resource Identification: Every entity (pet, device, schedule) has a unique URI
  • Standard Methods: GET (retrieve), POST (create), PUT (update), DELETE (remove)
  • Stateless Operations: Each request contains all necessary information
  • JSON Responses: Consistent, self-describing response format
  • HTTP Status Codes: Meaningful error and success indicators
  • Versioning: API version in URI or Accept header

5.2.2 Base URL Structure

All API endpoints follow a consistent hierarchical structure reflecting resource relationships:

https://api.petcare.example.com/v1/{resource}/{id}/{sub-resource}

Examples:
https://api.petcare.example.com/v1/pets/uuid-123
https://api.petcare.example.com/v1/pets/uuid-123/feeding-schedule
https://api.petcare.example.com/v1/devices/feeder-456/status
https://api.petcare.example.com/v1/events/feeding?petId=uuid-123&date=2025-12-29

5.3 Authentication and Authorization

5.3.1 OAuth 2.0 Authentication Flow

The standard mandates OAuth 2.0 for user authentication, supporting multiple grant types for different use cases. This industry-standard approach enables secure third-party integrations without sharing credentials.

Grant Type Use Case Security Level
Authorization Code Web applications, third-party integrations High (with PKCE)
Client Credentials Device-to-device, service accounts High
Device Authorization Smart home devices, limited input devices Medium-High
Refresh Token Long-term access without re-authentication High

5.3.2 Access Token Structure

Access tokens use JWT (JSON Web Token) format, enabling stateless validation and containing necessary authorization claims.

{
  "header": {
    "alg": "RS256",
    "typ": "JWT"
  },
  "payload": {
    "iss": "https://auth.petcare.example.com",
    "sub": "user-uuid-789",
    "aud": "petcare-api",
    "exp": 1735574400,
    "iat": 1735488000,
    "scope": "pet:read pet:write device:control",
    "pets": ["pet-uuid-123", "pet-uuid-456"],
    "role": "owner"
  }
}

5.3.3 Permission Scopes

Fine-grained scopes enable principle of least privilege, granting only necessary permissions to each client.

Scope Description Typical Users
pet:read View pet profiles and data All authenticated users
pet:write Modify pet information Owners, authorized family
device:control Control feeding, dispense treats Owners, trusted users
health:read Access health metrics and history Owners, veterinarians
health:write Record health observations Veterinarians
admin:full Complete account management Primary account owner

5.4 Pet Management APIs

5.4.1 Pet Profile Operations

GET /v1/pets

Retrieve list of pets associated with authenticated user.

Response 200: 
{
  "pets": [
    {
      "id": "pet-uuid-123",
      "name": "Luna",
      "species": "canis_familiaris",
      "breed": "Golden Retriever",
      "age": 3,
      "profileUrl": "/v1/pets/pet-uuid-123"

  "total": 1

GET /v1/pets/{petId}

Retrieve complete profile for specific pet.

Response 200: Returns full pet profile schema (see Chapter 4)

Create new pet profile.

Request Body: 
{
  "name": "Max",
  "species": "felis_catus",
  "breed": "Maine Coon",
  "birthDate": "2023-05-20",
  "sex": "male"
}
Response 201: Created pet profile with generated ID
PUT /v1/pets/{petId}

Update existing pet profile.

Response 200: Updated pet profile

5.5.3 Feeding History

GET /v1/events/feeding?petId={petId}&startDate={date}&endDate={date}

Retrieve feeding event history with optional filtering.

Query Parameters:
  • petId: Filter by specific pet (required)
  • startDate: Beginning of date range (ISO 8601)
  • endDate: End of date range (ISO 8601)
  • limit: Maximum number of results (default 100)
  • offset: Pagination offset

5.10 Error Handling and Status Codes

5.10.1 Standard HTTP Status Codes

Code Meaning Usage
200 OK Successful GET, PUT requests
201 Created Successful POST creating new resource
202 Accepted Request accepted for async processing
204 No Content Successful DELETE, no response body
400 Bad Request Invalid input, validation errors
401 Unauthorized Missing or invalid authentication
403 Forbidden Authenticated but insufficient permissions
404 Not Found Resource doesn't exist
409 Conflict Resource state conflict
429 Too Many Requests Rate limit exceeded
500 Internal Server Error Unexpected server error
503 Service Unavailable Temporary server unavailability

5.10.2 Error Response Format

{
  "error": {
    "code": "INVALID_PORTION_SIZE",
    "message": "Portion size must be between 10 and 500 grams",
    "details": {
      "field": "amount",
      "providedValue": 750,
      "allowedRange": {
        "min": 10,
        "max": 500
      }
    },
    "timestamp": "2025-12-29T16:55:00Z",
    "requestId": "req-uuid-xyz"
  }
}

5.11 Rate Limiting and Quotas

5.11.1 Request Limits

To ensure system stability and fair resource allocation, the API enforces rate limits based on authentication tier and endpoint sensitivity.

Endpoint Category Free Tier Standard Tier Premium Tier
Read Operations (GET) 100/hour 1,000/hour 10,000/hour
Write Operations (POST/PUT) 50/hour 500/hour 5,000/hour
Device Control 20/hour 200/hour Unlimited
Video Streaming 2 hours/day 12 hours/day Unlimited

Key Takeaways

  • RESTful API design ensures compatibility with existing tools and developer familiarity
  • OAuth 2.0 authentication provides secure, industry-standard access control
  • Fine-grained permission scopes implement principle of least privilege
  • Consistent URL structures and HTTP methods create predictable, intuitive interfaces
  • Comprehensive error responses enable effective debugging and user feedback
  • Rate limiting protects system resources while supporting various use cases
  • Real-time capabilities (streaming, audio) integrate seamlessly with REST architecture
  • API versioning ensures backward compatibility as standard evolves
