弘益人間 (Benefit All Humanity)

6.1 Beyond Request-Response: Real-Time Communication

While RESTful APIs excel at structured data exchange and control operations, pet care applications demand real-time, bidirectional communication for critical functions. Watching a live video feed, receiving instant notifications when a pet approaches the camera, or dispensing treats with sub-second latency requires protocols optimized for continuous, low-latency data streams.

Phase 3 of WIA-ROB-014 introduces communication protocols that complement the REST API layer, enabling real-time monitoring, streaming media, and responsive interaction. These protocols maintain the standard's principles of security, interoperability, and simplicity while addressing the unique requirements of live pet interaction.

6.2 Protocol Stack Overview

6.2.1 Multi-Protocol Approach

Different use cases demand different communication patterns. Rather than forcing all interactions through a single protocol, the standard defines appropriate protocols for each scenario, with clear guidance on when to use each.

Protocol Use Cases Latency Overhead
WebSocket Event streams, notifications, telemetry 50-200ms Low
WebRTC Video/audio streaming, peer-to-peer 100-300ms Medium
MQTT IoT device communication, telemetry 20-100ms Very Low
RTSP Camera streaming (legacy support) 200-500ms Medium

6.2.2 Protocol Selection Decision Tree

Choosing the Right Protocol:
  • Video/Audio Streaming: WebRTC (preferred) or RTSP (fallback)
  • Real-time Notifications: WebSocket
  • Device Telemetry: MQTT (for devices) or WebSocket (for apps)
  • Command and Control: MQTT (device-initiated) or REST API (user-initiated)
  • Bulk Data Transfer: REST API with chunking

6.3.4 Heartbeat and Connection Management

Long-lived connections require heartbeat mechanisms to detect disconnections and maintain session state.

Message Type Direction Frequency Purpose
ping Server → Client Every 30 seconds Verify connection alive
pong Client → Server Response to ping Acknowledge connection
keep_alive Client → Server Every 60 seconds Prevent timeout
connection_timeout Server → Client After 90s silence Warning before disconnect

6.4 WebRTC Video Streaming

6.4.1 Peer-to-Peer Architecture

WebRTC enables direct peer-to-peer video streaming between pet cameras and viewing applications, reducing latency and server bandwidth compared to server-relayed streams. When direct connection is impossible due to network restrictions, TURN servers provide relay functionality.

WebRTC Connection Flow:
  1. Signaling: Client requests stream session via REST API
  2. ICE Candidate Exchange: Both peers gather and exchange network connection options
  3. SDP Offer/Answer: Negotiate codec, resolution, and streaming parameters
  4. Connection Establishment: Optimal connection path selected (direct or relayed)
  5. Media Streaming: Encrypted video/audio flows between peers
  6. Adaptive Quality: Bitrate and resolution adjust to network conditions

6.4.2 Session Initiation Protocol

POST /v1/devices/{deviceId}/camera/stream
Authorization: Bearer {token}

6.5 MQTT for IoT Devices

6.5.1 Why MQTT for Devices

IoT devices have constraints that REST APIs don't address well: limited memory, intermittent connectivity, battery power concerns, and need for efficient protocols. MQTT (Message Queuing Telemetry Transport) was designed specifically for these constraints.

MQTT Advantages for Pet Care Devices:
  • Lightweight: Minimal protocol overhead, ideal for embedded systems
  • Persistent Sessions: Messages queued during disconnection, delivered upon reconnection
  • Quality of Service: Configurable delivery guarantees (at-most-once, at-least-once, exactly-once)
  • Last Will and Testament: Automatic notification when device unexpectedly disconnects
  • Topic-based Routing: Efficient message distribution to interested subscribers

6.5.2 Topic Hierarchy

MQTT uses hierarchical topics for message routing. The standard defines a consistent topic structure for all pet care devices.

wia-rob-014/{accountId}/{deviceId}/{category}/{action}

Examples:
wia-rob-014/account-123/feeder-456/feeding/schedule-updated
wia-rob-014/account-123/feeder-456/status/online
wia-rob-014/account-123/camera-789/motion/detected
wia-rob-014/account-123/tracker-321/vitals/heartrate

6.5.3 Message Quality of Service

QoS Level Guarantee Overhead Use Cases
QoS 0 At most once (fire and forget) Minimal Frequent telemetry, non-critical status
QoS 1 At least once (acknowledged) Low Feeding events, alerts, commands
QoS 2 Exactly once (guaranteed) High Critical health alerts, medication

6.5.4 Device Status Management

MQTT's Last Will and Testament feature enables automatic offline detection and notification when devices lose connectivity.

// Device connects and sets Last Will
CONNECT
  client_id: feeder-456
  username: device-feeder-456
  password: {device-token}
  clean_session: false
  keep_alive: 60
  will_topic: wia-rob-014/account-123/feeder-456/status/lwt
  will_payload: {
    "status": "offline",
    "lastSeen": "2025-12-29T18:30:00Z",
    "reason": "unexpected_disconnect"
  }
  will_qos: 1
  will_retain: true

// Normal status updates
PUBLISH wia-rob-014/account-123/feeder-456/status/online
{
  "status": "online",
  "firmwareVersion": "2.4.1",
  "batteryLevel": 100,
  "foodLevel": 65
}

6.6.2 Event-Driven Notifications

Not all data should be transmitted continuously. Event-driven protocols reduce bandwidth by transmitting only when significant changes occur.

Event Type Trigger Condition Priority Delivery Method
Heart Rate Anomaly >20% deviation from baseline High MQTT QoS 2 + Push notification
Activity Change State transition (rest↔active) Medium MQTT QoS 1
Feeding Complete Portion dispensed and consumed Low MQTT QoS 1 + WebSocket
Motion Detected Camera motion sensor triggered Low WebSocket (if connected)

6.7 Treat Dispensing and Interactive Control

6.7.1 Low-Latency Command Execution

Interactive features like treat dispensing or laser pointer control require sub-second response times. The protocol optimizes for minimal latency from user action to device response.

Command Flow:
  1. User Action: User taps "Dispense Treat" button in app
  2. Command Transmission: App sends command via WebSocket or MQTT (≈50ms)
  3. Device Receipt: Device receives and validates command (≈20ms)
  4. Execution: Motor activates to dispense treat (≈200ms)
  5. Confirmation: Device reports completion (≈50ms)
  6. Total Latency: ~320ms from tap to treat dispensed

6.7.2 Command Acknowledgment Protocol

// User initiates treat dispense
{
  "messageType": "command",
  "commandId": "cmd-uuid-xyz",
  "deviceId": "feeder-uuid-456",
  "action": "dispense_treat",
  "parameters": {
    "count": 3,
    "playSound": true
  },
  "timestamp": "2025-12-29T18:45:00Z"
}

// Device acknowledges receipt
{
  "messageType": "command_ack",
  "commandId": "cmd-uuid-xyz",
  "status": "received",
  "estimatedExecution": "2025-12-29T18:45:00.500Z"
}

// Device reports execution
{
  "messageType": "command_complete",
  "commandId": "cmd-uuid-xyz",
  "status": "success",
  "actualExecution": "2025-12-29T18:45:00.520Z",
  "result": {
    "dispensed": 3,
    "soundPlayed": true
  }
}

6.8 Offline Operation and Synchronization

6.8.1 Edge Computing Capabilities

Devices must continue operating during internet outages. Critical functions like scheduled feeding execute locally with synchronization occurring upon reconnection.

Offline-First Design Principles:
  • Local Schedules: Feeding schedules stored in device memory, executed regardless of connectivity
  • Event Buffering: Feeding events, alerts, and telemetry queued locally during outage
  • Synchronization: Upon reconnection, buffered data transmitted to cloud
  • Conflict Resolution: Cloud-side changes during outage merged with local state
  • Capacity Limits: Devices buffer minimum 7 days of events before oldest data pruned

6.8.2 Synchronization Protocol

// Device reconnects after outage
{
  "messageType": "sync_request",
  "deviceId": "feeder-uuid-456",
  "lastSyncTime": "2025-12-28T10:30:00Z",
  "currentTime": "2025-12-29T19:00:00Z",
  "bufferedEvents": 142,
  "configVersion": "1.2.5"
}

// Server responds with sync strategy
{
  "messageType": "sync_response",

  "strategy": "incremental",
  "configUpdated": true,
  "newConfigVersion": "1.2.7",
  "requestBufferedEvents": true,
  "maxBatchSize": 50
}

6.9 Security Considerations

6.9.1 Transport Layer Security

All protocol communications must use TLS 1.3 or higher, with certificate validation enforced. Unencrypted connections are prohibited for production deployments.

6.9.2 Message Authentication

Every message includes authentication information preventing spoofing and replay attacks.

Protocol Authentication Method Token Lifetime
WebSocket JWT in connection handshake 1 hour (renewable)
WebRTC Short-lived credentials per session Session duration (max 2 hours)
MQTT Device certificates or JWT credentials 24 hours (auto-renewal)

6.10 Performance Optimization

6.10.1 Connection Pooling and Reuse

Establishing new connections is expensive. The protocol encourages connection reuse and multiplexing multiple logical streams over single physical connections.

6.10.2 Compression

Text-based protocols benefit from compression, especially for telemetry data with high redundancy.

Data Type Compression Typical Reduction
JSON Events gzip 60-70%
Telemetry Batches Brotli 70-80%
Video Streams Codec-native (H.264/VP8) N/A
Audio Opus codec N/A

Key Takeaways

  • Real-time protocols complement REST APIs for latency-sensitive operations
  • WebSocket enables efficient bidirectional event streaming and notifications
  • WebRTC provides low-latency peer-to-peer video streaming with adaptive quality
  • MQTT optimizes IoT device communication with minimal overhead and offline resilience
  • Quality of Service levels match message criticality to delivery guarantees
  • Offline-first design ensures critical functions continue during connectivity loss
  • Interactive features achieve sub-second latency through protocol optimization
  • Security remains paramount across all communication channels
