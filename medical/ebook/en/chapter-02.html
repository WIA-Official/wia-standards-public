HL7 Standards Family

Health Level Seven International (HL7) has been the primary standards development organization for healthcare data exchange since 1987, creating a family of standards that enable electronic health information sharing across diverse healthcare systems worldwide.

Introduction to HL7 International

Health Level Seven International is a not-for-profit, ANSI-accredited standards developing organization dedicated to providing comprehensive framework and related standards for the exchange, integration, sharing, and retrieval of electronic health information. The "Level Seven" refers to the seventh layer (Application) of the Open Systems Interconnection (OSI) communications model, reflecting the organization's focus on application-level data exchange rather than lower-level network protocols.

Founded in 1987, HL7 has grown to include members from over 55 countries, with affiliate organizations worldwide that adapt HL7 standards to local requirements. The organization operates through work groups that develop and maintain standards across various healthcare domains, from clinical and administrative data to public health reporting and medical device integration.

HL7 Version 2.x Messaging

HL7 Version 2.x (commonly written as HL7v2 or HL7 v2.x) is the most widely implemented healthcare messaging standard in the world. Despite being superseded by newer standards, HL7v2 continues to dominate healthcare data exchange, with an estimated 95% of US healthcare organizations using some version of the standard. Its longevity is a testament to its flexibility and the massive investment healthcare organizations have made in HL7v2 infrastructure.

Message Structure

HL7v2 messages are composed of segments, each representing a specific category of information. Segments are delimited by carriage returns and contain fields separated by pipe characters (|). Fields can contain subfields (components) separated by carets (^), and subcomponents separated by ampersands (&). This delimiter-based structure, while not as readable as XML or JSON, is extremely efficient and well-suited for high-volume healthcare transactions. 

MSH|^~\&|SENDING_APP|SENDING_FAC|RECEIVING_APP|RECEIVING_FAC|20250108120000||ADT^A01|MSG00001|P|2.5.1
EVN|A01|20250108120000
PID|1||123456789^^^MRN||Doe^John^Q||19850315|M|||123 Main St^^Springfield^IL^62701||555-1234|||S
PV1|1|I|ICU^101^A^Hospital|||^Smith^John^Dr|||MED||||||||12345678^Smith^John^Dr

In this example ADT (Admit-Discharge-Transfer) message, the MSH segment contains message header information including sending and receiving applications, timestamp, message type, and HL7 version. The EVN segment provides event information, PID contains patient identification details, and PV1 describes the patient visit.

Version Evolution

HL7v2 has evolved through numerous versions since its inception, with each version adding new functionality while maintaining backward compatibility where possible. The most commonly deployed versions today are 2.3, 2.3.1, 2.4, 2.5, 2.5.1, and 2.7, though organizations frequently implement customized "Z-segments" to extend the standard for local requirements.

HL7v2 Version Timeline

  • v2.0 (1988): Initial release with basic message types
  • v2.1 (1990): Added query/response mechanisms
  • v2.2 (1994): Enhanced segment definitions
  • v2.3 (1997): Major expansion, still widely used
  • v2.3.1 (1999): Clarifications and corrections
  • v2.4 (2000): Added XML encoding option
  • v2.5 (2003): Conformance framework introduced
  • v2.5.1 (2007): Security and privacy enhancements
  • v2.7 (2011): Latest major version
  • v2.9 (2019): Current release with incremental updates

HL7 Version 3 and the Reference Information Model

HL7 Version 3 (HL7v3) represented a fundamental redesign of the HL7 approach to healthcare data exchange. Unlike the pragmatic, segment-based approach of HL7v2, Version 3 was built on a formal methodology centered on the Reference Information Model (RIM), an object-oriented data model that attempts to represent all healthcare information in a consistent, rigorous framework.

The Reference Information Model (RIM)

The RIM is an abstract model that serves as the foundation for all HL7v3 artifacts. It defines six core classes—Act, Entity, Role, Participation, ActRelationship, and RoleLink—that can represent any healthcare concept. From these backbone classes, more specific classes are derived to represent clinical acts, observations, procedures, substances, organizations, persons, and their relationships.

RIM Class Purpose Examples
Act Actions that can be documented Observations, procedures, substance administrations
Entity Physical things with identity Persons, organizations, devices, materials
Role Competency of an entity Patient, provider, employee, specimen
Participation Association of role with act Author, performer, subject, location
ActRelationship Association between acts Component, reason, support, fulfillment
RoleLink Association between roles Direct authority, indirect authority

HL7v3 Message Development Framework

The HL7v3 methodology uses a series of models derived from the RIM to create message specifications. Domain Message Information Models (D-MIMs) constrain the RIM for specific domains like patient administration or laboratory. Refined Message Information Models (R-MIMs) further constrain D-MIMs for specific interactions. Hierarchical Message Descriptions (HMDs) define the XML structure of messages.

While this methodology provides rigor and consistency, it also introduces significant complexity. The learning curve for HL7v3 is steep, and the resulting XML messages are verbose compared to both HL7v2 and newer formats like FHIR. This complexity has limited HL7v3 adoption, with most implementations focused on specific use cases like the Clinical Document Architecture (CDA) rather than general messaging.

Clinical Document Architecture (CDA)

The Clinical Document Architecture (CDA) is arguably the most successful application of HL7v3 methodology. CDA provides a standard structure for clinical documents—the narrative and structured content that comprises the patient's medical record. Unlike transactional messages, CDA documents are designed to be persistent, stewardship-tracked, and potentially legally authenticated.

CDA Structure

A CDA document consists of a header and a body. The header contains metadata including patient identification, author information, custodian organization, encounter details, and document provenance. The body contains the clinical content, organized into sections that can contain both human-readable narrative and machine-processable structured entries.

CDA Document Components:
  • Header: Patient, author, custodian, dates, authentication
  • Body: Can be non-structured (narrative only) or structured
  • Sections: Organized clinical content with standardized codes
  • Entries: Machine-readable clinical statements
  • Narrative: Human-readable text required for every section

CDA Implementation Guides

The base CDA specification is highly flexible, which necessitates implementation guides that constrain CDA for specific use cases. The most important implementation guide in the United States is the Consolidated CDA (C-CDA), which defines templates for common document types and sections.

Challenges with Legacy HL7 Standards

While HL7v2 and HL7v3/CDA have enabled tremendous progress in healthcare data exchange, both standards face significant challenges in the modern healthcare technology landscape.

HL7v2 Limitations

  • Optionality: The standard allows extensive optionality, meaning two "HL7-compliant" systems may be incompatible without additional interface work.
  • Z-Segments: Organizations extend HL7v2 with custom Z-segments, creating proprietary variations that hinder interoperability.
  • No REST Support: HL7v2 was designed for point-to-point messaging, not modern RESTful API architectures.
  • Limited Querying: While HL7v2 includes query mechanisms, they are cumbersome compared to modern API patterns.
  • Binary Encoding: The delimiter-based format, while efficient, is less developer-friendly than JSON or XML.

HL7v3/CDA Limitations

  • Complexity: The RIM-based methodology requires extensive training and creates verbose messages.
  • Implementation Burden: The formal approach creates high barriers to entry for implementers.
  • Limited Adoption: Outside of CDA, HL7v3 messaging has seen limited real-world deployment.
  • Static Documents: CDA's document-centric approach doesn't align with modern mobile and API needs.

The Path to FHIR

The limitations of existing HL7 standards, combined with advances in web technology, led HL7 to develop FHIR (Fast Healthcare Interoperability Resources). FHIR was designed to learn from both the practical success of HL7v2 and the rigorous modeling of HL7v3, while embracing modern web standards like REST, JSON, and OAuth. HL7 in Practice: Integration Patterns

Healthcare organizations typically implement HL7 standards through integration engines—middleware platforms that route, transform, and validate messages between clinical systems. Understanding common integration patterns is essential for healthcare IT professionals.

Point-to-Point Integration

In the simplest pattern, systems exchange HL7 messages directly. While straightforward for small numbers of systems, this approach quickly becomes unmanageable as the number of interfaces grows exponentially with each new system (n × (n-1) / 2 connections for n systems).

Hub-and-Spoke Integration

Most organizations implement a central integration engine (hub) that receives messages from source systems and routes them to destination systems (spokes). This reduces the number of interfaces to n connections and centralizes message transformation, routing logic, and error handling.

Enterprise Service Bus

Larger organizations may implement an Enterprise Service Bus (ESB) architecture that adds sophisticated routing, orchestration, and service mediation capabilities beyond simple message transformation. ESB patterns support complex workflows that span multiple systems and message types.

Summary

Key Takeaways

  • HL7 International: Founded in 1987, HL7 is the primary SDO for healthcare data exchange, with members in 55+ countries.
  • HL7v2 Dominance: Despite its age, HL7v2 remains the most widely deployed healthcare messaging standard, used by 95% of US healthcare organizations.
  • Segment-Based Structure: HL7v2 messages consist of segments (MSH, PID, OBR, OBX, etc.) with pipe-delimited fields, enabling efficient data exchange.
  • HL7v3 and RIM: Version 3 introduced a formal methodology based on the Reference Information Model, providing rigor but adding complexity.
  • CDA Success: The Clinical Document Architecture is HL7v3's most successful application, enabling standardized clinical document exchange.
  • Legacy Limitations: Both HL7v2 and HL7v3 face challenges in modern healthcare IT, including lack of REST support and implementation complexity.
  • Integration Engines: Healthcare organizations use integration engines to route, transform, and validate HL7 messages between systems.
