FHIR: Fast Healthcare Interoperability Resources

FHIR (Fast Healthcare Interoperability Resources) represents a paradigm shift in healthcare data exchange, combining the best aspects of HL7's previous standards with modern web technologies to create a flexible, implementer-friendly standard for the 21st century.

Introduction to FHIR

FHIR was first published by HL7 International in 2014 as a draft standard for trial use (DSTU). It emerged from recognition that existing standards—while valuable—were not meeting the needs of modern healthcare IT. HL7v2, though widely deployed, lacked support for RESTful APIs and modern data formats. HL7v3, while rigorous, proved too complex for practical implementation. FHIR was designed to learn from both, providing a standard that is both implementable and well-structured.

The name "FHIR" (pronounced "fire") reflects the standard's focus on speed and ease of implementation. Where HL7v3 projects often took years to complete, FHIR implementations can be prototyped in days. This rapid development cycle, combined with modern web technologies familiar to developers outside healthcare, has driven explosive growth in FHIR adoption.

FHIR Architecture and Design Principles

FHIR's architecture is built on several core principles that distinguish it from earlier healthcare standards and make it accessible to developers with general web development experience.

Resource-Centric Design

At the heart of FHIR is the concept of "resources"—discrete units of healthcare information that can be independently created, read, updated, and deleted. Unlike HL7v2's message-centric approach or HL7v3's document-centric approach, FHIR treats healthcare data as a collection of RESTful resources that can be accessed and manipulated through standard HTTP operations.

Multiple Representation Formats

FHIR resources can be represented in multiple formats, with JSON and XML being the most common. JSON has become the dominant format for new implementations due to its compact size, broad tool support, and familiarity to web developers. XML remains supported for compatibility and use cases requiring more robust validation.

RESTful API

FHIR defines a RESTful API that leverages standard HTTP methods for resource manipulation. This design choice makes FHIR immediately accessible to any developer familiar with REST APIs, regardless of their healthcare experience. The API supports standard CRUD operations as well as healthcare-specific operations like $everything (retrieving all information about a patient) and $validate (validating resource content).

HTTP Method FHIR Operation Description Example
GET Read Retrieve a resource GET /Patient/123
GET Search Search for resources GET /Patient?name=Smith
POST Create Create a new resource POST /Patient
PUT Update Update entire resource PUT /Patient/123
PATCH Patch Partial update PATCH /Patient/123
DELETE Delete Remove a resource DELETE /Patient/123

FHIR Resources

FHIR defines over 150 resource types organized into categories including Foundation, Base, Clinical, Financial, and Specialized. Each resource represents a specific concept in healthcare and has a defined structure with required and optional elements.

Foundation Resources

Foundation resources support the FHIR infrastructure itself. CapabilityStatement describes what a server can do, OperationDefinition defines custom operations, StructureDefinition describes resource profiles, and Bundle groups resources together for transactions or documents.

Clinical Resources

FHIR Search

FHIR's search capability is one of its most powerful features, allowing clients to find resources based on various criteria. Search parameters are defined for each resource type and can be combined to create complex queries.

Search Parameter Types

  • Number: Numeric comparisons (quantity values)
  • Date: Date/time matching with precision
  • String: String matching (starts with by default)
  • Token: Coded values, identifiers
  • Reference: References to other resources
  • Composite: Multiple parameters combined
  • Quantity: Measured values with units
  • URI: URI matching

Search Examples

# Find patient by name
GET /Patient?name=Smith

# Find patients born after 1985
GET /Patient?birthdate=gt1985-01-01

# Find observations for a specific patient
GET /Observation?subject=Patient/123

# Find active conditions with specific code
GET /Condition?clinical-status=active&code=http://snomed.info/sct|73211009

# Include related resources in results
GET /MedicationRequest?_include=MedicationRequest:medication

# Chain through references
GET /Observation?subject.name=Smith

US Core Implementation Guide

While FHIR provides a flexible framework, real-world interoperability requires agreement on how resources should be constrained for specific use cases. In the United States, the US Core Implementation Guide defines the minimum requirements for FHIR implementations to be conformant with US regulatory requirements.

US Core Key Requirements:
  • USCDI Data Classes: Profiles for each US Core Data for Interoperability element
  • Must Support: Defines which elements implementations must handle
  • Search Parameters: Mandatory search capabilities for each profile
  • Capability Statement: Required server capabilities
  • Security: SMART on FHIR authorization required

US Core Profiles

Profile Base Resource Key Constraints
US Core Patient Patient Race, ethnicity, birth sex extensions
US Core Condition Condition SNOMED or ICD-10 coding required
US Core Observation Observation LOINC codes, vital signs profiles
US Core Medication Medication RxNorm coding required
US Core AllergyIntolerance AllergyIntolerance SNOMED for substances
US Core Immunization Immunization CVX vaccine codes

SMART on FHIR

SMART on FHIR (Substitutable Medical Applications, Reusable Technologies) is a set of open specifications that enable third-party applications to launch from within EHR systems and access patient data through FHIR APIs. SMART provides a standardized authorization framework built on OAuth 2.0 and OpenID Connect.

SMART App Launch

The SMART App Launch Framework defines how applications authenticate with FHIR servers and obtain authorization to access specific patient data. The framework supports two launch contexts: EHR launch (application launched from within the EHR) and standalone launch (application launched independently).

SMART Launch Sequence

  1. Discovery: App queries /.well-known/smart-configuration for endpoints
  2. Authorization: App redirects user to authorization endpoint
  3. Authentication: User authenticates with EHR
  4. Consent: User approves requested scopes
  5. Token Exchange: App receives access token
  6. API Access: App uses token to access FHIR resources

SMART Scopes

SMART defines a scope syntax that specifies what resources and operations an application can access. Scopes follow the pattern: [context]/[resource].[permission] 

# Patient-level scopes (launched in patient context)
patient/Patient.read
patient/Observation.read
patient/MedicationRequest.read

# User-level scopes (all patients the user can access)
user/Patient.read
user/*.read

# System-level scopes (backend services)
system/Patient.read
system/*.read

# Special scopes
openid          # OpenID Connect authentication
fhirUser        # Get identity of current user
launch          # Receive launch context
launch/patient  # Receive patient context
offline_access  # Refresh tokens

FHIR Implementation Strategies

Greenfield vs. Brownfield

Greenfield FHIR implementations—new systems built natively on FHIR—can take full advantage of FHIR's resource model and RESTful architecture. Brownfield implementations must integrate FHIR with existing systems, often requiring translation layers between legacy HL7v2 interfaces and new FHIR APIs.

FHIR Facade Pattern

Many organizations implement FHIR as a "facade" over existing data systems. The FHIR server translates incoming FHIR requests into queries against legacy databases and transforms the results into FHIR resources. This approach allows organizations to expose standardized FHIR APIs without replacing existing systems.

Bulk Data Access

FHIR Bulk Data Access (also known as "Flat FHIR") enables efficient export of large datasets for analytics, research, and population health. The specification defines asynchronous operations for exporting data in NDJSON (Newline Delimited JSON) format, allowing clients to retrieve complete patient populations without individual resource requests.

Summary

Key Takeaways

  • Modern Standard: FHIR combines HL7's healthcare expertise with modern web technologies (REST, JSON, OAuth).
  • Resource-Centric: Over 150 resource types represent healthcare concepts as independently accessible entities.
  • RESTful API: Standard HTTP methods (GET, POST, PUT, DELETE) enable familiar development patterns.
  • Multiple Formats: Resources can be represented in JSON or XML, with JSON preferred for new implementations.
  • Powerful Search: Flexible search parameters support complex queries across resources.
  • US Core: Implementation guide defines minimum requirements for US regulatory compliance.
  • SMART on FHIR: OAuth-based authorization enables secure third-party application access.
  • R4 Normative: Release 4 is the first normative version and is required by ONC regulations.
