FHIR: Fast Healthcare Interoperability Resources

FHIR (Fast Healthcare Interoperability Resources)는 의료 데이터 교환의 패러다임 전환을 나타내며, HL7의 이전 표준의 가장 좋은 측면을 현대 웹 기술과 결합하여 21세기를 위한 유연하고 구현자 친화적인 표준을 만들었습니다.

FHIR 소개

FHIR는 2014년에 HL7 International에 의해 시험 사용을 위한 표준 초안 (DSTU)으로 처음 발표되었습니다. 기존 표준이 가치가 있지만 현대 의료 IT의 요구를 충족하지 못한다는 인식에서 나왔습니다. HL7v2는 널리 배포되었지만 RESTful API와 현대 데이터 형식에 대한 지원이 부족했습니다. HL7v3는 엄격했지만 실제 구현에는 너무 복잡한 것으로 판명되었습니다. FHIR는 둘 다에서 배우도록 설계되어 구현 가능하고 잘 구조화된 표준을 제공합니다.

"FHIR" (파이어로 발음)라는 이름은 속도와 구현 용이성에 대한 표준의 초점을 반영합니다. HL7v3 프로젝트가 종종 완료하는 데 수년이 걸리는 것과 달리 FHIR 구현은 며칠 만에 프로토타입을 만들 수 있습니다. 이러한 빠른 개발 주기와 의료 외부 개발자에게 익숙한 현대 웹 기술이 FHIR 채택의 폭발적인 성장을 이끌었습니다.

FHIR 아키텍처 및 설계 원칙

FHIR의 아키텍처는 초기 의료 표준과 구별하고 일반 웹 개발 경험이 있는 개발자가 접근할 수 있게 하는 여러 핵심 원칙 위에 구축되었습니다.

리소스 중심 설계

FHIR의 핵심은 "리소스" 개념입니다—독립적으로 생성, 읽기, 업데이트 및 삭제할 수 있는 의료 정보의 개별 단위입니다. HL7v2의 메시지 중심 접근 방식이나 HL7v3의 문서 중심 접근 방식과 달리 FHIR는 의료 데이터를 표준 HTTP 작업을 통해 접근하고 조작할 수 있는 RESTful 리소스 컬렉션으로 취급합니다.

RESTful API

FHIR는 리소스 조작을 위한 표준 HTTP 메서드를 활용하는 RESTful API를 정의합니다. 이 설계 선택은 의료 경험에 관계없이 REST API에 익숙한 모든 개발자가 FHIR에 즉시 접근할 수 있게 합니다. API는 표준 CRUD 작업뿐만 아니라 $everything (환자에 대한 모든 정보 검색) 및 $validate (리소스 콘텐츠 검증)와 같은 의료 특정 작업을 지원합니다.

HTTP 메서드 FHIR 작업 설명 예시
GET Read 리소스 검색 GET /Patient/123
GET Search 리소스 검색 GET /Patient?name=김
POST Create 새 리소스 생성 POST /Patient
PUT Update 전체 리소스 업데이트 PUT /Patient/123
PATCH Patch 부분 업데이트 PATCH /Patient/123
DELETE Delete 리소스 제거 DELETE /Patient/123

FHIR 리소스

FHIR는 Foundation, Base, Clinical, Financial 및 Specialized를 포함한 범주로 구성된 150개 이상의 리소스 유형을 정의합니다. 각 리소스는 의료의 특정 개념을 나타내며 필수 및 선택적 요소가 있는 정의된 구조를 가집니다.

임상 리소스

리소스 설명 주요 요소
Patient 인구통계 및 식별자 identifier, name, gender, birthDate, address
Observation 측정 및 소견 code, value, effectiveDateTime, status
Condition 진단 및 문제 code, clinicalStatus, subject, onsetDateTime
Medication 약물 정의 code, form, ingredient, amount
MedicationRequest 처방 주문 medication, subject, dosageInstruction, status
Procedure 임상 절차 code, subject, performedDateTime, performer
DiagnosticReport 검사/영상 결과 code, result, effectiveDateTime, conclusion
Encounter 환자 방문 class, status, subject, period, participant

FHIR 검색

FHIR의 검색 기능은 가장 강력한 기능 중 하나로, 클라이언트가 다양한 기준에 따라 리소스를 찾을 수 있게 합니다. 검색 매개변수는 각 리소스 유형에 대해 정의되며 복잡한 쿼리를 만들기 위해 결합할 수 있습니다.

검색 매개변수 유형

  • Number: 숫자 비교 (수량 값)
  • Date: 정밀도가 있는 날짜/시간 매칭
  • String: 문자열 매칭 (기본적으로 시작)
  • Token: 코딩된 값, 식별자
  • Reference: 다른 리소스에 대한 참조
  • Composite: 결합된 여러 매개변수
  • Quantity: 단위가 있는 측정값
  • URI: URI 매칭

검색 예시

# 이름으로 환자 찾기
GET /Patient?name=김

# 1985년 이후 출생 환자 찾기
GET /Patient?birthdate=gt1985-01-01

# 특정 환자의 관찰 찾기
GET /Observation?subject=Patient/123

# 특정 코드가 있는 활성 상태 찾기
GET /Condition?clinical-status=active&code=http://snomed.info/sct|73211009

# 관련 리소스 포함
GET /MedicationRequest?_include=MedicationRequest:medication

# 참조를 통한 체인
GET /Observation?subject.name=김

US Core Implementation Guide

FHIR가 유연한 프레임워크를 제공하는 반면, 실제 상호운용성은 특정 사용 사례에 대해 리소스가 어떻게 제약되어야 하는지에 대한 합의가 필요합니다. 미국에서는 US Core Implementation Guide가 미국 규제 요구사항을 준수하기 위한 FHIR 구현의 최소 요구사항을 정의합니다.

US Core 주요 요구사항:
  • USCDI 데이터 클래스: 각 US Core Data for Interoperability 요소에 대한 프로파일
  • Must Support: 구현이 처리해야 하는 요소 정의
  • 검색 매개변수: 각 프로파일에 대한 필수 검색 기능
  • Capability Statement: 필수 서버 기능
  • 보안: SMART on FHIR 인증 필수

SMART on FHIR

SMART on FHIR (Substitutable Medical Applications, Reusable Technologies)는 타사 애플리케이션이 EHR 시스템 내에서 실행되고 FHIR API를 통해 환자 데이터에 접근할 수 있게 하는 개방형 사양 세트입니다. SMART는 OAuth 2.0 및 OpenID Connect를 기반으로 구축된 표준화된 인증 프레임워크를 제공합니다.

SMART 실행 시퀀스

  1. Discovery: 앱이 /.well-known/smart-configuration에서 엔드포인트 쿼리
  2. Authorization: 앱이 사용자를 인증 엔드포인트로 리디렉션
  3. Authentication: 사용자가 EHR로 인증
  4. Consent: 사용자가 요청된 범위 승인
  5. Token Exchange: 앱이 액세스 토큰 수신
  6. API Access: 앱이 토큰을 사용하여 FHIR 리소스에 접근

SMART 범위

SMART는 애플리케이션이 접근할 수 있는 리소스와 작업을 지정하는 범위 구문을 정의합니다. 범위는 다음 패턴을 따릅니다: [context]/[resource].[permission] 

# 환자 수준 범위 (환자 맥락에서 실행)
patient/Patient.read
patient/Observation.read
patient/MedicationRequest.read

# 사용자 수준 범위 (사용자가 접근할 수 있는 모든 환자)
user/Patient.read
user/*.read

# 시스템 수준 범위 (백엔드 서비스)
system/Patient.read
system/*.read

# 특수 범위
openid          # OpenID Connect 인증
fhirUser        # 현재 사용자의 신원 가져오기
launch          # 실행 컨텍스트 수신
launch/patient  # 환자 컨텍스트 수신
offline_access  # 새로고침 토큰

요약

핵심 요점

  • 현대적 표준: FHIR는 HL7의 의료 전문성을 현대 웹 기술 (REST, JSON, OAuth)과 결합합니다.
  • 리소스 중심: 150개 이상의 리소스 유형이 의료 개념을 독립적으로 접근 가능한 엔티티로 표현합니다.
  • RESTful API: 표준 HTTP 메서드 (GET, POST, PUT, DELETE)가 익숙한 개발 패턴을 가능하게 합니다.
  • 다중 형식: 리소스는 JSON 또는 XML로 표현할 수 있으며 새 구현에는 JSON이 선호됩니다.
  • 강력한 검색: 유연한 검색 매개변수가 리소스 간 복잡한 쿼리를 지원합니다.
  • US Core: 구현 가이드가 미국 규제 준수를 위한 최소 요구사항을 정의합니다.
  • SMART on FHIR: OAuth 기반 인증이 안전한 타사 애플리케이션 접근을 가능하게 합니다.
  • R4 규범적: Release 4는 첫 번째 규범적 버전이며 ONC 규정에 의해 필수입니다.
