Chapter 7

Security, Privacy & OTA Updates

Secure boot, TrustZone, hardware root of trust, encrypted communication, OTA firmware updates with rollback protection

Overview

Security represents a fundamental requirement for IoT devices, not an optional feature. Smart sensors collect sensitive data (health metrics, location, behavioral patterns), connect to networks vulnerable to attack, and often operate in physically accessible locations where tampering is possible. The WIA-SEMI-015 standard mandates multiple security layers including secure boot, encrypted communication, and secure OTA updates to protect against evolving threat landscapes.

This chapter explores comprehensive security frameworks for smart sensors, from hardware root of trust through cryptographic protocols to secure firmware update mechanisms. Privacy-preserving techniques enable edge AI to keep sensitive data local rather than transmitting to cloud servers. Understanding and implementing these protections is essential for WIA-SEMI-015 compliance and customer trust.

Key Takeaways

  • Secure boot verifies firmware integrity before execution (RSA-2048, ECDSA P-256)
  • Hardware root of trust (TrustZone, secure element) protects cryptographic keys
  • Encrypted communication mandatory: TLS 1.3, DTLS 1.2, or AES-128-GCM minimum
  • OTA updates require signature verification, dual-bank storage, rollback protection
  • Edge AI preserves privacy by processing sensitive data locally

Threat Landscape for Smart Sensors

Understanding potential threats guides security architecture decisions. Smart sensors face multiple attack vectors, each requiring specific countermeasures.

Physical Attacks

Device tampering involves opening enclosure to access circuit board, probing debug interfaces (SWD, JTAG), and extracting firmware from flash memory. Side-channel attacks observe power consumption (differential power analysis), electromagnetic emissions (EM analysis), and timing variations (timing attacks). Protection includes tamper-evident seals, disabling debug interfaces in production, flash readout protection, constant-time cryptographic implementations, and power consumption randomization.

Network Attacks

Eavesdropping intercepts wireless communications to capture sensitive data or credentials. Man-in-the-middle (MITM) attacks impersonate legitimate servers or devices. Replay attacks capture and retransmit valid messages. Denial of service (DoS) floods device or network to disrupt operation. Countermeasures include end-to-end encryption (TLS/DTLS), mutual authentication (certificates), message freshness (nonces, timestamps), and rate limiting plus packet filtering.

Software Attacks

Malicious firmware update attempts install compromised firmware. Exploit attacks target buffer overflows, integer overflows, or format string vulnerabilities. Privilege escalation bypasses security boundaries. Protection requires signature verification for updates, secure coding practices, memory protection (MPU/MMU), and TrustZone or secure enclaves.

Secure Boot

Secure boot ensures that only authenticated firmware executes on the device, preventing malware installation even if attackers gain physical access. The WIA-SEMI-015 standard requires secure boot with RSA-2048, ECDSA P-256, or stronger signature algorithms.

Secure Boot Process

The boot sequence begins with immutable boot ROM (first-stage bootloader, factory-programmed, cannot be modified). It verifies second-stage bootloader signature using public key stored in OTP fuses or ROM. If valid, execution transfers to second-stage bootloader, which verifies application firmware signature using public key chain. If valid, execution transfers to application firmware. Any signature verification failure halts boot process.

Cryptographic Signature Schemes

RSA-2048 with SHA-256 provides 2048-bit keys with well-established security and widespread implementation, but has slower verification (50-200ms on Cortex-M4) and larger signatures (256 bytes). ECDSA P-256 with SHA-256 offers 256-bit keys with equivalent security to RSA-3072, faster verification (20-50ms on Cortex-M4), and smaller signatures (64 bytes), making it preferred for resource-constrained devices. Ed25519 (EdDSA) provides fast verification (10-30ms), 256-bit keys, simple implementation without complex parameter selection, and 64-byte signatures. 

// Example: Secure boot verification (simplified)
bool verify_firmware_signature(
    const uint8_t *firmware, size_t fw_len,
    const uint8_t *signature, size_t sig_len,
    const uint8_t *public_key)
{
    uint8_t hash[32];  // SHA-256 digest

    // Compute firmware hash
    sha256_compute(firmware, fw_len, hash);

    // Verify signature using ECDSA P-256
    if (ecdsa_p256_verify(hash, 32, signature, public_key)) {
        return true;  // Signature valid, boot firmware
    }

    // Signature invalid, halt boot
    return false;
}

Key Storage and Protection

Public keys for signature verification must be protected from modification. Options include OTP (One-Time Programmable) fuses (permanent, cannot be changed, highest security), immutable boot ROM (public key embedded at manufacturing), and flash readout protection (prevents external reading of flash contents).

Hardware Root of Trust

A hardware root of trust provides a secure foundation for cryptographic operations, protecting private keys from extraction even if firmware is compromised. The WIA-SEMI-015 standard requires secure key storage in hardware secure element, OTP fuses, or TrustZone secure world for Level 3 compliance.

ARM TrustZone-M

TrustZone-M (available on Cortex-M33, M35P, M55) partitions the system into secure world (isolated execution environment, access to all resources) and non-secure world (normal application code, restricted access). Secure code manages cryptographic keys, performs sensitive operations (signature verification, decryption), and provides API to non-secure world. Attack surface is minimized by keeping secure world code small and auditable.

Dedicated Secure Elements

Discrete secure elements provide tamper-resistant key storage and crypto operations. Options include NXP EdgeLock SE050 (I2C interface, ECC/RSA/AES, Common Criteria EAL 6+ certified), Microchip ATECC608 (I2C, ECDSA/ECDH/SHA-256, low cost ~$1), and Infineon OPTIGA Trust M (I2C, ECC/RSA/AES, automotive qualified). These devices protect private keys (never leave secure element), accelerate crypto operations (hardware offload), and provide certified tamper resistance.

Encrypted Communication

The WIA-SEMI-015 standard mandates encrypted wireless communication using TLS 1.3, DTLS 1.2, or AES-128-GCM minimum. This protects sensor data from eavesdropping and tampering during transmission.

TLS 1.3 for TCP-Based Protocols

TLS 1.3 provides encrypted, authenticated communication over TCP, suitable for Wi-Fi, cellular (NB-IoT, LTE-M), and Ethernet connections. Features include improved handshake (1-RTT, reduced latency), mandatory perfect forward secrecy (PFS), removal of obsolete ciphers (RC4, 3DES, SHA-1), and mutual authentication with client certificates.

DTLS 1.2 for UDP-Based Protocols

DTLS (Datagram TLS) adapts TLS for UDP transport, essential for CoAP and UDP-based IoT protocols. It handles packet loss and reordering, supports connection-less operation, and provides same security guarantees as TLS. Implementations include Mbed TLS (ARM, small footprint ~60KB code + 30KB RAM), wolfSSL (optimized for embedded, TLS 1.3 + DTLS 1.3), and TinyDTLS (minimal DTLS 1.2, <20KB code).

Application-Layer Encryption

For protocols without TLS/DTLS support, application-layer encryption secures payloads. AES-128-GCM provides authenticated encryption (confidentiality + integrity), 128-bit keys (sufficient for most applications), GCM mode with built-in authentication tag, and hardware acceleration on many MCUs (AES-NI, CRYPTO peripheral). Message format includes IV/nonce (96 bits, unique per message), ciphertext (encrypted payload), and authentication tag (128 bits, verifies integrity).

Over-The-Air (OTA) Firmware Updates

OTA updates enable remote firmware maintenance, security patching, and feature enhancements. The WIA-SEMI-015 standard requires OTA capability with cryptographic signature verification, atomic updates (dual-bank or equivalent), and rollback protection.

Dual-Bank Flash Architecture

Dual-bank architecture maintains two firmware images, allowing safe updates. Bank A holds currently running firmware, while Bank B receives new firmware download. After successful download and signature verification, the device reboots to Bank B. If Bank B boots successfully, it becomes the active firmware. If Bank B fails to boot, watchdog triggers rollback to Bank A. After successful operation, Bank A can be erased for the next update.

Update Process Flow

The secure update process begins with the server signing new firmware with private key. The device downloads firmware to inactive bank (Bank B), verifies signature using public key, and halts update if verification fails. Upon successful verification, the device sets boot flag to attempt Bank B, reboots and executes Bank B firmware, validates successful operation (application-specific checks), and commits update (marks Bank B as permanent). Failure at any stage triggers rollback to Bank A. 

// Example: OTA update verification and installation
typedef enum {
    OTA_SUCCESS,
    OTA_SIGNATURE_INVALID,
    OTA_FLASH_ERROR,
    OTA_ROLLBACK
} ota_status_t;

ota_status_t ota_install_update(
    const uint8_t *firmware_image, size_t fw_size,
    const uint8_t *signature, size_t sig_size)
{
    // 1. Verify signature
    if (!verify_firmware_signature(firmware_image, fw_size,
                                   signature, sig_size,
                                   ota_public_key)) {
        return OTA_SIGNATURE_INVALID;
    }

    // 2. Erase inactive bank (Bank B)
    if (!flash_erase(BANK_B_ADDRESS, fw_size)) {
        return OTA_FLASH_ERROR;
    }

    // 3. Write firmware to Bank B
    if (!flash_write(BANK_B_ADDRESS, firmware_image, fw_size)) {
        return OTA_FLASH_ERROR;
    }

    // 4. Verify written data
    if (!flash_verify(BANK_B_ADDRESS, firmware_image, fw_size)) {
        return OTA_FLASH_ERROR;
    }

    // 5. Set boot flag to attempt Bank B
    set_boot_bank(BANK_B);

    // 6. Reboot to new firmware
    system_reboot();

    return OTA_SUCCESS;  // Not reached
}

Update Size and Delta Updates

Full firmware updates transfer entire image (100KB-1MB typical), simple to implement and verify, but require significant bandwidth and time. Delta updates transfer only differences from current firmware (10-50KB typical for minor updates), reduce bandwidth by 80-95%, but require delta computation on server and patching on device. Binary diff algorithms include bsdiff (general purpose, compression-friendly) and Courgette (optimized for executable binaries).

Privacy-Preserving Edge AI

Edge AI enhances privacy by processing sensitive data locally rather than transmitting raw data to cloud servers. This approach aligns with regulations like GDPR and HIPAA.

Local Processing Benefits

Data minimization transmits only inference results (classifications, anomaly scores) rather than raw sensor data. Reduced attack surface eliminates cloud-side data breaches from processing. User control allows users to understand what data leaves device. Compliance with privacy regulations is simplified when sensitive data never leaves device.

Differential Privacy Techniques

For applications requiring aggregated data collection, differential privacy adds calibrated noise to prevent individual identification while preserving statistical properties. Local differential privacy (LDP) adds noise on-device before transmission, federated learning trains models on distributed devices without centralizing data, and secure aggregation combines individual contributions without revealing them.

Security Testing and Validation

WIA-SEMI-015 compliance requires documented security testing. Essential tests include penetration testing (attempt firmware extraction, debug interface access, and wireless protocol attacks), fuzzing (send malformed inputs to wireless stack, firmware update mechanism, and sensor interfaces), and cryptographic validation (verify proper random number generation, correct cryptographic implementation, and secure key storage).

Communication Protocols

Effective communication is essential for smart sensors to transmit data, receive commands, and integrate into larger systems. This chapter covers the protocols and standards that enable reliable, efficient sensor communication across wired and wireless interfaces.

Related Standard: WIA-UNIVERSAL-PROTOCOL provides the foundational communication framework that WIA-SMART-SENSOR builds upon for interoperability.

Wired Communication Interfaces

I²C (Inter-Integrated Circuit)

I²C is a two-wire serial protocol widely used for sensor interfacing due to its simplicity and multi-device support. The protocol uses open-drain signaling with pull-up resistors, supporting multiple devices on a single bus through addressing. Standard mode operates at 100 kHz, Fast mode at 400 kHz, and Fast mode Plus at 1 MHz.

Parameter Standard Fast Fast+ High-Speed
Max Clock 100 kHz 400 kHz 1 MHz 3.4 MHz
Max Capacitance 400 pF 400 pF 550 pF 100 pF
Addressing 7-bit (128 devices) or 10-bit (1024 devices)
// I²C Sensor Read Example #define SENSOR_ADDR 0x48 #define TEMP_REG 0x00 int16_t read_temperature_i2c(I2C_HandleTypeDef* hi2c) { uint8_t data[2]; // Write register address HAL_I2C_Master_Transmit(hi2c, SENSOR_ADDR << 1, &TEMP_REG, 1, 100); // Read temperature data HAL_I2C_Master_Receive(hi2c, SENSOR_ADDR << 1, data, 2, 100); // Convert to temperature (example: 12-bit, 0.0625°C resolution) int16_t raw = (data[0] << 4) | (data[1] >> 4); if (raw & 0x800) raw |= 0xF000; // Sign extend return raw; // Returns temperature in 1/16 °C units }

SPI (Serial Peripheral Interface)

SPI provides higher throughput than I²C through full-duplex operation and clock speeds up to 100 MHz. The four-wire interface (MOSI, MISO, SCK, CS) enables simple, fast communication at the cost of additional pins. SPI is preferred for high-data-rate sensors like accelerometers and ADCs.

UART (Universal Asynchronous Receiver-Transmitter)

UART provides simple point-to-point serial communication without a clock signal. Commonly used for GPS modules, cellular modems, and debugging interfaces. Standard baud rates range from 9600 to 115200, with some applications using higher rates up to several Mbps.

Wireless Communication Protocols

Bluetooth Low Energy (BLE)

BLE is the dominant wireless protocol for personal-area sensor applications. Its low power consumption, ubiquitous smartphone support, and flexible GATT profile system make it ideal for wearables, health monitors, and smart home sensors. BLE 5.x adds extended range, higher throughput, and improved coexistence.

BLE Key Features:
  • Connection Modes: Advertising (broadcast) and Connected (bidirectional)
  • GATT Profiles: Standardized data organization with Services and Characteristics
  • Power Optimization: Connection intervals, slave latency, advertising intervals
  • BLE 5.x: 2x speed (2 Mbps), 4x range (Coded PHY), 8x advertising capacity
  • Security: Pairing, bonding, encryption (AES-CCM)

Zigbee and Thread

Zigbee and Thread are mesh networking protocols designed for IoT applications. They enable self-healing networks where devices can route messages through multiple paths, improving reliability and extending coverage. Thread adds IPv6 support for direct internet connectivity.

LoRaWAN

LoRaWAN provides long-range, low-power communication for wide-area IoT deployments. Using chirp spread spectrum modulation, LoRa achieves ranges of 2-15 km with minimal power consumption. The trade-off is limited data rate (0.3-50 kbps) and duty cycle restrictions.

Application Layer Protocols

MQTT (Message Queuing Telemetry Transport)

MQTT is a lightweight publish-subscribe messaging protocol ideal for constrained IoT devices. Its small packet overhead and QoS levels (0: at most once, 1: at least once, 2: exactly once) make it suitable for unreliable networks. MQTT-SN (Sensor Networks) variant further reduces overhead for very constrained devices.

// MQTT Sensor Data Publishing Example void publish_sensor_data(MQTTClient* client, SensorData* data) { char topic[64]; char payload[256]; // Format topic: sensors/{device_id}/{sensor_type} snprintf(topic, sizeof(topic), "sensors/%s/temperature", device_id); // Format JSON payload snprintf(payload, sizeof(payload), "{" "\"value\": %.2f," "\"unit\": \"celsius\"," "\"timestamp\": %lu," "\"quality\": %d" "}", data->temperature, data->timestamp, data->quality_indicator ); // Publish with QoS 1 (at least once delivery) MQTTClient_publish(client, topic, strlen(payload), payload, 1, 0, NULL); }

CoAP (Constrained Application Protocol)

CoAP brings REST-like semantics to constrained devices using UDP transport. Its request-response model with GET, PUT, POST, DELETE methods mirrors HTTP but with minimal overhead. CoAP is well-suited for resource-constrained sensors that need to expose data as web resources.

Industrial Protocols

IO-Link

IO-Link is the standardized sensor interface for industrial automation. It provides bidirectional communication for configuration, diagnostics, and process data over standard 3-wire sensor cables. IO-Link enables smart sensor capabilities in industrial environments while maintaining compatibility with existing infrastructure.

OPC UA

OPC Unified Architecture provides secure, reliable communication for industrial systems. While traditionally used at higher levels, OPC UA Pub/Sub and field-level variants are bringing standardized industrial communication to smart sensors, enabling seamless integration with enterprise systems.

Data Formats and Encoding
