Chapter 4

Power Optimization & Energy Harvesting

Ultra-low-power design strategies, duty cycling, dynamic voltage scaling, battery life extension, and energy harvesting technologies

Overview

Power optimization represents the single most critical design consideration for battery-powered smart sensors. While cloud-connected devices can rely on wall power, edge sensors deployed in remote locations, wearables, or industrial environments must operate for months or years on compact batteries. The WIA-SEMI-015 standard mandates minimum one-year battery life on standard batteries (CR2032 coin cell or AA), requiring average power consumption well below 100 µA, and often targeting 10-50 µA for multi-year operation.

This chapter explores comprehensive power optimization strategies spanning hardware selection, firmware design, intelligent duty cycling, dynamic voltage/frequency scaling, and energy harvesting technologies that can extend battery life or even enable perpetual operation. Understanding the power budget and optimizing every component is essential for successful deployment.

Key Takeaways

  • Target average power consumption: <50 µA for multi-year battery life on coin cells
  • Duty cycling: spend 99%+ time in deep sleep (<50 µA) for battery-powered applications
  • Dynamic voltage/frequency scaling (DVFS) reduces active power by 30-60%
  • Energy harvesting (solar, vibration, thermal) can eliminate battery replacement
  • Power profiling essential: measure actual consumption, not just datasheet estimates

Power Budget Analysis

Effective power optimization begins with detailed power budget analysis, accounting for every component and operational state. A typical smart sensor alternates between active periods (sensor sampling, ML inference, wireless transmission) and sleep periods (waiting for next event or timer expiry).

Battery Capacity and Target Lifetime

Common battery options include CR2032 coin cell (220 mAh at 3V, compact form factor, 5-10 year shelf life), AA alkaline (2500 mAh at 1.5V, widely available, low cost), and AAA alkaline (1000 mAh at 1.5V, smaller than AA). Lithium primary cells (CR123A, 1500 mAh at 3V) offer higher energy density and better temperature performance.

For one-year battery life on CR2032 (220 mAh), the average current must not exceed 220 mAh / 8760 hours = 25 µA. Accounting for 80% usable capacity (temperature effects, end-of-life voltage drop), the target becomes approximately 20 µA average. For five-year lifetime, the target drops to just 4-5 µA average, demanding extreme optimization.

Power State Analysis

Break down operation into distinct power states and calculate time-weighted average consumption. For example, deep sleep state (<50 µA) spending 99.9% of time, sensor sampling (2-5 mA) for 100ms every 60 seconds (0.17% duty cycle), ML inference (20-50 mA) for 50ms every 60 seconds (0.083% duty cycle), and BLE transmission (8-15 mA) for 10ms every 60 seconds (0.017% duty cycle). 

// Example power budget calculation
// Assumptions: 60-second measurement interval
const float SLEEP_CURRENT_UA = 30.0;      // Deep sleep
const float SLEEP_TIME_S = 59.84;         // 99.73% duty cycle

const float SAMPLE_CURRENT_MA = 3.0;      // Sensor active
const float SAMPLE_TIME_S = 0.1;          // 100ms

const float INFERENCE_CURRENT_MA = 40.0;  // MCU + ML
const float INFERENCE_TIME_S = 0.05;      // 50ms

const float TX_CURRENT_MA = 10.0;         // BLE transmit
const float TX_TIME_S = 0.01;             // 10ms

// Calculate average current in µA
float avg_current_ua =
    (SLEEP_CURRENT_UA * SLEEP_TIME_S +
     SAMPLE_CURRENT_MA * 1000 * SAMPLE_TIME_S +
     INFERENCE_CURRENT_MA * 1000 * INFERENCE_TIME_S +
     TX_CURRENT_MA * 1000 * TX_TIME_S) / 60.0;

// Result: ~36.5 µA average
// Battery life on CR2032: 220mAh * 0.8 / 0.0365mA = ~4800 hours = 6.6 months

Ultra-Low-Power MCU Modes

Modern microcontrollers provide multiple power modes with varying levels of functionality, current consumption, and wake-up latency. Effective firmware design transitions between modes based on system requirements, maximizing time in the lowest-power states.

Active Mode

Active mode runs CPU at full speed with all peripherals available, consuming 40-200 µA/MHz depending on architecture and voltage. A Cortex-M4 at 80 MHz and 3.3V typically draws 6-10 mA. Minimize active time by completing tasks quickly and returning to sleep, using interrupt-driven architecture rather than polling, and leveraging DMA to transfer data without CPU intervention.

Sleep Mode

Sleep mode halts CPU execution while maintaining peripheral operation, with typical consumption of 1-5 mA. The CPU stops but SRAM retained, peripherals (timers, UART, I2C) continue operating, and wake-up is instantaneous (µs). This mode suits brief waits for peripheral operations like waiting for ADC conversion, SPI transfer completion, or UART reception.

Deep Sleep Mode

Deep sleep disables most peripherals while retaining SRAM and critical timers, consuming 10-50 µA. SRAM contents preserved, RTC (Real-Time Clock) continues, wake sources limited (RTC alarm, external interrupt), and wake-up time ranges from tens to hundreds of microseconds. This represents the primary power state for battery-operated sensors, where the system spends 99%+ of time.

Shutdown/Standby Mode

The lowest power mode (<5 µA, often 1-2 µA) loses SRAM contents requiring full system reset, maintains only critical registers (RTC backup), has limited wake sources (RTC alarm, reset pin), and incurs millisecond wake-up latency. Use for extreme power savings during long dormant periods or when sensor is disabled until user interaction.

Dynamic Voltage and Frequency Scaling (DVFS)

DVFS adjusts MCU operating voltage and clock frequency based on workload, reducing active power consumption by 30-60% for non-critical tasks. Power consumption relates to frequency and voltage: P ∝ f × V². Reducing voltage from 3.3V to 1.8V cuts power by roughly (1.8/3.3)² = 30%, and halving frequency provides another 50% reduction, combining for ~65% total savings.

Implementation Strategies

Run sensor sampling at low frequency (8-16 MHz, 1.8V) consuming 1-2 mA versus 6-8 mA at full speed. Execute ML inference at maximum frequency (80-120 MHz, 3.3V) to minimize active time and return to sleep faster. Handle wireless communication at protocol-required speed (typically 48-64 MHz for BLE). Configure peripherals (ADC, timers) to operate during sleep when possible. 

// Example: Dynamic frequency scaling for sensor reading
void read_sensor_low_power(void) {
    // Reduce clock to 8 MHz, voltage to 1.8V
    system_clock_set_frequency(CLOCK_8MHZ);
    power_set_voltage(VOLTAGE_1V8);

    // Sample sensor (takes longer, but uses less energy)
    i2c_read_sensor(&sensor_data);

    // Return to high performance for ML inference
    power_set_voltage(VOLTAGE_3V3);
    system_clock_set_frequency(CLOCK_80MHZ);

    // Run inference at full speed
    run_ml_inference(&sensor_data, &result);

    // Enter deep sleep
    enter_deep_sleep_until_next_interval();
}

Peripheral Power Management

Sensors and peripherals often dominate the power budget, especially in active state. Careful management is essential for meeting battery life targets.

Sensor Power Optimization

Many MEMS sensors provide multiple power modes. For example, Bosch BMA400 accelerometer offers normal mode (14 µA at 100 Hz), low-power mode (3.5 µA at 100 Hz), and sleep mode (0.8 µA). Strategies include power down sensors between measurements, use lowest sampling rate meeting application needs (1 Hz instead of 100 Hz saves 90% power), leverage built-in motion detection to wake system only when needed, and batch multiple measurements before waking MCU.

Wireless Radio Management

Wireless transceivers consume significant power during transmission and reception: BLE transmission (8-15 mA), BLE reception (8-12 mA), LoRaWAN transmission (30-120 mA peak, brief bursts), and Wi-Fi transmission/reception (80-200 mA). Optimization techniques include minimize connection intervals (advertise less frequently), reduce packet size to minimize airtime, use connection parameters optimized for power (longer intervals, more latency tolerance), implement sleep between transmissions, and consider LoRaWAN Class A (lowest power) versus Class C (always listening).

Software Optimization for Power

Firmware design profoundly impacts power consumption. Poorly optimized code can waste 50%+ energy through unnecessary wake-ups, inefficient algorithms, or failure to utilize hardware features.

Interrupt-Driven Architecture

Avoid polling loops that keep CPU active, instead use interrupts to wake from sleep on events (sensor data ready, timer expiry, wireless packet received). Configure peripherals to use interrupt-driven operation, leverage DMA for data transfers, and minimize interrupt service routine duration to return to sleep quickly.

Event-Driven State Machines

Structure firmware as state machines responding to events rather than continuous execution loops. Each state transition performs minimal work then returns to sleep, ensuring the system is never active without purpose. RTOS task notifications or event flags coordinate complex behaviors efficiently.

Computational Efficiency

Reduce active time through algorithmic optimization: use fixed-point instead of floating-point when possible (fewer CPU cycles), leverage hardware accelerators (CRC, AES, MAC), pre-compute constants and lookup tables, and optimize critical loops (unroll, vectorize with SIMD). A 2× speedup in computation halves active energy consumption.

Energy Harvesting Technologies

Energy harvesting captures ambient energy to supplement or replace batteries, enabling perpetual operation or dramatically extending battery life. The WIA-SEMI-015 standard includes optional support for energy harvesting interfaces.

Solar/Photovoltaic

Solar cells convert light to electricity with varying efficiency based on illumination. Indoor lighting provides 10-100 µW/cm² (fluorescent, LED office lighting), while outdoor sunlight delivers 1-10 mW/cm² (diffuse) to 10-100 mW/cm² (direct sun). A 5 cm² indoor cell generates 50-500 µW, sufficient for low-duty-cycle sensors (<1% duty cycle, deep sleep between samples). Outdoor deployment enables much higher power budgets.

Implementation uses maximum power point tracking (MPPT) to optimize energy extraction, supercapacitors or thin-film batteries for energy storage, and power management ICs like Texas Instruments BQ25570 (nanopower harvesting, cold-start from 330 mV) or Analog Devices ADP5090 (solar/TEG optimized).

Vibration/Piezoelectric

Piezoelectric generators convert mechanical vibration to electrical energy, suitable for industrial machinery monitoring, automotive applications, and structural health monitoring. Output power depends on vibration amplitude and frequency: low vibration (10 µW - 100 µW), moderate vibration (100 µW - 500 µW), and high vibration (0.5 mW - 5 mW).

Piezo harvesters require AC-to-DC rectification, impedance matching to maximize power transfer, and energy storage (supercapacitor) to buffer intermittent generation. Typical implementation charges supercapacitor during vibration, then powers sensor burst when sufficient energy accumulated.

Thermoelectric Generators (TEG)

TEGs convert temperature differentials to electricity using the Seebeck effect. Power output relates to temperature difference: 5°C delta generates 100-500 µW, 10°C delta produces 500 µW - 2 mW, and 20°C delta yields 2-10 mW. Applications include body heat harvesting (wearables using skin temperature vs. ambient), industrial process monitoring (hot pipes, machinery), and HVAC duct sensors.

TEG modules like Micropelt MPG-D751 (13mm × 13mm, 20 µW to 5 mW depending on ΔT) pair with PMICs offering ultra-low input voltage capability (Analog Devices LTC3108 starts from 20 mV input).

RF Energy Harvesting

RF harvesting captures ambient radio frequency energy or uses dedicated RF beacons. Ambient RF (Wi-Fi, cellular, broadcast) provides 1-10 µW at typical distances, while dedicated RF beacons (900 MHz, 2.4 GHz) deliver 100 µW - 1 mW at 1-10 meter range. Applications include RFID-like scenarios, warehouse asset tracking, and proximity-based sensors.

Power Measurement and Profiling

Accurate power measurement is essential for validating optimization efforts and ensuring WIA-SEMI-015 compliance. Relying solely on datasheet estimates often leads to 2-5× underestimation of actual consumption due to overlooked peripherals, suboptimal firmware, or board leakage currents.

Measurement Tools

Options include benchtop power supplies with built-in current measurement (µA to A range, logging capability), dedicated power profilers (Nordic PPK2, Otii Arc, Joulescope JS220), oscilloscope with current probe (captures transients, high bandwidth), and energy harvesting simulators (Keysight E3640A with energy harvesting simulation).

Measurement Techniques

Insert current sense resistor (1-10Ω) in supply path and measure voltage drop, use power profiler with automated data logging and analysis, capture full power cycles including sleep, wake, active, and transmission phases, and measure over extended periods (hours to days) to capture all operational modes. Calculate average, peak, and energy per cycle metrics.

Embedded Machine Learning

The integration of machine learning capabilities directly into smart sensors represents a paradigm shift from cloud-centric to edge-centric intelligence. TinyML enables real-time inference with minimal latency, reduced bandwidth, and enhanced privacy.

The TinyML Revolution

TinyML refers to machine learning algorithms optimized to run on microcontrollers with extremely limited resources—often less than 256KB of Flash memory and 64KB of RAM. This field has emerged from the convergence of advances in model compression, efficient neural network architectures, and specialized hardware accelerators. The result is the ability to deploy sophisticated AI capabilities directly on smart sensors without cloud connectivity.

The benefits of edge ML are compelling: sub-millisecond inference latency, operation without network connectivity, reduced data transmission costs, and enhanced privacy since raw sensor data never leaves the device. For applications like keyword spotting, anomaly detection, and gesture recognition, TinyML enables capabilities that were previously impossible on resource-constrained devices.

ML Model Architectures for Sensors

Not all neural network architectures are suitable for embedded deployment. Smart sensor applications typically use architectures specifically designed for efficiency, including MobileNets, EfficientNets, and custom architectures optimized for specific sensor modalities.

Convolutional Neural Networks (CNNs)

CNNs excel at processing spatial and temporal patterns in sensor data. Depthwise separable convolutions, as used in MobileNet, dramatically reduce computational requirements while maintaining accuracy. For time-series sensor data, 1D convolutions are particularly efficient.

Recurrent Neural Networks (RNNs)

For sequential sensor data like audio, vibration, or time-series measurements, RNN variants such as LSTM and GRU provide temporal modeling capabilities. However, their sequential nature can limit throughput. Alternatives like Temporal Convolutional Networks (TCNs) offer parallelizable processing with similar modeling capabilities.

Transformers for Sensors

Attention-based models have shown remarkable performance across domains. While standard Transformers are too resource-intensive for MCUs, efficient variants like Linear Attention and Performer architectures are being adapted for edge deployment. These models excel at capturing long-range dependencies in sensor data.

Model Optimization Techniques

Deploying ML models on microcontrollers requires aggressive optimization to meet memory and compute constraints. Multiple complementary techniques are employed to reduce model size and inference cost while preserving accuracy.

Quantization

Quantization reduces the precision of model weights and activations from 32-bit floating point to lower bit-widths. 8-bit integer quantization typically provides 4x memory reduction with minimal accuracy loss. More aggressive quantization to 4-bit or even binary can achieve further compression but requires careful training techniques.

# TensorFlow Lite Quantization Example import tensorflow as tf # Post-training quantization converter = tf.lite.TFLiteConverter.from_saved_model('saved_model') converter.optimizations = [tf.lite.Optimize.DEFAULT] # Full integer quantization for MCU deployment def representative_dataset(): for data in calibration_data: yield [data.astype(np.float32)] converter.representative_dataset = representative_dataset converter.target_spec.supported_ops = [tf.lite.OpsSet.TFLITE_BUILTINS_INT8] converter.inference_input_type = tf.int8 converter.inference_output_type = tf.int8 tflite_model = converter.convert() # Model size comparison print(f"Original: {original_size / 1024:.1f} KB") print(f"Quantized: {len(tflite_model) / 1024:.1f} KB") print(f"Compression: {original_size / len(tflite_model):.1f}x")

Pruning

Pruning removes redundant weights or entire neurons/filters that contribute minimally to model accuracy. Structured pruning removes entire channels or layers, enabling direct speedup on standard hardware. Unstructured pruning achieves higher compression but requires sparse computation support for acceleration.

Pruning Strategies:
  • Magnitude Pruning: Remove weights below threshold
  • Structured Pruning: Remove entire filters/channels
  • Movement Pruning: Prune based on weight changes during training
  • Lottery Ticket: Find sparse subnetworks that train well
  • Neural Architecture Search: Automatically find efficient architectures

Knowledge Distillation

Knowledge distillation trains a small "student" model to mimic a larger "teacher" model. The student learns not just the hard labels but the teacher's soft probability distributions, which encode additional information about class relationships. This often enables smaller models to achieve accuracy comparable to much larger models.

Application Examples

Keyword Spotting

Wake word detection is a canonical TinyML application. Models process audio spectrograms to detect specific keywords with high accuracy while consuming minimal power in always-on operation. State-of-the-art models achieve >95% accuracy with <500KB model size.

Anomaly Detection

Smart sensors can learn normal operating patterns and detect anomalies indicating potential failures or security threats. Autoencoder-based approaches compare reconstruction error against learned thresholds, while one-class classification models directly identify outliers.

Predictive Maintenance

Vibration analysis using ML can predict equipment failures before they occur. Models trained on historical failure data can identify subtle patterns in frequency spectra that precede failures, enabling proactive maintenance and reducing downtime.
