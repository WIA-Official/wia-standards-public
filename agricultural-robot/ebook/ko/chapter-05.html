제5장: 2단계: API 인터페이스

弘益人間 (홍익인간) - 널리 인간을 이롭게 하라

5.1 RESTful API 개요

WIA-ROB-007의 2단계는 농업 로봇을 제어하고 데이터를 조회하기 위한 표준 API를 정의합니다. REST (Representational State Transfer) 아키텍처를 기반으로 하며, HTTP/HTTPS 프로토콜을 사용합니다.

RESTful API는 직관적이고 사용하기 쉬우며, 웹 기술 표준을 따르므로 대부분의 개발 환경에서 쉽게 구현할 수 있습니다. 이는 弘益人間 철학에 따라 기술 접근성을 높이는 중요한 선택입니다.

5.2 API 기본 원칙

WIA-ROB-007 API 설계 원칙

  • 일관성: 모든 엔드포인트가 동일한 패턴과 규칙을 따름
  • 명확성: URL과 메서드만으로 기능을 파악 가능
  • 버전 관리: API 버전을 명시하여 하위 호환성 보장
  • 상태 비저장: 각 요청은 독립적이며 이전 상태에 의존하지 않음
  • 보안성: HTTPS 필수, 인증/인가 메커니즘 적용

5.3 API 엔드포인트 구조

WIA-ROB-007 API는 계층적 URL 구조를 사용합니다. 

https://api.farm.example.com/wia-rob-007/v1/{resource}/{id}/{action}

예시:
https://api.farm.example.com/wia-rob-007/v1/robots/ROB-001/status
https://api.farm.example.com/wia-rob-007/v1/tasks/TASK-001/start

5.3.1 URL 구성 요소

구성 요소 설명 예시
Base URL API 서버 주소 https://api.farm.example.com
Prefix 표준 식별자 /wia-rob-007
Version API 버전 /v1
Resource 리소스 타입 /robots, /tasks, /sensors
ID 리소스 식별자 /ROB-001
Action 작업 (선택) /start, /stop, /status

5.4 HTTP 메서드

WIA-ROB-007은 표준 HTTP 메서드를 사용하여 CRUD 작업을 수행합니다.

메서드 용도 예시 멱등성
GET 데이터 조회 GET /robots/ROB-001 Yes
POST 새 리소스 생성 POST /tasks No
PUT 리소스 전체 업데이트 PUT /robots/ROB-001/config Yes
PATCH 리소스 부분 업데이트 PATCH /robots/ROB-001 No
DELETE 리소스 삭제 DELETE /tasks/TASK-001 Yes

5.8 인증 및 보안

모든 API 요청은 인증이 필요하며, WIA-ROB-007은 OAuth 2.0 기반 인증을 사용합니다.

5.8.1 액세스 토큰 획득

POST /wia-rob-007/v1/auth/token

Request Body:
{
    "grant_type": "client_credentials",
    "client_id": "your-client-id",
    "client_secret": "your-client-secret",
    "scope": "robot:read robot:write task:manage"
}

Response 200 OK:
{
    "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "token_type": "Bearer",
    "expires_in": 3600,
    "scope": "robot:read robot:write task:manage"
}

5.8.2 API 요청 시 인증

GET /wia-rob-007/v1/robots/ROB-001
Headers:
  Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
  Content-Type: application/json

5.8.3 권한 범위 (Scopes)

Scope 설명 권한
robot:read 로봇 정보 읽기 GET /robots/*
robot:write 로봇 설정 변경 PATCH /robots/*/config
task:read 작업 정보 읽기 GET /tasks/*
task:manage 작업 생성/제어 POST /tasks, POST /tasks/*/start
sensor:read 센서 데이터 읽기 GET /sensors/*
admin 모든 권한 All endpoints

5.9 에러 처리

WIA-ROB-007은 표준 HTTP 상태 코드와 일관된 에러 응답 형식을 사용합니다.

5.9.1 HTTP 상태 코드

상태 코드 의미 사용 예
200 OK 성공 GET, PATCH 성공
201 Created 리소스 생성 성공 POST 성공
400 Bad Request 잘못된 요청 유효성 검증 실패
401 Unauthorized 인증 실패 토큰 없음/만료
403 Forbidden 권한 없음 Scope 부족
404 Not Found 리소스 없음 잘못된 robot_id
429 Too Many Requests 요청 제한 초과 Rate limit 도달
500 Internal Server Error 서버 오류 예상치 못한 오류

5.9.2 에러 응답 형식

Response 400 Bad Request:
{
    "status": "error",
    "error": {
        "code": "VALIDATION_ERROR",
        "message": "Invalid request parameters",
        "details": [
            {
                "field": "ripeness_threshold",
                "issue": "Must be between 0 and 1",
                "received": 1.5
            }
        ],
        "timestamp": "2025-01-15T14:30:00.000Z",
        "request_id": "req-20250115-143000-001"
    }
}

5.10 Rate Limiting 및 페이지네이션

5.10.1 Rate Limiting

서버 부하를 방지하기 위해 요청 제한을 적용합니다. 

Response Headers:
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 987
X-RateLimit-Reset: 1642252800

Response 429 Too Many Requests:
{
    "status": "error",
    "error": {
        "code": "RATE_LIMIT_EXCEEDED",
        "message": "Too many requests. Please retry after 300 seconds.",
        "retry_after": 300
    }
}

5.10.2 페이지네이션

GET /wia-rob-007/v1/sensors/data?limit=100&offset=200

Response Headers:
X-Total-Count: 10000
Link: </v1/sensors/data?limit=100&offset=300>; rel="next",
      </v1/sensors/data?limit=100&offset=100>; rel="prev"

핵심 내용 요약

  • WIA-ROB-007은 RESTful API를 표준 인터페이스로 사용
  • 계층적 URL 구조: /wia-rob-007/v1/{resource}/{id}/{action}
  • 표준 HTTP 메서드로 CRUD 작업 수행 (GET, POST, PUT, PATCH, DELETE)
  • 로봇 관리, 작업 관리, 센서 데이터 API 제공
  • OAuth 2.0 기반 인증 및 Scope 기반 권한 관리
  • 일관된 에러 응답 형식과 표준 HTTP 상태 코드 사용
  • Rate Limiting으로 서버 부하 방지
  • 페이지네이션으로 대량 데이터 효율적 조회
  • HTTPS 필수로 보안 통신 보장
