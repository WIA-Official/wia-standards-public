Chapter 5: Phase 2 - API Interface

5.1 RESTful API Architecture

The WIA-ROB-007 standard defines a comprehensive RESTful API architecture that enables seamless communication between agricultural robots, farm management systems, and third-party applications. This API provides a unified interface for controlling robots, managing fields, scheduling tasks, and coordinating fleets regardless of the underlying hardware or manufacturer.

The API follows REST (Representational State Transfer) principles, utilizing standard HTTP methods (GET, POST, PUT, DELETE) and JSON data formats. This approach ensures maximum compatibility with existing web technologies, programming languages, and integration platforms while maintaining simplicity and ease of implementation.

All API endpoints follow a consistent URL structure: https://api.farm.example/v1/{resource}/{id}/{action}. Version numbering in the URL path ensures backward compatibility as the standard evolves. The API supports both synchronous operations that return immediate results and asynchronous operations for long-running tasks like field mapping or harvest missions.

API Design Principles

  • Resource-Oriented: Every entity (robots, fields, tasks) is a resource with a unique identifier
  • Stateless: Each request contains all information needed for processing; no server-side session state
  • Cacheable: Responses include caching headers to reduce server load and improve performance
  • Uniform Interface: Consistent patterns across all endpoints for predictable behavior
  • Layered System: API can operate through proxies, gateways, and load balancers transparently

5.2 Robot Control Endpoints

The robot control API provides comprehensive functionality for managing individual agricultural robots. These endpoints enable remote operation, monitoring, and configuration of robots from any authorized client application.

5.3 Field Management API

Field management endpoints enable digital representation and manipulation of agricultural fields. These APIs support field boundaries, crop zones, obstacle mapping, and yield data management. Integration with GIS systems and farm management platforms ensures that field data remains synchronized across all systems.

Field data includes geometric boundaries (typically GeoJSON polygons), soil characteristics, crop information, historical yield data, and infrastructure elements like irrigation systems and fences. The API supports both manual field creation and automatic boundary detection using GPS traces from equipment operations.

5.4 Task Scheduling and Management

The task scheduling API enables creation, assignment, and monitoring of agricultural operations. Tasks represent discrete work units like "plant field A", "scout for pests in orchard B", or "harvest section 3". The scheduler coordinates task execution across available robots, considering constraints like weather, soil conditions, and equipment availability.

Tasks follow a lifecycle from creation through planning, assignment, execution, and completion. The API provides real-time status updates, progress tracking, and automatic rescheduling when tasks cannot be completed as planned.

Task State Description Next States Actions Available
Draft Task created but not scheduled Scheduled, Cancelled Edit, Delete, Schedule
Scheduled Assigned to robot(s) with start time In Progress, Cancelled, On Hold Reschedule, Cancel
In Progress Robot actively executing task Completed, Failed, Paused Monitor, Pause, Abort
Paused Temporarily stopped In Progress, Cancelled Resume, Cancel
Completed Successfully finished None (terminal) View Results
Failed Could not complete Scheduled (retry) Retry, Analyze Failure

5.5 Fleet Coordination API

Fleet coordination enables multiple robots to work together efficiently, avoiding conflicts, optimizing field coverage, and balancing workloads. The fleet API manages robot assignments, monitors collective progress, and handles dynamic reallocation when robots become unavailable or new tasks arrive.

The coordination system implements several algorithms including nearest-task assignment, coverage path planning for multiple robots, and collision avoidance for robots working in proximity. Fleet operators can set coordination policies that balance competing objectives like minimizing total completion time versus maximizing individual robot utilization.

Fleet Optimization

The fleet optimizer allocates tasks to robots considering multiple factors:

  • Task-Robot Compatibility: Not all robots can perform all tasks (e.g., harvesting robots cannot spray)
  • Geographic Proximity: Minimize travel time by assigning tasks to nearby robots
  • Resource Availability: Consider battery levels, fluid tanks, and maintenance schedules
  • Task Dependencies: Some tasks must complete before others can start (e.g., planting before spraying)
  • Weather Windows: Prioritize tasks with narrow weather requirement windows

Fleet Coordination Algorithm

The WIA-ROB-007 standard recommends a hierarchical task allocation approach:

  1. Global Planning: Cluster tasks by field and operation type
  2. Constraint Filtering: Eliminate robot-task pairs that violate constraints
  3. Cost Matrix: Calculate cost (time, energy, opportunity) for each valid assignment
  4. Hungarian Algorithm: Solve optimal assignment problem
  5. Path Deconfliction: Adjust paths to prevent robot collisions
  6. Continuous Replanning: Recalculate assignments as conditions change

5.6 Authentication and Authorization

Security is paramount for agricultural robot APIs. Unauthorized access could result in crop damage, chemical misapplication, or safety hazards. The WIA-ROB-007 standard specifies OAuth 2.0 and JWT (JSON Web Tokens) for authentication and role-based access control (RBAC) for authorization.

Authentication Flow

POST /api/v1/auth/login
{
  "username": "farmer@example.com",
  "password": "secure_password",
  "farm_id": "farm-12345"
}

Response:
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refresh_token": "dGhpcyBpcyBhIHJlZnJlc2ggdG9rZW4...",
  "expires_in": 3600,
  "token_type": "Bearer",

}

5.8 Data Formats and Validation

The API enforces strict data validation using JSON Schema. All request payloads are validated against published schemas, ensuring data consistency and preventing errors from malformed requests.

Common Data Types

  • Coordinates: WGS84 decimal degrees with precision sufficient for centimeter-level accuracy
  • Timestamps: ISO 8601 format with UTC timezone (YYYY-MM-DDTHH:MM:SSZ)
  • Measurements: Metric units with explicit unit specification (e.g., "speed_ms": 1.5)
  • Identifiers: UUID v4 or namespaced strings (e.g., "robot-001")
  • Status Enums: Predefined string values for consistent state representation

Validation Example

// JSON Schema for Task Creation
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "required": ["type", "field_id", "schedule"],
  "properties": {
    "type": {
      "type": "string",
      "enum": ["plant", "spray", "harvest", "scout", "weed"]
    },
    "field_id": {
      "type": "string",
      "pattern": "^field-[a-z0-9-]+$"
    },
    "schedule": {
      "type": "object",
      "required": ["start_time"],
      "properties": {
        "start_time": {
          "type": "string",
          "format": "date-time"
        }
      }
    }
  }
}

5.9 API Rate Limiting and Quotas

To ensure fair resource allocation and system stability, the API implements rate limiting. Rate limits vary by endpoint type and user subscription level. Headers in API responses indicate current usage and remaining quota.

Endpoint Category Rate Limit Quota Period Burst Allowed
Robot Control 100 requests Per minute 20 over limit
Telemetry Read 1000 requests Per minute 100 over limit
Task Management 500 requests Per hour 50 over limit
Field Management 200 requests Per hour 20 over limit
Authentication 10 requests Per minute No burst

Rate Limit Headers

X-RateLimit-Limit: 100
X-RateLimit-Remaining: 87
X-RateLimit-Reset: 1640808000
Retry-After: 45

// When rate limit exceeded:

X-RateLimit-Limit: 100

5.10 API Versioning and Deprecation

The API follows semantic versioning (MAJOR.MINOR.PATCH) with version specified in the URL path. The WIA-ROB-007 standard guarantees backward compatibility within major versions, providing a clear deprecation policy for changes that break compatibility.

Version Lifecycle

  • Active: Current version, receives new features and bug fixes
  • Maintained: Previous major version, receives security and critical bug fixes only
  • Deprecated: Announced end-of-life, minimum 12 months before shutdown
  • Retired: No longer available, returns 410 Gone status

API Deprecation Notice Example

HTTP/1.1 200 OK
Deprecation: true
Sunset: Wed, 31 Dec 2025 23:59:59 GMT
Link: <https://api.farm.example/v2/robots>; rel="successor-version"

{
  "warning": "API v1 is deprecated. Migrate to v2 by Dec 31, 2025.",
  "migration_guide": "https://docs.wia.org/migration-v1-to-v2",
  "data": {...}
}
