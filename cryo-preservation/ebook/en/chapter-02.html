2.1 SUBJECT IDENTIFICATION SCHEMA
--------------------------------------------------------------------------------

Every cryopreservation subject requires a unique, globally verifiable
identifier that persists throughout the preservation period.

[CRYO] SUBJECT DOCUMENT SCHEMA:

    {
      "@context": [
        "https://www.w3.org/2018/credentials/v1",
        "https://wia.org/cryo-preservation/v1"
      ],
      "type": ["VerifiableCredential", "CryoSubjectCredential"],
      "id": "urn:uuid:cryo:subject:a1b2c3d4-e5f6-7890-abcd-ef1234567890",

      "credentialSubject": {
        "subjectId": "CRYO-2025-001-WB-001",
        "subjectType": "WHOLE_BODY",
        "species": "HOMO_SAPIENS",

        "identifier": {
          "type": "FACILITY_ASSIGNED",
          "value": "ACME-CRYO-2025-00001",
          "issuer": "urn:facility:acme-cryonics"
        },

        "demographics": {
          "birthYear": 1960,
          "biologicalSex": "MALE",
          "bloodType": "A_POSITIVE"
        },

        "biometrics": {
          "heightCm": 175,
          "weightKg": 70,
          "bodyMassIndex": 22.9
        }
      },

      "preservationStatus": {
        "currentState": "VITRIFIED",
        "containerAssignment": "DEWAR-A-001-SLOT-05",
        "temperatureCelsius": -196,
        "preservationDate": "2025-03-15T14:30:00Z"
      }
    }

[i] SUBJECT ID FORMAT:
    CRYO-[YEAR]-[FACILITY]-[TYPE]-[SEQUENCE]

    Examples:
    - CRYO-2025-001-WB-001  (Whole Body, Facility 001)
    - CRYO-2025-042-NR-015  (Neuro, Facility 042)
    - CRYO-2025-007-EM-203  (Embryo, Facility 007)

2.2 CONSENT AND LEGAL DOCUMENTATION
--------------------------------------------------------------------------------

Cryopreservation requires robust consent documentation, especially for
human subjects where storage may span multiple generations.

[CRYO] CONSENT CREDENTIAL STRUCTURE:

    ConsentDocument
    +------------------------------------------------------------------+
    | consentId         | Unique consent document identifier           |
    | consentType       | PATIENT | NEXT_OF_KIN | LEGAL_GUARDIAN       |
    | consentDate       | ISO 8601 timestamp                           |
    | consentScope      | Array of authorized procedures               |
    +------------------------------------------------------------------+
    | signatories       | Array of SignatoryRecord                     |
    |   - name          | Full legal name                              |
    |   - relationship  | SELF | SPOUSE | CHILD | LEGAL_REP            |
    |   - signature     | Cryptographic signature                      |
    |   - witnessed     | Witness attestation                          |
    +------------------------------------------------------------------+
    | revivalConsent    | RevivalConsentRecord                         |
    |   - authorized    | Boolean                                      |
    |   - conditions    | Array of revival conditions                  |
    |   - decisionMaker | Who decides on revival                       |
    +------------------------------------------------------------------+

[!] LEGAL REQUIREMENTS:
    - Consent must be obtained BEFORE preservation begins
    - Multiple witnesses required for human subjects
    - Legal representative designation for revival decisions
    - Periodic consent renewal for long-term storage (if subject capable)

2.5 CHAIN OF CUSTODY DOCUMENTATION
--------------------------------------------------------------------------------

Every interaction with a preserved subject must be documented in an
immutable chain of custody record.

[CRYO] CUSTODY EVENT TYPES:

    INITIAL_INTAKE
      |-- Subject received at facility
      |-- Initial documentation verified
      +-- Preservation protocol assigned

    PROCEDURE_START
      |-- Specific procedure initiated
      |-- Operator credentials verified
      +-- Equipment calibration confirmed

    CONTAINER_ASSIGNMENT
      |-- Subject placed in storage container
      |-- Position coordinates recorded
      +-- Monitoring sensors activated

    INSPECTION
      |-- Routine or emergency inspection
      |-- Visual/instrumental assessment
      +-- Findings documented

    TRANSFER_OUT
      |-- Subject leaving current custody
      |-- Destination facility verified
      +-- Transport conditions specified

    TRANSFER_IN
      |-- Subject arriving from transfer
      |-- Condition assessment
      +-- New container assignment

    REVIVAL_INITIATION
      |-- Revival process authorized
      |-- Medical team assembled
      +-- Protocol selected

[CRYO] CUSTODY CHAIN EXAMPLE:

    Subject: CRYO-2025-001-WB-001

    Event 1: INITIAL_INTAKE

    | Witness: Nurse Maria Garcia (ID: OP-2025-0089)                   |

2.6 CREDENTIAL VERIFICATION
--------------------------------------------------------------------------------

All credentials must be verifiable through standard DID (Decentralized
Identifier) resolution and cryptographic signature verification.

[i] VERIFICATION FLOW:

    +----------------+     +------------------+     +----------------+
    |  CREDENTIAL    | --> |  DID RESOLVER    | --> |  PUBLIC KEY    |
    |  with signature|     |  did:wia:cryo:.. |     |  retrieval     |
    +----------------+     +------------------+     +----------------+
                                                            |
                                                            v
    +----------------+     +------------------+     +----------------+
    |  VERIFIED      | <-- |  SIGNATURE       | <-- |  VERIFY SIG    |
    |  CREDENTIAL    |     |  VALID/INVALID   |     |  against key   |
    +----------------+     +------------------+     +----------------+

[CRYO] VERIFICATION CODE EXAMPLE:

    async function verifySubjectCredential(credential) {
      // 1. Extract issuer DID
      const issuerDid = credential.issuer;

      // 2. Resolve DID document
      const didDocument = await didResolver.resolve(issuerDid);

      // 3. Get verification method
      const verificationMethod = didDocument.verificationMethod[0];

      // 4. Verify signature
      const isValid = await verifySignature(
        credential,
        credential.proof.jws,
        verificationMethod.publicKeyJwk
      );

      // 5. Check revocation status
      const isRevoked = await checkRevocationRegistry(credential.id);

      return {
        signatureValid: isValid,
        notRevoked: !isRevoked,
        issuerTrusted: await checkTrustRegistry(issuerDid)
      };
    }
