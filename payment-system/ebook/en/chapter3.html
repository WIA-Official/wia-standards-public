CHAPTER 3

API Endpoints

Payment Processing & Authorization APIs

3.1 Payment API Overview

Modern payment systems expose RESTful APIs that enable merchants, payment gateways, and financial applications to process transactions programmatically. This chapter covers the standard API endpoints for payment processing.

API Architecture Principles

  • RESTful Design: HTTP methods (GET, POST, PUT, DELETE) for resource operations
  • Stateless: Each request contains all necessary information
  • Idempotency: Repeated requests produce same result
  • Versioning: API version in URL or header (v1, v2, etc.)
  • JSON Payload: Standard data format for requests/responses
  • HTTPS Only: TLS 1.2+ encryption mandatory
  • Authentication: API keys, OAuth 2.0, or JWT tokens
  • Rate Limiting: Prevent abuse and ensure fairness

3.2 Authorization Endpoint

POST /v1/payments/authorize

Authorizes a payment without immediately capturing funds. Used in two-step payment flows.

Request

POST https://api.payment-gateway.com/v1/payments/authorize
Content-Type: application/json
Authorization: Bearer EXAMPLE_API_KEY_REPLACE_ME

{
  "amount": 10000,
  "currency": "USD",
  "payment_method": {
    "type": "card",
    "card": {
      "number": "4532123456789010",
      "exp_month": 12,
      "exp_year": 2025,
      "cvc": "123",
      "name": "John Doe"
    }
  },
  "billing_details": {
    "address": {
      "line1": "123 Main St",
      "city": "San Francisco",
      "state": "CA",
      "postal_code": "94102",
      "country": "US"
    }
  },
  "merchant_id": "merch_7h3k2m9p",
  "order_id": "ORDER-2025-001",
  "description": "Premium subscription",
  "metadata": {
    "customer_id": "cust_987xyz",
    "invoice_id": "INV-2025-001"
  },
  "three_d_secure": {
    "enabled": true,

3.3 Capture Endpoint

POST /v1/payments/{authorization_id}/capture

Captures funds from a previously authorized payment. Must be done within authorization expiry window (typically 7 days).

Request

POST https://api.payment-gateway.com/v1/payments/auth_9x7k3m2n/capture
Content-Type: application/json
Authorization: Bearer EXAMPLE_API_KEY_REPLACE_ME

{
  "amount": 10000,
  "final": true,
  "statement_descriptor": "ACME CORP PURCHASE"
}

Response

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": "cap_5k8x3m7n",
  "object": "capture",
  "authorization_id": "auth_9x7k3m2n",
  "amount": 10000,
  "amount_captured": 10000,
  "currency": "USD",
  "status": "succeeded",
  "created": "2025-01-15T14:20:30Z",
  "settlement_date": "2025-01-17",
  "network_transaction_id": "txn_visa_5k8x3m7n"
}

Request (Full Refund)

POST https://api.payment-gateway.com/v1/payments/pay_9x7k3m2n/refund
Content-Type: application/json
Authorization: Bearer EXAMPLE_API_KEY_REPLACE_ME

{
  "reason": "requested_by_customer",
  "metadata": {
    "refund_reason": "Product defective"

}

  "object": "refund",
  "payment_id": "pay_9x7k3m2n",
  "amount": 5000,
  "currency": "USD",
  "status": "succeeded",

  "estimated_arrival": "2025-01-23",

}

Content-Type: application/json
Authorization: Bearer EXAMPLE_API_KEY_REPLACE_ME

  "reason": "duplicate_transaction"
}

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": "void_2k9x7m3n",
  "object": "void",
  "authorization_id": "auth_9x7k3m2n",
  "status": "succeeded",
  "created": "2025-01-15T11:45:10Z",

3.10 Webhooks

Webhooks enable real-time notifications when payment events occur.

Webhook Event Types

  • payment.succeeded - Payment successfully processed
  • payment.failed - Payment declined or failed
  • payment.refunded - Refund processed
  • payment.disputed - Chargeback initiated
  • subscription.created - New subscription created
  • subscription.updated - Subscription modified
  • subscription.deleted - Subscription canceled
  • invoice.payment_succeeded - Invoice paid
  • invoice.payment_failed - Invoice payment failed

Webhook Payload Example

POST https://merchant.com/webhooks/payment
Content-Type: application/json
X-Signature: sha256=7h3k2m9p4x1q8n5w...

{
  "id": "evt_9x7k3m2n",
  "type": "payment.succeeded",
  "created": "2025-01-15T10:30:45Z",
  "data": {
    "object": {
      "id": "pay_9x7k3m2n",
      "amount": 10000,
      "currency": "USD",
      "status": "succeeded",
      "merchant_id": "merch_7h3k2m9p",
      "created": "2025-01-15T10:30:00Z"
    }
  },
  "livemode": true
}

Webhook Security

  • Signature Verification: Validate X-Signature header using shared secret
  • HTTPS Only: Webhook endpoints must use TLS
  • Idempotency: Handle duplicate webhook deliveries
  • Timeout: Respond within 5 seconds to avoid retries
  • Retry Logic: Failed webhooks retry with exponential backoff

API Best Practices

  • Always use idempotency keys for payment operations
  • Implement proper error handling and retry logic
  • Store payment IDs for reconciliation and support
  • Use webhooks for asynchronous event processing
  • Never log full card numbers or sensitive data
  • Implement rate limiting to prevent abuse
  • Use API versioning for backward compatibility
