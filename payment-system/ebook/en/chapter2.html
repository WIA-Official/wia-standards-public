Payment Data Structures & Card Formats

2.1 Card Data Structure

Understanding how payment card data is structured is fundamental to implementing secure payment systems. This chapter covers the data formats, validation rules, and security requirements for handling card information.

Primary Account Number (PAN)

The Primary Account Number is the unique identifier for a payment card account. It follows a structured format defined by ISO/IEC 7812.

PAN Structure

Card Number: 4532 1234 5678 9010
│    │              │         └── Check Digit (Luhn)
│    │              └──────────── Account Identifier
│    └─────────────────────────── Issuer Identification (BIN/IIN)
└──────────────────────────────── Major Industry Identifier (MII)

Breakdown:
Position 1:      MII (Major Industry Identifier)
Positions 1-6:   BIN/IIN (Bank/Issuer Identification Number)
Positions 7-15:  Account Number
Position 16:     Check Digit (Luhn algorithm)

Luhn Algorithm Validation

The Luhn algorithm (also called modulus 10 or mod 10 algorithm) validates card numbers and detects simple errors in digit entry.

Algorithm Steps:
1. Starting from rightmost digit (excluding check digit), double every second digit
2. If doubling results in two digits, add them together (e.g., 16 → 1+6 = 7)
3. Sum all the digits
4. If sum modulo 10 equals 0, the number is valid

Example: Validate 4532 1234 5678 9010

Card number:  4  5  3  2  1  2  3  4  5  6  7  8  9  0  1  0
Double alt:   4 10  3  4  1  4  3  8  5 12  7 16  9  0  1  0
Add digits:   4  1  3  4  1  4  3  8  5  3  7  7  9  0  1  0
Sum: 4+1+3+4+1+4+3+8+5+3+7+7+9+0+1+0 = 60
60 % 10 = 0 ✓ Valid

Implementation:
function luhnCheck(cardNumber) {
  const digits = cardNumber.replace(/\D/g, '').split('').reverse();
  const checksum = digits.reduce((sum, digit, index) => {
    let value = parseInt(digit);
    if (index % 2 === 1) {
      value *= 2;
      if (value > 9) value -= 9;
    }
    return sum + value;
  }, 0);
  return checksum % 10 === 0;
}

2.2 Magnetic Stripe Data

Magnetic stripe cards store data on three tracks. Most payment systems use Track 2 data.

Track 1 Format

Format: %B{PAN}^{Name}^{Expiry}{Service Code}{Discretionary Data}?{LRC}

Example: %B4532123456789010^DOE/JOHN^2512101123456789012345?

Components:
%B            : Start sentinel for Track 1
4532...9010   : Primary Account Number (PAN)
^             : Field separator
DOE/JOHN      : Cardholder name (surname/firstname)
^             : Field separator
25            : Expiry year (2025)
12            : Expiry month (December)
101           : Service code
123456789012345 : Discretionary data (optional)
?             : End sentinel
LRC           : Longitudinal Redundancy Check

Track 1 Specs:
- Density: 210 bits per inch
- Character set: ALPHA (alphanumeric)
- Max length: 79 characters
- Encoding: 7-bit including parity

Track 2 Format

Format: ;{PAN}={Expiry}{Service Code}{Discretionary Data}?{LRC}

Example: ;4532123456789010=25121011234?

Components:
;             : Start sentinel for Track 2
4532...9010   : Primary Account Number (PAN)
=             : Field separator
25            : Expiry year (2025)
12            : Expiry month (December)
101           : Service code
1234          : Discretionary data (PIN verification key indicator, etc.)
?             : End sentinel
LRC           : Longitudinal Redundancy Check

Track 2 Specs:
- Density: 75 bits per inch
- Character set: NUMERIC only (0-9, =, ?)
- Max length: 40 characters
- Encoding: 5-bit including parity
- Most commonly used for transactions

Service Code

The 3-digit service code defines card usage restrictions and authorization requirements.

Position Digit Meaning

(Authorization)		 1 International use, ICC (chip)

2nd Digit
(PIN)		 0 Normal authorization, no PIN required

  • 101: International, no restrictions, no PIN (most credit cards)
  • 121: International, no restrictions, online PIN required (debit cards)
  • 201: International, magnetic stripe, no restrictions, no PIN
  • 221: International, magnetic stripe, online PIN required

EMV Transaction Cryptograms

EMV chips generate cryptographic validation codes for each transaction:

ARQC - Authorization Request Cryptogram

  • Generated by card for online authorization
  • Sent to issuer for validation
  • Proves card authenticity to issuer
  • Includes transaction data in cryptogram calculation
  • Cannot be reused for another transaction

TC - Transaction Certificate

  • Card approves transaction offline
  • Generated when issuer parameters allow offline approval
  • Transaction completes without real-time authorization
  • Submitted in batch for clearing

AAC - Application Authentication Cryptogram

  • Card declines transaction offline
  • Card determines transaction too risky
  • Transaction does not proceed
  • Merchant receives decline message

ARPC - Authorization Response Cryptogram

  • Generated by issuer in response to ARQC
  • Sent back to card via terminal
  • Card validates issuer response
  • Ensures response came from legitimate issuer

2.4 PCI-DSS Data Classification

The Payment Card Industry Data Security Standard (PCI-DSS) classifies card data into different categories with specific security requirements.

Cardholder Data (CHD)

Primary Account Number (PAN)

⚠️ Highest Security Level

PAN must be protected at all times. If stored, it must be encrypted or tokenized. Never store full PAN in plain text.

  • Storage: Encrypted or tokenized only
  • Transmission: Encrypted (TLS 1.2+)
  • Display: Masked (show last 4 digits only)
  • Logging: Never log full PAN
  • Retention: Minimize storage duration

Cardholder Name

  • Storage: Allowed if business need justified
  • Protection: Encrypt if stored with PAN
  • Access: Restrict to authorized personnel

Expiration Date

  • Storage: Allowed if business need justified
  • Protection: Encrypt if stored with PAN
  • Usage: Required for recurring billing

Service Code

  • Storage: Allowed if business need justified
  • Protection: Encrypt if stored with PAN
  • Usage: Defines card authorization rules

Sensitive Authentication Data (SAD)

🚫 NEVER STORE AFTER AUTHORIZATION

Sensitive authentication data must NEVER be stored after authorization, even if encrypted. Violation is grounds for immediate decertification.

Full Magnetic Stripe Data

  • Storage: ❌ NEVER (not even encrypted)
  • Usage: For authorization only
  • Transmission: Encrypted, immediately purged
  • Violation Penalty: Up to $500,000 per incident

CAV2/CVC2/CVV2/CID (Card Verification Value)

  • Storage: ❌ NEVER (not even encrypted)
  • Usage: For authorization only
  • Purpose: Proves physical card possession
  • Location: 3 digits on back (4 on front for Amex)

PIN / PIN Block

  • Storage: ❌ NEVER (not even encrypted)
  • Transmission: Triple DES or stronger encryption
  • Purpose: Cardholder verification
  • Processing: Only in secure cryptographic devices (HSM)

Data Masking Requirements

PAN Masking Examples:

Full PAN:        4532 1234 5678 9010
Masked Display:  •••• •••• •••• 9010
Masked Display:  ****-****-****-9010
Masked Display:  XXXX-XXXX-XXXX-9010
Masked Display:  4532-****-****-9010 (first 6 + last 4 for support)

Rules:
- Maximum first 6 + last 4 digits can be displayed
- Middle digits must be masked
- Use consistent masking character (•, *, X)
- Masking applies to:
  * Web pages and forms
  * Mobile applications
  * Email communications
  * SMS notifications
  * Printed receipts
  * Call center displays
  * Log files and databases

2.5 Tokenization

Tokenization replaces sensitive card data with non-sensitive tokens, reducing PCI-DSS scope and enhancing security.

Tokenization Process

1. Initial Transaction:
   Customer:        4532 1234 5678 9010
   ↓ (secure transmission)
   Token Vault:     Creates token
   ↓
   Token:           tok_7h3k2m9p4x1q8n5w

2. Subsequent Transactions:
   Merchant sends:  tok_7h3k2m9p4x1q8n5w
   ↓
   Token Vault:     Detokenizes → 4532 1234 5678 9010
   ↓
   Card Network:    Processes real PAN
   ↓
   Response:        Authorization approved/declined

Benefits:
✓ Merchant never stores real PAN
✓ Reduced PCI-DSS scope
✓ Lower compliance costs
✓ Enhanced security
✓ Breach protection (tokens useless to attackers)

Token Formats

Format-Preserving Tokens

Real PAN:  4532 1234 5678 9010
Token:     4532 9876 5432 1098

Characteristics:
- Maintains PAN format (same length, Luhn valid)
- First 6 digits preserved for BIN routing
- Last 4 digits may be preserved for display
- Allows use in legacy systems expecting PAN format

Non-Format-Preserving Tokens

Real PAN:  4532 1234 5678 9010
Token:     tok_7h3k2m9p4x1q8n5w

Characteristics:
- Different format from PAN
- More flexible token structure
- Clearly distinguishable from real PAN
- Prevents accidental PAN exposure

Network Tokenization

Card networks provide tokenization services that create network-specific tokens:

  • Visa Token Service (VTS)
  • Mastercard Digital Enablement Service (MDES)
  • American Express Token Service

Network Token Benefits

  • Higher authorization rates (issuer recognizes token)
  • Automatic card updates (token persists through reissue)
  • Enhanced security (dynamic CVV)
  • Mobile wallet integration (Apple Pay, Google Pay)
  • Fraud reduction (tokens cannot be used outside domain)

2.6 Data Encryption Standards

Encryption at Rest

Required Standards

  • AES-256: Advanced Encryption Standard with 256-bit keys (recommended)
  • AES-128: 128-bit keys (minimum acceptable)
  • RSA-2048: For key encryption (minimum)

⚠️ Deprecated Algorithms

The following are NO LONGER ACCEPTABLE:

  • DES (Data Encryption Standard)
  • 3DES/TDEA (after 2023)
  • RSA-1024 or lower
  • MD5 hashing
  • SHA-1 hashing

Encryption in Transit

TLS Requirements

Required:
✓ TLS 1.2 (minimum)
✓ TLS 1.3 (recommended)

Prohibited:
✗ SSL 2.0
✗ SSL 3.0
✗ TLS 1.0
✗ TLS 1.1

Cipher Suites (Examples of Strong):
- TLS_AES_256_GCM_SHA384
- TLS_CHACHA20_POLY1305_SHA256
- TLS_AES_128_GCM_SHA256
- ECDHE-RSA-AES256-GCM-SHA384
- ECDHE-RSA-AES128-GCM-SHA256

Key Management

Proper cryptographic key management is critical for encryption security:

Key Generation

  • Use cryptographically secure random number generator (CSRNG)
  • Generate keys in secure Hardware Security Module (HSM)
  • Minimum key strength: 128-bit for symmetric, 2048-bit for asymmetric
  • Never use predictable or hardcoded keys

Key Storage

  • Store keys in HSM or secure key vault
  • Encrypt data encryption keys with key encryption keys
  • Maintain air gap between keys and encrypted data
  • Never store keys in application code or configuration files

Key Rotation

  • Rotate encryption keys annually at minimum
  • Rotate immediately if compromise suspected
  • Maintain versioned keys for data re-encryption
  • Automate rotation process where possible

Key Destruction

  • Securely destroy retired keys
  • Overwrite key material multiple times
  • Document key destruction in audit logs
  • Use cryptographic erasure for data destruction

2.7 Data Retention and Disposal

Retention Policies

Data Type Retention Period Justification Required
PAN (encrypted/tokenized) Business need only Yes - minimize duration
Cardholder name Business need only Yes
Transaction history 7 years (regulatory) Yes - tax/audit requirements
Authorization logs 90 days minimum Yes - fraud investigation
Full magnetic stripe ❌ Never store N/A - Prohibited
CVV/CVC/CVV2/CID ❌ Never store N/A - Prohibited
PIN/PIN Block ❌ Never store N/A - Prohibited

Secure Data Disposal

Electronic Media

  • Degaussing: Magnetic field disruption for HDDs
  • Physical Destruction: Shredding, pulverizing, incinerating
  • Cryptographic Erasure: Destroy encryption keys
  • Secure Wipe: DoD 5220.22-M (7-pass overwrite) minimum

Paper Media

  • Cross-Cut Shredding: Minimum 1mm x 5mm particles
  • Pulping: Industrial paper pulping
  • Incineration: Complete combustion

Best Practice: Minimize Data Collection

The most secure data is data you don't collect. Implement "data minimization" principles:

  • Collect only data necessary for business purpose
  • Use tokenization to avoid storing real PANs
  • Purge data as soon as business need expires
  • Never collect sensitive authentication data
  • Implement automatic data retention enforcement
