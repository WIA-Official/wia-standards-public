CHAPTER 2

Data Structures for Fraud Detection

Comprehensive guide to data formats, transaction structures, user profiles, and event representations in fraud prevention systems

2.1 Fraud Event Data Format

The fraud event data structure is the foundational element of any fraud prevention system. It captures all relevant information about a potential fraud incident, including transaction details, user context, detection methods, risk indicators, and recommended actions.

A well-designed fraud event structure must balance completeness with performance. It needs to capture sufficient detail for accurate fraud detection and post-incident analysis while remaining compact enough for real-time processing at scale.

Core Event Structure

Every fraud event contains several key components:

  • Event Metadata: Unique identifier (UUID), timestamp (ISO 8601), event type, severity level, and current status
  • Transaction Details: Amount, currency, merchant information, payment method, location data, and timing
  • User Context: User identifier, account age, risk score, verification level, device fingerprint, and behavioral profile
  • Detection Information: Method used (rule-based, ML model, anomaly detection), model version, confidence scores, and feature values
  • Risk Indicators: List of factors contributing to fraud score, each with type, weight, and description
  • Response Actions: Recommended action (allow, block, challenge, review), response time, reviewer information, and resolution notes

2.2 Transaction Monitoring Data

Transaction monitoring structures enable real-time analysis of payment activity, velocity checks, and pattern detection. These structures aggregate multiple transactions over time windows to identify suspicious behavior patterns.

Real-Time Metrics

Transaction monitoring systems track several key metrics in real-time:

  • Volume Metrics: Transaction count, total amount, average amount per transaction, peak transaction rate
  • Fraud Metrics: Suspicious count, blocked count, challenged count, fraud detection rate
  • Performance Metrics: False positive rate, false negative rate, average response time, throughput (TPS)
  • Model Metrics: Confidence distribution, model version in use, prediction latency

Velocity Checks

Velocity checks detect anomalous transaction patterns by monitoring activity rates over various time windows. Common velocity checks include:

  • Transaction Count Velocity: Number of transactions per hour/day from single user or device
  • Amount Velocity: Total dollar amount transacted within time window
  • Location Velocity: Geographic distance traveled between transactions (impossible travel detection)
  • Merchant Velocity: Transactions across different merchants in short timeframe
  • Card Velocity: Same card used across multiple devices or locations
Impossible Travel Example: If a user makes a transaction in New York at 10:00 AM and another in London at 10:30 AM, the system calculates they would need to travel 5,585 km in 30 minutes (11,170 km/h velocity). Since commercial flights max out around 900 km/h, this is physically impossible and flags as high-risk fraud.

Velocity Check Configuration

Check Type Time Window Typical Threshold Fraud Indication
Transaction Count 1 hour 10 transactions Rapid card testing
Amount Sum 24 hours $5,000 Account draining
Geographic Velocity 30 minutes 1,000 km/h Impossible travel
Merchant Diversity 1 hour 8 different merchants Card testing spree
Failed Attempts 15 minutes 3 failures Credential stuffing

2.3 User Behavioral Profiles

Behavioral profiles capture the typical patterns and preferences of individual users, enabling deviation detection. A comprehensive profile includes transaction patterns, location patterns, merchant preferences, device usage, and temporal behavior.

Transaction Pattern Components

Amount Distribution: Statistical distribution of transaction amounts including minimum, maximum, mean, median, standard deviation, and percentile values (p25, p50, p75, p95). This enables detection of unusually large or small transactions relative to user's history.

Frequency Patterns: Average daily, weekly, and monthly transaction counts. Seasonal variations and trend analysis over time. Sudden spikes in frequency indicate potential account compromise.

Temporal Patterns: Distribution of transactions by hour of day and day of week. Most users have consistent transaction timing—purchases during lunch breaks, evening shopping, weekend patterns. Transactions outside typical windows raise suspicion.

Location Pattern Analysis

Users typically transact from a limited set of locations. The behavioral profile tracks:

  • Primary Locations: Top 3-5 cities/regions where user transacts, with frequency percentages
  • Home Region: Geographic center of user's typical activity
  • Travel Patterns: Average distance between consecutive transactions, maximum observed velocity
  • International Activity: Percentage of international transactions, which countries
  • New Location Behavior: How user typically behaves when traveling (smaller transactions, specific merchant types)

Merchant and Category Preferences

Users show strong preferences for certain merchant categories and specific merchants:

  • Category Distribution: Percentage breakdown across MCC codes (grocery, restaurants, gas, entertainment, etc.)
  • Frequent Merchants: List of merchants user regularly patronizes
  • Merchant Diversity Score: How varied user's merchant selection is (scale 0-1)
  • High-Risk Category Exposure: Whether user typically transacts in high-risk categories (gambling, adult, cryptocurrency)

2.4 Machine Learning Training Data

ML training datasets must be carefully structured to enable effective model training while handling the extreme class imbalance inherent in fraud detection (typically 0.1-0.5% fraud rate).

Dataset Structure

A complete training dataset includes:

  • Feature Vectors: Numerical and categorical features extracted from transactions and user profiles
  • Labels: Ground truth classification (fraud / legitimate)
  • Timestamps: When transaction occurred (for temporal train/test splits)
  • Weights: Optional sample weights for handling class imbalance
  • Metadata: Additional context not used as features but valuable for analysis

Feature Categories

Category Example Features Data Type Importance
Transaction Amount Amount, amount/avg ratio, log(amount) Numeric High
Temporal Hour, day_of_week, is_weekend, time_since_last Numeric/Categorical Medium
Location Country, city, distance_from_home, velocity Categorical/Numeric Very High
Merchant MCC code, merchant_risk_score, is_new_merchant Categorical/Numeric High
Device Device_fingerprint, is_new_device, device_reputation Categorical/Numeric Very High
User History Account_age, transaction_count, previous_fraud Numeric Medium
Behavioral Deviation_score, pattern_match, anomaly_score Numeric Very High

Handling Class Imbalance

With fraud rates typically under 0.5%, standard ML algorithms struggle. Several techniques address this:

SMOTE (Synthetic Minority Over-sampling): Generates synthetic fraud examples by interpolating between existing fraud cases. Increases fraud representation to 5-10% of dataset.

Undersampling: Randomly reduces legitimate transactions to match fraud count. Fast but discards data. Often combined with ensemble methods.

Cost-Sensitive Learning: Assigns higher misclassification cost to false negatives (missing fraud). XGBoost scale_pos_weight parameter commonly used.

Anomaly Detection: Treats problem as outlier detection rather than classification. Trains on legitimate transactions only, identifies fraud as anomalies.

2.5 Alert and Notification Data

Alert structures enable communication of fraud incidents to relevant stakeholders with appropriate urgency and detail. Effective alert design balances completeness with actionability.

Alert Priority Levels

Alerts are categorized by priority to ensure appropriate response:

  • Critical: High-value fraud ($5,000+), account takeover, identity theft. Requires immediate action within 15 minutes.
  • High: Confirmed fraud pattern, multiple risk indicators. Response within 1 hour.
  • Medium: Suspicious activity requiring review. Response within 4 hours.
  • Low: Minor anomalies, informational alerts. Daily batch review acceptable.

Alert Delivery Channels

Multi-channel delivery ensures alerts reach appropriate personnel:

  • Email: Detailed reports with full analysis, suitable for medium/low priority
  • SMS: Brief notifications for high/critical priority, ensures immediate awareness
  • Push Notifications: Mobile app alerts for fraud analysts and users
  • Webhooks: Programmatic integration with incident management systems
  • Dashboard: Real-time alert feed in fraud operations center

2.7 Data Validation and Quality

Data quality directly impacts fraud detection accuracy. Rigorous validation ensures clean, consistent data throughout the system.

Validation Rules

  • Type Validation: Ensure fields match expected data types (string, number, boolean, etc.)
  • Range Validation: Values within acceptable ranges (amounts non-negative, scores 0-100, etc.)
  • Format Validation: Strings match expected patterns (UUIDs, ISO dates, currency codes)
  • Reference Validation: Foreign keys exist (user IDs, merchant IDs)
  • Business Logic Validation: Fields consistent with business rules (transaction date not in future)

Data Quality Metrics

Monitor these metrics to maintain high data quality:

  • Completeness: Percentage of records with all required fields populated
  • Accuracy: Percentage of records passing validation checks
  • Consistency: Percentage of records consistent across related systems
  • Timeliness: Lag between event occurrence and data availability
  • Uniqueness: Percentage of records with unique identifiers (no duplicates)

Key Takeaways

  • Comprehensive Structure: Fraud event data must capture transaction details, user context, detection methods, risk indicators, and response actions while maintaining real-time processing performance.
  • Standardization: Using standard formats (ISO 8601 timestamps, ISO 4217 currencies, UUID identifiers) ensures interoperability and simplifies system integration.
  • Velocity Monitoring: Transaction monitoring structures enable detection of suspicious patterns through velocity checks across multiple dimensions (amount, count, location, merchant).
  • Behavioral Profiles: Rich user profiles capturing transaction patterns, location preferences, merchant affinities, and temporal behavior enable effective deviation detection.
  • ML Training Data: Properly structured training data with balanced classes (via SMOTE, undersampling, cost-sensitive learning) and comprehensive features is critical for model accuracy.
  • Alert Design: Effective alerts balance detail with actionability, use appropriate channels for priority levels, and enable efficient fraud analyst workflows.
  • Data Quality: Rigorous validation and quality monitoring (completeness, accuracy, consistency, timeliness) directly impact fraud detection effectiveness.
