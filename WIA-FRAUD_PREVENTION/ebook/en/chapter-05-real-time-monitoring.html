5.1 Real-Time Architecture

Real-time fraud detection demands sub-100ms response times while processing thousands of transactions per second. This requires carefully designed architecture combining streaming data processing, low-latency model inference, efficient feature extraction, and intelligent caching. Every millisecond counts—delays translate directly to poor user experience and potential fraud losses.

Modern real-time systems use event-driven architectures where transactions flow through a processing pipeline: data ingestion layer receives transactions, feature extraction enriches with behavioral context, ML models score fraud risk, decision engine determines action (allow/block/challenge), and response returns to payment system. Each component must meet strict latency SLAs.

Key Architecture Components

  • Message Queue: Apache Kafka, AWS Kinesis, or RabbitMQ buffer incoming transactions, providing fault tolerance and load smoothing
  • Stream Processor: Apache Flink or Spark Streaming perform real-time feature calculations and aggregations
  • Model Serving: TensorFlow Serving, TorchServe, or custom APIs host ML models with GPU acceleration
  • Cache Layer: Redis or Memcached store user profiles, behavioral features, and recent transaction history
  • Decision Engine: Rules engine combines ML scores with business logic to determine final action

5.2 Velocity Checks and Rate Limiting

Velocity checks detect suspicious transaction patterns by monitoring activity rates over various time windows. A user making 20 transactions in 10 minutes or $10,000 in purchases within an hour triggers velocity alerts. These checks catch fraud testing, account draining, and bot-driven attacks that might not have obviously fraudulent individual transactions.

Implementation Strategy

Sliding window counters track transaction count and amount sum over rolling time windows (1 hour, 24 hours, 7 days). For each transaction, query recent activity: "SELECT COUNT(*), SUM(amount) FROM transactions WHERE user_id = ? AND timestamp > NOW() - INTERVAL 1 HOUR". Compare against user-specific or global thresholds. Exceeded thresholds trigger alerts or blocks.

Efficient implementation uses time-series databases (InfluxDB, TimescaleDB) or in-memory data structures (Redis sorted sets) for fast range queries. Pre-aggregated rollups (5-minute buckets) reduce query load. Approximate algorithms (HyperLogLog for cardinality, Count-Min Sketch for frequencies) enable extreme scale with bounded memory.

Velocity Check Types

  • Transaction Count Velocity: Number of transactions per hour/day—catches card testing (rapid succession of small purchases)
  • Amount Velocity: Total dollar amount per time window—catches account draining
  • Merchant Velocity: Transactions across different merchants—unusual diversity indicates stolen card
  • Geographic Velocity: Distance traveled between transactions—impossible travel detection
  • Failed Attempt Velocity: Failed login/payment attempts—credential stuffing or card testing
  • Device Velocity: Same card used from multiple devices—indicates card sharing or theft

5.3 Streaming ML Inference

Traditional batch ML prediction processes transactions in groups (hourly, daily), unacceptable for fraud where every second counts. Streaming inference evaluates each transaction immediately upon arrival, returning prediction in milliseconds. This requires optimized model deployment, efficient feature pipelines, and careful resource management.

Model Optimization

Production models undergo optimization for inference speed: pruning removes unimportant weights/neurons (10-30% speedup, minimal accuracy loss), quantization converts 32-bit floats to 8-bit integers (2-4x speedup on CPU), distillation trains smaller "student" model to mimic larger "teacher" (3-5x speedup, 2-3% accuracy loss), and batching processes multiple transactions together (2-10x throughput with slightly higher latency).

Feature Serving

ML models require features computed from current transaction plus historical user data. Feature serving systems (Tecton, Feast, AWS Feature Store) precompute expensive features, cache results, and serve with low latency. Real-time features (transaction amount, time) computed on-demand, while historical features (user's average amount, transaction frequency) retrieved from cache. Feature freshness—how recent the cached features are—impacts accuracy.

5.4 Scalability and Performance

Production fraud detection systems must scale to millions of transactions per day (thousands per second during peak) while maintaining consistent low latency. This requires horizontal scaling, load balancing, and careful capacity planning. Performance degrades non-linearly—a system handling 1,000 TPS comfortably may collapse at 1,200 TPS without proper scaling.

Horizontal Scaling Patterns

Stateless services (model inference, feature computation) scale linearly—double the servers, double the throughput. Deploy multiple instances behind load balancer, automatically scaling based on metrics (CPU usage, queue depth, request latency). Stateful services (user profile lookup, velocity checks) require partitioning—shard by user ID, ensuring all requests for a user route to same instance.

Load Testing and Capacity Planning

Before production, stress test under realistic load: simulate peak transaction rates (holiday shopping, Black Friday), measure latency distribution (p50, p95, p99), identify bottlenecks through profiling, and determine breaking point where latency degrades. Provision capacity for 3x peak load (headroom for traffic spikes) and test failover scenarios (what happens when 20% of servers fail).

5.5 High Availability and Fault Tolerance

Fraud detection systems must remain operational 24/7—downtime means fraud goes undetected and revenue is lost. High availability requires redundancy at every layer: multiple instances of each service, multi-region deployment, automatic failover, and graceful degradation when components fail.

Redundancy Patterns

Active-active configuration runs multiple instances simultaneously, load balancing across all instances—immediate failover with no downtime. Active-passive maintains hot standbys ready to take over—failover takes 30-60 seconds. Multi-region deployment places complete system in multiple AWS regions or data centers—protects against regional outages, requires data replication and careful latency management.

Graceful Degradation

When ML model service fails, fall back to rule-based detection—less accurate but functional. When velocity check database is unavailable, skip velocity checks temporarily (accept increased fraud risk) or use cached values (accept staleness). When fraud detection system is completely unavailable, allow transactions through (fraud prevention teams can review retroactively) or block everything (safe but terrible user experience)—decision depends on business priorities.

Key Takeaways

  • Latency Budget: Sub-100ms fraud detection requires careful allocation: 15ms feature extraction, 30ms ML inference, 5ms decision logic, 20ms database lookups, 10ms network—every component optimized.
  • Velocity Monitoring: Sliding window counters track transaction patterns (count, amount, merchant diversity, geographic movement) over time windows, catching fraud testing, account draining, and bot attacks.
  • Streaming Inference: Real-time ML predictions demand model optimization (pruning, quantization, distillation), feature caching, and batching to achieve thousands of predictions per second with millisecond latency.
  • Horizontal Scaling: Stateless services scale linearly with server count, stateful services require partitioning—deploy behind load balancers with auto-scaling based on metrics, provisioning 3x peak capacity.
  • Performance Monitoring: Track P50/P95/P99 latency, throughput, error rate, and queue depth—alert when thresholds exceeded, automatically scale or rollback to maintain SLAs.
  • High Availability: Multi-region active-active deployment with automatic failover ensures 99.99% uptime—graceful degradation (fallback to rules when ML fails) maintains fraud protection even during partial outages.
