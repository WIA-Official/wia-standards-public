CHAPTER 7

Error Handling and Recovery

← Contents ← Previous Next →

All systems fail sometimes. The difference between fragile and robust collaboration lies not in preventing all errors, but in how gracefully systems handle and recover from failures. This chapter explores strategies for resilient AI-human collaboration that maintains operational continuity even when components fail, guided by 弘익人間 principles that prioritize human safety and dignity.

Types of Errors

Understanding error types helps design appropriate handling strategies. Different errors require different responses.

AI Model Errors

  • Prediction errors: Wrong classification or forecast
  • Confidence miscalibration: Confidence doesn't match accuracy
  • Distribution shift: Real-world data differs from training data
  • Adversarial inputs: Maliciously crafted inputs fool the model
  • Edge cases: Novel situations outside training distribution

Human Errors

  • Automation bias: Over-relying on AI without verification
  • Attention failures: Missing critical information due to cognitive overload
  • Misunderstanding: Misinterpreting AI output or interface
  • Fatigue errors: Mistakes due to prolonged work
  • Policy violations: Decisions inconsistent with guidelines

System Errors

  • Infrastructure failures: Server downtime, network issues
  • Integration failures: Communication breakdown between components
  • Data pipeline errors: Missing or corrupted data
  • Scaling failures: Performance degradation under high load

Error Detection

Catching errors early limits their impact. Implement multiple layers of detection across the collaboration pipeline.

AI Self-Monitoring

AI systems can detect some errors themselves through confidence assessment and anomaly detection. 

interface ErrorDetector {
  async detectAnomalies(
    input: any,
    prediction: any
  ): Promise<{
    anomalies_detected: boolean;
    anomaly_types: string[];
    confidence_in_detection: number;
  }>;
}

class PredictionMonitor implements ErrorDetector {
  async detectAnomalies(
    input: any,
    prediction: any
  ): Promise {
    const anomalies: string[] = [];

    // Check for out-of-distribution inputs
    const input_anomaly_score = await this.calculateAnomalyScore(input);
    if (input_anomaly_score > 0.8) {
      anomalies.push('out_of_distribution_input');
    }

    // Check for confidence-prediction mismatch
    const expected_confidence = await this.estimateExpectedConfidence(input);
    if (Math.abs(prediction.confidence - expected_confidence) > 0.3) {
      anomalies.push('confidence_miscalibration');
    }

    // Check for adversarial patterns
    const adversarial_score = await this.detectAdversarial(input);
    if (adversarial_score > 0.7) {
      anomalies.push('possible_adversarial_input');
    }

    return {
      anomalies_detected: anomalies.length > 0,
      anomaly_types: anomalies,
      confidence_in_detection: this.calculateDetectionConfidence(anomalies)
    };
  }
}

Human Error Detection

Monitor human behavior patterns for signs of errors: 

class HumanErrorDetector {
  async detectPotentialErrors(
    reviewer_id: string,
    decision: any,
    context: any
  ): Promise<{
    error_likely: boolean;
    warning_signals: string[];
  }> {
    const signals: string[] = [];

    // Check for automation bias
    const recent_decisions = await this.getRecentDecisions(reviewer_id, 20);
    const agreement_rate = recent_decisions.filter(d =>
      d.decision === d.ai_suggestion
    ).length / recent_decisions.length;

    if (agreement_rate > 0.95) {
      signals.push('possible_automation_bias');
    }

    // Check for unusually fast decisions on complex cases
    const decision_time = decision.time_spent;
    const expected_time = await this.estimateExpectedReviewTime(context);

    if (decision_time < expected_time * 0.5 && context.complexity > 7) {
      signals.push('insufficient_review_time');
    }

    // Check for inconsistency with past decisions
    const similar_past = await this.findSimilarDecisions(context);
    const consistency_rate = similar_past.filter(d =>
      d.decision === decision.decision
    ).length / similar_past.length;

    if (consistency_rate < 0.3) {
      signals.push('inconsistent_with_history');
    }

    return {
      error_likely: signals.length >= 2,
      warning_signals: signals
    };
  }
}

Cross-Validation

Use multiple signals to validate decisions: 

class CrossValidator {
  async validateDecision(
    decision: any,
    context: any
  ): Promise<{
    validation_passed: boolean;
    issues: string[];
  }> {
    const issues: string[] = [];

    // Check multiple AI models agree
    const models = await this.getEnsembleModels();
    const predictions = await Promise.all(
      models.map(m => m.predict(context.features))
    );

    const consensus = this.checkConsensus(predictions);
    if (!consensus && decision.source === 'ai') {
      issues.push('model_disagreement');
    }

    // Check business rules
    const rule_violations = await this.checkBusinessRules(decision, context);
    if (rule_violations.length > 0) {
      issues.push(...rule_violations);
    }

    // Check statistical outliers
    const historical_similar = await this.getHistoricalDecisions(context);
    if (this.isOutlier(decision, historical_similar)) {
      issues.push('statistical_outlier');
    }

    return {
      validation_passed: issues.length === 0,
      issues
    };
  }
}

Graceful Degradation

When errors occur, systems should degrade gracefully rather than fail catastrophically. Provide reduced functionality rather than no functionality.

Degradation Levels

enum DegradationLevel {
  FULL_OPERATION = 0,      // All features working
  REDUCED_CONFIDENCE = 1,  // AI less certain, more human review
  AI_DISABLED = 2,         // AI offline, full manual operation
  QUEUE_ONLY = 3,          // Accept requests, process when restored
  READ_ONLY = 4            // View-only mode, no new decisions
}

class GracefulDegradation {
  async handleFailure(
    failure_type: string,
    severity: number
  ): Promise {
    switch (failure_type) {
      case 'ai_model_error':
        if (severity > 0.8) {
          // Critical AI failure - switch to manual
          await this.switchToManualMode();
          return DegradationLevel.AI_DISABLED;
        } else {
          // Reduce AI confidence, increase human review
          await this.lowerConfidenceThresholds();
          return DegradationLevel.REDUCED_CONFIDENCE;
        }

      case 'database_error':
        if (severity > 0.9) {
          // Can't write - read-only mode
          return DegradationLevel.READ_ONLY;
        } else {
          // Queue operations for later
          await this.enableQueueMode();
          return DegradationLevel.QUEUE_ONLY;
        }

      case 'high_load':
        // Prioritize critical operations
        await this.enablePriorityMode();
        return DegradationLevel.REDUCED_CONFIDENCE;

      default:
        return DegradationLevel.FULL_OPERATION;
    }
  }
}

Fallback Strategies

  • Previous model version: Roll back to last known good model
  • Simpler algorithm: Use faster, simpler backup model
  • Human-only mode: All decisions go to humans
  • Queue and defer: Accept inputs, process when recovered
  • Reduced feature set: Disable non-essential features

Error Recovery Strategies

Different errors require different recovery approaches. Design recovery strategies appropriate to error types and contexts.

Automatic Recovery

Some errors can be automatically resolved: 

class AutoRecovery {
  async attemptRecovery(
    error: Error,
    context: any
  ): Promise<{
    recovered: boolean;
    recovery_method: string;
  }> {
    // Retry with exponential backoff
    if (error.type === 'transient_network_error') {
      return await this.retryWithBackoff(context.operation, 3);
    }

    // Use cached data
    if (error.type === 'data_unavailable') {
      const cached = await this.getCachedData(context.data_key);
      if (cached && !this.isStale(cached)) {
        return {
          recovered: true,
          recovery_method: 'used_cached_data'
        };
      }
    }

    // Fallback to simpler model
    if (error.type === 'model_timeout') {
      const result = await this.useFallbackModel(context.input);
      return {
        recovered: true,
        recovery_method: 'fallback_model'
      };
    }

    return {
      recovered: false,
      recovery_method: 'none'
    };
  }
}

Human-Assisted Recovery

Complex errors may require human intervention: 

class HumanAssistedRecovery {
  async escalateForRecovery(
    error: Error,
    failed_operation: any
  ): Promise {
    // Create recovery task for human
    const recovery_task = {
      task_id: generateId(),
      error_type: error.type,
      error_message: error.message,
      failed_operation: failed_operation,
      context: await this.gatherContext(error),
      suggested_actions: await this.suggestRecoveryActions(error),
      priority: this.assessPriority(error)
    };

    // Route to appropriate expert
    const expert = await this.findExpert(error.type);
    await this.assignRecoveryTask(recovery_task, expert);

    // Notify stakeholders
    await this.notifyStakeholders(error, recovery_task);
  }

  async suggestRecoveryActions(
    error: Error
  ): Promise {
    const actions = [];

    if (error.type === 'data_corruption') {
      actions.push('Restore from backup');
      actions.push('Manually correct corrupted records');
      actions.push('Re-run data validation pipeline');
    }

    if (error.type === 'model_drift') {
      actions.push('Retrain model with recent data');
      actions.push('Adjust confidence thresholds');
      actions.push('Increase human review percentage');
    }

    return actions;
  }
}

Rollback and Replay

For serious errors, rollback to known good state and replay operations: 

class RollbackManager {
  async rollbackAndReplay(
    error_timestamp: Date,
    error_scope: string
  ): Promise {
    // Find last known good state
    const checkpoint = await this.findCheckpoint(error_timestamp);

    // Rollback to checkpoint
    await this.restoreFromCheckpoint(checkpoint);

    // Identify operations after checkpoint
    const operations_to_replay = await this.getOperations(
      checkpoint.timestamp,
      error_timestamp
    );

    // Filter out erroneous operations
    const valid_operations = operations_to_replay.filter(op =>
      !this.isErroneous(op, error_scope)
    );

    // Replay valid operations
    for (const op of valid_operations) {
      await this.replayOperation(op);
    }

    // Verify system state
    await this.verifyIntegrity();
  }
}

Communication During Failures

How errors are communicated affects user trust and response. Transparent, actionable communication maintains confidence even during failures.

User Communication Principles

  • Be honest: Don't hide or minimize errors
  • Be specific: Explain what went wrong
  • Be actionable: Tell users what to do next
  • Be timely: Communicate quickly, update regularly
  • Be empathetic: Acknowledge impact on users
interface ErrorCommunication {
  user_message: {
    headline: string;
    description: string;
    impact: string;
    user_action: string;
    timeline: string;
  };
  technical_details?: {
    error_code: string;
    stack_trace: string;
    debug_info: any;
  };
  updates: Array<{
    timestamp: Date;
    status: string;
    message: string;
  }>;
}

class ErrorCommunicator {
  async communicateError(
    error: Error,
    impact_assessment: any
  ): Promise {
    const message: ErrorCommunication = {
      user_message: {
        headline: this.generateHeadline(error),
        description: this.explainError(error),
        impact: this.describeImpact(impact_assessment),
        user_action: this.suggestUserAction(error),
        timeline: this.estimateRecovery(error)
      },
      updates: []
    };

    // Include technical details for power users
    if (this.userWantsTechnicalDetails()) {
      message.technical_details = {
        error_code: error.code,
        stack_trace: error.stack,
        debug_info: await this.gatherDebugInfo(error)
      };
    }

    await this.sendNotification(message);
    return message;
  }

  explainError(error: Error): string {
    const explanations = {
      'model_error': 'Our AI system encountered an issue processing your request. The prediction may be less accurate than usual.',
      'data_error': 'We're experiencing issues accessing some data. Some information may be incomplete.',
      'system_overload': 'We're handling higher than normal volume. Processing may be slower than usual.',
    };

    return explanations[error.type] ||
           'We encountered an unexpected issue. Our team has been notified.';
  }
}

Learning from Failures

Every error is an opportunity to improve. Systematic error analysis reveals patterns and informs preventive measures.

Post-Incident Analysis

interface IncidentReport {
  incident_id: string;
  incident_date: Date;
  detection_time: Date;
  resolution_time: Date;
  error_type: string;
  root_cause: string;
  contributing_factors: string[];
  impact: {
    users_affected: number;
    decisions_affected: number;
    downtime_minutes: number;
    financial_impact?: number;
  };
  timeline: Array<{
    timestamp: Date;
    event: string;
  }>;
  what_went_well: string[];
  what_went_wrong: string[];
  lessons_learned: string[];
  action_items: Array<{
    action: string;
    owner: string;
    due_date: Date;
    status: 'open' | 'in_progress' | 'completed';
  }>;
}

class PostIncidentReview {
  async conductReview(incident_id: string): Promise {
    // Gather incident data
    const incident = await this.getIncidentData(incident_id);

    // Conduct blameless retrospective
    const retrospective = await this.facilitateRetrospective(
      incident.team_members
    );

    // Identify root cause
    const root_cause = await this.performRootCauseAnalysis(incident);

    // Generate action items
    const actions = await this.identifyPreventiveMeasures(
      root_cause,
      retrospective
    );

    const report: IncidentReport = {
      incident_id,
      incident_date: incident.start_time,
      detection_time: incident.detected_at,
      resolution_time: incident.resolved_at,
      error_type: incident.type,
      root_cause: root_cause.primary_cause,
      contributing_factors: root_cause.contributing_factors,
      impact: await this.assessImpact(incident),
      timeline: incident.timeline,
      what_went_well: retrospective.went_well,
      what_went_wrong: retrospective.went_wrong,
      lessons_learned: retrospective.lessons,
      action_items: actions
    };

    await this.publishReport(report);
    return report;
  }
}

Preventive Measures

Use incident learnings to prevent future failures:

  • Enhanced monitoring: Add checks for error precursors
  • Improved testing: Add test cases for failure scenarios
  • Circuit breakers: Prevent cascading failures
  • Rate limiting: Protect against overload
  • Redundancy: Eliminate single points of failure
  • Training: Educate team on error patterns
弘益人間 in Error Handling

Error handling guided by 弘익人間 prioritizes human safety and dignity. Never hide errors that could harm users. Provide graceful degradation that maintains essential services. Communicate honestly about failures. Learn from mistakes to prevent future harm. The goal is not zero errors (impossible) but ethical, transparent, and compassionate handling when errors occur.

Chapter Summary

Resilient AI-human collaboration requires robust error handling and recovery. Key concepts include:

  • Understanding error types: AI model errors, human errors, and system errors
  • Multi-layered error detection through AI self-monitoring, human behavior analysis, and cross-validation
  • Graceful degradation that provides reduced functionality rather than complete failure
  • Recovery strategies including automatic recovery, human-assisted recovery, and rollback/replay
  • Transparent error communication that is honest, specific, actionable, timely, and empathetic
  • Post-incident analysis to learn from failures and implement preventive measures
  • Focus on human safety and dignity in all error handling decisions
