Chapter 4

Phase 1: Data Format

4.1 Phase 1 Overview: Data Format Standards

Phase 1 establishes the foundational data formats for representing all aspects of infrastructure integration. Using JSON-LD for semantic web compatibility and human readability, these formats enable consistent data exchange across systems, organizations, and borders.

4.2 Infrastructure Project Schema

The core data structure represents infrastructure projects from conception through operation:

{ "@context": "https://wiastandards.com/contexts/uni-005/v1", "@type": "InfrastructureProject", "id": "wia:uni:proj:2025-tkr-001", "projectType": "railway", "name": "Trans-Korean Railway Modernization Phase 1", "description": "Upgrade Gyeongui Line for high-speed operation", "location": { "startPoint": {"lat": 37.5665, "lon": 126.9780, "name": "Seoul Station"}, "endPoint": {"lat": 40.1098, "lon": 124.3953, "name": "Sinuiju Station"}, "corridor": "Gyeongui Line" }, "budget": { "total": 12000000000, "currency": "USD", "breakdown": { "construction": 8000000000, "equipment": 2500000000, "environmental": 500000000, "contingency": 1000000000 } }, "timeline": { "planningStart": "2025-01-01", "constructionStart": "2026-06-01", "targetCompletion": "2030-12-31", "phaseApproach": true }, "stakeholders": [ {"organization": "Korean Ministry of Land", "role": "primary"}, {"organization": "Korean State Railway", "role": "implementing"} ], "standards": ["WIA-UNI-005", "ERTMS-Level-2", "ISO-55000"], "certificationStatus": "pending" }

4.4 Geospatial Data Standards

Accurate geospatial representation is critical for infrastructure integration:

  • Coordinate System: WGS84 (EPSG:4326) for global compatibility
  • Precision: Minimum 6 decimal places (±0.11m accuracy)
  • Elevation: Meters above mean sea level (MSL)
  • Route Representation: GeoJSON LineString for linear assets
  • Area Boundaries: GeoJSON Polygon for zones and facilities
"geometry": { "type": "LineString", "coordinates": [ [126.9780, 37.5665, 25.7], [126.9234, 37.6543, 45.2], [126.8876, 37.7321, 67.8] ], "crs": "EPSG:4326" }

4.7 Environmental Monitoring Data

Environmental protection requires systematic monitoring and reporting:

  • Air quality measurements (PM2.5, PM10, NOx, SO2)
  • Water quality parameters (pH, dissolved oxygen, pollutants)
  • Noise levels (dBA at specified distances)
  • Wildlife observations and habitat assessments
  • Carbon emissions calculations and reporting

4.8 Validation Rules and Data Quality

All data must meet quality standards enforced through validation:

Validation Requirements

  • Schema Validation: JSON Schema validation for all documents
  • Range Checks: Values within physically possible ranges
  • Referential Integrity: Valid references to related entities
  • Temporal Consistency: Logical time ordering of events
  • Geospatial Validation: Coordinates within valid boundaries
  • Checksum Verification: Data integrity checks

4.9 Metadata and Provenance

Understanding data origin and transformation is essential for trust:

"metadata": { "createdBy": "system:asset-registry", "createdAt": "2025-12-25T10:00:00Z", "modifiedBy": "user:engineer-kim", "modifiedAt": "2025-12-25T14:30:00Z", "version": "1.2", "dataSource": "field-survey", "accuracy": "high", "confidenceLevel": 0.95 }

4.10 Extensibility and Custom Fields

While core fields are standardized, extensibility allows domain-specific additions:

  • Custom fields use namespaced prefixes (e.g., "custom:fieldName")
  • Extension schemas can be registered with WIA
  • Unknown fields are ignored by validators (forward compatibility)
  • Documentation required for all custom extensions

Chapter Summary

Key Takeaways

  • Phase 1 uses JSON-LD for semantic, interoperable data formats
  • Comprehensive schemas cover projects, assets, operations, and maintenance
  • Geospatial standards ensure accurate location representation
  • Validation rules enforce data quality and consistency
  • Metadata and provenance support data trust and lineage
