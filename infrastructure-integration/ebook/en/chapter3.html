Chapter 3

Standard Overview

3.1 WIA-UNI-005 Architecture Overview

The WIA-UNI-005 Infrastructure Integration Standard employs a four-phase architecture designed to systematically address the complexities of inter-Korean infrastructure integration. Each phase builds upon the previous, creating a comprehensive framework that spans from basic data representation to complete ecosystem integration.

PhaseFocus AreaKey DeliverablesDependencies
Phase 1Data FormatJSON schemas, validation rules, metadata standardsNone (foundation)
Phase 2API InterfaceREST APIs, SDKs, authenticationPhase 1 data formats
Phase 3ProtocolCommunication protocols, IoT integration, SCADAPhases 1-2
Phase 4IntegrationSystem connectors, international mappingPhases 1-3

3.2 Design Principles

The standard adheres to fundamental design principles that ensure long-term viability, broad adoption, and effective interoperability:

  • Open and Accessible: All specifications publicly available under open license
  • Technology Neutral: Not tied to specific vendors or platforms
  • Backward Compatible: Support for legacy systems through adapters
  • Forward Compatible: Extensibility for future technologies
  • Security by Design: Security integrated at every layer
  • Scalable: From pilot projects to peninsula-wide deployment
  • Modular: Independent implementation of phases
  • Interoperable: Works with international standards (ISO, IEC, ITU)

3.3 Phase 1: Data Format Foundation

Phase 1 establishes the fundamental data structures and formats for representing infrastructure projects, assets, and operations. Using JSON-LD for semantic interoperability, it defines schemas for:

Core Data Entities

  • Infrastructure Projects: Planning, budgeting, timeline data
  • Physical Assets: Bridges, tunnels, power stations, track sections
  • Geospatial Data: Coordinates, boundaries, elevation profiles
  • Operational Data: Traffic, power flow, sensor readings
  • Maintenance Records: Inspections, repairs, certifications
  • Environmental Data: Emissions, water quality, habitat monitoring

3.4 Phase 2: API Interface Layer

Phase 2 defines RESTful APIs that enable systems to exchange infrastructure data programmatically. It covers:

API CategoryPurposeExample Endpoints
Project ManagementCreate, update, query projects/api/v1/projects, /api/v1/projects/{id}
Asset RegistryRegister and track infrastructure assets/api/v1/assets, /api/v1/assets/{type}
Real-time MonitoringSensor data, operational status/api/v1/monitoring/sensors/{id}
MaintenanceSchedule and track maintenance/api/v1/maintenance/schedules

3.5 Phase 3: Protocol Standards

Phase 3 specifies communication protocols for real-time coordination and control across infrastructure systems. It addresses:

Protocol Domains

  1. IoT Sensor Networks: MQTT, CoAP for distributed sensors
  2. SCADA Integration: IEC 60870-5-104, DNP3 for power grids
  3. Railway Signaling: ERTMS/ETCS protocols
  4. Emergency Communications: Priority messaging, alert distribution
  5. Data Security: TLS 1.3, certificate management, encryption

3.6 Phase 4: Ecosystem Integration

Phase 4 connects WIA-UNI-005 systems with broader ecosystems including government platforms, international organizations, and other WIA standards:

Integration Targets

  • Korean Ministry of Land, Infrastructure and Transport systems
  • WIA Infrastructure Registry and Certification Platform
  • International standards (ISO 55000 Asset Management, IEC 61850)
  • Regional cooperation frameworks (UNESCAP, CAREC)
  • GIS platforms (OpenStreetMap, national mapping services)

3.8 Governance and Evolution

WIA-UNI-005 includes governance structures for maintaining and evolving the standard:

  • Technical Committee: Reviews proposals, manages versioning
  • Working Groups: Domain-specific expertise (railway, power, telecom)
  • Public Comment Process: Open feedback on proposed changes
  • Annual Review: Update cycle for incorporating improvements
  • Version Management: Semantic versioning, deprecation policies

3.9 Implementation Roadmap

Successful deployment follows a phased approach:

  1. Pilot Projects (6-12 months): Test implementations on limited scope
  2. Expansion (1-2 years): Scale to major infrastructure corridors
  3. Full Deployment (3-5 years): Peninsula-wide integration
  4. Optimization (ongoing): Continuous improvement and expansion

Chapter Summary

Key Takeaways

  • WIA-UNI-005 uses a four-phase architecture: Data Format, API, Protocol, Integration
  • Design principles emphasize openness, interoperability, and scalability
  • Formal certification ensures implementation compliance
  • Governance structures support standard evolution and maintenance
  • Integration with international standards ensures global compatibility
