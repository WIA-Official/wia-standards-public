Chapter 02: RL Fundamentals: MDPs, Value vs Policy Methods

2.1 Markov Decision Processes (MDPs)

The mathematical foundation of reinforcement learning. An MDP is a 5-tuple (S, A, P, R, γ): state space S, action space A, transition probability P(s\x27|s,a), reward function R(s,a), and discount factor γ ∈ [0,1]. The agent\x27s goal is to learn a policy π(a|s) that maximises expected discounted return. The standard\x27s environment_descriptor envelope captures all five tuple components in a vendor-neutral form.

2.2 Value-based methods

Q-learning, SARSA, DQN (Deep Q-Network). The agent learns a value function Q(s,a) representing expected return from taking action a in state s. Policy is implicit: act greedily with respect to Q. Strengths: data-efficient for discrete action spaces. Weaknesses: brittle for continuous action spaces, off-policy training instability.

2.3 Policy gradient methods

REINFORCE, PPO, TRPO. The agent directly parameterises the policy π(a|s; θ) and updates θ via gradient ascent on expected return. Strengths: handles continuous action spaces, on-policy training stable. Weaknesses: high variance, sample-inefficient.

2.4 Actor-critic methods

A2C, A3C, SAC, TD3. Combines value and policy: the actor parameterises π, the critic parameterises Q (or V). The critic reduces actor variance via advantage estimation. Currently the dominant family for continuous control.

2.5 Model-based methods

MuZero, Dreamer, PETS. The agent learns a world model P̂(s\x27|s,a) and uses it for planning (MCTS, MPC). Strengths: extreme sample efficiency. Weaknesses: model bias, computation cost.

2.6 Offline RL and imitation learning

CQL, IQL, behavioural cloning, GAIL. Trains from a fixed dataset (no environment interaction) or from expert demonstrations. Critical for domains where exploration is expensive or unsafe (clinical, industrial control).

Safety-First Design

Every choice in this standard is filtered through: does this surface enough evidence for downstream consumers to detect reward hacking and unsafe deployment, or just optimize for headline benchmark numbers? When the two conflict, safety surfacing wins. That is why the standard mandates safety_metrics on every evaluation_run, refuses to allow frozen agent_policy modification post-evaluation, and requires implementations to surface checkpoint provenance so downstream consumers can verify what model is actually deployed.

Implementer Notes

Three operational concerns recur in RL infrastructure and are baked into the standard. First, environment determinism is property of the environment + the framework + the GPU driver stack — the standard's environment_descriptor envelope captures all three so reproducibility studies can pin the exact stack. Second, checkpoint storage scales aggressively with experiment frequency; a typical research lab generates 100 GB of checkpoints per week from active hyperparameter sweeps. The standard's checkpoint envelope references model_url + sha256 rather than embedding bytes, so storage backend choices are deferred. Third, replay buffer size for offline RL — the replay_buffer_url field in the checkpoint envelope is optional but strongly recommended for offline RL workflows where buffer size can dominate disk usage.
