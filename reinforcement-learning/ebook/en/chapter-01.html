Chapter 01: Introduction & Why Open RL Standards Matter

1.1 Why this book exists

WIA-AI-025 is the open standard for reinforcement-learning data exchange — environment descriptors, training runs, evaluation runs, checkpoints, agent policies, and deployment records. This book is the practitioner introduction.

1.2 The vendor-locked RL landscape

RL research today is fragmented across vendor frameworks (Stable-Baselines3, RLlib, CleanRL, Tianshou, Acme, Dopamine), simulator backends (OpenAI Gymnasium, MuJoCo, Isaac Gym, Unity ML-Agents), and cloud orchestration platforms. A research lab using framework A cannot easily ingest checkpoints or evaluation results from framework B; reproducibility studies require manual reconstruction of the training environment.

1.3 What WIA-AI-025 changes

The standard collapses framework silos into one open wire format with explicit environment determinism, checkpoint provenance, and safety metric tracking baked into Phase 1. A Stable-Baselines3 training run and an RLlib training run can publish to the same host; downstream evaluation can compare them on identical environment seeds.

1.4 The 4-Phase architecture

Phase 1 (Data Format) defines JSON envelopes for environment descriptors, training runs, evaluation runs, checkpoints, agent policies, and deployment records. Phase 2 (API Interface) specifies the HTTP surface for publish, query, stream, and federation. Phase 3 (Federation Protocol) handles cross-organisation data flow. Phase 4 (Integration) bridges to OpenAI Gymnasium API, RLlib, Stable-Baselines3, MLflow tracking, and the WIA family.

1.5 Reward-hacking detection as a baked-in primitive

Earlier RL standards have treated agent safety evaluation as an external research concern. WIA-AI-025 treats it as a Phase 1 primitive: every evaluation_run envelope MUST include a safety_metrics block (reward variance, action distribution skew, observation-coverage entropy, declared safety constraint violations) so downstream consumers can detect reward hacking signatures without re-evaluating from scratch.

1.6 How this book is organised

Chapter 2 covers RL fundamentals. Chapter 3 introduces the standard at a glance. Chapters 4-7 walk the four phases. Chapter 8 closes with conformance, safety, and roadmap.

Safety-First Design

Every choice in this standard is filtered through: does this surface enough evidence for downstream consumers to detect reward hacking and unsafe deployment, or just optimize for headline benchmark numbers? When the two conflict, safety surfacing wins. That is why the standard mandates safety_metrics on every evaluation_run, refuses to allow frozen agent_policy modification post-evaluation, and requires implementations to surface checkpoint provenance so downstream consumers can verify what model is actually deployed.

Implementer Notes

Three operational concerns recur in RL infrastructure and are baked into the standard. First, environment determinism is property of the environment + the framework + the GPU driver stack — the standard's environment_descriptor envelope captures all three so reproducibility studies can pin the exact stack. Second, checkpoint storage scales aggressively with experiment frequency; a typical research lab generates 100 GB of checkpoints per week from active hyperparameter sweeps. The standard's checkpoint envelope references model_url + sha256 rather than embedding bytes, so storage backend choices are deferred. Third, replay buffer size for offline RL — the replay_buffer_url field in the checkpoint envelope is optional but strongly recommended for offline RL workflows where buffer size can dominate disk usage.
