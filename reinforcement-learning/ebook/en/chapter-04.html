Chapter 04: Phase 1 — Data Format

4.1 Phase 1 in one paragraph

Phase 1 specifies JSON envelopes for RL data. Every envelope is signed with Ed25519 (IETF RFC 8032).

4.2 Environment descriptor envelope

{
  "wia_reinforcement_learning_version": "1.0.0",
  "type": "environment_descriptor",
  "environment_id": "env_cartpole_v1_seed_42",
  "framework": "gymnasium",
  "framework_version": "1.0.0",
  "env_name": "CartPole-v1",
  "observation_space": { "type": "Box", "shape": [4], "low": [-4.8, -inf, -0.42, -inf], "high": [4.8, inf, 0.42, inf] },
  "action_space": { "type": "Discrete", "n": 2 },
  "discount_factor": 0.99,
  "max_episode_steps": 500,
  "seed": 42,
  "deterministic": true,
  "signature": { "alg": "Ed25519", "value": "..." }
}

4.3 Training run envelope

Required: environment_id, algorithm, algorithm_family, hyperparameters, started_at, ended_at, total_steps, total_wall_seconds, gpu_hours. Hyperparameters are framework-specific but the envelope preserves them verbatim for reproducibility.

4.4 Evaluation run envelope

Required: environment_id, agent_policy_id, episodes, return distribution (mean, std, percentiles 25/50/75/95), safety_metrics block (reward_variance, action_distribution_entropy, observation_coverage_entropy, constraint_violations).

4.5 Checkpoint envelope

Required: training_run_id, step_index, model_url, model_sha256, optimizer_state_url, replay_buffer_url (optional). The model_url + sha256 pair lets downstream consumers fetch and verify the parameters; the replay_buffer_url enables continuation training.

4.6 Agent policy envelope

Required: training_run_id, environment_id, model_url, model_sha256, evaluation_run_id, parameter_count, frozen (boolean). Frozen=true means the policy is locked for production deployment; further training would invalidate the policy_id.

4.7 Deployment record envelope

Required: agent_policy_id, target_environment_id, deployed_at, monitoring_stream_url, rollback_policy_id (optional). The monitoring_stream_url references a Phase 2 stream endpoint where post-deployment metrics flow.

Safety-First Design

Every choice in this standard is filtered through: does this surface enough evidence for downstream consumers to detect reward hacking and unsafe deployment, or just optimize for headline benchmark numbers? When the two conflict, safety surfacing wins. That is why the standard mandates safety_metrics on every evaluation_run, refuses to allow frozen agent_policy modification post-evaluation, and requires implementations to surface checkpoint provenance so downstream consumers can verify what model is actually deployed.

Implementer Notes

Three operational concerns recur in RL infrastructure and are baked into the standard. First, environment determinism is property of the environment + the framework + the GPU driver stack — the standard's environment_descriptor envelope captures all three so reproducibility studies can pin the exact stack. Second, checkpoint storage scales aggressively with experiment frequency; a typical research lab generates 100 GB of checkpoints per week from active hyperparameter sweeps. The standard's checkpoint envelope references model_url + sha256 rather than embedding bytes, so storage backend choices are deferred. Third, replay buffer size for offline RL — the replay_buffer_url field in the checkpoint envelope is optional but strongly recommended for offline RL workflows where buffer size can dominate disk usage.
