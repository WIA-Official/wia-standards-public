Chapter 2: Air Traffic Standards

Purpose

This chapter covers the Chapter 2: Air Traffic Standards area of the WIA Air Traffic Management standard. The standard defines a cross-vendor data and protocol layer for air traffic surveillance, conflict detection, flow management, and incident reporting. It is interoperable with ICAO Annex 11, EUROCAE ED-129B, and the FAA NextGen Concept of Operations.

Reference Standards (CITATION-POLICY ALLOW only)

  • ICAO Annex 11 — Air Traffic Services
  • ICAO Annex 14 — Aerodromes
  • EUROCAE ED-129B — Communications protocols for air traffic data
  • FAA NextGen Concept of Operations
  • OASIS Common Alerting Protocol (CAP) 1.2
  • OpenAPI Specification 3.1, JSON Schema 2020-12
  • IETF RFC 9700 — OAuth 2.1
  • IETF RFC 9457 — Problem Details for HTTP APIs
  • ISO/IEC 27001:2022 — Information security management
  • Sigstore (DSSE envelope, Rekor transparency log)

Implementation Procedure

Air-traffic data exchange follows a four-stage protocol: surveillance ingest, conflict detection, flow advisory, and incident reporting. Each stage produces a signed manifest anchored to a Sigstore Rekor entry so that downstream auditors can reconstruct the chain offline.

Surveillance Ingest

Surveillance data (radar, ADS-B, MLAT) is ingested at 1 Hz minimum cadence. Each frame is signed by the originating sensor and the gateway records both the sensor signature and a gateway-issued aggregate signature. The sensor identification follows ICAO 24-bit address conventions.

Conflict Detection

Conflict detection runs at the gateway with a configurable look-ahead window (default: 8 minutes for en-route, 2 minutes for terminal). Detected conflicts emit a CAP 1.2 alert in addition to the WIA-native envelope so that downstream public-warning systems can consume them without custom integration.

Flow Advisory

Flow advisories are issued by the responsible air-traffic-flow management unit. Advisories are valid for a declared window and are renewed by re-issuing the manifest signature; the prior advisory remains valid for audit purposes.

Incident Reporting

Incidents (loss of separation, runway incursion, ground collision) are reported via the protocol within 24 hours. Reports are signed by the reporting facility and forwarded to the relevant aviation authority and the WIA Standards working group for cross-vendor incident analysis.

Integration Topologies

Implementations declare their topology in the manifest:

  • Single-tenant edge — one runtime per air-navigation service provider (ANSP).
  • Multi-tenant gateway — shared runtime with ANSP-based isolation, used in regional facilities.
  • Federated mesh — multiple ANSPs peer to one another via the WIA directory service.
  • Air-gapped batch — used for certain military or sensitive routes; signed evidence packages are transported out-of-band.

Governance

Issues, errata, and proposals are tracked at github.com/WIA-Official/wia-standards/issues with the air-traffic-management label. The WIA Standards working group reviews open issues at the start of every minor release cycle and publishes the resulting decision log alongside the release notes. Errata are issued as patch releases; new normative requirements trigger minor bumps; backwards-incompatible changes trigger major bumps with a 90-day overlap window.

Reference Standards and Implementation Notes

This standard is designed to interoperate with established international frameworks. Implementations SHOULD treat the following as the single source of truth and use this PHASE document as the interoperability layer that maps existing data into a machine-readable, signed, and auditable shape.

  • OpenAPI Specification 3.1 — REST API definition
  • JSON Schema 2020-12 — schema validation
  • ISO/IEC 27001:2022 — information security management
  • ISO/IEC 27018:2019 — PII protection in public clouds
  • ISO/IEC 17065:2012 — product certification bodies
  • CycloneDX 1.5 / SPDX 2.3 — software bill of materials
  • Sigstore (DSSE envelope, Rekor transparency log) — signature anchoring
  • in-toto Attestation Framework 1.0 — build provenance
  • IETF RFC 9700 — OAuth 2.1
  • IETF RFC 9457 — Problem Details for HTTP APIs
  • IETF RFC 8615 — well-known URIs

Practical Implementation Checklist

  • Manifest integrity — every published manifest is signed via Sigstore DSSE. Tampering is detectable without trusting an intermediary. Updates are issued as new manifests; the prior manifest is preserved with a supersede marker.
  • Retention — keep signed evidence for at least seven years to satisfy ISO/IEC 17065:2012 audit expectations. Some regulators require longer retention; in that case the deployment policy MUST extend the window rather than relying on the defaults.
  • Versioning — follow Semantic Versioning 2.0.0. Major version bumps require at least a 90-day overlap with the prior major version on every WIA-published reference implementation.
  • Negative-result reporting — a skipped indicator without recorded justification is treated as a failed indicator at audit time.
  • Key rotation — manifest signing keys are rotated per the deployment policy. The previous key is preserved in a published key history so that auditors can verify historical manifests offline.

Reference Topologies

Implementations declare their operational topology in the manifest: single-tenant edge (one runtime per organization), multi-tenant gateway (shared runtime with header-based isolation), federated mesh (multiple runtimes peer to one another and publish manifests to a directory service), or air-gapped batch (signed evidence package transported out-of-band). Each topology ships with reference test vectors so that implementations can validate their deployment matches the canonical pattern.
