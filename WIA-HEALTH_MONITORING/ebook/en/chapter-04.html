Chapter 4

Continuous Health Monitoring

Learning Objectives:

Master continuous monitoring strategies including real-time data streaming, synchronization patterns, offline resilience, battery optimization, and quality assurance for health monitoring systems.

4.1 Real-Time vs Periodic Synchronization

Continuous health monitoring systems must balance data freshness, battery life, bandwidth consumption, and clinical requirements. Different metrics and use cases require different synchronization strategies.

4.1.1 Real-Time Continuous Sync

Critical for high-stakes medical scenarios where immediate data availability is essential:

  • Use Cases: Hospital ICU monitoring, post-surgical recovery, acute diabetes management with insulin pump integration
  • Metrics: ECG during cardiac events, CGM during insulin adjustments, SpO2 during respiratory distress
  • Technology: WebSocket connections, MQTT with QoS 2 (exactly once delivery)
  • Latency: Sub-5 second from measurement to cloud availability
  • Battery Impact: High (continuous radio active, frequent data transmission)
Sync Strategy Frequency Battery Impact Best For
Real-Time Continuous Immediate Very High Critical care, hospital monitoring
Periodic (5-15 min) Every 5-15 minutes Medium General wellness, fitness tracking
Opportunistic (Wi-Fi) When Wi-Fi available Low Bulk historical data, detailed sleep analysis
Manual/On-Demand User-initiated Minimal Infrequent measurements, manual entry

4.2 Offline Data Resilience

Wearable devices must function without continuous connectivity. Robust offline capabilities ensure no data loss during network outages, airplane mode, or remote locations.

4.2.1 Local Data Storage

Devices should buffer at least 7-14 days of data locally:

  • Heart Rate: Store aggregated data (average, min, max per minute) to reduce storage: ~2KB/day
  • CGM: Store all readings (288 per day at 5-minute intervals): ~10KB/day with compression
  • Activity: Store minute-by-minute summaries: ~5KB/day
  • Sleep: Store complete hypnograms: ~2KB per sleep session

4.2.2 Sync Conflict Resolution

When devices reconnect after extended offline periods, conflicts may arise if data was modified on multiple systems. Resolution strategies:

  1. Last-Write-Wins (LWW): Timestamp-based, simplest but can lose data
  2. Device Priority: Prefer device data over manual edits, higher quality scores win
  3. Merge Strategy: Combine non-conflicting changes, flag irreconcilable conflicts for user review
  4. Version Vectors: Track causality to intelligently merge concurrent updates

4.3 Battery Optimization Strategies

Battery life is often the limiting factor for continuous monitoring. Optimization requires balancing measurement frequency, computation, and communication against power consumption.

4.3.1 Sensor Management

  • Adaptive Sampling: Increase measurement frequency during detected activity, decrease during sleep
  • Sensor Fusion: Use low-power accelerometer to trigger higher-power PPG only when needed
  • Motion Gating: Skip PPG measurements during excessive motion that would yield poor quality data

4.3.2 Communication Optimization

  • Batch Uploads: Aggregate data and send in batches rather than individual transmissions
  • Compression: GZIP or Brotli compression can reduce data size by 60-80%
  • Delta Encoding: Send only changes from previous values, especially effective for slowly changing metrics
  • Wi-Fi Preferencing: Queue non-urgent data for Wi-Fi sync rather than cellular

4.4 Quality Assurance During Continuous Monitoring

With thousands of measurements per day, automated quality assurance is essential to filter artifacts and ensure clinically useful data.

4.4.1 Real-Time Quality Checks

  • Range Validation: Flag physiologically impossible values
  • Rate-of-Change Limits: Human heart rate cannot change by 50 bpm in one second
  • Signal Quality Metrics: PPG signal-to-noise ratio, motion artifact detection
  • Sensor Contact Detection: Verify device is properly worn before trusting readings

4.4.2 Post-Processing Quality Enhancement

After data collection, additional algorithms can improve data quality:

  • Outlier Detection: Statistical methods (z-score, IQR) to identify anomalous values
  • Smoothing Filters: Remove noise while preserving true signal (moving average, Kalman filters)
  • Gap Filling: Interpolate short gaps (<5 minutes) when appropriate
  • Artifact Removal: Machine learning models trained to recognize and remove motion artifacts

4.5 Handling Data Gaps

Despite best efforts, gaps in continuous monitoring data are inevitable due to device removal (charging, showering), connectivity issues, or device failures.

Gap Duration Likely Cause Handling Strategy
< 5 minutes Brief device removal, temporary connectivity loss Linear interpolation acceptable for most metrics
5-30 minutes Charging, shower, sensor warmup Mark as missing data, do not interpolate
30 min - 2 hours Extended charging, device off Treat as significant gap, may impact daily summaries
> 2 hours Intentional device non-wear, device failure Consider entire time period invalid for certain analyses

4.6 Time Zone and Daylight Saving Time

Continuous monitoring across time zones requires careful timestamp management to maintain accurate temporal relationships.

4.6.1 Best Practices

  • Store all data in UTC internally
  • Record device's local timezone for each measurement session
  • Handle DST transitions by preserving actual time-of-day for circadian analysis
  • Display data in user's current or preferred timezone
  • Mark time zone transitions explicitly in data streams
DST Transition Example: During "spring forward," the hour from 2:00 AM to 3:00 AM doesn't exist. During "fall back," 1:00 AM to 2:00 AM occurs twice. Systems must handle these edge cases to avoid duplicate or missing data.

4.7 Clinical Context Awareness

Effective continuous monitoring considers the user's current context—activity, sleep, meals, medications—to provide appropriate interpretations and alerts.

4.7.1 Activity Context

  • Resting: Lower heart rate thresholds, more sensitive to arrhythmia detection
  • Exercising: Elevated heart rate is normal, focus on recovery patterns
  • Sleeping: Different normal ranges for all metrics, sleep stage detection enabled
  • Eating: Expect glucose to rise, but rate and peak matter

4.7.2 Automated Context Detection

Systems can infer context from sensor data without explicit user input:

  • Sleep detected from extended periods of low motion + characteristic position changes
  • Exercise detected from sustained elevated heart rate + high accelerometer activity
  • Meals inferred from postprandial glucose rise patterns
  • Stress episodes detected from low HRV + elevated resting heart rate

Key Takeaways

  • Different monitoring scenarios require different sync strategies: real-time for critical care, periodic for wellness, opportunistic for bulk data
  • Devices must buffer 7-14 days of data locally to ensure resilience during connectivity outages
  • Battery optimization balances measurement frequency, computation, and communication against power consumption
  • Real-time quality checks and post-processing algorithms ensure clinically useful data
  • Proper timestamp handling with timezone information is critical for accurate temporal analysis
  • Context-aware systems adjust interpretations and alerts based on activity, sleep, meals, and other factors
