Chapter 2

Data Structures & Formats

Learning Objectives:

Master the standardized data formats for health metrics, understand validation rules, quality scoring mechanisms, and learn how to implement robust data structures that ensure interoperability across health monitoring systems.

2.1 Base Health Metric Structure

All health metrics in the WIA-HEALTH_MONITORING standard inherit from a common base structure that ensures consistency, traceability, and quality assurance across all metric types. This foundational design enables seamless integration with diverse systems while maintaining data integrity.

{ "metric_id": "uuid-v4", "metric_type": "HEART_RATE | BLOOD_GLUCOSE | SPO2 | ...", "timestamp": "ISO8601 datetime with timezone", "device_id": "unique device identifier", "user_id": "anonymized user identifier", "value": "metric-specific data structure", "unit": "standard unit of measurement", "quality_score": 0.0-1.0, "metadata": { "device_model": "string", "firmware_version": "string", "sensor_position": "WRIST | CHEST | FINGER | ARM | EAR", "activity_context": "RESTING | ACTIVE | SLEEPING | EXERCISING", "confidence_interval": [lower, upper] } }

2.1.1 Required Fields

Field Type Description Validation
metric_id String (UUID v4) Globally unique identifier for the metric Must be valid UUID v4 format
metric_type Enum Type of health metric being recorded Must be from predefined list
timestamp ISO 8601 DateTime When the measurement was taken Must include timezone offset
device_id String Identifier of the measuring device Non-empty, max 255 characters
user_id String Anonymized user identifier Non-empty, no PII allowed
value Object Metric-specific measurement data Must conform to metric type schema
unit String Unit of measurement (UCUM standard) Must be valid UCUM code
quality_score Float (0.0-1.0) Data quality indicator 0.0 ≤ value ≤ 1.0

2.2 Heart Rate Data Format

Heart rate monitoring is one of the most ubiquitous forms of health tracking. The standard defines comprehensive fields for capturing not just the current heart rate, but also variability, trends, and contextual information critical for accurate interpretation.

{ "metric_type": "HEART_RATE", "value": { "bpm": 72, "variability_ms": 45.3, "resting_hr": 65, "max_hr_24h": 145, "min_hr_24h": 58, "irregular_rhythm_detected": false, "raw_intervals_ms": [850, 840, 855, 845, 860] }, "unit": "beats_per_minute", "quality_score": 0.98, "metadata": { "measurement_duration_sec": 60, "ppg_signal_quality": "EXCELLENT", "motion_artifact_level": 0.02 } }

2.2.1 Heart Rate Variability (HRV)

HRV measures the variation in time intervals between consecutive heartbeats, providing insights into autonomic nervous system function, stress levels, and overall cardiovascular health. Higher HRV generally indicates better cardiovascular fitness and stress resilience.

The raw_intervals_ms array captures the R-R intervals (time between successive heartbeat peaks) measured in milliseconds. This raw data enables advanced HRV analysis including RMSSD (root mean square of successive differences), SDNN (standard deviation of NN intervals), and frequency domain analysis.

2.3 Continuous Glucose Monitoring Format

CGM systems revolutionized diabetes management by providing real-time glucose readings every 1-5 minutes. The data format captures not just the current value but trends, predictions, and time-in-range metrics.

{ "metric_type": "BLOOD_GLUCOSE", "value": { "glucose_mg_dl": 105, "glucose_mmol_l": 5.83, "trend_direction": "STABLE", "trend_rate_mg_dl_min": 0.5, "time_in_range_percent": 78.5, "prediction_30min": { "predicted_value": 110, "confidence": 0.85 }, "sensor_status": "ACTIVE" }, "unit": "mg/dL", "quality_score": 0.95, "metadata": { "sensor_age_days": 7, "calibration_status": "CALIBRATED", "alert_thresholds": { "low": 70, "high": 180 } } }

2.3.1 Glucose Trend Directions

Direction Rate of Change Clinical Significance Recommended Action
RISING_RAPIDLY > +2 mg/dL/min Rapid glucose increase Check carb intake, consider insulin
RISING +1 to +2 mg/dL/min Moderate increase Monitor closely
STABLE -1 to +1 mg/dL/min Relatively stable Continue current management
FALLING -2 to -1 mg/dL/min Moderate decrease Prepare fast-acting carbs if trending toward low
FALLING_RAPIDLY < -2 mg/dL/min Rapid glucose decrease Immediate action needed to prevent hypoglycemia

2.4 Blood Oxygen (SpO2) Format

Pulse oximetry measures oxygen saturation non-invasively using light absorption properties of oxygenated and deoxygenated hemoglobin. The format includes perfusion index and pleth variability for quality assessment.

{ "metric_type": "SPO2", "value": { "oxygen_saturation_percent": 98, "perfusion_index": 4.5, "pulse_rate": 72, "pleth_variability_index": 12.5 }, "unit": "percent", "quality_score": 0.97, "metadata": { "measurement_method": "CONTINUOUS", "ambient_light_level": "LOW", "motion_artifact_detected": false, "sensor_placement": "FINGER" } }

2.5 Sleep Tracking Format

Sleep data requires complex temporal structures to represent multiple sleep stages, transitions, and quality metrics. The format supports both automatic and manually-entered sleep data.

{ "metric_type": "SLEEP", "value": { "sleep_start": "2026-01-12T22:30:00-05:00", "sleep_end": "2026-01-13T06:06:00-05:00", "total_sleep_minutes": 456, "sleep_stages": [ { "stage": "LIGHT", "start_time": "2026-01-12T22:30:00-05:00", "duration_minutes": 85 }, { "stage": "DEEP", "start_time": "2026-01-12T23:55:00-05:00", "duration_minutes": 95 }, { "stage": "REM", "start_time": "2026-01-13T01:30:00-05:00", "duration_minutes": 110 } ], "sleep_efficiency_percent": 92.5, "wake_episodes": 2, "restlessness_score": 15, "sleep_score": 85 } }

2.6 Data Validation Rules

2.6.1 Range Validation

All numeric health metrics must fall within physiologically plausible ranges. Values outside these ranges should be flagged for review and potentially rejected.

Metric Minimum Value Maximum Value Typical Range
Heart Rate 30 bpm 220 bpm 60-100 bpm (resting adult)
Blood Glucose 20 mg/dL 600 mg/dL 70-180 mg/dL
SpO2 70% 100% 95-100%
Systolic BP 60 mmHg 250 mmHg 90-120 mmHg
Diastolic BP 40 mmHg 150 mmHg 60-80 mmHg
Body Temperature 32°C (89.6°F) 42°C (107.6°F) 36.5-37.5°C (97.7-99.5°F)

2.6.2 Relationship Validation

Certain metrics have intrinsic relationships that must be validated:

  • Diastolic blood pressure must be less than systolic blood pressure
  • Sleep end time must be after sleep start time
  • Pulse rate from SpO2 sensor should correlate with heart rate (within ±10%)
  • Total sleep time should not exceed time between sleep_start and sleep_end
  • Sum of sleep stage durations should approximately equal total sleep time
  • Activity calories should align with intensity and duration

2.7 Quality Score Calculation

Quality scores (0.0-1.0) indicate the reliability of a measurement. This multi-factor assessment helps downstream systems appropriately weigh data in analysis and clinical decision-making.

Factor Weight Description Scoring Criteria
Signal Strength 30% Quality of sensor signal 1.0 = excellent signal, 0.0 = poor/noisy signal
Motion Artifact 25% Movement interference 1.0 = no motion, 0.0 = severe motion artifacts
Sensor Calibration 20% Calibration status 1.0 = recently calibrated, decreases with age
Environmental Factors 15% Ambient conditions 1.0 = ideal conditions, lower for adverse conditions
Device Battery 10% Power level 1.0 = full battery, 0.5 = low battery warning threshold
Example Calculation: A heart rate measurement with excellent signal (0.95), minor motion (0.80), good calibration (0.90), ideal environment (1.0), and adequate battery (0.85) would score: (0.95×0.30) + (0.80×0.25) + (0.90×0.20) + (1.0×0.15) + (0.85×0.10) = 0.90

2.8 Units and Standardization

The standard uses UCUM (Unified Code for Units of Measure) for all unit specifications, ensuring global interoperability and unambiguous interpretation.

2.8.1 Common Unit Conversions

// Glucose mg_dl = mmol_l × 18.018 mmol_l = mg_dl / 18.018 // Temperature celsius = (fahrenheit - 32) × 5/9 fahrenheit = celsius × 9/5 + 32 // Distance kilometers = miles × 1.60934 miles = kilometers × 0.621371 // Weight kilograms = pounds × 0.453592 pounds = kilograms × 2.20462

2.9 Timestamp Handling

Proper timestamp management is critical for health monitoring, especially when data is collected across time zones or during daylight saving time transitions.

2.9.1 ISO 8601 Format Requirements

  • All timestamps MUST include timezone offset or Z for UTC
  • Millisecond precision is RECOMMENDED for high-frequency measurements
  • Format: YYYY-MM-DDTHH:mm:ss.sss±HH:mm
  • Example: 2026-01-12T14:35:22.123-05:00
Critical: Always store data in UTC internally and convert to local time only for display. Include the original device timezone in metadata for accurate reconstruction of the user's context.

2.10 Batch Upload Format

For efficient data transmission, especially with offline-capable devices, the standard supports batch uploads with compression and integrity verification.

{ "batch_id": "uuid-v4", "device_id": "device-identifier", "user_id": "user-identifier", "upload_timestamp": "2026-01-12T14:35:22.123+00:00", "data_points": [ { /* metric object 1 */ }, { /* metric object 2 */ }, { /* metric object 3 */ } ], "compression": "GZIP", "checksum": "SHA256 hash of uncompressed data", "total_count": 1250 }

Key Takeaways

  • All health metrics inherit from a common base structure ensuring consistency and interoperability across systems
  • Each metric type has specific fields optimized for clinical relevance and technical accuracy
  • Quality scores provide a standardized way to assess measurement reliability across diverse devices and conditions
  • Comprehensive validation rules prevent invalid data from entering systems while allowing physiologically extreme but valid values
  • ISO 8601 timestamps with timezone information ensure accurate temporal interpretation across global deployments
  • UCUM standard units enable unambiguous measurement interpretation and seamless unit conversions
  • Batch upload formats with compression and checksums optimize bandwidth while ensuring data integrity
