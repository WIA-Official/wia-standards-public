Chapter 8

Privacy, Security & Compliance

Learning Objectives:

Navigate healthcare data privacy and security requirements including HIPAA, GDPR, encryption standards, access controls, audit logging, and regulatory compliance for medical devices.

8.1 Healthcare Privacy Regulations

Health data is among the most sensitive personal information, requiring stringent privacy protections. Multiple regulations govern health monitoring systems depending on jurisdiction and use case.

8.1.1 HIPAA (United States)

The Health Insurance Portability and Accountability Act regulates Protected Health Information (PHI) in the United States.

HIPAA Component Requirement Implementation for Health Monitoring
Privacy Rule Limit use and disclosure of PHI Implement access controls, audit logging, minimum necessary principle
Security Rule Administrative, physical, and technical safeguards Encryption, authentication, workstation security, data backup
Breach Notification Rule Notify individuals of breaches Incident response plan, breach detection systems, notification procedures
Enforcement Rule Penalties for non-compliance Regular compliance audits, staff training, documentation
Important: Not all health monitoring apps are HIPAA covered entities. Apps marketed directly to consumers without healthcare provider involvement may not fall under HIPAA. However, when integrating with EMR systems or used in clinical care, HIPAA compliance becomes mandatory.

8.1.2 GDPR (European Union)

The General Data Protection Regulation applies to health data of EU residents, with stricter requirements than HIPAA in many areas.

  • Legal Basis for Processing: Must have explicit consent or other legal basis for processing health data
  • Right to Access: Individuals can request all their data in machine-readable format
  • Right to Erasure: "Right to be forgotten" - delete user data upon request
  • Data Portability: Export data in standard formats for transfer to other services
  • Privacy by Design: Build privacy protections into system architecture from the start
  • Data Protection Officer: Designate DPO for organizations processing large amounts of health data

8.2 Encryption Standards

Encryption protects data confidentiality in transit and at rest. Health monitoring systems must implement strong encryption at multiple layers.

8.2.1 Data in Transit

Protocol Minimum Version Use Case
TLS 1.3 (1.2 minimum) HTTPS API communications, WebSocket connections
Bluetooth BLE 4.2+ with encryption Device-to-phone communication
Wi-Fi WPA3 (WPA2 minimum) Device-to-router connections

8.2.2 Data at Rest

  • Database Encryption: AES-256 encryption for database files
  • File System Encryption: Encrypted volumes for stored backups
  • Mobile Device Encryption: Use device's built-in encryption (iOS Data Protection, Android File-Based Encryption)
  • Backup Encryption: Encrypt backups before transmission to cloud storage

8.2.3 End-to-End Encryption

For particularly sensitive data (CGM readings used for insulin dosing, mental health data), consider end-to-end encryption where data is encrypted on the device and only decrypted by authorized recipients:

// Simplified E2E encryption flow 1. User device generates key pair (public/private) 2. Public key stored on server, private key never leaves device 3. Data encrypted with recipient's public key before upload 4. Only recipient's device (with private key) can decrypt 5. Server cannot access plaintext data

8.3 Authentication and Authorization

Proper authentication ensures only authorized users access health data. Authorization controls what actions authenticated users can perform.

8.3.1 Authentication Methods

Method Security Level User Experience Recommendation
Password Low-Medium Familiar Require strong passwords (8+ chars, mixed case, numbers, symbols)
Biometric (Face/Touch ID) High Convenient Preferred for mobile apps, backed by device secure enclave
SMS/Email OTP Medium Additional friction Two-factor authentication for sensitive operations
Authenticator App (TOTP) High Moderate friction Recommended 2FA method for healthcare providers
Hardware Security Key Very High Requires hardware For administrative access, HIPAA-required systems

8.3.2 Role-Based Access Control (RBAC)

Define roles with specific permissions:

  • Patient: Full access to own data, can share with providers
  • Family Member/Caregiver: Read-only access to shared metrics, alert notifications
  • Healthcare Provider: Read access to patient data (with consent), can add clinical notes
  • Care Coordinator: Access to multiple patients, aggregate views, care plan management
  • Researcher: Access to de-identified, aggregated data only
  • Administrator: System configuration, user management (no access to patient data unless also provider)

8.4 Audit Logging

Comprehensive audit logs are required for HIPAA compliance and essential for security monitoring and incident response.

8.4.1 What to Log

  • Authentication events: Logins, logouts, failed login attempts
  • Data access: Who viewed which patient records, when
  • Data modifications: Creation, updates, deletions of health data
  • Permission changes: Access grants/revocations, role changes
  • Export operations: Data exports, report generation
  • System events: Configuration changes, backups, failures

8.4.2 Audit Log Format

{ "timestamp": "2026-01-12T14:30:00.123Z", "event_type": "DATA_ACCESS", "user_id": "provider-123", "user_role": "PHYSICIAN", "action": "VIEW_PATIENT_GLUCOSE_DATA", "resource_type": "OBSERVATION", "patient_id": "patient-456", "ip_address": "192.168.1.100", "user_agent": "Mozilla/5.0...", "result": "SUCCESS", "records_accessed": 288 }

8.4.3 Audit Log Protection

Audit logs themselves must be protected from tampering:

  • Write-only access for application, separate read access for auditors
  • Cryptographic signatures to detect modifications
  • Immutable storage (append-only, cannot delete or modify past entries)
  • Regular exports to secure, offline storage
  • Retention for minimum 6 years (HIPAA requirement)

8.5 Data Anonymization and De-identification

For research, quality improvement, and analytics, health data should be de-identified to protect privacy while enabling secondary use.

8.5.1 HIPAA Safe Harbor Method

Remove or generalize 18 specific identifiers:

  1. Names
  2. Geographic subdivisions smaller than state (except first 3 digits of zip code if region >20,000 people)
  3. Dates (except year) - birth dates, admission dates, etc.
  4. Telephone numbers
  5. Fax numbers
  6. Email addresses
  7. Social Security numbers
  8. Medical record numbers
  9. Health plan beneficiary numbers
  10. Account numbers
  11. Certificate/license numbers
  12. Vehicle identifiers and serial numbers
  13. Device identifiers and serial numbers
  14. Web URLs
  15. IP addresses
  16. Biometric identifiers (fingerprints, retinal scans)
  17. Full-face photographs
  18. Any other unique identifying number, characteristic, or code

8.5.2 K-Anonymity

Ensure that each individual is indistinguishable from at least k-1 other individuals in the dataset:

Example with k=3: Age, Gender, Zip Code, Diagnosis 30-40, F, 021**, Diabetes 30-40, F, 021**, Hypertension 30-40, F, 021**, Diabetes Each combination of quasi-identifiers (age range, gender, zip prefix) appears at least 3 times

8.6 Consent Management

Users must provide informed consent for data collection, use, and sharing. Consent management systems track permissions at a granular level.

8.6.1 Consent Types

Consent Type Scope Revocable
Data Collection Permission to collect health data from devices Yes - user can stop at any time
Clinical Use Share data with healthcare providers Yes - user controls which providers
Research De-identified data for research studies Yes - with exceptions for ongoing studies
Marketing Use data for product recommendations Yes - easily revocable
Third-Party Sharing Share with partner applications Yes - with per-partner granularity

8.7 Security Incident Response

Despite best efforts, security incidents may occur. Prepared incident response minimizes damage.

8.7.1 Incident Response Plan

  1. Detection: Automated monitoring, user reports, security audits
  2. Containment: Isolate affected systems, revoke compromised credentials
  3. Investigation: Determine scope, affected data, attack vector
  4. Eradication: Remove attacker access, patch vulnerabilities
  5. Recovery: Restore systems from clean backups, verify integrity
  6. Notification: Notify affected users (HIPAA: within 60 days), regulators, law enforcement if appropriate
  7. Post-Incident Review: Lessons learned, improve security controls

8.8 Medical Device Regulations

Health monitoring devices may be regulated as medical devices, requiring compliance with FDA (US), CE marking (EU), or other regulatory frameworks.

8.8.1 FDA Classification

Class Risk Level Examples Requirements
Class I Low risk Elastic bandages, exam gloves General controls
Class II Moderate risk Blood pressure cuffs, CGMs, pulse oximeters 510(k) premarket notification, special controls
Class III High risk Implantable pacemakers, heart valves Premarket Approval (PMA), clinical trials
Note: Many consumer wearables are classified as "wellness devices" and not regulated as medical devices. However, if the device makes medical claims (e.g., "detects AFib"), it may require FDA clearance. Similarly, if integrated with a clinical workflow for diagnosis or treatment decisions, regulatory requirements apply.

Key Takeaways

  • HIPAA and GDPR impose strict requirements on health data handling, including encryption, access controls, and breach notification
  • Data encryption must protect information in transit (TLS 1.3), at rest (AES-256), and consider end-to-end encryption for sensitive data
  • Strong authentication including biometrics and two-factor authentication, combined with role-based access control, protects data access
  • Comprehensive audit logging of all data access and modifications is required for HIPAA compliance and security monitoring
  • De-identification using HIPAA Safe Harbor or k-anonymity enables research and analytics while protecting privacy
  • Granular consent management gives users control over data collection, clinical use, research participation, and third-party sharing
  • Medical device regulations (FDA, CE marking) apply when devices make medical claims or integrate with clinical workflows
