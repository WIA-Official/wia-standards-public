Chapter 6

Alerts & Notifications

Learning Objectives:

Design effective alert systems for critical health events, understand alert prioritization, manage notification channels, configure thresholds, and prevent alert fatigue while ensuring patient safety.

6.1 Alert Priority Levels

Not all health alerts are equally urgent. A well-designed alert system uses priority levels to ensure critical alerts get immediate attention while less urgent notifications don't cause alarm fatigue.

Priority Urgency User Action Required Example
CRITICAL Immediate Act within minutes Severe hypoglycemia (<50 mg/dL), SpO2 <85%, suspected cardiac arrest
HIGH Urgent Act within 1 hour Mild hypoglycemia (50-70 mg/dL), prolonged tachycardia (>140 bpm resting), AFib detection
MEDIUM Important Review same day Hyperglycemia (>250 mg/dL), missed medication reminder, low activity warning
LOW Informational Review when convenient Daily summary available, weekly goal achieved, sensor needs replacement soon

6.1.1 Dynamic Priority Adjustment

Alert priority can adjust based on context:

  • Glucose of 65 mg/dL at 3 AM (sleeping) → CRITICAL (harder to wake up and treat)
  • Glucose of 65 mg/dL at 10 AM (awake) → HIGH (easily treatable)
  • Heart rate of 160 bpm during exercise → LOW (expected)
  • Heart rate of 160 bpm at rest → CRITICAL (abnormal)

6.2 Notification Channels

Different alert priorities warrant different notification mechanisms. Multi-channel delivery ensures critical alerts are received even if one channel fails.

6.2.1 Channel Selection by Priority

Channel Latency User Disruption Best Priority Level
Alarm Sound Immediate Very High CRITICAL only
Vibration Immediate High CRITICAL, HIGH
Push Notification Seconds Medium HIGH, MEDIUM
SMS Seconds to minutes Medium HIGH (backup), emergency contacts
Email Minutes to hours Low MEDIUM, LOW, daily summaries
In-App Badge Hours Minimal LOW, informational

6.2.2 Escalation Policies

If a critical alert is not acknowledged, escalate to ensure it's received:

Escalation for CRITICAL glucose alert: 1. Device vibration + alarm (immediate) 2. If not acknowledged in 5 min → Push notification 3. If not acknowledged in 10 min → SMS to user 4. If not acknowledged in 15 min → SMS to emergency contact 5. If not acknowledged in 20 min → Call emergency contact

6.3 Alert Thresholds

Thresholds define the boundary between normal and alertable conditions. They should be evidence-based, personalized, and adjustable.

6.3.1 Default Clinical Thresholds

Metric Low Alert High Alert Rationale
Blood Glucose <70 mg/dL >250 mg/dL ADA guidelines for diabetes management
SpO2 <90% N/A Clinical hypoxemia definition
Resting Heart Rate <40 bpm >120 bpm Bradycardia/tachycardia thresholds
Systolic BP <90 mmHg >180 mmHg Hypotension/hypertensive crisis

6.3.2 Personalized Thresholds

After establishing individual baselines, thresholds can be personalized:

  • Statistical approach: Alert when metric exceeds 2 standard deviations from personal mean
  • Physician-directed: Healthcare provider sets custom thresholds based on patient history
  • Adaptive learning: System learns from user responses to auto-adjust thresholds

6.4 Alert Fatigue Prevention

Alert fatigue occurs when users receive so many alerts that they begin ignoring them, potentially missing truly critical warnings. Prevention strategies are essential.

6.4.1 Intelligent Suppression

  • Hysteresis: Require metric to improve significantly before clearing alert (prevent alert flapping)
  • Rate limiting: Maximum of N alerts per hour for non-critical issues
  • Duplicate suppression: Don't re-alert for same condition within X minutes
  • Sleep mode: Adjust thresholds during sleep hours (unless critical)

6.4.2 Alert Aggregation

Combine multiple related alerts into a single notification:

Instead of: - "Glucose 185 mg/dL (High)" - "Glucose 195 mg/dL (High)" - "Glucose 205 mg/dL (High)" Aggregate to: - "Glucose elevated for 45 minutes, currently 205 mg/dL"

6.4.3 User Feedback Loop

Learn from user actions to improve alert relevance:

  • Track alert acknowledgment times
  • Monitor "snooze" vs "dismiss" patterns
  • Survey users about false positive alerts
  • Adjust thresholds based on ignored alerts

6.5 Alert Rules Configuration

Systems should support flexible alert rule configuration for different use cases and user preferences.

6.5.1 Rule Components

{ "rule_id": "uuid", "name": "Severe Hypoglycemia Alert", "metric_type": "BLOOD_GLUCOSE", "condition": { "operator": "LESS_THAN", "threshold": 55, "unit": "mg/dL" }, "duration": { "minutes": 0, "mode": "IMMEDIATE" }, "severity": "CRITICAL", "channels": ["ALARM", "VIBRATION", "PUSH", "SMS"], "active_hours": "00:00-23:59", "active_days": ["MON","TUE","WED","THU","FRI","SAT","SUN"], "snooze_duration_minutes": 5, "max_alerts_per_day": null }

6.5.2 Complex Alert Rules

Support compound conditions for sophisticated alerting:

  • Trend-based: Alert if glucose falling rapidly (>3 mg/dL/min) even if still in range
  • Multi-metric: Alert if heart rate >150 bpm AND SpO2 <92% simultaneously
  • Time-based: Alert if glucose >180 mg/dL for 3+ hours (sustained hyperglycemia)
  • Pattern-based: Alert if 3+ episodes of hypoglycemia in 24 hours

6.6 Emergency Contact Notifications

For critical alerts that go unacknowledged, automatically notify designated emergency contacts.

6.6.1 Emergency Contact Configuration

  • Primary contact (spouse, parent, adult child)
  • Secondary contact (backup if primary unreachable)
  • Healthcare provider (optional, for clinical team notification)
  • Contact preferences (SMS, call, both)
  • Escalation timing (when to notify each contact)

6.6.2 Privacy and Consent

Important: Sharing health alerts with third parties requires explicit consent. Users must actively opt-in and understand what information will be shared. Provide granular control over what metrics and alert types are shared with each contact.

6.7 Alert Performance Metrics

Monitor alert system effectiveness to identify and fix issues:

Metric Definition Target
True Positive Rate Legitimate alerts correctly triggered >95%
False Positive Rate Alerts for non-issues <5%
Mean Time to Acknowledge How quickly users respond to alerts <2 minutes for CRITICAL
Alert Dismissal Rate % of alerts dismissed without action <20%
Delivery Success Rate % of alerts successfully delivered >99.9%

Key Takeaways

  • Alert priority levels (CRITICAL, HIGH, MEDIUM, LOW) ensure urgent alerts get immediate attention while preventing alert fatigue
  • Multi-channel notification delivery (alarm, vibration, push, SMS, email) with escalation policies ensures critical alerts are received
  • Thresholds should be evidence-based, personalized, and adjustable based on individual baselines and clinical guidance
  • Alert fatigue prevention requires intelligent suppression, rate limiting, aggregation, and user feedback loops
  • Complex alert rules support trend-based, multi-metric, time-based, and pattern-based conditions
  • Emergency contact notifications with proper consent enable third-party awareness of critical health events
  • Monitoring alert performance metrics identifies false positives and system improvements needed
