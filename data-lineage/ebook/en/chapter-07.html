OpenLineage Standard

OpenLineage is an open standard for lineage metadata collection, designed to solve a critical problem: vendor lock-in and lack of interoperability in data lineage tools.

The Interoperability Problem

Before OpenLineage, each data tool maintained its own proprietary lineage format:

  • Airflow tracked job lineage in its metadata database
  • dbt captured model lineage in its manifest files
  • Spark maintained execution graphs internally
  • BI tools tracked report dependencies separately

Result: Fragmented lineage with no unified view across the data stack.

OpenLineage Solution: A common specification that any tool can emit and any lineage backend can consume, enabling end-to-end lineage across heterogeneous systems.

OpenLineage Architecture

Core Components

  1. Event Specification: JSON schema for lineage events
  2. Client Libraries: SDKs for emitting events (Python, Java, etc.)
  3. Integrations: Pre-built emitters for popular tools
  4. Backend: System that collects and stores lineage

OpenLineage Event Model

Event Structure

Every OpenLineage event contains:

  • Event Type: START, COMPLETE, FAIL, ABORT
  • Event Time: When the event occurred
  • Run: Unique execution instance
  • Job: The process or transformation
  • Inputs: Source datasets
  • Outputs: Destination datasets

Facets: Extensible Metadata

Facets are the key to OpenLineage's flexibility. They allow adding custom metadata without breaking the core specification.

Standard Facets

Run Facets

  • nominalTime: Scheduled execution time
  • parent: Parent job for nested workflows
  • errorMessage: Failure details

Job Facets

  • documentation: Job description
  • sourceCode: Transformation logic
  • sql: SQL query text
  • ownership: Team responsible

Dataset Facets

  • schema: Column definitions
  • dataSource: Connection details
  • lifecycleStateChange: CREATE, ALTER, DROP events
  • dataQuality: Metrics and validation results
  • columnLineage: Column-level dependencies

Custom Facets

Organizations can define their own facets:

Integration Examples

Python Client

Emitting OpenLineage Events in Python
from openlineage.client import OpenLineageClient
from openlineage.client.run import RunEvent, RunState, Run, Job
from openlineage.client.facet import SqlJobFacet, SchemaDatasetFacet, SchemaField
import uuid
from datetime import datetime

# Initialize client
client = OpenLineageClient(url="http://lineage-backend:5000")

# Create unique run ID
run_id = str(uuid.uuid4())

# START event
client.emit(
    RunEvent(
        eventType=RunState.START,
        eventTime=datetime.now().isoformat(),
        run=Run(runId=run_id),
        job=Job(
            namespace="my-data-pipeline",
            name="transform-customer-data",
            facets={
                "sql": SqlJobFacet(
                    query="SELECT * FROM raw_customers WHERE active = true"
                )
            }
        ),
        inputs=[
            Dataset(
                namespace="postgres://db.company.com:5432",
                name="public.raw_customers",
                facets={
                    "schema": SchemaDatasetFacet(
                        fields=[
                            SchemaField(name="id", type="INTEGER"),
                            SchemaField(name="email", type="VARCHAR"),
                            SchemaField(name="active", type="BOOLEAN")
                        ]
                    )
                }
            )
        ],
        outputs=[
            Dataset(
                namespace="postgres://db.company.com:5432",
                name="public.active_customers"
            )
        ]
    )
)

# ... perform transformation ...

# COMPLETE event
client.emit(
    RunEvent(
        eventType=RunState.COMPLETE,
        eventTime=datetime.now().isoformat(),
        run=Run(runId=run_id),
        job=Job(namespace="my-data-pipeline", name="transform-customer-data"),
        inputs=[...],
        outputs=[...]
    )
)

Benefits of OpenLineage

  1. Vendor Neutrality: Not locked into proprietary format
  2. Ecosystem Growth: More tools adding support
  3. Best Practices: Standardized approach to lineage
  4. Community Support: Active open-source community
  5. Future-Proof: Adapts as data stack evolves

Adoption Strategy

  1. Start Small: One pipeline or tool
  2. Choose Backend: Marquez or commercial option
  3. Instrument Jobs: Add OpenLineage emitters
  4. Validate Events: Ensure quality metadata
  5. Expand Coverage: Add more integrations
  6. Build on Top: Impact analysis, visualization, etc.
Key Takeaway: OpenLineage transforms lineage from a proprietary afterthought into a first-class, interoperable data product. It's the TCP/IP of data lineage.
