4.1 프로토콜 아키텍처

WIA-AIR-SHIELD는 모든 무선 통신 계층에 걸쳐 포괄적인 보호를 제공하는 다중 계층 프로토콜 스택을 구현합니다.

┌─────────────────────────────────────────────────────────┐
│                    애플리케이션                          │
├─────────────────────────────────────────────────────────┤
│  ┌─────────────────────────────────────────────────┐   │
│  │            WIA-AIR-SHIELD                        │   │
│  │  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌───────┐ │   │
│  │  │ 은폐    │ │ 교란    │ │ 검증    │ │ 경보  │ │   │
│  │  │ 계층    │ │ 계층    │ │ 계층    │ │ 계층  │ │   │
│  │  └─────────┘ └─────────┘ └─────────┘ └───────┘ │   │
│  └─────────────────────────────────────────────────┘   │
├─────────────────────────────────────────────────────────┤
│                 무선 매체                               │
│        (RF / WiFi / Bluetooth / 전력 전송)             │
└─────────────────────────────────────────────────────────┘

4.2.2 트래픽 패턴 마스킹

프로토콜은 타이밍 공격 및 트래픽 분석을 방지하기 위해 트래픽 패턴을 정규화합니다:

  • 패킷 크기 정규화: 모든 패킷을 표준 MTU 크기로 패딩
  • 타이밍 정규화: 랜덤 지터를 사용하여 일정한 간격으로 전송
  • 버스트 평활화: 폭발적인 트래픽을 일정한 속도 전송으로 평활화
  • 패턴 깨기: 예측 가능한 패턴을 의도적으로 방해

4.3.2 타이밍 랜덤화

// 타이밍 랜덤화 프로토콜
function sendWithRandomTiming(packet, config) {
  // 기본 간격
  const baseInterval = config.targetInterval;

  // 랜덤 지터
  const jitter = randomRange(
    -config.jitterMs,
    config.jitterMs
  );

  // 전송 예약
  const delay = baseInterval + jitter;
  scheduleTransmission(packet, delay);
}

// 예제 설정
{
  "targetInterval": 100, // ms
  "jitterMs": 50,        // ±50ms 랜덤 변동
  "distribution": "uniform" // 또는 "gaussian", "exponential"
}

4.4 검증 계층 프로토콜

검증 계층은 모든 무선 연결이 인증되고 합법적임을 보장합니다.

4.4.1 송신기 인증 프로토콜

// 3단계 인증 프로토콜
1단계: 챌린지-응답
┌──────────┐                    ┌──────────┐
│  기기    │                    │송신기    │
└──────────┘                    └──────────┘
     │                               │
     │─────── 인증 요청 ─────────────>│
     │                               │
     │<────── 챌린지 ─────────────────│
     │       (랜덤 논스)              │
     │                               │
     │──── 응답 + 인증서 ─────────────>│
     │   (서명된 챌린지)              │
     │                               │
     │<───── 검증 ────────────────────│
     │      (성공/실패)               │

2단계: 인증서 검증
- 인증서 체인 확인
- 폐기 상태 확인 (OCSP)
- 인증서 제약 검증
- WIA 인증 확인

3단계: 영지식 증명 (선택사항)
- 송신기가 개인키를 공개하지 않고 신원 증명
- 기기가 암호학적으로 증명 검증
- 비밀 정보 교환 없음

4.6 암호화 프로토콜

4.6.1 포스트 양자 암호화

// 포스트 양자 키 교환
프로토콜: WIA-PQ-KEX (CRYSTALS-Kyber 기반)

1단계: 클라이언트가 키쌍 생성
  (pk, sk) ← Kyber.KeyGen()

2단계: 클라이언트가 공개키 전송
  클라이언트 → 서버: pk

3단계: 서버가 공유 비밀과 암호문 생성
  (c, ss_server) ← Kyber.Encaps(pk)

4단계: 서버가 암호문 전송
  서버 → 클라이언트: c

5단계: 클라이언트가 공유 비밀 도출
  ss_client ← Kyber.Decaps(c, sk)

검증: ss_client == ss_server

6단계: 세션 키 도출
  session_key ← KDF(ss_client || context)

4.7 영지식 프로토콜

4.7.1 신원 증명 프로토콜

// Schnorr 기반 영지식 신원 증명
프로토콜: ZK-Identity-Proof

설정:
- 소수 차수 q의 그룹 G
- 생성자 g
- 개인키: x (비밀)
- 공개키: y = g^x (공개)

증명 (증명자가 x를 공개하지 않고 알고 있음을 증명):

1. 증명자가 랜덤 r 생성
   commitment = g^r

2. 증명자가 검증자에게 커밋먼트 전송
   증명자 → 검증자: commitment

3. 검증자가 랜덤 챌린지 c 생성
   검증자 → 증명자: c

4. 증명자가 응답 계산
   response = r + c*x (mod q)
   증명자 → 검증자: response

5. 검증자가 확인
   g^response == commitment * y^c

참이면: 증명자가 x를 공개하지 않고 알고 있음이 증명됨

프로토콜 보안 속성

  • 순방향 비밀성: 장기 키의 침해가 과거 세션을 침해하지 않음
  • 포스트 양자 보안: 양자 컴퓨터 공격에 저항
  • 완전 순방향 비밀성: 각 연결마다 새로운 세션 키
  • 인증: 상호 인증으로 사칭 방지
  • 기밀성: 다중 계층 암호화로 모든 데이터 보호
  • 무결성: 인증된 암호화로 변조 방지
  • 익명성: 영지식 증명으로 프라이버시 보존
  • 관찰 불가능성: 트래픽 분석 저항
