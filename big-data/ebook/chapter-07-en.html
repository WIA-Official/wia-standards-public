Distributed Computing Fundamentals

Distributed computing is the foundation of big data processing, enabling parallel execution across multiple nodes to handle datasets that exceed the capacity of single machines. The WIA-BIG_DATA standard defines requirements for building reliable, scalable, and efficient distributed systems.

Core Distributed Computing Concepts

  • Horizontal Scaling: Add more nodes to increase capacity
  • Data Locality: Move computation to data, not vice versa
  • Fault Tolerance: Continue operating despite node failures
  • Consistency Models: Trade-offs between consistency and availability
  • Partitioning: Distribute data across nodes efficiently

Data Partitioning Strategies

Effective data partitioning is crucial for distributed computing performance. Poor partitioning leads to data skew, where some nodes are overloaded while others are idle.

# Partitioning Strategies in PySpark
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, spark_partition_id, count

spark = SparkSession.builder.appName("PartitioningDemo").getOrCreate()

# Check current partition distribution
def analyze_partitions(df, name):
    """Analyze data distribution across partitions"""
    partition_stats = df.withColumn("partition_id", spark_partition_id()) \
        .groupBy("partition_id") \
        .agg(count("*").alias("record_count")) \
        .orderBy("partition_id")

    print(f"\n{name} Partition Analysis:")
    partition_stats.show()

    stats = partition_stats.agg(
        {"record_count": "min", "record_count": "max", "record_count": "avg"}
    ).collect()[0]
    print(f"Min: {stats[0]}, Max: {stats[1]}, Avg: {stats[2]}")

# Hash Partitioning - Even distribution
df_hash = df.repartition(100, "user_id")
analyze_partitions(df_hash, "Hash Partitioned")

# Range Partitioning - Ordered data
df_range = df.repartitionByRange(100, "timestamp")
analyze_partitions(df_range, "Range Partitioned")

# Custom Salting for Skewed Keys
from pyspark.sql.functions import concat, lit, rand

def salt_skewed_key(df, key_column, salt_factor=10):
    """Add salt to skewed keys for better distribution"""
    return df.withColumn(
        "salted_key",
        concat(col(key_column), lit("_"), (rand() * salt_factor).cast("int"))
    )

# Handle skewed join
skewed_df = salt_skewed_key(large_df, "customer_id", salt_factor=10)

# Broadcast join for small tables
from pyspark.sql.functions import broadcast

result = large_df.join(
    broadcast(small_df),
    "join_key"
)

# Adaptive Query Execution handles skew automatically
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionFactor", "5")
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes", "256MB")

Mechanism Description Use Case
RDD Lineage Recompute lost partitions from source Short transformations
Checkpointing Save intermediate state to storage Long lineage chains
Replication Store multiple copies of data Critical datasets
Task Retry Retry failed tasks on other nodes Transient failures
Speculation Run backup tasks for slow executors Straggler mitigation 
# Fault Tolerance Implementation
from pyspark.sql import SparkSession
from pyspark.streaming import StreamingContext

# Checkpointing for fault tolerance
def create_context():
    spark = SparkSession.builder \
        .appName("FaultTolerantApp") \
        .config("spark.streaming.receiver.writeAheadLog.enable", "true") \
        .getOrCreate()

    ssc = StreamingContext(spark.sparkContext, batchDuration=10)
    ssc.checkpoint("hdfs:///checkpoints/streaming")

    # Create DStream with receiver
    lines = ssc.socketTextStream("localhost", 9999)

    # Stateful transformation with checkpointing
    def update_function(new_values, running_count):
        return sum(new_values) + (running_count or 0)

    word_counts = lines.flatMap(lambda line: line.split(" ")) \
        .map(lambda word: (word, 1)) \
        .updateStateByKey(update_function)

    word_counts.pprint()
    return ssc

# Recover or create context
ssc = StreamingContext.getOrCreate("hdfs:///checkpoints/streaming", create_context)
ssc.start()
ssc.awaitTermination()

# RDD Checkpointing for iterative algorithms
def iterative_algorithm(initial_rdd, iterations):
    rdd = initial_rdd
    rdd.cache()

    for i in range(iterations):
        rdd = rdd.map(transform_function)

        # Checkpoint every 10 iterations to truncate lineage
        if i % 10 == 0:
            rdd.checkpoint()
            rdd.count()  # Force materialization

    return rdd

Performance Optimization

Optimization Techniques

  • Data Serialization: Use Kryo for efficient serialization
  • Broadcast Variables: Share read-only data efficiently
  • Accumulators: Aggregate values across executors
  • Caching: Persist frequently accessed data in memory
  • Coalesce: Reduce partitions without full shuffle
# Performance Optimization Examples
from pyspark.sql import SparkSession
from pyspark.sql.functions import broadcast

spark = SparkSession.builder \
    .config("spark.serializer", "org.apache.spark.serializer.KryoSerializer") \
    .config("spark.kryo.registrationRequired", "false") \
    .getOrCreate()

# Broadcast join optimization
small_lookup = spark.read.parquet("/data/lookup_table")
large_data = spark.read.parquet("/data/transactions")

# Without broadcast: shuffle join
result_slow = large_data.join(small_lookup, "key")

# With broadcast: no shuffle for small table
result_fast = large_data.join(broadcast(small_lookup), "key")

# Caching strategies
from pyspark import StorageLevel

# Memory only - fastest, may spill
df.cache()  # Equivalent to MEMORY_ONLY

# Memory and disk - fault tolerant
df.persist(StorageLevel.MEMORY_AND_DISK)

# Memory serialized - less memory, more CPU
df.persist(StorageLevel.MEMORY_ONLY_SER)

# Off-heap memory
df.persist(StorageLevel.OFF_HEAP)

# Partition coalescing
# After filter that reduces data significantly
filtered_df = large_df.filter("status = 'active'")

# Reduce partitions without shuffle
coalesced_df = filtered_df.coalesce(50)

# Full shuffle repartition only when needed
repartitioned_df = filtered_df.repartition(50)

# Column pruning and predicate pushdown
# Only read required columns and filter at source
optimized_read = spark.read.parquet("/data/large_table") \
    .select("col1", "col2", "col3") \
    .filter("date >= '2024-01-01'")

# Check execution plan
optimized_read.explain(extended=True)

// Spark Application Troubleshooting Checklist
- Check Spark UI for stage details and task distribution
- Look for skewed tasks (some tasks much slower than others)
- Monitor GC time in executor metrics
- Check shuffle read/write metrics for bottlenecks
- Review spill metrics for memory pressure
- Analyze failed tasks and their error messages
- Examine executor logs for OOM errors

Key Takeaways

  • Configure cluster resources based on workload characteristics
  • Use appropriate partitioning to avoid data skew
  • Implement checkpointing for fault tolerance in long jobs
  • Optimize joins with broadcast variables for small tables
  • Monitor cluster health and application metrics continuously
