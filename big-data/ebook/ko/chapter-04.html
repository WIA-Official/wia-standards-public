4장: 1단계 - 데이터 형식 스펙

"데이터 형식의 표준화는 모든 상호 운용성의 출발점입니다"

WIA-DATA-001의 첫 번째 단계는 데이터를 어떻게 정의하고, 저장하고, 전송할 것인가를 다룹니다. SDL(Standard Data Layout) 스키마 정의 언어와 WBF(WIA Binary Format) 직렬화 형식은 빅데이터 생태계 전반의 공통 언어 역할을 합니다.

4.1 SDL (Standard Data Layout) 스키마 정의 언어

4.1.1 SDL 개요 및 설계 철학

SDL은 WIA-DATA-001의 핵심 스키마 정의 언어로, 데이터 구조를 명확하고 일관되게 표현합니다. JSON 기반의 직관적인 문법과 강력한 타입 시스템을 결합하여 개발자 친화성과 엄격성을 모두 확보했습니다.

SDL 설계 원칙:

  • 가독성: 사람이 쉽게 읽고 작성할 수 있는 구조
  • 표현력: 복잡한 데이터 관계도 명확하게 표현
  • 호환성: JSON Schema, Avro, Protobuf와 양방향 변환 가능
  • 진화 가능성: 스키마 변경 시 하위 호환성 유지
  • 도구 친화성: IDE 자동완성, 검증 도구 지원

4.2 WBF (WIA Binary Format) 직렬화 형식

4.2.1 WBF 설계 목표

WBF는 빅데이터 워크로드에 최적화된 바이너리 직렬화 형식으로, 다음 목표를 달성하도록 설계되었습니다:

성능 목표:

  • 압축률: 원본 JSON 대비 70-80% 크기 절감
  • 직렬화 속도: 1M 레코드/초 이상 (단일 코어)
  • 역직렬화 속도: 1.5M 레코드/초 이상
  • 메모리 효율: 제로 카피(zero-copy) 가능
  • 스트리밍 친화성: 청크 단위 점진적 처리

4.2.2 WBF 바이너리 구조

WBF 파일 구조:
┌─────────────────────────────────────────────────────┐
│ Header (고정 크기: 64 bytes)                         │
├─────────────────────────────────────────────────────┤
│ - Magic Number: 0x57424601 (4 bytes)                │
│ - Version: 1.0.0 (3 bytes)                          │
│ - Flags: 압축/암호화 설정 (1 byte)                   │
│ - Schema ID: 스키마 레지스트리 참조 (8 bytes)         │
│ - Record Count: 레코드 개수 (8 bytes)                │
│ - Compressed Size: 압축 후 크기 (8 bytes)            │
│ - Uncompressed Size: 압축 전 크기 (8 bytes)          │
│ - Checksum: CRC32C (4 bytes)                        │
│ - Reserved: 향후 확장 (20 bytes)                     │
├─────────────────────────────────────────────────────┤
│ Sync Marker (16 bytes)                              │
├─────────────────────────────────────────────────────┤
│ Data Block 1 (가변 크기, 기본 64KB)                  │
│ - Block Header (16 bytes)                           │
│ - Compressed Records (N bytes)                      │
├─────────────────────────────────────────────────────┤
│ Sync Marker (16 bytes)                              │
├─────────────────────────────────────────────────────┤
│ Data Block 2                                        │
├─────────────────────────────────────────────────────┤
│ ...                                                 │
├─────────────────────────────────────────────────────┤
│ Footer (가변 크기)                                   │
│ - Schema (JSON)                                     │
│ - Metadata (key-value pairs)                        │
│ - Block Index (offset, size)                        │
└─────────────────────────────────────────────────────┘

4.2.3 인코딩 기법

데이터 타입 인코딩 방식 설명 효율성
정수 ZigZag + VarInt 작은 숫자는 1-2바이트로 인코딩 40-60% 절감
부동소수점 IEEE 754 표준 float/double 표현 고정 크기
문자열 Length + UTF-8 길이(VarInt) + UTF-8 바이트 최소 오버헤드
배열 Count + Elements 개수(VarInt) + 각 요소 최소 오버헤드
맵 Count + Key-Value Pairs 개수 + (키, 값) 쌍 최소 오버헤드
Null Bit Flag 비트맵으로 null 표시 1 bit/필드

4.2.4 VarInt 인코딩 상세

VarInt(Variable-length Integer)는 작은 숫자를 효율적으로 저장하는 핵심 기법입니다:

VarInt 인코딩 예시:
  0 (0x00)       → 1 byte:  0000 0000
  127 (0x7F)     → 1 byte:  0111 1111
  128 (0x80)     → 2 bytes: 1000 0000 0000 0001
  16,383         → 2 bytes: 1111 1111 0111 1111
  2,097,151      → 3 bytes: 1111 1111 1111 1111 0111 1111

ZigZag 인코딩 (부호 있는 정수):
  0   → 0
  -1  → 1
  1   → 2
  -2  → 3
  2   → 4
  ...

효율성 비교:
  값 범위              | 고정 8바이트 | VarInt  | 절감률
  -64 ~ 63            |  8 bytes     | 1 byte  | 87.5%
  -8,192 ~ 8,191      |  8 bytes     | 2 bytes | 75%
  -1M ~ 1M            |  8 bytes     | 3 bytes | 62.5%

4.3 스키마 레지스트리

4.3.1 스키마 레지스트리의 역할

스키마 레지스트리는 SDL 스키마의 중앙 저장소로, 버전 관리와 호환성 검증을 담당합니다:

주요 기능:

  • 스키마 저장: 모든 스키마 버전을 영구 보관
  • 버전 관리: 시맨틱 버저닝 기반 버전 추적
  • 호환성 검증: 스키마 변경 시 호환성 자동 검사
  • 스키마 조회: ID 또는 이름으로 빠른 검색
  • 메타데이터 관리: 소유자, 생성일, 설명 등 추적

4.3.2 스키마 진화 (Schema Evolution)

스키마는 시간이 지남에 따라 변경될 수 있습니다. WIA-DATA-001은 4가지 호환성 모드를 지원합니다:

호환성 모드 설명 허용 변경 사용 사례
BACKWARD 새 스키마로 이전 데이터 읽기 필드 삭제, 기본값 있는 필드 추가 Consumer 먼저 업그레이드
FORWARD 이전 스키마로 새 데이터 읽기 필드 추가, 기본값 있는 필드 삭제 Producer 먼저 업그레이드
FULL 양방향 호환 기본값 있는 필드만 추가/삭제 순서 무관 업그레이드
NONE 호환성 검사 없음 모든 변경 허용 개발 환경, 1회성 데이터

4.3.3 스키마 버전 관리 예시

Order 스키마 진화 과정:

v1.0.0 (초기 버전):
{

  "fields": [
    {"name": "order_id", "type": "string"},
    {"name": "customer_id", "type": "long"},
    {"name": "total_amount", "type": "decimal"}

v1.1.0 (하위 호환 - 필드 추가):

  "fields": [
    {"name": "order_id", "type": "string"},
    {"name": "customer_id", "type": "long"},
    {"name": "total_amount", "type": "decimal"},
    {"name": "status", "type": "string", "default": "PENDING"} // 추가
  ]
}

v1.2.0 (하위 호환 - 선택 필드 추가):
{
  "name": "Order",
  "fields": [
    {"name": "order_id", "type": "string"},
    {"name": "customer_id", "type": "long"},
    {"name": "total_amount", "type": "decimal"},
    {"name": "status", "type": "string", "default": "PENDING"},
    {"name": "notes", "type": ["null", "string"], "default": null} // 추가

{
  "name": "Order",
  "fields": [
    {"name": "order_id", "type": "string"},
    {"name": "customer_id", "type": "string"}, // long → string (BREAKING)
    {"name": "total_amount", "type": "decimal"},
    {"name": "status", "type": "string", "default": "PENDING"},
    {"name": "notes", "type": ["null", "string"], "default": null}

}

4.4 압축 알고리즘

4.4.1 지원 압축 코덱

코덱 압축률 속도 CPU 사용 권장 용도
None 1:1 가장 빠름 없음 이미 압축된 데이터, 개발/테스트
Snappy 1:2~3 매우 빠름 낮음 실시간 스트리밍, 로그
LZ4 1:2~3 매우 빠름 낮음 범용, 기본 권장
Zstd (Level 3) 1:3~5 빠름 중간 분석 워크로드, 장기 저장
Zstd (Level 9) 1:4~7 중간 높음 아카이빙, 백업
Gzip 1:3~4 느림 높음 레거시 호환, HTTP 전송

4.4.2 압축 성능 벤치마크

1GB 전자상거래 주문 데이터 기준 (JSON → WBF):

코덱 압축 시간 압축 후 크기 압축률 처리량
None 2초 450 MB 55% 500 MB/s
Snappy 3초 200 MB 80% 333 MB/s
LZ4 3.5초 195 MB 80.5% 285 MB/s
Zstd-3 5초 150 MB 85% 200 MB/s
Zstd-9 15초 120 MB 88% 67 MB/s
Gzip 20초 140 MB 86% 50 MB/s

4.4.3 압축 선택 가이드

사용 시나리오별 권장 코덱:

실시간 스트리밍 (Kafka, Pulsar):

  • 1순위: LZ4 (속도와 압축률 균형)
  • 2순위: Snappy (최고 속도)

배치 분석 (Spark, Flink):

  • 1순위: Zstd-3 (우수한 압축률, 합리적 속도)
  • 2순위: LZ4 (빠른 처리 필요 시)

장기 저장 (S3, HDFS):

  • 1순위: Zstd-9 (최고 압축률)
  • 2순위: Zstd-3 (균형)

HTTP API 전송:

  • 1순위: Gzip (브라우저 호환)
  • 2순위: Zstd (최신 브라우저)

4.5 파티셔닝 및 인덱싱

4.5.1 파티셔닝 전략

WBF는 대규모 데이터셋의 효율적인 쿼리를 위해 파티셔닝을 지원합니다:

디렉토리 기반 파티셔닝:
/data/orders/
  year=2025/
    month=01/
      day=15/
        part-00000.wbf
        part-00001.wbf
        ...
    month=02/
      day=01/
        part-00000.wbf
        ...

Hive 스타일 파티션 경로:
/data/orders/year=2025/month=01/region=ASIA/country=KR/part-00000.wbf

쿼리 예시:
SELECT * FROM orders
WHERE year = 2025 AND month = 1 AND region = 'ASIA'
-- 해당 파티션만 스캔 (파티션 프루닝)

4.5.2 컬럼 통계 및 인덱스

각 WBF 파일은 푸터에 컬럼 통계를 포함하여 쿼리 최적화를 지원합니다:

통계 항목 설명 활용
Row Count 총 레코드 수 COUNT(*) 쿼리 최적화
Min/Max 각 컬럼의 최소/최대값 범위 필터 프루닝
Null Count NULL 값 개수 IS NULL 쿼리 최적화
Distinct Count 유니크 값 개수 (추정) 카디널리티 추정
Bloom Filter 값 존재 여부 확률 IN 절 필터 프루닝

4.5.3 쿼리 최적화 예시

쿼리:
SELECT * FROM orders
WHERE order_date >= '2025-01-01'
  AND order_date < '2025-02-01'
  AND total_amount > 100000

최적화 과정:
1. 파티션 프루닝: year=2025/month=01/ 파티션만 스캔
2. Min/Max 프루닝: order_date 범위 밖 파일 스킵
3. Min/Max 프루닝: total_amount < 100000인 파일 스킵
4. Bloom Filter: 특정 customer_id IN (...) 조건 시 활용

결과:
- 스캔 대상: 1000개 파일 → 50개 파일 (95% 감소)
- 쿼리 시간: 120초 → 6초 (95% 단축)

4.6 다른 형식과의 호환성

4.6.1 레거시 형식 변환

WIA-DATA-001은 기존 형식과의 양방향 변환을 지원합니다:

형식 읽기 쓰기 스키마 변환 성능 오버헤드
JSON ✅ ✅ 자동 추론 가능 파싱: 중간
CSV ✅ ✅ 헤더 또는 스키마 필요 파싱: 낮음
Avro ✅ ✅ 직접 매핑 거의 없음 (5%)
Parquet ✅ ✅ 스키마 매핑 낮음 (10%)
ORC ✅ ✅ 스키마 매핑 낮음 (10%)
Protobuf ✅ ✅ .proto → SDL 변환 낮음 (15%)

4.6.2 변환 도구 사용 예시

# JSON → WBF 변환
wia-convert --from json --to wbf \
  --input orders.json \
  --output orders.wbf \
  --schema order-schema.sdl \
  --compress lz4

# Parquet → WBF 변환
wia-convert --from parquet --to wbf \
  --input orders.parquet \
  --output orders.wbf \
  --infer-schema

# WBF → JSON 변환 (디버깅용)
wia-convert --from wbf --to json \
  --input orders.wbf \
  --output orders.json \
  --pretty

# 스키마 추출
wia-schema extract \
  --input orders.wbf \
  --output order-schema.sdl

4.8 모범 사례 및 안티 패턴

4.8.1 스키마 설계 모범 사례

DO (권장):

  • ✅ 명확하고 일관된 필드명 사용 (customer_id, order_date)
  • ✅ 모든 필드에 doc 주석 추가
  • ✅ 기본값 제공 (default) 으로 호환성 확보
  • ✅ 적절한 타입 선택 (금액은 decimal, ID는 long)
  • ✅ 네임스페이스 사용으로 충돌 방지
  • ✅ 필드 추가는 선택 필드 또는 기본값 포함

DON'T (안티 패턴):

  • ❌ 모호한 약어 사용 (cid, amt 대신 전체 단어)
  • ❌ 깊은 중첩 (3단계 이상)
  • ❌ 필수 필드 삭제 (호환성 깨짐)
  • ❌ 타입 변경 (메이저 버전 업그레이드 필요)
  • ❌ 거대한 맵 타입 (map<string, any> 남용)
  • ❌ 문서화 없이 스키마 배포

4.9 장 요약

핵심 요점

  1. SDL 스키마 언어: WIA-DATA-001은 SDL(Standard Data Layout)이라는 JSON 기반의 직관적인 스키마 정의 언어를 제공합니다. 기본 타입(boolean, int, long, string 등), 특수 타입(decimal, timestamp), 복합 타입(array, map, record)을 지원하며, 중첩 레코드와 참조를 통해 복잡한 데이터 구조를 명확하게 표현할 수 있습니다.
  2. WBF 바이너리 형식: WBF는 빅데이터 워크로드에 최적화된 바이너리 직렬화 형식으로, VarInt/ZigZag 인코딩을 통해 작은 숫자를 1-2바이트로 압축합니다. 원본 JSON 대비 70-80% 크기 절감, 1M 레코드/초 이상의 직렬화 속도, 제로 카피 지원으로 메모리 효율성을 극대화합니다.
  3. 스키마 레지스트리: 중앙 집중식 스키마 저장소를 통해 버전 관리와 호환성 검증을 자동화합니다. BACKWARD, FORWARD, FULL, NONE 4가지 호환성 모드를 지원하여 스키마 진화 시 이전 데이터와의 호환성을 보장하며, 시맨틱 버저닝으로 명확한 버전 추적이 가능합니다.
  4. 압축 알고리즘: Snappy, LZ4, Zstd, Gzip 등 다양한 압축 코덱을 지원하며, 사용 시나리오에 따라 최적의 코덱을 선택할 수 있습니다. 실시간 스트리밍에는 LZ4, 배치 분석에는 Zstd-3, 장기 저장에는 Zstd-9를 권장하며, 1GB JSON 데이터를 120MB까지 압축 가능합니다.
  5. 파티셔닝과 인덱싱: Hive 스타일 디렉토리 파티셔닝으로 쿼리 성능을 극적으로 향상시키며, 각 파일의 푸터에 Min/Max, Null Count, Bloom Filter 등의 통계를 포함하여 파티션 프루닝과 필터 프루닝을 지원합니다. 실제 사례에서 1000개 파일 스캔을 50개로 줄여 95% 쿼리 시간 단축을 달성했습니다.
  6. 레거시 호환성: JSON, CSV, Avro, Parquet, ORC, Protobuf 등 모든 주요 형식과 양방향 변환을 지원하여 기존 시스템과의 통합이 용이합니다. 특히 Avro와는 거의 오버헤드 없이 (5% 이내) 직접 매핑이 가능하며, 변환 도구(wia-convert)를 통해 배치 변환을 자동화할 수 있습니다.
  7. 다중 언어 SDK: Python, Java, Go, TypeScript용 공식 SDK를 제공하여 WBF 파일의 읽기/쓰기를 간단하게 구현할 수 있습니다. 스키마 검증, 압축, 스트리밍 처리 등 모든 기능을 SDK에서 직접 지원하며, 개발자는 비즈니스 로직에만 집중할 수 있습니다.
  8. 모범 사례: 명확한 필드명 사용, 모든 필드에 문서 주석 추가, 기본값 제공으로 호환성 확보, 적절한 타입 선택, 네임스페이스 활용 등의 모범 사례를 따르면 유지보수성과 성능을 동시에 달성할 수 있습니다. 반대로 모호한 약어, 깊은 중첩, 필수 필드 삭제 등의 안티 패턴은 반드시 피해야 합니다.

4.10 복습 문제

문제 1: SDL 스키마에서 decimal 타입과 double 타입의 차이를 설명하고, 각각 어떤 상황에서 사용해야 하는지 예를 들어 설명하세요.

문제 2: WBF의 VarInt 인코딩이 일반 고정 크기 정수 인코딩보다 효율적인 이유를 설명하고, 값 127, 128, 16383을 VarInt로 인코딩했을 때 각각 몇 바이트가 필요한지 계산하세요.

문제 3: 스키마 레지스트리의 4가지 호환성 모드(BACKWARD, FORWARD, FULL, NONE)를 각각 설명하고, 실제 프로젝트에서 FULL 모드를 사용해야 하는 이유를 설명하세요.

문제 4: 다음 시나리오별로 최적의 압축 코덱을 선택하고 그 이유를 설명하세요: (1) 실시간 주문 처리 Kafka 토픽, (2) 월별 매출 분석 Spark 배치 작업, (3) 법적 의무 7년 보관 아카이브

문제 5: 전자상거래 주문 데이터를 year, month, region으로 파티셔닝했을 때, 다음 쿼리에서 스캔되는 파티션 수를 계산하세요: SELECT * FROM orders WHERE year=2025 AND month IN (1,2) AND region='ASIA'

문제 6: 기존 Order 스키마(v1.0.0)에 새로운 필드 payment_method를 추가하려고 합니다. FULL 호환성을 유지하면서 추가하는 방법을 SDL 코드로 작성하고, 호환성 검증 과정을 설명하세요.
