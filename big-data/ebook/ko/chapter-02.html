Chapter 2: 데이터 수집과 저장

빅데이터 파이프라인의 시작점은 데이터 수집입니다. 다양한 소스에서 발생하는 데이터를 효율적으로 수집하고, 적절한 저장 시스템에 안전하게 보관하는 것은 모든 빅데이터 프로젝트의 성공을 위한 기반입니다. 본 장에서는 데이터 소스 유형, 수집 패턴, 분산 스토리지 인프라, 그리고 데이터 레이크 아키텍처를 상세히 다룹니다.

2.1 데이터 소스 유형

현대 기업 환경에서 데이터는 다양한 소스에서 생성됩니다. 각 소스의 특성을 이해하고 적절한 수집 전략을 수립하는 것이 중요합니다.

2.2 데이터 수집 패턴

데이터 수집 방식은 크게 배치 수집과 실시간 수집으로 나뉩니다. 비즈니스 요구사항과 데이터 특성에 따라 적절한 패턴을 선택해야 합니다.

Change Data Capture (CDC)

CDC는 소스 데이터베이스의 변경사항을 실시간으로 감지하여 타겟 시스템에 전파하는 패턴입니다. 전체 데이터를 복사하는 것보다 효율적이며, 데이터 동기화에 널리 사용됩니다.

CDC 아키텍처
Change Data Capture 구현 패턴
═══════════════════════════════════════════════════════════════

1. 로그 기반 CDC (Log-based CDC)
   ─────────────────────────────────────────
   
   ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
   │   MySQL      │     │  Debezium    │     │   Kafka      │
   │  (binlog)    │────▶│  Connector   │────▶│   Topic      │
   └──────────────┘     └──────────────┘     └──────────────┘
                                                    │
                        ┌───────────────────────────┘
                        │
                        ▼
   ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
   │  Data Lake   │◀────│   Consumer   │◀────│   Kafka      │
   │              │     │   (Flink)    │     │   Topic      │
   └──────────────┘     └──────────────┘     └──────────────┘

   장점:
   • 소스 시스템 부하 최소화
   • 모든 변경사항 캡처
   • 삭제 작업도 추적 가능

2. 트리거 기반 CDC (Trigger-based CDC)
   ─────────────────────────────────────────
   
   ┌──────────────────────────────────────────────────────────┐
   │  CREATE TRIGGER capture_changes                          │
   │  AFTER INSERT OR UPDATE OR DELETE ON orders              │
   │  FOR EACH ROW                                            │
   │  BEGIN                                                    │
   │      INSERT INTO change_log (                            │
   │          table_name, operation, row_data, timestamp      │
   │      ) VALUES (                                          │
   │          'orders', TG_OP, row_to_json(NEW), NOW()        │
   │      );                                                   │
   │  END;                                                     │
   └──────────────────────────────────────────────────────────┘

   장점:
   • 구현 단순
   • 데이터베이스 독립적
   
   단점:
   • 소스 DB 성능 영향
   • 스키마 변경 추적 어려움

3. 타임스탬프 기반 CDC (Timestamp-based CDC)
   ─────────────────────────────────────────
   
   SELECT * FROM orders
   WHERE updated_at > :last_sync_timestamp
   ORDER BY updated_at;

   장점:
   • 가장 단순한 구현
   
   단점:
   • 삭제 추적 불가
   • 시계 동기화 이슈

2.3 분산 스토리지 인프라

빅데이터는 단일 서버로 저장할 수 없는 규모입니다. 분산 파일 시스템과 객체 스토리지는 대규모 데이터를 안정적으로 저장하기 위한 핵심 인프라입니다.

핵심 특성:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

• 블록 크기: 기본 128MB (대용량 파일에 최적화)
• 복제 계수: 기본 3 (데이터 내구성 보장)
• Write-Once-Read-Many: 한 번 쓰면 수정 불가
• 랙 인식: 장애 도메인 분리를 위한 복제 배치
• 고가용성: NameNode HA, Federation 지원

HDFS 명령어 예시:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

hdfs dfs -put local_file.csv /user/data/

# 디렉토리 생성
hdfs dfs -mkdir -p /user/data/raw/2025/01

# 파일 목록 조회
hdfs dfs -ls /user/data/

# 파일 읽기
hdfs dfs -cat /user/data/sample.txt

# 복제 계수 변경
hdfs dfs -setrep -w 5 /user/critical/important.parquet

2.4 데이터 레이크 아키텍처

데이터 레이크는 원시 형태의 데이터를 대규모로 저장하는 중앙 집중식 저장소입니다. 정형, 반정형, 비정형 데이터를 모두 수용할 수 있으며, 분석 시 스키마를 적용하는 "Schema-on-Read" 접근 방식을 사용합니다.

Delta Lake 예시:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

# Delta 테이블 생성
df.write.format("delta").mode("overwrite") \
    .partitionBy("year", "month") \
    .save("/lakehouse/sales/orders")

# ACID 트랜잭션
from delta.tables import DeltaTable

deltaTable = DeltaTable.forPath(spark, "/lakehouse/sales/orders")

# MERGE (Upsert)
deltaTable.merge(
    newData,
    "target.order_id = source.order_id"
).whenMatchedUpdate(set={
    "amount": "source.amount",
    "updated_at": "current_timestamp()"
}).whenNotMatchedInsert(values={
    "order_id": "source.order_id",
    "amount": "source.amount",
    "created_at": "current_timestamp()"
}).execute()

# 타임 트래블 (과거 데이터 조회)
df_history = spark.read.format("delta") \
    .option("timestampAsOf", "2025-01-18") \
    .load("/lakehouse/sales/orders")

WIA-BIG_DATA 저장소 설계 권장사항

1. 원본 데이터는 항상 보존하고 절대 삭제하지 않습니다. 2. 파티셔닝 전략을 데이터 접근 패턴에 맞게 설계합니다. 3. 적절한 파일 포맷(Parquet, ORC)을 선택하여 저장 공간과 쿼리 성능을 최적화합니다. 4. 메타데이터 관리를 자동화하여 데이터 카탈로그를 최신 상태로 유지합니다.
