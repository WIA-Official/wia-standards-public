Introduction to Stream Processing

Stream processing is the continuous processing of data as it arrives, enabling real-time analytics, monitoring, and decision-making. Unlike batch processing that operates on bounded datasets, stream processing handles unbounded, continuously flowing data with minimal latency.

The WIA-BIG_DATA standard defines requirements for stream processing implementations that ensure reliability, scalability, and consistency across diverse streaming platforms.

Processing Semantics

Delivery Guarantees

  • At-most-once: Messages may be lost, never duplicated (fastest)
  • At-least-once: Messages never lost, may be duplicated (reliable)
  • Exactly-once: Messages processed exactly once (strongest guarantee)

The WIA-BIG_DATA standard requires at-least-once semantics as the minimum guarantee, with exactly-once recommended for financial and critical applications.

Kafka Producer Implementation

// WIA-BIG_DATA Compliant Kafka Producer
import org.apache.kafka.clients.producer.*;
import org.apache.kafka.common.serialization.StringSerializer;
import io.confluent.kafka.serializers.KafkaAvroSerializer;

public class WIAKafkaProducer {
    private final KafkaProducer<String, GenericRecord> producer;
    private final String topic;

    public WIAKafkaProducer(String bootstrapServers, String topic) {
        this.topic = topic;

        Properties props = new Properties();
        props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, bootstrapServers);
        props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, KafkaAvroSerializer.class);

        // Exactly-once semantics
        props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, true);
        props.put(ProducerConfig.ACKS_CONFIG, "all");
        props.put(ProducerConfig.RETRIES_CONFIG, Integer.MAX_VALUE);
        props.put(ProducerConfig.MAX_IN_FLIGHT_REQUESTS_PER_CONNECTION, 5);

        // Performance tuning
        props.put(ProducerConfig.COMPRESSION_TYPE_CONFIG, "lz4");
        props.put(ProducerConfig.BATCH_SIZE_CONFIG, 32768);
        props.put(ProducerConfig.LINGER_MS_CONFIG, 5);

        // Schema Registry
        props.put("schema.registry.url", "http://schema-registry:8081");

        this.producer = new KafkaProducer<>(props);
    }

    public void sendWithCallback(String key, GenericRecord value) {
        ProducerRecord<String, GenericRecord> record =
            new ProducerRecord<>(topic, key, value);

        producer.send(record, (metadata, exception) -> {
            if (exception != null) {
                // Log and handle failure
                logger.error("Failed to send message", exception);
                handleFailure(record, exception);
            } else {
                // Log success with offset
                logger.debug("Message sent to partition {} offset {}",
                    metadata.partition(), metadata.offset());
            }
        });
    }

    public void close() {
        producer.flush();
        producer.close();
    }
}

Apache Flink Integration

Apache Flink provides true stream processing with powerful windowing, state management, and exactly-once guarantees. The WIA-BIG_DATA standard recommends Flink for complex event processing and stateful computations.

// WIA-BIG_DATA Flink Streaming Application
import org.apache.flink.streaming.api.scala._
import org.apache.flink.streaming.api.windowing.time.Time
import org.apache.flink.streaming.api.windowing.windows.TimeWindow
import org.apache.flink.streaming.api.CheckpointingMode

object WIAStreamProcessor {
  def main(args: Array[String]): Unit = {
    val env = StreamExecutionEnvironment.getExecutionEnvironment

    // Exactly-once checkpointing
    env.enableCheckpointing(60000, CheckpointingMode.EXACTLY_ONCE)
    env.getCheckpointConfig.setMinPauseBetweenCheckpoints(30000)
    env.getCheckpointConfig.setCheckpointTimeout(120000)
    env.getCheckpointConfig.setMaxConcurrentCheckpoints(1)
    env.getCheckpointConfig.enableExternalizedCheckpoints(
      ExternalizedCheckpointCleanup.RETAIN_ON_CANCELLATION
    )

    // State backend configuration
    env.setStateBackend(new RocksDBStateBackend("hdfs:///checkpoints", true))

    // Kafka source with watermarks
    val kafkaSource = KafkaSource.builder[Event]()
      .setBootstrapServers("kafka:9092")
      .setTopics("events")
      .setGroupId("wia-flink-consumer")
      .setStartingOffsets(OffsetsInitializer.committedOffsets())
      .setValueOnlyDeserializer(new EventDeserializer())
      .build()

    val eventStream = env.fromSource(
      kafkaSource,
      WatermarkStrategy
        .forBoundedOutOfOrderness[Event](Duration.ofSeconds(10))
        .withTimestampAssigner((event, _) => event.timestamp),
      "Kafka Events"
    )

    // Windowed aggregation
    val aggregatedStream = eventStream
      .keyBy(_.userId)
      .window(TumblingEventTimeWindows.of(Time.minutes(5)))
      .allowedLateness(Time.minutes(1))
      .sideOutputLateData(lateDataTag)
      .aggregate(new EventAggregator(), new WindowResultFunction())

    // Sink to multiple outputs
    aggregatedStream.addSink(new KafkaSink("aggregated-events"))
    aggregatedStream.addSink(new ElasticsearchSink(esConfig))

    env.execute("WIA Stream Processor")
  }
}

Windowing Strategies

Window Type Description Use Case
Tumbling Fixed-size, non-overlapping windows Hourly aggregations
Sliding Fixed-size, overlapping windows Moving averages
Session Activity-based, gap-driven windows User sessions
Global All elements in one window Custom triggers 
// Window Implementation Examples
// Tumbling Window: 5-minute aggregations
val tumblingResults = events
  .keyBy(_.customerId)
  .window(TumblingEventTimeWindows.of(Time.minutes(5)))
  .reduce((a, b) => a.merge(b))

// Sliding Window: 10-minute window, 5-minute slide
val slidingResults = events
  .keyBy(_.customerId)
  .window(SlidingEventTimeWindows.of(Time.minutes(10), Time.minutes(5)))
  .aggregate(new AvgAggregator())

// Session Window: 30-minute gap
val sessionResults = events
  .keyBy(_.userId)
  .window(EventTimeSessionWindows.withGap(Time.minutes(30)))
  .process(new SessionAnalyzer())

Complex Event Processing (CEP)

CEP Pattern Matching

Complex Event Processing enables detection of patterns across event streams, essential for fraud detection, anomaly identification, and business rule enforcement.

// Flink CEP Pattern for Fraud Detection
import org.apache.flink.cep.scala.CEP
import org.apache.flink.cep.scala.pattern.Pattern

// Define suspicious activity pattern
val fraudPattern = Pattern.begin[Transaction]("first")
  .where(_.amount > 1000)
  .next("second")
  .where(_.amount > 1000)
  .within(Time.minutes(10))
  .next("third")
  .where(tx => tx.amount > 5000 && tx.country != tx.homeCountry)
  .within(Time.minutes(30))

// Apply pattern to stream
val patternStream = CEP.pattern(transactionStream.keyBy(_.userId), fraudPattern)

val alerts = patternStream.select(matches => {
  val first = matches("first").head
  val second = matches("second").head
  val third = matches("third").head

  FraudAlert(
    userId = first.userId,
    transactions = List(first, second, third),
    totalAmount = first.amount + second.amount + third.amount,
    riskScore = calculateRisk(first, second, third),
    timestamp = System.currentTimeMillis()
  )
})

alerts.addSink(new AlertNotificationSink())

Kafka Streams API

For simpler streaming use cases, Kafka Streams provides a lightweight library that runs as a standard Java application without requiring a separate cluster.

// Kafka Streams Topology for Order Processing
import org.apache.kafka.streams.*;
import org.apache.kafka.streams.kstream.*;

public class OrderProcessor {
    public static void main(String[] args) {
        Properties props = new Properties();
        props.put(StreamsConfig.APPLICATION_ID_CONFIG, "order-processor");
        props.put(StreamsConfig.BOOTSTRAP_SERVERS_CONFIG, "kafka:9092");
        props.put(StreamsConfig.PROCESSING_GUARANTEE_CONFIG,
                  StreamsConfig.EXACTLY_ONCE_V2);

        StreamsBuilder builder = new StreamsBuilder();

        // Source: Orders topic
        KStream<String, Order> orders = builder.stream(
            "orders",
            Consumed.with(Serdes.String(), orderSerde)
        );

        // Branch orders by type
        Map<String, KStream<String, Order>> branches = orders.split(Named.as("branch-"))
            .branch((key, order) -> order.getType().equals("STANDARD"),
                    Branched.as("standard"))
            .branch((key, order) -> order.getType().equals("EXPRESS"),
                    Branched.as("express"))
            .defaultBranch(Branched.as("other"));

        // Process standard orders
        branches.get("branch-standard")
            .mapValues(order -> enrichOrder(order))
            .to("enriched-orders");

        // Join with customer data
        KTable<String, Customer> customers = builder.table("customers");

        KStream<String, EnrichedOrder> enrichedOrders = orders
            .selectKey((k, v) -> v.getCustomerId())
            .join(customers,
                  (order, customer) -> new EnrichedOrder(order, customer),
                  Joined.with(Serdes.String(), orderSerde, customerSerde));

        // Windowed aggregation
        enrichedOrders
            .groupBy((k, v) -> v.getRegion())
            .windowedBy(TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(15)))
            .aggregate(
                () -> new RegionMetrics(),
                (key, order, metrics) -> metrics.add(order),
                Materialized.as("region-metrics-store")
            )
            .toStream()
            .to("region-metrics");

        KafkaStreams streams = new KafkaStreams(builder.build(), props);
        streams.start();
    }
}

Stream Processing Monitoring

Key Metrics to Monitor

  • Throughput: Events processed per second
  • Latency: End-to-end processing time
  • Consumer Lag: Messages behind latest offset
  • Checkpoint Duration: Time to complete checkpoints
  • Backpressure: Queue buildup indicators

Best Practices Summary

WIA-BIG_DATA Stream Processing Checklist

  • Enable exactly-once semantics for critical data
  • Configure proper watermarks for out-of-order events
  • Implement dead-letter queues for failed messages
  • Use incremental checkpointing for large state
  • Monitor consumer lag and processing latency
  • Design for horizontal scalability
  • Implement graceful shutdown and restart
