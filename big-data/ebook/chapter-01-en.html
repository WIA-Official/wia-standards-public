Chapter 1: Big Data Architecture Patterns

Learning Objectives: Understand the three major big data architecture patterns (Lambda, Kappa, Delta) and learn when to apply each pattern based on your organization's requirements.

Introduction to Big Data Architecture

Big data architecture refers to the overall structure of a system designed to handle massive volumes of data. The choice of architecture fundamentally impacts how data flows through your organization, how quickly insights can be derived, and how complex the operational overhead will be.

The WIA-BIG_DATA standard recognizes three primary architecture patterns, each with distinct advantages and trade-offs. Understanding these patterns is essential for designing systems that meet your specific latency, throughput, and complexity requirements.

Lambda Architecture

Lambda Architecture, introduced by Nathan Marz, is a hybrid approach that combines batch and stream processing to achieve both comprehensive accuracy and low latency. It consists of three layers that work together to provide a complete view of the data.

Lambda Architecture Components

  • Batch Layer: Stores the master dataset and pre-computes batch views for comprehensive analytics
  • Speed Layer: Processes real-time data to compensate for batch layer latency
  • Serving Layer: Indexes batch and speed layer outputs for low-latency queries

Kappa Architecture

Kappa Architecture, proposed by Jay Kreps, simplifies Lambda by treating everything as a stream. Instead of maintaining two separate processing paths, Kappa uses a single stream processing engine for all data. Historical data is replayed through the same stream processor when reprocessing is needed.

Kappa Architecture Principles

  • Single Processing Path: All data flows through one stream processor
  • Replayable Log: An immutable log (like Kafka) serves as the system of record
  • Reprocessing via Replay: New computations are run by replaying the log
  • Simplified Operations: One codebase, one processing engine to maintain
// Kappa Architecture Implementation with Apache Flink
import org.apache.flink.streaming.api.scala._
import org.apache.flink.streaming.connectors.kafka._

object KappaProcessor {
  def main(args: Array[String]): Unit = {
    val env = StreamExecutionEnvironment.getExecutionEnvironment

    // Enable exactly-once processing
    env.enableCheckpointing(60000)
    env.getCheckpointConfig.setCheckpointingMode(
      CheckpointingMode.EXACTLY_ONCE
    )

    // Configure Kafka source with earliest offset for replay
    val kafkaSource = KafkaSource.builder[Event]()
      .setBootstrapServers("kafka:9092")
      .setTopics("events")
      .setStartingOffsets(OffsetsInitializer.earliest())
      .setValueOnlyDeserializer(new EventDeserializer())
      .build()

    // Single stream processing pipeline
    val eventStream = env.fromSource(
      kafkaSource,
      WatermarkStrategy.forBoundedOutOfOrderness(Duration.ofSeconds(30)),
      "Kafka Source"
    )

    // Process all events through unified logic
    val aggregatedStream = eventStream
      .keyBy(_.userId)
      .window(TumblingEventTimeWindows.of(Time.minutes(5)))
      .aggregate(new EventAggregator())

    // Sink to serving layer
    aggregatedStream.addSink(new DruidSink())

    env.execute("Kappa Pipeline")
  }
}

Delta Architecture (Lakehouse)

Delta Architecture, also known as the Lakehouse pattern, combines the flexibility of data lakes with the reliability of data warehouses. It uses technologies like Delta Lake to add ACID transactions, schema enforcement, and time travel capabilities to data lakes.

Delta Architecture Layers

  • Bronze Layer: Raw data ingestion, minimal processing
  • Silver Layer: Cleaned, deduplicated, schema-enforced data
  • Gold Layer: Business-level aggregations and curated datasets
# Delta Lake Implementation in Python
from delta import DeltaTable
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, current_timestamp

# Initialize Spark with Delta Lake
spark = SparkSession.builder \
    .appName("DeltaLakehouse") \
    .config("spark.sql.extensions",
            "io.delta.sql.DeltaSparkSessionExtension") \
    .config("spark.sql.catalog.spark_catalog",
            "org.apache.spark.sql.delta.catalog.DeltaCatalog") \
    .getOrCreate()

# Bronze Layer: Ingest raw data
raw_events = spark.readStream \
    .format("kafka") \
    .option("kafka.bootstrap.servers", "kafka:9092") \
    .option("subscribe", "events") \
    .load()

# Write to Bronze with append mode
raw_events.writeStream \
    .format("delta") \
    .outputMode("append") \
    .option("checkpointLocation", "/checkpoints/bronze") \
    .start("/data/bronze/events")

# Silver Layer: Clean and deduplicate
bronze_df = spark.read.format("delta").load("/data/bronze/events")

silver_df = bronze_df \
    .dropDuplicates(["event_id"]) \
    .filter(col("event_type").isNotNull()) \
    .withColumn("processed_at", current_timestamp())

silver_df.write \
    .format("delta") \
    .mode("overwrite") \
    .option("overwriteSchema", "true") \
    .save("/data/silver/events")

# Gold Layer: Business aggregations
silver_events = spark.read.format("delta").load("/data/silver/events")

daily_metrics = silver_events \
    .groupBy("date", "product_category") \
    .agg(
        count("event_id").alias("total_events"),
        sum("revenue").alias("total_revenue"),
        countDistinct("user_id").alias("unique_users")
    )

daily_metrics.write \
    .format("delta") \
    .mode("overwrite") \
    .partitionBy("date") \
    .save("/data/gold/daily_metrics")

# Time Travel Query
historical_data = spark.read \
    .format("delta") \
    .option("versionAsOf", 10) \
    .load("/data/silver/events")

Latency Low (speed layer) Low (streaming) Low to Medium
Reprocessing Batch layer Log replay Time travel
ACID Support Limited Via Kafka Full ACID
Best For Mixed workloads Stream-first apps Analytics-heavy

Choosing the Right Architecture

Decision Framework

Choose Lambda when: You have both batch and streaming requirements with different accuracy needs, and operational complexity is acceptable.

Choose Kappa when: Stream processing can handle all use cases, and you want to minimize code duplication and operational overhead.

Choose Delta when: You need ACID transactions, schema evolution, and time travel capabilities with unified batch and streaming support.

Key Takeaways

  • All three architectures are valid for WIA-BIG_DATA compliance
  • The choice depends on specific latency, accuracy, and operational requirements
  • Delta Architecture is becoming the preferred choice for new implementations
  • Hybrid approaches combining elements of each pattern are also acceptable
