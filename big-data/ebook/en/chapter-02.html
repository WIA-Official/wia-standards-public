Chapter 2: Data Architecture

A well-designed data architecture is the foundation of any successful big data implementation. This chapter explores the architectural patterns, storage strategies, and design principles that enable organizations to build scalable, maintainable big data systems.

2.1 Architectural Paradigms

Modern big data architecture has evolved through several paradigms, each addressing specific challenges and use cases. Understanding these paradigms helps organizations choose the right approach for their needs.

Traditional Data Warehouse

The data warehouse model, pioneered in the 1990s, centralizes structured data from multiple sources into a single repository optimized for analytical queries. While effective for structured data, traditional warehouses struggle with scale, variety, and real-time requirements.

Data Lake

Data lakes store raw data in its native format, deferring schema definition until query time (schema-on-read). This approach provides flexibility and supports diverse data types but can lead to "data swamps" without proper governance.

Data Lakehouse

The lakehouse architecture combines the best aspects of data warehouses and data lakes. It provides ACID transactions, schema enforcement, and governance features on top of scalable, low-cost storage.

2.2 The Medallion Architecture

The medallion architecture, popularized by Databricks, organizes data into three layers based on quality and transformation status:

Bronze Layer (Raw)

Raw data as ingested from source systems. This layer preserves the original data format and serves as the single source of truth. Data is immutable and timestamped for auditability.

  • Contains exact copies of source data
  • Minimal transformations applied
  • Supports data replay and recovery
  • Typically stored in cost-efficient formats (Parquet, ORC)
Silver Layer (Cleaned)

Cleansed and conformed data with consistent schemas. This layer applies data quality rules, deduplication, and standardization while maintaining granular detail.

  • Schema validation and enforcement
  • Data quality rules applied
  • Deduplication and cleansing
  • Joins across source systems
Gold Layer (Curated)

Business-level aggregations and data products ready for consumption. This layer contains purpose-built datasets optimized for specific analytical use cases.

  • Aggregated and summarized data
  • Business metrics and KPIs
  • Feature stores for ML
  • Optimized for query performance

2.4 Table Formats

Modern table formats provide database-like features on top of data lake storage:

Delta Lake

Developed by Databricks, Delta Lake provides ACID transactions, scalable metadata handling, and time travel capabilities on top of Parquet files. It integrates seamlessly with Spark.

Apache Iceberg

Designed for huge tables with multiple petabytes of data, Iceberg provides snapshot isolation, schema evolution, and hidden partitioning. It's vendor-neutral and supports multiple compute engines.

Apache Hudi

Hudi (Hadoop Upserts Deletes and Incrementals) excels at incremental data processing and provides record-level updates. It's particularly suited for CDC (Change Data Capture) workloads.

2.5 Partitioning Strategies

Effective partitioning is essential for query performance and cost optimization at scale:

Time-based Partitioning

Partition by date or timestamp for time-series data. This enables efficient pruning of historical data and supports common analytical patterns.

# Example: Time-based partitioning
/data/events/
├── year=2024/
│   ├── month=01/
│   │   ├── day=01/
│   │   ├── day=02/
│   │   └── ...
│   └── month=02/
└── year=2025/
    └── month=01/

Hash Partitioning

Distribute data evenly across partitions using a hash function. This ensures balanced workloads and prevents hot spots.

Range Partitioning

Partition by value ranges for ordered data. Useful for data with natural ordering like IDs or geographic regions.

2.6 Data Modeling

Data modeling for big data systems differs from traditional OLTP approaches:

Denormalization

Unlike normalized OLTP schemas, big data systems often use denormalized models to minimize joins and improve query performance. Storage is cheap; compute is expensive.

Star Schema

Fact tables with associated dimension tables remain effective for analytical workloads. Modern implementations use columnar storage for efficiency.

Wide Tables

Extremely wide tables with hundreds or thousands of columns are common in big data systems. Columnar formats handle this efficiently by reading only required columns.

2.7 Architecture Patterns

Lambda Architecture

Combines batch and stream processing with separate layers. The batch layer provides comprehensive historical views, while the speed layer handles real-time updates.

Kappa Architecture

Simplifies Lambda by using a single stream processing layer for all data. Historical reprocessing is done by replaying the event log.

2.8 Chapter Summary

Data architecture forms the foundation of any big data implementation. The choice between data lakes, warehouses, and lakehouses depends on organizational needs, existing infrastructure, and use case requirements. Modern table formats like Delta Lake, Iceberg, and Hudi bring reliability and manageability to data lake storage.

Effective partitioning, appropriate storage technologies, and well-designed data models ensure that big data systems can scale while maintaining performance. The medallion architecture provides a proven pattern for organizing data quality tiers.
