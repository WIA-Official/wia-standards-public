CHAPTER 04

Phase 1 - Data Format

4.1 Introduction to WIA Data Formats

Phase 1 of the WIA Environmental Sensor Standard establishes standardized data schemas that form the foundation for all higher phases. These schemas define how environmental sensor data is structured, what fields are required or optional, what data types are used, and how values should be represented. By standardizing these fundamental elements, Phase 1 enables the most critical form of interoperability—data-level compatibility.

The data format standard uses JSON (JavaScript Object Notation) as its primary serialization format. JSON was chosen for several compelling reasons: it's human-readable and easy to debug, it's supported natively by virtually all programming languages, it efficiently represents hierarchical data structures, it's well-suited for both storage and transmission, and it has robust schema validation tools through JSON Schema.

While JSON is the normative format, the standard also provides guidance for equivalent representations in XML, Protocol Buffers, and CBOR for scenarios where these formats are more appropriate. All representations must be semantically equivalent to the canonical JSON format.

4.2 Core Data Model Structure

All WIA environmental sensor data messages share a common core structure that provides essential context for measurements. This core structure includes standard fields that appear in every message regardless of sensor type.

Mandatory Core Fields

Every WIA-compliant sensor data message must include these fields:

  • version: String specifying the WIA standard version (e.g., "1.0.0"). This enables parsers to handle schema evolution gracefully.
  • standard: String identifying the WIA standard (e.g., "WIA-ENE-027"). This allows data from different WIA standards to coexist and be distinguished.
  • deviceId: Unique identifier for the sensor device. Should be globally unique, typically using UUID format or manufacturer-assigned serial numbers.
  • timestamp: ISO 8601 formatted timestamp indicating when the measurement was taken (e.g., "2024-01-15T14:23:45.123Z"). Must include timezone, preferably UTC.
  • sensorType: String indicating the category of sensor (e.g., "air_quality", "water_quality", "soil"). Maps to specific type definitions.

Optional Core Fields

The following fields are optional but strongly recommended:

  • location: Object containing latitude, longitude, and optionally altitude. Critical for spatial analysis and mapping.
  • metadata: Object containing additional context like firmware version, battery level, signal strength, sensor manufacturer, sensor model.
  • quality: Object containing quality indicators and flags (good, suspect, bad, calibration, maintenance_required).
  • calibration: Object containing calibration information including last calibration date, calibration method, calibration parameters.
{ "version": "1.0.0", "standard": "WIA-ENE-027", "deviceId": "ENV-AIR-00F3A2B5", "timestamp": "2024-12-26T10:30:00.000Z", "sensorType": "air_quality", "location": { "latitude": 37.5665, "longitude": 126.9780, "altitude": 38.5, "accuracy": 5.0 }, "metadata": { "manufacturer": "AirSense Corp", "model": "AS-3000", "firmware": "v2.4.1", "battery": 85, "signalStrength": -65 }, "quality": { "overall": "good", "flags": [] }, "readings": { // Sensor-specific measurements } }

4.3 Air Quality Sensor Data Format

Air quality sensors measure atmospheric pollutants and conditions. The WIA standard defines comprehensive schemas for various air quality parameters.

Particulate Matter Measurements

Particulate matter sensors measure the concentration of airborne particles of different sizes. The standard supports PM1.0, PM2.5, PM4.0, and PM10 measurements.

{ "version": "1.0.0", "standard": "WIA-ENE-027", "deviceId": "ENV-AIR-001", "timestamp": "2024-12-26T10:30:00Z", "sensorType": "air_quality", "readings": { "pm1_0": { "value": 8.5, "unit": "μg/m³", "method": "laser_scattering", "uncertainty": 1.5 }, "pm2_5": { "value": 15.3, "unit": "μg/m³", "method": "laser_scattering", "uncertainty": 2.0 }, "pm10": { "value": 22.7, "unit": "μg/m³", "method": "laser_scattering", "uncertainty": 3.0 }, "aqi": { "value": 58, "category": "moderate", "pollutant": "pm2_5" } } }

Gaseous Pollutants

Gas sensors measure concentrations of various atmospheric gases. Each gas measurement includes value, unit, detection method, and uncertainty.

Parameter Unit Typical Range Detection Method
CO2 (Carbon Dioxide) ppm 400-2000 NDIR (Non-Dispersive Infrared)
CO (Carbon Monoxide) ppm 0-50 Electrochemical
NO2 (Nitrogen Dioxide) ppb 0-200 Electrochemical
O3 (Ozone) ppb 0-150 Electrochemical/UV absorption
SO2 (Sulfur Dioxide) ppb 0-100 Electrochemical
VOC (Volatile Organic Compounds) ppb 0-1000 Metal oxide semiconductor (MOS)

4.4 Water Quality Sensor Data Format

Water quality sensors monitor physical, chemical, and biological properties of water. The standard defines schemas for common water quality parameters used in drinking water monitoring, wastewater treatment, aquaculture, and environmental assessment.

{ "version": "1.0.0", "standard": "WIA-ENE-027", "deviceId": "ENV-WATER-001", "timestamp": "2024-12-26T10:30:00Z", "sensorType": "water_quality", "location": { "latitude": 37.5665, "longitude": 126.9780, "waterBody": "Han River", "samplingDepth": 1.5 }, "readings": { "ph": { "value": 7.2, "unit": "pH", "method": "glass_electrode", "temperature_compensated": true, "uncertainty": 0.05 }, "dissolved_oxygen": { "value": 8.5, "unit": "mg/L", "method": "optical", "saturation": 92.3, "uncertainty": 0.2 }, "turbidity": { "value": 12.5, "unit": "NTU", "method": "nephelometric", "uncertainty": 1.0 }, "conductivity": { "value": 450, "unit": "μS/cm", "temperature": 20.0, "specific_conductance": 448, "uncertainty": 5.0 }, "temperature": { "value": 18.5, "unit": "°C", "uncertainty": 0.1 }, "orp": { "value": 220, "unit": "mV", "uncertainty": 10 } } }

4.5 Soil Sensor Data Format

Soil sensors measure parameters critical for agriculture, land management, and environmental monitoring. The standard supports soil moisture, temperature, electrical conductivity (salinity), and nutrient measurements.

Soil Moisture and Temperature

Soil moisture is typically expressed as volumetric water content (VWC) representing the ratio of water volume to total soil volume. Temperature affects many soil processes and is usually measured alongside moisture.

{ "version": "1.0.0", "standard": "WIA-ENE-027", "deviceId": "ENV-SOIL-001", "timestamp": "2024-12-26T10:30:00Z", "sensorType": "soil", "location": { "latitude": 37.5665, "longitude": 126.9780, "depth": 0.15, "soilType": "loamy_sand" }, "readings": { "moisture": { "value": 28.5, "unit": "%VWC", "method": "capacitance", "uncertainty": 2.0 }, "temperature": { "value": 22.3, "unit": "°C", "uncertainty": 0.3 }, "electrical_conductivity": { "value": 320, "unit": "μS/cm", "uncertainty": 15 }, "nutrients": { "nitrogen": { "value": 45, "unit": "ppm", "form": "nitrate_NO3", "uncertainty": 5 }, "phosphorus": { "value": 12, "unit": "ppm", "form": "available_P", "uncertainty": 2 }, "potassium": { "value": 180, "unit": "ppm", "form": "exchangeable_K", "uncertainty": 10 } } } }

4.7 Radiation and Noise Sensor Formats

Radiation and noise sensors serve specialized monitoring needs in environmental safety and urban planning.

Radiation Sensors

Radiation sensors detect ionizing radiation from natural and artificial sources. The standard supports measurements of alpha, beta, gamma, and neutron radiation.

{ "version": "1.0.0", "standard": "WIA-ENE-027", "deviceId": "ENV-RAD-001", "timestamp": "2024-12-26T10:30:00Z", "sensorType": "radiation", "readings": { "dose_rate": { "value": 0.12, "unit": "μSv/h", "type": "gamma", "detector": "geiger_mueller", "uncertainty": 0.02 }, "counts_per_minute": { "value": 45, "unit": "CPM", "background_subtracted": true } } }

Noise Level Sensors

Noise sensors measure sound levels, critical for urban planning and regulatory compliance. Measurements use various weighting curves (A, C, Z) to match different applications.

{ "version": "1.0.0", "standard": "WIA-ENE-027", "deviceId": "ENV-NOISE-001", "timestamp": "2024-12-26T10:30:00Z", "sensorType": "noise", "readings": { "sound_level": { "instantaneous": { "value": 65.5, "unit": "dB(A)", "weighting": "A" }, "leq": { "value": 62.3, "unit": "dB(A)", "period": 3600, "uncertainty": 1.5 }, "lmax": { "value": 78.2, "unit": "dB(A)" }, "lmin": { "value": 48.7, "unit": "dB(A)" } }, "frequency_spectrum": { "octave_bands": [ {"frequency": 63, "level": 55.2}, {"frequency": 125, "level": 58.5}, {"frequency": 250, "level": 60.1}, {"frequency": 500, "level": 62.8}, {"frequency": 1000, "level": 64.5}, {"frequency": 2000, "level": 61.2}, {"frequency": 4000, "level": 57.8}, {"frequency": 8000, "level": 52.3} ] } } }

4.8 Validation Rules and Data Quality

The WIA standard includes comprehensive validation rules that data should satisfy. These rules enable automated quality checking and ensure data reliability.

Range Validation

All measurements must fall within physically plausible ranges. Out-of-range values indicate sensor malfunction or communication errors.

Parameter Minimum Maximum Notes
Temperature (ambient) -50°C 60°C Typical environmental range
Relative Humidity 0% 100% Physical limits
PM2.5 0 μg/m³ 1000 μg/m³ Extreme pollution events
pH 0 14 Theoretical pH scale
Dissolved Oxygen 0 mg/L 20 mg/L Supersaturation possible

Temporal Consistency

Rate-of-change validation detects unrealistic rapid changes that likely indicate errors rather than genuine environmental variability. For example, ambient temperature shouldn't change more than 10°C per hour under normal circumstances.

Cross-Parameter Validation

Some parameters have expected relationships. Relative humidity and dew point should be consistent with temperature. PM2.5 should generally be less than or equal to PM10. These cross-checks identify inconsistent data.

4.9 Extensibility and Custom Fields

While the standard defines common fields, real-world deployments often require additional information. The standard supports extensibility through custom fields while maintaining core compatibility.

Custom fields should use vendor-specific prefixes to avoid namespace conflicts (e.g., "acme_proprietary_field"). The presence of unknown fields should not prevent parsing of standard fields. This approach enables vendors to add value while ensuring basic interoperability.

{ "version": "1.0.0", "standard": "WIA-ENE-027", "deviceId": "ENV-AIR-001", "timestamp": "2024-12-26T10:30:00Z", "sensorType": "air_quality", "readings": { "pm2_5": { "value": 15.3, "unit": "μg/m³" } }, "acme_calibration_curve": { "slope": 1.02, "intercept": -0.5, "r_squared": 0.995 }, "acme_last_maintenance": "2024-11-15T09:00:00Z" }
Best Practice: Metadata is Critical

While it's tempting to focus only on measurement values, comprehensive metadata is essential for data quality and usability. Always include location information for spatial analysis, calibration data for quality assessment, sensor specifications for interpretation, and quality flags for appropriate use. Rich metadata transforms raw numbers into scientifically valuable information that can be confidently used for decision-making.

Chapter Summary

Chapter 4 detailed Phase 1 of the WIA standard—Data Format standardization. We explored the core data model structure common to all sensor types, examined specific schemas for air quality, water quality, soil, meteorological, radiation, and noise sensors, and discussed validation rules and extensibility mechanisms that ensure data quality while supporting diverse deployment needs.

Key Takeaways

  • All WIA-compliant sensor data shares a common core structure with mandatory fields (version, standard, deviceId, timestamp, sensorType) and optional but recommended fields (location, metadata, quality, calibration) that provide essential context.
  • Sensor-specific schemas extend the core model with parameters appropriate to each sensor category, using consistent field naming, standard units, and uncertainty quantification to enable meaningful comparison across sensors.
  • Validation rules including range checks, temporal consistency validation, and cross-parameter verification enable automated quality assurance and detect sensor malfunctions or communication errors.
  • The standard uses JSON as the normative format for human readability and universal tool support while allowing equivalent representations in other formats (XML, Protocol Buffers, CBOR) for specialized applications.
  • Extensibility mechanisms using vendor-specific prefixes enable proprietary extensions while maintaining core interoperability, balancing standardization benefits with flexibility for innovation and specific deployment requirements.
