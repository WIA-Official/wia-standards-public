CHAPTER 05

Phase 2 - API Interface

5.1 Introduction to WIA API Standards

Phase 2 of the WIA Environmental Sensor Standard defines standardized Application Programming Interfaces (APIs) for accessing environmental sensor data and controlling sensor devices. While Phase 1 standardizes data format, Phase 2 standardizes how applications discover sensors, retrieve data, configure devices, and receive real-time updates. This API standardization transforms interoperability from a data-level concept to an operational reality.

The WIA API standard adopts RESTful (Representational State Transfer) design principles, leveraging HTTP as the underlying protocol. REST was chosen because it is universally supported, scales well from small to massive deployments, is cacheable for performance, and has extensive tooling and developer familiarity. The API standard also specifies real-time streaming interfaces using WebSockets and Server-Sent Events for applications requiring immediate data delivery.

A key design goal is consistency—applications interacting with one WIA-compliant API can immediately work with others. URL structures, query parameters, request/response formats, error handling, and authentication mechanisms are all standardized. This consistency dramatically reduces integration effort and enables developers to build applications that work with diverse environmental sensor platforms without custom per-platform code.

5.2 API Architecture and Principles

The WIA API architecture follows several core principles that guide all aspects of the specification.

Resource-Oriented Design

The API is resource-oriented, treating sensors, sensor data, locations, and other entities as resources accessible via URLs. Each resource has a canonical URL structure. For example, a specific sensor might be accessed at /api/v1/sensors/{sensorId}, while that sensor's data is accessed at /api/v1/sensors/{sensorId}/data. This intuitive, hierarchical structure makes APIs self-documenting and easy to learn.

Standard HTTP Methods

The API uses standard HTTP methods semantically: GET for retrieving resources (sensors, data), POST for creating resources (registering sensors, submitting commands), PUT for updating resources (sensor configuration), DELETE for removing resources (deregistering sensors). This alignment with HTTP semantics leverages existing infrastructure and developer knowledge.

Stateless Operations

API operations are stateless—each request contains all information needed to process it. Servers don't maintain session state between requests. This statelessness enables horizontal scaling, simplifies server implementation, and improves reliability through idempotent operations that can be safely retried.

Content Negotiation

The API supports content negotiation through standard HTTP headers. Clients specify desired response formats using Accept headers (application/json, application/xml, application/cbor). Servers indicate response format using Content-Type headers. This flexibility accommodates different client capabilities and preferences while maintaining a single API endpoint.

Versioning

API versioning is explicit in the URL path (/api/v1/, /api/v2/). This approach makes versioning transparent and allows multiple API versions to coexist during migration periods. Major version changes indicate breaking changes, while minor and patch versions maintain backward compatibility.

Principle Implementation Benefits
Resource-Oriented URL hierarchy, RESTful paths Intuitive navigation, self-documenting
HTTP Methods GET/POST/PUT/DELETE semantics Leverages HTTP infrastructure, familiar to developers
Stateless Complete requests, no server session Scalability, reliability, simplicity
Content Negotiation Accept/Content-Type headers Format flexibility, single endpoint
Versioning Path-based (/api/v1/) Transparent, migration-friendly

5.3 Core API Endpoints

The WIA API defines a set of core endpoints that all compliant implementations must support. These endpoints enable essential operations for sensor discovery, data retrieval, and device management.

Sensor Discovery and Registry

Sensor discovery enables applications to find available sensors and understand their capabilities.

GET /api/v1/sensors Returns list of all sensors accessible to the authenticated user Query parameters: - type: Filter by sensor type (air_quality, water_quality, etc.) - location: Geographic bounding box (lat1,lon1,lat2,lon2) - status: Filter by status (active, inactive, maintenance) - limit: Maximum results to return (default 100, max 1000) - offset: Pagination offset Response: { "total": 1547, "count": 100, "offset": 0, "sensors": [ { "deviceId": "ENV-AIR-001", "type": "air_quality", "location": {"latitude": 37.5665, "longitude": 126.9780}, "status": "active", "lastUpdate": "2024-12-26T10:30:00Z", "capabilities": ["pm2_5", "pm10", "temperature", "humidity"] }, // ... more sensors ], "links": { "next": "/api/v1/sensors?offset=100", "self": "/api/v1/sensors" } }

Retrieving detailed information about a specific sensor.

5.4 Data Retrieval Endpoints

Data retrieval endpoints provide flexible querying of historical and recent sensor measurements.

Latest Data

Retrieve the most recent measurement from a sensor.

GET /api/v1/sensors/{sensorId}/data/latest Response: { "version": "1.0.0", "standard": "WIA-ENE-027", "deviceId": "ENV-AIR-001", "timestamp": "2024-12-26T10:30:00Z", "sensorType": "air_quality", "readings": { "pm2_5": {"value": 15.3, "unit": "μg/m³"}, "pm10": {"value": 22.7, "unit": "μg/m³"}, "temperature": {"value": 23.5, "unit": "°C"}, "humidity": {"value": 65, "unit": "%RH"} }, "quality": {"overall": "good", "flags": []}, "location": { "latitude": 37.5665, "longitude": 126.9780 } }

Historical Data with Time Range Query

Retrieve historical data for a specified time period with flexible aggregation options.

GET /api/v1/sensors/{sensorId}/data Query parameters: - start: ISO 8601 timestamp (e.g., 2024-12-25T00:00:00Z) - end: ISO 8601 timestamp - parameters: Comma-separated list (e.g., pm2_5,temperature) - aggregation: none, hourly, daily, weekly, monthly - format: json, csv, xml Response (with aggregation=hourly): { "deviceId": "ENV-AIR-001", "start": "2024-12-25T00:00:00Z", "end": "2024-12-26T00:00:00Z", "aggregation": "hourly", "data": [ { "timestamp": "2024-12-25T00:00:00Z", "pm2_5": { "mean": 12.5, "min": 8.2, "max": 18.3, "stddev": 2.1, "count": 60 }, "temperature": { "mean": 22.1, "min": 20.5, "max": 23.8 } }, // ... more hourly aggregates ] }

Bulk Data Retrieval

Retrieve data from multiple sensors simultaneously.

POST /api/v1/data/bulk Request body: { "sensorIds": ["ENV-AIR-001", "ENV-AIR-002", "ENV-AIR-003"], "start": "2024-12-26T00:00:00Z", "end": "2024-12-26T12:00:00Z", "parameters": ["pm2_5", "temperature"], "aggregation": "hourly" } Response: { "sensors": [ { "deviceId": "ENV-AIR-001", "data": [/* aggregated data */] }, { "deviceId": "ENV-AIR-002", "data": [/* aggregated data */] } ] }

5.5 Real-Time Streaming Interfaces

For applications requiring immediate data delivery, the WIA standard specifies real-time streaming interfaces using WebSockets and Server-Sent Events (SSE).

WebSocket Streaming

WebSocket connections provide full-duplex communication for real-time data streaming and device control.

// WebSocket connection ws://api.example.com/v1/stream // Subscribe to sensor(s) { "action": "subscribe", "sensors": ["ENV-AIR-001", "ENV-AIR-002"], "parameters": ["pm2_5", "temperature"] } // Server sends data as it arrives { "type": "measurement", "deviceId": "ENV-AIR-001", "timestamp": "2024-12-26T10:30:00Z", "readings": { "pm2_5": {"value": 15.3, "unit": "μg/m³"} } } // Heartbeat messages maintain connection { "type": "ping", "timestamp": "2024-12-26T10:35:00Z" }

Server-Sent Events (SSE)

SSE provides one-way server-to-client streaming over HTTP, simpler than WebSockets for read-only scenarios.

GET /api/v1/sensors/ENV-AIR-001/stream Accept: text/event-stream Response (streaming): event: measurement data: {"timestamp":"2024-12-26T10:30:00Z","pm2_5":15.3} event: measurement data: {"timestamp":"2024-12-26T10:31:00Z","pm2_5":15.5} event: heartbeat data: {"timestamp":"2024-12-26T10:35:00Z"}

5.6 Authentication and Authorization

Security is fundamental to the WIA API standard. Multiple authentication mechanisms accommodate different security requirements and deployment scenarios.

API Keys

API keys provide simple authentication suitable for server-to-server communication and low-security applications.

GET /api/v1/sensors Authorization: Bearer YOUR_API_KEY_HERE // Or via query parameter (less secure) GET /api/v1/sensors?api_key=YOUR_API_KEY_HERE

OAuth 2.0

OAuth 2.0 enables secure delegated access, ideal for third-party applications accessing user data.

// Authorization Code Flow 1. Redirect user to authorization endpoint: GET /oauth/authorize?client_id=...&response_type=code&redirect_uri=... 2. Exchange authorization code for access token: POST /oauth/token { "grant_type": "authorization_code", "code": "AUTH_CODE", "client_id": "CLIENT_ID", "client_secret": "CLIENT_SECRET" } 3. Use access token: GET /api/v1/sensors Authorization: Bearer ACCESS_TOKEN

JWT (JSON Web Tokens)

JWTs provide stateless authentication with embedded claims, reducing server-side session management.

GET /api/v1/sensors Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9... // JWT payload includes: { "sub": "user@example.com", "iss": "https://auth.wia.io", "exp": 1704067200, "permissions": ["sensors:read", "data:read"] }

Role-Based Access Control (RBAC)

The API supports role-based permissions governing what operations users can perform.

Role Permissions Typical Use Case
Public Viewer Read public sensor data Open data platforms, citizen access
Data Consumer Read all sensor data, metadata Researchers, analysts
Sensor Operator Read data, update sensor config, submit commands Field technicians, maintenance staff
Administrator Full access including sensor registration, user management Platform administrators

5.7 Error Handling and Status Codes

Consistent error handling enables robust applications. The WIA API uses standard HTTP status codes with detailed error messages.

Standard HTTP Status Codes

  • 200 OK: Successful request
  • 201 Created: Resource successfully created
  • 204 No Content: Successful request with no response body
  • 400 Bad Request: Invalid request syntax or parameters
  • 401 Unauthorized: Authentication required or failed
  • 403 Forbidden: Authenticated but insufficient permissions
  • 404 Not Found: Requested resource doesn't exist
  • 429 Too Many Requests: Rate limit exceeded
  • 500 Internal Server Error: Server-side error
  • 503 Service Unavailable: Server temporarily unavailable

Error Response Format

{ "error": { "code": "SENSOR_NOT_FOUND", "message": "Sensor with ID 'ENV-AIR-999' not found", "details": { "requestedId": "ENV-AIR-999", "suggestion": "Check sensor ID or permissions" }, "timestamp": "2024-12-26T10:30:00Z", "requestId": "req_abc123def456" } }

To ensure fair resource usage and system stability, the API implements rate limiting.

5.9 Pagination and Performance

Large datasets require efficient pagination to maintain performance and usability.

Offset-Based Pagination

GET /api/v1/sensors?limit=100&offset=200 Response: { "total": 1547, "count": 100, "offset": 200, "limit": 100, "sensors": [/* ... */], "links": { "first": "/api/v1/sensors?limit=100&offset=0", "prev": "/api/v1/sensors?limit=100&offset=100", "self": "/api/v1/sensors?limit=100&offset=200", "next": "/api/v1/sensors?limit=100&offset=300", "last": "/api/v1/sensors?limit=100&offset=1500" } }

Cursor-Based Pagination

For real-time data streams, cursor-based pagination provides stable pagination even as data changes.

GET /api/v1/sensors/ENV-AIR-001/data?limit=100 Response: { "data": [/* ... */], "cursor": { "next": "eyJpZCI6MTIzNDU2LCJ0aW1lIjoiMjAyNC0xMi0yNlQxMDozMDowMFoifQ==", "hasMore": true } } // Next page GET /api/v1/sensors/ENV-AIR-001/data?limit=100&cursor=eyJ...
Best Practice: API Client Design

When building applications using WIA APIs, implement proper error handling and retry logic with exponential backoff, respect rate limits and use caching where appropriate, use streaming interfaces for real-time requirements rather than polling, implement timeout handling for network resilience, and log request IDs for troubleshooting. Well-designed API clients are robust to network issues and server-side problems.

Chapter Summary

Chapter 5 explored Phase 2 of the WIA standard—API Interface specification. We examined RESTful API design principles, core endpoints for sensor discovery and data retrieval, real-time streaming using WebSockets and SSE, authentication mechanisms, error handling, and performance optimization through pagination and rate limiting.

Key Takeaways

  • WIA APIs follow REST principles with resource-oriented design, standard HTTP methods, stateless operations, and explicit versioning, enabling consistent interaction across all compliant platforms.
  • Core endpoints support sensor discovery (GET /sensors), detailed sensor information (GET /sensors/{id}), latest and historical data retrieval with flexible querying and aggregation options.
  • Real-time streaming via WebSockets and Server-Sent Events provides immediate data delivery for time-critical applications, complementing request-response APIs for different use cases.
  • Multiple authentication mechanisms (API keys, OAuth 2.0, JWT) and role-based access control support diverse security requirements from public data access to enterprise deployments.
  • Standardized error handling, rate limiting, and pagination ensure robust, performant applications while protecting server resources and maintaining quality of service for all users.
