# 3장: 표준 개요 및 아키텍처

## 환경 센서 상호운용성을 위한 WIA-ENE-027 프레임워크

---

## 3.1 WIA-ENE-027 설계 철학

WIA-ENE-027 환경 센서 표준은 설계 및 구현을 안내하는 여러 핵심 원칙을 구현합니다.

### 과학적 엄격성

모든 사양은 확립된 과학적 관행 및 동료 심사를 거친 방법론에 기반합니다:
- **측정 표준**: EPA, NOAA, USGS 및 국제 지침 기반
- **품질 보증**: 규제 기관의 QA/QC 프레임워크
- **검증 방법**: 검증된 교정 및 검증 절차
- **단위 및 용어**: 표준 과학 단위 (SI 및 승인된 대안)

### 운영 실용성

표준은 과학적 정교함과 실제 구현의 균형을 맞춥니다:
- **리소스 제약**: 전력, 대역폭 및 비용 제한 인정
- **배포 현실**: 이상적 실험실이 아닌 실제 현장 조건을 위해 설계
- **유지보수 실현 가능성**: 특수 장비 없이 달성 가능한 교정 절차
- **확장성**: 10개 또는 10,000개 센서에 작동

### 벤더 중립성

표준은 어떤 제조사나 기술도 선호하지 않습니다:
- **개방형 사양**: 공개적으로 사용 가능, 독점 요소 없음
- **기술 불가지론**: 성능 요구사항을 충족하는 모든 센서와 작동
- **프로토콜 유연성**: MQTT, CoAP, LoRaWAN, HTTP 지원
- **클라우드 독립성**: 모든 클라우드 제공자 사용 또는 자체 호스팅 가능

### 미래 지향적 확장성

아키텍처는 진화와 새로운 기능을 수용합니다:
- **버전 관리**: 시맨틱 버전 관리로 하위 호환성 보장
- **확장 지점**: 기존 구현을 깨지 않고 새 센서 유형 추가 가능
- **프로토콜 진화**: 새로운 통신 기술 채택 가능
- **분석 통합**: 머신러닝 및 고급 분석을 위해 설계

## 3.2 4단계 표준 아키텍처

WIA-ENE-027 표준은 WIA 4단계 아키텍처를 따르며, 각 단계는 이전 계층 위에 구축됩니다:

```
┌────────────────────────────────────────────────────────────────┐
│                  WIA-ENE-027 아키텍처                          │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  PHASE 1: 데이터 형식 사양                                     │
│  ├── 핵심 데이터 모델 (모든 센서 유형)                        │
│  ├── 대기질 센서 스키마 (PM, 가스, AQI)                       │
│  ├── 수질 센서 스키마 (pH, DO, 탁도)                          │
│  ├── 토양 센서 스키마 (수분, 온도, 영양소)                    │
│  ├── 기상 스키마 (온도, 습도, 풍속, 강수)                     │
│  ├── 메타데이터 및 품질 플래그 정의                           │
│  └── JSON 스키마 검증 사양                                     │
│                                                                │
│  PHASE 2: API 인터페이스 사양                                  │
│  ├── RESTful 엔드포인트 설계                                   │
│  ├── 센서 탐색 및 등록                                        │
│  ├── 데이터 제출 (POST /sensors/{id}/data)                   │
│  ├── 데이터 검색 및 쿼리                                      │
│  ├── 실시간 스트리밍 (WebSocket, SSE)                         │
│  ├── 인증 및 권한 부여                                        │
│  └── 오류 처리 및 페이지네이션                                │
│                                                                │
│  PHASE 3: 프로토콜 사양                                        │
│  ├── MQTT 토픽 구조 및 QoS                                    │
│  ├── CoAP 리소스 경로 및 관찰                                 │
│  ├── LoRaWAN 페이로드 인코딩/디코딩                           │
│  ├── HTTP/REST 통신 패턴                                      │
│  ├── TLS/DTLS 보안 요구사항                                   │
│  ├── 데이터 검증 및 품질 관리                                 │
│  └── 엣지 컴퓨팅 및 게이트웨이 패턴                           │
│                                                                │
│  PHASE 4: 시스템 통합                                          │
│  ├── 클라우드 플랫폼 통합 (AWS, Azure, GCP)                  │
│  ├── 시계열 데이터베이스 통합                                 │
│  ├── 분석 및 시각화 플랫폼                                    │
│  ├── 규제 보고 자동화                                         │
│  ├── 다른 WIA 표준과 통합                                     │
│  ├── 연합 네트워크 데이터 공유                                │
│  └── 참조 배포 아키텍처                                       │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 단계 종속성

- **Phase 1**은 기초 - 모든 구현이 준수해야 함
- **Phase 2**는 Phase 1 준수 필요 - API는 Phase 1 형식의 데이터 전송
- **Phase 3**은 Phase 1과 2 필요 - 프로토콜은 Phase 2 인터페이스를 통해 Phase 1 데이터 전송
- **Phase 4**는 Phase 1-3 필요 - 통합은 준수 데이터, API 및 프로토콜 가정

### 준수 수준

조직은 다양한 준수 수준을 달성할 수 있습니다:

| 수준 | 요구사항 | 사용 사례 |
|------|---------|----------|
| **Phase 1 준수** | 데이터 형식만 | 센서 제조사, 데이터 제공자 |
| **Phase 2 준수** | 데이터 형식 + API | 애플리케이션 개발자, 플랫폼 제공자 |
| **Phase 3 준수** | 데이터 + API + 프로토콜 | 완전한 센서 시스템, IoT 플랫폼 |
| **Phase 4 준수** | 모든 단계 + 통합 | 엔터프라이즈 배포, 스마트 시티 |

## 3.3 센서 분류 프레임워크

표준은 센서를 계층적 분류 시스템으로 조직합니다:

### 센서 유형 계층

```
환경 센서
├── 대기질
│   ├── 미세먼지 (PM1.0, PM2.5, PM4.0, PM10)
│   ├── 가스 오염물질 (CO2, CO, NO2, O3, SO2, VOC)
│   └── 대기질 지수 (AQI)
├── 수질
│   ├── 물리적 매개변수 (pH, 탁도, 온도)
│   ├── 화학적 매개변수 (DO, 전도도, ORP)
│   └── 영양소 매개변수 (질산염, 인산염 등)
├── 토양
│   ├── 물리적 매개변수 (수분, 온도, EC)
│   └── 영양소 매개변수 (N, P, K)
└── 기상
    ├── 대기 (온도, 습도, 기압)
    ├── 풍속 (속도, 방향)
    ├── 강수 (강우, 눈)
    └── 태양 (복사, UV 지수)
```

### 센서 유형 식별자

`sensorType` 필드의 표준 센서 유형 식별자:

| 센서 유형 | 식별자 | 주요 매개변수 |
|-----------|--------|--------------|
| 대기질 | `"air_quality"` | PM2.5, PM10, CO2, NO2, O3 |
| 수질 | `"water_quality"` | pH, DO, 탁도, 전도도 |
| 토양 | `"soil"` | 수분, 온도, EC, 영양소 |
| 기상 | `"meteorological"` | 온도, 습도, 기압, 풍속 |
| 복사 | `"radiation"` | 태양 복사, UV 지수 |
| 소음 | `"noise"` | 음압 수준, 주파수 |

## 3.4 핵심 데이터 모델 설계

모든 WIA-ENE-027 준수 센서 데이터는 공통 핵심 구조를 공유합니다.

### 필수 핵심 필드

모든 센서 데이터 메시지는 반드시 포함해야 합니다:

```json
{
  "version": "1.0.0",
  "standard": "WIA-ENE-027",
  "deviceId": "ENV-AIR-001",
  "timestamp": "2025-01-09T10:30:00.000Z",
  "sensorType": "air_quality",
  "readings": {
    // 센서별 측정
  }
}
```

**필드 사양:**

- **version**: 시맨틱 버전 (MAJOR.MINOR.PATCH)
  - MAJOR: 호환성 깨는 변경
  - MINOR: 하위 호환 추가
  - PATCH: 하위 호환 수정

- **standard**: 이 표준의 경우 항상 `"WIA-ENE-027"`

- **deviceId**: 전역 고유 식별자
  - 형식: `[PREFIX]-[TYPE]-[IDENTIFIER]`
  - 예시: `ENV-AIR-001`, `WATER-PH-SITE42`
  - 센서의 모든 메시지에서 일관되어야 함

- **timestamp**: 시간대 포함 ISO 8601 형식
  - 시간대 포함 필수 (UTC의 경우 Z 또는 ±HH:MM)
  - 정밀도: 밀리초 권장
  - 예시: `"2025-01-09T10:30:00.000Z"`

- **sensorType**: 분류의 표준 식별자
  - 정의된 센서 유형 중 하나여야 함
  - 예상 `readings` 구조 결정

- **readings**: 센서별 측정
  - 각 센서 유형에 대해 정의된 구조
  - 실제 측정 데이터 포함

### 선택적 권장 필드

포괄적인 데이터 품질을 위해 매우 권장됨:

**위치:**
```json
"location": {
  "latitude": 37.5665,
  "longitude": 126.9780,
  "altitude": 38.5,
  "accuracy": 10.0,
  "datum": "WGS84"
}
```

**메타데이터:**
```json
"metadata": {
  "manufacturer": "AirSense Corp",
  "model": "AS-3000",
  "firmware": "v2.4.1",
  "battery": 85,
  "signalStrength": -65,
  "uptime": 86400
}
```

**품질:**
```json
"quality": {
  "overall": "good",
  "flags": [],
  "confidence": 0.95
}
```

**교정:**
```json
"calibration": {
  "lastCalibration": "2024-11-15T09:00:00Z",
  "nextCalibration": "2025-05-15T09:00:00Z",
  "method": "reference_colocated",
  "parameters": {"slope": 1.02, "intercept": -0.5},
  "certificateId": "CAL-2024-1234"
}
```

## 3.5 시스템 아키텍처 구성요소

완전한 WIA-ENE-027 준수 시스템은 여러 구성요소로 구성됩니다:

### 구성요소 아키텍처

```
┌─────────────┐
│   센서      │ (Phase 1: 데이터 형식)
└──────┬──────┘
       │ MQTT/CoAP/LoRaWAN/HTTP
       ↓
┌─────────────┐
│  게이트웨이  │ (Phase 3: 프로토콜)
└──────┬──────┘
       │ API 호출
       ↓
┌─────────────┐
│  API 계층   │ (Phase 2: REST API)
└──────┬──────┘
       │
       ├→┌──────────────┐
       │ │시계열 DB     │ (InfluxDB, Timestream)
       │ └──────────────┘
       │
       ├→┌──────────────┐
       │ │분석          │ (ML, 이상 탐지)
       │ └──────────────┘
       │
       └→┌──────────────┐
         │시각화        │ (대시보드, 앱)
         └──────────────┘
```

### 센서 계층 (Phase 1)

**책임:**
- Phase 1 데이터 형식에 따라 측정 생성
- 기본 품질 검사 적용 (범위 검증)
- 교정 메타데이터 포함
- 측정 정확하게 타임스탬프 지정

**예시 센서:**
- 대기질: PurpleAir, AirGradient, Clarity
- 수질: Atlas Scientific, YSI EXO
- 토양: METER Group, Stevens Water
- 기상: Davis Vantage, Campbell Scientific

### 게이트웨이 계층 (Phase 3)

**책임:**
- 프로토콜 변환 (MQTT/CoAP/LoRaWAN → HTTP/API)
- 연결 손실 시 데이터 버퍼링
- 엣지 처리 및 필터링
- 로컬 품질 검증
- 보안 및 인증

**예시 게이트웨이:**
- IoT 게이트웨이: AWS Greengrass, Azure IoT Edge
- LoRaWAN: The Things Gateway, RAK Wireless
- 커스텀: MQTT/HTTP 브리지가 있는 Raspberry Pi

### API 계층 (Phase 2)

**책임:**
- 데이터 제출/검색을 위한 RESTful 엔드포인트
- 인증 및 권한 부여
- JSON 스키마에 대한 데이터 검증
- 속도 제한 및 할당량 관리
- 실시간 스트리밍 인터페이스

**예시 플랫폼:**
- 클라우드: AWS IoT Core, Azure IoT Hub, Google Cloud IoT
- 자체 호스팅: FastAPI, Express.js, Spring Boot
- 오픈 소스: ThingsBoard, Grafana, Home Assistant

### 저장소 계층 (Phase 4)

**책임:**
- 시계열 센서 데이터 지속
- 시간, 위치, 센서별 효율적인 쿼리 활성화
- 집계 및 다운샘플링 지원
- 장기 아카이빙

**예시 데이터베이스:**
- 시계열: InfluxDB, TimescaleDB, AWS Timestream
- 문서: MongoDB, JSONB가 있는 PostgreSQL
- 객체 저장소: AWS S3, Google Cloud Storage

### 애플리케이션 계층 (Phase 4)

**책임:**
- 시각화 대시보드
- 알림 및 통지 시스템
- 분석 및 보고
- 타사 시스템과 통합
- 모바일 애플리케이션

**예시 애플리케이션:**
- Grafana: 실시간 대시보드
- 커스텀 웹 앱: React, Vue.js
- 모바일 앱: iOS, Android
- 분석: Python/Jupyter, Apache Spark

## 3.6 품질 보증 프레임워크

표준은 포괄적인 품질 보증 프레임워크를 확립합니다:

### 품질 플래그 시스템

데이터 조건을 나타내는 표준 품질 플래그:

| 플래그 | 의미 | 심각도 | 조치 |
|------|------|--------|------|
| `calibration_due` | 교정 기한 도래 | 경고 | 곧 교정 일정 잡기 |
| `calibration_overdue` | 교정 기한 크게 초과 | 심각 | 주의하여 데이터 사용 |
| `out_of_range` | 값이 타당한 범위 초과 | 심각 | 센서 조사 |
| `rapid_change` | 비현실적 변화율 | 경고 | 오작동 확인 |
| `low_battery` | 배터리 20% 미만 | 경고 | 곧 배터리 교체 |
| `critical_battery` | 배터리 10% 미만 | 심각 | 즉시 교체 |
| `poor_signal` | 연결 약함 | 경고 | 네트워크 확인 |
| `sensor_fault` | 센서 자가 진단 실패 | 심각 | 서비스 필요 |
| `maintenance_required` | 서비스 간격 초과 | 경고 | 유지보수 일정 잡기 |
| `estimated` | 값 추정/보간됨 | 정보 | 신뢰도 낮음 |

### 품질 수준

전반적인 품질 평가:

| 수준 | 설명 | 기준 | 사용 사례 |
|------|------|------|----------|
| `"good"` | 고품질 데이터 | 교정됨, 범위 내, 플래그 없음 | 모든 응용 |
| `"suspect"` | 의심스러운 품질 | 부 플래그, 경계선 값 | 주의하여 사용 |
| `"bad"` | 불량 품질 데이터 | 주요 플래그, 범위 밖 | 의사결정에 사용 금지 |
| `"missing"` | 데이터 없음 | 센서 오프라인, 전송 실패 | 기록 공백 |

### 검증 절차

**범위 검증:**
- 물리적으로 타당한 범위에 대한 측정 확인
- 예상 범위 밖 값 플래그 지정
- 환경 맥락 고려 (예: 사막에서 더 높은 PM10)

**시간 검증:**
- 불가능한 변화율 탐지
- 고장난 센서 식별 (변동 없음)
- 타임스탬프 이상 플래그 지정 (미래 날짜, 중복)

**교차 매개변수 검증:**
- PM2.5는 PM10 이하여야 함
- 이슬점 ≤ 기온
- 매개변수 간 관계 검증

## 3.7 준수 및 인증

조직은 검증을 통해 WIA-ENE-027 인증을 달성할 수 있습니다:

### 인증 수준

**레벨 1: 데이터 형식 준수**
- 요구사항:
  - 유효한 Phase 1 JSON 생성
  - 모든 필수 필드 포함
  - JSON 스키마 검증 통과
  - 품질 메타데이터 포함
- 검증: 자동화된 스키마 검증
- 감사: 샘플 데이터 검토

**레벨 2: API 준수**
- 요구사항:
  - 레벨 1 + Phase 2 API 구현
  - OpenAPI 문서가 있는 RESTful 엔드포인트
  - 적절한 인증 및 오류 처리
- 검증: API 적합성 테스트
- 감사: 통합 테스트

**레벨 3: 프로토콜 준수**
- 요구사항:
  - 레벨 2 + Phase 3 프로토콜 지원
  - MQTT/CoAP/LoRaWAN 또는 HTTP 구현
  - 보안 (TLS/DTLS)
  - 엣지 처리 기능
- 검증: 프로토콜 적합성 테스트
- 감사: 보안 검토

**레벨 4: 전체 시스템 준수**
- 요구사항:
  - 레벨 3 + Phase 4 통합
  - 클라우드 플랫폼 통합
  - 분석 기능
  - 문서 및 지원
- 검증: 종단 간 테스트
- 감사: 프로덕션 시스템 검토

3. **테스트**: 자동화 및 수동 적합성 테스트
4. **감사**: WIA 기술 위원회 검토
5. **인증**: 준수 인증서 발급
6. **등재**: WIA 준수 시스템 레지스트리에 추가
7. **유지보수**: 연간 재인증

## 3.8 복습 질문 및 핵심 요점

### 복습 질문

1. **설계 원칙**: WIA-ENE-027 표준이 과학적 엄격성과 운영 실용성의 균형을 어떻게 맞추는지 설명하세요. 각각의 예를 제시하세요.

2. **4단계 아키텍처**: Phase 1 (데이터 형식)이 다른 단계의 기초인 이유는 무엇입니까? 센서가 Phase 1을 준수하지 않으면 무엇이 깨집니까?

3. **준수 수준**: 조직이 센서 플랫폼을 구축하려고 합니다. 여러 제조사를 지원해야 하지만 프로토콜 사양은 필요하지 않습니다. 어떤 준수 수준을 목표로 해야 합니까?

4. **품질 플래그**: 대기질 센서의 품질 플래그 로직을 설계하세요. `calibration_due`, `calibration_overdue`, `out_of_range`, `rapid_change` 플래그를 언제 설정해야 합니까?

5. **시스템 구성요소**: 100개 센서가 있는 완전한 WIA-ENE-027 배포를 다이어그램으로 그리세요. 센서 계층, 게이트웨이 계층, API 계층, 저장소 및 애플리케이션을 보여주세요.

6. **인증 가치**: (a) 센서 제조사, (b) 시스템 통합자, (c) 최종 사용자에 대한 WIA-ENE-027 인증의 이점은 무엇입니까?

### 핵심 요점

1. **설계 철학**: 표준은 과학적 엄격성, 운영 실용성, 벤더 중립성 및 미래 확장성의 균형을 맞춰 실제 요구를 충족합니다.

2. **4단계 아키텍처**: 점진적 계층 (데이터 형식, API, 프로토콜, 통합)은 조직이 요구에 맞는 준수 수준을 달성하도록 합니다.

3. **Phase 1 기초**: 데이터 형식 표준화가 기초입니다 - 다른 모든 단계가 일관되고 유효한 데이터 구조에 의존합니다.

4. **센서 분류**: 계층적 조직 (대기/수질/토양/기상)과 표준 식별자는 일관된 시스템 설계를 가능하게 합니다.

5. **핵심 데이터 모델**: 모든 센서는 필수 필드 (version, standard, deviceId, timestamp, sensorType, readings)를 공유하여 기본 상호운용성을 보장합니다.

6. **선택적 메타데이터**: 위치, 품질, 교정 및 장치 메타데이터는 선택 사항이지만 포괄적인 데이터 관리를 위해 강력히 권장됩니다.

7. **시스템 구성요소**: 완전한 시스템은 표준 인터페이스를 통해 함께 작동하는 센서, 게이트웨이, API, 저장소 및 애플리케이션을 포함합니다.

8. **품질 보증**: 표준 품질 플래그 및 검증 절차는 자동화된 데이터 품질 평가 및 필터링을 가능하게 합니다.

9. **준수 수준**: 4개의 인증 수준은 조직이 생태계에서의 역할에 적합한 준수를 입증하도록 합니다.

10. **인증 프로세스**: WIA의 공식 검증 및 인증은 시스템이 진정으로 표준 요구사항을 충족하고 상호운용할 수 있음을 보장합니다.

## 장 요약

이 장에서는 WIA-ENE-027 표준 아키텍처 및 설계 원칙을 제시했습니다. 표준은 과학적 엄격성과 실용적 배포 요구의 균형을 맞추고, 벤더 중립성을 유지하며, 미래 진화를 위한 확장성을 제공합니다.

4단계 아키텍처 (데이터 형식, API, 프로토콜, 통합)는 점진적 준수를 가능하게 합니다. Phase 1 데이터 형식 표준화가 기초이며, 각 단계는 이전 계층 위에 구축됩니다. 조직은 생태계 역할에 맞는 준수 수준을 달성할 수 있습니다.

센서 분류 프레임워크는 센서를 계층적으로 조직하며 (대기질, 수질, 토양, 기상) 표준 유형 식별자가 있습니다. 핵심 데이터 모델은 모든 센서가 포함해야 하는 필수 필드와 포괄적인 데이터 품질을 위한 선택적 권장 메타데이터를 정의합니다.

완전한 시스템은 여러 계층으로 구성됩니다: Phase 1 데이터를 생성하는 센서, Phase 3 프로토콜을 처리하는 게이트웨이, Phase 2 인터페이스를 제공하는 API, 데이터를 지속하는 저장소, 사용자 가치를 제공하는 애플리케이션. 표준 품질 플래그 및 검증 절차는 자동화된 품질 평가를 가능하게 합니다.

WIA 인증은 테스트 및 감사를 통해 준수를 검증하며, 4단계에 해당하는 4개 수준이 있습니다. 인증은 상호운용성 및 표준 준수의 보증을 제공합니다.
