Chapter 04: Phase 1 — Data Format

4.1 Phase 1 in one paragraph

Phase 1 specifies JSON envelopes for precision-agriculture data. Every envelope is signed with Ed25519 (IETF RFC 8032).

4.2 Field/parcel envelope

{
  "wia_smart_farm_version": "1.0.0",
  "type": "field",
  "field_id": "fld_oconnor-farm-001",
  "farm_id": "farm_oconnor",
  "boundary": { "type": "Polygon", "coordinates": [...] },
  "area_hectares": 32.5,
  "current_crop": "maize",
  "soil_class": "Mollisol",
  "signature": { "alg": "Ed25519", "value": "..." }
}

4.3 Sensor reading envelope

Required: field_id, sensor_id, captured_at, measurement_kind, value, unit. Reserved measurement_kind tokens: soil_moisture_volumetric_pct, soil_temperature_c, soil_ec_ds_per_m, soil_ph, leaf_wetness_pct, air_temperature_c, air_humidity_pct, precipitation_mm, solar_radiation_w_per_m2, ndvi.

4.4 Irrigation event envelope

Required: field_id, started_at, ended_at, method, volume_m3, source. Reserved method tokens: drip, sprinkler, flood, micro-sprinkler, sub-surface.

4.5 Fertilisation event envelope

N/P/K rates per hectare, application method, equipment ID. Compatible with US EPA nutrient management plan reporting where required.

4.6 Harvest record envelope

Yield (t/ha), moisture pct, quality metrics (protein, test weight for grains; brix for fruit), equipment ID. Yield map embedded as GeoJSON for sub-field resolution.

4.7 Crop scouting observation envelope

Pest/disease/weed presence with geolocation. Reserved pest concepts align with FAO AgroVoc; downstream analytics can compute outbreak heatmaps from federated scouting data.

Farmer-First Design

Every choice in this standard is filtered through: does this make farm data more useful to the farmer who collected it, or just more accessible to the vendor who sold the equipment? When the two conflict, farmer sovereignty wins. That is why the standard mandates explicit farmer consent for cross-vendor data sharing, refuses to allow vendor terms-of-service to bypass the consent envelope, and requires implementations to surface vendor data flows so farmers can verify what their data is doing.

Implementer Notes

Three operational concerns recur in smart-farm infrastructure and are baked into the standard. First, sensor data volume during peak season: a 100-hectare farm with 20 soil sensors per hectare emitting at 5-minute cadence produces ~500 000 sensor readings per day. Hosts MUST provision for this peak load and SHOULD downsample to hourly aggregates after 30 days. Second, farmer consent revocation: when a farmer revokes consent, the data must stop flowing within minutes; the consent envelope MUST carry an explicit revocation timestamp and downstream consumers MUST honour it. Third, ISOBUS Task XML round-trip: variable-rate prescription maps emitted as ISOBUS Task XML must round-trip through WIA envelopes without losing the per-cell prescription rates; the harvest_record envelope embeds GeoJSON yield maps that are convertible back to ISOBUS Task XML for the next planting cycle.
