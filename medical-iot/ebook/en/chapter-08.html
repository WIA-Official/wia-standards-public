Chapter 8: WIA-MEDICAL-IOT Standard

Complete Technical Specification with 4-Phase Implementation Architecture

8.1 Standard Overview

The WIA-MEDICAL-IOT standard provides a comprehensive framework for implementing secure, interoperable Internet of Medical Things systems. Built on the WIA 4-phase architecture (Data, API, Protocol, Integration), this standard enables seamless connectivity between medical devices, remote monitoring platforms, clinical systems, and analytics applications.

Guided by the principle of 弘益人間 (Hongik Ingan) - "Benefit All Humanity" - this standard aims to make connected health technologies accessible globally, enabling continuous care and early intervention regardless of geographic or economic barriers.

8.2 WIA 4-Phase Architecture

The WIA-MEDICAL-IOT standard implements the WIA 4-phase architecture pattern:

Architecture Philosophy

The 4-phase architecture separates concerns into distinct layers: Phase 1 (Data) defines device and observation models compatible with FHIR; Phase 2 (API) provides RESTful endpoints for device management and data access; Phase 3 (Protocol) enables device communication via BLE, MQTT, and web protocols; Phase 4 (Integration) connects with EHR, analytics, and clinical systems.

Phase Name Purpose Components
Phase 1 Data Layer Information modeling Device, Observation, Patient models
Phase 2 API Layer Programmatic access FHIR APIs, device management, alerts
Phase 3 Protocol Layer Device communication BLE PHD, MQTT, IEEE 11073
Phase 4 Integration Layer External connectivity EHR, RPM platforms, analytics

8.4 Phase 2: API Layer Specification

The API Layer provides RESTful endpoints for device management and data access:

API Endpoints Overview

Category Endpoint Method Description
Devices /api/v1/devices GET List devices (filtered)
/api/v1/devices/{id} GET Get device details
/api/v1/devices/{id}/assign POST Assign device to patient
Observations /fhir/Observation GET Query observations
/fhir/Observation POST Submit observation
/api/v1/observations/aggregate GET Get aggregated data
Alerts /api/v1/alerts GET Query active alerts
/api/v1/alerts/{id}/acknowledge POST Acknowledge alert
/api/v1/alert-rules POST Configure alert rules
Programs /api/v1/programs GET List RPM programs
/api/v1/programs/{id}/enroll POST Enroll patient

Authentication

APIs use OAuth 2.0 with JWT tokens. Scopes include: devices:read, devices:manage, observations:read, observations:write, alerts:manage, programs:admin. SMART on FHIR enables EHR-launched applications with context-aware authorization.

8.5 Phase 3: Protocol Layer Specification

The Protocol Layer defines communication standards for device connectivity:

Supported Protocols

Protocol Use Case Standard
Bluetooth PHD Personal health devices IEEE 11073-20601, GATT profiles
MQTT IoT messaging MQTT 3.1.1/5.0 over TLS
HTTP/REST Gateway to cloud HTTPS, FHIR REST
WebSocket Real-time streaming WSS for live data
CoAP Constrained devices CoAP over DTLS

MQTT Topic Structure

// Device telemetry
wia/iomt/{org_id}/devices/{device_id}/telemetry

// Device status
wia/iomt/{org_id}/devices/{device_id}/status

// Alerts
wia/iomt/{org_id}/patients/{patient_id}/alerts

// Commands to device
wia/iomt/{org_id}/devices/{device_id}/commands

// Example message payload
{

8.6 Phase 4: Integration Layer Specification

The Integration Layer connects IoMT systems with healthcare ecosystem:

Integration Patterns

System Integration Method Data Exchanged
EHR FHIR API, SMART on FHIR Observations, devices, care plans
RPM Platform REST API, webhooks Device data, alerts, programs
Analytics Event streaming, batch export Time-series data, outcomes
Patient App Mobile SDK, push notifications Readings, education, reminders
Billing ADT events, charge capture RPM codes, transmission days

✓ Implementation Best Practices

  • Start with Phase 1 data models aligned to FHIR for maximum interoperability
  • Implement MQTT for real-time device communication with TLS encryption
  • Use FHIR Observation resources for clinical data integration
  • Design alert rules with clinical input to prevent alert fatigue
  • Enable SMART on FHIR for secure EHR-launched applications
  • Implement comprehensive audit logging for compliance

📌 Key Takeaways

  • WIA-MEDICAL-IOT implements 4-phase architecture: Data Layer (FHIR-aligned models), API Layer (REST, FHIR), Protocol Layer (BLE, MQTT, IEEE 11073), Integration Layer (EHR, analytics)
  • Core data models include Device, Observation, and Alert with full FHIR compatibility
  • API layer provides RESTful endpoints for device management, observations, alerts, and program enrollment
  • Protocol layer supports Bluetooth PHD, MQTT, HTTP/REST, WebSocket, and CoAP for device connectivity
  • Integration patterns connect IoMT to EHR (FHIR), RPM platforms, analytics systems, and patient apps
  • Security framework covers device authentication, transport encryption, API authorization, and data protection
