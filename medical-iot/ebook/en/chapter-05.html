Chapter 5: Data Management

Interoperability Standards and Data Flows for Medical IoT

5.1 IoMT Data Lifecycle

Medical IoT devices generate continuous streams of health data that must flow through collection, transmission, storage, processing, and ultimately integration with clinical systems. Managing this data lifecycle effectively determines whether IoMT delivers clinical value.

The data journey presents challenges at each stage: ensuring measurement accuracy at collection, maintaining integrity during transmission, enabling efficient storage at scale, supporting real-time processing for alerts, and achieving semantic interoperability for clinical use.

FHIR for IoMT

HL7 FHIR (Fast Healthcare Interoperability Resources) has become the preferred standard for modern healthcare interoperability. For IoMT, key FHIR resources include: Observation (measurements), Device (device information), Patient (demographics), DeviceMetric (calibration), and DeviceUseStatement (device assignment). SMART on FHIR enables secure app authorization.

Device Resource

The FHIR Device resource captures device metadata including manufacturer, model, serial number, UDI (Unique Device Identifier), and status. DeviceMetric represents calibration and operational information. These resources enable traceability from clinical observations back to the specific device that generated them.

5.5 Data Quality Management

Ensuring IoMT data quality is essential for clinical decision-making:

Data Quality Dimensions

  • Accuracy: Measurements reflect true physiological values
  • Completeness: No missing data points or metadata
  • Timeliness: Data available when clinically needed
  • Consistency: Measurements comparable across devices/time
  • Validity: Values within physiologically plausible ranges
  • Provenance: Source, device, and collection context known

Quality Assurance Processes

Process Description Implementation
Validation Rules Check plausible ranges BP 60-260/30-150, HR 30-250
Device Calibration Regular accuracy verification Per manufacturer schedule
Patient Verification Confirm measurement context Activity level, position, timing
Artifact Detection Identify erroneous readings Motion, interference detection

5.6 Data Storage Architecture

IoMT data storage must handle high-volume time-series data efficiently:

Storage Patterns

  • Time-Series Databases: Optimized for sequential sensor data (InfluxDB, TimescaleDB)
  • FHIR Servers: Healthcare-native storage with search (HAPI FHIR, Azure FHIR)
  • Data Lakes: Raw storage for analytics (S3, ADLS)
  • Operational Stores: Current state for real-time access
  • Archival Storage: Long-term retention with tiered cost

⚠️ Data Volume Challenge

A single continuous glucose monitor generates ~288 readings/day. A cardiac patch may generate 86,400+ samples/day at 1Hz. At scale, IoMT data volumes can overwhelm traditional EHR storage. Design for data aggregation (hourly/daily summaries vs. raw), tiered storage (hot/warm/cold), and selective EHR integration (clinically relevant vs. complete datasets).

✓ Data Management Best Practices

Adopt FHIR as the integration standard; use IEEE 11073 codes for device data; implement validation rules at ingestion; design for data volume from day one with appropriate storage tiers; maintain data provenance linking observations to devices and patients; and establish clear data governance policies covering retention, access, and sharing.

📌 Key Takeaways

  • IoMT data lifecycle spans collection, transmission, storage, processing, and clinical integration—challenges exist at each stage
  • Key interoperability standards include HL7 FHIR, IEEE 11073 PHD, LOINC codes, and Continua guidelines
  • FHIR resources (Observation, Device, Patient) provide standardized representation for modern healthcare integration
  • IEEE 11073 defines device profiles for blood pressure monitors, glucose meters, pulse oximeters, and other PHDs
  • Data quality requires attention to accuracy, completeness, timeliness, consistency, validity, and provenance
  • Storage architecture must handle high-volume time-series data with appropriate aggregation and tiering strategies
