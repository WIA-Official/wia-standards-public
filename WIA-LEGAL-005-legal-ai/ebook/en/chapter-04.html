Chapter 4: API Design for Legal Systems

Phase 2 of WIA-LEGAL-005 defines comprehensive API interfaces for legal AI systems. This chapter explores RESTful API design principles, GraphQL implementations, authentication mechanisms, and rate limiting strategies specific to legal technology applications. We examine how well-designed APIs enable seamless integration between legal research platforms, case management systems, and AI-powered analysis tools.

4.1 RESTful API Architecture for Legal Systems

Legal AI systems require robust API architectures that handle complex queries, large document collections, and real-time analysis requests. The WIA-LEGAL-005 standard defines a RESTful API specification that balances performance, security, and developer experience.

The API architecture follows REST principles while incorporating legal domain-specific requirements such as citation resolution, jurisdiction filtering, temporal queries, and privilege-aware access control. Every endpoint is designed to support both synchronous and asynchronous operations, enabling clients to handle long-running legal research tasks efficiently.

Core API Endpoints

Endpoint Method Purpose Response Time
/api/v1/documents GET, POST Search and ingest legal documents < 500ms
/api/v1/analysis POST Submit documents for AI analysis Async (webhook)
/api/v1/citations GET Resolve and validate citations < 200ms
/api/v1/precedents GET Find relevant case precedents < 1000ms
/api/v1/predictions POST Predict case outcomes < 2000ms

4.2 API Implementation Example

The following code demonstrates a production-ready implementation of the WIA-LEGAL-005 API specification using modern frameworks and best practices.

// WIA-LEGAL-005 API Server Implementation

import express from 'express';
import { rateLimit } from 'express-rate-limit';
import { authenticate, authorize } from './middleware/auth';
import { LegalDocumentService } from './services/documents';
import { AnalysisService } from './services/analysis';

const app = express();

// Rate limiting configuration
const limiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 100, // Limit each IP to 100 requests per window
  message: 'Too many requests from this IP'
});

app.use('/api/', limiter);
app.use(express.json({ limit: '10mb' }));

// Document Search Endpoint
app.get('/api/v1/documents',
  authenticate,
  async (req, res) => {
    try {
      const {
        query,
        jurisdiction,
        dateFrom,
        dateTo,
        docType,
        limit = 20,
        offset = 0
      } = req.query;

      const documentService = new LegalDocumentService();
      const results = await documentService.search({
        query: query as string,
        filters: {
          jurisdiction: jurisdiction ? (jurisdiction as string).split(',') : undefined,
          dateRange: dateFrom && dateTo ? {
            from: new Date(dateFrom as string),
            to: new Date(dateTo as string)
          } : undefined,
          documentType: docType as string
        },
        pagination: {
          limit: parseInt(limit as string),
          offset: parseInt(offset as string)
        }
      });

      res.json({
        success: true,
        data: {
          documents: results.documents,
          totalCount: results.total,
          pagination: {
            limit: results.limit,
            offset: results.offset,
            hasMore: results.offset + results.documents.length < results.total
          }
        },
        meta: {
          requestId: req.id,
          timestamp: new Date().toISOString()
        }
      });
    } catch (error) {
      res.status(500).json({
        success: false,
        error: {
          code: 'SEARCH_ERROR',
          message: error.message
        }
      });
    }
  }
);

// Document Analysis Endpoint (Async)
app.post('/api/v1/analysis',
  authenticate,
  authorize(['legal-ai:analyze']),
  async (req, res) => {
    try {
      const {
        documentId,
        analysisType,
        webhookUrl,
        options
      } = req.body;

      // Validate request
      if (!documentId || !analysisType) {
        return res.status(400).json({
          success: false,
          error: {
            code: 'INVALID_REQUEST',
            message: 'documentId and analysisType are required'
          }
        });
      }

      const analysisService = new AnalysisService();
      const job = await analysisService.submitAnalysis({
        documentId,
        type: analysisType,
        userId: req.user.id,
        webhook: webhookUrl,
        options
      });

      res.status(202).json({
        success: true,
        data: {
          jobId: job.id,
          status: 'pending',
          estimatedCompletion: job.estimatedTime,
          statusUrl: `/api/v1/analysis/${job.id}/status`
        },
        meta: {
          requestId: req.id,
          timestamp: new Date().toISOString()
        }
      });
    } catch (error) {
      res.status(500).json({
        success: false,
        error: {
          code: 'ANALYSIS_ERROR',
          message: error.message
        }
      });
    }
  }
);

// Citation Resolution Endpoint
app.get('/api/v1/citations/:citationId',
  authenticate,
  async (req, res) => {
    try {
      const { citationId } = req.params;
      const { format = 'bluebook' } = req.query;

      const citationService = new CitationService();
      const citation = await citationService.resolve(citationId);

      if (!citation) {
        return res.status(404).json({
          success: false,
          error: {
            code: 'CITATION_NOT_FOUND',
            message: `Citation ${citationId} not found`
          }
        });
      }

      const formatted = citationService.format(citation, format as string);

      res.json({
        success: true,
        data: {
          citation: {
            id: citation.id,
            canonical: formatted,
            components: citation.components,
            references: citation.references,
            treatment: citation.treatment
          }
        }
      });
    } catch (error) {
      res.status(500).json({
        success: false,
        error: {
          code: 'CITATION_ERROR',
          message: error.message
        }
      });
    }
  }
);

// Precedent Discovery Endpoint
app.get('/api/v1/precedents',
  authenticate,
  async (req, res) => {
    try {
      const {
        legalIssue,
        jurisdiction,
        minRelevance = 0.7,
        limit = 10
      } = req.query;

      const precedentService = new PrecedentService();
      const precedents = await precedentService.find({
        issue: legalIssue as string,
        jurisdiction: jurisdiction ? (jurisdiction as string).split(',') : undefined,
        minRelevance: parseFloat(minRelevance as string),
        limit: parseInt(limit as string)
      });

      res.json({
        success: true,
        data: {
          precedents: precedents.map(p => ({
            id: p.id,
            caseName: p.caseName,
            citation: p.citation,
            relevanceScore: p.relevance,
            keyHoldings: p.holdings,
            treatment: p.treatment,
            year: p.year
          })),
          analysis: {
            totalFound: precedents.length,
            averageRelevance: precedents.reduce((sum, p) => sum + p.relevance, 0) / precedents.length,
            jurisdictionCoverage: [...new Set(precedents.map(p => p.jurisdiction))]
          }
        }
      });
    } catch (error) {
      res.status(500).json({
        success: false,
        error: {
          code: 'PRECEDENT_ERROR',
          message: error.message
        }
      });
    }
  }
);

// Start server
const PORT = process.env.PORT || 3000;
app.listen(PORT, () => {
  console.log(`WIA-LEGAL-005 API server running on port ${PORT}`);
});

4.3 Authentication and Authorization

Legal AI APIs must implement robust security measures to protect sensitive legal information and ensure compliance with attorney-client privilege requirements. The WIA-LEGAL-005 standard mandates OAuth 2.0 with JWT tokens for authentication and role-based access control (RBAC) for authorization.

Security Feature Implementation Standard Purpose
Authentication OAuth 2.0 + JWT RFC 6749, RFC 7519 Verify user identity
Authorization RBAC with scopes WIA-LEGAL-005 Control access to resources
Transport Security TLS 1.3 RFC 8446 Encrypt data in transit
API Keys SHA-256 hashed keys WIA-LEGAL-005 Service authentication
Rate Limiting Token bucket algorithm Industry standard Prevent abuse

4.4 Error Handling and Status Codes

Consistent error handling is critical for API usability. The WIA-LEGAL-005 standard defines a comprehensive error taxonomy with specific status codes and error objects that provide actionable information to API consumers.

// WIA-LEGAL-005 Error Response Format

interface ErrorResponse {
  success: false;
  error: {
    code: string;           // Machine-readable error code
    message: string;        // Human-readable description
    details?: any;          // Additional context
    requestId: string;      // For support tracking
    documentation?: string; // Link to error docs
  };
  meta: {
    timestamp: string;
    version: string;
  };
}

// Error Codes
enum LegalAPIErrorCode {
  // Client Errors (4xx)
  INVALID_REQUEST = 'INVALID_REQUEST',
  UNAUTHORIZED = 'UNAUTHORIZED',
  FORBIDDEN = 'FORBIDDEN',
  NOT_FOUND = 'NOT_FOUND',
  RATE_LIMIT_EXCEEDED = 'RATE_LIMIT_EXCEEDED',
  INVALID_JURISDICTION = 'INVALID_JURISDICTION',
  INVALID_CITATION = 'INVALID_CITATION',

  // Server Errors (5xx)
  INTERNAL_ERROR = 'INTERNAL_ERROR',
  SERVICE_UNAVAILABLE = 'SERVICE_UNAVAILABLE',
  ANALYSIS_TIMEOUT = 'ANALYSIS_TIMEOUT',
  EXTERNAL_SERVICE_ERROR = 'EXTERNAL_SERVICE_ERROR'
}

4.5 API Versioning and Deprecation

APIs evolve over time. The WIA-LEGAL-005 standard defines a clear versioning strategy that ensures backward compatibility while enabling innovation. Version numbers appear in the URL path (/api/v1/), and deprecation warnings are communicated through HTTP headers.

Version Strategy Approach Timeline Migration Support
Major Version Breaking changes 12 month deprecation period Full documentation + migration guide
Minor Version New features (backward compatible) Immediate availability Release notes + examples
Patch Version Bug fixes Immediate deployment Changelog entry
Beta Features Experimental endpoints No guarantee of stability Beta documentation
