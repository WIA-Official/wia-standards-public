제6장: Phase 3 - 실시간 프로토콜

弘益人間 · 널리 인간을 이롭게 하라

개요

Phase 3는 디지털 패션 자산의 실시간 협업과 라이브 이벤트를 가능하게 하는 통신 프로토콜을 정의합니다. 여러 디자이너가 동시에 같은 의상을 편집하거나, 메타버스 패션쇼에서 실시간으로 의상이 변경되는 시나리오를 지원하기 위해 WebSocket 기반 이벤트 스트리밍, Operational Transform(OT), Conflict-free Replicated Data Types(CRDT) 등의 기술을 활용합니다.

이 장에서는 실시간 프로토콜의 설계 원칙, WebSocket 연결 관리, 이벤트 타입, 동시 편집 충돌 해결, 상태 동기화 전략, 그리고 실제 구현 예시를 다룹니다.

WebSocket 연결

1. 연결 수립

WIA-ART-008 실시간 프로토콜은 WebSocket을 사용하여 클라이언트와 서버 간 양방향 통신을 수립합니다. 

// TypeScript 클라이언트 예시
import { WIARealtimeClient } from '@wia/art-008-sdk';

const rtClient = new WIARealtimeClient({
  endpoint: 'wss://realtime.wiastandards.org/art-008/v1',
  accessToken: 'eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...'
});

// WebSocket 연결
await rtClient.connect();

// 연결 상태 이벤트
rtClient.on('connected', () => {
  console.log('Connected to WIA real-time server');
});

rtClient.on('disconnected', () => {
  console.log('Disconnected. Attempting reconnection...');
});

rtClient.on('error', (error) => {
  console.error('WebSocket error:', error);
});

WebSocket 연결 시 OAuth 2.0 access token 또는 Web3 서명을 사용하여 인증합니다.

이벤트 타입

WIA-ART-008 실시간 프로토콜은 다음 이벤트 타입을 정의합니다:

이벤트 타입 방향 설명 페이로드
garment.created 서버 → 클라이언트 새 의상 생성 알림 의상 메타데이터
garment.updated 서버 → 클라이언트 의상 수정 알림 변경된 필드만
garment.deleted 서버 → 클라이언트 의상 삭제 알림 의상 ID
edit.lock 클라이언트 ↔ 서버 편집 잠금 요청/응답 잠금 범위, 기간
edit.operation 클라이언트 → 서버 편집 작업 전송 OT operation
edit.broadcast 서버 → 클라이언트 편집 작업 브로드캐스트 OT operation
presence.join 서버 → 클라이언트 사용자 참여 사용자 정보
presence.leave 서버 → 클라이언트 사용자 퇴장 사용자 ID
cursor.move 클라이언트 ↔ 서버 커서 위치 공유 3D 좌표
live.stream 서버 → 클라이언트 라이브 이벤트 스트림 이벤트 데이터

실시간 협업 편집

1. 의상 구독 (Subscribe)

특정 의상의 변경사항을 실시간으로 받기 위해 구독합니다. 

// 의상 구독
await rtClient.subscribe('garment', 'urn:uuid:abc123');

// 변경 이벤트 수신
rtClient.on('garment.updated', (event) => {
  console.log('Garment updated:', event.garment_id);
  console.log('Changed fields:', event.changes);
  console.log('Editor:', event.editor.name);

  // UI 업데이트
  updateGarmentView(event.changes);
});

// 구독 해제
await rtClient.unsubscribe('garment', 'urn:uuid:abc123');

2. Operational Transform (OT)

여러 사용자가 동시에 같은 의상을 편집할 때 충돌을 방지하기 위해 Operational Transform 알고리즘을 사용합니다.

OT 동작 원리

시나리오: 디자이너 A와 B가 동시에 같은 드레스의 색상을 변경합니다.

  1. 초기 상태: color: "#FF0000"
  2. A가 "#00FF00"으로 변경 (Operation 1)
  3. B가 "#0000FF"로 변경 (Operation 2)
  4. 서버가 Operation 1을 먼저 받음 → 적용
  5. 서버가 Operation 2를 받음 → Operation 1에 대해 변환(transform) → 적용
  6. 최종 상태는 마지막 작업인 B의 "#0000FF"
// 편집 작업 전송
await rtClient.sendOperation({
  garment_id: 'urn:uuid:abc123',
  operation: {
    type: 'set',
    path: 'semantic.color_primary',
    value: '#00FF00',
    version: 42  // 로컬 버전 번호
  }
});

// 서버로부터 변환된 작업 수신
rtClient.on('edit.broadcast', (event) => {
  if (event.operation.version > localVersion) {
    applyOperation(event.operation);
    localVersion = event.operation.version;
  }
});

3. 편집 잠금 (Locking)

특정 부분을 배타적으로 편집하기 위해 잠금을 요청할 수 있습니다. 

// 특정 필드 잠금 요청
const lock = await rtClient.requestLock({
  garment_id: 'urn:uuid:abc123',
  scope: 'semantic.color_primary',
  duration: 300  // 5분
});

if (lock.granted) {
  console.log('Lock acquired. Editing...');

  // 편집 수행
  await client.garments.update(garmentId, {
    semantic: { color_primary: '#00FF00' }
  });

  // 잠금 해제
  await rtClient.releaseLock(lock.lock_id);
} else {
  console.log('Lock denied. Field is being edited by:', lock.owner);
}
잠금 범위 대상 허용 동시 편집
전체 의상 전체 ❌
섹션 metadata, technical, semantic 등 다른 섹션만
필드 semantic.color_primary 등 개별 필드 다른 필드만
레이어 특정 glTF 레이어 다른 레이어만

Presence (참여자 정보)

협업 세션에 참여 중인 사용자 목록과 각자의 활동 상태를 실시간으로 공유합니다. 

// Presence 구독
await rtClient.subscribePresence('garment', 'urn:uuid:abc123');

// 참여자 정보 수신
rtClient.on('presence.join', (user) => {
  console.log(`${user.name} joined the session`);
  displayUserCursor(user);
});

rtClient.on('presence.leave', (userId) => {
  console.log(`User ${userId} left the session`);
  removeUserCursor(userId);
});

// 현재 참여자 목록 조회
const participants = await rtClient.getPresence('garment', 'urn:uuid:abc123');
console.log('Active participants:', participants);

// 커서 위치 공유
rtClient.sendCursor({
  garment_id: 'urn:uuid:abc123',
  position: { x: 0.5, y: 0.3, z: 0.1 },
  viewport: { zoom: 1.5, rotation: [0, 45, 0] }
});

// 다른 사용자 커서 표시
rtClient.on('cursor.move', (event) => {
  updateCursor(event.user_id, event.position);
});

라이브 이벤트 스트리밍

1. 패션쇼 라이브 스트리밍

메타버스 패션쇼에서 의상이 실시간으로 변경되는 것을 수천 명의 시청자가 동시에 볼 수 있습니다.

라이브 스트림 구조

// 스트림 구독
await rtClient.subscribeLiveStream('fashion-show-2024');

// 이벤트 수신
rtClient.on('live.stream', (event) => {
  switch (event.event_type) {
    case 'garment_showcase':
      displayGarment(event.garment);
      break;
    case 'garment_transform':
      animateGarmentChange(event.from, event.to, event.duration);
      break;
    case 'announcement':
      showAnnouncement(event.message);
      break;
  }
});

// 이벤트 예시
{
  "stream_id": "fashion-show-2024",
  "event_type": "garment_transform",
  "timestamp": "2024-03-15T19:30:45Z",
  "sequence": 127,
  "payload": {
    "from": "urn:uuid:dress-001",
    "to": "urn:uuid:dress-002",
    "duration": 3000,
    "transition": "fade"
  }
}

2. Pub/Sub 패턴

WIA-ART-008 실시간 프로토콜은 Publish/Subscribe 패턴을 사용하여 확장 가능한 이벤트 배포를 지원합니다.

채널 타입 패턴 예시 용도
garment:{id} 단일 의상 garment:urn:uuid:abc123 특정 의상 실시간 편집
collection:{id} 컬렉션 collection:summer-2024 컬렉션 내 모든 의상 업데이트
creator:{wallet} 창작자 creator:0x742d35... 창작자의 모든 작품 업데이트
live:{event} 라이브 이벤트 live:fashion-show-2024 패션쇼, 런웨이 스트리밍
platform:{name} 플랫폼 platform:decentraland 플랫폼별 업데이트

충돌 해결 전략

1. Last-Write-Wins (LWW)

가장 최근의 쓰기 작업이 승리합니다. 간단하지만 데이터 손실 가능성이 있습니다. 

{
  "strategy": "last-write-wins",
  "timestamp": "2024-03-15T10:30:45.123Z",
  "operation": {
    "path": "semantic.color_primary",
    "value": "#FF0000"
  }
}

2. Version Vector

각 클라이언트의 버전을 추적하여 인과 관계(causality)를 보장합니다. 

{
  "strategy": "version-vector",
  "version": {
    "client_a": 5,
    "client_b": 3,
    "client_c": 7
  },
  "operation": { ... }
}

3. CRDT (Conflict-free Replicated Data Type)

수학적으로 증명된 충돌 없는 데이터 구조를 사용합니다. 

// CRDT 카운터 예시 (의상 좋아요 수)
{
  "type": "g-counter",
  "increments": {
    "client_a": 5,
    "client_b": 3,
    "client_c": 2
  },
  "value": 10  // 5 + 3 + 2
}

성능 최적화

1. 델타 동기화

전체 상태가 아닌 변경된 부분만 전송하여 대역폭을 절약합니다. 

// 전체 상태 전송 (비효율)
{
  "garment": {
    "id": "...",
    "name": "...",
    "semantic": {
      "color_primary": "#FF0000",
      "color_secondary": ["#00FF00", "#0000FF"],
      // ... 모든 필드
    }
  }
}

// 델타만 전송 (효율적)
{
  "garment_id": "urn:uuid:abc123",
  "delta": {
    "semantic.color_primary": "#FF0000"
  }
}

2. 배치 처리

여러 작은 이벤트를 모아서 한 번에 전송합니다. 

// 100ms마다 변경사항을 배치로 전송
const batchQueue = [];
setInterval(() => {
  if (batchQueue.length > 0) {
    rtClient.sendBatch(batchQueue);
    batchQueue.length = 0;
  }
}, 100);

3. 압축

대용량 메시지는 gzip 또는 Brotli로 압축하여 전송합니다.

메시지 크기 압축 사용 예상 압축률
< 1KB 없음 -
1KB - 10KB gzip 60-70%
> 10KB Brotli 70-80%

핵심 요점

  • WebSocket 기반: wss:// 프로토콜로 양방향 실시간 통신. OAuth 2.0 또는 Web3 서명으로 연결 인증. 자동 재연결 및 백오프 전략 지원.
  • 이벤트 타입: garment.created/updated/deleted, edit.lock/operation/broadcast, presence.join/leave, cursor.move, live.stream 등 10가지 표준 이벤트 정의.
  • 협업 편집: Operational Transform(OT)으로 동시 편집 충돌 해결. 편집 잠금(전체/섹션/필드/레이어 단위)으로 배타적 접근 보장. 실시간 커서 공유.
  • Presence: 참여자 목록, 활동 상태, 커서 위치 실시간 공유. 협업 세션의 투명성과 조율 향상.
  • Pub/Sub 패턴: 채널 기반 메시지 배포(garment:{id}, collection:{id}, creator:{wallet}, live:{event}). 확장 가능한 이벤트 스트리밍.
  • 성능 최적화: 델타 동기화(변경 부분만 전송), 배치 처리(100ms 간격), 메시지 압축(gzip/Brotli). 충돌 해결(LWW, Version Vector, CRDT).

복습 질문

  1. WIA-ART-008 WebSocket 연결 수립 과정을 설명하십시오. 인증 핸드셰이크에서 클라이언트가 전송하는 정보(access_token, client_id, capabilities)와 서버 응답(session_id, user_id, expires_at)의 역할을 서술하십시오.
  2. 실시간 프로토콜의 10가지 이벤트 타입을 표로 정리하고, 각 이벤트의 방향(클라이언트 → 서버, 서버 → 클라이언트, 양방향), 용도, 페이로드 구조를 설명하십시오.
  3. Operational Transform(OT)의 동작 원리를 예시와 함께 설명하십시오. 두 디자이너가 동시에 같은 의상의 색상을 변경할 때 충돌이 어떻게 해결되는지 단계별로 서술하십시오.
  4. 네 가지 편집 잠금 범위(전체, 섹션, 필드, 레이어)를 비교하고, 각 범위에서 허용되는 동시 편집의 수준을 설명하십시오. 편집 잠금 요청/해제 코드를 작성하십시오.
  5. Presence 기능을 설명하고, presence.join, presence.leave, cursor.move 이벤트가 협업 편집에서 어떤 역할을 하는지 서술하십시오. 다섯 가지 Pub/Sub 채널 타입(garment:{id}, collection:{id}, creator:{wallet}, live:{event}, platform:{name})의 용도를 설명하십시오.
  6. 세 가지 충돌 해결 전략(Last-Write-Wins, Version Vector, CRDT)을 비교하고 각각의 장단점을 분석하십시오. 성능 최적화 기법(델타 동기화, 배치 처리, 압축)이 대역폭과 응답 속도에 미치는 영향을 설명하십시오.
