제6장: dwell-time SLA와 안전 동작 트리거

오염을 측정하고 보고하는 것만으로는 안전이 보장되지 않는다. "오염이 얼마나 오래 잔존하면 어떤 안전 동작을 취하는가"에 대한 합의가 필요하다. WIA 표준은 dwell-time SLA(최대 가림 잔존 시간)와 다섯 가지 안전 동작 트리거를 정의한다. 이 장은 그 메커니즘을 다룬다.

6.1 왜 dwell-time인가

순간적인 명료도 저하와 지속적인 저하는 위험도가 다르다. 빗방울이 잠깐 렌즈에 떨어졌다가 와이퍼나 주행풍으로 즉시 제거되는 경우와, 진흙이 달라붙어 5분간 BLIND가 지속되는 경우는 전혀 다른 대응을 요구한다. dwell-time(잔존 시간)은 이 차이를 포착하는 핵심 변수다.

보고 메시지의 dwellSeconds 필드는 현재 비-CLEAR 상태가 연속으로 잔존한 시간(초)이다. 이 값이 미리 정의된 SLA 임계를 넘으면 해당 안전 동작이 트리거된다. SLA는 상태별·센서별로 다르게 설정할 수 있다.

6.2 dwell-time SLA의 정의

dwell-time SLA는 "특정 상태가 얼마나 오래 지속되면 안전 동작을 트리거해야 하는가"를 명시한 약속이다. 적합성 선언(ConformanceDeclaration)의 dwellSlaSeconds 필드에 센서 클래스별로 선언된다.

dwell-time SLA 개념 모델
==========================
상태가 DEGRADED 이상으로 진입
        │
        ▼  (dwellSeconds 카운트 시작)
   ┌──────────────────────────────┐
   │  dwellSeconds < SLA → 모니터링 │
   │  dwellSeconds ≥ SLA → 트리거   │
   └──────────────────────────────┘
        │
        ▼
   안전 동작 실행 (상태·SLA에 따라)

상태가 CLEAR로 복귀 → dwellSeconds = 0 (리셋)

예컨대 OBSTRUCTED 상태에서 dwellSlaSeconds = 30이라면, 카메라가 30초 이상 OBSTRUCTED를 유지할 때 감속·우회·세척 같은 동작이 트리거된다. BLIND 상태에서는 더 짧은 SLA(예: 즉시 또는 수 초)를 설정해 즉각 safe-state로 전환하도록 한다.

6.3 안전 동작 트리거 enum

WIA 표준은 다섯 가지 안전 동작 트리거를 정의한다. 적합성 선언의 safeActionTriggers 배열에 지원하는 동작을 열거한다.

safeActionTriggers: [
  "slow_down",        // 감속
  "reroute",          // 경로 우회
  "trigger_cleaning", // 세척 작동 요청
  "safe_state",       // 안전 정지·최소위험상태(MRC)
  "disable_sensor"    // 해당 센서 입력 배제
]
트리거의미전형적 발동 상황
slow_down주행·이동 속도 감소DEGRADED 지속, 인식거리 저하
reroute오염원 회피 경로로 변경특정 방향 역광·먼지 구간
trigger_cleaning세척 메커니즘 작동 요청제거 가능 오염(빗물·먼지)
safe_state최소위험상태로 전환·정지BLIND, 다중 센서 동시 저하
disable_sensor융합에서 해당 센서 배제단일 센서 BLIND, 잉여 센서 존재

6.4 trigger_cleaning과 세척 비표준화의 경계

주의할 점은 trigger_cleaning이 "세척을 요청한다"는 의도(intent)만 표준화하고, "어떻게 닦는가"는 표준화하지 않는다는 것이다. 노즐 분사·에어커튼·초음파 진동·발열 디포깅 등 실제 세척 방식은 벤더 자유다. WIA 표준은 단지 "지금 세척이 필요하다"는 트리거 신호와 그 발동 조건(dwell-time SLA)만 규정한다. 이는 세척 하드웨어 특허밭과의 충돌을 피하는 설계다.

세척 무한 반복 방지

스크래치 같은 영구 손상(제4장)은 세척으로 회복되지 않는다. 세척 후에도 PCI가 회복되지 않으면(예: lastCleanedAt 갱신 후에도 BLIND 지속), 시스템은 trigger_cleaning을 무한 반복하지 말고 disable_sensor 또는 safe_state로 에스컬레이션해야 한다. 회복 불가 판정은 "세척 후 N초 내 PCI 미회복"으로 정의할 수 있다.

6.5 동작 에스컬레이션 계층

안전 동작은 단계적으로 에스컬레이션된다. 경미한 저하에는 가벼운 대응을, 심각한 저하에는 강한 대응을 적용한다.

에스컬레이션 계층 (예시)
==========================
DEGRADED (PCI 60-89)
  → 모니터링 강화
  → dwell ≥ SLA_d : trigger_cleaning, slow_down

OBSTRUCTED (PCI 30-59)
  → slow_down (즉시)
  → trigger_cleaning
  → dwell ≥ SLA_o : reroute 또는 safe_state

BLIND (PCI 0-29)
  → disable_sensor (융합에서 배제)
  → 잉여 센서로 운용 불가 시 : safe_state (즉시)

이 계층은 SAE J3016의 fallback·최소위험상태(MRC) 개념과 연동된다. 인식 명료도가 ODD(운영 설계 영역)의 인식 요구를 만족하지 못하면, 시스템은 정의된 fallback을 실행해야 한다. PCI와 dwell-time은 "ODD를 벗어났는가"를 판정하는 정량적 근거를 제공한다.

6.6 잉여성과 disable_sensor

disable_sensor는 다중 센서 잉여성(redundancy)이 있을 때 유효하다. 예컨대 전방을 카메라·LiDAR·레이더 세 센서가 함께 인식하는 경우, 카메라가 BLIND에 빠지면 그 입력을 융합에서 배제하고 나머지 두 센서로 운용을 지속할 수 있다. 그러나 잉여 센서가 없거나 다중 센서가 동시에 저하되면 disable_sensor는 불가능하며 safe_state로 직행해야 한다.

이 판단은 융합 가중치(제8장)와 직결된다. 각 센서의 PCI가 융합 신뢰 가중을 결정하므로, BLIND 센서는 자연스럽게 가중이 0에 가까워져 융합 결과에서 배제된다.

6.7 SLA 선언 예시

적합성 선언 내 SLA·트리거 (발췌)
==================================
{
  "sensorClass": "lidar_window",
  "count": 2,
  "pciSupported": true,
  "reportingSupported": true,
  "dwellSlaSeconds": 20,
  "safeActionTriggers": ["slow_down", "trigger_cleaning", "safe_state", "disable_sensor"]
}

이 선언은 "이 제품의 LiDAR는 비-CLEAR 상태가 20초를 넘으면 안전 동작을 트리거하며, 감속·세척·safe-state·센서배제를 지원한다"는 약속이다. 관제와 감사자는 이 선언을 근거로 안전 동작의 예측 가능성을 검증한다.

dwell-time SLA와 안전 동작 트리거의 연동은 적합성 레벨 L-C의 요건이다. 다음 장에서는 L-A·L-B·L-C 세 적합성 레벨과 인증 절차를 다룬다.
