Chapter 6: Dwell-Time SLA & Safe-Action Triggers

Knowing a sensor is obstructed is useless unless something happens in response, within a bounded time. This chapter defines the dwell-time service-level agreement — the maximum time a sensor may remain obstructed before the agent must act — and the five standardized safe-action triggers: slow_down, reroute, trigger_cleaning, safe_state, and disable_sensor. Together they convert a clarity measurement into a closed safety loop, and they are the defining requirement of the highest conformance level, L-C.

6.1 Why Time Matters

Perception clarity fluctuates. A truck throws up a sheet of spray; a robot rolls through a puff of dust; sun glare flares for an instant on a curve. If an agent reacted to every momentary dip into a degraded state, it would oscillate uselessly — braking, rerouting, and firing cleaning cycles on transients that resolve on their own in a second. Conversely, a genuine obstruction that persists is a mounting hazard: each second the agent acts on a collapsed field of view increases the chance of an undetected obstacle.

The dwell-time SLA resolves this tension. By gating action on how long a sensor has continuously held a non-CLEAR state — the dwellSeconds field from Chapter 5 — the agent distinguishes transient noise from sustained degradation and acts proportionately.

6.2 The Dwell-Time SLA

A dwell-time SLA specifies, per sensor class and state, the maximum dwell permitted before a mandated response. It is declared in the conformance declaration's dwellSlaSeconds field. The principle is monotonic: the more severe the state, the shorter the tolerated dwell.

Dwell-time SLA model (illustrative, vendor-declared)
====================================================
State        Tolerated dwell      Mandated response on breach
-----------  -------------------  --------------------------------------
degraded     long  (e.g. minutes) monitor; trigger_cleaning if available
obstructed   short (e.g. seconds) slow_down and/or reroute and/or clean
blind        ~immediate           disable_sensor + safe_state

A BLIND sensor tolerates essentially no dwell — the agent cannot safely keep using a sensor it cannot trust. An OBSTRUCTED sensor permits a brief window in which cleaning or a speed reduction may restore clarity before more drastic action. A DEGRADED sensor may dwell longer under monitoring. Exact values are vendor-declared and depend on agent type, speed envelope, and operating domain — a highway vehicle's obstructed-dwell tolerance is far tighter than a 1 m/s indoor AMR's.

6.3 The Five Safe-Action Triggers

The standard fixes a closed set of five action triggers. A conformance declaration lists, per sensor class, which triggers the product supports via the safeActionTriggers array.

safeActionTriggers: [
  "slow_down",        // reduce speed to enlarge reaction margin
  "reroute",          // alter path to avoid reliance on the degraded sensor
  "trigger_cleaning", // invoke the (vendor-specific) cleaning mechanism
  "safe_state",       // bring the agent to a defined minimal-risk condition
  "disable_sensor"    // remove the sensor from the trusted fusion set
]
TriggerWhat it doesTypical state
slow_downReduces speed so the remaining clear sensors and shorter range still give adequate reaction timeobstructed
rerouteChooses a path that does not depend on the degraded sensor's coverage (e.g. avoids a turn requiring the blocked side)obstructed
trigger_cleaningInvokes the vendor's cleaning subsystem — spray, air, wipe, heat, vibration — which the standard does NOT specifydegraded / obstructed
safe_stateBrings the agent to a defined minimal-risk condition: controlled stop, pull-over, hover-and-hold, dockblind
disable_sensorDrops the sensor from the trusted set so fusion stops weighting its corrupted datablind

Note that trigger_cleaning is the only point where the standard touches cleaning at all — and even here it merely invokes whatever mechanism the vendor provides. The standard never specifies how cleaning is performed, preserving the deliberate boundary established in Chapter 1.

6.4 A Worked Escalation

Consider a vehicle's front camera during a muddy passage:

t=0s    PCI 88  degraded  dwell 0    -> monitor
t=12s   PCI 71  degraded  dwell 12   -> dwell < SLA, continue monitoring
t=20s   PCI 71  degraded  dwell 20   -> degraded-SLA breach -> trigger_cleaning
t=23s   PCI 54  obstructed dwell 3   -> cleaning insufficient, now obstructed
t=26s   PCI 51  obstructed dwell 6   -> obstructed-SLA breach -> slow_down + reroute
t=31s   PCI 22  blind      dwell 0*  -> state worsened; blind tolerates ~no dwell
                                      -> disable_sensor + safe_state

(*dwell resets the continuous-non-CLEAR clock per the implementation's state-transition policy; what matters is that BLIND triggers immediate action.) The escalation is graduated: monitor, then clean, then slow and reroute, then — when clarity collapses entirely — disable the sensor and enter a safe state. At each rung the response is proportional to both the severity (state) and the persistence (dwell).

6.5 Multi-Sensor Arbitration

A real agent runs many sensors, and safe actions must be arbitrated across all of them. If a single redundant camera goes BLIND while overlapping LiDAR and radar remain CLEAR, the correct action may be only disable_sensor — fusion continues safely on the survivors. If the sensors covering a critical direction all degrade, the agent has lost a whole region of awareness and must escalate to slow_down, reroute, or safe_state regardless of any single sensor's redundancy. The dwell-time SLA therefore operates not only per sensor but, in mature implementations, per safety-relevant coverage zone. Chapter 8 develops how PCI feeds fusion weights to make this arbitration principled.

6.6 Reporting the Loop

The actions an agent takes should themselves be observable. While the SensorClarityReport carries the clarity state and dwell, an L-C implementation closes the loop by ensuring the fleet can correlate a reported OBSTRUCTED-with-high-dwell condition against the agent having slowed, rerouted, cleaned, or stopped. This makes the safety behavior auditable: an operator can verify not just that a sensor degraded, but that the mandated response actually fired within the SLA window.

Chapter Summary

The dwell-time SLA caps how long a sensor may stay non-CLEAR before a mandated response, with shorter tolerances for more severe states — BLIND tolerates essentially none. Five fixed triggers convert clarity loss into action: slow_down, reroute, trigger_cleaning (the sole, mechanism-agnostic touchpoint with cleaning), safe_state, and disable_sensor. Responses escalate proportionally to severity and persistence and are arbitrated across redundant sensors and coverage zones. Implementing dwell-time SLAs with trigger integration is exactly what distinguishes the top conformance level, L-C (Chapter 7).
