Chapter 5: Clarity States & the Reporting Message

Measuring PCI is only half the standard; the other half is communicating it. This chapter defines the four-value clarity state enum and the SensorClarityReport message — a VDA5050-style JSON document by which an agent publishes, per sensor, its PCI, state, contaminants, dwell time, and confidence to fleet control or to peer agents. We cover the schema, the required and optional fields, the two roles of dwell-time and confidence, and the consistency rules that keep a report internally coherent.

5.1 The Four Clarity States

Every sensor is, at any instant, in exactly one of four states. The states are an ordered ladder of trust, and their order and meaning are invariant even if the underlying PCI thresholds are tuned for a given deployment.

enum ClarityState {
  CLEAR      = "clear"        // PCI 90-100  nominal
  DEGRADED   = "degraded"     // PCI 60-89   monitor / consider cleaning
  OBSTRUCTED = "obstructed"   // PCI 30-59   safe action: slow / reroute / clean
  BLIND      = "blind"        // PCI 0-29    stop using sensor; safe-state
}

invariant order:  CLEAR > DEGRADED > OBSTRUCTED > BLIND
StatePCI bandTrust postureExpected response
clear90 - 100Full trustNormal operation
degraded60 - 89Reduced trustMonitor; trigger cleaning if available
obstructed30 - 59Low trustSlow down, reroute, or clean
blind0 - 29No trustDisable sensor; enter safe-state

5.2 The SensorClarityReport Schema

The report carries a header identifying the agent and message, plus a sensors[] array with one entry per sensor. It follows the VDA5050 pattern of a versioned header and a structured body.

{
  "header": {
    "version": "1.0.0",
    "messageId": "uuid",
    "timestamp": "ISO 8601",
    "agentId": "string",
    "agentType": "vehicle | robot | drone | amr | other",
    "conformanceLevel": "L-A | L-B | L-C"
  },
  "sensors": [
    {
      "sensorId": "string",
      "sensorClass": "rgb_camera | ir_thermal | lidar_window | radar_radome | ultrasonic",
      "pci": 0-100,
      "state": "clear | degraded | obstructed | blind",
      "contaminants": [
        { "type": "ContaminantType | other", "coverage": 0.0-1.0, "note": "optional" }
      ],
      "axes":       { "occlusion": 0.0-1.0, "distanceDegradation": 0.0-1.0, "mtfReduction": 0.0-1.0 },
      "pciWeights": { "occlusion": number, "distance": number, "mtf": number },
      "lastCleanedAt": "ISO 8601 | null",
      "dwellSeconds": number,
      "confidence": 0.0-1.0
    }
  ]
}

5.3 Required versus Optional Fields

The standard distinguishes fields that must always be present from those recommended for transparency. The header must carry version, messageId, timestamp, agentId, and agentType. Each sensor entry must carry sensorId, sensorClass, pci, state, dwellSeconds, and confidence. Each contaminant must carry type and coverage.

The axes and pciWeights objects are optional but recommended: publishing them lets any consumer recompute and verify the PCI, turning the report into self-auditing evidence. lastCleanedAt may be null if the sensor has not been cleaned in the current session.

5.4 Dwell Time

dwellSeconds is the continuous time, in seconds, that the sensor has remained in a non-CLEAR state. When the sensor is CLEAR, dwell is 0. Dwell time is the foundation of the SLA logic developed in Chapter 6: a brief dip into DEGRADED during a passing splash is operationally different from a sustained OBSTRUCTED condition that has persisted for minutes. By reporting dwell explicitly, the agent lets the fleet or its own planner apply time-based thresholds rather than reacting to every momentary fluctuation. A consistency rule follows directly: if dwellSeconds > 0, the state must not be clear.

5.5 Confidence

confidence, in [0.0, 1.0], is the agent's certainty about the PCI value itself — the reliability of its own self-diagnosis, not the clarity of the scene. A sensor might report PCI 50 with confidence 0.9 (it is quite sure it is half-obstructed) or PCI 50 with confidence 0.4 (its estimate is shaky, perhaps because the contamination pattern confounds its measurement). Low confidence is itself actionable: a fusion stack or planner may treat an uncertain clarity estimate conservatively, much as it would treat a marginal one. Separating "how clear" (PCI) from "how sure" (confidence) prevents the two from being silently conflated.

5.6 Example — Rain DEGRADED Camera with a CLEAR LiDAR

{
  "header": {
    "version": "1.0.0",
    "messageId": "7c1f0a2e-3b9d-4a51-8e2c-91a0b6d4f7e2",
    "timestamp": "2025-06-05T08:14:32Z",
    "agentId": "veh-seoul-0421",
    "agentType": "vehicle",
    "conformanceLevel": "L-C"
  },
  "sensors": [
    {
      "sensorId": "front-cam-main",
      "sensorClass": "rgb_camera",
      "pci": 73,
      "state": "degraded",
      "contaminants": [ { "type": "rain_film", "coverage": 0.35 } ],
      "axes": { "occlusion": 0.18, "distanceDegradation": 0.22, "mtfReduction": 0.40 },
      "pciWeights": { "occlusion": 0.40, "distance": 0.25, "mtf": 0.35 },
      "lastCleanedAt": "2025-06-05T08:02:10Z",
      "dwellSeconds": 47,
      "confidence": 0.92
    },
    {
      "sensorId": "roof-lidar-1",
      "sensorClass": "lidar_window",
      "pci": 95,
      "state": "clear",
      "contaminants": [],
      "lastCleanedAt": "2025-06-05T07:55:00Z",
      "dwellSeconds": 0,
      "confidence": 0.98
    }
  ]
}

The camera has dwelt 47 seconds in DEGRADED with a 35% rain film; the LiDAR is CLEAR with dwell 0. A fleet receiving this can already reason: one sensor merits monitoring, the other is fully trusted.

5.7 Consistency Rules

A conformant report must be internally coherent:

  • state must agree with pci under the applied band thresholds; deviating thresholds require an explicit declaration.
  • dwellSeconds > 0 implies state ≠ clear.
  • pci is an integer in [0,100]; coverage, axes.*, and confidence are in [0.0,1.0]; dwellSeconds ≥ 0.
  • If present, pciWeights must sum to 1.00 (within ±0.001).
  • The sum of contaminant coverage values may exceed axes.occlusion (overlapping contamination), but each individual coverage stays in [0,1].

Chapter Summary

Four ordered states — clear / degraded / obstructed / blind — express trust posture and expected response. The SensorClarityReport is a VDA5050-style JSON with a versioned header and a per-sensor array; required fields are the header identity, plus sensorId/class/pci/state/dwellSeconds/confidence and contaminant type/coverage, while axes and pciWeights are recommended for verifiability. Dwell time measures how long a sensor has been non-CLEAR (feeding the SLA logic of Chapter 6); confidence measures certainty in the PCI itself. Consistency rules keep state, PCI, dwell, weights, and coverage mutually coherent.
