Chapter 5

API Specifications

This chapter defines comprehensive API specifications for agricultural IoT platforms, covering REST and GraphQL endpoints for farm management, sensor data ingestion, real-time updates via WebSockets, authentication and authorization patterns, and integration strategies for building interoperable agricultural data ecosystems.

1. API Architecture Principles

1.1 Design Philosophy

Agricultural IoT APIs must balance competing requirements: real-time responsiveness for operational decisions, batch efficiency for historical data queries, offline resilience for rural connectivity, and security for protecting sensitive farm data. The WIA Agricultural IoT standard defines a hybrid architecture combining REST for CRUD operations, GraphQL for flexible data retrieval, and WebSockets for real-time streaming.

APIs should embrace eventual consistency for distributed sensor networks, implement intelligent caching to reduce bandwidth in bandwidth-constrained environments, and support offline-first patterns where devices queue operations during connectivity outages. Version management through semantic versioning and deprecation policies ensures backward compatibility as standards evolve.

API Pattern Use Cases Advantages Limitations
REST Device management, field CRUD, user accounts Widely supported, cacheable, stateless Over-fetching, multiple round-trips
GraphQL Dashboard queries, analytics, reports Flexible queries, precise data retrieval Caching complexity, query cost analysis
WebSocket Live sensor feeds, alerts, status updates Real-time, bidirectional, low latency Scalability, connection management
MQTT Device-to-cloud telemetry, commands Lightweight, pub/sub, QoS levels Binary protocol, broker required

2. REST API Endpoints

2.1 Core Resource Models

The REST API organizes agricultural entities into logical resources: Farms, Fields, Crops, Devices, Sensors, and Observations. Each resource supports standard CRUD operations with consistent URL patterns, HTTP methods, and response formats. Resource relationships are expressed through hypermedia links (HATEOAS) enabling API discoverability and client evolution.

2.2 Batch Operations and Bulk Data

Agricultural IoT generates high-volume data requiring efficient batch processing. The API supports bulk observation submission where devices can queue measurements during connectivity outages and submit hundreds or thousands of readings in a single request. Batch endpoints implement idempotency through unique observation IDs, preventing duplicate processing if requests are retried.

API Rate Limiting

APIs implement tiered rate limiting: 100 requests/minute for interactive operations (dashboard queries), 1000 observations/minute for sensor data ingestion, unlimited for batch uploads via background jobs. Rate limit headers (X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset) enable clients to implement backoff strategies.

3. GraphQL API

3.1 Schema Design

GraphQL enables clients to request precisely the data they need, reducing over-fetching and under-fetching common in REST APIs. The agricultural IoT schema defines types for all domain entities with rich relationships, allowing clients to traverse from farm to fields to devices to observations in a single query. Field-level resolvers implement efficient data loading with DataLoader pattern to prevent N+1 query problems.

// TypeScript: GraphQL Schema and Resolvers import { GraphQLObjectType, GraphQLSchema, GraphQLString, GraphQLFloat, GraphQLList } from 'graphql'; // Type Definitions const FarmType = new GraphQLObjectType({ name: 'Farm', fields: () => ({ id: { type: GraphQLString }, name: { type: GraphQLString }, area: { type: GraphQLFloat }, fields: { type: new GraphQLList(FieldType), resolve: (farm, args, context) => context.dataLoaders.fieldsByFarm.load(farm.id) }, devices: { type: new GraphQLList(DeviceType), resolve: (farm, args, context) => context.dataLoaders.devicesByFarm.load(farm.id) }, statistics: { type: FarmStatisticsType, args: { period: { type: GraphQLString } }, resolve: (farm, { period }, context) => context.services.calculateFarmStats(farm.id, period) } }) }); 4. Real-Time Data Streaming

4.1 WebSocket Connections

WebSocket connections enable bidirectional real-time communication for live sensor feeds, instant alerts, and system status updates. Clients subscribe to specific data streams (field sensor updates, alert notifications, irrigation status) and receive push notifications as events occur, eliminating polling overhead.

4.2 MQTT for Device Communication

MQTT (Message Queuing Telemetry Transport) is the protocol of choice for device-to-cloud telemetry due to its lightweight binary format, pub/sub architecture, and Quality of Service levels. Devices publish sensor readings to topics organized by hierarchy (farm/field/device/sensor), and cloud services subscribe to relevant topics for ingestion and processing.

// TypeScript: Real-Time Streaming Implementation import WebSocket from 'ws'; import mqtt from 'mqtt'; // WebSocket Client for Dashboard class RealTimeAgriculturalClient { private ws: WebSocket; private subscriptions: Map; connect(url: string, apiKey: string): Promise { return new Promise((resolve, reject) => { this.ws = new WebSocket(url, { headers: { 'Authorization': `Bearer ${apiKey}` } }); this.ws.on('open', () => { console.log('WebSocket connected'); resolve(); }); this.ws.on('message', (data) => { const message = JSON.parse(data.toString()); this.handleMessage(message); }); this.ws.on('error', reject); }); } subscribe(channel: string, callback: Function): void { if (!this.subscriptions.has(channel)) { this.subscriptions.set(channel, []); // Send subscription message this.ws.send(JSON.stringify({ type: 'subscribe', channel })); } this.subscriptions.get(channel)!.push(callback); } subscribeSensorUpdates(deviceId: string, callback: (data: any) => void): void { this.subscribe(`sensor:${deviceId}`, callback); } subscribeAlerts(farmId: string, callback: (alert: any) => void): void { this.subscribe(`alerts:${farmId}`, callback); } private handleMessage(message: any): void { const { channel, data } = message; const callbacks = this.subscriptions.get(channel); if (callbacks) { callbacks.forEach(cb => cb(data)); } } } // MQTT Client for IoT Devices class MQTTDeviceClient { private client: mqtt.MqttClient; private deviceId: string; return new Promise((resolve, reject) => { this.client.on('connect', () => { console.log(`Device ${this.deviceId} connected to MQTT broker`); this.subscribeToCommands(); resolve(); }); this.client.on('error', reject); }); } async publishSensorData( sensorType: string, reading: any ): Promise { const topic = `farm/${this.getFarmId()}/device/${this.deviceId}/${sensorType}`; const message = { deviceId: this.deviceId, sensorType, timestamp: new Date().toISOString(), ...reading }; return new Promise((resolve, reject) => { this.client.publish( topic, JSON.stringify(message), { qos: 1, retain: false }, // QoS 1: At least once delivery (error) => { if (error) reject(error); else resolve(); const commandType = parts[parts.length - 1]; switch (commandType) { case 'config': this.updateConfiguration(command); break; case 'restart': this.restart(); break; case 'firmware': this.updateFirmware(command.url); break; } } private getFarmId(): string { // Extract from device configuration return 'farm-123'; } }

5. Authentication and Authorization

5.1 OAuth 2.0 and JWT

The API implements OAuth 2.0 for user authentication with JWT (JSON Web Tokens) for stateless session management. Machine-to-machine authentication for IoT devices uses client credentials flow with device-specific API keys. Fine-grained role-based access control (RBAC) ensures users can only access farms and data they own or have been granted permission to view.

5.2 API Key Management

Each device and integration receives unique API keys with scoped permissions. Keys can be time-limited, rotated regularly, and instantly revoked if compromised. Rate limiting and anomaly detection protect against API abuse and unauthorized access attempts.

Key Takeaways

  • Hybrid API architecture combines REST for CRUD, GraphQL for flexible queries, WebSocket for real-time, and MQTT for device telemetry
  • REST endpoints support batch operations with idempotency, enabling devices to queue and submit thousands of observations efficiently
  • GraphQL schema enables precise data retrieval with single queries traversing farm→field→device→sensor relationships
  • WebSocket subscriptions provide real-time sensor updates and alerts with push notifications, eliminating polling overhead
  • MQTT pub/sub architecture with QoS levels ensures reliable device-to-cloud telemetry even in unreliable network conditions
  • OAuth 2.0 + JWT authentication with RBAC authorization ensures secure, scalable access control for multi-tenant agricultural platforms

Reference Standards and Implementation Notes

This standard is designed to interoperate with established international frameworks. Implementations SHOULD treat the following as the single source of truth and use this PHASE document as the interoperability layer that maps existing data into a machine-readable, signed, and auditable shape.

  • OpenAPI Specification 3.1 — REST API definition
  • JSON Schema 2020-12 — schema validation
  • ISO/IEC 27001:2022 — information security management
  • ISO/IEC 27018:2019 — PII protection in public clouds
  • ISO/IEC 17065:2012 — product certification bodies
  • CycloneDX 1.5 / SPDX 2.3 — software bill of materials
  • Sigstore (DSSE envelope, Rekor transparency log) — signature anchoring
  • in-toto Attestation Framework 1.0 — build provenance
  • IETF RFC 9700 — OAuth 2.1
  • IETF RFC 9457 — Problem Details for HTTP APIs
  • IETF RFC 8615 — well-known URIs

Practical Implementation Checklist

  • Manifest integrity — every published manifest is signed via Sigstore DSSE. Tampering is detectable without trusting an intermediary. Updates are issued as new manifests; the prior manifest is preserved with a supersede marker.
  • Retention — keep signed evidence for at least seven years to satisfy ISO/IEC 17065:2012 audit expectations. Some regulators require longer retention; in that case the deployment policy MUST extend the window rather than relying on the defaults.
  • Versioning — follow Semantic Versioning 2.0.0. Major version bumps require at least a 90-day overlap with the prior major version on every WIA-published reference implementation.
  • Negative-result reporting — a skipped indicator without recorded justification is treated as a failed indicator at audit time.
  • Key rotation — manifest signing keys are rotated per the deployment policy. The previous key is preserved in a published key history so that auditors can verify historical manifests offline.

Reference Topologies

Implementations declare their operational topology in the manifest: single-tenant edge (one runtime per organization), multi-tenant gateway (shared runtime with header-based isolation), federated mesh (multiple runtimes peer to one another and publish manifests to a directory service), or air-gapped batch (signed evidence package transported out-of-band). Each topology ships with reference test vectors so that implementations can validate their deployment matches the canonical pattern.

Open Governance

Issues, errata, and proposals are tracked at github.com/WIA-Official/wia-standards/issues with the appropriate label. The WIA Standards working group reviews open issues at the start of every minor release cycle and publishes the resulting decision log alongside the release notes. Errata are issued as patch releases; new normative requirements trigger minor bumps; backwards-incompatible changes trigger major bumps.
