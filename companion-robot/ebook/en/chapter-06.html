6.1 Protocol Architecture Overview

Phase 3 of WIA-ROB-013 establishes communication protocols that enable real-time interaction, multi-modal data synchronization, and seamless coordination between companion robots, cloud services, and edge devices. These protocols prioritize low latency, reliability, and privacy while supporting diverse communication patterns.

6.1.1 Protocol Design Goals

  • Real-Time Performance: Sub-second latency for interactive experiences
  • Reliability: Guaranteed message delivery for critical operations
  • Scalability: Support for thousands of concurrent robot connections
  • Security: End-to-end encryption and authentication
  • Bandwidth Efficiency: Optimized for varied network conditions
  • Offline Resilience: Graceful degradation when connectivity is lost

6.2 Real-Time Messaging Protocol

WIA-ROB-013 uses WebSocket and MQTT protocols for real-time bidirectional communication between robots and cloud services.

6.2.1 WebSocket Connection Establishment

// Client connection request
wss://realtime.companion-robot.wia.org/v1/connect

Connection Headers:
  Authorization: Bearer {access_token}
  Sec-WebSocket-Protocol: wia-rob-013-v1
  X-Robot-ID: {robot_serial_number}
  X-User-ID: {user_id}

// Server handshake response
{
  "type": "connection_established",
  "connectionId": "conn-uuid",
  "serverTime": "2025-01-15T14:30:00Z",
  "protocolVersion": "1.0.0",
  "capabilities": [
    "real_time_emotions",
    "streaming_audio",
    "behavior_control"
  ]
}

6.2.2 Message Frame Structure

All WebSocket messages follow a standardized frame format:

{
  "messageId": "msg-uuid",
  "timestamp": "2025-01-15T14:30:00.123Z",
  "type": "emotion_detected|user_utterance|robot_response|behavior_command",
  "priority": "critical|high|normal|low",
  "sequenceNumber": 12345,

  "payload": {
    // Type-specific data
  },

  "metadata": {
    "sessionId": "session-uuid",
    "userId": "user-uuid",
    "robotId": "robot-serial"
  },

  "acknowledgment": {
    "required": true,
    "timeout": 5000
  }
}

6.2.3 Message Types and Flow

Message Type Direction Description Priority
emotion_detected Robot → Cloud Real-time emotion detection results Normal
user_utterance Robot → Cloud Streaming speech transcription High
robot_response Cloud → Robot Generated response with behavior High
behavior_command Cloud → Robot Execute specific behavior Normal
emergency_alert Robot → Cloud Critical incident notification Critical
heartbeat Bidirectional Connection keep-alive Low

6.3 MQTT for IoT Integration

For resource-constrained devices and IoT ecosystem integration, WIA-ROB-013 supports MQTT protocol alongside WebSocket.

6.3.1 MQTT Topic Hierarchy

Topic Structure:
  wia/rob013/{robot_id}/{category}/{subcategory}

Examples:
  wia/rob013/robot-001/emotions/detected
  wia/rob013/robot-001/speech/utterance
  wia/rob013/robot-001/behaviors/command
  wia/rob013/robot-001/health/vitals
  wia/rob013/robot-001/alerts/emergency

6.3.2 Quality of Service Levels

Message Category QoS Level Rationale
Emotion detections QoS 0 (At most once) High frequency, tolerance for occasional loss
User utterances QoS 1 (At least once) Important conversational data, duplicates acceptable
Emergency alerts QoS 2 (Exactly once) Critical safety events, no loss or duplication
Behavior commands QoS 1 (At least once) Ensure delivery, idempotent execution

6.3.3 MQTT Message Example

// Publishing emotion detection
Topic: wia/rob013/robot-001/emotions/detected
QoS: 0
Retain: false

Payload:
{
  "emotionId": "emotion-uuid",
  "timestamp": "2025-01-15T14:30:00Z",
  "primaryEmotion": {
    "category": "joy",
    "confidence": 0.87
  },
  "sessionId": "session-uuid"
}

6.4 Multi-Modal Data Synchronization

Companion robots collect data from multiple sensors simultaneously. The synchronization protocol ensures temporal alignment of multi-modal streams.

6.4.1 Timestamp Synchronization

All devices must synchronize their clocks using NTP (Network Time Protocol) or PTP (Precision Time Protocol) to ensure accurate temporal correlation.

Synchronization Requirements

  • Clock Accuracy: ±50ms maximum drift from UTC
  • Sync Frequency: Clock synchronization every 5 minutes minimum
  • Timestamp Format: ISO 8601 with millisecond precision
  • Timezone: All timestamps in UTC

6.6 Edge-Cloud Coordination Protocol

Many companion robots use hybrid edge-cloud architectures. The coordination protocol enables intelligent workload distribution between local processing and cloud services.

6.6.1 Task Distribution Strategy

Task Type Processing Location Rationale
Face detection Edge (on-device) Low latency, privacy preservation
Basic emotion recognition Edge (on-device) Real-time feedback, offline capability
Advanced emotion analysis Cloud Complex models, higher accuracy
Speech recognition Hybrid Local for simple commands, cloud for complex
Conversational AI Cloud Large language models, knowledge access
Behavior execution Edge (on-device) Real-time motor control

6.6.2 Task Handoff Protocol

// Edge to Cloud handoff
{
  "type": "task_handoff",
  "taskId": "task-uuid",
  "handoffReason": "complexity_threshold|resource_exhaustion|accuracy_required",

  "edgeResults": {
    "preliminaryEmotion": "neutral",
    "confidence": 0.62,
    "processingTime": 85
  },

  "cloudRequest": {
    "taskType": "advanced_emotion_analysis",
    "inputData": "...",
    "priority": "high",
    "maxLatency": 500
  }
}

// Cloud response
{
  "type": "task_result",
  "taskId": "task-uuid",
  "refinedResults": {
    "emotion": "contentment",
    "confidence": 0.88
  },
  "processingTime": 245,
  "totalLatency": 330
}

6.8.2 Reconnection and Sync

Synchronization Procedure

  1. Reconnect: Establish secure connection to cloud services
  2. Time Sync: Resynchronize device clock with NTP
  3. State Exchange: Compare local and cloud state versions
  4. Conflict Resolution: Apply conflict resolution rules (latest-write-wins, manual review, merge)
  5. Queue Upload: Upload queued data in priority order
  6. Confirmation: Receive acknowledgment and delete local queue

Key Takeaways

  • Phase 3 protocols enable real-time bidirectional communication using WebSocket and MQTT
  • Standardized message frames ensure consistent structure across all protocol communications
  • MQTT topic hierarchy organizes data streams with appropriate Quality of Service levels
  • Multi-modal synchronization protocols align data from different sensors with sub-100ms precision
  • Streaming audio protocol supports real-time voice conversations with adaptive quality
  • Edge-cloud coordination optimizes workload distribution between local and remote processing
  • Robot-to-robot communication enables multi-device coordination in shared environments
  • Offline operation protocol ensures graceful degradation and synchronization upon reconnection
  • TLS 1.3 encryption mandatory for all protocol communications with optional message-level encryption
  • Built-in monitoring provides real-time performance metrics and health diagnostics
