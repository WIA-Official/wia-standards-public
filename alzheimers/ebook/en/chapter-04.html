Chapter 4

Phase 1: Data Format Standards

Standardized schemas for NAD+ homeostasis, pathology tracking, and treatment response

4.1 Introduction to WIA-ALZHEIMERS Data Standards

The WIA-ALZHEIMERS standard defines a comprehensive set of data formats for representing, exchanging, and analyzing Alzheimer's disease clinical and research data with a focus on NAD+ homeostasis. These standards enable interoperability between clinical sites, research institutions, and health systems worldwide, facilitating data aggregation, meta-analysis, and the development of AI-powered diagnostic and therapeutic tools.

Design Principles

  • JSON-based: All data formats use JSON (JavaScript Object Notation) for human readability and universal compatibility
  • Schema validation: JSON Schema specifications ensure data integrity and enable automated validation
  • Semantic clarity: Field names are self-documenting with clear units and ranges
  • Extensibility: Custom fields can be added while maintaining core compatibility
  • Version control: Explicit schema versioning enables evolution while maintaining backward compatibility
  • FAIR principles: Data is Findable, Accessible, Interoperable, and Reusable

Why JSON for Medical Data?

JSON offers several advantages for medical data exchange:

  • Human-readable and writable (unlike binary formats)
  • Native support in all modern programming languages and databases
  • Lightweight compared to XML (less overhead)
  • Flexible schema validation with JSON Schema
  • Compatible with NoSQL databases and modern APIs
  • Easy integration with web applications and mobile apps

Core Data Objects

The WIA-ALZHEIMERS Phase 1 standard defines five core data object types:

Object Type Purpose Primary Use Cases
NAD+ Homeostasis Index Quantify NAD+ metabolic status Baseline assessment, treatment monitoring, research endpoints
Pathology Profile Document AD pathological burden Diagnosis, staging, treatment selection, outcome tracking
Treatment Response Track therapeutic interventions and outcomes Clinical care, trial data collection, comparative effectiveness
Patient Demographics Standardized demographic and clinical context Risk stratification, subgroup analysis, health equity research
Cognitive Assessment Standardized cognitive testing results Diagnosis, monitoring, trial endpoints

NHI Calculation Implementation

calculate-nhi.js
function calculateNHI(data) {
  const WEIGHTS = {
    L: 0.35,  // NAD+ level
    R: 0.25,  // NAD+/NADH ratio
    S: 0.25,  // Sirtuin activity
    P: 0.15   // PARP activity (inverse)
  };

  // Extract normalized values
  const L = data.nad_level.normalized_value;
  const R = data.nad_nadh_ratio;
  const S = data.sirtuin_activity.value;
  const P = data.parp_activity.value;

  // Calculate NHI (note: P is subtracted as higher PARP = worse)
  const nhi = WEIGHTS.L * L + WEIGHTS.R * R + WEIGHTS.S * S - WEIGHTS.P * P;

  // Determine category
  let category;
  if (nhi >= 0.8) category = "healthy";
  else if (nhi >= 0.5) category = "mild_depletion";
  else if (nhi >= 0.3) category = "moderate_depletion";
  else category = "severe_depletion";

  return {
    nhi_score: Math.round(nhi * 1000) / 1000,  // Round to 3 decimals
    nhi_category: category
  };
}

// Example usage
const patientData = {
  nad_level: { normalized_value: 0.42 },
  nad_nadh_ratio: 0.28,
  sirtuin_activity: { value: 0.31 },
  parp_activity: { value: 3.2 }
};

const result = calculateNHI(patientData);
console.log(result);
// Output: { nhi_score: -0.185, nhi_category: "severe_depletion" }

4.4 Treatment Response Schema

The Treatment Response schema tracks therapeutic interventions and their effects over time, enabling systematic evaluation of treatment efficacy and safety.

Schema Overview

Treatment Response Data Structure

The schema captures:

  • Intervention details: Drug name, dose, route, frequency, start/stop dates
  • Baseline assessments: NHI, cognition, pathology before treatment
  • Serial assessments: Time-series data showing response trajectory
  • Adverse events: Side effects, dose modifications, discontinuations
  • Outcomes: Clinical response classification, quality of life changes

4.5 Data Validation and Quality Control

Ensuring data quality is critical for reliable research and clinical decision-making. The WIA-ALZHEIMERS standard includes comprehensive validation rules and quality control procedures.

Validation Rules

Rule Category Examples Enforcement Level
Type validation Numbers must be numeric, dates must be ISO 8601 Hard (schema validation)
Range validation MMSE 0-30, NHI typically -2.0 to 2.0 Hard (schema validation)
Logical consistency Follow-up dates must be after baseline Hard (application logic)
Biological plausibility NAD+ levels should be >0, CSF Aβ42 typically 200-1500 pg/mL Soft (warnings)
Completeness Required fields must be present Hard (schema validation)
Cross-field validation NAD+/NADH ratio should match individual values Soft (warnings)

Validation Implementation Example

validate-nhi-data.js
const Ajv = require('ajv');
const ajv = new Ajv({ allErrors: true });

// Load schema
const nhiSchema = require('./nad-homeostasis-index-v1.0.schema.json');
const validate = ajv.compile(nhiSchema);

function validateNHIData(data) {
  const errors = [];
  const warnings = [];

  // Schema validation (hard errors)
  const valid = validate(data);
  if (!valid) {
    errors.push(...validate.errors.map(err => ({
      type: 'schema_violation',
      message: `${err.dataPath} ${err.message}`,
      severity: 'error'
    })));
  }

  // Cross-field validation
  if (data.nad_level && data.nadh_level) {
    const calculatedRatio = data.nad_level.value / data.nadh_level.value;
    const reportedRatio = data.nad_nadh_ratio;

    if (Math.abs(calculatedRatio - reportedRatio) > 0.05) {
      warnings.push({
        type: 'consistency_check',
        message: `NAD+/NADH ratio mismatch: calculated ${calculatedRatio.toFixed(3)} vs reported ${reportedRatio}`,
        severity: 'warning'
      });
    }
  }

  // Biological plausibility checks
  if (data.nad_level?.normalized_value > 1.5) {
    warnings.push({
      type: 'plausibility',
      message: `Unusually high NAD+ level (${data.nad_level.normalized_value}x reference)`,
      severity: 'warning'
    });
  }

  if (data.nad_level?.normalized_value < 0.1) {
    warnings.push({
      type: 'plausibility',
      message: `Extremely low NAD+ level (${data.nad_level.normalized_value}x reference) - verify measurement`,
      severity: 'warning'
    });
  }

  // Quality control checks
  if (data.quality_control?.hemolysis === true) {
    warnings.push({
      type: 'quality_control',
      message: 'Hemolysis detected - NAD+ measurements may be elevated',
      severity: 'warning'
    });
  }

  if (data.quality_control?.time_to_processing > 60) {
    warnings.push({
      type: 'quality_control',
      message: `Prolonged time to processing (${data.quality_control.time_to_processing} min) may affect NAD+ stability`,
      severity: 'warning'
    });
  }

  return {
    valid: errors.length === 0,
    errors,
    warnings
  };
}

// Example usage
const patientData = { /* ... patient NHI data ... */ };
const result = validateNHIData(patientData);

if (result.valid) {
  console.log('✓ Data is valid');
  if (result.warnings.length > 0) {
    console.log('⚠ Warnings:', result.warnings);
  }
} else {
  console.log('✗ Validation errors:', result.errors);
}

4.7 Privacy, Security, and De-identification

Patient privacy and data security are paramount in the WIA-ALZHEIMERS standard.

De-identification Requirements

Protected Health Information (PHI) Removal

All data submitted to WIA-ALZHEIMERS compliant systems MUST be de-identified according to HIPAA Safe Harbor or Expert Determination methods. The following identifiers must be removed or pseudonymized:

  • Names, addresses, contact information
  • Dates (except year) - use age or time from baseline instead
  • Medical record numbers (replace with study IDs)
  • Social security numbers
  • Any other identifying information

Patient IDs must be cryptographic pseudonyms (minimum 8 characters, alphanumeric) that cannot be reversed to identify individuals without access to a secure mapping key stored separately.

Data Encryption and Transmission Security

  • Transport encryption: All API communication must use TLS 1.3 or higher
  • At-rest encryption: Databases must use AES-256 encryption
  • Authentication: OAuth 2.0 or API keys with regular rotation
  • Authorization: Role-based access control (RBAC) with minimal necessary permissions
  • Audit logging: All data access must be logged with timestamp, user, and action

4.8 Future Extensions and Versioning

The WIA-ALZHEIMERS data standard is designed for evolution as science advances.

Planned Schema Extensions

Extension Planned Version Description
Genetic data v1.1 APOE status, polygenic risk scores, NAD+ metabolism SNPs
Lifestyle factors v1.1 Exercise, diet, sleep quality (factors affecting NAD+)
Multi-omics v1.2 Metabolomics, proteomics, transcriptomics integration
Digital biomarkers v1.2 Wearable device data, smartphone cognitive assessments
AI predictions v1.3 Risk scores, treatment response predictions

Version Compatibility

The WIA-ALZHEIMERS standard follows semantic versioning (MAJOR.MINOR.PATCH):

  • MAJOR: Incompatible changes (rare, requires data migration)
  • MINOR: Backward-compatible feature additions
  • PATCH: Backward-compatible bug fixes

Forward Compatibility Design

Systems implementing WIA-ALZHEIMERS v1.0 will automatically accept v1.x data by ignoring unknown fields. This allows researchers to extend schemas with custom fields while maintaining interoperability with the core standard.

Chapter Summary: Key Takeaways

  1. WIA-ALZHEIMERS Phase 1 defines five core JSON-based data objects: NAD+ Homeostasis Index, Pathology Profile, Treatment Response, Patient Demographics, and Cognitive Assessment, enabling universal data interoperability.
  2. The NAD+ Homeostasis Index schema captures all components (L, R, S, P) with full metadata including specimen type, measurement methods, quality control metrics, and calculated NHI score with category classification.
  3. Pathology Profile schema encompasses comprehensive AD biomarkers including CSF (Aβ42/40, p-tau181/217, total tau, NfL), plasma markers, imaging (amyloid PET, tau PET, MRI volumetrics), and ATN classification framework.

  5. Comprehensive validation includes schema-based type and range checks, logical consistency rules, biological plausibility warnings, and quality control flags to ensure data integrity and reliability.
  6. RESTful API standards facilitate seamless integration with health IT systems through standardized endpoints for data submission, retrieval, and validation, with OAuth 2.0 authentication and TLS 1.3 encryption.
  7. Strict de-identification requirements (HIPAA Safe Harbor), cryptographic pseudonymization, and role-based access control ensure patient privacy while enabling global data sharing for research and clinical advancement.
