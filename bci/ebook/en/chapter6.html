Chapter 6: Security & Privacy

Neural Data Protection and Consent for Brain Data

WIA BCI Standard v1.0.0

6.1 Overview

Brain data is among the most sensitive personal information. The WIA BCI standard incorporates privacy-by-design principles and security best practices to protect neural data throughout its lifecycle.

Critical Importance

Brain signals can reveal cognitive states, medical conditions, emotions, and potentially even thoughts. Protecting this data is not just a legal requirement—it's an ethical imperative.

6.2 Threat Model

Attack Vectors

Threat Description Mitigation
Data Interception Eavesdropping on BCI streams End-to-end encryption
Unauthorized Access Access to stored recordings Authentication, access control
Device Hijacking Control of BCI device Secure pairing, authentication
Data Leakage Unintentional disclosure Data minimization, anonymization
Re-identification Linking anonymized data to individuals Differential privacy, k-anonymity

6.3 Data Encryption

Transport Layer Security

// WebSocket with TLS
const bci = new WiaBci();
await bci.connect({
  type: 'eeg_headset',
  connection: {
    protocol: 'websocket',
    address: 'wss://secure-server.example.com',  // WSS (not WS)
    tls: {
      enabled: true,
      version: 'TLSv1.3',
      ciphers: ['TLS_AES_256_GCM_SHA384']
    }
  }
});

End-to-End Encryption

// Enable E2E encryption for sensitive applications
await bci.connect({
  type: 'eeg_headset',
  encryption: {
    enabled: true,
    algorithm: 'AES-256-GCM',
    keyExchange: 'ECDHE',
    publicKey: userPublicKey
  }
});

// Data is encrypted before transmission
// Only the recipient with the private key can decrypt

At-Rest Encryption

import { encryptRecording, decryptRecording } from 'wia-bci';

// Encrypt recording before saving
const encrypted = await encryptRecording({
  recording: './raw-data',
  algorithm: 'AES-256-GCM',
  password: userPassword,  // Or use key management system
  outputDir: './encrypted-data'
});

// Decrypt for analysis
const decrypted = await decryptRecording({
  recording: './encrypted-data',
  password: userPassword,
  outputDir: './decrypted-data'
});

Role-Based Access Control

// Define roles
enum Role {
  PATIENT = 'patient',        // Can view own data
  CLINICIAN = 'clinician',    // Can view assigned patients
  RESEARCHER = 'researcher',  // Can access anonymized data
  ADMIN = 'admin'             // Full access
}

// Check permissions
function canAccessRecording(user: User, recording: Recording): boolean {
  switch (user.role) {
    case Role.PATIENT:
      return recording.subjectId === user.id;
    case Role.CLINICIAN:
      return user.assignedPatients.includes(recording.subjectId);
    case Role.RESEARCHER:
      return recording.anonymized === true;
    case Role.ADMIN:
      return true;
    default:
      return false;
  }
}

6.5 Consent Management

Informed Consent

Required Elements

  • Purpose: Why brain data is being collected
  • Data Use: How data will be used, stored, shared
  • Risks: Potential privacy risks and mitigation
  • Rights: Right to withdraw, access, delete data
  • Duration: How long data will be retained
  • Third Parties: Who will have access to data

Consent Schema

interface Consent {
  // Consent Information
  consentId: string;
  subjectId: string;
  consentDate: string;  // ISO 8601
  expiryDate?: string;
  version: string;      // Consent form version

  // Consent Scope
  purposes: ConsentPurpose[];
  dataTypes: DataType[];
  sharingScope: SharingScope;

  // Rights Acknowledged
  rightsAcknowledged: {
    rightToWithdraw: boolean;
    rightToAccess: boolean;
    rightToDelete: boolean;
    rightToPortability: boolean;
  };

  // Legal Basis
  legalBasis: 'consent' | 'contract' | 'legal_obligation' | 'vital_interests';
  jurisdiction: string;  // e.g., "EU", "US-CA", "KR"

  // Ethics Approval
  ethicsApproval?: {
    committee: string;
    approvalNumber: string;
    approvalDate: string;
  };

  // Signature
  signature: {
    type: 'electronic' | 'written';
    timestamp: string;
    ipAddress?: string;
    witnessId?: string;
  };
}

type ConsentPurpose =
  | 'medical_treatment'
  | 'clinical_research'
  | 'product_improvement'
  | 'academic_research'
  | 'commercial_use';

type SharingScope =
  | 'no_sharing'
  | 'anonymized_only'
  | 'institution_only'
  | 'trusted_partners'
  | 'public_dataset';

Dynamic Consent

// Allow users to update consent
class ConsentManager {
  async updateConsent(
    subjectId: string,
    updates: Partial<Consent>
  ): Promise<void> {
    const currentConsent = await this.getConsent(subjectId);

    // Validate updates
    if (updates.sharingScope && updates.sharingScope !== currentConsent.sharingScope) {
      // Re-anonymize or un-share data if scope reduced
      await this.adjustDataSharing(subjectId, updates.sharingScope);
    }

    // Save updated consent
    await this.saveConsent({
      ...currentConsent,
      ...updates,
      lastUpdated: new Date().toISOString()
    });

    // Notify relevant parties
    await this.notifyConsentChange(subjectId, updates);
  }

  async withdrawConsent(subjectId: string): Promise<void> {
    // Stop any active data collection
    await this.stopDataCollection(subjectId);

    // Delete or anonymize existing data (per user request)
    await this.handleDataDeletion(subjectId);

    // Record withdrawal
    await this.recordWithdrawal(subjectId);
  }
}

6.6 Data Anonymization

De-identification Techniques

import { anonymizeRecording } from 'wia-bci';

const anonymized = await anonymizeRecording({
  recording: './raw-data',
  method: 'k-anonymity',
  k: 5,  // Each record indistinguishable from 4 others
  techniques: [
    'remove_identifiers',    // Remove name, ID, etc.
    'generalize_age',        // Age ranges instead of exact age
    'suppress_rare_values',  // Remove rare conditions
    'perturb_timestamps',    // Add noise to timestamps
    'aggregate_location'     // Generalize location data
  ],
  outputDir: './anonymized-data'
});

// Verify anonymization
const report = await validateAnonymization('./anonymized-data');
console.log('Re-identification risk:', report.riskScore);  // < 0.05 recommended

Differential Privacy

import { addDifferentialPrivacy } from 'wia-bci';

// Add noise to protect individual records

6.7 Secure Storage

Storage Best Practices

  • Encryption: Encrypt all stored brain data
  • Access Logs: Audit trail of all data access
  • Backup Security: Encrypt backups separately
  • Retention Policy: Delete data after retention period
  • Secure Deletion: Overwrite data multiple times

Key Management

const keyManager = new KeyManager({

  keyId: 'arn:aws:kms:...'

// Encrypt with managed key
const encrypted = await keyManager.encrypt(sensitiveData);

// Keys are rotated automatically

// Audit key usage
const auditLog = await keyManager.getAuditLog();

6.8 Regulatory Compliance

GDPR (EU)

Key Requirements

  • Lawful Basis: Explicit consent or legal basis for processing
  • Data Minimization: Collect only necessary data
  • Purpose Limitation: Use data only for stated purposes
  • Right to Access: Provide data upon request
  • Right to Erasure: Delete data upon request ("right to be forgotten")
  • Data Portability: Export data in machine-readable format
  • Breach Notification: Report breaches within 72 hours

HIPAA (US)

Key Requirements

  • Protected Health Information (PHI): Brain data is PHI
  • Minimum Necessary: Access only what's needed
  • Administrative Safeguards: Policies, training, audits
  • Physical Safeguards: Secure facilities, device controls
  • Technical Safeguards: Encryption, access controls, audit logs
  • Business Associate Agreements: Contracts with third parties

Compliance Implementation

class ComplianceManager {
  async handleDataRequest(request: DataRequest): Promise<void> {
    switch (request.type) {
      case 'access':
        // GDPR Article 15: Right to access
        return await this.provideDataCopy(request.subjectId);

      case 'rectification':
        // GDPR Article 16: Right to rectification
        return await this.correctData(request.subjectId, request.corrections);

      case 'erasure':
        // GDPR Article 17: Right to erasure
        return await this.deleteData(request.subjectId);

      case 'portability':
        // GDPR Article 20: Right to data portability
        return await this.exportData(request.subjectId, 'json');

      case 'restriction':
        // GDPR Article 18: Right to restriction
        return await this.restrictProcessing(request.subjectId);

      default:
        throw new Error('Unknown request type');
    }
  }

  async reportBreach(breach: SecurityBreach): Promise<void> {
    // GDPR Article 33: Breach notification
    if (breach.severity === 'high') {
      await this.notifySupervisoryAuthority(breach);  // Within 72 hours
      await this.notifyDataSubjects(breach);         // Without undue delay
    }

    // Log breach
    await this.logBreach(breach);
  }
}

6.9 Audit Logging

Audit Log Schema

interface AuditLog {
  id: string;
  timestamp: string;
  eventType: AuditEventType;
  actor: {
    userId: string;
    role: string;
    ipAddress: string;
    userAgent?: string;
  };
  resource: {
    type: 'recording' | 'device' | 'consent' | 'key';
    id: string;
  };
  action: 'create' | 'read' | 'update' | 'delete' | 'export';
  result: 'success' | 'failure';
  details?: Record<string, unknown>;
}

type AuditEventType =
  | 'data_access'
  | 'data_modification'
  | 'consent_change'
  | 'encryption_key_use'
  | 'authentication'
  | 'authorization_failure'
  | 'data_export'
  | 'data_deletion';

Tamper-Proof Logging

import { AuditLogger } from 'wia-bci';

const logger = new AuditLogger({
  storage: 'blockchain',  // Immutable audit trail
  encryption: true
});

// Log every data access
bci.on('signal', (event) => {
  logger.log({
    eventType: 'data_access',
    actor: { userId: currentUser.id, role: currentUser.role },
    resource: { type: 'recording', id: recording.id },
    action: 'read',
    result: 'success'
  });
});

// Query audit log
const accessLog = await logger.query({
  resource: { id: 'rec-001' },
  dateRange: { start: '2025-01-01', end: '2025-01-31' }

6.10 Security Checklist

Development Phase

  • ☐ Use TLS 1.3 for all network connections
  • ☐ Implement end-to-end encryption for sensitive data
  • ☐ Validate all inputs to prevent injection attacks
  • ☐ Use secure random number generators
  • ☐ Implement proper error handling (no sensitive info in errors)
  • ☐ Use parameterized queries to prevent SQL injection
  • ☐ Sanitize all user inputs
  • ☐ Implement rate limiting and throttling

Deployment Phase

  • ☐ Enable audit logging
  • ☐ Configure firewall rules
  • ☐ Set up intrusion detection
  • ☐ Enable automatic security updates
  • ☐ Configure secure backup procedures
  • ☐ Implement key rotation policies
  • ☐ Set up monitoring and alerting
  • ☐ Conduct penetration testing

Operational Phase

  • ☐ Regular security audits
  • ☐ Vulnerability scanning
  • ☐ Review access logs monthly
  • ☐ Update consent forms as needed
  • ☐ Conduct staff security training
  • ☐ Test incident response procedures
  • ☐ Review and update retention policies
  • ☐ Maintain compliance documentation

6.11 Incident Response

Incident Response Plan

class IncidentResponsePlan {
  async handleSecurityIncident(incident: SecurityIncident): Promise<void> {
    // 1. Contain
    await this.containIncident(incident);

    // 2. Assess
    const assessment = await this.assessImpact(incident);

    // 3. Notify
    if (assessment.requiresNotification) {
      await this.notifyAuthorities(assessment);
      await this.notifyAffectedUsers(assessment);
    }

    // 4. Remediate
    await this.remediateVulnerability(incident);

    // 5. Document
    await this.documentIncident(incident, assessment);

    // 6. Review
    await this.conductPostMortem(incident);
  }
}
